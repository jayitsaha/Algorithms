LeetCode #416

Partition Equal Subset Sum

Given an integer array nums, return true if you can partition the array into two subsets such that the sum of elements in both subsets is equal.

Medium Dynamic Programming 0/1 Knapsack Time: O(n · sum/2) Space: O(sum/2)
1 Understanding the Problem ▾

You're given a list of positive integers. Can you split it into two groups where both groups sum to the same value? Each number goes entirely into one group — no splitting. You cannot cut numbers in half.

💡 The Core Reframe

If two subsets are equal, each must equal total / 2. So the problem reduces to: "can you find any subset that sums to exactly target = sum(nums) / 2?" Find one subset and the rest automatically forms the other.

✓ Possible: [1, 5, 11, 5]

Total = 22, target = 11
Subset A: {11} = 11 ✓
Subset B: {1, 5, 5} = 11 ✓
→ True

✗ Impossible: [1, 2, 3, 5]

Total = 11, which is odd
No integer halves of 11 exist
→ False immediately

Key edge cases: odd total → always False (can't halve evenly). Single element → False (nothing to partition into). Any element greater than target → False (can't fit in either half).

⚠️ Why Not Greedy?

A greedy approach (sort descending, assign to lighter group) fails. For [3, 3, 3, 4, 5]: greedy assigns 5→A, 4→B, 3→A, 3→B, 3→A giving A=11, B=7. But the correct split is {5,4,3}=12 and {3,3,6}... Actually greedy can miss valid solutions when early choices foreclose later options. DP explores all possibilities systematically.

2 The Algorithm: 0/1 Knapsack DP ▾

This is the classic 0/1 Knapsack in disguise. The "knapsack" has capacity target = sum/2. Each number is an "item" that can fill part of the knapsack. We ask: can items exactly fill the knapsack? Each item either goes in (1) or stays out (0) — never split.

We maintain a boolean DP array: dp[j] = "can we achieve sum exactly j using items seen so far?" We start with only dp[0] = True (empty subset sums to 0), then process each number.

1 Quick-fail: Compute total. If total % 2 != 0 → return False (can't split odd sum). Set target = total // 2.
2 Initialize: dp = [False] × (target+1). Set dp[0] = True. The empty subset always achieves sum 0.
3 Process items right-to-left: For each num, scan j from target down to num. Update: dp[j] |= dp[j - num]. "If we could reach j - num before, we can now reach j by taking num."
4 Early exit + answer: After each item, check dp[target]. If True → return True immediately. After all items, return dp[target].

The binary decision at each item:

SKIP num
dp[j] unchanged
dp[j] |= dp[j]
For each num, each j
(right to left)
dp[j] |= dp[j - num]
TAKE num
dp[j] |= dp[j-num]
inherit reachability
🔑 Why Iterate Right-to-Left? (The 0/1 Constraint)

If we went left-to-right, when we process j=5 and set dp[5]=True from dp[0], then later reach j=10 and check dp[5] — we'd be "using num twice." Right-to-left guarantees dp[j-num] still reflects state before this item was processed, enforcing each item is used at most once.

3 Why It Works: Optimal Substructure & Overlapping Subproblems ▾

Why is this Dynamic Programming? Two conditions must hold simultaneously:

✓ Optimal Substructure

Whether we can reach sum j using items 0..i depends only on whether we could reach j or j - nums[i] using items 0..i-1. Each step builds on the previous.

✓ Overlapping Subproblems

Naive recursion recomputes "can I reach sum j from index i?" across many branches. The same (i, remaining) pair appears exponentially often in the recursion tree.

🔐 The Invariant

After processing the first k items: dp[j] = True if and only if some subset of nums[0..k-1] sums exactly to j. Established at init (dp[0]=True), maintained at every step: skip → dp[j] unchanged; take → dp[j] inherits dp[j-num] which was True for pre-item subsets.

📐 The Recurrence Relation
dp[j] = dp[j] OR dp[j - num]   // after processing item num

Base: dp[0] = True   // empty subset sums to 0, always
Base: dp[j] = False for j > 0   // nothing processed yet
Answer: dp[target]   // where target = total // 2

Full DP table trace (nums=[1,5,11,5], target=11) — each row is dp state after that item:

After item j=0j=1j=2j=3j=4j=5j=6j=10j=11
init TFFFFFFFF
num=1 TTFFFFFFF
num=5 TTFFFTTFF
num=11 TTFFFTTFT ✓

After num=11: dp[11] becomes True because dp[11-11]=dp[0]=True. We hit target early — return True!

⚠️ The Direction Trap: Left-to-Right = Wrong for 0/1 Knapsack

If you write for j in range(num, target+1) (left-to-right), consider num=5, dp=[T,F,F,F,F,T,...]: processing j=5 sets dp[5]=True from dp[0]. Then j=10 checks dp[5] which is already True from this very pass — accidentally using num=5 twice. The right-to-left scan prevents this: when we check dp[j-num], it still has its "before this item" value.

4 The Code (Annotated) ▾

Solution 1 — DP Tabulation / Bottom-Up (Interview Pick)

def canPartition(nums): ← O(n·T) time, O(T) space, T = sum//2
total = sum(nums) ← compute total once
if total % 2 != 0: ← odd total → can't make equal halves
return False ← O(1) early exit
target = total // 2 ← find one subset summing to this
dp = [False] * (target + 1) ← dp[j] = "can we make sum j?"
dp[0] = True ← base: empty subset → sum 0
for num in nums: ← each item processed exactly once (0/1)
for j in range(target, num - 1, -1): ← RIGHT-TO-LEFT: j from target down to num
dp[j] |= dp[j - num] ← take num: if j-num reachable, so is j
if dp[target]: ← check after each item for early exit
return True ← found! no need to process rest
return dp[target] ← final answer
💡 Complexity (Tabulation)

Time: O(n × target) = O(n × sum/2). Space: O(target) = O(sum/2). For LeetCode constraints (n≤200, values≤100), target ≤ 10,000. At most 2,000,000 operations — very fast.

Solution 2 — Memoization / Top-Down

def canPartition(nums):
total = sum(nums)
if total % 2 != 0: return False
target = total // 2
memo = {} ← cache: (index, remaining) → bool
def dp(i, rem): ← can we reach rem=0 from index i?
if rem == 0: return True ← hit target!
if rem < 0 or i == len(nums): return False ← overshot or out of items
if (i, rem) in memo: return memo[(i, rem)] ← cache hit: skip recomputation
result = (dp(i+1, rem) or ← SKIP nums[i]
dp(i+1, rem - nums[i])) ← TAKE nums[i], reduce budget
memo[(i, rem)] = result ← cache the answer
return result
return dp(0, target) ← start: index 0, full budget
⚠️ Memoization State: What Are We Caching?

Each state is (i, remaining): "at index i with remaining budget left, can we reach 0?" Without memo, the call tree has 2^n leaves. With memo, there are at most n × target unique states — the same O(n × target) as tabulation, but with extra recursion overhead (function calls, stack depth). Tabulation is preferred for large inputs.

Solution 3 — Brute Force Backtracking (O(2^n) — understand, don't submit)

def canPartition(nums):
total = sum(nums)
if total % 2: return False
target = total // 2
def bt(i, rem): ← no cache = exponential time
if rem == 0: return True
if rem < 0 or i == len(nums): return False
return bt(i+1, rem) or bt(i+1, rem - nums[i])← 2 branches per item → 2^n total
return bt(0, target) ← TLE for n > 20
5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Reframe to Subset Sum

"If a partition into two equal halves exists, each half equals total/2. If total is odd → impossible (return False). So I need: does any subset sum to total//2? This is the classic Subset Sum problem."

Step 2: Identify as DP — Name the Overlapping Subproblems

"Brute force tries all 2^n subsets — too slow. But 'can I reach sum j using items 0..i?' is asked repeatedly for the same (i, j) pairs. That's a DP signal. State: (item index, remaining budget). At most n × target unique states."

Step 3: Choose Bottom-Up with 1D Array

"I'll use a 1D boolean array dp[0..target]. dp[j]=True means some subset reaches sum j. For each item, I update right-to-left (to prevent reuse). This collapses the 2D DP table into O(target) space."

Step 4: Justify Right-to-Left

"Left-to-right would allow using the same item multiple times (unbounded knapsack). Right-to-left ensures dp[j-num] reflects the state before the current item was processed — enforcing the 0/1 constraint."

🗣️ What to Say to the Interviewer

"Two equal subsets → each equals total/2. Odd total → False.
Now it's Subset Sum: can some subset reach target = total//2?
Brute force is O(2^n). I'll use DP: dp[j] = can we make sum j?
Init: dp[0]=True. For each num: right-to-left, dp[j] |= dp[j-num].
Right-to-left = 0/1 constraint (each item used once).
Time O(n×target), Space O(target). Early exit if dp[target] becomes True."
🔮 Likely Follow-up Questions
  • "What if items are reusable?" → Change right-to-left to left-to-right (Coin Change / Unbounded Knapsack)
  • "Count the number of ways?" → Change dp from bool to int, += instead of |= (Target Sum #494)
  • "Minimum subset sum difference?" → Run same DP, find largest dp[j]=True for j ≤ target; answer = total - 2j (#2035, #1049)
  • "K equal subsets?" → Much harder: bitmask DP or backtracking (#698)

7 Summary & Key Takeaways ▾
Reframe: equal partition ↔ find subset summing to total / 2
Quick-fail on odd total — no integer halves possible
dp[j] = "can some subset of processed items sum to j?"
Right-to-left inner loop = 0/1 constraint (each item used once)
Left-to-right would give unbounded knapsack (items reusable) — different problem
Early exit when dp[target] becomes True saves remaining item processing
Space-optimized to O(target) by collapsing 2D table to rolling 1D array
ApproachTimeSpaceNotes
Brute Force (all subsets) O(2^n) O(n) TLE for n > 25
Memoization (top-down) O(n × T) O(n × T) Natural from recursion; T = sum/2
Tabulation (bottom-up) ✓ O(n × T) O(T) Best space; rolling 1D array
🎯 When to Recognize 0/1 Knapsack
  • "Can a subset reach exactly sum X?" → 0/1 Knapsack (boolean)
  • "Each element used at most once" + capacity/budget constraint → 0/1 Knapsack
  • State = (item index, remaining capacity) — O(n × capacity) states total
  • Right-to-left inner loop = 0/1. Left-to-right = unbounded (Coin Change)
  • Change dp from bool to int and += to count ways (Target Sum)
