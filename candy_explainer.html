LeetCode #135

Candy

Distribute the minimum total candies to n children in a line so every child has at least 1 candy and any child with a higher rating than a direct neighbour receives more candies than that neighbour.

Hard Greedy Two Passes Left/Right Time: O(n) Space: O(n)
1 Understanding the Problem ▾

You are distributing candies to n children standing in a line. Each child has a numeric rating. Rules: (1) every child gets at least 1 candy; (2) if child i has a strictly higher rating than an adjacent child, child i must receive strictly more candies than that adjacent child. Find the minimum total candies needed.

💡 Key Insight

The left-neighbour constraint and the right-neighbour constraint are independent. Solve each separately in a single scan, then take the element-wise maximum. The max is the tightest lower bound that simultaneously satisfies both.

✓ Example: ratings = [1, 0, 2]

child 0: rating 1 → 2 candies
child 1: rating 0 → 1 candy
child 2: rating 2 → 2 candies
Total = 5

✗ Naively giving 1 to all

ratings = [1, 0, 2]
candies = [1, 1, 1]
child 0 beats child 1 in rating
but both get 1 → violated!

Edge cases: n=1 (always 1), all equal ratings (all get 1), strictly increasing (1,2,...,n), strictly decreasing (n,...,2,1), V-shape valley, W-shape with multiple peaks.

2 The Algorithm: Two-Pass Left/Right Greedy ▾

We decompose the two-sided constraint into two independent single-sided problems, each solved greedily in O(n).

→
Left-to-Right Pass: Start with left[i] = 1 for all. Scan i=1..n-1: if ratings[i] > ratings[i-1], set left[i] = left[i-1] + 1. This greedily gives the minimum needed to beat the left neighbour.
←
Right-to-Left Pass: Start with right[i] = 1 for all. Scan i=n-2..0: if ratings[i] > ratings[i+1], set right[i] = right[i+1] + 1. Independently satisfies the right-neighbour constraint.
⊕
Merge: Final answer for child i is max(left[i], right[i]). The total is the sum of all maxes — minimum possible while satisfying both constraints everywhere.
1 Init: Create left = [1]*n.
2 L→R scan (i=1 to n-1): If ratings[i] > ratings[i-1], then left[i] = left[i-1] + 1.
3 Init: Create right = [1]*n.
4 R→L scan (i=n-2 to 0): If ratings[i] > ratings[i+1], then right[i] = right[i+1] + 1.
5 Return sum(max(left[i], right[i]) for i in range(n)).
🔑 Why Not One Pass?

A peak child (higher than both neighbours) must beat its left neighbour and its right neighbour. The right-constraint update at index i depends on right[i+1], which is not yet computed during a left-to-right scan. The passes are causally independent — finish one before starting the other.

3 Why It Works: The Invariant ▾
🔐 Invariant after Left Pass

After the L→R pass: left[i] equals 1 plus the length of the longest strictly-increasing run in ratings ending at index i from the left. It is the minimum number of candies child i must receive to satisfy every left-neighbour constraint along that run.

🔐 Invariant after Right Pass

After the R→L pass: right[i] equals 1 plus the length of the longest strictly-increasing run in ratings ending at index i from the right. It is the minimum candies to satisfy every right-neighbour constraint along that run.

Why max is globally minimum: At each index i, the two passes establish lower bounds left[i] and right[i]. Any valid assignment must be ≥ both. The minimum valid value is exactly max(left[i], right[i]). Summing these gives the global minimum total — we cannot do better at any single index without violating a constraint.

⚠️ Why Equal Ratings Don't Trigger More Candies

The problem says "strictly higher rating." If ratings[i] == ratings[i-1], child i has no obligation to beat child i-1. Resetting to 1 when equal is correct and necessary for minimality. Many beginners use >= instead of > and over-allocate.

Worked trace (ratings = [1, 0, 2, 2, 1]):
L→R: left = [1, 1, 2, 1, 1] (ratings[2]=2 > ratings[1]=0, so left[2]=left[1]+1=2; all others reset)
R→L: right = [2, 1, 1, 2, 1] (ratings[0]=1 > ratings[1]=0 → right[0]=2; ratings[3]=2 > ratings[4]=1 → right[3]=2)
max: [2, 1, 2, 2, 1] → total = 8

4 The Code (Annotated) ▾

Solution 1 — Two-Pass Greedy (Optimal · Interview Pick)

def candy(ratings: list[int]) -> int:
n = len(ratings) ← number of children
left = [1] * n ← left[i]: min candies satisfying left constraint
for i in range(1, n): ← left-to-right pass, start at i=1
if ratings[i] > ratings[i - 1]: ← strictly higher than left neighbour?
left[i] = left[i - 1] + 1 ← must beat left neighbour by exactly 1 (greedy minimum)
right = [1] * n ← right[i]: min candies satisfying right constraint
for i in range(n - 2, -1, -1): ← right-to-left pass, start at second-to-last
if ratings[i] > ratings[i + 1]: ← strictly higher than right neighbour?
right[i] = right[i + 1] + 1 ← must beat right neighbour by exactly 1
return sum(max(l, r) for l, r in zip(left, right))← max satisfies both; sum is the answer
⚠️ Why range(n-2, -1, -1) and not range(n-1, 0, -1)?

The last child (index n-1) has no right neighbour — it keeps its default value of 1. We start the right pass at index n-2. Starting at n-1 and accessing ratings[n-1+1] would be out of bounds.

Solution 2 — Brute Force with Repeated Scans (O(n²))

def candy_brute(ratings: list[int]) -> int:
n = len(ratings)
candies = [1] * n ← start everyone at 1
changed = True
while changed: ← repeat until stable (no violation)
changed = False
for i in range(n - 1): ← check left constraint
if ratings[i] > ratings[i+1] and candies[i] <= candies[i+1]:
candies[i] = candies[i+1] + 1; changed = True
return sum(candies) ← O(n) passes × O(n) scan = O(n²) worst case
💡 Complexity Summary

Two-Pass (Optimal): Time O(n) — two linear scans + sum. Space O(n) for two extra arrays.
Brute Force: Time O(n²) for strictly decreasing input. Space O(n).
O(1) Space (bonus): Single-pass slope counting — track ascending/descending run lengths, handle peak = max of both slopes. Clever but interview-complex; mention it, don't lead with it.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify the Constraint Structure

Draw [1,3,2] on the board. Work out by hand: child 1 (rating 3) must beat child 0 (left) AND child 2 (right). Realize: left-constraint flows left-to-right; right-constraint flows right-to-left. They're causal arrows pointing in opposite directions — handle separately.

Step 2: Propose Brute Force and Explain Its Flaw

"We could repeatedly scan until stable. Correct but O(n²). For [n, n-1, ..., 1], the first pass fixes only one index per iteration." Interviewers like seeing you identify the specific worst case.

Step 3: Decompose Into Two Independent Sub-Problems

"I can handle the left constraint with a single left-to-right scan (greedy: if ratings[i] > ratings[i-1], increment). Then independently handle the right constraint right-to-left. Taking the max at every position satisfies both in O(n) time."

Step 4: Trace the Example [1, 0, 2, 2, 1]

left=[1,1,2,1,1], right=[2,1,1,2,1], max=[2,1,2,2,1], sum=8. Tracing a non-trivial example that exercises the equal-ratings case (indices 2 and 3) shows you understand the strict inequality requirement.

🗣️ What to Say to the Interviewer

"The left-neighbour and right-neighbour constraints are independent. I'll handle each with one O(n) pass: Pass 1 (L→R): if ratings[i] > ratings[i-1], left[i] = left[i-1]+1, else left[i]=1. Pass 2 (R→L): if ratings[i] > ratings[i+1], right[i] = right[i+1]+1, else right[i]=1. Final answer = sum of max(left[i], right[i]). Total time O(n), space O(n). Could save space with single-pass slope counting if needed."
🔮 Likely Follow-up Questions
  • "Can you do O(1) space?" → Single-pass: track ascending/descending slope lengths, update peak retroactively
  • "What if equal-rated neighbours must get equal candies?" → Same algorithm; strict inequality already makes equal ratings give no bonus
  • "What if it's a circular arrangement?" → More complex: need a wraparound pass or restart detection
  • "Why does taking max work and not sum or average?" → Max is the tightest value that dominates both lower bounds; any smaller would violate one constraint

7 Summary & Key Takeaways ▾
Two independent directional constraints → one pass each, then take element-wise max
L→R pass: greedily give +1 over left neighbour when rating is strictly higher
R→L pass: greedily give +1 over right neighbour when rating is strictly higher
max(left[i], right[i]) is the minimum value satisfying both bounds simultaneously
Equal ratings → no extra candies required (strict inequality is the trigger)
Time O(n), Space O(n); O(1) space possible with single-pass slope trick
ApproachTimeSpaceNotes
Brute Force (repeated scans) O(n²) O(n) Too slow for large n
Two-Pass Greedy ✓ O(n) O(n) Optimal clarity; standard interview answer
Single-Pass Slope Counting O(n) O(1) Tricky to implement; mention as bonus
🎯 When to Recognize the Two-Pass Greedy Pattern
  • Problem has two directional local constraints (left and right neighbours)
  • Each constraint alone is greedily solvable in one direction
  • The combined solution is the element-wise max of the two single-constraint solutions
  • Keyword patterns: "higher than adjacent must get more", "distribute minimum subject to neighbour ordering"
  • Related structural pattern: left-product × right-product (Product of Array Except Self)
