LeetCode #1462

Course Schedule IV

Given n courses and prerequisite pairs, answer multiple queries: is course u a prerequisite (direct or indirect) of course v?

Medium Graph Topological Sort Reachability Time: O(n² + n·|E|) Space: O(n²)
1 Understanding the Problem ▾

You have n courses numbered 0 to n-1. A direct prerequisite pair [a, b] means you must take course a before course b. You receive multiple queries [u, v]. For each query: is course u a prerequisite (directly or indirectly) of course v?

💡 Key Insight

This is a reachability problem on a Directed Acyclic Graph (DAG). Course u is a prerequisite of v if and only if there exists a directed path from u to v. With many queries, it is far more efficient to precompute all reachability pairs into a matrix, then answer each query in O(1) with a single lookup.

✓ Example: n=4, prereqs=[[0,1],[1,2],[2,3]]

Graph: 0 → 1 → 2 → 3
Query [0,3]: path 0→1→2→3 → True
Query [1,3]: path 1→2→3 → True
Query [3,0]: no path → False

⚠️ Indirect prerequisites count!

0→1 (direct)
0→2 (indirect via 1)
0→3 (indirect via 1,2)
Not just direct edges!

Edge cases: A course is NOT a prerequisite of itself (no self-loops). If no prerequisites exist, all answers are False. The graph is always a DAG — no cycles. Queries can repeat or ask about entirely unconnected pairs.

2 The Algorithm: Topological Sort + Reachability Propagation ▾

We precompute a boolean reachability matrix reach[i][j] = True if course i can reach course j. We build this matrix by processing nodes in topological order (courses with no prerequisites first) so that when we process a node, all information about who can reach it is already finalized.

1 Build adjacency list & in-degrees: For each prerequisite [a, b], add b to graph[a] and increment indegree[b]. Also set reach[a][b] = True to seed direct edges.
2 Enqueue all in-degree-0 nodes: Nodes with no prerequisites are "roots" of the DAG — they can be processed first because nothing needs to come before them.
3 BFS / Kahn's Algorithm: Dequeue node u. For each neighbor v in graph[u], propagate reachability. Then decrement indegree[v]; if it becomes 0, enqueue v.
4 Key propagation step: for i in range(n): reach[i][v] |= reach[i][u]. Meaning: every node that can reach u can also reach v (since u→v exists). This builds the transitive closure in one pass.
5 Answer queries: Return [reach[u][v] for u, v in queries]. Each lookup is O(1) — constant time regardless of graph size.
🔑 Why Topological Order Is Critical

When we process edge u → v and execute reach[i][v] |= reach[i][u], we need reach[i][u] to be fully populated — meaning all paths leading into u have already been accounted for. Topological order guarantees that u is dequeued only after all its predecessors have been processed, so the propagation is complete and correct.

3 Why It Works: The Invariant ▾
🔐 The Invariant

When we dequeue and begin processing node u, reach[i][u] is finalized for all i. This is because every node that has a path to u must have been dequeued before u (since processing a node decrements in-degrees and u is only dequeued when its in-degree hits 0 — meaning all predecessors are done). Therefore, when we propagate reach[i][v] |= reach[i][u], we are working with complete and correct data.

Why the OR-assignment is correct: For any pair (i, v), node i can reach v if:

  • There is a direct edge i→v (seeded at initialization), OR
  • There is a path i→...→u, and the edge u→v exists (propagated by reach[i][v] |= reach[i][u])

Processing in topological order and combining all contributions with OR gives us the complete transitive closure of the graph.

⚠️ Common Mistake: Forgetting to Seed Direct Edges

If you initialize the entire reach matrix to False and only rely on the propagation loop, nothing will propagate — the loop can only flow existing True values forward. Seeding reach[a][b] = True for each direct edge [a, b] gives the propagation something to work with. Alternatively, you can set reach[u][v] = True during the propagation step for each edge u→v, but seeding upfront is cleaner.

Why the matrix is always correct after the BFS: Every node is processed exactly once (Kahn's guarantees this for DAGs). Every edge u→v is processed exactly once (when u is dequeued). At that point, all paths ending at u have been fully propagated. Therefore, after the BFS, reach[i][j] is True if and only if there is a directed path from i to j.

4 The Code (Annotated) ▾

Solution 1 — Topological Sort + Reachability Matrix (Optimal, Interview Pick)

from collections import deque
def checkIfPrerequisite(n, prereqs, queries):
graph = [[] for _ in range(n)] ← adjacency list: course → list of courses that depend on it
indegree = [0] * n ← how many prerequisite courses each course has
reach = [[False]*n for _ in range(n)] ← n×n matrix; reach[i][j]=True means i is a prereq of j
for a, b in prereqs: ← process each direct prerequisite pair [a, b]
graph[a].append(b) ← directed edge a→b in adjacency list
indegree[b] += 1 ← b has one more prerequisite
reach[a][b] = True ← seed: a directly reaches b (every path of length 1)
queue = deque(i for i in range(n) ← start BFS with all courses that have zero prerequisites
if indegree[i] == 0)
while queue: ← standard BFS / Kahn's topological sort loop
u = queue.popleft() ← dequeue next node in topological order
for v in graph[u]: ← visit each course that u is a direct prerequisite of
for i in range(n): ← check every node i in the graph
reach[i][v] |= reach[i][u] ← if i reaches u, then i also reaches v (via u→v)
indegree[v] -= 1 ← one more prerequisite of v has been fully processed
if indegree[v] == 0: ← all prerequisites of v are done → ready to process v
queue.append(v) ← enqueue v to process its outgoing edges next
return [reach[u][v] for u, v in queries] ← O(1) per query: just index the precomputed matrix
⚠️ Why reach[i][v] |= reach[i][u] and NOT reach[u][v] = True?

Setting reach[u][v] = True only records that u can directly reach v. But we need all ancestors of u to know they can also reach v. The OR-assignment over all i achieves this: if ancestor i already has reach[i][u] = True, it gets reach[i][v] = True automatically — without needing to re-traverse the graph.

Solution 2 — DFS/BFS per Query (Brute Force)

def checkIfPrerequisite_brute(n, prereqs, queries):
graph = [[] for _ in range(n)]
for a, b in prereqs: graph[a].append(b)
def bfs(src, dst): ← fresh BFS from src for each query
visited, queue = {src}, deque([src])
while queue:
node = queue.popleft()
if node == dst: return True
for nb in graph[node]:
if nb not in visited:
visited.add(nb); queue.append(nb)
return False
return [bfs(u, v) for u, v in queries] ← O(n+E) per query — slow for many queries
💡 Complexity Comparison

Topo Sort + Reach Matrix: O(n² + n·|E|) precompute + O(q) queries. Space O(n²).
Brute Force BFS/DFS: O(q × (n + |E|)) total — a new traversal per query. Space O(n) per query.

When q > n (which is common — e.g., n=100, q=10000), the matrix approach is decisively faster.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Graph + Reachability Pattern

Prerequisite relationships are inherently directed: "A must come before B" is a directed edge A→B. Multiple "can A reach B?" queries on a DAG is the textbook transitive closure problem. Recognize that precomputing all pairs is better than re-searching for each query.

Step 2: Propose Brute Force First

"For each query, I can run a BFS/DFS from u and see if I reach v. This is O(n+E) per query. With q queries total: O(q·(n+E)). If q and n are both up to 100, that's fine. But we can do much better by precomputing."

Step 3: Introduce Precomputed Reachability Matrix

"I'll build an n×n matrix: reach[i][j] = True means i can reach j. Computed once, queries become O(1). I fill it using topological sort (Kahn's BFS) so that when I propagate through edge u→v, all paths into u are already known."

Step 4: Explain the Propagation Formula

"The key step is: for each edge u→v, for every node i, reach[i][v] |= reach[i][u]. This means: all ancestors of u also become ancestors of v. Combined with topological order, this computes the full transitive closure efficiently."

🗣️ What to Say to the Interviewer

"This is a reachability problem on a DAG — multiple queries, so precompute.
I'll build an n×n reach matrix, seeded with direct edges.
Then I process nodes in topological order using Kahn's BFS.
For each edge u→v I run: reach[i][v] |= reach[i][u] for all i.
This propagates reachability to all ancestors of u simultaneously.
After precompute: O(1) per query. Total: O(n²+n·E) time, O(n²) space."
🔮 Likely Follow-up Questions
  • "Can you detect cycles with your approach?" → If any node still has non-zero in-degree after BFS, there's a cycle (#207 Course Schedule)
  • "What if n is huge (e.g., 10⁴)?" → n² matrix becomes 10⁸ booleans — too large. Use DFS with memoization or bitset compression
  • "Could you use Floyd-Warshall instead?" → Yes, O(n³), same O(n²) space. Topo sort approach is better for sparse DAGs
  • "What if the graph had cycles (not a DAG)?" → Kahn's wouldn't process all nodes. Need DFS + SCC, or Floyd-Warshall

from collections import deque
def checkIfPrerequisite(n, prereqs, queries):
graph = [[] for _ in range(n)]
indegree = [0] * n
reach = [[False]*n for _ in range(n)]
for a, b in prereqs:
graph[a].append(b); indegree[b] += 1; reach[a][b] = True
queue = deque(i for i in range(n) if indegree[i] == 0)
while queue:
u = queue.popleft()
for v in graph[u]:
for i in range(n):
reach[i][v] |= reach[i][u]
indegree[v] -= 1
if indegree[v] == 0: queue.append(v)
return [reach[u][v] for u, v in queries]

7 Summary & Key Takeaways ▾
Multiple reachability queries on a DAG → precompute an n×n reach matrix once, answer each in O(1)
Kahn's BFS gives topological order so each node is processed after all its predecessors
Propagation formula: reach[i][v] |= reach[i][u] for all i propagates all ancestors of u to v in one step
Seed direct edges before BFS so propagation has a starting point
The graph must be a DAG for this approach — Kahn's only terminates cleanly on acyclic graphs
