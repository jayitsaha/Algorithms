LeetCode #84

Largest Rectangle in Histogram

Given an array of integers heights representing bar heights in a histogram (each bar has width 1), find the area of the largest rectangle you can form.

Hard Monotonic Stack Stack for Left/Right Bounds Time: O(n) Space: O(n)
1 Understanding the Problem ▾

Imagine a bar chart where every bar has width 1. You can draw a rectangle that spans several consecutive bars — but its height is limited by the shortest bar in that span. Your job: find the largest possible such rectangle.

💡 Key Insight

Every maximum rectangle is limited by exactly one bar — the shortest bar in its horizontal span. So reframe the problem: for each bar i, "if I use heights[i] as the rectangle's height, how far left and right can I extend before hitting something shorter?" The answer for bar i is heights[i] × (right_bound − left_bound − 1). The global answer is the maximum across all bars.

For heights = [2, 1, 5, 6, 2, 3]:

❌ Naive: try all spans

For each (l, r) pair:
area = min(heights[l..r]) × (r−l+1)
O(n²) pairs → TLE at n=100,000

✓ Monotonic stack: O(n)

Each bar pushed once, popped once
On pop: instantly know left & right bounds
Amortized O(1) work per bar

Best rectangle for each bar in [2, 1, 5, 6, 2, 3]:

heights[0]=2: extends [0,0] → 2×1 = 2
heights[1]=1: extends [0,5] → 1×6 = 6 (shortest bar — spans everything!)
heights[2]=5: extends [2,3] → 5×2 = 10 ← ANSWER
heights[3]=6: extends [3,3] → 6×1 = 6
heights[4]=2: extends [2,5] → 2×4 = 8
heights[5]=3: extends [5,5] → 3×1 = 3

Edge Cases: single bar (area = that height), all same height (area = height × n), strictly increasing (largest bar × 1 wins for height, but shortest × n may win for width), all zeros (0), n = 1 (trivial).

2 The Algorithm: Monotonic Increasing Stack ▾

We maintain a stack of bar indices in strictly increasing height order (bottom-to-top). When we encounter a bar shorter than the stack top, the top bar has found both its left and right boundaries — we pop it and compute its rectangle area immediately.

1 Initialize: Empty stack, max_area = 0. Process indices 0 to n inclusive (n+1 iterations), using a virtual sentinel bar of height 0 at position n.
2 For each index i: Get h = heights[i] (or 0 for the sentinel at i=n). While the stack is non-empty AND the current bar is shorter than the stack's top bar, pop the stack top — that top bar has found its right boundary.
3 On pop (index h_idx): The popped bar's height is heights[h_idx]. Its right boundary is the current index i. Its left boundary is stack[-1] after the pop (or −1 if stack is empty). Width = i − left − 1. Area = height × width.
4 Push current index: After all pops, push i. The stack remains in increasing height order.
5 Sentinel handles the flush: The height-0 sentinel at index n is shorter than all real bars, so it triggers pops for all remaining stack entries. No separate cleanup loop needed.
🔑 Why an Increasing Stack?

When we pop bar h because bar i is shorter, the new stack top is the nearest bar to the left of h that is also shorter than h. This is exactly the left boundary of h's rectangle. The increasing order invariant guarantees that the stack top after popping is always the correct left boundary — no second pass needed.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every moment, the stack contains indices whose corresponding heights are in strictly increasing order from bottom to top. When bar i causes bar h_idx to be popped:

  • Right boundary of h_idx's rectangle = i — because heights[i] < heights[h_idx], and this is the first such bar to the right (any earlier shorter bar would have triggered the pop sooner).
  • Left boundary of h_idx's rectangle = stack[-1] after pop — because the new stack top is the nearest bar to the left of h_idx that is shorter than heights[h_idx] (the increasing invariant guarantees this).
  • Rectangle width = i − left − 1 — from left+1 to i−1 inclusive, all bars are ≥ heights[h_idx].

Proof of completeness: Every possible optimal rectangle is the "widest rectangle at some height h." That height h corresponds to some bar b — the leftmost minimum in the optimal span. Bar b will eventually be popped when the next shorter bar appears to its right. At that exact moment, both boundaries are available from the stack. Therefore every candidate rectangle is evaluated exactly once.

⚠️ Off-by-One in the Width Formula

Width = i − left − 1, NOT i − left. The bars at positions left and i are the first bars shorter than our rectangle — they are excluded. The rectangle spans bars from left+1 to i−1, a total of (i−1) − (left+1) + 1 = i − left − 1 bars. When stack is empty, use left = −1, giving width = i − (−1) − 1 = i, meaning all bars from 0 to i−1 are included.

Amortized O(n) analysis: Each index is pushed exactly once and popped at most once. Total push+pop operations across the entire algorithm ≤ 2n. So the total work is O(n) despite the nested while loop.

4 The Code (Annotated) ▾

Solution 1 — Monotonic Stack with Sentinel (Optimal, Interview Pick)

def largestRectangleArea(heights: list[int]) -> int:
stack = [] ← Indices of bars; heights are strictly increasing bottom→top
max_area = 0 ← Running best area across all evaluated rectangles
n = len(heights) ← We loop n+1 times (0..n) to include the sentinel
for i in range(n + 1): ← +1 to process the virtual sentinel bar at index n
h = 0 if i == n else heights[i] ← Sentinel height 0 forces flush of all remaining stack entries
while stack and h < heights[stack[-1]]: ← Current bar is shorter → pop; stack top has found its right bound
height = heights[stack.pop()] ← Height of the rectangle limited by the popped bar
left = stack[-1] if stack else -1 ← New top = nearest shorter bar to the left (or -1 if none)
width = i - left - 1 ← Bars from left+1 to i-1 all have height ≥ popped bar's height
max_area = max(max_area, height * width) ← Update running max
stack.append(i) ← Push current index; stack stays in increasing-height order
return max_area ← Largest rectangle area across all bars
💡 The Sentinel Bar Trick

Iterating to n + 1 with h = 0 at position n is elegant: height 0 is guaranteed to be shorter than every real bar (even if heights are all 1). This forces the while loop to drain the entire stack during the last iteration. Without it, you'd need a separate cleanup loop — the sentinel makes the code cleaner and avoids code duplication.

⚠️ Store Indices, Not Heights

The stack must hold indices. When computing width, you need the positions of boundaries. If you store heights instead, stack[-1] gives you a height, not a position — the width calculation breaks. Always store indices, then use heights[idx] to get the height when needed.

Solution 2 — Brute Force O(n²) — always mention this first in interviews

def largestRectangleArea(heights: list[int]) -> int:
max_area = 0
n = len(heights)
for left in range(n): ← Try every possible left boundary
min_h = heights[left] ← Track the bottleneck height in [left, right]
for right in range(left, n): ← Expand right boundary
min_h = min(min_h, heights[right]) ← Rectangle height = minimum in current span
max_area = max(max_area, min_h * (right - left + 1))
return max_area ← O(n²) — correct but TLE for large inputs
📊 Complexity Comparison

Monotonic Stack: O(n) time — each element pushed/popped at most once · O(n) space for the stack
Brute Force: O(n²) time — all pairs with running minimum · O(1) space
Divide & Conquer: O(n log n) average — range-min query + recursion (not worth in interview)

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Reframe the Problem

"Instead of thinking about all possible rectangles (O(n²) pairs), think per-bar: for each bar, what's the maximum rectangle that uses this bar as its height limit? That reduces to: find the nearest shorter bar to the left (left boundary) and right (right boundary). The rectangle spans exactly between those two boundaries."

Step 2: Recognize the Monotonic Stack Signal

"Finding 'next smaller element' and 'previous smaller element' for all positions simultaneously in O(n) is a classic monotonic stack problem. An increasing stack gives us the previous smaller for free (it's the new top after a pop), and the element that triggers the pop is the next smaller."

Step 3: Explain the Stack Invariant Clearly

"The stack holds indices in increasing height order. When bar i is shorter than the top, we pop — that gives us the right bound (= i) and the left bound (= new top after pop, or -1 if empty). Width = i - left - 1. Then we push i."

Step 4: Handle the Flush Elegantly

"After processing all real bars, any remaining stack entries have no shorter bar to their right. I can handle this with a separate loop, or more elegantly, by appending a virtual height-0 bar — it's shorter than everything, forcing all remaining bars to be popped and computed."

🗣️ What to Say to the Interviewer

"I'll start with O(n²) brute force: for each pair (l, r), area = min(heights[l..r]) * (r-l+1).
To optimize, I reframe: for each bar, find its max rectangle by locating the nearest shorter bar left and right.
A monotonic increasing stack does this in one pass: when I see a shorter bar at i, the stack top's right
boundary is i, and the new stack top after popping is the left boundary. Width = i - left - 1.
I add a sentinel height-0 bar at the end to flush the remaining stack.
Time: O(n) — each bar pushed/popped once. Space: O(n) for the stack."

7 Summary & Key Takeaways ▾
Every optimal rectangle is "pinned" at height of some bar — the shortest in its span
Monotonic increasing stack finds left AND right bounds simultaneously in one pass
On pop: right boundary = current i; left boundary = new stack top (or −1)
Store indices in the stack, not heights — you need positions for width calculation
Sentinel bar of height 0 at position n flushes the stack without extra code
Total work is O(n) — each bar pushed once, popped at most once (amortized O(1) per bar)
