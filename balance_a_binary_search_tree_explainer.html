LeetCode #1382

Balance a Binary Search Tree

Given the root of a BST, return a balanced BST with the same node values. A balanced BST is one where the depth of the two subtrees of every node never differs by more than 1.

Medium Trees In-order + Rebuild Time: O(n) Space: O(n)
1 Understanding the Problem ▾

A BST can become unbalanced through repeated insertions — it may degenerate into something like a linked list, giving O(n) search instead of the expected O(log n). Your task is to take any BST and return a new BST that contains the same values but is height-balanced: at every node, the left and right subtree heights differ by at most 1.

💡 The Central Insight

In-order traversal of any BST always visits nodes in strictly ascending sorted order (Left → Root → Right). So if we collect all values via in-order traversal, we get a sorted array. From a sorted array, the optimal balanced BST root is always the median (middle element) — it splits the array into two equal halves which become the left and right subtrees. Apply this recursively.

✗ Unbalanced (skewed right)
1 \ 2 \ 3 \ 4 \ 5
→
✓ Balanced BST (rebuilt)
3 / \ 2 4 / \ 1 5

Concrete walkthrough (n=5, sorted array = [1, 2, 3, 4, 5]):

  • Median of [1,2,3,4,5] is index 2 → value 3. Root = 3.
  • Left half [1,2]: median at index 0 → value 1. Right of 1 = 2 (only element). Left subtree = node(1, right=node(2)).
  • Right half [4,5]: median at index 0 → value 4. Right of 4 = 5. Right subtree = node(4, right=node(5)).
  • Result: root 3, left subtree height 2, right subtree height 2. Perfectly balanced!

Edge Cases: A single-node BST (already balanced, in-order gives [val], return a leaf). An already-balanced BST (algorithm still works, produces an equivalent balanced tree). Even-length arrays — either of the two middle elements works as the root, both produce valid balanced trees.

2 The Algorithm: In-Order Flatten → Divide-and-Conquer Rebuild ▾

The algorithm cleanly separates into two independent phases. Phase 1 extracts a sorted array from the original BST. Phase 2 builds the optimal balanced BST from that sorted array. Both phases use simple recursion with one base case.

1 Phase 1 — In-Order DFS (Flatten): Define a recursive helper inorder(node). Base case: if node is None, return. Recursive case: call inorder(node.left), then vals.append(node.val), then inorder(node.right). After the full traversal, vals holds all values in sorted ascending order.
2 Phase 2 — Rebuild (Divide and Conquer): Define build(lo, hi) that works on the subarray vals[lo..hi]. Base case: if lo > hi, return None (empty range = no node).
3 Pick the Median as Root: Compute mid = (lo + hi) // 2. Create TreeNode(vals[mid]). This median element is the perfect balanced root: it will have exactly ⌊(n-1)/2⌋ elements to its left and ⌈(n-1)/2⌉ to its right (where n = hi - lo + 1).
4 Recurse on Both Halves: node.left = build(lo, mid - 1) for the left subarray (smaller values). node.right = build(mid + 1, hi) for the right subarray (larger values). Return node.
5 Kick Off Both Phases: Call inorder(root) to fill vals, then return build(0, len(vals) - 1) as the new balanced root.
BST Property Auto-Preserved

Since vals is sorted, vals[lo..mid-1] are all < vals[mid] and vals[mid+1..hi] are all > vals[mid]. The BST property (left < root < right) is automatically satisfied at every node, with zero extra checks.

Height Optimality Guaranteed

With n nodes, height = ⌈log₂(n+1)⌉ — the mathematical minimum possible. Picking the median ensures both subtrees have at most ⌈n/2⌉ nodes each, so the height can never be suboptimal.

🔑 Why This Is Brilliant: Reducing to a Solved Problem

This problem reduces to two sub-problems you already know: (1) in-order traversal of a BST (fundamental traversal), and (2) convert sorted array to BST (#108, an Easy problem). Recognizing how to decompose a harder problem into known sub-problems is a core interview skill.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every call build(lo, hi), the subarray vals[lo..hi] is sorted and contains exactly the values that belong in this subtree. Picking mid = (lo+hi)//2 guarantees that the resulting subtree has height at most ⌈log₂(hi - lo + 2)⌉ — the minimum possible — and that the BST ordering property holds by the sorted-array guarantee.

Proof that BST property holds: Since vals is sorted ascending, for any call build(lo, hi) with mid = (lo+hi)//2: Every element at indices lo..mid-1 is strictly less than vals[mid] (sorted array). Every element at indices mid+1..hi is strictly greater. These become the left and right subtrees respectively — so the BST property (all left descendants < root < all right descendants) is automatically satisfied without any additional checks.

Proof that balance holds (by induction): For a subarray of length 1 (lo == hi), a single leaf node is trivially balanced. For length n, the left half has ⌊(n-1)/2⌋ elements and the right half has ⌈(n-1)/2⌉ elements — they differ by at most 1. If both halves are balanced by induction, the resulting node is balanced. QED.

⚠️ Common Mistake: In-Place Rotation Approach

A natural first instinct is to try AVL rotations (LL, RR, LR, RL) or red-black tree rebalancing. These are correct but extremely complex to implement under interview pressure. The flatten-rebuild approach is not only simpler — it is also optimal (O(n) time) and produces a provably perfect balanced tree, not just a "good enough" one. Never use rotations unless specifically asked.

Verification for n=5: vals = [1,2,3,4,5]. build(0,4): mid=2, root=3. Left: build(0,1) → mid=0, root=1. Left-left: build(0,-1) → None. Left-right: build(1,1) → mid=1, root=2, both children None. Right: build(3,4) → mid=3, root=4. Right-left: build(3,2) → None. Right-right: build(4,4) → mid=4, root=5. Heights: left subtree of 3 = 2, right = 2. Balanced!

4 The Code (Annotated) ▾

Solution 1 — In-Order + Sorted Array Rebuild (Optimal, Interview Pick)

class Solution:
def balanceBST(self, root: TreeNode) -> TreeNode:
vals = [] ← Sorted values from in-order traversal
def inorder(node): ← DFS helper: Left → Root → Right
if not node: return ← Base case: None means empty subtree
inorder(node.left) ← Visit all smaller values first
vals.append(node.val) ← Collect this value in sorted position
inorder(node.right) ← Then all larger values
inorder(root) ← Run traversal; after this vals is sorted
def build(lo, hi): ← Build balanced BST from vals[lo..hi]
if lo > hi: return None ← Empty range → no node (leaf's children)
mid = (lo + hi) // 2 ← Median = perfect balanced split point
node = TreeNode(vals[mid]) ← Median becomes root of this subtree
node.left = build(lo, mid - 1) ← All smaller values → left subtree
node.right = build(mid + 1, hi) ← All larger values → right subtree
return node ← Propagate subtree root upward
return build(0, len(vals) - 1) ← Kick off rebuild on full sorted array
💡 Complexity Summary

In-order traversal: O(n) time (visits every node once), O(n) space (vals array) + O(h) stack (original tree height)
Build: O(n) time (each index visited exactly once), O(log n) stack depth (balanced tree height)
Overall: O(n) time, O(n) space. This is optimal — we must read all n values at minimum, so O(n) is a lower bound.

⚠️ Why Brute Force Is Worse

The brute-force approach collects values in an arbitrary order and then sorts them explicitly — O(n log n). For a BST, we get sorted order for free via in-order traversal — no explicit sort needed, giving O(n). In an interview, always exploit BST properties rather than treating it as a generic binary tree. This distinction shows deep understanding.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify the BST Property

"The input is a BST. In-order traversal of a BST gives sorted order. Can I exploit this?" — This single observation transforms a hard problem into two easy ones. Practice saying this out loud in interviews to signal your BST fluency.

Step 2: Reduce to Known Sub-Problems

"Sorted array → balanced BST" is LeetCode #108. "In-order traversal" is a fundamental pattern. Explicitly reduce: BST → sorted array (in-order) → balanced BST (median-based rebuild). Stating this decomposition impresses interviewers because it shows you connect problems.

Step 3: Explain Optimality Before Coding

Briefly explain why this is O(n): "In-order visits each node once. Build recurses once per index in vals. Total O(n) time, O(n) space for the sorted array." Then also mention: "If O(1) space is needed, the Day-Stout-Warren algorithm does it in-place but is much more complex."

Step 4: Verify Correctness Verbally

"The BST property is preserved because vals is sorted, so left subarray < mid < right subarray. Balance is guaranteed because median split gives equal-size halves ±1 at every level." Saying this shows you understand why the solution works, not just that it does.

🗣️ What to Say to the Interviewer

"Since this is a BST, in-order traversal gives me sorted values — no explicit sort needed.
So my approach is two phases:
Phase 1: In-order DFS to collect all n values into a sorted array. O(n).
Phase 2: Build a balanced BST from the sorted array using divide-and-conquer.
Pick the middle element as root, recurse on left and right halves.
The BST property is preserved because sorted subarray guarantees left < mid < right.
Balance is guaranteed because median gives equal-size halves.
Total: O(n) time, O(n) space for the array (plus O(log n) recursion stack in build phase).
If O(1) space is needed, there's DSW (Day-Stout-Warren), but that's significantly more complex."
🔮 Likely Follow-up Questions
  • "Can you do it in O(1) space?" → DSW algorithm: convert to a "vine" (right-skewed), then compress by left-rotating. O(n)/O(1) but complex.
  • "What if it's a general binary tree, not a BST?" → Collect values (any traversal), sort them explicitly (O(n log n)), then rebuild the same way.
  • "How do you know your tree is valid after rebuilding?" → Prove BST from sorted subarray invariant, prove balance from equal-size halves at each level.
  • "What if there are duplicate values?" → Handled naturally — duplicates appear adjacent in sorted array, each becomes its own node, and can go to the left child by convention.
  • "LeetCode 108 — Convert Sorted Array to Binary Search Tree" → Exact same build() function! This problem is a superset of #108.

7 Summary & Key Takeaways ▾
In-order traversal of any BST produces values in sorted ascending order — use this to flatten the tree for free (no explicit sort).
The middle element of a sorted array is the ideal balanced BST root: it splits into two equal halves (±1), minimizing tree height.
BST property is automatically preserved when rebuilding from a sorted array: left subarray < mid < right subarray at every level.
This decomposes into two known sub-problems: in-order traversal + "sorted array to balanced BST" (#108). Recognizing reductions is a key interview skill.
Time O(n), Space O(n). Cannot be better than O(n) since we must read all n values. This is the optimal approach for interviews.
