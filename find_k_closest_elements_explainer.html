LeetCode #658

Find K Closest Elements

Given a sorted integer array arr and integers k and x, return the k closest integers to x in the array, sorted in ascending order.

Medium Binary Search BS on Left Bound Time: O(log n + k) Space: O(1)
1 Understanding the Problem ▾

You have a sorted integer array. You want to find the k numbers from that array that are closest in value to a target x. The result must be returned in ascending order.

Closeness is defined by absolute difference: |arr[i] - x|. When two elements are equally distant from x, we prefer the smaller one (the tie-breaking rule specified in the problem).

💡 Key Insight

Because the array is sorted, the k closest elements must form a contiguous subarray. You never scatter-pick — the answer is always a window of exactly k elements. Your entire job is to find where that window starts.

✓ arr=[1,2,3,4,5], k=4, x=3

Window [1,2,3,4]: dists = 2,1,0,1
Window [2,3,4,5]: dists = 1,0,1,2
Tied! Prefer smaller → [1,2,3,4]

✗ Why can't we scatter-pick?

Skip 2, pick {1,3,4,5}?
But |2-3|=1 < |5-3|=2,
so 2 must beat 5. Always contiguous.

Edge Cases: x smaller than all elements → return leftmost k; x larger than all → return rightmost k; k equals array length → return everything; duplicates in array (still works correctly); x exactly equals an element.

2 The Algorithm: Binary Search on Window Start ▾

Since the answer is always a contiguous window of size k, we binary search over all valid starting indices for that window. The window start lives in the range [0, n-k].

1 Set search bounds: lo = 0, hi = n - k. A window starting at hi fills the last k positions. Beyond hi, fewer than k elements remain — invalid.
2 Pick midpoint: mid = (lo + hi) // 2. This candidate window spans arr[mid .. mid+k-1].
3 Compare outer endpoints: Check whether arr[mid] (leftmost in window) or arr[mid+k] (first element just outside, which would enter if we shift right) is closer to x.
4 Decide which side to eliminate: If arr[mid] is strictly farther → shift right (lo = mid+1). Otherwise arr[mid] is as good or better → keep or shift left (hi = mid).
5 Extract answer: When lo == hi, return arr[lo : lo + k].
The Core Binary Search Decision:
→
x - arr[mid] > arr[mid + k] - x
arr[mid] is farther from x than arr[mid+k]. Replacing arr[mid] with arr[mid+k] makes the window strictly better. Shift right: lo = mid + 1.
←
x - arr[mid] ≤ arr[mid + k] - x
arr[mid] is at least as close as arr[mid+k]. On a tie, prefer smaller (arr[mid]). Window could start at mid or earlier. Set hi = mid.
🔑 Why Compare arr[mid] vs arr[mid+k], not arr[mid+k-1]?

Window starting at mid covers arr[mid..mid+k-1]. The element at arr[mid+k] is the one that would enter the window if we shift one step right (dropping arr[mid]). Comparing these two tells us definitively whether shifting right improves things.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every iteration, the optimal window start index is always within [lo, hi]. Each step eliminates indices that are provably not optimal, so the correct answer is never discarded.

Why eliminating mid when arr[mid] is strictly farther is safe: If x - arr[mid] > arr[mid+k] - x, then any window starting at index ≤ mid includes arr[mid] while excluding arr[mid+k]. Swapping arr[mid] out and bringing arr[mid+k] in (shifting right by 1) strictly improves the window. Therefore, starting at mid or anything to its left cannot be optimal. We safely set lo = mid + 1.

Why we use strict > not ≥: Ties favour the smaller element (arr[mid] is smaller than arr[mid+k] since the array is sorted). So when distances tie, arr[mid] wins — we must keep mid in contention and set hi = mid.

⚠️ The Most Common Binary Search Mistake Here

Setting hi = mid - 1 instead of hi = mid. This is a "find leftmost" binary search — when the condition is false (arr[mid] is not farther), mid could itself be the answer. Setting hi = mid - 1 would eliminate it. Always use hi = mid in "leftmost position" binary search patterns.

Termination: Each step either increases lo or decreases hi. The interval hi - lo strictly shrinks, so the loop terminates in O(log(n - k)) ≈ O(log n) iterations.

4 The Code (Annotated) ▾

Solution 1 — Binary Search on Left Bound (Optimal, Interview Pick)

def findClosestElements(arr, k, x):
lo, hi = 0, len(arr) - k ← window start can be in [0, n-k]
while lo < hi: ← standard "find leftmost" binary search loop
mid = (lo + hi) // 2 ← candidate window start (safe from overflow in Python)
if x - arr[mid] > arr[mid + k] - x: ← arr[mid] strictly farther from x than arr[mid+k]?
lo = mid + 1 ← shift window right; arr[mid] can't be in optimal window
else: ← arr[mid] closer or tied (ties break left = smaller)
hi = mid ← keep mid in play; optimal start ≤ mid
return arr[lo : lo + k] ← the k closest elements as a sorted subarray
⚠️ Why hi = mid, not hi = mid - 1?

We're searching for the leftmost valid start. When x - arr[mid] ≤ arr[mid+k] - x, mid could be the optimal start. Setting hi = mid - 1 discards it. Setting hi = mid keeps it in the search space. The loop terminates correctly because lo can only increase and hi can only decrease.

Solution 2 — Two-Pointer Shrink (O(n−k) time, simpler)

def findClosestElements(arr, k, x):
left, right = 0, len(arr) - 1 ← start with all elements in consideration
while right - left >= k: ← keep removing the farther endpoint
if x - arr[left] > arr[right] - x: ← left is farther? remove it
left += 1
else: ← right is farther or tied? remove it
right -= 1
return arr[left : right + 1] ← exactly k elements remain

Solution 3 — Sort by Distance (Brute Force, O(n log n))

def findClosestElements(arr, k, x):
# Sort by (distance, value) then re-sort top-k by value ← ignores sorted structure
return sorted(sorted(arr, key=lambda a: (abs(a-x), a))[:k]) ← O(n log n) + O(k log k)
💡 Complexity Summary

Binary Search (Optimal): Time O(log(n−k) + k), Space O(1)
Two-Pointer: Time O(n − k), Space O(1) — removes n−k elements one at a time
Sort by Distance: Time O(n log n), Space O(n) — overkill; ignores sorted order

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Exploit "Sorted Array"

The word "sorted" is the trigger. Sorted = structure to exploit. First observation: "If the array is sorted, the k closest elements must be contiguous." This single insight transforms the problem from a selection problem into a search problem.

Step 2: Brute Force First

"Sort all elements by distance to x, take the first k, sort them by value. O(n log n). This is correct but ignores the sorted structure entirely. Can we do better?" Always show the brute force to anchor the conversation.

Step 3: Two-Pointer (Good Interview Answer)

"Since the answer is a contiguous window, I can shrink from both ends: start with left=0, right=n-1 and remove the farther endpoint until exactly k remain. O(n-k). Easy to implement and explain." This alone scores well.

Step 4: Binary Search (Optimal, Great Answer)

"The window start lives in [0, n-k] — I can binary search for it. Comparison: if arr[mid] is strictly farther from x than arr[mid+k], shift right. Otherwise keep or go left. Terminates in O(log n), total O(log n + k)."

🗣️ What to Say to the Interviewer

"The array is sorted, so the k closest elements are always contiguous — I just need to find the window start.
The start is in range [0, n-k]. I'll binary search: at each mid, I compare arr[mid] vs arr[mid+k].
If x - arr[mid] > arr[mid+k] - x: arr[mid] is farther, shift right (lo = mid + 1).
Otherwise arr[mid] is at least as close — on ties, keep smaller — so hi = mid.
When lo == hi, return arr[lo : lo+k]. Time: O(log(n-k) + k). Space: O(1) excluding output."
🔮 Likely Follow-up Questions
  • "What if the array is unsorted?" → Sort first O(n log n), then binary search. Or max-heap of size k, O(n log k).
  • "What if ties should break right (prefer larger)?" → Change > to ≥ in the comparison.
  • "What if you need k closest across multiple sorted arrays?" → Merge with a min-heap, O(n log m).
  • "Streaming version — k closest in a sorted stream?" → Max-heap of size k, O(n log k).

7 Summary & Key Takeaways ▾
The k closest elements in a sorted array are always a contiguous subarray
Binary search on the window start index, not on a value — range is [0, n-k]
Compare arr[mid] vs arr[mid+k]: the competing endpoints for the window boundary
Strict > because ties break left (prefer smaller element)
hi = mid (not mid-1) — canonical "find leftmost position" binary search

🎯 When to Recognize Binary Search on Left Bound
  • "Find the k closest / best subarray of fixed size k in a sorted array"
  • You're searching for a position (window start), not a value
  • Search space is [0, n-k], not [lo_val, hi_val]
  • Answer is a contiguous block → the problem reduces to "where does it start?"
