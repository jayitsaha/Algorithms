LeetCode #546

Remove Boxes

Remove groups of same-colored boxes to maximize the total score, where removing k boxes of the same color scores k² points.

Hard Dynamic Programming 3D DP with Count Interval DP Time: O(n⁴) Space: O(n³)
1 Understanding the Problem ▾

You have a row of colored boxes (each box has an integer color). You can repeatedly choose a contiguous group of boxes of the same color and remove them. Removing k boxes earns k² points. Continue until all boxes are removed. Maximize the total score.

💡 Key Insight: k² Rewards Patience

Removing 1 box = 1 pt. Removing 2 = 4 pts. Removing 3 = 9 pts. Removing 5 = 25 pts. The quadratic scoring heavily favors merging groups. Instead of greedily removing the largest available group, it's often better to remove intervening boxes first so that two same-colored groups merge into one giant group.

❌ Greedy Remove: Score = 10

[1,1,1,2,1,1] → remove 2 → +1
[1,1,1,1,1] → remove 5 → +25
Total: 26 pts

[1,1,3,3,1,1,1,1] → remove 3,3 → +4
[1,1,1,1,1,1] → remove 6 → +36
Total: 40

✓ Optimal: Merge Groups First

Example: boxes=[1,3,2,2,2,3,4,3,1]
Remove [2,2,2] → +9
Remove [3,3,3] → +9 (merged!)
Remove [1,1] → +4
Remove [4] → +1
Total: 23 pts

Why this is hard: Future removals affect current scores. Removing a box in the middle can merge two groups of the same color, which might dramatically increase future points. This dependency on future decisions makes greedy fail and requires DP to enumerate all strategies.

Edge Cases: All boxes same color (remove all at once for n² pts), alternating colors (each removal scores 1 pt), single box (1 pt), n=1 (trivially 1).

2 The Algorithm: 3D Interval DP with Count ▾

Standard Interval DP tracks subproblems on ranges [l, r]. For Remove Boxes, this is insufficient because when we remove boxes[r], we need to know how many boxes of the same color are "attached" to the right of position r but outside our current range. We add a third dimension k to track that count.

🔑 The 3D State Definition

dp[l][r][k] = maximum points from removing all boxes in boxes[l..r], given that there are k additional boxes of the same color as boxes[r] attached to the right side of position r.

1 Base case: dp[i][i][k] = (k+1)² — a single box at position i with k extra same-color boxes to the right can be removed as a group of k+1.
2 Choice A — Remove the tail group now: Remove all boxes from position r along with the k extra attached boxes. Score = (k+1)² + dp[l][r-1][0]. We cash in the right group immediately and solve the rest independently.
3 Choice B — Merge with a matching box: Find any position m in [l, r-1] where boxes[m] == boxes[r]. Remove everything between m and r first (score = dp[m+1][r-1][0]), then boxes[m] merges with the k+1 tail group: dp[l][m][k+1].
4 Take the maximum over Choice A and all valid positions m in Choice B. Memoize with a 3D table to avoid recomputation.

The key genius is Choice B: by splitting the range at m, we defer the removal of boxes[m] and boxes[r] together, capturing the quadratic bonus of a larger group. Without the k dimension, we couldn't express this "attached extra boxes" concept.

3 Why Is This DP? — Recurrence, Invariant & State Machine ▾

Why is greedy wrong? Removing the current largest group ignores that a smaller removal now might enable a bigger group later. The optimal strategy requires seeing the future — which is exactly what DP provides.

Optimal Substructure

The maximum score for range [l,r] with k extras is built from optimal scores of smaller subranges [l, r-1] and [m+1, r-1]. Each sub-problem has its own optimal solution that we can combine.

Overlapping Subproblems

dp[l][r-1][0] and dp[l][m][k+1] appear in multiple parent calls with different (l, r, k) values. Without memoization, we'd recompute them exponentially many times.

📐 The Recurrence Relations
Base: dp[i][i][k] = (k + 1)²

Option A (remove tail immediately):
dp[l][r][k] = (k+1)² + dp[l][r-1][0]

Option B (merge with matching box at m, where boxes[m] == boxes[r]):
dp[l][r][k] = max(dp[l][r][k],
dp[m+1][r-1][0] + dp[l][m][k+1])
for all m in [l, r-1] with boxes[m] == boxes[r]

Final: dp[l][r][k] = max(Option A, all Option B variants)

🔐 The Invariant

At every call dp[l][r][k], the k extra boxes to the right of r will eventually be removed in the same operation as boxes[r]. We commit to removing them together; the only question is whether we do it now (Option A) or wait until more matching boxes are merged in (Option B). This invariant ensures we never double-count or leave boxes unaccounted.

4 The Code (Annotated) ▾

Solution 1 — Top-Down Memoization (Interview Pick) · O(n⁴) time, O(n³) space

def removeBoxes(boxes):
n = len(boxes)← Total number of boxes
memo = {} ← Cache: (l, r, k) → max points
def dp(l, r, k): ← Max pts from boxes[l..r] with k extra boxes[r]-colored boxes on right
if l > r: return 0 ← Empty range: no boxes to remove
if (l, r, k) in memo: ← Already computed? Return cached result
return memo[(l, r, k)]
# Optimization: extend k for consecutive matching boxes at right
while r > l and boxes[r-1] == boxes[r]:← Pull in adjacent same-color boxes at tail
r -= 1; k += 1 ← Shrink range, grow the k-group
# Choice A: remove the tail group (boxes[r] + k extras) immediately
res = (k + 1) ** 2 + dp(l, r - 1, 0)← Score k+1 boxes, then solve remaining [l, r-1]
# Choice B: find a matching box at m and merge with the k+1 group
for m in range(l, r): ← Scan left of r for same-colored boxes
if boxes[m] == boxes[r]:← Found a match — can merge!
res = max(res,
dp(m + 1, r - 1, 0) +← Remove everything between m and r
dp(l, m, k + 1)) ← Now boxes[m] joins the k+1-group → solve [l,m] with k+1
memo[(l, r, k)] = res ← Cache result before returning
return res
return dp(0, n - 1, 0) ← Solve full range with 0 extra boxes initially
⚠️ Why k starts at 0 and not 1?

k counts extra same-color boxes outside the current range. At the top level, boxes[r] is inside the range — it's the 1st box of its potential group, not an "extra". So k=0 means "just boxes[r] itself". When we call dp[l][m][k+1], we're saying: boxes[m] now has k+1 extras to its right (the original k plus boxes[r] itself has been conceptually shifted out).

Solution 2 — Bottom-Up Tabulation · O(n⁴) time, O(n³) space

def removeBoxes(boxes):
n = len(boxes)
# dp[l][r][k]: max pts from boxes[l..r] with k extra boxes[r]-colored on right
dp = [[[0]*n for _ in range(n)] for _ in range(n)]← 3D table n×n×n
# Fill by increasing interval length
for length in range(1, n + 1): ← Subproblem length: 1 to n
for l in range(n - length + 1):← Left endpoint of range
r = l + length - 1 ← Right endpoint
for k in range(n): ← k extra same-colored boxes on right
dp[l][r][k] = (k+1)**2 + (dp[l][r-1][0] if r > l else 0)← Choice A baseline
for m in range(l, r):← Try every split point m
if boxes[m] == boxes[r]:
score = dp[m+1][r-1][0] + dp[l][m][k+1] if r > m+1 else dp[l][m][k+1]
dp[l][r][k] = max(dp[l][r][k], score)← Choice B
return dp[0][n-1][0] ← Answer: full range, no extras
⚠️ Tabulation Order Matters

We fill by increasing interval length, not by l or r alone. This guarantees that when we compute dp[l][r][k], all sub-ranges dp[m+1][r-1][0] and dp[l][m][k+1] (which have shorter lengths) are already computed. Filling by l or r directly would access uninitialized cells.

💡 Complexity Analysis

States: O(n³) — all (l, r, k) with 0 ≤ l ≤ r < n and 0 ≤ k < n
Transitions per state: O(n) — scan through up to n positions m
Total time: O(n⁴)
Space: O(n³) for the memo table / DP array
In practice: With the consecutive-box optimization (merging the tail while l < r), the constant factor is much better than the worst case.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize Greedy Fails

The scoring is non-linear (k²). Removing a smaller group now to set up a larger group later is always worth considering. Say: "Greedy doesn't work because future removals affect current scoring decisions — this screams DP."

Step 2: Start with 2D Interval DP (and show it fails)

A natural first attempt: dp[l][r] = max score from boxes[l..r]. But when we try to formulate the recurrence, we realize we can't express "boxes[r] might be removed together with same-colored boxes outside [l,r]." This insight motivates adding the third dimension k.

Step 3: Introduce k — The Extra Count

k encodes the "context" we need: how many same-color boxes as boxes[r] are attached to its right (but outside the current range). Now dp[l][r][k] is self-contained. Two choices: (A) remove boxes[r] with its k extras now, or (B) find a matching box inside the range and merge.

Step 4: Write the Recurrence, Then Code

Write the recurrence on the whiteboard before coding. Confirm base case (l > r → 0, single box → (k+1)²), Option A formula, Option B formula. Then code a recursive solution with memo, starting simple and adding the consecutive-box optimization last.

🗣️ What to Say to the Interviewer

"The k² scoring makes greedy fail — merging groups is always worth exploring.
Interval DP seems right, but dp[l][r] alone can't capture same-color boxes outside the range.
I'll add a third dimension k: dp[l][r][k] = best score from boxes[l..r] with k extra
same-color-as-boxes[r] boxes attached outside on the right.
Two choices: (A) remove the k+1 tail group now, or (B) find a matching box at m,
clear the middle [m+1, r-1], and merge boxes[m] into the k+1 group.
Recurrence: dp[l][r][k] = max((k+1)² + dp[l][r-1][0],
max over m where boxes[m]==boxes[r] of dp[m+1][r-1][0] + dp[l][m][k+1])
Complexity: O(n⁴) time, O(n³) space."
🔮 Likely Follow-up Questions
  • "Can you optimize space?" → Not significantly; the n³ states are needed
  • "Why not bottom-up?" → Bottom-up works; fill by interval length to avoid dependency issues
  • "What if scoring is k instead of k²?" → Then greedy (always remove largest group) works optimally

7 Summary & Key Takeaways ▾
Standard 2D Interval DP is insufficient — the 3rd dimension k tracks "attached" extra same-color boxes outside the range
The k² scoring means patience pays: always consider merging groups before removal
Two choices per subproblem: remove tail now (Option A) or merge with a matching interior box (Option B)
Top-down memoization is cleaner for this problem; bottom-up requires careful ordering by interval length
The "count attached boxes" trick generalizes to any problem where interval subproblems have context from outside the range
ApproachTimeSpaceNotes
Brute Force (all permutations) O(n! × n) O(n) Try every removal order — completely intractable
2D Interval DP (incomplete) Incorrect O(n²) Cannot capture cross-range merges
3D DP Memoization ✓ O(n⁴) O(n³) Interview pick; cleaner recursion
3D DP Tabulation O(n⁴) O(n³) Same complexity; fill by interval length
🎯 When to Recognize 3D Interval DP
  • Removing elements from a sequence with non-linear scoring (k², bonuses, cascades)
  • The value of removing a subset depends on what's adjacent to the subset (outside the interval)
  • Merging equal/compatible elements before removal unlocks bonuses
  • Classic signal: "remove k adjacent identical elements for k² points" or similar
