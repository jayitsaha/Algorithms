LeetCode #164

Maximum Gap

Given an unsorted array, find the maximum difference between successive elements in its sorted form — in O(n) time and O(n) space.

Medium Sorting Pigeonhole Buckets Time: O(n) Space: O(n)
1 Understanding the Problem ▾

Given an unsorted integer array nums, return the maximum difference between successive elements in the array's sorted form. If the array has fewer than 2 elements, return 0.

💡 Key Insight

If we sorted the array, the answer is trivially the largest difference between adjacent elements. But the problem demands O(n) time — which rules out any comparison-based sorting (those have an Ω(n log n) lower bound). The question is really: how do we find the largest consecutive gap without sorting?

✓ Example: [3, 6, 9, 1, 15]

Sorted: [1, 3, 6, 9, 15]
Gaps: 2, 3, 3, 6
Maximum Gap = 6

⚡ Edge Cases

n < 2 → return 0
all same → return 0
n = 2 → |a − b|

The sorting constraint is the entire puzzle. Values can be up to 10⁹, so a simple count-array of that size is out. We need a clever O(n) bucketing approach using only O(n) extra memory.

2 The Algorithm: Pigeonhole Buckets ▾

The key is the Pigeonhole Principle: spread n numbers across n−1 equal-width buckets, and at least one bucket must be empty. The maximum gap must then cross an empty bucket — so it is the gap from the maximum of some non-empty bucket to the minimum of the next non-empty bucket.

🔑 Why Bucket Width = ⌈(max−min)/(n−1)⌉?

With n numbers in sorted order, there are n−1 consecutive gaps summing to max − min. By the averaging argument, at least one gap is ≥ (max−min)/(n−1). So if we make each bucket exactly that wide, no two numbers within the same bucket can be the answer — their distance is at most bucket_width − 1, which is less than the guaranteed maximum gap.

1 Guard: Return 0 if n < 2.
2 Find global min/max: One linear pass. If lo == hi, all elements are equal → return 0 (prevents division by zero).
3 Compute bucket width: bw = max(1, (hi − lo) // (n − 1)). Create nb = (hi − lo) // bw + 1 buckets, each storing only lo (min seen) and hi (max seen), initialized to +∞ and −∞.
4 Fill buckets: For each num, compute idx = (num − global_lo) // bw. Update blo[idx] and bhi[idx]. We never store individual values — just bucket min/max.
5 Scan for max cross-bucket gap: Sweep buckets left to right, skipping empty ones. For each non-empty bucket, compute blo[i] − prev_hi (where prev_hi is the max of the last non-empty bucket). Track the running maximum. Return it.
✗ Why Not Sort?

Comparison-based sorts need O(n log n) time. The problem explicitly requires O(n). Radix sort or bucket sort work because we exploit the integer nature and known range of the input.

✓ Pigeonhole Guarantee

With n numbers in n−1 buckets of width bw, at least one bucket is empty. The maximum gap crosses it. So comparing adjacent non-empty buckets (right.lo − left.hi) always finds the answer.

3 Why It Works: The Invariant ▾
🔐 The Core Invariant

Any two elements that land in the same bucket have a difference of at most bw − 1. But the true maximum gap is ≥ bw (by averaging). Therefore, the maximum gap always spans across at least one bucket boundary, and is always captured by blo[j] − bhi[i] for some adjacent non-empty buckets i < j.

Formal proof that max_gap ≥ bw:

Sum of all n−1 consecutive gaps in sorted order = hi − lo
bw = (hi − lo) // (n − 1) ≤ (hi − lo) / (n − 1)
Average gap = (hi − lo) / (n − 1) ≥ bw
At least one gap ≥ average ≥ bw → max_gap ≥ bw ✓

Why same-bucket elements can't be the answer: Two elements in the same bucket differ by at most bw − 1. Since max_gap ≥ bw, any same-bucket pair is strictly smaller than the true answer. We safely discard internal bucket structure — only lo and hi of each bucket matter.

⚠️ Common Confusion: "Does bw need to be exact ceiling?"

No. Floor division (hi−lo)//(n−1) makes bw slightly smaller than the average, which can produce one extra bucket. That's fine — correctness only requires that bw ≤ average_gap, so same-bucket differences stay below max_gap. The max(1, ...) guard handles the case where hi−lo < n−1 (more elements than the range, so duplicates exist).

Edge case: all elements equal (lo == hi). Every consecutive gap is 0. We detect this before bucket creation and return 0, avoiding division by zero in the bw calculation.

4 The Code (Annotated) ▾

Solution 1 — Pigeonhole Buckets (Optimal, O(n) time & space)

def maximumGap(nums):
n = len(nums)
if n < 2: ← guard: can't have a gap with fewer than 2 elements
return 0
lo, hi = min(nums), max(nums) ← O(n) scan for global range
if lo == hi: ← all equal → every gap is 0; also prevents div by 0
return 0
bw = max(1, (hi - lo) // (n - 1)) ← bucket width ≥ 1; floor div keeps bucket count manageable
nb = (hi - lo) // bw + 1 ← number of buckets needed to cover [lo, hi]
blo = [float('inf')] * nb ← track min of each bucket; inf = empty sentinel
bhi = [float('-inf')] * nb ← track max of each bucket; -inf = empty sentinel
for num in nums: ← assign every element to its bucket
idx = (num - lo) // bw ← index: offset from lo, divided by bucket width
blo[idx] = min(blo[idx], num) ← update bucket minimum
bhi[idx] = max(bhi[idx], num) ← update bucket maximum
max_gap, prev_hi = 0, lo ← prev_hi anchors from global minimum
for i in range(nb): ← sweep buckets left to right
if blo[i] == float('inf'): ← bucket is empty — skip it (this is the pigeonhole!)
continue
max_gap = max(max_gap, blo[i] - prev_hi) ← cross-bucket gap: this bucket's min minus prev bucket's max
prev_hi = bhi[i] ← advance the left anchor to this bucket's max
return max_gap
⚠️ Why prev_hi starts at lo (not 0)?

The global minimum lo always falls in bucket 0 with bhi[0] ≥ lo. When we process bucket 0, blo[0] − prev_hi = blo[0] − lo. Since lo is guaranteed to be in bucket 0, blo[0] = lo, so this gap is 0 — correct, because there's nothing to the left of the minimum. Starting at lo avoids a special case for the leftmost element.

Solution 2 — Sort and Scan (Simple, O(n log n), violates constraint)

def maximumGap(nums):
if len(nums) < 2: return 0
nums.sort() ← O(n log n) — correct but violates the O(n) time constraint
return max(b-a for a,b in zip(nums, nums[1:])) ← scan adjacent pairs for max difference
💡 Complexity Summary

Pigeonhole Buckets (Interview Pick): Time O(n), Space O(n) — meets the constraint
Sort and Scan: Time O(n log n), Space O(1) — fails the O(n) time requirement
Radix Sort + Scan: Time O(n), Space O(n) — also valid; more code, same idea

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Name the Obvious Solution

"The simplest approach: sort the array and scan for max consecutive difference. O(n log n), O(1) extra space. Correct, but the constraint says O(n) time — so we need something smarter."

Step 2: Recognize the Comparison Lower Bound

"O(n) time means we can't use any comparison-based sort. But the values are bounded integers — so we can use a non-comparison method like bucket sort or radix sort that doesn't have the Ω(n log n) lower bound."

Step 3: Invoke the Pigeonhole Principle

"Key insight: with n numbers and n−1 gaps averaging (max−min)/(n−1), at least one gap ≥ this average. If I create buckets of this width, any two elements in the same bucket differ by less than the true max gap — so I only need to look at cross-bucket gaps."

Step 4: Only Store lo/hi Per Bucket

"I don't need to know which values are in each bucket. The cross-bucket gap is always right_bucket.min − left_bucket.max. So I store only two numbers per bucket: its minimum and maximum value. O(n) total space."

🗣️ What to Say to the Interviewer

"Brute force: sort and scan adjacent pairs — O(n log n). But we need O(n).
Since O(n) rules out comparison sorting, I'll use the Pigeonhole Principle.
Key math: n numbers have n−1 gaps. Average gap = (max−min)/(n−1).
So max gap ≥ average. I'll make buckets of exactly that width.
Any two values in the same bucket differ by less than bucket_width < max_gap.
So the max gap always crosses a bucket boundary — I compare adjacent non-empty buckets.
Each bucket stores only its min and max. O(n) time, O(n) space."
🔮 Likely Follow-up Questions
  • "What if there are duplicates?" → No change. They land in the same bucket; only lo/hi matter. Duplicates never increase the gap.
  • "Can you use radix sort instead?" → Yes. Radix sort integers in O(n), then scan consecutive pairs. Same asymptotic complexity, more code.
  • "What's the minimum possible max gap?" → ⌈(max−min)/(n−1)⌉ — the ceiling of the average gap. This is tight (achieved by arithmetic progressions).
  • "Why not a count-array like counting sort?" → Values can be up to 10⁹. A direct count-array would need 10⁹ space — too much. Bucket abstraction solves this.

7 Summary & Key Takeaways ▾
The Pigeonhole Principle guarantees the max gap always crosses a bucket boundary
Bucket width ≤ average gap ≤ max gap — so same-bucket pairs can never be the answer
Only two numbers per bucket (lo, hi) are needed — O(n) total space
Skip empty buckets during the sweep; the answer is always right.lo − left.hi across non-empty neighbors
prev_hi anchors from global min, naturally handling the leftmost element as a boundary
