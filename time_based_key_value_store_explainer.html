LeetCode #981

Time Based Key-Value Store

Design a data structure that stores key-value pairs with timestamps, and retrieves the value with the largest timestamp that is less than or equal to a given timestamp.

Medium Design Hash Map + Binary Search set: O(1) get: O(log n)
1 Understanding the Problem ▾

Imagine a versioned dictionary. The same key can hold different values at different points in time. You store ("color", "red", t=1) and ("color", "blue", t=3). Now if someone queries "what was color at t=2?", the answer is "red" — the most recent value at or before t=2.

💡 Key Insight

The problem guarantees that all set() calls have strictly increasing timestamps. This single constraint is the unlock: for each key, the stored list of (timestamp, value) pairs is always sorted by timestamp. We never need to sort — and we can binary search directly.

set() — Append Only

set("foo","a",1)
set("foo","b",3)
set("foo","c",5)
→ store["foo"] = [(1,"a"),(3,"b"),(5,"c")]

get() — Floor Binary Search

get("foo",4) → "b" (ts=3 ≤ 4)
get("foo",2) → "a" (ts=1 ≤ 2)
get("foo",6) → "c" (ts=5 ≤ 6)
get("foo",0) → "" (none ≤ 0)

Edge cases to consider: Query before any stored timestamp → return "". Key not in store → return "". Query exactly matching a stored timestamp → return that exact value. Different keys are fully independent.

1 Data structure choice: A dict mapping each key to a list of (timestamp, value) tuples. The list is always sorted by timestamp because the problem guarantees strictly increasing timestamps per key on every set() call.
2 set(key, value, timestamp): Simply append (timestamp, value) to store[key]. O(1) amortized. No sorting needed — the guarantee does the work for us.
3 get(key, timestamp): We need the largest stored timestamp that is ≤ the query. This is a classic "floor search" — binary search the timestamps list with bisect_right(timestamps, query) - 1. The result index is the answer, or -1 if none qualifies.
4 Return: If idx ≥ 0, return pairs[idx][1] (the value). Otherwise return "". The idx < 0 case handles queries before all stored timestamps.
🔑 The bisect_right - 1 Pattern

bisect_right(arr, x) returns the index where x would be inserted after any existing equal elements. So bisect_right(arr, x) - 1 gives the index of the rightmost element ≤ x — exactly the "floor" we need. If the result is 0, 0 - 1 = -1, correctly signaling no valid entry.

3 Why It Works: The Invariant ▾
🔐 The Invariant

For any key k, store[k] is a list of (timestamp, value) tuples in strictly increasing order by timestamp. This holds because: (1) the problem guarantees timestamps are non-decreasing on each set() call, and (2) we only ever append. Appending a larger value to an already-sorted list keeps it sorted.

Why binary search is correct: Binary search requires a sorted sequence. The invariant above provides exactly that. bisect_right(timestamps, query) - 1 computes the floor in O(log n) — this works because the sorted order is guaranteed, not constructed.

Proof of correctness for the edge case idx = -1: bisect_right(arr, x) returns 0 when all elements of arr are > x (the insertion point is before the start). Subtracting 1 gives -1. Our code checks if idx < 0: return "" — this handles "query is before all stored timestamps" correctly.

⚠️ Common Mistake: Using bisect_left for floor search

Both bisect_left and bisect_right work here, but they differ on exact matches. bisect_right(arr, x) - 1 gives index of rightmost element ≤ x. bisect_left(arr, x) - 1 gives index of rightmost element strictly less than x — wrong for exact matches. Prefer bisect_right for floor queries.

Walk-through of get("foo", 4) with pairs=[(1,"a"),(3,"b"),(5,"c")]: timestamps=[1,3,5]. bisect_right([1,3,5], 4) = 2 (4 would go at index 2). idx = 2-1 = 1. pairs[1] = (3, "b"). Return "b". Correct — 3 is the largest timestamp ≤ 4.

4 The Code (Annotated) ▾

Solution 1 — Hash Map + Binary Search (Optimal, Interview Pick)

from bisect import bisect_right← stdlib; O(log n) floor search
class TimeMap:
def __init__(self):
self.store = {} ← key → [(ts, val), ...] always sorted by ts
def set(self, key, value, timestamp): ← O(1) amortized
if key not in self.store:
self.store[key] = [] ← first write for this key
self.store[key].append((timestamp, value)) ← guaranteed increasing ts → always sorted
def get(self, key, timestamp): ← O(log n)
pairs = self.store.get(key, []) ← [] if key never set
if not pairs: return "" ← key missing → empty
ts_list = [ts for ts, _ in pairs] ← extract timestamps for bisect
idx = bisect_right(ts_list, timestamp) - 1 ← floor index
return pairs[idx][1] if idx >= 0 else "" ← idx=-1 means no ts ≤ query
⚠️ O(n) Extraction on Every get()

The list comprehension [ts for ts, _ in pairs] is O(n) per call. For LeetCode constraints this is acceptable. In production, use two separate parallel lists per key — one for timestamps, one for values — so get() can binary search directly without extraction.

Production-Grade Variant (Parallel Lists, Recommended)

class TimeMap:
def __init__(self):
self.ts = {} ← key → [t₁, t₂, ...] sorted timestamps
self.vals = {} ← key → [v₁, v₂, ...] parallel values
def set(self, key, value, timestamp):
self.ts.setdefault(key, []).append(timestamp)
self.vals.setdefault(key, []).append(value)
def get(self, key, timestamp):
if key not in self.ts: return ""
idx = bisect_right(self.ts[key], timestamp) - 1 ← O(log n), no extraction
return self.vals[key][idx] if idx >= 0 else ""

💡 Complexity Summary

set(): O(1) amortized — append to list
get() optimal: O(log n) — binary search, n = entries for that key
get() brute: O(n) — linear scan; acceptable for small n
Space: O(N) total — one entry per set() call across all keys

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Mine the Constraints

The line "All the timestamps of set are strictly increasing" is the key. It means: the list stays sorted automatically. Without this, you'd need O(log n) inserts with bisect.insort or a SortedList. Always quote constraints back to the interviewer to show you noticed them.

Step 2: Decompose into Two Dimensions

The problem has two axes: which key and which timestamp. Key lookup → hash map (O(1)). Timestamp floor query → sorted array + binary search (O(log n)). Nesting these gives the solution: dict[key] → sorted list of (ts, val).

Step 3: Nail the Binary Search Variant

"Largest timestamp ≤ query" is a floor search. The pattern is: bisect_right(arr, x) - 1. Memorize: bisect_right gives insertion point after any equal elements → subtract 1 for floor. Edge case: result is -1 when all elements exceed the query.

Step 4: Walk Through Edge Cases Proactively

Before coding: "Three edge cases I want to handle: (1) key never set → return '', (2) query timestamp before all stored values → idx = -1, return '', (3) exact timestamp match → bisect_right correctly returns the match." Say this out loud — it signals thoroughness.

🗣️ What to Say to the Interviewer

"I'll design this with a dictionary mapping each key to a sorted list of (timestamp, value) pairs.
set() just appends — O(1) — because the problem guarantees timestamps are strictly increasing, so the list stays sorted automatically.
get() needs the largest timestamp ≤ query. Since the list is sorted, I binary search with bisect_right(timestamps, query) - 1 to get the floor index. If that index is negative, no valid entry exists and I return empty string.
set: O(1) amortized. get: O(log n). Space: O(N) for N total insertions.
For an even cleaner production version, I'd keep timestamps and values in separate parallel lists to avoid extracting timestamps on every get()."
🔮 Likely Follow-up Questions
  • "What if timestamps aren't guaranteed increasing?" → Use bisect.insort for O(n) insert, or a SortedList for O(log n) insert + O(log n) search.
  • "How to support delete?" → Soft delete: append a sentinel value (None or tombstone). Hard delete: O(n) reconstruction.
  • "How to get all values in a time range [t1, t2]?" → Two bisect calls: left = bisect_left(ts, t1), right = bisect_right(ts, t2). Slice pairs[left:right].
  • "Scale this to millions of keys?" → Shard by key hash. Each shard is an independent TimeMap instance.

🎯 When to Recognize Hash Map + Binary Search
  • "Multiple values per key, find value at a given index/time" → sorted list + binary search
  • "Data arrives sorted, floor/ceiling query needed" → append + bisect
  • Design problems with two independent lookup dimensions → nest data structures
  • Constraint says "timestamps are strictly increasing" → never sort; binary search directly
