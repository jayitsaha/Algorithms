LeetCode #72

Edit Distance

Given two strings word1 and word2, return the minimum number of operations required to convert word1 to word2. Each operation is an insert, delete, or replace of a single character.

Medium Dynamic Programming DP: LCS / Insert-Delete-Replace Time: O(m·n) Space: O(m·n)
1 Understanding the Problem ▾

Imagine you have a word on a whiteboard and want to erase-and-rewrite it into a different word, one character at a time. Each step you can: insert a new character anywhere, delete an existing character, or replace one character with another. The edit distance (Levenshtein distance) is the fewest steps needed.

💡 Key Insight

At every position (i, j) in the two strings, we face a binary choice: the characters either match (free — inherit the diagonal) or they don't (pay 1 operation — but which? Try all three and keep the cheapest). This "try everything, cache results" structure is the signature of dynamic programming.

Example: "horse" → "ros" (answer: 3)

horse → rorse (replace h→r)
rorse → rose (delete 2nd r)
rose → ros (delete e)
3 operations total

Example: "intention" → "execution" (answer: 5)

intention → inention (delete t)
inention → enention (replace i→e)
enention → exention (replace n→x)
exention → exection (replace n→c)
exection → execution (insert u)
5 operations total

Edge cases to watch: If either string is empty — converting empty to "abc" costs 3 (all inserts); converting "abc" to empty costs 3 (all deletes). Both empty → 0. Strings up to 500 characters (so O(n²) is fine, O(3^n) is not).

🔑 Why Can't We Use Greedy?

A greedy approach — always picking the locally cheapest operation — fails here. For "horse" → "ros", deleting 'h' greedily creates "orse" and leads to a longer path. Replacing 'h'→'r' first opens a much cheaper path overall. We need to explore all subproblem branches, which is exactly what DP does.

2 The Algorithm: 2D Tabulation DP ▾

Define dp[i][j] = minimum edits to convert the first i characters of word1 into the first j characters of word2. We build a (m+1) × (n+1) table row by row, left to right.

1 Create the table: dp[0..m][0..n]. The extra row/column (index 0) represents the empty-string prefix. Initialize all to 0.
2 Base case — first column: dp[i][0] = i. Converting the first i chars of word1 to an empty string requires exactly i deletions.
3 Base case — first row: dp[0][j] = j. Converting an empty string to the first j chars of word2 requires exactly j insertions.
4 Fill the table: For each cell (i, j) — if word1[i-1] == word2[j-1]: free match, dp[i][j] = dp[i-1][j-1]. Otherwise: dp[i][j] = 1 + min(dp[i-1][j], dp[i][j-1], dp[i-1][j-1]) — cheapest of delete, insert, replace.
5 Return dp[m][n] — the answer for the full strings.
📐 The Recurrence Relation
dp[i][0] = i  (delete all i chars to reach empty string)
dp[0][j] = j  (insert all j chars from empty string)

If word1[i-1] == word2[j-1]: ← characters match
    dp[i][j] = dp[i-1][j-1]  (free — no cost)

Else: ← characters differ
    dp[i][j] = 1 + min(
        dp[i-1][j],  ← delete word1[i-1]
        dp[i][j-1],  ← insert word2[j-1]
        dp[i-1][j-1]  ← replace word1[i-1] with word2[j-1]
    )

⚠️ What Each Direction Means

↖ dp[i-1][j-1] = replace word1[i-1] with word2[j-1] — align both chars
↑ dp[i-1][j] = delete word1[i-1] — word1 gets one char shorter
← dp[i][j-1] = insert word2[j-1] into word1 — word2 gets one more char covered

3 Why It Works: DP Invariant & Correctness ▾
🔐 Why Is This Dynamic Programming?

Optimal Substructure: The optimal edit distance for "horse"→"ros" depends on the optimal distance for their prefixes. Knowing the best way to convert "hors"→"ro" lets you extend it to "horse"→"ros" — smaller answers compose into larger ones without interference.

Overlapping Subproblems: In naive recursion, the subproblem dp("hor","r") would be recomputed from multiple parent calls. With an (m+1)×(n+1) table, we solve each of the O(m·n) subproblems exactly once.

The Invariant: When we compute dp[i][j], every cell above and to its left is already the correct optimal value. We always read from precomputed correct cells — the invariant holds by induction from the base cases.

Proof sketch for the recurrence:

  • If characters match, aligning them costs nothing — we simply inherit the optimal cost for the smaller prefixes.
  • If they don't match, any optimal edit sequence must eventually handle this position. It either: (a) deletes word1[i-1] — leaving the subproblem dp[i-1][j]; (b) inserts word2[j-1] — leaving dp[i][j-1]; or (c) replaces word1[i-1] — leaving dp[i-1][j-1]. Taking 1 + min of these three covers all cases.
⚠️ Common Misconception: Replace = Delete + Insert?

Mathematically yes, a replace can be simulated as delete then insert (2 ops). But in Levenshtein distance, replace is defined as one single operation. The recurrence correctly assigns cost 1 to replace. If you used cost 2 for replace, you'd get a different (and wrong for this problem) distance metric.

Edge case correctness: dp[i][0] = i is trivially correct (delete all i chars). dp[0][j] = j is trivially correct (insert all j chars). Both sequences of deletions/insertions are the only option when one string is empty, so they are trivially optimal.

4 The Code (Annotated) ▾

Solution 1 — Tabulation / Bottom-Up DP (Interview Pick)

def minDistance(word1: str, word2: str) -> int:
m, n = len(word1), len(word2) ← store lengths (cleaner than re-calling len repeatedly)
dp = [[0]*(n+1) for _ in range(m+1)] ← (m+1)×(n+1) table; extra row/col for empty-string prefixes
for i in range(m+1): ← base: dp[i][0] = i (delete all i chars to get to empty)
dp[i][0] = i
for j in range(n+1): ← base: dp[0][j] = j (insert all j chars from empty)
dp[0][j] = j
for i in range(1, m+1): ← iterate over each character of word1 (1-indexed)
for j in range(1, n+1): ← iterate over each character of word2 (1-indexed)
if word1[i-1] == word2[j-1]: ← i-th char of word1 = word1[i-1] (0-indexed string!)
dp[i][j] = dp[i-1][j-1] ← match: inherit diagonal, no cost
else:
dp[i][j] = 1 + min( ← mismatch: pay 1 for whichever op is cheapest
dp[i-1][j], ← delete word1[i-1] (look up = word1 shrinks)
dp[i][j-1], ← insert word2[j-1] (look left = word2 covered)
dp[i-1][j-1] ← replace word1[i-1] with word2[j-1] (diagonal)
)
return dp[m][n] ← answer: full word1 (len m) converted to full word2 (len n)
⚠️ Top Bug: word1[i-1] vs word1[i]

Our DP table uses 1-based prefix lengths: dp[i][j] = "first i chars of word1." So the i-th character (1-based) is at position i-1 in the 0-indexed Python string. Forgetting this off-by-one causes silent wrong answers — a frequent interview mistake.

Solution 2 — Memoization / Top-Down DP

from functools import lru_cache
def minDistance(word1: str, word2: str) -> int:
@lru_cache(maxsize=None) ← Python caches every (i,j) pair automatically
def dp(i, j): ← solve: min edits for word1[:i] → word2[:j]
if i == 0: return j ← base: word1 exhausted, insert remaining j chars of word2
if j == 0: return i ← base: word2 exhausted, delete remaining i chars of word1
if word1[i-1] == word2[j-1]: ← match: recurse on smaller subproblem, no cost
return dp(i-1, j-1)
return 1 + min( ← mismatch: try all three ops, pick cheapest
dp(i-1, j), ← delete from word1
dp(i, j-1), ← insert into word1
dp(i-1, j-1) ← replace in word1
)
return dp(len(word1), len(word2))

Solution 3 — Brute Force Recursion (for understanding only):

def minDistance(word1, word2): ← WARNING: O(3^max(m,n)) — time limit exceeded
def rec(i, j):
if i == 0: return j
if j == 0: return i
if word1[i-1] == word2[j-1]: return rec(i-1, j-1)
return 1 + min(rec(i-1,j), rec(i,j-1), rec(i-1,j-1))
return rec(len(word1), len(word2)) ← recomputes rec("hor","r") many times without cache
💡 Complexity Summary

Tabulation: Time O(m·n), Space O(m·n) for the full table.
Memoization: Time O(m·n), Space O(m·n) for cache + O(m+n) call stack.
Space-optimized tabulation: O(min(m,n)) — use two 1D arrays (current and previous row).
m = len(word1), n = len(word2)

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify the DP Signal

"We want the minimum number of operations to go from A to B. 'Minimum cost to transform X into Y' is the canonical DP signal — especially when there are multiple ways to make progress and local decisions affect future cost."

Step 2: Define State Clearly

"Let dp[i][j] = min edits to convert word1[0..i) to word2[0..j). This is a 2D state over prefix lengths. Final answer = dp[m][n]."

Step 3: Enumerate Transitions

"At (i,j): if chars match, free — dp[i-1][j-1]. If not: try delete (dp[i-1][j]), insert (dp[i][j-1]), replace (dp[i-1][j-1]). Take 1 + min of the three."

Step 4: Set Base Cases

"dp[i][0] = i (all deletes), dp[0][j] = j (all inserts). These seed the first row and column of the table."

Step 5: Mention Optimization

"For bonus points: since dp[i][j] only needs the current and previous row, space reduces from O(m·n) to O(min(m,n)) with a rolling array."

🗣️ What to Say to the Interviewer

"This is a classic 2D DP problem — the edit distance (Levenshtein distance).
I'll define dp[i][j] = min edits to convert word1's first i chars to word2's first j chars.
Base cases: dp[i][0] = i, dp[0][j] = j.
Recurrence: if chars match → dp[i-1][j-1]. If not → 1 + min(delete=dp[i-1][j], insert=dp[i][j-1], replace=dp[i-1][j-1]).
Time O(m·n), Space O(m·n), reducible to O(min(m,n)) with rolling rows."
🔮 Likely Follow-up Questions
  • "Reduce space to O(n)?" → Keep only previous row; use one extra variable for the diagonal (dp[i-1][j-1]).
  • "Print the actual edit sequence?" → Backtrack from dp[m][n]: at each cell, check which neighbor it came from (diagonal=replace or match, up=delete, left=insert).
  • "Allow only insert and delete?" → Replace = delete + insert (cost 2). Adjust recurrence: only min(dp[i-1][j], dp[i][j-1]) + 1. This also gives 2*(m+n−2*LCS) as a formula.
  • "Weighted operations?" → Replace 1 + with individual costs c_del, c_ins, c_rep.

7 Summary & Key Takeaways ▾
Edit distance = Levenshtein distance: min insert/delete/replace to go from word1 to word2
State: dp[i][j] = min edits for word1[0..i) → word2[0..j)
Match = free (inherit diagonal); mismatch = 1 + min(delete, insert, replace)
Base cases: dp[i][0] = i and dp[0][j] = j (empty string conversions)
Off-by-one trap: word1[i-1] not word1[i] — DP is 1-indexed, strings are 0-indexed
Space optimization: rolling two rows cuts O(m·n) → O(min(m,n))
ApproachTimeSpaceNotes
Naive Recursion O(3^max(m,n)) O(m+n) Exponential — understanding only
Memoization (Top-Down) O(m·n) O(m·n) Natural recursion, easy to derive
Tabulation (Bottom-Up) ✓ O(m·n) O(m·n) Interview pick; iterative, no stack
Space-Optimized Tabulation O(m·n) O(min(m,n)) Mention for bonus points
🎯 When to Recognize the Edit Distance Pattern
  • "Minimum operations to transform sequence X into sequence Y"
  • Two strings/arrays, comparing subproblems on prefixes of both
  • Three or more choices at each mismatch (generalize to weighted costs)
  • DNA sequence alignment, spell-checking, diff tools (e.g., git diff)
