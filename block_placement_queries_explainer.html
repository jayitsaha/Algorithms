LeetCode #3161

Block Placement Queries

Given obstacle placement and block-size queries on a number line [0, N], determine for each query the maximum block width that fits in any gap before the query position.

Hard Advanced Data Structures Segment Tree Time: O(Q log N) Space: O(N)
1 Understanding the Problem ▾

You have an infinite number line. Position 0 always has an implicit obstacle. You receive a sequence of queries:

  • Type 1: Place an obstacle — [1, x] places an obstacle at position x.
  • Type 2: Block query — [2, x, sz] asks: can a block of size sz fit anywhere in the range [0, x] without touching an obstacle?

A block of size sz fits in a gap if the gap (distance between consecutive obstacles) is strictly greater than sz. We answer true or false for each type-2 query.

💡 Key Insight

The problem reduces to: given the set of obstacles placed so far, what is the maximum gap between any two consecutive obstacles in the range [0, x]? If max_gap > sz, return true. We need a data structure that answers this "maximum gap in a prefix" query after dynamic obstacle insertions.

✓ Block Fits

Obstacles: [0, 3, 8]
Gaps: 3, 5
Query [2, 10, 4]
max_gap=5 > sz=4 → True

✗ Block Doesn't Fit

Obstacles: [0, 3, 6, 8]
Gaps: 3, 3, 2
Query [2, 10, 4]
max_gap=3 < sz=4 → False

Key subtleties:

  • Position 0 is always an obstacle (acts as the left boundary).
  • For a type-2 query at position x, we also treat x itself as a right boundary — the maximum gap is computed from obstacles in [0, x] PLUS a virtual obstacle at x.
  • Queries are processed online — obstacles accumulate across queries.
2 The Algorithm: Offline Sort + Segment Tree on Max Gap ▾

We process queries offline — sort all events by position, then sweep left to right. As we encounter obstacles, we update a segment tree that tracks the maximum gap in any prefix. For each type-2 query at position x, we query the segment tree for the maximum gap in [0, x].

1 Coordinate compress: Collect all x-values from both query types. Sort and deduplicate. Map to indices 1..M. This lets the segment tree work on indices instead of large x values.
2 Segment tree on gaps: Build a segment tree over positions 1..M. Each leaf i stores the gap between obstacle at position i and its predecessor obstacle. Internal nodes store the maximum gap in their range. Initially all leaves = 0 (no obstacles except 0).
3 Sort events by x: Sort all type-1 (obstacles) and type-2 (queries) events by their x-coordinate. Process in order.
4 Type-1 (obstacle at x): Find x's predecessor in the sorted obstacle set. The gap that previously existed between predecessor and successor is now split. Update the segment tree: remove old gap, add two smaller gaps.
5 Type-2 (query x, sz): The answer is whether seg.query_max(1, idx(x)) > sz, where idx(x) is x's compressed index. We also account for the gap from the last obstacle to x itself.
🔑 Why Offline Works Here

By sorting all queries by x, we can maintain a sorted set of obstacles and sweep through them once. Each obstacle insertion splits exactly one gap into two — a classic "interval management" pattern that the segment tree handles in O(log N).

3 🔬 Algorithm Deep-Dive: Segment Tree ▾

A Segment Tree is a binary tree where each node stores an aggregate (sum, min, max, GCD, etc.) over a contiguous subarray. It supports both point updates and range queries in O(log N) time.

🏛️ Origin & Use Cases

Segment trees were formalized in computational geometry in the 1970s. They are the workhorse for problems requiring dynamic range queries — where the array changes between queries. Classic problems: range sum with updates, range min/max with updates, lazy propagation for range updates.

Core Invariant: Every internal node v covering range [l, r] stores aggregate(arr[l..r]). After any point update at index i, exactly O(log N) nodes (the path from leaf i to root) need to be refreshed.

class SegTree:
def __init__(self, n):
self.n = n
self.tree = [0] * (4 * n) ← 4n size is safe for any n
def update(self, node, start, end, idx, val):
if start == end: ← Leaf node: store the value directly
self.tree[node] = val
return
mid = (start + end) // 2
if idx <= mid:
self.update(2*node, start, mid, idx, val)
else:
self.update(2*node+1, mid+1, end, idx, val)
self.tree[node] = max(self.tree[2*node], self.tree[2*node+1]) ← Pull up
def query(self, node, start, end, l, r):
if r < start or end < l: ← Out of range: return identity (0 for max)
return 0
if l <= start and end <= r: ← Fully covered: return this node's value
return self.tree[node]
mid = (start + end) // 2
return max(self.query(2*node, start, mid, l, r),
self.query(2*node+1, mid+1, end, l, r))
💡 Why O(log N) Range Query Works

The query range [l, r] can intersect at most O(log N) nodes at each level of the tree. Nodes fully covered by [l, r] return immediately; nodes partially covered recurse. The recursion tree has depth log N and visits at most 4 log N nodes total.

⚠️ Common Mistakes with Segment Trees
  • Array size: Always allocate 4*n nodes. Using 2*n is common but can overflow for non-power-of-2 sizes.
  • Identity element: For max queries, return 0 (or -∞) when out of range. For min, return +∞. Getting this wrong corrupts all answers.
  • 1-indexed vs 0-indexed: Be consistent — the root is at index 1, left child at 2*node, right at 2*node+1.

Generalizations: Segment trees extend to lazy propagation (range updates in O(log N)), persistent segment trees (keep all historical versions), and 2D segment trees (for 2D range queries). Recognizing "dynamic range query" is the trigger to reach for this tool.

4 Why It Works: The Invariant ▾
🔐 The Invariant

At all times, seg.tree[node] for a node covering range [l, r] equals the maximum gap between consecutive obstacles whose right endpoint falls in [l, r]. This allows O(log N) max-gap queries over any prefix [0, x].

Why splitting works correctly: When obstacle at position x is inserted:

  1. Find predecessor prev and successor next in the sorted obstacle set.
  2. The gap (next - prev) was stored at the compressed index of next. Set it to next - x.
  3. Store x - prev at the compressed index of x.
  4. The two updates take O(log N) each via the segment tree.
⚠️ The Boundary Trick for Type-2 Queries

For a query [2, x, sz], the block must fit within [0, x]. We treat x as a virtual right boundary. So we compute: gap from last obstacle before x to x itself (x - prev_obstacle), AND the maximum gap from the segment tree in [0, idx(x)-1]. The answer is max(these two) > sz.

Edge cases: If no obstacles have been placed (only position 0 exists), the only gap is from 0 to x, which is x itself. For obstacles at x's position, that gap contributes 0 to the right side. The segment tree always returns the correct max because it only contains committed obstacles.

5 The Code (Annotated) ▾

Solution 1 — Offline + Coordinate Compression + Segment Tree (Interview Pick)

from sortedcontainers import SortedList
class Solution:
def getResults(self, queries):
# Collect all x-coords; 0 is always an obstacle
xs = sorted(set([0] + [q[1] for q in queries])) ← Coordinate compression
rank = {v: i+1 for i, v in enumerate(xs)} ← 1-indexed rank map
n = len(xs)
tree = [0] * (4 * n) ← Segment tree: max gap
def update(node, s, e, idx, val):
if s == e: tree[node] = val; return
mid = (s + e) // 2
if idx <= mid: update(2*node, s, mid, idx, val)
else: update(2*node+1, mid+1, e, idx, val)
tree[node] = max(tree[2*node], tree[2*node+1])
def query(node, s, e, l, r): ← Max gap in range [l,r]
if r < s or e < l: return 0
if l <= s and e <= r: return tree[node]
mid = (s + e) // 2
return max(query(2*node, s, mid, l, r),
query(2*node+1, mid+1, e, l, r))
obstacles = SortedList([0]) ← Always start with obstacle at 0
ans = []
for q in queries:
x = q[1]
if q[0] == 1: ← Type 1: place obstacle at x
idx = obstacles.bisect_left(x) ← Find insertion point
if idx == len(obstacles) or obstacles[idx] != x:
prev = obstacles[idx - 1] ← Left neighbor obstacle
succ = obstacles[idx] if idx < len(obstacles) else None
obstacles.add(x)
update(1, 1, n, rank[x], x - prev) ← Gap: x to its left neighbor
if succ is not None:
update(1, 1, n, rank[succ], succ - x) ← Shrink succ's gap
else: ← Type 2: can block of sz fit in [0,x]?
sz = q[2]
prev_obs = obstacles[obstacles.bisect_right(x) - 1]
gap_to_x = x - prev_obs ← Gap from last obstacle to x
max_gap = max(gap_to_x,
query(1, 1, n, 1, rank[x])) ← Max gap in all of [0,x]
ans.append(max_gap > sz) ← True if block fits
return ans
⚠️ Why SortedList for obstacles?

SortedList (from sortedcontainers) gives O(log N) insert and predecessor queries. You could also use Python's bisect module on a plain list for O(log N) search with O(N) insert — acceptable for interview, but O(N) insert is too slow for the hard constraint. In interviews, mention both and use bisect if library not allowed.

Solution 2 — Brute Force (for intuition only, TLE)

def getResults_brute(queries):
obstacles = sorted({0})
ans = []
for q in queries:
if q[0] == 1:
import bisect; bisect.insort(obstacles, q[1]) ← O(N) insert
else:
x, sz = q[1], q[2]
obs_in_range = [o for o in obstacles if o <= x] + [x]
max_gap = max(obs_in_range[i]-obs_in_range[i-1] ← O(N) scan
for i in range(1, len(obs_in_range)))
ans.append(max_gap > sz)
return ans ← O(Q*N) overall — TLE
💡 Complexity Summary

Optimal (Segment Tree): Time O(Q log Q) for sorting + O(Q log N) for queries = O(Q log Q). Space O(N) for segment tree + O(Q) for SortedList.
Brute Force: Time O(Q · N) for N obstacles and Q queries. Fails for N, Q up to 5×10⁴.

6 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Reframe the Core Question

"For each type-2 query at x: what is the largest gap between consecutive obstacles in [0, x]?" This is a dynamic range-max query — the classic signal for a segment tree.

Step 2: Propose Brute Force

"Maintain a sorted list of obstacles. For each type-2 query, scan all obstacles ≤ x to find the max gap. O(Q·N) — too slow for 5×10⁴ queries."

Step 3: Identify the Bottleneck

"The bottleneck is finding max gap in a prefix after each obstacle insertion. If I precompute gaps and store them in a max segment tree, both operations become O(log N)."

Step 4: Coordinate Compress

"x-values can be up to 5×10⁴, but there are at most Q distinct values. Map them to 1..Q ranks. The segment tree works on these compressed indices — O(Q log Q) total."

Step 5: Handle the Boundary Gap

"For type-2, also consider the gap from the last obstacle before x to x itself — this gap exists even if x has no obstacle. Take max of this and the segment tree result."

🗣️ What to Say to the Interviewer

"Let me restate: for a block of size sz to fit in [0, x], I need some gap between consecutive obstacles in that range to be strictly greater than sz.
So the question is: what is max(gap) over all consecutive obstacle pairs in [0, x]?
This is a dynamic range-max query — segment tree is perfect.
I'll coordinate-compress x-values, build a segment tree where each leaf stores the gap to its left obstacle, and internal nodes store the max in their range.
Each obstacle insert: split one gap into two — two O(log N) updates.
Each type-2 query: O(log N) range-max query plus O(log N) predecessor search.
Overall: O(Q log Q) time, O(Q) space."
🔮 Likely Follow-up Questions
  • "What if obstacles can be removed?" → Need a different structure (order-statistic tree or lazy segment tree with deletion)
  • "What if queries are online (no offline trick)?" → Still works — we process queries in arrival order with the same segment tree
  • "Can you use a Fenwick tree instead?" → No — Fenwick trees support prefix sum, not prefix max (unless values are non-decreasing)
  • "How would lazy propagation help?" → Not needed here; each update touches O(log N) nodes with simple point updates

8 Summary & Key Takeaways ▾
Dynamic range-max after point updates = Segment Tree
Each obstacle insertion splits one gap into two — O(log N) with seg tree
Coordinate compression keeps the tree small when x-values are sparse
Always handle the boundary: gap from last obstacle to query position x
SortedList (or bisect) finds predecessor in O(log N) for gap computation
🎯 When to Use Segment Tree
  • Need range aggregate (sum / min / max / GCD) with point or range updates
  • Both operations must be O(log N), not O(N)
  • Array changes between queries (otherwise prefix arrays suffice)
  • Keywords: "after each update, what is the max/min/sum in range [l, r]?"
Range Sum Query - Mutable Medium — Classic seg tree / Fenwick tree on sum
The Skyline Problem Hard — Segment tree on max height with sweep line
Count of Smaller Numbers After Self Hard — Segment tree / Fenwick for rank queries
Falling Squares Hard — Segment tree with lazy propagation for interval max
Rectangle Area II Hard — Coordinate compress + seg tree for area sweep
Minimum Interval to Include Each Query Hard — Offline sort + seg tree / heap
