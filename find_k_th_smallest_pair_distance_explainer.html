LeetCode #719

Find K-th Smallest Pair Distance

Given an integer array nums and an integer k, return the k-th smallest distance among all pairs (i, j) where i < j.

Hard Binary Search BS on Answer + Count Pairs Time: O(n log n + n log W) Space: O(1)
1 Understanding the Problem ▾

Consider every ordered pair of indices (i, j) where i < j. For each pair, compute the absolute difference |nums[i] − nums[j]|. If you sorted all these distances, which one lands at position k (1-indexed)? That's the k-th smallest pair distance.

💡 Key Insight

With n up to 10,000, there are up to 50 million pairs — far too many to enumerate and sort. The breakthrough: binary search on the answer itself. Instead of listing all distances, we ask: "How many pairs have distance ≤ d?" This count function is monotone in d, so binary search finds the exact k-th smallest in O(n log W) time.

2 The Algorithm: Binary Search on Answer + Sliding Window Count ▾

We never sort pair distances. Instead, we binary search over distance values in [0, max−min], and for each candidate distance mid, we count valid pairs in O(n) using a two-pointer sliding window over the sorted array.

1 Sort nums. Once sorted, all differences nums[j] − nums[i] (j>i) are non-negative and the array has the property that pairs with small differences are adjacent. This enables efficient two-pointer counting.
2 Set search bounds. lo = 0 (minimum possible distance), hi = nums[-1] − nums[0] (maximum, since array is sorted). The k-th smallest distance must lie in [lo, hi].
3 Binary search. Pick mid = (lo+hi)//2. Count how many pairs have distance ≤ mid. If count ≥ k, the answer is ≤ mid (set hi=mid). Otherwise, the answer is > mid (set lo=mid+1).
4 Count in O(n). For each right pointer j, advance left pointer i while nums[j]−nums[i] > mid. Then j−i valid pairs end at j (partners are indices i, i+1, ..., j-1). Sum over all j.
5 Return lo. When lo == hi, binary search converges. lo is the smallest distance d such that at least k pairs have distance ≤ d — that is exactly the k-th smallest pair distance.
📐 The Count Function (Two-Pointer Sliding Window)
def count_pairs(nums, mid): # O(n) after sorting
  count = i = 0
  for j in range(len(nums)): # right pointer scans right
    while nums[j] - nums[i] > mid: i += 1 # shrink from left
    count += j - i # indices i..j-1 all pair with j
  return count
🔑 Why the Count is Monotone (the BS prerequisite)

As mid increases, more pairs satisfy nums[j]−nums[i] ≤ mid, so count is non-decreasing. This is exactly the monotone property needed for binary search: we're finding the smallest mid where count ≥ k.

3 Why It Works: The Invariant ▾
🔐 Binary Search Invariant

Throughout the search, we maintain: count_pairs(lo−1) < k and count_pairs(hi) ≥ k. When lo == hi, both conditions hold for the same value, so lo IS the smallest distance where count ≥ k — the k-th smallest pair distance.

Why the answer always exists in [lo, hi]: The k-th smallest distance d* is a real pair distance in the array. After sorting, d* is achievable by some pair (nums[i], nums[j]), so d* lies in [0, nums[-1]−nums[0]] = our initial [lo, hi]. The binary search never eliminates d* because: if count(mid) ≥ k and mid < d*, then at least k pairs have distance ≤ mid < d*, but d* is the k-th smallest — contradiction. So whenever count(mid) ≥ k, the answer is truly ≤ mid.

Why two-pointer count is O(n) total: The left pointer i only moves right and is never reset between j iterations. Over the entire loop, i moves at most n times total. So the inner while loop contributes O(n) amortized, making the full count O(n).

⚠️ Most Common Mistake: hi = mid−1 instead of mid

When count ≥ k, you might be tempted to set hi = mid−1 like in a "find exact value" binary search. Don't. mid itself could be the answer. Using hi = mid keeps mid as a candidate and shrinks the range. Combined with lo = mid+1 on the other branch, the loop terminates when lo == hi == the exact answer.

Handling duplicates (distance = 0): lo starts at 0, so distance 0 is always in the search space. If there are k or more pairs with the same value, count(0) ≥ k and hi shrinks to 0 immediately — the answer is 0. Correct.

4 The Code (Annotated) ▾
def smallestDistancePair(nums, k): ← nums: integer list, k: rank (1-indexed)
nums.sort() ← sort in-place; required for two-pointer counting
n = len(nums) ← cache length
lo, hi = 0, nums[-1] - nums[0] ← distance search space: 0 to max possible diff
while lo < hi: ← continue until converged; lo==hi is the answer
mid = (lo + hi) // 2 ← candidate distance threshold (floor division)
# Count pairs with dist <= mid ← two-pointer sliding window; O(n) amortized
count = i = 0 ← count: valid pair tally; i: left pointer (never resets)
for j in range(n): ← j: right pointer, fixes the larger element of pair
while nums[j]-nums[i]>mid: ← if window too wide, shrink from left
i += 1 ← discard nums[i] as left partner for this j
count += j - i ← nums[i..j-1] all form valid pairs with nums[j]
if count >= k: ← enough pairs have dist <= mid; answer could be <= mid
hi = mid ← keep mid as candidate, shrink upper bound
else: ← too few pairs; answer must be > mid
lo = mid + 1 ← exclude mid, raise lower bound
return lo ← lo==hi: smallest d with count(d)>=k = k-th smallest dist
⚠️ Do Not Reset i Between j Iterations

Notice i is initialized once before the for loop and carries its value across j iterations. This is intentional and critical. Since j always increases and the array is sorted, i never needs to go left. If you wrote i = 0 inside the for loop, counting would still be correct but each j would scan from index 0 — making the inner while loop O(n) per j, giving O(n²) total and defeating the purpose entirely.

Complexity Analysis: Sorting takes O(n log n). Binary search runs O(log(nums[-1]−nums[0])) = O(log W) iterations where W ≤ 10⁶. Each iteration runs the two-pointer count in O(n) amortized. Total: O(n log n + n log W). Space: O(1) extra (sort is in-place).

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Brute force and identify the bottleneck

Generate all C(n,2) pairs, compute distances, sort, index. O(n² log n) and O(n²) space. For n=10,000, that's 50M pairs — TLE. The bottleneck is enumerating all pair distances.

Step 2: Recognize the "k-th smallest" pattern

"K-th smallest of an implicitly huge set" is the canonical signal for binary search on answer. The key question: can I check "is the answer ≤ d?" efficiently? Yes: count_pairs(d) — does it count ≥ k pairs?

Step 3: Establish monotonicity

As d grows, count_pairs(d) grows or stays the same — it never decreases. This guarantees the binary search property: there's a clean threshold where count flips from <k to ≥k.

Step 4: Implement count_pairs in O(n)

Sort the array first. Then two-pointer: j is the right end, i is the left. For each j, advance i until nums[j]−nums[i] ≤ mid. All elements from i to j-1 are valid left partners. Add j−i to count. Amortized O(n) — i never resets.

Step 5: Nail the binary search termination

lo=0, hi=max−min. Use while lo<hi. When count≥k → hi=mid. When count<k → lo=mid+1. Return lo. This converges to the smallest d where count(d)≥k.
