LeetCode #237

Delete Node in a Linked List

Given only the node to delete (not the head), remove it from the singly linked list in-place.

Medium Linked List Copy Next Node Value Time: O(1) Space: O(1)
1 Understanding the Problem ▾

In a normal linked list deletion, you find the previous node and redirect its next pointer to skip the target. But this problem hands you only the node to delete — no head, no previous node, nothing before it. The standard approach is simply impossible.

💡 Key Insight

You cannot remove this specific node from the list. But you can make it impersonate the next node: copy the successor's value into the current node, then unlink the successor. From any external perspective, the target's value has vanished from the sequence.

✗ Standard Approach (Impossible Here)

List: [4] → [5] → [1] → [9]
Want: prev.next = node.next
Problem: "prev" (the 4-node)
is completely unreachable.
We only have a ref to [5].

✓ Copy-Forward Trick (Works!)

node.val = node.next.val → 1
node.next = node.next.next → 9
Before: [4]→[5]→[1]→[9]
After: [4]→[1]→[9] ✓
The "5" is gone from the chain.

Guaranteed constraints (key to why the trick works): The problem guarantees the given node is never the tail — it always has a valid node.next. If the tail could be given, this trick would break because there'd be no successor to copy from.

Edge cases: Two-node list where the given node is index 0 (the head) — still works fine: copy next value, point next to None. Duplicate values in the list — irrelevant, we're copying a value not an identity. All values the same — irrelevant for the same reason.

2 The Algorithm: Copy Next and Skip ▾

The trick is elegant: instead of removing this node, we delete the next node by copying its content here first. We have full access to our successor, so we can be its "predecessor" and cut it out.

1 Copy successor's value: node.val = node.next.val. The given node now carries the value that was in the next node. The original value we wanted to delete has been overwritten and is effectively gone.
2 Skip the successor: node.next = node.next.next. We redirect our next pointer to jump over the now-redundant successor node. It becomes unreferenced and eligible for garbage collection.
3 Done — O(1) time and space. No loops, no traversal, just two pointer/value assignments. The list length has decreased by one and the deleted value is gone.

3 Why It Works: The Invariant ▾
🔐 The Core Invariant

After both operations, the sequence of values reachable from any pointer into the list is identical to the original list with the deleted value removed. The physical node objects may differ from what you'd expect, but the observable list is correct.

Proof of correctness: Let the list be ... → A → target(v) → B(w) → C → ... and we want to delete target (the node with value v).

After Step 1: target.val = w. The list reads ... → A → target(w) → B(w) → C → ....

After Step 2: target.next = C. The list reads ... → A → target(w) → C → ....

Value sequence: ..., val(A), w, val(C), ... — exactly what you'd get by conventionally removing target(v). The value v is gone. ✓

⚠️ Why the Tail Constraint is Critical

If target were the tail node, node.next would be None. Then node.val = node.next.val would raise AttributeError: 'NoneType' object has no attribute 'val'. The problem guarantees the given node is never the tail — this constraint is not arbitrary, it is necessary for the trick to work.

Two-node edge case: List = [target(5), last(9)]. Step 1: target.val = 9. Step 2: target.next = None. Result: [9] — a one-node list. Correct.

⚠️ What About the Orphaned Node?

After Step 2, the original node.next object (the old "B" node with value w) is no longer reachable from the list. In Python it will be garbage-collected. If any external code held a reference to that node object directly, it now points to an orphaned node — this is by design and the problem guarantees it won't happen in the test cases.

4 The Code (Annotated) ▾

Solution 1 — Copy Next Value (Optimal · Interview Pick, O(1) time & space)

class Solution:
def deleteNode(self, node: ListNode) -> None:← in-place; return type is None (modifies the node object directly)
node.val = node.next.val ← Line 1: overwrite this node's value with successor's value
node.next = node.next.next ← Line 2: skip the now-duplicate successor out of the chain
💡 That's the Entire Solution — Two Lines

Time O(1) — two assignments with no loops. Space O(1) — no extra memory. The simplicity is deceptive: arriving at this independently requires a shift in how you think about "deletion" in a linked list.

Alternative: Save reference explicitly (same logic, more readable)

def deleteNode(self, node: ListNode) -> None:
successor = node.next ← save reference to successor node before any changes
node.val = successor.val ← steal successor's value
node.next = successor.next ← skip successor; it is now unreferenced and will be GC'd
⚠️ Order Matters

You MUST copy the value before updating node.next. If you reversed the order — setting node.next = node.next.next first — you would lose the reference to the original successor and could no longer access node.next.val. Always: copy value → relink pointer.

🚫 Common Wrong Answer: node = None

Writing node = None only rebinds the local variable node in Python. It does NOT modify the list at all — the previous node's .next still points to the original node object. The list is completely unchanged. This is a fundamental Python gotcha: you cannot "delete" an object by rebinding a reference to it.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Read the Function Signature Carefully

The function takes node — not head. Say: "I notice this function only receives the node to delete, not the head of the list. That means I have no way to find the previous node by traversal. Standard deletion is impossible here." This immediately demonstrates careful problem analysis.

Step 2: Reason About What You CAN Reach

"From the given node, I can reach: the node itself (read/write its val and next), and everything after it — node.next, node.next.next, etc. I cannot reach anything before it. So any solution must only use forward references."

Step 3: Reframe the Goal

"I can't delete THIS node. But what if I make this node become its successor? If I copy the next node's value here, then delete the NEXT node (for which I AM the predecessor), the effect is the same. I can delete node.next because node is its predecessor."

Step 4: Verify Constraints Enable the Trick

"The problem guarantees the given node is never the tail, so node.next always exists. Two assignments: node.val = node.next.val, then node.next = node.next.next. O(1) time, O(1) space."

🗣️ What to Say to the Interviewer

"I only have the node to delete — no head, no previous node. So I can't use the
standard prev.next = node.next approach. But I can reach node.next from here.
Instead of deleting THIS node, I'll make it impersonate node.next:
Line 1: node.val = node.next.val — copy successor's value into current node
Line 2: node.next = node.next.next — unlink the now-duplicate successor
The problem guarantees node is not the tail, so node.next is always valid.
Time O(1), Space O(1). The list content is correct after both assignments."
🔮 Likely Follow-up Questions
  • "What if the node IS the tail?" → The trick fails — node.next is None. You'd need the head and a full traversal to find the predecessor, then set prev.next = None.
  • "What if someone holds a Python reference to node.next?" → After Step 2, that reference is to an orphaned node not in the list. It's valid Python but the object is logically deleted.
  • "Delete the Kth node from the end?" → Use two pointers offset by K: advance fast by K, then move both until fast.next is None, then delete slow.next (#19).
  • "Remove all nodes with a given value?" → Dummy head + standard traversal (#203).

7 Summary & Key Takeaways ▾
When prev is unreachable, make node impersonate its successor instead of removing itself
Two O(1) operations: copy successor's value, then unlink the successor
Works only for non-tail nodes — successor must exist (guaranteed by the problem)
node = None is wrong in Python — it rebinds only a local variable, not the list
Linked list "identity" is defined by structure and pointers, not which physical object holds a value

  • Guarantee that node is never the tail → safe to access node.next unconditionally
