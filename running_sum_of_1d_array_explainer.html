LeetCode #1480

Running Sum of 1d Array

Given an array nums, return a running sum where runningSum[i] = sum(nums[0] … nums[i]).

Easy Array Manipulation Prefix Sum Time: O(n) Space: O(1)*
1 Understanding the Problem ▾

Given an array of numbers, produce a new array of the same length where each element is the cumulative total of all elements up to and including that index. Think of it like a running bank balance: after each transaction, you record the new total — not just the transaction amount.

💡 Key Insight

The running sum at index i equals the running sum at index i−1 plus the current element: result[i] = result[i−1] + nums[i]. You only ever need the previous total — there is no need to re-sum from the beginning each time.

❌ Naive — recompute from scratch

result[0] = 1
result[1] = 1 + 2 = 3
result[2] = 1 + 2 + 3 = 6
result[3] = 1 + 2 + 3 + 4 = 10
→ O(n²) total additions

✓ Optimal — carry the running total

result[0] = 1
result[1] = prev(1) + 2 = 3
result[2] = prev(3) + 3 = 6
result[3] = prev(6) + 4 = 10
→ O(n) total additions

Edge cases: A single-element array returns itself unchanged. An empty array returns an empty array. Negative numbers work identically — the running sum decreases when a negative element is added. Mixed positive and negative values are all handled by the same formula.

2 The Algorithm: In-Place Prefix Accumulation ▾

The cleanest approach is in-place prefix accumulation: overwrite each element of nums with the running total so far. By scanning left to right, each element we need for the next computation has already been updated to its running sum — so we can safely overwrite in place without any extra array.

1 Skip index 0 — it's already correct. nums[0] is the running sum of a single-element prefix: itself. Starting the loop at index 1 means we never touch it.
2 For each index i from 1 to n−1: execute nums[i] += nums[i-1]. At this point nums[i-1] already holds the running sum up to i−1 (by induction), so adding it to nums[i] gives the correct running sum up to i.
3 Return nums. After one pass the array has been transformed in place — every element holds its correct prefix sum. No extra data structure is needed.
🔑 Why This Is the "Prefix Sum" Pattern

A prefix of an array is any slice starting from index 0. The prefix sum array stores the cumulative total at every position. Once built in O(n), it enables O(1) range sum queries: sum(l, r) = prefix[r] − prefix[l−1]. This problem is literally asking you to build that prefix sum array — the foundational building block of a hugely powerful technique.

⚠️ Alternative: Separate Output Array

If the problem says "do not modify the input," allocate a new result array and write result[i] = result[i−1] + nums[i] (with result[0] = nums[0]). Same O(n) time, but O(n) extra space. In an interview always ask: "Can I modify the input?" before choosing the approach.

3 Why It Works: The Invariant ▾
🔐 The Invariant

After processing index i, every element nums[j] for j ≤ i holds the sum of the original elements from index 0 through j. In other words: at every step, all elements to the left of (and including) the current index have been correctly converted to their running sums.

Why in-place overwriting is safe (left-to-right only): When we compute nums[i] += nums[i−1], we depend on nums[i−1]. By the invariant, that cell already holds the correct prefix sum up to index i−1 — exactly what we need. We only look backward, never forward. Overwriting left-to-right never corrupts any value we still need.

Proof by induction:

  • Base case (i = 0): nums[0] is untouched. It equals sum(original[0..0]) = original[0]. ✓
  • Inductive step: Assume nums[i−1] = sum(original[0..i−1]). Then nums[i] += nums[i−1] = original[i] + sum(original[0..i−1]) = sum(original[0..i]). ✓
🚫 Right-to-Left Would Break Everything

If you iterated from right to left (i = n−1 down to 1), nums[i−1] at the time of computation would still hold its original value — not the running sum. You'd only get a one-element look-back, not the cumulative total from index 0. Always left-to-right for prefix accumulation.

4 The Code (Annotated) ▾

Solution 1 — In-Place Prefix Sum (Optimal · Interview Pick)

def runningSum(nums: list[int]) -> list[int]:
for i in range(1, len(nums)): ← Start at 1; index 0 is already correct
nums[i] += nums[i - 1] ← nums[i-1] already holds running sum up to i-1
return nums ← nums is now the prefix-sum array
💡 Complexity

Time: O(n) — one left-to-right pass, one addition per element.
Space: O(1) extra — we overwrite the input array. The output IS the modified input. (If the caller relies on the original nums not being mutated, see Solution 2.)

Solution 2 — Explicit Output Array (Input Preserved)

def runningSum(nums: list[int]) -> list[int]:
result = [0] * len(nums) ← Allocate output; original nums untouched
result[0] = nums[0] ← Base case: prefix sum of one element
for i in range(1, len(nums)): ← Build from index 1 forward
result[i] = result[i-1] + nums[i] ← Previous running sum + current value
return result

Solution 3 — Python One-liner with itertools.accumulate

from itertools import accumulate
def runningSum(nums: list[int]) -> list[int]:
return list(accumulate(nums)) ← Stdlib lazy generator; same O(n) prefix logic
⚠️ Which Solution to Present in an Interview

Always start with the in-place approach — it demonstrates space awareness. Then mention: "If the caller needs the original array unchanged, I'd use a separate result array." The accumulate one-liner is a nice bonus but implement the manual version first to show you understand the invariant.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Pattern

Keywords: "running," "cumulative," "sum of all elements up to index i." Any time you need the sum of a prefix of an array, think Prefix Sum. This problem literally asks you to build the prefix sum array — the most basic form of the pattern.

Step 2: Observe the One-Directional Dependency

result[i] depends only on result[i−1] and nums[i]. No look-ahead. This means a single left-to-right scan is sufficient, and the in-place overwrite is safe because we only ever look backward.

Step 3: Clarify the Mutation Constraint

Ask: "Can I modify the input array?" This determines O(1) vs O(n) extra space. Showing you think about this constraint signals engineering maturity to the interviewer.

Step 4: Name the Broader Technique

Volunteer: "This prefix sum array is a building block — once built, I can answer any range sum query in O(1): sum(l, r) = prefix[r] − prefix[l−1]." This shows you understand why the pattern matters beyond this single problem and opens the door to harder follow-ups.

🗣️ What to Say to the Interviewer

"This is a prefix sum problem — each output element is the cumulative sum up to that index.
Brute force: recompute each sum from index 0 each time — O(n²). But I can do better.
Key observation: result[i] = result[i−1] + nums[i]. One-step dependency, left to right.
Can I modify the input? If yes: loop from index 1, do nums[i] += nums[i−1], return nums.
One pass, O(n) time, O(1) extra space. Let me code it up."
🔮 Likely Follow-up Questions

  • "Count subarrays summing to k?" → Prefix sum + hash map to track seen sums (#560)
