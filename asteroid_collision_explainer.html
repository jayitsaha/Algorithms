LeetCode #735

Asteroid Collision

Given a row of asteroids, simulate all collisions: the smaller explodes, same size both explode, same direction never collide.

Medium Stacks Stack Simulation Time: O(n) Space: O(n)
1 Understanding the Problem ▾

You are given an array asteroids of integers. The absolute value is each asteroid's size; the sign is its direction: positive (+) moves right, negative (−) moves left. They all travel at the same speed.

Asteroids can collide only when a rightward one is followed by (i.e., to the left of) a leftward one — they are approaching each other. When they collide, the smaller explodes; if equal, both explode. Asteroids moving the same direction never meet. Return the state after all collisions.

💡 Key Insight

A collision only happens between a positive (right-mover) and a negative (left-mover) when the positive is to the left of the negative. Two positives diverge; two negatives diverge; a negative then a positive diverge (they move apart, not toward each other).

✓ Collision: [5, 10, -5]

5→ and 10→ both travel right.
-5← meets 10→: |10| > |-5| → -5 dies.
Result: [5, 10]

✗ No collision: [-2, -1, 1, 2]

-2← and -1← both go left (diverge).
1→ and 2→ both go right (diverge).
Result: [-2, -1, 1, 2] unchanged

Collision rules (when negative meets positive top):

  • Current wins — |current| > |top|: pop top, current keeps fighting next element
  • Top wins — |top| > |current|: current dies, stop fighting
  • Tie — |current| == |top|: both die (pop top, discard current)
  • No collision — stack empty, or top is also negative: push current

Edge cases: all right-movers (result = input), all left-movers (result = input), a giant left-mover wipes the entire stack, equal-size mutual destruction, a negative asteroid with an empty stack (just pushes).

2 The Algorithm: Stack Simulation with Collision Rules ▾

Use a stack as a buffer of "still-alive, not-yet-cancelled" asteroids. As we scan left-to-right, right-movers are always safe to push — they can only be threatened by future left-movers. When a left-mover arrives, we resolve its fight against the stack top in a loop.

1 Initialize an empty stack. It holds surviving asteroids.
2 For each asteroid a, set alive = True and enter the collision loop.
3 While alive AND a < 0 AND stack is non-empty AND stack[-1] > 0: a collision is occurring.
4 Resolve: if top is smaller → pop (top explodes, current continues). If top equals -a → pop and set alive=False (both die). If top is larger → set alive=False (current dies, top stays).
5 After the loop: if alive is still True, push a onto the stack.
6 Return the stack — it holds all survivors in their original relative order.
Decision Tree: What happens at each asteroid?
Case 1 — Push Directly
Current is positive — OR — current is negative but stack is empty or top is also negative. No collision → push.
Case 2 — Current Wins
Current (negative) vs top (positive): |current| > |top|. Top explodes — pop and loop back. Current may fight the new top.
Case 3 — Mutual Destruction
|current| == |top|. Both explode — pop top, mark alive=False. Exit the collision loop.
Case 4 — Top Wins
|top| > |current|. Current dies — mark alive=False. Top stays on stack. Exit loop.
3 Why It Works: The Invariant ▾
🔐 The Stack Invariant

At every step, the stack contains only asteroids that cannot currently collide with each other. Any asteroid already on the stack either: (a) is moving right and hasn't yet met a left-mover, or (b) is moving left and has already passed all right-movers.

Why a while loop, not an if statement? A single left-mover can destroy multiple consecutive right-movers on the stack before itself dying — a chain reaction. Example: [1, 2, 3, -100]. The -100 first fights 3 and wins, then fights 2 and wins, then fights 1 and wins, then has an empty stack — and it pushes. Each asteroid is pushed at most once and popped at most once, giving O(n) total.

Why not a queue or two-pointer? The key operation is accessing the most recently added right-mover. That is exactly what a stack top provides in O(1). Any other approach needs O(n) to find the relevant asteroid.

⚠️ Common Mistake: Using a=0 as the "dead" flag

Some implementations use a = 0 to mark the asteroid as dead, then check if a != 0 before pushing. This works, but breaks if 0 is ever a valid asteroid (problem constraints say it isn't, but the alive boolean flag is more explicit and readable).

Amortized analysis: There is a while inside a for, which looks like O(n²). But each element is pushed exactly once and popped at most once — total work is at most 2n operations, so O(n) amortized.

4 The Code (Annotated) ▾
def asteroidCollision(asteroids):
stack = []← survivors buffer; holds asteroids still in-flight
for a in asteroids:← process left-to-right in original order
alive = True← assume this asteroid survives until proven otherwise
while alive and a < 0 and stack and stack[-1] > 0:← all 4 conditions must hold for a collision
if stack[-1] < -a:← top's magnitude smaller than current's → top explodes
stack.pop()← destroy top; current survives and may fight the next top
elif stack[-1] == -a:← equal magnitudes → mutual destruction
stack.pop()← remove top from stack…
alive = False← …and mark current as dead too
else:← top's magnitude larger → current explodes
alive = False← current dies; top stays unchanged on the stack
if alive:← only push if this asteroid survived all collisions
stack.append(a)← add to survivors (could be positive or a negative that won)
return stack← all surviving asteroids in original relative order
⚠️ Why -a instead of abs(a)?

Inside the while loop we know a < 0, so -a gives the positive magnitude. Writing stack[-1] < -a cleanly compares two positive numbers. Using abs(a) works identically but adds a function call. Both are correct — understand that -a == abs(a) only when a < 0.

💡 Brute Force Comparison

A brute force O(n²) approach would repeatedly scan backwards from each left-mover to find the nearest right-mover. The stack eliminates this scan: the nearest right-mover is always at stack[-1] in O(1).

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Stack Signal

When a new element's fate depends on the most recently seen element of a specific type, that is the stack pattern. Here, a left-mover only fights the nearest right-mover — the most recent one added to a stack. Any time you hear "the last unsettled element interacts with the new arrival," think stack.

Step 2: Define What the Stack Holds

Ask: "What do I need to remember about past elements for future decisions?" Answer: right-movers that haven't been cancelled yet. A stack of right-movers gives O(1) access to the nearest one. Left-movers that survive (because the stack is empty when they arrive) also go on the stack — they can't collide with anything to their left.

Step 3: Enumerate Cases Before Coding

Explicitly list every branch: (1) positive → push, (2) negative + empty stack → push, (3) negative + top is negative → push, (4) negative + top positive + top smaller → pop loop, (5) tie → pop both, (6) negative + top positive + top larger → discard. Code these cases directly as if/elif/else.

Step 4: Choose a While Loop for the Chain

One left-mover can destroy many right-movers sequentially. An if only resolves one fight. A while keeps fighting until the current asteroid dies or the stack is exhausted — this is the key structural decision that separates a correct solution from a broken one.

🗣️ What to Say to the Interviewer

"The pattern here is 'most recent pending element interacts with new arrival' — that's a stack.
I'll maintain a stack of surviving asteroids. A collision only happens when the current is negative and the stack top is positive. I enumerate four cases: push, current wins (pop + loop), tie (pop both), top wins (discard current).
The while loop handles chain reactions where one left-mover takes out multiple right-movers.
Time: O(n) amortized since each asteroid is pushed and popped at most once. Space: O(n)."
🎯 Follow-up Questions to Expect
  • Can you do it in O(1) space? → No: output can be O(n) in the worst case.
  • What if all asteroids go right? → Stack never triggers collision; O(n) all pushes.
  • What if you have 0-valued asteroids? → Need an explicit boolean flag (not the a=0 trick).
  • How does this compare to Valid Parentheses? → Same "match last unsettled" structure.

7 Summary & Key Takeaways ▾
Collision only between a negative (left-mover) and a positive (right-mover) at the stack top
The stack holds right-movers that haven't been cancelled yet (plus any surviving left-movers)
One left-mover can chain-destroy multiple stack entries — a while loop, not an if
Use -a for current's magnitude (since a < 0, -a is positive)
Time is O(n) amortized: each asteroid is pushed and popped at most once total
Space is O(n): in worst case all asteroids are same-direction, entire array on stack
