LeetCode #688

Knight Probability in Chessboard

On an n×n chessboard, a knight starts at (row, col) and makes exactly k moves. Return the probability it stays on the board.

Medium Dynamic Programming Probability DP Time: O(k·n²) Space: O(n²)
1 Understanding the Problem ▾

A chess knight moves in an L-shape: 2 squares in one direction, then 1 square perpendicular (or vice versa). From any square, up to 8 possible destinations exist. If the knight would land outside the n×n board, that path ends — the knight "falls off".

The question: "After exactly k random moves, what is the probability the knight is still on the board?" At each step, the knight picks one of (up to) 8 moves uniformly at random — probability 1/8 each.

💡 Key Insight

Instead of counting individual paths (there are 8^k of them!), we track a probability distribution over the board. We start with probability 1.0 at the start cell, then spread that probability across valid knight destinations — dividing by 8 at each step. After k steps, summing all remaining probability gives the answer, because any probability that "fell off" is gone forever.

2 The Algorithm: Bottom-Up Probability DP ▾

We maintain an n×n grid dp where dp[r][c] = probability the knight is at (r, c) after the most recent move step. We iterate k times, each time computing a fresh next_dp by spreading probability forward.

1 Base case: Create an n×n grid of zeros. Set dp[row][col] = 1.0 — before any moves, the knight is certainly at the start position.
2 Iterate k times: Create a fresh next_dp (all zeros) before processing each move step.
3 Spread probability: For each cell (r, c) where dp[r][c] > 0, try all 8 knight moves. For each valid in-bounds destination (nr, nc), add dp[r][c] / 8 to next_dp[nr][nc]. Probability going off-board vanishes.
4 Swap: Replace dp = next_dp. The old grid is discarded. Each step only needs the previous step's values.
5 Sum and return: After k steps, sum(sum(row) for row in dp) gives the total probability the knight is still on the board.
📐 Recurrence Relation
next_dp[nr][nc] += dp[r][c] / 8
  for each (r, c) where dp[r][c] > 0
  for each (dr, dc) in the 8 knight deltas
  where 0 ≤ nr < n and 0 ≤ nc < n

Answer = Σ dp[r][c] over all (r, c)
🔑 Probability Conservation Principle

Before any moves, total probability = 1.0. After each step, some fraction "falls off" the board and is irrecoverably lost. The remaining total after k steps is exactly the probability we want — that the knight stayed on the board for all k moves.

3 Why It Works: The Invariant ▾
🔐 The Invariant

After step m, dp[r][c] = the probability that a random sequence of exactly m knight moves from the starting position ends at (r, c) while staying on the board at every intermediate step (including step m).

Why this is Dynamic Programming:

Optimal substructure: The probability of being at (nr, nc) after step m+1 equals (1/8) × sum of probabilities of being at each of the up-to-8 knight-ancestors of (nr, nc) after step m. Each step depends only on the previous step — no need to recall the full history.

Overlapping subproblems: Many distinct path sequences can reach the same cell at the same step. A naive DFS would recompute the probability for (step=5, r=2, c=3) once per path leading to it — exponentially many times. DP stores it once: O(k · n²) unique subproblems.

⚠️ Why you MUST use two grids (not in-place update)

If you update dp[nr][nc] directly while still scanning dp[r][c], you might later use an already-updated cell as a source. That source's value now includes contributions from the current step — you'd be counting probability that "already moved once this step" as a source for another move this same step. This double-counts paths. Always build next_dp separately, then swap.

k=0 edge case: The loop never executes. dp still holds the base case: 1.0 at start, 0 everywhere else. Sum = 1.0. Correct — with zero moves, the knight is certainly on the board.

Probability decay: Each step multiplies total on-board probability by at most (valid_moves/8) ≤ 1. For corner cells with only 2 valid moves, we lose 6/8 = 75% probability per step at that cell. On large boards, corners are rare so decay is slower.

4 The Code (Annotated) ▾

Solution 1 — Bottom-Up DP (Tabulation) — Interview Pick

def knightProbability(n, k, row, col):
MOVES = [(-2,-1),(-2,1),(-1,-2),(-1,2), ← All 8 L-shape move deltas
(1,-2),(1,2),(2,-1),(2,1)]
dp = [[0.0]*n for _ in range(n)] ← n×n grid, all zero probability
dp[row][col] = 1.0 ← Start: certainty at (row, col)
for _ in range(k): ← Perform k move steps
ndp = [[0.0]*n for _ in range(n)] ← Fresh grid for this step's output
for r in range(n): ← Scan every source cell
for c in range(n):
if dp[r][c] == 0: continue ← Skip zero-prob cells (optimization)
for dr, dc in MOVES: ← Try all 8 destinations
nr, nc = r+dr, c+dc ← Compute destination cell
if 0<=nr← Bounds check: must stay on board
ndp[nr][nc] += dp[r][c]/8 ← Spread 1/8 of prob to destination
dp = ndp ← Advance state; old dp discarded
return sum(dp[r][c] for r in range(n) ← Sum all remaining on-board prob
for c in range(n))
⚠️ Why ndp, not dp in-place?

Each step, a source cell at (r,c) sends probability to destinations. If we updated dp directly, cell (1,2) might receive probability from (0,0) on line 14, then on line 8 when we reach (1,2) as a source, we'd spread that freshly-received probability again — as if it had already made another move this step. Using a separate ndp ensures sources are always from the previous step only.

💡 Complexity Summary

Bottom-Up (Interview Pick): Time O(k·n²) — k steps × n² cells × 8 moves = O(8kn²) = O(kn²). Space O(n²) — only 2 grids at a time.
Memoization: Same time O(k·n²) states × O(8) work each. Space O(k·n²) for the cache — worse than bottom-up.
Brute DFS (no memo): O(8^k) — exponential, TLE for k>10.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the DP Signal

Key signals: "probability after k steps", "random walk on a grid", "fixed number of moves". These suggest DP where state = (step, position) and we track a probability distribution, not a single path.

Step 2: Propose Brute Force and Explain Its Failure

"Naive approach: DFS/recursion — at each cell try all 8 moves, recurse for k steps. Total paths = 8^k. For k=25, that's 8^25 ≈ 10^22 paths. Also, the same (step, cell) pair gets recomputed along many different paths — overlapping subproblems. DP is the natural fix."

Step 3: Define State and Recurrence

Say: "Let dp[r][c] = probability of being at (r,c) after the current move count. Base case: dp[row][col] = 1. Transition: next_dp[nr][nc] += dp[r][c]/8 for each valid knight destination. Repeat k times. Answer = sum of all dp values."

Step 4: Justify Two-Grid (Not In-Place)

"I use a separate next_dp to avoid using already-updated values as sources in the same step — that would mix contributions from step m and step m+1."

🗣️ What to Say to the Interviewer

"The key observation: I don't need to enumerate 8^k paths.
Instead I track the probability distribution over board cells after each move.
State: dp[r][c] = probability of being at (r,c) on the current move number.
Base: 1.0 at (row, col). Transition: spread each cell's probability to
valid knight destinations, dividing by 8. Any probability that goes
off-board is permanently lost. After k steps, sum the grid.
Time: O(k·n²), Space: O(n²) using two grids."
🔮 Likely Follow-Up Questions
  • "Can you use O(n²) space?" → Already do! Two grids of size n×n. Just rolling the array.
  • "Expected number of on-board moves before falling off?" → E = Σ_{m=1}^{inf} P(still on board after step m).
  • "What if moves have different probabilities?" → Replace /8 with the specific prob for each direction.
  • "Most likely landing cell after k moves?" → Return argmax of dp instead of sum.
  • "What if knight teleports back when it falls off?" → Accumulate off-board probability and add it back to start at each step.

7 Summary & Key Takeaways ▾
Probability DP: track distribution P(at cell X after step m), not individual paths
Each move: spread dp[r][c] / 8 to valid knight destinations — off-board probability is permanently lost
Total probability remaining after k steps = answer (probability of staying on board)
Two separate grids per step — never update in-place (corrupts source values for this step)
Memoization top-down gives same time complexity but O(k·n²) cache space vs O(n²) for bottom-up
Pattern generalizes: any "probability after fixed steps in random walk on grid" → same dp[r][c] spread approach
ApproachTimeSpaceNotes
Brute-force DFS O(8^k) O(k) Exponential; TLE for k>12
Memoization (top-down) O(k·n²) O(k·n²) Natural to derive; cache overhead
Bottom-Up DP ✓ Interview Pick O(k·n²) O(n²) Optimal space; no recursion
🎯 When to Recognize Probability DP
  • "Probability after k steps/moves/rounds" in a random process → Probability DP
  • State space is 2D grid or linear, steps are fixed → dp[step][position]
  • Each state distributes probability to its neighbors uniformly → divide by degree
  • Sum over all terminal states (vs. optimize) → sum the final dp grid
