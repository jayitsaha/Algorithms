LeetCode #1203

Sort Items by Groups Respecting Dependencies

Given n items each in a group, sort them so group members appear consecutively and all before/after dependencies between items are respected.

Hard Graph Two-Level Topo Sort Time: O(n+m+E) Space: O(n+m+E)
1 Understanding the Problem ▾

You have n items numbered 0..n-1. Each item belongs to exactly one of m groups (or is ungrouped when group[i] == -1). You are given a list beforeItems[i] = items that must appear before item i in any valid output ordering.

Two constraints must hold simultaneously:

  • Dependency constraint: For every pair (b, i) in beforeItems, b must appear before i.
  • Group-contiguity constraint: All items belonging to the same group must appear consecutively in the output. No interleaving of groups is allowed.

Return any valid ordering, or an empty list [] if no valid ordering exists (i.e., a cycle is detected).

💡 Key Insight

The group-contiguity rule implies that if item B (in group X) must precede item A (in group Y), then the entire group X must appear before the entire group Y. So there is a hidden dependency between groups, derived from cross-group item dependencies. We need to sort both the items and the groups — hence: Two-Level Topological Sort.

2 The Algorithm: Two-Level Topological Sort (Kahn's BFS) ▾

We apply Kahn's BFS-based topological sort at two levels: items and groups. The key idea is that both levels form independent DAGs that we can sort separately, then merge.

1 Assign synthetic group IDs: For each item with group[i] == -1, assign a unique new group ID starting at m. This ensures every item belongs to exactly one group, simplifying subsequent logic.
2 Build two graphs simultaneously: For each dependency b → i: add an item-graph edge and increment item_indeg[i]. If group[b] != group[i], also add a group-graph edge and increment grp_indeg[group[i]]. (Skip if groups are equal — same group, no cross-group edge.)
3 Kahn's topo-sort on items: Initialize a queue with all items of in-degree 0. Process them BFS-style: emit each item, decrement its successors' in-degrees, enqueue those hitting 0. If final order length < n, there's a cycle — return [].
4 Kahn's topo-sort on groups: Same algorithm, applied to the group-level DAG. If result length < total number of groups (real + synthetic), there's a group-level cycle — return [].
5 Bucket and emit: Build buckets[group] = items of that group, in the order they appear in item_topo_order. Then for each group in group_topo_order, extend the result with that group's bucket. This emits groups as contiguous blocks, in the correct dependency order.
🔑 Why Run Two Separate Sorts?

Item topo-sort alone doesn't guarantee groups are contiguous (it could interleave items from different groups). Group topo-sort alone doesn't say how to order items within a group. Only by running both and combining them do you satisfy all constraints at once.

3 Why It Works: The Invariants ▾
🔐 Kahn's Algorithm: Core Invariant

At every step of Kahn's BFS, a node is emitted only when its in-degree reaches 0 — meaning every one of its prerequisite nodes has already been emitted. This guarantees that the emitted sequence is a valid topological order of the DAG. If any cycle exists, at least one node in the cycle will never reach in-degree 0 and won't be included, giving us the cycle-detection check for free.

🔐 Item-Level Invariant

After item topo-sort, for every dependency edge b → i, item b appears before item i in item_order. Therefore, when we populate each group's bucket by iterating item_order in sequence, items within each bucket are already in a valid dependency order.

🔐 Group-Level Invariant

After group topo-sort, for every cross-group dependency (item in group A must precede item in group B), group A appears before group B in grp_order. Therefore, when we emit all of group A's bucket before any of group B's bucket, every cross-group item dependency is satisfied.

Proof of Correctness (Combine Step): The final result is built by iterating groups in group-topo-order and emitting items within each group in item-topo-order. Consider any dependency b → i:

  • If b and i are in the same group: item topo-order guarantees b appears before i within that group's bucket. ✓
  • If b and i are in different groups: group topo-order guarantees group[b] appears before group[i], so all of group[b]'s items (including b) are emitted before any of group[i]'s items (including i). ✓
⚠️ Pitfall: Duplicate Group Edges Inflate In-Degree

If two items in group A both have dependencies on items in group B, a naive implementation adds the group edge A→B twice and increments grp_indeg[B] twice. Kahn's then expects to decrement it twice, but the edge is in the graph only once — so B may never reach in-degree 0, causing a false "cycle" report. Fix: track which group pairs you've already added an edge for and skip duplicates.

⚠️ Pitfall: Wrong In-Degree Array Size

The group in-degree array must have size gid (real groups + synthetic groups), not just m. Using size m causes index-out-of-bounds for synthetic group IDs ≥ m.

4 The Code (Annotated) ▾
from collections import defaultdict, deque
def sortItems(n, m, group, beforeItems):
# ── Step 1: Assign synthetic group IDs ──────────────────────────
gid = m ← first synthetic ID; real groups are 0..m-1
for i in range(n):
if group[i] == -1:
group[i] = gid ← give ungrouped item its own private group
gid += 1
# ── Step 2: Build item graph and group graph ─────────────────────
item_graph = defaultdict(list) ← item adjacency list: before → item
item_indeg = [0] * n ← in-degree for each item
grp_graph = defaultdict(set) ← group adjacency (set avoids dup edges!)
grp_indeg = [0] * gid ← size gid, not m — covers synthetic groups
for i in range(n):
for b in beforeItems[i]: ← b must come before i
item_graph[b].append(i)
item_indeg[i] += 1
if group[b] != group[i]: ← cross-group dep
if group[i] not in grp_graph[group[b]]:← dedup!
grp_graph[group[b]].add(group[i])
grp_indeg[group[i]] += 1
# ── Step 3: Kahn's topo-sort helper ─────────────────────────────
def topo(nodes, graph, indeg):
q = deque(v for v in nodes if indeg[v] == 0)← seed with 0-indegree nodes
order = []
while q:
v = q.popleft() ← emit: all prereqs already done
order.append(v)
for nb in graph[v]: ← "complete" v; unblock successors
indeg[nb] -= 1
if indeg[nb] == 0:
q.append(nb)
return order
# ── Step 4: Run topo-sort on items, then groups ──────────────────
item_order = topo(range(n), item_graph, item_indeg)
if len(item_order) != n: ← item-level cycle detected
return []
grp_order = topo(range(gid), grp_graph, grp_indeg)
if len(grp_order) != gid: ← group-level cycle detected
return []
# ── Step 5: Bucket items by group (preserving item_order) ────────
buckets = defaultdict(list)
for item in item_order: ← iterate in dep-sorted order
buckets[group[item]].append(item)← bucket preserves item dep order
# ── Step 6: Emit groups in group-topo order ──────────────────────
result = []
for g in grp_order:
result.extend(buckets[g]) ← all of group g's items, dep-sorted, contiguously
return result
⚠️ The Deduplication Fix Is Critical

Using defaultdict(set) for grp_graph and checking if group[i] not in grp_graph[group[b]] before adding ensures each group edge is counted in grp_indeg exactly once. Without this, multiple items in the same group creating the same cross-group dependency will over-count in-degrees, making it appear there's a cycle when there isn't.

Time & Space Complexity: Building graphs: O(n + E) where E = total dependencies. Each topo-sort: O(V + E). Overall O(n + m + E) time and O(n + m + E) space. This is optimal since we must at minimum read all dependencies.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify the Core Pattern

Whenever you see "ordering items subject to must-come-before constraints," your first thought should be topological sort. The words "before," "dependency," "prerequisite," or "order respecting constraints" are the trigger. This is standard Kahn's BFS or DFS topo sort.

Step 2: Spot the Two-Level Structure

The added group-contiguity constraint tells you there's a second DAG hiding in the problem. If item B (group X) precedes item A (group Y), then everything in group X must precede everything in group Y. Draw this out — you'll see a group dependency graph emerging. Recognize: two DAGs = two topo sorts.

Step 3: Handle the Ungrouped Edge Case Early

When you spot group[i] == -1, convert it immediately to a synthetic group ID. This removes special-case handling from the rest of the algorithm — ungrouped items just become singleton groups, and everything else works uniformly.

Step 4: Avoid the Duplicate-Edge Trap

Proactively mention deduplication. Interviewers often use this to distinguish candidates who think about correctness carefully. Use a set (not a list) for group adjacency to automatically deduplicate, and only increment grp_indeg once per unique group-pair.

🗣️ What to Say to the Interviewer

"I see before-dependencies, so this is a topological sort problem. The group constraint adds a second layer: if any item in group A must precede an item in group B, the entire group A must precede group B. So I need TWO topological sorts — one at the item level, one at the group level. I'll use Kahn's BFS for both since it also gives free cycle detection. Step one: assign synthetic IDs to ungrouped items so every item belongs to a group. Step two: build item and group DAGs from the dependency list, deduplicating group edges to avoid false cycle reports. Step three: Kahn's on items → check length == n, Kahn's on groups → check length == gid. Step four: bucket items by group in item-topo order, then emit buckets in group-topo order. Time: O(n + m + E), Space: O(n + m + E) where E is total dependencies."
💡 Follow-Up Questions
  • Can you do it with DFS-based topo sort instead? (Yes — use GRAY/BLACK coloring)
  • What if items within a group also have their own ordering priority? (Add weights; use priority queue in Kahn's)
  • What if we want lexicographically smallest valid order? (Priority queue instead of FIFO queue in Kahn's)
  • What if groups can form hierarchies (groups of groups)? (Generalize to k-level topo sort)

7 Summary & Key Takeaways ▾
Two-Level Topo Sort: Kahn's BFS independently on items AND groups, then merge by bucketing.
Assign synthetic group IDs to ungrouped items so every item uniformly belongs to exactly one group.
Build group edges only for cross-group item dependencies (same-group deps don't affect group ordering).
Deduplicate group edges (use a set) to prevent inflated in-degrees causing false cycle detection.
Cycle detection is free: if topo result length < node count, a cycle exists — return [].
Final merge: bucket items by group (in item-topo order), emit buckets in group-topo order.
