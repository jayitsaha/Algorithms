LeetCode #2610

Apple Redistribution into Boxes

Find the minimum number of boxes needed to hold all apple packs, using the largest boxes first.

Easy Greedy Sort + Fill Largest First Time: O(n + m log m) Space: O(1)
1 Understanding the Problem ▾

You have n packs of apples, where apple[i] is the number of apples in pack i. You also have m boxes, where capacity[j] is the maximum apples box j can hold. Each box can hold multiple packs as long as the total apples doesn't exceed its capacity. The goal is to find the minimum number of boxes needed to hold all the apple packs.

The key freedom: you can choose which boxes to use, and you can arrange packs into boxes however you like. You want to use as few boxes as possible.

💡 Key Insight

To minimize the count of boxes used, always pick the largest available box first. A bigger box packs more apples, so it covers more ground per box "spent." We never need to match specific packs to specific boxes — only the total capacity vs total apple count matters. Sort boxes descending, accumulate capacity, stop when the running total meets or exceeds the total apple count.

✓ Optimal: Largest-First

apple=[1,3,2] → total=6
capacity=[4,3,1,5]
sorted desc: [5,4,3,1]
Box cap=5: cumsum=5 < 6
Box cap=4: cumsum=9 ≥ 6 ✓
Answer: 2 boxes

✗ Suboptimal: Smallest-First

sorted asc: [1,3,4,5]
Box cap=1: cumsum=1 < 6
Box cap=3: cumsum=4 < 6
Box cap=4: cumsum=8 ≥ 6
Answer: 3 boxes — worse!

Edge Cases: A single giant box that fits everything (answer = 1). All boxes identical capacity (answer = ceil(total / single_cap)). Problem guarantees the total capacity of all boxes is at least total apples, so a valid solution always exists.

2 The Algorithm: Greedy — Sort Capacities Descending ▾

The approach has three clean phases: compute the target (total apples), sort boxes descending, then greedily pick boxes one by one until the running capacity sum covers the target. No complex data structures needed.

1 Compute total apples: total = sum(apple). This is our threshold. We need the combined capacity of chosen boxes to reach or exceed this value. Individual pack assignments don't matter — only the aggregate does.
2 Sort capacity descending: capacity.sort(reverse=True). The biggest boxes come first. Each box we pick should contribute as much capacity as possible — so we always pick the largest remaining box.
3 Walk and accumulate: Iterate through sorted capacities, keeping a running cumsum. The moment cumsum >= total, return i + 1 (one-indexed count of boxes used). We stop immediately — there is no benefit to selecting more boxes.
🔑 Why the apple array doesn't need sorting

Intuitively, you might think we need to sort both arrays and do a one-to-one assignment — matching each apple pack to the best-fitting box. But the problem lets us put multiple packs in one box. This means packing is always feasible as long as total capacity is sufficient. We reduced a packing problem to a simple sum comparison.

🎯 Goal: Find the shortest prefix of the descending-sorted capacity array whose sum ≥ total apples.
📦 Greedy choice: At each step, select the largest remaining box. It contributes the most capacity per "slot" used, minimizing the total boxes needed.
✋ Early stop: The moment the cumulative capacity meets the target, we have our answer. All remaining (smaller) boxes are irrelevant.
3 Why It Works: The Greedy Invariant ▾
🔐 The Invariant

At every step of the iteration, the set of boxes chosen so far achieves the maximum possible total capacity for that number of boxes. Equivalently: among all possible selections of k boxes from the given set, our greedy selection of the top-k largest boxes provides the greatest combined capacity.

Exchange Argument (why greedy is optimal): Suppose there exists a better solution S that uses fewer boxes than our greedy solution G. Let G use k boxes and S use fewer than k. Since S has fewer boxes and covers total apples, S's total capacity ≥ total. But our greedy G selects the k largest boxes, which maximizes total capacity among all k-box sets. If even a k-box set (G) needs k boxes, then S uses fewer than k boxes — but each of those boxes must be from our available set, and the best possible subset of that size is the top-k, which was our selection. Contradiction.

Why pack-level assignment doesn't change anything: Once we've confirmed that total box capacity ≥ total apples, we can always distribute packs. Think of it like water filling containers: as long as total container volume ≥ total water, water can always be distributed (no individual container overflow since we can choose how many packs go in each box).

⚠️ Common Mistake: Sorting Apples Too

Some approaches sort both apple packs (descending) and boxes (descending), then try to match the biggest pack to the biggest box one-to-one. This treats the problem like a bin-packing or assignment problem, which is much harder. But here, you can put multiple packs in one box — so the feasibility check is just a sum comparison. No matching needed.

💡 Analogy: Filling Containers with Sand

Imagine you have bags of sand (apple packs) and boxes of different sizes. You need to fit all the sand using the fewest boxes. You don't need to keep each bag intact inside a specific box — you could theoretically pour sand from one bag into multiple boxes. So the only question is: do my chosen boxes have enough total volume? Use the biggest containers first.

4 The Code (Annotated) ▾

Solution 1 — Greedy Sort (Optimal, Interview Pick)

def minimumBoxes(apple: list[int], capacity: list[int]) -> int:
total = sum(apple) ← Total apples we must fit; this is our threshold
capacity.sort(reverse=True) ← Largest box first — maximizes capacity per box used
cumsum = 0 ← Running total capacity of boxes selected so far
for i, cap in enumerate(capacity): ← Walk sorted capacities from largest to smallest
cumsum += cap ← Commit this box; add its capacity to running total
if cumsum >= total: ← Do we have enough capacity now?
return i + 1 ← Yes! i is 0-indexed, so i+1 gives the box count
return len(capacity) ← All boxes needed (problem guarantees this suffices)
⚠️ Off-by-One: Why i + 1?

enumerate gives 0-based indices. When we successfully cover all apples at index i, we've used boxes 0 through i inclusive — that's i + 1 boxes. Returning i would always be one short. A classic fencepost error to watch for in any "count items up to index" situation.

Solution 2 — Brute Force (for conceptual comparison)

from itertools import combinations
def minimumBoxes_brute(apple, capacity):
total = sum(apple) ← Same target threshold
m = len(capacity)
for k in range(1, m + 1): ← Try every possible number of boxes k
for subset in combinations(capacity, k): ← All C(m,k) subsets of size k
if sum(subset) >= total: ← Does this subset have enough total capacity?
return k ← First k where a valid subset exists
💡 Complexity Summary

Greedy Sort: Time O(n + m log m) where n=len(apple), m=len(capacity). Space O(1) (in-place sort).
Brute Force: Time O(2m) — exponential, only for tiny toy examples.
Key insight: Sorting converts an exponential combinatorial search into a single O(m) linear scan.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify What We're Minimizing

"We want to minimize the count of boxes, not the wasted space or the cost. To minimize count, I should maximize what each chosen box contributes — so I should use the biggest boxes first."

Step 2: Simplify the Packing Constraint

"Do I need to assign specific packs to specific boxes? No — I can pack them however I want. So the only feasibility check is: does the combined capacity of chosen boxes cover the total apple count? This reduces to a sum comparison."

Step 3: Recognize the Greedy Pattern

"Whenever we're minimizing a count of items needed to cover a total, and items can be ranked by their contribution, sorting and greedily selecting is usually optimal. The exchange argument confirms: swapping in a smaller box can only hurt us."

Step 4: Translate to Code and Verify

"Sort capacity descending. Compute total = sum(apple). Walk sorted capacities, accumulate cumsum, return i+1 when cumsum ≥ total. Time O(n + m log m), Space O(1). Verify on the example: apple=[1,3,2]→total=6, capacity sorted=[5,4,3,1], after boxes 5 and 4: cumsum=9≥6 → return 2. Correct."

🔮 Likely Follow-up Questions
  • "What if each box can hold at most one apple pack?" → Now it's an assignment problem. Sort both; use two pointers to match each pack to the smallest sufficient box (#455 Assign Cookies).
  • "Minimize total wasted space instead of box count?" → After greedy selection, wasted = sum(chosen capacities) − total_apples. But which boxes minimize waste given same count? A variant problem.
  • "What if boxes have costs and you want to minimize cost?" → Greedy by capacity/cost ratio (fractional knapsack). Or DP if costs are discrete.
  • "What's the minimum total apples that can go unboxed?" → 0, by definition — the problem guarantees all fit.

7 Summary & Key Takeaways ▾
To minimize box count, use the largest boxes first — greedy by capacity descending
The key simplification: feasibility depends on total capacity vs total apples, not individual pack-to-box assignments
Sort capacity descending, keep a running sum, return index+1 at first sufficiency
The fallback return len(capacity) handles when all boxes are needed (guaranteed sufficient by problem)
Exchange argument proves greedy optimality: top-k largest boxes always maximize combined capacity among all k-box subsets

🎯 When to Recognize "Greedy — Sort + Fill Largest First"
  • "Minimum number of containers to hold a given total" → sort containers descending, accumulate
  • Objective is the count of items selected, not their assignment or arrangement
  • Feasibility is a pure sum comparison (no position or order constraints)
  • Items can be ranked by their "contribution" (capacity, value, coverage, etc.)
