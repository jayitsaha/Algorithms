LeetCode #931

Minimum Falling Path Sum

Given an n×n integer matrix, return the minimum sum of any falling path — from any cell in the top row, stepping down-left, down, or down-right each row.

Medium Dynamic Programming DP from Previous Row Time: O(n²) Space: O(1)
1 Understanding the Problem ▾

You are given an n × n matrix of integers. A falling path starts at any element in row 0 and moves down one row at a time. From position (r, c) you may step to (r+1, c-1), (r+1, c), or (r+1, c+1) — but never out of bounds. The path visits exactly one cell per row and ends in the last row. Find the path whose elements sum to the minimum.

💡 Key Insight

You cannot solve this greedily — a locally small number now might lead to a globally expensive path below. The optimal substructure is: the cheapest path to (r, c) equals matrix[r][c] plus the cheapest path to the best of its up-to-three parents above. This single equation is the entire DP.

Edge Cases: n = 1 (the answer is the single cell value); a matrix with negative numbers (the DP handles negatives correctly); a single column (only one parent per cell); a matrix where all values are equal.

2 The Algorithm: In-place Row-by-Row DP ▾

We process rows from top to bottom. The first row is the base case — no parents exist, so each cell's DP cost equals its value. For every subsequent row, we look up to three parents in the row above, take the minimum, and add the current cell's value. We do this in-place, overwriting the matrix so no extra array is needed.

1 Base case (row 0): No parents. dp[0][c] = matrix[0][c] for all c. The matrix row 0 is already correct — we don't touch it.
2 For each row r = 1 … n−1, each column c: Collect valid parents from row r−1: directly above (r−1, c), upper-left (r−1, c−1) if c > 0, upper-right (r−1, c+1) if c < n−1. Compute best = min(valid parents).
3 Update in-place: matrix[r][c] += best. Now this cell holds the minimum cumulative cost to reach it from any starting cell in row 0.
4 Answer: After processing all rows, return min(matrix[n-1]). Every cell in the last row holds the cheapest cost to arrive there — the global minimum is the answer.
📐 The Recurrence Relation
dp[r][c] = matrix[r][c] + min( dp[r−1][c−1] (if c>0),  dp[r−1][c],  dp[r−1][c+1] (if c<n−1) )
Base: dp[0][c] = matrix[0][c]
Answer: min(dp[n−1][c] for all c)
🔑 Why In-Place Works

When computing row r, we only read row r−1. Row r−1 was already fully computed and will never be read again. Therefore, we can safely overwrite row r while reading row r−1, reducing extra memory to O(1). If mutating the input is unacceptable, keep a 1D prev array — still O(n).

3 Why It Works: Optimal Substructure & Invariant ▾
🔐 The Invariant

After processing row r, every cell matrix[r][c] holds the minimum cumulative sum of any falling path from any starting cell in row 0 down to (r, c). This invariant holds initially (row 0) and is preserved by induction across every row.

Why this is Dynamic Programming — the two pillars:

1. Optimal Substructure. The best path to (r, c) is: (best path to its best parent) + value at (r, c). Any globally optimal path, when you look at the sub-path to any intermediate cell, must itself be optimal for reaching that cell. If it weren't, you could swap in the cheaper sub-path and improve the global answer — contradiction.

2. Overlapping Subproblems. A brute-force recursion from each last-row cell back to row 0 would recompute the same intermediate cells many times — cell (r−1, c) is a valid parent of (r, c−1), (r, c), and (r, c+1). Without memoization we'd do exponential work. DP computes each cell's cost exactly once.

⚠️ Why Greedy Fails Here

Greedy always steps to the smallest neighbour. But consider a matrix where row 0 has values [100, 1], row 1 has [−1000, 5000], and row 2 is anything. Greedy from row 0 picks 1, then must step into 5000. But starting at 100 and going to −1000 gives a far better total. Greedy is irreversibly committed to each step — DP considers all futures simultaneously.

Corner handling: At the left edge (c = 0), there is no upper-left parent — only above and upper-right. At the right edge (c = n−1), there is no upper-right parent. The recurrence simply skips non-existent neighbours. For a 1×1 matrix, the loop body never executes and we return the single element directly.

4 The Code (Annotated) ▾

Solution 1 — In-place Tabulation (Interview Pick)

def minFallingPathSum(matrix):
n = len(matrix) ← n×n matrix; works for any n ≥ 1
for r in range(1, n): ← row 0 is the base case; skip it
for c in range(n): ← every column in current row
best = matrix[r-1][c] ← start with directly-above parent (always valid)
if c > 0: ← upper-left parent exists?
best = min(best, matrix[r-1][c-1]) ← consider upper-left
if c < n-1: ← upper-right parent exists?
best = min(best, matrix[r-1][c+1]) ← consider upper-right
matrix[r][c] += best ← overwrite: self value + best parent cost
return min(matrix[n-1]) ← cheapest end of any path in the last row
⚠️ += Not = best

We write matrix[r][c] += best, not = best. The cell's own value is the cost of visiting that position. We accumulate: total cost to reach (r, c) = (cost to reach best parent) + (cost at (r, c) itself). Forgetting += and writing = best would silently discard the current cell's contribution.

Solution 2 — Top-Down Memoization (easy to derive from recurrence)

from functools import lru_cache
def minFallingPathSum(matrix):
n = len(matrix)
@lru_cache(maxsize=None) ← memoize (r, c) → min cost
def dp(r, c): ← min falling path cost ending at (r, c)
if r == 0: return matrix[0][c] ← base case: top row, cost equals value
parents = [c] ← always include directly-above
if c > 0: parents.append(c-1) ← upper-left if exists
if c < n-1: parents.append(c+1) ← upper-right if exists
return matrix[r][c] + min(dp(r-1, pc) for pc in parents)← recurse upward
return min(dp(n-1, c) for c in range(n)) ← best over all last-row endpoints
💡 Complexity Summary

In-place Tabulation: Time O(n²) — every cell visited once. Space O(1) extra (modifies input).
Memoization: Time O(n²) — same subproblems, each solved once. Space O(n²) for cache + O(n) recursion depth.
Brute-force DFS (no memo): O(3ⁿ) — exponential; each cell fans out to 3 children.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Spot the DP Signal

The problem asks for a minimum sum path with constrained choices at each step. Whenever you see "minimum/maximum over a path with limited moves" in a grid, think DP. Greedy is ruled out because local choices don't guarantee global optimality.

Step 2: Define the State Clearly

"I'll define dp[r][c] = minimum sum of any falling path that ends at cell (r, c)." Say this out loud. Interviewers want to see that you can articulate the state before writing code.

Step 3: Write the Recurrence

"dp[r][c] = matrix[r][c] + min(parents), where parents are the up-to-three cells in row r−1 at columns c−1, c, c+1 (within bounds). Base case: row 0 is just the matrix values."

Step 4: Propose Space Optimization Proactively

"Since row r only reads row r−1, I can modify the matrix in-place — O(1) extra space. If the interviewer objects to mutating input, I'll use a 1D prev-row array — O(n) extra space."

🗣️ What to Say to the Interviewer

"This is a grid DP problem. I'll define dp[r][c] as the cheapest falling path that ends at (r, c).
Base case: row 0 — dp[0][c] = matrix[0][c] for all c.
Transition: dp[r][c] = matrix[r][c] + min of valid parents in row r−1.
I'll compute this in-place, row by row — O(n²) time, O(1) space.
After processing all rows, the answer is min of the last row."
🔮 Likely Follow-ups
  • "Can you reconstruct the actual path?" → Track a parent[r][c] table during fill, then walk back from the min cell in the last row.
  • "What if you cannot mutate the matrix?" → Keep a 1D prev array — O(n) extra space.
  • "What if diagonal jumps are not allowed (down only)?" → Recurrence simplifies to one parent per cell (Triangle problem, #120).
  • "What if you can skip a column (any column, not adjacent)?" → Minimum Falling Path Sum II (#1289) — track top-2 minimum of previous row.

7 Summary & Key Takeaways ▾
Greedy fails — locally cheap choices can force expensive steps later
State: dp[r][c] = min cost to reach (r, c) from any start in row 0
Recurrence: dp[r][c] = matrix[r][c] + min(up to 3 parents in row r−1)
Only the previous row is ever needed — update in-place for O(1) extra space
Answer = min of the entire last row (every cell is a valid path end)
Memoization derives naturally from the recurrence; tabulation is the interview pick
ApproachTimeSpaceNotes
Brute-force DFS O(3ⁿ) O(n) Exponential; impractical for n > 10
Memoization (top-down) O(n²) O(n²) Easy to derive from recurrence; cache overhead
In-place Tabulation ✓ O(n²) O(1) Optimal; row-by-row, modify in-place
🎯 When to Recognize "DP from Previous Row"
  • Grid path problem with downward-only movement (no going back up)
  • Each cell's answer depends on a small, bounded window of the row above
  • Multiple valid starting positions AND/OR ending positions
  • "Minimum/maximum sum/product/cost over any path from top to bottom"
