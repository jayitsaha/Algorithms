LeetCode #42

Trapping Rain Water

Given n non-negative integers representing an elevation map (bar width = 1), compute how much water it can trap after raining.

Hard Two Pointers Opposite Direction Time: O(n) Space: O(1)
1 Understanding the Problem ▾

Imagine a cross-section of terrain: an array of bar heights. After rain, water collects in the "valleys" between taller walls. We want the total volume of trapped water.

The critical physical rule: water above position i can only be as high as the shorter of the tallest walls on its left and right. Any water above that level spills over the shorter wall.

💡 Key Insight

At each column i, the water level = min(max_height_left, max_height_right). If this minimum exceeds height[i], column i traps min(lmax, rmax) - height[i] units. If not, it contributes zero (it is a wall).

✓ Formula per column i

water[i] = max(0,
 min(left_max, right_max)
 - height[i])

Answer = sum(water[i])

Example: [0,1,0,2,1,0,1,3,1,0,1,2]

Answer = 6 units.
idx 2: min(1,3)-0 = 1

idx 5: min(2,3)-0 = 2

idx 9: min(3,2)-0 = 2 (via rmax)

Edge cases: length 0 or 1 → return 0. All increasing → water only at one side (but no valley, so 0). Monotone decreasing → same. Single valley between two walls → formula handles it correctly.

2 The Algorithm: Two Pointers (Opposite Direction) ▾

Start with left at index 0 and right at index n-1. Track the running maximum seen from each side. At each step, process whichever side has the smaller maximum — because that side's max is definitively the bottleneck.

🔐 Core Observation

When left_max <= right_max: we know the actual right-side maximum (even unseen parts) is at least right_max. So min(left_max, actual_right_max) = left_max. We can compute column left's water immediately — no need to look further right.

1 Init: left=0, right=n-1, left_max=0, right_max=0, water=0
2 Loop while left <= right: compare left_max vs right_max
3 If left_max <= right_max: update left_max = max(left_max, height[left]), then water += left_max - height[left], advance left++
4 Else: update right_max = max(right_max, height[right]), then water += right_max - height[right], retreat right--
5 Return water when pointers cross

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every iteration: left_max = max(height[0..left]) and right_max = max(height[right..n-1]). The side with the smaller max is definitively the bottleneck for any column on that side.

Formal proof for the left-side case (left_max <= right_max):

  • We have seen right_max = max(height[right..n-1]).
  • The actual right maximum over ALL of [left+1..n-1] is ≥ right_max (we've already seen right_max).
  • Therefore: min(left_max, actual_rmax) = left_max, since left_max ≤ right_max ≤ actual_rmax.
  • Water at column left = left_max - height[left] (guaranteed, no right context needed).
⚠️ Common Misconception

"What if there's a taller wall to the right that we haven't visited yet?" — Doesn't matter! The water level at left is capped by left_max (the smaller side). Even if the right side has a wall of height 1000, it doesn't lift the water level at left above left_max.

💡 Update-then-compute order is critical

We update left_max before computing left_max - height[left]. When height[left] is itself a new max, left_max - height[left] = 0 — the wall contributes zero water. If we computed water first, we'd subtract a higher value and get a negative number.

4 The Code (Annotated) ▾

Optimal two-pointer solution — O(n) time, O(1) space, single pass:

def trap(height: list[int]) -> int:
left, right = 0, len(height) - 1 ← Pointers at opposite ends of array
left_max = right_max = 0 ← Tallest wall seen from each side
water = 0 ← Accumulator for total trapped water
while left <= right: ← Process until pointers meet or cross
if left_max <= right_max: ← Left max is the bottleneck
left_max = max(left_max, height[left]) ← Update max BEFORE computing water
water += left_max - height[left] ← Headroom between wall and bar
left += 1 ← Advance left pointer inward
else: ← Right max is the bottleneck
right_max = max(right_max, height[right]) ← Update max BEFORE computing water
water += right_max - height[right] ← Headroom between wall and bar
right -= 1 ← Retreat right pointer inward
return water ← All columns processed exactly once

Brute-force variant (O(n²), for reference):

def trap_brute(height):
water = 0
for i in range(1, len(height) - 1): ← Skip edges (no trapping possible)
lmax = max(height[:i+1]) ← Scan left for max — O(n) each time
rmax = max(height[i:]) ← Scan right for max — O(n) each time
water += min(lmax, rmax) - height[i] ← The formula, applied directly
return water ← O(n²) total, O(1) space
5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1 — Identify the Structure

Water at any column depends on context from both sides. This rules out a simple left-to-right scan. The pattern: "each element's contribution depends on global max to its left AND right" = precomputed max arrays or two pointers.

Step 2 — State the Brute Force

For each index, scan left for left_max and right for right_max. O(n²) time, O(1) space. Always say this first — it shows you understand the formula before optimizing.

Step 3 — O(n) with Arrays

Precompute lmax[i] and rmax[i] in two passes, then sum min(lmax[i], rmax[i]) - height[i] in a third pass. O(n) time, O(n) space. This is already a strong answer.

Step 4 — Eliminate Arrays with Two Pointers

Key question: "Do I need the exact right max, or just know whether it's bigger than my left max?" — I only need the comparison. Two pointers from both ends, always process the side with the smaller max. O(n) time, O(1) space.

🗣️ What to Say

"For each column, water = min(left_max, right_max) - height[i].
Brute force: scan both sides for each column, O(n²).
Optimization: precompute lmax and rmax arrays, O(n) time and space.
But I can do O(1) space: two pointers, always process smaller max side.
When left_max <= right_max, I know left_max IS the bottleneck —
even unseen right walls are >= right_max >= left_max.
So I can safely compute water at left and advance inward."
🎯 Follow-up Questions to Expect
  • "Solve with a monotonic stack?" — Yes, O(n)/O(n); pop when a taller bar is found.
  • "What if the grid is 2D?" — LC 407, Trapping Rain Water II; min-heap approach.
  • "How to test?" — Single valley, W-shape, all equal, monotone increasing/decreasing, length=1.
  • "Why does left <= right not double-count the middle element?" — Only one branch executes per iteration, each advances its pointer, so the element is processed once.

7 Summary & Key Takeaways ▾
Water at column i = max(0, min(left_max, right_max) − height[i])
The side with the smaller running max is the bottleneck — always process it
Two pointers from both ends: O(n) time, O(1) space, single pass
Update the running max BEFORE computing water — avoids false negatives
When pointers meet, every column has been accounted for exactly once
The smaller max governs even if we haven't seen the actual other-side maximum yet

🎯 When to Recognize Two Pointers (Opposite Direction)
  • Each element's contribution depends on the max/min in both directions
  • You can process the element with the smaller boundary value without knowing the other side exactly
  • Looking to eliminate O(n) space while keeping O(n) time
  • Two converging pointers naturally visit every element exactly once
