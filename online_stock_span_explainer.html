LeetCode #901

Online Stock Span

Design a data structure that collects daily stock prices and, for each call, returns the span — the number of consecutive days (including today) where the price was less than or equal to today's price.

Medium Monotonic Stack Previous Greater Element Count Time: O(1) amortized Space: O(n)
1 Understanding the Problem ▾

Imagine you're a stock trader. Each day you receive today's price and want to answer: how many consecutive days in a row, counting backwards including today, has the price been at or below today's price? That count is the span.

For example, with prices called in order [100, 80, 60, 70, 60, 75, 85]:

  • next(100) → 1 — no prior days
  • next(80) → 1 — yesterday was 100 > 80, stops immediately
  • next(60) → 1 — yesterday was 80 > 60, stops immediately
  • next(70) → 2 — yesterday 60 ≤ 70 ✓; two days back 80 > 70 ✗
  • next(60) → 1 — yesterday 70 > 60, stops immediately
  • next(75) → 4 — 60 ≤ 75 ✓, 70 ≤ 75 ✓, 60 ≤ 75 ✓, then 80 > 75 ✗
  • next(85) → 6 — 75,60,70,60,80 all ≤ 85 ✓; only 100 > 85 ✗
💡 Key Insight

The span is: "How far back can I look before hitting a strictly greater price?" This is the classic Previous Greater Element problem. A monotonic stack answers it in amortized O(1) by storing compressed spans — when a new higher price arrives, it can skip entire ranges already counted, rather than re-examining each day individually.

API design: The problem asks for a class StockSpanner with a next(price) method. Prices come in one at a time — we must process them online (incrementally), without seeing the future.

Edge Cases: Strictly decreasing prices (every span = 1 — stack grows unboundedly). Strictly increasing prices (spans grow monotonically). All equal prices (span = call number). Single call.

2 The Algorithm: Monotonic Decreasing Stack with Span Compression ▾

We maintain a stack of (price, span) pairs. The key invariant: prices in the stack are always strictly decreasing from bottom to top. Each entry is the "last surviving barrier" for a range of days it represents.

1 Initialize span = 1. Today always counts itself. We'll accumulate more as we pop the stack.
2 While the stack top price ≤ today's price: pop the top (top_price, top_span) and add top_span to our running span. Those days all had price ≤ top_price ≤ today's price, so today beats them all — absorb their entire count in one step.
3 Push (today_price, span) onto the stack. This compressed entry now represents all the days we just absorbed, plus today. Future calls with higher prices will absorb this whole range by popping just one entry.
4 Return span. That is the count of consecutive days (including today) where price ≤ today.
🔑 The Compression Insight

When we push (75, span=4), we're recording: "Behind price 75 are 4 days, all with prices ≤ 75." Any future price ≥ 75 can absorb all 4 of those days simply by popping this single entry. We collapse entire monotone runs into one "representative" — this is what makes the algorithm amortized O(1).

next(60): top=80 > 60, no pop → span=1 → push(60,1) → stack=[(100,1),(80,1),(60,1)] → return 1

next(60): top=70 > 60, no pop → span=1 → push(60,1) → stack=[(100,1),(80,1),(70,2),(60,1)] → return 1

3 Why It Works: The Invariant ▾
🔐 The Stack Invariant

After every next() call, the stack contains (price, span) pairs in strictly decreasing price order from bottom to top. Each entry (p, s) means: "There are s consecutive days (a run) where p was the maximum price, and this run immediately precedes the current position — no day in this run was superseded by a later higher price."

Why popping ≤ is correct: If top_price ≤ today, then all days in the top's range had prices ≤ top_price ≤ today. So today's price is ≥ all of them — they all satisfy the span's ≤ condition. We absorb them correctly.

Why strictly decreasing stays maintained: When we push (today_price, span), we've just popped everything with price ≤ today_price. So the new top of the stack has price > today_price — the strictly decreasing invariant holds.

Amortized O(1) proof: Each price is pushed exactly once and popped at most once. For n calls, total push + pop operations ≤ 2n. Total work = O(n) across all calls → amortized O(1) per call.

⚠️ Why ≤ not < in the pop condition?

The problem asks for days with price less than or equal to today. So equal prices also count toward the span. When the stack top has the same price as today, we pop it and absorb its days — they all had prices ≤ today. If the problem said "strictly less than," we'd use < in the pop condition.

💡 Analogy: Rising Flood Water

Imagine filling a valley (the stock chart) with water to today's price level. The span counts all days submerged (price ≤ water level). The stack entries are the "peaks sticking above water" — they block your view backward. A new high-water day floods more peaks and the span grows. Each peak is only encountered once as it gets submerged.

4 The Code (Annotated) ▾

Solution 1 — Monotonic Stack with Span Compression (Optimal · Interview Pick)

class StockSpanner:
def __init__(self): ← Constructor: set up once when object is created
self.stack = [] ← Stack of (price, span) pairs; always strictly decreasing by price
def next(self, price: int) -> int:
span = 1 ← Today always counts itself → start at 1
while self.stack and self.stack[-1][0] <= price: ← Pop while top price ≤ today (today beats those days)
span += self.stack.pop()[1] ← Absorb popped entry's span (range compression!)
self.stack.append((price, span)) ← Push today's compressed entry for future reuse
return span ← Consecutive days with price ≤ today, including today
⚠️ Why store span in the stack instead of just price?

If we only stored prices, popping and absorbing would require scanning all individual days in the stack's collapsed range. By storing the pre-computed span with each price entry, we can absorb an entire range of days (possibly many) in O(1) — just add the stored span. This is the key trick that achieves amortized O(1).

Solution 2 — Brute Force Array Scan (O(n) per call — for baseline understanding only)

class StockSpannerBrute:
def __init__(self):
self.prices = [] ← Store all prices seen
def next(self, price: int) -> int:
self.prices.append(price) ← Record today's price
span = 0
for p in reversed(self.prices): ← Scan backwards through all stored prices
if p <= price: span += 1 ← Count if within span
else: break ← First price > today breaks the consecutive run
return span
💡 Complexity Summary

Brute Force: O(n) per call, O(n²) for n calls, O(n) space
Monotonic Stack: O(1) amortized per call, O(n) total, O(n) worst-case stack space
Worst-case stack: strictly decreasing prices → all n entries survive → O(n) space always.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Name the Core Pattern

"This is asking for the previous greater element problem — specifically, how far back until I hit a strictly greater price. I've seen this pattern before with monotonic stacks."

Step 2: Propose Brute Force First

"A simple approach: store all prices, and for each new call scan backwards until I find price > today. O(n) per call, O(n²) total. Correct but too slow for large n."

Step 3: Spot the Redundancy

"Key observation: If I already know that price=75 had span=4 (it covered 4 prior days), and today is ≥ 75, I don't need to look at those 4 days again. 85 automatically beats all of them. I can skip the entire block by storing the pre-computed span."

Step 4: Design the Stack

"I'll maintain a monotonic decreasing stack of (price, span) pairs. New price arrives → pop everything ≤ it, accumulating spans → push new compressed entry. Each price pushed once, popped once → amortized O(1)."

🗣️ What to Say to the Interviewer

"This is a previous greater element problem, and we're processing prices online one at a time.
Brute force: scan backward each call — O(n) per call, O(n²) total.
Key insight: if we've already computed the span for a past price, any future higher price inherits it.
I'll use a monotonic decreasing stack of (price, span) pairs.
For each new price: start span=1, pop all ≤ today absorbing their spans, push the new entry.
Each price is pushed once and popped at most once → amortized O(1) per call, O(n) total."

7 Summary & Key Takeaways ▾
Span = consecutive days back until price strictly greater — a Previous Greater Element problem
Store (price, span) pairs, not just prices — the span enables range compression in O(1)
Pop condition is ≤ (not <) because equal prices also count toward the span
Each price is pushed exactly once, popped at most once → amortized O(1) per call
The stack always has strictly decreasing prices — it records the surviving "barriers"
