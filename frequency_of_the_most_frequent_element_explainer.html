LeetCode #1838

Frequency of the Most Frequent Element

The frequency of an element is the number of times it occurs in an array. You have a budget of k operations (each increments one element by 1). Return the maximum possible frequency of any element after using at most k operations.

Medium Sliding Window Sort + Window Sum Check Time: O(n log n) Space: O(1)
1 Understanding the Problem ▾

You have an integer array and a budget k of increment operations. In each operation you pick any element and add 1 to it (you cannot decrement). Goal: after spending at most k total increments, what is the highest count any single value can appear?

💡 Key Insight

Because we can only increment, the target value for any group must be the maximum in that group — everything else must be raised up to meet it. After sorting the array, the target for any window is always nums[right] (the rightmost/largest element). The cost to equalize a window is: nums[right] × window_size − window_sum. If that cost ≤ k, the window is valid.

2 The Algorithm: Sort + Sliding Window Sum Check ▾

The fundamental tension: we want to group as many elements as possible and raise them all to the same value — but we have limited operations. Sorting resolves the "which value to target" question: after sorting, the target is always the rightmost element in any window.

1 Sort the array ascending. Now for any contiguous window [left..right], nums[right] is the maximum. Every element to its left is ≤ target, so all we need is increments — no decrements.
2 Expand right pointer one step. Add nums[right] to window_sum. We're now considering making every element in [left..right] equal to nums[right].
3 Compute the cost to equalize this window: cost = nums[right] × (right−left+1) − window_sum. This formula works because each element nums[i] costs nums[right] − nums[i] increments, and summing gives target×size − sum.
4 Shrink left while cost > k. Remove nums[left] from window_sum and advance left. Repeat. Since the array is sorted, nums[left] is the cheapest-to-target element we're removing, reducing cost by nums[right] − nums[left].
5 Record the window size as a candidate answer: ans = max(ans, right−left+1). After shrinking, the window [left..right] is always valid (cost ≤ k).
🔑 Why Sort Changes Everything

In an unsorted array, finding which elements to group is combinatorial. After sorting, any optimal group is a contiguous subarray (you can always rearrange elements in a group without increasing cost). Sorting collapses the 2D search space into a single left-right sweep. The target is always nums[right] — there is no other candidate that could be cheaper.

⚠️ Why Not Binary Search on the Answer?

You can binary search on window size, checking if a window of that size exists within budget — O(n log² n). But the sliding window is O(n log n) total (sort is the bottleneck) and avoids nested logic. The cost function is monotone: a wider window (for the same right) always costs more or equal. This monotonicity is exactly what makes the sliding window valid.

3 Why It Works: The Invariant ▾
🔐 The Invariant

After every iteration, window_sum equals the exact sum of nums[left..right], and the cost to equalize the window — nums[right] × (right−left+1) − window_sum — is always ≤ k. The window always represents a valid, achievable assignment of elements to the target value.

The cost formula derived:

total_cost = Σ (target − nums[i]) for i in [left, right]
= target × (right−left+1) − Σ nums[i]
= nums[right] × window_size − window_sum

Why shrinking left never misses the optimal answer:

  • The left pointer only ever moves right (never left) — each element is added and removed at most once. This gives amortized O(n) for the window part.
  • When we shrink, we remove the element with the greatest individual cost (the smallest sorted element, farthest from target). This is always the right choice to reduce cost fastest.
  • We never "skip" a right value: every index becomes right exactly once, and for each, we find the widest valid window. The globally maximum valid window must appear as one of these candidates.
⚠️ Does Shrinking Miss Answers? (No — here's why)

When we shrink left and advance right, the previous left is gone. Is this safe? Yes — we're tracking the maximum window size seen. The only way to improve the answer is if the new window is strictly larger. A window of size ≤ current best with a new right is not interesting unless it grows by expanding right more, which we will do on the next iteration.

4 The Code (Annotated) ▾

Solution 1 — Sort + Sliding Window (Optimal, Interview Pick)

def maxFrequency(nums: list, k: int) -> int:
nums.sort() ← Sort ascending: nums[right] will always be our target
left = 0 ← Left boundary of sliding window
window_sum = 0 ← Running sum of elements in [left..right]
ans = 0 ← Best (max) frequency found so far
for right in range(len(nums)): ← Expand right pointer one index at a time
window_sum += nums[right] ← Include new element in the running sum
# cost = target*size - sum ← Operations needed to raise all to nums[right]
while nums[right]*(right-left+1) - window_sum > k:
window_sum -= nums[left] ← Remove leftmost (cheapest-cost) element from window
left += 1 ← Shrink window from the left
ans = max(ans, right - left + 1) ← Window is now valid; record its size
return ans ← Maximum achievable frequency
⚠️ Common Mistake: Using if Instead of while for the Shrink

If you write if cost > k instead of while cost > k, the window may still be invalid after a single shrink step. You must keep shrinking until the window satisfies the invariant. A single removal can reduce cost by at most nums[right] − nums[left], which may not be enough if the window is very over-budget.

Solution 2 — Brute Force (O(n²), for contrast)

def maxFrequency_brute(nums: list, k: int) -> int:
nums.sort() ← Still need to sort for correct grouping
ans = 1
for i in range(len(nums)): ← Each element is the target
ops = 0
for j in range(i-1, -1, -1): ← Try pulling in elements to the left
ops += nums[i] - nums[j] ← Cost to raise nums[j] to nums[i]
if ops > k: break ← Over budget; stop extending
ans = max(ans, i - j + 1) ← Valid group [j..i]; update answer
return ans ← O(n²) — TLE on n=10^5
💡 Complexity Summary

Sort + Sliding Window (optimal): Time O(n log n) [sort bottleneck; window is amortized O(n)], Space O(1)
Brute Force: Time O(n²), Space O(1)
Binary Search on Answer: Time O(n log² n), Space O(1) — correct but more complex

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Notice "increment only" → target must be maximum

The problem says "increment any element." This asymmetry is the key. You can only raise values — so within any group, the target must be the maximum element, and everything else gets raised up. This rules out choosing a target that isn't in the array.

Step 2: Sort → contiguous windows cover all cases

In an unsorted array, the optimal group could be any subset. After sorting, the optimal group is always a contiguous window — because any out-of-order subset can be rearranged into a contiguous window of the same or lower cost.

Step 3: Derive the cost formula, then recognize sliding window

Cost = Σ(target − nums[i]) = target×size − window_sum. This is O(1) with a running sum. Cost increases monotonically as the window grows for a fixed right — classic signal for a sliding window (expand right, shrink left when over budget).

Step 4: Confirm complexity and edge cases

O(n log n) from sort, O(n) amortized window → O(n log n) overall. O(1) extra space. Edge cases: k=0 → return 1; all equal → return n; verify formula at window size = 1 (cost = 0, always valid).

🗣️ What to Say to the Interviewer

"Since we can only increment, the target for any group must be its maximum element.
I'll sort the array so that for any window [left, right], the target is always nums[right].
The cost to equalize: target × window_size − window_sum. Track this with a running sum.
Sliding window: expand right always. When cost exceeds k, shrink left until valid.
Record window size at each step. Return the maximum. Time O(n log n), Space O(1)."
🔮 Likely Follow-up Questions
  • "Can you also decrement?" → Target could be any value, not just array elements. Harder; still solvable with sort + prefix sums.
  • "Prove no valid window is missed by the sliding approach." → Every index becomes right exactly once; for each right we find the maximum valid left. All candidates are covered.
  • "What if increment cost varies by element?" → Cannot use the simple sum formula. Need priority queue or different approach.
