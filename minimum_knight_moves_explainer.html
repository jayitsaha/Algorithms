LeetCode #1197

Minimum Knight Moves

In an infinite chessboard, find the minimum number of moves a knight needs to get from (0,0) to (x,y).

Medium Graph BFS Time: O(|x|·|y|) Space: O(|x|·|y|)
1 Understanding the Problem ▾

A chess knight moves in an "L-shape": exactly two squares in one direction, then one square perpendicular. From any square, the knight has up to 8 possible landing squares. On an infinite board, there are no walls — the knight can go anywhere. Starting at (0, 0), what's the minimum number of knight moves needed to reach (x, y)?

💡 Key Insight

This is a shortest path on an unweighted graph problem. Each board square is a node; each valid knight move is an edge of weight 1. BFS finds the shortest path on unweighted graphs in guaranteed O(V + E) time — because every "layer" of BFS corresponds to exactly one more move.

🔑 Critical Symmetry Observation

The chessboard is symmetric under reflections across both axes. The answer for (x, y) equals the answer for (|x|, |y|), (|y|, |x|), (-x, y), etc. This means we can always work in the first quadrant (x≥0, y≥0), cutting the search space by 4×.

Edge cases to consider:

  • (0, 0): already at destination — return 0 immediately.
  • (1, 0) or (0, 1): surprisingly need 3 moves (can't reach in 1 or 2 from origin).
  • (2, 2): needs 4 moves, not 2 (common trap — feels intuitively reachable in 2 but it isn't).
2 The Algorithm: BFS with Symmetry Reduction ▾

BFS (Breadth-First Search) explores all nodes reachable in exactly k moves before exploring any node reachable in k+1 moves. This level-by-level property is precisely what we need: the first time we reach the target, the BFS level equals the minimum moves.

1 Symmetry normalize: Replace x = abs(x), y = abs(y). The answer is identical in all four quadrants. We now only explore the first quadrant.
2 Handle trivial case: If x == 0 and y == 0, return 0 — already at destination.
3 Initialize BFS: Create a queue (deque) with (0, 0, dist=0). Create a visited set with (0, 0). Define all 8 knight move offsets: [(±1,±2), (±2,±1)].
4 BFS loop: Dequeue the front cell. For each of the 8 knight moves, compute the neighbor (nx, ny). If it equals (x, y), return dist + 1. If it's unvisited and nx >= -1 and ny >= -1, add to visited and enqueue.
5 Why allow -1? Near-origin targets like (1, 0) require the knight to briefly step into negative coordinates and bounce back. Bounding at -1 (not 0) handles these correctly without expanding the search space significantly.
🔑 Why BFS and Not DFS or Dijkstra?

DFS finds some path but not necessarily the shortest — it may dive arbitrarily deep.
Dijkstra is correct but overkill: it's BFS with a priority queue for weighted edges. Here all edges weigh 1, so plain BFS is both simpler and faster.
BFS is optimal for unweighted shortest path: it naturally processes nodes in order of increasing distance from the source.

3 Why It Works: The BFS Invariant ▾
🔐 The BFS Invariant

When BFS processes a node v with distance d, every node in the queue ahead of v has distance ≤ d, and every node behind has distance ≥ d. This means the first time BFS dequeues the target, it has found the optimal (shortest) path — no shorter path could exist, because any path of length < d would have been discovered in an earlier BFS level.

Proof by contradiction: Suppose there exists a shorter path of length d* < d to the target. That path passes through some intermediate node at distance d* - 1. BFS would have enqueued that node at level d* - 1 and processed it before level d. From there, the target would have been discovered at level d* — before our alleged discovery at level d. Contradiction. Therefore, the first discovery is optimal.

Why the visited set is essential: Without marking cells as visited, BFS could enqueue the same cell multiple times from different parents — exponentially multiplying work. Once a cell is visited (enqueued), any future path to it via BFS has length ≥ the current path. We never need to revisit.

⚠️ The nx >= -1 Lower Bound (Not 0!)

If we restrict neighbors to nx >= 0, ny >= 0, we cut off valid paths for near-origin targets.
Example: target (1, 0). The correct 3-move path is: (0,0) → (-1,2) → (1,1) → (0,-1)... wait — or try (0,0) → (2,1) → (1,-1) → (-1,0)... the actual optimal involves stepping to negative coordinates. Allowing -1 as the floor gives BFS room to find these paths while keeping the search bounded.

Search space bound: With symmetry reduction (first quadrant only) and the -1 floor, the visited set is bounded by roughly (x+2) × (y+2) cells, giving time and space complexity of O(|x| · |y|).

4 The Code (Annotated) ▾

Solution 1 — BFS with Symmetry Reduction (Optimal, Interview Pick)

from collections import deque ← O(1) popleft (critical — list.pop(0) is O(n))
def minKnightMoves(x: int, y: int) -> int:
x, y = abs(x), abs(y) ← use symmetry: same answer in all 4 quadrants
if x == 0 and y == 0: return 0 ← already there
DIRS = [(1,2),(1,-2),(-1,2),(-1,-2), ← all 8 L-shaped offsets
(2,1),(2,-1),(-2,1),(-2,-1)]
queue = deque([(0, 0, 0)]) ← (x_pos, y_pos, distance)
visited = {(0, 0)} ← mark visited before adding to queue
while queue:
cx, cy, dist = queue.popleft() ← dequeue front: O(1) with deque
for dx, dy in DIRS: ← try all 8 knight moves
nx, ny = cx + dx, cy + dy ← compute neighbor position
if (nx, ny) == (x, y): ← found target! BFS guarantees min distance
return dist + 1 ← one more move from current position
if (nx,ny) not in visited and nx>=-1 and ny>=-1:← skip visited; allow coords down to -1
visited.add((nx, ny)) ← mark now (not after dequeue) to avoid dup
queue.append((nx, ny, dist + 1)) ← enqueue with incremented distance
⚠️ Mark Visited BEFORE Enqueue, Not After Dequeue

A common BFS bug: marking cells visited only when dequeued rather than when enqueued. This lets the same cell be enqueued multiple times — once from each neighbor that discovers it. For large graphs, this causes duplicate work and potentially incorrect behavior. Always add to visited set at the moment you enqueue.

Solution 2 — Mathematical Closed Form (O(1) — mention if you know it)

def minKnightMoves(x: int, y: int) -> int:
x, y = abs(x), abs(y) ← normalize to first quadrant
if x + y == 0: return 0 ← origin
if x + y == 1: return 3 ← (0,1) or (1,0): special case needing 3 moves
if x == 2 and y == 2: return 4 ← special case: formula gives wrong 2
t = max(max(x, y), (x + y + 2) // 3) ← base estimate from triangle inequality
return t + (t - x - y) % 2 ← parity correction: adjust for color constraint
💡 Complexity Summary

BFS + Symmetry: Time O(|x|·|y|), Space O(|x|·|y|) — search bounded to first quadrant with small buffer.
Mathematical: Time O(1), Space O(1) — but the formula is nearly impossible to derive under pressure. Cite it only as a bonus observation.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify the Graph Structure

"This is a shortest path problem. Nodes are (x,y) board squares; edges are knight moves (all weight 1, unweighted graph). Minimum moves = shortest path length. This is the canonical BFS use case."

Step 2: Apply Symmetry Optimization

"The board is symmetric: answer(x,y) = answer(|x|,|y|). I only need to explore the first quadrant, cutting the search space by 4x. This bounds the BFS to roughly O(|x|·|y|) instead of an unbounded infinite grid."

Step 3: Handle the -1 Edge Case

"I'll allow coordinates down to -1 (not 0) because near-origin targets like (1,0) require a brief excursion into negative territory before bouncing back. This is a subtle but critical correctness detail."

Step 4: Implement Clean BFS

"Standard BFS with deque for O(1) popleft. Mark cells visited when enqueued (not dequeued) to prevent duplicates. First time we generate the target as a neighbor — return dist + 1."

🗣️ What to Say to the Interviewer

🔮 Likely Follow-up Questions
  • "Can you do better than BFS?" → Yes: O(1) mathematical formula. Hard to derive — better to say "theoretically yes" and discuss the intuition.
  • "What about bidirectional BFS?" → Search from both ends simultaneously; meet in the middle. Reduces explored nodes from O(b^d) to O(b^(d/2)) — roughly halves the search radius.
  • "If knight moves had different costs?" → Switch to Dijkstra's algorithm with a priority queue.
  • "Bounded N×N board?" → Standard BFS with boundary checks; no -1 trick needed.

7 Summary & Key Takeaways ▾
Model "minimum moves" problems as shortest path on an unweighted graph — BFS is the right tool
abs(x), abs(y) symmetry cuts the search space to one quadrant — always apply first
Allow coordinates down to -1 (not 0) to correctly handle near-origin special cases
Use deque.popleft() not list.pop(0) — the difference is O(1) vs O(n) per dequeue
Mark visited when enqueuing (not dequeuing) to prevent the same cell entering the queue multiple times

🎯 When to Use BFS for Shortest Path
  • "Minimum number of moves/steps/operations" → BFS if unweighted, Dijkstra if weighted
  • Grid traversal with uniform step cost → BFS
  • State-space search where each state transition has cost 1 → BFS
  • If the graph is huge/infinite, look for symmetry to bound the search
