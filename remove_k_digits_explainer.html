LeetCode #402

Remove K Digits

Given string num representing a non-negative integer and integer k, remove k digits to produce the smallest possible number.

Medium Monotonic Stack Greedy Time: O(n) Space: O(n)
1 Understanding the Problem ▾

We have a number stored as a string — for example "1432219" — and we must remove exactly k of its digits, preserving the relative order of the ones we keep. The goal is to make the resulting number as small as possible.

This is really an ordering problem: which k positions do we sacrifice? We can't try every combination (that's exponential), so we need a smarter strategy.

💡 Key Insight

In any number, the leftmost digit has the highest place value. So the way to make a number smaller is: whenever you see a digit that is smaller than the digit before it, removing that earlier (larger) digit moves a smaller digit into a higher position — which shrinks the number significantly. Do this greedily from left to right.

✗ Removing wrong digits

num="1432219", k=3
Remove indices 0,2,5 → "4221"
Result: 4221 — too large

✓ Greedy: remove larger digits

num="1432219", k=3
Remove 4(>3), 3(>2), 2(>1) → "1219"
Result: 1219 — minimal!

Edge cases to consider: If k = len(num), return "0". If leading zeros appear after removal, strip them (e.g., "10200" with k=1 → "200"). If the number is already non-decreasing (e.g., "12345"), all removals must come from the right tail.

2 The Algorithm: Greedy + Monotonic Stack ▾

We maintain a stack that holds the digits of our answer-in-progress. The stack is kept monotonically non-decreasing — meaning each digit on the stack is ≥ the one below it. When an incoming digit is smaller than the stack top (and we have removals left), we pop the top because keeping it would give a larger number at that position.

1 Initialize: Empty stack []. It will hold candidate answer digits in order.
2 Scan left to right: For each digit d in num: while k > 0 AND stack is non-empty AND stack[-1] > d — pop the top (uses one removal, k -= 1).
3 Push current digit: After all eligible pops, append d to stack.
4 Exhaust leftover k: If k > 0 after the full scan, the stack is non-decreasing — the largest digits are at the right. Remove the last k digits: stack[:len(stack)-k].
5 Clean up: Join, strip leading zeros: ''.join(result).lstrip('0') or '0'.

🔑 Why Non-Decreasing Stack?

A non-decreasing stack guarantees that at every prefix of our answer, we have the smallest possible number achievable with the digits seen and removals used. Whenever we encounter a smaller digit, we greedily "replace" any larger predecessors — reducing the number at that place value.

3 Why It Works: The Invariant ▾
🔐 The Stack Invariant

At any moment during the scan, the stack contains the lexicographically smallest subsequence formable from the digits processed so far, using exactly the removals consumed so far. It is always the best possible prefix for our final answer.

The greedy exchange argument: Suppose we have digits ...A B... on the stack where A > B, and we just received B. Keeping A at its current position and B right after gives ...AB.... But removing A and letting B occupy that position gives ...B... which is smaller (since B < A). Any answer that keeps A in front of B can always be improved — so popping is always correct.

Why remove from the tail when k remains: After a full scan with removals still left, the stack is already non-decreasing (each digit ≥ the one before it). In a non-decreasing sequence, every digit ≤ its successor, so the largest digits are at the right. Removing from the right minimizes the number. Removing from the left would expose a larger digit to a higher place value.

⚠️ Subtle Bug: stack[:-k] vs stack[:len(stack)-k]

In Python, stack[:-0] returns an empty list, not the full list. So when k=0 (all removals used during scan), stack[:-k] silently erases everything. Always use stack[:len(stack)-k], which correctly returns the whole stack when k=0.

Amortized O(n) time: The outer loop runs n times. The inner while loop pops elements, but each element is pushed exactly once and popped at most once across the entire algorithm. So total operations = O(n) pushes + O(n) pops = O(n).

4 The Code (Annotated) ▾

Solution 1 — Monotonic Stack, Optimal (Interview Pick)

def removeKdigits(num: str, k: int) -> str:
stack = [] ← Candidate result digits, maintained non-decreasing
for d in num: ← Scan each digit left to right
while stack and k and stack[-1] > d:← Pop if top is larger and removals remain
stack.pop() ← Remove the bad (larger) digit at that position
k -= 1 ← One removal used
stack.append(d) ← Incoming digit fits — push it
result = stack[:len(stack) - k] ← If k>0 remain, trim k from tail (not stack[:-k]!)
return ''.join(result).lstrip('0') or '0'← Strip leading zeros; "0" if empty
⚠️ Why use len(stack)-k instead of -k?

Python's stack[:-0] returns [] (empty), not the full stack. When k=0 after the loop (all removals used), you must keep everything. stack[:len(stack)-0] = stack[:len(stack)] = full stack. Using stack[:-k] would silently break for this case.

Solution 2 — Brute Force: Try All Subsequences (Exponential)

from itertools import combinations
def removeKdigits_brute(num: str, k: int) -> str:
n, target = len(num), len(num) - k ← We keep (n-k) digits
best = '9' * target ← Sentinel worst value
for keep in combinations(range(n), target): ← C(n, target) subsequences
cand = ''.join(num[i] for i in keep).lstrip('0') or '0'
if cand < best: best = cand ← Lexicographic comparison
return best ← O(C(n,k)·n) — fine for n≤15, hopeless for n=1000
💡 Complexity Summary

Monotonic Stack (optimal): Time O(n) amortized, Space O(n)
Brute Force: Time O(C(n,k)·n) — exponential. For n=20, k=10: C(20,10) = 184,756 iterations, manageable. For n=105: completely infeasible.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Reframe as a Greedy Problem

"The leftmost digit contributes most to the number's magnitude. So I should minimize the leftmost digit first, then the second-leftmost, and so on. This is a classic greedy setup."

Step 2: Identify the Exchange Argument

"If I see digit 3 followed by digit 2, and I have removals left, removing 3 puts 2 at a higher place value — that strictly decreases the number. This works for any pair where the left is larger. So I should always remove the first 'peak' (digit greater than its successor) first."

Step 3: Stack as the Natural Data Structure

"I need to compare each incoming digit against the most recently accepted digit. A stack gives me exactly that: O(1) access to the top. When I pop, I automatically look at the next candidate — no backtracking needed."

Step 4: Handle the Remaining k and Edge Cases

"After scanning: if k > 0 remains, the stack is non-decreasing — remove from the tail. Then strip leading zeros. If result is empty, return '0'."

🗣️ What to Say to the Interviewer

"I'll use a greedy approach with a monotonic stack.
For each digit, while the stack top is larger than the current digit and k > 0, I pop — using a removal. Then push the current digit.
After scanning all digits, if k > 0 remains, I trim k digits from the right of the stack.
Finally, strip leading zeros and return, defaulting to '0' if empty.
Time is O(n) amortized — each digit is pushed and popped at most once.
Key edge case: use stack[:len(stack)-k] not stack[:-k] to handle k=0 safely."

7 Summary & Key Takeaways ▾
The leftmost digit has the highest place value — always minimize it first (greedy exchange argument)
A monotonically non-decreasing stack enforces the greedy choice at every step automatically
Pop when stack[-1] > incoming digit AND k > 0 — this replaces a larger digit at a high place value with a smaller one
After scan: if k > 0 remains, trim k digits from the right tail of the (non-decreasing) stack
Each digit is pushed and popped at most once → O(n) amortized — despite the nested while loop
Use stack[:len(stack)-k] not stack[:-k] — Python's [:-0] returns empty list!
Always strip leading zeros and handle empty result → return "0"
