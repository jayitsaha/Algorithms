LeetCode #1143

Longest Common Subsequence

Given two strings text1 and text2, return the length of their longest common subsequence. A subsequence is derived by deleting some elements without changing the order of the rest.

Medium Dynamic Programming Classic 2D DP Time: O(m·n) Space: O(m·n)
1 Understanding the Problem ▾

A subsequence is not a substring. You can skip characters anywhere in the string — they just have to appear in the same left-to-right order. The characters you keep don't need to be adjacent.

For example, from "abcde" you can pull out "ace" by taking positions 0, 2, 4 — skipping "b" and "d". That's a valid subsequence.

💡 Key Insight

Think of LCS as independently "threading" characters from each string such that both threads spell the same word in the same order. The Longest Common Subsequence is the longest shared thread you can find.

✓ Valid LCS: text1="abcde", text2="ace"

text1: abcde (pick positions 0,2,4)
text2: ace (pick all three)
LCS = "ace" → length 3

⚠ Order matters: text1="abc", text2="cba"

"abc" reversed is "cba" — but LCS
isn't 3. You can't reverse order.
LCS = 1 (just "a", "b", or "c")

Why is this hard? There are 2ᵐ possible subsequences of text1 and 2ⁿ of text2. Comparing every pair would be exponential. We need overlapping-subproblem structure to reduce this to polynomial time.

Edge Cases: one or both empty (LCS = 0); identical strings (LCS = full length); completely different character sets (LCS = 0); one string is a subsequence of the other (LCS = shorter length).

2 The Algorithm: Classic 2D DP (Bottom-Up Tabulation) ▾

We build a 2D table dp[i][j] where each cell answers: "What is the LCS length of text1[0..i-1] and text2[0..j-1]?" We fill it row by row using the recurrence below.

1 Create dp table: Allocate a (m+1)×(n+1) table, all zeros. The +1 leaves room for the empty-string base case: row 0 and column 0 are always 0.
2 Characters match? → dp[i][j] = dp[i-1][j-1] + 1. Both strings contributed the same character — the LCS can extend by 1 from the previous prefixes (the diagonal cell).
3 Characters don't match? → dp[i][j] = max(dp[i-1][j], dp[i][j-1]). We can't use both current characters. Take the best we get by skipping one or the other.
4 Read the answer: dp[m][n] — the bottom-right cell holds the LCS of the complete text1 and text2.
📐 The Recurrence Relations
if text1[i-1] == text2[j-1]:
    dp[i][j] = dp[i-1][j-1] + 1   ← extend diagonal (match!)
else:
    dp[i][j] = max( dp[i-1][j], dp[i][j-1] )  ← skip one char

3 Why It Works: The Invariant & Why This is DP ▾
🔐 The DP Invariant

At any point during the fill, dp[i][j] holds the exact LCS length for text1[0..i-1] and text2[0..j-1]. When we fill row by row left to right, every dependency (dp[i-1][j-1], dp[i-1][j], dp[i][j-1]) is already computed before we need it.

Optimal Substructure: When both current characters match (text1[i-1] == text2[j-1]), we can extend ANY common subsequence of the shorter prefixes by appending this character. The LCS of the longer prefixes is exactly 1 more than the LCS of the shorter ones. When they don't match, the LCS of the longer prefixes is the best of the two options where we ignore one character.

Overlapping Subproblems: A naive recursive approach to compute LCS("abc","ace") calls LCS("ab","ace") and LCS("abc","ac") — both of which call LCS("ab","ac"). The same sub-problem is recomputed exponentially many times. The DP table ensures each cell is computed exactly once in O(1) time.

⚠️ Why Diagonal Only When Characters Match?

Some beginners write max(dp[i-1][j-1]+1, dp[i-1][j], dp[i][j-1]) for the match case. This is correct but unnecessary. When characters match, dp[i-1][j-1]+1 is always ≥ both neighbors: the LCS can only gain by matching, not by ignoring one of the matching characters. Trust the diagonal — it's strictly better in the match case.

Space optimization: We only ever look at the current row and the previous row. We can keep just two 1D arrays (O(n) space) instead of the full table, with the same correctness.

4 The Code (Annotated) ▾

Solution 1 — 2D Tabulation (Interview Pick, O(m·n) time & space)

def longestCommonSubsequence(text1: str, text2: str) -> int:
m, n = len(text1), len(text2) ← store lengths; text1 has m chars, text2 has n
dp = [[0]*(n+1) for _ in range(m+1)] ← (m+1)×(n+1) table; extra row/col = empty-string base case
for i in range(1, m+1): ← i is 1-indexed into dp; text1[i-1] is the character
for j in range(1, n+1): ← j is 1-indexed into dp; text2[j-1] is the character
if text1[i-1] == text2[j-1]: ← characters match!
dp[i][j] = dp[i-1][j-1] + 1 ← extend diagonal: LCS of shorter prefixes + 1
else: ← characters differ
dp[i][j] = max(dp[i-1][j], dp[i][j-1]) ← best of skipping text1[i] or skipping text2[j]
return dp[m][n] ← bottom-right cell = LCS of full strings
⚠️ Off-by-One: Why text1[i-1] not text1[i]?

The dp table has dimensions (m+1)×(n+1). Row i in dp represents the prefix of text1 of length i, which is text1[0..i-1]. When i=1 we're looking at text1[0]. The -1 bridges dp indices to string indices. This is the most common mistake — draw the table and trace it by hand.

Solution 2 — Space-Optimized (O(n) space — two rolling arrays)

def longestCommonSubsequence(text1: str, text2: str) -> int:
m, n = len(text1), len(text2)
prev = [0] * (n+1) ← represents "previous row" of the 2D table
for i in range(1, m+1):
curr = [0] * (n+1) ← build current row; starts with 0 for base case (j=0)
for j in range(1, n+1):
if text1[i-1] == text2[j-1]:
curr[j] = prev[j-1] + 1 ← prev[j-1] is the diagonal (row i-1, col j-1)
else:
curr[j] = max(prev[j], curr[j-1]) ← prev[j]=(i-1,j) is "above"; curr[j-1]=(i,j-1) is "left"
prev = curr ← slide window: current becomes previous for next row
return prev[n]

Solution 3 — Top-Down Memoization (natural from recursion)

def longestCommonSubsequence(text1: str, text2: str) -> int:
from functools import lru_cache
@lru_cache(maxsize=None)
def dp(i, j): ← i = position in text1, j = position in text2
if i == len(text1) or j == len(text2): ← base case: one string exhausted
return 0
if text1[i] == text2[j]: ← characters match — take both and advance both
return 1 + dp(i+1, j+1) ← +1 for the matched char, then recurse on the rest
return max(dp(i+1, j), dp(i, j+1)) ← skip text1[i] or skip text2[j]; take the better
return dp(0, 0)
💡 Complexity Summary

2D Tabulation: Time O(m·n), Space O(m·n) — clearest for interviews; fills entire table
Space-Optimized: Time O(m·n), Space O(n) — best practical solution
Top-Down Memo: Time O(m·n), Space O(m·n) — natural derivation from recursion

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Spot the DP Signals

"Two strings + find the longest common subsequence." Every word here is a signal. Two inputs that interact character by character → 2D DP. "Longest" → optimization → DP. "Subsequence" (not substring) → we can skip characters → the DP recurrence uses max, not just a counter. Clarify: subsequence means order preserved but not necessarily contiguous. Substring would be different.

Step 2: Define the State Clearly

Say aloud: "I'll let dp[i][j] = the length of the LCS of text1[0..i-1] and text2[0..j-1]. The +1 indexing gives me a free zero-row and zero-column for base cases." Draw a 3×4 table for a tiny example to show you understand the shape.

Step 3: Derive the Recurrence by Cases

Two characters, two cases. "If text1[i-1] == text2[j-1]: both contribute to the LCS — extend the diagonal by 1. Else: we have to sacrifice one character — take the max of looking up (skip text1[i]) or looking left (skip text2[j])."

Step 4: Mention Space Optimization

"Since each row only depends on the previous row, I can reduce space to O(n) with a rolling array. But O(m·n) space is cleaner to code and explain first."

🗣️ What to Say to the Interviewer

"Two strings, finding longest common subsequence — classic 2D DP.
State: dp[i][j] = LCS of text1[0..i-1] and text2[0..j-1]. Base: row 0 and col 0 are 0.
Recurrence: match → dp[i-1][j-1]+1. No match → max(dp[i-1][j], dp[i][j-1]).
Fill row by row. Answer at dp[m][n]. Time O(m·n), Space O(m·n), reducible to O(n)."
🔮 Likely Follow-up Questions
  • "Return the actual LCS string, not just the length?" → Backtrack the dp table from dp[m][n]: if chars matched, go diagonal; otherwise follow the larger neighbor.
  • "Edit Distance between text1 and text2?" → Very similar 2D DP — costs for insert/delete/replace (#72)
  • "Minimum deletions to make text1 == text2?" → m + n - 2·LCS (#583)
  • "Reduce space from O(m·n) to O(n)?" → Rolling array (Solution 2 above)

7 Summary & Key Takeaways ▾
LCS is 2D DP: dp[i][j] = LCS of text1[0..i-1] and text2[0..j-1]
Match → dp[i-1][j-1]+1 (diagonal). No match → max(up, left)
Base cases are zero for free: row 0 and column 0 are all 0 (empty-string LCS)
Space reduces from O(m·n) to O(n) by keeping only the previous row (rolling array)
Backtrack the table from dp[m][n] to reconstruct the actual LCS string
LCS underlies diff tools, Edit Distance, DNA sequence alignment, and patch generation
ApproachTimeSpaceNotes
Brute Force Recursion O(2ᵐ⁺ⁿ) O(m+n) Exponential — TLE on any real input
2D DP Tabulation ✓ O(m·n) O(m·n) Standard interview pick; clear table visualization
Space-Optimized DP O(m·n) O(n) Best practical; same complexity, less memory
Top-Down Memoization O(m·n) O(m·n) Natural from recursion; adds call-stack overhead
🎯 When to Recognize Classic 2D DP (LCS Pattern)
  • Two sequences + "longest/shortest common X" → Classic 2D DP (LCS family)
  • "Subsequence" (skip allowed) vs "substring" (must be contiguous) → different recurrences
  • The match/no-match two-case structure appears in Edit Distance, Wildcard Match, Regex
  • Real-world: Unix diff, git blame, DNA sequence alignment, file versioning all use LCS
