LeetCode #391

Perfect Rectangle

Given an array of axis-aligned rectangles, determine if they all together form an exact cover of a rectangular region — no gaps, no overlaps.

Hard Advanced Data Structures Corner Count + Area Check Time: O(n) Space: O(n)
1 Understanding the Problem ▾

You receive n axis-aligned rectangles, each encoded as [x1, y1, x2, y2] where (x1, y1) is the bottom-left corner and (x2, y2) is the top-right corner. The question: do these rectangles tile a single, solid, gap-free, overlap-free rectangular region?

💡 Key Insight — Two Conditions Fully Characterize a Perfect Rectangle

Condition 1 (Quantitative): The sum of all sub-rectangle areas must equal the area of the overall bounding box. This rules out gaps (area too small) and overlaps (area too large).

Condition 2 (Structural): Each rectangle contributes 4 corner points. In a perfect tiling, every interior corner is shared by an even number of rectangles (2 or 4), so they cancel out. Only the 4 extreme corners of the bounding box appear an odd number of times (exactly once each).

✓ Perfect Rectangle: 4 rects → 2×2 grid

R1=[0,0,1,1], R2=[0,1,1,2]
R3=[1,0,2,1], R4=[1,1,2,2]
Total area = 1+1+1+1 = 4
Bounding box = 2×2 = 4 ✓
Corner set after XOR = {(0,0),(0,2),(2,0),(2,2)} ✓
→ True

✗ Not Perfect — gap in bottom-right

R1=[0,0,1,1], R2=[0,1,1,2]
R3=[1,1,2,2] (only 3 rects!)
Total area = 1+1+1 = 3
Bounding box = 2×2 = 4
3 ≠ 4 → Area check fails
→ False

Corner counting intuition: Each rectangle "stamps" its 4 corners into a toggle-set (XOR semantics). Shared corners between adjacent rectangles appear twice and cancel. Only unmatched corners survive. In a perfect tiling, only the 4 outer corners survive.

Edge Cases: A single rectangle (n=1) is always a perfect rectangle — area equals bounding box, only 4 corners in set. Overlapping rectangles inflate the area sum beyond the bounding box. Gaps cause the area sum to fall short.

2 The Algorithm: Corner Count + Area Check ▾

A single O(n) pass over all rectangles. For each rectangle, we accumulate its area and XOR-toggle its 4 corners into a set. After the pass, we verify two conditions.

1 Initialize. Set total_area = 0, corners = set(), and bounding-box trackers min_x, min_y = +∞, max_x, max_y = −∞.
2 Process each rectangle [x1, y1, x2, y2]. Add (x2−x1)×(y2−y1) to total_area. Update the bounding box with min/max of coordinates.
3 Toggle the 4 corners. For each of (x1,y1), (x1,y2), (x2,y1), (x2,y2): if the point is already in corners, remove it (appeared even times — cancels). Otherwise add it (first or odd occurrence).
4 Build the expected corner set. The 4 outer corners of the bounding box: {(min_x,min_y), (min_x,max_y), (max_x,min_y), (max_x,max_y)}.
5 Return True if and only if both conditions hold. corners == expected AND total_area == (max_x−min_x)×(max_y−min_y). Either condition alone is insufficient.
🔑 Why Both Conditions Are Necessary

Area check alone: Two identical rectangles stacked on top of each other have area = 2×bbox — fails. But a more subtle arrangement might have correct total area without tiling correctly. The corner check catches structural errors. Corner check alone: A single rectangle with a tiny rectangle cut from its interior might have correct corner parity but wrong area. You need both.

3 Why It Works: The Invariant ▾
🔐 The Invariant

The toggle-set at any point contains exactly the corner points that have appeared an odd number of times. Adding a point that is already present removes it (simulating XOR). After processing all rectangles, interior corners will have been seen an even number of times and vanished. Only the 4 outer corners — each owned by exactly one rectangle — remain.

Why interior corners always appear an even number of times: In a valid tiling, rectangles meet edge-to-edge. At any interior point where rectangles touch, they do so in one of two ways: (a) two rectangles share an edge segment — the shared endpoint appears in both → 2 times (even); (b) four rectangles meet at a cross-point — each contributes once → 4 times (even). Both are even, so these corners disappear from the set.

Why the 4 outer corners appear exactly once: The extreme corners of the bounding box can only belong to one sub-rectangle. No other rectangle can have that corner as one of its corners — that would place part of it outside the bounding box. So these 4 corners appear exactly once each — odd count — and stay in the set.

Why area check is also needed: The corner check verifies structural corner parity but doesn't rule out a pathological case where rectangles cleverly mis-tile yet have the right corner XOR. The area check provides an independent arithmetic guard: if there are gaps, total area is less than bounding box area; if overlaps, greater. Together the two checks are sufficient.

⚠️ Common Trap: Corner Check Alone Can Fail

Counterexample where corner check alone passes but there's an overlap: Consider R1=[0,0,3,3] and R2=[1,1,2,2] — a rectangle fully inside another. The inner rectangle's corners are each introduced once (net +4 to the set) and the outer also introduces its 4 corners (net +4), giving 8 odd-count corners. This fails the corner check correctly. But it's a reminder: always apply both checks.

4 The Code (Annotated) ▾

The optimal solution is a clean single-pass O(n) algorithm — one loop, one set, one accumulator:

def isRectangleCover(rectangles): ← takes list of [x1,y1,x2,y2]
total_area = 0 ← running sum of sub-rectangle areas
corners = set() ← XOR-toggle set: odd-count corners survive
min_x = min_y = float('inf') ← bounding box (start at extremes)
max_x = max_y = float('-inf') ← bounding box (other extreme)
for x1, y1, x2, y2 in rectangles: ← unpack each rectangle's corners
total_area += (x2 - x1) * (y2 - y1) ← add this rect's area to running total
min_x = min(min_x, x1) ← expand bounding box left
min_y = min(min_y, y1) ← expand bounding box down
max_x = max(max_x, x2) ← expand bounding box right
max_y = max(max_y, y2) ← expand bounding box up
for pt in [(x1,y1),(x1,y2),(x2,y1),(x2,y2)]: ← all 4 corners of this rectangle
if pt in corners: ← seen before? → second (or even) occurrence
corners.remove(pt) ← remove: even count → cancels out
else: ← first time seeing this point
corners.add(pt) ← add: odd count → survives for now
expected = {(min_x, min_y), (min_x, max_y), ← 4 outer corners of bounding box
(max_x, min_y), (max_x, max_y)} ← (continued from above)
bbox_area = (max_x - min_x) * (max_y - min_y) ← area of bounding box
return corners == expected and total_area == bbox_area ← both must hold for perfect rectangle
⚠️ Why Not Use XOR Operator (^) Directly?

Python's ^ operator works on integers, not tuples. You can't do seen ^= (x1, y1). Instead, we simulate XOR semantics manually: add if absent, remove if present. This is exactly what XOR does for bit sets — the set IS our XOR accumulator.

💡 The Brute-Force Approach (O(n²)) — Mention First

For every pair of rectangles, check if they overlap (6 comparisons per pair → O(n²) total). Then verify their union covers exactly a rectangle with no holes. This is correct but O(n²) — too slow. After mentioning it, pivot to the XOR insight for O(n).

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Start with the Quantitative Check

The most obvious condition: if all sub-rectangles perfectly tile a region, their areas must sum to the bounding box area. Say this first — it immediately shows you've reduced the problem to two verifiable properties.

Step 2: Recognize That Area Alone Isn't Enough

Give a quick counterexample: two rectangles that don't line up but happen to have the right total area. This motivates the structural check. Interviewers love when you immediately poke holes in your own naive approach.

Step 3: Think About What Distinguishes Meeting Points

Draw a 2×2 tiling on paper. At every interior point, rectangles meet. Ask: "what happens to corners at each meeting?" Interior corners are shared by 2 or 4 rectangles — even counts. Outer corners belong to just 1 — odd counts. Parity is the insight.

Step 4: XOR Semantics as Parity Counter

Once you see it's about even vs. odd counts, XOR is the natural tool. Add if absent, remove if present — this is XOR on a set. After all rectangles, only odd-count points remain. Verify they are exactly the 4 bounding-box corners.

🗣️ What to Say to the Interviewer

"First condition: the sum of all sub-rectangle areas must equal the bounding box area. This catches gaps and overlaps quantitatively. But area alone is insufficient — I can construct counterexamples. I need a structural check. Key observation: in a perfect tiling, each rectangle contributes 4 corners. Interior corners are shared by 2 or 4 rectangles — always even — so they cancel. Only the 4 extreme corners of the bounding box appear exactly once each. So I'll use a toggle-set: add a corner if it's new, remove it if it's already present. This simulates XOR — only odd-count corners survive. After all rectangles, the set should contain exactly the 4 bounding-box corners. I return True only if BOTH conditions hold: area matches AND corner set matches. Time O(n), Space O(n). Let me code it up."
💡 Expected Follow-Up Questions

Q: What if rectangles can have fractional coordinates? Same algorithm — Python tuples of floats hash correctly.
Q: What if n=1? One rectangle: area equals its own bounding box, 4 corners are the 4 outer corners → True.
Q: Can you do O(1) space? Not easily — the corner set fundamentally needs O(n) in the worst case (all corners distinct).
Q: What about 3D? Extend to 8 corners per box; same XOR parity logic applies.

7 Summary & Key Takeaways ▾
Two conditions together are necessary and sufficient: area match + correct corner parity
XOR-toggle (add if absent, remove if present) naturally tracks odd/even count using a plain set
Interior corners always appear 2 or 4 times in a perfect tiling — they always cancel out
Only the 4 extreme corners of the bounding box appear exactly once — they survive the XOR
Single O(n) pass suffices — no pairwise comparison is needed
Time O(n), Space O(n) for the corner set (up to 4n points, each tuple is O(1) hashed)
