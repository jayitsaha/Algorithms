LeetCode #33

Search in Rotated Sorted Array

A sorted array was rotated at an unknown pivot. Find the target in O(log n) time, or return -1.

Medium Binary Search Identify Sorted Half Time: O(log n) Space: O(1)
1 Understanding the Problem ▾

Start with a sorted array: [1, 3, 5, 7, 9, 11, 13]. A "rotation" takes a suffix of the array and moves it to the front. For example, rotating at index 4 gives [9, 11, 13, 1, 3, 5, 7]. The result is no longer fully sorted — but it has a special structure: two sorted sub-sequences joined end-to-end, with one discontinuity ("drop") somewhere in the middle.

Your goal: given this rotated array and a target, return the target's index, or -1. And you must do it in O(log n) — meaning binary search of some kind.

💡 Key Insight

No matter where mid falls, one of the two halves around mid is always fully sorted. A rotation creates exactly one drop; splitting at any mid point places that drop entirely in one half. The other half has no drop — it's sorted. Use that sorted half's boundaries to decide where to search next.

Rotation Anatomy

Original: [1, 3, 5, 7, 9, 11, 13]
Rotated at pivot 4 →
[9, 11, 13, | 1, 3, 5, 7]
Left block: sorted 9→13
Right block: sorted 1→7
Drop: between idx 2 and idx 3

Why Naive Binary Search Fails

[9, 11, 13, 1, 3, 5, 7], target=3
mid = 3, nums[mid] = 1
1 < 3 → "go right" in normal BS
But 3 IS on the right! (ok here)
[4,5,6,7,0,1,2], target=0, mid=7
7 > 0 → "go left" — RIGHT has 0!

Edge cases to consider: single-element array, no rotation at all (array fully sorted), target equals first or last element, target at the pivot element, target absent entirely.

2 The Algorithm: Identify the Sorted Half ▾

The strategy is to modify binary search with one extra question before deciding which direction to move: "which half is sorted?" Once we know, we check whether the target could live in that sorted range. If yes, go there. If no, go to the other half.

1 Set up the search window: lo = 0, hi = len(nums) - 1. Loop while lo ≤ hi.
2 Compute mid and check for target: mid = (lo + hi) // 2. If nums[mid] == target, return mid.
3 Identify which half is sorted: If nums[lo] ≤ nums[mid], the left half [lo..mid] is sorted (no drop in it). Otherwise, the right half [mid..hi] is sorted.
4 Left sorted — decide direction: Is nums[lo] ≤ target < nums[mid]? If yes → hi = mid - 1 (search left). If no → lo = mid + 1 (search right).
5 Right sorted — decide direction: Is nums[mid] < target ≤ nums[hi]? If yes → lo = mid + 1 (search right). If no → hi = mid - 1 (search left).
6 Return -1 if loop exhausted: Target does not exist in the array.

🔑 Why One Half Is Always Sorted

A rotated sorted array has exactly one "drop point" (where a value is followed by a smaller value). When you pick any mid, this drop falls entirely in either the left half or the right half — never in both. Whichever half does NOT contain the drop is fully sorted. Comparing nums[lo] to nums[mid] tells you which half the drop is in.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every iteration: if the target exists in the array, it is within nums[lo..hi]. Each iteration cuts this window in half while preserving the invariant — guaranteeing we find the target or correctly determine it's absent in O(log n) steps.

Proof sketch — why each branch is safe:

  • Left is sorted, target in [lo, mid): The sorted left half spans exactly [nums[lo], nums[mid]). If target is in that range, it must be somewhere in indices [lo, mid-1]. Setting hi = mid-1 keeps it in window.
  • Left is sorted, target NOT in [lo, mid): Target cannot be in the sorted left half. By the invariant, it must be in the right portion. Setting lo = mid+1 is safe.
  • Right is sorted, target in (mid, hi]: Symmetric argument — target is in right, set lo = mid+1.
  • Right is sorted, target NOT in (mid, hi]: Must be in left, set hi = mid-1.
⚠️ The ≤ in nums[lo] ≤ nums[mid] Is Critical

When the window shrinks to 2 elements (lo = mid-1, hi = mid), mid equals lo. In this case nums[lo] == nums[mid], so we need ≤ (not strict <) to correctly classify the left half as sorted. Using strict < would falsely say "right sorted" and might skip the target.

💡 No-Rotation Case Works Too

If the array was never rotated (nums[0] ≤ nums[-1]), the left half always satisfies nums[lo] ≤ nums[mid], and the algorithm degenerates into exactly standard binary search — no special handling needed.

4 The Code (Annotated) ▾

Solution 1 — Identify Sorted Half (Optimal, O(log n), Interview Pick)

def search(nums: list[int], target: int) -> int:
lo, hi = 0, len(nums) - 1 ← Initialize the search window to the whole array
while lo <= hi: ← ≤ because a single element (lo==hi) is still a valid candidate
mid = (lo + hi) // 2 ← Integer midpoint; no overflow risk in Python
if nums[mid] == target: ← Direct match: return immediately
return mid
if nums[lo] <= nums[mid]: ← No drop in [lo..mid] → left half is sorted
if nums[lo] <= target < nums[mid]: ← Target within sorted left range?
hi = mid - 1 ← Yes: eliminate right half, search left
else: ← No: target can't be in sorted left → must be in right
lo = mid + 1
else: ← Drop is in [lo..mid] → right half [mid..hi] is sorted
if nums[mid] < target <= nums[hi]: ← Target within sorted right range?
lo = mid + 1 ← Yes: eliminate left half, search right
else: ← No: target can't be in sorted right → must be in left
hi = mid - 1
return -1 ← Window collapsed to empty; target is not in array
⚠️ Common Mistake: Comparing to target without checking sorted half first

It's tempting to write if nums[mid] < target: lo = mid + 1 — the standard binary search move. In a rotated array, this is wrong. nums[mid] might be in the elevated left run, and the target in the low right run. The boundary check against the sorted half is what saves you.

Solution 2 — Linear Scan (Brute Force, O(n))

def search_linear(nums, target):
for i, v in enumerate(nums): ← Scan every element — works but O(n), fails the constraint
if v == target: return i
return -1
💡 Complexity Summary

Identify Sorted Half: Time O(log n) — halves the window each iteration. Space O(1) — three pointer variables.
Linear Scan: Time O(n), Space O(1) — correct but disqualified by the O(log n) requirement.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: O(log n) signals Binary Search

When a problem states "you must achieve O(log n) runtime," that is a near-certain invitation to use binary search. This problem has an ordered structure (sorted + rotated) — it has structure we can exploit. Propose binary search immediately.

Step 2: Acknowledge Why Standard Binary Search Fails

"Standard binary search decides direction by comparing nums[mid] to target. After rotation, that comparison alone is unreliable because the array isn't monotone — a small target might be 'below' mid numerically but actually stored to the right."

Step 3: State the Key Observation

"Even in a rotated array, splitting at any mid always leaves one fully-sorted half. I can detect which half is sorted by checking nums[lo] vs nums[mid]. Once I know the sorted half, I can use its endpoints to reliably determine if target belongs there."

Step 4: Dry-Run a Small Example Before Coding

Show: [4,5,6,7,0,1,2], target=0. lo=0,hi=6 → mid=3,nums[3]=7 ≠ 0. nums[0]=4 ≤ 7 → left sorted. Is 0 in [4,7)? No → lo=4. mid=5,nums[5]=1 ≠ 0. nums[4]=0 ≤ 1 → left sorted. Is 0 in [0,1)? Yes → hi=4. mid=4,nums[4]=0 == target → return 4. Done in 3 steps.

🗣️ What to Say to the Interviewer

"The O(log n) requirement tells me to use binary search.
The challenge is the rotation — standard BS fails on non-monotone data.
Key insight: at any split, one half is always fully sorted.
I identify which half by comparing nums[lo] to nums[mid].
Then I check if target is in the sorted half's range [lo..mid) or (mid..hi].
If yes, search there. If no, search the other half.
Either way I halve the window — so the loop runs at most log2(n) times.
Time O(log n), Space O(1). Let me code it."
🔮 Likely Follow-up Questions
  • "What if duplicates are allowed? (LC #81)" → When nums[lo] == nums[mid] you can't tell which half is sorted → lo += 1 and continue. Worst case becomes O(n) for all-equal arrays.
  • "Find the rotation pivot / minimum? (LC #153)" → Binary search for the single "drop point" instead of a target value.
  • "What if rotated multiple times?" → Multiple rotations reduce to a single net rotation — same algorithm applies.
  • "How would you search if you don't know n? (unbounded/infinite array)" → First exponentially expand until you bracket target, then apply this search.
