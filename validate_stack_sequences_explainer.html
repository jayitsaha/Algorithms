LeetCode #946

Validate Stack Sequences

Given two integer arrays pushed and popped of distinct values, return true if this could have been the result of a sequence of push and pop operations on an initially empty stack.

Medium Stacks Simulate Push/Pop Time: O(n) Space: O(n)
1 Understanding the Problem ▾

You are given a push sequence — the exact order in which elements are pushed onto a stack — and a pop sequence — the order in which they come out. The question is: is the pop sequence physically achievable? Can you interleave the pushes (which must happen in the given order) with pops (which must happen in LIFO order) to produce the given pop sequence?

💡 Key Insight

A stack lets you only remove the top element. So if the next expected pop is not on top, your only option is to keep pushing until it arrives on top. If you push everything and it still never arrived on top at the right moment — the sequence is invalid. Simulate the process directly.

✓ Valid: pushed=[1,2,3,4,5], popped=[4,5,3,2,1]

push 1,2,3,4 → pop 4
push 5 → pop 5
pop 3 → pop 2 → pop 1 ✓

✗ Invalid: pushed=[1,2,3,4,5], popped=[4,3,5,1,2]

push 1..4 → pop 4,3
push 5 → pop 5
stack=[1,2], want 1 before 2 ✗

Key constraints: All values are distinct (no duplicate confusion) and popped is guaranteed to be a permutation of pushed (every value appears in both). Edge cases: single element (always valid), pop sequence = push sequence reversed (always valid — push all then pop all), pop sequence = push sequence (valid only if popping as soon as each is pushed).

2 The Algorithm: Greedy Stack Simulation ▾

We replay the scenario literally: maintain a real simulation stack stk and a pointer pop_idx into the popped array. Push every element from pushed in order — and after each push, greedily pop as many times as possible while the stack top matches the next expected pop.

1 Initialize: stk = [] (empty simulation stack), pop_idx = 0 (next expected pop is popped[0]).
2 Push unconditionally: For each x in pushed, do stk.append(x). Every element must be pushed exactly once in the given order — no choice here.
3 Greedy pop cascade: After pushing, enter a while loop: while the stack is non-empty AND stk[-1] == popped[pop_idx], pop the stack and advance pop_idx. One push may trigger many pops.
4 Final verdict: After exhausting all pushes, check stk == []. Empty = every element was consumed in exactly the right order = True. Non-empty = some element couldn't be popped at the right time = False.

🔑 Why Is the Greedy Pop Always Safe?

If stk[-1] == popped[pop_idx], delaying this pop only makes things harder. Any future push buries this element deeper — and to eventually pop it, all future elements on top must be popped first. But those future elements appear after our element in the popped sequence, meaning our element's turn comes before theirs. Delaying is strictly worse. Therefore: pop immediately, always.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every point in the simulation: stk contains exactly the elements that have been pushed but not yet popped (in push order, bottom to top). The elements popped[0..pop_idx-1] have all been successfully matched and removed from the stack. No element is ever popped except when it is provably the correct next pop.

Why empty stack means True: Since pushed and popped are permutations of each other with equal length, if every pop succeeds in the correct order, then pop_idx must reach len(popped) — meaning all elements were consumed. At that point, the stack is empty by construction (we popped every element we pushed).

Why non-empty stack means False: Some element remains in the stack, which means we ran out of pushes before it could be reached at the top. But the pop sequence still expects it somewhere — that position has passed and we cannot go back. Contradiction: sequence is invalid.

⚠️ Common Trap: Using if Instead of while

Writing if stk and stk[-1] == popped[pop_idx] (single pop per push) misses cascade pops. After pushing the last element in a sequence, you may need to pop many elements in a row. The while loop is essential — it processes the entire cascade triggered by that single push.

⚠️ Common Trap: Early Exit on Mismatch

After pushing x, if stk[-1] != popped[pop_idx], you cannot immediately return False — the required pop might arrive after several more pushes. Only after exhausting all pushes can you check the final stack state.

4 The Code (Annotated) ▾

Solution 1 — Greedy Stack Simulation (Optimal, Interview Pick)

def validateStackSequences(pushed, popped):
stk = [] ← Simulation stack, starts empty
pop_idx = 0 ← Pointer: next expected pop is popped[pop_idx]
for x in pushed: ← Must push every element in order (mandatory)
stk.append(x) ← Push is unconditional
while stk and stk[-1] == popped[pop_idx]: ← Greedy: pop as long as top matches expected
stk.pop() ← Remove from simulation stack
pop_idx += 1 ← Advance expected-pop pointer
return stk == [] ← All consumed in correct order? → True
⚠️ Why no bounds check on pop_idx?

Since popped is a permutation of pushed (same distinct values, same length), pop_idx can reach at most len(popped). But the while guard stk and ... ensures we only access popped[pop_idx] while the stack is non-empty. When stk is empty (all n elements popped), the while exits — at that point pop_idx == n but we never index it again.

Solution 2 — In-Place O(1) Space Variant (Follow-up)

def validateStackSequences(pushed, popped):
top = 0 ← Reuse pushed[] as the stack; top = stack top index
pop_idx = 0 ← Same pointer logic as before
for x in pushed: ← Write x into pushed[top], then advance top
pushed[top] = x; top += 1 ← Overwrites positions we've already processed
while top and pushed[top-1] == popped[pop_idx]: ← top>0 means stack non-empty
top -= 1; pop_idx += 1 ← "Pop" by decrementing top
return top == 0 ← Stack empty iff top==0
💡 Complexity Summary

Greedy Simulation: Time O(n), Space O(n) — each element pushed once, popped at most once (amortized O(1) per element across the while loop).
In-place variant: Time O(n), Space O(1) — reuses the input array as the stack, no extra allocations.
Brute force (enumerate interleavings): O(2ⁿ) — exponential, impractical.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Pattern

Signals: "stack operations," "push/pop sequence," "could this be the result of stack operations?" Whenever you must validate a sequence under LIFO constraints, the answer is almost always: simulate the stack. Do not overthink it — stack validation = stack simulation.

Step 2: Trace a Small Example by Hand

Take pushed=[1,2,3,4,5], popped=[4,5,3,2,1]. Mentally trace: push 1,2,3,4 → top is 4, expected pop is 4 → pop. Push 5 → top is 5, expected is 5 → pop. Expected is 3 → pop 3,2,1 in cascade. Stack empty → True. This trace reveals the while-loop cascade naturally.

Step 3: Implement with the While Loop

The key implementation detail is using while not if. After one push you may need to pop many elements consecutively. Write while stk and stk[-1] == popped[pop_idx]: — this handles the cascade automatically.

Step 4: Offer the O(1) Space Optimization

"I can also do this in O(1) extra space by reusing the pushed array itself as the stack — tracking a top pointer instead of a real list. This avoids any extra allocation." This shows depth and often earns follow-up brownie points.

🗣️ What to Say to the Interviewer

"I'll simulate the stack operations directly. I iterate through pushed — each element must be pushed in order.
After each push, I greedily pop while the stack top matches the next expected pop from popped[].
I track the next expected pop with a pointer pop_idx.
At the end: if the stack is empty, all pops matched perfectly → True. Otherwise → False.
Time O(n) amortized: each element pushed once, popped at most once.
Space O(n) for the stack. Can reduce to O(1) by reusing the input array as the stack."
🔮 Likely Follow-up Questions
  • "Can you do O(1) space?" → Yes, reuse pushed[] as stack (write pointer trick)
  • "What if values are not distinct?" → Algorithm still works; LIFO order disambiguates
  • "How many valid pop sequences exist for n elements?" → Catalan number C(n) = (2n choose n)/(n+1)
  • "Generate all valid pop sequences?" → Backtracking: at each step choose push or pop if valid

Simulate directly — replay the push/pop operations using a real stack
Push is always mandatory and unconditional; ordering is fixed

Use while (not if) — one push can trigger a cascade of pops
