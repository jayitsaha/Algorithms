LeetCode #228

Summary Ranges

Given a sorted unique integer array, return the smallest sorted list of ranges that cover all array numbers exactly.

Easy Intervals Track Range Start/End Time: O(n) Space: O(1)
1 Understanding the Problem ▾

You have a sorted array of unique integers. Your job is to compress consecutive runs into range strings. A range is any maximal sequence of consecutive integers. Format single values as "a" and multi-value runs as "a->b".

💡 Key Insight

Two adjacent numbers nums[i] and nums[i+1] belong to the same range if and only if nums[i+1] - nums[i] == 1 (they are consecutive). Any gap > 1 means a new range must start. Track a range-start pointer and emit a range whenever a gap is detected or the array ends.

✓ Example 1: [0,1,2,4,5,7]

0,1,2 → consecutive → "0->2"
4,5 → consecutive → "4->5"
7 → single → "7"
Output: ["0->2","4->5","7"]

✓ Example 2: [0,2,3,4,6,8,9]

0 → single → "0"
2,3,4 → consecutive → "2->4"
6 → single → "6"
8,9 → consecutive → "8->9"

Edge cases: Empty array → return []. Single element → one range string with just that number. All elements consecutive → one single range. No consecutive elements → each element is its own range string.

🔑 What "maximal" means

A range must be as long as possible. You cannot split [2,3,4] into "2->3" and "4" — that's not maximal. You must output "2->4". The inner loop keeps extending until the run breaks.

2 The Algorithm: Single-Pass Range Tracker ▾

We scan the array once, maintaining a start variable that remembers where the current range began. At each position we check: is the next element consecutive? If yes, the current range extends. If no (or we hit the end), we emit the completed range and start a fresh one.

1 Outer loop — set start: When the outer loop begins a new iteration, record start = nums[i]. This is the beginning of a new range we don't yet know the end of.
2 Inner loop — extend range: Advance i while nums[i+1] == nums[i] + 1. Each step confirms the next element is consecutive. When the inner loop exits, nums[i] is the last element of the current range.
3 Emit the range: If start == nums[i], emit "start" (single element). Otherwise emit "start->nums[i]" (multi-element). Append to result.
4 Advance past range: i += 1 moves the outer loop past the last element of the just-emitted range. The next outer iteration will record the new start.

At each element, one of two things happens:

nums[i+1] = nums[i]+1
Continue range — advance i
or
gap / end of array
Emit range, reset start
🔑 Why this is O(n) despite nested loops

The outer and inner loops together advance i exactly once per element. The inner loop consumes elements by incrementing i, and the outer loop starts from wherever i left off. No element is ever visited twice.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At any point during the scan, start holds the first element of the current (open) range. Every element from index of start up to the current i forms a single consecutive sequence. When a gap is detected or the array ends, nums[i] is the last element of this range.

Why the invariant is maintained: We set start = nums[i] at the top of each outer iteration — exactly when a new range begins. The inner loop only advances i when the consecutive condition holds. When it exits, the invariant is ready for the emit step.

Why sorted + unique is required: If elements were unsorted, gaps could appear anywhere and we'd need a different approach (sort first, or use a set). If elements could repeat, the adjacency check nums[i+1] == nums[i] + 1 could miss ranges. The given constraints make the single-pass adjacent comparison the complete solution.

⚠️ Common Mistake: Forgetting to emit the final range

The last element of the array never triggers the gap condition — there's no "next element" to compare against. A common bug is a loop that only emits on gaps, forgetting the last open range. The nested-loop structure in the solution handles this naturally: the outer loop's next i check eventually fails, and the emit happens inside the outer iteration before looping.

Proof of completeness: Every element is "owned" by exactly one outer iteration. The outer loop starts a range, the inner loop extends it, then the emit step closes it. Since outer iterations are non-overlapping and cover all indices, every element appears in exactly one emitted range.

4 The Code (Annotated) ▾

Solution 1 — Single-Pass Range Tracker (Optimal, Interview Pick)

def summaryRanges(nums): ← Takes sorted list of unique ints
result, n, i = [], len(nums), 0 ← result accumulates range strings; i is our scan pointer
while i < n: ← Process every element (outer loop)
start = nums[i] ← Record start of this new range
while i+1 < n and nums[i+1] == nums[i]+1: ← Extend while next is consecutive
i += 1 ← Advance into the range
if start == nums[i]: ← Did we stay in place? Single element.
result.append(str(start)) ← Format as "a"
else: ← We advanced; multi-element range
result.append(f"{start}->{nums[i]}") ← Format as "a->b"
i += 1 ← Move past end of this range; start fresh
return result ← All ranges collected
⚠️ Why compare start == nums[i] rather than tracking a length?

After the inner loop, nums[i] is the last value in the current range. Comparing it against start tells us whether the range is a single element without needing a separate counter. If start == nums[i], the inner loop never ran — we stayed at the same index and this is a solo element.

Solution 2 — Explicit Two-Pointer (i = start, j = end)

def summaryRanges(nums):
result, n = [], len(nums)
i = 0
while i < n:
j = i ← j scans to find range end; i is anchored at range start
while j+1 < n and nums[j+1] == nums[j]+1:
j += 1 ← Extend range end
if i == j: ← i and j same → single element
result.append(str(nums[i]))
else:
result.append(f"{nums[i]}->{nums[j]}")
i = j + 1 ← Jump i past entire range to next start position
return result
💡 Complexity Summary

Both solutions: Time O(n) — the outer + inner loops together advance the pointer by 1 per element, never revisiting. Space O(1) extra (result list is required output).
Brute Force (check all pairs): O(n²) — completely unnecessary given sorted input.
Sort-based (if input unsorted): O(n log n) sort + O(n) scan.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Core Check

"The problem wants me to group consecutive numbers. The simplest possible check: are two adjacent numbers exactly 1 apart? If yes, same group. If no, different group. Given sorted + unique input, this is the only check needed."

Step 2: Identify What State to Track

"I need to remember where the current group started. A single variable start is sufficient. I record it at the beginning of each group, then compare it to the end of the group when I emit."

Step 3: Handle the Two Emit Conditions

"I emit when: (a) I find a gap between adjacent elements, or (b) I reach the end of the array. This second condition is critical — many beginners forget the final flush."

Step 4: Format Output Correctly

"If start equals end of range: str(start). If start differs from end: f'{start}->{end}'. This distinction is just string formatting."

🗣️ What to Say to the Interviewer

"I'll use a single-pass scan with a start variable.
For each element, I record start, then use an inner loop to advance while consecutive.
When the inner loop exits, nums[i] is the range end.
If start == end, emit 'start'. Otherwise emit 'start->end'.
Then advance past the range and repeat. Time O(n), Space O(1) extra."
🔮 Likely Follow-up Questions
  • "What if the input is unsorted?" → Sort first O(n log n), then same scan. Or use a hash set to check for num-1.
  • "What if there are duplicates?" → Deduplicate first, then same approach.
  • "Given ranges, find missing ranges between lo and hi?" → Similar scan, emit gaps between consecutive values (#163).
  • "Merge overlapping intervals?" → Sort by start, then check if current interval's start ≤ prev end (#56).
  • "Can you implement this as a generator?" → Yes, replace append with yield — same logic.
