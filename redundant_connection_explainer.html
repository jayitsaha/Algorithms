LeetCode #684

Redundant Connection

In an undirected graph that started as a tree with one extra edge added, find and return the redundant edge that forms the cycle.

Medium Graph Union-Find Time: O(n · α(n)) Space: O(n)
1 Understanding the Problem ▾

We're given a graph with n nodes (labeled 1 to n) and n edges — one more edge than a tree needs. The graph was originally a valid spanning tree (n nodes, n−1 edges, fully connected, no cycles). Then one extra edge was secretly added, creating exactly one cycle somewhere. Our job: identify and return that one redundant edge.

💡 Key Insight

A tree with n nodes has exactly n−1 edges. Adding one more always creates exactly one cycle. The redundant edge is the one that, when added, connects two nodes that are already in the same connected component. We process edges in order and return the first edge where both endpoints share the same root in Union-Find.

What "redundant" means precisely: Removing the returned edge leaves a valid spanning tree — fully connected, no cycles, exactly n−1 edges. The problem guarantees exactly one such edge exists. If multiple edges could be the answer (they're all part of the cycle), return the one that appears last in the input array.

Edge cases to consider: Minimum input is n=3 with a triangle [[1,2],[1,3],[2,3]]. All three form a cycle; the last one [2,3] is the answer. Nodes are 1-indexed. The input is guaranteed to form a tree plus exactly one extra edge — no self-loops, no disconnected components.

2 The Algorithm: Union-Find (Disjoint Set Union) ▾

Union-Find (also called Disjoint Set Union, DSU) is a data structure that efficiently tracks which nodes belong to the same connected component. It supports two core operations in amortized nearly-O(1) time each: find (who is this node's root/representative?) and union (merge two components into one). We use it here to detect — in a single pass over the edge list — exactly which edge closes the first cycle.

1 Initialize: Create a parent array of size n+1 where parent[i] = i. Every node starts as its own component — it is its own root. Also initialize rank[i] = 0 for union-by-rank.
2 For each edge [u, v] in order: Compute find(u) and find(v) using path compression — follow parent pointers up to the root, and compress the path on the way back.
3 Cycle check: If find(u) == find(v), both nodes already share the same root — they're in the same component. Adding this edge creates a cycle. Return [u, v] immediately.
4 Union otherwise: If roots differ, the edge safely connects two separate components. Merge them by attaching the smaller-rank root under the larger-rank root (union by rank). Continue to the next edge.

🔑 Path Compression + Union by Rank — Why They Matter

Path compression in find(x): while recursing up to the root, set each node's parent directly to the root. This flattens the tree so future lookups are O(1). Union by rank: always attach the shorter tree under the taller one, keeping the tree shallow. Together they give amortized O(α(n)) per operation — α is the inverse Ackermann function, which is ≤ 4 for any input you'll ever encounter in practice. Effectively constant time.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At any point during processing, the Union-Find structure captures exactly the connected components formed by the edges added so far. Two nodes share the same root if and only if there is already a path between them in the current partial graph. This invariant is maintained by every union operation.

Why same-root means cycle: If we're about to add edge [u, v] and find(u) == find(v), there is already a path from u to v through previously added edges. The new edge [u, v] would create a second path between u and v — by definition, that is a cycle. We return [u, v] immediately.

Why processing in order finds the "last" cycle-forming edge: The problem guarantees exactly one extra edge. There is exactly one cycle. All edges that are part of the cycle are "candidates." When we process edges left to right and encounter the first cycle-closing edge, that edge is the last one in the cycle's sequence as presented by the input — because all earlier edges in the cycle were safely added without detecting a cycle. This is why we can return on first detection.

Correctness after n−1 edges: A tree has n−1 edges. Once we've successfully added n−1 edges without finding a cycle, all n nodes are fully connected. The very next edge (the n-th one) must connect two already-connected nodes — it must close a cycle. This guarantees the algorithm always returns before or at the last edge.

⚠️ Common Mistake: Skipping Path Compression

Without path compression, find() degenerates to O(n) per call in the worst case (a long chain of parent pointers: 1→2→3→...→n). With n edges each requiring O(n) find, the overall algorithm becomes O(n²). Always implement path compression — parent[x] = find(parent[x]) on the way back from recursion — to maintain O(α(n)) amortized.

⚠️ Common Mistake: Using find(u) == find(v) AFTER union

Always check find(u) == find(v) before calling union. After union, they will always share the same root — so checking after gives a meaningless "always true" result. The cycle check must happen first: find, check same root, then conditionally union.

4 The Code (Annotated) ▾

Solution 1 — Union-Find with Path Compression + Union by Rank (Optimal, Interview Pick)

def findRedundantConnection(edges):
n = len(edges) ← n edges → n nodes (labeled 1..n)
parent = list(range(n + 1)) ← parent[i]=i: each node is its own root initially
rank = [0] * (n + 1) ← rank[i]: height of subtree rooted at i (for union)
def find(x): ← find root with path compression
if parent[x] != x: ← if x is not the root, recurse upward
parent[x] = find(parent[x]) ← path compression: point x directly to root
return parent[x] ← return root representative
def union(x, y): ← merge two components; return False if already same
rx, ry = find(x), find(y) ← find both roots
if rx == ry: return False ← same root = cycle; signal failure to caller
if rank[rx] < rank[ry]: ← union by rank: attach smaller tree under larger
parent[rx] = ry ← rx's subtree goes under ry
elif rank[rx] > rank[ry]:
parent[ry] = rx ← ry's subtree goes under rx
else: ← equal ranks: arbitrary choice, increment winner
parent[ry] = rx; rank[rx] += 1
return True ← successfully merged two distinct components
for u, v in edges: ← process each edge in the given order
if not union(u, v): ← if union returns False: same root → cycle
return [u, v] ← this is the redundant edge; return it
⚠️ Why allocate parent with size n+1?

Nodes are labeled 1 to n (1-indexed). Allocating list(range(n+1)) gives indices 0..n where we simply ignore index 0. This avoids constant node - 1 offsets everywhere. A common bug is using 0-indexed arrays for 1-indexed node problems, causing off-by-one errors in parent lookups.

💡 Complexity Summary

Union-Find (optimal): Time O(n·α(n)) ≈ O(n), Space O(n) for parent/rank arrays
DFS per edge: Time O(n²) worst case — each DFS takes O(n) and we run up to n DFS calls
α(n) is the inverse Ackermann function: grows so slowly that α(n) ≤ 4 for any input in the observable universe

5 🔬 Algorithm Deep-Dive: Union-Find / DSU ▾

Origin: Union-Find was described by Galler and Fischer (1964) and subsequently optimized by Hopcroft and Ullman (1973) and Tarjan (1975). Tarjan proved that path compression combined with union by rank achieves an amortized bound of O(α(n)) per operation — nearly constant for all practical purposes.

🔐 Core Invariant

The parent array encodes a forest of trees, one tree per connected component. The root of each tree is the "champion" (representative) of its component. Two nodes are in the same component if and only if find(x) == find(y). Path compression ensures future lookups stay instant by flattening the tree toward the root on every find.

Why it works — the algebra of merging: When we union two components with roots rx and ry, we make one root the parent of the other. This is safe because all nodes in rx's component still have rx (or a descendant of rx) on the path to the new merged root. Path compression handles the rest — the next find from any node in the merged tree will follow at most two hops.

Generalization across problem types: Union-Find appears wherever you need to dynamically track connected groups: number of connected components, merging accounts by shared email, detecting redundant edges, satisfiability of equality equations. The template is always: initialize → find with path compression → union by rank → check same root.

When to recognize Union-Find in an interview:

  • Problem involves grouping or merging elements dynamically as you process input
  • Query: "are X and Y in the same group?" must be answered repeatedly and efficiently
  • Detecting cycles in an undirected graph (not directed — use DFS/topological sort for directed graphs)
  • "Connected components" appears in the problem description
  • Offline queries — you don't need to undo merges (no "split" operation needed)
⚠️ Union-Find vs DFS for Cycle Detection

Use Union-Find when building the graph edge-by-edge and need to detect cycles incrementally (online). Use DFS/BFS when you have the full graph upfront and need to find or reconstruct specific cycle members. Union-Find tells you which edge closes the first cycle; it does not reconstruct the cycle path itself.

6 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Pattern

Keywords: "tree + extra edge", "cycle in undirected graph", "connected components built incrementally". When a problem asks you to detect cycles while adding edges one by one, think Union-Find. When it needs the specific edge that closes the first cycle, Union-Find with per-edge check is the cleanest solution.

Step 2: Propose DFS First, Then Optimize

"A naive approach: for each edge [u,v], run DFS to check if v is already reachable from u. O(n) per edge × n edges = O(n²). Correct, but slow. We can optimize to O(n·α(n)) using Union-Find — process each edge in one pass, check same root instead of running DFS."

Step 3: Explain Union-Find Clearly

"Union-Find tracks connected components. Adding an edge between nodes in different components merges them. If they're already in the same component (same root), the edge creates a cycle — return it. Process edges in order, return the first cycle-closing edge."

Step 4: Proactively Mention Optimizations

"I'll implement path compression to flatten the tree on each find call, and union by rank to keep the forest shallow. Together these give amortized O(α(n)) — effectively O(1) per operation. The complete algorithm runs in O(n·α(n)) ≈ O(n)."

🗣️ What to Say to the Interviewer

"The problem asks: which edge, when added, creates the first cycle in the graph?
A tree needs exactly n-1 edges; the n-th edge must close a cycle.
I'll use Union-Find to track connected components incrementally.
For each edge [u,v]: call find(u) and find(v). If they share the same root,
they're already connected — this edge is redundant, return it.
Otherwise, union them and continue. Time O(n·alpha(n)), Space O(n).
Path compression + union by rank keep each find/union call effectively constant."
🔮 Likely Follow-up Questions
  • "What if it's a directed graph?" → Union-Find doesn't capture direction; use DFS with in-stack tracking or in-degree analysis (see LC #685)
  • "How would you reconstruct the actual cycle?" → From the redundant edge [u,v], run DFS from u to v in the graph without that edge to find the cycle path
  • "Can there be multiple valid answers?" → No — with exactly n edges and n nodes, there's exactly one cycle, but if multiple edges form the same cycle, return the one appearing last in input
  • "How do you handle edge removal (undo)?" → Standard Union-Find is union-only. For dynamic connectivity with deletions, use Link-Cut Trees (offline) or segment tree + DSU (online)
