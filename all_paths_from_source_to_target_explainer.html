LeetCode #797

All Paths From Source to Target

Given a DAG of n nodes, find all paths from node 0 (source) to node n-1 (target) and return them in any order.

Medium Graph DFS + Backtracking Time: O(2ⁿ · n) Space: O(n)
1 Understanding the Problem ▾

We have a Directed Acyclic Graph (DAG) with n nodes (0 through n-1), given as an adjacency list: graph[i] is the list of nodes you can reach directly from node i. Our mission is to find every possible path from node 0 to node n-1.

💡 Key Insight

The graph is a DAG — no cycles. This is critical: it means DFS will always terminate without needing a visited set. We can freely explore every branch by building a path as we go deeper and backtracking when we return. Reach the target → save a copy of the current path.

✓ Example: graph = [[1,2],[3],[3],[]]

0 → [1, 2]
1 → [3]
2 → [3]
3 → [] (target = n-1 = 3)

Output: [[0,1,3], [0,2,3]]

⚠ What Makes This Tricky

• Need ALL paths, not just shortest
• Path must be rebuilt carefully
• Copying at target is mandatory
• Up to 2ⁿ⁻¹ paths in worst case
• Must undo choices after recursion

Edge Cases: If n=1, source equals target → return [[0]]. A linear chain has exactly one path. A "diamond" graph (like our example) has two. A complete DAG can have exponentially many.

2 The Algorithm: DFS + Backtracking ▾

We use Depth-First Search with backtracking. Imagine exploring a maze where you leave a trail of breadcrumbs — when you reach the exit, you write down the trail. Then you pick up the last breadcrumb and try a different turn. This is backtracking: explore fully, undo the choice, try the next one.

1 Initialize: Start with path = [0] (source pre-seeded). Call dfs(0).
2 Base case — target reached: If node == n-1, append path[:] (a copy!) to result and return. The copy is critical — without it, the stored path would mutate as we backtrack.
3 Recursive case — explore neighbors: For each neighbor of the current node: (a) Choose: path.append(neighbor). (b) Explore: dfs(neighbor). (c) Unchoose: path.pop().
4 Why no visited set? The graph is a DAG — edges only go "forward" in topological order. We can never revisit an ancestor, so there are no cycles to guard against. Every path terminates.
DFS Call Tree — graph = [[1,2],[3],[3],[]]
dfs(0, path=[0])
├── Choose 1 → path=[0,1] → dfs(1)
│   └── Choose 3 → path=[0,1,3] → dfs(3)
│       └── BASE CASE: node 3 == target → result.append([0,1,3]) ✓
│       Unchoose 3 → path=[0,1]
│   Unchoose 1 → path=[0]
└── Choose 2 → path=[0,2] → dfs(2)
    └── Choose 3 → path=[0,2,3] → dfs(3)
        └── BASE CASE: node 3 == target → result.append([0,2,3]) ✓
        Unchoose 3 → path=[0,2]
    Unchoose 2 → path=[0]

Final result: [[0,1,3], [0,2,3]]
🔑 The Choose → Explore → Unchoose Template

Every backtracking problem follows exactly this 3-step pattern. Learning to recognize it turns any "find all…" problem into a mechanical exercise. The path.pop() after dfs(neighbor) is the undo — it restores the path to its state before we made that choice.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every point during DFS, path contains exactly the nodes on the route from node 0 to the current node, in traversal order. When dfs(node) returns, path is in the exact same state it was before dfs(node) was called — this is the backtracking guarantee.

Why no visited set is needed: In a DAG, edges form a strict topological ordering. Every call to dfs(neighbor) processes a node that is "topologically after" the current one. We can never return to a previously visited ancestor, so the recursion always terminates. Adding a visited set would actually be wrong for this problem — it would prevent us from visiting node 3 via two different paths!

Why we must copy the path: In Python, lists are objects. If we write result.append(path), we store a reference — every stored "result" now points to the same mutable list. As backtracking removes elements, all stored results become corrupted. path[:] creates a snapshot, frozen at this moment.

⚠️ The #1 Bug in Backtracking Problems

result.append(path) stores a reference to the same list object. After full backtracking, every entry in result will be the empty list []. Always write result.append(path[:]) — or equivalently result.append(list(path)) — at every base case where you record mutable state.

Termination proof: Each recursive call descends to a neighbor reachable from the current node. In a DAG, path length is strictly increasing in topological order and bounded by n. Therefore recursion depth ≤ n, and the algorithm always terminates.

4 The Code (Annotated) ▾

Solution 1 — DFS + Backtracking (Optimal / Interview Pick)

def allPathsSourceTarget(graph): ← graph[i] = neighbors reachable from node i
result = [] ← will collect all complete paths
path = [0] ← start at source (node 0), pre-seeded
target = len(graph) - 1 ← target is always the last node (n-1)
def dfs(node): ← closure: path and result are shared by reference
if node == target: ← BASE CASE: we've reached the destination
result.append(path[:]) ← COPY the path — critical! path[:] is a snapshot
return ← stop here; caller will backtrack
for neighbor in graph[node]: ← iterate over every outgoing edge
path.append(neighbor) ← CHOOSE: extend path with this neighbor
dfs(neighbor) ← EXPLORE: recurse into the full subtree
path.pop() ← UNCHOOSE: undo → path is restored
dfs(0) ← launch DFS from node 0
return result ← all paths collected
💡 Why a Closure vs Passing path as Parameter

Using a closure (path defined in outer scope) means we share one list object and mutate it in-place — only O(n) memory for the current path. If we passed path + [neighbor] as a parameter, we'd create a new list at every recursion level, using O(n²) total memory across the call stack. The closure + pop approach is more efficient.

Solution 2 — Iterative DFS (Stack of Paths)

def allPathsSourceTarget(graph):
target = len(graph) - 1 ← same target node
stack = [[0]] ← stack holds ENTIRE PARTIAL PATHS
result = []
while stack: ← process until all paths explored
path = stack.pop() ← get the deepest partial path
node = path[-1] ← current node = last element
if node == target: ← target reached: save this path
result.append(path) ← no copy needed — path is a fresh list already
for neighbor in graph[node]: ← extend this path to each neighbor
stack.append(path + [neighbor]) ← NEW list created — not the same object
return result
⚠️ Memory Trade-off: Recursive vs Iterative

The iterative approach stores entire partial paths on the stack — at peak, it may hold O(2ⁿ) paths simultaneously, consuming O(2ⁿ · n) space. The recursive approach reuses one path array and uses O(n) space for the call stack. For large sparse DAGs, the recursive approach is far more memory-efficient. Both have the same time complexity.

💡 Complexity Summary

DFS + Backtracking (Recursive): Time O(2ⁿ · n) — up to 2ⁿ⁻¹ paths each of length ≤ n. Space O(n) call stack + O(n) current path = O(n) excluding output.
Iterative DFS: Same time; O(2ⁿ · n) space (stores all partial paths simultaneously).
Note: The output itself takes O(2ⁿ · n) space — we can't do better since we must enumerate all paths.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify the Structure

"This is a graph traversal problem. Key signal: 'find all paths' — this immediately points to DFS + backtracking. Problems asking for a single shortest path use BFS; problems asking for ALL solutions use backtracking."

Step 2: Exploit the DAG Property

"The graph is a DAG — no cycles. I can safely omit a visited set. In fact, if I used a visited set, I'd get wrong answers: node 3 in our example is reachable via two separate paths, and marking it 'visited' after the first path would prevent the second from being found."

Step 3: Design the Backtracking Template

"The pattern is universal: path.append(node) → recurse → path.pop(). Base case: when we reach the target, record a copy of path. The pop restores path to its pre-call state, so the next iteration sees the unmodified path."

Step 4: Nail the Copy Gotcha

"One subtle bug: appending path (not path[:]) stores a reference to the same list. As we backtrack, all recorded results would be corrupted. Always use path[:] or list(path) to freeze a snapshot at the base case."

🗣️ What to Say to the Interviewer

"Since the problem says 'find all paths', I'll use DFS with backtracking.
I maintain a 'path' list starting with [0]. When I reach node n-1, I record a copy of path.
For each neighbor: append it, recurse, then pop it — this is the choose/explore/unchoose pattern.
Because the graph is a DAG, I don't need a visited set — cycles are impossible, so every DFS terminates.
Important: at the base case I use path[:] not path, because lists are mutable references in Python.
Time: O(2ⁿ · n) — exponential paths, each of length n. Space: O(n) excluding the output."
🔮 Likely Follow-up Questions
  • "What if the graph had cycles?" → Track a visited set; mark on entry, unmark on backtrack.
  • "Find the shortest path only?" → Use BFS instead of DFS.
  • "Find paths that avoid certain nodes?" → Add a condition before each recursive call.
  • "Count paths without listing them?" → Return an integer counter at the base case instead of appending.
  • "Weighted graph — path with min cost?" → Track running cost; update best-so-far at the target.

7 Summary & Key Takeaways ▾
DAG means no cycles → DFS terminates without a visited set
The backtracking template is always: Choose → Explore → Unchoose
Always copy mutable state (path[:]) when recording at base cases
Recursive DFS uses O(n) space; iterative stores all partial paths at O(2ⁿ·n)
Do NOT use a visited set here — it would incorrectly prune valid paths
ApproachTimeSpace (excl. output)Notes
DFS + Backtracking ✓ O(2ⁿ · n) O(n) Interview pick — elegant, memory-efficient
Iterative DFS (stack of paths) O(2ⁿ · n) O(2ⁿ · n) Same time, worse space for large graphs

🎯 When to Recognize This Pattern
  • "Find ALL paths / combinations / subsets / permutations" → Backtracking
  • Graph with no cycles (DAG) + path enumeration → DFS without visited set
  • Graph with cycles + path enumeration → DFS with visited set (unmark on backtrack)
  • "Count paths" (not enumerate) → DP on DAG (memoize) — much faster for counting
