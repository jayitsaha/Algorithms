LeetCode #106

Construct Binary Tree from Inorder and Postorder Traversal

Given the inorder and postorder traversals, reconstruct the original binary tree.

Medium Trees Build Right Subtree First Time: O(n) Space: O(n)
1 Understanding the Problem ▾

You are given two arrays that record how a binary tree was visited. Inorder traversal visits: Left subtree → Root → Right subtree. Postorder traversal visits: Left subtree → Right subtree → Root. From only these two sequences, you must rebuild the exact original tree — including its shape and every node's parent–child relationship.

💡 Key Insight

The last element of postorder is always the root of the current subtree. Once we know the root, we find it in the inorder array: everything to its left in inorder belongs to the left subtree; everything to its right belongs to the right subtree. This gives us a perfect divide-and-conquer split — then we recurse.

Consider this tree and its traversals:

     3
    / \
   9  20
      / \
     15  7
Inorder (L → Root → R)
[9], 3, [15, 20, 7]

Root (3) splits array into left and right groups.

Postorder (L → R → Root)
[9], [15, 7, 20], 3

Root (3) is the last element — always.

Edge cases: An empty array means an empty tree (return None). A single-element array is a leaf node — it has no children. The algorithm requires all node values to be distinct (LeetCode guarantees this).

2 The Algorithm: Recursive Root-Split with HashMap ▾

Postorder and inorder encode complementary information: postorder says "which node is root"; inorder says "which nodes are left vs right." We exploit both at every level of recursion.

1 Pre-build a hash map from inorder values to their indices. This lets us find the root's position in O(1) instead of O(n), giving O(n) overall instead of O(n²).
2 Base case: If the current inorder window is empty (in_left > in_right), return None.
3 Identify root: Read postorder[post_idx] — this is the root of the current subtree. Decrement post_idx (consuming postorder right-to-left).
4 Split inorder: Look up the root's position mid in the hash map. Inorder elements at [in_left..mid-1] form the left subtree; [mid+1..in_right] form the right subtree.
5 Recurse right first, then left. When reading postorder right-to-left, the sequence is Root → (right subtree roots) → (left subtree roots). So we build right first to consume postorder in the correct order.
🔑 Why left_size = mid (the inorder split index)

Postorder places all left subtree nodes before all right subtree nodes (then root last). So if there are mid nodes in the left subtree (those at indices 0..mid-1 from inorder's perspective), the first mid elements of the current postorder slice are exactly the left subtree's postorder. The next batch is the right subtree's postorder. This count-based alignment is the engine of the algorithm.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every recursive call, the window inorder[in_left..in_right] and the next unconsumed elements of postorder (tracked by post_idx) describe exactly the same set of nodes — one self-contained subtree. The element at postorder[post_idx] is always the root of that subtree. The inorder window tells us how to partition nodes into left and right groups.

Uniqueness guarantee: Every binary tree with distinct node values has a unique pair of inorder + postorder traversals. Reconstructing from them is always unambiguous — there is exactly one tree consistent with both sequences. (Note: preorder + postorder alone is not sufficient for unique reconstruction.)

Why we need both traversals: With postorder alone we can find the root at every level, but we cannot split left from right — e.g., postorder [9, 3] is consistent with 9 being either a left or right child of 3. Inorder resolves this: if 9 appears before 3, it must be in the left subtree.

⚠️ Requires Distinct Values

If values could repeat, finding the root's index in the inorder array would be ambiguous — you might match the wrong occurrence. LeetCode #106 guarantees distinct values. For duplicate-value trees, you'd need an index-based approach rather than value-based lookup.

Naive approach — O(n²)

At each level, scan inorder to find the root — O(n) per level. With n levels in the worst case (skewed tree), total is O(n²). Also creates O(n²) array slices in memory.

Optimal — O(n) with HashMap

Pre-compute val → index map once in O(n). Each lookup is O(1). Use index pointers instead of array slicing — O(1) per level. Total: O(n) time, O(n) space.

Solution 1 — Naive Recursive with Array Slicing (O(n²) time) — Brute Force

def buildTree(inorder, postorder):
if not inorder or not postorder: ← Base case: empty subtree → return None
return None
root_val = postorder[-1] ← Last element of postorder = current root
root = TreeNode(root_val)
mid = inorder.index(root_val) ← O(n) linear scan — bottleneck
root.left = buildTree(
inorder[:mid], ← Left inorder slice (elements before root)
postorder[:mid]) ← Left postorder slice (first mid elements)
root.right = buildTree(
inorder[mid+1:], ← Right inorder slice (elements after root)
postorder[mid:-1]) ← Right postorder (elements mid to second-to-last)
return root

Solution 2 — Optimal with HashMap + Index Pointers (O(n) time) — Interview Pick ✓

def buildTree(inorder, postorder):
idx_map = {v:i for i,v in enumerate(inorder)} ← Pre-map val→index, O(n) once
post_idx = [len(postorder) - 1] ← Mutable pointer; list lets inner func mutate it
def build(in_left, in_right): ← in_left..in_right = current inorder window
if in_left > in_right: ← Empty window → no node here
return None
root_val = postorder[post_idx[0]] ← Consume root from postorder right-to-left
post_idx[0] -= 1 ← Advance pointer before recursing
root = TreeNode(root_val)
mid = idx_map[root_val] ← O(1) lookup — split index in inorder
root.right = build(mid+1, in_right) ← RIGHT first! Mirrors postorder's right-before-root
root.left = build(in_left, mid-1) ← Then left
return root
return build(0, len(inorder) - 1)
⚠️ Why recurse on right BEFORE left?

When we consume postorder from right-to-left, we see: root first, then the right subtree's root, then the left subtree's root (because postorder is L→R→Root, so reversed is Root→R→L). If we recursed left-first, we'd consume the postorder pointer in the wrong order. Building right first keeps the pointer aligned with the reversed postorder sequence.

💡 Complexity Summary

Naive: Time O(n²) due to .index() at each of n levels; Space O(n²) from array copies
Optimal (HashMap): Time O(n) — one hash map build + n O(1) lookups; Space O(n) for hash map + O(h) recursion stack (h = tree height, O(n) worst case for skewed tree)

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Know Your Traversal Properties

"In postorder, the last element is always the root. In inorder, the root splits left from right subtrees." If you know these two facts, the algorithm falls out naturally.

Step 2: Propose the Naive Solution First

"My first instinct: take postorder[-1] as root, find it in inorder, slice both arrays, recurse. This is O(n²) due to .index() at each level and O(n²) space for the slices."

Step 3: Optimize with a Hash Map

"To improve: precompute a val → inorder_index map — now lookup is O(1). Instead of slicing arrays, pass index pointers. Use a mutable counter for the postorder pointer. Total: O(n) time, O(n) space."

Step 4: Explain the Right-First Recursion

"When consuming postorder right-to-left, we see the right subtree's root before the left subtree's root. So we must recurse right first to consume the pointer in order."

🗣️ What to Say to the Interviewer

"Two key observations: postorder[-1] is always the current root; inorder splits nodes into left vs right.
Naive approach: slice arrays recursively — O(n²) time. I'll optimize with a hashmap for O(1) lookups.
I'll precompute val→index for inorder, then use a single mutable pointer into postorder consumed right-to-left.
I recurse on the right subtree first because reading postorder backwards gives: Root, Right roots, Left roots.
Overall: O(n) time, O(n) space for the hashmap plus O(h) stack depth."
🔮 Likely Follow-up Questions
  • "What if values are not unique?" → Use parent index context or serialize with null markers to break ambiguity
  • "Construct from Preorder + Inorder?" → First element of preorder is root; same technique, recurse left first (#105)
  • "Can you do Preorder + Postorder?" → Not uniquely in general (ambiguous when node has only one child)
  • "Iterative version?" → Use an explicit stack; push nodes and track when to connect left vs right children
  • "What is the space complexity for a balanced tree?" → O(log n) stack depth + O(n) hash map = O(n) total

7 Summary & Key Takeaways ▾
The last element of postorder is always the root of the current subtree — at every level of recursion
Finding root in inorder splits the remaining elements into left vs right subtrees
left_size = split index (mid), which lets us slice postorder correctly without a second scan
Pre-build a val→index hash map for O(1) inorder lookup: O(n) total vs O(n²) naive
Recurse on right subtree before left when consuming postorder right-to-left — critical implementation detail
Requires distinct node values; preorder+inorder solves the same problem with first-element root convention
