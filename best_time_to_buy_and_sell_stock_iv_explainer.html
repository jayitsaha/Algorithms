LeetCode #188

Best Time to Buy and Sell Stock IV

Given an integer k and a price array, find the maximum profit using at most k transactions.

Hard Dynamic Programming K Transactions DP State Machine Time: O(kn) Space: O(k)
1 Understanding the Problem ▾

You receive an integer k (max transactions allowed) and prices[] where prices[i] is the stock price on day i. One transaction = one buy + one subsequent sell. You may hold at most one stock at a time. Find the maximum profit.

💡 Key Insight

At any day, for each transaction slot j, you are in exactly one of two states: HOLD (you bought and are waiting to sell) or CASH (you sold or have not bought yet). Tracking the best profit achievable in each of these 2k states — and the transitions between them — gives the exact answer.

Example: prices=[3,2,6,5,0,3], k=2

Buy day 1 (price=2), Sell day 2 (price=6) → +4
Buy day 4 (price=0), Sell day 5 (price=3) → +3
Total = 7 ✓

Why greedy fails with k limit

k is a hard budget. Every buy+sell
uses one transaction. Greedy would
grab every uptick, ignoring the limit.
We need DP to budget wisely.

Critical optimization: if k ≥ n//2, we effectively have unlimited transactions — any profitable move can be taken. We fall back to a simple greedy: sum all positive day-over-day differences. This prevents O(kn) runtime from being unnecessarily large when k is huge.

Edge cases: empty prices → 0, k=0 → 0, strictly decreasing prices → 0, k ≥ n//2 → unlimited greedy.

2 The Algorithm: K-Transaction State Machine DP ▾

We maintain two arrays, each of size k+1: hold[j] tracks the best profit when currently holding a stock while having started j buy operations, and cash[j] tracks the best profit when not holding after completing j full transactions.

1 Unlimited escape hatch: If k ≥ n//2, return sum of all positive price diffs (greedy).
2 Initialize: hold[j] = −∞ (no stock held before any day), cash[j] = 0 for all j. cash[0] = 0 always (free baseline).
3 For each price, iterate j from k down to 1: This inner loop order prevents "using a transaction twice in one day."
4 Update hold[j]: max(hold[j], cash[j−1] − price). Either keep holding, or buy today using the profit from completing j−1 transactions.
5 Update cash[j]: max(cash[j], hold[j] + price). Either stay in cash, or sell today to complete the j-th transaction.
6 Return cash[k]: maximum profit from at most k completed transactions without holding any stock.
📐 The Recurrence Relations
hold[j] = max(hold[j], cash[j−1] − price[i]) ← buy into j-th transaction cash[j] = max(cash[j], hold[j] + price[i]) ← sell, completing j-th transaction

3 Why It Works: The Invariant ▾
🔐 The Invariant

After processing each day's price, hold[j] = the best achievable profit if we are currently holding a stock having used j buys total; cash[j] = the best achievable profit if we are not holding having completed exactly j full transactions.

Why optimal substructure holds: Today's optimal HOLD[j] is exhaustively: either "keep the stock I bought before" (hold[j] from yesterday) or "buy today from the best cash state after j−1 completions" (cash[j−1] − price). These are the only two options — no other history matters. Same logic applies to CASH[j].

Why j iterates high-to-low: When we update hold[j], we use cash[j−1]. If j went low-to-high, we might update cash[j−1] before using it for hold[j] — that would let us buy and sell in the same day on the same transaction, which isn't allowed. High-to-low preserves yesterday's cash[j−1] when computing hold[j].

⚠️ The Classic Confusion: hold uses cash[j-1], not cash[j]

When you buy into transaction j, you come from j−1 completed transactions (cash[j−1]). When you sell in transaction j, you arrive at j completed transactions (cash[j]). If you accidentally use cash[j] when buying, you allow "pre-using" a transaction you haven't done yet — producing incorrect results.

Why the unlimited escape hatch works: With n prices, you can make at most n//2 non-overlapping transactions (each needs a buy-day and a sell-day). When k is large enough, every single profitable buy-sell can be captured. At that point, the problem reduces to: sum all consecutive day-over-day gains where price increases.

4 The Code (Annotated) ▾
def maxProfit(k: int, prices: list[int]) -> int:
n = len(prices) ← how many days we have
if n == 0 or k == 0: return 0 ← no prices or no transactions allowed
if k >= n // 2: ← unlimited transactions escape hatch
return sum(max(prices[i]-prices[i-1], 0) for i in range(1, n)) ← grab every uptick
hold = [-float('inf')] * (k + 1) ← -inf means "not holding" (impossible state initially)
cash = [0] * (k + 1) ← 0 profit before any transaction; cash[0]=0 is the base
for price in prices: ← process each day left-to-right
for j in range(k, 0, -1): ← j from k down to 1 (preserve yesterday's cash[j-1])
hold[j] = max(hold[j], cash[j-1] - price) ← keep holding OR buy today for txn j
cash[j] = max(cash[j], hold[j] + price) ← stay cash OR sell today (complete txn j)
return cash[k] ← best profit with at most k completed transactions
⚠️ Why range(k, 0, -1) and not range(1, k+1)?

Going high-to-low means when we compute hold[j] = cash[j−1] − price, the cash[j−1] value is still yesterday's (not today's updated version). If we went low-to-high, we might update cash[1] first, then use that updated cash[1] for hold[2] — effectively buying and selling in one day.

💡 Space Complexity: O(k) not O(kn)

We only maintain two 1D arrays of length k+1, not a 2D table of k×n. Each price iteration overwrites the same arrays. This is the "rolling array" technique — valid because each day only depends on the previous day's values.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Stock Problem Family

Signal: "buy and sell stock", "at most k transactions", "maximize profit". Immediately think state machine DP. This is a generalization of LC 121 (k=1), LC 122 (unlimited), LC 123 (k=2). Naming the family earns instant points.

Step 2: State the unlimited escape hatch immediately

"If k ≥ n//2, I can treat this as unlimited transactions." This shows you understand the problem at a deep level and avoids potential TLE from doing O(kn) when k is astronomically large.

Step 3: Define states clearly before writing code

"At each day, for each transaction count j from 1 to k, I track two states: hold[j] = best profit while holding, having started j buys; cash[j] = best profit not holding, having completed j sells."

Step 4: Derive transitions, then argue iteration order

Write the recurrences. Then say: "I iterate j from k down to 1 so that when I update hold[j] using cash[j−1], the cash[j−1] still reflects yesterday's value — preventing double-counting a transaction in one day."

Step 5: Mention space optimization

A 2D table would be O(kn) space. With rolling arrays it's O(k). Proactively mentioning this often clinches the hire signal.

🗣️ What to Say to the Interviewer

"This is a generalization of the stock problems. My first observation: if k is at least n//2, I can take every profit, so I sum all positive differences — O(n) solution."

"For smaller k, I'll use state machine DP. For each transaction slot j and each day, I track hold[j] — best profit if I'm currently holding having started j buys — and cash[j] — best profit not holding after completing j sells."

"Recurrences: hold[j] = max(hold[j], cash[j−1] − price) and cash[j] = max(cash[j], hold[j] + price). I iterate j from k down to 1 each day so cash[j−1] is always yesterday's value."

🧠 Follow-Up Questions to Expect
  • What if k=1? Simplifies to: track min price seen so far, answer = max(price − min_price).
  • What if k=2? Same DP but with only 2 slots — 4 variables: hold1, cash1, hold2, cash2.
  • With cooldown? Add a 3rd "resting" state; transitions change to account for cooldown day.
  • With transaction fee? Subtract fee when selling: cash[j] = max(cash[j], hold[j] + price − fee).

7 Summary & Key Takeaways ▾
If k ≥ n//2, reduce to unlimited transactions — greedy sum of positive differences
State machine DP: hold[j] and cash[j] for each transaction slot j = 1..k
Buying into slot j uses cash[j−1] (profit from j−1 prior completed transactions)
Iterate j from k down to 1 each day — prevents double-using a transaction in one day
O(k) space with rolling arrays — no need for a 2D table
The answer is cash[k] — best profit after at most k completed transactions
