LeetCode #493

Reverse Pairs

Given an integer array nums, return the number of reverse pairs — pairs (i, j) where i < j and nums[i] > 2 × nums[j].

Hard Sorting Merge Sort + Count Time: O(n log n) Space: O(n)
1 Understanding the Problem ▾

A reverse pair is any pair of indices (i, j) where i < j AND nums[i] > 2 × nums[j]. Two conditions must both hold: the left element appears earlier in the array, and it is more than twice the right element — strict inequality, not ≥. We must count how many such pairs exist.

💡 Key Insight

This is a generalized inversion counting problem. A standard inversion is any (i, j) with i < j and nums[i] > nums[j]. Here the threshold is stricter: nums[i] must exceed 2 × nums[j]. Both problems — and the general form nums[i] > k × nums[j] — share the same efficient solution: count qualifying cross-pairs during merge sort, exploiting the fact that each half is sorted when we reach the merge step.

✓ Example: nums = [1, 3, 2, 3, 1]

(i=3,j=4): 3 > 2×1=2 ✓
(i=2,j=4): 2 > 2×1=2? No (equal)
Answer = 2

✗ Brute Force Fails at Scale

Check all pairs (i,j): i < j

O(n²) → Time Limit Exceeded

Edge Cases: Empty array or single element (answer is 0 — no pairs possible). All same values (e.g., [3,3,3]): is 3 > 2×3=6? No, so answer is 0. Negative numbers: be careful — 2 × (−5) = −10, and −1 > −10 is true, so negatives can create pairs. In Python, no overflow risk; in C++/Java, multiply to long before comparison.

2 The Algorithm: Modified Merge Sort ▾

Merge sort splits an array in half, recursively sorts each half, then merges them. The key observation is that at the moment we merge, both halves are already sorted. This lets us count qualifying cross-pairs (one element from each half) in O(n) with a two-pointer sweep — far better than O(n²) pairwise comparison.

1 Base case: If subarray has 0 or 1 elements, return 0. A single element has no pair partners within itself.
2 Divide: Find mid = (lo + hi) // 2. Recursively call merge_count(lo, mid) and merge_count(mid, hi). Each call sorts its subarray in-place and returns the count from within that half.
3 Count cross-pairs (BEFORE merging): Both halves are now sorted. Use pointer i over the left half and pointer j starting at mid (start of right half). For each i, advance j while nums[i] > 2 × nums[j]. Add j − mid to count (all right elements at indices [mid, j) are valid partners for this i). Since left is sorted ascending, j is monotonically non-decreasing — never backtrack.
4 Merge (AFTER counting): Standard merge — produce sorted subarray from the two sorted halves. This prepares the correct sorted state for the parent call's counting step. The merge changes array contents, which is why counting must come first.
🔑 Why Count Before Merge (Not During)?

The merge operation interleaves elements from the two halves. Once merged, the original "left half" and "right half" positions are lost. The counting step needs to know which elements came from left vs right. By counting first (on still-separated sorted halves), then merging, both steps operate on clean, well-defined data.

💡 Why the Two-Pointer Is O(n) Per Merge Level

Pointer i scans left half (m elements) left-to-right. Pointer j scans right half (m elements) left-to-right. Since left[i] only increases as i advances, the condition left[i] > 2 × right[j] can only become harder to satisfy as i increases — so j never needs to go back. Total j movements across all i values: at most m. So the counting loop is O(n) per merge level. With O(log n) levels: O(n log n) total.

3 Why It Works: The Invariant ▾
🔐 The Invariant

When merge_count(lo, hi) is called: all reverse pairs (i, j) with both i, j inside [lo, hi) will be counted exactly once — either by a recursive sub-call (if both indices are in the same half) or by the current call's cross-pair counting step (if i is in [lo, mid) and j is in [mid, hi)). After the function returns, nums[lo:hi] is sorted ascending.

Every pair counted exactly once: For any pair (i, j) with i < j globally, they fall into one of three categories:

  • Both i and j in the left half → handled by the recursive left call
  • Both i and j in the right half → handled by the recursive right call
  • i in left half, j in right half → handled by the cross-pair counting at the current level

Since every pair falls into exactly one category, no pair is double-counted or missed.

Why the two-pointer gives correct counts: When we process i in ascending order over the sorted left half, and j scans forward through the sorted right half, the key monotonicity holds: if nums[i₁] ≤ nums[i₂] and nums[i₁] > 2×nums[j], then nums[i₂] > 2×nums[j] too. So when we move from i₁ to i₂, j can only stay or move forward. This means count += j − mid is always correct — all elements right[mid..j) qualify for this left[i].

⚠️ Critical Mistake: Trying to Count During the Merge

During the standard merge operation, we compare left[i] ≤ right[j] to decide which element to place next. This is a completely different comparison from left[i] > 2 × right[j]. If you try to combine counting with merging in a single loop, one comparison corrupts the other — you'll either miss pairs or count the wrong ones. Always use two separate, sequential loops: count first, then merge.

4 The Code (Annotated) ▾

Solution 1 — Brute Force O(n²) — Understand the problem, then optimize

def reversePairs_brute(nums):
count = 0 ← Accumulate qualifying pairs
n = len(nums)
for i in range(n): ← Try every left index i
for j in range(i+1, n): ← Try every right index j > i
if nums[i] > 2*nums[j]: ← Check the reverse-pair condition
count += 1
return count ← O(n²): works but TLEs on large n

Solution 2 — Merge Sort + Count During Merge (Interview Pick) — O(n log n)

def reversePairs(nums):
def merge_count(lo, hi): ← Sorts nums[lo:hi], returns cross-pair count
if hi - lo <= 1: ← Base case: 0 or 1 elements, already sorted
return 0
mid = (lo + hi) // 2 ← Find the midpoint
count = merge_count(lo, mid) ← Left half: sort + count pairs within
count += merge_count(mid, hi) ← Right half: sort + count pairs within
# ── COUNT cross-pairs (halves are now sorted) ──
j = mid ← j starts at the beginning of the right half
for i in range(lo, mid): ← For each element in the LEFT half
while j < hi and nums[i] > 2*nums[j]: ← Advance j while pair is valid
j += 1 ← j never goes backward (monotone property)
count += j - mid ← All right[mid..j) are valid partners for left[i]
# ── MERGE the two halves (separate step) ──
nums[lo:hi] = sorted(nums[lo:hi]) ← Standard merge; prepares sorted state for parent
return count
return merge_count(0, len(nums))
⚠️ Why count += j − mid and not j − lo?

j starts at mid (the start of the right half, not lo). After the while loop, j is the first right-half index where the pair condition FAILS. So the valid right indices are exactly [mid, j), which is j − mid elements. Using j − lo would include left-half indices — that's wrong.

💡 Complexity Summary

Brute Force: Time O(n²), Space O(1)
Merge Sort + Count (Interview Pick): Time O(n log n), Space O(n) — for the sorted() call's temporary copy
BIT / Fenwick Tree: Time O(n log n), Space O(n) — coordinate compress then prefix-sum queries; more code
Segment Tree: Same complexity; even more code; rarely needed in interviews

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Pattern

The phrase "count pairs (i, j) where i < j and some comparison of values" is the signal for inversion-counting. This family of problems (standard inversions, reverse pairs, count of smaller to the right) all have O(n log n) solutions based on divide-and-conquer or balanced BSTs/BITs.

Step 2: Propose Brute Force and Name the Bottleneck

"The naive approach is O(n²): nested loops checking every pair. For n = 50,000 that's 2.5 billion operations — way too slow. I need to count many pairs at once instead of one-by-one."

Step 3: Articulate the Core Insight

"If both halves are sorted, I can use a two-pointer to count cross-pairs in O(n). As the left pointer advances (elements only get larger), the right pointer also only advances — it's monotone. Merge sort gives me sorted halves for free as part of its recursion."

Step 4: Clarify the Two-Pass Design

"I must count BEFORE I merge. The counting pass needs the original left/right separation. The merge destroys that separation. So: count cross-pairs first (two-pointer over sorted halves), then merge (standard)."

🗣️ What to Say to the Interviewer

"This is an inversion-counting variant. Brute force O(n²) is too slow.
Key insight: merge sort gives me two sorted halves at every level.
With sorted halves, a two-pointer counts cross-pairs in O(n) per level.
j is monotone: as left[i] increases, j only advances — never backtracks.
count += j - mid for each i: all right[mid..j) are valid pairs with left[i].
IMPORTANT: count pass must come BEFORE the merge pass (separate loops).
Total: O(n log n) time, O(n) space for the merge buffer."
🔮 Likely Follow-up Questions
  • "Change 2× to k×?" → Same code, just replace 2*nums[j] with k*nums[j]
  • "Standard inversions (nums[i] > nums[j])?" → Same structure with k=1 — this is #315
  • "Can you use a BIT instead?" → Yes: iterate right to left, for each j query "how many inserted values > 2×nums[j]", then insert nums[j]. Needs coordinate compression
  • "Integer overflow?" → Python: fine. Java/C++: cast nums[j] to long before multiplying

7 Summary & Key Takeaways ▾
Reverse Pairs is generalized inversion counting — replace nums[i]>nums[j] with nums[i]>2×nums[j]
Merge sort provides sorted halves at every recursion level — exploit this for O(n) cross-pair counting
Count BEFORE merge (separate loops) — merging destroys the left/right separation needed for counting
j pointer is monotone: as left[i] increases across sorted left half, j only moves forward
count += j − mid (not j − lo): j starts at mid, so j − mid counts valid right-half elements
Python: no overflow. C++/Java: cast to long before 2 × nums[j] to avoid 32-bit overflow
