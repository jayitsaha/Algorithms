LeetCode #23

Merge k Sorted Lists

Given an array of k linked lists each sorted in ascending order, merge them into one sorted linked list and return it.

Hard Heaps Min Heap with K Pointers Time: O(N log k) Space: O(k)
1 Understanding the Problem ▾

You have k sorted linked lists and need to stitch them into one globally sorted linked list. Each individual list is already in ascending order — your job is to interleave nodes from all k lists correctly without losing any nodes.

Input: 3 sorted lists

List 0: 1 → 4 → 5
List 1: 1 → 3 → 4
List 2: 2 → 6

Output: one merged list

1 → 1 → 2 → 3 → 4 → 4 → 5 → 6

💡 Key Insight

Because each list is already sorted, the next node to add to the output must be the minimum of all current front-elements. We never need to look past the first node of each list — the global minimum is always among those k candidates. This observation makes the problem tractable without sorting everything from scratch.

Edge cases: k = 0 (return null), k = 1 (return the list unchanged), some lists are null/empty (skip them during seeding), all values equal (still works, order preserved by index tie-breaker), lists of wildly different lengths.

2 The Algorithm: Min-Heap with K Pointers ▾

We use a min-heap (priority queue) as our "tournament bracket." At any moment the heap holds the current front-element of each active list — at most k items. Popping gives us the globally smallest in O(log k), and pushing the next candidate from that list restores the invariant.

1 Seed the heap: For each non-null list head, push (node.val, idx, node) into the min-heap. The idx counter is a tie-breaker so Python never compares ListNode objects directly. After seeding: at most k elements in the heap.
2 Pop the global minimum: heapq.heappop() extracts the root in O(log k). This is the next node for our output list. Append it: tail.next = node; tail = tail.next.
3 Advance that list's pointer: If node.next exists, push (node.next.val, idx, node.next) into the heap. This "refills" the heap with the new front of that list. If node.next is None, the list is exhausted — no push.
4 Repeat until heap is empty: Each pop gives the globally smallest remaining node. When the heap empties, every node has been processed and appended. Return dummy.next to skip the sentinel.

🔑 Python Heap Trick: Tuple (val, idx, node)

Python's heapq compares tuples lexicographically. If two nodes have equal val, it would try to compare the ListNode objects — raising TypeError. We insert a unique integer idx as the second element. Since indices are always distinct, comparison never reaches the third element (the node). Safe and clean.

3 Why It Works: The Invariant ▾
🔐 The Loop Invariant

At every iteration of the while loop, the heap contains exactly one node per non-exhausted list — always the current front node of that list. Therefore the heap root is always the globally smallest unprocessed value across all k lists. Popping it is always the correct next output.

Why O(N log k): There are N nodes total. Each node is pushed exactly once and popped exactly once. Each push/pop costs O(log k) because the heap never exceeds k elements. Total: 2N × O(log k) = O(N log k). Compare to brute force O(N log N) — when k is small (say k=3, N=10^6), this is dramatically faster since log k ≈ 1.6 vs log N ≈ 20.

Invariant maintenance: Initially we push all k non-null heads → invariant holds. Each iteration: pop one node (its list loses a representative), then push node.next if it exists (its list regains a representative). If node.next is None, that list is exhausted — no representative is correct. The invariant is preserved at every step by construction.

⚠️ Common Mistakes

1. Forgetting null-check on seed: If any list head is None, pushing it crashes. Always if node: before seeding.
2. Forgetting null-check on advance: If node.next is None and you push it, you'll crash or get wrong results. Check before pushing.
3. Comparing ListNode directly: Push (val, idx, node) not (val, node). The idx prevents TypeError.

def mergeKLists(lists):
dummy = ListNode(0) ← Sentinel; result = dummy.next at the end
tail = dummy ← Growing tail of the output list
heap = [] ← Min-heap of (val, idx, node)
idx = 0 ← Unique tie-breaker counter
for node in lists: ← Seed: one entry per non-null list head
if node: ← Skip null lists
heapq.heappush(heap, (node.val, idx, node))
idx += 1 ← Increment tie-breaker
while heap: ← Process until all lists exhausted
val, _, node = heapq.heappop(heap) ← Pop global min in O(log k)
tail.next = node ← Append to output
tail = tail.next ← Advance output tail
if node.next: ← Push next from same list (if exists)
heapq.heappush(heap, (node.next.val, idx, node.next))
idx += 1
return dummy.next ← Skip sentinel, return real head

💡 Complexity Summary

Brute Force: Time O(N log N), Space O(N) — ignores the sorted property
Min-Heap ✓: Time O(N log k), Space O(k) — heap never exceeds k nodes
Divide & Conquer: Time O(N log k), Space O(log k) recursion — same asymptotic
N = total nodes, k = number of lists. When k << N: O(N log k) is much faster than O(N log N).

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the "k-way merge" signal

Any problem saying "merge k sorted [lists / arrays / streams]" is the k-way merge pattern. The key observation: because inputs are sorted, you only ever need to compare their current front elements. You never need to look beyond position 0 of each list — that's the crucial insight that makes a simple heap sufficient.

Step 2: Propose brute force, then critique it

"The naive approach: collect all N values into an array, sort it O(N log N), rebuild the list. Correct, but it throws away the fact that all inputs are already sorted. We're doing unnecessary work."

Step 3: Exploit sorted structure with a heap

"Since each list is sorted, the next output is always the minimum of the k current fronts. A min-heap maintains that minimum in O(log k) — much better than O(k) linear scan when k is large. Heap never exceeds k elements so space is O(k), not O(N)."

Step 4: Mention the Python gotcha proactively

"In Python I push (val, idx, node) tuples. Without the idx, equal values would cause Python to compare ListNode objects directly — TypeError. The idx tie-breaker costs O(1) and makes the code safe."

🗣️ What to Say to the Interviewer

"I'll use a min-heap seeded with each list's head node — at most k items.
Each iteration: pop the minimum (O(log k)), append it to output,
then push its next node (if any) — keeping the heap at most k items.
N total nodes × O(log k) per pop = O(N log k) time, O(k) space.
Python trick: push (val, idx, node) tuples to avoid ListNode comparison errors.
I'll also handle null list heads by checking if node before seeding."

7 Summary & Key Takeaways ▾
The next output is always the minimum of k current list fronts — a min-heap finds it in O(log k)
Heap size never exceeds k — one representative per active list at all times
Push (val, idx, node) tuples in Python — the idx tie-breaker prevents TypeError on equal values
O(N log k) is dramatically better than O(N log N) when k is small (e.g. k=10, N=10^6)
Divide & conquer gives the same O(N log k) with O(log k) space — worth mentioning as alternative
