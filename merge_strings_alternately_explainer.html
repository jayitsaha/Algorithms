LeetCode #1768

Merge Strings Alternately

You are given two strings word1 and word2. Merge them by adding letters in alternating order, starting with word1. Append remaining letters from the longer string to the end.

Easy Two Pointers Same Direction Interleave Time: O(m+n) Space: O(m+n)
1 Understanding the Problem ▾

Picture a zipper: teeth from the left track and right track interlock one at a time, alternating perfectly. When one track is shorter, the remaining teeth from the longer track just extend past the end without a partner. That physical zipper is exactly this algorithm.

More concretely: take word1[0], then word2[0], then word1[1], then word2[1]… and so on. Once either string is exhausted, everything that remains in the other string gets appended verbatim to the result.

💡 Key Insight

The output always starts with word1[0]. At every even position (0, 2, 4 …) in the result sits a character from word1; at every odd position (1, 3, 5 …) sits a character from word2 — until one string runs out, after which the tail fills the remaining positions.

Edge cases to consider: one string is empty (result = the other string entirely); both strings length 1 (result = word1[0] + word2[0]); word1 is the longer string (the word1 tail is appended, not word2).

2 The Algorithm: Two-Pointer Same-Direction Interleave ▾

We use two pointers i and j, both starting at index 0 and advancing in the same direction (forward) through word1 and word2 respectively. This is structurally the same as the merge step in merge sort — except instead of picking the smaller element we strictly alternate. There are two clean implementations; we focus on the explicit two-pointer approach because it makes the pointer motion visible.

1 Initialize: i = 0 into word1, j = 0 into word2, res = [] (list, not string — to avoid O(n²) concatenation).
2 Interleave loop: While both i < len(word1) and j < len(word2), append word1[i] then word2[j], then advance both pointers by 1. This appends one complete pair per iteration.
3 Append word1 tail: res.append(word1[i:]). If word1 is already exhausted, word1[i:] returns "" — harmless. Otherwise it appends all remaining characters of word1 in one shot.
4 Append word2 tail: res.append(word2[j:]). Same logic — at most one of steps 3 and 4 appends non-empty content. They are mutually exclusive, so both can be called unconditionally.
5 Join and return: return "".join(res). One linear scan to build the final string. O(m+n) total.
🔑 Single-Loop Variant (compact)

Alternatively, loop a single index k from 0 to max(m, n) - 1, and use if k < m: res.append(word1[k]) and if k < n: res.append(word2[k]). The bounds checks naturally handle both the interleave and the tail — no explicit tail logic needed. Same O(m+n) complexity.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At the start of iteration k of the while loop, res contains exactly 2k characters: the k-th pair is at positions 2k−2 (from word1) and 2k−1 (from word2), for all pairs so far. After the loop, res contains 2·min(m,n) characters. The tail appends bring it to the full m+n.

Why steps 3 and 4 are mutually exclusive: The while loop exits when at least one pointer reaches its string's end. At that point, the other pointer is either also at its end (equal lengths → both tail appends produce "") or still has characters (unequal lengths → exactly one tail is non-empty). This means you never double-count a character.

Why no index-out-of-bounds occurs: Inside the loop, both i and j are only accessed after the guard conditions confirm they are in-bounds. The tail slicing word1[i:] never raises an error in Python regardless of i — past-end slices return "".

⚠️ The O(n²) String Concatenation Trap

In Python, strings are immutable. Writing result += word1[i] inside a loop allocates a brand-new string on every iteration — O(k) work at step k, totalling O(n²). Always accumulate into a list and call "".join() once at the end. This is O(n) total and is the idiomatic Python approach.

Order of appends matters: We append word1[i] before word2[j] per pair, because the problem specifies "starting with word1." Swapping the append order produces an incorrect result where word2 leads each pair.

4 The Code (Annotated) ▾

Solution 1 — Two-Pointer Explicit (Interview Pick, O(m+n))

def mergeAlternately(word1: str, word2: str) -> str:
i, j = 0, 0 ← Two forward-moving pointers, one per string
res = [] ← List avoids O(n²) string concat; join once at end
while i < len(word1) and j < len(word2):← Both must still have chars
res.append(word1[i]) ← word1 always leads each pair (per problem spec)
res.append(word2[j]) ← word2 follows immediately in the same iteration
i += 1 ← Advance word1 pointer
j += 1 ← Advance word2 pointer
res.append(word1[i:]) ← Tail of word1; "" if word1 already exhausted
res.append(word2[j:]) ← Tail of word2; "" if word2 already exhausted
return "".join(res) ← O(m+n) linear scan. Done!
💡 Python Shortcut: Unconditional Tail Appends

Because word1[i:] returns "" when i == len(word1), you can append both tails unconditionally without any if guards. Exactly one will contribute real characters; the other contributes the empty string, which join ignores. This halves the number of conditional branches in the tail logic.

Solution 2 — Single-Loop with Bounds Checks (compact)

def mergeAlternately(word1: str, word2: str) -> str:
res = []
for k in range(max(len(word1), len(word2))):← Up to the longer string
if k < len(word1): res.append(word1[k]) ← Guard prevents IndexError
if k < len(word2): res.append(word2[k]) ← Guard prevents IndexError
return "".join(res)

Solution 3 — zip_longest (Pythonic library approach)

from itertools import zip_longest
def mergeAlternately(word1: str, word2: str) -> str:
return "".join(a + b for a, b in zip_longest(word1, word2, fillvalue=""))← One line
⚠️ Interviewer Expectation on zip_longest

Using zip_longest is elegant but obscures the two-pointer thinking. In an interview, propose it as the "Pythonic alternative" after presenting the explicit two-pointer version. The explicit version is easier to reason about, extend, and port to other languages.

💡 Complexity Summary

All three solutions: Time O(m+n) — every character visited exactly once.
Space: O(m+n) — unavoidable, the output itself has m+n characters.
String concat in loop (naïve): O((m+n)²) — reject this immediately.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Interleave Signal

"Alternating order" and "merge" are the keywords. Whenever you see "take one from each, in turns," think same-direction two pointers. This is the same conceptual structure as merge sort's merge — just with alternation instead of minimum-first selection. State this aloud to show pattern recognition.

Step 2: Identify the Main Complication

The strings can differ in length. This means your loop must handle a "leftover tail." Ask yourself whether to handle it inside the loop (single-loop with guards) or after the loop (explicit tail appends). Either works — say which you're choosing and why.

Step 3: Nail the String Building Pattern

Before writing a single line of code, say: "I'll collect characters into a list and join once at the end to stay O(n) — string concatenation inside a loop is O(n²) in Python." This single sentence distinguishes you from candidates who don't know Python internals.

Step 4: Walk Through an Example

Pick a case where word2 is longer than word1 (like word1="ab", word2="pqrs"). Trace the loop — it exits after two iterations with i=2 (exhausted), j=2. Then the tail word2[2:]="rs" gets appended. This shows you've considered the unequal-length case explicitly.

🗣️ What to Say to the Interviewer

"I'll use two pointers i and j, both starting at index 0, moving through word1 and word2 in sync.
While both strings still have characters, I take one from word1 then one from word2 — word1 always leads.
Once the shorter string is exhausted, the loop ends. I then append whatever remains of word1 and word2.
Since Python string slicing past the end returns empty string, both tail appends are unconditional and safe.
I build into a list and join once at the end — O(m+n) time, O(m+n) space, which is optimal."
🔮 Likely Follow-up Questions
  • "Merge k strings alternately?" → Generalize: use a list of (string, index) pairs; round-robin with a pointer per string
  • "Interleave with ratio 2:1?" → Take 2 from word1 per loop iteration, then 1 from word2
  • "Can you do it without a result array?" → In Python, not without O(n²) cost due to string immutability. In C++ with a mutable char array you could write in-place if given extra space
  • "Is this related to merge sort?" → Yes! The merge step of merge sort is the same two-pointer pattern but picks the minimum instead of alternating

7 Summary & Key Takeaways ▾
Two pointers in the same direction — i walks word1, j walks word2, both advance together in each iteration
The while loop handles the interleave; the two tail appends handle unequal lengths — mutually exclusive, unconditional
Use list + join, never string concatenation in a loop — the former is O(n), the latter O(n²)
word1 always leads each pair; reversing the append order inside the loop gives the wrong answer
Single-loop with bounds checks and zip_longest are valid compact alternatives — same complexity, different readability trade-offs
