LeetCode #876

Middle of the Linked List

Given the head of a singly linked list, return the middle node. If there are two middle nodes, return the second middle node.

Easy Linked List Fast & Slow Pointers Time: O(n) Space: O(1)
1 Understanding the Problem ▾

A singly linked list is a chain of nodes where each node only knows about the next node — there is no length field, no index, no way to jump directly to position n/2. You can only traverse forward, one node at a time.

The problem asks: "Without converting the list to an array, how do I find the middle node efficiently?"

💡 Key Insight

If one pointer moves at twice the speed of another, by the time the fast pointer exits the list, the slow pointer has covered exactly half the distance — it is at the middle. This is the classic Tortoise and Hare race.

Odd-length list: [1 → 2 → 3 → 4 → 5]

5 nodes → one clear middle
Middle index = ⌊5/2⌋ = 2
Answer: node(3) at index 2

Even-length list: [1 → 2 → 3 → 4 → 5 → 6]

6 nodes → two middles (nodes 3 and 4)
Problem says: return the second
Answer: node(4) at index 3

Edge Cases to consider: A single-node list (head is the only and therefore middle node), a two-node list (return the second node), long lists where we must not traverse twice, and even vs odd length (which affects where "middle" lands).

2 The Algorithm: Tortoise & Hare (Fast-Slow Pointers) ▾

We use two pointers, both starting at head. The slow pointer ("tortoise") moves one node per iteration. The fast pointer ("hare") moves two nodes per iteration. When the fast pointer can no longer advance two steps, the slow pointer is at the middle.

1 Initialize: Set slow = head and fast = head. Both point to the first node — they race from the same starting line.
2 Loop while safe: Continue as long as fast is not None AND fast.next is not None. Both conditions are required: fast must exist to check fast.next, and fast.next must exist to safely do fast.next.next.
3 Advance: Move slow = slow.next (one hop) and fast = fast.next.next (two hops). The 2:1 speed ratio is the key invariant.
4 Return slow: Loop exits when fast can't take two more steps. At this point, slow is exactly at the middle. For even-length lists, slow lands on the second middle, which is exactly what the problem requires.

🔑 Why Not Just Count and Walk?

The naive approach: traverse the entire list counting n nodes, then reset to head and walk n//2 steps. This is correct, O(n) time, O(1) space — but it visits every node twice. The fast/slow pointer approach traverses the list once, visiting each node at most once, giving us the middle in a single pass.

3 Why It Works: The Invariant ▾
🔐 The Invariant

After every iteration: fast has moved exactly twice as many steps as slow. If slow is at position k (from head), fast is at position 2k. When fast reaches position n (or can't proceed further because n is exhausted), slow is at position n/2 — the middle.

Proof by case analysis:

  • Odd length n (e.g., 5 nodes, 0-indexed 0–4): The loop runs 2 iterations. slow moves to index 2. fast: 0 → 2 → 4. fast.next = null → loop exits. slow = index 2 = ⌊5/2⌋. That is the unique middle. ✓
  • Even length n (e.g., 6 nodes, 0-indexed 0–5): The loop runs 3 iterations. slow moves to index 3. fast: 0 → 2 → 4 → then fast.next = index 5 ≠ null, so one more: fast → 6 (null). Loop exits. slow = index 3 = the second middle. ✓
⚠️ The Loop Condition Bug Trap: "while fast" vs "while fast and fast.next"

If you write while fast: alone, then for an even-length list, fast lands on the last node (not null) after the last safe iteration. The loop enters again, and fast = fast.next.next tries to dereference fast.next which is None — crash! You must guard with fast.next is not None.

Single-node edge case: fast.next is None from the start, so the loop condition fails immediately. We return slow = head, which is the only node — correct.

Two-node edge case: fast = head. Check: fast ✓, fast.next (node 2) ✓. Enter loop: slow → node 2, fast → null. Now fast fails the check. Return slow = node 2. That is the second (and second-middle) node — correct.

4 The Code (Annotated) ▾

Solution 1 — Fast & Slow Pointers (Optimal, Interview Pick)

def middleNode(head):
slow = fast = head ← Both start at the first node
while fast and fast.next: ← fast exists AND fast.next exists (safe to do .next.next)
slow = slow.next ← slow advances 1 step
fast = fast.next.next ← fast advances 2 steps; safe because fast.next checked above
return slow ← fast hit end → slow is exactly at middle
⚠️ Order of Updates: slow first or fast first?

It does not matter here — we read slow.next and fast.next.next independently, then assign both. There is no dependency between the two assignments. Either order works. But do NOT do slow = slow.next; fast = slow.next (that would use the NEW slow, giving wrong results).

Solution 2 — Two-Pass Length Count (Brute Force, O(n) but 2 passes)

def middleNode(head):
n, cur = 0, head ← Count total nodes in first pass
while cur:
n += 1; cur = cur.next
cur = head ← Reset to head for second pass
for _ in range(n // 2): ← Walk n//2 steps to reach middle
cur = cur.next
return cur ← This is the middle node
💡 Complexity Summary

Fast & Slow Pointers: Time O(n), Space O(1), 1 pass — optimal for this problem
Two-Pass Count: Time O(n), Space O(1), 2 passes — correct but visits every node twice
Array Copy: Time O(n), Space O(n) — copy all to list, return list[n//2] — wasteful

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify the Structural Constraint

Linked lists have no random access. There is no list[n//2]. You must traverse. This constraint alone rules out O(1) solutions and tells you traversal is unavoidable. You then ask: "Can I find middle in one traversal instead of two?"

Step 2: Propose the Naive Solution First

"I can traverse once to count n, then traverse again n//2 steps. That's O(n) time, O(1) space — but two passes. Let me think if I can do it in one."

Step 3: The Tortoise and Hare Insight

"What if I use two pointers at different speeds? A fast pointer moving 2x faster than a slow pointer. By the time fast exits, slow has gone half as far — that's the middle! One pass, same complexity."

Step 4: Handle the Even-Length Case Explicitly

"For even-length lists there are two middles — the problem says return the second. My loop condition while fast and fast.next naturally makes slow land on the second middle. Let me trace [1,2,3,4]: slow ends at node 3, which is the second of {2,3}."

🗣️ What to Say to the Interviewer

"Linked lists have no index, so I can't do O(1) middle access.
The naive approach counts n nodes, then walks n//2 — two passes, O(n) time, O(1) space.
I can do better with fast and slow pointers.
Both start at head. Slow moves 1 step, fast moves 2 steps per iteration.
When fast can't take 2 more steps, slow is at the middle — one pass.
Loop condition: 'while fast and fast.next' guards against null dereference on the 2-step advance.
For even length, slow naturally lands on the second middle, as required.
Time: O(n). Space: O(1)."
🔮 Likely Follow-up Questions
  • "Is there a cycle in the list?" → Same fast/slow pointers — if they ever meet, there's a cycle. If fast reaches null, there's no cycle. (#141)
  • "Where does the cycle start?" → Floyd's full algorithm — after detecting meeting point, reset one pointer to head, advance both by 1 until they meet again at the cycle entrance. (#142)
  • "Is the linked list a palindrome?" → Use this problem to find middle, reverse the second half, compare both halves. (#234)
  • "Delete the middle node?" → Modify the approach to stop slow one step early so you have the node before middle. (#2095)

7 Summary & Key Takeaways ▾
Fast & slow pointers find the middle in ONE pass — 2× faster than the two-pass count approach
Loop condition must be while fast and fast.next — guards against null dereference on the two-step advance
Speed ratio 2:1 means slow travels exactly half the distance fast does — it stops at the middle
For even-length lists, slow lands on the second middle automatically — exactly what this problem requires
This is the gateway to Floyd's Cycle Detection — same pointers, different stopping condition (they meet = cycle)
