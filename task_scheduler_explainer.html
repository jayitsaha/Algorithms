LeetCode #621

Task Scheduler

Given a CPU task list and cooldown n, return the minimum number of intervals needed to complete all tasks.

Medium Greedy Max Frequency Determines Idle Time: O(n) Space: O(1)
1 Understanding the Problem ▾

You have a CPU executing tasks, each identified by a letter ('A', 'B', 'C', …). The CPU executes one task per interval. The same task type cannot run again for at least n intervals after its last execution. During any interval, the CPU can either run a task or be idle. We want to minimize total intervals (including idle time).

💡 Key Insight

The most frequent task forces the minimum number of gaps. Think of building a schedule as filling a grid: the most frequent task anchors each row, and other tasks fill in the gaps. The formula drops out naturally from this grid structure.

✓ Example: tasks=[A,A,A,B,B,B], n=2

A→B→idle→A→B→idle→A→B
= 8 intervals
idle fills gap between A's
(n=2 → must wait 2 slots)

⚠ Dense case: [A,A,B,B,C,C,D], n=2

A→B→C→A→B→C→D
= 7 = len(tasks)
enough variety → zero idle
answer = task count

Edge cases: n=0 (no cooldown → answer = len(tasks)); all tasks same type (maximum idle every slot); tasks so diverse that every cooldown slot is filled (no idle at all).

2 The Algorithm: Greedy Math Formula ▾

Rather than simulating the schedule slot by slot, we derive the answer mathematically by observing the structure of an optimal greedy schedule.

1 Count frequencies. Build a frequency table. For uppercase letters, this is at most 26 entries — effectively O(1) space.
2 Find max_f = the highest frequency. This is the task that appears most often — the bottleneck that forces idle slots.
3 Count max_count = how many tasks share frequency max_f. Multiple tasks at max frequency together fill the last "frame" of the grid.
4 Apply the formula: result = max((max_f − 1) × (n + 1) + max_count, len(tasks)). First term = grid structure including idle; second term = dense case where no idle is needed.
📐 The Core Formula
result = max( (max_f − 1) × (n + 1) + max_count, len(tasks) )

(max_f − 1) = number of complete "frames" (all but the last occurrence)
× (n + 1) = each frame is n+1 intervals wide (1 task + n cooldown slots)
+ max_count = the final partial frame (all tasks tied at max frequency)
max(…, len(tasks)) = can never schedule fewer slots than tasks

🔑 The Grid Mental Model

Picture a grid with (n+1) columns. Task A (most frequent, max_f times) appears in the leftmost column of every row. There are (max_f − 1) complete rows, then a partial last row with max_count tasks. Other tasks fill the grid left-to-right. If there are more task types than columns, tasks spill over — no idle needed — and the schedule length is just the total task count.

3 Why It Works: The Invariant ▾
🔐 The Core Invariant

The most frequent task imposes a structural minimum on the schedule. If task A has max_f = 3 and n = 2, then A must appear at positions 0, 3, 6 (at minimum). That means at least 2 × 3 = 6 intervals before the last A, plus the A itself = 7 intervals minimum just from A's constraint. Other tasks either fill those gaps or get appended at the end.

Why (max_f − 1) × (n + 1)? A must run max_f times, separated by at least n slots each time. Between the first and last A there are (max_f − 1) gaps of at least n. A "frame" = one A + its n cooldown slots = (n+1) intervals. There are (max_f − 1) such frames (the last A has no required trailing cooldown).

Why + max_count? Tasks that share max frequency also need a final occurrence in the last partial row. If A and B are both at max_f = 3, the last row is [A, B] — two tasks with no idle after them.

Why max(…, len(tasks))? Consider many diverse tasks with n=2. There are enough different task types to fill every cooldown slot — no idle at all! The grid formula might give a number smaller than len(tasks), which is impossible (each task needs at least one slot). The max() enforces this lower bound.

⚠️ Simulation Alternative (and why we avoid it here)

A correct but slower approach: simulate the schedule using a max-heap (by remaining count). Each cycle of (n+1) slots, pop the most frequent tasks, run them, decrement counts, push back. This is O(result × 26) in the worst case. The formula is O(n) in one pass — much cleaner for an interview once you understand the grid insight.

4 The Code (Annotated) ▾

Solution 1 — Greedy Math Formula (Optimal, O(n) · Interview Pick)

from collections import Counter
def leastInterval(tasks: list, n: int) -> int:
counts = Counter(tasks) ← frequency of each task type, e.g. {A:3, B:3, C:2}
max_f = max(counts.values()) ← highest frequency: bottleneck task; O(26)=O(1)
max_count = sum(1 for f in ← how many tasks share the maximum frequency
counts.values() if f == max_f) (needed for the final partial frame)
part1 = (max_f - 1) * (n + 1) ← (frames) × (frame width): body of the grid
+ max_count plus the last partial row
return max(part1, len(tasks)) ← never fewer slots than total tasks
⚠️ When does max(part1, len(tasks)) matter?

Try tasks = [A,A,B,B,C,C,D,D,E,E], n=2. max_f=2, max_count=5. part1 = (2−1)×3+5 = 8. But len(tasks)=10. Result = max(8,10) = 10. The formula "under-predicts" because there are so many diverse tasks that every cooling gap is occupied — no idle at all, and the schedule is exactly 10 slots long.

Solution 2 — Heap Simulation (Intuitive, O(n×k) time)

import heapq
def leastInterval(tasks, n):
freq = list(Counter(tasks).values()) ← frequency counts only (task labels don't matter)
max_heap = [-c for c in freq] ← negate for Python's min-heap (simulating max-heap)
heapq.heapify(max_heap)
time_elapsed = 0
while max_heap:
cycle, k = [], min(n+1, len(max_heap))← one cooling frame: up to n+1 tasks
for _ in range(k):
cycle.append(heapq.heappop(max_heap))← greedily pick most frequent available task
for c in cycle:
if c + 1 < 0: ← task still has remaining occurrences?
heapq.heappush(max_heap, c + 1)
time_elapsed += n + 1 if max_heap else len(cycle)← full frame or just remaining tasks
return time_elapsed
💡 Complexity Comparison

Math Formula: Time O(n), Space O(1) [26-letter alphabet] — derive this in your interview
Heap Simulation: Time O(result × 26), Space O(26) = O(1) — more intuitive, derive first
Both are effectively O(n) given the 26-letter constraint.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Scheduling Pattern

"Minimize total time" + "same task must wait n slots" → scheduling with cooldown. Ask yourself: what creates idle time? Answer: whichever task appears most frequently forces the most gaps. That's your bottleneck — the key greedy observation.

Step 2: Propose Simulation First

"My first approach: simulate the schedule. Use a max-heap sorted by remaining count. Each interval, pick the most frequent available task (or idle if none). This is O(result × 26) — correct but not optimal. Let me optimize."

Step 3: Derive the Formula via the Grid Insight

"Think of the schedule as a grid: (n+1) columns wide. The most frequent task anchors the left column. (max_f − 1) complete rows, each of width (n+1). Plus a final partial row of max_count tasks. Total: (max_f−1)×(n+1)+max_count. But if tasks overflow the grid → no idle → answer = len(tasks). Take the max of both."

Step 4: Verify with Two Examples

[A,A,A,B,B,B], n=2: max_f=3, max_count=2 → (3-1)×3+2=8 ✓. Dense: [A,B,C,D,E,F,G], n=2: max_f=1, max_count=7 → 0×3+7=7, len=7, max(7,7)=7 ✓.

🗣️ What to Say to the Interviewer

"The bottleneck is the most frequent task — call its count max_f.
I'll arrange the schedule like a grid: (n+1) columns, (max_f-1) complete rows, plus a last partial row.
The most frequent task anchors each row. Other tasks fill the cooldown gaps.
Formula: (max_f-1) × (n+1) + max_count. But if tasks are so diverse that every slot is filled,
we need no idle at all, so the answer is just len(tasks).
Final result: max((max_f-1)*(n+1)+max_count, len(tasks)). O(n) time, O(1) space."
🔮 Likely Follow-up Questions
  • "Output the actual schedule, not just the length?" → Construct the grid: fill left-to-right by frequency rank
  • "Different cooldowns per task type?" → Formula breaks down, use heap simulation
  • "Tasks have priorities?" → Priority queue ordered by (priority, remaining_count)
  • "What if tasks can be split?" → Preemptive scheduling variant (OS theory)
