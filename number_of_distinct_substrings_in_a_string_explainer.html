LeetCode #1698

Number of Distinct Substrings in a String

Given a string s, return the number of distinct substrings of s (including the empty substring).

Medium String Suffix Array · Rolling Hash Time: O(n²) best practical Space: O(n)
1 Understanding the Problem ▾

A substring is any contiguous sequence of characters within a string — including the empty string "". For s = "aab", the raw substrings are: "", "a", "a", "b", "aa", "ab", "aab" — but "a" appears twice, so we count it only once. The answer is 6.

💡 Key Insight

A string of length n has exactly n(n+1)/2 + 1 substrings total (counting the empty string). The challenge is counting only unique ones efficiently without materializing and comparing every pair. The three approaches — brute force set, rolling hash, and suffix array — trade implementation complexity for time complexity.

Edge Cases: empty string ("" → 1, just the empty substring), single char ("a" → 2), all-same characters ("aaaa" → n+1 = 5: "", "a", "aa", "aaa", "aaaa"), all-distinct characters (no duplicates — answer equals n(n+1)/2 + 1).

2 The Algorithm: Three Approaches ▾

There are three clean approaches ordered from simplest to most optimal.

Approach 1 — Brute Force Hash Set (O(n³) time)

1 Start with a set containing {""} (the empty string).
2 For every start index i in [0, n) and end index j in (i, n], extract s[i:j] and add to the set. Python's set hashes the string in O(length) time.
3 Return len(seen). The set automatically deduplicates.

Approach 2 — Rolling Hash (O(n²) time, O(n) space) — Interview Pick

1 Start distinct = 1 (for the empty string).
2 For each window length L from 1 to n, slide a window of size L across s. Use polynomial rolling hash to compute each window's hash in O(1) per slide.
3 Maintain a hash set for this length L. Each new hash = a new distinct substring. After all windows of length L, add |seen| to distinct.
4 Return distinct after all lengths 1 to n.

Approach 3 — Suffix Array + LCP (O(n log n) time, O(n) space)

1 Build the Suffix Array: sort all suffixes of s lexicographically. Store their start indices.
2 Build the LCP Array: for each adjacent pair in the sorted suffix array, compute their Longest Common Prefix length using Kasai's algorithm (O(n)).
3 Apply the formula: distinct = n(n+1)/2 − sum(LCP) + 1. The LCP values represent substrings shared between adjacent sorted suffixes — already counted, so subtract them once.
🔑 The interactive walkthrough below uses Rolling Hash on s = "abab"

Rolling hash is the most practical interview approach — it is explainable, implementable in ~15 lines, and generalizes to many other substring problems (longest duplicate substring, Rabin-Karp pattern matching, repeated DNA sequences).

3 Why It Works: The Invariants ▾

Why Suffix Array + LCP gives the exact count:

🔐 The Suffix Array Invariant

Every non-empty substring of s is a prefix of exactly one (or more) suffixes. When we sort suffixes, adjacent suffixes share the longest common prefixes — those are the duplicates we must remove. Suffix sa[i] of length L_i contributes exactly L_i − lcp[i] new distinct substrings (its unique prefixes not shared with the previous sorted suffix). Summing across all suffixes gives total distinct non-empty substrings.

Concrete walkthrough — s = "aab":

RankSuffix (sorted)LengthLCP prevNew distinct substrings
1"aab"303 (unique prefixes: "a","aa","aab")
2"ab"211 ("ab" only — "a" shared with prev)
3"b"101 ("b")

Total non-empty distinct = 3+1+1 = 5. + 1 for empty = 6. Formula check: n(n+1)/2 − sum(LCP) + 1 = 6 − (0+1+0) + 1 = 6. ✓

Why Rolling Hash works:

🔐 The Rolling Hash Invariant

Two strings that are equal must have the same polynomial hash (assuming no collision). When we slide a window of size L, each step computes the new hash from the old one in O(1) via:

h_new = (h_old × BASE − s[i−L] × BASE^L + s[i]) mod MOD

We "roll off" the leftmost character and "roll on" the rightmost. For each length L, we maintain a set of hashes seen so far — each new hash = one more distinct substring of that length.

Polynomial Hash Formula (window size L, starting at index i)
H(i) = s[i]·BASE^(L-1) + s[i+1]·BASE^(L-2) + ... + s[i+L-1]·BASE^0 (all mod MOD)

Slide by 1: H(i+1) = (H(i) · BASE − s[i]·BASE^L + s[i+L]) mod MOD

Each slide is O(1) — only 3 arithmetic operations!
⚠️ Hash Collisions

Two different strings may rarely produce the same hash — this is a collision. To minimize risk: use a large prime modulus (e.g., 10⁹+7) and a prime base (e.g., 31 or 131). For safety in critical code, use double hashing (two independent (BASE, MOD) pairs). The collision probability per pair drops to ~1/MOD² ≈ 10⁻¹⁸.

4 The Code (Annotated) ▾

Solution 1 — Brute Force Set (O(n³) time, most readable)

def countDistinct(s: str) -> int:
n = len(s) ← String length
seen = {""} ← Seed with empty string
for i in range(n): ← Every start index
for j in range(i + 1, n + 1): ← Every end index (exclusive)
seen.add(s[i:j]) ← O(j-i) to slice+hash; set deduplicates
return len(seen) ← Count including the empty string

Solution 2 — Rolling Hash (O(n²) time, O(n) space — Interview Pick)

def countDistinct(s: str) -> int:
n = len(s)
MOD = 10**9 + 7 ← Large prime; reduces collision probability
BASE = 31 ← Prime base; maps a→1, b→2, ..., z→26
distinct = 1 ← Count the empty string "" immediately
for length in range(1, n + 1): ← Process each possible substring length
seen = set() ← Hash set scoped to this length only
h = 0 ← Rolling hash accumulator
power = pow(BASE, length, MOD) ← BASE^length mod MOD (precompute for roll-off)
for i in range(n): ← Slide window across string
c = ord(s[i]) - ord('a') + 1 ← Map 'a'→1, 'b'→2, ... 'z'→26 (avoid 0!)
h = (h * BASE + c) % MOD ← Roll new right character in
if i >= length: ← Window overflows? Roll leftmost char out
lc = ord(s[i-length]) - ord('a') + 1
h = (h - lc * power) % MOD ← Subtract leftmost char's contribution
if i >= length - 1: ← Full window formed? Record the hash
seen.add(h) ← New hash = new distinct substring
distinct += len(seen) ← Count distinct substrings of this length
return distinct
⚠️ Why map 'a' → 1 (not 0)?

If 'a' maps to 0, then "a" and "aa" and "aaa" could all hash to 0 under certain BASE values (since 0×BASE^k = 0 for any k). Mapping 'a'→1 ensures every character contributes a non-zero polynomial term, eliminating this class of spurious collisions.

Solution 3 — Suffix Array + LCP (O(n log n) asymptotically optimal)

def countDistinct(s: str) -> int:
n = len(s)
# Sort suffix start indices lexicographically by suffix string
sa = sorted(range(n), key=lambda i: s[i:]) ← O(n² log n) simple; SA-IS is O(n log n)
def lcp(a, b): ← Longest Common Prefix of s[a:] and s[b:]
l = 0
while a+l < n and b+l < n and s[a+l] == s[b+l]:
l += 1
return l
# n(n+1)/2 total non-empty substrings, minus shared prefixes, plus 1 for empty
total = n * (n + 1) // 2 + 1 ← Start with every possible substring + empty
for i in range(1, n): ← For each adjacent pair in sorted suffix array
total -= lcp(sa[i-1], sa[i]) ← Subtract shared prefixes (already counted)
return total
💡 Complexity Summary

Brute Force Set: Time O(n³), Space O(n²) — enumerate all, hash each
Rolling Hash: Time O(n²), Space O(n) — O(1) per window; practical interview choice
Suffix Array + LCP: Time O(n log n), Space O(n) — optimal but complex to implement live

🔬 Algorithm Deep-Dive: Rabin-Karp Rolling Hash

Origin: Introduced by Michael O. Rabin and Richard M. Karp in 1987 for efficient string pattern matching. The core idea — treating a string as a polynomial and computing its value modulo a prime — makes hash updates O(1) per slide rather than O(L) per recompute.

Core invariant: For window starting at index i with length L:
H(i) = Σ char(s[i+k]) × BASE^(L-1-k) mod MOD, for k ∈ [0, L-1].
Slide: H(i+1) = (H(i)×BASE − char(s[i])×BASE^L + char(s[i+L])) mod MOD. Only 3 operations — O(1).

Why it works: Polynomial hashing is a surjective map from strings to integers. Distinct polynomials evaluated at the same BASE (mod MOD) produce the same result only if BASE is a root of their difference polynomial — with probability at most (L-1)/MOD per pair, negligible for large MOD.

When to recognize it: "Distinct substrings," "duplicate substrings," "pattern matching in O(n+m)," or "find repeated k-length windows." The rolling update is the tell — any problem where you slide a window and need to identify/deduplicate contents efficiently.

Generalization: 2D rolling hash extends to matrix/grid pattern matching. Double hashing (two (BASE, MOD) pairs) reduces collision probability to ~1/MOD² ≈ 10⁻¹⁸ — effectively zero for competitive programming.

🔬 Algorithm Deep-Dive: Suffix Array + LCP Array

Origin: Suffix Arrays were introduced by Manber and Myers (1993) as a space-efficient replacement for Suffix Trees. The combination of SA + LCP (using Kasai's 2001 algorithm) is the foundation of most modern competitive programming string algorithms.

Core invariant: Every substring of s is a prefix of some suffix. Sorting suffixes groups all substrings that share a common prefix — adjacent sorted suffixes have the "most similar" prefixes. The LCP array measures exactly how much they share.

The formula explained: suffix sa[i] has len_i = n − sa[i] substrings (its prefixes). But the first lcp[i] of them were already counted as prefixes of sa[i-1] (the previous suffix in sorted order). So this suffix contributes exactly len_i − lcp[i] new substrings. Sum over all → total distinct non-empty.

SA-IS / DC3: The simple sorting approach is O(n² log n) (sorting n strings of average length n/2). SA-IS (Suffix Array — Induced Sorting) builds the SA in O(n) time and is one of the most elegant algorithms in computer science. In interviews, the simple sort is acceptable for small n.

When to recognize it: Any problem requiring reasoning about ALL substrings simultaneously — "longest repeated substring," "most frequent substring," "minimum unique substring," "number of distinct substrings" — the SA+LCP structure handles them all in O(n log n) or better.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify the Core Task

"We have O(n²) possible substrings. We need to count the distinct ones. The naive approach generates them all and uses a hash set to deduplicate — O(n³) time since each substring takes O(length) to hash."

Step 2: Propose Brute Force First

"The simplest: nested loops with a set. seen.add(s[i:j]) for all i, j. O(n³) time, O(n²) space. Correct and fast enough for small n. Let me optimize."

Step 3: Recognize the Rolling Hash Optimization

"The bottleneck is hashing each substring fresh — O(length). If I process substrings by fixed length and use polynomial rolling hash, each new window updates in O(1). This gives O(n²) total — n lengths, n windows each."

Step 4: Mention Suffix Array as Optimal

"For O(n log n), I would build a suffix array, compute LCP between adjacent sorted suffixes, then apply: distinct = n(n+1)/2 + 1 − sum(LCP). Each LCP value subtracts the shared prefixes already counted. Complex to implement live but optimal."

🗣️ What to Say to the Interviewer

"A string of length n has n(n+1)/2 non-empty substrings. I want to count the distinct ones.
Brute force: generate all O(n²) substrings, add to a set → O(n³) total (hashing each costs O(length)).
Better: rolling hash. Fix length L, slide window across s — each step updates hash in O(1).
Track seen hashes per length. New hash = new distinct substring. Sum across all L from 1 to n.
Total time O(n²), space O(n) per length. Add 1 at the start for the empty string.
Optional optimal: suffix array + LCP in O(n log n) using the formula distinct = n(n+1)/2 + 1 − sum(LCP)."
🔮 Likely Follow-up Questions
  • "What about hash collisions?" → Double hashing with two (BASE, MOD) pairs; probability ~1/MOD².
  • "Find the longest duplicate substring?" → Binary search on length + rolling hash (#1044).
  • "Distinct substrings of exactly length k?" → Single pass of rolling hash for that length — O(n).
  • "How is Kasai's algorithm for LCP built?" → Use the inverse suffix array; linear scan with a "h" variable that decreases at most n times total.

7 Summary & Key Takeaways ▾
A string of length n has n(n+1)/2 + 1 total substrings (including empty). Count distinct by deduplicating.
Rolling hash reduces per-window hashing from O(n) to O(1) by incrementally updating via: h_new = (h_old × BASE − left_char × power + right_char) mod MOD.
Suffix Array + LCP formula: distinct = n(n+1)/2 + 1 − sum(LCP). Each LCP value subtracts already-counted shared prefixes.
Always map characters to [1, 26] (not [0, 25]) in polynomial hashing to avoid zero-value collisions.
Rolling hash generalizes immediately to: find duplicate substrings, Rabin-Karp search, binary-search for longest repeated substring.
