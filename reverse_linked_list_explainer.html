LeetCode #206

Reverse Linked List

Given the head of a singly linked list, reverse the list, and return the reversed list's head.

Easy Linked List Iterative Pointer Reversal Time: O(n) Space: O(1)
1 Understanding the Problem ▾

A singly linked list is a chain of nodes. Each node holds a value and a next pointer to the node after it. The final node's next is None. We can only travel forward — each node doesn't know who came before it.

Reversing means flipping every next pointer so the chain runs the opposite direction. The original tail becomes the new head.

💡 Key Insight

Because we can only travel forward, we must redirect each node's pointer to its predecessor as we walk the list. The catch: the moment we overwrite curr.next, we lose our link to the rest of the list. So we must save the next node BEFORE redirecting. This single observation gives us the whole algorithm.

Before: 1 → 2 → 3 → 4 → 5 → None

1.next = 2
2.next = 3
3.next = 4
4.next = 5
5.next = None

After: 5 → 4 → 3 → 2 → 1 → None

5.next = 4
4.next = 3
3.next = 2
2.next = 1
1.next = None ← new tail

Edge Cases: empty list (head = None → return None), single node (head.next = None → return head unchanged), two-node list (the most common off-by-one mistake in pointer re-wiring).

2 The Algorithm: Iterative Three-Pointer Reversal ▾

We use exactly three pointers: prev (behind), curr (current), and nxt (lookahead). Think of laying a single-direction road in reverse: you pick up the road segment behind the current car and place it ahead, reversing the direction one segment at a time.

1 Initialize: prev = None, curr = head. We initialize prev to None because once reversed, the original head will be the new tail — and tails must point to None.
2 Save the successor: nxt = curr.next. Before touching curr.next, save it. This is non-negotiable — after we redirect, curr.next will no longer point forward, so nxt is our only way to continue.
3 Redirect the pointer: curr.next = prev. This is the actual reversal. The current node now points backward instead of forward. The original forward connection to nxt is intentionally severed here.
4 Advance both pointers: prev = curr, then curr = nxt. Slide the window one position to the right. prev now points to the freshly-reversed node; curr is the next to process.
5 Repeat until done: When curr becomes None, we've processed every node. prev is now at the original tail — which is the new head. Return prev.

⚠️ Order of Operations is Strict

The four assignments inside the loop must happen in exactly this sequence: (1) save nxt → (2) redirect curr.next → (3) advance prev → (4) advance curr. Any swap causes bugs: if you redirect before saving, the rest of the list is gone; if you advance prev before redirecting, you create a cycle pointing back to prev.

3 Why It Works: The Invariant ▾
🔐 The Loop Invariant

At the start of every iteration: all nodes before curr have already had their next pointers reversed. prev points to the most recently reversed node (the "rightmost" member of the growing reversed chain). The sub-list from curr onwards is still intact in its original order.

Why this guarantees correctness: When curr becomes None, the invariant tells us all nodes have been reversed. prev holds the last node we processed — the original tail — which is now the new head. Returning prev is correct.

Edge case verification via the invariant:

  • Empty list: curr = None at start — loop never runs — return prev = None. Correct.
  • Single node: One iteration: curr.next = None (it was already None), prev = node. Return prev. Correct.
  • Two nodes: After first iteration, node 1 points to None; after second, node 2 points to node 1. prev = node 2 = new head. Correct.
🚨 Classic Bug: Returning curr Instead of prev

When the loop exits, curr is None — one position past the end. prev is at the last node (new head). Many beginners instinctively return curr because that's the pointer they've been tracking progress with, but curr has stepped off the list. Always return prev.

4 The Code (Annotated) ▾

Solution 1 — Iterative Three-Pointer (Interview Pick)

def reverseList(head):
prev = None ← Predecessor of curr; starts None (head becomes the tail, so tail.next = None)
curr = head ← The node we are currently reversing; starts at head
while curr: ← Keep going until we fall off the end of the list
nxt = curr.next ← Save successor FIRST — after redirect, curr.next is gone
curr.next = prev ← THE REVERSAL: point this node backward
prev = curr ← Advance prev to the node we just reversed
curr = nxt ← Advance curr to the node we saved earlier
return prev ← curr is None (past end); prev is the original tail = new head
💡 Why Only 4 Lines Inside the Loop?

The elegance of this algorithm is that exactly 4 pointer assignments — save, redirect, advance, advance — are sufficient to reverse any node. No conditionals, no length computation, no extra allocations. One pass, O(1) space, O(n) time.

Solution 2 — Recursive (elegant alternative)

def reverseList(head):
if not head or not head.next: ← Base: empty list OR last node → it IS the new head, return it
return head
new_head = reverseList(head.next) ← Recurse: reverse the tail; new_head = original last node
head.next.next = head ← The node after us should now point BACK to us
head.next = None ← Cut our forward link (we'll be the tail once all frames unwind)
return new_head ← Bubble the real new head (original tail) all the way up
⚠️ Recursive vs Iterative in Interviews

Both are valid. Use iterative first — it demonstrates O(1) space mastery. The recursive solution uses O(n) call-stack space, which can cause a stack overflow for very long lists. Offer recursive as a follow-up to show you know both approaches.

💡 Complexity Summary

Iterative: Time O(n) — one pass through n nodes. Space O(1) — only 3 pointer variables.
Recursive: Time O(n) — n frames. Space O(n) — call stack depth equals list length.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify the Core Constraint

Singly linked list → forward-only traversal. You cannot simply "read from the end" like an array. This forces an in-place approach as you walk forward. The signal: "reverse" + "linked list" + no extra space → three-pointer iterative.

Step 2: Discover the Pointer-Loss Problem

Draw nodes A → B → C on a whiteboard. Ask: what happens if you write A.next = None (redirect) without saving B first? Nodes B and C become unreachable — lost forever. This "aha" naturally leads to needing to save nxt before every redirect. Articulating this to the interviewer shows deep understanding.

Step 3: Count What You Need to Track

At any point you need: (1) where you've been (prev), (2) where you are (curr), (3) where you're going (nxt). That's exactly 3 things → 3 pointers. The loop body directly maps: save nxt, redirect curr, advance prev, advance curr.

Step 4: Always State the Return Value

Explicitly say "I return prev, not curr" and explain why. This is a well-known pitfall. Saying it proactively signals that you've thought through the post-loop state carefully, not just guessed.

🗣️ What to Say to the Interviewer

"I'll use three pointers. prev starts at None, curr at head.
Each iteration: save curr.next before I redirect — critical, or I lose the list.
Then point curr.next backward to prev. Advance prev to curr, curr to saved nxt.
When curr hits None, prev is at the original tail — the new head. Return prev.
Time O(n), Space O(1). Want to see the recursive version too?"
🔮 Likely Follow-up Questions
  • "Can you do it recursively?" → Base case on empty/last node; reverse suffix; fix pointers on unwind.
  • "Reverse from position left to right?" → Locate left-1 node; reverse the segment; reconnect endpoints. (#92)
  • "Reverse in groups of k?" → Reversal applied in k-node chunks; count and reconnect groups. (#25)
  • "Is this list a palindrome?" → Find middle (slow/fast), reverse second half, compare. (#234)

7 Summary & Key Takeaways ▾
Three pointers (prev, curr, nxt) are all you ever need — no extra data structure
Save nxt before redirecting curr.next — never lose the rest of the list
When the loop ends, curr = None (past end), prev = new head
Iterative = O(1) space; recursive = O(n) call-stack space
This exact pattern extends to partial reversal, k-group reversal, and palindrome checking
