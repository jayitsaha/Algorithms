LeetCode #45

Jump Game II

Given an integer array nums, return the minimum number of jumps to reach the last index. You can jump at most nums[i] steps forward from index i. It is guaranteed you can always reach the last index.

Medium Greedy Track Current / Next Boundary Time: O(n) Space: O(1)
1 Understanding the Problem ▾

You're standing at index 0 of an array. nums[i] tells you the maximum number of positions you can jump forward from index i — you don't have to jump that many, you can jump any amount from 1 to nums[i]. Your goal: reach the last index in the fewest total jumps.

💡 Key Insight

Think of it like a BFS on levels. From index 0 (level 0), you can reach some set of indices with 1 jump (level 1). From those, you can reach a further set with 2 jumps (level 2). The answer is the level number at which the last index first becomes reachable. The greedy trick: instead of tracking a queue, just track the farthest boundary each level can reach.

✓ Optimal: nums = [2, 3, 1, 1, 4]

Jump 1: 0 → 1 (use 1 of 2 steps)
Jump 2: 1 → 4 (use 3 of 3 steps)
Answer: 2 jumps

✗ Greedy-max fails: always jump max

Jump max: 0 → 2 (use all 2 steps)
Jump max: 2 → 3 (use 1 of 1 step)
Jump max: 3 → 4 → 3 jumps!
Suboptimal — don't always max jump

What makes this "greedy" and not just BFS? BFS would maintain a set of all reachable positions at each level. But we only need to know the farthest position reachable at the next level — we never need the intermediate positions because any position up to farthest is reachable and any of them could be a valid jumping-off point.

Edge Cases: nums = [0] (single element, 0 jumps). nums = [1,1,1] (must jump each step, 2 jumps). nums = [10,0,0,0,0] (1 jump covers everything). Problem guarantees reachability, so zeros won't strand you.

2 The Algorithm: Implicit BFS via Boundary Tracking ▾

We simulate BFS levels without storing any explicit set. We track two pointers through the array: cur_end marks where the current BFS level ends, and farthest records the best reach discovered while scanning the current level.

1 Initialize: jumps = 0, cur_end = 0, farthest = 0. We're at level 0, which spans only index 0.
2 Scan i from 0 to n−2 (stop before the last index — no jump needed from there). At each i: update farthest = max(farthest, i + nums[i]). This finds the best position we could reach with one more jump from anywhere in the current level.
3 When i == cur_end: We've exhausted the current BFS level. We must take a jump. Do: jumps += 1; cur_end = farthest. This opens the next level — every index from old_cur_end + 1 to farthest is now in scope.
4 Return jumps at end. The loop counting stops at n−2, and the final jump that makes cur_end ≥ n−1 was already incremented when we crossed that boundary.
nums = [2, 3, 1, 1, 4] — BFS level expansion:
Level 0:
idx 0 only (cur_end=0)
Level 1:
idx 1–2 (cur_end=2, farthest seen=4)
Level 2:
idx 3–4 ✓ last index reached!
🔑 Why Only Loop to n−2?

We count a jump when we cross the level boundary, not when we land at the destination. When cur_end advances to cover index n−1, that jump has already been counted. Processing index n−1 would just trigger an extra unwanted jump increment. Stopping at n−2 is the precise off-by-one prevention.

3 Why It Works: The Greedy Invariant ▾
🔐 The Invariant

After processing each level boundary, cur_end equals the maximum index reachable in exactly jumps total jumps from index 0. Every index ≤ cur_end is reachable in ≤ jumps jumps. We never undercount or overcount a level.

Why greedy is optimal here (informal proof):

Suppose an optimal solution uses k jumps. Our algorithm also takes exactly k jumps. Here's why: at each level, our algorithm records farthest as the maximum of i + nums[i] over all i in that level. Any optimal strategy must also pick a jump point within that level — and the best it can do is land at most at farthest. Our algorithm reaches at least as far as any strategy after each level, so it never needs more levels (jumps) than optimal.

Why we don't track which position to jump from: The minimum-jumps count depends only on how far each level extends, not on which specific position within the level we jump from. By taking the max over the whole level, we capture the best possible reach for the next level — regardless of where the jump originates.

⚠️ Common Mistake: "Greedy" = Always Jump Maximum Steps

Greedy here does NOT mean always jumping the full nums[i] steps from the current position. That leads to suboptimal paths (as shown in Section 1). The greedy choice is: which position within the current level allows the next level to reach the farthest? We find this by scanning all positions in the level — the maximum i + nums[i] wins.

Single-element proof: If n == 1, the loop runs over range(0) — zero iterations. We return jumps = 0. Correct: already at the destination.

4 The Code (Annotated) ▾

Solution 1 — Greedy Boundary Tracking (Interview Pick)

def jump(nums: list[int]) -> int:
jumps = 0 ← answer: how many jumps taken so far
cur_end = 0 ← rightmost index in the current BFS level
farthest = 0 ← farthest index we can reach from any position in current level
for i in range(len(nums) - 1): ← stop at n-2; no need to jump from last index
farthest = max(farthest, i + nums[i]) ← best reach from i
if i == cur_end: ← exhausted current level — must jump
jumps += 1 ← take the jump (greedy: we jump to farthest possible)
cur_end = farthest ← next level extends to best reach we found
return jumps ← minimum jumps needed
⚠️ Why update farthest BEFORE checking i == cur_end?

When i == cur_end, index i is still part of the current level and we should count its reach too. If we checked the boundary first, we'd skip including position cur_end's own jump range in farthest. Always update farthest first, then check the boundary.

Solution 2 — DP (O(n²), for understanding)

def jump(nums: list[int]) -> int:
n = len(nums)
dp = [float('inf')] * n ← dp[i] = min jumps to reach index i
dp[0] = 0 ← start: 0 jumps to reach index 0
for i in range(n): ← from each position...
for j in range(i+1, min(i+nums[i]+1, n)): ← to each reachable position j
dp[j] = min(dp[j], dp[i]+1) ← relax: one more jump from i
return dp[n-1] ← min jumps to reach last index
💡 Complexity Summary

Greedy (Optimal): Time O(n) — single scan; Space O(1) — three variables only
DP: Time O(n²) — nested loops over all reachable pairs; Space O(n) — dp array
Explicit BFS: Time O(n) — each index visited once; Space O(1) — same as greedy, just more verbose

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize "Minimum Steps" = BFS Thinking

"We want the minimum number of jumps. Minimum-steps problems often map to BFS where each BFS level represents one more step taken. Each jump expands which indices we can reach — just like BFS expands a frontier."

Step 2: Offer DP First (Baseline)

"A straightforward approach: dp[i] = minimum jumps to reach i. For each i, update all reachable j = dp[j] = min(dp[j], dp[i]+1). O(n²) time and O(n) space. Correct, but I can do better."

Step 3: Collapse BFS to Boundary Tracking

"Key insight: in BFS, I don't care which specific position in the current level I jump from — I only care how far the next level can extend. So I track cur_end (where current level ends) and farthest (best reach seen so far). When i hits cur_end, I take a jump and advance the boundary."

Step 4: Handle the Off-by-One

"I loop to n−2 (not n−1) because I never need to take a jump from the last index — reaching it is the goal. The jump that extends cur_end to cover n−1 was already counted."

🗣️ What to Say to the Interviewer

"Minimum jumps — I'll think BFS. Each BFS level = one jump.
Instead of a queue, I track cur_end: where this level ends.
While scanning the current level, I track farthest: best we can reach with one more jump.
When i == cur_end: increment jumps, set cur_end = farthest.
Loop stops at n−2 — no jump needed from the last index.
Time O(n), Space O(1)."
🔮 Likely Follow-up Questions

7 Summary & Key Takeaways ▾
Minimum jumps = implicit BFS — each jump increments the BFS level
Track two boundaries: cur_end (current level limit) and farthest (next level limit)
Jump only when you exhaust the current level (i == cur_end), never sooner
Loop to n−2, not n−1 — no jump needed from or past the last index
Update farthest before checking the boundary, to include cur_end's own range
O(n) time, O(1) space — strictly better than DP's O(n²) time
ApproachTimeSpaceNotes
DP (relax all pairs) O(n²) O(n) Intuitive; too slow for n=10⁵

Greedy Boundary ✓ O(n) O(1) Optimal — interview pick
