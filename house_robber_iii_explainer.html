LeetCode #337

House Robber III

Rob houses arranged in a binary tree — no two directly connected houses can be robbed. Return the maximum amount you can steal.

Medium Dynamic Programming Tree DP Return Pair Time: O(n) Space: O(h)
1 Understanding the Problem ▾

You are a robber. The city's houses are arranged as a binary tree. Each node holds money. The constraint: if you rob a house, you cannot rob its direct parent or either direct child — the alarm system connects adjacent nodes. What is the maximum total you can steal?

💡 Key Insight

At every node, you face a binary choice: rob it (gaining its value but locking out its children) or skip it (gaining nothing but freeing its children to be robbed). The optimal choice at each node depends on what both subtrees can offer under each state — a classic tree DP setup requiring a return pair, not just a scalar.

Example — Optimal Strategy

Tree: root=3, left=4, right=5
4's children: 1 (left), 3 (right)
5's children: null, 1 (right)

Skip root(3), rob 4, rob 5:
4 + 5 = 9 ✓ Answer = 9

Why Simple Greedy Fails

"Greedy: always take the root":
Rob 3 → skip 4 and 5
Best below 4: 1+3=4 (blocked!)
Best below 5: 1 (blocked!)
Total = 3. Way worse than 9!

🔑 State Identification — Why We Need TWO Values Per Node

Returning only max(rob, skip) from a child loses critical information. The parent needs to know both options separately: if the parent is robbed, it must use the child's skip value specifically. If it only has the max, it can't enforce the constraint. Returning a pair (rob_val, skip_val) is the key technique of Tree DP.

Edge cases: single node (answer = node.val), leaf nodes (rob = val, skip = 0), skewed tree (recursion depth = n), tree with all-same values.

2 The Algorithm: Return-Pair Tree DP (Postorder DFS) ▾

We perform a postorder DFS (left → right → root). Each call returns a pair (rob, skip) where rob = best loot from this subtree if we rob this node, and skip = best loot if we skip it. The parent combines both children's pairs to form its own pair and propagates it upward.

1 Base case — null node: Return (0, 0). A null node contributes nothing regardless of state. This cleanly handles missing children without if-guards.
2 Recurse left then right: left = dp(node.left), right = dp(node.right). Postorder — collect children's results before processing the current node.
3 Compute rob[node]: Rob this node → gain node.val, but children must be skipped. Use left[1] (skip-left) and right[1] (skip-right). Formula: rob = node.val + left[1] + right[1].
4 Compute skip[node]: Skip this node → each child independently picks its best option. Formula: skip = max(left) + max(right) — let each child choose rob or skip freely.
5 Return pair and combine at root: Each node returns (rob, skip). At the root, answer is max(dp(root)) — pick whichever is better.

3 Why It Works: The Invariant ▾
🔐 The Invariant

For every node v, the pair (rob[v], skip[v]) represents the globally optimal loot from the entire subtree rooted at v, conditioned on whether v itself is robbed or skipped. Because we enforce children-must-skip when a parent is robbed, no two adjacent nodes are ever both robbed. This invariant propagates correctly up the tree by construction.

Why this is DP:

  • Optimal substructure: The best answer for a subtree is computed exactly from the best answers of its left and right sub-subtrees — no need to look further.
  • Overlapping subproblems: In a naive recursive solution without memoization, nodes lower in the tree would be evaluated multiple times when trying all subset combinations. The pair DP processes each node exactly once.
Naive — Try All Subsets

For n nodes: 2ⁿ possible subsets.
Check each for validity → O(2ⁿ · n).
Completely unusable for n > 20.

Return-Pair DP — O(n)

Each node visited once in postorder.
Pair propagates constraint upward.
No cache needed — single pass!

⚠️ The HashMap Memo Anti-Pattern (Slower Alternative)

A common intermediate solution: memo = {}, define rob_value(node) that computes max loot when robbing node, calling rob_value on grandchildren and rob_value on children for the skip case. This achieves O(n) with memoization, but requires an O(n) cache and makes more recursive calls per node. The pair approach is cleaner: no cache, no extra calls, one pass.

4 The Code (Annotated) ▾

Solution 1 — Return-Pair Tree DP (Optimal, Interview Pick)

def rob(root): ← Main entry point
def dp(node): ← Inner DFS: returns (rob_val, skip_val)
if not node: ← Base case: null node
return (0, 0) ← Null contributes 0 in all states
left = dp(node.left) ← Recurse left (postorder: children first)
right = dp(node.right) ← Recurse right
rob_node = node.val + left[1] + right[1] ← Rob: children must be skipped (index 1)
skip_node = max(left) + max(right) ← Skip: children pick their own best
return (rob_node, skip_node) ← Return BOTH to parent
return max(dp(root)) ← Root picks the better of rob or skip
⚠️ Why left[1] and right[1], not left[0]?

Index 0 = rob that child, index 1 = skip that child. When we rob the current node, we must force both children to be skipped — so we use left[1] and right[1]. When we skip the current node, children may choose freely, so max(left) and max(right) let each child pick its own optimum. Never confuse the two indices!

Solution 2 — HashMap Memoization (Easier to Derive, Good Starting Point)

def rob(root):
memo = {} ← Cache: node → best loot from subtree
def helper(node):
if not node: return 0 ← Base case
if node in memo: return memo[node] ← Cache hit
rob_this = node.val ← Option A: rob this node
if node.left:
rob_this += helper(node.left.left) + helper(node.left.right) ← Skip left child, use grandchildren
if node.right:
rob_this += helper(node.right.left) + helper(node.right.right)← Skip right child
skip_this = helper(node.left) + helper(node.right)← Option B: skip, children free
memo[node] = max(rob_this, skip_this) ← Cache best choice
return memo[node]
return helper(root)
💡 Complexity Summary

Return-Pair DP: Time O(n) — each of n nodes visited once. Space O(h) for call stack (h = height; O(log n) balanced, O(n) worst-case skewed).
HashMap Memo: Time O(n) — same asymptotically. Space O(n) for cache plus O(h) stack. More memory, more calls.
Prefer the pair approach for interviews — it's cleaner and shows deeper understanding.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize Tree DP Signals

Binary tree + maximize/minimize + "no two adjacent nodes" constraint. The constraint is local (parent↔child), not global (e.g., not "no two in same level"), so postorder DFS with state propagation works perfectly. Say: "Since the constraint is strictly between parent and child, I can use a bottom-up tree DP."

Step 2: Start with Memoization (Easier to Derive)

It's acceptable to start with the HashMap memo approach in an interview — say "I can solve this recursively: at each node, rob it (skip children) or skip it (children decide freely). I'll memoize to avoid recomputation." Then offer to optimize: "Actually, I can eliminate the cache by returning a pair from each node instead."

Step 3: Explain the Pair Return Type

The key move: "Instead of returning a single max value, I return two values — the best loot if I rob this node, and the best loot if I skip it. The parent uses these two values to compute its own pair without needing to revisit any node."

Step 4: Write the Recurrence Clearly

On the whiteboard: rob = val + left.skip + right.skip, skip = max(left) + max(right). Emphasize: "left.skip" not "left.rob" — the constraint forces children to skip, not to rob. Then code it up cleanly.

🗣️ What to Say to the Interviewer

"This is a tree DP problem. The adjacency constraint is parent-to-child, so I'll use postorder DFS.
At each node I track two values: loot if I rob this node, and loot if I skip it.
Rob case: gain node.val, but both children must be skipped — use their skip values.
Skip case: each child independently picks its better option — use max of each child's pair.
I return both values to the parent as a tuple. At the root, I take the max of the pair.
Time: O(n) — one pass over all nodes. Space: O(h) — recursion stack only, no cache."
🔮 Likely Follow-up Questions
  • "Can you do this iteratively?" → Iterative postorder with explicit stack; store pairs in a hash map keyed by node.
  • "What if it's an n-ary tree?" → Same technique: rob = val + sum(child.skip for all children); skip = sum(max(child) for all children).
  • "How is this different from House Robber I?" → HR I is a path graph (linear array). This is a general binary tree. The pair generalizes the even/odd DP of HR I to arbitrary tree shapes.
  • "Space complexity for balanced vs skewed?" → O(log n) for balanced binary tree, O(n) for a completely skewed tree (like a linked list).

7 Summary & Key Takeaways ▾
Tree DP uses postorder DFS — solve children before using their results in the parent
Return a (rob, skip) PAIR from every node — returning only the max loses constraint context
rob = node.val + left[1] + right[1] — when you rob a node, both children MUST be skipped
skip = max(left) + max(right) — when you skip a node, each child picks its own best independently
No extra memoization hash map needed — the pair naturally encodes all necessary state
Final answer = max(dp(root)) at the root — pick rob or skip, whichever is larger

🎯 When to Recognize the Return-Pair Tree DP Pattern
  • Tree structure + optimization (maximize/minimize) + adjacent-node constraint
  • Each node has a binary decision (include/exclude, rob/skip) that constrains its children
  • The constraint is parent↔child only (local), not path-wide or level-wide
  • Key phrase in problem: "two directly connected nodes cannot both be selected"
  • Signal: "what if we must select nodes but can't select adjacent ones?"
