LeetCode #690

Employee Importance

Given a list of employees (each with an id, importance value, and list of direct subordinate ids) and a target id, return the total importance of that employee and all their direct and indirect subordinates.

Medium Graph DFS Sum Values Time: O(n) Space: O(n)
1 Understanding the Problem ▾

Picture a company org chart. Every employee has an ID, an importance value (think: performance score or salary weight), and a list of direct subordinate IDs. You pick one employee and ask: "What is the total importance of this employee plus everyone who reports to them, directly or indirectly?"

💡 Key Insight

The org chart is a tree (directed, acyclic). Starting from the target employee, you need to visit every node in their subtree and sum the importance values. This is a classic graph traversal (DFS or BFS) combined with a running sum — the "DFS Sum Values" pattern.

2 The Algorithm: DFS with HashMap Lookup ▾

The challenge has two parts: how to look up employees efficiently, and how to visit all subordinates. We solve them separately. First, build a hash map (id → Employee) for O(1) lookups. Then run a DFS from the target, accumulating importance at each node.

1 Preprocess: build the map. emp_map = {e.id: e for e in employees}. This one-time O(n) pass allows every subsequent lookup by id to run in O(1). Without this, each lookup costs O(n) — the whole algorithm becomes O(n²).
2 DFS from the target. Call dfs(target_id). Look up the employee, add their importance to a running total, then recursively call dfs(sub_id) for each subordinate.
3 Natural base case. When an employee has no subordinates, the for loop over emp.subordinates simply has zero iterations. No explicit base-case check is needed — the recursion bottoms out naturally at leaf employees.
4 Return the total. Each DFS call returns the sum for its subtree. The caller adds this to its own running total. When dfs(target_id) returns, the answer is complete.
🔑 BFS Alternative (equally valid)

Use a deque starting with the target id. While the queue is non-empty: dequeue an id, look up the employee, add their importance, enqueue all their subordinates. Both DFS and BFS are O(n). DFS via recursion is more concise; BFS avoids Python's default recursion limit (~1000 frames) for very deep hierarchies. Mention this trade-off in an interview.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At any point during DFS, the running total equals the sum of importance values of all employees that have been fully processed (i.e., visited and all of whose subtrees have been explored). When the root DFS call returns, every employee reachable from the target has been processed exactly once.

Why no "visited" set is needed: The employee hierarchy is guaranteed to be a tree — every employee has at most one manager, and there are no cycles. Following only the subordinate links (parent → children), you can never revisit an already-seen employee. DFS always moves "downward" toward leaf employees and never loops back.

Why the time complexity is O(n): Building the hash map is O(n). The DFS visits each employee exactly once and follows each subordinate link exactly once. The total number of links across all employees equals n−1 (tree property). So DFS is O(n). Total: O(n).

Why space is O(n): The hash map uses O(n). The recursive call stack uses O(h) where h is the maximum depth of the hierarchy. In the worst case (one long chain), h = n, giving O(n) total space.

⚠️ What if there were cycles?

If the data could form cycles (employee A reports to B who reports to A), DFS would infinite-loop. In a general graph you'd add a visited = set() and skip already-seen ids. The problem guarantees a tree structure, so cycles are impossible here — but always verify this assumption with your interviewer before omitting the visited set.

4 The Code (Annotated) ▾

Solution 1 — DFS with HashMap (Optimal, Interview Pick) — O(n) time, O(n) space

def getImportance(employees, id): ← employees: list of Employee; id: target employee's id
emp_map = {e.id: e for e in employees} ← O(n) build: id → Employee for O(1) lookup later
def dfs(eid): ← Inner function: closes over emp_map (no extra param needed)
emp = emp_map[eid] ← O(1) hash lookup — why we built the map first
total = emp.importance ← Count this employee's own importance first
for sub_id in emp.subordinates: ← Iterate over direct subordinate ids
total += dfs(sub_id) ← Recursively sum entire subtree of each subordinate
return total ← Return this subtree's total to the caller
return dfs(id) ← Launch DFS from the target employee
⚠️ Why define dfs() as an inner function?

The inner dfs() closes over emp_map — no need to pass it as a parameter on every recursive call. This is idiomatic Python. Alternatively, you could write a standalone helper that takes emp_map as a parameter, but the closure is cleaner. Both are correct in interviews.

Solution 2 — BFS with deque (Iterative, avoids recursion depth limits)

from collections import deque
def getImportance(employees, id):
emp_map = {e.id: e for e in employees} ← Same O(1) lookup map
total, queue = 0, deque([id]) ← Initialize running total; seed queue with target id
while queue: ← Process until all reachable employees are visited
eid = queue.popleft() ← popleft() is O(1) with deque (list.pop(0) would be O(n))
emp = emp_map[eid] ← Look up employee in O(1)
total += emp.importance ← Accumulate this employee's importance
queue.extend(emp.subordinates) ← Enqueue all direct subordinates for later processing
return total ← All reachable employees visited, sum is complete

Brute Force: Time O(n²), Space O(h) — correct but slow

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the "Sum All Reachable Nodes" Pattern

The word "subordinates" is the signal — hierarchical structure = tree. The question asks for the total value of the target node plus all descendants. This is a textbook tree/graph traversal with accumulation: DFS or BFS while summing values.

Step 2: Spot the Lookup Bottleneck — Propose HashMap

The data comes as a list, but traversal requires looking up employees by ID. Say: "Without preprocessing, each lookup scans the whole list — O(n) per call, O(n²) total. I'll build a hash map first: O(n) once, then O(1) per lookup during traversal."

Step 3: Decide DFS vs BFS — and Justify

DFS via recursion is more concise. BFS with a deque is safer for very deep hierarchies (avoids Python's ~1000 frame recursion limit). Offer both and ask the interviewer which they prefer — showing you know the trade-off earns points.

Step 4: Mention the "visited set" consideration — then dismiss it

"In a general graph I'd maintain a visited set. Here the structure is a tree by definition — no cycles possible — so I'll skip it. But I'd confirm this guarantee with you before omitting it in production."

🗣️ What to Say to the Interviewer

"I see a hierarchical structure — employees with subordinate links — so this is a tree traversal problem.
The ask is: sum importance of the target plus all descendants. Classic DFS with accumulation.
Problem: the data is a list, but I need to look up employees by id during traversal. O(n) scans would make it O(n²).
Fix: build emp_map = {e.id: e} first — O(n) once, then O(1) per lookup.
Then DFS from the target: add importance, recurse into each subordinate. No visited set needed — it's a tree.
Time: O(n) map + O(n) DFS = O(n). Space: O(n) map + O(h) call stack = O(n) worst case."
🔮 Likely Follow-up Questions
  • "What if an employee can report to multiple managers?" → General graph; add visited set to prevent revisits
  • "Return the list of all subordinate IDs instead of the sum?" → Same DFS/BFS, collect IDs into a list instead of summing
  • "Find which employee has the highest total importance?" → Run this for every employee (O(n²) naive) or use tree DP in O(n)
  • "Data given as parent pointers instead of children?" → Build children adjacency list first, then same DFS

7 Summary & Key Takeaways ▾
Always build a HashMap first when you need O(1) lookups by key during traversal
Hierarchical "subordinate" structures are trees — use DFS or BFS to visit all descendants
No "visited" set needed for trees — the directed structure prevents revisiting nodes
DFS recursion is concise; BFS with deque avoids Python's recursion depth limit on deep hierarchies
"Sum all reachable nodes" is a recurring graph pattern — accumulate during traversal
