LeetCode #881

Boats to Save People

Each boat can carry at most two people, subject to a weight limit. Find the minimum number of boats needed to rescue everyone.

Medium Two Pointers Greedy Time: O(n log n) Space: O(1)
1 Understanding the Problem ▾

You have a group of people, each with a given weight, and an unlimited number of rescue boats. Each boat can carry at most 2 people and has a weight limit. You want to rescue everyone using the fewest boats possible.

💡 Key Insight

The minimum number of boats is achieved by greedily pairing the heaviest available person with the lightest. If they fit together, you save a boat. If the heaviest cannot share with even the lightest remaining person, they must ride alone. Sorting enables O(1) access to the best possible pairing at every step.

Example 1

people = [1, 2], limit = 3
Sort → [1, 2]
lo=0(1), hi=1(2): 1+2=3 ≤ 3 → pair!
Boats needed: 1

Example 2

people = [3, 2, 2, 1], limit = 3
Sort → [1, 2, 2, 3]
Pair(1,2) ✓ · Solo(2) · Solo(3)
Boats needed: 3

Key constraints: Each boat carries at most 2 people (not n people). Everyone's weight is guaranteed to be ≤ limit (so a solo always fits). You want the minimum number of boats. This "at most 2" constraint is what makes the greedy pairing approach optimal — if boats could hold 3+ people, the problem would require a more complex algorithm.

⚠️ Why Can't We Just Pair Adjacent Elements After Sorting?

Consider people=[1,2,2,3], limit=3. Pairing adjacent: [1,2] (sum 3, ok) and [2,3] (sum 5 > 3, fails). That gives 3 boats. Pairing heaviest with lightest: [1,3]=4>3 (3 goes solo), then [2,2]=4>3 (both solo) = 3 boats too. But try: pair [1,2] and [2,3 solo] = 3. Best possible is 3 here. The greedy always matches heaviest with lightest — not adjacent — to minimize missed opportunities.

2 The Algorithm: Greedy Two-Pointer (Opposite Direction) ▾

After sorting, we place two pointers at opposite ends — lo pointing to the lightest person and hi pointing to the heaviest. At each step, we always send the heaviest person (hi) on a boat. We ask: can the lightest person ride along?

1 Sort the people array in ascending order. This gives us O(1) access to the lightest (index 0) and heaviest (index n-1) at any point during the scan.
2 Initialize two pointers: lo = 0 (lightest), hi = n-1 (heaviest), and boats = 0.
3 At each iteration: Check if people[lo] + people[hi] ≤ limit. If yes, the lightest can join — move lo right. Regardless, the heaviest always takes a boat: hi--, boats++.
4 Repeat while lo ≤ hi. When lo == hi, exactly one person is left who must ride alone — the loop handles this automatically (the pair check fails trivially since you'd compare a person with themselves).
✓ Can Pair: people[lo] + people[hi] ≤ limit

Send both on one boat. lo++, hi--, boats++. We successfully saved a boat by placing two people on it.

✗ Can't Pair: people[lo] + people[hi] > limit

The heaviest person rides alone. hi--, boats++. Since lo is the lightest available, no one lighter exists — solo is unavoidable.

🔑 Why We Always Handle the Heaviest First

The heaviest person is the hardest to pair. If they can share with the lightest available, that's the best possible pairing for them. If they can't share with even the lightest, they definitely can't share with anyone heavier — so they must go alone. Handling them every iteration ensures we never miss a valid pairing and never defer a problem we can't solve later.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every iteration, people[lo..hi] is exactly the set of unrescued people. Everyone outside this range has already been placed on a boat. The heaviest unrescued person always boards the current boat, possibly with the lightest unrescued person.

Greedy correctness (exchange argument): Suppose the heaviest remaining person is at index hi with weight H, and the lightest is at lo with weight L.

Case 1 — Can pair (L + H ≤ limit): Pairing them is optimal. Any alternative — pair L with someone else (weight M) and let H go alone — uses 2 boat-slots total (one for H alone, one for L+M). Our approach also uses 2 boat-slots (L+H together, one boat; plus one for whoever M eventually rides with). No worse, and we've handled H, which is the most constrained element.

Case 2 — Can't pair (L + H > limit): If even the lightest person can't share with H, then every remaining person is at least as heavy as L, so no one can share with H. H must go alone. There is no alternative — we lose no optimality here.

💡 Analogy: Loading a Ferry

Imagine loading a small ferry that holds 2 passengers up to a weight capacity. You load passengers in pairs from each end of the weight-sorted line — heaviest at the back, lightest at the front. If they fit together, both board. If not, the heaviest boards alone and the lightest waits for the next trip with the next-heaviest. This always minimizes trips.

Edge cases handled naturally:

  • Single person: lo=0, hi=0. Loop runs once. Pair check: person + person > limit (or not, irrelevant — lo and hi are the same, can't actually pair). hi--, boats=1. Correct.
  • All same weight ≤ limit/2: Every pair succeeds. n/2 boats (ceil for odd n).
  • All same weight > limit/2: No pairs succeed. n solo boats.
  • Two people: Either they pair (1 boat) or they don't (2 boats). lo≤hi=1 until both processed.
4 The Code (Annotated) ▾

Solution 1 — Two-Pointer Greedy (Interview Pick, Optimal)

def numRescueBoats(people: list[int], limit: int) -> int:
people.sort() ← Sort ascending; enables O(1) access to lightest and heaviest
lo, hi = 0, len(people) - 1 ← Two pointers: lo=lightest remaining, hi=heaviest remaining
boats = 0 ← Accumulate count of boats used
while lo <= hi: ← Continue while there are unrescued people (lo==hi means 1 left)
if people[lo] + people[hi] <= limit: ← Can lightest ride with heaviest?
lo += 1 ← Yes: lightest also boards this boat
hi -= 1 ← Heaviest ALWAYS boards (unconditional — outside the if!)
boats += 1 ← One boat deployed per iteration, whether pair or solo
return boats ← Minimum boats needed to rescue everyone
⚠️ The Unconditional hi-- Is Critical

hi -= 1 is outside the if-block. Both branches — successful pair and solo — result in the heaviest person boarding a boat. A common bug is placing hi -= 1 and boats += 1 inside the if-block, which forgets the solo case entirely. The heaviest person always boards every iteration.

Solution 2 — Sort + Scan (Brute Force, O(n²))

def numRescueBoats_brute(people, limit): ← O(n²): for understanding only
people.sort()
used = [False] * len(people) ← Track rescued status
boats = 0
for i in range(len(people)-1, -1, -1): ← Process heaviest first
if used[i]: continue
for j in range(i): ← Scan for lightest available partner — O(n)
if not used[j] and people[j]+people[i]<=limit:
used[j] = True; break ← Pair found; inner scan = O(n) total → O(n²) overall
used[i] = True; boats += 1
return boats
💡 Complexity Summary

Two-Pointer Greedy: Time O(n log n) — dominated by sort; O(n) for the pointer scan. Space O(1) auxiliary (sort in-place in Python with Timsort).
Brute Force: Time O(n²) worst case (inner scan per person). Space O(n) for the used array. Only for conceptual clarity.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Notice the "at most 2 people" constraint

This is the unlock. With k=2, every boat is a binary decision: solo or pair. This binary structure screams greedy. (If k were 3+, you'd need DP or a different approach — always ask about the limit up front.)

Step 2: Think about who is hardest to pair

The heaviest person is the bottleneck. Ask: "Who could possibly share a boat with the heaviest?" Only someone light enough. So: sort, try to pair heaviest with lightest. If it works, great. If not, heaviest goes solo — because no lighter person exists.

Step 3: Propose sort + opposite two-pointer

"Sort the array. Put lo=0 (lightest) and hi=n-1 (heaviest). Each iteration: always move hi (heaviest boards), move lo only if they can share. boats++ each time. Return boats when lo>hi." Then trace through the example live.

Step 4: Justify the greedy choice formally

"Why is pairing lightest+heaviest always optimal? If heaviest CAN pair with lightest — this is the best partner for heaviest. If heaviest CANNOT — they definitely can't pair with anyone heavier. So the decision for the heaviest person is always uniquely determined."

🗣️ What to Say to the Interviewer

"The key observation is each boat holds at most 2 people — so every boat is either a solo or a pair.
To minimize boats, I want to maximize pairs. Greedy insight: after sorting, always try to pair the
heaviest remaining person with the lightest remaining person. If they fit, great — both board. If
not, the heaviest must go alone — no lighter partner exists. Two pointers: lo starts at 0, hi at n-1.
hi always decrements; lo only when a pair succeeds. Time O(n log n), Space O(1)."
🔮 Likely Follow-up Questions
  • "What if boats hold 3 people?" → Greedy becomes much harder — O(n²) or DP approaches needed
  • "Can you do better than O(n log n)?" → Only with counting sort if weights are bounded integers (e.g., 1..limit)
  • "What's the minimum boats for n people all weighing exactly limit/2?" → ceil(n/2) — everyone pairs
  • "What if the boat can carry any number of people but you want to minimize total weight per boat?" → Completely different objective; greedy doesn't directly apply

7 Summary & Key Takeaways ▾
Sort first — opposite-direction two pointers require sorted order to be meaningful
Always handle the hardest-to-pair element (the heaviest) at each step
Greedy pairing: lightest + heaviest is always the best possible pairing for the heaviest
hi always decrements (unconditionally); lo only decrements when a valid pair forms
The "at most 2 people per boat" constraint is what makes the greedy approach provably optimal
ApproachTimeSpaceNotes
Brute Force (nested loops) O(n²) O(n) For understanding; never in interviews

🎯 When to Recognize This Pattern (Opposite-Direction Two Pointers + Greedy)
  • Problem asks to minimize groups/pairs subject to a sum/weight constraint
  • Each group has exactly 2 participants (or "at most 2")
  • After sorting, the "hardest" element is at one end; the "easiest partner" is at the other
  • You want to greedily match the most constrained element with the best available partner
