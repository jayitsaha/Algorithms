LeetCode #659

Split Array into Consecutive Subsequences

Given a sorted integer array, determine if it can be split into subsequences, each of length at least 3, consisting of consecutive integers.

Medium Hash Tables Greedy + Two Maps Time: O(n) Space: O(n)
1 Understanding the Problem ▾

You are given a sorted array of integers. Your task: determine whether every element can be placed into one of several subsequences, where each subsequence is a run of consecutive integers (like 1,2,3 or 4,5,6,7) and has length at least 3.

This is not about partitioning into contiguous subarrays — subsequences are groups of elements drawn from the array that form a consecutive number sequence. If a number appears multiple times (e.g., two 3s), it may go into two different subsequences.

💡 Key Insight

Think of it as scheduling tasks. When you see a number n, you have two choices: extend an existing subsequence that ends at n-1, or start a brand-new subsequence [n, n+1, n+2]. If neither is possible, return False.

✓ Valid: [1,2,3,3,4,5]

Subsequence 1: [1, 2, 3]
Subsequence 2: [3, 4, 5]
Both ≥ 3 elements → True

✗ Invalid: [1,2,3,4,4,5]

Subsequence 1: [1, 2, 3, 4]
Remaining 4 and 5 can't form a run of 3
→ False

Edge Cases: single-element arrays (False — can't make length-3 run), arrays where counts don't match up (e.g., three 4s but only two 5s), and numbers with gaps (gap means no consecutive continuation is possible).

2 The Algorithm: Greedy with Two Hash Maps ▾

We maintain two hash maps simultaneously: freq counts how many times each number still needs to be placed, and tails tracks how many open subsequences currently end at each value. The greedy strategy: always prefer extending an existing subsequence over starting a new one.

1 Build freq map: Count every number's frequency. freq = Counter(nums). This tells us how many of each number we must place.
2 Iterate through unique sorted values: For each number n (in order), decide what to do with it based on the two maps.
3 Priority 1 — Extend existing subsequence: If tails[n-1] > 0, an open subsequence ends at n-1. Append n to it: tails[n-1] -= 1, tails[n] += 1, freq[n] -= 1. This is preferred because it avoids consuming future numbers unnecessarily.
4 Priority 2 — Start new subsequence: If tails[n-1] == 0 but we have freq[n+1] > 0 and freq[n+2] > 0, we can start a new 3-element run. Consume all three: decrease freq for n, n+1, n+2, and add 1 to tails[n+2].
5 Fail condition: If neither option works (no tail to extend AND no two successors available), return False. The number n cannot legally be placed anywhere.
🔑 Why Greedy Works Here

Extending an existing subsequence is never worse than starting a new one. If we have a tail at n-1 and we don't use it, that open subsequence might fail to reach length 3 later. Using the tail consumes n and keeps the longer sequence alive without spending extra future numbers. This local greedy choice is globally optimal.

Decision Tree for Each Number n:

tails[n-1] > 0 ? (existing run to extend)
YES
Extend: tails[n-1]--, tails[n]++
NO
freq[n+1] > 0 AND freq[n+2] > 0 ?
YES
Start new run: consume n, n+1, n+2
NO
Return False — n cannot be placed
3 Why It Works: The Invariant ▾
🔐 The Invariant

At every step, tails[v] exactly equals the number of open subsequences that currently end at value v and are still "growing" (waiting to receive their next number). No subsequence is double-counted, and no subsequence is forgotten.

Why extending is always safe: Suppose we have a tail at n-1 and also the capacity to start a new run. If we start a new run instead, we consume n+1 and n+2 from the freq map. But some later number might have needed those. By extending instead, we use only n and don't touch future numbers — strictly less consumption.

Why starting a new run is correct when no tail exists: If tails[n-1] == 0, there is no ongoing run to attach n to. The only way to place n is to start a fresh one. If we don't have n+1 and n+2 available, we provably cannot place n — it's impossible regardless of how we rearrange other decisions (since the array is sorted, anything to the left of n is already placed).

⚠️ Common Trap: Counting from the Wrong End

New learners sometimes track starts of subsequences instead of tails. But when you process a number, you need to know what came before it (the tail), not what came after. The tails map is the right state to maintain.

Completeness proof sketch: Because we process numbers in sorted order and we always prefer to extend before starting new runs, every element that can legally be placed is placed. If any number cannot be placed — either because it's stranded (no predecessors and not enough successors) — the true answer is False, and we correctly return False at that point.

4 The Code (Annotated) ▾

Optimal Solution — Greedy + Two Maps (O(n) time, O(n) space)

from collections import Counter
def isPossible(nums): ← nums is sorted
freq = Counter(nums) ← How many of each value remain to be placed
tails = Counter() ← tails[v] = # open subsequences ending at v
for n in nums: ← Process each element left to right
if freq[n] == 0: ← Already placed this copy elsewhere; skip
continue
if tails[n-1] > 0: ← A run ends at n-1: extend it
tails[n-1] -= 1 ← That run no longer ends at n-1...
tails[n] += 1 ← ...now it ends at n
freq[n] -= 1 ← n has been placed
elif freq[n+1] > 0 and freq[n+2] > 0: ← Can start a new 3-run
freq[n] -= 1 ← Place n
freq[n+1] -= 1 ← Place n+1
freq[n+2] -= 1 ← Place n+2
tails[n+2] += 1 ← New run ends at n+2
else: ← Cannot place n anywhere → impossible
return False
return True ← All elements placed legally
⚠️ Why iterate over nums (not unique values)?

We iterate over the original sorted nums list and skip already-placed copies via the freq[n] == 0 check. This ensures we process each logical occurrence in order. An alternative is to iterate over sorted(freq.keys()), applying the same logic — both are O(n).

💡 Complexity Summary

Greedy + Two Maps: Time O(n), Space O(n) — each element visited once; both maps have at most O(n) distinct entries.
Brute Force: Time O(n²), Space O(1) extra — but O(n) removal makes it impractical for large inputs.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Pattern Signals

The sorted input + "can we assign elements to groups" structure screams greedy. When you see "can something be partitioned/split," and the array is sorted, greedy + frequency counting is a go-to pattern. The minimum group size (≥3) is the key constraint.

Step 2: Start with Brute Force

Propose: "Repeatedly find the smallest number, try to extend a run of 3 starting there." This is O(n²) due to list removals. It shows correctness intuition before optimization.

Step 3: Identify the Bottleneck and Optimize

"The brute force is slow because searching and removing from a list is O(n). What if I kept frequency counts in a hash map and also tracked open subsequence tails? Then each step is O(1) lookup."

Step 4: Justify the Greedy Choice

"Prefer to extend an existing run over starting a new one because extending costs only 1 element whereas starting costs 3. If an open tail exists, we'd be wasting future elements by not using it."

🗣️ What to Say to the Interviewer

"The input is sorted, so I process elements left to right. For each n, I check: is there a subsequence that ends at n-1? If yes, extend it — this is always at least as good as starting new. If no, check if n+1 and n+2 are available; if so, start a fresh 3-element run. If neither option works, return False — n is unplaceable. I maintain two counters: freq (remaining counts) and tails (open run endpoints). Time O(n), Space O(n)."
🔮 Likely Follow-up Questions
  • "What if minimum subsequence length is k instead of 3?" → Same structure, change start-run check to require k successors
  • "What if the array is not sorted?" → Sort it first; still O(n log n) overall
  • "Can you reconstruct the actual subsequences?" → Store list of subsequences, append to them; heap helps pick which to extend
  • "How does this relate to hand scheduling problems?" → It's essentially assigning tasks to workers where each task chain must have length ≥ 3
