Custom Problem · Hard

Minimum Cost Path with Teleportations

Find the minimum cost to travel from node 0 to node n−1 in a weighted graph, where each node may offer a one-time free teleportation to a designated target node.

Hard Dynamic Programming DP with State Time: O(n²) Space: O(n)
1 Understanding the Problem ▾

You have n nodes numbered 0 to n−1. You can walk from any node i to i+1 paying edge weight w[i]. Additionally, each node i has a teleporter: pay cost tp_cost[i] and jump instantly to node tp_dest[i]. You start at node 0 and want to reach node n−1 with minimum total cost.

💡 Key Insight

At every node you make a binary decision: walk forward (pay the edge cost) or teleport (pay the teleport cost and jump). Optimal decisions at each node depend only on the best cost to reach that node — classic DP substructure.

Concrete small example: 5 nodes (0–4), edge costs [3, 1, 2, 4], teleporters: node 2 → node 4 costs 1, node 1 → node 3 costs 2.

2→3: cost 2

Optimal path (with teleport)

Edge cases: A teleport to a lower-numbered node (going backwards) is usually suboptimal but theoretically possible if costs allow. Teleport cost 0 is a free skip. The walk edges must be non-negative for this DP to be correct (otherwise Bellman-Ford or Dijkstra is needed).

2 The Algorithm: 1-D State Machine DP ▾

We define a 1-D DP table dp[i] = minimum cost to reach node i. We fill it left to right. At each node i, we update all nodes reachable from i in a single step: the next node i+1 (walk) and the teleport destination tp_dest[i].

1 Initialize: dp[0] = 0 (we're already at the start, no cost). All other dp[i] = ∞.
2 Process node i (0 to n−2): For each node, compute two transitions: walk to i+1 with cost dp[i] + w[i]; teleport to tp_dest[i] with cost dp[i] + tp_cost[i].
3 Relax each destination: If the computed cost is better than the current best for that destination, update: dp[dest] = min(dp[dest], new_cost).
4 Answer: After processing all nodes, dp[n−1] holds the minimum cost to reach the final node.
🔑 Why Process Left to Right?

If all walk-edge costs are non-negative (and teleports only go forward), node i is finalized before node i+1 is ever processed. This is exactly the same ordering guarantee that Dijkstra exploits on DAGs — once we pop a node, its shortest path is settled. Here the topological order is just node index.

3 Why It Works: Optimal Substructure & Overlapping Subproblems ▾
🔐 Why Is This DP?

Optimal substructure: The cheapest path to node k must go through some intermediate node j < k via the cheapest path to j. Replacing that sub-path with a cheaper one to j would give a cheaper overall path — contradiction with optimality of dp[j]. So dp[k] depends optimally on dp[j].

Overlapping subproblems: Multiple later nodes can be reached from the same intermediate node (via walk or teleport). Without memoizing dp[i], naive recursion would recompute the cheapest path to i every time it's a stepping stone.

📐 The Recurrence Relation
dp[0] = 0 // base case

dp[i+1] = min(dp[i+1], dp[i] + w[i]) // walk forward
dp[tp_dest[i]] = min(dp[tp_dest[i]], dp[i] + tp_cost[i]) // teleport
🔐 The Invariant

At the start of processing node i, dp[i] is the final, optimal minimum cost to reach node i. No future processing step can improve it (assuming forward-only teleports and non-negative walk costs). This is the key correctness guarantee that lets us process each node exactly once.

Proof sketch: By induction. Base: dp[0] = 0 is trivially optimal. Inductive step: when we process node i, all nodes j < i are already optimal (inductive hypothesis). The two transitions (walk and teleport) from j correctly propagate those optimal costs forward. Therefore dp[i], which was relaxed by all j < i that can reach i, contains the true optimum.

4 The Code (Annotated) ▾

Solution 1 — DP (Bottom-Up Tabulation, Optimal)

def min_cost_teleport(n, walk_costs, tp_costs, tp_dests):
INF = float('inf') ← sentinel: no known path yet
dp = [INF] * n ← dp[i] = cheapest cost to reach node i
dp[0] = 0 ← base case: already at node 0
for i in range(n - 1): ← process each node 0..n-2 in order
if dp[i] == INF: ← skip unreachable nodes (safety)
continue
# Walk: move to next node
walk_cost = dp[i] + walk_costs[i] ← cost of stepping right one node
dp[i + 1] = min(dp[i + 1], walk_cost)← relax neighbor — take it if cheaper
# Teleport: jump to tp_dests[i]
dest = tp_dests[i] ← teleport landing node
tp_total = dp[i] + tp_costs[i] ← cost of teleporting from i
dp[dest] = min(dp[dest], tp_total) ← relax dest — update only if cheaper
return dp[n - 1] ← answer: cheapest way to reach final node
⚠️ Why is the skip-INF check needed?

If an earlier teleport jumped past node i without anyone providing a path to it, dp[i] stays INF. Adding INF + walk_cost could overflow integers (not in Python, but in other languages it would). The guard also prevents meaningless relaxations. Skip any node you can't yet reach.

Solution 2 — Brute Force Recursive (Exponential)

def min_cost_brute(n, walk_costs, tp_costs, tp_dests, i=0, cost=0, best=[]):
# DFS: try all walk/teleport combinations at every node
if i == n - 1:
best[0] = min(best[0], cost); return← O(2^n) — try every subset of teleports
if i < n - 1:
min_cost_brute(..., i+1, cost+walk_costs[i], best) ← walk
min_cost_brute(..., tp_dests[i], cost+tp_costs[i], best) ← teleport

Solution 3 — Top-Down Memoization

from functools import lru_cache
def min_cost_memo(n, walk_costs, tp_costs, tp_dests):
@lru_cache(maxsize=None)
def dp(i): ← minimum cost to reach node i from 0
if i == 0: return 0 ← base case
# Best cost among all nodes j that can reach i
best = float('inf')
if i - 1 >= 0: ← walk: predecessor is i-1
best = min(best, dp(i-1) + walk_costs[i-1])
for j in range(n): ← any j whose teleport lands on i
if tp_dests[j] == i:
best = min(best, dp(j) + tp_costs[j])
return best
return dp(n - 1)
💡 Bottom-Up vs Top-Down Trade-offs

Bottom-up (Tabulation): O(n) space, processes each node once. Very cache-friendly. Preferred for production and interviews because it is iterative, avoids recursion stack, and makes the DP table explicit.
Top-down (Memoization): Same O(n²) time (inner loop over teleporters), but O(n) recursion depth. Easier to derive directly from the recurrence, but the inner scan for "who teleports to me" is awkward compared to "where do I teleport from here."

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the DP Signal

"We want the minimum cost path in a graph where decisions at each node affect future costs. These two signals — optimisation + sequential decisions — almost always mean DP."

Step 2: Define the State

"At node i, all I need to know is the cheapest cost to have arrived here. Future decisions only depend on current position and current cost — no need to remember the full path."

Step 3: Write the Recurrence

"dp[i+1] = min(dp[i+1], dp[i] + walk[i]) and dp[tp_dest[i]] = min(dp[tp_dest[i]], dp[i] + tp_cost[i]). Two transitions, one min — that's the full recurrence."

Step 4: Choose Tabulation (Bottom-Up)

"Because edges only go forward (or teleport to known targets), processing left to right ensures each dp[i] is finalized before it's used. No Dijkstra needed if all weights non-negative."

🗣️ What to Say to the Interviewer

"This is a shortest-path problem on a DAG, which maps directly to DP.
Let dp[i] = minimum cost to reach node i. Base: dp[0] = 0.
At each node i, I have two choices: walk to i+1 (cost dp[i] + w[i]) or teleport to tp_dest[i] (cost dp[i] + tp_cost[i]).
I process nodes left to right. Since edges go forward, dp[i] is final when I process it.
Time: O(n) for the main loop plus O(1) per node = O(n). Space: O(n) for the dp array.
Edge case: if a teleport goes to a lower index, we'd need to handle cycles — that changes it to Bellman-Ford."
🔮 Likely Follow-up Questions
  • "What if teleports can go backwards?" → Now there are cycles — use Dijkstra or Bellman-Ford
  • "What if edge costs can be negative?" → Bellman-Ford or detect negative cycles
  • "What if there are multiple teleports per node?" → Add one more loop inside, still O(nk)
  • "Print the actual path, not just cost?" → Maintain a parent array, reconstruct from end to start

7 Summary & Key Takeaways ▾
State = node index; value = minimum cost to reach that node
Two transitions per node: walk (+edge cost) and teleport (+tp cost to dest)
Left-to-right order works when all transitions go forward (non-negative costs, forward teleports)
Relax with min() — never blindly overwrite; only update if strictly cheaper
Bottom-up tabulation is preferred over recursion for this problem — cleaner and avoids stack depth issues
ApproachTimeSpaceNotes
Brute Force DFS O(2ⁿ) O(n) Tries all walk/teleport combos
Top-Down Memo O(n²) O(n) Inner scan for "who teleports to me"
Bottom-Up DP ✓ O(n) O(n) Optimal; single left-to-right pass
Dijkstra (general) O(n log n) O(n) Works for backward teleports too
🎯 When to Recognize DP with State Transitions
  • "Minimum/maximum cost to reach a node/state" → DP
  • At each step you choose one of several actions, each with a cost → State Machine DP
  • Optimal path depends only on current state (Markov property) → DP, not greedy
  • If the graph has cycles or backward edges → Dijkstra or Bellman-Ford instead
