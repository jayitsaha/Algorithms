LeetCode #1268

Search Suggestions System

Given a list of products and a search word, return at most 3 lexicographically smallest products with the search word prefix for each character typed.

Medium Tries Trie + DFS Top 3 Time: O(M¬∑L + N¬∑L) Space: O(M¬∑L)
1 Understanding the Problem ‚ñæ

Imagine typing "mouse" into a search bar one character at a time. After each keystroke, the autocomplete system should return up to 3 product names that begin with what you've typed so far, ordered alphabetically. This problem asks us to build that system.

üí° Key Insight

A Trie (prefix tree) is the natural structure for prefix-based lookups. If we sort the products first, then insert them into the Trie, each node along any word's path can cache up to 3 suggestions ‚Äî because the first 3 insertions at any node are already the 3 lex-smallest products sharing that prefix.

Worked example:
products = ["mobile", "mouse", "moneypot", "monitor", "mousepad"], searchWord = "mouse"

(3 lex-smallest of all 5)

(same ‚Äî all still start "mo")

‚úì After typing "mou"

["mouse", "mousepad"]
(only 2 products start with "mou")

‚úì After typing "mous" / "mouse"

["mouse", "mousepad"]
(same 2 products)

Edge Cases: empty product list (all empty suggestions), searchWord prefix with no matches part-way through (all subsequent steps are also []), very long searchWord exceeding all product lengths, duplicate product names, all products sharing the same prefix.

üîë Two Approaches Worth Knowing

Approach A ‚Äî Trie + cached suggestions: Build the Trie once O(M¬∑L). Each query step O(L). Space O(M¬∑L). Best when the product catalogue is fixed and queried repeatedly.

Approach B ‚Äî Sort + Binary Search: Sort O(M log M). Each query O(N ¬∑ log M). Space O(1) extra. Simpler code, great for one-off queries. Both are valid interview answers ‚Äî propose B first, then offer A as the scalable solution.

2 The Algorithm: Trie + Cached Top-3 Suggestions ‚ñæ

The Trie stores words character by character. The key optimisation: every node along a product's path caches that product in its suggestions list ‚Äî but only if the list has fewer than 3 items. Since products are inserted in sorted order, the first 3 products to reach any node are the 3 lex-smallest products with that prefix.

1 Sort the products list lexicographically. This is the critical prerequisite. Sorting ensures that when we insert products one-by-one, the first 3 that pass through any Trie node are guaranteed to be the 3 lex-smallest.
2 Build the Trie: For each product, walk its characters from the root, creating new nodes as needed. At every node along the path, if len(node.suggestions) < 3, append the product. This bounds each node's suggestions list at exactly 3 items.
3 Walk the search prefix: Starting from the root, follow one character of searchWord at a time. At each step, if the child node exists, add node.suggestions to the result. If the child is missing, set a dead flag ‚Äî all remaining steps return [].
4 Return the result list ‚Äî one sublist of up to 3 suggestions per character of searchWord.

üí° Why Sort Before Inserting?

Because we cap suggestions at 3, only the first 3 products to arrive at any node get stored. Sort beforehand so those first 3 are the lex-smallest. This eliminates any need for sorting during or after search ‚Äî suggestions are already in order.

3 Why It Works: The Invariant ‚ñæ
üîê The Invariant

At any Trie node reached by path p, node.suggestions holds exactly the up to 3 lexicographically smallest products in the entire sorted product list whose string starts with p. This property is established during insertion and requires no maintenance during search.

Why the invariant holds (proof sketch):
Let P = all products with prefix p, sorted lexicographically: [p1, p2, p3, ...]. Because we insert the full sorted product list in order, p1 is inserted first (it's smallest), then p2, then p3. Each of p1, p2, p3 passes through node N and gets appended to N.suggestions. When p4 arrives, len == 3, so it is skipped. Therefore N.suggestions == [p1, p2, p3] exactly.

Why "dead" flag is correct:
If at some prefix p no Trie child exists for the next character, no product shares prefix p+c. Since any longer prefix must include p+c as a substring, no product can match any longer prefix either. So we short-circuit all remaining steps.

‚öÝÔ∏è Common Mistake: Skipping the Sort

If you insert products in an unsorted or arbitrary order and still cap at 3, you'll store the first 3 products to arrive ‚Äî which may not be the lex-smallest. The sort step is not optional; it's what makes the cap work correctly. Always call products.sort() before building the Trie.

Binary Search alternative (for completeness):
After sorting, for each prefix, bisect_left finds the insertion point ‚Äî the first index where a product could start with or equal the prefix. Then scan forward collecting at most 3 products that start with the prefix. This works because sorted order groups all prefix-matching products consecutively.

4 The Code (Annotated) ‚ñæ

Solution 1 ‚Äî Trie + Cached Suggestions (Interview Pick)

class TrieNode:
def __init__(self): ‚Üê Each node: 26 children + cached suggestions
self.children = {} ‚Üê char ‚Üí child TrieNode
self.suggestions = [] ‚Üê up to 3 lex-smallest products through here
def suggestedProducts(products, searchWord):
products.sort() ‚Üê CRITICAL: must sort before inserting
root = TrieNode()
for product in products:
node = root ‚Üê Reset to root for each new product
for ch in product:
if ch not in node.children:
node.children[ch] = TrieNode() ‚Üê Create node on first visit
node = node.children[ch] ‚Üê Descend to next char
if len(node.suggestions) < 3: ‚Üê Cap at 3 (sorted order ensures lex-smallest)
node.suggestions.append(product)
result = []
node = root
dead = False ‚Üê Once off-Trie, all future prefixes are also []
for ch in searchWord:
if dead or ch not in node.children:
dead = True ‚Üê Short-circuit remaining steps
result.append([]) ‚Üê No product matches this prefix
else:
node = node.children[ch]
result.append(node.suggestions) ‚Üê Pre-computed answer at this node
return result
‚öÝÔ∏è Why Returning node.suggestions Directly is Safe

The suggestions list is built once during Trie construction and never modified during search. So returning it by reference is fine ‚Äî no copying needed. If this were a mutable system (products added/removed), you'd want to return a copy to prevent aliasing bugs.

Solution 2 ‚Äî Sort + Binary Search (Simpler, O(1) Space)

import bisect
def suggestedProducts(products, searchWord):
products.sort() ‚Üê Sort once ‚Äî O(M log M)
result, prefix = [], ""
for ch in searchWord:
prefix += ch ‚Üê Build prefix one char at a time
i = bisect.bisect_left(products, prefix)‚Üê First index >= prefix
sugg = []
for j in range(i, min(i + 3, len(products))):
if products[j].startswith(prefix): ‚Üê Guard: must actually start with prefix
sugg.append(products[j])
result.append(sugg)
return result
üí° Complexity Summary

Trie: Build O(M¬∑L), Search O(N¬∑L), Space O(M¬∑L). Best for repeated queries.
Binary Search: Sort O(M log M), Search O(N¬∑(log M + L)), Space O(1). Best for one-off queries.
M = # products, L = avg product length, N = len(searchWord).

5 üé§ Interview Strategy: How to Arrive at This ‚ñæ
Step 1: Spot the Prefix Pattern

Key words: "starts with", "each character typed", "suggestions". These are prefix-lookup signals. When you see prefix + "top K" results ‚Üí think Trie with cached suggestions per node.

Step 2: Propose Binary Search First (Simpler)

"Quick approach: sort products, binary search for each prefix, collect up to 3 matching. O(M log M + N log M) time, O(1) space. Simple and correct." Show you know the simple solution before the complex one.

Step 3: Motivate the Trie

"If we query many different search words on the same catalogue, building a Trie once pays off ‚Äî each search prefix lookup is O(N¬∑L), not O(N¬∑log M). The trade-off is O(M¬∑L) space."

Step 4: Highlight the Sort-Before-Insert Trick

"Sort first. Then cap each node at 3 suggestions. The first 3 products to reach any node are the 3 lex-smallest ‚Äî no further sorting needed. This is the elegant trick that makes the solution clean."

üó£Ô∏è What to Say to the Interviewer

"This is a prefix search problem ‚Äî for each prefix of searchWord, I need the 3 lex-smallest products that start with it.
Simple approach: sort products, binary-search per prefix, O(M log M + N log M), O(1) space.
For better per-query performance: Trie. Sort first, insert all products, cap suggestions at 3 per node.
During search, follow each character ‚Äî if the child exists, read pre-cached suggestions; if not, flag dead and emit [] for the rest.
Build: O(M¬∑L). Search: O(N¬∑L). Space: O(M¬∑L). The key insight: sorting before inserting means the first 3 words reaching any node are automatically lex-smallest."
üîÆ Likely Follow-up Questions
  • "Top-K by popularity, not alphabetically?" ‚Üí Store (frequency, name) pairs per node, use a max-heap of size K.
  • "What if products are added/removed dynamically?" ‚Üí Maintain a sorted set per node instead of a fixed list; re-sort on updates.
  • "Implement autocomplete for a search engine." ‚Üí Same Trie pattern with frequency-weighted suggestions + LRU cache per prefix.
  • "Can you reduce space?" ‚Üí Compressed Trie (Radix Tree) reduces nodes for long shared prefixes.

7 Summary & Key Takeaways ‚ñæ
Sort products before building the Trie ‚Äî this makes the first 3 words at any node automatically lex-smallest
Cache up to 3 suggestions per node ‚Äî O(1) retrieval at each search step instead of DFS per query
The "dead" flag short-circuits all remaining steps once no Trie child is found
Binary search is simpler and uses O(1) space; Trie is faster per-query when the catalogue is reused
In an interview: propose binary search first, then offer Trie as the scalable alternative
