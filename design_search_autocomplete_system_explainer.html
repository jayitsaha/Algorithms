LeetCode #642

Design Search Autocomplete System

Design a search autocomplete system that returns the top 3 historical hot sentences matching the typed prefix, ranked by frequency then lexicographically.

Hard Design Trie Priority Queue Input: O(L²) per char Space: O(N·L)
1 Understanding the Problem ▾

You're building the autocomplete dropdown you see in every search bar — when you type "i l", it suggests "i love you", "i love leetcode", etc. The system must be stateful: it remembers what sentences you've typed before (with frequency counts) and offers the top 3 completions live, one character at a time.

💡 Key Insight

This problem combines two data structures working together: a Trie to efficiently find all sentences matching a typed prefix, and a Min-Heap (priority queue) to efficiently extract the top 3 by (frequency, lexicographic order) from potentially many candidates. The trie routes the query; the heap ranks the results.

The two operations we must support:

input(c) — one character at a time

If c ≠ '#': append c to current prefix, return top 3 matching sentences
If c = '#': user pressed Enter — save current prefix as a sentence, increment its frequency, reset prefix to ""

Initialization

Seed the system with sentences array and times array — each sentence[i] has been searched times[i] times historically. We must preload all of them into the trie.

Ranking rule: Top 3 sentences that: (1) share the typed prefix, (2) sorted by frequency descending, (3) ties broken by lexicographic (alphabetical) order ascending.

🔑 The Stateful Session Design

Between consecutive input() calls (before a '#'), the system is mid-session — building up a prefix character by character. The '#' character ends the session: we record the completed sentence and start fresh. Every character typed triggers a full prefix lookup returning up to 3 results.

Example walkthrough:

Init: ["i love you", "island", "ironman", "i love leetcode"] with times [5, 3, 2, 2]
input('i') → prefix="i" → ["i love you"(5), "island"(3), "ironman"(2)]
input(' ') → prefix="i " → ["i love you"(5), "i love leetcode"(2)]
input('a') → prefix="i a" → [] (no sentence starts with "i a")
input('#') → save "i a" with freq=1, reset prefix → []
2 The Algorithm: Trie + Priority Queue ▾

The solution breaks into two independent concerns: prefix lookup (handled by Trie) and top-K extraction (handled by a min-heap). Understanding each piece separately makes the design clear.

User Input

Character stream 'i', ' ', 'l', 'o', ...
Each char triggers a lookup

→
Trie

Navigate to prefix node, collect ALL descendant sentences in O(subtree size)

→
Min-Heap

From all candidates, extract top 3 by (freq↓, lex↑)

How the Trie is structured: Each node has up to 27 children (a–z plus space). When a complete sentence is stored, its terminal node holds the sentence text and its frequency count. During insertion, we walk character by character, creating nodes as needed.

1 Build the Trie: For each (sentence, count) pair from initialization, insert the sentence into the trie. At the terminal node, store the sentence string and its frequency.
2 On input(c) where c ≠ '#': Append c to the current prefix. Navigate the trie from root to the node representing the full current prefix. If no such path exists, return []. Otherwise, do a DFS from that node to collect all sentences in the subtree.
3 Rank with a Min-Heap: Push all collected (freq, sentence) pairs into a min-heap of size 3. The heap compares by (-freq, sentence) so the "minimum" element is actually the least desirable — it gets evicted when we have more than 3 candidates.
4 On input('#'): The current prefix is now a completed sentence. Insert it (or increment its count) in the trie. Reset the prefix to "".
⚠️ Why a Min-Heap (not Max-Heap) for Top-K?

Counterintuitively, a min-heap of size K gives you the top K efficiently. We store (-freq, sentence) so the smallest heap element is the least desirable. As we process candidates, if a new candidate beats the heap's minimum, we replace it. At the end, the heap holds exactly the top K. Max-heap would work too but requires different eviction logic.

Trie node definition:

class TrieNode:
def __init__(self):
self.children = {} ← dict: char → TrieNode (sparse, up to 27 children)
self.sentence = "" ← non-empty only at terminal nodes (end of a sentence)
self.freq = 0 ← how many times this sentence has been searched
3 Why It Works: The Invariant ▾
🔐 The Trie Invariant

In a Trie, every node on the path from root to a terminal represents a prefix of the stored sentence. Therefore, all sentences in the subtree rooted at node N share the prefix path that leads to N. This is exactly what we need: navigate to the prefix node, collect the subtree, get all matching sentences.

Why DFS from prefix node gives all matches: Every terminal node in the subtree of the prefix node represents a sentence that starts with our typed prefix. There is no way a matching sentence could exist outside this subtree — its first characters would have to travel the same trie path as the prefix, landing exactly at or below the prefix node.

Why the heap correctly extracts top 3: We use Python's heapq which is a min-heap. We store tuples (-freq, sentence). When two sentences have the same frequency, Python compares the second element — the sentence string — lexicographically. Smaller lexicographic value = smaller tuple = min-heap keeps it over the larger one. By maintaining only 3 items in the heap, we efficiently discard less desirable candidates.

💡 The Tie-Breaking Trick

We negate the frequency because min-heap pops the smallest, but we want the largest frequency. For ties, Python's string comparison is already lexicographic — "a" < "b" — which matches our requirement of sorting by lexicographic order when frequencies are equal. No extra comparison logic needed!

⚠️ Edge Cases
  • Prefix not in trie: Navigate returns None — return [] immediately
  • Fewer than 3 matches: Return all that exist (no need to pad)
  • New sentence via '#': Insert with freq=1 (may already exist with freq=N → bump to N+1)
  • Space character: Treated as a regular character in trie children (use a dict, not array, to avoid wasting 26 slots)
4 The Code (Annotated) ▾

Solution 1 — Trie + Min-Heap (Optimal, Interview Pick)

import heapq
class TrieNode:
def __init__(self):
self.children = {} ← char → TrieNode (dict handles spaces + letters)
self.sentence = "" ← non-empty marks this as a terminal node
self.freq = 0 ← search frequency for this sentence
class AutocompleteSystem:
def __init__(self, sentences, times):
self.root = TrieNode() ← root of the trie (represents empty prefix)
self.prefix = "" ← accumulates typed chars between '#' resets
for s, t in zip(sentences, times):
self._insert(s, t) ← seed the trie with historical data
def _insert(self, sentence, freq):
node = self.root
for c in sentence: ← walk each character
if c not in node.children:
node.children[c] = TrieNode() ← create missing nodes
node = node.children[c]
node.sentence = sentence ← mark terminal with full sentence
node.freq += freq ← += handles re-inserting an existing sentence
def _dfs(self, node, results): ← collect all sentences in subtree
if node.sentence: ← this node is a terminal
results.append((-node.freq, node.sentence)) ← negate for min-heap trick
for child in node.children.values():
self._dfs(child, results) ← recurse into all children
def input(self, c):
if c == '#': ← sentence completion signal
self._insert(self.prefix, 1) ← save/increment this sentence
self.prefix = "" ← reset for next session
return []
self.prefix += c ← extend the current prefix
node = self.root ← navigate to prefix node
for ch in self.prefix:
if ch not in node.children:
return [] ← prefix path broken → no matches
node = node.children[ch]
results = []
self._dfs(node, results) ← collect all sentences in subtree
return [s for _, s in heapq.nsmallest(3, results)] ← top 3: smallest(-freq,lex)
⚠️ Why heapq.nsmallest(3, results) and not sorted(results)[:3]?

heapq.nsmallest(k, iterable) runs in O(N log k) time — much better than sorted's O(N log N) when k (=3) is much smaller than N. For autocomplete with many potential matches, this matters. Both return the same answer, but nsmallest is the interview-preferred idiom for "top K from large collection".

Solution 2 — HashMap + Sorting (Brute Force)
Skip the trie entirely. Store all sentences in a dict {sentence: freq}. On each input, iterate through all sentences, filter those starting with the prefix, sort by (-freq, sentence), return first 3.

class AutocompleteSystem:
def __init__(self, sentences, times):
self.counts = {} ← {sentence: frequency}
for s, t in zip(sentences, times):
self.counts[s] = t
self.prefix = ""
def input(self, c):
if c == '#':
self.counts[self.prefix] = self.counts.get(self.prefix, 0) + 1
self.prefix = ""; return []
self.prefix += c
p = self.prefix
matches = [(v, k) for k, v in self.counts.items() if k.startswith(p)] ← O(N·L)
matches.sort(key=lambda x: (-x[0], x[1])) ← O(N log N)
return [s for _, s in matches[:3]]
💡 Complexity Comparison

Trie + Heap: Init O(N·L), each input() O(L + S log 3) where S = subtree size
HashMap + Sort: Init O(N), each input() O(N·L + N log N) — degrades with scale
The trie wins when queries are frequent and N is large (real search systems).

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Design Pattern Signals

"The problem says Design and gives us a class with methods — this is a systems-design-lite problem. The key question is: what data structure supports prefix lookup efficiently?" The answer for strings/prefixes is almost always a Trie. If you need ranking, add a heap.

Step 2: Propose Brute Force First

"My simplest approach: store sentences in a dict with their frequencies. On each character, iterate all sentences, filter startswith, sort, return top 3. O(N·L) per character typed. This works but is slow for large N — can we do better?"

Step 3: Motivate the Trie

"The bottleneck is the prefix scan over all N sentences. A Trie eliminates this: instead of checking every sentence, we navigate directly to the prefix node in O(L) steps, then collect only sentences in that subtree. The trie organizes data by prefix structure — that's the key insight."

Step 4: Address the '#' Edge Case Explicitly

"When '#' is received, we need to: (1) save the completed prefix as a sentence, (2) increment its frequency in the trie using freq += 1, (3) reset prefix to '''. A new sentence gets freq=1; an existing one gets bumped. The _insert(sentence, 1) method handles both via node.freq += freq."

🗣️ What to Say to the Interviewer

"I see two sub-problems: prefix lookup and ranking. I'll use a Trie for prefix lookup and a min-heap for ranking.
In the Trie, each terminal node stores the full sentence and its frequency count.
On input(c): append c to prefix, navigate the trie to the prefix node, DFS to collect all sentences in the subtree, use heapq.nsmallest(3) with key (-freq, sentence) to get top 3.
On input('#'): insert current prefix with freq += 1 into the trie, reset prefix to ''.
Time: O(L) to reach prefix node + O(S) to collect matches + O(S log 3) to rank — S is subtree size.
Space: O(N·L) for the trie where N = number of sentences, L = average length."
🔮 Likely Follow-up Questions
  • "What if we want top-5 instead of top-3?" → Same code, change 3 to 5. Heap size is parameterized.
  • "How to handle very large datasets?" → Cache suggestions at each trie node to avoid re-DFS; trade space for time.
  • "What about case-insensitive matching?" → Normalize to lowercase on insert and on prefix lookup.
  • "Can you store top-3 at each trie node?" → Yes — update node's sorted top-3 list on every insert. O(1) lookup but O(N·L) update cost per insertion.

7 Summary & Key Takeaways ▾
Trie organizes strings by prefix — navigate to prefix node in O(L) steps
DFS from prefix node collects ALL matching sentences (any in subtree)
Min-heap with (-freq, sentence) tuples extracts top-K with correct tie-breaking
heapq.nsmallest(3, results) is O(N log 3) — better than full sort O(N log N)
'#' input → save sentence (insert with freq=1), reset prefix — stateful design
Dict-based children (not array) handles spaces and any character set
ApproachInit Timeinput() TimeSpace
HashMap + Sort O(N) O(N·L + N log N) O(N·L)
Trie + Heap ✓ O(N·L) O(L + S log 3) O(N·L)
Trie + Cached Top-3 O(N·L²) O(L) O(N·L·K)
🎯 When to Recognize the Trie + Priority Pattern
  • "Design a system with prefix lookups" → Trie for navigation
  • "Return top K results from a large collection" → Min-heap of size K
  • "Rank by multiple criteria (freq then lex)" → Tuple comparison in heap
  • "Stateful input that accumulates a string" → Track prefix, reset on delimiter
