LeetCode #1653

Minimum Deletions to Make String Balanced

Find the minimum number of characters to delete so that no 'b' appears before any 'a' in a string of only 'a's and 'b's.

Medium Dynamic Programming Track Count and Deletions Time: O(n) Space: O(1)
1 Understanding the Problem ▾

A string of 'a' and 'b' characters is called balanced if all the 'a's appear before all the 'b's. That means the string has the form aaa...abbb...b — some (possibly zero) 'a's followed by some (possibly zero) 'b's. "aabb", "aaa", "bbb", and even "" are all balanced.

Given a string s, we want to find the minimum number of characters to delete to make it balanced. We cannot reorder characters — only delete.

💡 Key Insight

Think of balance as a cut point: there is some index i (0 through n) where everything to the left is 'a' (or deleted) and everything to the right is 'b' (or deleted). For any fixed cut position, the minimum deletions = b's on the left of cut + a's on the right of cut. We want the best cut.

✗ Not Balanced: "aababbab"

a a b a b b a b
Several 'b's precede 'a's
Needs 2 deletions

✓ Balanced after 2 deletes

a a [b] a [b] b a b → "aaaabb"
Delete b@2 and b@4
All a's now precede b's

Concrete example: s = "aababbab"

  • Try cut at index 3: b's on left = 1 (index 2), a's on right = 2 (indices 3, 6) → cost = 3
  • Try cut at index 4: b's on left = 2 (indices 2,3... wait, 3='a'). b's left = 1, a's right = 2 → cost = 3
  • Try cut at index 7: b's on left = 4 (indices 2,4,5,7→wait). Let's just run the algorithm!
  • Optimal answer: 2
⚠️ Edge Cases
  • All 'a's ("aaaa") → already balanced, answer = 0
  • All 'b's ("bbbb") → already balanced, answer = 0
  • Single char → balanced, answer = 0
  • "ba" → delete 'b' or 'a', answer = 1
  • "bbbbaaa" → delete all 4 b's or all 3 a's → answer = min(4,3) = 3
2 The Algorithm: One-Pass DP with Running State ▾

We scan left to right, maintaining two running values: b_count (number of 'b's seen so far that haven't been accounted for in a deletion) and deletions (minimum deletions to balance everything we've seen so far). At each character we make a local optimal decision.

1 Initialize: b_count = 0, deletions = 0. No characters seen yet, nothing to fix.
2 If current char is 'b': It extends the 'b' region. Increment b_count += 1. No deletion required yet — this 'b' might end up on the right side of the cut.
3 If current char is 'a' and b_count > 0: Conflict! There's at least one 'b' to our left that would violate balance. Best fix: delete one 'b' from the pool (b_count -= 1, deletions += 1). This is always at least as good as deleting this 'a' — same cost, but reduces future conflict risk.
4 If current char is 'a' and b_count == 0: No conflict. This 'a' can sit safely to the left of all b's seen. No action.
5 Return deletions after scanning the entire string.
📐 The Recurrence Relations
When s[i] == 'b': b_count += 1
When s[i] == 'a': if b_count > 0:
                       b_count -= 1   (absorb one 'b' into deletions)
                       deletions += 1
                 else: no action (this 'a' is already safe)

Alternative view — Prefix/Suffix Array DP:

For each cut point i (0 to n), cost = prefix_b[i] + suffix_a[i] where prefix_b[i] = count of b's in s[0..i-1] and suffix_a[i] = count of a's in s[i..n-1]. The one-pass approach computes this minimum without storing arrays.

3 Why It Works: DP Invariant & Proof ▾
🔐 Why Is This Dynamic Programming?

Optimal Substructure: The minimum deletions for s[0..i] can be derived from the minimum deletions for s[0..i-1] plus one local decision about s[i]. We don't need to look further back.

Overlapping Subproblems: A naive recursive approach would repeatedly recompute "best solution for prefix of length k." By keeping two running variables, we effectively cache the answer in O(1) space — the DP table is implicit.

💡 The Invariant

At any point after processing s[0..i-1]:
• deletions = minimum deletions to make s[0..i-1] balanced
• b_count = number of 'b's in s[0..i-1] that are currently "on the right side of the cut" (i.e., after the last 'a' we kept) and available for retroactive deletion

This invariant is maintained by construction at each step.

Why absorbing 'b' is always optimal over deleting 'a':

When we encounter 'a' with b_count > 0: both options cost exactly +1 deletion. But absorbing a 'b' also decreases b_count by 1, meaning there's one fewer 'b' in the pool that could conflict with future 'a's. Deleting this 'a' instead leaves b_count unchanged — all those same 'b's are still there to potentially conflict again. So absorbing 'b' is at least as good and usually strictly better.

⚠️ Does This Need a 2D DP Table?

First instinct: dp[i][j] = min deletions for s[0..i] with cut at j. That's O(n²) states! The insight is that we don't need to track the cut position explicitly. The variable b_count captures everything we need: "how many misplaced b's are available for retroactive deletion." This collapses the 2D problem to two O(1) variables.

Connection to prefix/suffix formulation:

At cut point i: cost = (b's in s[0..i-1]) + (a's in s[i..n-1]) = prefix_b[i] + suffix_a[i]. The one-pass algorithm implicitly minimizes this by "greedily" choosing the best cut position as it scans — when we absorb a 'b', we're effectively moving the cut one step right; when b_count is 0, the cut is at the current position.

4 The Code (Annotated) ▾

Solution 1 — One-Pass DP (Interview Pick) — O(n) time, O(1) space

def minimumDeletions(s: str) -> int:
b_count = 0 ← pool of 'b's available for retroactive deletion
deletions = 0 ← running minimum deletions for processed prefix
for c in s:
if c == 'b':
b_count += 1 ← add to pool; no deletion needed yet
elif b_count > 0: # c == 'a'
b_count -= 1 ← absorb one 'b' from pool (retroactively delete it)
deletions += 1 ← that absorption costs 1 deletion
# else: c == 'a' and b_count == 0 → no conflict, keep 'a'
return deletions
⚠️ Why no else clause for 'a' when b_count == 0?

When b_count = 0, the 'a' we're looking at has no 'b' before it in the current pool. It's already "on the correct side of the cut." We simply skip it — no deletion, no pool update needed. Adding a deletion here would overcount.

Solution 2 — Prefix + Suffix Arrays (Clearest to explain) — O(n) time, O(n) space

def minimumDeletions(s: str) -> int:
n = len(s)
prefix_b = [0] * (n + 1) ← prefix_b[i] = # of b's in s[0..i-1]
for i in range(n):
prefix_b[i+1] = prefix_b[i] + (s[i] == 'b') ← running count
suffix_a = [0] * (n + 1) ← suffix_a[i] = # of a's in s[i..n-1]
for i in range(n - 1, -1, -1):
suffix_a[i] = suffix_a[i+1] + (s[i] == 'a') ← reverse pass
# For each cut point, cost = b's on left + a's on right
return min(prefix_b[i] + suffix_a[i] for i in range(n + 1))

Solution 3 — Top-Down Memoization — O(n) time, O(n) space

from functools import lru_cache
def minimumDeletions(s: str) -> int:
@lru_cache(maxsize=None)
def dp(i, b_count):
# min deletions for s[i:] given b_count unresolved b's before it
if i == len(s): return 0 ← base: empty suffix, done
if s[i] == 'b':
return dp(i + 1, b_count + 1) ← accumulate into pool
# s[i] == 'a': conflict if b_count > 0
if b_count == 0:
return dp(i + 1, 0) ← no conflict, keep 'a'
delete_a = 1 + dp(i + 1, b_count) ← delete this 'a'
delete_b = 1 + dp(i + 1, b_count-1) ← delete one 'b' from pool
return min(delete_a, delete_b)
return dp(0, 0)
💡 Why Solutions 1 & 3 are equivalent

In the memoized version, when both delete_a and delete_b have equal cost (+1), we still pick delete_b because it reduces b_count for future calls. This is exactly what Solution 1 does greedily — they always make the same choice, and Solution 1 skips storing the cache because the state is fully captured in two integers.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Clarify What "Balanced" Means

"Balanced = all a's before all b's = there exists a cut point i such that s[0..i-1] are all 'a' and s[i..n-1] are all 'b' (after deletions). Confirm: we can delete any character, but not reorder."

Step 2: Propose Brute Force — O(n²)

"For each cut point i from 0 to n: count b's in s[0..i-1] and a's in s[i..n-1] in O(n) each. Total O(n²). Correct but too slow for n up to 10⁵."

Step 3: Optimize with Prefix Arrays — O(n)

"Precompute prefix_b[i] = b count in s[0..i-1] and suffix_a[i] = a count in s[i..n-1]. Then each cut evaluation is O(1). Total O(n) time, O(n) space."

Step 4: Collapse to O(1) Space — One-Pass DP

"I notice that as I scan, I can maintain b_count (b pool) and deletions incrementally. When I see 'a' after b's, absorbing a b is always optimal — same cost, fewer future conflicts. Single pass, constant space."

🗣️ What to Say to the Interviewer

"Balanced means all a's before all b's — equivalent to picking a cut point.
Cost of any cut = b's on left + a's on right. I want to minimize over all n+1 cut positions.
Brute force: O(n²). Optimize with prefix_b and suffix_a arrays: O(n) time, O(n) space.
Best: scan once, track b_count (pool of b's available to delete) and running deletions.
See 'b': add to pool. See 'a' with pool non-empty: absorb one b (both cost 1, absorbing is never worse).
Final answer: deletions. Time O(n), Space O(1) — optimal."
🔮 Likely Follow-up Questions
  • "Return the actual resulting string?" → Track which characters to mark deleted (indices where you absorbed or skipped)
  • "What if we could also swap adjacent characters?" → Different problem; count inversions
  • "String has a, b, c — make all a's before b's before c's?" → 2D DP with states (a_count, b_count)
  • "Minimum swaps instead of deletions?" → Count inversions — number of (b,a) pairs where b comes before a

7 Summary & Key Takeaways ▾
Balanced string = all 'a's before all 'b's = there exists a valid cut point
Cost at cut i = b's on left + a's on right; minimize over all n+1 cut positions
One-pass DP: b_count is the pool of b's available for retroactive deletion
When 'a' appears after some b's: absorbing one 'b' is always ≥ as good as deleting this 'a'
Space collapses from O(n) prefix arrays to O(1) running integers — same logical computation
This exact pattern (track count + optimize deletion cost) recurs in many string DP problems

  • Optimization over cut points or partition positions in a string
  • Running count of one type can absorb cost of another — suggests O(1) space DP
