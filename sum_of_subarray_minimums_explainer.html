LeetCode #907

Sum of Subarray Minimums

Given an array of integers arr, find the sum of min(b) for every contiguous subarray b of arr. Return the answer modulo 10⁹ + 7.

Medium Monotonic Stack Left/Right Boundaries Time: O(n) Space: O(n)
1 Understanding the Problem ▾

For every contiguous subarray of arr, find its minimum, then sum all those minimums. For arr = [3, 1, 2, 4] there are 10 subarrays — every possible (start, end) pair.

All 10 subarrays of [3, 1, 2, 4]

[3]→3, [1]→1, [2]→2, [4]→4
[3,1]→1, [1,2]→1, [2,4]→2
[3,1,2]→1, [1,2,4]→1
[3,1,2,4]→1
Sum = 3+1+2+4+1+1+2+1+1+1 = 17

Why naive is too slow

n=3×10⁴ → n²=9×10⁸ subarrays
Even O(n²) DP: too slow

💡 The Contribution Flip

Instead of asking "what is the minimum of each subarray?", ask "for each element arr[i], how many subarrays have arr[i] as their minimum?" If arr[i] is the min of exactly k subarrays, it contributes arr[i] × k. Sum contributions from every element. This transforms the problem from iterating subarrays to finding boundaries around each element.

Edge Cases: single element (answer = arr[0] mod MOD), all equal elements (requires careful duplicate tie-breaking), array with negative numbers (algorithm still works — we only compare, never assume positivity).

2 The Algorithm: Contribution Counting with Monotonic Stack ▾

For element arr[i] to be the minimum of a subarray, that subarray must not extend left past any element strictly smaller than arr[i], and must not extend right past any element smaller or equal to arr[i] (to handle duplicates). We need to efficiently find those boundary indices.

1 Compute left[i] — "previous strictly smaller" distance: Scan left to right with a monotone increasing stack. When we arrive at i, pop all indices with arr[stack.top()] ≥ arr[i]. The remaining top is the previous smaller element. left[i] = i − top (or i+1 if stack is empty).
2 Compute right[i] — "next smaller or equal" distance: Scan right to left with the same stack, but pop when arr[stack.top()] > arr[i] (strictly greater). right[i] = top − i (or n−i if stack empty). The asymmetry in comparisons is the duplicate tie-breaking trick.
3 Sum contributions: For each i, arr[i] is the minimum of exactly left[i] × right[i] subarrays (any combination of left endpoint and right endpoint within the allowed ranges). Add arr[i] × left[i] × right[i] to the running total, taking modulo 10⁹+7 at the end.
🔑 The Duplicate Tie-Breaking Rule

When arr has duplicates like [1, 1, 1], both 1s would claim [1,1,1] as their minimum — double-counting. Fix: for left boundaries use strictly less than (<, meaning we pop when stack.top() ≥ current), and for right boundaries use strictly greater than (>, meaning we pop only when strictly greater). This makes the leftmost equal element "own" the tie, and the rightmost one defers. Each subarray gets exactly one owner.

3 Why It Works: The Invariant ▾
🔐 The Monotone Stack Invariant

At any point during the left-to-right scan, the stack contains indices of elements in strictly increasing order of value from bottom to top. When we pop index j because arr[i] < arr[j], index i is the first element to the right of j that is smaller — i.e., the right boundary of j's "dominance region." Meanwhile, the new stack top after the pop is the left boundary of j's region.

Why left × right counts subarrays correctly: If arr[i] is the minimum of a subarray [L, R], then: (a) no element in [L, i-1] is smaller than arr[i] (otherwise that element would be the minimum), and (b) no element in [i+1, R] is smaller or equal (same reason, with tie-breaking). So L can be anything in [prev_smaller+1, i] — that is left[i] choices. R can be anything in [i, next_smaller_eq-1] — that is right[i] choices. All left[i] × right[i] combinations are valid.

Partition completeness: Every subarray has exactly one minimum (under our tie-breaking, the leftmost among ties is the "owner"). So every subarray is counted exactly once, by exactly one element i. The sum of all contributions equals the sum of minimums across all subarrays. No subarray is missed; none is double-counted.

⚠️ Common Mistake: Symmetric Comparisons

Using strictly less than for BOTH left and right passes causes double-counting on duplicates. Example: arr=[2,2]. Left pass (strict <): left=[1,2]. Right pass (also strict <): right=[2,1]. Sum = 2×1×2 + 2×2×1 = 4+4 = 8. But correct answer is 2+2+2=6 (subarrays: [2],[2],[2,2]). The asymmetric rule gives left=[1,2], right=[1,1]: sum = 2×1×1 + 2×2×1 = 2+4 = 6. Correct!

4 The Code (Annotated) ▾

Solution 1 — Brute Force, O(n²) (propose first, then optimize)

def sumSubarrayMins(arr): ← O(n²) — too slow for n=3×10⁴
MOD, total, n = 10**9+7, 0, len(arr)
for i in range(n): ← fix left endpoint
cur_min = arr[i] ← rolling minimum as right extends
for j in range(i, n): ← extend right endpoint
cur_min = min(cur_min, arr[j])
total = (total + cur_min) % MOD
return total

Solution 2 — Monotonic Stack (Optimal, O(n)) — Interview Pick

def sumSubarrayMins(arr):
MOD = 10**9 + 7
n = len(arr)
left = [0] * n ← left[i] = # left endpoint choices
right = [0] * n ← right[i] = # right endpoint choices
stack = [] ← monotone increasing stack of indices
for i in range(n): ← LEFT PASS: previous strictly smaller
while stack and arr[stack[-1]] >= arr[i]: ← >= means "not strictly smaller" → pop
stack.pop()
left[i] = i - stack[-1] if stack else i + 1← distance to prev smaller, or to start
stack.append(i)
stack = [] ← reset stack for right pass
for i in range(n - 1, -1, -1): ← RIGHT PASS: next smaller or equal
while stack and arr[stack[-1]] > arr[i]: ← strictly > means equal defers to left owner
stack.pop()
right[i] = stack[-1] - i if stack else n - i← distance to next smaller/equal or to end
stack.append(i)
return sum(arr[i]*left[i]*right[i] for i in range(n)) % MOD← sum contributions mod MOD
⚠️ The Asymmetric Comparison is Not Optional

Left pass: arr[stack[-1]] >= arr[i] (pop when top is greater-than-or-equal). Right pass: arr[stack[-1]] > arr[i] (pop only when strictly greater). This one-character difference in each while-condition is what prevents double-counting when equal elements exist.

💡 Complexity

Brute Force: Time O(n²), Space O(1)
Monotonic Stack: Time O(n) — each index pushed and popped at most once across both passes; Space O(n) for left[], right[], stack
Modulo: Apply at the final sum. Individual products arr[i]×left×right can reach ~3×10⁴³ in other languages — Python handles big ints natively but other languages need MOD applied during accumulation.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the "Contribution" Pattern

"Sum over all subarrays" of a min/max is a signal to switch perspective: instead of iterating subarrays, iterate elements and count how many subarrays each element dominates as the minimum. This flips an O(n²) enumeration into a boundary-finding problem.

Step 2: Connect Boundaries to Monotonic Stack

"Previous smaller element" and "next smaller element" are the two classic monotone stack problems. Mention them explicitly: "I'll use a monotone increasing stack to find, for each i, the nearest smaller element on each side in O(n) total."

Step 3: Proactively Address Duplicates

Before the interviewer asks: "To handle equal elements, I'll use an asymmetric comparison — strictly-less for left boundaries, less-or-equal for right. This ensures each equal group assigns ownership to the leftmost element, preventing double-counting."

Step 4: State Complexity and Modulo Reason

"Two O(n) passes, O(n) final loop → O(n) total. O(n) space for the arrays and stack. The modulo 10⁹+7 is required by the problem because the answer can be astronomically large — up to O(n³ × max_val)."

🗣️ What to Say to the Interviewer

"My brute-force: iterate all O(n²) subarrays and track the rolling minimum. O(n²) time.
To optimize, I flip the perspective: for each element arr[i], count how many subarrays have it as minimum.
That count is left[i] × right[i] — the number of valid (left, right) endpoint pairs.
I find those boundaries using a monotone increasing stack in two O(n) passes.
For duplicates, I use ≥ in the left pass and > in the right pass — leftmost equal element owns the tie.
Final answer: sum of arr[i] × left[i] × right[i] for all i, mod 10⁹+7. O(n) time, O(n) space."
🔮 Likely Follow-up Questions
  • "Sum of subarray maximums?" → Same technique, flip to "next greater" and "previous greater or equal"
  • "Can you solve in one pass?" → Yes: use a stack of (value, count) pairs and accumulate contribution on each pop
  • "Sum of (max − min) for all subarrays?" → Compute sum-of-maxima and sum-of-minima separately, subtract (#2104)
  • "Largest rectangle in histogram?" → Same left/right smaller-element boundaries, but for area computation (#84)

7 Summary & Key Takeaways ▾
Flip perspective: count subarray memberships per element, not minima per subarray
A monotone increasing stack finds previous-smaller and next-smaller boundaries in O(n)
Contribution of arr[i] = arr[i] × left[i] × right[i] — two independent choices multiplied
Asymmetric comparisons (≥ left, > right) handle duplicates — leftmost equal element owns ties
Same pattern works for sum-of-maxima by flipping comparisons to find next/prev greater
