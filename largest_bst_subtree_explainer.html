LeetCode #333

Largest BST Subtree

Given a binary tree root, find the largest subtree which is a Binary Search Tree and return its node count.

Medium Trees Post-order with Range Time: O(n) Space: O(h)
1 Understanding the Problem ▾

You are given a binary tree (NOT necessarily a BST). Your job: find the largest connected subtree rooted at some node that satisfies BST rules, and return the number of nodes in it. If the whole tree is a BST, the answer is n (total nodes).

💡 Key Insight

To check if a subtree at node X is a valid BST, you need more than just "are both children BSTs?" — you also need to know that every value in the left subtree is less than X.val, and every value in the right subtree is greater. Tracking the min and max values propagated from each subtree handles this in O(1) per node — no re-scanning needed.

Key definitions:

  • A subtree includes a node AND all its descendants — it's a connected rooted subgraph, not just any subset of nodes.
  • A BST requires every node in the left subtree to be strictly less than the root, every node in the right subtree to be strictly greater — and this must hold recursively for every node in the subtree.
  • Edge cases: A single leaf node is always a valid BST (size 1). An empty tree returns 0. Nodes with only one child are valid BSTs if that one child's whole subtree satisfies BST rules relative to the root.
2 The Algorithm: Post-order DFS with Min/Max Range ▾

The key design decision: each recursive call returns a 4-tuple (is_bst, size, min_val, max_val) for the subtree it covers. With this tuple, the parent can validate itself in O(1) — no re-traversal of descendants is needed. We gather this information bottom-up using post-order DFS.

1 Base case — null node: Return (True, 0, +∞, -∞). The sentinel values are the elegant heart of this solution: min=+∞ means "any parent value will be ≤ to this empty side's minimum" and max=-∞ means "any parent value will be ≥ to this empty side's maximum." No special-casing for leaf nodes needed.
2 Recurse post-order: Call dfs(node.left) and dfs(node.right). Unpack each into (l_bst, l_sz, l_min, l_max) and (r_bst, r_sz, r_min, r_max). We have both children's full range information now.
3 Three conditions for current node: The current subtree is a valid BST iff: (a) left subtree is BST, (b) right subtree is BST, and (c) l_max < node.val < r_min. Condition (c) uses the global max/min of each child's entire subtree — this catches violations from any depth, not just direct children.
4 If valid: Compute sz = l_sz + r_sz + 1, update global answer, and return (True, sz, min(node.val, l_min), max(node.val, r_max)). The returned min/max are the global extremes of the current subtree.
5 If invalid: Return (False, 0, 0, 0). This signals to the parent that they cannot form a larger BST by extending through this node. The parent will also return False (a BST cannot be built on an invalid subtree).
🔑 Why Sentinel Values for Null Nodes

For a leaf node X with no children: left DFS returns max=-∞, so check l_max (-∞) < X.val is always true. Right DFS returns min=+∞, so check X.val < r_min (+∞) is always true. Both children are BSTs (base case). So every leaf is automatically a valid BST of size 1 — no special case needed. This is the elegance of sentinel values.

3 Why It Works: The Invariant ▾
🔐 The Invariant

When dfs(node) returns (True, sz, min_val, max_val), it guarantees that min_val is the global minimum across ALL nodes in node's subtree, and max_val is the global maximum. These are tight bounds — not approximations. Every node in the subtree lies within [min_val, max_val].

Why checking l_max and r_min is sufficient: The BST condition requires that ALL values in the left subtree are less than the current node — not just the direct left child. If the largest value in the entire left subtree (l_max) is less than node.val, then every other value in the left subtree is also less than node.val (since l_max is the maximum). This makes the check complete without traversing the left subtree again.

Why post-order is necessary: We need l_max and r_min from the children before we can validate the current node. Pre-order or in-order would require passing constraints downward and separately collecting results — more complex. Post-order naturally gives bottom-up information flow.

⚠️ Classic Mistake: Checking Only Direct Children

It is tempting to write: "if node.left.val < node.val < node.right.val: it's a BST." This only checks the two direct children's values, not all descendants. Counterexample: a node with value 10, right child 15, right grandchild 3. Checking 15 passes, but 3 is in the right subtree and 3 < 10 — a BST violation. The l_max / r_min propagation from the entire subtree catches this correctly.

Edge-case proof — leaf node X: Both children are null → both return (True, 0, +∞, -∞). Check: l_bst=True ✓, r_bst=True ✓, l_max (-∞) < X.val ✓, X.val < r_min (+∞) ✓. All pass. X is a valid BST of size 1. Returns (True, 1, X.val, X.val). ✓

Solution 1 — Post-order DFS with Range Tracking (Optimal — Interview Pick)

ans = [0] ← List so the nested function can mutate it (Python closure)
def dfs(node): ← Returns (is_bst, size, min_val, max_val)
if not node: ← Base case: null node
return True, 0, float('inf'), float('-inf') ← Sentinel: +inf min, -inf max
l_bst, l_sz, l_min, l_max = dfs(node.left) ← Recurse left (post-order)
r_bst, r_sz, r_min, r_max = dfs(node.right) ← Recurse right (post-order)
if l_bst and r_bst and l_max < node.val < r_min:
# l_max: biggest value in entire left subtree must be < node.val
# r_min: smallest value in entire right subtree must be > node.val
sz = l_sz + r_sz + 1 ← Size = left + right + current node (1)
ans[0] = max(ans[0], sz)← Update global best if this subtree is larger
# Return global min/max of this entire subtree
return True, sz, min(node.val, l_min), max(node.val, r_max)
return False, 0, 0, 0 ← Not a BST: signal failure, parent can't extend through here
dfs(root) ← Start DFS from root; ans[0] updated as side-effect
return ans[0] ← Return the largest BST subtree size found
⚠️ Tuple Index Confusion Is the #1 Bug

The return tuple is (is_bst, size, min_val, max_val). To validate: check l_max (index 3 of left) and r_min (index 2 of right). Using unpacking names like l_min, l_max eliminates index bugs. This is worth doing explicitly in an interview — it shows careful thinking.

💡 Complexity

Time: O(n) — each node is visited exactly once.
Space: O(h) — recursion stack. O(log n) for balanced trees, O(n) worst case for a skewed tree (line of nodes).

Solution 2 — Brute Force: Validate Each Subtree Separately (O(n²))

def largestBSTSubtree_brute(root) -> int:
def is_bst(node, lo, hi): ← Standard BST validation with range
if not node: return True
if not (lo < node.val < hi): return False
return (is_bst(node.left, lo, node.val) and
is_bst(node.right, node.val, hi))
def count(node): ← Count all nodes in subtree
if not node: return 0
return 1 + count(node.left) + count(node.right)
def solve(node): ← Try every node as a BST root
if not node: return 0
if is_bst(node, float('-inf'), float('inf')):
return count(node) ← If valid BST, its whole subtree is the answer from here
return max(solve(node.left), solve(node.right)) ← Otherwise recurse into children
return solve(root)

O(n²) time: for each of the n nodes, we may do O(n) work to validate its subtree. Mention this as your starting approach, then optimize to O(n).

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Pattern — "Subtree Property"

Any time a problem asks about a property of a subtree (BST, balanced, max path, etc.), think: post-order DFS. The property at a node depends on both children's properties — so process children first, then combine. This is the classic bottom-up tree pattern.

Step 2: Define the Return Type

Ask yourself: "What does the parent need from each child to validate itself?" For BST: (1) is the child a valid BST? (2) how large is it? (3) what are the min and max values in it? This gives the 4-tuple: (is_bst, size, min_val, max_val). Design the return type before writing any code.

Step 3: Handle the Null Base Case Cleverly

A null node should return (True, 0, +inf, -inf). Walk through why with the interviewer: the sentinel values eliminate all special-casing for leaf nodes. Explain this out loud — it demonstrates you understand invariants.

Step 4: Write the Three-Condition Check

The current node is valid if: l_bst and r_bst and l_max < node.val < r_min. Emphasize to the interviewer that l_max and r_min are the global extremes of each child's entire subtree — not just the direct children's values. This is what makes the solution correct for all depths.

🗣️ What to Say to the Interviewer

"Let me start with brute force: for each node, run an O(n) BST validation, giving O(n^2) total. We can do better.

The key observation is: when we recurse into a child, we can have it return not just 'is it a BST' but also its size and the min/max values in it. With those four values, the parent can validate in O(1).

I'll use post-order DFS — children first, then combine. The null base case returns (True, 0, +inf, -inf). Those sentinel values ensure leaf nodes are automatically valid BSTs without special-casing.

At each node: if both children are BSTs and l_max < node.val < r_min, the current subtree is a valid BST. I update the global answer and propagate (True, size, min, max) upward."
💡 Follow-up Questions
  • "What about duplicate values?" — Standard BST is strict (no duplicates). If duplicates allowed, adjust to l_max <= node.val <= r_min accordingly.
  • "Can you do it iteratively?" — Yes with an explicit stack (post-order iterative), but the recursive version is clearer. Mention the space trade-off is the same O(h).
  • "What if you need to return the subtree root, not just size?" — Track the best_root as well in the global ans variable.

7 Summary & Key Takeaways ▾
Post-order DFS enables bottom-up information flow — children provide everything the parent needs in one pass
The 4-tuple return value (is_bst, size, min_val, max_val) is the core design — the parent needs all four pieces to validate in O(1)
Sentinel values (+∞ for min, -∞ for max) for null nodes eliminate all special-casing for leaves — elegant and bug-free
Checking l_max and r_min (global extremes) instead of direct children's values catches all deep violations
O(n) time — each node visited exactly once, vs O(n²) for brute-force validate-each-node approach

🎯 When to Reach for Post-order with Returned State

This pattern applies whenever: (1) the problem is about a subtree property, (2) the property at a node depends on the same property of both children, and (3) you need aggregated information from all descendants (min, max, count, height). Other prime examples: Diameter of Binary Tree, Maximum Path Sum, Balanced Binary Tree, Binary Tree Cameras, Count Good Nodes in Binary Tree.
