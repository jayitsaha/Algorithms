LeetCode #232

Implement Queue using Stacks

Design a FIFO queue using only two stacks — supporting push, pop, peek, and empty with amortized O(1) time per operation.

Easy Design Two Stacks Push: O(1) Pop/Peek: O(1) amortized Space: O(n)
1 Understanding the Problem ▾

A stack is LIFO — Last In, First Out. Think of a pile of plates: you add to the top and take from the top. A queue is FIFO — First In, First Out. Think of a line at a store: you join the back and leave from the front. These are fundamentally opposite orderings.

The challenge: implement a queue using only stack operations — push(top), pop(top), peek(top), isEmpty(). No arrays accessed by index, no doubly-linked lists, just stacks.

Stack (LIFO) — Last pushed, first out

push(1), push(2), push(3)
pop() → 3 (LAST in)
pop() → 2
pop() → 1

Queue (FIFO) — First pushed, first out

push(1), push(2), push(3)
pop() → 1 (FIRST in)
pop() → 2
pop() → 3

💡 Key Insight

A stack reverses insertion order. Two stacks cancel each other: stack A reverses the order, pouring A into stack B reverses again — restoring original insertion order. So: inbox (stack_in) receives pushes; outbox (stack_out) serves pops in FIFO order after a one-time transfer.

Required operations and expected behavior:

  • push(x) — Add element x to the back of the queue
  • pop() — Remove and return the element from the front
  • peek() — Return front element without removing it
  • empty() — Return True if the queue has no elements

Edge cases: Alternating push/pop (stress-tests lazy transfer), calling empty() with elements only in stack_in (must check both stacks), calling pop/peek on an empty queue (problem guarantees this won't happen, but mention it).

2 The Algorithm: Two Stacks — Lazy Transfer ▾

We use two stacks: stack_in (inbox) receives all new pushes, and stack_out (outbox) serves all pops and peeks. The critical design decision is when to transfer elements from inbox to outbox.

1 push(x): Always stack_in.append(x). Simple O(1) — no conditions, no transfer.
2 _transfer() helper (lazy): Only runs if stack_out is empty. Pop everything from stack_in and push to stack_out. This reverses order: the oldest element (queue front) lands at the top of stack_out.
3 pop(): Call _transfer() (no-op if stack_out not empty), then stack_out.pop(). The top of stack_out is always the queue's front element.
4 peek(): Same as pop() but use stack_out[-1] (read without removing) instead of stack_out.pop().
5 empty(): Return not stack_in and not stack_out. Must check both — an element might be in either stack.
🔑 Why "Lazy" is the Key Word

We delay the expensive transfer until strictly necessary. Once stack_out is loaded, consecutive pops cost O(1) each — no re-transfer until stack_out empties again. This amortizes the O(n) transfer cost across the n elements that will be popped from it.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At all times, the front of the queue is either the top of stack_out (if non-empty), or the bottom of stack_in (the element that has been there longest). After transfer, the bottom of stack_in becomes the top of stack_out — so we can always pop from stack_out to get FIFO order.

Why two reversals restore FIFO: Push [1, 2, 3] to stack_in → stack is [1(bottom), 2, 3(top)]. Stack pops give 3, 2, 1 — reversed. Pour into stack_out: each pop from stack_in pushes to stack_out, giving stack_out = [3(bottom), 2, 1(top)]. Now stack_out.pop() returns 1 — the element pushed first. Correct FIFO order restored.

Amortized O(1) proof sketch: Every element follows exactly this lifecycle:

  • Pushed to stack_in once — 1 operation
  • Transferred (popped from stack_in, pushed to stack_out) at most once — 2 operations
  • Popped from stack_out once — 1 operation
  • Total: 4 operations per element → O(4n) = O(n) for all n elements → O(1) amortized per operation
⚠️ "But a single pop can cost O(n)!"

True — when transfer triggers, that pop costs O(n). But the next n−1 pops each cost O(1). Think of it as pre-paying: the expensive pop "pays for" all the cheap ones. This is valid amortized analysis, and it's the expected answer in interviews. If worst-case O(1) is required, that's provably impossible with only stacks.

⚠️ Critical: empty() Must Check Both Stacks

Elements can be waiting in stack_in (not yet transferred) or ready in stack_out. Checking only one will give wrong results. If you only check stack_out: you'd return True when items are in stack_in. Always: not stack_in and not stack_out.

Solution 2 — Two Stacks, Eager Transfer (Simpler O(n) push, O(1) pop)

def push(self, x): ← O(n): transfer on every push
while self.stack_out: ← Pour outbox back to inbox
self.stack_in.append(self.stack_out.pop())
self.stack_in.append(x) ← Insert new element at bottom
while self.stack_in: ← Pour back to outbox (restores order)
self.stack_out.append(self.stack_in.pop())
def pop(self): return self.stack_out.pop() ← O(1): outbox always ready
💡 Complexity Comparison

Lazy Transfer (Sol 1): push O(1), pop/peek O(1) amortized, O(n) worst-case single pop
Eager Transfer (Sol 2): push O(n), pop O(1), peek O(1)
Interview pick: Solution 1 — amortized O(1) across all ops is the optimal design.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Observe the Core Conflict

Stack is LIFO, queue is FIFO — opposites. Think: "If I push [1,2,3] and pop immediately from a stack, I get 3,2,1. I need 1,2,3. How do I reverse the reversal?"

Step 2: One Stack Reverses, Two Restore

Key insight: "If I pour stack A into stack B, the order reverses. So if I start with LIFO order and reverse it, I get FIFO order. I need a second stack to hold the reversed sequence ready for serving."

Step 3: When Should the Transfer Happen?

Option A: Transfer on every push (O(n) push, O(1) pop). Option B: Transfer lazily only when outbox is empty (O(1) push, O(1) amortized pop). Option B is better — push is more common, and amortized is acceptable here.

Step 4: Explain Amortized Analysis

Walk through: "Each element is pushed to stack_in once, transferred at most once, and popped from stack_out once. That's at most 4 operations per element. So n elements cost at most 4n ops — O(1) amortized per operation."

🗣️ What to Say to the Interviewer

"I'll use two stacks: stack_in for incoming pushes and stack_out for outgoing pops.
push() always appends to stack_in — O(1), no conditions needed.
For pop()/peek(), I call _transfer(): if stack_out is empty, pour all of stack_in into it.
This reversal reverses the LIFO order, giving us the oldest element at the top of stack_out.
Then pop just does stack_out.pop() — the queue's front element.
empty() checks both stacks — elements could be in either.
Each element is moved at most once, so amortized O(1) per operation."
