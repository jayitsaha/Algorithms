LeetCode #2353

Design a Food Rating System

Build a system that rates foods, supports dynamic rating updates, and can answer "what's the highest-rated food for a given cuisine?" efficiently.

Medium Design Hash Maps + Heap changeRating: O(log n) highestRated: O(log n) amortized
1 Understanding the Problem ▾

We're asked to design a class FoodRatings initialized with three parallel arrays: foods, cuisines, and ratings. It must support two operations:

  • changeRating(food, newRating) — Update a specific food's rating to a new value
  • highestRated(cuisine) — Return the name of the highest-rated food for that cuisine. If there's a tie in rating, return the lexicographically (alphabetically) smaller name.
💡 Key Insight

This is a "top of a dynamically-updated sorted collection" problem. We need O(log n) for both operations. The trick: maintain a max-heap per cuisine using negated ratings, a food→rating dict as ground truth, and use lazy deletion — old heap entries become "stale ghosts" that we skip when they surface.

Why negated ratings? Python's heapq is a min-heap. To get the maximum rating efficiently, we store -rating. The smallest negative = the largest positive rating.

Why tuples (-rating, food_name)? Python compares tuples element-by-element. If two foods have the same (negated) rating, the heap picks the lexicographically smaller food_name first — exactly the tie-breaking rule the problem requires.

2 The Algorithm: Lazy Deletion Heap ▾

The algorithm combines three data structures. Understanding why each exists is the key to arriving at this solution:

1 __init__: Build food_rating (food → rating), food_cuisine (food → cuisine), and one min-heap per cuisine containing tuples (-rating, food_name). Each food is pushed into its cuisine's heap exactly once during initialization.
2 changeRating(food, newRating): Update food_rating[food] = newRating (this is the ground truth). Then heappush(cuisine_heap, (-newRating, food)). The old entry (-oldRating, food) remains in the heap — it is now a "stale ghost". We do NOT remove it.
3 highestRated(cuisine): Peek at heap[0]. Check: does -heap[0][0] == food_rating[heap[0][1]]? If yes, the top is valid — return the food name. If no, this entry is stale — pop and discard it, then repeat the check on the new top.
4 Amortized cost: Each changeRating pushes exactly one stale entry. Each stale entry is discarded at most once. Across all operations, total heap pops ≤ total pushes. So amortized O(log n) per operation, even if individual highestRated calls must pop several stale entries.
🔑 Why "Lazy Deletion" Instead of Real Deletion?

Standard binary heaps support push and pop in O(log n), but arbitrary deletion requires finding the element first (O(n) scan) or maintaining an index (complex). Lazy deletion avoids this entirely: push the new value, let the old one sit. When it rises to the top, detect it's stale via the rating dict and discard it then. Simple, correct, and amortized efficient.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At all times, food_rating[food] holds the current, true rating of every food. An entry (-r, food) in the heap is valid if and only if r == food_rating[food]. Since a food's rating changes atomically in the dict, only the most recently pushed heap entry for that food can be valid.

Why the highest-rated food is always correctly found: The heap always contains at least one valid entry for every food (the one pushed in __init__, or the most recent one pushed in changeRating). When we call highestRated, we pop stale entries until a valid one is at the top. Valid entries are kept in their heap-order position, so the first valid entry we encounter truly has the highest (negated-lowest) rating.

Tie-breaking correctness: Two entries with the same rating produce equal first tuple elements. Python then compares the second element: the food name string. "apple" < "banana" lexicographically, so the min-heap returns "apple" first — the alphabetically smaller name wins, exactly as required.

⚠️ The Classic Stale-Entry Bug

Beginners often write return heapq.heappop(heap)[1] — popping without checking staleness. This is wrong: after changeRating("sushi", 5), the heap may still have (-16,"sushi") at the top. Returning "sushi" would be correct, but the rating claim of 16 is a lie. Always validate: while -heap[0][0] != food_rating[heap[0][1]]: heapq.heappop(heap).

Proof that the heap never empties: Each food starts with one valid heap entry. A changeRating call pushes a new valid entry before the old becomes stale. So there is always ≥ 1 valid entry per food, and the heap for a cuisine always has ≥ 1 valid entry (since at least one food exists per cuisine).

4 The Code (Annotated) ▾

Solution 1 — Lazy Deletion Heap (Interview Pick, stdlib only)

import heapq
from collections import defaultdict
class FoodRatings:
def __init__(self, foods, cuisines, ratings):
self.food_rating = {} ← source of truth: food → current rating
self.food_cuisine = {} ← which cuisine owns this food
self.heaps = defaultdict(list) ← per-cuisine max-heap (negated ratings)
for food, cuisine, rating in zip(foods, cuisines, ratings):
self.food_rating[food] = rating
self.food_cuisine[food] = cuisine
heapq.heappush(self.heaps[cuisine], (-rating, food)) ← negate for max-heap
def changeRating(self, food: str, newRating: int) -> None:
self.food_rating[food] = newRating ← update dict FIRST — old heap entry now stale
cuisine = self.food_cuisine[food]
heapq.heappush(self.heaps[cuisine], (-newRating, food)) ← push new; DON'T remove old
def highestRated(self, cuisine: str) -> str:
heap = self.heaps[cuisine]
while True:
neg_r, food = heap[0] ← PEEK (don't pop) the heap top
if -neg_r == self.food_rating[food]: ← valid? rating matches ground truth
return food ← yes — this is truly the highest-rated
heapq.heappop(heap) ← stale ghost — pop and discard
⚠️ Why peek with heap[0] before deciding to pop?

We use heap[0] to look without removing. If the entry is valid, we return immediately — leaving the valid entry at the top for future queries (no wasted pops). Only stale entries get popped. If we always popped first, we'd remove the valid top entry and have to re-push it, adding complexity.

Solution 2 — SortedList (Cleaner, exact O(log n) everywhere)

from sortedcontainers import SortedList
class FoodRatings:
def __init__(self, foods, cuisines, ratings):
self.food_rating = dict(zip(foods, ratings))
self.food_cuisine = dict(zip(foods, cuisines))
self.sorted = defaultdict(SortedList) ← cuisine → sorted ascending by (-rating, name)
for food, cuisine, rating in zip(foods, cuisines, ratings):
self.sorted[cuisine].add((-rating, food))
def changeRating(self, food, newRating):
c = self.food_cuisine[food]
old = self.food_rating[food]
self.sorted[c].remove((-old, food)) ← O(log n) exact removal — no ghost entries
self.food_rating[food] = newRating
self.sorted[c].add((-newRating, food))
def highestRated(self, cuisine):
return self.sorted[cuisine][0][1] ← O(1): minimum = highest-rated food
💡 When to Propose Each in an Interview

Lazy Heap: Standard library, classic approach, shows you understand amortized analysis. Present this first.
SortedList: Cleaner semantics, exact complexity, O(1) highestRated. Mention as an upgrade if the interviewer allows third-party libraries. In LeetCode, sortedcontainers is available.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the "Dynamic Top-K" Pattern

Key signals: "design a system", "find the highest/lowest rated", "ratings change over time". Whenever you need repeated access to the maximum/minimum of a mutable collection, think heap. Multiple independent groups (cuisines) → one heap per group.

Step 2: Identify What Lookup Tables You Need

For changeRating: Which cuisine's heap do I update? → food_cuisine dict. For lazy deletion validation: What's the current rating? → food_rating dict. For highestRated: Where's the heap? → heaps dict. Draw these three structures on the whiteboard first — they drive the solution.

Step 3: Propose Brute Force, Then Motivate the Heap

Say: "Brute force: a list per cuisine, scan the whole list for highestRated — O(n) per query. Unacceptable with 2×10⁴ calls. I need O(log n). A heap gives O(1) peek at the max. The only challenge is updates. I'll handle that with lazy deletion."

Step 4: Articulate the Amortized Argument

Interviewers often push on "worst case for highestRated". Say: "Each changeRating call pushes at most one stale entry. Every stale entry is popped at most once total, across all highestRated calls. So total pops across all queries ≤ total pushes. Amortized O(log n) per operation."

🗣️ What to Say to the Interviewer

"I need three structures: food_rating as my ground truth, food_cuisine for routing updates,
and a max-heap per cuisine. Python's heapq is a min-heap, so I store (-rating, food_name).
The tuple comparison automatically breaks rating ties alphabetically — that's free.
changeRating: update the dict, push the new tuple. Leave the old entry — it's now stale.
highestRated: peek at heap top. If it doesn't match food_rating, pop it as stale. Repeat.
Each stale entry is pushed once and discarded once. Amortized O(log n) per call."
🔮 Likely Follow-up Questions
  • "Why not use a sorted set instead?" → SortedList from sortedcontainers gives exact O(log n) removal and O(1) peek. Cleaner but third-party. Mention it as an upgrade.
  • "What if ratings can only increase (or only decrease)?" → Lazy deletion still works. Old entries are stale regardless of direction of change.
  • "What's the worst-case heap size?" → O(n + total_changeRating_calls). In the problem: n ≤ 2×10⁴, calls ≤ 2×10⁴, so at most ~4×10⁴ entries total.
  • "Extend to top-K highest rated?" → Pop and validate until you've collected K valid entries, then re-push them. Or keep K valid entries in a sorted structure.

7 Summary & Key Takeaways ▾
Lazy deletion = push new value + leave old as a "stale ghost" — no real deletion needed
Negate ratings to turn Python's min-heap into a max-heap
Tuple (-rating, food_name) handles tie-breaking for free via lexicographic comparison
Always validate against the ground-truth dict before trusting a heap entry
One heap per category (cuisine) is the standard pattern for "top per group" queries

🎯 When to Recognize the Lazy Deletion Heap Pattern
  • Need the max/min of a collection that's updated dynamically
  • True heap deletion is costly or unavailable
  • You can detect staleness by comparing to an external ground-truth dict
  • Amortized complexity is fine (each update produces exactly one future discard)
