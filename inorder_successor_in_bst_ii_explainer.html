LeetCode #510

Inorder Successor in BST II

Given a node in a BST that also has a parent pointer, return its inorder successor — the node with the next larger key in sorted order.

Medium Trees With Parent Pointers Time: O(h) Space: O(1)
1 Understanding the Problem ▾

In a Binary Search Tree (BST), every node satisfies: all keys in the left subtree are smaller, all keys in the right subtree are larger. If you visit nodes in inorder (left → root → right), you get all values in ascending sorted order. The inorder successor of node N is the node that comes immediately after N in this sorted sequence — the node with the smallest key strictly greater than N.val.

💡 Key Insight

There are exactly two cases: if the node has a right subtree, the successor is the leftmost node of that right subtree. If it has no right subtree, the successor is the lowest ancestor from which we ascend via a left child. Parent pointers let us do this in O(1) space — no stack, no recursion needed.

This problem (LeetCode #510) differs from the classic Inorder Successor in BST (#285): in #285 you are given the root and a target value and can search top-down. Here you only receive the node itself, with a .parent pointer. You cannot do a top-down search.

✓ Case A — Has right child

node = 2, right child = 4
4.left = 3 → go left
3.left = None → stop
Successor = 3

↑ Case B — No right child

node = 4, no right child
4 is right child of 2 → climb
2 is left child of 5 → STOP
Successor = 5

Edge Cases: Node is the BST maximum (rightmost node) — climb all the way to root still as right child → return None. Single-node tree — no successor, return None. Node is the root with no right child — curr.parent is None → return None.

2 The Algorithm: Two-Case Parent-Pointer Walk ▾

The algorithm forks on one binary question: does the node have a right subtree? This produces two independent, clean pointer walks. No stack, no visited set, no recursion.

Case A: node.right exists

Enter the right subtree, then follow left pointers until exhausted. The leftmost node of the right subtree is the smallest value greater than node.val — that is the successor.

Case B: node.right is None

Climb via parent pointers as long as we are the right child. The moment we are a left child of a parent, that parent is the answer. If we exhaust all parents, the node was the maximum — no successor.

1 Branch on right child: If node.right is not None, we are in Case A.
2 [Case A] Walk leftmost: Set curr = node.right. While curr.left exists, do curr = curr.left. Return curr.
3 [Case B] Climb loop: Set curr = node. While curr.parent is not None AND curr == curr.parent.right, do curr = curr.parent.
4 [Case B] Return parent: Return curr.parent. This is the first ancestor approached from the left, or None if none exists.
⚠️ Why Can We Not Just Do a BST Search?

A BST search starts at the root and uses the key comparison (val > target → go right) to navigate down. But we are not given the root here — only the node. We must use parent pointers to navigate upward instead.

3 Why It Works: The Invariant ▾
🔐 The Core BST Invariant

In a BST, the inorder traversal produces a strictly increasing sequence. The inorder successor of node N is the node with the minimum value that is strictly greater than N.val. This gives us a structural guarantee we can exploit without visiting all nodes.

Why Case A works: If N has a right subtree, every node in it has a value greater than N.val (BST property). Among all those nodes, the one visited first in inorder is the leftmost — it has no left child to visit before it. Therefore, leftmost of right subtree = minimum of right subtree = smallest value greater than N.val = successor.

Why Case B works: If N has no right subtree, there are no keys in N's own subtree that are greater than N. The successor must be an ancestor. As we climb: any ancestor from whose right subtree we came has a value smaller than N (BST property says: N is in that ancestor's right subtree → ancestor.val < N.val). Such ancestors cannot be the successor. We skip them. The first ancestor from whose left subtree we came has a value greater than N.val (N is in its left subtree → ancestor.val > N.val) — and it's the smallest such ancestor (every ancestor above it is even larger). That is the successor.

💡 Analogy: Reading a Book

Imagine each BST level is a chapter. Inorder traversal reads the book cover to cover. When you finish page 4 (our node), the next page is 5. If there are sub-pages after 4 (a right subtree), the first sub-page (leftmost) comes next. If 4 is the last sub-page in a chapter (no right subtree), you return to the chapter heading (the parent from whose left you came) — that chapter title is the next thing you read.

Edge: rightmost node has no successor. If we keep climbing and are always a right child, the loop exits with curr.parent == None (we reached the root and were its right child). Returning None is correct.

4 The Code (Annotated) ▾

Solution 1 — Parent-Pointer Walk (Optimal / Interview Pick)

def inorderSuccessor(node: 'Node') -> 'Node': ← Input: node with .val, .left, .right, .parent
if node.right: ← Case A: does a right subtree exist?
curr = node.right ← Enter right subtree
while curr.left: ← Walk left as far as possible
curr = curr.left ← Each step: smaller value in right subtree
return curr ← Leftmost = minimum of right subtree = successor
curr = node ← Case B: start climbing from node itself
while curr.parent and curr == curr.parent.right: ← Keep climbing while we are a right child
curr = curr.parent ← These ancestors are already before us in order
return curr.parent ← First ancestor from whose left we came (or None)
⚠️ The Most Common Mistake: Returning curr Instead of curr.parent

After the climbing loop, curr is the node where the loop stopped — either because it is a left child (we arrived at the successor's left subtree node) or because curr.parent is None. The successor is curr.parent, not curr. Students who return curr fail all cases where no right subtree exists.

Solution 2 — Brute Force Inorder Traversal (requires root; worse complexity)

def inorderSuccessor_bf(root, node) -> 'Node': ← Needs root — NOT available in this problem
prev, result = None, None ← Track previous node in inorder
def inorder(n):
nonlocal prev, result
if not n or result: return ← Early exit once found
inorder(n.left)
if prev is node: result = n ← n immediately follows target in inorder
prev = n
inorder(n.right)
inorder(root); return result ← O(n) time, O(h) space — never optimal here
💡 Complexity Summary

Parent-Pointer Walk: Time O(h), Space O(1) — h = tree height; O(log n) balanced, O(n) skewed
Brute Force Inorder: Time O(n), Space O(h) — visits every node; also requires root
Why O(h)? Case A descends at most h levels. Case B climbs at most h levels. Together they never exceed h pointer hops.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Signal

"BST" + "inorder successor" + "parent pointer" = O(1) space traversal without recursion. The parent pointer is the hint that an upward-climbing solution exists. Immediately think: two cases based on right subtree existence.

Step 2: Restate the Goal Structurally

Reframe: "Find the smallest key in the BST that is strictly greater than node.val, using only the node and its parent pointer chain." This makes the two cases obvious.

Step 3: Draw the Two Cases on Paper

Sketch a 5-7 node BST. Pick a node with a right subtree and trace Case A. Pick a node without a right subtree and trace Case B (climb, checking left vs. right at each step). Confirm your loop terminates correctly.

Step 4: Double-Check the Return

After the Case B loop: return curr.parent, not curr. Say this explicitly to the interviewer — it's the most common gotcha in this problem.

🗣️ What to Say to the Interviewer

"The inorder successor is the next node in sorted order — smallest value greater than node.val.

There are two cases based on whether the node has a right subtree:

Case A: If node.right exists — go right, then walk left as far as possible.
The leftmost node of the right subtree is the minimum of all values greater than ours.

Case B: No right child — climb via parent pointers. Skip ancestors where
we came from their right (those ancestors are smaller than us in BST context).
Stop when we came from an ancestor's left — that ancestor is the successor.
Return curr.parent, not curr. If no such ancestor, return None.

Time O(h), Space O(1) — no recursion, no stack."
🔮 Likely Follow-up Questions
  • "What if we only had the root?" → LeetCode #285: BST search comparing node.val vs target.val
  • "Find the inorder predecessor?" → Mirror this: go left then rightmost; or climb while we are a left child
  • "What's the worst-case time?" → O(n) on a completely skewed (degenerate) BST
  • "Find kth inorder successor?" → Repeat k times, still O(k·h) total
  • "Does your solution handle a single-node tree?" → Yes: node.right is None, curr = node, loop doesn't run, curr.parent is None → return None

7 Summary & Key Takeaways ▾
Two cases determined by the presence of a right subtree — handle each independently
Case A: go right, then leftmost — finds minimum of right subtree in O(h)
Case B: climb via parent pointers while we are a right child — first left-child ancestor is the successor
Return curr.parent after the climbing loop, NOT curr — the most common mistake in this problem
Parent pointers give us O(1) space — we never need a recursion stack or an explicit stack data structure
If curr.parent is None after climbing, this was the rightmost node — return None (no successor)
