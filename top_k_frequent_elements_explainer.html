LeetCode #347

Top K Frequent Elements

Given an integer array nums and an integer k, return the k most frequent elements. You may return the answer in any order.

Medium Heaps Bucket Sort Time: O(n) Space: O(n)
1 Understanding the Problem ▾

You have an array of integers. Some numbers appear once, some appear many times. Your task: find the k numbers that appear most frequently. The order of your answer doesn't matter — just name the right k elements.

💡 Key Insight

This is a ranking by frequency problem. First count how often each element appears — that takes one hash-map pass. Then select the top k by count. The elegant O(n) trick is to use the array index itself as a frequency level — known as bucket sort — which avoids sorting entirely.

✓ Example: nums=[1,1,1,2,2,3], k=2

1 appears 3 times
2 appears 2 times
3 appears 1 time
Top 2 by frequency → [1, 2]

✗ Wrong: returning [1, 3] or [3]

3 only appears 1× — least frequent
1 and 3 is not top-2 by count
Must rank by frequency count

Edge cases to consider: all elements distinct (each frequency = 1), all elements identical (one element with freq = n), negative integers, k = 1 (single most frequent), k = number of distinct elements (return everything). The problem guarantees no tie at position k — the answer is always unique.

2 The Algorithm: Bucket Sort (O(n) Optimal) ▾

The pivotal observation: element frequency is a bounded integer from 1 to n. Bounded integer keys let us skip comparison-based sorting and use bucket sort — placing elements into buckets indexed by their frequency, then reading the top k from the highest-index buckets.

1 Count frequencies: One linear pass through nums builds a hash map: element → count. Cost: O(n).
2 Build frequency buckets: Create an array of n+1 empty lists. buckets[f] will hold all elements with exactly frequency f. Since no element can appear more than n times, n+1 slots covers the full range. Cost: O(n).
3 Fill buckets: For each (element, count) pair in the frequency map, append the element into buckets[count]. Cost: O(n) for at most n distinct elements.
4 Collect top k from the back: Scan the bucket array from index n down to 1. For each bucket, append its elements into the result. Stop as soon as the result has k elements. Cost: O(n) worst-case scan.

🔑 Why Not Just Use a Heap? (O(n log k) alternative)

A min-heap of size k is the canonical interview answer. Iterate the frequency map; push (freq, element) pairs; whenever heap size exceeds k, pop the minimum. The k elements remaining in the heap are the top k. Time: O(n log k), Space: O(n). This is simpler to explain and works even when the frequency range is unbounded. In an interview, propose the heap first, then offer bucket sort as the O(n) bonus optimization.

3 Why It Works: The Invariant ▾
🔐 The Invariant

After filling the buckets, buckets[f] contains exactly the set of elements whose frequency in nums is exactly f. Because bucket indices are the frequency values themselves, scanning from index n downward visits element groups in strictly non-increasing frequency order. The first k elements encountered are therefore guaranteed to be the k most frequent.

Why the back-scan is safe without explicit sorting: Bucket index is the frequency. Index 5 holds elements with higher frequency than index 4. By reading right-to-left we automatically get descending frequency order — without comparing any two elements. This works only because frequencies are bounded integers: the bucket array converts frequency into an array position, giving O(1) access to each frequency tier.

Why n+1 buckets is always sufficient: An element can appear at most n times (if every element in the array is the same). So the maximum possible frequency is n, and we need indices 0 through n — exactly n+1 slots. Index 0 is always empty (an element must appear at least once to be counted).

⚠️ Multiple elements in the same bucket

If two distinct elements have the same frequency they share a bucket (e.g., both 2 and 5 appear 3 times → both go into bucket[3]). The back-scan will encounter both and add them equally. Since the problem guarantees uniqueness at position k, we won't straddle a tie at the kth slot. If ties did occur at position k (a variant), any element from that bucket would be a valid answer — you can stop mid-bucket.

4 The Code (Annotated) ▾

Solution 1 — Bucket Sort (Optimal, O(n) time) — Interview Pick

def topKFrequent(nums, k):
count = {} ← freq map: element → how many times it appears
for n in nums: ← Pass 1: count every element
count[n] = count.get(n, 0) + 1 ← get(n,0) safely handles first occurrence
freq = [[] for _ in range(len(nums) + 1)] ← n+1 buckets; freq[f] = elements with that freq
for n, c in count.items(): ← Pass 2: place each element into its bucket
freq[c].append(n) ← element n goes in bucket at index c (its count)
res = [] ← accumulate our k results here
for i in range(len(freq) - 1, 0, -1): ← scan from index n downward (highest freq first)
for n in freq[i]: ← for each element in this frequency bucket
res.append(n) ← add to results
if len(res) == k: ← stop as soon as we have k elements
return res ← early exit — no need to continue scanning
⚠️ Why stop at index 1, not 0?

The range range(len(freq)-1, 0, -1) stops at index 1, excluding index 0. Index 0 represents "appeared 0 times" — which is always empty (an element must appear at least once). Stopping at 0 would also be correct (visiting an empty list is harmless) but visiting index 1 is the smallest meaningful bucket.

Solution 2 — Min-Heap of size k (O(n log k), classic interview approach)

import heapq
def topKFrequent(nums, k):
count = {}
for n in nums:
count[n] = count.get(n, 0) + 1
return heapq.nlargest(k, count.keys(), ← find k keys with the largest values
key=count.get) ← key function looks up each element's frequency

Solution 3 — Sort by frequency (O(n log n), simplest code)

from collections import Counter
def topKFrequent(nums, k):
return [x for x, _ in Counter(nums) ← Counter builds freq map; most_common returns sorted
.most_common(k)] ← take only the first k (element, count) pairs
💡 Complexity Summary

Bucket Sort: Time O(n), Space O(n) — theoretically optimal; reading input is already O(n)
Min-Heap (nlargest): Time O(n log k), Space O(n) — standard interview answer
Sort + slice: Time O(n log n), Space O(n) — simplest to write; fine for small n

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Clarify the Problem

Ask: "Can elements be negative? Is k always valid? Are ties at position k possible?" The problem guarantees k is valid and the answer is unique — no tiebreaking needed. If those guarantees weren't given, you'd need extra logic.

Step 2: Propose the Naive Approach

"I'll count frequencies with a hash map, then sort by frequency descending, return the first k. That's O(n log n) — correct and simple. Can I optimize?" Always start here; it shows you understand the problem before jumping to tricks.

Step 3: Optimize to O(n log k) with a Heap

"Instead of sorting all n items, I keep only k items in a min-heap. For each element in the freq map, I push (freq, element). If the heap exceeds k, pop the minimum. At the end, the heap contains exactly the top-k by frequency." This is the most-expected good answer in interviews.

Step 4: Blow Their Minds — O(n) with Bucket Sort

"There's a key observation: frequencies are bounded integers from 1 to n. I can create n+1 buckets indexed by frequency. No element can appear more than n times, so the buckets cover the full range. Place each element into its frequency's bucket. Then scan from the right, collecting elements until I have k. No sorting, no heap — pure O(n)."

🗣️ What to Say to the Interviewer

"First I build a frequency map in O(n) — one pass through nums.
Key observation: frequencies are bounded integers in [1, n].
So I create a bucket array of size n+1, where index i holds all elements with frequency i.
I fill the buckets in O(n), then scan from right to left collecting until I have k elements.
Total: O(n) time, O(n) space. This beats even the heap approach."
🔮 Likely Follow-up Questions
  • "What if the answer is not unique (ties at position k)?" → Collect all elements from the tie bucket; any subset of size k is valid.
  • "Top k frequent words?" → Same pattern but break ties alphabetically (#692) — heap with (−count, word) tuples handles it neatly.
  • "What is the kth most frequent element?" → Same bucket scan but stop after collecting the kth individual element, not k elements.
  • "Can you do this in O(1) space?" → Not without O(n log n) sorting. You need at least a freq map to know frequencies.

7 Summary & Key Takeaways ▾
Always start with a frequency hash map — it's the foundation of every "top k by frequency" problem
Bucket sort achieves O(n) by treating frequencies as array indices — only valid when keys are bounded integers
Min-heap of size k is O(n log k) and easier to explain — propose it first in interviews, then offer bucket sort
The early exit in the back-scan (return as soon as k elements collected) keeps the constant factor small
Python's heapq.nlargest(k, count, key=count.get) is the idiomatic one-liner; bucket sort shows algorithmic depth
ApproachTimeSpaceNotes
Sort by frequency O(n log n) O(n) Counter.most_common; simplest code

🎯 When to Recognize This Pattern
  • "Find the k most/least frequent elements" → freq map + heap (or bucket sort for O(n))
  • "Rank items by a bounded integer score" → bucket sort to avoid O(n log n) comparison sort
  • "Sort characters by frequency, then rebuild string" → same freq+bucket pattern (#451)
  • "Kth largest in a stream" → maintain heap of size k; root is always the kth largest (#703)
