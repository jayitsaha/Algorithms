LeetCode #321

Create Maximum Number

Given two arrays of single digits and a length k, create the lexicographically largest number of length k using digits from both arrays while preserving relative order within each array.

Hard Monotonic Stack Greedy Merge Time: O(k·(m+n+k)) Space: O(k)
1 Understanding the Problem ▾

You receive two arrays nums1 and nums2 of single digits, and a target length k. Your task: pick exactly k digits in total from the two arrays — preserving the original left-to-right order within each — and form the lexicographically largest (i.e., numerically maximum) possible number.

💡 Key Insight — Decompose into Three Sub-Problems

Fix how many digits to take from each array: i from nums1 and k−i from nums2. Then solve three independent sub-problems: (1) best i-length subsequence of nums1, (2) best (k−i)-length subsequence of nums2, (3) largest possible merge of those two sequences. Iterate over all valid values of i and keep the overall winner.

✓ Correct: nums1=[3,4,6,5], nums2=[9,1,2,5,8,3], k=5

Best split: 2 from nums1, 3 from nums2
nums1 → [6, 5] (best 2-subseq)
nums2 → [9, 8, 3] (best 3-subseq)
Merge → [9, 8, 6, 5, 3] ✓

✗ Wrong: "just take the 5 biggest digits"

nums1 has: 3,4,6,5 → digits 6,5,4,3
nums2 has: 9,1,2,5,8,3 → digits 9,8,5
Top-5 raw: 9,8,6,5,5 — but the 5 in nums2
comes AFTER 8, so [9,8,5] violates order.
Relative order is sacred — we can't reorder!

Edge cases to consider: k equals m+n (take all digits, just merge them); k=0 (return empty); all same digits; both arrays of length 1; nums1 or nums2 empty (take all k from the other).

2 The Algorithm: Monotonic Stack + Greedy Merge ▾

Two classical techniques combine here. Monotonic Stack (decreasing) extracts the best p-length subsequence from a single array. Greedy Merge combines two ordered sequences into the lexicographically largest possible interleaving.

1 Enumerate valid splits. Let i range from max(0, k−len(nums2)) to min(k, len(nums1)). For each i, we take i digits from nums1 and k−i from nums2. The range ensures we don't exceed either array's length.
2 max_subseq(nums, p) — Monotonic Stack. We want the best p-element subsequence. Let drop = len(nums) − p (the discard budget). Walk through nums: if the current digit exceeds the stack top AND drop > 0, pop the stack top (spend one discard). Then push current digit. At the end, take the first p elements of the stack.
3 merge(A, B) — Greedy Merge. Use two pointers. At each step, compare the remaining suffix of A (from pointer i) with the remaining suffix of B (from pointer j). Pick the digit from whichever suffix is lexicographically greater. This ensures the most significant digit placed at each position is maximized.
4 Keep global best. After merging, compare the candidate with the current best using list comparison (lexicographic). Update if the new candidate is greater. After all splits, return best.

Monotonic Stack Walk-through — extracting best 3 from [9,1,2,5,8,3], drop=3:

d=9: stack empty → push 9. stack=[9], drop=3
d=1: 1 < 9, no pop → push 1. stack=[9,1], drop=3
d=2: 2 > 1, drop=3>0 → pop 1, push 2. stack=[9,2], drop=2
d=5: 5 > 2, drop=2>0 → pop 2, push 5. stack=[9,5], drop=1
d=8: 8 > 5, drop=1>0 → pop 5, push 8. stack=[9,8], drop=0
d=3: drop=0, must keep → push 3. stack=[9,8,3], drop=0
Result: stack[:3] = [9, 8, 3] ✓
🔑 Why Suffix Comparison in Merge?

When A[i] == B[j] (same digit), you cannot simply pick either — you must compare the full remaining suffix. Example: A=[6,5,3], B=[6,5,4]. At the tie on 6 and again on 5, you must reach the 3 vs 4 to know that B leads to a better result. Python list comparison (A[i:] >= B[j:]) handles this correctly in O(k) per comparison.

3 Why It Works: The Invariant ▾
🔐 Monotonic Stack Invariant

At any point during the pass, the stack contains the lexicographically largest subsequence of digits seen so far, given the remaining budget of discards and the constraint that we must reach exactly p elements by the end. The stack is decreasing (every element is ≤ the one before it) because we greedily replaced any smaller element with a larger one when given the chance.

Why popping is always correct: If we see digit d and the stack top is smaller than d, and we still have remaining capacity to fill p elements without the popped element — then replacing that earlier, smaller digit with this larger one always makes the subsequence lexicographically larger. A larger digit in an earlier position strictly dominates a smaller digit in the same position.

🔐 Greedy Merge Invariant

At each merge step, picking the digit from the sequence with the larger remaining suffix is optimal. If A[i:] > B[j:], then placing A[i] and continuing from A gives a result that is ≥ any result that starts with B[j]. This is because the result of remaining characters from A will be larger than from B, no matter what happens later.

Why enumerate all splits: The optimal number of digits from each array is not known in advance — it depends on the specific digit values. Consider nums1=[8,9] and nums2=[3,6,5] with k=2: taking 1+1 gives max([9],[6])=[9,6], while taking 2+0 gives [8,9] and 0+2 gives [6,5]. Trying all splits (0, 1, 2) and comparing [6,5], [9,6], [8,9] → the winner is [9,6]. We need to try them all.

⚠️ Common Implementation Bug

Using A[i] >= B[j] instead of A[i:] >= B[j:] in merge will fail on tie cases. Example: A=[6,5,3], B=[6,5,4]. Both start with [6,5]. If you just compare A[0] >= B[0] (6>=6, tie → pick A), you'd get [6,5,3,...] but the correct answer starts with B's branch: [6,5,4,...]. The full suffix comparison is not optional — it's the correctness guarantee.

4 The Code (Annotated) ▾

Solution 1 — Monotonic Stack + Greedy Merge (Optimal, Interview Pick)

def maxNumber(nums1, nums2, k):
def max_subseq(nums, p): ← returns best p-length subseq of nums (order preserved)
drop = len(nums) - p ← how many digits we can afford to discard
stack = []
for d in nums:
while drop and stack and stack[-1] < d: ← pop smaller top if we have budget
stack.pop()
drop -= 1 ← spend one discard
stack.append(d) ← always push current digit
return stack[:p] ← trim to p (in case some discards unused)
def merge(A, B): ← greedy merge: lexicographically largest interleaving
res, i, j = [], 0, 0
while i < len(A) or j < len(B):
if A[i:] >= B[j:]: ← suffix comparison — handles ties correctly
res.append(A[i]); i += 1
else:
res.append(B[j]); j += 1
return res
m, n, best = len(nums1), len(nums2), []
for i in range(max(0, k-n), min(k, m)+1): ← enumerate valid splits: i from nums1, k-i from nums2
s1 = max_subseq(nums1, i)
s2 = max_subseq(nums2, k - i)
candidate = merge(s1, s2)
if candidate > best: ← Python list comparison is lexicographic
best = candidate
return best
⚠️ Why return stack[:p] and not just stack?

If we never exhaust the discard budget (e.g., digits are all non-increasing), we push every digit we see, so the stack ends up with len(nums) elements. But we want only p. Slicing to [:p] ensures we return exactly the requested length. This handles the case where the discard budget is not fully spent.

💡 Complexity

Time: O(k) splits × (O(m) + O(n) for max_subseq + O(m+n) for merge + O(k) for comparison) = O(k·(m+n+k)).
Space: O(k) for the stack and result arrays; O(m+n) intermediate but reused each split.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Decompose the problem

"If I fix how many digits I take from each array — say i from nums1 — then I have three independent sub-problems: best i-subseq of nums1, best (k-i)-subseq of nums2, and merge them optimally. I'll try all valid values of i (from max(0,k-n) to min(k,m)) and keep the best result."

Step 2: Recognize max subsequence = Remove K Digits pattern

"Finding the best p-length subsequence is the same as 'Remove (n-p) digits to maximize the number' — a classic monotonic stack problem. I maintain a decreasing stack, popping when the current digit is larger AND I still have discards left. The discard budget is drop = len(nums) - p."

Step 3: Recognize merge = Largest Number merge pattern

"Merging two sequences to maximize the result: at each step pick from the sequence whose remaining suffix is lexicographically larger. This is similar to the custom comparator in 'Largest Number' (#179). The key is comparing full suffixes, not just the first digit."

Step 4: State the gotchas

"Two implementation traps: (1) The split range: i must satisfy both i ≤ m and k-i ≤ n, so range(max(0,k-n), min(k,m)+1). (2) Merge comparison must use A[i:] >= B[j:] (suffix slices, not single elements) to handle ties correctly."

🗣️ What to Say to the Interviewer

"I'll decompose this: fix split i (digits from nums1), find best subseqs, merge them, maximize.
For max_subseq: monotonic decreasing stack with drop=len-p discards allowed.
For merge: greedy — always take from whichever remaining suffix is lexicographically larger.
Critical detail: use A[i:] >= B[j:] not just A[i] >= B[j] to handle ties.
Split range: range(max(0,k-n), min(k,m)+1) to respect both array bounds.
Overall: O(k*(m+n+k)) time, O(k) space."
🔮 Likely Follow-up Questions
  • "Three arrays?" → Extend to merge 3: build best-subseq per array, then merge pairwise.
  • "What if digits can repeat?" → Already handled correctly — the stack and suffix comparison both work with repeated digits.
  • "Can you optimize the merge?" → Suffix comparison is O(k) worst case. For large arrays, deque-based approach can reduce average case but not worst case.
  • "Relate to Remove K Digits (#402)?" → max_subseq IS the Remove K Digits solution with k replaced by drop=n-p. Identical logic.

7 Summary & Key Takeaways ▾
Decompose: fix the split i → solve max_subseq × 2 → merge → maximize over all valid i
Monotonic decreasing stack extracts the best p-length subsequence in O(n) with a discard budget of drop = n − p
Greedy merge: always pick from the sequence whose remaining suffix is lexicographically larger
A[i:] >= B[j:] (not A[i] >= B[j]) is the merge correctness detail — full suffix, not single digit
Valid split range: range(max(0, k-n), min(k, m)+1) — ensures both array bounds are respected
max_subseq is identical to "Remove K Digits to maximize" — the same pattern in disguise
ApproachTimeSpaceNotes
Brute Force — all subsequences Exponential O(k!) Completely impractical

🎯 When to Recognize This Pattern
  • "Pick k total from two arrays, preserve order, maximize value" → enumerate splits + max_subseq + merge
  • "Remove k digits from one array to maximize the number" → Monotonic stack, drop budget = k
  • "Merge two ordered sequences to maximize result" → Greedy suffix comparison merge
  • Any problem where "which element to pick" depends on all future elements → Monotonic stack
