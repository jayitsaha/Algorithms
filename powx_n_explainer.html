LeetCode #50

Pow(x, n)

Implement pow(x, n), which calculates x raised to the power n (i.e., xn). n can be negative.

Medium Recursion Binary Exponentiation Time: O(log n) Space: O(log n)
1 Understanding the Problem ▾

You must implement pow(x, n) — raising a floating-point base x to an integer exponent n — without using Python's built-in ** operator or math.pow. The exponent can be negative: x-n = 1 / xn. The challenge is to do this efficiently — the naive approach of multiplying x by itself n times takes O(n) operations, which is far too slow for n up to 231−1.

💡 Key Insight

Squaring cuts the exponent in half: xn = (xn/2)2. If we already have xn/2, we only need one more multiplication to get xn. Repeating this halving, we go from n multiplications down to just O(log n).

Edge cases to consider: n = 0 (any base → 1, by definition), n < 0 (flip to 1/x and negate), x = 0.0 with positive n (result is 0), x = 1.0 (always 1), and INT_MIN for n in statically typed languages (negating overflows — handle with long).

2 The Algorithm: Binary Exponentiation (Fast Power) ▾

Binary Exponentiation is a divide-and-conquer technique. At each step we cut the exponent in half and square the base. If the exponent is odd, we have one leftover factor of x to account for.

1 Base case n=0: Return 1. Any number to the 0th power is 1. This terminates the recursion.
2 Handle negative n: Convert myPow(x, n) to myPow(1/x, -n). This is valid because x-n = (1/x)n. After this, n is always non-negative in recursion.
3 Recurse on half: Compute half = myPow(x, n // 2). This solves a sub-problem half the size. The recursion depth is O(log n).
4 Combine (even n): Return half * half. Mathematically: (xn/2)2 = xn. One multiplication from two sub-results.
5 Combine (odd n): Return half * half * x. For odd n = 2k+1: x2k+1 = (xk)2 × x. The extra ×x handles the +1 in the exponent.

🔑 Why Exactly O(log n)?

Every recursive call divides n by 2 (integer division). Starting from n=10: 10 → 5 → 2 → 1 → 0. Four levels deep. In general, this is ⌊log₂(n)⌋ + 1 levels. Each level does at most 2 multiplications. Total: O(log n) multiplications. This is an exponential speedup — for n=109, we need only ~30 multiplications instead of a billion.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every call myPow(x, n), the function returns exactly xn. This is guaranteed by the recurrence: for even n, (xn/2)2 = xn; for odd n, (x(n-1)/2)2 × x = xn-1+1 = xn. The base case x0 = 1 is correct by definition.

Proof by strong induction on n:

  • Base case n=0: We return 1 = x0. Correct by convention.
  • Even n > 0: By hypothesis, myPow(x, n//2) returns xn/2. We return (xn/2)2 = xn. ✓
  • Odd n > 0: By hypothesis, myPow(x, n//2) returns x(n-1)/2. We return (x(n-1)/2)2 × x = xn-1 × x = xn. ✓
  • Negative n: We convert to myPow(1/x, -n). By the above, this returns (1/x)-n = xn (when n was negative). ✓
⚠️ The INT_MIN Overflow Trap (Java/C++)

In Java and C++, if n = Integer.MIN_VALUE (−2,147,483,648), then −n also equals −2,147,483,648 due to two's complement overflow! Python's integers are arbitrary precision, so this doesn't occur. In Java/C++, cast n to long before negating, or add a special case: if n == Integer.MIN_VALUE, return myPow(x, n+1) / x.

⚠️ The Duplicate-Call Trap

Never write return myPow(x, n//2) * myPow(x, n//2). This makes two separate recursive calls of the same size, creating an exponential call tree with O(n) total calls — exactly as slow as brute force. Always compute half = myPow(x, n//2) once, store it, then square it.

4 The Code (Annotated) ▾

Solution 1 — Recursive Binary Exponentiation (Interview Pick)

def myPow(x: float, n: int) -> float:
if n == 0: ← Base case: x⁰ = 1 for any x
return 1
if n < 0: ← Negative exponent: x⁻ⁿ = (1/x)ⁿ
return myPow(1 / x, -n) ← Flip base, make n positive
half = myPow(x, n // 2) ← CRITICAL: compute once, not twice!
if n % 2 == 0: ← Even n: x^n = (x^(n/2))²
return half * half ← One multiplication — done
else: ← Odd n: x^n = (x^((n-1)/2))² × x
return half * half * x ← Two multiplications; extra x for odd

Solution 2 — Iterative (O(1) space, O(log n) time) — the "bit-reading" approach

def myPow(x: float, n: int) -> float:
if n < 0: ← Handle negative up front
x, n = 1 / x, -n ← Flip base, negate exponent
result = 1.0 ← Accumulator — will absorb x^(2^k) for each '1' bit
while n: ← Loop until all bits processed
if n % 2 == 1: ← LSB is 1: current power of x is "active"
result *= x ← Include x^(current power) in result
x *= x ← Square x: next iteration represents next power of 2
n //= 2 ← Right-shift through bits of n
return result ← Product of selected powers = x^original_n
💡 The Bit-Reading Intuition (Iterative Version)

The iterative version reads the binary representation of n from LSB to MSB. For n=13 = 1101₂: we collect x1 (bit 0), x4 (bit 2), x8 (bit 3) and skip x2 (bit 1 is 0). Product: x1 × x4 × x8 = x13. This is the square-and-multiply algorithm used in RSA and cryptography.

Solution 3 — Brute Force (O(n) — never use this for large n)

def myPow(x: float, n: int) -> float: ← TLE for n > 10^5
if n < 0: x, n = 1/x, -n
result = 1.0
for _ in range(n): ← O(n) loop — multiply x n times
result *= x
return result
5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Name the Bottleneck in Brute Force

Start by proposing the brute force: "Loop n times, multiply x each time — O(n)." Then immediately flag the problem: "For n near 2 billion, that's 2 billion multiplications. We need O(log n)." This shows you think about scale from the start.

Step 2: Find the Halving Structure

Ask: "Is there redundancy I can exploit?" Key observation: if I know xn/2, I can square it to get xn in one step. Say: "x to the n = (x to the n/2) squared. I can solve the half-problem and combine." This halving is O(log n) depth.

Step 3: Handle Odd Exponent

Walk through n=5: n//2=2, so half = x². Squaring gives x⁴, but we need x⁵. We're off by one x. Insight: for odd n, n = 2k+1, so xn = (xk)² × x. Return half * half * x for odd n.

Step 4: Address Edge Cases Proactively

Say: "Three edge cases: n=0 returns 1 (base case). n negative: flip base to 1/x, negate n. And in C++/Java, n=INT_MIN can't be negated safely — use long or add 1 before negating." Covering INT_MIN is the senior-level signal.

🗣️ What to Say to the Interviewer

"Brute force: multiply x by itself n times — O(n). That's too slow for large n.
Key observation: x^n = (x^(n/2))^2. If I know x^(n/2), squaring it gives x^n in one step.
For odd n: I get one extra x because n = 2k+1, so result = half*half*x.
Base case: n=0 returns 1. Negative n: use 1/x with positive exponent.
I'll store half = myPow(x, n//2) once — if I call it twice, I'd get O(n) again.
Complexity: O(log n) time, O(log n) space for call stack.
I can also do O(1) space iteratively by reading the bits of n from LSB to MSB."
🔮 Likely Follow-up Questions
  • "Can you do it iteratively?" → Yes — read bits of n from LSB: if bit is 1, multiply result by x; then square x; then shift n right.
  • "What about n = INT_MIN in Java/C++?" → Overflow danger when negating. Use long, or handle separately.
  • "How does this apply to matrices?" → Same algorithm with matrix multiply — gives Fibonacci(n) in O(log n).
  • "How about (x^n) mod m?" → Same recursion, just add % m at each step — used in RSA, competitive programming.

7 Summary & Key Takeaways ▾
Binary Exponentiation halves n each recursion — O(log n) total multiplications, not O(n)
Always store half = myPow(x, n//2) in a variable — calling it twice would give O(n)
Odd n needs one extra factor: return half * half * x (one multiplication more)
Negative n: flip base to 1/x and negate n before recursing
n=0 is the base case: any number to the 0th power is 1
Iterative version reads bits of n LSB-first — O(1) space, same O(log n) time
Same algorithm powers RSA, matrix exponentiation, and competitive programming

🎯 When to Recognize the Binary Exponentiation Pattern
  • "Compute x to the power n efficiently" — always fast power
  • "Count something after n steps" where steps compose — matrix exponentiation
  • "Fibonacci(n) in O(log n)" — matrix exponentiation with fast power
  • "(x^n) mod m" — same algorithm, add mod at each step
  • Any problem reducible to repeated squaring of an associative operation
