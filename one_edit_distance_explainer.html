LeetCode #161

One Edit Distance

Medium String Processing Check Insert / Delete / Replace O(n) Time · O(1) Space
1 Understanding the Problem

Given strings s and t, return True if they are exactly one edit apart — not zero edits, not two or more. Exactly one. There are exactly three atomic edits:

Edit TypeExampleWhen
REPLACE"cat" → "cut" (c→c, a→u, t→t)Same length; exactly 1 char differs
INSERT"abc" → "abdc" (insert 'd')len(t) = len(s) + 1
DELETE"abdc" → "abc" (delete 'd')len(t) = len(s) - 1
💡 Key Insight: Collapse Three Cases into Two

INSERT and DELETE are the same operation viewed from opposite directions. If we normalize by always treating the shorter string as s, we only need to handle two cases: same-length → replace and length-diff-of-1 → insert. Everything simplifies from there.

⚠️ The "Exactly One" Trap

If s == t, that's zero edits — return False. Many candidates forget this and pass "at most one" tests but fail "exactly one" tests. Identical strings are valid is-substring but not one-edit-apart.

Edge cases to keep in mind: empty strings ("" and "a" are one edit apart), single character strings, and the prefix case ("abc" vs "abcd") where no character ever differs but there's still a valid insertion at the end.

2 The Algorithm: Length-Based Case Split

The algorithm hinges on one critical observation: once you find the first character difference, you know exactly where the edit happened and you can verify the rest in O(1) comparisons (a single tail equality check).

1 Boundary check: If |len(s) - len(t)| > 1, return False immediately. A single edit can change length by at most 1.
2 Normalize: Swap s and t if len(s) > len(t), ensuring s is always the shorter (or equal-length) string. This unifies INSERT and DELETE into one code path.
3 Scan for first mismatch: Walk index i through s. Compare s[i] with t[i] at each step. If all match and i reaches len(s), jump to step 5.
4 Mismatch found at index i:
• Same length (replace): return s[i+1:] == t[i+1:] — tails after the replaced char must match.
• Different length (insert): return s[i:] == t[i+1:] — skip the "extra" char at t[i], then the rest must align.
5 No mismatch (s is a prefix of t): return len(t) - len(s) == 1. Valid only when t has exactly one extra character appended at the end.
🔐 Why "First Mismatch" Is Enough

Since we can make only one edit, the first mismatch must be where that edit occurs. After applying this one hypothetical edit, the remainder of both strings must be identical. The tail equality check s[i:] == t[i+1:] verifies exactly that — no further scanning needed.

3 Why It Works: The Invariant
🔐 The Invariant

At every step i: all characters s[0..i-1] and t[0..i-1] are identical. The edit budget is intact — no edit has been "spent" yet.

Replace case: We've confirmed s[0..i-1] == t[0..i-1]. Replacing s[i] with t[i] costs our one edit. For the strings to be exactly one edit apart, everything after — s[i+1:] and t[i+1:] — must already be the same. One slice comparison confirms or denies this.

Insert case: We've confirmed s[0..i-1] == t[0..i-1]. Inserting t[i] into s at position i costs our one edit. After the insertion, s[i:] must equal t[i+1:] — s's remaining characters align with t after the newly inserted one.

❌ Two Edits Needed

s = "abc"
t = "xyd"
Mismatch at 0: s="abc", t[1:]="yd"
"bc" ≠ "yd" → False

✓ Exactly One Edit

s = "abc"
t = "abdc"
Mismatch at 2: s[2:]="c", t[3:]="c"
"c" == "c" → True

4 The Code (Annotated)
def isOneEditDistance(s: str, t: str) -> bool: m, n = len(s), len(t) # Lengths of both strings if abs(m - n) > 1: # Length diff > 1 → impossible return False if m > n: # Normalize: ensure s is shorter/equal return isOneEditDistance(t, s) # (recursive swap — O(1) extra work) for i in range(m): # Scan through shorter string if s[i] != t[i]: # First mismatch found at index i if m == n: # Same length → REPLACE case return s[i+1:] == t[i+1:] # Tails after i must match else: # Diff length → INSERT case return s[i:] == t[i+1:] # t[i] is the inserted char; align s[i:] to t[i+1:] return m + 1 == n # No mismatch: s is prefix of t (insert at end)
⚠️ Common Mistake: Forgetting the Prefix Case

The return m + 1 == n at the end handles s="abc", t="abcd" where s is a strict prefix of t. Without it, you'd return False unconditionally after the loop — which would also (wrongly) make s="abc", t="abc" return False... oh wait, that's actually correct! But s="abc", t="abcd" would also return False, which is wrong. The check m + 1 == n elegantly covers both: identical strings (m==n → False) and true prefixes (m+1==n → True).

⏱ Complexity

Time: O(n) — single pass until mismatch, then O(n) slice comparison. Total O(n).
Space: O(n) due to slicing (s[i:] creates a new string). Use the two-pointer variant below for true O(1) space.

5 🎤 Interview Strategy: How to Arrive at This
Step 1: Enumerate the Edit Types

Say aloud: "There are exactly three operations: replace, insert, delete." Draw one example each. Immediately notice that INSERT(s,t) == DELETE(t,s) — they're symmetric. This collapse means I only need two code branches.

Step 2: State the Length Constraint

Before writing any code: "If the lengths differ by more than 1, it's impossible — one edit can change length by at most 1." This is a free O(1) guard that impresses interviewers. Mention it before touching the loop.

Step 3: The "First Mismatch" Reduction

Key insight to state: "I only need to find where strings first diverge. Once found, the edit must happen there. Then I just verify the tails match." This is the crux — no need to track a running count or use DP.

Step 4: Don't Forget Edge Cases

Verbalize: "s and t identical → return False (zero edits). s is prefix of t → return True only if len diff is 1. Empty string vs single char → return True."

🗣️ What to Say to the Interviewer

"Three edit types collapse to two once I normalize to keep s shorter. I check the length constraint first — it's O(1) and prunes impossible cases. Then I scan until the first mismatch: same-length means replace (check both tails from i+1), different-length means insert (check s[i:] vs t[i+1:]). If I exhaust s without a mismatch, I only return True if t is exactly one character longer — that covers the append-at-end case."

7 Summary & Key Takeaways
INSERT and DELETE are symmetric — normalize to keep the shorter string as s
|len(s) - len(t)| > 1 is an immediate O(1) pruning check — always do it first
The edit must happen at the first mismatch — after that, just check tails
Replace case: verify s[i+1:] == t[i+1:] — both strings share the same suffix
Insert case: verify s[i:] == t[i+1:] — one extra char in t, then the rest aligns
No mismatch loop? Check m + 1 == n — covers the append-at-end prefix case
Identical strings return False — "exactly one edit" is not the same as "at most one"
🎯 When to Use This Pattern
  • Any "is this string reachable in exactly k steps?" question where k is small
  • Spell-checker candidate generation (adjacent keys, transpositions)
  • BFS word ladder where each "step" changes one character
  • Variant recognition signal: "differ by exactly one insert/delete/replace"
