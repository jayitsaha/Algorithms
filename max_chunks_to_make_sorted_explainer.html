LeetCode #769

Max Chunks To Make Sorted

Given a permutation of [0,1,...,n-1], find the maximum number of chunks you can split it into so that sorting each chunk individually produces the globally sorted array.

Medium Monotonic Stack Track Max So Far Time: O(n) Space: O(1)
1 Understanding the Problem ▾

You're given an array that is a permutation of [0, 1, 2, ..., n-1] — every integer appears exactly once. You may cut this array into as many contiguous chunks as you like. After sorting each chunk independently, you concatenate them back. The goal is to maximize the number of chunks while still ending up with the globally sorted array [0, 1, 2, ..., n-1].

💡 Key Insight

A chunk boundary can be placed after index i if and only if the maximum value in arr[0..i] equals i. Why? In the sorted output, positions 0..i must hold values {0,1,...,i}. Because the input is a permutation, if the largest value in the prefix is exactly i, the prefix contains exactly {0,...,i} — it is self-contained and can be sorted independently.

1 chunk only: arr=[4,3,2,1,0]

max at i=0 → 4 ≠ 0 ✗
max at i=1 → 4 ≠ 1 ✗
max at i=4 → 4 == 4 ✓
Only 1 chunk (whole array)

4 chunks: arr=[1,0,2,3,4]

max at i=1 → 1 == 1 ✓ → [1,0]
max at i=2 → 2 == 2 ✓ → [2]
max at i=3 → 3 == 3 ✓ → [3]
max at i=4 → 4 == 4 ✓ → [4]

Edge Cases: Single-element array → always 1 chunk. Already sorted (arr=[0,1,...,n-1]) → n chunks (every element is its own chunk, since arr[i]=i always fires). Reverse sorted (arr=[n-1,...,1,0]) → 1 chunk (max stays n-1 until the very last index).

2 The Algorithm: Track Max So Far ▾

We make a single left-to-right pass. We keep a running maximum of all elements seen so far. Whenever this running max equals the current index, the prefix is self-contained — we found a valid chunk boundary and increment our counter. No sorting needed.

1 Initialize: max_so_far = 0, chunks = 0. max_so_far tracks the largest element seen in the current unfinished chunk — it tells us the rightmost sorted position any seen element needs to land in.
2 Iterate with index: For each (i, val) pair, update max_so_far = max(max_so_far, val). This running max represents: "the furthest right that any element in the current chunk must eventually go in the sorted output."
3 Check boundary: If max_so_far == i, then all elements in positions 0..i have their sorted destinations within 0..i. The permutation guarantees they form exactly {0,...,i}. Cut here — chunks += 1.
4 Return chunks: After scanning all n elements, we have the maximum number of valid partitions. The algorithm is purely greedy — taking every valid cut is always optimal because skipping cuts can only merge, never split, later boundaries.
🔑 Why "max so far" not "sum so far"?

You could also use sum(arr[0..i]) == i*(i+1)/2 as the boundary check — for a permutation, this is mathematically equivalent. However, the max check is simpler, faster (no accumulation), and generalizes more intuitively. The max check directly encodes "no element needs to go further right than i," while the sum check is an indirect arithmetic property that can have false positives with negative numbers.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every index i, max_so_far = the maximum value in arr[0..i]. Because values are target indices in the sorted output, max_so_far equals the rightmost destination of any element seen in the current chunk. When max_so_far == i, no element in the prefix needs to move past position i — the prefix is self-contained.

Formal argument: The input is a permutation of {0,...,n-1}. A prefix arr[0..i] is a subset S of {0,...,n-1} with |S| = i+1. The prefix forms a valid chunk iff S = {0,1,...,i} (each element belongs at a position ≤ i). Since elements in S are distinct non-negative integers, S = {0,...,i} iff max(S) = i (the only subset of size i+1 from non-negatives with max = i is exactly {0,1,...,i}).

⚠️ Greedy Is Optimal — Why Not Wait?

You might wonder: "What if I skip a valid boundary and get more chunks later?" This is impossible. Every valid chunk boundary is independent of the others — confirming a chunk at index i doesn't affect whether a boundary exists at any j > i. Skipping a valid boundary only merges it with the next chunk; it never creates a new boundary elsewhere. So cutting greedily as soon as valid is always optimal.

Worked counterexample for "arr=[4,3,2,1,0]": After i=0, max_so_far=4. The element 4 must end up at sorted position 4, so positions 0..4 must all be in the same chunk. We can never cut until we've seen element 4's rightful neighbors (0,1,2,3). Only at i=4 do we have all five elements, and max_so_far=4=i fires.

4 The Code (Annotated) ▾

Solution 1 — Track Max So Far (Interview Pick, O(n) time / O(1) space)

def maxChunksToSorted(arr: list[int]) -> int:
max_so_far = 0 ← running maximum of arr[0..i]; tracks "furthest right destination" in current chunk
chunks = 0 ← count of valid chunk boundaries found so far
for i, val in enumerate(arr): ← iterate with both index and value; we need i for the boundary check
max_so_far = max(max_so_far, val) ← update running max: does val push our "need to reach" index further right?
if max_so_far == i: ← the prefix arr[0..i] contains exactly {0,1,...,i}: a self-contained sortable chunk
chunks += 1 ← cut here — this chunk can be sorted independently without touching arr[i+1..]
return chunks ← maximum valid partitions (always ≥ 1 since the whole array is always valid)
⚠️ Off-by-one trap: does max_so_far start at 0 or -1?

Starting at 0 is fine because in the first iteration (i=0), we immediately compute max(0, arr[0]). If arr[0]=0, we get max_so_far=0=i=0 and count a chunk. If arr[0] > 0, max_so_far stays arr[0] > 0 and no cut happens. Starting at -1 or float('-inf') also works identically — the first max() call overwrites it.

Solution 2 — Brute Force Sort (O(n² log n), for comparison)

def maxChunksToSorted_brute(arr: list[int]) -> int:
chunks = 0
for i in range(len(arr)): ← try every position as a possible chunk end
if sorted(arr[:i+1]) == list(range(i+1)): ← sort prefix, check if == [0,1,...,i]
chunks += 1 ← valid boundary found
return chunks

Track Max So Far: Time O(n), Space O(1) — single pass, two integer variables

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Rephrase "valid chunk" in concrete terms

A valid chunk [l..r] must contain exactly the values {l, l+1, ..., r} — because after sorting, positions l..r should hold exactly those values in the globally sorted array. Restate this until it clicks: "the chunk must be a self-contained set."

Step 2: Exploit the permutation constraint

Because it's a permutation of {0..n-1}, element values equal their sorted positions. So for the prefix ending at i to be a valid chunk, we need the elements to form {0,...,i}. For a set of distinct non-negatives, this means max == i. This observation collapses the check from "sort and compare" to "track max."

Step 3: Propose brute force first

"Sort each prefix and compare to [0..i]. Correct but O(n² log n). Can we check in O(1) per step? Yes — just track the running max. If max == i, the prefix contains exactly {0..i} since they're distinct non-negatives."

Step 4: Address the follow-up (LC #768 with duplicates)

If duplicates exist, max == i no longer suffices. Instead: precompute suffix minimums, then check prefix_max[i] <= suffix_min[i+1]. This still runs O(n) time with O(n) space. Alternatively: use a monotonic stack counting valid boundaries where the stack max equals index.

🗣️ What to Say to the Interviewer

"The input is a permutation of [0..n-1], so element values are target positions.
A prefix [0..i] is a valid chunk iff it contains exactly {0,1,...,i}.
Since values are distinct non-negatives, that's equivalent to max(arr[0..i]) == i.
I track the running max in one variable. Each time max_so_far == i, increment chunks.
Time O(n), Space O(1). Greedy is optimal — skipping a valid boundary never helps."
🔮 Likely Follow-up Questions
  • "What if the array has duplicates?" → LC #768 — compare prefix max with suffix min: valid cut at i iff max(arr[0..i]) ≤ min(arr[i+1..n-1])
  • "Why is greedy optimal?" → Taking every valid cut is safe because cuts are independent; skipping only merges boundaries, never creates new ones
  • "Alternative valid check?" → sum(arr[0..i]) == i*(i+1)/2 works for non-negative permutations but breaks with negatives
  • "What's the minimum chunks?" → Always 1 — the whole array sorted is valid by definition
