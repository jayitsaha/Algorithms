LeetCode #421

Maximum XOR of Two Numbers in an Array

Given an integer array nums, return the maximum result of nums[i] XOR nums[j], where 0 ≤ i ≤ j < n.

Medium Tries Bit Trie Time: O(n) Space: O(n)
1 Understanding the Problem ▾

XOR (exclusive-or) compares two bits: it produces 1 if the bits are different, and 0 if they are the same. So to maximize an XOR result, we want to pair numbers whose bits are as opposite as possible — especially at the highest (most significant) bit positions, since those contribute the most value.

💡 Key Insight

To maximize XOR, we greedily try to make each bit of the result a 1, starting from the most significant bit (MSB). For each bit of a given number, we want to find another number in the array whose same bit is the opposite. A Bit Trie lets us answer "does any number exist with this prefix?" in O(1) per bit — turning an O(n²) brute force into O(n).

✗ Naive: O(n²) — TLE

for i in range(n):
  for j in range(i, n):

2 The Algorithm: Bit Trie with Greedy Query ▾

A Bit Trie (binary prefix tree) stores numbers bit-by-bit from the most significant bit down to the least significant bit. Each internal node has at most two children: one for bit=0, one for bit=1. The path from root to a leaf spells out one full binary number.

1 Determine bit length L: Find the maximum number in nums and compute its bit_length(). All numbers will be treated as L-bit integers. For nums up to 2³¹ this gives L ≤ 31.
2 Build the trie: Insert every number in nums into the trie, bit by bit from bit L-1 (MSB) down to bit 0 (LSB). Use an array trie[node][0/1] where the value is a child node ID (0 = no child). Each insertion takes O(L) = O(1) time.
3 Greedy query for each number: For each number x, traverse the trie from root. At each bit i, compute bit = (x >> i) & 1. Try to go to the child for 1 - bit (the opposite). If that child exists, set bit i of the result to 1 and move there. Otherwise, forced to take the same-bit child (XOR bit stays 0).
4 Track and return maximum: After querying all numbers, the maximum XOR value found across all queries is the answer.
🔑 Why Array-Based Trie?

Instead of class TrieNode: children = [None, None], we use a 2D list: trie = [[0, 0]]. trie[node][bit] gives the child node ID. A value of 0 means "no child exists" (since the root occupies index 0, all real children are at indices ≥ 1). This is significantly faster in Python — no object creation overhead, better cache locality.

⚠️ Why MSB to LSB?

Bit k contributes 2^k to the value. Since 2^k > 2^(k−1) + 2^(k−2) + … + 2^0 = 2^k − 1, maximizing higher bits is always better than maximizing all lower bits. Processing MSB first lets us make the greedy choice at each level independently and correctly.

3 Why It Works: The Greedy Invariant ▾
🔐 The Invariant

At each bit level during a query for number x: if any number in the array has the opposite bit from x at the current position, the trie guarantees we find and follow it. This is because the trie stores all numbers — every possible path that exists in the array exists in the trie.

Greedy correctness: Suppose the optimal pair is (a, b) giving XOR = OPT. When we query for a, at each bit level where a's bit differs from b's bit, the child for b's bit exists in the trie (since b was inserted). Our greedy takes that child at every such level. So the query for a recovers a XOR b = OPT. Since we query every number and take the global max, we find OPT.

💡 Proof by Contradiction

Suppose the greedy query for a gives result R < OPT = a XOR b. Then at some bit k, the greedy chose 0 (same-bit child) when OPT has 1 at bit k. But OPT having 1 at bit k means b's bit k is opposite to a's bit k, so the opposite child must exist in the trie (b was inserted). Therefore the greedy would have taken that child — contradiction. So R ≥ OPT, meaning R = OPT.

Edge cases handled automatically:

  • Single element: query finds same number's path → XOR = 0 ✓
  • All identical: every bit at every level is forced → XOR = 0 ✓
  • One number much larger: its MSBs differ from all others, giving high XOR ✓
4 The Code (Annotated) ▾

Solution 1 — Bit Trie (Optimal, O(n) time / O(n) space) · Interview Pick

def findMaximumXOR(nums): ← main function
max_num = max(nums) ← largest value determines bit length
if max_num == 0: return 0 ← edge case: all zeros
L = max_num.bit_length() ← e.g., 25 (11001) → bit_length() = 5
trie = [[0, 0]] ← root at index 0; trie[node][bit]=child_id
def insert(num): ← insert one number into trie
node = 0 ← start at root
for i in range(L - 1, -1, -1): ← MSB first: bit L-1 down to bit 0
bit = (num >> i) & 1 ← extract bit i: shift right i positions, mask off the rest
if not trie[node][bit]: ← child doesn't exist? (0 means null)
trie.append([0, 0]) ← create new child node
trie[node][bit] = len(trie) - 1 ← link: parent → new child node id
node = trie[node][bit] ← walk down to child
def query(num): ← greedy query for max XOR partner
node, xor_val = 0, 0 ← start at root, accumulated XOR = 0
for i in range(L - 1, -1, -1): ← same bit order as insert
bit = (num >> i) & 1 ← current bit of num
want = 1 - bit ← we want the OPPOSITE bit to make XOR=1
if trie[node][want]: ← can we get a 1 at this bit position?
xor_val |= (1 << i) ← yes! set bit i of result to 1
node = trie[node][want] ← take the "good" (opposite) path
else: ← no opposite child — forced
node = trie[node][bit] ← XOR bit stays 0, follow same-bit path
return xor_val
for num in nums: insert(num) ← build trie with all numbers
return max(query(num) for num in nums) ← query each, return global max
⚠️ Why child ID 0 means "null" (not the root)?

The root is at index 0. When we create new nodes, they start at index 1 (root already occupies 0). So trie[node][bit] = 0 unambiguously means "no child" — because 0 is the root, and nothing should point back to the root as a child. Checking if not trie[node][bit] correctly detects missing children. This is why we use indices ≥ 1 for all real nodes created during insertion.

Solution 2 — Brute Force (O(n²), for reference only)

def findMaximumXOR_brute(nums):
ans = 0
for i in range(len(nums)):
for j in range(i, len(nums)):
ans = max(ans, nums[i] ^ nums[j]) ← try every pair — O(n²)
return ans
💡 Complexity Summary

Bit Trie: Time O(n × L) = O(n) (L ≤ 31 is constant), Space O(n × L) = O(n).
Brute Force: Time O(n²), Space O(1). Correct but too slow.
L is the bit-length of the maximum element — at most 31 for 32-bit non-negative integers.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Propose Brute Force

"The naive approach is to try all O(n²) pairs and XOR them, keeping the max. Correct, but too slow for n up to 3×10⁴."

Step 2: Think Bitwise — Greedy at Each Bit

"XOR maximization is a greedy problem: I want as many 1-bits as possible, starting from the most significant. For each bit of a given number, I need to ask: 'does any other number have the opposite bit here?' If yes, take that bit in the XOR result."

Step 3: Recognize the Trie Data Structure

"The query 'does any number exist with this binary prefix?' is exactly what a prefix trie answers in O(bits). I'll store all numbers in a bit trie, then greedily query each one."

Step 4: Implement with Array Trie for Speed

"Rather than Node objects, I'll use a 2D list trie[node][bit] = child_id. Root at index 0, new children appended. O(n × 31) = O(n) total."

🗣️ What to Say to the Interviewer

"To maximize XOR, I think bit-by-bit from MSB to LSB — greedy works because
higher bits are always more valuable than all lower bits combined.
For each number, I need to find a partner with the opposite bit at each position.

def findMaximumXOR(nums):
L = max(nums).bit_length() # L=5 (max=25=11001)
trie = [[0, 0]] # root node at index 0
def insert(num): # — INSERT PHASE —
node = 0
for i in range(L-1, -1, -1): # MSB first
bit = (num >> i) & 1
if not trie[node][bit]: # create child if missing
trie.append([0, 0]); trie[node][bit] = len(trie)-1
node = trie[node][bit]
def query(num): # — QUERY PHASE —
node, xor_val = 0, 0
for i in range(L-1, -1, -1):
bit = (num >> i) & 1; want = 1 - bit
if trie[node][want]: # can we get a 1 here?
xor_val |= (1 << i); node = trie[node][want]
else: node = trie[node][bit] # forced: XOR bit=0
return xor_val
for num in nums: insert(num) # build trie
return max(query(num) for num in nums) # query all

7 Summary & Key Takeaways ▾
To maximize XOR, think greedily bit-by-bit from MSB — higher bits always dominate all lower bits combined
A Bit Trie stores all number prefixes, answering "does a number with opposite prefix exist?" in O(bits)
Array-based trie: trie[node][0/1] = child_id. Child ID 0 means null (root is 0, children start at 1)
Insert all numbers first, then query each one. The answer is symmetric — query(a) finds b just as query(b) finds a
Time O(n × L) = O(n) since L ≤ 31 is a constant for 32-bit integers. Space O(n × L) = O(n) for trie nodes
