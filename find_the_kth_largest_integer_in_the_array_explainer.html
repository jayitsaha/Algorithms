LeetCode #1985

Find the Kth Largest Integer in the Array

Given an array of numeric strings where each represents a non-negative integer, return the k-th largest value as a string. Numbers can have up to 100 digits.

Medium Sorting String Compare QuickSelect Time: O(n) avg Space: O(1)
1 Understanding the Problem ▾

You have an array of strings where each string represents a non-negative integer — but these integers can be enormous (up to 100 digits). Find the k-th largest among them. Return it as a string.

Example: nums = ["3","6","7","10","2"], k = 2. Sorted descending: 10, 7, 6, 3, 2. The 2nd largest is "7".

💡 The Key Trap — Why You Can't Just Sort Alphabetically

Alphabetical (lexicographic) string sorting puts "10" before "2" because '1' < '2'. But numerically, 10 > 2. You need a custom comparator that respects numeric magnitude. The numbers are too large for standard integers in most languages — so compare the strings directly.

The String-Compare Rule: For two numeric strings with no leading zeros:

a="999", b="1000" len(a)=3 < len(b)=4 → b is longer → b is bigger a < b
a="999", b="998" Same length → compare lex: "999" > "998" (first diff: '9'>'8') a > b
a="10", b="2" len(a)=2 > len(b)=1 → a is longer → a is bigger a > b (10 > 2 ✓)
✗ Alphabetical Sort (Wrong!)

Input: ["3","6","7","10","2"]
Sorted alpha: "10","2","3","6","7"
k=2 → "2"?? That's wrong!
10 < 2 alphabetically (bad)

✓ Custom Numeric String Sort

Input: ["3","6","7","10","2"]
Sorted: "2","3","6","7","10"
k=2 → index n-k=3 → "7" ✓
Length-first compare works!

Edge Cases: k=1 returns the largest element. All same strings: return any one. Single element array with k=1: return the element. No leading zeros in input (per constraints).

2 The Algorithm: QuickSelect with Custom Comparator ▾

QuickSelect — invented by Tony Hoare in 1961 — finds the k-th order statistic (the element at position k in sorted order) in O(n) average time, without sorting the entire array. It's essentially QuickSort that only recurses into the half containing the target.

1 Map k to target index: k-th largest in an ascending array of size n is at index target = n − k. For k=2, n=5: target = 3. This is what QuickSelect will land on.
2 Pick a pivot: Choose nums[hi] (last element in window). Randomizing the pivot is better for adversarial inputs, but Lomuto with last element is simpler to explain.
3 Partition around pivot: Walk pointer i from lo to hi−1. Every element ≤ pivot (using string comparator) gets swapped to position store and store advances. Finally, place pivot at store. Now pivot is at its exact final sorted position p.
4 Decide which half to search: If p == target, return it immediately. If p < target, target is to the right of p (set lo = p+1). If p > target, target is to the left (set hi = p−1). Repeat until found.

⚠️ QuickSelect Worst Case

If the pivot is always the min or max (e.g., already-sorted input with last-element pivot), each pass only eliminates one element: O(n²) total. Fix: randomize the pivot before each partition step. In Python: import random; r = random.randint(lo, hi); nums[r], nums[hi] = nums[hi], nums[r].

3 Why It Works: The Invariant ▾
🔐 The Partition Invariant

After partition(lo, hi) returns index p:
• Every element at index < p is numerically ≤ nums[p]
• nums[p] is at its exact final position in the sorted array
• Every element at index > p is numerically ≥ nums[p]

This means: if p == target, nums[p] is our answer. If p < target, the answer is strictly to the right; if p > target, strictly to the left. We only recurse into one half each time.

Why string comparison is correct: For non-negative integers with no leading zeros, two strings a and b compare correctly by: (1) length — more digits means larger; (2) same length — lexicographic order matches numeric order because the most significant digit comes first. This is a total order over non-negative integers.

Why O(n) average: On a random pivot, each partition call reduces the search window by roughly half on average. Work done: n (first pass) + n/2 (second) + n/4 + ... = 2n ≈ O(n). Unlike QuickSort, we never recurse into BOTH halves.

Proof that the algorithm terminates: Each iteration strictly reduces hi − lo: either lo increases (lo = p+1) or hi decreases (hi = p−1). Since the range is finite and decreases by at least 1 each iteration, the while loop always terminates.

4 The Code (Annotated) ▾

Solution 1 — Custom Sort (Simple, O(n log n))

from functools import cmp_to_key
def kthLargestNumber_sort(nums, k):
def cmp(a, b): ← comparator: negative = a<b, positive = a>b, 0 = equal
if len(a) != len(b): ← different lengths → longer is bigger
return len(a) - len(b) ← negative if a shorter (a<b), positive if a longer (a>b)
return (a > b) - (a < b) ← same length: lex compare maps to -1/0/+1
nums.sort(key=cmp_to_key(cmp)) ← ascending sort with our custom comparator
return nums[-k] ← k-th largest in ascending array = index -k

Solution 2 — QuickSelect (Optimal, O(n) average) ← Interview Pick

def kthLargestNumber(nums, k):
def num_gt(a, b): ← True if string a represents a numerically greater value than b
if len(a) != len(b): ← longer string = bigger number (no leading zeros)
return len(a) > len(b)
return a > b ← same length: lex order matches numeric order
def partition(lo, hi):
pivot = nums[hi] ← Lomuto scheme: pivot is last element in window
store = lo ← store: next position for elements that are <= pivot
for i in range(lo, hi): ← scan every element except the pivot
if not num_gt(nums[i], pivot): ← nums[i] <= pivot numerically
nums[i], nums[store] = nums[store], nums[i] ← move small element to front region
store += 1 ← advance boundary for next small element
nums[store], nums[hi] = nums[hi], nums[store] ← place pivot at its final sorted position
return store ← p = pivot's final index
target = len(nums) - k ← k-th largest in ascending order is at index n-k
lo, hi = 0, len(nums) - 1
while lo <= hi:
p = partition(lo, hi) ← partition returns pivot's final sorted position
if p == target: ← pivot landed exactly where we need it!
return nums[p]
elif p < target: ← answer is to the right of pivot
lo = p + 1
else: ← answer is to the left of pivot
hi = p - 1
⚠️ Why Not Just int(s) in Python?

Python handles big integers natively, so sorted(nums, key=int)[-k] works. But in an interview: (1) it hides the insight that string comparison is sufficient; (2) it's not portable to Java/C++/Go; (3) interviewers typically want you to reason about the comparison. Always show the custom comparator — even if you mention Python's big-int as an aside.

Custom Sort: Time O(n log n · L), Space O(log n) for sort stack
Min-Heap (size k): Time O(n log k · L), Space O(k)
QuickSelect: Time O(n · L) avg / O(n² · L) worst-case, Space O(1) in-place

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Spot the "string numbers" trap

The problem says "strings representing integers" with up to 100 digits. Immediately say: "I can't parse these as standard integers in most languages. I'll write a custom comparator." This shows you read the constraints carefully.

Step 2: Derive the string comparator from first principles

Don't memorize — derive it. "With no leading zeros, the longer string is the bigger number. If same length, the left-most differing character decides, which is exactly lexicographic order." State the two rules: length first, then lex if tied.

Step 3: Propose Sort → then optimize to QuickSelect

"My first approach: custom sort O(n log n), return index n−k. Optimal: QuickSelect O(n) average. After partitioning, the pivot is at its final sorted position — I only recurse into the half containing my target." This shows depth of knowledge.

Step 4: Carefully map k to target index

"k=1 means largest → index n−1 in ascending array. k=2 means 2nd largest → index n−2. So target = n−k." Say this explicitly — the k-to-index mapping is the most common bug point.

🗣️ What to Say to the Interviewer

"The numbers can be 100 digits long so I'll skip int conversion and write a custom comparator: two strings compare by length first — longer is bigger — then lexicographically if same length, since no leading zeros means left-most digit is most significant. Brute force: sort with this comparator in O(n log n), return nums[-k]. Optimal: QuickSelect in O(n) average. Partition around a pivot — pivot ends up at its final sorted position p. If p == target = n-k, done. Otherwise search only the relevant half. Work is n + n/2 + n/4 + ... = 2n = O(n)."
🔎 Follow-up Questions to Expect
  • Worst case of QuickSelect? O(n²) if pivot always min/max — fix: random pivot
  • What if k > n? Constraint says 1 ≤ k ≤ n, but add a guard if unsure
  • What about leading zeros? Not in constraints, but strip with lstrip('0') if needed
  • Can you do it in O(n log k)? Yes — min-heap of size k
  • What if we needed the top k, not just the k-th? QuickSelect still works: after finding target, left partition is unordered top-k

7 Summary & Key Takeaways ▾
Numeric strings (no leading zeros) compare correctly: longer length wins; same length uses lexicographic order
QuickSelect finds the k-th order statistic in O(n) average time — never fully sorts the array
Target index mapping: k-th largest in ascending array of size n is at index n−k
After partition, pivot is at its exact final sorted position — the core invariant enabling QuickSelect
Worst case O(n²) is avoidable by randomizing the pivot before each partition
🎯 When to Reach for QuickSelect
  • Finding k-th smallest or largest in an unsorted array
  • Finding the median (k = n/2)
  • Finding top-k elements when you need O(n) not O(n log n)
  • In-place required (O(1) space) and the array can be mutated
🔑 QuickSelect Algorithm Deep-Dive

Origin: Tony Hoare, 1961. The same genius who gave us QuickSort, Hoare Logic, and the null reference (for which he later apologized). QuickSelect finds the k-th order statistic — the element at position k in a sorted array — in expected O(n) time without sorting.

Core invariant: After partition, pivot is at its exact final sorted index. Elements left are smaller, elements right are larger.

Why O(n) average: Work per round is proportional to window size. Expected window halves each round: n + n/2 + n/4 + ... = 2n. QuickSort recurses BOTH halves (n log n); QuickSelect recurses only ONE (O(n)).

Recognize when: "k-th element", "median", "top k" with O(n) hint, or in-place constraint.
