LeetCode #143

Reorder List

Given the head of a singly linked list L0→L1→…→Ln−1→Ln, reorder it in-place to: L0→Ln→L1→Ln−1→L2→Ln−2→…

Medium Linked List Find Mid + Reverse + Merge Time: O(n) Space: O(1)
1 Understanding the Problem ▾

You have a singly linked list. Your task is to rearrange it in-place so that the new order zips the front and back halves together: first node, last node, second node, second-to-last, and so on — all without creating new nodes or using extra data structures.

Input: 1 → 2 → 3 → 4 → 5

Front half: [1, 2, 3]
Back half: [4, 5]
Interleave: 1,5,2,4,3

Output: 1 → 5 → 2 → 4 → 3

front[0]=1, back[-1]=5
front[1]=2, back[-2]=4
front[2]=3 (middle, stays)

💡 Key Insight

The reordered list is exactly the first half interleaved with the reversed second half. Once you see that, the solution falls out naturally: find the midpoint, reverse the back half, then weave the two halves together — all in O(1) extra space.

Edge cases: A list of 0 or 1 nodes is already in order. A 2-node list needs no change either. For odd-length lists, the middle node belongs to the first half and becomes the natural tail after merging. The constraint "do not modify node values" means you must rewire pointers, not just swap values.

2 The Algorithm: Find Mid → Reverse → Merge ▾

A seemingly complex rearrangement decomposes into three classic linked list primitives, each taking O(n) time and O(1) space:

1 Find the middle node. Use two pointers: slow advances one step per round, fast advances two. When fast can no longer move two steps, slow sits at the midpoint. Then sever: set slow.next = None. This creates two independent lists: [head … slow] and [slow.next … tail].
2 Reverse the second half. Classic three-pointer reversal starting from the node that was slow.next. Maintain prev=None and curr. For each node: save nxt = curr.next, flip curr.next = prev, advance both. When the loop ends, prev is the new head of the reversed second half.
3 Interleave the two halves. l1 = original head, l2 = reversed second head. Each round: save n1 = l1.next and n2 = l2.next, then wire l1 → l2 → n1. Advance l1 = n1, l2 = n2. Stop when l2 is None — the second half is always ≤ the first in length.
🔑 Why We Must Sever at the Middle

Without slow.next = None, the tail of the first half still points into the second half. During merging when we set l2.next = n1, we'd reconnect back into the (now partially reversed) second half — creating a cycle or skipping nodes. The explicit sever gives the first half a clean null terminator so the merge loop ends correctly.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every merge iteration, l1 points to the next unprocessed node from the front, and l2 points to the next unprocessed node from the reversed back. One node from each is attached per iteration, maintaining strict alternation: front → back → front → back.

Fast/Slow correctness: After k steps, slow is at position k, fast is at position 2k. When fast reaches the last or second-to-last node (for even and odd n respectively), slow is at ⌊n/2⌋ — the middle. For even n: slow is at the last node of the first half. For odd n: slow is at the exact middle, which we include in the first half.

Reversal correctness: Before each step, prev is the head of the already-reversed suffix. After curr.next = prev, the suffix now includes curr. The invariant: at any point, all nodes from the start of second half up to (but not including) curr have been reversed and are reachable from prev.

Merge termination: The second half has ⌊n/2⌋ nodes (at most as many as the first half). The loop condition while l2 stops exactly when all second-half nodes are placed. Any remaining first-half nodes (just the middle node for odd n) are already correctly linked — nothing extra to do.

⚠️ Critical: Save Both next Pointers Before Any Rewiring

In the merge phase, you must capture n1 = l1.next and n2 = l2.next before touching any .next field. Once you execute l1.next = l2, the original l1.next (the remaining first half) is permanently gone. Forgetting either save corrupts the list irreversibly.

4 The Code (Annotated) ▾

Solution 1 — Find Mid + Reverse + Merge (Optimal · Interview Pick)

def reorderList(head): ← in-place: no return value, modifies list
if not head or not head.next: ← 0 or 1 node: trivially ordered
return
# ── Phase 1: Find the middle ──
slow, fast = head, head ← both start at head
while fast and fast.next: ← fast needs 2 nodes ahead to step
slow = slow.next ← slow: 1 step
fast = fast.next.next ← fast: 2 steps; when done, slow = mid
second = slow.next ← second half begins just after mid
slow.next = None ← SEVER: first half now ends at null
# ── Phase 2: Reverse second half ──
prev, curr = None, second ← classic reversal setup
while curr: ← walk every node in second half
nxt = curr.next ← save next BEFORE overwriting
curr.next = prev ← flip pointer backward
prev = curr ← prev advances to current node
curr = nxt ← curr advances to saved next
second = prev ← prev is new head of reversed second half
# ── Phase 3: Interleave merge ──
l1, l2 = head, second ← l1 = front head, l2 = reversed back head
while l2: ← second half drives: shorter or equal
n1, n2 = l1.next, l2.next ← CRITICAL: save both nexts first
l1.next = l2 ← insert l2 right after l1
l2.next = n1 ← attach remaining first half after l2
l1 = n1 ← advance l1 to next first-half node
l2 = n2 ← advance l2 to next second-half node

Solution 2 — Collect Into Array (O(n) space, simpler to derive)

def reorderList(head):
if not head: return
nodes = []
cur = head ← collect all node references
while cur:
nodes.append(cur); cur = cur.next
l, r = 0, len(nodes) - 1 ← two-pointer from both ends
while l < r: ← weave until pointers meet
nodes[l].next = nodes[r] ← front → back
l += 1
if l == r: break ← odd length: avoid double-linking middle
nodes[r].next = nodes[l] ← back → next front
r -= 1
nodes[r].next = None ← null-terminate the tail
💡 Interview Strategy: Offer Both

Array approach: O(n) time, O(n) space. Derive it first — it's immediate and demonstrates the insight.
In-place approach: O(n) time, O(1) space. Then say "Can I eliminate the O(n) space? Yes — find the middle with fast/slow, reverse in-place, merge." This shows optimization thinking.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Pattern

Signals: "linked list", "in-place", "interleave front and back". Any time you need to process a linked list from both ends simultaneously — and can't use random access — think fast/slow to find the middle, then work from both ends using a reversed second half.

Step 2: Propose the Array Approach First

"Simple O(n) space approach: collect all nodes into an array, then use left/right two-pointers to rewire them. Correct and easy to implement, but O(n) extra space." Then offer to optimize.

Step 3: Drive Toward the O(1) Solution

"The reordered list is exactly first-half interleaved with reversed second-half. I don't need an array — I can operate on the list structure directly. Three phases: fast/slow to find middle, reverse second half in-place, then merge the two halves."

Step 4: Trace Through an Example

Walk through [1,2,3,4,5] out loud. After Phase 1: [1→2→3→null] and [4→5→null]. After Phase 2: [5→4→null]. After Phase 3: [1→5→2→4→3→null]. Confirm matches expected output.

🗣️ What to Say to the Interviewer

"The reordered list is the first half zipped with the reversed second half.
Phase 1: fast/slow pointer — fast moves 2x, slow moves 1x. At loop end, slow is at the middle.
I'll sever slow.next = None to get two independent halves.
Phase 2: in-place reversal with prev/curr on the second half.
Phase 3: interleave — save both nexts, wire l1→l2→saved_l1_next, advance both pointers.
Total: O(n) time, O(1) space. Three passes, each O(n) and O(1)."
🔮 Likely Follow-up Questions
  • "What if the list has 1 or 2 nodes?" → The early return handles 0–1. For 2 nodes: fast.next is null after 0 steps, slow stays at head, slow.next = node 2 = second, sever gives [1→null] and [2→null], merge produces [1→2→null]. Correct!
  • "How does odd vs. even length differ?" → Even: both halves equal length. Odd: first half has one extra (the middle), which naturally becomes the tail after merging. The while l2 condition handles it automatically.
  • "This problem combines which sub-problems?" → #876 Middle of Linked List, #206 Reverse Linked List, and the merge structure from #21 Merge Two Sorted Lists.
  • "Can you do it recursively?" → Yes, but it uses O(n) call stack space and is harder to implement cleanly. The iterative solution is better.

7 Summary & Key Takeaways ▾
Decompose into three primitives: find midpoint (fast/slow), reverse second half, interleave merge
Always sever the list at the midpoint before reversing — prevents pointer corruption during merge
Save both l1.next and l2.next before any rewiring in the merge phase — the "two saves" rule
Second half drives the merge loop (while l2) — it is always ≤ first half in length
For odd-length lists, the middle node lands naturally at the tail after merge — no special case needed
This three-phase pattern recurs: Palindrome Linked List uses the same find-mid + reverse structure
