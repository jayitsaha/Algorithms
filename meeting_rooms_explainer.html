LeetCode #252

Meeting Rooms

Given an array of meeting time intervals, determine if a person could attend all meetings without any time conflict.

Easy Intervals Sort + Check Overlap Time: O(n log n) Space: O(1)
1 Understanding the Problem ▾

Imagine you have a busy schedule: three meetings booked at [0,30], [5,10], and [15,20]. Can you attend all three? The meeting at 5 starts while the meeting at 0 is still running (it doesn't end until 30). You'd have to be in two places at once — impossible. Return False.

The question is simply: do any two meetings overlap? Two intervals [a, b] and [c, d] overlap when c < b (the later-starting one starts before the earlier-starting one ends). If even one such pair exists, the answer is False.

💡 Key Insight

Sort all meetings by start time. After sorting, an overlap can only happen between adjacent meetings in the sorted order. If meeting i ends after meeting i+1 starts, that's a conflict. No conflict among adjacent pairs → no conflict at all → True.

Edge cases: Empty array → True (no meetings, no conflicts). Single meeting → True. Touching boundaries [0,5],[5,10] → True (not overlapping — one ends exactly when the next begins). Two identical meetings [1,5],[1,5] → False (after sort, start[i] < end[i-1] catches same-start conflicts).

2 The Algorithm: Sort Then Scan Adjacent Pairs ▾

Random intervals are hard to analyze — two conflicting meetings might be far apart in the input array. The trick is to sort by start time so that any conflict becomes visible between neighboring entries.

1 Sort by start time: intervals.sort(key=lambda x: x[0]). This places all meetings in chronological order. O(n log n) — the bottleneck of the whole algorithm.
2 Scan from index 1: For each meeting at index i (starting from 1), compare it against the previous meeting at index i-1.
3 Check overlap condition: If intervals[i][0] < intervals[i-1][1] — the current meeting's start is before the previous meeting's end — we have a conflict.
4 Fail fast on conflict: Return False immediately. We don't need to check the rest — one conflict is enough to make attendance impossible.
5 Return True if no conflicts: If we complete the scan without finding any overlap, return True — the person can attend all meetings.
🔑 Why Checking Only Adjacent Pairs Is Enough

After sorting, suppose meetings are ordered A, B, C by start time. If A and C conflict (A ends after C starts), then since B starts between A and C, we'd have B.start ≤ C.start < A.end, which means A and B also conflict. So any non-adjacent conflict implies a nearer adjacent conflict. Checking adjacent pairs catches everything.

3 Why It Works: The Invariant ▾
🔐 The Invariant

After sorting, at any index i, all meetings with indices 0 through i-1 have been confirmed non-conflicting with each other. Adding meeting i can only create a conflict with meeting i-1 (the one immediately before it), because i-1 has the largest end time among all "cleared" meetings — and meeting i starts after all of them by the sort guarantee.

Formal proof sketch: Let s₀ ≤ s₁ ≤ ... ≤ sₙ₋₁ be the start times after sorting. Suppose meetings j and k conflict where k > j+1 (non-adjacent). That means sₖ < eⱼ. Since sⱼ₊₁ ≤ sₖ (sorted order), we have sⱼ₊₁ ≤ sₖ < eⱼ, so meetings j and j+1 also conflict. Therefore every non-adjacent conflict implies an adjacent conflict. QED.

Boundary behavior (touching is not overlapping): The condition is strict: intervals[i][0] < intervals[i-1][1]. Meeting [0,5] followed by [5,10] — the check is 5 < 5 which is False, so no conflict. The person finishes one meeting and immediately starts the next. This models real-world scheduling correctly.

⚠️ Common Mistake: Wrong Comparison Direction

Don't accidentally flip the comparison. The overlap condition is next.start < prev.end (current starts before previous ends = conflict). Some write it as prev.end > next.start — that's equivalent, but writing the "conflict condition" directly (and returning False on it) is clearer than writing the "no-conflict condition" and inverting it.

4 The Code (Annotated) ▾

Solution 1 — Sort + Linear Scan (Optimal · Interview Pick)

def canAttendMeetings(intervals: list[list[int]]) -> bool:
intervals.sort(key=lambda x: x[0]) ← Sort in-place by start time; O(n log n)
for i in range(1, len(intervals)): ← Compare each meeting with the one before it
if intervals[i][0] < intervals[i-1][1]: ← Overlap: current starts before previous ends
return False ← Conflict found — can't attend both
return True ← No conflicts found — all meetings attendable
⚠️ In-Place Sort Modifies the Input

intervals.sort() modifies the caller's list. In an interview, mention this: "I'm sorting in-place. If the input list must be preserved, I'd use sorted(intervals, key=lambda x: x[0]) instead, which returns a new sorted list and leaves the original unchanged." LeetCode doesn't care, but showing this awareness impresses interviewers.

Solution 2 — Brute Force (All Pairs, O(n²) — mention then discard)

def canAttendMeetings_brute(intervals):
n = len(intervals)
for i in range(n): ← Outer loop: every meeting
for j in range(i+1, n): ← Inner loop: compare with every later meeting
a, b = intervals[i], intervals[j]
if a[0] < b[1] and b[0] < a[1]: ← Overlap iff NOT (a ends before b starts OR b ends before a starts)
return False ← O(n²) — correct but impractical
return True
💡 Complexity Summary

Sort + Scan (optimal): Time O(n log n) for sort, O(n) for scan → O(n log n). Space O(1) in-place.
Brute Force: Time O(n²), Space O(1). Correct but never acceptable in an interview for n > a few thousand.
Lower bound: We can't do better than O(n log n) in general — any algorithm must examine all intervals at least once, and comparison-based sorting can't beat O(n log n).

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize Interval Pattern

Any time you see "intervals" + "conflict/overlap/merge" → your first move is sort by start time. This is the universal first step for almost all interval problems. It transforms a 2D, unordered problem into a 1D, ordered one where adjacent comparisons dominate.

Step 2: Mention Brute Force First

"I could check all O(n²) pairs using the overlap condition a.start < b.end and b.start < a.end. That's correct but O(n²). Let me think about how sorting helps."

Step 3: Derive Adjacent-Check Property

"If I sort by start time, any conflict between non-adjacent meetings implies a conflict between adjacent ones — so I only need to check n-1 consecutive pairs. That's O(n) after the O(n log n) sort."

Step 4: Clarify Edge Cases

"Empty → True. Single → True. Touching boundaries use strict less-than so they don't count as overlaps. Same-start meetings: after sort, the second one's start equals the first one's start which is less than the first one's end → conflict caught."

🗣️ What to Say to the Interviewer

"This is an interval overlap problem. The brute-force is O(n²) — check all pairs.
To optimize: sort by start time in O(n log n). After sorting, a conflict can only
occur between adjacent meetings — I proved this because any non-adjacent conflict
implies an adjacent conflict in sorted order.
So I do a single O(n) scan: if intervals[i][0] < intervals[i-1][1], return False.
Total: O(n log n) time, O(1) extra space (in-place sort)."
🔮 Likely Follow-up Questions
  • "Minimum number of meeting rooms needed?" → Meeting Rooms II (#253): sort by start, min-heap of end times
  • "Merge all overlapping intervals?" → Merge Intervals (#56): sort + extend end greedily
  • "Can you attend meetings with at most one removal?" → Extension: try removing each meeting once, O(n²)
  • "Return all conflicting pairs?" → Collect instead of fail-fast; still O(n) scan after sort

7 Summary & Key Takeaways ▾
Sorting by start time reduces the pair-check problem from O(n²) to O(n log n)
After sorting, only adjacent pairs can conflict — this is provably sufficient
Use strict less-than: next.start < prev.end — touching boundaries are allowed
Fail fast: return False the moment a conflict is detected, no need to scan further
This is the gateway pattern — mastering it unlocks Meeting Rooms II, Merge Intervals, and all other interval problems
