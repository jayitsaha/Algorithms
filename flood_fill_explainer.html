LeetCode #733

Flood Fill

Given an image (2D pixel grid), a starting pixel, and a new color — flood-fill all 4-directionally connected pixels that share the original color to the new color.

Easy Graph Traversal DFS / BFS from Start Time: O(m·n) Space: O(m·n)
1 Understanding the Problem ▾

Think of the paint bucket tool in Microsoft Paint. You click on a pixel and the tool spreads outward in all four cardinal directions — up, down, left, right — changing every connected pixel that shares the clicked color. It stops where the color changes, creating natural boundaries.

💡 Key Insight

The 2D image is an implicit graph. Each pixel is a node connected to its 4 cardinal neighbors. Flood Fill is graph traversal (DFS or BFS) starting from the seed pixel, visiting every reachable node with the original color and painting it to the new color. The traversal stops at boundary pixels or grid edges.

Input: sr=1, sc=1, color=2

1  1  1
1  1  0  ← start
1  0  0

Output: all connected 1s → 2

2  2  2
2  2  0
2  0  0

Formal definitions:

  • 4-directional connectivity — only up/down/left/right count; no diagonals
  • Original color — image[sr][sc] at call time
  • Reachable — connected to start via a path of same-color pixels
  • In-place — we modify the original grid and return it
⚠️ Critical Edge Case: start pixel already has the target color

If image[sr][sc] == color, return the image unchanged immediately. If you don't, the DFS's "visited" mechanism breaks: we mark visited by painting to color, but if orig == color, a painted pixel still looks unvisited — infinite recursion!

Other edge cases: single 1×1 grid (paint it, return), entire grid same color (whole grid changes), start pixel surrounded by different colors (only start pixel changes), color=0 (works fine — 0 is a valid new color).

2 The Algorithm: DFS from Starting Pixel ▾

We use Depth-First Search (DFS) — a recursive exploration that dives deep into the grid before backtracking. BFS (breadth-first with a queue) is equally valid and produces the same result; DFS is simpler to write for this problem since path order doesn't matter.

1 Capture the original color: orig = image[sr][sc]. If orig == color, return immediately — no fill needed and we avoid the infinite recursion trap.
2 Define the DFS function: dfs(r, c) — takes a pixel coordinate. Checks bounds and color match. If valid, paints the pixel, then recurses into all 4 neighbors.
3 Base cases (stop recursing): (a) r < 0 or r >= rows — out of grid vertically; (b) c < 0 or c >= cols — out of grid horizontally; (c) image[r][c] != orig — different color (boundary) or already painted (= visited).
4 Paint then recurse: Set image[r][c] = color first (marks as visited), then call dfs(r-1,c), dfs(r+1,c), dfs(r,c-1), dfs(r,c+1). Order of direction calls doesn't matter.
5 Kick off and return: Call dfs(sr, sc). Since we modify in-place, just return image.

DFS (Recursive)

Explores one branch fully before backtracking. Uses the call stack implicitly. Simpler code. Can hit Python's recursion limit (~1000 calls) on very large grids.

BFS (Iterative Queue)

Explores all neighbors level by level. Uses an explicit deque. More code, but safer for huge grids — no recursion limit risk. Preferred in production.

🔑 Why No Explicit Visited Set?

We use the grid itself as our visited marker. When we paint image[r][c] = color, that pixel now has a different value than orig. Any future DFS call to that cell will hit the image[r][c] != orig check and return immediately — exactly what we want. Zero extra memory for tracking!

3 Why It Works: The Invariant ▾
🔐 The DFS Invariant

When dfs(r, c) is entered and the color check passes, it guarantees two things: (1) image[r][c] will be painted to color, and (2) every 4-connected pixel reachable from (r, c) that still has orig will also be painted before this call returns.

Why painting before recursing prevents infinite loops: After we set image[r][c] = color, any neighbor that tries to recurse back to (r, c) will see image[r][c] == color != orig and immediately return. This is why orig != color must hold — if they were equal, painting wouldn't change the value, the check would pass again, and we'd recurse forever.

Why every connected pixel is guaranteed to be visited: Suppose pixel P has color orig and is reachable from the start via a path of same-color pixels. DFS will reach P because: (1) it explores all 4 directions from each visited pixel, (2) only the color check stops it, and (3) every pixel on the connecting path also has orig and will be visited in turn. By induction, the entire connected component is covered.

⚠️ Common Mistake: Adding a Separate Visited Set

Many beginners write visited = set() and add (r,c) to it. This is correct but wastes O(m·n) extra space. The grid already encodes visited state via the color change. Trust the color check — it's doing double duty as both a boundary test and a visited filter.

What stops the spread: A pixel stops DFS if it's (a) out of bounds — the grid has finite size, (b) color 0 or any non-orig value — a "wall" that acts as a natural boundary, or (c) already painted — a previously visited pixel that acts as a closed border.

4 The Code (Annotated) ▾

Solution 1 — DFS Recursive (Optimal, Interview Pick)

def floodFill(image, sr, sc, color): ← modifies image in-place
orig = image[sr][sc] ← save the color we are replacing
if orig == color: ← CRITICAL early exit: prevents infinite recursion
return image ← nothing to do — already the right color
def dfs(r, c): ← inner fn captures orig, color from outer scope
if r < 0 or r >= len(image): ← row out of bounds → stop
return
if c < 0 or c >= len(image[0]): ← col out of bounds → stop
return
if image[r][c] != orig: ← wrong color (boundary) or already visited → stop
return
image[r][c] = color ← paint NOW → acts as "visited" marker
dfs(r - 1, c) ← explore UP
dfs(r + 1, c) ← explore DOWN
dfs(r, c - 1) ← explore LEFT
dfs(r, c + 1) ← explore RIGHT
dfs(sr, sc) ← kick off DFS from the seed pixel
return image ← grid modified in-place; just return it
💡 Complexity Summary

Time: O(m·n) — each pixel is visited at most once (once painted to color, the check image[r][c] != orig prevents revisiting)
Space: O(m·n) — worst-case recursion depth equals the grid size (e.g., a 1×n or serpentine all-same-color grid forces one recursive call per pixel)

Solution 2 — BFS Iterative (Safe for Large Grids)

from collections import deque
def floodFill(image, sr, sc, color):
orig = image[sr][sc]
if orig == color: return image
image[sr][sc] = color ← paint seed BEFORE enqueuing (avoids double-enqueue)
queue = deque([(sr, sc)]) ← BFS frontier
while queue:
r, c = queue.popleft()
for dr, dc in [(-1,0),(1,0),(0,-1),(0,1)]: ← 4 directions
nr, nc = r + dr, c + dc
if 0 <= nr < len(image) and 0 <= nc < len(image[0]):
if image[nr][nc] == orig: ← unvisited neighbor with original color
image[nr][nc] = color ← paint before enqueue (prevents double-add)
queue.append((nr, nc))
return image
⚠️ BFS Subtlety: Paint Before Enqueueing

In BFS, paint the pixel before adding it to the queue. If two neighbors both see an unvisited cell and enqueue it twice, you'd paint it twice (harmless but wasteful) and push it on the queue twice (causes extra iterations and potential wrong logic in extended problems). Pre-painting marks it visited immediately on discovery.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Grid Graph Pattern

Trigger words: "spread," "connected," "4-directional," "flood," "all pixels." When a problem asks to explore or modify a region of a grid starting from one cell, it's almost always DFS or BFS on an implicit graph. Name this out loud to your interviewer.

Step 2: Choose DFS (and mention BFS)

"I'll use recursive DFS — it's compact and reads like pseudocode. For production I'd use iterative BFS to avoid Python's recursion limit on very large grids. Both are O(m·n) time and space." This shows you know the tradeoffs.

Step 3: Proactively Handle the Same-Color Edge Case

Before writing any recursion, say: "Let me handle the edge case where the start pixel already has the target color — I'll return early to prevent infinite recursion since our visited check relies on color change." This immediately impresses the interviewer.

Step 4: Explain the Visited Trick

"I don't need a separate visited set — painting the pixel to the new color serves as the visited marker. Since orig ≠ color, any revisited cell will fail the color check and return immediately." This demonstrates deep understanding.

🗣️ What to Say to the Interviewer

"This is the paint-bucket problem — flood fill from (sr, sc) to all 4-connected pixels sharing the same color.
First, edge case: if the start pixel already has the target color, return early — otherwise we'd recurse infinitely.
Then I'll DFS from (sr, sc). At each cell: if out of bounds or not the original color, return.
Otherwise paint it (this also marks it visited), then recurse into all 4 neighbors.
No extra visited set needed — the color change IS the visited marker.
Time O(m·n), Space O(m·n) for the recursion stack in worst case."
🔮 Likely Follow-up Questions
  • "What if we needed 8-directional connectivity?" → Add diagonal directions to the 4-dir list: (-1,-1), (-1,1), (1,-1), (1,1)
  • "Stack overflow on a 1000×1000 grid?" → Use iterative BFS with a deque or explicit stack
  • "Number of Islands?" → Same DFS pattern; count how many times you start a new DFS (#200)
  • "Surrounded Regions?" → DFS from border O-cells to mark safe, then flip the rest (#130)
