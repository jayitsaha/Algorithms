LeetCode #30

Substring with Concatenation of All Words

Find all starting indices of substrings in s that are a concatenation of every word in words exactly once, in any order, with no gaps.

Hard Sliding Window Multiple Starting Points + Hash Time: O(n · w) Space: O(k)
1 Understanding the Problem ▾

You are given a string s and a list of words words where every word has the same fixed length w. Return all starting indices in s where a contiguous substring of length k × w (k = number of words) is a permutation-concatenation of all words. Every word must be used exactly its required number of times — including duplicates.

💡 Key Insight

Because all words have the same length w, any valid answer window is exactly k × w characters long, and can be divided into exactly k non-overlapping chunks of size w. The ordering of chunks doesn't matter — only the multiset of chunks must equal the multiset of words. This is a fixed-size sliding window problem with a frequency-map check.

✓ Valid: s="barfoobar", words=["foo","bar"]

Index 0: chunks="bar"+"foo" ✓
Index 3: chunks="foo"+"bar" ✓
Result: [0, 3]

✗ Invalid: index 1 chunk "arf" not in words

Index 1: chunk "arf" not in words ✗
Index 2: chunk "rfo" not in words ✗
Indexes 1, 2 invalid

Edge Cases: duplicate words in the list (e.g., ["foo","foo"] requires "foofoo" — the Counter handles multiplicity); s too short to contain any window; all characters valid but misaligned; a single word (window size = w).

⚠️ The O(n²) Trap

Checking every starting index by rebuilding a full Counter for each window is O(n × k). The key optimization is realizing that windows for indices in the same residue class mod w share overlapping word slots — they can reuse computation via a sliding window that shifts one word at a time. This drops to O(n) per offset class, O(n × w) total.

2 The Algorithm: w Interleaved Sliding Windows ▾

Because any valid window must be word-boundary-aligned, and words have length w, every possible valid starting index belongs to exactly one of the w residue classes 0, 1, …, w−1 modulo w. We run one independent sliding window per residue class.

1 Build word_count: Counter(words). This maps each word to how many times it must appear in any valid window. Track distinct = len(word_count) — number of distinct word types we must satisfy.
2 For each offset i ∈ [0, w): Run an independent pass starting at position i. Left and right pointers begin at i, stepping in increments of w — processing one word-sized chunk per step.
3 Expand right: Read the next word chunk. If it's in word_count, add it to window. If window[word] == word_count[word], increment matched. If window[word] > word_count[word], shrink left until no longer over-counted.
4 Record and slide: When matched == distinct, the window contains all required words → record left as a valid start index, then slide the window one word-position right (evict left word, advance left pointer).
5 Reset on invalid word: If the chunk is not in word_count, the entire current window is invalid. Clear window, set matched = 0, advance left past the invalid chunk.
Expand Right

Read word chunk at j. Valid word → add to window, update matched. Continue expanding.

Record Match

matched == distinct → record left as answer. Evict leftmost word, advance left.

Shrink / Reset

Over-counted → shrink left. Unknown word → reset window entirely to j + w.

🔑 Why Exactly w Passes?

If word length is w=3, valid starts at index 5 yield chunks at 5, 8, 11, … All these are at positions ≡ 2 (mod 3). No starting index ≡ 0 or 1 can ever align with index 5. So offset 0 covers indices 0, 3, 6, …; offset 1 covers 1, 4, 7, …; offset 2 covers 2, 5, 8, … Together these three passes cover every index in s. Each pass is O(n/w) word-steps × O(w) to extract a word = O(n). Total: O(n × w). Since window size = k × w, we have w = window/k, and k ≈ n/w in worst case, so O(n × k) equivalently.

3 Why It Works: The Invariant ▾
🔐 The Window Invariant

At all times, window[word] equals the count of that word-chunk in the current window [left, right). The value matched counts how many distinct word types satisfy window[w] == word_count[w] exactly. When matched == distinct, the window is a valid concatenation of exactly all required words.

Why shrinking is safe: Each left/right pointer only moves forward — never backward. This is the standard two-pointer guarantee. When we shrink left due to over-counting, we remove words from the window one at a time, updating matched when a satisfied word type falls below its target count. The "never go below zero" property of the shrink ensures we don't over-remove.

Why resetting on unknown word is correct: Any window containing an unknown word can never be valid (it would have an extra unwanted word). So we cannot begin a valid window anywhere before we pass the unknown word. Setting left = j + w skips the bad word and starts fresh — in word-aligned steps this is the earliest possible new window start.

💡 Handling Duplicate Words

If words = ["foo", "foo"], then word_count["foo"] = 2. The matched counter only increments when window["foo"] reaches exactly 2, not at 1. If a third "foo" appears, window["foo"] = 3 > word_count["foo"] = 2, triggering the shrink loop. The shrink evicts one "foo" from the left, bringing the count back to 2 and restoring matched. Duplicates are handled automatically without special casing.

⚠️ Off-by-One: Window Slide After Recording

After recording a valid start, the window must slide one word right (evict leftmost word, advance left). Forgetting this slide means the same window gets recorded on the next step when the right pointer moves — producing duplicate answers. Always slide after recording, even if it drops matched below the threshold.

4 The Code (Annotated) ▾

Solution 1 — Brute Force O(n × k): Start point

from collections import Counter
def findSubstring_naive(s, words):
w, k = len(words[0]), len(words) ← word length and word count
wc = Counter(words) ← required word multiset
res = []
for i in range(len(s) - w*k + 1): ← every possible start index
chunks = [s[i+j*w:i+j*w+w] for j in range(k)]
if Counter(chunks) == wc: ← O(k) comparison, O(k*w) to build
res.append(i)
return res ← O(n × k × w) total — rebuilds per step

Solution 2 — Optimal: w Sliding Windows (Interview Pick) ✓

def findSubstring(s, words):
if not s or not words: return [] ← guard empty inputs
w, k = len(words[0]), len(words) ← word length w, word count k
n = len(s)
word_count = Counter(words) ← required multiset: {word: needed_count}
distinct = len(word_count) ← how many distinct types must be satisfied
res = []
for i in range(w): ← w separate offset passes: 0, 1, ..., w-1
left = i ← left edge of window, word-aligned
window = {} ← current word counts in window [left, j+w)
matched = 0 ← # distinct word types exactly satisfied
for j in range(i, n - w + 1, w): ← slide right ptr in w-sized steps
word = s[j : j + w] ← extract word-sized chunk at right edge
if word in word_count: ← is this a required word?
window[word] = window.get(word, 0) + 1
if window[word] == word_count[word]: ← just reached exact count → satisfied!
matched += 1
while window[word] > word_count[word]: ← over-counted → shrink left
lw = s[left : left + w] ← word to evict from left
if window[lw] == word_count[lw]: ← was satisfied, now losing one
matched -= 1
window[lw] -= 1
left += w ← left advances by w (one word)
if matched == distinct: ← all words satisfied → valid start!
res.append(left) ← record this start index
lw = s[left : left + w] ← slide window one word right
if window[lw] == word_count[lw]:
matched -= 1
window[lw] -= 1
left += w
else: ← unknown word → entire window invalid
window.clear() ← reset: can't salvage any prefix
matched = 0
left = j + w ← skip past the bad word
return res
💡 Complexity Summary

Brute Force: Time O(n × k × w), Space O(k) — rebuilds Counter every step
Sliding Window: Time O(n × w), Space O(k) — w passes, each O(n) amortized (each word enters/leaves window at most once per pass)
Since total window size = k × w, the two formulations are equivalent when k ≈ w.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Notice the Fixed Word Length

"All words have the same length — this is the crucial constraint. It means any valid window has a fixed total length k×w. The problem reduces to: find all fixed-size windows whose word-chunks match the multiset of words."

Step 2: Propose Brute Force, Then Identify the Overlap

"Naive: try each starting index, extract k chunks, compare with Counter — O(n×k). The optimization: adjacent valid starts (like 0 and 3 for w=3) share most word-slots. Sliding window lets us add one word right and remove one word left — O(1) update."

Step 3: Explain Why w Passes Are Needed

"A window at index 5 with w=3 has chunks at 5,8,11 — offset 2 mod 3. A window at index 0 has chunks at 0,3,6 — offset 0 mod 3. These windows can't share a sliding pointer. So we need w=3 independent sliding windows, one per alignment class."

Step 4: Explain the matched Counter Trick

"Instead of comparing two full dicts each step (O(k)), I track matched — the number of distinct word types exactly satisfied. When matched == distinct, the window is complete. This gives O(1) check per step."

🗣️ What to Say to the Interviewer

"Since all words have the same length w, any valid concatenation spans exactly k*w characters.
I'll use w separate sliding windows — one per offset class mod w — so every possible start index is covered.
Each window slides in word-sized steps: add right word, evict left word when over-counted or after a match.
I track 'matched' — a counter of how many distinct word types are fully satisfied in the window.
When matched == len(word_count), record the start. Total time: O(n*w), Space: O(k)."
🔮 Likely Follow-up Questions
  • "What if words have different lengths?" → Fixed-size window breaks; need trie or Aho-Corasick.
  • "Duplicates in words[]?" → Counter handles it — word_count["foo"]=2 means need exactly 2 occurrences.
  • "Could s be a stream?" → Yes — the sliding window naturally processes s left-to-right, O(1) per word-step.

matched += 1 # newly satisfied

7 Summary & Key Takeaways ▾
Fixed word length → fixed window size → sliding window, not brute-force Counter rebuild
Run w separate passes (one per offset mod w) to cover every possible starting index
Use matched counter instead of full dict comparison — O(1) check per step
Two shrink triggers: unknown word (reset to j+w) and over-counted word (shrink left gradually)
Duplicate words in list handled automatically via Counter — no special casing needed
Always slide window one position after recording a match — prevents duplicate results
