LeetCode #43

Multiply Strings

Given two non-negative integers num1 and num2 represented as strings, return the product of num1 and num2, also as a string — without converting directly to integer.

Medium Math Grade School Multiplication Time: O(m·n) Space: O(m+n)
1 Understanding the Problem ▾

We are given two large non-negative integers as strings — not ints. We must multiply them and return the result as a string. The key constraint: no int(num1), no BigInteger library. We must simulate the arithmetic digit by digit.

💡 Key Insight

When you multiply num1[i] by num2[j], their product's units digit goes to position i+j+1 in the result, and the tens digit (carry) goes to position i+j — where index 0 is the most significant digit. This lets us accumulate all partial products into a position array first, then resolve carries in a single sweep.

Think about how you multiply on paper: 123 × 45. You multiply each digit of 45 by each digit of 123, shift by the right power of 10, and add up partial products. We do the same — instead of paper, we write to an array indexed by position (power of 10).

🔐 The Position Rule

If num1 has m digits and num2 has n digits, the product has at most m+n digits. The digit at index i from the left of num1 and index j from the left of num2 contributes to position i+j (carry/high digit) and position i+j+1 (units/low digit) in the result array — where index 0 is the leftmost (most significant) position.

Concrete example: num1="123" (m=3), num2="45" (n=2). num1[2]='3' and num2[1]='5': their product 15 contributes lo=5 to pos[4] and hi=1 to pos[3]. num1[0]='1' and num2[0]='4': product 4 contributes lo=4 to pos[1] and nothing (0) to pos[0].

2 The Algorithm: Position Array Method ▾

The key technique is a position accumulation array. Instead of resolving carries after every digit multiplication (which requires propagation chains), we accumulate all raw products first and normalize in one pass at the end.

1
Edge case: If either input is "0", return "0" immediately — avoids leading-zero issues.
2
Allocate pos array of size m+n filled with zeros. This is always large enough: (10^m − 1)(10^n − 1) < 10^(m+n).
3
Double loop (right-to-left): For each i in num1 and j in num2, compute mul = digit1 × digit2. Set hi = i+j, lo = i+j+1. Then total = mul + pos[lo]; pos[lo] = total % 10; pos[hi] += total // 10.
4
Build string: Join all digits of pos into a string, then strip leading zeros with lstrip('0'). The guard or "0" handles the case where all digits are 0.

3 Why It Works: The Position Invariant ▾
🔐 The Core Invariant

After processing all (i, j) pairs, pos[k] contains the sum of all contributions to position k from the right. This sum may exceed 9 (e.g., pos[2] might hold 13), but carry normalization reduces it. The total sum across all positions, when carries are resolved, equals the true product.

Mathematical justification: num1 represents the value Σ a_i · 10^(m−1−i) and num2 represents Σ b_j · 10^(n−1−j). Their product has a term a_i · b_j · 10^(m+n−2−i−j) for each pair (i,j). The exponent m+n−2−i−j maps to index i+j+1 (the "ones" digit of a_i·b_j) and i+j (the "tens" carry digit) in a zero-indexed array of size m+n from the left. Adding raw products to these positions accumulates the correct sum before normalization.

💡 Why Defer Carry Resolution?

If we normalized carries immediately, each multiplication might trigger a cascade of carries upward, making the code complex. By accumulating raw sums first (positions can temporarily hold values like 45 or 81 if many digits contribute to one spot), we can do a single right-to-left carry sweep in O(m+n) at the end. This is both simpler and cache-friendly.

Why m+n slots is always enough? The maximum m-digit number is 10^m − 1. The maximum n-digit number is 10^n − 1. Their product is at most (10^m − 1)(10^n − 1) = 10^(m+n) − 10^m − 10^n + 1 < 10^(m+n), which has exactly m+n digits. So pos[0] can hold at most a small carry (often 0), never a multi-digit value after normalization.

⚠️ Edge Case: Input "0"

Without the early return for "0", the code would produce pos = [0,0,...,0], join to "000...0", lstrip to "", and the or "0" guard would return "0". But it's cleaner and faster to check upfront. Also, "00" is not a valid input per constraints, but "0" alone is common.

4 The Code (Annotated) ▾
def multiply(num1: str, num2: str) -> str:
if num1 == "0" or num2 == "0": return "0" ← shortcut: anything × 0 = 0
m, n = len(num1), len(num2) ← digit counts of each input
pos = [0] * (m + n) ← result array; product has at most m+n digits
for i in range(m - 1, -1, -1): ← num1 right-to-left (units → leading digit)
for j in range(n - 1, -1, -1): ← num2 right-to-left
mul = (ord(num1[i])-48)*(ord(num2[j])-48) ← char→int without int(); ord('0')=48
hi, lo = i + j, i + j + 1 ← hi gets carry/tens, lo gets units
total = mul + pos[lo] ← include already-accumulated value at lo
pos[lo] = total % 10 ← units digit stays at lo
pos[hi] += total // 10 ← tens digit (carry) propagates to hi
return ''.join(map(str,pos)).lstrip('0') or "0" ← join, strip leading zeros, guard
⚠️ Common Mistake: Carry Direction

In our array, index 0 is the most significant digit (leftmost). So the "tens" carry goes to pos[hi] = pos[i+j], which is a smaller index (leftward, more significant). The units digit stays at pos[lo] = pos[i+j+1] (rightward, less significant). Getting this backwards produces a wrong answer.

⚠️ Common Mistake: Missing Accumulated Value in Total

When computing total = mul + pos[lo], we must include pos[lo]'s current value. If we just set pos[lo] = mul % 10 and pos[hi] += mul // 10 without accumulating, the carry from previous iterations at pos[lo] is lost and the result is wrong. This is the most frequent bug.

5 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Pattern

Signal: numbers as strings + no built-in arithmetic = simulate the operation. The prompt "you may not convert inputs to integer directly" is a direct hint to simulate grade-school multiplication digit by digit.

Step 2: Identify the State

The state is a position array of size m+n. Each slot accumulates contributions from all digit pairs that map to it. The key formula: num1[i] × num2[j] → positions i+j (carry) and i+j+1 (units).

Step 3: Derive the Position Formula

Convince yourself with a 1×1 example: num1="3" (m=1), num2="5" (n=1). Result = "15". pos has size 2. i=0, j=0: mul=15, hi=0, lo=1. pos[1]=5, pos[0]=1. Join → "15". Correct! Now generalize.

Step 4: Handle Carries Simply

Key insight: don't resolve carries inside the loop. Accumulate raw sums. A position may temporarily hold a value like 13 or 27 — that's fine. One final pass (implicit in the lstrip or explicit carry sweep) normalizes everything.

What to Say to the Interviewer

"Since we can't convert to int, I'll simulate grade-school multiplication. The key observation: multiplying num1[i] by num2[j] produces a two-digit result whose units go to position i+j+1 and whose tens digit (carry) goes to position i+j in a result array of size m+n (0-indexed from the left, most significant first). I'll allocate that array, accumulate all digit-pair products into it — including the existing value at pos[lo] before splitting — then join and strip leading zeros. Time: O(m*n) for the double loop. Space: O(m+n) for the result array."
💡 Likely Follow-ups
  • Add Strings without int() → same idea, simpler (1D, single-digit at a time)
  • Can you do O(n log n)? → FFT-based polynomial multiplication
  • Subtract Strings? → Handle borrow instead of carry

7 Summary & Extensions ▾
num1[i] × num2[j] always contributes to positions i+j (carry) and i+j+1 (units) — index 0 is the most significant digit
Accumulate all raw products first; resolve carries implicitly by including pos[lo] in total before splitting
Result array of size m+n is always sufficient: max product of m-digit × n-digit numbers has exactly m+n digits
ord(c) − 48 converts a digit character to int without using int() — or use int(c) if the constraint only bans full-string conversion
lstrip('0') strips leading zeros; the "or '0'" guard returns "0" when the entire array was zeros (edge case)
🎯 When to Use This Pattern
  • Numbers too large for 64-bit integers, represented as strings
  • Any problem requiring big-integer arithmetic from scratch
  • Interviewer says "no built-in bignum" or "no direct conversion"
  • Extends to: Add Strings (1D, simpler), Subtract Strings, polynomial multiplication
