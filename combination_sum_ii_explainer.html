LeetCode #40

Combination Sum II

Find all unique combinations in candidates that sum to target, where each number may be used at most once.

Medium Backtracking Single Use + Skip Dups Time: O(2ⁿ) Space: O(n)
1 Understanding the Problem ▾

You are given a candidate array that may contain duplicates and a target integer. Find all unique combinations where the chosen numbers sum to the target. Two constraints distinguish this from Combination Sum I: (1) each candidate may only be used once per combination, and (2) the result must contain no duplicate combinations.

💡 Key Insight

Both constraints are solved elegantly together: sort the candidates first. Sorting groups equal values adjacently — enabling a skip rule that prevents duplicate combinations — and enables early break pruning when candidates exceed the remaining target.

✓ candidates=[10,1,2,7,6,1,5], target=8

Sorted: [1,1,2,5,6,7,10]
Valid combos:
[1,1,6] → 1+1+6 = 8 ✓
[1,2,5] → 1+2+5 = 8 ✓
[1,7] → 1+7 = 8 ✓
[2,6] → 2+6 = 8 ✓

✗ Two things to avoid

Reusing same index:
[1,1,1,...] from idx=0 alone

Duplicate combos:
[1,7] listed twice because
two 1s exist in candidates

Edge cases: empty candidates (return []), no valid combo (return []), all candidates exceed target (prune immediately), single candidate exactly equals target (one-element combo).

2 The Algorithm: Sort + Backtrack + Skip Dups ▾

We explore a decision tree: at each recursive call we try including each candidate from index start onward, backtracking whenever a path exceeds the target or we've exhausted all options.

1 Sort the candidates. Groups equal values adjacently (enables skip rule) and puts values in ascending order (enables early break when a value exceeds the remaining target).
2 Recurse with (start, remaining, path). start = first index allowed (moves forward to prevent reuse). remaining = sum still needed. path = combination being built.
3 Base case — found: if remaining == 0, append path[:] (a copy!) to results and return.
4 Prune — too large: if candidates[i] > remaining, break (not continue). Since the array is sorted, all subsequent candidates are at least as large — none can work.
5 Skip dups: if i > start and candidates[i] == candidates[i-1], skip this index with continue. The i > start (not i > 0) condition is critical: it allows the first occurrence at each depth to be used, while blocking all duplicate siblings at the same depth.
6 Choose, Explore, Unchoose: append candidates[i], recurse with start=i+1 (single-use: advance past i), then pop to backtrack.
🔑 i+1 vs i — The Key Difference from Combination Sum I

In Combination Sum I (unlimited reuse), recursion uses start=i — the same index can be picked again. Here each element is single-use, so we use start=i+1. One character change separates the two problems.

Decision tree for [1,1,2,5], target=3:

candidates sorted: [1, 1, 2, 5], target=3 depth=0 (start=0) ├─ i=0 pick 1 → path=[1], rem=2 │ ├─ i=1 pick 1 → path=[1,1], rem=1 │ │ ├─ i=2 pick 2 → rem=-1 prune (2>1, break) │ └─ i=2 pick 2 → path=[1,2], rem=0 → [1,2] FOUND ✓ │ └─ i=3 pick 5 → prune (5>2, break) │ backtrack → path=[] ├─ i=1 (val=1) == candidates[0]=1, i>start → SKIP duplicate ├─ i=2 pick 2 → path=[2], rem=1 │ └─ i=3 pick 5 → prune (5>1, break) │ backtrack → path=[] └─ i=3 pick 5 → prune (5>3, break)
3 Why It Works: The Invariant ▾
🔐 The Invariant

At every recursive call, the start parameter guarantees: (a) we never reuse an index from a parent call, and (b) all candidates we consider are in non-decreasing order relative to the current path. Together these ensure each unique combination is generated exactly once, in sorted order.

Why the skip rule uses i > start, not i > 0:

Consider [1,1,2], target=4. The combo [1,1,2] is valid. At depth 0 we pick index 0 (val=1). At depth 1 (start=1), we try index 1 (val=1). The check is i=1 > start=1 → False → we are allowed to pick it. This is correct: we're picking the second 1 at a new depth, not a sibling duplicate. If we used i > 0, the check would be 1 > 0 → True → we'd wrongly skip it, missing [1,1,2].

⚠️ Most Common Bug: i > 0 instead of i > start

Using if i > 0 and candidates[i] == candidates[i-1] is subtly wrong. It blocks valid combinations where the same value is needed at different recursion depths. The correct guard is i > start — "is this a sibling we've already tried at this recursion level?"

Why break (not continue) when candidates[i] > remaining: The candidates are sorted in ascending order. If candidates[i] > remaining, then candidates[i+1] ≥ candidates[i] > remaining, so all subsequent candidates are also too large. A break exits the entire loop immediately; continue would pointlessly try each remaining (and larger) candidate one by one.

Why append path[:] not path: Python list is passed by reference. Appending path stores a reference to the same object, which will be mutated as we backtrack. By the end all entries would be empty. path[:] creates a snapshot copy.

4 The Code (Annotated) ▾

Solution 1 — Sort + Backtrack with Duplicate Skip (Interview Pick)

def combinationSum2(candidates, target):
candidates.sort() ← Sort: groups dups + enables early break pruning
results = []
def backtrack(start, remaining, path):
if remaining == 0: ← Base case: found a valid combination
results.append(path[:]) ← path[:] is a copy; path itself will be mutated
return
for i in range(start, len(candidates)):
if candidates[i] > remaining: ← Prune: sorted → all subsequent also too large
break ← break, not continue! Exit entire loop.
if i > start and candidates[i] == candidates[i-1]:← Skip dup sibling at this depth
continue ← i > start (not i > 0): allows same val at deeper levels
path.append(candidates[i]) ← Choose: add this candidate
backtrack(i + 1, remaining - candidates[i], path) ← i+1 = single use
path.pop() ← Unchoose: undo the append (backtrack)
backtrack(0, target, []) ← Entry: start=0, full target, empty path
return results

Solution 2 — Counter-based (alternative, avoids skip rule)

from collections import Counter
def combinationSum2(candidates, target):
counts = sorted(Counter(candidates).items()) ← (value, freq) pairs sorted by value
results = []
def backtrack(idx, remaining, path):
if remaining == 0: results.append(path[:]); return
if idx == len(counts): return
val, freq = counts[idx]
for k in range(0, min(freq, remaining // val) + 1): ← use 0 to k copies of val
path.extend([val] * k)
backtrack(idx + 1, remaining - val * k, path)
del path[-k:] ← remove the k values we added (handle k=0 safely)
backtrack(0, target, [])
return results
💡 Complexity Summary

Both approaches: Time O(2ⁿ) worst case — in the worst case we explore all subsets of n candidates.
Space O(n) for the recursion stack (depth ≤ n) plus O(output size) for results.
The pruning (break when candidates[i] > remaining) dramatically reduces the constant factor in practice.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize "Find All" → Backtracking

"The problem asks for all combinations, not just one or the count. This is the key signal for backtracking/DFS on a decision tree. Greedy won't work because there are multiple valid solutions we need to enumerate."

Step 2: Identify the Two Constraints

"Compared to Combination Sum I: constraint 1 = each element used once → recurse with i+1. Constraint 2 = no duplicate combos despite duplicate inputs → sort + skip rule. I'll address each explicitly."

Step 3: Explain the Skip Rule Carefully

"After sorting, if candidates[i] == candidates[i-1] and i > start (i is a sibling, not a child), skip it. The condition is i > start, not i > 0 — this is the subtle part. Using i > 0 would wrongly block valid combos that need the same value at a deeper level."

Step 4: Mention Pruning and Break vs Continue

"Because I sorted, if candidates[i] > remaining I use break (not continue) — sorted means all following candidates are also too large. This exits the loop immediately rather than testing each one."

🗣️ What to Say to the Interviewer

"This needs backtracking — we want all combinations.
Sort first: enables skip-dup rule and early break pruning.
Recurse with backtrack(start, remaining, path).
Base: remaining==0 → save path copy to results.
Prune: candidates[i]>remaining → break (sorted, so all later also exceed).
Skip: i>start AND candidates[i]==candidates[i-1] → continue (sibling dup).
Choose: path.append(candidates[i]).
Recurse: backtrack(i+1, remaining-candidates[i], path). i+1 = single-use.
Unchoose: path.pop().
Time O(2^n) worst case, O(n) stack depth."
