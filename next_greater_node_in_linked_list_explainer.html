LeetCode #1019

Next Greater Node In Linked List

Given a linked list, return an array where answer[i] is the value of the next greater node for node i, or 0 if none exists.

Medium Monotonic Stack Convert to Array + Stack Time: O(n) Space: O(n)
1 Understanding the Problem ▾

You are given a singly linked list: 2 → 7 → 4 → 3 → 5. For each node, find the first node to its right whose value is strictly greater. Return those values as an array. If no such node exists, use 0 in that slot.

💡 Key Insight

This is a classic Next Greater Element problem dressed up in a linked list. The trick: convert the list to an array first (giving index access), then apply a monotonic decreasing stack that keeps "candidates" alive until they find the first element larger than them.

Walking through [2, 7, 4, 3, 5] manually:

Node → Next Greater Value

Node 0 (val=2) → 7 (first val > 2)
Node 1 (val=7) → 0 (nothing > 7 after it)
Node 2 (val=4) → 5 (first val > 4)
Node 3 (val=3) → 5 (first val > 3)
Node 4 (val=5) → 0 (last node)
Output: [7, 0, 5, 5, 0]

Why It's Non-Trivial

Linked list = no random access
Naive: O(n²) nested scan forward
Stack approach: O(n) — brilliant!
Node 3 (val=3) answer = 5 at idx 4
Node 2 (val=4) answer = also 5!
One scan fills both at once

Edge cases: Single-element list → [0]. Strictly decreasing [5,4,3,2,1] → [0,0,0,0,0]. Strictly increasing [1,2,3,4,5] → [2,3,4,5,0]. Empty list → [].

2 The Algorithm: Convert to Array + Monotonic Stack ▾

The linked list's lack of random access is the only complication. Resolve it immediately: flatten to an array, then apply the classic monotonic stack pattern for Next Greater Element.

1 Flatten to array: Traverse the list once, collecting node.val into vals = []. Now every position has O(1) index access. Initialize result = [0] * len(vals) — default 0 means "no next greater element."
2 Initialize a monotonic stack: Create stack = []. This stack stores indices (not values) of nodes that haven't yet found their next greater element. The values at those indices form a non-increasing sequence from bottom to top.
3 Scan left to right: For each index i and value val: while the stack is non-empty AND vals[stack[-1]] < val, pop index j and set result[j] = val. That element at j just found its answer. Then push i onto the stack.
4 Remaining stack = no answer: After the scan, any index still in the stack has no larger element to its right. Their result slots are already 0 from initialization — nothing to do.
🔑 Why Store Indices, Not Values?

We need to write the answer to the correct output position. Storing index j lets us do result[j] = val — the current element is the next greater for every waiting index whose value it exceeds. We can always retrieve vals[j] for the comparison.

Stack ordering invariant — at any point the stack holds a non-increasing sequence (bottom → top), waiting for a value larger than each:

3 Why It Works: The Invariant ▾
🔐 The Monotonic Stack Invariant

At every point during the scan, the stack contains indices of elements that are waiting for their first next greater element. Their values form a non-increasing sequence from bottom to top. An element can only be popped when a strictly greater value arrives.

Why the non-increasing order holds: Before pushing index i, we pop all indices at the top whose values are strictly less than vals[i]. So vals[i] is only pushed when it is ≤ the current top value (or stack is empty). This means each newly pushed element is ≤ the previous top — maintaining the invariant.

Why popping gives the correct (first) greater element: When we pop index j because vals[i] > vals[j], index i is the first index to the right of j that is larger. Why? Because if any index k with j < k < i had a value greater than vals[j], it would have already popped j when we processed k. The stack guarantees nothing between j and i was large enough.

Total work is O(n): Each index is pushed exactly once and popped at most once — giving at most 2n operations total across all iterations of the while loop, regardless of how nested the pops appear. The amortized cost per element is O(1).

⚠️ Decreasing vs Increasing Stack Direction

A monotonic decreasing stack (bottom-to-top values are non-increasing) solves "Next Greater." We pop when the incoming element is greater than the top — that's what makes it decreasing. For "Next Smaller," you'd want a monotonic increasing stack and pop when incoming is smaller.

4 The Code (Annotated) ▾

Solution 1 — Monotonic Stack (Optimal, Interview Pick)

def nextLargerNodes(head):
vals = [] ← Collect all values from the linked list
node = head ← Traversal pointer starts at head
while node: ← Until we fall off the end of the list
vals.append(node.val) ← Grab this node's value
node = node.next ← Advance to next node
result = [0] * len(vals) ← Default 0 = "no next greater element found"
stack = [] ← Monotonic stack: stores indices waiting for answer
for i, val in enumerate(vals): ← Scan left to right with index
while stack and vals[stack[-1]] < val: ← Top of stack has smaller value → pop it
result[stack.pop()] = val ← That index found its next greater: val
stack.append(i) ← Push current index; wait for something larger
return result ← Remaining stack indices → 0 (already initialized)
⚠️ Why stack stores INDICES, not values?

We need to write the answer to result[j] for the correct position j. If we stored the value, we'd know what the waiting element is, but not where to write its answer. Indices give us both: vals[stack[-1]] for the comparison, and stack.pop() as the key into result.

Solution 2 — Brute Force O(n²) (for reference)

def nextLargerNodes_brute(head):
vals, node = [], head
while node: vals.append(node.val); node = node.next
result = [0] * len(vals)
for i in range(len(vals)): ← For each element...
for j in range(i+1, len(vals)): ← ...scan all elements to its right
if vals[j] > vals[i]: ← Found the first greater
result[i] = vals[j] ← Record and break
break
return result
💡 Complexity Comparison

Monotonic Stack: Time O(n) amortized — each index pushed + popped at most once. Space O(n) for stack + result.
Brute Force: Time O(n²) — for each of n elements, scan up to n elements to the right. Space O(n) for result only.
For n=10,000: stack processes ~20,000 ops; brute force runs ~50,000,000 ops.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Strip Away the Linked List

"The linked list structure is just packaging. The real question is: for each position in a sequence, find the first larger element to its right. That's a canonical Next Greater Element problem. I'll flatten to an array in O(n) first."

Step 2: State the Brute Force

"A naive O(n²) approach: for each element, scan everything to the right until we find a larger value. Correct, but too slow for large inputs. We can use a stack to reduce this to O(n)."

Step 3: Describe the Monotonic Stack

"Use a monotonic decreasing stack of indices. Scan left to right. For each new element: pop indices from the stack whose values are smaller — they've found their answer. Then push the current index. Whatever remains in the stack at the end has no next greater element."

Step 4: Verify Complexity

"O(n) time — each index is pushed once and popped at most once across all iterations. O(n) space for the stack and result array. We can't do better than O(n) since we must read every node."

🗣️ What to Say to the Interviewer

"This is a Next Greater Element problem on a linked list.
First pass: flatten the list to an array — O(n) time, O(n) space.
Then: use a monotonic stack storing indices. Initialize result = [0, ..., 0].
Scan left to right. For each i, pop all stack tops where vals[top] < val.
For each popped index j: result[j] = val — j found its answer.
Push i. Leftover stack after the scan = no answer (stays 0).
Time: O(n). Space: O(n). Interview pick."
🔮 Likely Follow-up Questions
  • "Circular list variant?" → Scan twice (or double the array); mod indexing (#503)
  • "Next smaller instead of next greater?" → Pop when current val < stack top val (increasing stack)
  • "Previous greater element?" → Reverse scan or maintain a different stack ordering
  • "Daily Temperatures?" → Exact same pattern, output number of days (#739)
  • "Without flattening?" → Process list nodes directly, same O(n) — stack still stores indices

7 Summary & Key Takeaways ▾
Flatten linked list to array first — decouples structure from algorithm, costs O(n)
Monotonic stack stores indices (not values) — needed to write back to result[j]
Stack invariant: values are non-increasing bottom→top; a larger arrival pops smaller ones
Each element pushed once, popped at most once → O(n) amortized, never O(n²)
Leftover stack after full scan = elements with no next greater → result stays 0
ApproachTimeSpaceNotes

🎯 When to Recognize the Monotonic Stack Pattern
  • "Next greater / smaller / warmer / taller element" → Monotonic stack immediately
  • "For each element, find the first X to its right" → Monotonic stack
  • Answer for each element depends on future values not yet seen → Stack keeps them waiting
  • "Largest rectangle" / histogram problems → Monotonic stack (previous smaller variant)
