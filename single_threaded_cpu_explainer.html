LeetCode #1834

Single-Threaded CPU

Given n tasks with enqueue times and processing times, simulate a CPU that always picks the shortest available task. Return the order in which tasks are executed.

Medium Heaps & Priority Queues Sort + Min Heap Time: O(n log n) Space: O(n)
1 Understanding the Problem ▾

You have a single CPU and n tasks. Each task is described by [enqueueTime, processingTime]. The CPU follows these rules at every scheduling decision:

→ At any moment, the CPU picks the available task with the shortest processing time.
→ A task is "available" if its enqueueTime <= currentTime. It may not be selected before it arrives.
→ On ties in processing time, pick the task with the smaller original index.
→ If no task is available when the CPU finishes, jump time directly to the next task's enqueue time.
💡 Key Insight

This is a greedy scheduling simulation. At each decision point, we want the shortest available job — Shortest Job First (SJF). A min-heap keyed by (processing_time, original_index) gives us the winner in O(log n). The challenge is efficiently tracking which tasks have "arrived" as time advances.

2 The Algorithm: Sort + Min Heap Simulation ▾

We need to efficiently answer "which available task has the shortest processing time?" at each moment. The approach: pre-sort by enqueue time, then use a min-heap to always surface the optimal task.

Tag each task with its original index. Sort by enqueue time so arrivals are ordered.

2 · Push to Heap
As time advances, sweep new arrivals (enqueue ≤ time) into the min-heap.

Pop min. Run it. Advance time. If heap empty → fast-forward. Repeat.
1 Tag & Sort: Create indexed = sorted(enumerate(tasks), key=lambda x: x[1][0]). This pairs each task with its original index and sorts by enqueue time, keeping indices intact.
2 Initialize: current_time = first task's enqueue time. Use pointer j = 0 to sweep through the sorted list.
3 Push available tasks: While indexed[j][1][0] <= current_time, push (proc_time, orig_idx) onto the heap. Increment j.
4 Fast-forward (idle): If heap is empty after the sweep, set current_time = indexed[j][1][0] and continue (this skips idle time).
5 Execute: Pop (proc, orig_i) from heap. Append orig_i to result. Advance current_time += proc.
🔑 Why Sort Before Heaping?

Sorting by enqueue time lets us use a simple linear sweep with pointer j — we only ever need to look at indexed[j] to know if the next task is ready. Without sorting, we'd scan all n remaining tasks every scheduling step, giving O(n²) total — far too slow for n up to 10⁵.

3 Why It Works: The Invariant ▾
🔐 The Heap Invariant

At every scheduling decision, the heap contains exactly all tasks that have arrived (enqueueTime ≤ current_time) but have not yet been executed. Because of the heap's min-property on (proc, idx), the top is always the correct greedy choice — shortest job with smallest index on tie.

Why Shortest Job First is correct here: The problem asks for the order defined by the SJF discipline — this is the specification, not an optimization. Any tie (equal processing times) must be broken by original index. Our heap key (proc_time, original_idx) encodes both rules directly into Python's natural lexicographic tuple comparison.

Why fast-forward is safe: When the heap empties and j < n, no task can arrive between current_time and indexed[j][1][0] — if any had, we would have pushed them (they'd have enqueue ≤ current_time). So jumping directly is lossless.

⚠️ Most Common Bug: Incrementing Time by 1

If you write current_time += 1 when the heap is empty instead of jumping directly to the next arrival, you'll get the right answer on small inputs but Time Limit Exceeded on large inputs where enqueue times can be up to 10⁹. Always jump: current_time = indexed[j][1][0].

Second most common bug — losing original index: If you sort the tasks array directly without preserving original indices, you cannot output the correct answer. Always use enumerate before sorting, or store index alongside each task before any rearrangement.

4 The Code (Annotated) ▾

Solution 1 — Sort + Min Heap (Optimal, O(n log n)) — Interview Pick

import heapq
def getOrder(tasks):
# Tag each task with its original index, then sort by enqueue time
indexed = sorted(enumerate(tasks), key=lambda x: x[1][0]) ← enumerate gives (orig_idx, [enq, proc])
result = [] ← Output: original task indices in execution order
heap = [] ← Min-heap of (proc_time, orig_idx) for available tasks
j = 0 ← Pointer into sorted indexed list
current_time = indexed[0][1][0] ← Start at first task's arrival time, not 0
while len(result) < len(tasks):
# Push all tasks that have arrived by current_time onto heap
while j < len(indexed) and indexed[j][1][0] <= current_time: ← enqueue ≤ now → available
orig_i, (enq, proc) = indexed[j]
heapq.heappush(heap, (proc, orig_i)) ← (proc, idx): min-heap sorts by proc first, then idx
j += 1
if not heap: # CPU idle — no available tasks
current_time = indexed[j][1][0] ← Fast-forward; do NOT increment by 1
continue
proc, orig_i = heapq.heappop(heap) ← Shortest available job (and smallest idx on tie)
current_time += proc ← CPU runs task for proc time units
result.append(orig_i) ← Record the original task index, not sorted index
return result
⚠️ Why current_time starts at indexed[0][1][0], not 0?

The CPU is idle until the first task arrives. If task 0 arrives at t=1000, there's nothing to do before that. Starting at 0 would loop doing nothing (or fast-forward, which is equivalent, but starting at the right time is cleaner and clearer).

💡 Complexity Summary

Sort + Min Heap (Optimal): O(n log n) sort + O(n log n) for n heap operations = O(n log n) total. Space O(n).
Brute Force: O(n² + T) where T = max enqueue time. TLE when n = 10⁵ or T = 10⁹.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Heap Pattern

"We repeatedly need the minimum from a dynamic, growing set. Classic min-heap. The wrinkle: tasks aren't all available from the start. They arrive over time." This is the Sort + Min Heap template.

Step 2: Explain the Sort-First Insight

"If I sort by enqueue time, I can use a pointer to efficiently sweep new arrivals into the heap as time passes — one pass, O(n log n) total, instead of O(n) scans per step."

Step 3: Address the Idle CPU Problem

"When the heap is empty and no tasks are available yet, I jump directly to the next task's arrival time. This avoids an O(T) loop — critical for correctness under TLE."

Step 4: Confirm Tiebreaking is Automatic

"Heap stores (proc_time, original_idx). Python compares tuples lexicographically — equal proc_times resolve by smaller index first. No custom comparator needed."

🗣️ What to Say to the Interviewer

"We need the shortest available task at each step — that's a min-heap.
First I sort tasks by enqueue time (tagging original indices before sorting).
I simulate time: sweep all tasks with enqueue ≤ current_time into a (proc, idx) heap.
If heap is empty, fast-forward current_time to next task's arrival — never increment by 1.
Pop min heap → record original index → advance time by proc_time → repeat.
Time O(n log n), Space O(n). Handles up to n=10^5 and enqueue up to 10^9."

7 Summary & Key Takeaways ▾
Sort by enqueue time first — enables O(n) sweep with pointer instead of O(n²) linear scans
Tag with original index BEFORE sorting — heap must output task identity, not sorted position
Min-heap on (proc_time, orig_idx) encodes both the greedy rule and tiebreaker simultaneously
Fast-forward idle time to next arrival — never increment current_time by 1 during idle
Python tuple comparison does the tiebreaking for you — (2,1) < (2,2) naturally
