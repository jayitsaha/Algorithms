LeetCode #815

Bus Routes

Find the minimum number of buses you must take to travel from source stop to target stop. Each bus route cycles through its stops. Return -1 if not possible.

Hard Graph BFS on Routes Graph Time: O(N·R) Space: O(N·R)
1 Understanding the Problem ▾

You have a city bus network. Each route is an ordered list of stops the bus visits (and cycles through indefinitely). You start at stop source and want to reach stop target. At any stop served by multiple routes, you can switch buses freely. You want the minimum number of buses to board, not the minimum number of stops to pass through.

💡 Key Insight

This is "minimum hops" — which signals BFS. But the unit of counting is bus rides taken, not individual stops traversed. The trick: once you board a bus, you can ride to any stop on that route at no additional cost. So BFS should expand by entire routes, not individual stops. Each new route boarded = one more bus = one BFS level.

✓ 2 buses: routes=[[1,2,7],[3,6,7]]

✗ Not possible: routes=[[1,2],[3,4]]

Edge cases: source == target → return 0 immediately. A single route containing both source and target → return 1. Routes with only one stop. Very large inputs (up to 500 routes, total stops up to 10⁵) — solution must be efficient.

2 The Algorithm: BFS on Routes Graph ▾

Reframe the problem as a bipartite graph: stops and routes are two kinds of nodes. A stop is connected to every route that serves it, and vice versa. We want the minimum number of route nodes visited on a path from any route containing source to any route containing target.

1 Trivial check: If source == target, return 0. Already at the destination.
2 Build reverse index: Iterate over every route. For each stop on that route, append the route's index to stop_to_routes[stop]. This lets us look up, in O(1), which routes are boardable from any stop.
3 BFS initialization: Seed the queue with source. Add source to visited_stops. Initialize visited_routes = set() and buses = 0.
4 BFS level loop: Each iteration of the outer while loop = one more bus boarded. Increment buses. Then process ALL stops in the current queue level. For each stop, look up its routes. For each unvisited route, mark it visited, then check all its stops — if any equals target, return buses. Otherwise enqueue all unvisited stops on that route.
5 No path: If the queue empties without finding target, return -1.
🔑 Why BFS Levels = Bus Count

Standard BFS gives shortest hops in an unweighted graph. Here each "hop" is boarding one new bus route. All stops reachable with k buses are processed in level k. Because BFS explores level-by-level, the first time we find target is guaranteed to be after the minimum number of buses. This is exactly why BFS (not DFS) is the right tool.

3 Why It Works: The Invariant ▾
🔐 The BFS Invariant

At the start of BFS level k, the queue contains every stop reachable by taking at most k−1 buses. Expanding level k (incrementing buses to k) discovers all routes boardable from those stops. All stops on those routes become reachable with exactly k buses. BFS guarantees the first time target appears, it's via the minimum k.

Why visited_routes is essential: Consider a route R that serves stops A, B, C. Without tracking visited routes, every time we encounter A, B, or C we'd re-expand R — scanning all its stops again. That's O(stops_on_R × times_R_is_encountered) work. With visited_routes, we expand R exactly once. This is the key to keeping complexity at O(N·R) rather than O(N²·R).

Why not just BFS on stops with stop-adjacency edges? Two stops are "adjacent" if they share a route. But a route with M stops has M×(M−1)/2 adjacency edges — O(M²). With many large routes this explodes. Worse, each edge would count as one "bus," but crossing the same route multiple times via stop-to-stop edges would count as multiple buses — the wrong objective entirely.

⚠️ The Crucial Subtlety: Bus Count vs Stop Count

Every stop-to-stop movement on the same bus should be free. Only changing buses costs 1. If you BFS on stops and count hops, each stop-to-stop edge counts as 1 even within the same route. This gives the wrong answer. BFS over routes is the correct model because an entire route is traversable for free once boarded.

4 The Code (Annotated) ▾

Solution 1 — BFS on Routes Graph (Optimal · Interview Pick)

from collections import defaultdict, deque
def numBusesToDestination(routes, source, target):
if source == target: return 0 ← trivial: already at destination
stop_to_routes = defaultdict(list) ← stop_id -> list of route indices
for i, route in enumerate(routes): ← build reverse index in O(total stops)
for stop in route:
stop_to_routes[stop].append(i)
queue = deque([source]) ← BFS starts from source stop
visited_stops = {source} ← don't re-enqueue stops
visited_routes = set() ← don't re-expand routes (key optimization)
buses = 0
while queue: ← process until reachable stops exhausted
buses += 1 ← boarding one more bus this BFS level
for _ in range(len(queue)): ← process exactly this level's stops
stop = queue.popleft()
for ri in stop_to_routes[stop]: ← which routes board here?
if ri in visited_routes: continue ← already boarded this route
visited_routes.add(ri) ← mark route boarded
for next_stop in routes[ri]: ← all stops accessible on this route
if next_stop == target: ← found! minimum buses achieved
return buses
if next_stop not in visited_stops:
visited_stops.add(next_stop)
queue.append(next_stop) ← reachable by buses buses
return -1 ← target unreachable
⚠️ Why check target BEFORE enqueueing (not after dequeuing)?

We check next_stop == target the moment we discover it while scanning a route. This is fine because: (1) at this point buses already holds the current level count, and (2) there's no point enqueuing target — if we find it, we return immediately. If we instead checked only when dequeuing, we'd enqueue target, then on the next level increment buses by 1 unnecessarily before dequeuing it.

💡 Complexity Summary

Let N = total stops across all routes (sum of all route lengths), R = number of routes.
BFS on Routes: Time O(N + N·R) ≈ O(N·R), Space O(N·R)
Each stop processed once; each route expanded once. N·R bounds total work.
Stop Adjacency: Time O(M² per route) — TLE + semantically wrong

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Pattern — Minimum Hops = BFS

"Minimum number of buses to reach target" — the word "minimum" with a discrete counting objective almost always means BFS. BFS in an unweighted graph guarantees the shortest path (fewest hops). The question becomes: what's the "graph" here?

Step 2: Choose the Right Abstraction — Routes Not Stops

Naively, stops are the nodes. But then each stop-to-stop move on the same bus counts as a hop — wrong objective. The insight: boarding a new bus is the hop. So BFS should expand over routes. Once you board a route, all its stops are reachable at zero additional cost.

Step 3: Bridge Stops and Routes with a Reverse Index

We start and end at stops, but the BFS expansion unit is routes. Build stop_to_routes: for each stop, which route indices serve it? This bridge lets us go from "current stop" → "boardable routes" → "newly reachable stops" in each BFS level.

Step 4: Two Visited Sets — Never Forget Routes

Track visited stops (avoids re-enqueuing) AND visited routes (avoids re-expanding the same route from multiple of its stops). Without visited_routes, you re-scan a 100-stop route 100 times — once from each stop. This is the difference between O(N·R) and O(N²·R).

🗣️ What to Say to the Interviewer

"This is a minimum-count problem, so I want BFS — it guarantees shortest hops.
The key insight: I should count bus rides, not stop transitions.
So I'll BFS with routes as the expansion unit. Each BFS level = one more bus boarded.
I'll build a reverse index: stop → route indices, to quickly find which buses I can board.
From each stop, I look up all routes, board any unvisited ones, then add all their stops to the next level.
Two visited sets are critical: stops (don't re-enqueue) and routes (don't re-expand).
The moment I encounter target while scanning a route, I return the current bus count.
Time O(N·R), Space O(N·R) where N = total stops, R = routes."
🔮 Likely Follow-up Questions
  • "What if buses have different fares?" → Dijkstra with priority queue; fare = edge weight
  • "Minimize total stops traversed, not buses?" → BFS on stops with stop-adjacency-per-route edges
  • "Bidirectional BFS?" → Yes — expand from source and target simultaneously, meet in the middle. Roughly halves the BFS depth.
  • "Real-time updates to routes?" → Maintain stop_to_routes dynamically; invalidate and rerun BFS on query

7 Summary & Key Takeaways ▾
"Minimum X to reach target" → BFS; the BFS level counter = the answer
The BFS expansion unit must match what you're counting — routes, not stops
Build stop→routes reverse index to bridge between stops and routes efficiently
Two visited sets: stops (don't re-enqueue) and routes (don't re-expand) — both are essential
Boarding a route gives ALL its stops "for free" — that's the graph model insight

🎯 When to Use BFS on Group/Route Expansion
  • Moving within a group is free; changing groups costs 1 (bus routes, word-change steps, etc.)
  • "Minimum number of services / routes / transformations" → BFS where service = one hop
  • Entities belong to sets; transitions happen at set intersections (shared members)
  • Hint: if you need a reverse index (element → which groups contain it), this pattern applies
