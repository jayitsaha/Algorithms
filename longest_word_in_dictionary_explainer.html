LeetCode #720

Longest Word in Dictionary

Find the longest word that can be built one character at a time using other words in the dictionary. On a tie, return the lexicographically smallest.

Medium Tries Trie + BFS Valid Words Time: O(Σ|w|) Space: O(Σ|w|)
1 Understanding the Problem ▾

We are given a list of words. We want to find the longest word that can be built one character at a time, where every intermediate prefix must also be a word in the list. Think of it as spelling out a word letter-by-letter, where each partial spelling must itself be a valid dictionary entry.

💡 Key Insight

A word W of length n is valid if and only if all of its proper prefixes — W[0:1], W[0:2], …, W[0:n-1] — are also words in the dictionary. This is a prefix-chain constraint, and a Trie physically encodes exactly this structure.

Consider the input ["w","wo","wor","worl","world","write"]:

"w" ∈ dict ✓

"wor" ∈ dict ✓

Tiebreaking rule: When multiple words share the maximum length, return the lexicographically smallest. E.g., if "abc" and "xyz" both have length 3 and both are fully buildable, return "abc".

Edge Cases: Empty dictionary returns "". A single-character word always qualifies (the "empty prefix" is trivially valid). If no word has a full prefix chain, return "".

2 The Algorithm: Trie + BFS on Valid Nodes ▾

A Trie (prefix tree) is the ideal data structure: it stores all words such that nodes along a path from root represent prefixes. We insert all words, marking each terminal node. Then we run BFS, but with a twist: we only expand from nodes that are word-endings. This automatically enforces the "every prefix must be a word" constraint.

1 Build the Trie. Insert every word. At each word's final character node, set node.word = word (the full string). Nodes that are not word-endings have node.word = "".
2 Seed BFS. Push all direct children of root into the queue. These represent single-character words. BFS starts here because single-character words trivially satisfy the prefix requirement.
3 BFS main loop. Dequeue a node. If node.word == "", skip it entirely — its prefix chain is broken. Otherwise, it's a valid buildable word.
4 Update best. If this word is longer than our current best, or same length but lexicographically smaller, update the answer.
5 Expand only from valid nodes. Push all children of the current node into the queue. They will be checked in step 3. If a child has word = "", it gets skipped (and its subtree is unreachable — BFS never expands from skipped nodes).
🔐 Why BFS, not DFS?

Both work for correctness. BFS visits all words of length 1 before length 2, length 2 before length 3, etc. This means the first word we see at any depth is guaranteed to have shorter prefixes already processed. However, the real constraint enforcement comes from only expanding word-end nodes, not from BFS vs DFS order. DFS would also work; BFS is conventional and slightly cleaner for this problem.

3 Why It Works: The Invariant ▾
🔐 The BFS Reachability Invariant

A Trie node N at depth D is reachable via our BFS if and only if the string formed by the path root → N (the length-D word) has every prefix of length 1 through D in the dictionary. This invariant holds because we only ever push a node's children into the queue from nodes that already satisfy the invariant.

Formal proof by induction on depth D:

  • Base case (D=1): We seed the queue with root's children. A node at depth 1 represents a single-character word. The only prefix of such a word (of any length less than 1) is the empty string, which we treat as trivially in the dictionary. So every node we seed is valid if and only if it is a word-end (which we check in step 3).
  • Inductive step: Assume every node at depth D in the queue satisfies the invariant. When we dequeue a node at depth D, we first check node.word != "". If it passes, the word W (length D) is in the dictionary, and by the inductive hypothesis, all prefixes W[0:1]…W[0:D-1] are also in the dictionary. When we push a child C (depth D+1), C forms the word W + c.char. Its prefixes are W[0:1]…W[0:D-1], W — all confirmed in the dictionary. The invariant holds at depth D+1.
⚠️ The Subtle Correctness Point

Notice that we push ALL children into the queue, not just the word-end children. The filtering happens on dequeue: if not node.word: continue. This means invalid nodes get added to the queue but never expanded. Their children are never pushed. So the "barrier" is the dequeue check, not the enqueue step. This keeps the code simple while maintaining correctness.

💡 Why the Sort+HashSet Alternative Also Works

When we sort words lexicographically, shorter words always appear before longer words (because shorter strings are lex-smaller than their extensions with the same prefix). So when we encounter a word of length L, we've already processed all words of length < L. If word[:-1] is in our "seen" set, we know the word of length L-1 was valid (it satisfied its own prefix chain), and by the prefix chain property, all shorter prefixes were valid too.

4 The Code (Annotated) ▾

Solution 1 — Trie + BFS (Interview Pick, O(Σ|w|) time)

from collections import deque
class TrieNode:
def __init__(self):
self.children = {} maps char → TrieNode child
self.word = "" full word string if this is a word-end, else ""
def longestWord(words):
root = TrieNode() root has no character; represents empty prefix
for word in words:
node = root
for ch in word:
if ch not in node.children:
node.children[ch] = TrieNode()create node if not exists
node = node.children[ch]walk down
node.word = word mark terminal: store the full word here
res = ""
queue = deque(root.children.values()) start BFS from depth-1 nodes
while queue:
node = queue.popleft()
if not node.word:not a complete word → prefix chain broken → skip
continue
w = node.word
if len(w) > len(res) or (len(w) == len(res) and w < res):
res = w update: longer, or equal-length but lex-smaller
queue.extend(node.children.values()) all children go to queue; invalid ones filtered on dequeue
return res
⚠️ Why store the full word in node.word?

We store the entire word string (not just a boolean is_end = True) so that during BFS we can retrieve the answer directly without re-tracing the path from root to the current node. This is a standard Trie trick for problems that need the actual word, not just a flag.

Solution 2 — Sort + Hash Set (Simpler to Code, O(N log N + Σ|w|) time)

def longestWord(words):
words.sort() lex sort: shorter words come before their extensions
seen = {""} empty string = valid base; any single-char word's prefix is ""
res = ""
for word in words:
if word[:-1] in seen: prefix of length L-1 already confirmed valid?
seen.add(word) this word is now a valid prefix for longer words
if len(word) > len(res):sorted order ensures first-seen at this length = lex smallest
res = word
return res
💡 Why sorting enables the O(1) prefix check

After sorting, when we encounter "world", we've already processed "w", "wo", "wor", "worl" (if present). Each was validated and added to seen. So checking only word[:-1] in seen is enough — if the immediate (length-1) prefix is valid, all shorter prefixes must have been valid too (validated even earlier in the sorted iteration). The lex sort also gives us tiebreaking for free: first word of a given length encountered is lex smallest.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Prefix Pattern

Keywords: "built one character at a time", "dictionary", "every prefix must exist". Whenever a problem is about prefix relationships over a set of strings, the Trie is your first instinct. This problem is essentially "find the deepest reachable node in a Trie where every ancestor node is a word-end."

Step 2: Eliminate Brute Force Verbally

Say: "I could iterate over all words, and for each word check all of its prefixes in a hash set — that's O(N × L²) which may be acceptable, but a Trie reduces this to O(∑|w|) because prefix structure is baked in." This shows you know both options.

Step 3: State the Trie BFS Insight

Say: "If I insert all words into a Trie, marking terminal nodes, then I can BFS from root — but only expand from nodes that are word-ends. Any reachable node is guaranteed to have a valid prefix chain by induction. The deepest such node gives my answer."

Step 4: Address Tiebreaking Explicitly

Don't forget: "For equal-length words, I return the lex smallest. In BFS I compare: update if longer, or if same length and lex-smaller. In the Sort approach, sorted order guarantees first-seen is lex smallest automatically."

Step 5: Offer the Alternative

Bonus: "There's also a O(N log N) alternative: sort + hash set. Sort ensures we process shorter words first. For each word, if its immediate prefix (word[:-1]) is in 'seen', add it. This is often cleaner to code under time pressure." Showing two approaches signals strong problem-solving depth.

🗣️ What to Say to the Interviewer

"This is a prefix-chain validation problem. Each candidate word must have
every prefix (length 1 through n-1) also in the dictionary — a natural
Trie invariant.

Plan: build Trie, insert all words marking terminals. Then BFS: only expand

💡 Follow-up Questions You Might Get
  • What if words are given as a stream? (Trie handles incremental inserts.)
  • What if we want all qualifying words, not just the longest? (Collect every non-empty BFS node.)
  • Can we do it with O(1) extra space? (No — we need at minimum a set or Trie of size O(∑|w|).)
  • What if words have uppercase letters? (Expand Trie node children from 26 to 52.)

7 Summary & Key Takeaways ▾
A word is valid if and only if every prefix is also in the dictionary — this is the "prefix chain" constraint
A Trie encodes all prefix relationships physically; BFS traversal can enforce the prefix chain naturally
Only expand BFS from nodes that are complete words — this single rule enforces the entire invariant
Store the full word in terminal nodes (not just a flag) to avoid retracing paths during BFS
Sort + Hash Set is a valid O(N log N) alternative: simpler code, slightly worse asymptotically but often faster in practice
💡 When to Reach for Trie + BFS/DFS
  • Problem requires checking all prefixes of strings in a collection
  • Words must be "built incrementally" from shorter to longer through a dictionary
  • Multiple strings share long common prefixes (Trie gives amortized O(1) per char)
  • You need to traverse only "valid" paths (mark terminals, BFS/DFS with validity check)
