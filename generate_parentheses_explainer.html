LeetCode #22

Generate Parentheses

Given n pairs of parentheses, generate all combinations of well-formed parentheses.

Medium Backtracking Track Open/Close Counts Time: O(4ⁿ/√n) Space: O(n)
1 Understanding the Problem ▾

Given n, return every possible string made of exactly n opening brackets ( and n closing brackets ) that form a well-formed (valid) sequence.

💡 Key Insight

A parentheses string is well-formed if and only if two conditions hold: (1) it uses exactly n of each bracket type, and (2) at every prefix of the string, the count of ( is always greater than or equal to the count of ). These two rules are all we need.

Edge Cases: n=1 → only (). The problem guarantees 1 ≤ n ≤ 8, so the output is always small enough to collect in a list. For n=8, there are 1430 valid strings (C₈).

🔑 Why Not Generate All Strings and Filter?

For n=4 that's 2⁸ = 256 strings to generate and validate. For n=8: 65,536 strings — but only 1430 are valid. That's a 98% waste rate. Backtracking prunes invalid branches before they grow, visiting only the ~1430 leaves instead of 65,536.

2 The Algorithm: Backtracking with Count Guards ▾

At each position in the string we're building, we have a binary decision: add ( or add ). We track two integers: open (opening brackets placed) and close (closing brackets placed). Two simple guards determine which choices are legal:

1 Can add (? Only if open < n. We still have unused opening bracket budget. Recurse with open+1.
2 Can add )? Only if close < open. There is at least one unmatched ( to close. This prevents strings like )( or ()). Recurse with close+1.
3 Base case: When open == n AND close == n, the string uses all brackets and every one is matched. Append to results.
4 No explicit undo needed: We pass curr + "(" (a new string) rather than mutating a shared buffer. Each branch of recursion holds its own copy. Python's immutable strings do the backtracking for us.

🔑 Why This Generates ONLY Valid Strings — No Post-Filtering Needed

The guard close < open before adding ) guarantees every ) has a matching ( to its left. The base case open==close==n guarantees all brackets are used. Together: every string that reaches the base case is structurally valid by construction.

3 Why It Works: The Invariant ▾
🔐 The Invariant

Throughout the recursion: close ≤ open ≤ n. This single compound constraint encodes two facts simultaneously: (a) we never exceed our bracket budget, and (b) at any prefix of the string, opening brackets never fall behind closing brackets.

Why close < open ensures validity: Consider what this check means at the moment we add ). We require close < open — meaning the number of ) we've placed so far is strictly less than the number of (. This is exactly the condition for "there is an unmatched opening bracket." Every ) we add is immediately matched to one of those unmatched openers. Invalid sequences like )( are impossible because we can never add ) when close == 0 == open.

Completeness — do we miss any valid strings? At each step, we try every legal choice: ( if open < n, ) if close < open. We never skip a legal option. Since both branches are always explored when they're legal, the recursion visits every valid string exactly once — no duplicates, no omissions.

Connection to Catalan numbers: The number of valid parentheses combinations for n pairs is the nth Catalan number, Cₙ = C(2n,n)/(n+1). For n=1: 1, n=2: 2, n=3: 5, n=4: 14, n=5: 42. This is why the time complexity involves 4ⁿ/√n — it bounds the total work across all Catalan branches.

⚠️ Common Mistake: Checking Length Instead of Counts

Some write if len(curr) == 2*n as the base case. This works, but misses the opportunity to prune: with separate count guards, we can reject invalid branches immediately. The count-based approach is cleaner, more efficient, and directly expresses the two logical rules of well-formed parentheses.

4 The Code (Annotated) ▾

Solution 1 — Backtracking via String Concatenation (Interview Pick)

def generateParenthesis(n: int) -> list[str]:
result = [] ← accumulate all valid strings
def backtrack(curr, open, close): ← curr = string so far; open/close = counts placed
if open == n and close == n: ← base case: used all n of each → valid string
result.append(curr) ← save it; implicit return causes backtracking
return
if open < n: ← can we still add an opening bracket?
backtrack(curr + "(", open+1, close)← new string object per call; no undo needed
if close < open: ← unmatched ( exists → safe to close
backtrack(curr + ")", open, close+1)← recurse with one more close bracket
backtrack("", 0, 0) ← kick off: empty string, zero of each
return result
💡 Why String Concatenation Means "Free" Backtracking

In Python, curr + "(" creates a new string object. The original curr is unchanged. So when the recursive call returns, curr in the caller's frame still has its old value — the "undo" happens automatically. No explicit pop or delete needed.

Solution 2 — Backtracking with Mutable List (Explicit Undo / More Efficient)

def generateParenthesis(n: int) -> list[str]:
result, path = [], [] ← path is shared; we append/pop explicitly
def backtrack(open, close):
if open == n and close == n: ← base case: join once at the leaf (cheaper)
result.append("".join(path))
return
if open < n:
path.append("(") ← mutate: choose '('
backtrack(open+1, close)
path.pop() ← undo: remove the '(' we just added
if close < open:
path.append(")")
backtrack(open, close+1)
path.pop() ← undo: crucial — without this, path accumulates garbage
backtrack(0, 0)
return result
⚠️ The pop() IS the Backtracking

path.pop() after each recursive call is not optional cleanup — it is the backtracking step. Without it, the path accumulates characters from previous branches and produces completely wrong results. This is why the pattern is called backtracking: you literally go back and undo the last choice before trying the next one.

💡 Complexity Summary

Time: O(4ⁿ / √n) — the nth Catalan number bounds the number of valid strings, each of length 2n to construct.
Space: O(n) for the recursion call stack (at most 2n levels deep before hitting the base case).
Catalan Numbers: C₁=1, C₂=2, C₃=5, C₄=14, C₅=42, C₆=132, C₇=429, C₈=1430.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Spot the "Generate All" Signal

"Generate all combinations" is the single biggest signal for backtracking. You're not finding one optimal answer — you're exploring a search space and collecting all leaves that satisfy a constraint. Think: decision tree with pruning.

Step 2: Define What State You Need

Ask: "What information determines which choices are legal at each step?" For this problem, you need open and close counts — nothing else. The string we're building is derived state, not decision state.

Step 3: Enumerate the Legal Choices + Guards

At each position: choice 1 is add (, valid when open < n. Choice 2 is add ), valid when close < open. Expressing these as explicit guards gives you the pruning automatically — no separate validation function needed.

Step 4: Define the Base Case and Return

When open == n and close == n, the string is complete and valid. Append to results and return. Because both guards prevent us from adding more than n of each, this base case is the only termination condition.

🗣️ What to Say to the Interviewer

"This is a backtracking problem — we need to generate all valid combinations.
I'll track two integers: open brackets placed and close brackets placed.
At each step, I can add '(' if open is still below n,
and ')' only if close is strictly less than open — guaranteeing every ')' has a match.
The base case is when both equal n: we have a complete valid string.
Time complexity is O(Cₙ × n) where Cₙ is the nth Catalan number (~4ⁿ/√n).
Space is O(n) for the recursion depth."
🔮 Likely Follow-up Questions
  • "Can you do it iteratively?" → Yes — BFS with a queue of (curr, open, close) tuples. Same guards, queue instead of call stack.
  • "Count without generating?" → It's C(2n, n)/(n+1), the nth Catalan number. For n=3: C(6,3)/4 = 20/4 = 5.
  • "Remove invalid parentheses from a given string?" → LeetCode #301 — BFS removing one bracket at a time, checking validity at each level.
  • "Multiple bracket types: {}, [], ()?" → Same framework, but the stack-based validity check needs to verify matching pairs, not just counts.

7 Summary & Key Takeaways ▾
"Generate all valid X" → backtracking. Binary decision at each step, prune invalid branches early.
Two integers (open, close) encode the full state. No need to inspect the string itself.
Guard: add '(' only if open < n. Add ')' only if close < open. These two rules are all the logic.
The invariant close ≤ open ≤ n guarantees every generated string is valid — no post-validation needed.
String immutability makes backtracking "free" — each recursive call gets its own string copy.
Output count = Catalan number Cₙ. C₃=5, C₄=14, C₅=42. Time is O(Cₙ × n) ≈ O(4ⁿ/√n).
