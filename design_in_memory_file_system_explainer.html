LeetCode #588

Design In-Memory File System

Implement a file system supporting mkdir, addContentToFile, readContentFromFile, and ls operations using a Trie of directory nodes.

Hard Design Trie of Directories Time: O(L) per op Space: O(N·L)
1 Understanding the Problem ▾

We need to build a simplified in-memory file system that supports four operations: creating directories, adding content to files, reading file content, and listing directory entries. Think of it like a tiny Unix filesystem living entirely in RAM — no disk, no OS.

💡 Key Insight

A filesystem is a tree of named nodes. Each node is either a directory (which can have children) or a file (which stores content). An absolute path like /a/b/c is just a sequence of edge labels to follow from the root. This maps perfectly to a Trie — where each "character" is an entire directory/file name rather than a single letter.

What's tricky

• ls("/") could return a directory OR a file listing
• mkdir must create all intermediate dirs
• addContentToFile must auto-create parent dirs
• Paths always start with "/" — split carefully

Concrete example

mkdir("/a/b/c")
addContentToFile("/a/b/c/d","hello")
ls("/a/b") → ["c"]
readContent("/a/b/c/d") → "hello"

The four operations explained:

  • ls(path): If path is a file → return [filename]. If it's a dir → return sorted list of its children names.
  • mkdir(path): Create the full path, auto-creating any missing intermediate directories. No return value.
  • addContentToFile(filePath, content): If file exists, append content. If not, create it (auto-create parent dirs). No return value.
  • readContentFromFile(filePath): Return the full content string of the file at that path.

Edge cases: root "/" is always present, paths always start with "/", ls("/") returns sorted children of root, a file can be created at the same time as intermediate dirs, content is appended (not replaced) on repeated addContentToFile calls.

2 The Algorithm: Trie of Directory Nodes ▾

We model the filesystem as a Trie where each node represents either a directory or a file. Edges are labeled by name (e.g., "a", "b", "myfile.txt"). Splitting a path by "/" gives us the sequence of edges to traverse.

🔑 Why a Trie (Not a Hash Map of Full Paths)?

We could store paths as flat keys in a dict, but then ls("/a") would require scanning all keys with the prefix "/a/" — O(N) per ls. With a Trie, ls is O(L) path traversal + O(k log k) for sorting k children. Structural queries are instantaneous.

1 TrieNode: Each node holds a dict children: {name → TrieNode} and a string content. If content is non-empty, the node is a file; otherwise it's a directory. A single node type handles both cases elegantly.
2 Path traversal helper: Split path by "/" and drop the empty string at index 0 (since paths start with "/"). Walk from root, creating missing nodes on demand (for mkdir and addContentToFile).
3 ls(path): Walk to the node. If it has content, it's a file — return [last component of path]. Otherwise it's a dir — return sorted(node.children.keys()).
4 mkdir(path): Walk each component, creating child nodes as needed. No return value, no content set.
5 addContentToFile(filePath, content): Walk all but the last component (creating dirs), then create/access the file node and append content to it.
6 readContentFromFile(filePath): Walk to the file node and return its content string. O(L) traversal, O(1) read.
🧠 Analogy: A Trie of Words vs. a Trie of Paths

In a classic string Trie, each edge is a single character. In this filesystem Trie, each edge is a full name (like "documents" or "readme.txt"). It's the same traversal logic — just with a dict of names instead of an array of 26 letters. This is sometimes called a "radix Trie" at the word boundary.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At all times, every path from the root to any node in the Trie corresponds to a valid absolute path in the filesystem. The depth of a node equals the number of components in its path. Any node with a non-empty content string is a file; any node with an empty content string is a directory (even if it has no children — an empty directory).

Why content="" means directory: When we create a directory node via mkdir, we never set content. We only set content when addContentToFile is called on the final node. A file can have children in our model (reflecting the simplification in the problem — a file path component can appear as a parent in another path), but in the constraints given, this never occurs.

⚠️ The ls Edge Case: Path Points to a File

When ls is called with a path that points to a file (not a directory), the expected output is a list containing just the filename itself — [last_component] — NOT an empty list, and NOT the file's content. You must distinguish files from dirs in the final node. A quick if node.content check handles this cleanly.

Why "walk and create" is safe: The _walk helper creates nodes on demand (setdefault pattern). Because we always start from the root and walk forward, we can never accidentally skip a level. Each intermediate component gets its own node, so the invariant holds after every mutation.

⚠️ Splitting "/" — Common Off-by-One

"/a/b/c".split("/") yields ["", "a", "b", "c"]. The leading empty string must be filtered out with [x for x in path.split("/") if x]. Forgetting this causes a spurious lookup for the empty-string key in root's children.

4 The Code (Annotated) ▾

Solution 1 — Trie of Nodes (Optimal, Interview Pick)

class TrieNode:
def __init__(self): ← Each node = one dir or file entry
self.children = {} ← name (str) → TrieNode, maps child names
self.content = "" ← "" = directory, non-empty = file with this content
class FileSystem:
def __init__(self): ← Create the root "/"
self.root = TrieNode() ← Root node always exists; represents "/"
def _walk(self, path, create=False):← Helper: traverse path, optionally create nodes
parts = [x for x in path.split("/") if x] ← Split + drop "" from leading "/"
node = self.root ← Start at root
for p in parts: ← Walk each path component
if create: ← mkdir/addContentToFile mode
node.children.setdefault(p, TrieNode())← Create child only if missing
node = node.children[p] ← Descend into child
return node ← Return the final node
def ls(self, path: str):
node = self._walk(path) ← Traverse without creating
if node.content: ← Non-empty content → this IS a file
return [path.split("/")[-1]] ← Return just the filename
return sorted(node.children.keys())← Directory → sorted children names
def mkdir(self, path: str):
self._walk(path, create=True) ← Walk with creation; that's it
def addContentToFile(self, filePath: str, content: str):
node = self._walk(filePath, create=True) ← Auto-create all parent dirs
node.content += content ← Append (not replace) content
def readContentFromFile(self, filePath: str):
return self._walk(filePath).content ← Walk, then return stored content
⚠️ Why += instead of = for content?

The problem says repeated addContentToFile calls on the same path append to existing content. Using += handles both the first write (empty string + content) and subsequent appends uniformly. Using = would silently discard prior content.

💡 Complexity Summary

Let L = length of path (number of components), k = number of children in a directory.
ls: O(L + k log k) — L for traversal, k log k for sorting
mkdir: O(L) — create up to L nodes
addContentToFile: O(L + |content|) — traverse + string append
readContentFromFile: O(L) — traverse + return
Space: O(N · L) where N = total nodes, L = average path depth

Solution 2 — Flat HashMap (Simpler, Slower ls)

class FileSystem:
def __init__(self): ← No trie needed
self.dirs = {"/"} ← Set of existing directory paths
self.files = {} ← filePath → content string
def ls(self, path):
if path in self.files: ← It's a file
return [path.rpartition("/")[2]] ← Extract filename
# It's a dir: find direct children
prefix = path if path == "/" else path + "/"
children = set()
for p in list(self.dirs) + list(self.files): ← O(N) scan — expensive
if p.startswith(prefix):
rest = p[len(prefix):]
children.add(rest.split("/")[0])
return sorted(children)
def mkdir(self, path):
parts = path.split("/")
for i in range(2, len(parts)+1):
self.dirs.add("/".join(parts[:i])) ← Add each prefix
def addContentToFile(self, fp, content):
self.mkdir(fp.rpartition("/")[0] or "/") ← Ensure parent dirs
self.files[fp] = self.files.get(fp, "") + content
def readContentFromFile(self, fp):
return self.files[fp]
🔑 Trie vs. Flat Map: The Trade-off

The Trie solution has O(L) ls (after traversal, children are directly accessible). The flat-map solution has O(N·L) ls since it scans all stored paths. For an interview, always prefer the Trie — it's the canonical answer and shows understanding of tree data structures for hierarchical data.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Hierarchy

"A filesystem has a hierarchical, tree-like structure. Paths are sequences of names. This screams Trie — a tree where each edge is a name segment from splitting the path on '/'."

Step 2: Design the TrieNode

"Each node needs: (1) a dict mapping child names to child nodes, and (2) a content string. Non-empty content distinguishes files from directories. One class handles both — I don't need separate File and Directory classes."

Step 3: Extract the Walk Helper

"All four operations need path traversal. I'll write a single _walk(path, create=False) helper. With create=True it uses setdefault to auto-create nodes; with create=False it just navigates. This eliminates code duplication."

Step 4: Handle the ls File Case

"The trickiest part is ls on a file path. I check if the final node has content — if so, it's a file, return [just the filename]. Otherwise it's a directory, return sorted children. Easy check, but easy to forget."

🗣️ What to Say to the Interviewer

"Filesystems are inherently hierarchical — paths are sequences of names. I'll model this as a Trie where each node represents one directory or file. Each node has a children dict mapping names to TrieNode objects, plus a content string — empty for directories, non-empty for files.

The key insight: all four operations share the same path traversal logic. I'll extract a _walk helper that splits the path by '/', filters the empty leading string, then walks the Trie, optionally creating missing nodes.

For ls, after traversal I check if the node has content — it's a file, so return [filename]. Otherwise return sorted children keys. For mkdir, just walk with creation. For addContentToFile, walk with creation and append. For readContentFromFile, walk and return content.

Time: O(L) per operation (L = path depth), plus O(k log k) sort in ls. Space: O(N·L) total nodes."
🔮 Likely Follow-up Questions
  • "Support delete file/dir?" → Walk to parent, remove child key from parent.children
  • "Support rename?" → Walk to parent, rename the key in parent.children dict
  • "Support symlinks?" → TrieNode gets a symlink_target field; walk resolves symlinks recursively
  • "Make it thread-safe?" → Add a threading.RLock; acquire on all mutating operations
  • "Persist to disk?" → Serialize the Trie with BFS/DFS; deserialize on load

7 Summary & Key Takeaways ▾
A filesystem is a Trie where edges are full names (not single characters)
One TrieNode class handles both files (content ≠ "") and directories (content = "")
A single _walk helper with create=True/False eliminates code duplication across all four operations
ls must handle the file case: return [filename], not sorted children
Split by "/" and filter empty strings to handle the leading "/" in absolute paths
addContentToFile uses += to append content (not = to replace)
