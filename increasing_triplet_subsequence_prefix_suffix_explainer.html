Increasing Triplet Subsequence

Prefix Min / Suffix Max Approach — Deep Dive

#334 Medium Prefix Min / Suffix Max Array Precomputation O(n) Time · O(n) Space
1 Understanding the Problem

Given an array nums, determine if there exist three indices i < j < k such that nums[i] < nums[j] < nums[k].

💡 Key Insight

For any middle element nums[j] to be part of a valid triplet, two conditions must be true:

  • Something smaller exists to the LEFT — at least one element in nums[0..j-1] is smaller than nums[j]
  • Something larger exists to the RIGHT — at least one element in nums[j+1..n-1] is larger than nums[j]

If we precompute the minimum to the left and maximum to the right for each position, we can check each middle in O(1) time!

2 The Algorithm: Prefix Min / Suffix Max

📊 prefix_min[i]

The smallest value in nums[0..i]. Answers: "What's the minimum value I've seen so far from the left?"

📊 suffix_max[i]

The largest value in nums[i..n-1]. Answers: "What's the maximum value remaining to the right?"

1 Build prefix_min[] — Scan left-to-right, tracking running minimum
2 Build suffix_max[] — Scan right-to-left, tracking running maximum
3 Check each middle j — For j from 1 to n-2: if prefix_min[j-1] < nums[j] < suffix_max[j+1], return True
📐 The Recurrence Relations
prefix_min[i] = min(prefix_min[i-1], nums[i])
↳ Smallest so far = min(previous smallest, current value)
suffix_max[i] = max(suffix_max[i+1], nums[i])
↳ Largest remaining = max(next largest, current value)
3 Why It Works: The Invariant
🔐 The Invariant

For any index j:

  • prefix_min[j-1] is the minimum of ALL elements strictly to the left of j
  • suffix_max[j+1] is the maximum of ALL elements strictly to the right of j

If prefix_min[j-1] < nums[j] < suffix_max[j+1], then there MUST exist indices i and k such that nums[i] < nums[j] < nums[k].

⚠️ Why Check j from 1 to n-2?

The first element (j=0) has nothing to its left — no valid i exists. The last element (j=n-1) has nothing to its right — no valid k exists. Only middle indices can be the "j" of a triplet!

Edge Cases: If no valid middle exists, we've checked every possibility in O(n) time — return False.

4 The Code (Annotated)
# Build auxiliary arrays for O(1) lookups at each position
def increasingTriplet(nums):
n = len(nums)
prefix_min = [0] * n # prefix_min[i] = min of nums[0..i]
suffix_max = [0] * n # suffix_max[i] = max of nums[i..n-1]
# Pass 1: Build prefix minimums (left-to-right)
prefix_min[0] = nums[0]
for i in range(1, n):
prefix_min[i] = min(prefix_min[i-1], nums[i])
# Pass 2: Build suffix maximums (right-to-left)
suffix_max[-1] = nums[-1]
for i in range(n-2, -1, -1):
suffix_max[i] = max(suffix_max[i+1], nums[i])
# Pass 3: Check each middle element
for j in range(1, n-1):
if prefix_min[j-1] < nums[j] < suffix_max[j+1]:
return True # Found valid triplet!
return False # No triplet exists

5 🎤 Interview Strategy: How to Arrive at This
Step 1: Recognize the Pattern

"Looking for a subsequence with relative ordering" → Consider what information each position needs about elements before/after it.

Step 2: Identify the Bottleneck

For each middle element, we need to know: "Is there something smaller to the left? Something larger to the right?" Checking naively is O(n) per element → O(n²) total.

Step 3: Precomputation Insight

Precompute these answers once! Running min from left, running max from right. Now each middle check is O(1).

Step 4: Mention the Trade-off

"This uses O(n) space. There's also an O(1) space solution using two variables, but this prefix/suffix approach is more intuitive and easier to verify correct."

🗣️ What to Say to the Interviewer
"For each position j to be a valid middle element, I need to know if there's a smaller value to its left and a larger value to its right. If I precompute prefix_min[i] = minimum of nums[0..i] and suffix_max[i] = maximum of nums[i..n-1], then I can check each middle position in O(1). This gives O(n) time with O(n) space. There's also an O(1) space approach tracking just two smallest values, but I'll start with this clearer solution first."

7 Summary & Key Takeaways
✓ Prefix/suffix precomputation converts O(n) per-query lookups into O(1)
✓ For triplets: check if "something smaller left" AND "something larger right"
✓ Three passes: build prefix_min → build suffix_max → scan for valid middle
✓ O(n) time, O(n) space — clear and easy to verify
🎯 When to Use This Pattern
  • Need min/max from "everything to the left" or "everything to the right"
  • Problem asks about subsequences with relative ordering
  • O(n) space is acceptable for clarity
  • When simpler to code than O(1) space alternatives

💡 Comparison: This vs O(1) Space Solution

There's also an O(1) space solution that tracks just first and second smallest values. It's more efficient but trickier to understand (why does updating first not break the invariant?).
