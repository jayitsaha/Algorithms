LeetCode #155

Min Stack

Design a stack that supports push, pop, top, and getMin — all in O(1) time.

Medium Stacks Auxiliary Stack for Mins Time: O(1) all ops Space: O(n)
1 Understanding the Problem ▾

We need a custom stack data structure that behaves exactly like a normal stack (LIFO — Last In, First Out) but also lets us retrieve the current minimum element instantly at any point in time. A normal stack doesn't track minimums — if you want the minimum of a stack, you'd have to scan every element (O(n)). We need O(1) for all four operations simultaneously.

💡 Key Insight

The minimum changes dynamically: pushing a smaller value lowers it; popping the current minimum reverts it to whatever it was before. We must remember the minimum at every historical depth — not just the current one.

Consider the sequence: push(-2), push(0), push(-3), getMin() → -3, pop(), getMin() → -2. After popping -3, the minimum immediately snaps back to -2 without any scanning — this is the challenge.

✗ Naive: one stack + one min variable

push(-2): stack=[-2], min=-2
push(0): stack=[-2,0], min=-2
push(-3): stack=[-2,0,-3], min=-3
pop(): stack=[-2,0], min=???
Must scan to find -2. O(n)!

✓ Two parallel stacks

push(-2): main=[-2], mins=[-2]
push(0): main=[-2,0], mins=[-2,-2]
push(-3): main=[-2,0,-3], mins=[-2,-2,-3]
pop(): main=[-2,0], mins=[-2,-2]
getMin()=mins[-1]=-2. O(1)!

Edge cases to consider: pushing INT_MIN, duplicate values (push(1), push(1) — after one pop, min is still 1), a stack with a single element, repeated getMin() calls without modifications.

2 The Algorithm: Parallel Auxiliary Min Stack ▾

Maintain two stacks in lockstep: self.stack (main — stores every pushed value) and self.min_stack (auxiliary — at position i, stores the minimum of all elements from index 0 through i of the main stack). They always have equal length.

1 Initialize: self.stack = [] and self.min_stack = []. Both empty.
2 push(val): Append val to self.stack. For self.min_stack: if it is non-empty, append min(val, self.min_stack[-1]); otherwise append val directly. This records "what is the global min when this element sits on top."
3 pop(): Pop the top from both stacks simultaneously. When the main value leaves, its paired min record in the auxiliary stack also leaves — automatically revealing the previous minimum with no computation.
4 top(): Return self.stack[-1]. Single O(1) array index into the main stack.
5 getMin(): Return self.min_stack[-1]. The top of the auxiliary stack is always the minimum of all currently-present elements. O(1).
🔑 The Pairing Invariant

At all times: min_stack[i] == min(stack[0], stack[1], ..., stack[i]). The two stacks are always equal length. This means whenever element i is on top of the main stack, min_stack[-1] accurately reflects the minimum across the entire stack — with zero scanning.

3 Why It Works: The Invariant ▾
🔐 The Invariant

After every push or pop, min_stack[-1] equals the minimum of all elements currently in stack. This invariant is maintained by construction at every operation.

Why push maintains the invariant: Before pushing val, min_stack[-1] holds the current global min (call it m). After pushing val, the new global min is min(val, m). We record exactly this value as the new top of the min stack. It cannot be wrong.

Why pop maintains the invariant: After popping the top of main stack, the new global min is whatever the min was just before that element was pushed — which is now the new top of the min stack (we popped both in tandem). Correct, with zero recomputation.

Duplicate values — the subtle case: Push 5, push 5 again. min_stack = [5, 5]. Pop one 5 → min_stack = [5]. The minimum is still correctly 5. This works because we record a min entry for every push regardless of whether the min changed — each depth level has its own record.

⚠️ Alternative: Only Push to Min Stack When Value ≤ Current Min

To save memory, some solutions only push to min_stack when val <= min_stack[-1]. This works but requires extra pop logic: only pop from min_stack when the popped main-stack value equals min_stack[-1]. The "always push" approach is simpler and more interview-friendly — it avoids conditional complexity at the cost of slightly more memory in runs with many non-minimum pushes.

Solution 1 — Auxiliary Min Stack (Interview Pick, O(1) all ops)

def __init__(self): ← Constructor: two parallel stacks
self.stack = [] ← Main stack: stores all pushed values
self.min_stack = [] ← Aux stack: min_stack[i] = min(stack[0..i])
def push(self, val: int) -> None:
self.stack.append(val) ← Always push val onto main stack
if self.min_stack: ← If min_stack is non-empty, compare with current min
self.min_stack.append(
min(val, self.min_stack[-1])) ← New min = min(incoming val, existing global min)
else:
self.min_stack.append(val) ← First element is trivially the minimum
def pop(self) -> None:
self.stack.pop() ← Remove top from main stack
self.min_stack.pop() ← Remove paired min record — reveals previous min
def top(self) -> int:
return self.stack[-1] ← Peek at top of main stack; O(1)
def getMin(self) -> int:
return self.min_stack[-1] ← Top of min stack = global minimum; O(1)
⚠️ Why not just track a single min variable?

A single self.cur_min works for push: update it if val < cur_min. But on pop, if the popped element was the minimum, you can't recover the previous minimum without scanning the entire remaining stack — O(n). The min stack stores this history explicitly, one entry per depth level.

Solution 2 — Store (val, current_min) Tuples (same complexity, single structure)

class MinStack:
def __init__(self):
self.stack = [] ← Each entry is a (value, min_at_this_depth) tuple
def push(self, val: int) -> None:
cur_min = min(val, self.stack[-1][1]) if self.stack else val
self.stack.append((val, cur_min)) ← Pack val + its depth-min together in one tuple
def pop(self) -> None: self.stack.pop()
def top(self) -> int: return self.stack[-1][0] ← First element of top tuple
def getMin(self) -> int: return self.stack[-1][1] ← Second element of top tuple
💡 Complexity Summary

All operations (push, pop, top, getMin): Time O(1) — all are simple list appends or index reads
Space: O(n) total — the auxiliary min stack has at most n elements (exactly matching the main stack)
Trade-off: 2× memory in the two-stack variant vs 1× with tuples — both are O(n) but tuples use less memory in practice

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify the Constraint

The key signal is "O(1) getMin" on a mutable stack. If the stack were immutable, you could compute the min once and be done. But pop changes the stack — so we need to maintain min across arbitrary state changes. This signals "track history."

Step 2: Propose and Dismiss the Single-Variable Approach

"I could track a single cur_min variable. Updating on push is easy. But on pop — if I remove the minimum element, I can't find the previous minimum without scanning. That's O(n). I need the min history at each depth."

Step 3: Arrive at Parallel Stack by Thinking About Pairing

"What if at each depth level I record 'the minimum when this element is on top'? Then push stores a new record, pop removes that record — automatically revealing the previous min. This is a second stack synchronized with the first."

Step 4: Trace the Duplicate Edge Case

Always trace: push(3), push(3). min_stack = [3, 3]. Pop one 3 → min_stack = [3]. Min is still 3. Correct. This demonstrates you handle duplicates without any special casing — the "always record" approach handles it naturally.

🗣️ What to Say to the Interviewer

"The challenge: getMin() must be O(1) but pop() can change the minimum.
A single min variable fails after popping the min — we lose the previous min.
Solution: a parallel 'min stack' where each entry records the global min at that depth.
push → append to both; pop → remove from both; getMin → read top of min stack.
All four ops are O(1). Space is O(n). Let me trace through an example to verify..."
🔮 Likely Follow-up Questions
  • "Can you save space?" → Only push to min_stack when val <= current min (saves entries, adds conditional pop logic)
  • "What about a Max Stack?" → Identical approach, replace min() with max()
  • "Need both min and max in O(1)?" → Two auxiliary stacks — one for mins, one for maxs
  • "Design a queue with O(1) min?" → Two-stack queue where each half has its own min stack

7 Summary & Key Takeaways ▾
A parallel auxiliary stack preserves the minimum history at every depth level
Both stacks always stay the same length — they grow and shrink in perfect lockstep
getMin() is O(1): just read min_stack[-1], no scanning required
Duplicate values are handled correctly — each push records its own min entry
Trade-off: 2× memory in exchange for O(1) time on all four operations

🎯 When to Recognize the Auxiliary Stack Pattern
  • "O(1) access to aggregate (min/max/sum) of a changing stack or queue"
  • "History-dependent query" — answer depends on when elements were pushed
  • Any design problem requiring rollback, undo, or "snapshot" capabilities
  • When a naive approach requires scanning but you need constant-time retrieval
