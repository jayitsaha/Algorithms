LeetCode #1168

Optimize Water Distribution in a Village

Find the minimum total cost to supply water to all houses — each house can dig a well (direct cost) or connect to another house via a pipe.

Hard Graph Virtual Node + MST Kruskal's + Union-Find Time: O(E log E) Space: O(V + E)
1 Understanding the Problem ▾

You have n houses (numbered 1 to n) in a village. Each house needs water. Water can be obtained in exactly two ways:

  • Dig a well at house i — costs wells[i-1]. The house gets its own water source.
  • Lay a pipe between two houses u and v — costs cost. Once connected, water can flow between them (transitively).

A house gets water if it either has its own well, or it is connected (directly or transitively through pipes) to any house that has a well. You want to supply all houses with water at minimum total cost.

💡 Key Insight — This Is a Graph Problem In Disguise

Think of "digging a well at house i" as "connecting house i to an unlimited, free water source." If we model that water source as a virtual node 0, then digging a well at house i becomes a pipe from 0 to i with cost wells[i-1]. Now every option (well or pipe) is just an edge in a graph. We want to connect all nodes (0 through n) at minimum cost — that's exactly a Minimum Spanning Tree problem.

MST picks edges cheapest first: (0→1, 1) + (1→2, 1) + (2→3, 1) = total cost 3. House 1 drills a well; houses 2 and 3 get water via pipes.

Edge Cases: n=1 (just pick min of drilling or no pipes), all pipes cheaper than all wells (one well + all pipes forms a star), all wells cheaper than any pipe (each house drills independently).

2 The Algorithm: Virtual Node + Kruskal's MST ▾

The approach has two distinct phases: graph construction (add the virtual node and its edges) and MST extraction (Kruskal's greedy edge selection with Union-Find).

1 Add virtual node 0. For each house i (1-indexed), add an edge (wells[i-1], 0, i). This represents "drill a well at house i with cost wells[i-1]." Node 0 is the shared water source; connecting to it = having a well.
2 Add all pipe edges. For each entry [u, v, cost] in pipes, add edge (cost, u, v). All edges are now in the same list, formatted as (cost, u, v).
3 Sort all edges by cost. Kruskal's greedy principle: always try the cheapest connection first. Sorting takes O(E log E) time.
4 Initialize Union-Find for nodes 0 through n. Each node starts as its own component. parent = list(range(n + 1)).
5 Kruskal's: process edges greedily. For each edge (cost, u, v) in sorted order: find the root of u and v. If they are in different components, merge them and add cost to total. If they are already connected, skip (adding this edge would form a cycle).
6 Return total. The MST of the augmented graph (nodes 0..n) has exactly n edges and connects all n+1 nodes at minimum cost. The MST cost = minimum cost to supply all houses.
🔬 Algorithm Deep-Dive: Kruskal's + Union-Find

Kruskal's MST Algorithm (Joseph Kruskal, 1956) builds a minimum spanning tree by greedily adding the cheapest edge that doesn't create a cycle. It works on any connected weighted graph. The cycle check is done efficiently with Union-Find (Disjoint Set Union).

Union-Find with Path Compression + Union by Rank achieves near-O(1) per operation (amortized O(α(n)), where α is the inverse Ackermann function — effectively constant). The find function traces to the root; path compression makes future lookups O(1) by wiring all nodes directly to root.

Core Invariant: At any point in Kruskal's, the selected edges form a forest (acyclic subgraph). Adding an edge between two different components grows the forest by one tree. We stop when all nodes are in one component = the MST.

Why it gives the global optimum: The "Cut Property" of MSTs: for any cut (partition) of the graph, the minimum-weight crossing edge is always in some MST. Since Kruskal's always picks the global cheapest safe edge, it exploits this property at each step.

⚠️ Why Virtual Node Works — The Critical Insight

Once we add virtual node 0, the problem becomes: "Find an MST over nodes {0, 1, 2, …, n}." The MST will select exactly n edges to form a spanning tree. Since node 0 must be connected to some house (otherwise we have no water source at all), at least one edge of the form (0, i) will be selected — representing at least one well being drilled. Additional (0, i) edges in the MST mean additional wells are drilled. All other MST edges are pipes. The minimum cost MST automatically balances how many wells to drill versus how many pipes to lay.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every step of Kruskal's, the selected edges form a minimum-cost spanning forest of the nodes seen so far. When we process the cheapest remaining edge: if it connects two different components, merging them is always optimal (Cut Property). If it connects nodes already in the same component, adding it would create a cycle — MSTs are acyclic, so we skip it.

Why the virtual node guarantees correctness: In the original problem, houses need water from either a local well or via a chain of pipes to any house with a well. In the augmented graph, "house has water" = "house is connected to node 0." After Kruskal's MST, every node 1..n is connected to node 0 through the spanning tree. Edges touching node 0 are wells; edges between non-zero nodes are pipes. This exactly models all valid water supply configurations.

Optimality proof sketch: Suppose there exists a cheaper valid configuration. In that configuration, some subset of houses have wells, others use pipes. Map wells → edges to node 0, pipes → edges between non-zero nodes. This gives a connected subgraph over {0..n}. Its spanning tree has cost ≤ total configuration cost. But Kruskal's finds the globally minimum spanning tree, contradiction.

⚠️ Common Mistake: Forgetting Node 0 in Union-Find

The Union-Find must be initialized for nodes 0 through n — that's n+1 nodes total. A common bug is initializing only n nodes and then getting an index error or wrong answer when processing virtual edges. Use parent = list(range(n + 1)).

Edge cases handled: If n=1, the only possible action is to drill a well (no pipes possible). The MST of {0, 1} picks the single edge (0→1, wells[0]). If all wells are very cheap, Kruskal's picks n edges of the form (0→i) — every house drills independently. If all pipes are very cheap, Kruskal's picks one well (the cheapest) and then n-1 pipes to connect everyone.

4 The Code (Annotated) ▾

Solution 1 — Virtual Node + Kruskal's MST (Optimal, Interview Pick)

def minCostToSupplyWater(n, wells, pipes): ← n houses, wells[i] = cost to drill at house i+1
# Build augmented edge list
edges = [(w, 0, i+1) for i, w in enumerate(wells)] ← virtual edges: node 0 → house i+1, cost = well cost
edges += [(c, u, v) for u, v, c in pipes] ← real pipe edges; reformat to (cost, u, v)
edges.sort() ← Kruskal's: sort by cost (cheapest first)
# Union-Find (path compression)
parent = list(range(n + 1)) ← n+1 nodes: 0 (virtual) and 1..n (houses)
def find(x):
if parent[x] != x:
parent[x] = find(parent[x]) ← path compression: wire directly to root
return parent[x]
total = 0 ← accumulate MST cost
for cost, u, v in edges: ← process each edge cheapest-first
pu, pv = find(u), find(v) ← find component roots of u and v
if pu != pv: ← different components → safe to merge (no cycle)
parent[pu] = pv ← union: merge components
total += cost ← add edge to MST
return total ← MST cost = minimum water supply cost
⚠️ Why We Don't Need a Rank/Size for Union?

Union by rank optimizes union operations, but for this problem size (n ≤ 10,000), path compression alone is sufficient and the code is simpler. In competitive programming, parent[pu] = pv is idiomatic — it's fast enough. In a production system, add rank or size tracking for robustness.

Solution 2 — Brute Force: Try All Subsets (Exponential — DO NOT USE)

# For each subset of houses that will have wells: ← 2^n subsets
# cost = sum of well costs in subset
# + minimum pipe cost to connect remaining houses ← Steiner tree problem — NP-hard in general
# Return minimum over all subsets ← O(2^n * n^2) — impractical for n > 20
💡 Complexity Summary

Virtual Node + Kruskal's: Time O((n + p) log(n + p)) where p = number of pipes, Space O(n + p)
Brute Force Subset: Time O(2ⁿ · n²) — exponential, unusable
The virtual node trick reduces a hard mixed-optimization problem to a standard MST with zero loss in optimality.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Graph Pattern

The problem has houses and connections with costs — classic graph setup. Two types of costs (well vs. pipe) might initially seem like a complication, but notice both are just "ways to get water to a house." Think: "Is this a connectivity / spanning tree problem?" Yes — we need all n houses connected to water at minimum cost.

Step 2: Identify the Blocker — Two Cost Types

Standard MST algorithms assume edges between existing nodes. But wells are "self-connections" — they don't fit the usual graph model. The insight: wells are really connections to a hypothetical global water source. Give that source a node number (0) and convert wells to edges. Now everything is uniform.

Step 3: Apply Kruskal's with Union-Find

Once all options are edges, sort by cost and greedily add cheapest non-cycle-forming edges until all houses are connected (to node 0, directly or transitively). This is exactly Kruskal's algorithm. Implement Union-Find for O(log n) per operation.

Step 4: Trace Through an Example

Show the interviewer: n=2, wells=[1,2], pipes=[[1,2,1]]. Edges sorted: (1, 0→1), (1, 1→2), (2, 0→2). Add (0→1): cost 1. Add (1→2): cost 1. Skip (0→2): 0 and 2 already connected. Total = 2. ✓

🗣️ What to Say to the Interviewer

"I see two types of choices: drill wells and lay pipes. Both are ways to supply water. Let me unify them.
I'll model this as a graph problem. Add a virtual node 0 as the water source.
Drilling a well at house i = adding an edge from 0 to i with cost wells[i-1].
Laying a pipe = an edge between two houses as given.
Now I need to connect all houses to node 0 at minimum cost — that's a Minimum Spanning Tree.
I'll use Kruskal's: sort all edges by cost, and greedily add them if they don't create a cycle.
Union-Find handles cycle detection in near-O(1) per edge.
Total time: O(E log E) for sorting. Space: O(V) for Union-Find."
🔮 Likely Follow-up Questions
  • "Can you do it with Prim's instead?" → Yes! Start from virtual node 0, use a min-heap to greedily expand. O((V+E) log V). Useful when E is dense.
  • "What if each pipe has capacity and houses have demand?" → Network flow problem (max-flow / min-cut); MST no longer applies.
  • "What if wells can supply limited houses?" → Facility location problem; much harder (NP-hard in general).
  • "What's path compression doing exactly?" → When we call find(x) and trace up to root r, we set every node along the path to point directly to r. Future calls to find for those nodes are O(1).

7 Summary & Key Takeaways ▾
The "virtual node" trick converts a mixed optimization problem (well vs. pipe) into a pure MST problem with zero loss in optimality.
Kruskal's algorithm + Union-Find solves MST in O(E log E) — dominated by the sort step.
Path compression in Union-Find makes find() effectively O(1) amortized — critical for performance.
The MST automatically decides how many wells to drill: one "virtual edge" in the MST = one well drilled.
When a problem has two "types" of connections, consider whether one type can be modeled as connections to a shared abstract node.
🎯 When to Use the Virtual Node + MST Pattern
  • Two types of costs: "self-supply" cost per node vs. "connection" cost between nodes
  • Need to minimize total cost to make all nodes "reachable" to a resource
  • Graph connectivity + cost minimization in the same problem
  • The signal phrase: "each node can either pay a fixed cost OR connect to another node"

🔑 The Unifying Insight

The virtual node pattern is an instance of a general technique: model abstract choices as concrete graph edges. When an algorithm needs a single unified input (all edges) but the problem gives you heterogeneous options (wells vs. pipes), introduce a dummy node that represents the "free resource" and convert every option into an edge. Then apply your standard algorithm unchanged. This technique appears in network design, facility location simplifications, and competitive programming frequently.
