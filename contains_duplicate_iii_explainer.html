LeetCode #220

Contains Duplicate III

Given an integer array nums and integers indexDiff (k) and valueDiff (t), return true if there exist indices i, j such that |i−j| ≤ k and |nums[i]−nums[j]| ≤ t.

Hard Sorting Bucket Sort + Sliding Window Time: O(n) Space: O(k)
1 Understanding the Problem ▾

We have an integer array and two thresholds: indexDiff k (how close two indices can be) and valueDiff t (how close two values can be). We want to know: is there any pair of distinct positions where both constraints hold simultaneously?

💡 Key Insight

Think of a sliding window of size k sweeping across the array. For each new element, we need to ask: "Is there any element already in the window whose value is within t of mine?" The challenge is answering that question in O(1) rather than O(k). Bucket sort achieves this by partitioning the value space into slots of size (t+1).

✓ Found: nums=[1,5,6,2], k=2, t=3

i=1: num=5, bucket 1
i=2: num=6, bucket 1 ← SAME BUCKET!
|5−6|=1 ≤ t=3 and |1−2|=1 ≤ k=2
→ return True

✗ Not found: nums=[1,5,9,1,5,9], k=2, t=3

Adjacent pairs: (1,5)→diff=4, (5,9)→diff=4
4 > t=3 for every pair in any window
Duplicates (1,1) at dist=3 > k=2
→ return False

Edge Cases: t=0 (exact duplicates within k — each bucket size 1), k=0 (no valid distinct pair since i≠j required), single-element (always False), all-same values with k≥1 and t=0 (True immediately), negative numbers (Python floor division handles correctly, Java/C++ needs custom floor-div).

2 The Algorithm: Bucket Sort with Sliding Window ▾

We divide the entire integer number line into buckets of width (t+1). Every bucket is an interval of exactly t+1 consecutive integers. Any two integers in the same bucket are guaranteed to differ by at most t — no further check needed. Two integers in adjacent buckets might still be close enough, so we verify those explicitly. Two integers two or more buckets apart are too far.

1 Set bucket width w = t+1. Edge case: if t < 0, return False immediately (impossible constraint). Each bucket B covers integers [B·w, (B+1)·w − 1].
2 Compute bucket ID: For integer x, bid = x // w. Python's floor division handles negatives correctly. Check 3 buckets: bid, bid−1, bid+1.
3 Same-bucket hit → instant True: If bid already exists in the map, the stored value and x are in the same bucket, so their difference is ≤ t. No absolute-value check needed.
4 Neighbor buckets → verify then True: If bucket bid±1 exists, compute |x − stored_value|. If ≤ t, return True. Adjacent buckets might just barely qualify.
5 Insert x: Store buckets[bid] = x. At most one value per bucket — if there were two, we'd have already returned True.
6 Slide window (evict oldest): When i ≥ k, delete buckets[nums[i−k] // w]. This removes the element that is now more than k positions behind, keeping the window exactly size k.
🔑 Why bucket size = t+1, not t?

A bucket of width t+1 can hold integers 0, 1, 2, ..., t — exactly t+1 values. The maximum difference between any two is t. If we used width t, we'd need to also check the same bucket's boundary values carefully. Width (t+1) makes same-bucket → guaranteed ≤ t, clean and simple.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every iteration i, the bucket map contains at most one entry per bucket, and only for elements at indices [i−k, i−1]. Since each bucket holds at most one value, the map is a valid dict. Any match in the map is automatically within k positions of i.

Why at most one value per bucket at any time? Suppose two values x and y are both in bucket B simultaneously. They're both in the window (within k of current i). They differ by at most t. That means we would have found and returned True when inserting the second one. So having two values in the same bucket coexist is impossible — it's a contradiction with our algorithm's early return.

Why only 3 buckets to check? Let x be in bucket B (covering [B·w, (B+1)·w−1]) and y be in bucket C. If |C−B| ≥ 2, then y ≥ (B+2)·w and x ≤ (B+1)·w − 1. So y − x ≥ (B+2)·w − (B+1)·w + 1 = w = t+1 > t. Guaranteed too far. Only C = B−1, B, B+1 can possibly satisfy |x−y| ≤ t.

⚠️ Negative Numbers: Floor Division Matters

Python's // always floors toward −∞: −7 // 4 = −2 (not −1). This ensures bucket continuity across zero. In Java/C++, / truncates toward zero, breaking the bucket for negatives. Python users get this for free; other languages need a helper: if x >= 0: return x // w else: return (x + 1) // w − 1.

Index constraint maintenance: We evict element at index i−k right after inserting element at index i. This means whenever we check the bucket map, every stored element is at an index in [i−k+1, i−1], i.e., within k steps. The eviction step is what enforces the |i−j| ≤ k constraint.

4 The Code (Annotated) ▾

Solution 1 — Bucket Sort + Sliding Window (Interview Pick, O(n) time)

def containsNearbyAlmostDuplicate(nums, k, t): ← k=indexDiff, t=valueDiff
if t < 0: return False ← negative valueDiff: impossible
buckets = {} # bid → value ← at most 1 value per bucket at any time
w = t + 1 # bucket width ← integers in same bucket differ by ≤ t
for i, x in enumerate(nums): ← slide i from 0 to n-1
bid = x // w ← Python floor div: correct for negatives
if bid in buckets: ← same bucket → |diff| ≤ t guaranteed
return True
if bid-1 in buckets and abs(x - buckets[bid-1]) <= t:← left neighbor
return True ← verify actual diff for adjacent buckets
if bid+1 in buckets and abs(x - buckets[bid+1]) <= t:← right neighbor
return True
buckets[bid] = x ← insert x; at most 1 per bucket
if i >= k: ← window about to exceed k elements
del buckets[nums[i - k] // w] ← evict oldest: remove its bucket entry
return False ← no valid pair found
⚠️ Why no absolute check for same bucket?

Bucket B covers [B·w, (B+1)·w−1] — exactly w = t+1 integers. The max difference between any two values in the same bucket is (B+1)·w−1 − B·w = w−1 = t. So same-bucket always satisfies |diff| ≤ t. For adjacent buckets (B−1 or B+1), the min cross-bucket value could be 0 (if they're right next to each other at the boundary) or as large as 2t. We must verify.

Solution 2 — Sorted Set / BST (O(n log k) time, cleaner to explain)

from sortedcontainers import SortedList
def containsNearbyAlmostDuplicate(nums, k, t):
window = SortedList() ← sorted structure of last k elements
for i, x in enumerate(nums):
pos = window.bisect_left(x - t) ← find leftmost value ≥ x−t
if pos < len(window) and window[pos] <= x + t:← value in range [x-t, x+t]?
return True
window.add(x) ← O(log k) insert
if len(window) > k: ← evict oldest
window.remove(nums[i - k]) ← O(log k) removal
return False

Solution 3 — Brute Force (O(n·k), for building intuition only)

def containsNearbyAlmostDuplicate(nums, k, t):
n = len(nums)
for i in range(n):
for j in range(i+1, min(i+k+1, n)): ← check at most k neighbors
if abs(nums[i] - nums[j]) <= t: ← value constraint satisfied?
return True
return False
💡 Complexity Summary

Bucket Sort: Time O(n), Space O(k) — each element is one O(1) dict op + one O(1) eviction
Sorted Set (BST): Time O(n log k), Space O(k) — good if bucket trick not known
Brute Force: Time O(n·k), Space O(1) — TLE on large inputs

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Decompose the two constraints

"I notice two independent constraints: a positional one (|i−j| ≤ k) and a value one (|nums[i]−nums[j]| ≤ t). I'll handle position with a sliding window of size k, then focus on efficiently querying within that window for the value constraint."

Step 2: Brute force first — O(n·k)

"Naively, for each element I scan the previous k elements and check the value difference. That's O(k) per element, O(n·k) overall. For k close to n that's O(n²) — too slow."

Step 3: Propose BST / sorted structure — O(n log k)

"If I maintain the window in a sorted structure (balanced BST or Python's SortedList), I can binary search for values in [x−t, x+t] in O(log k). Insert and delete are also O(log k). Total O(n log k)."

Step 4: The bucket insight for O(n)

"Can we do O(1) per element? If we partition the value axis into buckets of size (t+1), values in the same bucket are guaranteed to differ by ≤ t — no check needed. We only need to look at 3 buckets. Total O(n)."

Step 5: Call out edge cases proactively

"t=0 gives bucket size 1 — exact duplicates, handled perfectly. Negative numbers: Python floor division is correct. In Java I'd need a custom floor-div. The map has at most k entries at any time — O(k) space."

🗣️ What to Say to the Interviewer

"I'll decompose this: sliding window handles the index constraint, bucket sort handles value proximity in O(1).
Bucket width = t+1. Same bucket → diff ≤ t guaranteed (no check). Adjacent → verify actual diff.
For each element: compute bid = x // (t+1). Check buckets bid, bid−1, bid+1. Insert. Evict oldest when i ≥ k.
At most one value per bucket at any time — if two landed in same bucket we'd have returned True already.
Python floor division handles negatives. O(n) time, O(k) space."
🔮 Likely Follow-up Questions
  • "What if t is INT_MAX?" → Degenerates to Contains Duplicate II — any pair within k
  • "How in Java?" → Use TreeSet + floor()/ceiling(), or custom bucket floor-div
  • "Why can't we have two values in the same bucket simultaneously?" → If we did, we'd have returned True when inserting the second — contradiction with reaching that point
  • "What if k=0?" → Loop never finds valid j≠i with |i−j|≤0, so always False

7 Summary & Key Takeaways ▾
Two constraints (index distance, value distance) are handled independently: sliding window for position, bucket sort for value.
Bucket width (t+1) guarantees same-bucket → |diff| ≤ t always. No explicit check needed for same bucket.
Only 3 buckets to inspect per element: own bucket and two neighbors. Buckets 2+ away are provably too far.
At most one value per bucket in the map at any time — enforced by our own early-return invariant.
Python's floor division handles negatives correctly. Mention this to the interviewer for Java/C++ awareness.

🎯 When to Use Bucket Sort for Proximity
  • Sliding window + "are any two values within t of each other?"
  • You have a natural bucket size from the value threshold
  • O(1) per element required (BST gives O(log k) which might be enough)
  • The value domain is dense integers (not floats or strings)
