LeetCode #670

Maximum Swap

Given a non-negative integer, you may swap two digits at most once to get the maximum valued number. Return the maximum valued number you can get.

Medium Greedy Find Rightmost Larger Time: O(n) Space: O(1)
1 Understanding the Problem ▾

You have a non-negative integer (up to 108). You may perform at most one swap of any two of its digits. Your goal is to produce the largest possible number from that single operation — or leave it unchanged if no swap helps.

💡 Key Insight

To maximize a number, the leftmost (most significant) digit should be as large as possible. Scan left-to-right: at the first position where a larger digit exists somewhere to its right, swap the current digit with the rightmost occurrence of that largest available digit. That single swap produces the global maximum.

✓ Swap helps: 2736 → 7236

pos 0: digit '2'
Largest to right: '7' at idx 1
Swap idx 0 and 1 → 7236

✗ Already optimal: 9973

pos 0: '9' — nothing larger
pos 1: '9' — nothing larger
No swap improves → 9973

Tricky case — 98368: Position 0 has '9' (max possible). Position 1 has '8'; rightmost '8' is also at index 4 — swapping identical digits changes nothing. Position 2 has '3'; rightmost of max-to-right ('8' at index 4) gives 98863. That's the answer!

Why rightmost? If '9' appears at both index 3 and index 7, and we're at index 0, bringing the '9' from index 7 (rightmost) to index 0 leaves the '9' at index 3 in a better position than if we had swapped the one at index 3 and displaced the one at 7 further right. Rightmost always wins in tie-breaking.

Edge cases: Single-digit number (return as-is). All same digits (no improvement). Input like 1999 → swap '1' with rightmost '9' → 9991.

2 The Algorithm: Greedy + Last-Occurrence Map ▾

The brute-force approach tries all O(n²) pairs of digit positions, swaps them, and tracks the maximum. That works since n ≤ 9 (for 108), but the greedy approach is O(n) and scales to arbitrarily large numbers.

The O(n) trick: precompute last[d] = the rightmost index where digit d appears. Then for each position i left to right, check digits 9, 8, 7, … down to digits[i]+1. The first k where last[k] > i is our swap partner — execute it and stop.

1 Convert to digit list: digits = list(str(num)). This lets us index and swap individual digits.
2 Build last-occurrence map: One pass: last = {int(d): i for i, d in enumerate(digits)}. Later indices overwrite earlier ones, so last[d] automatically holds the rightmost position of digit d.
3 Greedy scan: For each position i, try digits 9 down to int(digits[i])+1. If last[k] > i, swap digits[i] and digits[last[k]] and return immediately.
4 No swap found: If we exhaust all positions without swapping, the number is already in descending order and cannot be improved. Return num unchanged.
🔑 Why Try k=9 Down to k=digits[i]+1?

We want the largest possible digit to move to position i. Checking from 9 downward guarantees we always try the biggest possible improvement first. The moment we find any valid k (digit present AND to the right of i), that is the globally optimal swap — there's no larger digit available, and position i is the leftmost improvable position. Stopping immediately is correct.

3 Why It Works: The Greedy Invariant ▾
🔐 The Invariant

When we scan position i left to right, we maintain the invariant that digits at positions 0..i-1 are already at their maximum possible values given no larger digit exists to their right. If position i can be improved (a larger digit exists to its right), our greedy swap is globally optimal: no swap starting anywhere else can produce a larger result.

Why the leftmost improvable position dominates: Number magnitude is determined left-to-right lexicographically. Improving the digit at position 0 by even 1 matters more than maximally rearranging all remaining digits. For example, 2999 → 9992 beats any swap of positions 1, 2, 3 (which can only produce 2999 still).

Exchange argument (proof sketch): Suppose our algorithm identifies swap (i, j) as optimal. Claim: no other swap (i', j') produces a larger number. If i' > i, then position i' matters less than position i (position i is more significant). If i' = i, we chose the largest available digit for that position (checking 9 first), so (i, j') with a different j' cannot give a larger digit at position i. If i' < i, positions 0..i'-1 already had no larger digit available to their right (we passed them without swapping), so no swap can improve them either. QED.

⚠️ Common Mistake: last.get(k, 0) instead of k in last

If you write last.get(k, 0) > i to check whether digit k exists to the right of i, this has a bug: if k is not present, get returns 0, and if i == 0, the condition 0 > 0 is False — accidentally correct, but fragile. Better: k in last and last[k] > i. Always be explicit.

Rightmost correctness: Suppose digits 7 appears at index 1 and index 5, and we're swapping from index 0. Swapping with index 5 gives …7…7… (the 7 at index 1 stays). Swapping with index 1 also gives a 7 at position 0, but the 7 that was at index 1 is now somewhere else. Since we want to maximize the number after the leftmost position, keeping the second '7' at its earlier (more significant) index is better, which is exactly what swapping with the rightmost occurrence achieves.

4 The Code (Annotated) ▾

Solution 1 — Greedy + Last-Occurrence Map (Interview Pick)

def maximumSwap(num: int) -> int:
digits = list(str(num)) ← "2736" → ['2','7','3','6']; allows index-based swapping
last = {int(d): i ← Map each digit to its rightmost index
for i, d in enumerate(digits)} ← Later iterations overwrite earlier ones → rightmost wins automatically
for i, d in enumerate(digits): ← Scan left to right; i is current position, d is current digit char
for k in range(9, int(d), -1): ← Try digits 9,8,…,d+1 — always try largest first
if k in last and last[k] > i: ← k exists in number AND is strictly to the right of position i
digits[i], digits[last[k]] = digits[last[k]], digits[i] ← Swap the digits
return int(''.join(digits)) ← One swap done — convert back and return immediately
return num ← No swap improved the number; original is already maximum
⚠️ Why int(d) in the inner loop, not d?

The digits list contains character strings like '2', '7'. The inner loop variable k is an integer (from range(9,...)). Comparing an int to a char '2' would compare different types. We cast int(d) to get the numeric value so range(9, int(d), -1) works correctly. Similarly, the last dict uses integer keys (via the comprehension's int(d)).

Solution 2 — Brute Force O(n²): Try Every Pair

def maximumSwap(num: int) -> int:
digits = list(str(num))
best = num ← Track the best result seen so far
for i in range(len(digits)): ← Try all positions as the first swap partner
for j in range(i+1, len(digits)): ← Try all positions to the right as second partner
digits[i], digits[j] = digits[j], digits[i] ← Swap
best = max(best, int(''.join(digits))) ← Compare with best
digits[i], digits[j] = digits[j], digits[i] ← Undo the swap
return best
💡 Complexity Summary

Greedy: Time O(n) — one pass to build last[], one pass with inner loop bounded by 10 digit values → O(10n) = O(n). Space O(1) — digit array and last[] are bounded by the number of digits (≤9 for 108).

Brute Force: Time O(n²). Space O(n). For n ≤ 9, both are effectively constant — but the greedy demonstrates algorithmic thinking.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Spot the Greedy Signal

"Maximize with at most one operation" — this is a classic greedy setup. We don't need to try every option; there's a single locally-optimal choice that's also globally optimal: bring the largest possible digit to the leftmost improvable position.

Step 2: Propose Brute Force First

"I can try all O(n²) pairs, swap, and track the max. For n ≤ 9 digits this is fine, but let me optimize. O(n²) time, O(n) space."

Step 3: Derive the Greedy

"Scan left to right. At each position, ask: is there a larger digit somewhere to my right? If yes, swap with the rightmost occurrence of the largest such digit — that maximizes the position I'm fixing and leaves later digits as favorable as possible."

Step 4: Optimize the Lookup

"To avoid re-scanning right on every position, precompute last[d] = rightmost index of digit d. One pass. Then each position check is O(10) = O(1). Total: O(n)."

🗣️ What to Say to the Interviewer

"This is a greedy problem. I want the largest digit as far left as possible. I'll precompute
last[d] = rightmost index where digit d appears, using a dict comprehension — later entries
naturally overwrite earlier ones, giving me the rightmost index for free.
Then I scan left to right. At each position i, I try digits 9, 8, ..., down to digits[i]+1.
The first k where last[k] > i tells me I can bring a larger digit to position i. I swap and return.
Time O(n), Space O(1) since there are at most 10 distinct digits."
🔮 Likely Follow-up Questions
  • "What if you can do k swaps?" → Repeat the greedy k times: each time bring the global max to the leftmost improvable position (like k rounds of selection sort)
  • "What if you want to minimize instead?" → Reverse greedy: bring the smallest digit leftward, being careful not to lead with 0
  • "What if the input is a string representing a huge number?" → Same algorithm works — we're already treating digits as characters internally
  • "Prove your greedy is optimal." → Exchange argument: any other swap either fixes a less significant position (worse) or uses a smaller digit at the same position (worse)

7 Summary & Key Takeaways ▾
Greedily bring the largest available digit to the leftmost improvable position
Use rightmost occurrence of each digit — the dict comprehension achieves this for free
Try candidate digits from 9 downward to always pick the best possible improvement
One beneficial swap is always optimal — stop and return immediately after performing it
If the scan ends without swapping, the digits are in non-increasing order: already maximal
