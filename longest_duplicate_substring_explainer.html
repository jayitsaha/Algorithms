LeetCode #1044

Longest Duplicate Substring

Find the longest substring that appears at least twice in a given string s.

Hard String Processing Binary Search Rabin-Karp Rolling Hash Time: O(n log n) Space: O(n)
1 Understanding the Problem ▾

Given a string s, find the longest substring that appears at least twice. The two occurrences may overlap.

Example: s = "banana"

"an" appears at index 1 and 3 ✓
"ana" appears at index 1 and 3 ✓
"anan" appears at index 1 only ✗
Answer: "ana" (length 3)

Why is this Hard?

n up to 30,000
Brute force: O(n³) — TLE
Need: O(n log n) approach
Key challenge: Compare all substrings fast

💡 Key Insight: The Monotonic Property

If a duplicate substring of length k exists, then a duplicate of length k−1 also exists (just take any k-1 prefix of that substring). Conversely, if no duplicate of length k exists, no duplicate of length k+1 can exist. This monotone binary structure — "all shorter lengths are feasible, all longer lengths are not" — is the trigger for binary search on the answer.

Problem reformulation: Binary search on length L. For a candidate length L, check: "Does any substring of length L appear more than once?" If yes, L is feasible — try longer. If no, L is too large — try shorter. The check runs in O(n) using Rabin-Karp rolling hash.

Edge cases: All characters identical (e.g., "aaa") → answer is s[:-1]. All distinct characters → answer is "". Single character → "". Overlapping duplicates are allowed.

2 The Algorithm: Binary Search + Rabin-Karp Rolling Hash ▾

We combine two powerful ideas: binary search on the answer (the length) and Rabin-Karp rolling hash (fast substring comparison). Together they bring the solution from O(n³) to O(n log n).

1 Binary search on length L in range [1, n−1]. Invariant: if a duplicate of length L exists, try L+1. If not, try L−1. Converge on the maximum feasible L. This takes O(log n) rounds.
2 Feasibility check via rolling hash: Slide a window of length L over s. Compute a polynomial hash for each window in O(1) per step (one addition and one removal). Store all seen hashes in a set. A collision means a duplicate exists.
3 Track the best result: When the feasibility check finds a match at positions i and j, record s[i:i+L] as the current best answer. After binary search converges, return the best recorded substring.
4 Handle hash collisions: When two hashes match, verify with actual string comparison s[i:i+L] == s[j:j+L]. In practice with large primes, collisions are near-impossible, but comparison makes the solution provably correct.

🔑 Rolling Hash Formula

For a window [i, i+L), the hash is: H = (s[i]·BASE^(L-1) + s[i+1]·BASE^(L-2) + ... + s[i+L-1]) mod MOD

Rolling update from position i to i+1: H_new = (H_old × BASE − s[i] × BASE^L + s[i+L]) mod MOD

This computes each new hash in O(1) — the key insight that makes the check O(n).

3 Why It Works: The Invariant ▾
🔐 Binary Search Invariant

At all times during binary search: lo ≤ answer ≤ hi. We maintain: any length ≤ current best is known feasible; any length > hi is known infeasible. When lo > hi, the answer has been found.

Why the monotone property holds: If "abc" appears twice, then "ab" also appears twice (at the same positions). So if length k is feasible, all lengths 1..k are feasible. This makes the search space a clean binary partition — exactly the condition for binary search to work.

🔐 Rolling Hash Invariant

At every window position i, the current hash value represents the polynomial fingerprint of s[i:i+L]. Two windows with identical hashes are the same string (with overwhelming probability). The seen-set tracks all fingerprints encountered so far — a hit means we found a duplicate at an earlier position.

Why rolling hash is correct: The polynomial hash function is a linear transformation over the string characters. When we slide the window right by one position, we: (1) multiply the entire sum by BASE (shifting all characters left), (2) subtract the contribution of the character that left the window, (3) add the new character. The result is identical to computing the hash of the new window from scratch — but in O(1) instead of O(L).

⚠️ Hash Collisions

Two different strings can produce the same hash (a collision). Use a large prime MOD (e.g., 2^61−1) and verify with actual string comparison on collision. Alternatively, use double hashing (two different MOD values) to reduce false-positive probability to near zero.

Edge case — lo=0: When lo stays 0 throughout (no feasible length found at all), the answer is "" — no duplicate substring exists.

3b 🔬 Algorithm Deep-Dive: Binary Search + Rabin-Karp ▾
Binary Search on Answer

Origin: Classical technique. Any time you need to find the optimal value and the feasibility function is monotone (true for all values below a threshold, false above), binary search on the value is applicable.

Core invariant: lo is always a feasible length (or 0), hi is always an infeasible length (or n). Every iteration halves the search space. After O(log n) rounds, lo and hi converge.

When to recognize it: "Find the maximum/minimum X such that condition(X) holds" where condition is monotone. Typical signals: "minimum time", "maximum length", "smallest k".

Generalization: Binary search on answer can be combined with ANY O(n) feasibility check. Here it's rolling hash; elsewhere it could be BFS, two-pointer, greedy, etc.

Rabin-Karp Rolling Hash

Origin: Michael O. Rabin and Richard M. Karp, 1987. Originally designed for multi-pattern string search, but the rolling hash mechanism is universally applicable to any fixed-window substring comparison.

Core invariant: For any window [i, i+L), H(i) = Σ s[i+k] × BASE^(L-1-k) mod MOD for k in [0, L). The rolling update H(i+1) = (H(i) × BASE − s[i] × BASE^L + s[i+L]) mod MOD preserves this invariant in O(1).

Why it works: Polynomial hashing is a homomorphism — the hash of a concatenation and the hash of its parts are related by simple arithmetic. Sliding the window = one multiply + one add + one subtract. No re-traversal of the window is needed.

When to recognize it: "Find repeated/matching substrings of fixed length." "Does pattern P appear in text T?" "Are two windows of the same string identical?" Any problem requiring O(1) window-to-window comparison.

Generalization: 2D Rabin-Karp for matrix pattern matching. Multi-hash for near-zero collision probability. Z-algorithm and KMP solve similar problems with different tradeoffs (exact, no collisions, but O(n+m) per query vs. O(n) amortized for Rabin-Karp).

# Rolling Hash Template
MOD = (1 << 61) - 1 ← Mersenne prime — large, fast modular arithmetic
BASE = 31 ← Larger than alphabet size (26 for a-z)
h = 0 ← Initial hash for empty window
power = 1 ← BASE^L mod MOD (precompute for the remove step)
for i in range(L):
h = (h * BASE + ord(s[i])) % MOD ← Build initial window hash
power = power * BASE % MOD ← power = BASE^L
seen = {h: 0} ← Map hash → start index for O(1) lookup
for i in range(1, n - L + 1):
h = (h * BASE - ord(s[i-1]) * power + ord(s[i+L-1])) % MOD
if h in seen: ← Hash collision → potential duplicate
if s[seen[h]:seen[h]+L] == s[i:i+L]: return s[i:i+L]
seen[h] = i
4 The Code (Annotated) ▾

Solution 1 — Binary Search + Rabin-Karp (Optimal, O(n log n)) — Interview Pick

def longestDupSubstring(s: str) -> str:
n = len(s) ← Input length
MOD = (1 << 61) - 1 ← 2^61 - 1: large Mersenne prime for low collision probability
BASE = 131 ← Base > 128 (all ASCII values) ensures unique char encoding
nums = [ord(c) for c in s] ← Convert chars to numbers once (reuse across all hash calls)
def check(L):
# Build initial hash for window s[0:L]
h, power = 0, 1
for i in range(L):
h = (h * BASE + nums[i]) % MOD ← Horner's method for polynomial evaluation
power = power * BASE % MOD ← power = BASE^L after loop
seen = {h: 0} ← Hash → start index; 0 is the start of the first window
for i in range(1, n - L + 1):
# Roll: remove s[i-1], add s[i+L-1]
h = (h * BASE - nums[i-1] * power + nums[i+L-1]) % MOD
if h in seen:
j = seen[h] ← j is the earlier occurrence's start index
if s[j:j+L] == s[i:i+L]: ← Confirm: not a hash collision
return s[i:i+L] ← Return the actual duplicate substring
seen[h] = i
return "" ← No duplicate of length L found
lo, hi = 1, n - 1 ← Min possible duplicate length=1; max=n-1
ans = ""
while lo <= hi:
mid = (lo + hi) // 2
result = check(mid)
if result: ← Feasible: a duplicate of length mid exists
ans = result ← Update best answer
lo = mid + 1 ← Try to find a longer duplicate
else:
hi = mid - 1 ← No duplicate of this length; try shorter
return ans ← Best duplicate substring found
⚠️ Common Pitfall: Negative Hash Values

The rolling step h * BASE - nums[i-1] * power can go negative in Python due to Python's integer semantics. Always apply % MOD after the entire expression, and in Python specifically, % MOD always returns a non-negative result (Python's modulo preserves sign of divisor, not dividend). In C++/Java you'd need an extra + MOD before the final % MOD.

Solution 2 — Brute Force (O(n³) — for conceptual clarity only)

def longestDupSubstring_brute(s: str) -> str:
n = len(s)
ans = ""
for length in range(n, 0, -1): ← Try from longest to shortest
seen = set()
for i in range(n - length + 1): ← O(n) windows per length
sub = s[i:i+length] ← O(L) string slice — the O(n³) culprit
if sub in seen: return sub ← First match is the longest duplicate
seen.add(sub)
return ""
💡 Complexity Summary

Brute Force: Time O(n³) — n lengths × n windows × O(n) hash of string slice. TLE on n=30,000.
Binary Search + Rolling Hash: Time O(n log n) — O(log n) binary search rounds × O(n) rolling hash check. Space O(n) for the seen-set.
Suffix Array (advanced): Time O(n log n) or O(n) with SA-IS. Space O(n). More complex to implement — not expected in interviews.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Brute Force and Its Bottleneck

"The brute force tries all O(n²) substrings and stores them in a set. Checking membership is O(L) for string hashing. Total: O(n³). For n=30,000 that's 27 billion operations — won't work. The bottleneck is the number of substrings we check."

Step 2: Spot the Monotone Property → Binary Search on Length

"I notice: if a duplicate of length k exists, then length k−1 also works. This monotone structure means I can binary search on the length. Instead of checking ALL lengths, I do O(log n) rounds. Each round I just need to check: 'does any window of length mid appear twice?'"

Step 3: Recognize the Feasibility Check Bottleneck → Rolling Hash

"The feasibility check — finding if any substring of length L repeats — requires comparing all n windows. Naively that's O(n·L). To make it O(n), I use Rabin-Karp rolling hash: compute the first window's hash in O(L), then slide in O(1) per step. Total check: O(n). Total solution: O(n log n)."

Step 4: Address Hash Collision Risk

"Use a large Mersenne prime like 2^61−1. When hashes collide, confirm with actual string comparison. Alternatively, use double hashing with two independent (BASE, MOD) pairs — reduces false positive probability to ~1/MOD² ≈ 10^−37."

🗣️ What to Say to the Interviewer

"I see the monotone property: if length k has a duplicate, length k-1 does too.
That means I can binary search on the length — O(log n) rounds.
For each candidate length mid, I need to check in O(n) if any window repeats.
I'll use Rabin-Karp rolling hash: O(L) to build the first hash, then O(1) per slide.
Store all hashes in a set. A collision means a duplicate — confirm with string equality.
Total: O(n log n) time, O(n) space. BASE=131, MOD=2^61−1 for collision resistance."
🔮 Likely Follow-up Questions
  • "What if we want ALL duplicate substrings?" → Run check for every length; collect all matches. O(n² log n) — suffix array is better.
  • "What if characters are Unicode (not just lowercase a-z)?" → Use ord(c) directly; just choose BASE > max Unicode code point.
  • "Can you solve it with suffix arrays?" → Build SA + LCP array in O(n log n); answer is max LCP value. More code complexity, same complexity.
  • "How do you handle hash collisions provably?" → Double hashing or confirm with string comparison (shown in code).
