LeetCode #593

Valid Square

Given four 2D points, return true if they form a valid square.

Medium Mathematics & Geometry Check Equal Sides and Diagonals Time: O(1) Space: O(1)
1 Understanding the Problem ▾

You are given four 2D points — p1, p2, p3, p4 — each as an [x, y] pair. The points can arrive in any order (not necessarily corner-by-corner). Return true if and only if they form a valid square.

💡 What Defines a Square?

A square has exactly two geometric properties:
1. All four sides are equal in length (and non-zero — no degenerate shapes).
2. Both diagonals are equal in length.

Property 1 alone describes a rhombus. Property 2 alone describes a rectangle. Both together uniquely define a square.

The tricky part: the four points can label the corners in any sequence. So we cannot assume p1→p2→p3→p4 traces the perimeter. We need an approach that works regardless of input ordering.

✓ Valid Square

p1=[0,0], p2=[1,1]
p3=[1,0], p4=[0,1]
4 sides² = 1, diags² = 2
→ True

✗ Rhombus — not a square

p1=[0,0], p2=[2,0]
p3=[3,2], p4=[1,2]
4 equal sides
But diags differ → False

Edge cases: All four points identical (zero area → False). Two points coinciding. Three collinear points. Input coordinates can be negative.

2 The Algorithm: Sort 6 Pairwise Distances ▾

With 4 points, there are C(4,2) = 6 pairwise distances. In a square, these 6 distances split into exactly two groups: the 4 equal shorter distances (sides) and 2 equal longer distances (diagonals). Sorting the 6 distances reveals this pattern directly.

🔑 Why Squared Distances?

We compute dx² + dy² instead of sqrt(dx²+dy²). Since we only compare distances for equality, any monotonic transformation preserves the ordering. Squaring avoids floating-point rounding errors entirely — all comparisons are between exact integers.

1 Compute all 6 pairwise squared distances between (p1,p2), (p1,p3), (p1,p4), (p2,p3), (p2,p4), (p3,p4). These cover every possible pair.
2 Sort the 6 distances ascending. For a square: the 4 shorter distances are the sides (sorted[0..3]), the 2 longer distances are the diagonals (sorted[4..5]).
3 Verify three conditions: (a) sorted[0] > 0 — no coincident points; (b) sorted[0] == sorted[1] == sorted[2] == sorted[3] — all sides equal; (c) sorted[4] == sorted[5] — both diagonals equal.

Why this works for any input ordering: No matter how the four square corners are labelled p1–p4, the set of 6 pairwise distances is the same. The sorted pattern [s,s,s,s,d,d] is an invariant of the point set, independent of ordering. By Pythagoras, for a square of side length s, the diagonal squared = 2s², so sorted[4] = 2 × sorted[0].

⚠️ Why Not Just Check 2 Distinct Values?

You might think: "check if there are exactly 2 distinct distance values." This fails for a degenerate case: 3 collinear equidistant points + 1 middle point can produce exactly 2 distinct values that are NOT a square. The sort-and-index approach is more robust — it explicitly verifies which positions the equal values occupy.

3 Why It Works: The Invariant ▾
🔐 The Invariant

For four points forming a square with side² = s:
— Exactly 4 of the 6 pairwise distances² equal s (the four sides).
— Exactly 2 of the 6 pairwise distances² equal 2s (the two diagonals, by Pythagoras).
— All distances² are positive (non-degenerate).

After sorting: [s, s, s, s, 2s, 2s]. This pattern is both necessary and sufficient for a square.

Proof of sufficiency (why these two conditions fully characterise a square and not other shapes):

  • A rhombus has 4 equal sides but unequal diagonals → sorted[4] ≠ sorted[5] → rejected.
  • A rectangle has 2 pairs of equal sides and equal diagonals → sorted[0] ≠ sorted[3] → rejected.
  • A general quadrilateral fails both conditions.
  • The degenerate case (all same point) produces all-zeros → sorted[0] = 0 → rejected.
⚠️ The Zero-Distance Trap

If two or more input points are identical, their pairwise squared distance is 0. Without the sorted[0] > 0 guard, four coincident points would pass both other conditions (trivially: 0==0==0==0 and 0==0). Always check for degeneracy first.

Optional cross-check (Pythagoras verification): For a valid square, sorted[4] == 2 * sorted[0] must hold. Our condition already implies this: if sorted[0..3] are all equal to s and sorted[4,5] are equal to each other, and these form a non-degenerate convex quadrilateral, then the diagonal relationship follows geometrically. But you can add the explicit check as an extra guard if desired.

4 The Code (Annotated) ▾

Solution 1 — Sort 6 Distances (Optimal, O(1) — Interview Pick)

def validSquare(p1, p2, p3, p4):
def d2(a, b): ← squared Euclidean distance, avoids sqrt
return (a[0]-b[0])**2 + (a[1]-b[1])**2
dists = sorted([ ← compute all C(4,2)=6 pairwise distances
d2(p1,p2), d2(p1,p3), d2(p1,p4),
d2(p2,p3), d2(p2,p4), d2(p3,p4) ← sort: sides come first, diagonals last
])
return (dists[0] > 0 ← guard: no two points coincide
and dists[0] == dists[1] ← all 4 sides equal (4-way check)
and dists[1] == dists[2]
and dists[2] == dists[3]
and dists[4] == dists[5]) ← both diagonals equal
⚠️ Why Not Check dists[3] == dists[4]?

In a valid square, dists[3] (the longest side) is strictly shorter than dists[4] (the shortest diagonal). Checking dists[3] == dists[4] would require side length to equal diagonal length — impossible in a non-degenerate square. The two groups (sides vs diagonals) must remain separate.

Solution 2 — Try All 24 Orderings (Brute Force)

from itertools import permutations
def validSquare(p1, p2, p3, p4):
def d2(a, b):
return (a[0]-b[0])**2 + (a[1]-b[1])**2
def is_square(a, b, c, d): ← treat a,b,c,d as consecutive corners
return (d2(a,b)==d2(b,c)==d2(c,d)==d2(d,a) ← 4 equal sides
and d2(a,c)==d2(b,d) ← 2 equal diagonals
and d2(a,b) > 0) ← non-degenerate
return any(is_square(*p) for p in permutations([p1,p2,p3,p4]))
💡 Complexity Summary

Sort 6 distances: Time O(6 log 6) = O(1), Space O(6) = O(1). Preferred: clean and minimal.
Brute force permutations: Time O(24) = O(1), Space O(1). More lines but very explicit.
Both are constant time and space — with exactly 4 input points, all work is bounded.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Formally Define a Square

"Before writing code, let me define a square geometrically: 4 equal non-zero sides AND 2 equal diagonals. A rhombus has equal sides but unequal diagonals. A rectangle has equal diagonals but unequal sides. Both conditions together are sufficient."

Step 2: Handle Arbitrary Point Ordering

"The four points can label corners in any sequence, so I can't hardcode 'p1-p2 is a side.' Instead, I'll compute all C(4,2)=6 pairwise distances. These 6 numbers capture the entire geometry of the quadrilateral regardless of labelling."

Step 3: Use Squared Distances

"I'll use squared distances (dx²+dy²) to keep everything in integer arithmetic. No sqrt, no floating-point errors. Monotone transformation preserves all ordering relationships."

Step 4: Sort and Verify Pattern

"Sort the 6 distances. A square always gives pattern [s,s,s,s,d,d] where d=2s (Pythagoras). Three checks: sorted[0]>0 (non-degenerate), sorted[0..3] all equal (4 sides), sorted[4]==sorted[5] (2 diagonals)."

🗣️ What to Say to the Interviewer

"A square requires 4 equal sides AND 2 equal diagonals — both conditions needed.
Since the points can come in any order, I'll compute all 6 pairwise squared distances.
After sorting, the 4 smaller ones are sides, the 2 larger are diagonals — for a valid square.
I check: smallest distance > 0 (no coincident points), first 4 equal (equal sides),
last 2 equal (equal diagonals). O(1) time and space since we always have exactly 4 points."
