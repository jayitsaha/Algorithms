LeetCode #234

Palindrome Linked List

Given the head of a singly linked list, return true if it is a palindrome or false otherwise — in O(n) time and O(1) extra space.

Easy Linked List Fast-Slow Pointer Reversal Time: O(n) Space: O(1)
1 Understanding the Problem ▾

A palindrome reads the same forwards and backwards: "racecar", [1, 2, 1], [1, 2, 2, 1]. For a linked list this means the value at position i from the start must equal the value at position i from the end.

The challenge: unlike arrays, linked lists don't support random access. You can't jump to the middle or the end in O(1). And the O(1) space constraint rules out simply copying values into an array for comparison. You have to be clever about using the structure of the list itself.

💡 Key Insight

A palindrome's first half equals its second half reversed. Find the middle of the list, reverse the second half in-place, then do a simple head-to-head value comparison. No extra memory needed — only pointer variables.

✓ Palindrome: [1, 2, 3, 2, 1]

First half: [1, 2]
Second half: [2, 1] → reversed: [1, 2]
1==1, 2==2 → True

✗ Not Palindrome: [1, 2, 3, 1]

First half: [1, 2]
Second half: [3, 1] → reversed: [1, 3]
1==1, 2≠3 → False

Edge cases: single-node list (always palindrome), two-node list ([1,1] → True, [1,2] → False), odd vs even length (for odd, the center node need not be compared since it mirrors itself).

2 The Algorithm: Find Middle → Reverse Second Half → Compare ▾

The optimal solution combines two classic linked list techniques in sequence: the fast-slow (tortoise-hare) pointer to locate the midpoint in one pass, then an in-place reversal of the second half, then a linear comparison walk.

1 Find middle: slow = fast = head. While fast and fast.next exist, advance slow by 1, fast by 2. Loop ends when fast cannot take 2 more steps — slow is at the midpoint (end of the first half for even-length; true center for odd-length).
2 Reverse second half: prev = None, curr = slow.next (skip center for odd). For each node: save nxt = curr.next, flip curr.next = prev, advance prev = curr and curr = nxt. After the loop, prev is the new head of the reversed second half.
3 Compare: left = head, right = prev. Walk both forward. If left.val != right.val at any point, return False. Stop when right is None (reversed half exhausted). Return True — all pairs matched.
🔑 Why O(1) Space is Possible

The brute-force approach copies values into a list and checks if it equals its reverse — O(n) space. The in-place reversal avoids this: we modify the pointers of the existing nodes, not their values, using only a handful of pointer variables. The list structure itself becomes our "reversed copy."

3 Why It Works: The Invariant ▾
🔐 Fast-Slow Pointer Invariant

At every step, fast has traveled exactly twice the distance of slow. When fast can't advance 2 more steps, slow must be at the halfway point. For a list of n nodes: if n is even, slow ends at node n/2 − 1 (last of first half); if n is odd, slow ends at the true center node ⌊n/2⌋.

Reversal correctness: The three-pointer reversal maintains a precise invariant — at any point during the loop, prev is the already-reversed suffix (a valid reversed sub-list), and curr is the remaining un-reversed prefix. When curr becomes None, the entire suffix is reversed and prev is its new head.

Why start from slow.next not slow: For an odd-length list like [1, 2, 3, 2, 1], the node with value 3 is the center. It mirrors itself — no comparison needed. By starting reversal at slow.next we correctly skip the center, reversing [2, 1] into [1, 2] and comparing that against [1, 2] from the head. For even-length lists, slow is already the last node of the first half, so slow.next is the correct start for the second half.

Why stop when right is None: The reversed second half has length ≤ first half (equal for even, 1 shorter for odd). Stopping when right exhausts naturally handles both cases without any length tracking.

⚠️ The List is Mutated After This Algorithm

After comparing, the second half of the list is still reversed. In production code (or if asked in an interview) you should restore the original structure: reverse the second half again before returning. For this LeetCode problem it's not tested, but mentioning it shows professional awareness. Code: run the same reversal again on prev.

4 The Code (Annotated) ▾

Solution 1 — Reverse Second Half (Optimal, O(n) time, O(1) space) — Interview Pick

def isPalindrome(head: ListNode) -> bool:
slow, fast = head, head ← both start at head for the midpoint search
while fast and fast.next: ← fast needs 2 hops; check both exist
slow = slow.next ← slow advances 1 step
fast = fast.next.next ← fast advances 2 steps
prev, curr = None, slow.next ← skip center node; start reversal at slow.next
while curr: ← reverse the second half in-place
nxt = curr.next ← save next before we overwrite the pointer
curr.next = prev ← flip: current node now points backward
prev = curr ← prev advances to current node
curr = nxt ← curr advances to saved next
left, right = head, prev ← left walks first half, right walks reversed second
while right: ← reversed half drives termination (may be shorter)
if left.val != right.val: ← any mismatch = not palindrome
return False
left = left.next ← advance both simultaneously
right = right.next
return True ← all pairs matched → palindrome!
⚠️ Why curr = slow.next not curr = slow?

For odd-length lists, slow lands on the true center node (e.g., value 3 in [1,2,3,2,1]). This center doesn't need to be compared — it always matches itself. Starting reversal from slow.next skips it. If you instead used curr = slow, you'd include the center in the reversed half, causing right to be 1 element longer than expected, but the algorithm would still return correct results (the extra comparison would be center vs center, which always matches). So it's correct either way — starting at slow.next is just cleaner.

Solution 2 — Array Conversion (Simple, O(n) space)

def isPalindrome(head: ListNode) -> bool:
vals = [] ← collect all node values into a list
while head: ← traverse the entire linked list
vals.append(head.val) ← append each value in order
head = head.next
return vals == vals[::-1] ← Python palindrome check: forward equals reversed
💡 Complexity Summary

Reverse Second Half: Time O(n) — 3 linear passes. Space O(1) — only pointer variables.
Array Conversion: Time O(n). Space O(n) — copy of all node values.
For interviews: propose Solution 2 first as the "obvious" approach, then offer Solution 1 as the O(1) space optimization.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Notice the O(1) Space Constraint

The problem says "singly linked list" and the follow-up asks for O(1) space. Array conversion is obvious but uses O(n) space. This immediately signals: we need to work in-place. Ask: "Is the O(1) space constraint required?" — it usually is for this problem.

Step 2: Propose the Brute Force

"The simple approach: walk the list, collect values into a Python list, return vals == vals[::-1]. O(n) time, O(n) space. Simple, correct, but uses extra memory." Write this up quickly, then say "Can I optimize to O(1) space?"

Step 3: State the Key Observation

"A palindrome's first half equals its second half reversed. If I can reverse the second half in-place — without extra memory — I can compare the two halves directly." Then break it into 3 phases: find middle, reverse, compare.

Step 4: Explain Each Phase Clearly

Be explicit: "I'll use the fast-slow pointer pattern to find the middle in O(n). Then the standard three-pointer reversal — prev, curr, nxt — to flip the second half. Then two pointers walking both halves forward." Draw this on a whiteboard if possible.

Step 5: Mention Restoration

"One caveat: this mutates the list. In production I'd reverse the second half again after comparing, to restore the original structure. For this problem it's not required, but it's good practice." This shows maturity beyond just "make tests pass."

🗣️ What to Say to the Interviewer

"My first approach: copy all values to an array and check if it equals its reverse. O(n) time, O(n) space.
For O(1) space: a palindrome's first half equals its reversed second half.
Phase 1: find the midpoint using fast-slow pointers — when fast can't move 2 steps, slow is at the middle.
Phase 2: reverse the second half in-place with the three-pointer trick (prev/curr/nxt).
Phase 3: walk left from head, right from the reversed second-half head — mismatch returns False, else True.
Total: O(n) time (3 linear passes), O(1) space (just pointer variables)."
🔮 Likely Follow-up Questions
  • "What if we can't modify the list?" → Use a stack: push first half while finding middle, then compare stack pops against second half.
  • "How would you restore the original list?" → Reverse the second half again after comparison (same code, applied to prev).
  • "What about a doubly linked list?" → Two pointers: one at head, one at tail; walk inward comparing. O(n) time, O(1) space.
  • "Can you do it recursively?" → Yes, with a nonlocal/outer pointer advancing from head while recursion unwinds from tail — O(n) space though.

7 Summary & Key Takeaways ▾
Palindrome linked list = first half equals reversed second half
Fast-slow (tortoise-hare) pointer finds the midpoint in O(n) with O(1) space
Three-pointer reversal (prev/curr/nxt) is the fundamental in-place linked list reversal pattern
For odd-length lists start reversal at slow.next — skip the center node (it mirrors itself)
Stop comparing when the reversed half (right) is exhausted — handles odd/even length cleanly
