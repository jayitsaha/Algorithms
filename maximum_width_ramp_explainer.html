LeetCode #962

Maximum Width Ramp

Given an integer array nums, a ramp is a pair (i, j) where i < j and nums[i] ≤ nums[j]. Return the maximum width j−i of such a ramp, or 0 if none exists.

Medium Monotonic Stack Decreasing Stack + Right Scan Time: O(n) Space: O(n)
1 Understanding the Problem ▾

A ramp is a pair of indices (i, j) where i is to the left, j is to the right, and the value at i does not exceed the value at j: nums[i] ≤ nums[j]. We want the maximum possible distance j−i.

Think of it physically: standing on a hillside, you can slide from position i down (or across) to position j as long as j's elevation is at least as high as i's. You want the widest possible slide.

Example: [6, 0, 8, 2, 1, 5]

Best ramp: i=1 (val=0), j=5 (val=5). Width = 5−1 = 4. We need 0 ≤ 5 → ✓

Why Not Check All Pairs?

O(n²) pairs. For n=50,000 that's ~2.5B comparisons. We need something smarter.

💡 Key Insight

Only "new minimum" left endpoints matter. If index a comes before index b and nums[a] ≤ nums[b], then a dominates b as a left endpoint: a is further left and its value threshold is lower, so any j that pairs with b also pairs with a at a wider width. We only need to keep indices where the value is a strict new minimum from the left.

This leads to a clean two-phase algorithm: Phase 1 collects candidate left endpoints into a decreasing-value stack. Phase 2 scans j from right to left and greedily pops the stack to measure ramp widths.

2 The Algorithm: Two-Phase Monotonic Stack ▾
Phase 1: Build Stack
Scan i left→right
Push if new minimum
Stack values strictly decrease
Phase 2: Pop & Record
Scan j right→left
Pop stack while nums[top] ≤ nums[j]
Record j−popped as candidate width

The key principle: a candidate left endpoint is dominated if a strictly earlier index has a smaller or equal value. The dominated one can never produce a wider ramp. So we only keep a monotonically decreasing sequence of values in our stack (stored as indices).

1 Phase 1 — Build decreasing stack: Scan left→right. For each index i, push it only if the stack is empty OR nums[i] is strictly less than nums[stack.top]. At the end, stack contains all "local left minima" with strictly decreasing values.
2 Phase 2 — Greedy right-to-left scan: Scan j from n−1 down to 0. While the stack is non-empty and nums[stack.top] ≤ nums[j], pop the top, compute width = j − popped_index, update max_width.
3 Why scan j right→left? For any fixed left index i in the stack, a larger j always gives a wider ramp. Scanning right-to-left means the first time we pop i (with some j), that j is the rightmost valid j for i — guaranteed maximum width for that i. We never need to revisit i.
4 Return max_width. If the stack was never matched, no valid ramp exists — return 0.
🔐 Why Only Decreasing Values in the Stack?

Suppose the stack has indices a < b and nums[a] ≥ nums[b]. For any j where nums[b] ≤ nums[j], we also have nums[a] ≤ nums[j]. The ramp (a, j) has width j−a > j−b. So b is dominated — it can never produce a better answer. Pruning dominated candidates keeps the stack strictly decreasing.

3 Why It Works: The Invariants ▾
🔐 Invariant 1 (Stack after Phase 1)

Stack contains exactly those indices i where nums[i] is a new left-to-right minimum. Values are strictly decreasing. The optimal left endpoint i* is guaranteed to be in this stack.

🔐 Invariant 2 (Phase 2 Pops)

When we pop index i at position j, j is the rightmost index seen so far (since we scan right→left) where nums[i] ≤ nums[j]. Therefore j−i is the maximum width achievable with left endpoint i. Once popped, i is permanently retired.

Proof of completeness: Suppose the optimal pair is (i*, j*). If i* is not in the stack, some earlier index a < i* with nums[a] ≤ nums[i*] was pushed first. Since nums[a] ≤ nums[i*] ≤ nums[j*], (a, j*) is a valid ramp with width j*−a > j*−i*. This contradicts optimality of i*, so i* must be in the stack.

O(n) time proof: Each index is pushed at most once (Phase 1) and popped at most once (Phase 2). Total operations = O(n).

⚠ Common Mistake: Stopping After First Pop

In Phase 2, after popping one index, keep checking the new stack top. The next entry might also satisfy the condition with the current j, giving an even wider ramp (it has a smaller index = further left). Always continue the while loop until the condition fails or the stack is empty.

Edge cases: All decreasing (e.g., [5,4,3,2,1]) → Phase 2 never pops, return 0. All equal (e.g., [3,3,3,3]) → index 0 is in stack; Phase 2 pops at j=n−1 giving width n−1. Single element → no pair possible, return 0.

4 The Code (Annotated) ▾

Optimal solution — O(n) time, O(n) space:

def maxWidthRamp(nums): → function entry
n = len(nums) → store length
stack = [] → holds candidate left-endpoint INDICES
for i in range(n): # Phase 1 → scan left-to-right
if not stack or nums[i] < nums[stack[-1]]: → push only if new minimum (strictly <)
stack.append(i) → record index (for width calculation)
max_width = 0 → initialize result to 0 (no ramp)
for j in range(n - 1, -1, -1): # Phase 2 → scan right-to-left
while stack and nums[stack[-1]] <= nums[j]: → ramp condition: nums[i] ≤ nums[j]
max_width = max(max_width, j - stack.pop())→ pop index, compute width
return max_width → 0 if no ramp found
⚠ Why < in Phase 1 but ≤ in Phase 2?

Phase 1 uses strictly less than (<) because we want a strictly decreasing sequence in the stack. If nums[i] == nums[stack.top], the earlier index already dominates (it's further left at the same value), so we don't push duplicates. Phase 2 uses less-than-or-equal (≤) because the ramp definition allows nums[i] == nums[j] as a valid pair.

Brute force for comparison:

def maxWidthRamp_brute(nums): # O(n^2) TLE
max_w = 0
for i in range(len(nums)):
for j in range(i + 1, len(nums)):
if nums[i] <= nums[j]:
max_w = max(max_w, j - i)
return max_w

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify the Pattern Signal

The problem says "maximize j−i subject to nums[i] ≤ nums[j]." This is a "widest valid pair" problem. The O(n²) brute force is obvious. Ask: can we prune the search space? That's the key question.

Step 2: Find the Domination Argument

Ask: "Are all left endpoints useful?" Consider two candidates a < b. If nums[a] ≤ nums[b], then a dominates b — anything valid for b is valid for a at wider width. So: only keep indices where nums[i] is a new minimum. That's a monotonic decreasing stack.

Step 3: Figure Out How to Use the Stack

Stack built. Now how to match? For each stack entry, we want the rightmost j where the condition holds. Scan j from right to left. First match = widest ramp. Pop and record. This is the classic "decreasing stack + reverse scan" pattern.

Step 4: Verify with Example

nums=[6,0,8,2,1,5]. Phase 1: push 0 (val=6), push 1 (val=0), skip 2,3,4,5. Stack=[0,1]. Phase 2: j=5 (5): pop 1 (0≤5, w=4), check 0 (6>5, stop). j=4,3: 6>1,2, no pop. j=2 (8): pop 0 (6≤8, w=2). max=4. ✓

🗣 What to Say to the Interviewer

"The brute force is O(n squared) -- try all pairs. Can we prune?

Key insight: for left endpoints, some are dominated. If index a is earlier and has a smaller value than index b, any right endpoint that works for b also works for a at greater width. So b is useless.

We only keep a monotonically decreasing sequence of values -- a decreasing stack of indices.

Phase 2: scan j right to left. First time we find nums[stack.top] <= nums[j], that's the maximum width for that left endpoint. Pop it, record the width, continue checking the next stack entry.

Total time: O(n) -- each index pushed/popped at most once."
💡 The Mental Model

Think of the stack as a list of "ski jump launch pads" sorted by height (shortest first, i.e., leftmost index has highest value, rightmost has lowest). We scan the mountain right-to-left looking for landings. The first valid landing for each pad (from the right) is the best landing for that pad. Once landed, the pad is retired.

7 Summary & Key Takeaways ▾
A monotonically decreasing stack captures all "useful" left endpoints — those that are new left-to-right minima.
Dominated candidates (index b where an earlier index a has nums[a] ≤ nums[b]) can never produce a better ramp and are safely skipped.
Scanning j right-to-left in Phase 2 guarantees each pop gives the widest possible ramp for that left endpoint.
Total O(n) time: each index is pushed at most once and popped at most once.
The "build candidates + greedy scan" two-phase pattern recurs in many monotonic stack problems.
🎯 When to Use This Pattern
  • Maximize j−i subject to a value-comparison constraint on nums[i] and nums[j]
  • "Widest", "maximum distance", "span", combined with ≤/≥ value constraint
  • O(n²) brute force is too slow and there's a monotone dominance structure
  • Pair-finding where one endpoint has a "better" version if scanned from one direction

🔗 The Monotonic Stack Mindset

Any time you can argue "if candidate A dominates candidate B, we can discard B permanently," you can build a monotonic stack. The stack maintains exactly the non-dominated candidates. This appears in dozens of problems: span problems, histogram problems, stock price problems, and more.
