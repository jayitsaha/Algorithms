LeetCode #76

Minimum Window Substring

Given strings s and t, return the minimum window in s which contains all characters of t. Return "" if no such window exists.

Hard Sliding Window Expand Until Valid, Shrink to Optimize Time: O(|s|+|t|) Space: O(|t|)
1 Understanding the Problem ▾

You have a long source string s and a short target string t. You must find the shortest contiguous substring of s that contains every character of t — respecting multiplicity. If t = "ABC", any window with at least one A, one B, and one C qualifies. The challenge is minimizing its length.

💡 Key Insight

A window is valid when it covers all character requirements of t. Once a valid window is found, we want the smallest one — so we shrink from the left while it stays valid. This "expand right until valid, shrink left to minimize" rhythm is the heartbeat of this algorithm.

✓ Example: s="ADOBECODEBANC", t="ABC"

"ADOBEC" contains A,B,C → len 6
"DOBECODEBA" → len 10 (also valid)
"BANC" → len 4 ← MINIMUM!
Answer: "BANC"

✗ Edge Cases → ""

s="a", t="aa" → "" (only one 'a')
s="a", t="b" → "" ('b' not in s)
s="", t="A" → "" (empty s)
s="aa", t="a" → "a" (single char)

Multiplicity matters: If t = "AAB", the window must contain at least 2 A's and 1 B. Extra copies of characters in the window are fine — they don't break validity, they just mean we can shrink more. The key question is always: "does this window cover every character of t at least as many times as t requires?"

Edge cases to consider: t contains characters not in s (return ""), |t| > |s| (impossible, return ""), single character strings, all characters in t are the same (need[c] might be > 1).

2 The Algorithm: Sliding Window with formed Counter ▾

We maintain a two-pointer window [left, right] over s and two frequency maps. The elegant trick is a single integer formed that acts as an O(1) validity signal — avoiding re-scanning the entire window map each step.

1 Build the need map: Count each character in t. Store required = len(need) — the number of distinct characters that must each be fully satisfied.
2 Expand right: Move right one step. Add s[right] to window map. If this character is in need AND window[c] == need[c] exactly, increment formed.
3 While valid (formed == required) — shrink from left: Record window size if it's the smallest so far. Remove s[left], advance left. If removing it makes window[c] < need[c], decrement formed and exit the shrink loop.
4 Repeat: Continue expanding right. Every time we reach formed == required again, we try to shrink. At the end, return the recorded minimum window.
🔑 The formed Counter (the clever O(1) trick)

formed counts distinct characters whose window frequency has reached (or exceeded) their need. When formed == required, all characters are covered — no need to iterate through the whole map. This single integer replaces an O(|t|) validity check with O(1).

⚠️ Why == and not >= when updating formed

We increment formed only when window[c] == need[c], not >= need[c]. The equality check fires exactly once per character per "satisfaction crossing" — the precise moment we tip from "not enough" to "enough". Using >= would fire every time we add an excess copy, over-counting.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every step, the window [left, right] is the tightest valid window ending at or before right. Whenever formed == required, we immediately shrink left to its tightest valid position and record the result. No valid window ending anywhere in s can be missed.

Why shrinking left is safe: When valid, shrinking removes s[left]. If s[left] is not in need (or window[left] > need[left], i.e., excess), the window remains valid. Only when we remove the last critical copy of a character does validity break — and we immediately stop and flip back to expanding.

Why no valid window is missed: The algorithm considers every possible right boundary (one per iteration of the for loop). For each right, it greedily minimizes the left boundary. So every (left, right) valid pair where left is optimal for that right is evaluated. The global minimum is therefore found.

Complexity proof: Both left and right are monotonically increasing — they never go backward. Each character is added to the window (right expansion) exactly once and removed at most once (left contraction). Total operations: O(2|s|) = O(|s|).

⚠️ Common Mistake: Checking all need keys after shrink

Beginners sometimes re-check all(window[c] >= need[c] for c in need) after each shrink step. This is O(|t|) per step, giving O(|s|·|t|) overall — no better than brute force. The formed counter exists precisely to avoid this. Trust it: if formed == required, the window is valid.

4 The Code (Annotated) ▾

Solution 1 — Sliding Window (Optimal, O(|s|+|t|)) — Interview Pick

def minWindow(s: str, t: str) -> str:
if not t or not s: return "" ← Guard: empty input → no window possible
need = {} ← Frequency map of chars we need from t
for c in t: need[c] = need.get(c,0)+1 ← e.g., t="ABC" → need={'A':1,'B':1,'C':1}
required = len(need) ← # distinct chars to fully satisfy
window = {}; formed = 0 ← window: current counts; formed: satisfied count
left = 0; best = float('inf'), 0, 0 ← (window_len, left_idx, right_idx)
for right in range(len(s)): ← Expand: right sweeps from 0 to end of s
c = s[right] ← Incoming character at right boundary
window[c] = window.get(c,0) + 1 ← Add it to current window frequency map
if c in need and window[c]==need[c]:← Did we JUST satisfy this char's requirement?
formed += 1 ← Yes: one more distinct char fully covered
while formed == required: ← Window is VALID: shrink from left
if right-left+1 < best[0]: ← Is this the smallest valid window seen?
best = right-left+1,left,right← Store (length, left, right) — not the string
l = s[left]; left += 1 ← Grab leftmost char, advance left boundary
window[l] -= 1 ← Remove it from window counts
if l in need and window[l]<need[l]:← Did removing break a requirement?
formed -= 1 ← Yes: window is no longer valid → exit while
if best[0] == float('inf'): return "" ← No valid window found
return s[best[1] : best[2]+1] ← Reconstruct: single slice using stored indices
⚠️ Why store (length, left, right) instead of slicing each time?

Slicing s[left:right+1] at every valid step is O(window_length) extra work per step. Storing indices is O(1). We do one O(answer_length) slice at the very end. In interviews, mentioning this shows you care about constants, not just big-O.

Solution 2 — Brute Force (O(|s|² · |t|)) — mention first, then optimize

def minWindow_brute(s: str, t: str) -> str:
from collections import Counter
need = Counter(t) ← What we need
best = ""
for i in range(len(s)): ← Try every start index i
for j in range(i, len(s)): ← Try every end index j
win = Counter(s[i:j+1]) ← O(j-i) to count this window
if all(win[c]>=need[c] for c in need):← O(|t|) validity check
if not best or j-i+1 < len(best):
best = s[i:j+1]
return best
💡 Complexity Summary

Brute Force: Time O(|s|² · |t|), Space O(|s| + |t|) — fails for large inputs
Sliding Window: Time O(|s| + |t|), Space O(|t|) — each char visited at most twice
The sliding window's key insight: instead of re-counting windows from scratch, we maintain a running count and update it incrementally.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Pattern

Signal phrases: "minimum subarray/substring containing all [requirements]." Anytime you see "find the shortest contiguous portion of s that satisfies every constraint from t," the answer is a variable-size sliding window. The window expands until valid, then shrinks to minimize length.

Step 2: Define State and Validity

Name your state: (a) need — what's required from t, (b) window — what the current window provides, (c) formed vs required — the scalar validity signal. Explain why you use a scalar counter instead of iterating all keys each step: it gives O(1) validity check.

Step 3: Code the Expand/Shrink Loop

The outer for right loop always expands. The inner while formed == required loop always shrinks. Inside the while: record the minimum, then remove s[left], advance left, update formed. Only one condition breaks the while: when formed drops below required.

Step 4: Handle Return Value

Store best as (length, left_idx, right_idx). Initialize to (float('inf'), 0, 0). At the end, check if best[0] is still infinity (no valid window found → return ""). Otherwise return s[best[1]:best[2]+1].

🗣️ What to Say to the Interviewer

"'Minimum window containing all of t' — this is a classic variable sliding window.
I'll use two pointers: right expands until the window is valid (contains all of t),
then left shrinks while validity holds, recording the minimum each time.
To check validity in O(1), I track formed — the count of distinct chars from t
whose window frequency equals their need. When formed == required, window is valid.
Total: O(|s| + |t|) time, each char processed at most twice (one add, one remove)."
🔮 Likely Follow-up Questions
  • "What if t has repeated characters?" → Already handled — need[c] could be 2 or more; formed fires only at window[c] == need[c]
  • "Can you optimize further for large |s| with small |t|?" → Yes: pre-filter s to only positions containing characters in t, then slide over that filtered list
  • "Find all minimum windows?" → Track all (left, right) pairs where right-left+1 == best_length
  • "Minimum window containing t as a subsequence?" → Different problem (#727); order matters, requires two-pointer scan per right

7 Summary & Key Takeaways ▾
Expand right until window is valid (formed == required), then shrink left to minimize
The formed counter gives O(1) validity check — never scan all keys at each step
Both pointers only move forward — each character processed at most twice → O(|s|)
Store best as (length, left, right) — reconstruct the substring once at the end
formed fires on window[c] == need[c], not ≥ — prevents double-counting
Characters not in t still enter window but never affect formed — safely ignored
ApproachTimeSpaceNotes
Brute Force (all windows) O(|s|² · |t|) O(|s|+|t|) Try every O(|s|²) window, check O(|t|) each
Sliding Window ✓ O(|s|+|t|) O(|t|) Optimal — standard interview solution
Filtered SW (only chars in t) O(|s|+|t|) O(|t|) Same O, better constant when t ≪ s
🎯 When to Recognize This Pattern
  • "Minimum/shortest subarray or substring containing all [conditions]" → Expand until valid, shrink to optimize
  • "Maximum window that satisfies [constraint]" → Expand until invalid, shrink to restore (opposite direction)
  • Character-frequency matching over a window → need map + formed counter
  • "Does window cover all of t?" with t having duplicates → formed tracks distinct-char satisfaction
