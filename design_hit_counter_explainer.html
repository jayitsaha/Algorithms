LeetCode #362

Design Hit Counter

Design a hit counter that counts the number of hits received in the past 5 minutes (300 seconds). Implement hit(timestamp) and getHits(timestamp).

Medium Design Circular Buffer Time: O(1) Space: O(300)
1 Understanding the Problem ▾

Imagine a web analytics system tracking API requests. You need a data structure that answers: "How many requests hit the server in the last 300 seconds?" At any given moment, you want a sliding 5-minute window count.

Two operations must be supported:

  • hit(timestamp) — Record that a hit occurred at this Unix timestamp (integer seconds)
  • getHits(timestamp) — Return the count of all hits where timestamp - 299 ≤ hit_time ≤ timestamp (the past 300 seconds inclusive)
💡 Key Insight

The window is exactly 300 seconds wide. We only need to aggregate hits per-second, and only the last 300 seconds ever matter. Data older than 300 seconds is stale and can be safely discarded — or better, overwritten. This is the classic definition of a circular buffer (ring buffer) of size 300.

✓ Valid: hit(1), hit(2), hit(3) then getHits(4)

Window for getHits(4) = [−295..4]
All 3 hits fall inside → return 3

✗ Stale: hit(1) then getHits(301)

Window for getHits(301) = [2..301]
t=1 is outside (1 ≤ 1) → return 0

Key constraints: Timestamps are monotonically non-decreasing (each call's timestamp ≥ previous call's timestamp). Multiple hits can share the same timestamp. The window is always exactly 300 seconds.

⚠️ Off-by-one on the boundary

The window includes timestamps strictly greater than current - 300. A hit exactly 300 seconds old (at timestamp - 300) is outside the window. Use strict inequality > not >=.

2 The Algorithm: Circular Buffer (Ring Buffer) ▾

We maintain two fixed-size arrays of length 300: times[300] stores the actual timestamp each slot currently represents, and hits[300] stores the hit count for that timestamp. The slot for any timestamp t is simply t % 300.

1 Map timestamp to slot: Compute i = timestamp % 300. Two timestamps 300 seconds apart land on the same slot — which is precisely when the older one becomes stale and should be evicted. The modulo acts as a natural clock hand.
2 hit(t): Check if times[i] == t. If yes, this slot already belongs to this second — increment hits[i]. If no, the slot holds stale data from an older second — overwrite it: set times[i] = t and reset hits[i] = 1.
3 getHits(t): Scan all 300 slots. For each slot i, check if times[i] > t - 300. If so, that slot's second is within the window — add hits[i] to the running total. Return the total after scanning all slots.

slot is reused

🔑 Why Size 300?

The window is exactly 300 seconds. Slot i = t % 300 won't be needed again for another 300 seconds. By then, the old data is outside any valid query window — overwriting is always safe. The modulo is the cleverness: it maps an infinite timeline onto 300 recyclable slots.

💡 getHits is O(1) even though it scans 300 slots

Scanning 300 slots is constant work — it does not grow with the number of hits or the passage of time. Since 300 is a fixed compile-time constant, O(300) = O(1). This is a standard trick in systems design: bound by the window size, not by traffic volume.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At any time, slot i holds valid data if and only if times[i] > current_timestamp - 300. If times[i] represents a second that is 300 or more seconds in the past, the slot is stale: ignore it in queries, overwrite it on the next hit at that slot index.

Why overwriting is always safe (proof sketch):

  • Two timestamps that share slot index i satisfy t2 - t1 = k × 300 for some positive integer k ≥ 1.
  • Therefore t1 ≤ t2 - 300, meaning t1 is at least 300 seconds before t2.
  • For any future query at timestamp q ≥ t2, the window is [q - 299, q]. Since t1 ≤ t2 - 300 ≤ q - 300 < q - 299, the old hit at t1 is outside the window.
  • Conclusion: overwriting is safe — the evicted data would never contribute to any future query.

Why the query sum is correct: Each slot holds the cumulative count for exactly one second. That second is either in-window (contribute) or not (skip). We never double-count since each second maps to exactly one slot, and each slot stores exactly one second's worth of hits.

⚠️ Boundary edge case: hit at t=1, then hit at t=301

Both map to slot 1 (1 % 300 = 1, 301 % 300 = 1). When hit(301) runs: times[1] = 1 ≠ 301, so we overwrite. The old count of 2 at t=1 is erased. This is correct: getHits(301) would compute window [2..301], and t=1 is outside (1 < 2), so the erased data would not have been counted anyway. No loss of accuracy.

4 The Code (Annotated) ▾

Solution 1 — Circular Buffer (Optimal — Interview Pick)

class HitCounter:
def __init__(self): ← Set up 300-slot ring buffer
self.times = [0] * 300 ← Which second each slot records; 0 = unused
self.hits = [0] * 300 ← How many hits at that second
def hit(self, timestamp): ← Record a hit at this timestamp
i = timestamp % 300 ← Map to slot [0..299]
if self.times[i] != timestamp: ← Slot has stale data?
self.times[i] = timestamp ← Claim slot for this second
self.hits[i] = 1 ← Reset count (first hit this second)
else:
self.hits[i] += 1 ← Same second: just increment
def getHits(self, timestamp): ← Count hits in past 300 seconds
total = 0
for i in range(300): ← Scan all slots (O(300) = O(1))
if self.times[i] > timestamp - 300: ← In-window?
total += self.hits[i] ← Yes: add hit count
return total
⚠️ Strict greater-than in the window check

times[i] > timestamp - 300 excludes hits that are exactly 300 seconds old. A hit at timestamp - 300 is outside the 300-second window (the window is [timestamp - 299, timestamp]). This is intentional and correct — don't change it to >=.

Solution 2 — Deque / Queue (Brute Force, O(n) space)

from collections import deque
class HitCounter:
def __init__(self):
self.q = deque() ← Store every individual hit timestamp
def hit(self, timestamp):
self.q.append(timestamp) ← O(1): just push to the right
def getHits(self, timestamp):
while self.q and self.q[0] <= timestamp - 300:
self.q.popleft() ← Evict stale hits (monotone: front is oldest)
return len(self.q) ← Remaining = all in-window hits

The deque is simpler to derive and handles arbitrary window sizes. The circular buffer is the engineering-grade solution: O(1) fixed space regardless of traffic volume, and O(1) query time. In a high-throughput system, the space savings alone justify the complexity.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Clarify the Contract

Ask: "Are timestamps guaranteed to be non-decreasing?" (Yes — this unlocks O(1) eviction.) "Can multiple hits share a timestamp?" (Yes — we need to aggregate.) "Is the window exactly 300 seconds?" (Yes — fixed window enables fixed-size buffer.)

Step 2: Propose the Obvious Solution First

"Naive approach: a queue. Push every hit; on getHits, pop from the front while the front is stale. O(1) hit, O(n) getHits, O(n) space. Correct, but unbounded memory — bad for high-traffic production."

Step 3: Derive the Key Observation

"Since the window is exactly 300 seconds, I never need more than 300 distinct per-second counters. And because timestamps are monotonically increasing, a slot for second t can be safely overwritten at second t + 300 — the old hit is now outside the window."

Step 4: Describe the Circular Buffer

"Two fixed arrays of size 300: times and hits. Slot = t % 300. On hit: if slot is current, increment; else overwrite. On getHits: scan all 300 slots, sum those with times[i] > t - 300. O(1) time, O(1) space."

🗣️ What to Say to the Interviewer

"The window is fixed at 5 minutes = 300 seconds. So I need at most 300 counters — one per second.
I map any timestamp t to slot t mod 300. Two timestamps 300 seconds apart share a slot.
By then the older one is outside any query window, so overwriting is safe — timestamps are monotone.
hit() is O(1): check the slot, reset or increment.
getHits() scans all 300 slots — O(300) = O(1). Sum those with a recent enough timestamp.
Total space: O(300) = O(1), regardless of how many hits come in."
🔮 Likely Follow-up Questions
  • "What if the window was configurable?" → Parameterize: self.n = window_size, arrays of size n. Same logic.
  • "What if timestamps were not monotone?" → The queue approach still works. The circular buffer would need a sorted structure or pre-sorting.
  • "Make it thread-safe?" → Add locks (mutex) around both methods, or use atomic integer arrays.
  • "Handle sub-second precision?" → Scale: use milliseconds, window = 300,000ms, arrays of size 300,000.

7 Summary & Key Takeaways ▾
Fixed-size sliding window → fixed-size circular buffer. Window size = buffer size.
Slot index = timestamp % window_size. The modulo maps an infinite timeline onto finite, recyclable slots.
Stale detection is trivial: times[i] != timestamp means the slot belongs to an older second.
Monotonically increasing timestamps are the invariant that makes overwrites safe without any explicit eviction logic.
getHits is O(300) = O(1) — scanning a fixed-size buffer is constant time regardless of traffic volume.
🎯 When to Use the Circular Buffer Pattern
  • Fixed-size sliding window over time: rate limiting, analytics, moving averages
  • You need O(1) space regardless of event volume (high-throughput systems)
  • Input is monotonically increasing (e.g., Unix timestamps from real-time systems)
  • The window size is small enough that scanning all slots is acceptable (≤ a few thousand)
