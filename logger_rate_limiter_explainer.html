LeetCode #359

Logger Rate Limiter

Design a logger system that limits duplicate messages: a given message can only be printed at most once every 10 seconds.

Easy Design Hash Map Timestamp Time: O(1) Space: O(n)
1 Understanding the Problem ▾

You're building a rate-limiting system for a log printer. In production systems, the same error can fire thousands of times per second — flooding logs and making real issues invisible. The rule: a message can only be printed (return True) if it has never been printed before, or if at least 10 seconds have passed since its last print.

The class has two methods: __init__() to set up the data structure, and shouldPrint(timestamp, message) which returns True if the message should print now, or False to suppress it. Timestamps are given in seconds and increase monotonically.

💡 Key Insight

For each unique message, we only need to remember one thing: the timestamp when it was last allowed through. A hash map from message → last_allowed_timestamp gives us an O(1) lookup to answer each query.

✓ shouldPrint Returns True

shouldPrint(1, "foo")
→ "foo" never seen → True

shouldPrint(11, "foo")
→ 11 - 1 = 10 ≥ 10 → True

✗ shouldPrint Returns False

shouldPrint(3, "foo")
→ 3 - 1 = 2 < 10 → False

shouldPrint(10, "foo")
→ 10 - 1 = 9 < 10 → False

Edge cases to consider: Exactly 10 seconds apart (diff = 10) is allowed — the condition is ≥ 10 not > 10. Different messages are completely independent. An empty message string is a valid key. Repeated denied requests should NOT block a message permanently.

🔑 The Rate-Limiting Condition

Message m at timestamp t is allowed if: m not in self.log OR t - self.log[m] >= 10. On allow: self.log[m] = t. On deny: do NOT update the map.

2 The Algorithm: Hash Map Timestamp Tracking ▾

The design stores a dictionary mapping each message string to the last timestamp at which that message was successfully printed. Every shouldPrint call is a pure O(1) hash-map lookup followed by a simple arithmetic comparison.

1 Initialize: Create self.log = {} in __init__. This dictionary maps each message string to the timestamp of its last allowed print.
2 Look up the message: Check if message is in self.log. If absent, the message is brand-new — skip to step 4.
3 Check the cooldown: If found, compute timestamp - self.log[message]. If the result is less than 10, the message is still in its cooldown window — return False without updating the map.
4 Allow and record: If the message is new OR the difference is ≥ 10, update self.log[message] = timestamp to reset the cooldown clock, then return True.
⚠️ The Most Common Mistake

Never update self.log[message] when returning False. Imagine "foo" is allowed at t=1, then denied at t=5, denied at t=7, denied at t=9. If you kept resetting to 5, 7, 9 on each denial, "foo" would be permanently locked out — a new cooldown starts from each rejected request. The clock must only reset when we actually print.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At all times, self.log[message] holds the timestamp of the most recent call that returned True for that message. This is the "last print time" — the origin of the 10-second cooldown clock. Since we only update on True, this invariant is preserved by every operation.

Why this guarantees correctness: The question "has it been at least 10 seconds since we last printed this?" is answered precisely by timestamp - self.log[message] >= 10. Our invariant ensures the stored timestamp is exactly the last print time, making the comparison exact.

Proof the invariant holds under denials: When we deny (return False), we do NOT write to the map. The stored timestamp stays anchored to the last successful print. Therefore, on any future call with timestamp T, the comparison T - self.log[m] correctly measures elapsed time since the last print — not elapsed time since the last request.

Why exactly 10 is allowed: The condition is diff < 10 to deny, which means diff ≥ 10 allows. So 10 seconds apart (same as "every 10 seconds") is permitted. If the first print is at t=1, next allowed is t=11 (11-1=10 ≥ 10).

💡 Analogy: Newspaper Printing

Think of each message as a news story. The rule is "don't reprint the same story within 10 days." You keep a ledger of the last publication date for each story. When a new submission arrives, you check the ledger. The ledger date only updates when you actually publish — rejected submissions don't count as publications and don't reset the clock.

⚠️ Why Brute Force Fails at Scale

In a production logging system, millions of requests may arrive per second. With brute force, the 1,000,000th call scans a million records — O(n) quickly becomes seconds of latency. The hash map solution handles any call volume in constant time.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Design Pattern

A class with persistent state + a stateful query method = Design problem. The key question: "What is the minimum state needed to answer each query?" Here, each query asks "did this message print recently?" — so we need exactly the last-print-timestamp per message. One value per key → hash map.

Step 2: Propose brute force first, then optimize

Say: "A naive approach would keep every (timestamp, message) pair and scan on each query — O(n) per call where n is total calls. But notice I only need the most recent allowed timestamp per message — the earlier ones are irrelevant." This telescopes naturally to the O(1) hash map solution.

Step 3: Implement and emphasize the critical update rule

When coding, explicitly comment: "I update the map only when returning True." This is the detail that shows depth. Most candidates will correctly build the lookup but miss the subtle invariant of not updating on deny.

Step 4: Analyze complexity and discuss trade-offs

State: O(1) per call, O(m) space where m = unique messages. Then proactively bring up the follow-up about memory: "In production I'd add TTL-based eviction to bound memory when there are many unique messages over a long period."

🗣️ What to Say to the Interviewer

"This is a rate-limiting design problem. For each message, all I need to
track is when it was last printed — so a hash map from message to
last-allowed-timestamp is the perfect data structure.

The key invariant: I only update the map when returning True. If I also

💡 Follow-Up Questions to Anticipate
  • "Memory concern if millions of unique messages?" → Pair with LRU cache or TTL-based eviction; entries unseen for 10+ seconds can be safely evicted
  • "Timestamps arrive out of order?" → Standard hash map still works; condition T - stored < 10 remains correct for any T
  • "Make it thread-safe?" → Wrap the dictionary in a lock; or use ConcurrentHashMap in Java
  • "Distributed system version?" → Redis SETNX with TTL 10; or sorted-set sliding window per message key
  • "Configurable window (not hardcoded 10)?" → Pass window as constructor parameter: def __init__(self, window=10)

7 Summary & Key Takeaways ▾
Store message → last_allowed_timestamp — one entry per unique message, not one per call.
Only update the map when returning True — denied requests must NOT shift the stored timestamp (the invariant).
Diff ≥ 10 is allowed (exactly 10 seconds is the boundary, not 11).
O(1) per call: hash map lookup and update are both constant time regardless of call history volume.
In production, bound memory with a TTL eviction strategy — entries unseen for ≥ 10s are safe to evict.
🎯 When to Use This Pattern (Hash Map Timestamp)
  • Design class with persistent state; queries answered by one value per key
  • Rate limiting, cooldown windows, deduplication within a time range
  • O(1) per operation is the stated or implied requirement
  • Signal: "print at most once every N seconds" or "limit to K requests per window"
