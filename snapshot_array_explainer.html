LeetCode #1146

Snapshot Array

Design a data structure that supports setting values, taking snapshots, and querying the value at a specific index at the time of a given snapshot.

Medium Design Binary Search set: O(1) get: O(log s) snap: O(1)
1 Understanding the Problem ▾

Imagine a spreadsheet that tracks history. You can edit cells freely, then press "Save Snapshot". Later you can ask: "What was cell B3's value in version 4 of this spreadsheet?" The system must preserve every past version so you can look anything up.

💡 Key Insight

You do not need to store the entire array on every snapshot. Most cells don't change between snapshots. Instead, record only what changes — as (snap_id, value) pairs per index. When querying, binary-search the history for that index to find the most recent write at or before the requested snapshot.

✗ Naive: Copy Whole Array on snap()

snap() → copy all n elements
n=50,000 and 50,000 snaps
→ 2.5 billion stored entries
Memory limit exceeded!

✓ Optimal: Only Record Changes

set(i, v) → append (snap_id, v) to data[i]
snap() → snap_id += 1, O(1)
get(i, sid) → bisect_right on data[i]
Space = O(total set() calls)

The API to implement:

  • SnapshotArray(length) — Initialize array of length elements, all zeros
  • set(index, val) — Set arr[index] = val under the current snapshot epoch
  • snap() — Take a snapshot; return the current snap_id (starts at 0), then advance
  • get(index, snap_id) — Return the value of arr[index] at the moment snapshot snap_id was taken

Edge Cases to think through: What if get() is called for an index that was never set()? (Return 0 — handled by baseline initialization.) What if multiple set() calls happen in the same snapshot? (Last write wins.) What if get() asks about snap_id 3, but the last write was at snap_id 1? (Binary search finds the predecessor.)

2 The Algorithm: Per-Index Version List + Binary Search ▾

Think of each array index as having its own changelog — a sorted list of (snap_id, value) entries. Each entry records "at snapshot X, this index held value Y." Since snap_id only ever increases, new entries are always appended — keeping the list sorted automatically. Binary search then gives O(log s) lookup for any historical query.

1 Initialize: Create data as a list of length changelogs. Each starts as [(0, 0)] — snap_id 0, value 0. Set snap_id = 0. The (0,0) baseline guarantees every index has a valid value to return even if never explicitly set.
2 set(index, val): If the last entry in data[index] already has the current snap_id, overwrite it (last write within a snapshot wins, and avoids duplicating the same snap_id). Otherwise, append a new (snap_id, val) entry.
3 snap(): Simply increment snap_id and return the old value (the snapshot just taken). O(1) — no data copying whatsoever.
4 get(index, snap_id): Binary-search data[index] using bisect_right(timeline, (snap_id, float('inf'))). This finds the position after all entries at or before the requested snap_id. Step back one position to get the most recent write at or before that snapshot. Return its value.

How get() uses binary search: finding the right historical version

data[0] timeline: (0, 5) (1, 8) (2, 2)
Query: get(0, snap_id=1) → bisect_right(timeline, (1, inf)) → position 2 (after (1,8))
Step back one: timeline[2-1] = (1, 8) → return 8 ✓
🔑 The bisect_right + float('inf') Sentinel Trick

Python tuples compare lexicographically: (1, inf) > (1, 999) for any real value. So bisect_right(timeline, (snap_id, float('inf'))) lands at the position after all entries whose first element equals snap_id. Subtracting 1 gives the last entry at or before the queried snapshot — precisely what we need.

3 Why It Works: The Invariant ▾
🔐 The Invariant

For every index i, data[i] is always a list of (snap_id, value) pairs in strictly ascending snap_id order. This is maintained automatically because snap_id is monotonically increasing — every new entry appended has a larger snap_id than all previous ones.

Why the sorted order guarantees correct answers: Binary search on a sorted list runs in O(log n). The sorted-by-snap_id invariant means we can always find the "most recent write at or before query snap_id" in O(log s) time, where s is the total number of snapshots.

Why initializing with (0, 0) is critical: If you never call set(3, ...), then data[3] only has [(0, 0)]. Any call to get(3, any_snap_id) will have bisect_right return 1, step back to position 0, and correctly return value 0. Without this seed, the binary search would have nothing to fall back on.

⚠️ Common Trap: Why Not Just Use a Dict per Index?

You could store data[i] = {snap_id: value} as a dictionary, but then get() would need to scan all keys to find the maximum snap_id ≤ query. That's O(s) per query — the same as brute force. The sorted list + binary search is what gives us the O(log s) advantage.

Multiple set() calls in the same snapshot: The overwrite rule (check if data[i][-1][0] == snap_id before appending) ensures that only the final write within a snapshot epoch is stored. This also preserves the strictly-ascending invariant — no two entries in the same timeline share the same snap_id.

4 The Code (Annotated) ▾

Solution 1 — Version List + Binary Search (Optimal, Interview Pick)

from bisect import bisect_right ← Python's O(log n) binary search, finds insertion point from the right
class SnapshotArray:
def __init__(self, length: int):
self.snap_id = 0 ← Current snapshot counter; starts at 0
self.data = [[(0, 0)] for _ in range(length)] ← Per-index changelog, seeded with (snap_id=0, val=0)
def set(self, index: int, val: int) -> None:
if self.data[index][-1][0] == self.snap_id: ← Already have an entry for this snap epoch?
self.data[index][-1] = (self.snap_id, val) ← Overwrite in-place (last write wins, no dup snap_ids)
else:
self.data[index].append((self.snap_id, val)) ← New snap epoch → append a fresh (snap_id, val) entry
def snap(self) -> int:
self.snap_id += 1 ← Advance to next epoch; future set() calls record under new id
return self.snap_id - 1 ← Return the id of the snapshot just taken (the old value)
def get(self, index: int, snap_id: int) -> int:
timeline = self.data[index] ← Sorted list of (snap_id, val) pairs for this index
pos = bisect_right(timeline, (snap_id, float('inf'))) ← Find position after all entries at or before snap_id
return timeline[pos - 1][1] ← Step back one: latest entry at or before target snap
⚠️ Why (snap_id, float('inf')) and not just snap_id?

The changelog stores tuples, so binary search compares tuples. If we searched for just an integer, Python would raise a TypeError (can't compare int to tuple). By using a tuple (snap_id, float('inf')), we search for the position after any real tuple (snap_id, any_actual_value), since float('inf') is larger than any stored value. This correctly places us after all entries for that snap_id.

Solution 2 — Brute Force: Full Array Copy Per Snapshot

class SnapshotArrayBrute:
def __init__(self, length: int):
self.arr = [0] * length ← Working copy of the array
self.history = [] ← List of full-array snapshots
def set(self, index, val):
self.arr[index] = val ← Simple in-place update
def snap(self):
self.history.append(self.arr[:]) ← O(n) copy! Disaster for large n
return len(self.history) - 1
def get(self, index, snap_id):
return self.history[snap_id][index] ← O(1) lookup but memory O(n × s)
💡 Complexity Summary

Version List + Binary Search (Optimal): set O(1) amortized · snap O(1) · get O(log s) · Space O(total set calls)
Brute Force (Full Copy): set O(1) · snap O(n) · get O(1) · Space O(n × s)
s = number of snapshots taken. The optimal approach shines when snapshots far outnumber actual writes.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Propose Brute Force and Identify the Bottleneck

"The naive approach: copy the whole array on every snap(). That's O(n) per snap — with n=50,000 and 50,000 snapshots, we're storing 2.5 billion values. TLE and MLE. I need a smarter approach."

Step 2: Observe Sparsity of Changes

"Most indices don't change between snapshots. Instead of storing the full array, I only need to record what actually changes — and when. If I attach a snapshot timestamp to each change, I can reconstruct any past state."

Step 3: Define the Data Structure

"Per index, maintain a sorted list of (snap_id, value) pairs. set() appends in O(1). snap() just increments a counter in O(1). get() binary-searches the per-index history in O(log s)."

Step 4: Handle Edge Cases Proactively

"Initialize every index with (0, 0) to handle queries before any set(). Detect when the last entry already has the current snap_id and overwrite instead of appending, so the list stays strictly sorted."

🗣️ What to Say to the Interviewer

"The key observation is that set() calls are sparse relative to the snap × index space.
I'll give each index its own sorted changelog of (snap_id, value) tuples.
set() appends or overwrites the last entry — O(1).
snap() just increments a global counter — O(1), no copying.
get() binary-searches the changelog with bisect_right and a sentinel — O(log s).
I seed each changelog with (0, 0) so unset indices always return 0."
🔮 Likely Follow-up Questions
  • "Range queries — sum of arr[l..r] at snap_id?" → Prefix-sum array, same versioning technique
  • "How to garbage-collect old snapshots?" → Track minimum snap_id still needed; drop earlier entries
  • "Thread safety for concurrent set()/snap()?" → Lock per-index for set, global lock for snap
  • "What if values are objects, not ints?" → Same approach; bisect still works on the snap_id key

7 Summary & Key Takeaways ▾
Store only what changes — per-index (snap_id, val) changelogs are sparse and memory-efficient
snap() is O(1) — just increment a counter, never copy the array
get() uses bisect_right with a sentinel tuple for O(log s) predecessor lookup
Initialize each changelog with (0, 0) to ensure every query has a valid fallback
Overwrite instead of append when the same snap_id already has an entry — keeps the sorted invariant
🎯 When to Use This Pattern
  • Design problems requiring historical / versioned queries on mutable data
  • Changes are sparse (few indices modified per snapshot cycle)
  • Need fast O(log n) or O(1) read/write on past versions
  • Any "what was the value at time T?" problem with monotonically increasing timestamps
