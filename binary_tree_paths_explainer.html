LeetCode #257

Binary Tree Paths

Given the root of a binary tree, return all root-to-leaf paths in any order.

Easy Trees DFS Preorder Build Path String Time: O(n) Space: O(h)
1 Understanding the Problem ▾

A root-to-leaf path starts at the tree's root and ends at a leaf node — any node that has no children at all. We must find every such path and return each as a string like "1->2->5", with "->" separating node values.

💡 Key Insight

A leaf is any node where both left and right children are None. The moment we reach one, we've completed a full path — there's nowhere deeper to go. We record it, then backtrack to explore other branches.

✓ Example Tree [1, 2, 3, null, 5]

    1
  /    \
 2       3
  \
   5
Leaves: 5, 3
Output: ["1->2->5", "1->3"]

What is NOT a leaf?

Node 2 has child 5
 → NOT a leaf
Node 5 has no children
 → IS a leaf ✓
Node 3 has no children
 → IS a leaf ✓

Edge cases to consider: A single-node tree — the root is itself a leaf, output is ["root_val"]. An empty tree (root is None) — output is []. A fully skewed tree (every node has only one child) — only one path exists, equal to all node values in order.

2 The Algorithm: Recursive DFS Preorder ▾

We use Depth-First Search in preorder — we process the current node first (adding its value to the running path string), then recurse into left and right subtrees. The key trick: pass an immutable string down the recursion so backtracking is automatic.

1 Handle null node: If node is None, return immediately. This handles missing children cleanly — we just don't go there.
2 Extend the path string: Append the current node's value to the running path. Use "->" as a separator — but only if the path is not empty (no separator before the root value).
3 Leaf detection — commit path: If not node.left and not node.right, we've reached a leaf. Append the completed path string to the results list and return.
4 Recurse into children: Call dfs(node.left, path) and dfs(node.right, path). The extended path string is passed down — each subtree gets the same prefix up to this node, then diverges independently.

Why preorder (not inorder or postorder)? Preorder visits Root → Left → Right. We need to record root's value before going deeper, so it appears at the front of the path string. Inorder or postorder would give values in the wrong order.

🔑 The Backtracking Trick

Python strings are immutable. When we write path = path + "->" + str(node.val), we create a new string object — the parent call's path variable is completely unchanged. So when we return from the left subtree and enter the right subtree, both get the exact same path prefix. No explicit undo step needed.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At any call dfs(node, path), the string path contains exactly the sequence of node values from the root down to node's parent, formatted with "->" separators. It is always correct and complete up to the current depth — never stale, never contaminated by sibling branches.

Why this guarantees completeness: DFS is exhaustive — it visits every node in the tree. Since the invariant holds at every call, when we reach any leaf, the path we've built is exactly the root-to-leaf string for that leaf. We collect every leaf's path, so we collect every root-to-leaf path.

Why immutability is essential: After dfs(node.left, path) returns, we call dfs(node.right, path). If path were a mutable list, the left recursion would have modified it and the right subtree would get a corrupted path. With strings (or by explicitly copying), each branch gets a pristine copy of the path up to the branching point.

⚠️ The Mutable List Bug — Most Common Mistake

If you use a list: path.append(node.val) and forget path.pop() after recursing, left-subtree values bleed into right-subtree paths. Always either (a) use immutable strings, or (b) use a list with explicit pop() after each recursive return. Both work; strings are safer for beginners.

Trace on the example tree [1, 2, 3, null, 5]:

dfs(1, "") → path = "1" → recurse left
  dfs(2, "1") → path = "1->2" → recurse left
    dfs(None) → return (null child)
  dfs(2, "1") → recurse right
    dfs(5, "1->2") → path = "1->2->5" → LEAF → result.append("1->2->5")
dfs(1, "") → recurse right
  dfs(3, "1") → path = "1->3" → LEAF → result.append("1->3")
Final: ["1->2->5", "1->3"] ✓
4 The Code (Annotated) ▾

Solution 1 — Recursive DFS with Path String (Optimal, Interview Pick)

def binaryTreePaths(root): ← entry point; returns list of path strings
result = [] ← collects all completed root-to-leaf paths
def dfs(node, path): ← inner DFS helper; path = string built so far
if not node: ← null child: nothing to do, stop here
return
path += ("->" if path else "") + str(node.val) ← extend: no "->" before the root
if not node.left and not node.right: ← leaf: both children are None
result.append(path) ← commit this complete root→leaf path
return ← nowhere deeper to go from a leaf
dfs(node.left, path) ← explore left subtree with this path prefix
dfs(node.right, path) ← explore right subtree with same prefix
dfs(root, "") ← start DFS from root, empty path
return result ← all root-to-leaf paths collected
⚠️ Why "->" if path else "" ?

At the root, path is "" (empty). If we always prepended "->", the root would produce "->1" instead of "1". The conditional check "->" if path else "" says: only add the separator if there are already values in the path. For all subsequent nodes, path is non-empty, so "->" is always added.

Solution 2 — Iterative DFS with Explicit Stack

def binaryTreePaths(root):
if not root: return [] ← empty tree: no paths at all
result, stack = [], [(root, str(root.val))] ← stack stores (node, path_string) pairs
while stack:
node, path = stack.pop() ← pop a (node, path) pair to process
if not node.left and not node.right: ← leaf: record completed path
result.append(path)
if node.right: ← push right first so left is processed first
stack.append((node.right, path + "->" + str(node.right.val)))
if node.left: ← left pushed last → popped next
stack.append((node.left, path + "->" + str(node.left.val)))
return result
💡 Complexity Summary

Recursive DFS: Time O(n) — every node visited once. Space O(h) call stack (h = tree height). O(n·h) total for stored path strings across all results.
Iterative DFS: Same O(n) time. Stack holds at most O(n) frames in skewed tree; each frame carries a path string up to O(h) length.
Tip: For very deep trees (depth > 1000), Python's recursion limit can bite — use the iterative version to be safe in production.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Pattern

"The problem asks for all root-to-leaf paths. Any time I need to explore all paths in a tree, I think DFS immediately — it naturally traces one path at a time, going deep. BFS would work too but is more awkward for path recording."

Step 2: Decide What to Carry Down

"Each recursive call needs the current node and the path built so far. I'll pass a growing string. The key question: should I use a string or a list? String is immutable — backtracking is automatic. List is mutable — I'd need explicit pop(). I'll go with string for safety."

Step 3: Write the Base Cases First

"Base case 1: node is None — return. Base case 2 (success): node is a leaf — record the path and return. These two cases cover all terminal conditions. Everything else recurses."

Step 4: Discuss Tradeoffs if Asked

"Recursive is cleaner. Iterative avoids Python's recursion depth limit — useful if the tree can be very deep (e.g., 10,000 nodes in a linked-list shape). I can code either."

🗣️ What to Say to the Interviewer

"I'll use DFS. Root-to-leaf paths map perfectly to preorder traversal — we build the path string as we descend.
My helper takes a node and the path string so far.
At each node: extend the string with '->' and the node's value.
If we hit a leaf (no children), append the completed path to results.
Otherwise recurse left and right — Python strings are immutable so backtracking is free.
Time: O(n), Space: O(h) for the call stack."

7 Summary & Key Takeaways ▾
A leaf = node with both children None; this is your "commit path" trigger
Preorder DFS (Root → Left → Right) naturally builds root-to-leaf path strings
Immutable strings give free backtracking — no explicit undo step needed
If using a mutable list, always pop() after each recursive return
Time O(n): every node visited once. Stack space O(h) = O(log n) balanced, O(n) skewed
ApproachTimeSpace (stack)Notes
Recursive DFS (string) ✓ O(n) O(h) Cleanest; implicit backtrack via immutability
Iterative DFS (explicit stack) O(n) O(n) worst Avoids Python recursion limit; safe for deep trees
Recursive DFS (list + pop) O(n) O(h) Slightly better memory; requires explicit backtrack
🎯 When to Recognize This Pattern
  • "Return all root-to-leaf paths" → DFS preorder + string accumulation
  • "Does any path satisfy condition X?" → DFS with running value, check at leaf
  • Leaf is always the commit point — the deepest reachable point for a path
  • Branching at internal nodes = need backtracking → immutable string or explicit pop
