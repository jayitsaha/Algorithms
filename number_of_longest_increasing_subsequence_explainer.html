LeetCode #673

Number of Longest Increasing Subsequence

Given an integer array nums, return the number of longest strictly increasing subsequences.

Medium Dynamic Programming DP with Count Time: O(n²) Space: O(n)
1 Understanding the Problem ▾

An increasing subsequence is a selection of elements from the array — chosen by position, in order — where each value is strictly greater than the previous. The elements do not need to be contiguous. This problem has two parts: first find the maximum length of any increasing subsequence, then count how many distinct subsequences achieve that maximum length.

💡 Key Insight

Classic LIS DP tracks one value per index: the length of the longest increasing subsequence ending there. This problem adds a second parallel value: the count of how many distinct LIS of that length end there. Both arrays evolve together with a two-case update rule — reset the count when you find a strictly better length, accumulate when you find a tie.

✓ Example: nums = [1, 3, 5, 4, 7]

LIS length = 4
[1, 3, 5, 7] → length 4 ✓
[1, 3, 4, 7] → length 4 ✓
Answer = 2

✗ Trap: nums = [2, 2, 2]

Strictly increasing: 2 is NOT > 2
No pair can form a longer sub
Each lone element = LIS of length 1
Answer = 3 (n individual elements)

Critical constraint — strictly increasing: Equal elements cannot extend a subsequence. This means duplicates always form isolated single-element LIS. The answer can be large (all elements equal → answer = n), but fits in a 32-bit integer for all given constraints.

Distinctness by position: Two subsequences are distinct if they use different index positions, even if the values happen to be identical (e.g., in an array with duplicate values, two different positions contribute two different subsequences).

2 The Algorithm: DP with Parallel Count Array ▾

We maintain two parallel DP arrays of size n, computed simultaneously in a nested loop:

length[i]
Length of the longest strictly increasing subsequence ending exactly at index i
 + 
count[i]
Number of distinct LIS of that length ending exactly at index i
1 Initialize: Set length[i] = 1 and count[i] = 1 for every index. Every single element is a valid subsequence of length 1, and there is exactly 1 way to form it — picking that element alone.
2 Outer loop — fix ending index i: For i = 1 to n−1, we determine the best LIS that can end at index i. At this point, length[0..i−1] and count[0..i−1] are already finalized (computed in earlier iterations).
3 Inner loop — try each prior index j: For j = 0 to i−1, if nums[j] < nums[i], then element i can extend any LIS ending at j. Compute the candidate length: length[j] + 1.
4 Two-case update rule:
• RESET: If length[j]+1 > length[i] — new best found! Update length[i] = length[j]+1 and reset count[i] = count[j]. Old paths are now suboptimal, discard their counts.
• ACCUMULATE: If length[j]+1 == length[i] — tie! Another set of paths reaches the same best length. Add: count[i] += count[j].
5 Collect the answer: After filling both arrays, find max_len = max(length). Return the sum of count[i] for all i where length[i] == max_len. Do NOT just return count[n−1] — the longest LIS may end at any index, not necessarily the last one.
🔑 Why copy count[j] instead of always adding?

Every LIS ending at j can be extended to a new LIS ending at i (by appending nums[i]). So all count[j] paths become valid paths to i. If j gives a strictly better length than what i currently holds, the old paths at i were for a shorter LIS — they're no longer the best, so we replace count[i] with count[j], discarding the inferior paths.

3 Why It Works: DP Invariant, Recurrence & Correctness ▾

Why is this Dynamic Programming? Two properties must hold:

✓ Optimal Substructure

The best LIS ending at i is built from the best LIS ending at some earlier j (where nums[j] < nums[i]), plus element i. The optimum of the subproblem (best LIS ending at j, and how many such sequences exist) feeds directly into the optimum at i.

✓ Overlapping Subproblems

A naive recursive approach recomputes "best LIS ending at j" separately for every i that uses j as an extension. With n elements, this creates O(2ⁿ) redundancy. Memoizing these subproblems collapses exponential work to O(n²).

📐 Recurrence Relations
length[i] = 1, count[i] = 1 for all i (base case: solo element) For each j < i where nums[j] < nums[i]: Case A — strictly better: if length[j] + 1 > length[i]: length[i] ← length[j] + 1 count[i] ← count[j] ← RESET (new best, old paths gone) Case B — exact tie: elif length[j] + 1 == length[i]: count[i] += count[j] ← ACCUMULATE (tie, merge path counts) Answer = Σ count[i] for all i where length[i] == max(length)
🔐 The Invariant

After the outer loop reaches index i: length[i] is exactly the length of the longest strictly increasing subsequence of nums[0..i] that ends at nums[i]. And count[i] is exactly how many distinct such subsequences exist. Both are guaranteed final because all j < i are processed before we compute anything for i.

Full trace on nums = [1, 3, 5, 4, 7]:

⚠️ Common Mistake: Always Adding Instead of Sometimes Resetting

A frequent bug is replacing the reset case with another accumulation: count[i] += count[j] even when length[j]+1 > length[i]. This is wrong because you'd be mixing counts from subsequences of different lengths. The reset step clears out inferior paths that are no longer the best. Remember: reset when strictly better, accumulate only when tied.

⚠️ Don't return count[n-1]!

The longest LIS can end at any index, not necessarily the last element. For nums=[5,3,4,1,2], the LIS of length 2 ends at multiple positions. You must sum count[i] over all i where length[i] equals the maximum — one pass over the final arrays after the nested loops finish.

Solution 2 — Top-Down Memoization (recursive, same complexity)

from functools import lru_cache
def findNumberOfLIS(nums):
n = len(nums)
@lru_cache(maxsize=None) ← cache each call to dp(i)
def dp(i): ← returns (best_length, count) ending at i
best_len, best_cnt = 1, 1 ← base: element alone
for j in range(i):
if nums[j] < nums[i]:
l, c = dp(j) ← RECURSE: get best (length, count) at j (cached)
if l + 1 > best_len: ← Case A: strictly better
best_len, best_cnt = l+1, c
elif l + 1 == best_len:← Case B: tie
best_cnt += c
return best_len, best_cnt
all_dp = [dp(i) for i in range(n)]← fill all (with caching, each computed once)
max_len = max(l for l,_ in all_dp)
return sum(c for l,c in all_dp if l==max_len)
💡 Complexity for Both Solutions

Time O(n²): The nested loop (or the recursive calls under memoization) visits every (i, j) pair exactly once — n × (n−1)/2 pairs.
Space O(n): Two arrays of size n for tabulation; O(n) memo cache for the recursive version (plus O(n) recursion stack, which Python handles).
Optimal O(n log n) exists using a Fenwick/segment tree indexed by value to query (max_length, total_count) — but implement the O(n²) version in an interview unless explicitly pushed for the log n solution.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Start from Classic LIS

Say: "This is a variant of Longest Increasing Subsequence. LIS uses dp[i] = length of LIS ending at i. Here I need both a length AND a count, so I'll run two parallel arrays using the same nested loop structure." This shows the interviewer you recognize the base problem.

Step 2: Define Both States Clearly

Explicitly define: "length[i] = length of the longest strictly increasing subsequence ending at index i. count[i] = number of distinct LIS of that maximum length ending at index i." State the base case: both start at 1 for all i.

Step 3: Articulate the Two-Case Update Rule

The crux: "For each j < i with nums[j] < nums[i], I compare length[j]+1 to length[i]. If strictly greater → I've found a new best at i, so reset count[i] to count[j]. If equal → another way to achieve the same best, so add count[j] to count[i]. If less → nothing to do." Write both if/elif branches explicitly.

Step 4: Careful Final Aggregation

Don't forget: "After both loops, find max_len = max(length), then sum count[i] for all i where length[i] == max_len. I cannot just return count[n−1] because the LIS may end anywhere in the array." This is a common error to call out proactively.

🗣️ What to Say to the Interviewer

"I recognize this as LIS with an extra dimension — I need both length and count.
I'll maintain two O(n) arrays: length[i] for LIS length ending at i, count[i] for how many.
Both initialize to 1. For each i, I scan all j < i where nums[j] < nums[i].
If length[j]+1 beats length[i]: update length and RESET count to count[j].
If it ties: accumulate count += count[j].
Finally find max_len, sum count[i] for all i achieving it.
Time O(n²), Space O(n). There's an O(n log n) BIT approach but this is clean."
🔮 Likely Follow-up Questions
  • "Can you do O(n log n)?" → Binary Indexed Tree / segment tree indexed by compressed values, storing (max_len, total_count) pairs. Complex but achievable.
  • "What if non-strictly increasing (allow equal)?" → Change nums[j] < nums[i] to nums[j] <= nums[i].
  • "Reconstruct one of the LIS?" → After filling arrays, backtrack from any i with length[i]==max_len, following valid j links.
  • "What if we only want the LIS length, not count?" → Drop the count array entirely, return max(length) — that's classic LIS (#300).
