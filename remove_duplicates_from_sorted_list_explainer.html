LeetCode #83

Remove Duplicates from Sorted List

Given the head of a sorted linked list, delete all duplicates such that each element appears only once. Return the linked list sorted as well.

Easy Linked List Compare Adjacent Nodes Time: O(n) Space: O(1)
1 Understanding the Problem ▾

We have a sorted linked list — nodes arranged in non-decreasing order. Duplicates therefore appear consecutively: a run of equal values always sits together. Our job is to trim those runs down to a single representative node, keeping the list otherwise intact and in order.

💡 Key Insight

Because the list is sorted, all duplicates of a value are adjacent. We never need to look far ahead — if curr.val == curr.next.val, that next node is a duplicate. This lets us solve the problem with a single pointer in one pass, using no extra memory. Just bypass the duplicate by rewiring the .next pointer.

Input: 1 → 1 → 2 → 3 → 3

node[0]: val=1
node[1]: val=1 ← DUPLICATE
node[2]: val=2
node[3]: val=3
node[4]: val=3 ← DUPLICATE

Output: 1 → 2 → 3

Keep first 1, remove second 1
Keep 2 (unique)
Keep first 3, remove second 3

3 unique nodes remain

Edge cases to consider:

  • Empty list (head is None) — return None; no nodes to process
  • Single node — curr.next is already None; loop never runs, return head
  • All same value (1 → 1 → 1) — repeatedly bypass until only one node remains
  • No duplicates — traverse without removing anything, return head unchanged
  • Duplicates at the tail — handled naturally: after bypass, curr.next becomes None, loop exits
🔑 Why "Sorted" Is the Critical Constraint

If the list were unsorted (e.g. 3 → 1 → 3 → 2), duplicates could be scattered anywhere. You'd need a hash set to track seen values — O(n) space. The sorted property clusters duplicates together, making O(1) space possible: we only ever need to compare the current node to its immediate successor.

2 The Algorithm: Compare-and-Bypass ▾

We walk through the list with a single pointer curr. At each position, we compare curr.val to curr.next.val. Equal values mean a duplicate: bypass it. Different values: advance. This is the minimal-pointer solution that fully exploits sorted order.

1 Initialize: Set curr = head. We keep head fixed as our return value — curr is our walking pointer. (No dummy node needed since head itself is never removed.)
2 Loop condition: while curr and curr.next — we need both a current node AND a next node to compare. When either is missing, there's nothing left to check.
3 Duplicate check: if curr.val == curr.next.val — adjacent equal values in a sorted list must be duplicates. Bypass the duplicate: curr.next = curr.next.next. This "stitches over" the duplicate node, removing it from the chain.
4 Stay put after bypass (critical!): Do NOT move curr after removing a duplicate. The new curr.next might also equal curr.val — there could be a run of three or more identical values. Re-check from the same position.
5 Advance when different: else: curr = curr.next. Values differ, so the current node is confirmed unique in its neighborhood. Move forward.
6 Return head: We only ever modify .next pointers, never the head node itself. Return head directly.

⚠️ The Stay-Put Rule: Why We Don't Advance After a Removal

Consider 1 → 1 → 1 → 2. After bypassing the second 1: curr is still at first 1, curr.next is now the third 1. If we moved curr forward here, we'd skip removing it. Only advancing on a different next value guarantees we catch chains of three or more duplicates.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At the start of every loop iteration, all nodes from head to curr (inclusive) are distinct. The sub-list head → ... → curr contains no duplicates and represents the deduplicated prefix built so far. We only advance curr when we are certain the new current node won't match anything before it (because the list is sorted, this is guaranteed by curr.val ≠ curr.next.val).

Why pointer rewiring works: Setting curr.next = curr.next.next removes the duplicate from the chain — nothing points to it anymore. In Python, the garbage collector reclaims it. In languages with manual memory (C/C++), you'd free() it explicitly, but the logic is identical.

Why head is unchanged: We start curr at head and only ever modify curr.next. Even if head.val == head.next.val, we bypass head.next — head itself stays in the list as the single representative of its value. We never need to create a dummy head node for this problem.

Termination proof: On every iteration, either we remove curr.next (shortening the list by 1) or we advance curr forward by 1. The list is finite, so the total number of operations is bounded by 2n. The algorithm always terminates.

💡 Analogy: Proofreading a Sorted Word List

Imagine you're proofreading a sorted dictionary printout, pointing your finger at each word. You compare it to the next word. If they're the same, you cross out the next one (bypass it) and keep your finger in place — there might be a third copy right after. If they're different, you slide your finger to the next word. When your finger falls off the page, every remaining word is unique.

4 The Code (Annotated) ▾

Solution 1 — Single Pointer, In-Place Bypass (Optimal · Interview Pick)

def deleteDuplicates(head): ← head: first node of sorted linked list
curr = head ← Walking pointer; head stays fixed for return
while curr and curr.next: ← Need curr (to read) AND curr.next (to compare)
if curr.val == curr.next.val: ← Adjacent equal = duplicate (sorted → adjacent)
curr.next = curr.next.next ← Bypass: skip the duplicate node entirely
else: ← Different values → current node is safe
curr = curr.next ← Advance to next distinct value
return head ← Head never changed; return directly
⚠️ Common Mistake: curr = curr.next After Every Removal

If you write curr = curr.next after the bypass (inside the if branch), you'd silently skip over triple or higher duplicates. After bypassing, stay at curr — let the next loop iteration re-evaluate the new curr.next.

Solution 2 — Two-Pointer with Dummy Head (works for unsorted lists too)

def deleteDuplicates_twoptr(head): ← Brute-force-friendly; also handles unsorted with set
seen = set() ← O(n) space: track every value we encounter
dummy = ListNode(0, head) ← Dummy node simplifies head-removal edge case
prev, curr = dummy, head
while curr:
if curr.val in seen: ← Already seen → remove this node
prev.next = curr.next ← Unlink by advancing prev.next over curr
else: ← First time seeing this value
seen.add(curr.val) ← Record it
prev = curr ← Advance prev only for kept nodes
curr = curr.next ← Always advance curr
return dummy.next ← dummy.next skips the sentinel node
💡 Complexity Comparison

Single Pointer (Optimal): Time O(n) · Space O(1) — no hash set, no dummy node
Two Pointer + Hash Set: Time O(n) · Space O(n) — necessary for unsorted lists
Recursive: Time O(n) · Space O(n) call stack — elegant but not constant space
Interview choice: Always single pointer when input is sorted.

Bonus: Recursive Solution (for code golf / follow-up)

def deleteDuplicates_rec(head): ← O(n) space due to call stack
if not head or not head.next: ← Base case: 0 or 1 nodes left
return head
head.next = deleteDuplicates_rec(head.next) ← Recurse: deduplicate the rest first
if head.val == head.next.val: ← After recursion, head.next is deduped → compare
return head.next ← Skip current head (it's a duplicate)
return head ← Keep current head (it's unique)
5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Seize on "Sorted"

The moment you see "sorted linked list," your first thought should be: "Duplicates must be adjacent." This is the O(1) space insight. Without sorted order you'd need a hash set; with it, one pointer comparing curr to curr.next is enough. State this explicitly: "Since the list is sorted, I only need to compare each node to its immediate neighbour."

Step 2: Think in Pointer Operations

For any linked list in-place removal, the operation is always: "reroute a .next pointer." To remove node B sitting between A and C, you write A.next = C. Here: curr.next = curr.next.next. Draw this on a whiteboard — it makes the logic crystal clear and shows the interviewer you reason from first principles.

Step 3: Clarify the Stay-Put Rule

Proactively mention: "After removing a duplicate, I don't advance curr — the new curr.next could also be a duplicate." This shows you've thought about triplicates and longer runs. Example: "Consider 1 → 1 → 1 → 2. Without staying put, we'd miss the third 1."

Step 4: Confirm Head Invariant

Unlike removal-from-end problems, here the head node is always kept. Point out: "I return head directly — I only modify .next pointers, never head itself. No dummy node needed." This shows you're not cargo-culting patterns without understanding why they're needed.

🗣️ What to Say to the Interviewer

"The list is sorted, so duplicates must be adjacent — I only need to compare curr to curr.next.
I'll use one pointer curr starting at head. At each step:
— if curr.val equals curr.next.val, bypass: curr.next = curr.next.next. Stay at curr.
— if they differ, advance: curr = curr.next.
Loop terminates when curr or curr.next is None. Return head — it's never moved.
Time: O(n) one pass. Space: O(1) no extra structures.
Edge cases: empty list returns None; all-same list reduces to one node."
🔮 Likely Follow-up Questions
  • "Remove ALL nodes with any duplicates (not just extras)?" → LeetCode #82. Need prev pointer + dummy head; skip entire runs of equal values.
  • "What if the list is unsorted?" → Use a hash set: O(n) time + O(n) space.
  • "Can you do it recursively?" → Yes: recurse on head.next first, then compare. Elegant but O(n) stack space.
  • "What if there are more than 2 duplicates (e.g. 1→1→1)?" → Already handled: staying put after each bypass catches chains of any length.
