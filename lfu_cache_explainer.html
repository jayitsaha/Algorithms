LeetCode #460

LFU Cache

Design and implement a Least Frequently Used (LFU) cache — evict the item accessed fewest times; break ties by Least Recently Used order.

Hard Design Multiple Data Structures Time: O(1) Space: O(capacity)
1 Understanding the Problem ▾

An LFU Cache is a fixed-size key-value store with an automatic eviction policy: when full and a new key arrives, it evicts the key that has been accessed the fewest times. If multiple keys tie for the minimum access count, the one that was least recently used (among those tied) is evicted.

We must support two operations, each in O(1) time:

  • get(key) — return value if key exists, else -1. Counts as one access.
  • put(key, value) — insert or update. If full and new key, evict first. Counts as one access.
💡 Key Insight

Scanning all keys to find the minimum-frequency one takes O(n) — too slow. Instead, group items by their access count using a hash map of ordered buckets. Keep a single min_freq integer so you can jump directly to the eviction target in O(1). The only data structures you need are two dicts and Python's OrderedDict.

Edge cases: capacity=0 (always -1 on get, no-op on put), updating an existing key (no eviction, just update value + increment freq), single-capacity cache (every put of a new key evicts the previous one).

Each frequency bucket is a Python OrderedDict — a hash map that also preserves insertion order. This gives us O(1) insert, O(1) delete-by-key, and O(1) "peek at oldest entry" via next(iter(d)) or popitem(last=False). The oldest entry in a bucket is the LRU within that frequency tier — exactly the one we want to evict.

G get(key): If key not in key_map → return -1. Otherwise call _increment(key) to move the key up one freq bucket, then return key_map[key][0].
U put(key, value) — existing key: Update value in key_map[key][0]. Call _increment(key) to promote its frequency. No eviction — we're not adding a new key.
I put(key, value) — new key, cache not full: Set key_map[key] = [value, 1]. Add key to freq_map[1] (tail = MRU). Set min_freq = 1. Done.
E put(key, value) — new key, cache full → evict first: freq_map[min_freq].popitem(last=False) removes the oldest key from the min-frequency bucket. Delete that key from key_map. Then insert the new key at freq=1 and reset min_freq = 1.
△ _increment(key) helper: Read old_freq. Remove key from freq_map[old_freq]. If that bucket is now empty and old_freq == min_freq → min_freq += 1. Insert key at tail of freq_map[old_freq+1]. Update key_map[key][1] = old_freq+1.
🔑 Why min_freq resets to 1 on every new insertion

Any brand-new key starts at frequency 1, regardless of what other frequencies exist. Even if all live keys have freq=5, a newly inserted key has freq=1 — the new global minimum. So after any successful new insertion, min_freq is always reset to 1. This is the only event that can lower min_freq.

3 Why It Works: The Three Invariants ▾
🔐 Invariant 1 — key_map is the single source of truth

At all times, key_map[key] = [value, freq] for every live key. A key is "in the cache" if and only if it is in key_map. Eviction removes the key from key_map immediately, so any future get on that key correctly returns -1.

🔐 Invariant 2 — freq_map mirrors key_map exactly

Every live key appears in exactly one bucket in freq_map — the bucket numbered by its current frequency. When we call _increment(key), we atomically remove from the old bucket and append to the new bucket's tail. Ordering within each bucket is insertion order = access-time order = LRU order for that frequency tier.

🔐 Invariant 3 — min_freq is always exact

min_freq can only change in two ways: (1) increment by 1 when _increment empties the current min-freq bucket, or (2) reset to 1 when a new key is inserted. Because increments happen one-at-a-time and each moves a key exactly one level up, and because new insertions always enter at freq=1, min_freq always equals the true minimum frequency across all live keys.

Why eviction is O(1): freq_map[min_freq] is a direct dict lookup — O(1). popitem(last=False) on an OrderedDict removes the oldest entry — O(1). No heap, no scan, no sort.

⚠️ Common Bug: Not checking if old bucket empties at min_freq

After removing a key from freq_map[old_freq], if the bucket is now empty and old_freq equals min_freq, you must increment min_freq. If you skip this check, min_freq becomes stale — pointing to an empty bucket — and the next eviction will crash or evict incorrectly. The condition is both: empty AND equals min_freq (it's fine if a non-min bucket becomes empty).

⚠️ Common Bug: popitem() vs popitem(last=False)

Python's OrderedDict.popitem() with no arguments removes the last (newest) item — that's MRU, which is backwards. You need popitem(last=False) to remove the first (oldest) = LRU. Always verify this in your implementation.

4 The Code (Annotated) ▾

Solution — HashMap + OrderedDict Buckets (Interview Pick, O(1) all ops)

from collections import defaultdict, OrderedDict← both stdlib; no external deps
class LFUCache:
def __init__(self, capacity: int):
self.cap = capacity ← 0 is valid; handled in put
self.min_freq = 0 ← will be set to 1 on first insert
self.key_map = {} ← key → [value, freq] (mutable list)
self.freq_map = defaultdict(OrderedDict)← freq → {key: val}; auto-creates new buckets
def get(self, key: int) -> int:
if key not in self.key_map:← cache miss
return -1
self._increment(key) ← access counts; move key up one freq bucket
return self.key_map[key][0] ← return value (index 0)
def put(self, key: int, value: int) -> None:
if self.cap == 0: return ← edge: zero-capacity cache is no-op
if key in self.key_map: ← UPDATE existing key (no eviction!)
self.key_map[key][0] = value ← overwrite value in place
self._increment(key) ← count this put as an access
return
if len(self.key_map) == self.cap: ← EVICT: new key but cache is full
evict_key, _ = self.freq_map[self.min_freq].popitem(last=False)← pop LRU from min-freq bucket
del self.key_map[evict_key] ← remove from key_map too
self.key_map[key] = [value, 1] ← INSERT: new key starts at freq=1
self.freq_map[1][key] = value ← add to tail of freq-1 bucket (MRU position)
self.min_freq = 1 ← new key has freq=1 → global min must be 1
def _increment(self, key: int):
val, freq = self.key_map[key] ← read current state
del self.freq_map[freq][key] ← remove from old bucket
if not self.freq_map[freq]: ← if old bucket is now empty…
if freq == self.min_freq: ← …AND it was the min bucket…
self.min_freq += 1 ← …raise min_freq by exactly 1
self.freq_map[freq + 1][key] = val ← append at tail of next bucket (MRU position)
self.key_map[key] = [val, freq + 1] ← update freq in key_map

Brute Force Alternative — O(n) per eviction (for comparison)

def put_slow(self, key, value): ← just to illustrate what we avoid

💡 Complexity Summary

get: O(1) — one dict lookup, one _increment call (all O(1) dict ops)
put: O(1) — one lookup + at most one eviction (OrderedDict popitem) + one insert
_increment: O(1) — one del + one insert in two OrderedDicts + two key_map writes
Space: O(capacity) — key_map has ≤ capacity entries; freq_map mirrors them

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Start from LRU Cache (the simpler sibling)

LRU uses a dict + doubly-linked list (or OrderedDict). "LFU is LRU plus a frequency dimension. Instead of one recency list, I need one recency list per frequency. That maps to a dict of OrderedDicts."

Step 2: Name the three structures and their roles

"I need: (1) key_map for O(1) value + frequency lookup by key; (2) freq_map to find all keys at a given frequency with O(1) LRU ordering; (3) min_freq integer so I know which bucket to evict from instantly."

Step 3: Walk through _increment logic carefully

Interviewers specifically probe the min_freq update: "min_freq only rises when the current min bucket empties due to an increment. It resets to 1 only on new insertion. I never need to scan for the next minimum — incrementing by 1 is always correct because you can only reach freq f+1 via freq f."

Step 4: Clarify "update" vs "insert" for put()

Many candidates forget: if the key already exists, put() is an update — no eviction, just change the value and increment frequency. Only when the key is truly new and the cache is full do we evict. State this explicitly before coding.

🗣️ What to Say to the Interviewer

"I'll use three structures: key_map maps each key to [value, freq]. freq_map maps each
frequency to an OrderedDict of keys at that frequency, ordered oldest-first so I can
evict the LRU in O(1) via popitem(last=False). min_freq tracks the current minimum.

For get: look up key_map, call _increment to promote the key's frequency, return value.
For put with existing key: update value, call _increment, return early — no eviction.
For put with new key and full cache: evict oldest from freq_map[min_freq], delete from
key_map, then insert new key at freq=1 and reset min_freq=1.

In _increment: move key from freq f to f+1 in freq_map. If bucket f becomes empty
AND f == min_freq, bump min_freq to f+1."
🔮 Likely Follow-up Questions
  • "How would you implement in Java?" → LinkedHashSet per freq bucket gives O(1) add/remove/first; or a manual DLL with a set for O(1) everything
  • "What if we want most-frequently-used eviction?" → Symmetric: track max_freq, evict from freq_map[max_freq] newest item
  • "Can freq_map grow unboundedly?" → Yes in pathological cases (always access the same key), but bounded by capacity × max_freq; in practice freq_map holds at most capacity non-empty buckets at any time
  • "LFU vs LRU — which is better in practice?" → LRU is simpler and usually better for temporal locality; LFU is better when popular items should stay even if not recently accessed (e.g., CDN assets)

7 Summary & Key Takeaways ▾
Three structures: key_map (key→val+freq), freq_map (freq→OrderedDict), min_freq integer
min_freq rises by 1 when the min bucket empties on increment; resets to 1 on new insert
Eviction = popitem(last=False) on freq_map[min_freq] — O(1), no scan
Updating an existing key's value is NOT eviction — just value update + _increment
popitem(last=False) = remove oldest (LRU); popitem() = remove newest — critical distinction

🎯 When to Recognize the Multiple Data Structures Pattern
  • "All operations must be O(1)" with a non-trivial eviction/ordering rule → need ≥2 structures
  • "Design a cache / store / counter" → identify what you must look up instantly → one structure per lookup type
  • Evict by frequency + recency → two orderings needed → two structures per key
  • Any time you see "O(1) inc/dec/getMax/getMin" — think dict + ordered set/list pair
