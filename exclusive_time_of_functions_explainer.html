LeetCode #636

Exclusive Time of Functions

Given n functions and their execution logs (start/end events), compute the exclusive execution time of each function.

Medium Stacks Stack + Time Tracking Time: O(m) Space: O(n)
1 Understanding the Problem ▾

Imagine a CPU profiler logging when each function starts and stops. Functions can interrupt each other — when function B starts while A is running, A is paused and B runs exclusively. When B ends, A resumes. We need the total exclusive time each function ran — time it was the active function, not waiting for a callee.

💡 Key Insight

This is a call stack simulation. When a function starts, push it onto the stack — it becomes the "running" function. When it ends, pop it and credit the time it just used. The trick is tracking how much time to credit at each event using a prev timestamp bookmark.

Each log entry is "fn_id:type:timestamp". Example: n=2, logs = ["0:start:0","1:start:2","1:end:5","0:end:6"]:

⚠️ What "start" means for time

A "start" at time t means the function begins at unit t. It owns that unit going forward. Elapsed on start = ts - prev (no +1).

✓ What "end" means for time

An "end" at time t means the function runs through unit t. It owns unit t. Elapsed on end = ts - prev + 1 (inclusive).

Edge cases: A function can be called recursively (same fn_id on stack multiple times). Functions can be non-overlapping (no nesting at all). Single function running the entire duration.

2 The Algorithm: Stack + Previous Timestamp Bookmark ▾

We maintain a stack (call hierarchy) and a variable prev — the timestamp of the "last accounting checkpoint." Whenever we see a log event, we first settle who gets the elapsed time since prev, then update prev.

1 Parse each log into fn_id, log_type ("start" or "end"), and ts (timestamp).
2 On "start": If the stack is non-empty, credit the current top-of-stack for ts - prev time units (it ran from prev to ts-1). Then push fn_id and set prev = ts.
3 On "end": Pop the top-of-stack function and credit it ts - prev + 1 time units (it ran from prev through ts inclusive). Then set prev = ts + 1.
4 Return the result array after all logs are processed.
🔑 Why prev is the Key

prev is a bookmark: "which time unit does the next accounting start from?" On a start event, the previous runner owned [prev, ts-1]. On an end event, the ending function owned [prev, ts]. After an end, the next segment starts at ts+1. This ensures every time unit from 0 to the last timestamp is credited to exactly one function.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every point during processing: prev is the first unaccounted time unit. All time units before prev have been credited to exactly one function. The function at the top of the stack is actively running from prev onward — it will be credited when the next event arrives.

On "start" at ts: The top-of-stack ran from prev to ts-1 (it does not own unit ts — the new function does). So elapsed = ts - prev. We push the new function and set prev = ts (now the new function "owns" from ts onward).

On "end" at ts: The ending function ran from prev through ts (inclusive — the end timestamp is the last unit it owns). So elapsed = ts - prev + 1. After popping, the next function on the stack (if any) resumes from ts+1, so prev = ts+1.

⚠️ The Off-By-One Trap

This problem's most common bug: forgetting the +1 on end events and/or setting prev = ts instead of prev = ts+1 after an end. Verify: if fn starts at t=2 and ends at t=5, it ran 4 units (t=2,3,4,5). Formula check: 5 - 2 + 1 = 4. After end at t=5, next segment starts at t=6, so prev = 5+1 = 6.

Total time conservation: Every unit from 0 to the final end timestamp is credited exactly once. There is no double-counting because prev always advances past what was just credited, and no gap is skipped because prev starts exactly where the last account ended.

4 The Code (Annotated) ▾

Solution 1 — Stack + Previous Timestamp (Optimal, O(m) time) — Interview Pick

def exclusiveTime(n: int, logs: list[str]) -> list[int]:
result = [0] * n ← Exclusive time per function; n functions total
stack = [] ← Call stack; stores fn_id of currently-running functions (LIFO)
prev = 0 ← "Last settled timestamp" — first unaccounted time unit
for log in logs:
fn_id, log_type, ts = log.split(":") ← Split "0:start:2" into three parts
fn_id, ts = int(fn_id), int(ts) ← Convert strings to integers
if log_type == "start":
if stack: ← If someone is already running, credit their time up to ts-1
result[stack[-1]] += ts - prev ← They ran from prev to ts-1; no +1 (ts belongs to new fn)
stack.append(fn_id) ← New function takes over; push onto call stack
prev = ts ← New fn owns from ts onward
else: # "end"
result[stack.pop()] += ts - prev + 1 ← +1: end ts is inclusive (function owns that last unit)
prev = ts + 1 ← Next fn resumes from unit AFTER this end (critical!)
return result
⚠️ Why prev = ts + 1 (not ts) after an end event?

The ending function was credited through unit ts. Unit ts is now fully accounted for. The next function that resumes (or the next event) starts from unit ts + 1. If you wrote prev = ts, then the next accounting would include unit ts again — double-counting the last unit of the ended function.

💡 Complexity Summary

Stack + prev (optimal): Time O(m) for m log entries; Space O(n) result + O(d) stack where d = max nesting depth.
Brute force: Time O(T) where T = max timestamp value — can be huge. Not acceptable for this problem.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Spot the Call Stack Pattern

"Functions interrupt each other" → LIFO order → stack. The most recently started function is always the first to finish. This is exactly how real call stacks work, so simulate one.

Step 2: Identify What to Credit at Each Event

The core question is: "Who gets the time from the last event to now?" Whoever is on top of the stack owns that interval. So at every event, we settle up with the current runner before changing the stack.

Step 3: Work Out the Off-by-One Carefully

Explicitly verify on a tiny example: fn starts at t=0, ends at t=2. How many units? (t=0, t=1, t=2) = 3 units = 2-0+1. So end events need +1. And after end at t=2, next segment starts at t=3, so prev = ts+1. Say all this aloud in the interview.

Step 4: Code the Clean Single-Pass Solution

One pass, four operations per event: (on start) credit top if stack non-empty, push, update prev. (on end) pop and credit with +1, update prev to ts+1. Trace through the example to verify.

🗣️ What to Say to the Interviewer

"The nesting structure of start/end events is exactly a call stack — LIFO.
I'll maintain a stack of active function IDs and a prev pointer — the last settled time unit.
On each 'start': credit the current top-of-stack for ts-prev units (it ran from prev to ts-1).
Then push the new function and set prev=ts.
On each 'end': credit the top (pop it) for ts-prev+1 units — end timestamp is inclusive.
Set prev=ts+1 so the resume point is the next unit after this end.
Time O(m) for m logs, space O(n) result + O(depth) stack."
🔮 Likely Follow-up Questions
  • "What about recursive calls (same fn_id on stack twice)?" → Stack handles naturally — same fn_id can be pushed multiple times, each call gets its own accounting.
  • "What if logs arrive out of order?" → Sort by timestamp first, then apply the same algorithm.
  • "Find the function that ran the longest?" → result.index(max(result)) after computing result.
  • "Could you do this in O(1) space?" → Not cleanly; the stack depth mirrors actual call nesting, which can be O(n) in worst case.

Nested start/end events → use a stack to track the active call hierarchy

🎯 When to Recognize Stack + Time Tracking
  • Functions/processes that can interrupt each other (LIFO resume) → call stack
  • "Exclusive" or "net" time per entity → credit time at each stack transition
  • Start/end events with nested structure → push on start, pop on end
  • Need to apportion shared resource (CPU time, bandwidth) among nested intervals
