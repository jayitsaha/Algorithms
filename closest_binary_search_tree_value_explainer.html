LeetCode #270

Closest Binary Search Tree Value

Given the root of a BST and a floating-point target, return the integer node value closest to target.

Easy Trees BST Property Time: O(h) Space: O(1)
1 Understanding the Problem ▾

You have a valid Binary Search Tree (BST) and a floating-point target. Find the single integer node value in the tree with the smallest absolute difference to target: minimize |node.val − target|.

💡 Key Insight

A BST is inherently sorted: every left-subtree value is smaller than the root, and every right-subtree value is larger. This means we can navigate toward target like binary search — at each node, we know exactly which half of the remaining tree could contain a closer value. We never need to explore both children.

Edge Cases: Exact match (diff = 0 — return immediately). Single-node tree. Target smaller than all values (leftmost closest). Target larger than all (rightmost closest). The problem guarantees exactly one valid answer.

2 The Algorithm: Track Closest While Searching ▾

We walk a single root-to-null path through the BST, updating our best candidate at every node. The BST ordering property tells us which direction to proceed at each step.

1 Initialize: closest = root.val (first candidate), node = root.
2 Update: At every node, if |node.val − target| < |closest − target|, set closest = node.val.
3 Navigate: If target < node.val, go left (smaller values are closer to target). Otherwise go right (larger values are closer).
4 Stop: When node is None (fallen off the tree). Return closest.
🔑 Why Can We Prune the Other Subtree?

Say the current node has value v and target < v. Every value in the right subtree is ≥ v. On the number line, all those values are on the same side of target as v but farther away. So they cannot improve upon closest. We skip the entire right subtree — reducing work from O(n) to O(h).

3 Why It Works: The Invariant ▾
🔐 The Invariant

At any point in the search, closest holds the best candidate among all visited nodes. When we choose a direction (say left, because target < node.val), every unvisited node in the other direction (right subtree) has values strictly farther from target than node.val. So the global optimum lies on the path we are walking.

Formal sketch: Let current node have value v. Suppose target < v. Every value r in the right subtree satisfies r ≥ v. Since target < v ≤ r, we have |r − target| = r − target ≥ v − target = |v − target|. So no right-subtree value can be strictly closer than v. We safely skip it.

Time: We visit at most one node per level, so O(h). For a balanced BST that is O(log n). For a pathological skewed tree (essentially a linked list), O(n).

⚠️ Common Mistake: Stopping at a Leaf

It is tempting to stop when you reach a leaf node (both children are null). But the invariant only guarantees correctness when you follow the full search path to null — meaning you tried to go in the correct direction and hit a null child. Some problems have the closest value two levels below a leaf in the other direction; always run until node = None.

💡 Analogy: Finding a Name in a Phone Book

You're looking for "Smith" in a sorted phone book. Open to a page — if the current name comes after "Smith", you know the back half of the book can only have names farther from "Smith". You only need to search one half at each step. BST search is exactly this, but on a tree instead of pages.

4 The Code (Annotated) ▾

Solution 1 — Iterative BST Search (Interview Pick: O(h) time, O(1) space)

def closestValue(root, target: float) -> int:
closest = root.val ← Root is our first candidate (guaranteed non-empty)
node = root ← Start traversal at root
while node: ← Walk until we fall off the tree
if abs(node.val - target) < abs(closest - target):← Better candidate?
closest = node.val ← Yes — update the running best
if target < node.val: ← BST navigation: target is left of current
node = node.left ← Go left (right subtree values are farther)
else: ← target >= node.val: go right
node = node.right ← Go right (left subtree values are farther)
return closest ← Best seen along the root-to-null search path
⚠️ Why Iterative Over Recursive?

A recursive solution works correctly with the same O(h) time, but uses O(h) call stack space. In a degenerate BST (sorted insertion order → linked list), that is O(n) stack frames. The iterative version always uses O(1) space — always prefer it in interviews unless asked for recursion.

Solution 2 — Recursive (correct one-child recursion, O(h) time, O(h) space)

def closestValue(root, target: float) -> int:
if not root: return float('inf') ← Base: off the tree, return sentinel
if target == root.val: return root.val ← Exact match — can't do better than 0 diff
if target < root.val: ← Navigate left only
child = closestValue(root.left, target)
else: ← Navigate right only
child = closestValue(root.right, target)
if child == float('inf'): return root.val ← No child in that direction
if abs(child - target) < abs(root.val - target):
return child ← Child wins
return root.val ← Root wins
🚫 Never Recurse Into Both Children

A common beginner mistake: recursing into BOTH left and right, then comparing. That is O(n) — it visits every node and completely ignores the BST ordering property. Always recurse into exactly ONE child, chosen by the target comparison.

💡 Complexity Summary

Inorder traversal + min scan: Time O(n), Space O(n) — never use
Recursive (one child): Time O(h), Space O(h) call stack
Iterative (Solution 1) ✓: Time O(h), Space O(1) — interview pick
h = O(log n) balanced BST, O(n) worst-case skewed

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Spot the BST Signal

The problem says "BST" and asks you to "find closest." BST = sorted structure. "Find closest" = navigate toward target tracking best candidate. This is the classic BST property pattern: use ordering to prune subtrees.

Step 2: Mention Brute Force First

"I could do an inorder traversal — it visits all n nodes in sorted order, then I scan for minimum difference. O(n) time, O(n) space. This ignores the BST structure entirely." Say this, then immediately offer to optimize.

Step 3: Derive the O(h) Insight

"Since the tree is sorted, when target is less than the current node, all right-subtree values are farther from target — I can skip the entire right subtree. I just walk from root toward target, updating my best at each node. That's O(h) — one step per level."

Step 4: Choose Iterative

"I'll implement this iteratively for O(1) space — no call stack needed, just a while loop until node is None." This shows you care about space complexity, not just time.

🗣️ What to Say to the Interviewer

"This is a BST, so at every node I can determine which direction to go.
If target is less than the current node, every right-subtree value is farther from target.
I walk a single path from root to null, keeping track of the closest value seen.
Time: O(h) — one node per level of the tree.
Space: O(1) iterative — no extra data structures or call stack."
🔮 Likely Follow-up Questions
  • "Find k closest values?" → Inorder traversal + sliding window of size k (#272, Hard)
  • "Not a BST, just a binary tree?" → Full O(n) DFS/BFS — can't prune anything
  • "Guarantee O(log n)?" → Only if tree is balanced (AVL, Red-Black); mention self-balancing BSTs
  • "Handle negative targets or values?" → The algorithm works with any numeric comparison — no special handling needed

BST ordering lets you prune one entire subtree at every step — reduces O(n) to O(h)
Track closest as you walk; every node on the path is a candidate
Direction rule: target < node.val → go left; else go right
Iterative (O(1) space) beats recursive (O(h) stack) — always prefer it
O(log n) for balanced BST; O(n) worst case for skewed (degenerate) BST

🎯 When to Recognize "Track Closest While Searching"
  • "Find node in BST closest to X" → Walk BST toward X, update best
  • "BST floor/ceiling/predecessor/successor" → Same navigation, different update condition
  • "Search for X but X might not exist" → Record nearest as you descend
  • Any BST + "find nearest match" combination
