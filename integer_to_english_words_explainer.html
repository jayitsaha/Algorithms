LeetCode #273

Integer to English Words

Convert a non-negative integer to its English words representation. 1,234,567 → "One Million Two Hundred Thirty Four Thousand Five Hundred Sixty Seven".

Hard Recursion Recursive Grouping by 1000s Time: O(1) Space: O(1)
1 Understanding the Problem ▾

Given a non-negative integer num up to 2,147,483,647 (INT_MAX), convert it to its English words string. The problem looks daunting because it seems like endless special cases — ones, teens, tens, hundreds, thousands, millions, billions. But there is a powerful recursive pattern hiding in plain sight.

💡 Key Insight

Every English number is built from groups of three digits. 1,234,567 = 1 (million) + 234 (thousand) + 567 (ones). Each group is itself a three-digit number in English, then we append the scale word ("Million", "Thousand"). We only need one helper that converts 0–999 to words, and then apply it to each chunk with the right suffix.

2 The Algorithm: Recursive Grouping by 1000s ▾

The strategy is a divide-and-conquer by thousands. We split the number into 3-digit chunks from the largest scale down, convert each chunk with a recursive helper, then concatenate with the scale word. The helper itself handles the hundred/tens/ones sub-structure.

1 Base case: num = 0. Return "Zero". This is the only time "Zero" appears in output — never from inside the grouping logic.
2 Build lookup tables. ones[0..19] for direct names — ones[0]="" handles implicit zeros silently. tens[0..9] for multiples of ten (tens[2]="Twenty", tens[3]="Thirty", etc.).
3 Write helper(n) for 0–999. n=0 → "". n<20 → ones[n]. n<100 → (tens[n//10] + " " + ones[n%10]).strip(). Otherwise → (ones[n//100] + " Hundred " + helper(n%100)).strip().
4 Peel scale groups. Process Billion (÷10⁹), Million (÷10⁶), Thousand (÷10³), then ones remainder. For each non-zero chunk, append helper(chunk) + scale name to parts list.
5 Join and return. " ".join(parts). Trailing/leading whitespace from empty ones[0] strings is cleaned by .strip() inside helper.

🔐 Key Pattern: ones[0] = ""

The ones array uses "" at index 0 so that helper(0) returns "" silently. This prevents "One Hundred Zero" or "One Thousand Zero Hundred" — zero sub-components are naturally invisible. The true zero input is caught at the very top with a special return of "Zero".

3 Why It Works: The Invariant ▾
🔐 The Invariant

At any point during the algorithm, helper(n) correctly converts any integer 0–999 to English words (or "" for 0). This guarantee propagates upward: since each thousands chunk is guaranteed to be in [0, 999], and helper is correct for that range, the outer scale-group loop is correct by composition.

Why does the grouping work? English uses the short scale system — Billion = 10⁹, Million = 10⁶, Thousand = 10³. INT_MAX ≈ 2.1 × 10⁹ so only four tiers suffice. Integer division by each scale extracts the chunk exactly; modulo removes it from the remainder. The chunks are independent.

Proof that helper(n) is correct for all 0–999:

0 n = 0: Returns "". Base case. Correct by definition.
1 1–19: Direct ones[] lookup. All 19 English words are irregular (no arithmetic pattern), so we must store them explicitly. The lookup is exact.
2 20–99: Regular structure: tens[n//10] gives the decade ("Forty"), ones[n%10] gives the digit ("Seven"). .strip() handles n%10 = 0 (e.g., 40 → "Forty ", strip → "Forty").
3 100–999: ones[n//100] gives the hundreds digit, " Hundred " is the fixed word, then helper(n%100) recursively handles [0,99] which is already correct. Induction holds.

Edge case verification:

  • num = 0 → "Zero" (top-level special case before the loop)
  • num = 100 → chunk=100, helper(100): ones[1]+"Hundred"+helper(0) → "One Hundred" + "" → strip → "One Hundred"
  • num = 1000 → chunk_T=1, helper(1)="One", append "One Thousand". chunk_O=0, skip.
  • num = 1000000 → chunk_M=1, "One Million". chunk_T=0, skip. chunk_O=0, skip.
  • num = 1000010 → "One Million Ten". The zero thousands chunk is skipped cleanly.
⚠️ Common Mistake: ones[0] = "Zero"

If you set ones[0] = "Zero", then helper(100) calls helper(0) = "Zero", giving "One Hundred Zero". Similarly, numbers like 1,000,000 would produce "One Million Zero Thousand Zero". Always keep ones[0] = "" and handle global zero separately.

⚠️ Why .strip() appears in multiple places

When ones[n%10] = "" (e.g., n=40), concatenation gives "Forty " with a trailing space. When n%100=0 in the hundreds branch, we get "Three Hundred ". .strip() normalizes all of these. Without it, the final join would contain double spaces and you'd fail test cases.

⏱️ Complexity Analysis

Time: O(1) — The number has at most 10 decimal digits (INT_MAX). This means at most 4 scale groups, and helper is called at most 4 times (once per chunk) with O(1) work each. The total number of operations is bounded by a constant regardless of input value.

Space: O(1) — The lookup tables are fixed-size constants. The output string has a bounded maximum length (~66 characters for 1,000,000,000 → "One Billion"). Call stack depth is at most 2 (helper calling helper for 100–999 sub-problem).

Brute force approach (for completeness): You could write explicit conditionals for every possible digit position and combination. This technically works but requires ~100+ branches, is error-prone, hard to extend (adding Trillion), and impossible to maintain. The grouping approach above is strictly superior in every dimension.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify the repeating structure

When you see a problem that seems like "hundreds of cases", look for repetition. Ask: "Is there a group of numbers that all get converted the same way?" Yes — every three-digit group is converted identically. That's your subproblem.

Step 2: Define the scale levels explicitly

INT_MAX ≈ 2.1 × 10⁹, so Billion is the maximum scale. State this clearly: "I'll define four scale levels — Billion, Million, Thousand, ones — and process from largest to smallest." This shows you understand the input range.

Step 3: Handle edge cases before writing helper

Mention the two critical edge cases before coding: (a) num=0 must return "Zero" specifically, and (b) ones[0] must be "" not "Zero" to avoid corrupting larger numbers. Interviewers love seeing edge case awareness upfront.

Step 4: Test helper independently on small cases

Before using helper in the outer loop, trace through: helper(0)="", helper(5)="Five", helper(12)="Twelve", helper(21)="Twenty One", helper(100)="One Hundred", helper(115)="One Hundred Fifteen". If all pass, the outer loop is trivially correct.

🗣️ What to Say to the Interviewer

"The key insight is that English number words group in threes by scale.
I'll build a helper that converts any three-digit number 0–999 to words
using two lookup tables: ones (0–19 including teens) and tens (20–90).
Then in the outer loop I process four scales — Billion, Million, Thousand, ones —
extracting each chunk with integer division, skipping zero chunks,
and joining non-empty parts with spaces.
Two critical edge cases: num=0 returns 'Zero' before the loop,
and ones[0]='' (not 'Zero') so that helper(0) returns '' silently."
💡 Common Follow-up Questions
  • Extend to Trillion? — Add (10**12, "Trillion") to the scales list. Nothing else changes.
  • Handle negative numbers? — Prepend "Negative " and negate num if negative. Two lines.
  • Handle a huge number given as a string? — Chunk the string in groups of 3 from the right, process each chunk with helper, append the scale word. Same grouping logic.
  • Can you do it without recursion? — Yes: unroll helper into three if-branches with an iterative structure instead of recursive call for the sub-hundreds portion.

7 Summary & Key Takeaways ▾
English numbers group in threes by scale — exploit that structure with a recursive helper for 0–999.
ones[0] = "" (not "Zero") prevents spurious "Zero" tokens appearing inside larger numbers.
Special-case num = 0 at the very top — it's the only time "Zero" should appear in output.
.strip() on every concatenation handles trailing spaces when sub-parts (ones[n%10] or helper sub-call) are empty.
Skip zero chunks entirely — never produce "Zero Thousand" or "Zero Million".
Adding Trillion requires one extra tuple in the scales list. The rest is unchanged — this is the power of the pattern.
