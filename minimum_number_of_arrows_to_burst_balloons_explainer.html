LeetCode #452

Minimum Number of Arrows to Burst Balloons

Find the minimum number of arrows to burst all balloons, where each balloon spans an interval [x_start, x_end] on the x-axis.

Medium Intervals Greedy · Sort by End Time: O(n log n) Space: O(1)
1 Understanding the Problem ▾

You are given an array of balloons. Each balloon i spans from x_start to x_end on the x-axis (inclusive). You shoot arrows vertically upward from position x. An arrow at x bursts any balloon where x_start ≤ x ≤ x_end. One arrow can burst multiple overlapping balloons simultaneously. What is the minimum number of arrows needed to burst every balloon?

💡 Key Insight

A single arrow can burst multiple balloons if they all share a common x-coordinate. So the question is really: what is the minimum number of "overlap groups" we need, where each group of balloons all intersect at one point? Each group needs exactly one arrow.

Worked example: points = [[10,16],[2,8],[1,6],[7,12]]

✓ Optimal: 2 arrows

Arrow 1 at x=6:
  bursts [1,6] ✓ and [2,8] ✓
Arrow 2 at x=11:
  bursts [7,12] ✓ and [10,16] ✓
Total: 2 arrows

✗ Naive: one arrow each

Arrow at x=1: bursts [1,6]
Arrow at x=2: bursts [2,8]
Arrow at x=7: bursts [7,12]
Arrow at x=10: bursts [10,16]
Total: 4 arrows (wasteful!)

Critical edge cases:

  • Touching endpoints: [1,4] and [4,6] overlap at x=4 — one arrow suffices (the condition uses ≤, not <)
  • Single balloon: always needs exactly 1 arrow
  • All identical: [[5,10],[5,10],[5,10]] → 1 arrow
  • None overlapping: n non-intersecting balloons → n arrows
  • Empty input: 0 arrows (check before sorting)
2 The Algorithm: Greedy — Sort by End, Scan Once ▾

Sort balloons by their right endpoint. Then make a single left-to-right pass: maintain arrow_pos, the position of the last arrow fired. When a balloon's start exceeds arrow_pos, fire a new arrow at that balloon's end. Otherwise, the current arrow already covers it.

1 Handle empty input. If points is empty, return 0. No balloons, no arrows.
2 Sort by end coordinate: points.sort(key=lambda x: x[1]). This is the crucial greedy decision — we want to process the most "urgent" balloon first (the one that ends soonest, giving us the least flexibility).
3 Initialize: arrows = 1, arrow_pos = points[0][1]. We always need at least one arrow. We fire it at the end of the first (earliest-ending) balloon — this is the latest position that still bursts it, and maximizes the chance of also bursting later balloons.
4 Scan from index 1: For each balloon [start, end]: if start > arrow_pos, this balloon is beyond our current arrow's reach. Fire a new arrow at end, increment arrows, update arrow_pos = end. Otherwise (start ≤ arrow_pos), it's already burst — skip.
5 Return arrows. The count equals the number of disjoint overlap groups found — each group shares at least one x-coordinate, requiring exactly one arrow.
🔑 Why Sort by End, Not Start?

Imagine a very long balloon [1, 100] alongside many short ones like [2,5], [3,6], [4,7]. If we sort by start, the long balloon comes first and we shoot at x=100 — missing all the short ones clustered near x=5 and wasting arrows. Sorting by end puts the short (tightest) balloons first, so we shoot at x=5 and burst all of them with one arrow.

3 Why It Works: The Invariant ▾
🔐 The Greedy Invariant

After each arrow is fired: every balloon processed so far has been burst, and arrow_pos is exactly the position of the most recent arrow. Any future balloon with start ≤ arrow_pos is guaranteed to also have been burst by this arrow (because its interval contains arrow_pos).

Exchange argument (proof that shooting at end is optimal): Suppose the optimal solution fires an arrow at position p to burst balloon B (among others). We can move this arrow to B.end (if p < B.end) without losing coverage: every balloon that overlapped with position p also overlaps with any position in [p, B.end] (since their starts are ≤ p ≤ B.end). Moving right can only gain more balloons whose start falls between p and B.end. So shooting at B.end is always at least as good as any position before it — never worse.

Why we don't update arrow_pos when a balloon overlaps: When a new balloon's start ≤ arrow_pos, we know it's already burst. If we updated arrow_pos = end here, we might shrink our coverage (the new balloon might end earlier than our current arrow, pulling arrow_pos left and causing future balloons to be incorrectly classified as needing new arrows). We leave arrow_pos alone — the original arrow position is correct.

⚠️ The > vs >= Trap: Touching Endpoints

Two balloons [1,4] and [4,6] share the single point x=4. One arrow at x=4 bursts both. Our condition is start > arrow_pos (strictly greater), so start=4 > arrow_pos=4 is False — correctly no new arrow. Using >= would fire an unnecessary second arrow.

Correctness for non-overlapping groups: When we find start > arrow_pos, we've definitively identified a new group — no matter where we shoot in this group, it cannot reach any previous balloon (their ends are all ≤ arrow_pos < start). This ensures we count every distinct group exactly once.

4 The Code (Annotated) ▾

Solution 1 — Greedy Sort by End (Optimal, O(n log n) time, O(1) space — Interview Pick)

def findMinArrowShots(points):
if not points: ← Guard: no balloons means 0 arrows needed
return 0
points.sort(key=lambda x: x[1]) ← Sort by END coordinate — the greedy key decision
arrows = 1 ← At least 1 arrow is always needed if non-empty
arrow_pos = points[0][1] ← Shoot at end of first (earliest-ending) balloon
for start, end in points[1:]: ← Check every remaining balloon
if start > arrow_pos: ← This balloon is NOT reached by current arrow
arrows += 1 ← New disjoint group → fire a new arrow
arrow_pos = end ← New arrow shoots at this balloon's end
# else: start <= arrow_pos → burst ← Implicit: current arrow covers this balloon
return arrows ← Count of disjoint overlap groups = min arrows
⚠️ Why NOT update arrow_pos when balloons overlap?

If balloon [3,9] overlaps with our arrow at x=6, we should NOT set arrow_pos = 9. That would suggest we can cover up to x=9 with this arrow — but our arrow is at x=6, not x=9. Updating would break the invariant and cause future balloons between x=6 and x=9 to be incorrectly skipped.

Solution 2 — Sort by Start (Alternative, same complexity, different style)

def findMinArrowShots_alt(points):
if not points: return 0
points.sort() ← Sort by start (and end as tiebreak)
arrows = 1
cur_end = points[0][1] ← Track the minimum end seen in current group
for start, end in points[1:]:
if start > cur_end: ← No overlap with current group's minimum end
arrows += 1
cur_end = end ← Reset to this balloon's end
else:
cur_end = min(cur_end, end) ← Shrink to common overlap region's right bound
return arrows
💡 Complexity Analysis

Greedy (Sort by End): O(n log n) time (sort dominates), O(1) extra space (in-place sort in Python modifies the list; we only use a few variables).

Greedy (Sort by Start): Same O(n log n) / O(1). Slightly more complex — must track minimum end of current group, not just fixed arrow position.

Why can't we do O(n)? Sorting is a necessary step, and comparison-based sorting has an Omega(n log n) lower bound. If coordinates are bounded, counting sort could give O(n + range) but "range" can be 2^31 here.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Interval Scheduling Pattern

Key signals: the problem involves intervals, asks for "minimum number of [something]" to cover/hit all intervals. This is classic interval scheduling — greedy with sorting almost always works here.

Step 2: Think About What One Arrow Can Achieve

"One arrow can burst all balloons it passes through — all those sharing a common point." This reframes the problem as: minimize the number of disjoint groups, where balloons in the same group share at least one common x-coordinate.

Step 3: Justify Sorting by End

"I want to process the most constrained balloon first — the one with the earliest end gives me the least flexibility. I should fire an arrow as far right as possible while still hitting it (at its end), to maximize coverage of future balloons."

Step 4: Trace Through the Example

Always trace through a small example on the whiteboard before coding. Sorted: [[1,6],[2,8],[7,12],[10,16]]. Arrow at 6 → bursts first two. Arrow at 12 → bursts last two. Answer: 2. Matches expectation.

🗣️ What to Say to the Interviewer

"I recognize this as an interval problem — I want to minimize the number of arrows, so I want each arrow to cover as many balloons as possible.

My greedy insight: sort by end coordinate. The balloon ending soonest is most constrained — I must shoot at or before its end. Shooting at exactly its end maximizes the chance of also catching later-starting balloons.

Then I scan: whenever a new balloon's start exceeds my last arrow position, I fire a new arrow at this balloon's end. Otherwise, it's already burst.

Time: O(n log n) for sort, O(n) for the scan = O(n log n) overall. Space: O(1)."
🔮 Likely Follow-up Questions
  • "What if arrows can't be at fractional positions?" → Same solution; x-coordinates are integers (endpoints are integers, and optimal arrow is always at an integer endpoint).
  • "What if you cannot modify the input?" → Use sorted(points, key=lambda x: x[1]) — costs O(n) extra space for the sorted copy.
  • "How many intervals must be removed so none overlap?" → LeetCode #435 — identical greedy logic, different counting.
  • "What's the maximum number of balloons one arrow can hit?" → A different problem; requires checking all possible overlap groups.

7 Summary & Key Takeaways ▾
Sort intervals by end coordinate — the most constrained balloon (earliest end) must be addressed first
Shoot each arrow at the end of the first unpopped balloon in sorted order — maximizes what it bursts
Only fire a new arrow when a balloon's start strictly exceeds the current arrow position
Touching endpoints count as overlapping — condition is start > arrow_pos, not >=
Never update arrow_pos when a balloon overlaps — that would shrink coverage, not expand it
This is equivalent to counting disjoint overlap groups — a fundamental interval scheduling concept
