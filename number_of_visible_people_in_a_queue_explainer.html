LeetCode #1944

Number of Visible People in a Queue

There are n people in a line. For each person, count how many others to their right they can see — a taller person standing between two people blocks visibility.

Hard Monotonic Stack Decreasing Stack from Right Time: O(n) Space: O(n)
1 Understanding the Problem ▾

There are n people in a queue with distinct heights. Person i can see person j (where j > i) if and only if every person standing between them has a height less than min(heights[i], heights[j]). In plain English: no one in between is as tall as either of them.

💡 Key Insight

The first person to the right of i who is taller than heights[i] acts as an absolute wall — person i can still see them, but cannot see anyone beyond that wall. Everyone shorter that person i encounters before hitting that wall is also visible.

Worked Example: heights = [10, 6, 8, 5, 11, 9]

Person 0 (h=10) → Sees person 1 (h=6, shorter ✓), person 2 (h=8, shorter ✓), person 4 (h=11, taller — last visible, blocks beyond). Count = 3
Person 1 (h=6) → Sees person 2 (h=8, taller — blocks immediately). Count = 1
Person 2 (h=8) → Sees person 3 (h=5, shorter ✓), person 4 (h=11, taller — last visible). Count = 2
Person 3 (h=5) → Sees person 4 (h=11, taller). Count = 1
Person 4 (h=11) → Sees person 5 (h=9, shorter). Count = 1
Person 5 (h=9) → Rightmost person, no one to see. Count = 0

Expected output: [3, 1, 2, 1, 1, 0]

Edge Cases: All same height → each person sees exactly one person to their right. Strictly decreasing → every person can see all to their right. Single person → [0]. Strictly increasing → each sees exactly their immediate neighbour (then a taller wall appears).

2 The Algorithm: Monotonic Decreasing Stack (Right → Left) ▾

We scan from right to left. For each person i, we want to answer: "how many people to my right can I see?" A monotonic decreasing stack stores exactly the people that still matter — those that haven't been "blocked" by someone taller who appeared later in the queue.

🔑 Stack Invariant

After processing position j, the stack contains the heights of persons at positions ≥ j that could potentially be seen by anyone to their left — stored in strictly decreasing order from bottom to top. Any height that is ≤ a later height in the stack would be permanently blocked and is never stored.

1 Scan i from n-1 down to 0. Initialize count=0 for each person.
2 Pop while stack-top < heights[i]. Person i can see over each shorter person we pop — increment count per pop. Popped persons are removed because person i now "overshadows" them for everyone to the left of i.
3 If stack is non-empty, add 1. The remaining top is ≥ heights[i]. Person i can see that person, but cannot see past them (they're taller, so they're a wall). This +1 is critical and easy to forget.
4 Set ans[i] = count. Push heights[i]. Person i is now in the stack as a candidate for people to their left.

Why does each element get pushed/popped at most once? Each of the n heights is pushed exactly once (when we process position i) and popped at most once (when someone to the left is taller). Total work is O(2n) = O(n).

3 Why It Works: The Invariant & Proof ▾

Claim: When we process person i, every element popped from the stack is a person that i can directly see, and the remaining stack top (if any) is also visible to i but blocks everyone beyond it.

Why popped elements are visible: The stack is decreasing. When we pop value v (because v < heights[i]), there is no element between i and the position of v that is ≥ v — if there were, it would already be in the stack below v, contradicting the decreasing property. So no one blocks the line of sight between i and v. Person i can see person at position of v.

Why the remaining top is visible but blocking: Once we stop popping, stack top w ≥ heights[i]. The stack is decreasing, so there is nothing between the current position of i and the position of w that is ≥ w. So person i can see person at w. But since w ≥ heights[i], everyone beyond w is blocked from i's perspective (a wall at least as tall as i stands in the way).

⚠️ The +1 Bug — Most Common Interview Mistake

Beginners only count the popped elements and forget that the remaining taller stack top is ALSO visible. Always: count += (1 if stack else 0) after the while-loop. Missing this produces wrong answers on any non-monotone input where a taller blocker exists.

🔑 Why Not Left-to-Right?

Going left to right, when we process person i, we don't yet know what's to their right. We'd need to either pre-compute or scan forward. Going right to left, by the time we reach person i, the entire right side is already summarized in the stack — a perfect fit for this query direction.

# TLE for n=100,000

✓ Monotonic Stack — O(n)

Each element: 1 push + at most 1 pop.
Total operations: O(2n) = O(n).
Stack space: O(n) worst case
(strictly decreasing heights).

Accepted for n up to 100,000.

def canSeePersonsCount(heights: list[int]) -> list[int]:
n = len(heights) → total people in queue
ans = [0] * n → answer array, default 0 for each person
stack = [] → monotone decreasing stack of heights (right-side candidates)
for i in range(n - 1, -1, -1): → right to left: process each person
count = 0 → reset visibility count for this person
while stack and stack[-1] < heights[i]: → pop everyone person i can see over
stack.pop() → remove this shorter person from candidates
count += 1 → person i CAN see this shorter person
if stack: → a taller (or equal) blocker still exists to the right
count += 1 → person i sees this taller person too (but not beyond)
ans[i] = count → record total visible for person i
stack.append(heights[i]) → push i onto stack as candidate for people to the left
return ans
⚠️ Why heights not indices on the stack?

We only need height values to do the comparison stack[-1] < heights[i]. We don't need the actual positions because visibility is determined purely by relative heights — there's no index lookup needed. If a problem required knowing which specific person was visible, we'd store indices instead.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify the Pattern — "Blocking" = Monotonic Stack

The moment a problem says one element "blocks" visibility of everything behind it, you should think: Monotonic Stack. The classic signals are: heights/buildings, "next greater element", "how many can you see". This problem combines all three.

Step 2: Verbalize the Brute Force

For each person i, scan right and track max_height. A person at j is visible if heights[j] > max_height OR j == i+1 (first neighbour is always visible). Update max_height = max(max_height, heights[j]). This is O(n²) but confirms you understand the problem.

Step 3: Observe Redundancy — The Stack Optimization

Going left to right, each person repeatedly re-scans the right side. Instead, if we go right to left and maintain a decreasing stack, the stack already summarizes "who matters on the right" — each pop answers one visibility query in O(1) amortized.

Step 4: Handle the Taller-Blocker Edge Case Explicitly

After the while-loop, if the stack is non-empty, there is one more visible person: the first one taller than you. Mention this explicitly — it shows you've thought about the case deeply rather than just pattern-matching.

🗣️ What to Say

"Brute force would be nested loops — O(n²). For each i, scan right until height[j] is taller than heights[i], counting who's visible."

"I notice that when I process person i from left to right, I'm repeatedly doing the same right-side scan. If I flip to right-to-left processing, I can maintain a stack of right-side candidates."

"The stack stays decreasing. When I process person i, I pop everything shorter — each pop is one visible person. If something taller remains, that's one more visible person but it blocks the rest."

"Each height enters and exits the stack at most once, so it's O(n) overall."

7 Summary & Key Takeaways ▾
Process right to left — by the time you handle person i, the right side is already summarized in the stack.
The stack is monotone decreasing: every element in it is a real visibility candidate from the left; no redundant shorter elements exist.
Each pop during the while-loop = one visible person (shorter, directly visible, now blocked by person i for future leftward queries).
If the stack is non-empty after the while-loop, add exactly 1 more — this is the first taller person who blocks the rest.
Total time O(n): each element pushed once, popped at most once — amortized O(1) per person despite the nested while-loop.
