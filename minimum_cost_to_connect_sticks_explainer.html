LeetCode #1167

Minimum Cost to Connect Sticks

Given an array of stick lengths, return the minimum cost of connecting all sticks into one. The cost of connecting two sticks is the sum of their lengths.

Medium Heaps Min Heap Greedy Time: O(n log n) Space: O(n)
1 Understanding the Problem ▾

You have an array of sticks, each with a positive integer length. You can pick any two sticks and connect them — the cost of that connection equals the sum of their lengths, and the result is a single new stick of that length. Repeat until one stick remains. Minimize the total cost.

💡 Key Insight

Every time you connect two sticks into one, you pay their combined length. The resulting composite stick participates in all future connections. So a stick's length contributes to cost once for every connection it is part of. To minimize total cost, you want large values to appear in as few connections as possible — which means merging the smallest sticks first so that only small values grow.

✗ Merge Largest First (BAD)

sticks = [1, 8, 3, 5]
Merge 8+5=13 → cost 13
Merge 13+3=16 → cost 16
Merge 16+1=17 → cost 17
Total = 46

✓ Merge Smallest First (OPTIMAL)

sticks = [1, 8, 3, 5]
Merge 1+3=4 → cost 4
Merge 4+5=9 → cost 9
Merge 9+8=17 → cost 17
Total = 30

Edge Cases: Single stick — cost is 0 (nothing to merge). Two sticks — cost is exactly their sum. All sticks of equal length — any merge order gives the same result. Very large n — needs efficient O(log n) min-extraction, not O(n) linear scan each step.

2 The Algorithm: Min Heap Greedy ▾

The greedy strategy is: always merge the two smallest sticks. To do this efficiently at every step, we use a min-heap (priority queue). A heap lets us extract the minimum in O(log n) and re-insert in O(log n).

1 Heapify: Convert the input array into a min-heap in O(n). The smallest stick is at index 0.
2 Loop (n-1 times): Pop the two smallest sticks a and b. Compute merged = a + b. Add merged to total cost. Push merged back onto the heap.
3 Return total: After n-1 iterations the heap contains one element (the final stick). Return the accumulated total cost.
Greedy Decision at Every Step
Which two to merge? → Always the TWO SMALLEST (min-heap gives them in O(log n))
Why not merge two large ones? → Creates an even larger stick; that value will be re-counted in all future merges
🔐 Connection to Huffman Coding

This problem is structurally identical to Huffman coding — the classic data compression algorithm. Huffman builds an optimal prefix tree by greedily merging the two least-frequent symbols using a min-heap. Same structure, same proof of optimality. Mentioning this in an interview earns bonus points.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every step, the heap holds exactly the current set of "available sticks" — some original, some composites from prior merges. Extracting the two smallest and merging them is always locally and globally optimal.

Exchange Argument (Proof of Optimality): Suppose an optimal solution performs a merge at step k that does NOT pick the two smallest sticks. We can "swap" that merge with an earlier merge that would have used the two smallest, and this swap cannot increase the total cost. (The smaller values participate in fewer merges, so their contribution to total cost is lower.) Therefore any deviation from the greedy choice can be transformed into the greedy solution without increasing cost — the greedy is optimal.

Why a heap and not a sorted array? After each merge, the composite stick needs to be reinserted in sorted position. An array would require O(n) shifts. A heap reinserts in O(log n) — critical for O(n log n) total complexity.

⚠️ Common Trap: Sort Once, Then Pick Front

Sorting once and always merging the first two elements fails because after pushing the composite back, it might not be the smallest — you'd need to re-sort or binary-search for its position. The heap maintains the ordering automatically with every push/pop.

4 The Code (Annotated) ▾

Python's heapq is a min-heap by default — perfect for this problem. No custom comparator needed.

import heapq ← Python's built-in min-heap module (smallest element always at index 0)
def connectSticks(sticks):
heapq.heapify(sticks) ← O(n): convert list to min-heap IN-PLACE. sticks[0] is always the minimum.
total = 0 ← running sum of all merge costs
while len(sticks) > 1: ← n-1 iterations; exits when only the final stick remains
a = heapq.heappop(sticks) ← O(log n): remove and return the SMALLEST value; heap re-adjusts
b = heapq.heappop(sticks) ← O(log n): remove the new minimum (second smallest overall)
merged = a + b ← cost of this merge = length of the new composite stick
total += merged ← accumulate the merge cost; merged also becomes the new stick's length
heapq.heappush(sticks, merged) ← O(log n): insert composite back so it can be merged in a future step
return total ← final answer: sum of all n-1 merge costs
💡 Complexity Summary

Time: O(n log n) — heapify O(n) + (n-1) iterations × O(log n) per pop/push.
Space: O(n) — heap holds all sticks in-place (we mutate the input array); technically O(1) extra.

⚠️ Edge Case: Single Stick

If len(sticks) == 1, the while condition len(sticks) > 1 is false immediately. The loop doesn't execute. total = 0 is returned. This is correct — connecting zero pairs costs nothing.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify the Signal — "Repeated Minimum"

Whenever a problem says "repeatedly pick/combine the two smallest (or largest)" — that's a heap signal. Here: we always want the two lightest sticks. Min-heap hands them to us in O(log n) at each step.

Step 2: Justify the Greedy — Say This Aloud

"Every merged stick participates in all future merges. A large stick grows the running total more than a small one. By merging small first, I keep large sticks out of the game as long as possible — minimizing re-counting." This exchange argument satisfies the interviewer that you understand correctness, not just mechanics.

Step 3: State Complexity Before Coding

Brute force: sort the array, merge the first two, re-sort → O(n² log n). With a heap: O(n log n) total — each of the n-1 merges is O(log n). State this before writing code so the interviewer sees your complexity thinking upfront.

Step 4: Optional Bonus — Huffman Connection

If you know Huffman coding from CS theory: mention it. "This is exactly Huffman coding — build a binary tree by merging the two least-frequent nodes; here lengths replace frequencies." Most interviewers will reward this observation.

🗣️ What to Say to the Interviewer

"I notice that each merge cost propagates — the result participates in all future merges. So I want large sticks to be merged last. Greedy strategy: always merge the two smallest. To get the two smallest efficiently at every step, I'll use a min-heap. heapq.heapify gives me O(n) setup, then each of the n-1 iterations is O(log n) — total O(n log n). Let me trace: [2,4,3] → heapify → [2,3,4]. Pop 2,3 → cost 5, push 5 → [4,5]. Pop 4,5 → cost 9, push 9 → [9]. Done. Total = 14."
🔎 Follow-up Questions
  • "Can you solve with a sorted deque instead?" — Yes: after merging, insert composite maintaining sorted order with binary search. Same O(n log n), but more code.
  • "What if we can merge k sticks at once?" — Pop k each iteration instead of 2; heap still applies.
  • "What's the connection to Huffman coding?" — Same greedy, same proof, different domain.
  • "Why not use a max-heap?" — We want minimums, not maximums. Max-heap would give us the wrong pair.

7 Summary & Key Takeaways ▾
Merge the two smallest sticks first — they contribute least to future merges
Min-heap gives O(log n) extract-min at every step without re-sorting
n-1 total merges: O(n log n) overall; O(n) heapify + O(n log n) loop
Proven optimal via exchange argument: any non-greedy merge can be swapped for a greedy one without increasing cost
Isomorphic to Huffman coding — both use the same greedy min-heap structure
ApproachTimeSpaceNotes
Sort + Re-sort each merge O(n² log n) O(n) Re-sorts entire array each step — prohibitive
Sort + Deque (binary search insert) O(n log n) O(n) Works but more code; O(n) shifts for insertion
Min-Heap (heapq) ✓ O(n log n) O(n) Optimal; cleanest; interview pick
🎯 When to Recognize the Min Heap Greedy Pattern
  • "Repeatedly pick the two smallest/largest and combine" → Min/Max Heap
  • "Merge result re-enters the pool" → Push merged value back to heap
  • "Minimize total cost of sequential operations" → Greedy + Heap
  • "Top K / K smallest / K largest" → Fixed-size heap of size K
  • "Scheduling, task allocation with priorities" → Priority queue (heap)
