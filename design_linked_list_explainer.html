LeetCode #707

Design Linked List

Implement a doubly linked list supporting get, addAtHead, addAtTail, addAtIndex, and deleteAtIndex — with zero edge-case bugs.

Medium Linked List Sentinel Nodes Time: O(n) per op Space: O(n)
1 Understanding the Problem ▾

You are building a linked list from scratch. It must support five operations:

  • get(index) — return the value at 0-based index, or −1 if out of bounds
  • addAtHead(val) — insert a new node with val at the front
  • addAtTail(val) — append a new node at the back
  • addAtIndex(index, val) — insert before the node at index; if index = size, append; if index > size, do nothing
  • deleteAtIndex(index) — remove the node at index; if out of bounds, do nothing
💡 Key Insight: Sentinel Nodes Eliminate All Edge Cases

The hardest part of a linked list is boundary handling: what do you do when the list is empty? When you insert at the head? When you delete the only node? A doubly-linked list with two permanent sentinel (dummy) nodes — one at each end — makes every single operation identical. You always insert or delete between two existing nodes, because the sentinels are always there.

✗ Without Sentinels — Edge Cases

addAtHead to empty list → update self.head
addAtTail to empty list → update self.tail
delete head → update self.head
delete tail → update self.tail
delete only node → null both
→ 5+ separate special cases

✓ With Sentinel Nodes — One Pattern

dummy_head ↔ [real nodes] ↔ dummy_tail
addAtHead → insert_after(dummy_head)
addAtTail → insert_after(dummy_tail.prev)
delete any → bypass between its neighbours
empty list: dummy_head.next IS dummy_tail
→ 1 universal pattern, zero special cases

Edge cases to consider: get on a negative or ≥ size index returns −1. addAtIndex when index = size is a valid append; index > size is a silent no-op. deleteAtIndex on an out-of-bounds index is a silent no-op.

2 The Algorithm: Doubly-Linked List with Sentinels ▾

The structure at all times: dummy_head ↔ node_0 ↔ node_1 ↔ … ↔ node_(n-1) ↔ dummy_tail. Every real node has a non-null prev (at least dummy_head) and a non-null next (at least dummy_tail).

1 Initialise: Create dummy_head and dummy_tail. Wire them: dummy_head.next = dummy_tail, dummy_tail.prev = dummy_head. Set self.size = 0. These two nodes are permanent — never removed.
2 get(index): Bounds check first (O(1)). If valid, start at dummy_head.next and advance index steps. Return cur.val.
3 addAtHead / addAtTail: Delegate to _insert_after(dummy_head, new) and _insert_after(dummy_tail.prev, new) respectively. Both use the same four-pointer helper.
4 addAtIndex(index, val): Walk index steps from dummy_head to find the predecessor. Call _insert_after(prev, Node(val)). This handles all boundary cases naturally — index 0 leaves prev = dummy_head.
5 deleteAtIndex(index): Walk index steps from dummy_head.next to the target node. Bypass it with two pointer updates: node.prev.next = node.next and node.next.prev = node.prev. Decrement size.
6 _insert_after(prev, new): The universal helper. Four steps in safe order: (1) new.next = prev.next, (2) new.prev = prev, (3) prev.next.prev = new, (4) prev.next = new. Increment size.
🔑 Why Doubly-Linked, Not Singly-Linked?

In a singly-linked list, deleting a node requires knowing its predecessor — you must traverse to index - 1. A doubly-linked list gives node.prev for free. More importantly, the sentinel tail's prev pointer gives you O(1) access to the last real node for addAtTail. The two extra pointer fields per node are worth the gain in simplicity.

3 Why It Works: The Invariant ▾
🔐 The Structural Invariant

At all times: every node in the list has a non-null prev and a non-null next. This holds for real nodes (bounded by their neighbours) and for the sentinels (dummy_head.prev = None, dummy_tail.next = None — their external edges, never touched by insert/delete). The list is always a correctly-wired doubly-linked chain.

Why _insert_after cannot violate this: Given prev (which has a valid next by invariant), we wire new between prev and prev.next. After the four steps, every node still has valid prev and next. The invariant is preserved.

Why delete cannot violate this: We bypass the target by updating its predecessor's next and its successor's prev. The sentinels guarantee there is always a predecessor (at least dummy_head) and a successor (at least dummy_tail). We never touch a null pointer.

⚠️ The Critical Ordering Bug in _insert_after

Step 4 (prev.next = new) must come after Step 3 (prev.next.prev = new). If you swap them: after Step 4, prev.next is now new — so Step 3 becomes new.prev = new, a self-loop. Always save the old successor first: new.next = prev.next (Step 1) captures it before any overwrite.

Solution 1 — Doubly-Linked List with Sentinel Head and Tail (Interview Pick)

class Node: ← Internal data container, nested or standalone
def __init__(self, val=0): ← Default val=0 so sentinels are Node() calls
self.val = val
self.prev = None ← Points to predecessor; None only on external edges
self.next = None ← Points to successor
class MyLinkedList:
def __init__(self):
self.head = Node() ← Dummy sentinel head — permanent, never removed
self.tail = Node() ← Dummy sentinel tail — permanent, never removed
self.head.next = self.tail ← Wire them: empty list looks like H ↔ T
self.tail.prev = self.head
self.size = 0 ← Track count for O(1) bounds checking
def get(self, index: int) -> int:
if index < 0 or index >= self.size: ← O(1) bounds check before any traversal
return -1
cur = self.head.next ← Skip sentinel, start at real node 0
for _ in range(index):
cur = cur.next ← Advance index steps to the target
return cur.val
def addAtHead(self, val: int) -> None:
self._insert_after(self.head, Node(val)) ← Predecessor is always dummy_head
def addAtTail(self, val: int) -> None:
self._insert_after(self.tail.prev, Node(val)) ← Predecessor is last real node (or dummy_head if empty)
def addAtIndex(self, index: int, val: int) -> None:
if index > self.size: ← index > size: invalid (index == size is valid append)
return
prev = self.head ← Walk from sentinel: 0 steps → prev stays dummy_head (addAtHead)
for _ in range(index):
prev = prev.next
self._insert_after(prev, Node(val)) ← Universal helper does the rest
def deleteAtIndex(self, index: int) -> None:
if index < 0 or index >= self.size: ← Bounds check
return
node = self.head.next ← Start at real node 0
for _ in range(index): ← Walk index steps to target
node = node.next
node.prev.next = node.next ← Left side bypass
node.next.prev = node.prev ← Right side bypass
self.size -= 1
def _insert_after(self, prev, new): ← The only insertion code in the entire class
new.next = prev.next ← Step 1: save old successor FIRST
new.prev = prev ← Step 2: wire new backwards
prev.next.prev = new ← Step 3: old successor points back to new (safe: new.next still valid)
prev.next = new ← Step 4: predecessor points to new (do this last!)
self.size += 1
⚠️ addAtIndex Bug with index < 0

The problem guarantees index ≥ 0 for addAtIndex, but if you're being defensive, add if index < 0: index = 0. Without this, walking a negative number of steps is a no-op in Python (range of negative is empty), so prev stays dummy_head — which actually inserts at head. Whether that's a bug or a feature depends on the spec. Be explicit with the interviewer.

Solution 2 — Singly-Linked List with Dummy Head (simpler, valid alternative)

class MyLinkedList:
def __init__(self):
self.head = Node() ← Single dummy head only (no tail sentinel)
self.size = 0
def addAtIndex(self, index, val):
if index > self.size: return
prev = self.head
for _ in range(index): prev = prev.next
new = Node(val)
new.next = prev.next ← Only 2 pointer updates (no prev pointers)
prev.next = new
self.size += 1
def deleteAtIndex(self, index, val):
if index < 0 or index >= self.size: return
prev = self.head ← Walk to predecessor (one extra step vs doubly-linked)
for _ in range(index): prev = prev.next
prev.next = prev.next.next ← Skip over target
self.size -= 1
💡 Complexity

Doubly-Linked (both sentinels): get O(n), addAtHead O(1)*, addAtTail O(1)*, addAtIndex O(n), deleteAtIndex O(n). Space O(n + 2 sentinels).
Singly-Linked (one sentinel): Same asymptotic, but addAtTail is O(n) without a tail pointer. Code simpler.
* O(1) only if you wire directly to sentinels; addAtIndex with index=0 or index=size also walks 0/size steps so it's still O(1) for those cases.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Enumerate the Edge Cases First

Before any code: "The tricky parts are inserting at head/tail, deleting the first/last node, and operating on an empty list. I want a design that makes all of these identical to the general case." This signals you think about correctness first.

Step 2: Propose Sentinel Nodes Explicitly

"I'll use two permanent dummy nodes — a sentinel head and a sentinel tail — that are never inserted or removed. Every real node always has at least the sentinel as a neighbour. This means I never need to special-case boundary conditions."

Step 3: Draw the Four-Pointer Insert Dance

On the whiteboard: draw prev → [old next], then show new being inserted between them. Label the four pointer assignments in order and stress that prev.next = new must come last. This is a classic signal of deep understanding.

Step 4: Show How All 5 Operations Reduce to Two Helpers

addAtHead, addAtTail, addAtIndex → all call _insert_after. deleteAtIndex → always the same bypass. Tracking size makes bounds checking O(1). The API is clean: two primitives, five public methods.

🗣️ What to Say to the Interviewer

"I'll use a doubly-linked list with two sentinel nodes — a dummy head and dummy tail — that are never removed. They frame the list so every node always has a non-null predecessor and successor.
Every insertion reduces to: find the predecessor, call _insert_after(prev, new) — four pointer assignments in a specific safe order.
Every deletion: find the target, bypass it left and right.
I track size as an integer so all bounds checks are O(1). All operations are O(n) due to linear traversal to the target index."
🔮 Likely Follow-up Questions
  • "How would you get O(1) for all head/tail ops?" → Already have it via sentinels: self.tail.prev is always the last real node.
  • "Why doubly-linked vs singly-linked?" → Deletion needs the predecessor; doubly-linked gives it via node.prev instead of extra traversal.
  • "How does this relate to LRU Cache?" → LRU Cache (#146) uses exactly this structure plus a hash map for O(1) access to arbitrary nodes.
  • "What if you needed O(log n) get?" → Skip list or segment tree — different data structure entirely.

7 Summary & Key Takeaways ▾
Sentinel nodes eliminate every edge case — every op is "insert/delete between two existing nodes"
In _insert_after, always assign prev.next = new last — doing it earlier loses the old successor reference
Track size as an integer for O(1) bounds checking — avoids traversal on every bad call
addAtHead and addAtTail reduce to addAtIndex(0, val) and addAtIndex(size, val) — or direct sentinel access
This exact doubly-linked + sentinel structure is the backbone of LRU Cache — learn it once, use it everywhere
