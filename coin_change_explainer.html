LeetCode #322

Coin Change

Given an array of coin denominations and a target amount, find the fewest number of coins that you need to make up that amount. Return -1 if no combination can reach it.

Medium Dynamic Programming Unbounded Knapsack Time: O(amount × n) Space: O(amount)
1 Understanding the Problem ▾

You are at a cash register with an unlimited supply of coins of certain denominations — say 1¢, 3¢, and 4¢. A customer owes you a specific amount — say 6¢. Your job: make change using the fewest possible coins. You can use the same denomination multiple times (that's the "unbounded" part). If no combination can hit the exact amount, return -1.

💡 Key Insight

The optimal number of coins to make amount A depends on the optimal number of coins to make A − c for each available coin c. This recursive dependency on subproblems is the textbook definition of Dynamic Programming.

✓ coins=[1,3,4], amount=6

Option A: 4+1+1 = 3 coins
Option B: 3+3 = 2 coins ✓
Option C: 1×6 = 6 coins
Best: two 3-cent coins

✗ coins=[2], amount=3

Only even amounts reachable.
2, 4, 6, 8... never 3.
Return -1 (impossible).

Edge cases: amount=0 → return 0 (zero coins needed). A coin denomination larger than amount → that coin is never used. coins=[1] → answer is always amount. Amount perfectly divisible by largest coin → that coin dominates.

⚠️ Why Greedy Fails Here

With coins=[1,3,4] and amount=6: Greedy picks the largest valid coin repeatedly. 4 → then 1+1 → total 3 coins. But 3+3 = only 2 coins. Greedy has no look-ahead; it commits to each choice without seeing future consequences. Greedy works for canonical denominations (e.g., US coins: 25,10,5,1) but fails in the general case. This is why we need DP.

2 The Algorithm: Bottom-Up Unbounded Knapsack DP ▾

We build a 1D array dp of size amount + 1. The value dp[a] represents the minimum number of coins needed to make exactly amount a. We fill it from small amounts to large, using previously solved sub-problems.

1 Initialize: Create dp = [∞] × (amount+1) — every amount starts as "impossible." Then set dp[0] = 0: the base case. It takes zero coins to make zero.
2 Iterate amounts left-to-right (1 → amount): For each sub-amount a, try every coin denomination. Ask: "If I use one coin of value c, the remaining problem is dp[a−c]. So this path costs dp[a−c] + 1 coins."
3 Apply recurrence: dp[a] = min(dp[a], dp[a−c] + 1) for every coin c ≤ a. We take the minimum across all coin choices — because we want fewest coins, we pick whichever denomination leads to the best sub-problem.
4 Return answer: If dp[amount] is still ∞, no combination reached the target → return -1. Otherwise return dp[amount].
📐 The Recurrence Relation
dp[0] = 0                                          (base case: 0 coins for 0 amount)
dp[a] = min over all c in coins where c ≤ a of (dp[a − c] + 1)
answer = dp[amount] if finite, else −1

1D vs 2D — why does unbounded knapsack only need 1D?
In 0/1 Knapsack (each item used at most once), we track dp[item][capacity] in 2D because using item i twice must be prevented. Here, every coin can be used unlimited times. So dp[a] doesn't track which coins we used — only the minimum count. When we fill left-to-right and look up dp[a-c], that sub-answer may already include coin c — and that's fine! That's the "unbounded" property.

3 Why It Works: The DP Invariants ▾
🔐 The Invariant

When we compute dp[a], every dp[x] for x < a has already been computed correctly and holds the true minimum. This is guaranteed by filling left-to-right — smaller amounts before larger. So when we look up dp[a−c], the answer there is exact, not an estimate.

Optimal Substructure: Suppose the optimal solution for amount a uses coins ending with coin c. Then the remaining a−c must also be solved optimally — if it wasn't, we could swap in a better sub-solution and improve the whole answer (contradiction). Therefore the optimal solution for a is built from the optimal solution for a−c.

Overlapping Subproblems: Without memoization, a recursive approach recomputes the same sub-amounts exponentially. With coins=[1] and amount=100, computing f(100) naively involves computing f(99), which computes f(98) ... but f(98) is also needed by f(99) of the sibling branch, etc. DP computes each dp[a] exactly once in O(n) inner work per cell.

Why left-to-right fills correctly for unbounded: In 0/1 Knapsack, we fill right-to-left to prevent an item from being "re-used" — once we take item i in a smaller capacity, we don't want to take it again for the same item row. But here, reuse is desired. Filling left-to-right means dp[a−c] may have already used coin c, and that's exactly correct for the unlimited-supply scenario.

⚠️ float('inf') vs sys.maxsize

We use float('inf') rather than sys.maxsize or a large sentinel like 99999. The reason: when we compute dp[a-c] + 1, if dp[a-c] were sys.maxsize (an int), adding 1 overflows. float('inf') + 1 == float('inf') — the arithmetic stays safe, and the minimum comparison correctly discards unreachable sub-amounts.

4 The Code (Annotated) ▾

Solution 1 — Bottom-Up DP / Tabulation (Interview Pick)

def coinChange(coins: list[int], amount: int) -> int:
dp = [float('inf')] * (amount + 1) ← Array of size amount+1; all "unreachable" initially
dp[0] = 0 ← Base case: 0 coins to make 0. Everything else built from here.
for a in range(1, amount + 1): ← Fill amounts 1, 2, 3 ... amount in order (small → large)
for c in coins: ← Try each coin denomination for this amount
if c <= a: ← Only if coin value doesn't exceed current amount
dp[a] = min(dp[a], dp[a - c] + 1) ← 1 coin (c) + optimal for remainder (a-c)
return dp[amount] if dp[amount] != float('inf') else -1 ← Still inf = impossible
⚠️ Order Matters: Outer loop = amounts, Inner = coins

This is the Unbounded Knapsack order. If you swapped to outer=coins, inner=amounts, you'd get 0/1 Knapsack (each coin used at most once). With outer=amounts and inner=coins, by the time we compute dp[a], all smaller dp values are finalized — including ones that already used coin c. That reuse is what we want here.

Solution 2 — Top-Down DP / Memoization (Recursive clarity)

def coinChange(coins: list[int], amount: int) -> int:
from functools import lru_cache ← Python's built-in memoization decorator
@lru_cache(maxsize=None)
def dp(rem): ← Recursively: min coins to make `rem` amount
if rem == 0: return 0 ← Base case: 0 coins for 0 remaining
if rem < 0: return float('inf') ← Coin exceeded remaining — invalid path
return min(dp(rem - c) + 1 for c in coins) ← Try all coins; cache prevents re-computation
ans = dp(amount)
return ans if ans != float('inf') else -1

Solution 3 — Brute Force Recursion (context only — exponential)

def coinChange_brute(coins, amount): ← O(n^amount) — unusable on real inputs
if amount == 0: return 0
best = float('inf')
for c in coins:
if c <= amount:
sub = coinChange_brute(coins, amount - c) ← Recomputes same subproblems exponentially
if sub != float('inf'):
best = min(best, sub + 1)
return best
💡 Interview: Which Solution to Present?

Start by mentioning brute force (shows you understand the recursive structure). Then offer memoization ("cache each sub-amount to avoid recomputation — same O(amount×n) complexity"). Finally present tabulation as the "cleanest iterative solution with no recursion overhead." Interviewers typically love the tabulation walk-through because it clearly shows you understand the DP table structure.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Spot the DP signals

Keywords "minimum number", "count ways", "can we reach exactly X?" with reusable items signal DP. "Unlimited coins" specifically signals Unbounded Knapsack — the 1D variant where left-to-right filling is correct.

Step 2: Kill greedy with a counterexample

"Greedy (take largest coin first) fails with coins=[1,3,4], amount=6: picks 4+1+1=3 coins, but 3+3=2 coins is better. So I need DP to explore all options." This one sentence justifies the approach.

Step 3: Define state and recurrence out loud

"Let dp[a] = minimum coins for amount a. Base: dp[0]=0. For each amount a, try all coins c: dp[a] = min(dp[a], dp[a-c]+1). Fill left-to-right because smaller sub-problems must be solved before larger ones."

Step 4: State complexity and handle edge cases

Time: O(amount × n_coins). Space: O(amount). Edge: amount=0 → return 0. If dp[amount] remains ∞ at the end → return -1 (no valid combination exists).

🗣️ What to Say to the Interviewer

"This is a minimization problem with unlimited items — classic Unbounded Knapsack DP.

Greedy fails here: with coins [1,3,4] and amount 6, greedy picks 4+1+1=3 coins, but 3+3=2 coins is better.

I'll define dp[a] as the minimum coins to make exactly amount a. Base case: dp[0]=0, all others infinity.
For each amount from 1 to target, I try each coin c: if I use coin c last, the cost is dp[a-c]+1.
I take the minimum over all valid coins.

Time O(amount * n_coins), Space O(amount). If dp[amount] is still infinity, return -1."
🔮 Likely Follow-up Questions
  • "Count the number of ways to make the amount?" → Coin Change II: change min→sum, dp[0]=1 (#518)
  • "Each coin can only be used once?" → 0/1 Knapsack: use 2D dp or iterate coins outer, amounts right-to-left
  • "Which coins were used? Can you reconstruct?" → Track a parent[] array: parent[a]=c means coin c was used to reach a
  • "Minimum cost, coins have different values AND weights?" → Classic 2D Knapsack with 2D dp table

7 Summary & Key Takeaways ▾
Greedy fails here — only DP explores all coin combinations correctly
Core recurrence: dp[a] = min(dp[a−c] + 1) over all valid coins c
Fill left-to-right because coins are unbounded — reuse is allowed and desired
Initialize dp[0]=0 (base), all others to float('inf') for safe arithmetic
If dp[amount] is still ∞ after filling → return -1 (amount unreachable)
Time O(amount × n_coins), Space O(amount) — optimal complexity for this class
ApproachTimeSpaceNotes
Brute Force Recursion O(n^amount) O(amount) Exponential; unusable for real inputs
Top-Down Memoization O(amount × n) O(amount) Recursive; intuitive; same complexity
Bottom-Up Tabulation ✓ O(amount × n) O(amount) Iterative; no stack overflow risk; interview pick
🎯 When to Recognize Unbounded Knapsack DP
  • "Minimum / maximum / count using items with unlimited supply"
  • Items can be reused any number of times (no "use each once" constraint)
  • Greedy fails → no canonical structure in denominations
  • Pattern: 1D dp array, outer loop = capacity/amount, inner = item/coin
  • Signature recurrence: dp[a] = opt(dp[a−item] + cost)
