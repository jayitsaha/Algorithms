LeetCode #1095

Find in Mountain Array

Given a MountainArray (accessible only via get()), find the minimum index where get(index) == target. Return -1 if not found.

Hard Binary Search Find Peak + Two Searches Time: O(log n) Space: O(1)
1 Understanding the Problem ▾

A mountain array strictly increases to a single peak, then strictly decreases. You cannot access elements directly — only via mountainArr.get(index). The problem limits how many times you can call get(), enforcing an O(log n) solution.

💡 Key Insight

A mountain is secretly two sorted arrays glued at the peak. The left half is sorted ascending; the right half is sorted descending. Binary search works on any sorted (monotonic) sequence — so we run it three times: once to locate the peak, then once on each sorted half to hunt the target. The left search is done first to guarantee we return the minimum index.

✓ Mountain: [1, 3, 5, 8, 6, 4, 2]

Indices 0-3: 1<3<5<8 ↑ ascending
Index 3 (value 8): peak
Indices 3-6: 8>6>4>2 ↓ descending
Two sorted halves joined at peak

🎯 Search Strategy: target = 3

Step 1: Binary search → peak = 3
Step 2: Search [0..3] asc → found at idx 1
Step 3: Would search [3..6] desc
Return 1 (leftmost index wins)

Why the minimum index matters: Target value 3 might appear on both the ascending and descending sides. The left (ascending) side always has smaller indices than the right (descending) side, so searching left first and returning immediately gives us the minimum index for free — no comparison needed between the two results.

Edge cases: target equals the peak value (found right at peak index); target appears only on the descending side; target on both sides (left answer is smaller); target not in array (return -1); smallest valid mountain has 3 elements; peak at index 1 (very left) or n-2 (very right).

2 The Algorithm: Triple Binary Search ▾

Three sequential binary searches, each independently O(log n). Because log n + log n + log n = O(log n), the total is still logarithmic.

1 Phase 1 — Find the Peak Index: Run binary search using adjacent-element comparison. At any mid, compare get(mid) vs get(mid+1). If get(mid) < get(mid+1), you're on the ascending slope — peak is strictly right → lo = mid + 1. Otherwise you're at or past the peak → hi = mid. Terminate when lo == hi: that's the peak.
2 Phase 2 — Binary Search the Ascending Side [0 .. peak]: Standard ascending binary search. get(mid) == target → return mid immediately (leftmost by construction). get(mid) < target → target is bigger, go right: lo = mid + 1. get(mid) > target → go left: hi = mid - 1.
3 Phase 3 — Binary Search the Descending Side [peak .. n-1]: Same logic, but comparisons are flipped because values decrease going right. get(mid) < target → target is larger, but larger values are to the left → hi = mid - 1. get(mid) > target → go right: lo = mid + 1.
4 Return -1: Target was not found on either side of the mountain.
⚠️ The Descending Side Flip — The Most Common Bug

On the descending side, larger values are to the left. If get(mid) < target, you need a bigger value — but bigger values are leftward, so you move hi = mid - 1. New developers almost always apply standard ascending comparisons here and get wrong answers. Always ask yourself: "Which direction do values increase on this side?"

🔑 Peak Search: Why while lo < hi (not ≤)?

The peak search compares get(mid) with get(mid+1), requiring mid+1 to always be valid. Using lo < hi (strict) guarantees mid = (lo+hi)//2 < hi, so mid+1 ≤ hi ≤ n-1 is always a valid index. The value searches use lo <= hi (inclusive) since they don't need the adjacent element.

3 Why It Works: Three Invariants ▾
🔐 Invariant 1 — Peak Search Correctness

After each iteration, the true peak index remains within [lo, hi]. Proof: if get(mid) < get(mid+1), the peak cannot be at or before mid (the sequence is still rising), so the peak is in [mid+1, hi] — safe to set lo = mid+1. Otherwise, the sequence is descending or at a local max at mid, so peak is in [lo, mid] — safe to set hi = mid. Each step shrinks the range by at least 1, so we converge.

🔐 Invariant 2 — Ascending Search

Standard binary search invariant on the range [0, peak]: if the target exists on the left side, it lies within [lo, hi] at every step. The left side is strictly sorted ascending, so each comparison correctly eliminates half the range without discarding the target.

🔐 Invariant 3 — Descending Search

Mirror of invariant 2 on [peak, n-1], which is strictly sorted descending. By flipping comparison directions (smaller value → go left, larger value → go right), we restore the standard invariant: if target exists on the right side, it stays within [lo, hi] at every step.

Why left result is always the minimum index: Every index in the ascending half is in range [0, peak-1] (or peak itself). Every index in the descending half is in range [peak+1, n-1] (the peak appears in both searches but only once). If the same value V appears in both halves, its left-side index is at most peak, while its right-side index is at least peak+1. So the left-side answer is always strictly less than (or equal to, if V equals the peak value) the right-side answer. Returning immediately when found on the left guarantees the minimum.

Call count analysis: Phase 1: ≤ ⌈log₂ n⌉ calls. Phase 2: ≤ ⌈log₂(peak+1)⌉ calls. Phase 3: ≤ ⌈log₂(n-peak)⌉ calls. Worst case all three phases: ~3 log₂ n = O(log n). In practice the constraint allows up to 100 calls for arrays up to length 10,000 — log₂(10000) ≈ 13, so 3 phases × 13 = 39 calls, well within limit.

4 The Code (Annotated) ▾
def findInMountainArray(target, mountainArr):
n = mountainArr.length() ← cache length; don't call repeatedly
 
# ─── Phase 1: Find the Peak ───────────────────────
lo, hi = 0, n - 1 ← search the entire array for the peak
while lo < hi: ← strict: need mid+1 to be valid
mid = (lo + hi) // 2 ← integer mid; mid < hi guaranteed
if mountainArr.get(mid) < mountainArr.get(mid + 1):
lo = mid + 1 ← still ascending → peak is to the right
else:
hi = mid ← descending or at peak → go left
peak = lo ← lo == hi == peak index
 
# ─── Phase 2: Search Ascending Side [0 .. peak] ──
lo, hi = 0, peak
while lo <= hi: ← inclusive: search left side
mid = (lo + hi) // 2
val = mountainArr.get(mid)
if val == target: ← found! left side always gives min index
return mid
elif val < target: ← too small → go right (asc order)
lo = mid + 1
else: ← too large → go left
hi = mid - 1
 
# ─── Phase 3: Search Descending Side [peak .. n-1] ─
lo, hi = peak, n - 1
while lo <= hi:
mid = (lo + hi) // 2
val = mountainArr.get(mid)
if val == target:
return mid
elif val < target: ← FLIPPED: too small → go LEFT (values increase leftward)
hi = mid - 1
else: ← FLIPPED: too large → go right
lo = mid + 1
 
return -1 ← not found on either side
⚠️ Common Bug: Wrong loop bound in peak search

Using while lo <= hi in the peak search causes an infinite loop when lo == hi, because mid = lo = hi and the condition get(mid) < get(mid+1) would try to access get(hi+1) — an out-of-bounds index. Always use while lo < hi for the peak search.

💡 Brute Force Alternative (for comparison)

Linear scan: iterate all indices, return first where get(i) == target. Time: O(n) get() calls. The constraint forces O(log n). The triple binary search reduces ~10,000 calls to ~39 — crucial when each get() might involve a network round-trip or database lookup.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1 — Notice the API Constraint

The array is "opaque" — only accessible via get(). This is a classic signal that O(n) approaches will TLE or exceed the API call budget. The problem screams O(log n). Any approach requiring sorting or full traversal is ruled out immediately.

Step 2 — Identify the Structure

"Mountain array" = bitonic = strictly increases then decreases. This isn't globally sorted, but it IS composed of two sorted (monotonic) halves. Binary search works on monotonic sequences. Decompose the problem: find where the two halves meet (the peak), then apply binary search to each half.

Step 3 — Peak-Finding via Adjacent Comparison

How to find the peak without knowing it? At any midpoint, compare get(mid) and get(mid+1). This local pair tells you which slope you're on: rising → peak is right; falling → peak is left or at mid. This shrinks the range by half each iteration. Classic "find peak in bitonic array" pattern.

Step 4 — Two Different Binary Searches

Left half: standard ascending binary search. Right half: descending — all comparisons flip. Always run left first to get the minimum index guarantee for free. Mention this explicitly to the interviewer; it shows you're thinking about correctness, not just functionality.

🗣️ What to Say to the Interviewer

"The interface-only access and the mountain property together point to O(log n).
A mountain is two sorted arrays joined at the peak — each half is monotonic.
So: three binary searches. First, find the peak using adjacent comparisons.
Then search the ascending left half — standard binary search.
If not found, search the descending right half with flipped comparisons.
Left side first guarantees we return the minimum index without extra work.
Time: O(log n) get() calls. Space: O(1)."
💡 Follow-up Questions to Expect
  • "Can you cache get() results?" — Yes, use a dict to avoid re-calling the same index if the peak overlaps with both searches
  • "What if multiple peaks exist?" — This breaks the binary search; would need different approach (e.g., 162. Find Peak Element handles any local peak)
  • "Could target appear more than twice?" — In a valid mountain array, each value except the peak appears at most once per side (strictly monotone), so at most twice total

7 Summary & Key Takeaways ▾
Mountain array = ascending half + peak + descending half: three binary-searchable segments
Find the peak first via adjacent comparison: get(mid) < get(mid+1) → peak is right; else left/at mid
Search ascending side first — if target found there, that index is automatically the minimum
Flip comparison directions on the descending side (val < target → hi = mid-1)
Total O(log n) get() calls and O(1) extra space — optimal for any interface-constrained search

💡 When to Recognise This Pattern
  • Array is bitonic / mountain-shaped (one peak, strictly mono on each side)
  • Access restricted to an API — can't sort or slice
  • Must minimise the number of accesses (O(log n) budget)
  • Need to find an element with a minimum/maximum index constraint
