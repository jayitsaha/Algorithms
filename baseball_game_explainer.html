LeetCode #682

Baseball Game

Process a sequence of baseball scoring operations on a stack. Return the total score after all operations.

Easy Stacks Stack Simulation Time: O(n) Space: O(n)
1 Understanding the Problem ▾

You are keeping score for a baseball game with a peculiar ruleset. You're given a list of operations and must apply them one by one, maintaining a record of valid past scores. At the end, return the sum of all scores in the record.

Each operation is one of four types:

OperationMeaningAction on Record
"5" (integer) A real score earned this round Add that integer to the record
"+" Score equals sum of last two valid rounds Push (prev1 + prev2) onto record
"D" Score equals double the last valid round Push (prev × 2) onto record
"C" Last valid round was invalid — cancel it Remove the last entry from record
💡 Key Insight

Every operation refers to the "last valid scores" — meaning only scores currently in the record, not the raw input. A C removes the most recent record entry; a + looks at the top two record entries. This "look back at recent items" pattern is exactly what a stack is designed for.

Concrete Example: ops = ["5","2","C","D","+"]

Step-by-step record

"5" → push 5 → record: [5]
"2" → push 2 → record: [5, 2]
"C" → pop → record: [5]
"D" → push 5×2 → record: [5, 10]
"+" → push 5+10→ record: [5, 10, 15]
Total = 5 + 10 + 15 = 30

Edge cases to handle

Negative integers: "-3" is valid
"C" after "D": remove the doubled score
"+" always has 2+ entries (guaranteed)
Single op: just return that value
All "C": record empties → return 0

2 The Algorithm: Stack Simulation ▾

This problem is a direct translation of operations to stack manipulations. We maintain a stack as our score record. Each operation maps cleanly to a stack operation.

1 Initialize: Create an empty stack record = [] to store all valid scores.
2 Process each operation in order:
  • If it's an integer string (e.g., "5", "-3"): convert to int and push onto record.
  • If it's "+": push record[-1] + record[-2] (sum of last two).
  • If it's "D": push record[-1] * 2 (double the last).
  • If it's "C": pop the last element (cancel it).
3 Return: sum(record) — add up all remaining valid scores.
🔑 Why a Stack?

Both + and D need to access the most recent entries. C needs to remove the most recent entry. This LIFO (Last-In, First-Out) access pattern is exactly what a stack provides efficiently — each operation is O(1).

Operation Decision Tree:

op received →
  ├── is integer? → push int(op)
  ├── is "+"? → push stack[-1] + stack[-2]
  ├── is "D"? → push stack[-1] * 2
  └── is "C"? → pop stack (cancel)
3 Why It Works: The Invariant ▾
🔐 The Invariant

At all times, the stack contains exactly the valid score record up to this point — every entry currently on the stack is a legitimate past score that hasn't been cancelled. The most recently-added entry is always at stack[-1].

Why each operation preserves the invariant:

  • Integer push: We add a new valid score — invariant holds trivially.
  • "+" push: The problem guarantees ≥2 entries exist. We read the top two (both valid by invariant) and add their sum as a new valid entry.
  • "D" push: Reads top entry (valid by invariant), doubles it, adds as a new valid entry.
  • "C" pop: The problem guarantees ≥1 entry exists. Removing the top entry leaves the remaining entries still valid.
⚠️ Common Mistake: Using Input Operations as Record

Some beginners try to reference back into the operations list itself (e.g., operations[i-1]) when handling "+" or "D". This is wrong — you need the actual computed score values from the record, not the raw operation strings. Always read from the stack (record), never from the input array.

Why sum(record) at the end is correct: After processing all operations, every entry in the stack is a valid uncancelled score. The total score is simply their sum — the problem asks for exactly this.

💡 Complexity

Time: O(n) — each operation is processed exactly once; each push/pop is O(1). The final sum is O(n) in the worst case. Total O(n).
Space: O(n) — the stack can hold at most n entries (when no "C" operations exist).

4 The Code (Annotated) ▾

Solution 1 — Stack Simulation (Optimal, Interview Pick)

def calPoints(operations: list[str]) -> int: ← take list of operation strings
record = [] ← our stack of valid scores
for op in operations: ← process each operation left to right
if op == '+': ← sum of last two: both must exist (guaranteed)
record.append(record[-1] + record[-2])← [-1] is top, [-2] is one below
elif op == 'D': ← double the last valid score
record.append(record[-1] * 2) ← read top, multiply, push new entry
elif op == 'C': ← cancel: remove last valid score
record.pop() ← O(1) removal from end of list
else: ← must be an integer string (e.g. "5" or "-3")
record.append(int(op)) ← convert string to int and push
return sum(record) ← total of all uncancelled valid scores
⚠️ Why check "+" and "D" before the else?

The else branch handles any string that is not "+", "D", or "C" — which must be an integer string. If we put the integer check first with something like if op.lstrip('-').isdigit(), it works too, but the order shown (C/D/+ first) is cleaner. Any string not matching the three special tokens must be numeric by problem definition.

Solution 2 — Brute Force (Simulation with explicit index)

def calPoints(operations: list[str]) -> int:
record = []
for op in operations:
if op[0].lstrip('-').isdigit(): ← explicit numeric check (handles negatives)
record.append(int(op))
elif op == '+' and len(record) >= 2:
record.append(record[-1] + record[-2])
elif op == 'D' and len(record) >= 1:
record.append(record[-1] * 2)
elif op == 'C' and len(record) >= 1:
record.pop()
return sum(record) ← same result, just with defensive guards
💡 When to Use the Defensive Guards?

The problem guarantees valid input (enough elements always exist for + and D), so the guards are unnecessary for LeetCode. However, in production code or if you're unsure about input validity, the defensive version is safer. In an interview, note that you trust the constraints and omit the guards for cleanliness — but mention you'd add them in production.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify the "Look Back" Pattern

The key signal is that "+" and "D" need the most recent entries in the record, and "C" needs to remove the most recent entry. This "access/modify the most recent element" pattern always suggests a stack. Say: "Each operation refers to the most recently added scores, which is a classic LIFO pattern — I'll use a stack."

Step 2: Map Each Operation to a Stack Action

Before coding, explicitly map each operation type: integer → push; "+" → read 2, push sum; "D" → read 1, push double; "C" → pop. Drawing this on a whiteboard builds confidence and makes the code almost write itself.

Step 3: Walk Through an Example

Trace through ops = ["5","2","C","D","+"] manually: push 5 → push 2 → pop (→[5]) → push 10 (→[5,10]) → push 15 (→[5,10,15]) → return 30. This validates the logic and shows the interviewer you verify your solution.

Step 4: State Complexity and Edge Cases

Time O(n), Space O(n). Edge cases: negative integers ("−3"), all "C" operations (sum = 0), single element. The problem guarantees valid input, so no need for guards.

🗣️ What to Say to the Interviewer

"I notice that '+' and 'D' always refer to the last valid score(s), and 'C' removes the last one.
This LIFO access pattern calls for a stack — I'll maintain a record list as a stack.
For each op: if it's an integer, push it; '+' pushes the sum of the top two; 'D' pushes double the top; 'C' pops.
At the end I return sum of the stack. Time O(n), Space O(n).
Let me trace through the example to verify before coding..."
🔮 Likely Follow-up Questions
  • "What if operations could be invalid?" → Add length guards before accessing stack[-1], stack[-2]
  • "Can you do it in O(1) space?" → No — we need the history for "+" and "D" to look back
  • "What if we needed to undo the entire history?" → This is the Undo pattern; use a stack of state snapshots
  • "Similar problem: Basic Calculator (#224)" → Harder version with parentheses and real arithmetic

Stack = score record; each operation maps 1-to-1 to a stack action
"+" and "D" need the top of the stack — O(1) random access at the end of a list
"C" is just pop() — the most natural stack operation
Order of if/elif matters: check special strings first, integer falls to else
sum(record) at the end is O(n) — acceptable since we've already done O(n) work

🎯 When to Recognize the Stack Simulation Pattern

  • Parentheses matching / expression evaluation → Stack
  • "Valid" bracket tracking → Stack push/pop
  • History-based computation where latest matters most → Stack
