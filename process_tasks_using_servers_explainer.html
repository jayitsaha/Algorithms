LeetCode #1882

Process Tasks Using Servers

Assign each task to the lowest-weight free server (break ties by index). If no free server exists, wait until the earliest busy one finishes.

Medium Heaps Two Heaps (Free/Busy) Time: O((n+m) log n) Space: O(n)
1 Understanding the Problem ▾

You have n servers (each with a weight) and m tasks (each with a processing time). Task j becomes available at time j. The assignment rules are strict and deterministic:

1 Task j can only start at time ≥ j. It cannot be assigned before second j.
2 Among all free servers, pick the one with the smallest weight. If there's a tie in weight, pick the one with the smallest index.
3 If no server is free at time j, the task waits. Advance time to the moment the earliest busy server finishes, and assign there.
💡 Key Insight

This is a priority-based simulation. At any moment we need two things in O(log n): (1) the best free server by priority, and (2) the earliest finishing busy server. These require two separate ordered structures — which means two heaps.

Example Input

servers = [3, 3, 2]
tasks = [1, 2, 3, 2, 1, 2]

ans = [2, 0, 1, 2, 2, 0]
(Server 2, weight=2, is picked first
when free. Weight-3 servers follow.)

Why brute force fails

Scanning all n servers per task = O(n·m).
With n = m = 105: 1010 ops → TLE.

We need O(log n) access to the "best free"
and the "earliest finishing" server.

Key edge cases: all tasks arrive before any server frees (cascaded waiting); all servers have the same weight (always pick lowest index); a single server must handle all tasks sequentially; tasks where duration=1 meaning the same server might be reused on the very next step.

2 The Algorithm: Two-Heap Simulation ▾

We run a greedy event-driven simulation using two min-heaps. Servers move between heaps as they become busy or free. The simulation processes tasks in order, and time advances only when tasks are forced to wait.

1 Build free heap: Push all servers as (weight, index) tuples. Python's heapq is a min-heap, so the tuple ordering naturally gives us "smallest weight, then smallest index first."
2 For each task j: Set time = j (task's arrival time). Then release servers: pop all entries from busy heap whose finish_time ≤ time, and push them back to the free heap.
3 If free heap is empty: Jump time forward to busy_heap[0][0] (the earliest finish time). Release all servers finishing at that exact moment (there may be multiple!).
4 Assign task: Pop the best server from the free heap → record ans[j] = index. Push (time + duration, weight, index) onto the busy heap.

🔑 Why Two Heaps, Not One?

Free servers need ordering by (weight, index) — the greedy assignment priority. Busy servers need ordering by finish_time — who becomes free soonest. These are fundamentally different orderings. A single heap can only satisfy one. Two heaps gives O(log n) for both, independently.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At the moment we assign task j, every server in the free heap is genuinely available (its finish time has passed or it never started), and every server in the busy heap is still processing (finish time is strictly after current_time). The "release" step at the start of each iteration maintains this separation.

Why greedy is correct: The problem mandates a specific greedy priority: lowest weight, then lowest index. This is not our choice — it's the specification. Our job is to efficiently implement it. The free min-heap with (weight, index) tuples does this in O(log n) per operation rather than O(n) scan.

Why time-jumping is safe: When no server is free, the only way a server can become available is for a currently-running task to finish. The next such event is the minimum of all finish times, which sits at the top of the busy heap. Jumping directly to that time is equivalent to fast-forwarding the simulation — all intermediate time is empty (no events happen).

⚠️ The "Two Releases" Subtlety

Notice the code has two while-loops for releasing: one at the start (normal case, release all done by time j), and a second inside the "if not free" branch (release all done at the jumped-to time). Both use while busy and busy[0][0] <= time because multiple servers can finish at the exact same moment and all must be released before assignment.

Time vs. task index: Task j starts "at or after" time j. When servers are busy, the actual start time may be j+k for some k > 0. Tasks always execute in order 0, 1, ..., m-1 (no reordering of task assignments), but the clock can jump forward when waiting.

4 The Code (Annotated) ▾

Optimal Solution — Two-Heap Greedy (Interview Pick)

import heapq
def assignTasks(servers, tasks):
free = [(w, i) for i, w in enumerate(servers)]← (weight, index) — tuples auto-sort correctly
heapq.heapify(free) ← O(n) in-place heapify — all servers start free
busy = [] ← (finish_time, weight, index)
ans = [0] * len(tasks)
for j, duration in enumerate(tasks): ← j = task index = task arrival time
time = j ← Nominal start; may be pushed later
while busy and busy[0][0] <= time: ← Release all servers done by now
ft, w, idx = heapq.heappop(busy)
heapq.heappush(free, (w, idx))
if not free: ← Nobody free? Must wait for earliest finish
time = busy[0][0] ← Jump clock to next completion event
while busy and busy[0][0] <= time: ← Release ALL finishing at exactly that time
ft, w, idx = heapq.heappop(busy)
heapq.heappush(free, (w, idx))
w, idx = heapq.heappop(free) ← Best available: min weight, then min index
ans[j] = idx ← Record which server handles task j
heapq.heappush(busy, (time+duration, w, idx)) ← Mark busy until time+duration
return ans
⚠️ Why store weight in the busy heap?

When a server is released from the busy heap, we need to know its (weight, index) to push it back onto the free heap correctly. We store these in the busy heap tuple to avoid a separate lookup. The ordering of the busy heap is still solely by finish_time — weight and index are just cargo for reconstruction.

🚨 Common Bug: Using if not free with only one release loop

Some implementations check if not free first, then jump time, but forget to release multiple servers finishing simultaneously. Always use while (not if) to drain all servers finishing at the jumped-to time. Missing even one causes incorrect heap state in subsequent iterations.

Brute Force for reference (O(n·m), TLE)

def assignTasks_brute(servers, tasks):
finish = [0] * len(servers) ← finish[i] = when server i is next free
ans = []
for j, dur in enumerate(tasks):
t = j; best = -1
for i in range(len(servers)): ← O(n) scan per task
if finish[i] <= t:
if best == -1 or (servers[i],i) < (servers[best],best): best = i
if best == -1: ← No free server → jump to earliest finish
t = min(finish)
for i in range(len(servers)): ← Rescan for ties at min finish time
if finish[i]==t and (best==-1 or (servers[i],i)<(servers[best],best)): best=i
finish[best] = t + dur; ans.append(best)
return ans
💡 Complexity

Time: O((n + m) log n) — n heap operations to build the free heap (O(n)), then m task iterations each doing O(log n) heap push/pop operations. Each server is pushed/popped at most once per task.
Space: O(n) — the two heaps together hold at most n entries (every server is in exactly one heap at any time).

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize — "Best available resource, dynamically changing"

When a problem says "pick the cheapest/lightest/earliest available resource" and resources dynamically join/leave the available set, think priority queue. The key signal here: servers have a fixed priority order (weight → index) but their availability changes over time.

Step 2: State the brute force and its bottleneck

"I can maintain a finish_time[i] array and scan all n servers for each task. O(n) per task → O(n·m) total. For n=m=105 that's 1010 operations — too slow. I need O(log n) per task."

Step 3: Identify the two query types

"I need two types of fast query: (A) minimum by (weight, index) among free servers — for assignment; (B) minimum by finish_time among busy servers — for detecting when to release. These need different orderings, so I'll use two separate heaps."

Step 4: Describe the simulation loop

"For each task j: set time=j, release all busy servers with finish≤time, if free is empty jump time to earliest busy finish and release those, then pop best free server and push to busy with finish=time+duration."

🗣️ What to Say to the Interviewer

"I'll simulate task assignment with two min-heaps.
Free heap: (weight, index) — Python's tuple comparison gives me the right priority for free.
Busy heap: (finish_time, weight, index) — so I can quickly find who finishes first.
For each task j: release all servers finishing by time j (or wait if none free),
then pop the best free server and push it to busy with finish=time+duration.
Time: O((n+m) log n), Space: O(n)."
🔮 Likely Follow-up Questions
  • "What if tasks can be split?" → Track partial loads; the server model changes fundamentally.
  • "Minimum number of servers to handle all tasks with no waiting?" → Event-based analysis (like Meeting Rooms II).
  • "What if servers have capacity > 1?" → Each server gets a count; it stays in free heap until count=0.
  • "Tasks with deadlines — skip if too late?" → Add deadline check; skip task if time > deadline.
  • "Order of assignments for tied priority?" → Must be stable (FIFO for same weight); index tie-breaking handles it.

7 Summary & Key Takeaways ▾
Use two heaps: free heap ordered by (weight, idx), busy heap ordered by finish_time
Release finished servers with a while loop — never an if, to handle ties at the same finish time
When free heap is empty, jump time to busy[0].finish_time (the earliest finisher)
Tasks start at or after their nominal time j — they can be delayed but not early
Each server is in exactly one heap at any time — heap sizes always sum to n

🎯 When to Recognize the Two-Heaps Scheduling Pattern
  • Resources have two distinct states with different ordering priorities (free vs. busy)
  • You need "best available" and "earliest releasing" simultaneously → two heaps
  • The problem involves event-driven time advancement (jump to next completion)
  • Keywords: "assign to server / worker / machine", "earliest finish", "priority scheduling"
