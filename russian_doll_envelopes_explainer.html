LeetCode #354

Russian Doll Envelopes

Find the maximum number of envelopes you can nest inside each other, where each must be strictly narrower and shorter than the one enclosing it.

Hard Dynamic Programming Sort + LIS (Binary Search) Time: O(n log n) Space: O(n)
1 Understanding the Problem ▾

You are given a list of envelopes, each described by [width, height]. Envelope A fits inside envelope B if and only if A's width is strictly less than B's width AND A's height is strictly less than B's height. You want to find the longest chain of envelopes you can nest — like Russian dolls.

✓ Valid 3-Chain

[2,3] fits in [5,4]?
  2<5 ✓ and 3<4 ✓
[5,4] fits in [6,7]?
  5<6 ✓ and 4<7 ✓
Chain: [2,3]→[5,4]→[6,7] = 3

✗ Same Width Fails

[6,4] fits in [6,7]?
  6<6? NO — equal, not strict
Even though 4<7, the width
check fails → cannot nest!

💡 Key Insight — 2D Problem → 1D LIS

If we sort envelopes by width, the width constraint is handled automatically for consecutive envelopes in our sorted order. We then only need to find the Longest Increasing Subsequence (LIS) on heights. But same-width envelopes cannot both appear in a chain, so we need a sorting trick to handle ties.

🔑 The Same-Width Trick

Sort by width ASC, but for envelopes with equal width, sort by height DESC. This guarantees that within any same-width group, at most one envelope can enter the LIS — because their heights appear in decreasing order, which can never form an increasing subsequence of length > 1.

Edge Cases: All same size → answer is 1. All same width → answer is 1 (heights in a single group, sorted DESC, LIS length 1). One envelope → answer is 1. Widths strictly increasing but heights decreasing → answer is 1.

2 The Algorithm: Sort + Patience Sort LIS ▾

The solution is a two-phase algorithm. Phase 1 collapses the 2D problem into 1D. Phase 2 solves LIS in O(n log n) using patience sorting — maintaining a tails array where tails[i] is the smallest possible ending value for an increasing subsequence of length i+1.

1 Sort: envelopes.sort(key=lambda x: (x[0], -x[1])) — width ascending, height descending for ties. This is the clever trick: same-width heights appear largest-first, so LIS can pick at most one.
2 Initialize tails = []: An empty array that will grow to represent the optimal ending values for increasing subsequences of each length.
3 For each height h: Binary search tails for the first index pos where tails[pos] ≥ h (using bisect_left).
4 Extend or Replace: If pos == len(tails), append h (LIS grows). Otherwise, set tails[pos] = h (we found a better — smaller — ending value for a subsequence of this length).
5 Return len(tails): The length of the tails array equals the LIS length, which equals the maximum nesting depth.

Why does sorting heights DESC for same-width groups prevent double-counting?

Without trick (height ASC)

With trick (height DESC)

After sorting by (w ASC, h ASC):
dp[i] = 1 + max(dp[j] for j < i if w[j]<w[i] and h[j]<h[i])
dp[i] = 1                         (base: every envelope alone = chain of 1)
answer = max(dp)

The O(n log n) version replaces the inner loop with binary search on the tails array — that is what we animate below.

3 Why It Works: The Tails Invariant ▾
🔐 The Invariant

After processing every element, tails is a strictly increasing array where tails[k] holds the minimum possible ending value of any increasing subsequence of length k+1 seen so far.

Why is tails always strictly sorted? If we have an IS of length 3 ending at value X, and one of length 4, the length-4 IS must end with something > X (since it extends length-3 by one more strictly larger element). So tails[2] < tails[3] always. This sorted property is what allows binary search.

Why does replacement give the correct final length? When we replace tails[pos] with a smaller value h, we're recording: "I can now achieve a subsequence of length pos+1 ending at the smaller value h." This never decreases any subsequence lengths — it only improves the minimum tail for that length, making future extensions more likely. The final array length is still the LIS length.

⚠️ Critical Misconception: tails is NOT a valid subsequence

The elements in tails may come from different incompatible subsequences. After processing [3,4,7,4], tails = [3,4,4]. The value at position 2 was replaced from 7 to... wait, 4 replaces position 1 (tails[1]=4, already 4). The actual answer 3 comes from the length, not from reading the array as a subsequence. Never try to reconstruct the subsequence from tails — use a separate predecessor array for that.

This is DP because:

Optimal Substructure: The longest chain ending at envelope i is 1 + (longest chain ending at any j where j fits inside i). The global optimum depends on optimal solutions to smaller subproblems.

Overlapping Subproblems: Many envelopes share the same "compatible predecessors." Without memoization, we'd recompute the same sub-chains repeatedly. The tails array is an O(n) compressed representation of all these overlapping computations.

4 The Code (Annotated) ▾

Solution 1 — Sort + LIS Binary Search (O(n log n), Interview Pick)

import bisectbuilt-in binary search; bisect_left finds leftmost insertion point
def maxEnvelopes(envelopes):
envelopes.sort(key=lambda x: (x[0], -x[1]))width ASC; for same width, height DESC — the crucial trick
tails = []patience sort tails; tails[i] = min tail for IS of length i+1
for _, h in envelopes:we only care about heights after sort handles widths
pos = bisect.bisect_left(tails, h)O(log n): first position where tails[pos] >= h
if pos == len(tails):h is bigger than everything — extends the LIS
tails.append(h)LIS grows by 1
else:
tails[pos] = hreplace with smaller tail — same IS length but more extensible
return len(tails)length of tails = LIS length = max nesting depth
⚠️ Why -x[1] and not just x[1]?

With same-width envelopes sorted height ASC, their heights form an increasing sequence. LIS on that increasing sequence would pick all of them — but we can't nest same-width envelopes! Sorting height DESC makes same-width heights decrease, so LIS can pick at most one from each same-width group. This is the single trickiest insight in the problem.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Pattern

The problem asks for a "longest chain with strictly increasing properties" — a classic LIS signal. The two-dimensional twist is new, but ask: "If I fix one dimension, is this just LIS on the other?" Yes → reduce to 1D.

Step 2: Propose Sorting

Sort by width ascending. Now any later envelope has a greater-or-equal width. We only need to ensure heights are increasing. But flag immediately: "Same width is not strictly less than — I need to handle ties."

Step 3: Handle Same-Width Ties

Give a concrete counterexample: [[3,3],[3,5]]. Sorting height ASC gives heights [3,5], LIS=2 (wrong). Sorting height DESC gives [5,3], LIS=1 (correct). Explain why: decreasing heights within a group prevent the LIS algorithm from picking more than one.

Step 4: Propose O(n log n) LIS

Mention the O(n²) DP, then immediately improve: "I can solve LIS in O(n log n) using a patience-sort tails array and binary search. The tails array stays sorted, so bisect_left finds the right position in O(log n) per element."

🗣️ What to Say to the Interviewer

"This is 2D LIS. Sort by width ascending to handle the width constraint automatically, then run LIS on heights. The tricky part: equal widths can't nest, so I sort heights descending within same-width groups. That way, the LIS algorithm can pick at most one envelope per width group, respecting the strict-less-than requirement. For the LIS itself, I'll use the O(n log n) patience sort: maintain a tails array where tails[i] is the smallest ending value for an IS of length i+1. Binary search each height to extend or replace. Total: O(n log n) time, O(n) space."
🎯 Follow-Up Questions to Expect
  • "Can you allow equal width or height?" — Change bisect_left to bisect_right for non-strict LIS.
  • "Reconstruct the actual chain?" — Track a separate predecessor array in the O(n²) DP; back-trace for the O(n log n) version.
  • "3D version (cuboids)?" — Sort by two dimensions and LIS on the third; same pattern applies.
  • "What if widths are all the same?" — After DESC sort by height, LIS = 1. Correct.

7 Summary & Key Takeaways ▾
Sort by width ASC, height DESC for same-width ties — this is the non-obvious trick that correctly reduces the 2D problem to 1D LIS.
After sorting, we only run LIS on heights — a clean 1D problem solved optimally in O(n log n) via patience sorting.
The tails array is always sorted, enabling O(log n) binary search per element. Its length is the LIS length, but its contents are NOT a valid subsequence.
For each height, bisect_left finds the optimal replacement position: extend if larger than all tails, replace otherwise.
This is DP: optimal substructure (chain ending at i depends on best chain ending at compatible j) + overlapping subproblems (many envelopes share ancestors).
🎯 When to Use Sort + LIS
  • Multi-dimensional "strictly fits inside" or "strictly dominates" problems
  • Longest chain where items must satisfy multiple strictly-increasing constraints
  • When sorting on one dimension reduces the problem to 1D LIS on another
  • Keywords: "nesting," "enclosing," "stacking," "contains," "dominates," "strictly increasing in multiple dimensions"
