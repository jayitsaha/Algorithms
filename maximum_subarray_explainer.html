LeetCode #53

Maximum Subarray

Given an integer array nums, find the subarray with the largest sum and return its sum.

Medium Array Manipulation Kadane's Algorithm Time: O(n) Space: O(1)
1 Understanding the Problem ▾

We have an array of integers — some positive, some negative — and we want to find a contiguous subarray (a slice of consecutive elements with no gaps) whose values sum to the largest possible total. We return only the sum itself, not the actual subarray.

💡 Key Insight

A negative prefix always hurts any sum that follows it. So whenever our running total drops below zero, it is strictly better to abandon that prefix and start fresh at the next element rather than carrying dead weight into the future.

✓ Example: [−2, 1, −3, 4, −1, 2, 1, −5, 4]

Sum = 4 + (−1) + 2 + 1 = 6

The subarray must be contiguous.
No gaps allowed!

Critical edge cases: When all elements are negative (e.g., [−3, −1, −2]), the answer is −1 (the maximum single element). The problem guarantees at least one element, so an empty subarray is never valid — never return 0 as a default.

2 The Algorithm: Kadane's Algorithm ▾

Kadane's Algorithm (1984, Joseph Kadane) is the classic linear-time solution. It makes a greedy local decision at every index: "Is it better to extend my current subarray by including this element, or should I start a brand-new subarray right here?"

1 Seed both trackers with nums[0]: Set curr_sum = nums[0] and max_sum = nums[0]. Using the first element (not 0) correctly handles all-negative inputs — the problem requires at least one element.
2 Scan from index 1: For each element num, compute curr_sum = max(num, curr_sum + num). If curr_sum + num < num, the previous running total was negative — restart from num alone.
3 Track the global best: After each update, max_sum = max(max_sum, curr_sum). This captures the best we've ever seen at any position.
4 Return max_sum: After visiting every element, max_sum is the answer — the largest subarray sum anywhere in the array.
🔑 The One Critical Line

curr_sum = max(num, curr_sum + num)

Two options at every step: extend the current subarray (curr_sum + num) or restart a new subarray here (num alone). We pick the larger. If curr_sum was negative, extending would only hurt — so we restart.

3 Why It Works: The Invariant ▾
🔐 The Invariant

After processing index i, curr_sum holds the maximum sum of any subarray that ends exactly at position i. The global max_sum is the running maximum of all such "best ending here" values across indices 0 through i.

Why this is sufficient: The answer subarray ends at some index — we don't know which one. But we compute the best-subarray-ending-here value at every index and remember the global maximum. So the true answer is necessarily captured.

Why restarting beats extending when curr_sum is negative: Say curr_sum = −3 and the next element is 5. Extending gives 5 + (−3) = 2. Restarting gives 5. Fresh start wins. Generalizing: a negative curr_sum reduces any subsequent value it's added to — it can never help.

⚠️ The Classic Initialization Mistake

Many beginners write curr_sum = 0; max_sum = 0. This breaks on all-negative inputs like [−1, −2, −3]: the algorithm would return 0 (the non-existent "empty subarray") instead of −1. Always initialize both variables to nums[0] and begin the loop at index 1.

Optimality proof sketch: Let the true optimal subarray be [l..r]. By induction on the invariant, when the scan reaches index r, curr_sum equals the maximum sum of any subarray ending at r — which is at least the sum of [l..r]. Therefore max_sum will have recorded this value. The algorithm is optimal.

4 The Code (Annotated) ▾

Solution 1 — Kadane's Algorithm (Optimal, Interview Pick) · O(n) time · O(1) space

def maxSubArray(nums: list[int]) -> int:
curr_sum = nums[0] ← Best-ending-here sum; seed with first element (not 0!)
max_sum = nums[0] ← Global best seen so far; same seed handles all-negative case
for num in nums[1:]: ← Start at index 1 (index 0 already seeded)
curr_sum = max(num, curr_sum + num) ← Restart or extend — whichever is larger
max_sum = max(max_sum, curr_sum) ← Did we just beat the global record?
return max_sum ← Largest subarray sum found anywhere in the array
⚠️ Why max(num, curr_sum + num) and not max(0, curr_sum + num)?

Using 0 as one branch allows returning the empty-subarray sum (0), which is invalid when all elements are negative. Using num forces at least one element in the subarray — correct per the problem constraints.

Solution 2 — Brute Force (for comparison only) · O(n²) time

def maxSubArray_brute(nums):
best = nums[0] ← Correct seed (not −∞ or 0)
for i in range(len(nums)): ← Every possible start index
total = 0
for j in range(i, len(nums)): ← Every possible end index from i
total += nums[j]
best = max(best, total) ← O(n²) total — TLE on large inputs
return best
💡 Complexity Summary

Kadane's (Optimal): Time O(n) — single linear scan. Space O(1) — two variables.
Brute Force: Time O(n²) — all start/end pairs. Space O(1).
Divide & Conquer: Time O(n log n) — split at midpoint recursively. Space O(log n) call stack.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Pattern

Signals: "optimal contiguous range," "maximize subarray sum." Brute force tries all O(n²) start/end pairs. The key question to ask yourself: can the decision at index i depend only on what happened at index i−1?

Step 2: Propose Brute Force First

"Naive approach: for each start i, extend j to the right and track the running sum. O(n²). Correct but too slow. Can we avoid recomputing overlapping subproblems?"

Step 3: Derive the Optimization

"At position i, I only need the best subarray ending at i−1 to know the best ending at i. If that previous best was negative, I'm better off starting fresh at i. So: curr_sum = max(nums[i], curr_sum + nums[i])."

Step 4: State Edge Cases

"I initialize to nums[0] — not 0 — so all-negative arrays are handled correctly. The subarray must have at least one element."

🗣️ What to Say to the Interviewer

"At each position, there's a local decision: extend the current window or restart here.
If the running sum going into this element is negative, including it only hurts.
So: curr_sum = max(nums[i], curr_sum + nums[i]).
I track a separate global max that never decreases.
Initialize both to nums[0] — not 0 — to handle all-negative arrays.
Time O(n), Space O(1) — single pass, two variables."
🔮 Likely Follow-up Questions
  • "Also return the actual subarray indices?" → Track start/end/tempStart; reset tempStart on restart
  • "Maximum product subarray?" → Track both max AND min (negative × negative = positive) (#152)
  • "Maximum sum after deleting one element?" → Forward + backward prefix Kadane (#1186)

7 Summary & Key Takeaways ▾
A negative running sum is dead weight — always restart when extending would give a worse result than starting fresh
curr_sum tracks the best subarray ending at the current index; max_sum tracks the overall global best
Initialize both to nums[0], never 0 — handles all-negative arrays without returning an invalid empty-subarray sum
Single pass through the array — O(n) time with O(1) space; no extra arrays needed
The pattern generalizes: max product, circular max, one-deletion max all build on the same extend-or-restart idea
