LeetCode #1423

Maximum Points You Can Obtain from Cards

Pick exactly k cards from either end of a row — maximize your total score using the complement window trick.

Medium Sliding Window Minimum Middle Window Time: O(n) Space: O(1)
1 Understanding the Problem ▾

You have a row of cards face-down, each with a point value. In each step, you take a card from the very left end or the very right end — never from the middle. You must take exactly k cards. What is the maximum total points you can collect?

💡 The Counterintuitive Insight: Flip the Problem

Instead of asking "which k cards should I take from the ends?", ask "which n − k cards should I leave behind?" Since you always take from both ends, whatever you leave behind is always a contiguous block in the middle. To maximize what you take, you must minimize what you skip — find the minimum-sum subarray of length n − k.

2 The Algorithm: Complement Sliding Window ▾

We invert the problem. The n − k cards we skip always form a contiguous middle block. We slide a fixed-size window of length n − k across the array, tracking the minimum window sum. The answer is total − min_window_sum.

1 Compute total sum. This is the maximum we could ever earn. We subtract the "damage" (the skipped window) from it.
2 Set window size w = n − k. If w = 0, we take all cards — return total immediately. Otherwise, initialize the first window sum as sum(cardPoints[:w]).
3 Slide the window right, k times. Each slide: add the new right element, remove the old left element. This maintains the window sum in O(1) per step. Track min_win as we go.
4 Return total − min_win. The minimum skip leaves the maximum take.
🔑 Why Fixed-Size Window

Every valid selection of k cards from the ends leaves exactly n − k consecutive cards untouched in the middle. This fixed-size constraint is what makes a simple sliding window (not a dynamic shrink/expand) appropriate. There are exactly k + 1 window positions (from taking 0-from-left to taking k-from-left), so we iterate k+1 times total.

3 Why It Works: The Invariant ▾
🔐 The Invariant

There is a one-to-one correspondence between every valid selection of k cards from both ends and every window of size n − k in the middle. Specifically, if you take i cards from the left (indices 0 to i−1) and k−i cards from the right (indices n−(k−i) to n−1), the middle window spans indices i to n−k+i−1. As i ranges 0 to k, you visit every valid window position exactly once.

Proof sketch: Let S be any valid selection of k cards from both ends. The unselected cards form a contiguous segment — because you can only take from either end, there is no way to "skip" a card in the middle without taking the card adjacent to it. Conversely, any contiguous window of size n−k corresponds to a valid selection (take everything outside the window). So sliding the window through all k+1 positions exhausts all possible selections, and tracking the minimum window sum gives us the maximum take.

⚠️ Common Confusion: When is window size 0?

If k = n, you take every card. The "middle window" has size 0, meaning nothing is skipped. The answer is simply the total sum — handle this as a special case early to avoid an off-by-one in the loop setup.

⚠️ Off-By-One in the Slide Loop

The initial window (take 0 from left, k from right) is computed before the loop. The loop runs from i=1 to i=k (k iterations), covering the remaining k window positions. This gives k+1 total configurations checked — exactly correct. Don't be tempted to use range(0, k) without the initial setup.

4 The Code (Annotated) ▾

Solution 1 — Complement Sliding Window (Interview Pick, O(n) time, O(1) space)

def maxScore(cardPoints: list[int], k: int) -> int:
n = len(cardPoints) ← total number of cards
total = sum(cardPoints) ← baseline: taking every card
w = n - k ← size of the middle "skip" window
if w == 0: ← k == n edge case: take all cards
return total
win_sum = sum(cardPoints[:w]) ← initial window at leftmost position (i=0)
min_win = win_sum ← best (minimum) window sum found so far
for i in range(1, k + 1): ← slide window k positions; i = cards taken from left
win_sum += cardPoints[i + w - 1] ← add new right element entering the window
win_sum -= cardPoints[i - 1] ← remove old left element leaving the window
min_win = min(min_win, win_sum) ← update minimum window sum
return total - min_win ← take as much as possible by skipping as little as possible
⚠️ Index math for the sliding step

When we've taken i cards from the left, the skip window occupies [i, i+w−1] (0-indexed). The new element entering at position i=2 (second slide) is cardPoints[2 + 4 − 1] = cardPoints[5]. The element leaving is cardPoints[2 − 1] = cardPoints[1]. Always verify with a small example if the index math feels shaky.

Solution 2 — Direct Two-Pointer (O(k) time, O(1) space)

def maxScore(cardPoints: list[int], k: int) -> int:
left_sum = sum(cardPoints[:k]) ← start: take all k from the left
best = left_sum
right_sum = 0
for i in range(1, k + 1): ← trade one left card for one right card
left_sum -= cardPoints[k - i] ← give back the rightmost left card
right_sum += cardPoints[-i] ← pick up the i-th card from the right end
best = max(best, left_sum + right_sum) ← is this split better?
return best
💡 Complexity Summary

Complement Window (Sol 1): Time O(n) (sum of all n, then k+1 window steps), Space O(1)
Two-Pointer (Sol 2): Time O(k) (sum of first k, then k swaps), Space O(1)
Both are effectively O(n) worst case (k ≤ n). In interviews, the complement window is preferred because it demonstrates a known pattern (minimum subarray complement).

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify the Structural Constraint

The problem says "from either end." This is not a free selection — the cards you take are always a prefix plus a suffix. The untouched cards are always one contiguous block. This structure is the key unlock.

Step 2: Propose Greedy, Then Disprove It

"My first thought is greedy — always pick the larger end. But consider [1,100,100,100,2], k=4: greedy would take 1 (left), then 2,100,100 from right = 203. Optimal is to take 100+100+100+2 = 302 from right. Greedy fails, so I need something systematic."

Step 3: Flip to the Complement

"Instead of choosing what to take, I'll choose what to skip. The skipped cards form a fixed-size window of length n−k. Minimizing the skipped sum = maximizing the taken sum. I can find the minimum window sum in O(n) with a sliding window."

Step 4: Code and Verify

Compute total, initialize window of size n−k, slide it k times tracking the minimum. Return total − min. Verify with the provided example before submitting.

🗣️ What to Say to the Interviewer

"Taking k cards from both ends always leaves n−k consecutive cards in the middle untouched.
So: maximize taken = minimize skipped = find minimum-sum window of size n−k.
Compute total sum in O(n). Initialize a window of size n−k over the leftmost position.
Slide it k times, each time adding the new right element and removing the old left element.
Track the minimum window sum. Return total − min_window. Time O(n), Space O(1)."
🔮 Likely Follow-up Questions
  • "What if you could pick from any position?" → Then it becomes a standard max-subarray problem (Kadane's) or prefix-sum query.
  • "What if each card can only be used once per round of picks?" → Same problem — cards are already distinct positions.
  • "What if picks from left cost 1 extra point each?" → Track left-pick count separately; adjust the score accordingly.
  • "What if k can be 0?" → Edge case: return 0.

7 Summary & Key Takeaways ▾
Taking k cards from both ends = always leaving n−k consecutive cards in the middle
Maximize taken ↔ minimize skipped: complement window trick
Fixed-size sliding window of size n−k finds the minimum skip in O(n), O(1) space
There are exactly k+1 window positions, one per possible left-take count 0..k
Greedy (always pick larger end) fails — use the complement approach instead
ApproachTimeSpaceNotes
Enumerate all splits (brute) O(k) O(k) Recompute prefix/suffix sums for each of k+1 splits
Prefix + Suffix arrays O(n) O(n) Build prefix/suffix, then iterate splits — cleaner but O(n) space
Complement Window ✓ O(n) O(1) Canonical interview answer; elegant inversion
Direct Two-Pointer O(k) O(1) Directly trades left-for-right cards; simpler mental model
🎯 When to Recognize the Minimum Middle Window Pattern
  • "Take exactly k elements from either end" → complement = fixed window in the middle
  • When "what you DON'T pick is always contiguous," optimize the complement instead
  • Any time maximize(selection) can be reframed as minimize(complement) with fixed complement size
  • Related pattern family: prefix-suffix duality, complement subarray problems
