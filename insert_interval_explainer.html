LeetCode #57

Insert Interval

Given a sorted, non-overlapping list of intervals, insert a new interval — merging all overlapping intervals — and return the sorted result.

Medium Intervals Merge with New Time: O(n) Space: O(n)
1 Understanding the Problem ▾

Imagine a calendar with booked time slots — sorted, non-overlapping. You want to add a new meeting. Some existing meetings might overlap your new one. Your job: merge all conflicts and return the final, clean calendar.

💡 Key Insight

Every existing interval falls into exactly one of three zones relative to the new interval: completely before it, overlapping with it, or completely after it. Handle each zone with one rule and the algorithm writes itself.

Example 1 — Simple

intervals = [[1,3],[6,9]]
newInterval = [2,5]
[1,3] + [2,5] → merge → [1,5]
[6,9]: after new end 5 → copy
Result: [[1,5],[6,9]]

Example 2 — Multiple Merges

intervals = [[1,2],[3,5],[6,7],[8,10],[12,16]]
newInterval = [4,8]
[1,2]: before → copy
[3,5],[6,7],[8,10]: overlap → merge to [3,10]
[12,16]: after → copy
Result: [[1,2],[3,10],[12,16]]

Edge Cases: Empty intervals list (return [newInterval]). New interval entirely before all (prepend). New interval entirely after all (append). New interval swallows every existing one (return one giant interval). Touching endpoints like [1,3] and [3,5] — do they merge? Yes: they share point 3, so the overlap condition start <= end is satisfied.

2 The Algorithm: Three-Phase Linear Sweep ▾

We make a single left-to-right pass through the sorted intervals, classifying each one into its zone. The key realization: because the input is sorted and non-overlapping, once you've passed all "before" intervals and entered the "overlap" zone, you know you'll eventually exit it — and everything after is automatically "after". The three zones are contiguous.

1 Phase 1 — Copy "before" intervals: While intervals[i][1] < newInterval[0], the current interval ends strictly before the new one starts. Zero overlap is possible. Append directly to result.
2 Phase 2 — Merge overlapping intervals: While intervals[i][0] <= newInterval[1], the current interval starts at or before the new interval ends — they overlap. Expand newInterval by taking min(starts) and max(ends).
3 Insert merged result: After Phase 2 ends, append the fully-expanded newInterval. This is the single merged interval that absorbed all overlaps.
4 Phase 3 — Copy "after" intervals: All remaining intervals start after newInterval ends. Copy them verbatim. No additional merging is possible (input was already non-overlapping).
🔐 Why the Three Zones Are Mutually Exclusive

Two intervals [a,b] and [c,d] overlap iff a <= d AND b >= c. In Phase 2, we check intervals[i][0] <= newInterval[1]. We only reach Phase 2 after Phase 1, which guarantees intervals[i][1] >= newInterval[0]. Both overlap conditions are satisfied — implicitly and cleanly.

3 Why It Works: The Invariant ▾
🔐 The Invariant

Throughout Phase 2, newInterval = the union of the original new interval plus every existing interval encountered so far that overlaps with it. When Phase 2 exits, newInterval is the complete merged result — no interval can extend it further.

Why sorted input is crucial: Because intervals are sorted by start time, once an interval's start exceeds newInterval[1], every subsequent interval also starts after it (starts only increase). So we can exit Phase 2 with confidence — no future interval can overlap our merged result.

Merging by expansion is correct: If intervals [3,5] and [6,7] both overlap new interval [4,8], their union is [3,7]. But [3,7] is contained within [4,8] expanded, so min(4,3)=3 and max(8,7)=8 gives [3,8], which correctly spans all three intervals together.

⚠️ The Touching-Endpoint Trap

The Phase 2 condition uses <=, not <. Intervals [1,3] and [3,5] share endpoint 3. With <, they'd be treated as non-overlapping. With <=, they correctly merge to [1,5]. Always ask: "Does my definition of overlap include shared endpoints?" For this problem, yes.

4 The Code (Annotated) ▾
def insert(intervals, newInterval):
result = [] ← output; grows through 3 phases
i = 0 ← current index into intervals
n = len(intervals)
 
# Phase 1: intervals that end before newInterval starts
while i < n and intervals[i][1] < newInterval[0]: ← strictly before: end < new_start
result.append(intervals[i]) ← no overlap; copy unchanged
i += 1
 
# Phase 2: merge all overlapping intervals into newInterval
while i < n and intervals[i][0] <= newInterval[1]: ← start ≤ new_end → overlap
newInterval[0] = min(newInterval[0], intervals[i][0]) ← expand left edge
newInterval[1] = max(newInterval[1], intervals[i][1]) ← expand right edge
i += 1
result.append(newInterval) ← the one merged interval goes here
 
# Phase 3: intervals that start after newInterval ends
while i < n: ← all remaining are strictly after
result.append(intervals[i]) ← copy unchanged
i += 1
 
return result

⚠️ Why not O(1) space?

The output itself is O(n) in the worst case (no merging = all intervals returned). Output space is unavoidable. We count auxiliary space as O(1) — we only track index i and mutate newInterval in place — but the full space is O(n) for the result list.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Pattern

Signal words: "sorted intervals", "insert a new interval", "non-overlapping list", "merge overlapping". This is the canonical interval insertion problem. The sorted + non-overlapping precondition means you skip the sort step (no O(n log n) needed).

Step 2: Draw the Three Zones

On scratch paper, draw a number line. Draw the new interval as a thick bar. Show that every existing interval must land in one of three zones — before, overlapping, or after. This visual immediately reveals three separate processing rules.

Step 3: Work Out the Boundary Conditions

Before coding, state: "Phase 1 exits when intervals[i][1] >= newInterval[0]. Phase 2 exits when intervals[i][0] > newInterval[1]." Verify with a concrete example — especially the touching endpoint case. Then negate to get the while-loop conditions.

Step 4: Three Clean While-Loops

Code each phase as its own while-loop with a clear invariant. This is more readable and debuggable than one loop with complex if-else branches. Place the merged interval append exactly once, after Phase 2 finishes — not inside the loop.

🗣️ What to Say to the Interviewer

"Since intervals are already sorted and non-overlapping, I can sweep once in O(n).

I'll classify each existing interval into one of three zones:
Zone 1: ends before new interval starts — copy directly.
Zone 2: overlaps new interval — expand new to absorb it.
Zone 3: starts after new interval ends — copy directly.

The overlap condition: interval i overlaps new when i.start <= new.end.
The min/max expansion ensures the merged interval covers all overlapping ranges.

I insert the merged newInterval exactly once, after all overlaps are consumed.
Time: O(n) — single pass. Space: O(n) — for the output."

Likely follow-up questions:

  • What if intervals are unsorted? — Sort first: O(n log n). Then run this algorithm on the sorted list plus the new interval.
  • What if we insert many new intervals? — See LC 56 (Merge Intervals): sort all intervals together, then do a single merge sweep.
  • Can you do it in-place without a result array? — Technically yes with list splicing, but it's O(n²) due to shifts. Not worth it.

7 Summary & Key Takeaways ▾
The sorted, non-overlapping precondition partitions all intervals into three contiguous zones. No binary search or sorting needed — a simple linear sweep suffices for O(n).
Merging = expanding: expand newInterval's start leftward with min() and rightward with max(). This absorbs any number of overlapping intervals into one.
The Phase 2 exit condition (intervals[i][0] > newInterval[1]) is implicit — once Phase 2's while exits, everything remaining is automatically "after". No extra check needed.
Append the merged newInterval exactly once, after Phase 2. A common bug is appending inside the Phase 2 loop (generates duplicates) or forgetting to append at all.
Use <= for the overlap check to correctly merge touching intervals like [1,3] and [3,5]. Strict < would incorrectly separate them.
🎯 When to Use This Pattern
  • Inserting one new range into a sorted, non-overlapping interval list
  • Any problem where you process a timeline and must merge a new segment with existing ones
  • Calendar scheduling — finding open slots, booking meetings
  • Generalization: if intervals are unsorted, sort first; then use the same sweep
