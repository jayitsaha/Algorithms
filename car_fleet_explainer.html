LeetCode #853

Car Fleet

n cars drive toward a target. Faster cars catch slower ones and merge into fleets. How many fleets arrive?

Medium Stacks Sort + Monotonic Stack Time: O(n log n) Space: O(n)
1 Understanding the Problem ▾

You have n cars on a one-lane road, all heading to target miles away. Each car starts at position position[i] and travels at speed[i] mph. Crucially, no car can pass another — if a faster car catches up to a slower one ahead of it, they merge into a fleet and travel together at the slower car's speed. Count the distinct fleets that arrive at the target.

💡 Key Insight

Forget tracking positions at every moment. Instead, compute each car's solo arrival time: time = (target - position) / speed. Two adjacent cars (sorted by position) form a fleet if the car behind has a smaller or equal arrival time — meaning it would reach (or match) the car in front before the target. A car starts a new fleet only if it takes strictly longer than the fleet immediately ahead.

Edge Cases: single car (always 1 fleet); cars at the same position; cars starting at target (time=0); ties in arrival time count as one fleet; all cars in one fleet.

2 The Algorithm: Sort + Monotonic Stack by Time ▾

Think from the target backward. The car closest to the target sets the pace — every car behind must wait for whatever is ahead. Sort by position descending (closest first), compute solo arrival times, then maintain a monotonically increasing stack of distinct fleet arrival times.

1 Pair and sort: Zip position and speed together, then sort by position descending. This orders cars front-to-back from the target's perspective.
2 Compute arrival time: For each car, time = (target - pos) / spd. This is how long the car would take without any obstacles.
3 Stack decision: If time > stack[-1] (or stack is empty), this car is slower than all fleets ahead — it cannot catch them. Push its time as a new fleet. Otherwise, it merges into the fleet ahead and we skip it.
4 Return len(stack): Each stack entry represents exactly one distinct fleet that will arrive at the target independently.

🔑 Why No Pop Is Needed

A stack normally has push and pop. Here we never pop, because when a car's time is ≤ the stack top, it simply merges — we skip the push entirely. The stack grows monotonically (each new entry is strictly larger than the one below), which encodes a strictly decreasing "speed limit as you go back from the target."

3 Why It Works: The Invariant ▾
🔐 The Invariant

At all times, the stack contains the arrival times of all distinct fleets formed so far, in strictly increasing order from bottom to top. The stack top is the arrival time of the most-recently-encountered fleet (the one farthest from the target that has been processed).

Why time comparison is sufficient: Consider two adjacent cars A (closer to target) and B (farther). If B would arrive at or before A's time, B would catch A before the target — they merge. If B arrives after A, B is too slow to catch A — separate fleet. The arrival time encodes all the physics we need; no simulation required.

Why compare only against the stack top? Once a fleet is formed, it travels as a unit at the speed of its slowest member. Any car behind must clear that entire fleet to start its own. The stack top is the most recent fleet's time — the first barrier. If B can't catch fleet Fk (stack top), it definitely can't catch F1, F2, … Fk-1 which arrive even earlier.

⚠️ Off-by-one on strictly greater

Use > (strictly greater), not >=. If two cars have equal arrival times, the one behind would reach the target at precisely the same moment as the fleet ahead — that counts as one fleet, not two. Using >= would incorrectly create a new fleet for ties.

4 The Code (Annotated) ▾

Solution 1 — Sort + Monotonic Stack (Optimal, Interview Pick)

def carFleet(target: int, position: list, speed: list) -> int:
pairs = sorted(zip(position, speed), reverse=True) ← pair cars; sort descending by position (closest to target first)
stack = [] ← stack of distinct fleet arrival times
for pos, spd in pairs: ← process front-to-back
time = (target - pos) / spd ← solo arrival time if no obstacles
if not stack or time > stack[-1]: ← strictly slower than last fleet? → new fleet
stack.append(time) ← push: this car is a new fleet leader
# else: time <= stack[-1] → merges into fleet ahead, no push
return len(stack) ← each stack entry = one distinct fleet
⚠️ Why zip before sort?

zip(position, speed) pairs each car's position with its speed so they stay together after sorting. If you sorted position and speed separately, you'd scramble the pairing — the wrong speed would be assigned to each position.

Solution 2 — Sort + max_time Counter (O(1) extra space)

def carFleet(target: int, position: list, speed: list) -> int:
pairs = sorted(zip(position, speed), reverse=True) ← same sort
fleets = 0; max_time = 0 ← count fleets; track slowest time seen
for pos, spd in pairs:
time = (target - pos) / spd
if time > max_time: ← slower than everything ahead → new fleet
fleets += 1; max_time = time ← update running max
return fleets
💡 Why Solution 2 works with just max_time

The stack top is always the largest time seen so far (stack is monotonically increasing). So stack[-1] is always the running maximum. We can replace the entire stack with a single max_time variable. The trade-off: we lose the ability to inspect intermediate fleet times (only the final count matters here).

Complexity Summary

Brute Force (pairwise simulation): O(n²) — compare every car against every car ahead.
Sort + Stack: O(n log n) time, O(n) space — optimal for the general case.
Sort + max_time: O(n log n) time, O(1) extra space — slightly better space, same logic.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Convert the Problem Domain

The trap is trying to simulate positions at each time unit. Instead, realize that a car's behavior is fully determined by a single number: its solo arrival time. Converting from (position, speed) to time collapses 2D physics into 1D comparison.

Step 2: Determine the Processing Order

Fleets form because of blocking — a car is blocked by whatever is directly ahead. To know what's "ahead" of each car, you must process cars in road order from front to back. Sorting by position descending achieves this.

Step 3: Identify the Stack Pattern

You need to compare each car against "the slowest fleet currently ahead." That's exactly a monotonic stack — each push represents a new, slower fleet. When a car can catch the fleet above, no push needed. Stack size = fleet count.

Step 4: Offer the Space Optimization

Since we only ever look at the stack top, mention that the stack can be replaced with a single max_time variable. This drops space to O(1) extra (beyond the sorted array). Present this as a follow-up, not the first solution.

🗣️ What to Say to the Interviewer

"Instead of simulating motion, I'll compute each car's solo arrival time: (target − pos) / speed.
Then I sort cars by position descending — closest to the target processed first.
I scan front-to-back maintaining a stack of distinct fleet times.
If a car's time is strictly greater than the stack top, it can't catch the fleet ahead — new fleet, push.
Otherwise it merges into the existing fleet — skip.
The stack size is the answer. Time O(n log n), Space O(n).
I can also use just a max_time variable instead of the stack to get O(1) extra space."
🔮 Likely Follow-up Questions
  • "Which cars are in each fleet?" → Store car IDs instead of just times in the stack; accumulate IDs on merge.
  • "Cars in both directions?" → Split by direction, run the algorithm separately on each group.
  • "What if cars can overtake but slow down?" → The arrival-time abstraction still holds; same algorithm.
  • "O(1) space beyond sorting?" → Yes: use max_time instead of stack (Solution 2 above).
