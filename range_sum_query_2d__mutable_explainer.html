LeetCode #308

Range Sum Query 2D - Mutable

Design a data structure that supports both submatrix sum queries and point updates in O(log²n) time.

Medium Advanced Data Structures 2D BIT Update: O(log²n) Query: O(log²n)
1 Understanding the Problem ▾

You are given an m × n matrix of integers. You need to build a data structure that supports two operations efficiently:

  • update(row, col, val) — change the value at matrix[row][col] to val
  • sumRegion(row1, col1, row2, col2) — return the sum of all elements in the rectangle whose top-left corner is (row1, col1) and bottom-right corner is (row2, col2)
💡 Key Insight

This is the 2D generalisation of the classic "Range Sum Query - Mutable" (LC 307). A 1D BIT (Binary Indexed Tree / Fenwick Tree) answers point-update + prefix-sum queries in O(log n). Extending it to 2D gives us O(log² n) for both operations — far better than naïvely recomputing.

✗ Naïve approach

Update: O(1)
Query: O(m·n) per call
Problem: too slow when queries >> updates

✓ 2D BIT approach

Update: O(log m · log n)
Query: O(log m · log n)
Perfect when both ops are frequent!

Concrete example: matrix = [[3,0,1,4,2],[5,6,3,2,1],[1,2,0,1,5],[4,1,0,1,7],[1,0,3,0,5]]. Query sumRegion(2,1,4,3) → elements at rows 2–4, cols 1–3 → 2+0+1 + 1+0+1 + 0+3+0 = 8. After update(3,2,2) → that 0 becomes 2, same query returns 10.

Edge cases: 1×1 matrix; query covering the full matrix; multiple updates to the same cell; update to value 0.

2 The Algorithm: 2D Binary Indexed Tree (Fenwick Tree) ▾

A Binary Indexed Tree (BIT / Fenwick Tree) stores a compact representation of prefix sums that can be updated and queried in O(log n). In 2D, we nest two BITs — one for each axis — giving O(log m · log n) per operation.

The key operation is lowbit(i) = i & (-i) — the lowest set bit of i in binary. It determines how many positions each BIT node "covers".

1 Build: Initialise a 2D array bit[m+1][n+1] of zeros (1-indexed). Process each matrix cell via update(r, c, matrix[r][c]). This builds the BIT in O(mn log m log n).
2 Point update update(r, c, val): Compute delta = val - self.mat[r][c]. Store the new raw value. Then propagate delta up through the BIT: outer loop steps through row indices (add lowbit), inner loop steps through column indices (add lowbit).
3 Prefix sum _prefix(r, c): Walk backwards through BIT (subtract lowbit) for both row and column to accumulate sum for top-left rectangle [0..r][0..c].
4 Region sum query: Use the inclusion-exclusion principle: sumRegion = prefix(r2,c2) - prefix(r1-1,c2) - prefix(r2,c1-1) + prefix(r1-1,c1-1).
📐 Core BIT Formulas
lowbit(i) = i & (-i) ← isolates lowest set bit

Update (r→r+lowbit(r)): bit[r][c] += delta
Query (r→r-lowbit(r)): sum += bit[r][c]
💡 Inclusion-Exclusion for 2D

sumRegion(r1,c1,r2,c2) = prefix(r2,c2) − prefix(r1−1,c2) − prefix(r2,c1−1) + prefix(r1−1,c1−1)

Think of it like Venn diagrams: you subtract the two strips that extend beyond your region, then add back the corner you subtracted twice.

3 Why It Works: The Invariant ▾
🔐 The Invariant

bit[i][j] stores the sum of matrix values in the 2D "responsibility rectangle" whose bottom-right corner is (i, j) and which spans lowbit(i) rows and lowbit(j) columns.

Why O(log n) per axis? When you query prefix(r, c), you walk from r towards 0 by repeatedly subtracting lowbit(r). Each step removes at least one bit from r's binary representation, so after at most log₂(n) steps, r reaches 0. Same for columns. Total: O(log m · log n).

Why does the update propagate correctly? When we add lowbit(i) to move to the next BIT node, we "tell" all nodes whose responsibility window contains index i about the change. The parent-child relationship in BIT guarantees every ancestor is updated exactly once.

Query path for prefix(6)

i=6 (110): sum += bit[6], i=4
i=4 (100): sum += bit[4], i=0
stop. Total: 2 steps.

Update path for index 5

i=5 (101): bit[5]+=d, i=6
i=6 (110): bit[6]+=d, i=8
i=8 (1000): bit[8]+=d, done.

Edge cases handled: 1-indexed BIT sidesteps the lowbit(0)=0 infinite-loop. Single-cell queries work because prefix(r,c) − prefix(r-1,c) − prefix(r,c-1) + prefix(r-1,c-1) = exactly one cell.

4 The Code (Annotated) ▾

Solution 1 — 2D BIT (Interview Pick)

class NumMatrix: ← class wraps BIT state
def __init__(self, matrix): ← constructor builds BIT
self.m = len(matrix) ← rows count
self.n = len(matrix[0]) ← cols count
self.mat = [[0]*(self.n) for _ in range(self.m)] ← stores raw values (0-indexed)
self.bit = [[0]*(self.n+1) for _ in range(self.m+1)]← BIT is 1-indexed → +1 padding
for r in range(self.m): ← build BIT from matrix
for c in range(self.n):
self._add(r+1, c+1, matrix[r][c]) ← 1-indexed BIT add
self.mat[r][c] = matrix[r][c] ← remember raw value for delta calc
def _add(self, r, c, delta): ← propagate delta up through BIT
i = r
while i <= self.m: ← traverse rows upward
j = c
while j <= self.n: ← traverse cols upward
self.bit[i][j] += delta ← update BIT node
j += j & (-j) ← j += lowbit(j) → next col node
i += i & (-i) ← i += lowbit(i) → next row node
def update(self, row, col, val): ← public update (0-indexed input)
delta = val - self.mat[row][col] ← change amount, not new value
self.mat[row][col] = val ← store new raw value
self._add(row+1, col+1, delta) ← propagate (convert to 1-indexed)
def _prefix(self, r, c): ← prefix sum [0..r][0..c] (1-indexed)
total = 0
i = r
while i > 0: ← walk rows backwards
j = c
while j > 0: ← walk cols backwards
total += self.bit[i][j] ← accumulate BIT node value
j -= j & (-j) ← j -= lowbit(j) → prev col node
i -= i & (-i) ← i -= lowbit(i) → prev row node
return total
def sumRegion(self, r1, c1, r2, c2): ← public query (0-indexed)
return (self._prefix(r2+1, c2+1)
- self._prefix(r1, c2+1) ← subtract top strip
- self._prefix(r2+1, c1) ← subtract left strip
+ self._prefix(r1, c1)) ← add back double-subtracted corner
⚠️ Common Mistake: Forgetting Delta vs Value

The BIT stores deltas, not absolute values. When you call update(row, col, val), you must compute delta = val - old_value and propagate only the delta. This is why we store self.mat — to know what was there before.

⚠️ Index Offset: 0-indexed input, 1-indexed BIT

LeetCode calls update(row, col, val) and sumRegion(r1,c1,r2,c2) with 0-indexed positions, but the BIT itself is 1-indexed (so lowbit never gets called on 0). Always add +1 when writing to/reading from the BIT.

5 🔬 Algorithm Deep-Dive: 2D Binary Indexed Tree ▾

Origin: The Binary Indexed Tree (BIT) was introduced by Peter Fenwick in 1994 (hence also called a Fenwick Tree) as a space-efficient alternative to segment trees for prefix sum queries with point updates. The 2D extension naturally nests two BITs.

🔐 Core Invariant

bit[i][j] holds the sum of all original matrix values mat[r][c] where i - lowbit(i) < r ≤ i and j - lowbit(j) < c ≤ j. In other words, each BIT cell is responsible for a specific rectangular "chunk" of the original matrix.

Why lowbit works: In binary, lowbit(i) = i & (-i) isolates the least-significant set bit. For i = 6 = 110₂, lowbit(6) = 010₂ = 2, meaning BIT[6] covers 2 elements. For the update path, adding lowbit moves to the next ancestor. For the query path, subtracting lowbit moves to the previous disjoint region.

810008[1..8] — 8 elements

Generalization: BIT is perfect whenever you need both prefix-range queries and point updates. Use a Segment Tree instead when you need range updates (lazy propagation) or non-associative operations (like max/min with range updates).

💡 When to Recognize 2D BIT

Problem signals: "given a 2D grid", "support update(row, col, val)", "support sumRegion(r1,c1,r2,c2)", "O(log n) time required for each operation". If the problem says "immutable", use 2D prefix sum instead — it's simpler and O(1) per query.

6 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Pattern

Signals: two operations (update + query), 2D grid, both must be efficient. This screams "data structure design problem". The 2D nature with sub-rectangle sums points to BIT or Segment Tree.

Step 2: Start with the 1D Simpler Version

Say: "I know the 1D version — LC 307 Range Sum Query Mutable uses a 1D BIT for O(log n) update + prefix-sum." Then: "Can I extend BIT to 2D?" The answer is yes — just nest the column loop inside the row loop.

Step 3: Derive the Key Operations

Write the lowbit operation first. Derive update (add lowbit) and query (subtract lowbit). State the invariant: BIT[i][j] covers a rectangular block. Apply inclusion-exclusion for region queries.

Step 4: Handle the Index Offset Cleanly

Immediately establish: "I'll keep self.mat for raw values (0-indexed) and self.bit 1-indexed with +1 padding." This avoids bugs in the interview.

Step 5: State Complexity

Build: O(mn log m log n). Update: O(log m log n). Query: O(log m log n). Space: O(mn). Compare to naïve: update O(1) + query O(mn) — BIT wins when queries are frequent.

🗣️ What to Say to the Interviewer

"The problem requires frequent updates AND range sum queries in 2D.
A naïve 2D prefix sum has O(1) query but O(mn) rebuild on every update — too slow.
I'll use a 2D Binary Indexed Tree (Fenwick Tree), which gives O(log m · log n) for both.
The core trick is lowbit(i) = i & (-i) — it tells us how many elements each BIT node covers.
For region queries, I apply 2D inclusion-exclusion using four prefix-sum calls.
I'll store the raw matrix separately to compute deltas on update."
💡 Follow-Up Questions to Anticipate
  • "Can you extend to range updates?" → Need Segment Tree with lazy propagation.
  • "What about max queries?" → BIT only works with invertible operations (sum). Use Segment Tree for max.
  • "Memory-optimize?" → BIT already uses O(mn) — same as the matrix itself.

8 Summary & Key Takeaways ▾
A 2D BIT gives O(log m · log n) for both point-update and rectangle-sum queries.
lowbit(i) = i & (-i) is the single operation that makes BIT work: it defines each node's responsibility window.
Store raw matrix values separately — updates propagate deltas, not new values, into the BIT.
Region queries reduce to 4 prefix-sum calls via 2D inclusion-exclusion: top-right − top-strip − left-strip + top-left-corner.
The BIT is 1-indexed; always add +1 when converting from the 0-indexed LeetCode API.
🎯 When to Use 2D BIT
  • 2D grid with frequent point updates + prefix/rectangle sum queries
  • Need O(log² n) per operation (BIT is simpler to code than 2D Segment Tree)
  • Operations are prefix-sum based (not range-update, not max/min — use Seg Tree then)
