LeetCode #502

IPO

Maximize capital after completing at most k projects, each with a required starting capital and a fixed profit — choosing greedily with two heaps.

Hard Heaps Greedy + Heap Time: O(n log n) Space: O(n)
1 Understanding the Problem ▾

You are a startup founder with w dollars and a menu of n projects. Each project requires a minimum starting capital and pays out a fixed profit once completed. You can complete at most k projects in sequence — the profit from one adds to your wallet before the next. Which projects should you pick to maximize your final capital?

💡 Key Insight

At any moment, greedily pick the highest-profit project you can currently afford. As your wallet grows from each completed project, more projects become affordable. Since profits are non-negative, more money never hurts — so this local greedy choice is globally optimal.

Edge Cases: k=0 (return w unchanged), all capitals > w forever (break early — wallet never grows), k > n (only n choices exist), all profits equal (any affordable order is fine).

2 The Algorithm: Two-Heap Greedy ▾

Two specialized data structures work together. A min-heap keyed by required capital holds all not-yet-affordable projects. A max-heap keyed by profit holds everything currently affordable. Each of the k rounds drains newly affordable projects from the first heap into the second, then pops the maximum profit.

1 Build min-heap from all (capital[i], profit[i]) pairs. Use heapify() for O(n) construction — it's faster than n individual pushes.
2 Repeat for each of k rounds (or until no affordable project exists).
3 Unlock phase: While min-heap is non-empty and the top project's capital ≤ current wallet, pop it and push its profit (negated) onto the max-heap. This loop runs amortized O(n) total across all rounds — each project is moved at most once.
4 Pick phase: If max-heap is non-empty, pop the highest-profit project and add to wallet. If max-heap is empty, no project is affordable — break immediately (wallet can't grow further).
5 Return wallet after all rounds complete.
Why Two Separate Heaps?

A single sorted structure can't serve both roles efficiently. The min-heap answers "what just became affordable as my wallet grew?" in O(log n). The max-heap answers "what's the best currently available project?" in O(log n). Together they encode the greedy policy with no wasted work.

3 Why It Works: The Invariant ▾
The Invariant

At the start of every pick phase, the max-heap contains exactly the set of all projects whose required capital ≤ current wallet. The profit we pop is therefore the globally optimal choice from all currently executable projects.

Greedy exchange argument: Suppose an optimal solution picks k projects P1, P2, ..., Pk in some order. Any permutation of these k projects yields the same total profit (each profit is added exactly once, regardless of order). Therefore, the only question is which k projects to pick, not the order. At each step, greedily choosing the highest-profit affordable project ensures we pick the best possible subset — any deviation can only leave a higher-profit project unpicked while a lower-profit one is chosen instead.

Why non-negative profits are required: If profits could be negative, picking a project could reduce wallet and prevent more projects from becoming affordable. The problem guarantees profits[i] ≥ 0, so every completed project either helps or is neutral — it never hurts.

Common Mistake: Not Checking Empty Max-Heap

After the unlock phase, the max-heap may still be empty if every remaining project costs more than the current wallet. In this case, wallet can never grow (no project can be executed), so future iterations of the for-loop would be wasted. Always add if not max_heap: break after the unlock while-loop.

The Negation Trick for Max-Heap

Python's heapq is a min-heap only. To simulate a max-heap: store -profit instead of profit. The smallest negative number is the largest profit. When popping: -heapq.heappop(max_heap) restores the real value. Always document this with a comment — it's easy to forget.

Complexity Analysis

Time: O(n log n + k log n)
Building min-heap: O(n). Each project is pushed/popped from each heap at most once: O(n log n). The k profit pops: O(k log n). Since k ≤ n, overall O(n log n).

Space: O(n) — both heaps together hold at most n items total.

5 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Pattern

Signal words: "maximize after k operations", "items unlock as a resource grows", "feasible set changes dynamically". This screams Greedy + Heap. The expanding feasibility set (wallet grows → more projects unlock) is the heap's job to track.

Step 2: Justify the Greedy

Ask yourself: "Does picking a lower-profit project now ever make a higher-profit project available later?" Answer: No — profits are non-negative, so our wallet only grows. The exchange argument shows any permutation of the chosen set yields the same total. Greedy is provably optimal.

Step 3: Design the Data Structures

We need two things each round: (a) find all projects that just became affordable as wallet grew → min-heap by capital. (b) pick the highest-profit one → max-heap by profit. Two heaps is the natural fit for these two distinct operations.

Step 4: Handle the Python Constraint

Python only provides min-heap. To get max-heap behavior: negate the profit on push, negate again on pop. Mention this explicitly to the interviewer — it shows awareness of language-specific details.

What to Say to the Interviewer

"This is a greedy problem. At each step, we should pick the highest-profit
project we can currently afford — because profits are non-negative, earning
more now only helps, never hurts future options.

The greedy is correct by an exchange argument: swapping order among the

pick matters. Always pick max profit from the affordable set.

For the implementation, I'll use two heaps:
- Min-heap by capital: efficiently identifies projects that just unlocked.
- Max-heap by profit: gives the best current choice in O(log n).

Shall I code it up?"
Follow-Up Questions to Prepare For
  • What if profits can be negative? — Greedy breaks. Need DP or more careful selection.
  • What if projects have durations or deadlines? — Different problem class (interval scheduling).
  • What if k > n? — We only have n projects; just process all n. The for-loop exits naturally when min-heap is empty.
  • Can you do O(n) space instead of O(n)? — Not generally; we need to store all projects in heaps.

7 Summary & Key Takeaways ▾
Use a min-heap by capital to track when projects become affordable — pop them as wallet grows.
Use a max-heap by profit (negate values in Python) to always pick the best currently available project.
Greedy is correct because profits ≥ 0: more capital never hurts, and project order doesn't change total gain.
Early termination: break when max-heap is empty after unlock phase — wallet can no longer grow.
heapify() is O(n) — always prefer it over n individual pushes for initial heap construction.
When to Use Greedy + Two Heaps
  • Maximize/minimize after at most k operations
  • Feasible set expands as a resource (capital, time, slots) grows
  • Local greedy choice is globally optimal (non-negative incremental gains)
  • Need O(log n) "best available" query after each step
