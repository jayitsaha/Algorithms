LeetCode #199

Binary Tree Right Side View

Given the root of a binary tree, imagine yourself standing on the right side of it. Return the values of the nodes you can see, ordered from top to bottom.

Medium Trees BFS Level Order Time: O(n) Space: O(w)
1 Understanding the Problem ▾

Imagine you're standing far to the right of a binary tree, looking left. At each horizontal level of the tree, you can only see the rightmost node — everything behind it is hidden. The question asks: which nodes can you actually see, from top to bottom?

💡 Key Insight

The right side view is the last node encountered at each depth level. You don't need to always follow the right child — if a node has no right child but its left child exists, that left child is still the rightmost visible node at its depth.

Consider the example tree with input [1, 2, 3, null, 5, null, 4]:

1 ← visible from right: 1 / \ 2 3 ← visible from right: 3 (blocks 2) \ \ 5 4 ← visible from right: 4 (blocks 5)

Output: [1, 3, 4] — exactly one value per level, always the rightmost node you can see.

✓ When right child exists

At depth 1: node 3 blocks node 2.
At depth 2: node 4 blocks node 5.
Right child is always the rightmost.

⚠️ When only left child exists

If node 3 had no right child,
node 5 would be the only node
at depth 2 — still visible!

Edge Cases: null root → return []. Single node → [root.val]. Left-skewed tree — every visible node is technically a left child, but it's still the rightmost at its depth. Perfectly balanced tree — always the rightmost child at each level.

2 The Algorithm: BFS with Level-Size Snapshot ▾

The cleanest solution uses Breadth-First Search (BFS) — processing the tree level by level using a queue. The critical trick is: after finishing all nodes at one level, the last node we dequeued is the rightmost visible one.

1 Handle null root: If root is null, return [] immediately — nothing is visible.
2 Seed the queue: Create q = deque([root]) and an empty result list.
3 Outer loop — one level per iteration: While the queue is non-empty, take a snapshot: level_size = len(q). This count captures exactly how many nodes belong to the current level, even as we enqueue children.
4 Inner loop — consume the level: Dequeue exactly level_size nodes. For each node, enqueue its left child then right child (if they exist). After the last iteration, node holds the rightmost node of this level.
5 Record the right side: After the inner loop ends, append node.val to result. This is the last-seen node at this depth — exactly what you'd see from the right.
6 Return result when the queue is empty (all levels processed).
🔑 DFS Right-First Alternative

You can solve this recursively by traversing right child before left child. Pass the current depth down. If depth == len(result), this is the first time we visit this depth — and since we go right-first, that first visit is always the rightmost node. Append it. O(n) time, O(h) space (call stack).

3 Why It Works: The Invariant ▾
🔐 The BFS Level Invariant

At the start of each outer loop iteration, the queue contains exactly all nodes at the current depth, ordered left to right. This property is maintained because: (1) we start with just the root at depth 0, and (2) each node's children are enqueued left-before-right, preserving the left-to-right ordering for the next level.

Why does the level_size snapshot work? When we enter the inner loop, len(q) equals the number of nodes at the current level. As we dequeue them, we enqueue their children — but those children belong to the next level. By looping exactly level_size times, we process only the current level. Afterward, the queue holds precisely the next level's nodes.

Why is the last dequeued node the rightmost? Within a level, nodes are ordered left-to-right in the queue (proven by induction from the BFS enqueue order). Dequeuing FIFO means we process them left-to-right. The last one dequeued is therefore the rightmost node at that depth — exactly what's visible from the right side.

⚠️ Common Misconception: "Always Follow Right Pointer"

A beginner might think: "just traverse right children until null, recording each." This only works for a right-skewed tree. In a tree where a node has a left child but no right child, the left child is the rightmost at its depth — a pure right-child traversal would miss it entirely.

DFS correctness: In right-first DFS, we visit right subtrees before left subtrees. At any depth, the first node visited is the one farthest to the right. depth == len(result) catches exactly this "first visit per depth" condition. All subsequent nodes at the same depth are more leftward — we skip them (just recurse without appending).

4 The Code (Annotated) ▾

Solution 1 — BFS Level Order (Interview Pick) · O(n) time · O(w) space

from collections import deque
def rightSideView(root): ← Entry point; root is a TreeNode or None
if not root: ← Null tree: nothing visible → empty list
return []
q = deque([root]) ← BFS queue seeded with just the root
result = []
while q: ← Each outer iteration = one tree level
level_size = len(q) ← CRITICAL: snapshot before inner loop
for _ in range(level_size): ← Process exactly this level's nodes
node = q.popleft() ← FIFO: dequeue leftmost of current level
if node.left: ← Enqueue left child first (left → right order)
q.append(node.left)
if node.right: ← Then enqueue right child
q.append(node.right)
result.append(node.val) ← After inner loop, node = last = rightmost
return result ← One value per tree level
⚠️ Why collections.deque and not a plain list?

list.pop(0) costs O(n) because it shifts all remaining elements left. deque.popleft() costs O(1) because a deque uses a doubly-linked structure. For a tree with n nodes, this changes total BFS cost from O(n²) to O(n). Always use collections.deque for BFS queues.

Solution 2 — DFS Right-First (Recursive) · O(n) time · O(h) space

def rightSideView(root): ← DFS approach; good for balanced trees (O(log n) space)
result = []
def dfs(node, depth): ← depth starts at 0 for root
if not node: ← Base case: null node
return
if depth == len(result): ← First node visited at this depth (going right-first)
result.append(node.val) ← This is the rightmost node at this depth
dfs(node.right, depth + 1) ← Explore right subtree first
dfs(node.left, depth + 1) ← Then left subtree (may be skipped if depth already seen)
dfs(root, 0)
return result
💡 Complexity Summary

BFS (Solution 1): Time O(n) — each node enqueued and dequeued once. Space O(w) — queue holds at most one full level (at most n/2 nodes for a perfect tree → O(n) worst case).
DFS (Solution 2): Time O(n) — visits every node once. Space O(h) — call stack = tree height. O(log n) for balanced; O(n) for a skewed (linked-list-like) tree.
Trade-off: BFS is more intuitive for "level-based" problems. DFS uses less space when the tree is balanced.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Reframe the Problem

"Standing on the right and looking left — I see one node per level: the rightmost at each depth. This is a level-by-level traversal problem. The output has exactly as many elements as the tree has levels."

Step 2: Identify the Pattern

"Level-by-level traversal → BFS with a queue. I'll process nodes in groups (one group = one level). After each group, I record the last node I processed — that's the rightmost visible one."

Step 3: Explain the level_size Trick

"The key is snapshotting level_size = len(q) before the inner loop. When I dequeue nodes and enqueue their children, the children go to the back of the queue. The snapshot tells me how many nodes belong to this level — I loop exactly that many times."

Step 4: Mention the DFS Alternative

"Alternatively: right-first DFS. I recurse right before left, passing depth. The first node visited at each depth is the rightmost — checked by depth == len(result). Same O(n) time, O(h) space instead of O(w)."

🗣️ What to Say to the Interviewer

"I see one visible node per level — the rightmost one. That means BFS level-order traversal.
I use a deque for O(1) popleft. Seed with root. While queue not empty:
Snapshot level_size = len(q). Then dequeue exactly level_size nodes.
For each: enqueue left then right child if they exist.
After the inner loop: node is the last dequeued = rightmost at this level.
Append node.val to result. Repeat for next level.
Time: O(n), Space: O(w) — queue holds at most one full level.
For a balanced tree, w = O(n/2), so worst case O(n) queue space."

7 Summary & Key Takeaways ▾
Right side view = last node at each depth, not just the rightmost child pointer
BFS with level_size snapshot is the canonical pattern for level-by-level tree problems
Always use collections.deque for BFS queues — popleft() is O(1), list.pop(0) is O(n)
DFS right-first is an elegant alternative: first visit per depth = rightmost node
The same BFS skeleton works for: left side view (first node per level), level averages, level max/min, zigzag traversal
