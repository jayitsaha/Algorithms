LeetCode #1014

Best Sightseeing Pair

Given an integer array values, return the maximum score of a sightseeing pair (i, j) where i < j and score = values[i] + values[j] + i - j.

Medium Kadane's Algorithm Track Best i + A[i] Time: O(n) Space: O(1)
1 Understanding the Problem ▾

Imagine you're visiting scenic spots, each with a beauty score. When you travel from spot i to spot j (you must visit i before j), your total enjoyment is the sum of both spots' scores minus the distance you traveled. The formula is:

The Score Formula
score(i, j) = values[i] + values[j] + i − j

Think of +i as a "nearness bonus" for the starting spot (spots earlier in the array are rewarded) and -j as a "distance penalty" (the farther right you travel, the more it costs). You want the pair (i, j) with i < j that maximizes this score.

💡 Key Insight: Split the Formula by Index

Rearrange the formula: score(i, j) = (values[i] + i) + (values[j] − j). The first group depends only on i. The second depends only on j. They're completely independent! So for a fixed j, the best partner i is simply the one that maximizes values[i] + i. We can track this as a running maximum — no need to re-examine previous i values.

✓ Example: values = [8, 1, 5, 2, 6]

i=0, j=2: 8+5+0−2 = 11 ✓
i=0, j=4: 8+6+0−4 = 10
i=2, j=4: 5+6+2−4 = 9
Answer: 11

✗ Naive O(n²): Try All Pairs

n=50,000 → 2.5 billion pairs
Time Limit Exceeded!
We need a smarter approach.

Edge Cases: Minimum two elements (only one pair to evaluate). All equal values (answer dominated by smallest j−i). Monotonically decreasing values (best i is always index 0 since values[i]+i is maximized early).

2 The Algorithm: One-Pass Running Maximum ▾

We scan j from left to right (index 1 onward). At each j, we combine the right term (values[j] − j) with the best left term best_i = max(values[k] + k) seen for any valid k < j. After computing the candidate score, we update best_i to include j itself (since j will serve as a valid i for future pairs).

1 Initialize: best_i = values[0] + 0 — the left term for the only valid starting index. ans = 0 — best score found.
2 For each j from 1 to n-1: Compute candidate = best_i + (values[j] − j). Update ans = max(ans, candidate).
3 Update best_i (after ans): best_i = max(best_i, values[j] + j). This allows j to serve as the left endpoint for future pairs. Critically, this happens after updating ans.
4 Return ans: After scanning all j, ans is the maximum sightseeing score over all valid (i, j) pairs.
🔑 Order of Operations is Critical

Always compute ans = max(ans, best_i + values[j] − j) before doing best_i = max(best_i, values[j] + j). Reversing these would allow i = j (same index for both endpoints), which violates the constraint i < j and can inflate the score incorrectly.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At the start of every iteration for index j, the variable best_i holds the maximum of (values[k] + k) for all k in [0, j−1]. This means when we evaluate j as the right endpoint, we always have the globally optimal left contribution available in O(1).

Proof of correctness: The maximum over all pairs (i, j) with i < j can be written as:

Reformulation
max over all j: [ max over i < j of (values[i]+i) ] + (values[j]−j)

For each fixed j, the inner maximization is independent of j — it's just the best "left term" seen before j. A running maximum captures this exactly, updated in constant time per step.

Connection to Kadane's Algorithm: In classic Kadane (max subarray), you track the best subarray ending at position i−1 to extend it at position i. Here, you track the best "left endpoint contribution" seen before j, and combine it with j's "right endpoint contribution". The structure is identical: carry the best past value, then update the carried value to include the current position.

⚠️ Common Mistake: Initializing best_i Incorrectly

A beginner might set best_i = values[0] (forgetting the +0) or never update best_i inside the loop. The first gives wrong answers when the index component matters. The second only considers index 0 as the left endpoint, missing cases like [1, 9, 1, 1] where index 1 (left term = 9+1=10) beats index 0 (left term = 1+0=1).

4 The Code (Annotated) ▾

Solution 1 — One-Pass Kadane Variant (Optimal · Interview Pick)

def maxScoreSightseeingPair(values):
best_i = values[0] + 0 ← left term for index 0: values[0]+i where i=0
ans = 0 ← tracks the maximum score found so far
for j in range(1, len(values)): ← j is the right endpoint; must be > 0
ans = max(ans, best_i + values[j] - j) ← combine best left term with current right term
best_i = max(best_i, values[j] + j) ← update: can j be a better left for future pairs?
return ans ← maximum score over all valid (i,j) with i
⚠️ Why Not Use Brute Force?

The brute force nested loop (try every pair) is O(n²). For n = 50,000, that's 2.5 billion iterations — well past the typical 10⁸ operations/second limit. The one-pass approach is O(n): a single linear scan, two comparisons per step, no extra memory.

Solution 2 — Brute Force (O(n²), for understanding only)

def maxScoreSightseeingPair_brute(values):
n, ans = len(values), 0
for i in range(n): ← every possible left endpoint
for j in range(i+1, n): ← every possible right endpoint > i
score = values[i] + values[j] + i - j
ans = max(ans, score) ← O(n²) — TLE for n > ~10⁴
return ans
💡 Complexity Summary

One-Pass Kadane Variant: Time O(n), Space O(1) — optimal
Brute Force: Time O(n²), Space O(1) — correct but too slow
The Kadane approach compresses a 2D search into a 1D scan by carrying a single variable: best_i.

🔬 Algorithm Deep-Dive: Kadane's Algorithm

Origin: Joseph Kadane, 1984. Originally solved the Maximum Subarray problem in O(n) time and O(1) space. Generalized, it applies to any problem where an optimal 2-endpoint solution can be decomposed into a "left part" and a "right part" with no cross-dependency.

Core invariant: At each position, maintain the single best "past contribution" that can be combined with the current element. Discard all other history.

Generalization pattern: If score(i, j) = f(i) + g(j) for functions f and g that depend only on i and j respectively, you can solve "maximize over all i < j" in O(n): scan j, maintain running max of f(i) for i < j.

Recognize when: The problem involves pairs (i, j) with a score formula separable by index. Problems: Best Time to Buy/Sell Stock (profit = price[j]−price[i] = −price[i] + price[j]), this problem, Maximum Subarray (sum of subarray from i to j).

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Write Out the Formula Explicitly

Write score = values[i] + values[j] + i − j on the board. This is your starting point. The formula contains everything you need to find the optimization strategy.

Step 2: Algebraically Separate by Index

Rearrange: (values[i] + i) + (values[j] − j). Verbalize: "The first group involves only i, the second only j — they don't interact. This is separable."

Step 3: Recognize the Running Maximum Pattern

For a fixed j, we want the maximum (values[i] + i) over all i < j. Since we scan j left to right and this set grows by one per step, a running maximum maintains it in O(1) per update — no nested loop needed.

Step 4: State the Update Order

Explain proactively: "I compute ans using the current best_i before updating best_i to include j. This ensures we never use the same index as both i and j."

🗣️ What to Say to the Interviewer

"The score is values[i] + values[j] + i - j. I'll rearrange it into two parts:
(values[i] + i) + (values[j] - j). These are independent — one depends only on i, one on j.
So for each right endpoint j, the best left endpoint is whichever i maximizes values[i]+i.
I'll track that as a running maximum — best_i — updated as I scan.
Key: I update ans first, then update best_i. This enforces i < j strictly.
Total: O(n) time, O(1) space. No nested loop needed."
🔮 Likely Follow-up Questions
  • "What if i can equal j?" → Then best_i should be updated before ans. But the problem says i < j strictly.
  • "Return the actual indices, not just the score?" → Track best_i_idx and record best_j when ans is updated.
  • "What if the score formula was values[i] * values[j] / (j - i)?" → Cannot separate; different approach needed (no clean Kadane).
  • "Extend to 3 spots (i < j < k)?" → Two-layer Kadane: carry best(values[i]+i), then best pair ending at j, then combine at k.

Split the formula: (values[i]+i) + (values[j]−j) — the algebraic insight that unlocks O(n)
Track only max(values[i]+i) as a running variable — one number replaces an entire inner loop
Update ans BEFORE best_i each iteration — enforces the strict i < j constraint
This is Kadane's generalized pattern: any score(i,j) = f(i) + g(j) can be solved this way
