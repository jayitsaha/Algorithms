LeetCode #664

Strange Printer

Find the minimum number of turns a strange printer needs to print a string, where each turn prints one character from position i to j (overwriting existing content).

Hard Dynamic Programming Interval DP Time: O(n³) Space: O(n²)
1 Understanding the Problem ▾

A strange printer works in turns. On each turn it selects a single character and prints it across any contiguous range of positions, overwriting whatever was there before. Given a target string, find the minimum number of turns needed to produce it (starting from a blank page).

💡 Key Insight

Think of each turn as a horizontal paint stroke of one color. Strokes can overlap — a later stroke covers an earlier one. The optimal strategy finds the fewest strokes to produce the final "painting." Crucially: if the first and last character of an interval are the same, we can extend the first character's stroke to also cover the last position at no extra cost (it gets painted over by inner strokes, but the endpoints match).

Example: s = "aaabbb"

Turn 1: print 'a' at [0..5] → "aaaaaa"
Turn 2: print 'b' at [3..5] → "aaabbb"
Answer: 2 turns

Example: s = "aba" (walkthrough target)

Turn 1: print 'a' at [0..2] → "aaa"
Turn 2: print 'b' at [1..1] → "aba"
Answer: 2 turns

Why is this hard? Greedy approaches fail because local decisions interact globally. Printing 'a' first and then 'b' may seem obvious, but with strings like "abacaba", the optimal order requires analyzing how distant matching characters can share strokes. This demands examining all possible splits of every sub-interval — the hallmark of Interval DP.

⚠️ Preprocessing: Collapse Consecutive Duplicates

If s = "aab", the two 'a's can be printed in a single stroke — one turn, not two. So first compress: remove any character equal to its predecessor. "aabbc" → "abc". This strictly reduces n without changing the answer and is important for correctness in the merge step.

Edge cases: Empty string → 0 turns. Single character → 1 turn. All same characters → 1 turn (one big stroke). Alternating like "ababab" → n/2 + 1 turns in the worst case.

2 The Algorithm: Interval DP with Merge Trick ▾

Interval DP solves problems over contiguous subarrays or substrings by building solutions for small intervals and combining them into larger ones. The key definition here: dp[i][j] = minimum turns to print exactly the substring s[i..j].

1 Compress: Remove consecutive duplicate characters. Consecutive same characters cost one stroke regardless of count — only the boundaries matter.
2 Base case: Every single character takes exactly 1 turn: dp[i][i] = 1 for all i.
3 Fill by increasing length: For lengths L = 2, 3, …, n and starting index i, set j = i + L − 1. Compute dp[i][j] from already-computed smaller subproblems.
4 Merge trick (the critical optimization): If s[i] == s[j], set dp[i][j] = dp[i][j-1]. The stroke printing s[i] can extend to also cover s[j] for free — s[j] is "absorbed" into the stroke that handles s[i]. The inner characters s[i+1..j-1] will overwrite the middle portion as needed.
5 Split search (when s[i] != s[j]): Try every split point k ∈ [i, j−1]. Combine the cost of s[i..k] and s[k+1..j]: dp[i][j] = min over all k of (dp[i][k] + dp[k+1][j]).
6 Answer: dp[0][n-1] gives the minimum turns for the entire string.
📐 Recurrence Relations
Base: dp[i][i] = 1
Merge: if s[i] == s[j]: dp[i][j] = dp[i][j−1]
Split: else: dp[i][j] = min(dp[i][k] + dp[k+1][j]) for k in [i, j−1]
Answer: dp[0][n−1]
🔑 Why the Merge Trick Is Correct

Suppose s[i] == s[j]. Consider any optimal printing strategy for s[i..j-1]. That strategy must include at least one stroke that covers position i (to print s[i]). We can extend that same stroke to also reach position j, since s[j] is the same character. The intermediate positions i+1..j-1 will be overwritten by subsequent strokes — and that's fine. So dp[i][j] ≤ dp[i][j-1]. Since dp[i][j] ≥ dp[i][j-1] (we can't do better than ignoring one character), we get equality.

3 Why It Works: Optimal Substructure & Overlapping Subproblems ▾
🔐 The Invariant

When we compute dp[i][j], every smaller subproblem dp[a][b] with b−a < j−i has already been computed. This bottom-up property means dp[i][j] is built from correct, finalized values — no circular dependencies.

Optimal substructure: Any optimal printing strategy for s[i..j] can be decomposed at some "last boundary" — a split point k where the strokes for the left half and right half are independent. For every possible k, the cost is dp[i][k] + dp[k+1][j]. By trying all k and taking the minimum, we consider every valid decomposition.

Overlapping subproblems: Without memoization, a recursive approach recomputes dp[2][5] whenever it appears as a sub-case of different parent intervals (e.g., dp[0][7] split at k=1 and dp[1][7] split at k=1 both need dp[2][7], which needs dp[2][5]). The 2D table ensures each of the O(n²) subproblems is solved exactly once.

Why fill diagonal-by-diagonal (by length)? dp[i][j] depends on dp[i][k] and dp[k+1][j] for k in [i,j−1]. These are strictly shorter intervals. By processing length 1, then 2, then 3, ..., all dependencies are ready when needed.

💡 Interval DP Template Recognition

The three-loop structure (length, i, k) is the canonical interval DP skeleton. You'll see it in Burst Balloons, Matrix Chain Multiplication, Minimum Cost to Cut a Stick, and Palindrome Partitioning II. Master this pattern and all of them become approachable.

4 The Code (Annotated) ▾

Solution 1 — Bottom-Up Tabulation (Optimal, Interview Pick)

def strangePrinter(s: str) -> int:
# Compress consecutive duplicates ← "aab" → "ab": same stroke covers both 'a's
s = ''.join(c for i,c in enumerate(s)
if i == 0 or c != s[i-1])
n = len(s)
if n == 0: return 0 ← edge case
dp = [[0]*n for _ in range(n)] ← n×n table; only upper triangle used
for i in range(n): ← Base: single chars take 1 turn each
dp[i][i] = 1
for length in range(2, n + 1): ← Outer loop: growing interval lengths
for i in range(n - length + 1): ← i = left bound of interval
j = i + length - 1 ← j = right bound
if s[i] == s[j]: ← Merge trick: extend s[i]'s stroke to cover j
dp[i][j] = dp[i][j-1] ← s[j] is free — same char as s[i]
else: ← No free merge; try all split points
dp[i][j] = float('inf')
for k in range(i, j): ← k splits [i..j] into [i..k] + [k+1..j]
dp[i][j] = min(dp[i][j],
dp[i][k] + dp[k+1][j]) ← combine two independent sub-intervals
return dp[0][n-1] ← Min turns for full string
⚠️ Common Mistake: Not Compressing Duplicates

If you skip the compression step, the merge trick can give wrong results. For "aab": without compression, when we evaluate dp[0][2] for "aab", s[0]='a'==s[2]='b' is false — but the real answer is 2 (not 3). Wait, "aab" → s[0]='a', s[1]='a', s[2]='b'. Here dp[0][1] = 1 (one stroke 'a' covers both), dp[0][2] = 2. Actually the algorithm handles this correctly even without compression, but compression can save up to n/2 operations in degenerate cases like "aaaa…".

Solution 2 — Top-Down Memoization (Alternative)

from functools import lru_cache
def strangePrinter(s: str) -> int:
s = ''.join(c for i,c in enumerate(s) if i==0 or c!=s[i-1])
n = len(s)
@lru_cache(maxsize=None) ← Python's built-in memoization
def dp(i, j): ← Recursive: min turns to print s[i..j]
if i > j: return 0 ← Empty range: 0 turns
if i == j: return 1 ← Single char: 1 turn
if s[i] == s[j]: ← Merge trick
return dp(i, j-1)
return min(dp(i,k) + dp(k+1,j) for k in range(i,j)) ← try all splits
return dp(0, n-1)
💡 Complexity Summary

Both approaches: Time O(n³) — O(n²) subproblems × O(n) splits each. Space O(n²) — the DP table.
After compression: n is at most the number of distinct character transitions — can be much smaller than original input length in practice.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Interval DP Signal

Key signals: (1) we operate on a string/array as a whole, (2) optimal answer depends on combining optimal answers for sub-intervals, (3) no greedy choice is obvious — what's optimal at position i depends on positions far away. This pattern screams "Interval DP".

Step 2: Define dp[i][j] Precisely

Always start by saying exactly what dp[i][j] means: "dp[i][j] = minimum number of printer turns to produce the substring s[i..j]." Then state the base case: dp[i][i] = 1.

Step 3: Find the Merge Condition

Ask: "Is there a free way to combine two subproblems?" Here, if s[i] == s[j], we can extend i's stroke to j for free. This is the "aha" insight. Without it the algorithm is still correct — it just misses a constant factor optimization in practice.

Step 4: State the Time Complexity Clearly

Three nested loops: O(n) lengths × O(n) starting positions × O(n) split points = O(n³). Space O(n²) for the table. Mention the compression reduces n in practice.

🗣️ What to Say to the Interviewer

"This is an Interval DP problem. I'll define dp[i][j] = min turns to print s[i..j].
Base case: dp[i][i] = 1 — one character needs one stroke.
Key insight: if s[i] == s[j], we can extend the stroke printing s[i] to also cover j for free.
So dp[i][j] = dp[i][j-1] when s[i] == s[j].
Otherwise try all split points k: dp[i][j] = min(dp[i][k] + dp[k+1][j]).
Fill bottom-up by increasing interval length. Total O(n³) time, O(n²) space.
I'll also compress consecutive duplicates first as an optimization."
🔮 Likely Follow-up Questions
  • "Reconstruct the actual print sequence?" → Track which k gave the minimum and whether merge was used; backtrack through the table
  • "What if each character has a different printing cost?" → Replace the implicit "+1 per turn" with the character-specific cost
  • "Can we do better than O(n³)?" → Not known for this problem; O(n³) is the best published bound
  • "How is this related to Burst Balloons?" → Same skeleton: 3-loop interval DP, but Burst Balloons uses a "do last" paradigm; Strange Printer uses "split into independent halves"

7 Summary & Key Takeaways ▾
Interval DP: dp[i][j] = minimum cost to handle the substring s[i..j]
Fill by increasing interval length — smaller intervals are dependencies
Merge trick: s[i] == s[j] means dp[i][j] = dp[i][j-1] — extend the stroke for free
Compress consecutive duplicates as preprocessing to reduce n
Three-loop skeleton (length, i, k) is shared across all Interval DP problems
ApproachTimeSpaceNotes
Naive Recursive Exponential O(n) Recomputes same intervals exponentially
Top-Down Memoization O(n³) O(n²) Natural recursion, easier to derive live
Bottom-Up Tabulation ✓ O(n³) O(n²) No recursion overhead; preferred in interviews
🎯 When to Recognize Interval DP
  • Problem operates on a contiguous range of elements (string or array)
  • Optimal solution for a range depends on optimal solutions for sub-ranges
  • No obvious greedy — the best action at i depends on j (distant element)
  • Keywords: "minimum operations on substring", "split/merge optimally", "print/erase range"
