LeetCode #1

Two Sum

Given an array of integers nums and an integer target, return the indices of the two numbers that add up to target.

Easy Hash Map Two Pointers Time: O(n) Space: O(n)
1 Understanding the Problem ▾

You have a list of numbers and a target sum. Find the two numbers that add up to the target and return their indices. The problem guarantees exactly one valid answer exists, and you cannot use the same element twice.

💡 Key Insight

For any number x you encounter, its complement is target - x. If you've already seen the complement at some earlier index, you've found your pair. A hash map lets you check this in O(1) — making the whole scan O(n) instead of O(n²).

✓ nums=[2,7,11,15], target=9

i=0: num=2, comp=7, 7 not in map
store {2:0}
i=1: num=7, comp=2, 2 IS in map!
return [0, 1]

✗ Brute Force O(n²)

Check 2+7=9 ✓ (found after 1 pair)
But in worst case checks n*(n-1)/2 pairs.
For n=100,000 → 5 billion checks.
Way too slow

Key edge cases: Duplicate values like [3, 3] with target=6 (must return [0, 1], using distinct indices). Negative numbers (complement can be negative). Answer always at distinct indices — you cannot pair an element with itself.

2 The Algorithm: One-Pass Hash Map ▾

Instead of comparing every pair (O(n²)), we scan the array once left to right. At each element, we ask: "Does the complement already exist in our running memory?" If yes — done. If no — add current to memory and continue.

1 Create an empty hash map seen = {} mapping each number's value to its index.
2 For each element nums[i], compute the complement: complement = target - nums[i].
3 Check the map: if complement in seen, we found our pair. Return [seen[complement], i] immediately.
4 Otherwise store current: seen[nums[i]] = i. We check before storing — this is critical to avoid pairing an element with itself.
🔑 Why One Pass is Enough

The valid pair consists of elements at indices j < k. When the loop reaches index k, element nums[j] has already been added to seen. The complement check at k will find it. We never need to look ahead — only backward — which the map handles perfectly.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At the start of iteration i, the map seen contains exactly the values and indices of nums[0..i-1]. When we check complement in seen, we are looking only at elements before index i — guaranteeing distinct indices and no self-pairing.

Why check before storing: If you stored then checked, you could match an element with itself. Example: nums=[3, 5], target=6. At i=0: num=3, complement=3. If we stored first, seen={3:0}, and then complement 3 is in seen — we'd return [0, 0], pairing index 0 with itself. Checking first prevents this.

Why no valid pair is missed: Suppose the answer is at indices j < k. When i=j, complement=target-nums[j] is not in seen yet (since nums[k] hasn't been seen). We store nums[j]. When i=k, complement=target-nums[k]=nums[j], and nums[j] is in seen at index j. We return [j, k]. Perfect.

⚠️ What About Duplicate Values like [3, 3], target=6?

i=0: num=3, comp=3, map is empty → no match. Store {3:0}. i=1: num=3, comp=3, map has 3 at index 0 → return [0, 1]. Correct! We do NOT pair index 1 with itself because we check before storing, so at i=1 the map still shows the previous index 0 for value 3.

4 The Code (Annotated) ▾

Solution 1 — One-Pass Hash Map (Optimal) ← Interview Pick

def twoSum(nums: list, target: int) -> list:
seen = {} ← hash map: {value: index} for all previously visited elements
for i, num in enumerate(nums): ← iterate with both index and value
complement = target - num ← the number we need to pair with num to hit target
if complement in seen: ← O(1) hash map lookup — has the partner been seen before?
return [seen[complement], i] ← partner's earlier index, then current index
seen[num] = i ← AFTER checking: store current so future elements can find it
⚠️ Critical: Check complement BEFORE storing

The line seen[num] = i comes AFTER the complement check. This is intentional. If we stored first, a number whose complement equals itself (e.g. target=4, num=2, comp=2) would incorrectly pair with its own index. The check-then-store order prevents all self-pairing bugs.

Solution 2 — Brute Force (O(n²), O(1) space — propose first in interview)

def twoSum(nums: list, target: int) -> list:
n = len(nums)
for i in range(n): ← fix first element
for j in range(i + 1, n): ← try every later element (j > i guarantees distinct)
if nums[i] + nums[j] == target:
return [i, j]

Solution 3 — Two-Pointer (only if array is SORTED — #167 variant)

def twoSum_sorted(nums: list, target: int) -> list:
left, right = 0, len(nums) - 1 ← converging pointers from both ends
while left < right:← stop when pointers meet
s = nums[left] + nums[right]
if s == target: return [left, right]
elif s < target: left += 1 ← sum too small: move left pointer right to increase
else: right -= 1 ← sum too big: move right pointer left to decrease
💡 Complexity Summary

Brute Force: Time O(n²), Space O(1)
One-Pass Hash Map: Time O(n), Space O(n) — optimal for unsorted input
Two-Pointer (sorted input only): Time O(n), Space O(1) — best constants, no extra memory

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Clarify constraints

Ask: "Is exactly one solution guaranteed? Can I use the same element twice? Are all values positive?" Two Sum guarantees one solution and distinct indices. Knowing this removes edge-case anxiety.

Step 2: Propose brute force first

"The naive approach: nested loops checking every pair. O(n²) time, O(1) space. Works correctly but won't scale. Can I do better?" Always start here — it shows you reason from fundamentals.

Step 3: Identify the bottleneck

"The inner loop is doing a linear search for the complement. If I could make that O(1), the whole algorithm would be O(n). A hash map gives O(1) lookups."

Step 4: Derive and describe one-pass

"For each element, I compute complement = target - current. I check if complement is in the map first (before storing), to avoid self-pairing. If yes, done. Otherwise store current and continue."

🗣️ What to Say to the Interviewer

"I'll use a hash map to store each number's index as I scan left to right.
For each element, complement = target - current.
If complement is already in the map, we're done — return [map[complement], i].
Otherwise add current to the map and continue.
This is one pass, O(n) time, O(n) space."
🔮 Likely Follow-up Questions
  • "What if the array is sorted?" → Two-pointer O(1) space (#167, Two Sum II)
  • "Three Sum to zero?" → Sort + fix one element + two-pointer (#15)
  • "Return all pairs, not just first?" → Collect all, deduplicate
  • "What if there are multiple valid answers?" → Hash map still works, collect all matches
  • "Integer overflow?" → Python no issue; in Java/C++ cast to long

7 Summary & Key Takeaways ▾
Hash map trades O(n) space to reduce time from O(n²) to O(n)
Always check complement before storing — prevents self-pairing bugs
One-pass enables early exit as soon as the pair is found
The complement pattern (target - x) is reused in 3Sum, 4Sum, and k-Sum variants
If input is sorted, two-pointer achieves O(n) time with only O(1) space (LeetCode #167)

🎯 When to Recognize the Complement + Hash Map Pattern
  • "Find two elements summing to target" → Hash map stores seen values for O(1) complement lookup
  • "Return indices" (not values) → hash map is usually required (sorting loses original indices)
  • O(n²) nested loop feels slow → ask "what information from the past could a hash map store?"
  • Single scan through array but need to cross-reference earlier elements → hash map running memory
