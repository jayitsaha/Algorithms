LeetCode #1049

Last Stone Weight II

You have an array of stones with positive weights. Each turn you pick the two heaviest, smash them: if equal they vanish, else the difference survives. Return the minimum possible weight of the final remaining stone (0 if none).

Medium Dynamic Programming 0/1 Knapsack Minimize Subset Difference Time: O(n·S) Space: O(S)
1 Understanding the Problem ▾

Each smash operation takes two stones of weights x and y (x ≤ y) and replaces them with a single stone of weight y − x (or removes both if equal). We repeat until at most one stone remains, and we want that final weight to be as small as possible.

💡 The Critical Reframe: Signs and Subsets

Think of each stone as getting a sign: +1 (put in group A) or −1 (put in group B). Any sequence of smashes is equivalent to one such assignment. The final result always equals |sum_A − sum_B|.

If total weight S = sum_A + sum_B and we call sum_A = P, then the result is |(S−P) − P| = |S − 2P|. To minimize this, maximize P subject to P ≤ S//2. This is exactly the 0/1 Knapsack problem with capacity S//2.

Brute force attempt

stones = [2,7,4,1,8,1]

Edge cases: Single stone → return that stone's weight. All stones identical → answer is 0 or 1 stone weight depending on count parity. Two stones → answer is |a − b|. Large stones make S big but still bounded (n ≤ 30, w ≤ 100 → S ≤ 3000).

2 Why Is This Dynamic Programming? The 0/1 Knapsack Deep-Dive ▾

After the sign-assignment reframe, we need: "find a subset of stones summing as close to target = S//2 as possible." This has the two hallmarks of DP:

Optimal Substructure

Whether sum j is reachable from the first i stones depends entirely on: (a) can we reach j from the first i−1 stones (skip stone i), OR (b) can we reach j − w[i] from the first i−1 stones then add stone i. Each answer breaks into strictly smaller subproblems.

Overlapping Subproblems

A naive recursive approach recomputes "can I form sum j using stones i..n-1?" for the same (i, j) pair via many different paths. DP fills each dp[j] exactly once. Without memoization, the recursion tree has exponential nodes.

📐 The Recurrence Relation (0/1 Knapsack)
dp[j] = dp[j] OR dp[j − stone]
Meaning: "can I reach sum j?" = "was it reachable before?" OR "can I reach j−stone and then add stone?"
Base case: dp[0] = True (empty subset always sums to 0)
Iterate j from target DOWN to stone — ensures each stone used at most once

The decision at each stone — for every stone w, we ask for each capacity j: "Could I fill exactly j by including this stone?" Yes, if I could fill j−w without it. This is the OR-update in one line: dp[j] |= dp[j - w].

⚠️ Right-to-Left: The 0/1 Knapsack Trick

Iterating j from target down to stone is mandatory. If we went left-to-right, then when we compute dp[j], dp[j−w] might have already been updated in this same pass (with this same stone already included) — effectively counting the stone twice. Right-to-left guarantees that when we read dp[j−w], it still reflects the state before this stone was considered.

3 Why It Works: The Invariant ▾
🔐 The DP Invariant

After processing the first i stones, dp[j] = True means "there exists a subset of {stones[0], …, stones[i-1]} whose weights sum to exactly j." The invariant is maintained because: at initialization dp[0]=True (empty subset). At each step, we only set dp[j]=True if we can prove it is reachable. We never unset True values.

Why total − 2 * best_P is the answer: Let best_P be the largest j ≤ target = S//2 with dp[j] = True. Put those stones into Group A (sum = best_P). The rest go into Group B (sum = S − best_P). When we let them smash each other, the result is (S − best_P) − best_P = S − 2 × best_P. Since best_P ≤ S//2, Group B ≥ Group A, so this is non-negative. No absolute value needed.

Why any sign-assignment equals some smash sequence: Formally, for any assignment of +1/−1 to the stones, there is an ordering of smash operations producing exactly |Σ(±w[i])|. This follows by induction: at each step, smash one + stone with one − stone. The difference is the sum of remaining assigned values.

Special case — answer is 0: When S is even and a subset exactly summing to S/2 exists (dp[S/2] = True), the two groups are equal and all cancel. This is exactly the Partition Equal Subset Sum problem (#416).

4 The Code (Annotated) ▾

Solution 1 — Bottom-Up Tabulation (Interview Pick) · O(n·S) time, O(S) space

def lastStoneWeightII(stones: list[int]) -> int:
total = sum(stones) ← S = sum of all stones
target = total // 2 ← We want subset sum as close to S//2 as possible
dp = [False] * (target + 1) ← dp[j] = "can some subset sum to exactly j?"
dp[0] = True ← Empty subset always sums to 0
for stone in stones: ← Process each stone exactly once (0/1 knapsack)
for j in range(target, stone - 1, -1): ← RIGHT-TO-LEFT: prevent reusing stone in same pass
dp[j] |= dp[j - stone] ← Include stone: if j−stone reachable, j is too
best_P = max(j for j in range(target + 1) if dp[j]) ← Largest reachable sum ≤ target
return total - 2 * best_P ← |Group_B − Group_A| = S − 2×P (non-negative)
⚠️ Why total − 2 × best_P (not total − best_P)?

Group A sums to best_P. Group B sums to total − best_P. When smashed: (total − best_P) − best_P = total − 2 × best_P. The factor of 2 comes from subtracting best_P twice — once because it left the total, and once because it counters an equal amount from Group B.

Solution 2 — Top-Down Memoization · O(n·S) time, O(n·S) space

from functools import lru_cache
def lastStoneWeightII(stones: list[int]) -> int:
total = sum(stones)
target = total // 2
@lru_cache(None)
def can_reach(i, rem): ← Can stones[i:] form exactly rem?
if rem == 0: return True ← Perfectly filled — done
if i == len(stones) or rem < 0: return False ← Ran out or overshot
# Two choices: take stone[i] or skip it
return (can_reach(i+1, rem - stones[i]) or ← Include stones[i]
can_reach(i+1, rem)) ← Skip stones[i]
best_P = max(j for j in range(target+1) if can_reach(0, j)) ← Same final step
return total - 2 * best_P
💡 Tabulation vs Memoization Trade-offs

Tabulation: O(S) space (one 1D array). Iterative — no stack overflow risk. Fills all states. Better for interviews: simple nested loops, easy to reason about.
Memoization: O(n·S) space (2D cache). Only fills states actually queried — may be faster in sparse cases. Easier to write by directly translating the recurrence relation. Slightly slower due to function call overhead and cache lookups.

Solution 3 — Brute Force · O(2^n) time — mention to establish baseline, then reject

from itertools import combinations
def lastStoneWeightII_brute(stones: list[int]) -> int:
total, best = sum(stones), float('inf')
for r in range(len(stones) + 1): ← All subset sizes 0..n
for combo in combinations(stones, r): ← All subsets of size r
P = sum(combo)
best = min(best, abs(total - 2 * P)) ← Check this partition
return best ← O(2^n) — too slow for n=30
5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Make the Non-Obvious Reframe

Say aloud: "Each stone gets a sign. The final result equals |Σ(±stones[i])|. Splitting stones into two groups and comparing, the result is |GroupA − GroupB|. To minimize this, I want the two groups as equal as possible — that means finding a subset summing to S//2."

Step 2: Name the Pattern

"This is 0/1 Knapsack — each stone is an item, I want to fill a knapsack of capacity S//2 as fully as possible. Classic DP: dp[j] = can we form sum j using stones seen so far?"

Step 3: Implement and Justify the Inner Loop Direction

"I iterate j from target down to stone. This ensures each stone is used at most once — if I went left-to-right, dp[j−w] would have already incorporated this same stone in this pass, double-counting it."

Step 4: State Complexity Confidently

"Outer loop: n stones. Inner loop: at most S//2 iterations. n ≤ 30, each stone ≤ 100, so S ≤ 3000. Total: O(30 × 1500) = O(45,000) operations. Space O(S) = O(1500) for the boolean array."

🗣️ What to Say to the Interviewer

"Each smash assigns +/− signs to stones. Final result = |sum+ − sum−| = |S − 2P|.
To minimize: maximize P (subset sum) subject to P ≤ S//2.
This is 0/1 Knapsack. dp[j] = True if some subset sums to j.
For each stone, iterate j from target down to stone: dp[j] |= dp[j−stone].
Right-to-left guarantees each stone used at most once.
Final answer: S − 2 × max{j : dp[j]=True, j ≤ S//2}.
Time: O(n×S/2) ≈ O(45000). Space: O(S/2) ≈ O(1500)."
🔮 Likely Follow-up Questions
  • "When is the answer exactly 0?" → When dp[S//2]=True and S is even (perfect partition).
  • "What if stones can be reused?" → Iterate inner loop left-to-right (Unbounded Knapsack).
  • "How does this relate to Partition Equal Subset Sum?" → Identical DP; that problem asks if S//2 is reachable (answer = 0).
  • "Count the number of minimum-weight outcomes?" → Change dp from bool to int (count of ways).

7 Summary & Key Takeaways ▾
Smashing stones = assigning +/− signs. Minimize |S − 2P| by maximizing subset sum P ≤ S//2.
This is 0/1 Knapsack. dp[j] = True if some subset of processed stones sums to j.
Right-to-left inner loop (j from target down to stone) ensures each stone is used at most once.
Final answer: total − 2 × best_P. The factor of 2 comes from Group B sum = S − P, so difference = S − 2P.
Space O(S) ≤ O(3000) — one 1D boolean array is enough; no 2D table needed.
Partition Equal Subset Sum (#416) is the special case asking if min answer = 0.

🎯 When to Recognize 0/1 Knapsack / Minimize Subset Difference
  • "Partition items into two groups with minimum difference" → 0/1 Knapsack Minimize Subset Difference
  • "Can we select a subset summing to exactly k?" → Subset Sum (same DP, boolean answer)
  • "Items 'cancel each other' or 'collide'" → Think sign assignment, then subset difference
  • Each item used at most once, bounded total weight → 0/1 Knapsack (right-to-left inner loop)
  • Unlimited copies of each item → Unbounded Knapsack (left-to-right inner loop)
