LeetCode #740

Delete and Earn

Delete any element, earn its value, then all copies of num−1 and num+1 are removed. Maximize total earnings.

Medium Dynamic Programming Transform to House Robber Time: O(n + M) Space: O(M)
1 Understanding the Problem ▾

You have an array nums. In one operation: pick any element x, earn x points, then all elements equal to x−1 and x+1 are deleted from the array permanently. Repeat until no elements remain. Return the maximum points you can earn.

💡 Key Insight: This Is House Robber in Disguise

The moment you pick any copy of value x, all neighbors (x−1 and x+1) die. Since the penalty is the same whether you take one or many copies of x, you should take all copies of x if you take any. This "take all or none per value" rule creates an adjacency conflict on the value axis — exactly the House Robber constraint.

Worked example — nums = [3, 4, 2]:

✓ Optimal: take 2 and 4 (earn 6)

Take 4 → delete all 3s and 5s → +4
Take 2 → delete all 1s and 3s → +2
2 and 4 are not adjacent values!
Total = 2 + 4 = 6

✗ Suboptimal: take 3 (earn only 3)

Take 3 → deletes 2 and 4 → +3
Array is now empty
Lost 6 points for 3!
Total = 3

The transformation: Build sum[v] = v × count(v) for each value 0 to max(nums). Now the problem says: choose a subset of indices from this sum array such that no two chosen indices are adjacent, maximizing the total. That is House Robber.

Edge cases: single element (take it all), all same values (no adjacency conflict, take everything), very large values with sparse coverage (sum array mostly zeros), nums containing 0s (earns 0 anyway).

2 The Algorithm: Transform → House Robber DP ▾

Two clean phases: transform the problem, then solve it with constant-space DP.

1 Build sum array: Let M = max(nums). Create sum[0..M] where sum[v] = v × count(v). Taking all copies of value v earns sum[v]. Adjacency in value-space maps to adjacency in this array.
2 Initialize base cases: prev2 = sum[0], prev1 = max(sum[0], sum[1]). These are dp[0] and dp[1] — the best earnings considering only the first one or two values.
3 Iterate v = 2 to M: curr = max(prev1, prev2 + sum[v]). Skip value v (keep prev1) or take all copies of v (prev2 + sum[v], skipping adjacent v−1).
4 Advance window: prev2, prev1 = prev1, curr. Discard oldest; keep only last two DP values.
5 Return prev1 — the maximum earnings after considering all values 0..M.
📐 The Recurrence Relations
sum[v] = v × count(v)    ← pre-processing (O(n))
dp[v] = max(dp[v−1], dp[v−2] + sum[v])   ← House Robber recurrence

Space-optimized rolling form:
curr = max(prev1, prev2 + sum[v])
🔑 Why "Take All or None" Is Always Optimal

Suppose you take one copy of value 3. The rule instantly deletes all 2s and 4s. Taking a second copy of 3 now costs you nothing extra — those neighbors are already gone. So we always earn the maximum by grabbing every copy of 3 (sum[3] = 3 × count(3)). The adjacency penalty is flat, not per-copy. Hence the transformation is exact.

3 Why It Works: The Invariant ▾
🔐 The DP Invariant

At every step, prev1 holds the maximum earnings achievable using only values 0 through v−1, and prev2 holds the maximum for values 0 through v−2. Together they encode all information needed to extend the solution by one more value.

Why is this DP (optimal substructure)? The best answer for values 0..v decomposes into two exclusive cases: (A) we don't take value v — best answer is dp[v−1]; or (B) we take value v — we can't have taken v−1, so best is dp[v−2] + sum[v]. The max of these two is dp[v]. No other history matters.

Overlapping subproblems: A naive recursion computing "what's the best for values 0..v" would recompute dp[v−1] once for each higher value. Memoization or tabulation avoids this, giving O(M) time instead of O(2^M).

⚠️ Common Mistake: Operating on nums Directly

Sorting nums and running two-pointer or greedy on the original array doesn't work because deleting one copy of 3 kills all 2s and 4s, not just neighbors in a sorted list. The transform to the sum array is the critical realization — without it you can't correctly model "take all copies together."

Edge case proof: If all elements equal k, then sum = [0,…,0,k×n] and the DP trivially returns k×n (the only non-zero entry has no neighbors to conflict with). If nums = [1,2], sum = [0,1,2], dp[0]=0, dp[1]=1, dp[2]=max(1, 0+2)=2 — we take only value 2.

4 The Code (Annotated) ▾

Solution 1 — Bottom-Up Tabulation (Interview Pick)

def deleteAndEarn(nums):
M = max(nums) ← largest value; defines our DP range [0..M]
total = [0] * (M + 1) ← total[v] = v × count(v); index IS the value
for n in nums: ← build total[] in O(n)
total[n] += n ← add n each time; three 3s → total[3] = 9 = 3×3
if M == 1: return total[1] ← edge: only value 1; no adjacency conflict
prev2 = total[0] ← dp[0]: best earnings with only value 0 (always 0)
prev1 = max(total[0], total[1]) ← dp[1]: take value 1 or value 0, whichever earns more
for v in range(2, M + 1): ← process each value from 2 up to M
curr = max(prev1, prev2 + total[v]) ← skip v → keep prev1; take v → prev2 + sum[v]
prev2, prev1 = prev1, curr ← roll window; discard oldest, store newest
return prev1 ← prev1 = dp[M] = answer
⚠️ Why total[n] += n instead of += 1?

We want sum[v] = v × count(v) — the total points from taking every copy of v. By adding n (not 1) each time we encounter value n, we get this directly: see three 3s → total[3] = 9. A common variant builds a count array first and multiplies: both are correct, but += n is a one-pass elegance.

Solution 2 — Top-Down Memoization

def deleteAndEarn(nums):
M = max(nums)
total = [0] * (M + 1)
for n in nums:
total[n] += n ← same pre-processing
from functools import lru_cache
@lru_cache(maxsize=None) ← Python's built-in memoization
def dp(v): ← returns best earnings for values 0..v
if v == 0: return total[0] ← base case
if v == 1: return max(total[0], total[1]) ← base case
return max(dp(v-1), dp(v-2) + total[v]) ← recurrence
return dp(M) ← solve from top; recursion fills cache bottom-up
💡 Complexity Summary

Both approaches: Time O(n + M), where n = len(nums), M = max(nums)
Tabulation space: O(M) for total[] + O(1) for two rolling variables
Memoization space: O(M) for total[] + O(M) cache + O(M) call stack
Interview pick: Tabulation — iterative, no stack overflow risk, easier to trace on whiteboard

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Parse the Constraint Carefully

The key phrase is "all elements equal to num−1 and num+1 are deleted." This is a global deletion by value, not by position. Ask yourself: does it matter which copy of 3 I pick? No — the penalty is the same. So I should take ALL copies at once.

Step 2: Recognize the Pattern

Once "take all copies or none" crystallizes, draw the values on a number line: 0, 1, 2, 3, …, M. Each has a payoff of sum[v] = v × count(v). Adjacent values can't both be chosen. This is verbatim House Robber.

Step 3: Code the Transform + Recurrence

Build total[] in one pass. Initialize prev2 = total[0], prev1 = max(total[0], total[1]). Loop v=2..M with curr = max(prev1, prev2 + total[v]). Roll the window. Return prev1. That's 12 lines of clean code.

Step 4: Verify with the Example

For [3,4,2]: total = [0,0,2,3,4]. dp[0]=0, dp[1]=0, dp[2]=2, dp[3]=max(2,0+3)=3, dp[4]=max(3,2+4)=6. Answer 6. Confirm: take all 2s (+2) and all 4s (+4), skip 3. Correct!

🗣️ What to Say to the Interviewer

"I notice that taking value x deletes all x-1 and x+1 globally — so I might as well take all copies of x at once.
That transforms this into: build sum[v] = v × count(v), then find a non-adjacent-value subset with max sum.
That's exactly House Robber! Recurrence: dp[v] = max(dp[v-1], dp[v-2] + sum[v]).
I'll use two rolling variables — prev2 and prev1 — to keep space O(1) beyond the sum array.
Time O(n + M), Space O(M) for the total array."
🔮 Likely Follow-up Questions
  • "What if deletion extended two values (x−2 and x+2)?" → Same transform; recurrence changes to dp[v] = max(dp[v-1], dp[v-3] + sum[v]) — skip 2 indices instead of 1.
  • "What if you could only delete K times?" → 2D DP: dp[v][k] = max earnings using values 0..v with at most k deletions left.
  • "What if M is huge but nums is small?" → Compress values to only the distinct ones present; O(n log n) instead of O(M).
  • "Circular version — value 1 is adjacent to value M?" → Run House Robber twice (excluding value 1 once, excluding value M once), take the max.

7 Summary & Key Takeaways ▾
"Delete neighbors" means taking value x is all-or-nothing — grab all copies at once
Build total[v] = v × count(v) to collapse the problem onto a value-indexed array
The transformed problem is exactly House Robber on total[]: no two adjacent values
Two rolling variables prev2/prev1 give O(1) space beyond the total[] array
Time O(n + M): O(n) for the sum build, O(M) for the DP sweep
ApproachTimeSpaceNotes
Brute Force (all subsets) O(2^n) O(n) Infeasible; try all deletion sequences
Memoization (top-down) O(n + M) O(M) Correct; recursion + cache; stack overhead
Tabulation ✓ O(n + M) O(M) Iterative, cleanest to explain and code
🎯 When to Recognize "Transform to House Robber"
  • Taking one instance of something prevents taking "adjacent" instances
  • Items can be bucketed by value; all instances of same value behave identically
  • "Maximize sum where neighbors on the value axis cannot coexist"
  • The exclusion is value-based, not position-based
