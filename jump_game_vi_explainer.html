LeetCode #1696

Jump Game VI

Find the maximum score path from index 0 to index n−1, jumping 1–k steps at a time and collecting values along the way.

Medium Monotonic Queue DP + Monotonic Deque Time: O(n) Space: O(n)
1 Understanding the Problem ▾

You stand at index 0 of an integer array nums. At each position i, you may jump to any index j where i < j ≤ i + k. You must reach index n−1. Your total score is the sum of values at every index you visit (including start and end). Find the maximum possible score.

💡 Key Insight

This is a DP problem. Define dp[i] = max score to reach index i. The recurrence is dp[i] = nums[i] + max(dp[j]) for j ∈ [i−k, i−1]. The bottleneck: computing that inner sliding window maximum naïvely costs O(k) per step → O(nk) total. A monotonic deque solves it in O(1) per step → O(n) total.

✓ Example: nums=[1,−1,−2,4,−7,3], k=2

Path 0→1→3→5: 1+(−1)+4+3 = 7
Path 0→2→3→5: 1+(−2)+4+3 = 6
Path 0→2→4→5: 1+(−2)+(−7)+3 = −5
Best: 0→1→3→5 = 7

✗ Naïve O(nk) approach — TLE

For each i, check all j in [i−k, i−1]
That is k inner checks × n outer steps
O(nk) — fails when n, k ≈ 10⁵
10¹⁰ operations → time limit exceeded

Edge cases: n = 1 (answer is nums[0]), all negative values (still pick the least-bad path), k ≥ n (can jump to any index in one jump), and the window [i−k, i−1] shrinks naturally at boundaries without special handling.

2 The Algorithm: DP + Monotonic Deque ▾

We fill the DP table left-to-right. For each index i, we need the maximum dp[j] over the window [i−k, i−1]. A monotonic decreasing deque maintains a front-to-back list of candidate indices, ordered so that the front always holds the index with the highest dp value in the current window.

1 Initialize: dp[0] = nums[0]. The deque starts with [0]. We always start at index 0 and collect its value — no predecessor to consider.
2 Front eviction (range constraint): For each i, pop from the front while dq[0] < i − k. Those indices are too far back to jump from in ≤ k steps.
3 Compute dp[i]: The front of the deque is now the best valid predecessor. dp[i] = nums[i] + dp[dq[0]].
4 Back eviction (dominance): Pop from the back while dp[dq[-1]] ≤ dp[i]. If index i has dp ≥ some older index j, then j is dominated — any future position can reach i instead of j, with equal or better score.
5 Push current: Append i to the back. The deque now includes i as a candidate for future positions. Return dp[n−1].
📐 The Recurrence
dp[0] = nums[0]
dp[i] = nums[i] + max( dp[j] )    for j ∈ [max(0, i−k), i−1]
answer = dp[n−1]
🔑 Why deque beats heap here

A max-heap gives O(log n) per insert/query → O(n log n) total. The deque is O(1) amortized because each index enters the deque exactly once (via append) and leaves exactly once (via popleft or pop). Total operations = 2n → O(n). The key insight enabling this: an index with lower dp that is also older (smaller index) can never win — we discard it immediately instead of lazily filtering it later.

3 Why It Works: The Invariant ▾
🔐 The Two-Pronged Deque Invariant

After processing each index, the deque satisfies two properties simultaneously:
(A) Indices increase from front to back — the oldest valid predecessor is at the front.
(B) dp values decrease from front to back — the best predecessor is at the front.
Together these mean: the front is always the maximum-dp index in the current window.

Why dominance eviction is correct: Suppose we're about to push index i, and the current back of the deque is index j with dp[j] ≤ dp[i]. For any future position c > i: if c can reach j (i.e., c − j ≤ k), then since i > j, c − i < c − j ≤ k, so c can also reach i. And dp[i] ≥ dp[j]. So j can never be chosen over i by any future position — we safely discard it.

Why front eviction is correct: Index j at the front is valid only if j ≥ i − k. Once j < i − k, we can't jump from j to i in ≤ k steps, so j must be removed. Front eviction maintains the range constraint.

⚠️ Critical Order of Operations

Always perform steps in this order for index i: (1) front-evict → (2) compute dp[i] → (3) back-evict → (4) push i. If you push i before computing dp[i], you'll read a stale dp[i]. If you skip front-eviction first, you'll use an out-of-range predecessor. If you skip back-eviction before pushing, you'll violate the monotone property.

4 The Code (Annotated) ▾

Solution 1 — DP + Monotonic Deque (Optimal, Interview Pick)

from collections import deque ← O(1) appendleft/popleft unlike list
def maxResult(nums, k):
n = len(nums) ← cache length for clarity
dp = [0] * n ← dp[i] = max score to reach index i
dp[0] = nums[0] ← base case: always collect index 0
dq = deque([0]) ← deque of indices; front = best predecessor
for i in range(1, n): ← process each remaining index
while dq[0] < i - k: ← front is out of jump range: evict
dq.popleft() ← O(1) front removal
dp[i] = nums[i] + dp[dq[0]] ← best predecessor dp value is at front
while dq and dp[dq[-1]] <= dp[i]: ← back dominated by current: evict
dq.pop() ← O(1) back removal
dq.append(i) ← push i as a future predecessor candidate
return dp[-1] ← max score at last index is the answer
⚠️ Why <= in back-eviction (not strict <)?

We use dp[dq[-1]] <= dp[i]. When dp values are equal, we prefer the more recent index (i). A more recent index with equal dp has a larger useful future window — it remains valid as a predecessor for more future positions. Keeping an older equal-dp index wastes a deque slot without benefit.

Solution 2 — Naïve DP (Brute Force, O(nk))

def maxResult_naive(nums, k):
n = len(nums)
dp = [float('-inf')] * n ← -inf = unreachable
dp[0] = nums[0]
for i in range(1, n):
for j in range(max(0, i-k), i): ← check all k predecessors: O(k) per step
dp[i] = max(dp[i], dp[j] + nums[i]) ← O(nk) total — TLE for large inputs
return dp[-1]

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the DP Shape

"Maximum score to reach the end with constrained choices at each step" → classic DP. Define dp[i] as max score to reach index i. Answer is dp[n−1].

Step 2: Write the Naïve Recurrence First

Write dp[i] = nums[i] + max(dp[j]) for j ∈ [i−k, i−1]. State the O(nk) solution clearly. Then ask: "The inner max is always over the last k dp values — can I speed up this sliding window maximum?"

Step 3: Connect to Sliding Window Maximum

Sliding Window Maximum (LeetCode #239) is the classic deque problem. The inner loop in our DP is exactly that: maximum over a sliding window of size k. Mention this by name — it signals pattern recognition to the interviewer.

Step 4: Explain the Deque Operations

Walk through front eviction (out-of-range) and back eviction (dominated). Emphasize the O(1) amortized argument: each index enters/exits the deque at most once, so total work is O(n).

7 Summary & Key Takeaways ▾
DP recurrence: dp[i] = nums[i] + max(dp[j]) for j ∈ [i−k, i−1]
The inner max is a sliding window maximum — solved in O(n) with a monotonic deque
Deque invariant: indices increase front→back; dp values decrease front→back
Front eviction enforces the jump range; back eviction removes dominated predecessors
Each index enters and exits the deque at most once → O(n) amortized total work
Optimal path for example: 0→1→3→5 collecting 1+(−1)+4+3 = 7
🎯 When to Reach for DP + Monotonic Deque
  • DP recurrence requires max or min over a sliding window of previous states
  • The window has a fixed size k (front eviction) and candidates become dominated (back eviction)
  • Naïve O(nk) inner loop makes total complexity O(nk) — need O(n)
  • Signal phrases: "jump at most k steps", "within distance k", "last k elements"
  • Related: Constrained Subsequence Sum (#1425) uses the identical deque technique
