LeetCode #110

Balanced Binary Tree

Given a binary tree, determine if it is height-balanced — every node's left and right subtree heights differ by at most 1.

Easy Trees DFS: Postorder Time: O(n) Space: O(h)
1 Understanding the Problem ▾

A binary tree is height-balanced when, at every single node, the height of its left subtree and the height of its right subtree differ by at most 1. This must hold globally — not just at the root. If even one internal node fails this condition, the whole tree is unbalanced.

💡 Key Insight

The balance condition is recursive and local. A tree is balanced if and only if: (1) the root's left and right subtrees differ in height by at most 1, AND (2) the left subtree is itself balanced, AND (3) the right subtree is itself balanced. All three must hold simultaneously.

Key definitions: The height of a node is the number of edges on the longest downward path to a leaf. A null node has height 0. A leaf has height 1. Edge cases to consider: empty tree (always balanced), single-node tree (always balanced), skewed trees that look like linked lists.

2 The Algorithm: Postorder DFS with −1 Sentinel ▾

The optimal approach fuses the height computation and balance check into a single bottom-up DFS pass. Instead of calling a separate height() function per node (which causes O(n²) redundancy), we compute the height and check balance simultaneously, bubbling up a special sentinel value −1 to signal failure.

1 Base case: A null node returns height 0. Null is always balanced and has zero height. This stops the recursion.
2 Recurse left (postorder step 1): Call check(node.left) to get the left subtree's height. Postorder means children are processed before the parent.
3 Early exit if left failed: If left_h == −1, some node in the left subtree was already unbalanced. Return −1 immediately — no need to visit the right subtree.
4 Recurse right (postorder step 2): Call check(node.right) similarly. Apply the same early exit check.
5 Check balance at current node: If |left_h − right_h| > 1, this is the first unbalanced node encountered. Return −1.
6 Return height (postorder step 3): If balanced, return 1 + max(left_h, right_h) — the height of the current subtree for the parent's use.
🔑 Why Postorder Specifically?

In postorder traversal (Left → Right → Root), a node is processed only after both its children. By the time we arrive at a node, we already hold the computed heights from both subtrees. This lets us check balance and compute the node's own height in a single step — eliminating redundant work entirely. Preorder or inorder wouldn't give us this property.

⚠️ The O(n²) Trap — Why Naive Fails

A common first attempt: write isBalanced recursively, and call a separate height(node) for each node to check balance. This looks clean but is O(n²) — every call to isBalanced(node) triggers O(n) work from height(), and there are O(n) such calls. The postorder sentinel approach visits each node exactly once: O(n) total.

3 Why It Works: The Invariant ▾
🔐 The Core Invariant

The inner function check(node) maintains a strict contract: it returns the actual height of the subtree rooted at node (a non-negative integer) if that entire subtree is balanced, OR it returns −1 if any node in that subtree is unbalanced. The return value is never ambiguous.

Why −1 works as a sentinel: Tree heights are always non-negative integers (0 for null, 1 for a leaf, 2+ for deeper nodes). The value −1 can never be a legitimate height, so it serves as a perfect out-of-band signal for "failure detected below." No extra boolean flag or exception is needed.

Why the invariant guarantees correctness: Each node checks two things — (a) did anything fail below? (check for −1) and (b) am I myself balanced? (check |left_h − right_h| ≤ 1). If either fails, the failure signal propagates all the way to the root. If the root's check() returns anything other than −1, then every single node in the tree passed both checks. The outer function's logic is simply: return check(root) != −1.

The empty tree case: check(None) returns 0. Since 0 ≠ −1, isBalanced returns True for an empty tree. Mathematically, an empty tree is trivially balanced — it has no nodes to violate the condition.

💡 Analogy: The Bubble-Up Pattern

Think of it like a building inspection. Each floor (node) checks its own condition and both the floors below it. If any floor is compromised, it sends a red flag (−1) up the chain. The moment the inspector at any level sees a red flag, they immediately pass it up without inspecting further. If the top floor receives no red flag, the whole building is safe.

4 The Code (Annotated) ▾

Solution 1 — Postorder DFS with Sentinel (Optimal, O(n) time) — Interview Pick

def isBalanced(root: TreeNode) -> bool:
def check(node): ← helper returns height (≥0) or -1 (failure)
if node is None: ← base case: null node is always balanced
return 0 ← height 0 (null contributes nothing)
left_h = check(node.left) ← POSTORDER step 1: recurse left first
if left_h == -1: ← early termination: left subtree already unbalanced
return -1 ← propagate failure, skip right subtree entirely
right_h = check(node.right) ← POSTORDER step 2: recurse right
if right_h == -1: ← early termination: right subtree unbalanced
return -1
if abs(left_h - right_h) > 1: ← POSTORDER step 3: check THIS node's balance
return -1 ← signal: this node is the culprit
return 1 + max(left_h, right_h)← balanced → return actual height for parent's use
return check(root) != -1 ← outer: balanced iff helper never returned -1
⚠️ Subtle Ordering Bug: Check for -1 Right After Each Call

Some write both checks together: if left_h == -1 or right_h == -1: return -1 after getting both values. This is correct but misses the optimization — if left_h == -1, we call check(node.right) unnecessarily. Checking immediately after each call avoids traversing the right subtree when the left already failed.

Solution 2 — Naive Top-Down Recursion (O(n²), good as a starting point)

def isBalanced(root: TreeNode) -> bool:
if not root: return True ← null tree is balanced
def height(n): ← separate height helper — O(n) per call
if not n: return 0
return 1 + max(height(n.left), height(n.right))
if abs(height(root.left) - height(root.right)) > 1:← O(n) work here
return False
return isBalanced(root.left) and isBalanced(root.right) ← recurse → O(n²)
💡 Complexity Summary

Optimal (postorder sentinel): Time O(n) — every node visited exactly once. Space O(h) — recursion stack, where h is tree height. Worst case O(n) for a completely skewed tree (like a linked list).
Naive top-down: Time O(n²) — height() called O(n) times, each doing O(n) work. Same O(h) space. Never acceptable for large inputs.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Clarify the Balance Definition

Ask: "Is the balance condition at every node, or just the root?" The answer (every node) tells you this is a fully recursive tree problem. This single question shows you understand the problem deeply and prevents the common mistake of only checking the root.

Step 2: Identify the Naive Approach and Its Flaw

"My first idea: at each node, call height(left) and height(right), check the difference, then recurse on both children. This is O(n²) — height() recomputes everything below for each node call. I can do better."

Step 3: Recognize the Postorder Fusion Opportunity

"The key insight: when I recurse into a child and get its height, I've done the traversal work already. I can merge the height computation with the balance check in one pass. I'll use −1 as a sentinel since heights are always non-negative."

Step 4: Design the Return Contract

"My helper returns the actual height if the subtree is balanced, or −1 if not. The outer function just checks if check(root) != −1. This is clean and composable — each node only needs to look at its children's return values."

🗣️ What to Say to the Interviewer

"This is a tree height problem with a balance condition that must hold at every node — so I need a recursive approach.

My naive solution would call a separate height() helper at each node and recurse on children — but height() itself traverses the whole subtree, making the overall complexity O(n²).

The optimization: I'll merge the height computation with the balance check in a single postorder DFS. A helper returns the subtree's height if balanced, or -1 as a sentinel if unbalanced. When a child returns -1, I immediately propagate it without processing further — early termination.

Time O(n), Space O(h) where h is the tree height — O(log n) for balanced, O(n) worst case for skewed."
🔮 Likely Follow-up Questions
  • "How would you balance an unbalanced BST?" → Inorder traversal → sorted array → build balanced BST (#1382)
  • "Can you do it iteratively?" → Yes, simulate postorder with an explicit stack (harder code, same complexity)
  • "What if you need to find the first unbalanced node, not just true/false?" → Return the node itself instead of -1
  • "How does this extend to AVL trees?" → AVL trees maintain balance by tracking heights per node and rotating on insert/delete

7 Summary & Key Takeaways ▾
Postorder DFS (children before parent) lets us compute height and check balance in a single pass
−1 as a sentinel works because tree heights are always non-negative — no ambiguity
Check for −1 right after each recursive call to enable true early termination
The outer function is just: return check(root) != -1 — keep it simple
This "merge two passes into one with a sentinel" pattern appears across many tree problems

🎯 When to Recognize the Postorder + Sentinel Pattern
  • "Compute a property of each node that depends on its children's properties" → postorder DFS
  • "Check a global condition that could fail at any node" → sentinel propagation
  • "Height / depth / diameter / path sum" problems → bottom-up single DFS pass
  • If you find yourself writing a separate helper that re-traverses the tree → merge it
