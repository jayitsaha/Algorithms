LeetCode #2402

Meeting Rooms III

Given n rooms and a sorted list of meetings, schedule each to the lowest-numbered free room (or delay to the earliest-finishing room). Return the room that hosted the most meetings.

Hard Intervals Two Heaps Time: O(m log n) Space: O(n)
1 Understanding the Problem ▾

Imagine you run a conference centre with n numbered meeting rooms (0 through n−1). A scheduler sends you meetings in order of start time. Your job: assign each meeting according to strict priority rules, then report which room was busiest.

Assignment rules (applied in this exact order):

1 Room free? If any room is available at the meeting's start time, assign it to the lowest-numbered free room. The meeting runs [start, end) as given.
2 All rooms busy? Find the room whose current meeting finishes earliest (break ties by lowest room number). Wait for it, then assign this meeting there. Its new end time = earliest_finish + (end − start) — the meeting keeps its original duration.
💡 Key Insight

At each step we need two different "best" answers: (a) the lowest-numbered free room and (b) the room finishing soonest. These are two separate orderings of the same set of rooms. The Two Heaps pattern exists precisely for this: one min-heap per ordering.

Worked Example — n=2, meetings=[[0,10],[1,5],[2,7],[3,4]]

[0,10] → Room 0 free ✓
[1,5] → Room 1 free ✓
[2,7] → Both busy. Room 1 ends at 5 (earliest). Delayed to [5,10]. count[1]=2
[3,4] → Both busy. Tie: both end at 10. Room 0 wins. Delayed to [10,11]. count[0]=2
Result: max count=2, first at index 0 → Room 0

Common Mistake: Wrong Delay Duration

✗ new_end = meeting.end
  (ignores when room became free)

✓ new_end = earliest_end + (end − start)
  (preserves the meeting duration)

Edge cases: n=1 (all meetings queue to Room 0); all meetings concurrent (n−1 rooms fill, rest delay); a delayed meeting's end equals another's start (the freed room becomes available just in time).

2 The Algorithm: Two Min-Heaps ▾

We maintain two min-heaps that together track every room. Rooms migrate between them as meetings start and finish.

Free Heap
Key: room_number
Top = lowest free room
Init: [0, 1, …, n−1]
assign →
⇄
← release
Busy Heap
Key: (end_time, room_number)
Top = earliest-finishing room
Init: empty
1 Setup: Sort meetings by start time. Fill the free heap with [0..n−1]. Create an empty busy heap. Create count[0..n−1] = 0.
2 Release finished rooms: For each meeting [start, end], pop all entries from the busy heap where end_time ≤ start. Each popped room goes back into the free heap.
3 Case A — free heap non-empty: Pop the minimum room number (lowest free room). Increment count[room]. Push (end, room) to busy heap.
4 Case B — free heap empty: Pop (earliest_end, room) from busy heap. Increment count[room]. Push (earliest_end + end − start, room) to busy heap.
5 Answer: Return count.index(max(count)) — the first index with the maximum count (automatically lowest room number on ties).
🔑 Why Sorting Matters

Processing meetings chronologically ensures that when we check "which rooms are free by this start time," the busy heap's condition end_time ≤ start is correct and complete. Without sorting we'd miss rooms that freed up between meetings.

3 Why It Works: The Invariant ▾
🔐 The Invariant

When processing any meeting, every room is in exactly one heap. The free heap holds all currently available rooms (ordered by number); the busy heap holds all currently occupied rooms (ordered by finish time, then room number). Together they partition {0, …, n−1}.

Why two heaps and not one? When a room is free, we want the minimum room number. When all rooms are busy, we want the minimum end time. These are different sort keys. A single heap sorted by one key is optimal for that query but slow (O(n) scan) for the other. Two heaps give O(log n) for both.

Why "release before assign"? If we assigned before releasing, a room that finishes right at the current start time would still look busy, causing unnecessary delays. Releasing first (condition end_time ≤ start, i.e., room is free by the meeting's start) is strictly correct per the problem spec.

⚠️ The Duration Bug (Most Common Wrong Answer)

When delaying, the meeting starts at earliest_end, not at start. Its length must stay the same: end − start. So the new finish = earliest_end + (end − start). Writing heapq.heappush(busy, (end, room)) in the else branch uses the original end — that's wrong whenever a delay happened.

Tie-breaking is free: Storing tuples (end_time, room_number) in the busy heap gives Python's built-in tuple comparison both sort keys simultaneously. When two rooms finish at the same time, the lower room_number sorts first — exactly the tie-breaking rule we need, at zero extra cost.

4 The Code (Annotated) ▾

Solution 1 — Two Heaps (Optimal · O(m log n) · Interview Pick)

import heapq
def mostBooked(n, meetings): ← n rooms (0..n-1), meetings = [[start,end],...]
meetings.sort() ← sort by start time; O(m log m) preprocessing
free = list(range(n)) ← [0,1,...,n-1] — all rooms start available
heapq.heapify(free) ← make it a proper min-heap (already sorted, but needed)
busy = [] ← min-heap of (end_time, room_number)
count = [0] * n ← count[i] = total meetings hosted in room i
for start, end in meetings:
# Release rooms whose meetings ended at or before 'start'
while busy and busy[0][0] <= start: ← busy[0][0] is the earliest end_time in busy heap
_, room = heapq.heappop(busy) ← pop (end_time, room); room is now free
heapq.heappush(free, room) ← return room to free heap (sorted by room number)
if free: ← Case A: at least one room available
room = heapq.heappop(free)← lowest-numbered free room
count[room] += 1
heapq.heappush(busy, (end, room)) ← room occupied until 'end'
else: ← Case B: all rooms busy — must delay
end_time, room = heapq.heappop(busy)← earliest finish, lowest room# on tie
count[room] += 1
new_end = end_time + (end - start)← CRITICAL: preserve duration, not original end
heapq.heappush(busy, (new_end, room))
return count.index(max(count)) ← first index of max = lowest room# that ties
⚠️ Most Common Bug: Using end Instead of new_end

In the else branch, many developers write (end, room) — the original meeting end. This is wrong when end_time > start because the meeting was delayed. The room won't be free until end_time + (end − start). Always compute new_end.

Solution 2 — Brute Force O(m·n) (for comparison)

def mostBooked_brute(n, meetings): ← correct but TLE on large inputs
meetings.sort()
end_times = [0] * n ← end_times[i] = when room i next becomes free
count = [0] * n
for start, end in meetings:
chosen = -1
for i in range(n): ← O(n) scan every meeting = bottleneck
if end_times[i] <= start:
chosen = i; break ← first free room (lowest index, since we scan 0..n-1)
if chosen == -1:
chosen = min(range(n), key=lambda i: (end_times[i], i))← O(n) scan for earliest end
end_times[chosen] += end - start
else:
end_times[chosen] = end
count[chosen] += 1
return count.index(max(count))
💡 Complexity Summary

Two Heaps: Time O(m log n + m log m) ≈ O(m log(m+n)). Space O(n) for heaps + count.
Brute Force: Time O(m·n). Space O(n). TLE when m, n ≈ 105.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Read the Signals

"Two different minimum queries at each step: min by room number AND min by end time. Two different minima = Two Heaps. This is the same pattern as Find Median from Data Stream."

Step 2: Propose Brute Force First

"Trivially: sort meetings, then for each one scan all n rooms — O(m·n). Correct, but with m=n=105 that's 1010 ops. We need log n per query, not linear."

Step 3: Identify Both Heap Contents

"Free heap: room numbers (so the top = smallest available room number). Busy heap: (end_time, room_number) tuples — Python tuple comparison gives free tiebreaking by room number at no cost."

Step 4: Nail the Transfer Logic

"Before assigning: drain busy heap of entries where end_time ≤ start. This is the 'state synchronisation' step — rooms that finished before this meeting's start are free again."

Step 5: Get the Delay Formula Right

"When delaying: new_end = earliest_end + (end − start). Not just 'end'. The meeting preserves its duration; it just starts later. This is the line most wrong answers get wrong — mention it explicitly."

🗣️ What to Say to the Interviewer

"I need two different 'minimum' queries — that's the Two Heaps signal.
Sort meetings by start. Free heap holds room numbers, busy heap holds (end, room) pairs.
For each meeting: first release any room from busy heap with end_time ≤ start.
If free heap non-empty: pop lowest room#, push (end, room) to busy, increment count.
Else: pop (earliest_end, room) from busy, new_end = earliest_end + duration, push (new_end, room).
Finally: count.index(max(count)) gives lowest-index maximum. O(m log n) time, O(n) space."
🔮 Likely Follow-up Questions
  • "How many rooms do you need for all meetings to run without delay?" → Meeting Rooms II: one heap of end times, track max heap size.
  • "What if we want the room with fewest meetings (not most)?" → Same algorithm, return argmin instead of argmax.
  • "What if meetings can be rejected (not just delayed)?" → Greedy: if all busy and delay cost too high, reject. Track via priority queue.

7 Summary & Key Takeaways ▾
Two different "min" queries → two min-heaps: free (by room#), busy (by end_time, room#)
Sort meetings first to enable clean chronological processing
Release rooms from busy heap before each assignment (end_time ≤ start)
Delayed meetings: new_end = earliest_end + (end − start) — preserve duration, not absolute end
Tuple ordering (end_time, room_number) gives free tie-breaking by lowest room number
count.index(max(count)) naturally returns the lowest-numbered room on equal counts

🎯 When to Use Two Heaps
  • You need the minimum of two different orderings of the same elements simultaneously
  • Elements "move" between two pools based on time or threshold events
  • Scheduling problems: servers, workers, meeting rooms, CPU tasks
  • Streaming median problems (upper half / lower half split)
Meeting Rooms IIMedium — Count min rooms needed; one heap of end times, max heap size = answer
Find Median from Data StreamHard — Classic two heaps: max-heap (lower half) + min-heap (upper half)
Process Tasks Using ServersMedium — Identical two-heap pattern: free servers (weight, idx) + busy servers (end, weight, idx)
IPOHard — Min heap by capital unlocks projects; max heap by profit picks best unlocked
Task SchedulerMedium — Max heap of task frequencies with cooldown; greedy assignment
Sliding Window MedianHard — Two heaps with lazy deletion as elements enter/leave the window
