LeetCode #1976

Number of Ways to Arrive at Destination

Count the number of ways to travel from node 0 to node n-1 in minimum time in a weighted undirected graph.

Medium Dynamic Programming Dijkstra + Count Paths Time: O((V+E) log V) Space: O(V + E)
1 Understanding the Problem ▾

You are given n cities (labeled 0 to n-1) and a list of bidirectional roads, each with a travel time. Start at city 0, end at city n-1. Find how many distinct routes achieve the minimum possible travel time. Return the count modulo 10⁹+7.

💡 Key Insight

This is NOT just "find the shortest path" — it's "count ALL paths tied for shortest." The trick is to run Dijkstra's algorithm but maintain a parallel ways[v] array alongside dist[v]. Every time you discover a new shortest path to v, reset the count. Every time you find a tie, add to the count. The counts accumulate from predecessors on the optimal subgraph — this is pure DP!

✓ The Two Critical Cases

dist[u]+w < dist[v]:
→ Reset: ways[v] = ways[u]

dist[u]+w == dist[v]:
→ Accumulate: ways[v] += ways[u]

✗ What doesn't work: plain BFS

BFS counts hops, not weights.
3 hops at weight 1 beats
1 hop at weight 10 in BFS,
but NOT in travel time.

Edge cases: n=1 (answer is 1, already at destination), two nodes with one road (answer is 1), multiple parallel edges between same pair, and very large graphs needing mod 10⁹+7 to prevent overflow.

2 The Algorithm: Dijkstra + Count Paths ▾

We augment standard Dijkstra's algorithm with a ways[ ] array. The insight is that Dijkstra processes nodes in non-decreasing distance order — this is the exact ordering we need to safely accumulate path counts, because all optimal predecessors are finalized before their successors.

1 Build adjacency list: Convert edge list to adj[u] = [(v, w), ...]. Each road [u, v, t] is bidirectional — add both u→v and v→u.
2 Initialize: dist[0]=0, ways[0]=1. Everything else: dist[i]=∞, ways[i]=0. Push (0, 0) onto a min-heap. The source has 1 way to reach itself (the empty path).
3 Pop minimum from heap: Extract (d, u). If d > dist[u], skip (stale entry — a shorter path was found since this was pushed). Otherwise, node u is now being processed.
4 Relax edges — the DP step: For each neighbor v via edge w, compute nd = dist[u] + w.
— If nd < dist[v]: update dist[v]=nd, reset ways[v]=ways[u], push (nd,v).
— If nd == dist[v]: accumulate ways[v] = (ways[v]+ways[u]) % MOD.
5 Return ways[n-1]: When the heap empties, ways[n-1] holds the count of all shortest paths to the destination.
📐 The Recurrence Relations
dist[v] = min(dist[v], dist[u] + w)     ← Dijkstra relaxation
ways[v] = ways[u]                    ← if new shortest (reset)
ways[v] = (ways[v] + ways[u]) % MOD  ← if equal (accumulate)
🔐 Why This Is DP on a Graph

The counting part obeys optimal substructure: the number of shortest paths to v equals the sum of shortest paths to all its optimal predecessors u. Dijkstra's order ensures we process u before v (since dist[u] < dist[v] on any shortest path), so we never recompute a subproblem — it's exactly DAG path counting where the DAG is the "optimal subgraph" induced by shortest paths.

3 Why It Works: Dijkstra's Invariant & Count Correctness ▾

Dijkstra's Algorithm — invented by Edsger W. Dijkstra in 1956, published 1959. It solves Single Source Shortest Path (SSSP) in graphs with non-negative edge weights. With a binary heap, it runs in O((V+E) log V).

🔐 Dijkstra's Core Invariant

When a node u is popped from the min-heap, dist[u] is permanently optimal — no future discovery can improve it. Proof: all remaining nodes in the heap have distances ≥ dist[u] (it's a min-heap), and all edges are non-negative, so any path through future nodes can only be longer.

Why counting is correct: Because Dijkstra finalizes nodes in non-decreasing distance order, when we pop u and update ways[v], we know ways[u] is already its final count (all of u's predecessors were processed earlier). The accumulation ways[v] += ways[u] is summing over all optimal predecessor edges — exactly the DAG path counting formula. No subproblem is ever revisited.

Why stale entries don't corrupt counts: When a stale entry (d, u) with d > dist[u] is popped, we skip it with continue. If we didn't skip, we'd process u a second time, adding duplicate ways[u] contributions to its neighbors — producing wrong (too large) counts.

⚠️ The Stale Entry Problem

The min-heap can have multiple entries for the same node at different distances. Example: push (7, node3) first, then find a shorter path and push (4, node3). When (4, node3) is popped and processed, (7, node3) becomes stale. The guard if d > dist[u]: continue is essential — without it, you'd process node3 twice and corrupt path counts.

💡 When to Recognize This Pattern

The signal combination is: weighted graph (non-negative) + "shortest path" + "number of ways / count paths". Any time a DP problem asks for an optimal value AND a count of ways to achieve that optimal, think about carrying a second parallel DP array alongside the first.

4 The Code (Annotated) ▾
import heapq— Python's min-heap module for Dijkstra's priority queue
MOD = 10**9 + 7— standard large prime modulus to prevent integer overflow
def countPaths(n, roads):— n = node count, roads = [[u,v,time], ...]
adj = [[] for _ in range(n)]— adjacency list: adj[u] = [(v, w), ...]
for u, v, w in roads:— build both directions (undirected graph)
adj[u].append((v, w))— u can reach v at cost w
adj[v].append((u, w))— v can also reach u at cost w (bidirectional)
dist = [float('inf')] * n— best known distance from source; ∞ until discovered
ways = [0] * n— number of shortest paths to each node; 0 until reached
dist[0] = 0— source costs 0 time to reach itself
ways[0] = 1— exactly 1 way to "be at" the source (the empty path)
heap = [(0, 0)]— min-heap: (distance, node); start with source at dist 0
while heap:— process until all reachable nodes finalized
d, u = heapq.heappop(heap)— extract node with smallest current tentative distance
if d > dist[u]:— stale heap entry: we already found a shorter path to u
continue— skip it — reprocessing would add duplicate way counts
for v, w in adj[u]:— examine all neighbors v reachable from u via edge weight w
nd = dist[u] + w— candidate new distance to v if we go through u
if nd < dist[v]:— STRICTLY BETTER: found a new shortest path to v!
dist[v] = nd— update shortest known distance to v
ways[v] = ways[u]— RESET count: all paths through u are now the optimal ones
heapq.heappush(heap, (nd, v))— push v with its new shorter distance for later processing
elif nd == dist[v]:— EQUAL: another set of paths tied for the same optimal cost!
ways[v] = (ways[v] + ways[u]) % MOD— ACCUMULATE: add u's path count to v's running count
return ways[n - 1]— ways to reach destination at minimum cost
⚠️ Common Mistake 1: Applying MOD on the Reset

ways[v] = ways[u] — no MOD here! The value ways[u] is already taken mod at each step. Applying mod again is redundant and wastes operations. MOD is only needed in the addition case: (ways[v] + ways[u]) % MOD.

⚠️ Common Mistake 2: Not Skipping Stale Entries

Without if d > dist[u]: continue, the algorithm works for shortest distances (stale entries just don't improve anything), but it CORRUPTS path counts. A stale entry processes node u a second time, re-adding ways[u] to its neighbors' counts — giving answers that are too large.

🚫 Common Mistake 3: Using ways[0] = 0

If you forget to set ways[0] = 1, no path counts will ever propagate from the source. Every ways[v] = ways[u] copy will copy 0, and every accumulation will add 0. The final answer would be 0 regardless of graph structure.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Pattern

Key signals: (1) weighted undirected graph, (2) "shortest path", (3) "how many ways/routes". Pattern = Dijkstra + count. If weights were all 1, you'd use BFS + count. If there were negative weights, you'd need Bellman-Ford. Non-negative weights → Dijkstra is optimal.

Step 2: Augment Standard Dijkstra

Start with the Dijkstra skeleton: adj list, dist array initialized to ∞ except source=0, min-heap. Then ask: "what second state do I need?" Answer: ways[] initialized to 0 except ways[0]=1. Now define how ways propagate during edge relaxation.

Step 3: Define the Two Relaxation Cases Precisely

There are exactly two non-trivial cases: (a) new shortest found → ways[v] = ways[u] and push to heap; (b) equal distance tie → ways[v] = (ways[v]+ways[u]) % MOD (no push needed, dist[v] doesn't change). Don't forget the stale-entry skip guard.

Step 4: State Complexity and Why MOD

Time: O((V+E) log V) — each node processed once, each edge relaxed once, heap operations O(log V). Space: O(V+E) for adj list + O(V) for dist/ways. MOD is required because path count can grow exponentially in dense graphs (each node could fan out to many optimal successors).

🗣️ What to Say to the Interviewer

"This is a weighted shortest-path problem — I'll use Dijkstra's algorithm.
But we also need to COUNT paths, not just find the minimum distance.
I'll maintain a ways[] array alongside dist[], with ways[0]=1 as the seed.
During edge relaxation: if I find a strictly shorter path to v,
I update dist[v] and RESET ways[v] = ways[u].
If I find a tie (same distance), I ACCUMULATE: ways[v] += ways[u] mod M.
I also need to skip stale heap entries — check d == dist[u] before relaxing.
Final answer: ways[n-1]. Time: O((V+E) log V). Space: O(V+E)."
💡 Follow-Up Questions the Interviewer Might Ask
  • "Why not BFS?" — BFS counts hops, not weighted distances. Only works for unit-weight graphs.
  • "Can this have negative weights?" — No! Dijkstra is wrong for negative weights. Use Bellman-Ford.
  • "What if you need all the actual paths?" — Backtrack from n-1: for each node, store predecessor(s) on shortest paths, then DFS/BFS over the predecessor graph.
  • "Why not Floyd-Warshall?" — O(V³) is too slow for large V; Dijkstra gives O((V+E) log V) which is much better for sparse graphs.

7 Summary & Key Takeaways ▾
Dijkstra finds shortest distances; augment with ways[] to count optimal paths
Two relaxation cases: new shortest → reset ways[v]=ways[u]; tie → add ways[v]+=ways[u] % MOD
Always skip stale heap entries (d > dist[u]) — they corrupt path counts
ways[0]=1 seeds the DP: the source has exactly one "empty" path to itself
Result must be taken mod 10⁹+7 — path count can grow exponentially in dense graphs
