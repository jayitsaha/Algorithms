LeetCode #357

Count Numbers with Unique Digits

Given an integer n, return the count of all numbers x with unique digits, where 0 ≤ x < 10n.

Medium Dynamic Programming Permutation Count Time: O(1) Space: O(1)
1 Understanding the Problem ▾

We want to count every integer x in [0, 10n) whose decimal representation has no repeated digit. For n=2 the range is [0, 100), so we count 0,1,2,...,98,99 — but skip numbers like 11, 22, 33, ... where a digit appears more than once.

💡 Key Insight

Instead of inspecting each number individually, we can count directly by digit length. A k-digit number with all unique digits is a combinatorics question: how many ways can we fill k position slots without ever repeating a digit? This is the multiplication principle applied to constrained slots.

Tracing n=2 by hand:

Why 81 for 2-digit numbers? Position 1 (leading) has 9 choices: {1,2,...,9} — no leading zero. Position 2 has 9 choices: {0,1,...,9} minus whichever digit was chosen first. So 9 × 9 = 81.

Key edge cases:

  • n=0: range is [0, 1), only x=0 qualifies → answer is 1.
  • n≥10: there are only 10 unique digits (0–9). Any 11+ digit number must repeat a digit by pigeonhole. Valid count stops growing after n=10 and plateaus at 8,877,690.
2 The Algorithm: Digit-by-Digit Permutation Counting ▾

We iterate over each possible digit-length k from 1 to min(n, 10), count how many k-digit numbers have all unique digits using the multiplication principle, then sum everything together.

1 Initialize result = 1. The single number 0 is always in range and always valid (only one digit, trivially unique). We pre-count it.
2 Set choices = 9, unique_count = 9. For the very first digit position (pos 1) we have 9 choices (1–9, no leading zero). unique_count will accumulate the product for the current digit-length.
3 For each k from 1 to min(n, 10): Add unique_count to result (it holds the count for exactly k-digit numbers). Then decrement choices by 1 and multiply unique_count by the new choices. This prepares the product for the next digit-length k+1.
4 Return result. After the loop, result contains 1 (for 0) + count(1-digit) + count(2-digit) + ... + count(k-digit) for all lengths up to min(n,10).
📐 The Counting Formula
count(1-digit) = 9 — choices {1..9}
count(2-digit) = 9 × 9 — pos1: {1..9}, pos2: {0..9} minus pos1
count(3-digit) = 9 × 9 × 8
count(k-digit) = 9 × 9 × 8 × 7 × … × (10 − k + 1)

answer(n) = 1 + sum of count(k) for k = 1 … min(n, 10)
🔑 Why 9 × 9 × 8 × 7 and NOT 9 × 9 × 9 × 8?

Position 1 picks from {1,...,9}: 9 options. Position 2 picks from {0,...,9} minus whatever pos 1 used — we lose pos1's digit but regain 0! Still 9 options. Position 3: we've now used 2 digits → 8 remaining. Position 4: 7 remaining. The asymmetry at position 1 (no 0) is balanced by regaining 0 at position 2.

3 Why It Works: The Invariant ▾
🔐 The Invariant

After each iteration with digit-length k, the variable unique_count holds exactly the number of k-digit integers with all distinct digits. The variable choices holds the count of remaining available digits for the next position. Together, they encode the full state needed for the next step without any extra bookkeeping.

Why the multiplication principle applies: At each digit position, the number of valid choices is independent given what was chosen before — we simply know "one more digit is forbidden." So the total count for k positions is the product of choices at each individual position. This is the fundamental counting principle.

Why does it plateau at n=10? After placing 10 digits in 10 positions using all of {0–9}, the 11th position would require a digit that's already present — impossible. So count(11-digit) = 0, count(12-digit) = 0, and so on. Capping the loop at k=10 is mathematically exact, not an approximation.

⚠️ Common Mistake: Off-by-One in the Update Order

In the code, we add unique_count to result before updating it. If you accidentally update first and then add, you skip the 1-digit count and double-count the 2-digit one. Always: add → decrement choices → multiply.

Edge case n=0 proof: range(1, min(0, 10) + 1) = range(1, 1) which is empty. The loop body never runs, and we return the initialized result=1. Correct — only {0} is in [0, 100) = [0,1).

4 The Code (Annotated) ▾

Solution 1 — Iterative Permutation Count (Optimal, Interview Pick)

def countNumbersWithUniqueDigits(n: int) -> int:
if n == 0: ← 10^0=1 → only x=0 → return 1 directly
return 1
result = 1 ← pre-count 0 (always valid)
choices = 9 ← next-position options (starts at 9 for pos2)
unique_count = 9 ← product for current k; 9 = count of 1-digit valid numbers
for k in range(1, min(n, 10) + 1): ← process lengths 1, 2, …, min(n,10)
result += unique_count ← add count for k-digit numbers to total
choices -= 1 ← one more digit slot is locked; next pos loses one option
unique_count *= choices ← extend product for k+1 digits
return result
⚠️ Why add unique_count BEFORE updating it?

At iteration k=1, unique_count=9 (correct for 1-digit). We add 9 to result, THEN multiply by 9 (for the 2nd position). At k=2, unique_count=81 (correct for 2-digit). This staged "add, then grow" pattern keeps the product in sync with the loop counter. Reversing the order would make k=2 use the k=3 product — off by one length.

💡 Complexity Comparison

Iterative Count (Solution 1): Time O(min(n,10)) = O(1), Space O(1).
Brute Force (Solution 2): Time O(10n · n) — completely impractical for n ≥ 7.
The closed-form iterative count is the only viable approach at scale.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Pattern

Key signals: "count numbers with a property over a range [0, 10n)". When a range has a power-of-10 upper bound and the property is digit-based, think digit-by-digit counting, not iteration. The answer is a formula, not a search.

Step 2: State the Brute Force and Reject It

"I could loop x from 0 to 10n-1 and check each with a set — that's O(10n · n). For n=10 that's 10 billion iterations. We need a counting formula instead."

Step 3: Group by Digit Length

"I'll count valid numbers by their number of digits. For each length k, count k-digit numbers with all unique digits. Sum those counts, plus 1 for zero, for the answer."

Step 4: Apply Multiplication Principle

"For k digits: pos 1 = 9 choices (no leading zero). Pos 2 = 9 remaining (0 is back, but pos1's digit is gone). Pos 3 = 8. So count(k) = 9 × 9 × 8 × 7 × … Cap at k=10 by pigeonhole."

🗣️ What to Say to the Interviewer

"This is a counting problem, not a search problem. I'll count by digit length.
Initialize result=1 for x=0. Then for each length k from 1 to min(n,10):
pos1: 9 options (1-9, no leading zero)
pos2: 9 options (0-9 minus the first digit — 0 rejoins!)
pos3: 8 options, pos4: 7, ... (decreasing by 1 each time)
So count(k) = 9 × 9 × 8 × 7 × (10-k+1).
Total time is O(min(n,10)) = O(1). Space O(1)."
🔮 Likely Follow-up Questions
  • "What if the digits must be strictly increasing?" → C(10, k) — choose k digits then arrange in one fixed order
  • "Base-b version?" → Replace 9 with b−1 (no leading zero), 9 with b−1 (pool minus one), then decrement as before
  • "n=3 answer?" → 1+9+81+648 = 739 — be ready to compute live
  • "Why stop at 10?" → Only 10 distinct decimal digits; 11th position must repeat by pigeonhole

7 Summary & Key Takeaways ▾
Count by digit-length, not by iterating over all numbers
Leading digit: 9 choices (1–9); second digit: 9 choices (0 rejoins); subsequent: decreasing
Formula pattern: 9 × 9 × 8 × 7 × ... × (10 − k + 1) for k-digit numbers
Always add 1 upfront for the number 0 (zero is always valid)
Cap loop at k=10 — beyond 10 digits, repetition is unavoidable by pigeonhole
Time is O(min(n,10)) = O(1); Space O(1) — no arrays, just two accumulators
