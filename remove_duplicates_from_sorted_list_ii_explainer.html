LeetCode #82

Remove Duplicates from Sorted List II

Given the head of a sorted linked list, delete all nodes that have duplicate numbers, leaving only distinct numbers from the original list. Return the head of the cleaned list.

Medium Linked List Skip All Duplicates Time: O(n) Space: O(1)
1 Understanding the Problem ▾

You have a sorted linked list. Your mission: delete every node whose value appears more than once — not just the extras, but all copies of that value. Return the head of the cleaned list.

This differs from LC #83 (Remove Duplicates from Sorted List I), where you keep one copy. Here, even two occurrences of a value means that value vanishes entirely from the result.

💡 Key Insight

The list is sorted, so all copies of any value sit together in a contiguous run. We can detect a run by peeking ahead: if curr.val == curr.next.val, that value is duplicated. Once detected, skip the entire run — not just one node — before relinking.

Without Dummy Head

If head itself is duplicated, we'd have no predecessor to relink. We'd need messy special-case code to find the new head.

With Dummy Sentinel

dummy.next = head. The answer is always dummy.next. No special cases — even if head is deleted, it just works.

2 The Algorithm: Dummy Head + Skip All Duplicates ▾

We use two pointer roles: prev (last confirmed-clean node, the tail of our result) and curr (the node currently under inspection). When we detect a duplicate run, prev stays put while curr fast-forwards through the entire group. Then prev.next = curr performs the bypass in a single assignment.

1 Create sentinel: dummy = ListNode(0, head). Set prev = dummy.
2 Outer loop: curr = prev.next. Repeat while curr exists.
3 Peek ahead: if curr.next exists and curr.val == curr.next.val — duplicate detected. Record dup_val = curr.val.
4 Inner skip loop: while curr is valid and curr.val == dup_val, advance curr = curr.next. Do NOT touch prev.
5 Bypass: prev.next = curr. This single assignment cuts the entire duplicate run from the result chain.
6 Unique node: if no duplicate, this node is safe — advance prev = curr, then curr = curr.next.
7 Return: dummy.next — always the correct new head, regardless of what happened to the original head.
🔑 The Core Branch

Is curr the start of a duplicate group? — check by peeking at curr.next. If yes: record value, fast-forward curr past the whole group, relink prev.next. If no: it's safe, advance prev forward.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every moment, prev points to the last confirmed-distinct node. Every node from dummy through prev (inclusive) has been verified as a unique value and belongs in the final answer.

Why the invariant is sufficient: When we detect a duplicate group, prev does not advance — it stays at the last safe node. The inner loop exhausts the entire group, leaving curr pointing at whatever comes next. Setting prev.next = curr splices out the group atomically. The invariant is re-established immediately after the bypass.

Why sorted matters: In a sorted list, all nodes with the same value are contiguous. Once we've walked past a group of 3s, we will never encounter another 3. This guarantees correctness with a single pass and no hash map needed.

⚠️ Edge Cases Covered
  • Head is a duplicate: prev stays at dummy; dummy.next skips the run. Works seamlessly.
  • All nodes duplicated: prev never leaves dummy; returns empty list (dummy.next = null).
  • Single node: curr.next is null — goes straight to the unique path; returns that node.
  • Two identical nodes: run detected immediately; both skipped; empty list returned.
  • Consecutive different duplicate groups: handled naturally — the outer loop restarts correctly after each bypass.
4 The Code (Annotated) ▾

Solution 1 — Dummy Head + Skip All (Optimal · Interview Pick)

def deleteDuplicates(head: ListNode) -> ListNode:
dummy = ListNode(0, head) // sentinel; value 0 is safe since list values are positive integers
prev = dummy // prev = tail of the confirmed-clean result list; starts at sentinel
while prev.next: // iterate as long as there's a candidate to evaluate
curr = prev.next // curr is the node under examination this iteration
if curr.next and curr.val == curr.next.val: // peek: does curr start a duplicate run?
dup_val = curr.val // record the value to eliminate from the list
while curr and curr.val == dup_val: // inner loop: advance past ALL nodes with this value
curr = curr.next // prev stays still — we're just walking curr forward
prev.next = curr // bypass: entire run is cut from the result chain
else: // no duplicate: curr is a unique node, safe to include
prev = prev.next // advance prev — this node is locked into the result
return dummy.next // dummy.next is the new head of the cleaned list
⚠️ Most Common Bug

Skipping only one node instead of the whole group. If you write curr = curr.next; curr = curr.next instead of the inner while loop, you'll leave behind extra duplicates when there are three or more copies. Always exhaust the group with the inner while curr and curr.val == dup_val.

✅ Complexity

Time O(n): Every node is visited at most twice (outer scan + inner skip). Space O(1): Only prev, curr, and dup_val — no extra data structures regardless of input size.

Solution 2 — Hash Set (Two Pass, For Contrast)

def deleteDuplicates_set(head):
seen, multi = set(), set() // Pass 1: collect all values seen twice or more
curr = head
while curr:
if curr.val in seen: multi.add(curr.val)
seen.add(curr.val); curr = curr.next
dummy = ListNode(0, head); curr = dummy // Pass 2: rebuild list omitting multi values
while curr.next:
if curr.next.val in multi: curr.next = curr.next.next
else: curr = curr.next
return dummy.next

O(n) time, O(n) space. Works on unsorted lists too, but wastes memory when the sorted property already lets us solve this in O(1) space.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify the Pattern Signals

Two signals: (1) sorted list — same values grouped; (2) remove ALL duplicates — not keep one, remove everything. Sorted means one pass; "remove all" means a simple advance won't work — the predecessor can't move during a skip.

Step 2: Tackle the Head Edge Case Proactively

Before writing any code, ask: what if the head is a duplicate? You'd need to return a different head. Classic fix: create a dummy sentinel before the real list. The answer is always dummy.next — no special cases ever.

Step 3: Define the Two-Pointer Roles

prev = tail of the clean result (moves only when we've confirmed a node is unique). curr = candidate under inspection. Peek at curr.next: if it matches curr.val, we have a group to skip.

Step 4: Inner Loop Exhausts the Group

Once you know a value is duplicated, advance curr through the entire run with an inner while loop. Then prev.next = curr performs the bypass. This is the "skip all" pattern — atomic, clean, no leftover copies.

🗣️ What to Say to the Interviewer

"The list is sorted, so duplicates are grouped — a single pass works.
 I'll use a dummy sentinel to cleanly handle head deletions.
 Two roles: prev = last confirmed-unique node, curr = node being examined.
 At each curr, I peek at curr.next.
 If they match: record dup_val, inner-loop past the entire group, relink prev.next.
 If they differ: curr is unique — advance prev.
 Return dummy.next as the new head.
 Time O(n), Space O(1)."
📌 Expect These Follow-ups
  • Can you do it recursively? — Yes: if head and head.next share a value, skip both and recurse on the remainder; else head.next = recursive(head.next); return head.
  • Unsorted list? — Need a hash set (two passes). O(n) space, no longer exploitable.
  • Difference from LC #83? — #83 keeps one copy; this removes all. Code difference: the inner while loop and the non-advancement of prev.
  • What if there are at most 2 of each value? — Same logic still correct; inner loop just exits after one advance.

7 Summary & Key Takeaways ▾
Dummy sentinel eliminates all head-deletion edge cases in one stroke.
Sorted property lets a single O(n) pass replace an O(n) hash set.
prev never advances during a skip — it's the anchor that makes relinking safe.
Inner while loop exhausts the entire duplicate group before relinking.
Time O(n), Space O(1) — the canonical optimal solution.
