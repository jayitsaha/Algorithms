LeetCode #958

Check Completeness of a Binary Tree

Given the root of a binary tree, determine if it is a complete binary tree. In a complete binary tree every level is fully filled except possibly the last, which is filled from left to right.

Medium Trees BFS Level Order Time: O(n) Space: O(n)
1 Understanding the Problem ▾

A complete binary tree has a very specific shape constraint: every level is completely filled from left to right, with the only allowed "gap" being at the rightmost positions of the bottom level. Nodes cannot be "skipped" — you cannot have a right child without a left child, and you cannot have any node on the last level if there is an empty slot to its left.

💡 Key Insight

In a complete binary tree, if you do a level-order (BFS) traversal and ever encounter a null node, then every subsequent node in the traversal must also be null. The moment you see a real node after a null, the tree has a "gap" and is NOT complete.

Edge Cases to consider: A single node is trivially complete. A fully perfect binary tree (all leaves at same depth) is complete. A tree where any non-leaf node has only a right child (but no left child) is immediately invalid. The subtle case: a node on the last level of the second-to-last level can have only a left child — that is fine as long as no sibling to its right has any children.

2 The Algorithm: BFS with Null-Sentinel Flag ▾

We perform a BFS (level-order traversal) with a critical modification: we add null children to the queue, not just real nodes. Once we dequeue a null, we flip a boolean flag found_null = True. If we then dequeue any real (non-null) node while the flag is set, we immediately return False. If BFS completes without that, the tree is complete.

1 Initialize: Push the root into the queue. Set found_null = False. (Root being null would mean an empty tree — handle as edge case.)
2 BFS loop: While the queue is non-empty, dequeue the front element curr.
3 Null check: If curr is None, set found_null = True and continue to the next iteration. We have seen our first gap position.
4 Violation check: If curr is a real node but found_null is already True, return False immediately. A gap appeared before this node — the tree cannot be complete.
5 Enqueue children unconditionally: Push curr.left AND curr.right onto the queue — whether they are real nodes or None. This preserves the slot ordering.
6 Return True: The queue drained with no violation. The tree is complete.

🔑 Why Enqueue Nulls?

Standard BFS only enqueues non-null children. But here we need to know exactly where null positions appear relative to real nodes. By enqueuing both children unconditionally, the queue faithfully represents all "slots" in left-to-right, level-by-level order. A null in the queue is an empty slot; if a real node follows it, there is a gap.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every point during BFS, the queue contains the "frontier" of tree slots in exact left-to-right, level-by-level order — including null slots. Because we always enqueue left child before right child, this ordering is preserved. If the dequeue sequence ever produces a null followed by a non-null, the null slot came before the non-null in the canonical complete-tree ordering, which means there is a gap.

Formal correctness sketch: A tree is complete if and only if its BFS slot-sequence (including nulls) has the form non-null* null* — some real nodes, then only nulls. Our flag precisely tests this: found_null becomes True at the first null, and returning False at the first subsequent non-null checks whether any non-null appears after the first null.

Why no node-with-only-right-child breaks the invariant: If node X has a right child but no left child, we enqueue None (X.left) first, then the right child. When we dequeue None, found_null becomes True. When we then dequeue the right child (a real node), we detect the violation and return False immediately. This is exactly correct.

⚠️ Common Mistake: Only Enqueuing Real Nodes

Many beginners use the standard BFS pattern: if node.left: queue.append(node.left). This silently skips null slots. The queue then has no way to represent gaps — all gaps are invisible, and even a completely wrong tree shape might pass. You must enqueue both children unconditionally for the sentinel trick to work.

Alternate approach — node count: An equivalent method is to count all nodes (n), then verify that every node's 1-indexed position (root=1, left child=2i, right child=2i+1) is ≤ n. Any position > n means there is a gap. This runs in the same O(n) time but uses O(log n) recursion stack space instead of O(n) queue space.

4 The Code (Annotated) ▾

Solution 1 — BFS Null-Sentinel (Optimal, Interview Pick)

from collections import deque
def isCompleteTree(root) -> bool:
queue = deque([root]) ← Seed queue with root (may be None for empty tree)
found_null = False ← Two-state flag: has a null slot been dequeued?
while queue: ← Process every slot until queue empty
curr = queue.popleft() ← Dequeue front slot (real node OR None)
if curr is None: ← Empty slot encountered
found_null = True ← Mark it — from now, only nulls are acceptable
else: ← Real node encountered
if found_null: ← Real node after a null = gap exists
return False ← Completeness violated — stop immediately
queue.append(curr.left) ← Always enqueue left child, even if None
queue.append(curr.right) ← Always enqueue right child, even if None
return True ← No violation found → tree is complete

Solution 2 — Node Count + DFS Index Check (O(n) time, O(log n) space)

def isCompleteTree(root) -> bool:
def count_nodes(node): ← Count all nodes in the tree
if not node: return 0
return 1 + count_nodes(node.left) + count_nodes(node.right)
n = count_nodes(root) ← Total node count
def check(node, idx): ← idx = 1-based array index of this node
if not node: return True ← Null child is fine — no node to misplace
if idx > n: return False ← Index beyond count → gap exists before it
return (check(node.left, 2*idx) and
check(node.right, 2*idx+1)) ← Check subtrees with standard heap indexing
return check(root, 1) ← Root is at index 1
💡 Complexity Summary

BFS Null-Sentinel: Time O(n), Space O(w) where w = max level width. In a complete tree the last level can hold n/2 nodes so worst-case queue size is O(n).
Node Count + DFS: Time O(n), Space O(h) = O(log n) for recursion stack. Better space if the tree is near-complete and tall.
Optimal (height-based binary search): O(log²n) — compare heights of leftmost and rightmost paths at each recursive level. Harder to code correctly.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Shape Constraint

"This is a shape-validation problem on a binary tree. Complete binary tree means level-by-level left-to-right fill. BFS naturally visits nodes in this exact order — it is the right tool."

Step 2: Identify the Key Observation

"In a BFS traversal, if I ever see a null position followed by a real node, there must be a gap — incompleteness. So I can use a boolean flag: once I dequeue a null, any subsequent real node is a violation."

Step 3: Mention the Null-Enqueue Trick

"The key implementation detail: I must enqueue both children of every real node unconditionally — even if they are null. This keeps null slots in the queue in their correct left-to-right positions."

Step 4: Offer the Alternative (Node Count)

"Alternatively, count n total nodes, then verify every node's heap-style index (root=1, left=2i, right=2i+1) is at most n. Same O(n) time but O(log n) recursion space instead of O(n) queue. Trade-off depends on interview context."

🗣️ What to Say to the Interviewer

"A complete binary tree is filled left-to-right at every level — any gap invalidates it.
I'll use BFS but with a twist: I enqueue null children too, not just real nodes.
I maintain a boolean found_null. The first time I dequeue a null, I set it.
If I later dequeue a real node while found_null is True, there's a gap — return False.
If BFS completes without that, return True.
Time O(n), space O(n) for the queue (worst case the last level)."
🔮 Likely Follow-up Questions
  • "Can you count nodes in O(log² n)?" → Yes, LeetCode #222. Use the height difference between leftmost and rightmost paths at each recursive call.
  • "What if the tree has a million nodes? Where is the bottleneck?" → BFS queue holds up to n/2 nodes on the last level — that is the O(n) space cost. The DFS index approach uses only O(log n) recursion stack.
  • "Difference between complete, perfect, and full binary tree?" → Perfect: all leaves at the same depth. Complete: left-to-right fill (last level may be partial). Full: every node has 0 or 2 children (no node with exactly one child).
  • "How would you serialize a complete binary tree more efficiently?" → Store as a 1-indexed array (heap representation). Array index encodes position without needing null markers.

7 Summary & Key Takeaways ▾
Complete binary tree = every level filled left to right, last level may have rightmost gap only
BFS is the natural fit — it visits nodes in the same left-to-right, level-by-level order as the completeness definition
Enqueue null children too — they hold the place of empty slots in the traversal order
The found_null flag is a two-state machine: NORMAL → NULL_SEEN. Any real node in NULL_SEEN state is a violation
Alternative: count n nodes, check every 1-indexed position ≤ n using DFS with O(log n) space
