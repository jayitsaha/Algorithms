LeetCode #227

Basic Calculator II

Evaluate a string expression with +, −, *, / operators and non-negative integers. No parentheses. Integer division truncates toward zero.

Medium Stacks Stack + Precedence Time: O(n) Space: O(n)
1 Understanding the Problem ▾

You receive a string like "3+2*2" or "14-3/2" and must evaluate it as a proper arithmetic expression — meaning multiplication and division bind tighter than addition and subtraction. There are no parentheses in this version.

💡 Key Insight

A stack lets us defer additions and subtractions until the end, while immediately resolving multiplications and divisions. The stack accumulates a list of signed partial results that we sum at the very end. This cleanly models operator precedence without any explicit precedence rules or recursion.

Naive Left-to-Right (WRONG)

"3+2*2"
Step 1: 3+2 = 5
Step 2: 5*2 = 10 ← wrong!
Answer: 10 (should be 7)

Stack Approach (CORRECT)

"3+2*2"
Push +3 → stack=[3]
Push +2 → stack=[3,2]
See *: pop 2, push 2*2=4 → [3,4]
sum([3,4]) = 7 ✓

Operator precedence:

OperatorsPrecedenceStack behaviorExample
*, / HIGH — evaluate NOW Pop top, compute, push result 2*3 → pop, push 6
+, - LOW — defer to end Push ±number onto stack +5 → push 5; -3 → push -3

Edge cases: Single number "3", multi-digit numbers "142", leading/trailing spaces, consecutive spaces "3 + 5", integer division truncating toward zero (not floor division), and the last number which has no following operator to trigger evaluation.

2 The Algorithm: Single-Pass Stack with Deferred Addition ▾

We scan the string exactly once, character by character. We always act on the previous operator — not the current one — because we need the complete next number first before we can evaluate.

1 Initialize: stack = [], num = 0, op = '+'. The sentinel op = '+' means the first number will be pushed as-is (as if the expression opens with an implicit +).
2 Digit character: Extend the current number: num = num * 10 + int(ch). This handles multi-digit numbers like "142" — after seeing '1', '4', '2' in order, num becomes 142.
3 Operator character OR last character: Evaluate the previous op against the fully-built num:
  + → push +num     - → push -num     * → pop, push top * num     / → pop, push int(top / num)
4 Save state: op = ch (current char becomes the next "previous op"), num = 0 (reset accumulator).
5 Return: sum(stack). Each element in the stack is one signed addend — the result of a fully-resolved multiplicative term. Summing them applies all the deferred additions and subtractions in one shot.
🔑 Why the "previous operator" pattern?

When we see the character +, we cannot act on it yet — we haven't seen the next number. But we can now act on whatever operator we saw before this +, because num is its complete right operand. This one-character lookahead is the elegant heart of the algorithm.

3 Why It Works: The Invariant ▾
🔐 The Stack Invariant

At any point during the scan, the stack contains a set of signed integers that can be freely summed — in any order — to produce the correct cumulative result. Every item in the stack is the completely-resolved value of one additive term: either a raw number (from +/-) or the result of chained multiplications/divisions. Because high-precedence operators are resolved immediately (by pop-compute-push), their term is collapsed before any addition can steal a factor from it.

Why immediate resolution of * and / is correct: When we see *, the next number (num) is its complete right operand — there's no higher-precedence operator that could interpose. So we pop the left operand (which was pushed as a "pending addend" when we saw the previous +/-), multiply, and push the combined result. That combined result is now the new pending addend for the eventual addition.

Why deferred + and − is correct: When we see +, the next number might still be part of a multiplication chain ahead. We cannot add it yet. Pushing it (signed) defers the add until sum(stack), by which point all chaining is done.

⚠️ Python's // vs int(): Truncation Toward Zero

Python's floor division // rounds toward negative infinity: -7 // 2 = -4. But the problem requires truncation toward zero: -7 / 2 = -3. Use int(top / num) — floating-point division followed by int() — which correctly truncates toward zero for both positive and negative results. This is a classic Python interview gotcha.

Full trace of "3+2*2":

i=0: ch='3', digit → num=3
i=1: ch='+', operator → act on op='+' with num=3 → push +3. stack=[3], op='+', num=0
i=2: ch='2', digit → num=2
i=3: ch='*', operator → act on op='+' with num=2 → push +2. stack=[3,2], op='*', num=0
i=4: ch='2', digit → num=2; also i==len(s)-1 → act on op='*' with num=2
→ pop 2, push 2*2=4. stack=[3,4], done
return sum([3, 4]) = 7 ✓
4 The Code (Annotated) ▾

Solution 1 — Stack + Deferred Addition (Interview Pick, O(n) time)

def calculate(s: str) -> int:
stack = [] ← accumulates signed partial sums (pending addends)
num = 0 ← builds current multi-digit number
op = '+' ← sentinel: the "previous" operator before the first number
for i, ch in enumerate(s):
if ch.isdigit(): ← digit: shift and add (handles multi-digit like 142)
num = num * 10 + int(ch)
if ch in '+-*/' or i == len(s) - 1:← operator OR last char → act on prev op
if op == '+': ← defer: push positive number
stack.append(num)
elif op == '-': ← defer: push as negative (subtraction = add negative)
stack.append(-num)
elif op == '*': ← resolve immediately: pop, multiply, push
stack.append(stack.pop() * num)
elif op == '/': ← resolve immediately: pop, divide (truncate toward 0!), push
stack.append(int(stack.pop() / num))
op = ch ← current character becomes the "previous op" for next iteration
num = 0 ← reset accumulator for the next number
return sum(stack) ← all deferred additions/subtractions resolved here
⚠️ Why the "i == len(s) - 1" condition is critical

Evaluation only fires when we see an operator character. But the last token in the expression is always a number, not an operator — so without this condition, the last number would never be evaluated and would silently be discarded. The i == len(s) - 1 check ensures that even when the last character is a digit, we still evaluate op against the fully-built num.

Solution 2 — Two-Pass Tokenize (O(n) time, more readable)

def calculate_two_pass(s: str) -> int:
# Step 1: tokenize into numbers and operators, skip spaces
tokens, i, n = [], 0, len(s)
while i < n:
if s[i] == ' ': i += 1; continue
if s[i].isdigit():
j = i
while j < n and s[j].isdigit(): j += 1
tokens.append(int(s[i:j])); i = j
else: tokens.append(s[i]); i += 1
# Step 2: resolve * and / left-to-right
stack, i = [], 0
while i < len(tokens):
if tokens[i] == '*': stack.append(stack.pop() * tokens[i+1]); i += 2
elif tokens[i] == '/': stack.append(int(stack.pop() / tokens[i+1])); i += 2
else: stack.append(tokens[i]); i += 1
# Step 3: resolve + and - (sum with signs)
result, sign = 0, 1
for t in stack:
if t == '+': sign = 1
elif t == '-': sign = -1
else: result += sign * t
return result
💡 Complexity Summary

Solution 1 (Stack + Deferred): Time O(n), Space O(n) — single pass, elegant
Solution 2 (Two-Pass): Time O(n), Space O(n) — more code but explicit stages
Interview pick: Solution 1 — it's the standard expected answer for this problem.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify the Core Difficulty

"The real challenge isn't parsing — it's operator precedence. If I just evaluate left-to-right, 3+2*2 gives 10 instead of 7. I need to respect that *// bind more tightly than +/-."

Step 2: Propose the Simpler Approach First

"I could tokenize first, then make two passes: one resolving all *//, then sum the remaining values with signs. O(n) time and space but requires extra token storage."

Step 3: Optimize to Single Pass

"Better: use a stack where I push pending +/- terms and immediately resolve *//. Single pass, same complexity, no token list. The key trick: always act on the previous operator — by the time I see an operator, I have the complete left operand in num."

Step 4: Call Out Edge Cases Proactively

"Multi-digit numbers: accumulate with num = num*10 + digit. Spaces: skip with ch.isspace() check. Last number: the end-of-string condition i == len(s)-1 fires evaluation. Integer division: int(a/b) not a//b for truncation toward zero."

🗣️ What to Say to the Interviewer

"The challenge is operator precedence. I'll use a stack to store pending addends.
I scan left-to-right, building up multi-digit numbers digit by digit.
When I hit an operator or the end: evaluate the PREVIOUS operator against the built-up number.
For + or -: push the signed number (defer it). For * or /: pop, compute, push (resolve it now).
At the end, sum the stack. Time O(n), Space O(n).
One Python gotcha: use int(a/b) not a//b for truncation toward zero."
🔮 Likely Follow-up Questions
  • "What if there are parentheses?" → Use a recursive approach or maintain a sign stack for nested context (#224 Basic Calculator)
  • "Can you reduce space to O(1)?" → With only +/- you can; once *// are involved, O(n) worst-case is unavoidable
  • "What if there are variables?" → Basic Calculator IV (#770) — extend to polynomial expressions
  • "Handle floating-point output?" → Change int() cast to keep as float; adjust return type

7 Summary & Key Takeaways ▾
Stack stores pending addends — sum it at the end to resolve all + and −
High-precedence * and / are resolved immediately by pop-compute-push
Low-precedence + and − are deferred by pushing ±num onto the stack
Sentinel op='+' eliminates special-casing for the first number
Always act on the PREVIOUS operator, not the current one (you need the next number first)
Use int(a/b) not a//b — Python floor division gives wrong sign for negative inputs
Handle the last number with the i==len(s)-1 condition — no trailing operator fires it
