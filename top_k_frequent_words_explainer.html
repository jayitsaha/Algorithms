LeetCode #692

Top K Frequent Words

Given an array of strings words and an integer k, return the k most frequent words. The answer should be sorted by frequency from highest to lowest. Words with the same frequency should be sorted alphabetically.

Medium Sorting Bucket by Frequency Time: O(n + m log m) Space: O(n)
1 Understanding the Problem ▾

You have a list of words — some may repeat. Find the k words that appear most often. Return them sorted by frequency, highest first. If two words share the same frequency, the one that comes earlier in the alphabet wins.

💡 Key Insight

This is a two-phase problem. Phase 1 (count): tally each word's frequency with a hash map — O(n). Phase 2 (select): pick the top k by frequency, with alphabetical tie-breaking. The clever trick in the optimal approach is using bucket sort: instead of sorting all words against each other, group them by frequency and sweep buckets from highest to lowest.

✓ Example: k = 2

words = ["i","love","leetcode",
         "i","love","coding"]
i→2, love→2, leetcode→1, coding→1
Top 2: ["i", "love"]

⚠ The Tie-Break Rule

"i" and "love" both appear 2x.
"i" < "love" alphabetically.
→ "i" ranks before "love".
Not just frequency — also alpha!

Edge cases: k = 1 (return single most frequent), all words identical (only one distinct word), all words unique (tie-break dominates completely), k equals the number of distinct words (return everything sorted).

2 The Algorithm: Bucket Sort by Frequency ▾

We avoid comparing words across different frequencies entirely. Instead, we physically separate them by frequency using an array of lists (buckets), then sweep from the highest-frequency bucket down.

1 Count frequencies (O(n)): Iterate through words. For each word, increment freq[word]. After this pass, we know exactly how many times each distinct word appears.
2 Build frequency buckets (O(m)): Create an array of n+1 empty lists — index = frequency. For each word, place it in buckets[freq[word]]. A word appearing 3 times goes into buckets[3]. Words sharing the same frequency share the same bucket.
3 Sort each bucket alphabetically: Within a bucket, all words have identical frequency — alphabetical order must distinguish them. Sort each non-empty bucket's word list.
4 Harvest from highest bucket down: Sweep buckets[n] → buckets[1]. From each bucket (already sorted alphabetically), take words one by one and append to result. Stop the moment we have k words.

🔑 Simple Alternative (Good Interview Opening)

sorted(freq.keys(), key=lambda w: (-freq[w], w))[:k] — sort all distinct words using a tuple key: first by negative frequency (so highest first), then by word itself (alphabetical for ties). Slice first k. Same O(m log m) worst case but dead-simple to write and explain. Propose this first; then offer to optimize.

3 Why It Works: The Invariant ▾
🔐 The Invariant

After building buckets, every word in buckets[f] appears exactly f times. When we sweep from f = n down to f = 1, every word collected from bucket f genuinely appears more often than any word collected later from a lower bucket. Within a bucket, alphabetical sort guarantees tie-breaking. Together, these two orderings produce the correct final order.

Why bucket sort handles tie-breaking correctly: Words in the same bucket have identical frequency. A global sort with a compound key like (-freq, word) compares all words against all others — expensive and harder to reason about. With buckets, frequency ordering is structural (implicit in which bucket we're visiting) and alphabetical ordering is local (sort within each bucket independently). Clean separation of concerns.

Why array size must be n+1: The maximum frequency any word can achieve is n (if all n input words are identical). Valid bucket indices are 0 through n — so we need n+1 slots. Index 0 is always empty. Allocating only n slots causes an out-of-bounds error for that edge case.

⚠️ Common Mistake: Wrong Array Size

buckets = [[] for _ in range(n)] fails if all words are the same, because we'd try to access buckets[n] which doesn't exist. Always range(n + 1). Also: don't return from the outer loop — use early exit inside the inner loop when len(result) == k.

Correctness proof sketch: By construction, word A in bucket f_A and word B in bucket f_B with f_A > f_B → A appears before B in the result. For A and B in the same bucket (same frequency), sorted() puts the alphabetically-earlier one first. This matches the problem's required ordering exactly.

4 The Code (Annotated) ▾

Solution 1 — Bucket Sort by Frequency (Optimal, Interview Pick)

def topKFrequent(words, k):
freq = {} ← hash map: word → count
for w in words: ← O(n) counting pass over all words
freq[w] = freq.get(w, 0) + 1 ← get defaults to 0 for new words
n = len(words)
buckets = [[] for _ in range(n + 1)] ← n+1 slots; index = frequency; index 0 unused
for word, count in freq.items(): ← place each distinct word in its freq bucket
buckets[count].append(word)
result = []
for i in range(n, 0, -1): ← sweep from highest freq → lowest
for word in sorted(buckets[i]): ← sort bucket alphabetically; ties resolved here
result.append(word)
if len(result) == k: ← early exit: collected enough words
return result
return result ← fallback (reached only if k = distinct words)
⚠️ Why sort inside the inner loop, not outside?

We call sorted(buckets[i]) only on words that share the same frequency. We never compare words from different buckets — their relative order is determined by the bucket index, not by string comparison. This is both correct and efficient: alphabetical sort only happens within each frequency group.

Solution 2 — Custom Sort Key (Clean, O(m log m))

from collections import Counter
def topKFrequent(words, k):
freq = Counter(words) ← Pythonic O(n) count
return sorted(
freq.keys(), ← sort only distinct words (m ≤ n)
key=lambda w: (-freq[w], w) ← (-freq, word): highest freq first, alpha for ties
)[:k] ← slice first k; Python tuple comparison handles both criteria

Solution 3 — Min-Heap of Size k (O(n + m log k), best when k small)

import heapq
from collections import Counter
def topKFrequent(words, k):
freq = Counter(words)
heap = [(-cnt, word) ← negate count so min-heap = max-freq order
for word, cnt in freq.items()] ← word string provides alpha tie-break naturally
heapq.heapify(heap) ← O(m) heapify; much faster than O(m log m) sort
return [heapq.heappop(heap)[1] for _ in range(k)]← pop k times: O(k log m)
💡 Complexity Summary

Bucket Sort (n = words, m = distinct words): O(n) count + O(m) place + O(m log m) sort buckets + O(k) harvest = O(n + m log m). Space O(n) for buckets.
Custom Sort Key: O(n) count + O(m log m) sort = O(n + m log m). Space O(m).
Min-Heap: O(n) count + O(m) heapify + O(k log m) pop = O(n + m + k log m). Best when k ≪ m.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Decompose into Two Sub-Problems

"I see this as two distinct steps: (1) count frequencies — easy with a hash map, O(n). (2) Select and order the top k — this is where the interesting design choice lives."

Step 2: State the Simple Approach First

"The simplest correct approach: count with Counter, then sorted(freq, key=lambda w: (-freq[w], w))[:k]. The compound tuple key handles both frequency ordering and alphabetical tie-breaking. O(m log m). I'd code this first."

Step 3: Propose Bucket Optimization

"For a cleaner structural solution: bucket sort. Words in the same bucket have identical frequency — I only sort within each bucket (for alphabetical order). Then sweep high-to-low. This makes frequency ordering implicit rather than part of the comparator."

Step 4: Mention Heap for Streaming / Large Input

"If k is very small compared to distinct words, a min-heap of size k achieves O(n + m log k) — better asymptotically. Also useful for streaming word counts."

🗣️ What to Say to the Interviewer

"Step 1: count word frequencies — O(n) with a hash map.
Step 2: I need top k by frequency; ties go alphabetically.
Simple approach: sort distinct words by (-freq, word) as key, slice first k. O(m log m).
Optimal: bucket sort. Create n+1 buckets indexed by count. Place each word in its bucket. Sort each bucket alphabetically. Sweep high-to-low, collect k words. O(n + m log m).
Complexity: O(n + m log m) time, O(n) space."
🔮 Likely Follow-up Questions
  • "What if there are numbers instead of words?" → No alphabetical tie-break; use numeric ordering or just sort by freq
  • "Top K Frequent Elements (numbers, no tie-break)?" → Simpler bucket sort — no per-bucket sort needed (#347)
  • "Handle a continuous stream of words?" → Maintain freq map + sorted structure; heapq for incremental updates
  • "Return words in original insertion order for ties?" → Track insertion index and use it as third sort key

7 Summary & Key Takeaways ▾
Count first (O(n)), select second — always two phases for "top K" problems
Bucket sort makes frequency ordering structural: no comparison needed across different frequencies
Alphabetical tie-breaking = sort within each bucket independently
Bucket array must be size n+1 — a word can appear at most n times
The custom sort key (-freq, word) is the cleanest one-liner for interviews
Min-heap of size k wins when k is tiny (streaming scenarios)
ApproachTimeSpaceNotes

Bucket Sort ✓ O(n + m log m) O(n) Optimal; structural freq order; interview pick

🎯 When to Recognize the Bucket-by-Frequency Pattern
  • "Top K most frequent X" → frequency count + bucket sort or heap
  • "Sort by frequency, tie-break by Y" → compound sort key or bucket + inner sort by Y
  • Frequency values bounded by n → bucket array of size n+1
  • "Kth most frequent" (single result) → use quickselect for O(n) or heap for O(n log k)
