LeetCode #496

Next Greater Element I

For each element in nums1 (a subset of nums2), find the first element to its right in nums2 that is strictly greater. Return -1 if none exists.

Easy Stack & Queue Monotonic Stack Time: O(m+n) Space: O(n)
1 Understanding the Problem ▾

You have two arrays. nums1 is a subset of nums2 — every value in nums1 appears somewhere in nums2. For each value in nums1, find its next greater element in nums2: look at where that value sits in nums2, then scan rightward — the first value you encounter that is strictly larger is the answer. If you reach the end of nums2 without finding one, report -1.

💡 Key Insight

nums1 is just a list of queries. The real work is computing the next greater element for every value in nums2. Once we have that precomputed in a hash map, each query in nums1 is answered with a single O(1) lookup. This separation of concerns is the core breakthrough.

✓ Example: nums1=[4,1,2], nums2=[1,3,4,2]

4 is at index 2 in nums2
  To its right: [2] — all smaller → -1
1 is at index 0 in nums2
  To its right: [3,4,2] — first >1 is 3 → 3
2 is at index 3 in nums2
  To its right: [] — nothing → -1
Result: [-1, 3, -1]

✗ Naive O(m·n) — works but too slow

For each x in nums1:
  Scan nums2 to find x → O(n)
  Scan right for next greater → O(n)
Total: O(m·n)
Fails on large inputs (m,n up to 1000)

Edge cases: all elements in nums2 are decreasing (every answer is -1), the queried element is the last in nums2 (answer is -1), nums1 has one element, nums2 has two elements. All elements are distinct (the problem guarantees uniqueness — crucial for using values as hash map keys).

2 The Algorithm: Monotonic Decreasing Stack ▾

Imagine you are sorting a queue of job applicants from tallest to shortest. As you process each new person, anyone shorter than them who was waiting behind gets an immediate answer: "this person is taller than you." The stack works the same way — it holds a decreasing sequence of "pending" values, each waiting for a larger number to resolve their answer.

1 Initialize: Empty nge_map (will store value → next greater answer) and empty stack. The stack maintains values in decreasing order from bottom to top.
2 Scan nums2 left-to-right: For each element x, enter a while-loop: while the stack is non-empty AND the stack's top value is less than x, pop the top and record nge_map[popped] = x. The current element x is the first element to the right of popped that is larger — its answer is found.
3 Push x: After the while-loop exits (stack is empty or top ≥ x), push x onto the stack. Its next-greater answer is still unknown; we'll find it when a larger element arrives.
4 Finalize remaining: After scanning all of nums2, any value still on the stack never saw a larger element to its right → nge_map[v] = -1 for each.
5 Answer queries: For each element in nums1, return nge_map[x] — O(1) per lookup. Total time: O(m+n).

🔑 Algorithm Deep-Dive: Monotonic Stack

Origin: A fundamental data structure technique in competitive programming and algorithm design. Solves "next greater/smaller element" problems in O(n) instead of O(n²).

Core Invariant: At all times, the stack holds values in strictly decreasing order from bottom to top. Every value on the stack is still "searching" for a larger element to its right.

Why O(n): Each element is pushed exactly once and popped at most once. Total push + pop operations ≤ 2n.

Variants: Monotonic increasing stack → finds next smaller element. Circular array → iterate twice. Store indices instead of values when there are duplicates.

3 Why It Works: The Invariant ▾
🔐 The Stack Invariant

At any point during the scan, every value in the stack satisfies: no element has appeared to its right in nums2 (so far) that is strictly greater than it. The stack holds these "unresolved" values in decreasing order, guaranteeing efficient resolution later.

Why popping gives the correct (first) answer: When we see element x and pop value v from the top (because x > v), x is the first element after v's position in nums2 that is greater. Why "first"? Because if any element between v and x were greater than v, it would have already popped v when it was processed. Since v is still on the stack, every element processed between v and x was ≤ v. So x is indeed the next greater element for v.

Why the remaining stack → -1: After scanning all of nums2, any value still on the stack was processed, and nothing larger appeared to its right. That's exactly the definition of "no next greater element" → -1.

Why the stack remains decreasing: We only push after the while-loop ends (meaning the top is ≥ x). So x ≤ top — preserving the decreasing invariant.

⚠️ Common Mistake: Storing Values vs. Indices

This problem guarantees all values in nums2 are distinct (see constraints). So using values as keys is safe. If the problem allowed duplicates, you'd need to store indices on the stack and use indices as keys, then map back to values. Always verify uniqueness before choosing.

⚠️ Common Mistake: Using <= Instead of <

The condition is stack[-1] < x (strictly less than). Using <= would pop equal elements and record x as their "next greater" — but an equal element is NOT greater. This produces wrong answers for arrays with repeated values in nums2 (and even though this problem has all-distinct, forming good habits matters).

4 The Code (Annotated) ▾

Solution 1 — Monotonic Stack + Hash Map (Optimal · Interview Pick)

def nextGreaterElement(nums1, nums2):
nge_map = {} ← Hash map: value in nums2 → its next greater element (or -1)
stack = [] ← Monotonic decreasing stack; stores unresolved values
for x in nums2: ← Process nums2 left to right
while stack and stack[-1] < x:← While top of stack is smaller than current x
nge_map[stack.pop()] = x ← Pop it; x is its next greater. Record in map.
stack.append(x) ← x still awaits its answer; push onto stack
for v in stack: ← Remaining stack = never found a greater element
nge_map[v] = -1 ← Map each to -1
return [nge_map[x] for x in nums1]← Answer each query via O(1) hash lookup
⚠️ The while-loop vs. if-statement Difference

Notice we use while, not if. A single incoming element x might be larger than multiple pending elements on the stack — one pop would only resolve the top, leaving others incorrectly unresolved. The while-loop ensures we pop everything that x resolves in one sweep.

Solution 2 — Brute Force (O(m·n) time, simple fallback)

def nextGreaterElement(nums1, nums2):
idx_map = {v: i for i, v in enumerate(nums2)} ← Precompute position of each value in nums2
result = []
for x in nums1: ← For each query
pos = idx_map[x] ← Where is x in nums2?
ans = -1
for j in range(pos + 1, len(nums2)): ← Scan right
if nums2[j] > x:
ans = nums2[j]; break ← Found next greater
result.append(ans)
return result ← O(m·n) — mention this then optimize to stack
💡 Complexity Summary

Monotonic Stack: Time O(m+n) — each of n elements pushed once, popped at most once; m lookups. Space O(n) for stack + map.
Brute Force: Time O(m·n) — for each of m queries, scan up to n elements. Space O(n) for the index map (or O(1) with linear search).
Constraints here: m, n ≤ 1000, so brute force is fine — but always demonstrate the optimal approach in interviews.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Decompose the Problem

"I notice nums1 is a subset of nums2 — it's really just a set of queries. If I precompute the next greater element for every number in nums2, then each query becomes an O(1) lookup. Let me separate the precomputation from the querying."

Step 2: Recognize the Pattern

"The precomputation is: for each element in nums2, find the first element to its right that is greater. This is the classic 'next greater element' problem — I know this can be solved in O(n) with a monotonic decreasing stack."

Step 3: Walk Through the Invariant

"The stack maintains a decreasing sequence. When I see element x, I pop everything from the stack that is smaller than x — x is their next greater answer. Then push x. Anything left after the full scan gets -1."

Step 4: Clarify the Uniqueness Constraint

"I'll note that all values are distinct — so I can safely use values as map keys. If duplicates were possible, I'd store indices on the stack instead."

🗣️ What to Say to the Interviewer

"Step 1: nums1 is just queries — let me precompute next-greater for all of nums2.
Step 2: I'll use a monotonic decreasing stack. I scan nums2 left-to-right.
Step 3: For each x, I pop anything from the stack that is smaller than x — x is their answer.
Step 4: Then push x (its answer is still unknown).
Step 5: After the scan, whatever's still in the stack maps to -1.
Step 6: Answer nums1 queries from the hash map — O(1) each.
Total: O(n) precompute + O(m) queries = O(m+n). Space O(n) for stack and map."
