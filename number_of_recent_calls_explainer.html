LeetCode #933

Number of Recent Calls

Design a class to count the number of requests in the past 3000 milliseconds for each new ping.

Easy Queues Sliding Window Queue Time: O(1) amortized Space: O(W)
1 Understanding the Problem ▾

You are building a request-rate tracker. A RecentCounter object receives ping(t) calls where t is a timestamp in milliseconds. Each call must return the number of calls (including this one) made in the time range [t − 3000, t] — that is, within the past 3000 milliseconds, inclusive on both ends.

💡 Key Insight

The window [t − 3000, t] slides rightward with every new call. A request from far in the past will eventually fall behind the left edge and never contribute to any future answer. We can safely evict it. A FIFO queue — where we push new calls to the back and pop stale ones from the front — models this perfectly.

Let's trace through the four example calls:

ping(1)
Window [1−3000, 1] = [−2999, 1] · queue: [1]
→ 1
ping(100)
Window [−2900, 100] · queue: [1, 100]
→ 2
ping(3001)
Window [1, 3001] · queue: [1, 100, 3001] (1 is exactly on edge!)
→ 3
ping(3002)
Window [2, 3002] · evict 1 (1 < 2) · queue: [100, 3001, 3002]
→ 3

Critical constraint: Calls arrive with strictly increasing timestamps. This means the queue is always sorted — the oldest call is always at the front. No sorting or binary search is required — just evict from the front.

⚠️ Off-by-One Alert: Inclusive Boundary

The window is [t − 3000, t] — both ends are inclusive. When ping(3001) arrives, the left boundary is 3001 − 3000 = 1, and a request at timestamp 1 is still valid. The eviction condition is front < t − 3000 (strictly less than), not ≤.

2 The Algorithm: Sliding Window Queue ▾

We maintain a queue of timestamps — only those still inside the current window [t − 3000, t]. On each ping(t):

1 Enqueue the new timestamp t — every new request is trivially inside its own window (it is the right edge). Append t to the back of the queue first, before any eviction.
2 Evict stale timestamps from the front — while queue[0] < t − 3000, the oldest request is outside the left edge of the window and will never be counted in any future answer. Pop it with popleft().
3 Return the queue length — every element remaining in the queue satisfies t − 3000 ≤ q_i ≤ t. The count is exactly len(queue).
✓ Sliding Queue (Optimal)

O(1) amortized per call
Each item: 1 enqueue + 1 dequeue
Space: O(W) ≤ O(3001)
No unbounded growth

✗ Brute Force (All + Filter)

O(n) per call (n = total calls)
Keep every timestamp ever
Space: O(n) grows forever
Fails for long-running systems

🔑 Why collections.deque, Not list?

Python's collections.deque provides O(1) popleft(). A plain list.pop(0) is O(n) — it shifts all remaining elements left in memory. For any sliding window where you evict from the front, always use deque. The difference matters at scale: if the window holds 3000 elements and every call evicts one, that's 3000 element copies per call with a list.

3 Why It Works: The Invariant ▾
🔐 The Invariant

After every ping(t) call completes, the queue contains exactly the set of timestamps q_i such that t − 3000 ≤ q_i ≤ t. The queue size equals the answer.

Proof that the invariant holds:

  • t itself is included: We always append t before evicting. Since t − 3000 ≤ t ≤ t trivially, t is always in the window.
  • Stale elements are removed: The while-loop evicts every q_i < t − 3000. After eviction, all remaining q_i ≥ t − 3000.
  • All remaining elements have q_i ≤ t: They were appended on a previous or current call with timestamp ≤ t (since timestamps are monotone increasing).
  • Nothing valid is over-evicted: We stop as soon as queue[0] ≥ t − 3000. The front is the smallest element (queue is always sorted due to monotone insertion), so all remaining elements also satisfy this bound.

Amortized O(1) analysis: Over n calls, each timestamp is enqueued exactly once and dequeued at most once — so total work is O(n). Per call: O(1) amortized. A single call might evict O(W) elements, but W ≤ 3000 is a constant upper bound; and across n calls, total evictions ≤ n.

⚠️ Why Append Before Evicting?

If we tried to evict first, we'd need to guard against accessing self.q[0] on an empty queue (which raises IndexError). By appending t first, we guarantee the queue is non-empty when we check the front — since we just added t and t is never evicted (it satisfies the window condition). This is a subtle but important ordering choice.

4 The Code (Annotated) ▾

Solution 1 — Sliding Window Queue with deque (Optimal, O(1) amortized) — Interview Pick

from collections import deque ← deque has O(1) popleft(); list.pop(0) is O(n)
class RecentCounter:
def __init__(self): ← Constructor: O(1)
self.q = deque() ← Queue holds timestamps of all in-window requests
def ping(self, t: int) -> int: ← t is always strictly greater than all prior t's
self.q.append(t) ← Add t to back first — ensures queue never empty
while self.q[0] < t - 3000: ← Front = oldest. If outside window, evict
self.q.popleft() ← O(1) — this is why deque beats list
return len(self.q) ← All remaining are in [t-3000, t] — count them
⚠️ Common Mistake: Using list.pop(0)

Writing self.q = [] and then self.q.pop(0) looks correct and produces the right answer — but it's O(n) per eviction because Python shifts all elements left after removing index 0. With deque, popleft() is a O(1) pointer adjustment. Always mention this distinction in interviews.

Solution 2 — Brute Force: Keep All Timestamps

class RecentCounter:
def __init__(self):
self.calls = [] ← Grow forever — never discard old timestamps
def ping(self, t: int) -> int:
self.calls.append(t) ← Add this call
return sum(1 for c in self.calls if c >= t - 3000) ← O(n) filter each time
💡 Complexity Summary

Sliding Window Queue: Time O(1) amortized per call, Space O(W) ≤ O(3001) = O(1)
Brute Force: Time O(n) per call (n = total calls so far), Space O(n) growing unboundedly
Note on space: Since timestamps within a 3000ms window can be at most 3000ms apart, the queue holds at most 3001 elements — a hard constant upper bound. In practice, space is O(W) where W = window size in ms.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Spot the Sliding Window

"Within the past 3000 milliseconds" is the classic sliding window signal. As time advances, old data expires. Think: what can I throw away early? Answer: anything with timestamp < t − 3000.

Step 2: Monotone Timestamps → FIFO Queue

Since calls arrive in strictly increasing order, the oldest call is always at the front of any ordered collection. A simple FIFO queue maintains this — we never need to search or sort. Eviction is always from the front.

Step 3: Propose Brute Force First, Then Optimize

"I could store all timestamps and filter with sum(1 for c in calls if c >= t-3000) — O(n) per call. But I notice old requests become irrelevant. I'll evict them immediately with a queue — O(1) amortized."

Step 4: Justify deque Explicitly

Don't just say "I'll use a queue." Say: "I'll use collections.deque specifically because popleft() is O(1). list.pop(0) is O(n) — it would negate the optimization."

🗣️ What to Say to the Interviewer

"This is a sliding window problem: the window [t-3000, t] shifts right with each call.
I'll use a deque to store in-window timestamps.
On ping(t): append t first (guarantees non-empty queue), then evict stale fronts
while q[0] < t-3000: popleft().
Return len(q) — all remaining are in [t-3000, t].
Time: O(1) amortized (each timestamp enqueued once, dequeued at most once).

🔮 Likely Follow-up Questions
  • "What if calls don't arrive in order?" → Need a sorted structure (SortedList or heap); binary search the eviction boundary.
  • "Rate limiter: reject if > K requests in window?" → Same queue; before returning, check if len(q) > K: return reject.
  • "Sliding window max, not count?" → Monotonic deque (LC 239) — evict elements that are smaller than the new one.
  • "What's the worst-case per call?" → O(W) = O(3000) evictions, but amortized over n calls it's O(1) per call.
