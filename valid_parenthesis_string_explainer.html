LeetCode #678

Valid Parenthesis String

Given a string s containing '(', ')' and '*', return true if s is a valid parenthesis string. '*' can be '(', ')' or empty.

Medium Stacks Two Counters (Lo/Hi Range) Time: O(n) Space: O(1)
1 Understanding the Problem ▾

You have a string of three character types: '(', ')', and '*'. The string is valid if every '(' has a matching ')' that appears later in the string. The twist: '*' is a joker that can act as '(', ')', or the empty string — whichever choice makes the string valid.

💡 Key Insight

Don't try to decide what each * becomes. Instead, track a range [lo, hi] representing all possible open-parenthesis counts reachable at the current position. A valid assignment exists if and only if lo == 0 at the end (meaning zero is in the range).

✓ Valid: s = "(*)"

'(' → 1 open paren
'*' → treat as '' (empty)
')' → closes the '(' → 0
Final lo=0 → True

✗ Invalid: s = "((*"

'(' → lo=1, hi=1
'(' → lo=2, hi=2
'*' → lo=1, hi=3
Final lo=1 ≠ 0 → False

Why wildcards make this hard: With just ( and ), a single counter suffices. With *, exponentially many assignments are possible. Trying each one is O(3ⁿ). We need a smarter representation that encodes the full space of outcomes at once.

Edge cases: Empty string (valid — lo=hi=0 → return True). All stars (valid for any length — stars can all be empty). Leading ')' (hi drops below 0 → immediate False). Unclosed '(' after full scan (lo > 0 → False).

2 The Algorithm: Lo/Hi Greedy Range Tracking ▾

Maintain two counters — lo (the minimum achievable open count) and hi (the maximum achievable open count). Together they define the full set of possible states without enumerating them explicitly.

1 Initialize lo = hi = 0. Before processing any character, exactly 0 open parentheses are outstanding. Both the pessimistic and optimistic counts start at 0.
2 '(' encountered: This always opens a paren — no ambiguity. Both lo and hi increase by 1. Every possible assignment goes up by 1.
3 ')' encountered: This always closes a paren — no ambiguity. Both lo and hi decrease by 1. Every possible assignment goes down by 1.
4 '*' encountered: Three possible interpretations. Treating it as ')' lowers the count → lo decreases by 1. Treating it as '(' raises the count → hi increases by 1. Treating it as '' leaves count unchanged. Net: lo−1, hi+1.
5 After each update: clamp lo to 0. lo can never be logically negative (open count must be ≥ 0). If lo would go negative, the * can simply be treated as empty instead of ')'.
6 After each update: fail if hi < 0. hi is the most optimistic open count. If it drops below 0, even the best assignment has too many ')' — the string is provably broken. Return False immediately.
7 Final check: return lo == 0. lo == 0 means there exists at least one assignment where all opens are matched (0 unmatched opens). lo > 0 means even the most pessimistic scenario has leftover opens.

🔑 Alternative: Two-Stack Approach (Mention First)

Use one stack for '(' indices, another for '*' indices. When ')' arrives, match with the nearest '(' first, then a '*'. Afterward, match any remaining '(' with a '*' to its right (check indices). This is O(n) time and O(n) space — intuitive but uses more memory. Propose it first in an interview, then optimize to O(1) space with the lo/hi approach.

3 Why It Works: The Invariant ▾
🔐 The Invariant

After processing each character, lo equals the minimum unmatched open-paren count achievable across all valid wildcard assignments, and hi equals the maximum. Every integer in [lo, hi] of the same parity as lo is achievable. If lo = 0 is in this range at the end, a valid assignment exists.

Why the clamp (lo = max(lo, 0)) is correct: Suppose a * would drive lo to −1. That scenario requires treating the * as a ')' that closes a non-existent open paren — which is invalid. We can always substitute '' (empty) instead, giving lo = 0. So −1 is never actually reachable; the clamp reflects this reality.

Why hi < 0 means hard failure: hi is the count achieved when every * is greedily treated as '('. If even this most optimistic scenario goes negative, we have a ')' with no possible match — the string is broken regardless of wildcard choices. There's no way to recover.

Why lo == 0 (not lo == hi == 0) is the final check: We only need one valid assignment to exist. lo == 0 means the most pessimistic valid assignment ends with 0 open parens — perfect balance. hi may still be positive (if wildcards could have been used as extra '('), but we don't need that scenario to be valid.

⚠️ Common Trap: Checking hi < 0 before the clamp

The order of operations matters: first update lo and hi, then check if hi < 0 (early exit), then clamp lo. Some implementations accidentally clamp first and then check hi — this can mask a failure where the clamp hides a negative hi. Always: update → check hi → clamp lo.

⚠️ Common Trap: Forgetting to clamp lo

Without the clamp, lo can become negative when a * is processed. Then at the end, lo < 0 fails the lo == 0 test — incorrectly returning False for valid inputs like "*" (which should be True: treat * as empty). The clamp enforces that lo never goes below 0, correctly modeling the empty-string interpretation of *.

4 The Code (Annotated) ▾

Solution 1 — Lo/Hi Greedy Range (Interview Pick, O(n) time, O(1) space)

def checkValidString(s: str) -> bool:
lo = hi = 0 ← lo = min possible open count; hi = max possible open count
for c in s:
if c == '(':
lo += 1 ← '(' is unambiguous: always adds one open paren
hi += 1
elif c == ')':
lo -= 1 ← ')' is unambiguous: always closes one open paren
hi -= 1
else: ← '*' is ambiguous: widens the range of possibilities
lo -= 1 ← worst case: treat '*' as ')' → lowest possible count
hi += 1 ← best case: treat '*' as '(' → highest possible count
if hi < 0: ← even best case is broken: too many ')' → impossible
return False
lo = max(lo, 0) ← open count can't go below 0; '*' can always be ''
return lo == 0 ← some assignment achieves exactly 0 unmatched opens
⚠️ Why check hi < 0 BEFORE clamping lo?

After processing ')', both lo and hi decrease. If hi goes negative, the string is already broken — we should fail immediately, before the clamp could accidentally hide this. Clamping lo first doesn't affect hi, but it's cleaner to check hi first as an early-exit condition. The code above checks hi → then clamps lo: always in this order.

Solution 2 — Two Stacks (O(n) time, O(n) space)

def checkValidString(s: str) -> bool:
open_st = [] ← indices of unmatched '(' characters
star_st = [] ← indices of '*' characters (potential wildcards)
for i, c in enumerate(s):
if c == '(': open_st.append(i)
elif c == '*': star_st.append(i)
else: ← ')': try to match nearest '(' first
if open_st: open_st.pop()
elif star_st: star_st.pop() ← fallback: use '*' as '(' to match this ')'
else: return False ← no way to match this ')' → invalid
while open_st and star_st: ← match remaining '(' with '*' to their right
if open_st.pop() > star_st.pop(): return False ← '*' must appear AFTER '('
return not open_st ← any unmatched '(' still left?
💡 Complexity Summary

Lo/Hi Greedy (Interview Pick): Time O(n), Space O(1) — optimal on both dimensions
Two Stacks: Time O(n), Space O(n) — more intuitive, great first mention
DP (memoized recursion): Time O(n²), Space O(n²) — state = (index, open_count) — feasible but skip in interviews

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Base Pattern

This is a parentheses validity problem. Without wildcards, the solution is a simple counter: +1 for '(', −1 for ')', fail if counter goes negative, return True if counter == 0. Wildcards add ambiguity — that's the signal you need something smarter than a single counter.

Step 2: Propose the Stack Solution First

"I'll keep a stack of unmatched '(' positions and a stack of '*' positions. When I see ')', I match it with the nearest '(' first; if none, use a '*'. After scanning, any remaining '(' need a '*' to their right." O(n) time, O(n) space — solid, correct, interviewer-friendly.

Step 3: Offer to Optimize Space

"I can reduce space to O(1) by not tracking individual positions. Instead of tracking all possible states, I track the range of possible open-paren counts: lo (pessimistic) and hi (optimistic). '*' widens the range. If hi goes negative, fail. At the end, return lo == 0."

Step 4: Trace Through a Key Example

Always trace "(*)": [0,0] → '(' → [1,1] → '*' → [0,2] (lo=max(1−1,0)=0; hi=1+1=2) → ')' → [−1,1] → clamp → [0,1]. End: lo=0 → True. This trace hits the clamping, the range expansion, and the final check all in 3 characters.

🗣️ What to Say to the Interviewer

"Without wildcards this is a simple counter problem. With '*', I need to represent uncertainty.
My first approach: two stacks — one for '(' indices, one for '*' indices.
Match ')' greedily with '(' first, then '*'. At the end, match remaining '(' with '*' to their right.
O(n) time and space. [Implement this, then say:]

To reduce space to O(1): track a range [lo, hi] of possible open counts.
'(' increments both. ')' decrements both. '*' widens: lo--, hi++.
Clamp lo at 0 — open count can't go negative. If hi goes below 0, return False.
At the end, return lo == 0 — some valid assignment reaches 0 unmatched opens."
🔮 Likely Follow-up Questions
  • "What if '*' can only be '(' or ')'?" → lo/hi still works; just don't have the empty interpretation keeping lo at prior value
  • "Can you solve it recursively?" → Yes, with memoization on (index, open_count) — O(n²)
  • "Extend to multiple wildcard types?" → Generalize the range expansion rules per wildcard type
  • "Minimum insertions to make valid?" → Count unmatched lo and hi — related problem #1312
