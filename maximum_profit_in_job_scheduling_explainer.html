LeetCode #1235

Maximum Profit in Job Scheduling

Given n jobs with start time, end time, and profit — find the maximum profit you can earn by scheduling non-overlapping jobs.

Hard Dynamic Programming DP + Binary Search Time: O(n log n) Space: O(n)
1 Understanding the Problem ▾

Imagine you're a freelancer with a list of contract jobs. Each job has a start time, an end time, and a profit. You can only work on one job at a time — two jobs cannot overlap. Your goal: pick a subset of non-overlapping jobs that maximizes your total earnings.

💡 Key Insight

This is Weighted Interval Scheduling — the weighted version of the classic "maximum non-overlapping intervals" problem. When jobs have weights (profits), greedy strategies fail. We need DP: define subproblems over sorted jobs, and use binary search to jump to the "last compatible" job in O(log n) per step.

✓ Optimal Choice

Jobs: A[1→3, $50], B[2→5, $10]

✗ Why Greedy Fails

Then no room for D (overlaps C)
No room for B (overlaps C)
A fits! 50 + 70 = 120
Greedy works HERE but not always

Critical boundary rule: A job ending at time t and a job starting at time t are non-overlapping — you finish one gig and start the next immediately. Binary search must respect this: we want the last job with end ≤ start_of_current.

Edge cases: n=1 (answer = that profit), all jobs overlap (pick max-profit one), all non-overlapping (sum all), zero-profit jobs (skip naturally), very large profits with many short jobs.

2 The Algorithm: Sort + DP + Binary Search ▾

We sort jobs by end time, then define dp[i] = maximum profit achievable from the first i sorted jobs. For each new job i, we ask: skip it or take it? If we take it, we use binary search to instantly find the last job compatible with it.

1 Sort jobs by end time. This creates a canonical ordering where job i always ends before job i+1. It makes dp[i] meaningful: "best profit from the first i jobs" is well-defined because earlier-ending jobs could possibly run before later ones.
2 Initialize dp[0] = 0. Before considering any job, profit is zero. The dp array is 1-indexed: dp[i] corresponds to the first i sorted jobs. dp[n] is our answer.
3 For each job i (1-indexed): Option SKIP: dp[i] = dp[i-1] (pretend job i doesn't exist). Option TAKE: find j = number of jobs ending ≤ start of job i (via binary search). Then dp[i] = dp[j] + profit[i]. Take the max.
4 Binary search for j. The sorted ends[] array is searched with bisect_right(ends, start_of_job_i). This returns the count of jobs whose end time is ≤ start_of_job_i — exactly the compatible jobs. dp[j] is the best we can earn from them.
5 Answer = dp[n]. After processing all n jobs in sorted order, dp[n] holds the global maximum profit.
📐 Recurrence Relations
dp[0] = 0 (base case)

dp[i] = max(dp[i-1], (skip job i)
        dp[last_compat(i)] + profit[i]) (take job i)

last_compat(i) = bisect_right(ends, start[i])
                  = # jobs with end ≤ start[i]
🔑 Why Binary Search Unlocks O(n log n)

Without binary search, finding "last compatible job" for each of n jobs takes O(n) — total O(n²). But since we sorted by end time, the ends[] array is sorted. bisect_right finds our answer in O(log n). Total: O(n log n) for sort + n × O(log n) for queries = O(n log n).

3 Why It Works: The DP Invariant ▾
🔐 The Invariant

At every step, dp[i] stores the exact maximum profit achievable using any non-overlapping subset of the first i jobs (sorted by end time). This invariant is maintained by construction because we process jobs in end-time order, ensuring that when we evaluate job i, all dp[0..i-1] are already correct.

Optimal Substructure: The optimal solution for n jobs either uses job n or not.

  • Exclude job n: optimal = dp[n-1] (don't change anything from n-1 jobs)
  • Include job n: optimal = dp[last_compat(n)] + profit[n]

Taking max of these two gives dp[n]. This decomposition is provably optimal: any subset either contains the last job (sorted by end) or it doesn't. No other cases.

Overlapping Subproblems: A pure recursive approach computing f(i) = max profit from job i onwards would recompute f(j) every time any job < i finds j as its next compatible job. Memoization (or tabulation, as we do here) ensures each state is solved once.

⚠️ Why Greedy Truly Fails on the General Case

Consider: Job A [1→10, $5], Job B [1→5, $4], Job C [5→10, $4]. Greedy by profit picks A ($5) and misses B+C ($8). Greedy by end time picks B then C ($8) — correct here but fails on other inputs. There is no single greedy strategy that works for all weighted interval scheduling cases. DP is required.

Boundary case (j=0): When no job ends before the current job starts, bisect_right returns 0. So dp[0] = 0 is used — meaning we add the current job's profit to zero previous profit. This cleanly handles the first job and any job starting before all previous jobs end.

4 The Code (Annotated) ▾

Solution 1 — Tabulation + Binary Search (Optimal · Interview Pick)

from bisect import bisect_right
def jobScheduling(startTime, endTime, profit):
jobs = sorted(zip(endTime, startTime, profit)) ← zip then sort by END time (index 0)
ends = [j[0] for j in jobs] ← sorted end-times; we binary search this
dp = [0] * (len(jobs) + 1) ← dp[0]=0; dp[i]=best profit from first i jobs
for i, (end, start, prof) in enumerate(jobs, 1):← 1-indexed; enumerate gives (end,start,prof)
j = bisect_right(ends, start) ← #jobs ending ≤ start → dp[j] = compatible best
take = dp[j] + prof ← profit if we include this job
skip = dp[i - 1] ← profit if we skip (carry forward)
dp[i] = max(take, skip) ← optimal substructure: take the better option
return dp[len(jobs)] ← dp[n] = answer over all jobs
⚠️ bisect_right vs bisect_left — This Matters!

Use bisect_right(ends, start), not bisect_left. When a job ends exactly at time t and another starts at t, they're non-overlapping (you can finish then start immediately). bisect_right includes end=t in the "compatible" count; bisect_left would exclude it, wrongly reducing your options.

Solution 2 — Memoization (Top-Down)

from bisect import bisect_left
from functools import lru_cache
def jobScheduling(startTime, endTime, profit):
jobs = sorted(zip(startTime, endTime, profit)) ← sort by START for top-down (process forward)
starts = [j[0] for j in jobs] ← binary search on start times
@lru_cache(maxsize=None)
def dp(i): ← max profit from jobs[i..n-1]
if i == len(jobs): return 0 ← base case: no jobs left → zero profit
skip = dp(i + 1) ← skip job i
next_j = bisect_left(starts, jobs[i][1]) ← first job starting >= end of job i
take = jobs[i][2] + dp(next_j) ← take job i + best from compatible suffix
return max(skip, take)
return dp(0) ← start from first job

Solution 3 — Brute Force O(n²) DP (No Binary Search)

def jobScheduling(startTime, endTime, profit):
jobs = sorted(zip(endTime, startTime, profit)) ← same sort
n = len(jobs)
dp = [0] * (n + 1)
for i, (end, start, prof) in enumerate(jobs, 1):
j = 0
for k in range(i - 1, -1, -1): ← O(n) linear scan for last compatible job
if jobs[k][0] <= start: ← jobs[k] ends before current starts
j = k + 1; break
dp[i] = max(dp[i-1], dp[j] + prof) ← O(n²) overall — mention then optimize
return dp[n]
💡 Complexity Summary

Tabulation + Binary Search: Time O(n log n). Space O(n).
Memoization + Binary Search: Time O(n log n). Space O(n) memo + O(n) call stack.
Brute Force DP: Time O(n²). Space O(n). Correct but TLEs on large inputs.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize Weighted Interval Scheduling

"This is non-overlapping intervals with weights. Classic greedy (earliest end time, highest profit) doesn't work when jobs have different values. That signals DP."

Step 2: Define the DP State

"Sort by end time. Define dp[i] = max profit from first i jobs. For each job: skip it (dp[i-1]) or take it (dp[j] + profit, where j = last compatible job index). dp[i] = max of both."

Step 3: Identify the O(n²) bottleneck and fix it

"Finding the last compatible job naively is O(n) per step = O(n²) total. But jobs are sorted by end time, so end-times form a sorted array. Binary search gives O(log n) per step = O(n log n) total."

Step 4: Nail the bisect detail

"Use bisect_right(ends, start_of_current_job) — this finds how many jobs end ≤ start, which is exactly our count. bisect_right correctly includes end=start (non-overlapping boundary) while bisect_left would wrongly exclude them."

🗣️ What to Say to the Interviewer

"This is weighted interval scheduling — greedy won't work because of different profit values.
I'll sort jobs by end time, then define dp[i] = max profit from first i sorted jobs.
For each job i: either skip it → dp[i] = dp[i-1],
or take it → need best profit from jobs ending before it starts.
Since end-times are sorted, I binary search: j = bisect_right(ends, start_i).
dp[j] = best profit from those compatible jobs. Take = dp[j] + profit_i.
dp[i] = max(take, skip). Final answer: dp[n].
Time O(n log n) for sort + n binary searches. Space O(n)."
🔮 Likely Follow-Up Questions
  • "At most K jobs?" → 2D DP: dp[i][k] = max profit using first i jobs and k selected.
  • "Return the actual jobs, not just profit?" → Reconstruct: if dp[i] != dp[i-1], job i was taken. Trace backwards.
  • "Can jobs have zero/negative profit?" → Handled naturally — max() always prefers skip over taking a zero-profit job.
  • "What if n is huge (10⁶)?" → Our O(n log n) solution handles it. Discuss memory: dp array is O(n).

7 Summary & Key Takeaways ▾
Sort by END time — this is the foundational ordering that makes the DP valid and the binary search meaningful.
dp[i] = max profit from first i sorted jobs. Two choices per job: skip (dp[i-1]) or take (dp[j] + profit).
Binary search on the sorted end-times array with bisect_right finds the last compatible job in O(log n), lifting the total to O(n log n).
Use bisect_right so that end=t and start=t jobs are treated as non-overlapping. This is a subtle but testable detail.
This generalizes to any Weighted Interval Scheduling: non-overlapping selections with weights/profits always call for Sort + DP + Binary Search.
🎯 When to Use This Pattern
  • Problems with intervals that have weights/profits
  • You must pick a non-overlapping subset to maximize total weight
  • Keywords: schedule, maximize profit, non-overlapping intervals, weighted intervals
  • When you see "take or skip" + sorted input → think DP + binary search
