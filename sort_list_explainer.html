LeetCode #148

Sort List

Given the head of a linked list, return the list sorted in ascending order in O(n log n) time and O(1) space.

Medium Sorting Merge Sort on Linked List Time: O(n log n) Space: O(log n)
1 Understanding the Problem ▾

You receive the head of a singly-linked list — a chain of nodes where each node holds an integer value and a pointer to the next node. Rearrange the nodes so their values are in non-decreasing order and return the new head.

💡 Key Insight

Linked lists are the natural home of merge sort. Splitting costs O(n) via the fast/slow pointer trick. Merging two sorted linked lists needs only pointer rewiring — no shifting, no indexing. Every other O(n log n) sort (quicksort, heapsort) requires random access, which linked lists don't support efficiently.

Input (unsorted)

4 → 2 → 1 → 3 → NULL
Arbitrary node order

Output (sorted ascending)

1 → 2 → 3 → 4 → NULL
Same nodes, rewired

Why not quicksort or heapsort? Both require O(1) random access by index. A linked list takes O(k) to reach index k. Quicksort's pivot swap becomes O(n) per element → O(n²) overall. Merge sort only needs sequential traversal — exactly what linked lists provide natively.

Edge cases: empty list (return None), single-node list (already sorted), two-node list (tests the split boundary), all equal values, already-sorted input, reverse-sorted input.

2 The Algorithm: Top-Down Merge Sort ▾

Merge sort has two phases: Split (find midpoint, divide into two halves) and Merge (combine two sorted halves into one sorted result). The recursion applies these at every level of the call tree.

1 Base case: If the list has 0 or 1 nodes, it's already sorted — return immediately. This terminates the recursion.
2 Find midpoint (Fast/Slow pointers): slow = head, fast = head.next. Each iteration: slow moves 1 step, fast moves 2. When fast can't advance further, slow is at the end of the left half. Execute slow.next = None to cut the chain into two independent sub-lists.
3 Recurse: Call sortList(head) on the left half and sortList(mid) on the right half. Each returns a fully sorted sub-list.
4 Merge: Create a dummy sentinel node. Repeatedly compare the front values of left and right, pick the smaller, attach to result, advance that pointer. When one list empties, attach the remainder of the other. Return dummy.next.
sortList([4, 2, 1, 3])
  ├ sortList([4, 2])
  │   ├ sortList([4]) → [4]   (base case)
  │   ├ sortList([2]) → [2]   (base case)
  │   └ merge([4], [2]) → [2, 4]
  ├ sortList([1, 3])
  │   ├ sortList([1]) → [1]   (base case)
  │   ├ sortList([3]) → [3]   (base case)
  │   └ merge([1], [3]) → [1, 3]
  └ merge([2, 4], [1, 3]) → [1, 2, 3, 4]
🔑 Why fast = head.next (not head)?

For a 2-node list [A→B]: if fast starts at head (A), then after one loop iteration slow=B, mid=None → the right half is empty → infinite recursion on the left! Starting fast one step ahead ensures slow always lands at the last node of the left half, keeping both halves non-empty.

3 Why It Works: The Invariant ▾
🔐 The Merge Invariant

At every step of the merge loop, left and right point to the smallest remaining unmerged elements of two already-sorted sub-lists. The result chain (rooted at dummy.next) always contains the globally smallest elements seen so far, in sorted order.

Why comparing front elements is optimal: Since both sub-lists are sorted, their front nodes are their respective minimums. The globally smaller minimum must appear before any other element in either list. One comparison per step is sufficient — no lookahead needed.

Divide-and-conquer complexity: There are O(log n) recursion levels (halving each time). At each level, every node participates in exactly one merge → O(n) work per level. Total: O(n log n). The call stack is O(log n) deep.

⚠️ The Disconnect Step Is Non-Negotiable

After finding the midpoint, slow.next = None must execute before recursing. Without it, the left recursive call receives a list that still chains through the right half — it will never hit a base case, causing infinite recursion. The two halves must be physically independent chains before any recursive call.

Why a dummy sentinel in merge: Without dummy, you'd need to special-case "setting the result head for the first time." With dummy, cur.next = <chosen_node> works identically on every iteration. Skip past it at the end with return dummy.next.

Space complexity: Top-down uses O(log n) implicit stack space (the call stack). Bottom-up iterative merge sort — doubling sub-list sizes 1→2→4→8 — uses true O(1) space at the cost of more complex bookkeeping.

4 The Code (Annotated) ▾

Solution 1 — Top-Down Recursive Merge Sort (Interview Pick) — O(n log n) time, O(log n) space

def sortList(head): ← receives head of unsorted linked list
if not head or not head.next: ← 0 or 1 nodes: already sorted, base case
return head
slow, fast = head, head.next ← fast starts 1 ahead to prevent empty-right-half bug
while fast and fast.next: ← continue while fast can take 2 more steps
slow = slow.next ← tortoise: 1 step
fast = fast.next.next ← hare: 2 steps
mid = slow.next ← mid = first node of right half
slow.next = None ← ★ disconnect: left half now ends at slow
left = sortList(head) ← recursively sort left half [head..slow]
right = sortList(mid) ← recursively sort right half [mid..end]
dummy = ListNode(0); cur = dummy ← sentinel: simplifies result-chain building
while left and right: ← merge loop: both halves have remaining nodes
if left.val <= right.val: ← left front is smaller or equal
cur.next = left ← attach left node
left = left.next ← advance left pointer
else: ← right front is strictly smaller
cur.next = right ← attach right node
right = right.next ← advance right pointer
cur = cur.next ← advance result tail pointer
cur.next = left or right ← attach whichever remainder is non-empty
return dummy.next ← skip sentinel; return true sorted head
⚠️ cur.next = left or right — what does this do?

In Python, X or Y returns X if X is truthy, else Y. After the while loop at most one list has remaining nodes. This line safely attaches that remainder (or None if both are exhausted) in a single expression — no second loop needed. This works because the remaining list is already sorted.

Solution 2 — Brute Force: Collect into Array, Sort, Rebuild — O(n log n) time, O(n) space

def sortList_array(head): ← simpler but uses O(n) extra space
vals = [] ← collect all values into a Python list
cur = head
while cur: ← single pass through the linked list
vals.append(cur.val); cur = cur.next
vals.sort() ← Python's Timsort: O(n log n)
cur = head ← second pass: overwrite node values in order
for v in vals: ← write sorted values back; O(n)
cur.val = v; cur = cur.next
return head ← same structure, different values
💡 Complexity Summary

Array collect+sort: Time O(n log n), Space O(n) — simple but uses extra memory.
Top-Down Merge Sort: Time O(n log n), Space O(log n) — interview pick; elegant recursion.
Bottom-Up Merge Sort: Time O(n log n), Space O(1) — true constant space; more complex to implement.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Eliminate Wrong Approaches

"We need O(n log n). Bubble/insertion are O(n²). Quicksort on a linked list is also O(n²) without random access for pivot selection. Heapsort needs array indexing. The only clean O(n log n) sort on a linked list is merge sort."

Step 2: Decompose into Known Sub-Problems

"Merge sort needs two primitives: (1) split a list in half — the fast/slow pointer trick does this in O(n). (2) merge two sorted linked lists — that's exactly LeetCode #21 which I can implement in O(m+n). I know both sub-problems, so the full solution is just composing them."

Step 3: State the Key Subtleties

"Two gotchas: (a) fast must start at head.next not head to correctly handle 2-node lists; (b) I must set slow.next = None before recursing — without that disconnect both halves share the same chain and I get infinite recursion."

Step 4: Discuss the Space Trade-Off

"Top-down is O(log n) stack space. If you need true O(1), bottom-up iterative merge sort: sweep the list merging pairs of size 1, then size 2, then 4, doubling each pass — no recursion stack at all."

🗣️ What to Say to the Interviewer

"The O(n log n) constraint rules out bubble and insertion sort. Quicksort needs random access for pivoting — not great on a linked list. Merge sort is the natural fit: it only needs sequential access.

My approach: find the midpoint with fast/slow pointers, disconnect the list there, recursively sort each half, then merge using a dummy sentinel node.

Key subtleties: I start fast one step ahead of slow to handle 2-node lists correctly, and I explicitly set slow.next = None to create two truly independent chains before recursing.

Time: O(n log n) — log n recursion levels, O(n) merge work per level. Space: O(log n) for the call stack. Want me to show the O(1)-space bottom-up version as well?"
🔮 Likely Follow-Up Questions
  • "Can you do it in O(1) space?" → Bottom-up iterative: merge sub-lists of size 1, 2, 4, 8… without recursion.
  • "How would you sort a doubly-linked list?" → Same algorithm; update prev pointers during merge too.
  • "What if the list is already sorted?" → Still O(n log n) — merge sort doesn't detect this. Add an O(n) pre-check if needed.
  • "Can you use quicksort on a linked list?" → Yes but O(n²) worst case. You'd need a good pivot strategy to approach O(n log n) average.

Merge sort is the only natural O(n log n) sort on linked lists — it needs only sequential access
Fast/slow pointer (hare and tortoise) finds the midpoint in O(n) without knowing the length
