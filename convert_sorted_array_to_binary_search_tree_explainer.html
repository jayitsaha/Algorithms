LeetCode #108

Convert Sorted Array to Binary Search Tree

Given a sorted integer array, build a height-balanced BST by always picking the middle element as the root.

Easy Trees Divide & Conquer Time: O(n) Space: O(log n)
1 Understanding the Problem ▾

A Binary Search Tree (BST) is a binary tree where for every node, all values in the left subtree are strictly smaller and all values in the right subtree are strictly larger. A height-balanced BST additionally requires that the depth of the two subtrees of every node never differs by more than one.

We are given a sorted integer array and must construct such a tree. The output is the root node of the BST.

Key Insight

The middle element of any sorted array is the perfect root — it divides the remaining elements into two equal halves, so left and right subtrees get the same number of nodes. This split guarantees height balance at every level of the recursion.

Common Mistake — Picking the First Element as Root

Using nums[0] as root every time creates a degenerate chain — every node's left subtree is empty, and all remaining elements go to the right. This is a valid BST but has height O(n), not O(log n). It is NOT height-balanced.

Example: nums = [-10, -3, 0, 5, 9]

Balanced BST — Mid as Root

      0       ← root (index 2)
     / \
   -10   5   ← children
     \   \
    -3   9   ← grandchildren

Height = 3. Max depth difference = 0 at root. Balanced!

Degenerate BST — First as Root

-10
  \
  -3
    \
     0
      \
       5
        \
         9

Height = 5. NOT balanced.

Edge cases: Empty array returns null. Single element returns a leaf. Two elements: lower-mid (index 0) becomes root, upper element becomes right child. Even-length arrays always produce valid balanced BSTs with either mid choice.

2 The Algorithm: Recursive Divide & Conquer — Mid as Root ▾

At every level of recursion, the subproblem is identical: given a sorted subarray, build a height-balanced BST from it. We always pick the middle element as root, then recurse on both halves independently.

1 Base case: if lo > hi, the range is empty — return None. This becomes a null child pointer in the parent node.
2 Find mid: compute mid = (lo + hi) // 2. This is the index of the element that becomes the current root.
3 Create root: build a TreeNode(nums[mid]). This node is the root of the current subtree.
4 Recurse left: node.left = helper(lo, mid - 1). The left subarray is still sorted — same problem, smaller size.
5 Recurse right: node.right = helper(mid + 1, hi). The right subarray is also sorted. Attach result and return the node upward.

Recursion trace for nums = [-10, -3, 0, 5, 9]:

helper(0,4) → mid=2 → root = nums[2] = 0 helper(0,1) → mid=0 → node = nums[0] = -10 helper(0,-1) → lo > hi → None helper(1,1) → mid=1 → node = nums[1] = -3 helper(1,0) → lo > hi → None helper(2,1) → lo > hi → None helper(3,4) → mid=3 → node = nums[3] = 5 helper(3,2) → lo > hi → None helper(4,4) → mid=4 → node = nums[4] = 9 helper(4,3) → lo > hi → None helper(5,4) → lo > hi → None
Binary Search in Reverse

Binary search finds an element in a sorted array by always examining the middle. Here we build from a sorted array using the same midpoint logic — we are constructing the very tree that binary search implicitly navigates. The resulting BST IS the binary search structure over these values.

3 Why It Works: The Invariant ▾
The Invariant

At every recursive call, nums[lo..hi] is a sorted subarray, and the returned node is the root of a valid height-balanced BST containing exactly those values.

BST property is preserved automatically: Because nums is globally sorted, all elements to the left of any mid index are smaller than nums[mid], and all to the right are larger. The BST ordering invariant holds for free — we never need to compare or rearrange values.

Height-balance proof sketch: With mid = (lo + hi) // 2, the left subarray has mid - lo elements and the right has hi - mid elements. These counts differ by at most 1 (integer division rounding). Since both halves are built recursively with the same strategy, by induction their subtree heights differ by at most 1. Therefore, at every node, the depth difference between left and right subtrees is at most 1. QED.

For nums = [-10, -3, 0, 5, 9] (5 elements): Root (index 2) leaves 2 elements on each side — equal halves. Both subtrees have height 2. Overall tree has height 3, and is perfectly balanced.

Multiple Valid Answers

For even-length subarrays, (lo+hi)//2 (lower-mid) and (lo+hi+1)//2 (upper-mid) both yield valid height-balanced BSTs — just with slightly different structure. LeetCode accepts any valid answer. The key is being consistent throughout one recursive pass.

All edge cases handled correctly:

  • Empty array (lo > hi): Returns None immediately — a correct null child pointer.
  • Single element (lo == hi): mid = lo, creates a leaf node, both child calls return None.
  • Two elements: mid = lo (lower-mid), root gets left element, right call creates right child.
4 The Code (Annotated) ▾

Solution 1 — Recursive Mid as Root (Optimal, Interview Pick)

def sortedArrayToBST(nums): ← entry point; nums is sorted ascending
def helper(lo, hi): ← subproblem: build BST from nums[lo..hi]
if lo > hi: ← base case: empty range → return null
return None ← becomes a null left or right child
mid = (lo + hi) // 2 ← lower-mid index; perfect balance point
node = TreeNode(nums[mid]) ← create root of this subproblem
node.left = helper(lo, mid - 1) ← left subtree from left half of array
node.right = helper(mid + 1, hi) ← right subtree from right half
return node ← return root of this subtree upward
return helper(0, len(nums) - 1) ← kick off on the full array
Integer Overflow (C++ / Java)

In Python, (lo + hi) // 2 is always safe — Python integers have unlimited precision. In C++ or Java, lo + hi can overflow a 32-bit integer when both values are large. Use lo + (hi - lo) / 2 to avoid overflow in those languages.

Solution 2 — Iterative with Explicit Stack (No Recursion)

Same divide-and-conquer logic but with an explicit stack of (node, lo, hi) triples instead of the call stack. Each iteration pops a node, computes child mid indices, creates child nodes, and pushes them.

def sortedArrayToBST(nums): ← iterative version; avoids recursion depth limit
if not nums: return None ← guard: empty input
mid = len(nums) // 2 ← root: mid of full array
root = TreeNode(nums[mid])
stack = [(root, 0, len(nums) - 1)] ← each entry: (node, lo, hi) for that node
while stack: ← process until all nodes assigned children
node, lo, hi = stack.pop()
m = (lo + hi) // 2 ← this node's original mid
if lo <= m - 1: ← left half is non-empty
lm = (lo + m - 1) // 2 ← mid of left half
node.left = TreeNode(nums[lm])
stack.append((node.left, lo, m - 1))
if m + 1 <= hi: ← right half is non-empty
rm = (m + 1 + hi) // 2 ← mid of right half
node.right = TreeNode(nums[rm])
stack.append((node.right, m + 1, hi))
return root ← root never changes after creation
Complexity Summary

Recursive: Time O(n) — each element visited exactly once. Space O(log n) — recursion stack = tree height.
Iterative: Time O(n). Space O(log n) — explicit stack bounded by tree height (log n levels at a time).

5 Interview Strategy: How to Arrive at This ▾
Step 1 — Recognize the Signal

"Sorted array + build a BST" is a classic divide-and-conquer signal. The sorted order gives you BST property for free. The challenge is height balance. Ask yourself: what root placement guarantees balance?

Step 2 — Ask the Key Question

"What should be the root?" For a balanced BST, the root must split remaining elements equally. That means the middle element. It gives n//2 elements to each subtree — the most balanced possible split.

Step 3 — Express It Recursively

Left half → left subtree (same problem). Right half → right subtree (same problem). Both halves are still sorted, so the invariant holds. Base case: empty subarray → null. This is the complete recursive structure.

Step 4 — Implement with Index Arithmetic

Use lo and hi indices rather than slicing the array on each call. Array slicing would add O(n) space per level and O(n log n) total space. Index arithmetic keeps it O(log n) auxiliary space.

What to Say to the Interviewer

"I notice the input is sorted, which gives us BST ordering for free — any element's
left neighbors are smaller, right neighbors are larger. The challenge is height balance.

For balance I want equal-sized left and right subtrees at every node. That means always
picking the middle element as root — it splits the remaining array into equal halves.
Then I recurse left half for left subtree, right half for right subtree. Base case is
an empty subarray — return null.

I'll implement with a helper that takes lo and hi indices to avoid O(n) slicing.
Time O(n): each element becomes exactly one node. Space O(log n): recursion depth
equals tree height, which is log n for a balanced tree."
Likely Follow-up Questions
  • "Can you do it iteratively?" → Yes — replace the call stack with an explicit stack of (node, lo, hi) triples.
  • "What if there are duplicate values?" → Pick any consistent mid strategy. BST property (left <= root <= right) still holds.
  • "Can there be multiple valid answers?" → Yes. For even-length arrays, upper-mid and lower-mid both produce valid balanced BSTs.
  • "What if the input is a linked list?" → Use slow/fast pointer to find mid in O(n) — that's LeetCode #109.

7 Summary & Key Takeaways ▾
Mid as root = guaranteed height-balanced split — left and right halves differ by at most 1 element
Recursion terminates when lo > hi (empty range = null node, not an error to guard against)
O(n) time: each array element becomes exactly one tree node — no element is visited more than once
O(log n) space: recursion depth equals tree height, which is log n for a balanced BST
Multiple valid outputs exist — either mid choice for even-length arrays is acceptable on LeetCode
Use index arithmetic (lo/hi), NOT array slicing — slicing costs O(n) space per level of recursion
