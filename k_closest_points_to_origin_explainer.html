LeetCode #973

K Closest Points to Origin

Given an array of points on a 2D plane, return the k points closest to the origin (0, 0). Distance is Euclidean.

Medium Heaps Max Heap of Size K Time: O(n log k) Space: O(k)
1 Understanding the Problem ▾

We have a list of 2D points [x, y] and need to find the k closest to the origin (0, 0). Euclidean distance from point [x, y] to origin is √(x² + y²). We must return exactly k points; order doesn't matter.

💡 Key Insight

We never need to compute the square root. Since √ is a monotone increasing function, comparing x₁²+y₁² vs x₂²+y₂² gives the exact same ordering as comparing actual distances. Squaring avoids floating-point arithmetic entirely.

Example: points=[[1,3],[-2,2]], k=1

dist([1,3])² = 1 + 9 = 10
dist([-2,2])² = 4 + 4 = 8
k=1 → return [[-2, 2]]

Why not sort all n points?

Sort all n = O(n log n)
Heap keeps only k → O(n log k)
When k ≪ n, heap wins!

Edge cases: k equals n (return all), k equals 1 (single closest), all equidistant points (any k are valid), negative coordinates (x²+y² is still positive), single-element input.

2 The Algorithm: Max Heap of Size K ▾

We maintain a max heap capped at k elements, where the root always holds the farthest current candidate. For each new point, we push it in. If the heap grows past k, we evict the root — the farthest point. After all n points, the heap holds exactly the k closest.

1 Compute squared distance: For each [x, y], compute dist_sq = x*x + y*y. Negate it when pushing: Python's heapq is a min-heap, so -dist_sq makes the largest actual distance sit at the root.
2 Push onto heap: heappush(heap, (-dist_sq, x, y)). The heap automatically maintains the most-negative value (largest distance) at the top — our "eviction candidate."
3 Evict if over size k: If len(heap) > k, call heappop(heap). This removes the root — the point with the largest squared distance. Heap is back to size k, holding the best k seen so far.
4 Return the heap contents: Extract [x, y] from each tuple. These k points are the final answer. (No need to sort them — the problem doesn't require ordering.)
🔑 Max Heap vs Min Heap — What's the Difference Here?

A min heap of all n points (then pop k times) works but needs O(n) space. A max heap of size k uses O(k) space and processes each point online. The max at the top acts as a "bouncer threshold" — any new point closer than the bouncer gets in, and the bouncer leaves.

3 Why It Works: The Invariant ▾
🔐 The Invariant

After processing each point, the max heap contains at most k elements, all of which are the k closest points seen so far. No point outside the heap has a smaller distance than any point inside it.

Why evicting the max is always correct: When heap size exceeds k after a push, the heap has k+1 candidates for the "k closest." Exactly one must leave — and it should be the worst (farthest) of the k+1. The max heap root is always that worst element. After eviction, the invariant is restored.

Why squaring distances preserves order: For non-negative a, b: a < b ↔ a² < b² ↔ √a < √b. Squared distances are always non-negative, so comparing x₁²+y₁² vs x₂²+y₂² is equivalent to comparing √(x₁²+y₁²) vs √(x₂²+y₂²).

⚠️ The Negation Trick — Easy to Forget

Python's heapq only provides a min-heap. Push (-dist_sq, x, y) — negating the distance. The most-negative value (= largest actual dist) floats to the top. When we pop, we remove the farthest point. Forgetting to negate produces a min-heap that evicts the closest points — exactly backwards!

Proof by induction: Base case: empty heap is trivially the "0 closest of 0 seen." Inductive step: after processing point p — if heap < k elements, push p (heap now holds all seen, which are the k closest of those). If heap = k and dist(p) < max(heap), push p then pop the max: new heap holds k closest. If dist(p) ≥ max(heap), push p then immediately pop p (equivalently skip it): heap unchanged, still holds k closest.

4 The Code (Annotated) ▾

Solution 1 — Max Heap of Size K (Interview Pick, O(n log k))

import heapq
def kClosest(points, k): ← points=[[x,y],...], k=how many to return
heap = [] ← max heap simulated via negation; stores (-dist², x, y)
for x, y in points: ← iterate all n points — O(n) iterations
dist_sq = x*x + y*y ← squared distance; skip sqrt (monotone function)
heapq.heappush(heap, (-dist_sq, x, y)) ← negated: largest dist becomes most-negative → sits at root
if len(heap) > k: ← after push, do we have k+1 candidates?
heapq.heappop(heap) ← evict the root = farthest point; heap back to k
return [[x, y] for (_, x, y) in heap] ← unpack (neg_dist, x, y) tuples into [x,y] lists
⚠️ Why (-dist_sq, x, y) and not just -dist_sq?

Storing the tuple (-dist_sq, x, y) keeps the coordinates alongside the key. If two points have equal distance, Python compares the next tuple element (x) as a tiebreaker — this is fine and avoids errors. The underscore _ in the return unpacking discards the negated distance; we only want [x, y].

Solution 2 — Sort All Points (Simpler, O(n log n))

def kClosest(points, k):
points.sort(key=lambda p: p[0]**2 + p[1]**2) ← sort all n by dist² — O(n log n), in-place
return points[:k] ← first k are the k closest — slice O(k)

Solution 3 — QuickSelect (O(n) avg, O(n²) worst)

import random
def kClosest(points, k):
def dist(p): return p[0]**2 + p[1]**2
def quickselect(lo, hi): ← partition-based selection — like quicksort but one side
pivot = dist(points[random.randint(lo,hi)])← random pivot avoids worst-case O(n²)
l, r, m = lo, hi, lo ← 3-way Dutch National Flag partition
while m <= r:
d = dist(points[m])
if d < pivot: points[l],points[m]=points[m],points[l]; l+=1; m+=1
elif d > pivot: points[m],points[r]=points[r],points[m]; r-=1
else: m+=1
if k <= l: return quickselect(lo, l-1) ← answer is entirely in left partition
elif k <= m: return ← answer spans into pivot group — done
else: return quickselect(m, hi) ← answer is in right partition
quickselect(0, len(points)-1)
return points[:k] ← points[:k] now hold the k closest (not sorted)
💡 Complexity Summary

Max Heap of k: O(n log k) time, O(k) space — interview pick; optimal for k ≪ n and streaming
Sort all: O(n log n) time, O(1) extra — easiest to write; best when k ≈ n
QuickSelect: O(n) avg, O(n²) worst, O(1) space — fastest average but tricky under pressure

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Spot the "Top K" Pattern

The phrase "find the k closest" is a classic signal. Any "k smallest/largest/most-frequent/nearest" question maps to the heap pattern. Here "k closest distances" = "k smallest squared distances."

Step 2: Propose Sort First (Brute Force)

"I can sort all points by x²+y² and return the first k. That's O(n log n) — correct but potentially wasteful since we sorted more than we needed to." Establish a baseline, then optimize.

Step 3: Optimize with Max Heap of Size K

"I only need to track k candidates at any time. A max heap lets me maintain the k closest seen so far: push each point, if heap exceeds k evict the root (farthest). O(n log k) time, O(k) space — strictly better when k ≪ n."

Step 4: Bonus — Mention QuickSelect

"For O(n) average time, QuickSelect partitions in-place. But worst case is O(n²) and it's harder to code correctly without bugs. For an interview I'd stick with the heap solution unless the problem explicitly asks for O(n)."

🗣️ What to Say to the Interviewer

"This is a 'k smallest by a metric' problem — my go-to is a max heap of size k.
First: I'll use squared distance x²+y² instead of Euclidean — same ordering, avoids sqrt.
I iterate all n points; for each I push (-dist_sq, x, y) onto the heap.
Python heapq is a min-heap, so negating makes the farthest point bubble to the root.
If heap size exceeds k, I pop the root — evicting the farthest candidate.
At the end, the heap holds exactly the k closest. Time O(n log k), Space O(k)."
🔮 Likely Follow-up Questions
  • "What if new points keep arriving (online stream)?" → Heap handles this perfectly; process one at a time
  • "Return them sorted by distance?" → Sort the k results: O(k log k) extra; total still O(n log k)
  • "What about 3D points?" → dist² = x²+y²+z²; algorithm identical
  • "Kth closest (not k closest)?" → Same heap; the point evicted when size first reaches k+1 is the (k+1)-th closest
  • "What if k=n?" → Just return all points; heap approach still works, same as sorting

7 Summary & Key Takeaways ▾
Use squared distance (x²+y²) — same ordering as Euclidean, no sqrt needed
Max heap of size k: root always holds the current farthest candidate — the one to evict
Python's heapq is a min-heap; negate the key to simulate a max-heap
O(n log k) time, O(k) space — much better than O(n log n) sort when k ≪ n

🎯 When to Recognize the Max Heap of Size K Pattern
  • "Find k smallest/largest/closest/most-frequent" from n items → max heap of size k
  • Data is a stream (can't hold all n at once) → heap handles one element at a time in O(k) space
  • k is significantly smaller than n → heap beats full sort
  • You need the answer to update as data grows → maintain the bounded heap
