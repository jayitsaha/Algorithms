LeetCode #314

Binary Tree Vertical Order Traversal

Return the vertical order traversal of a binary tree: group nodes by column (left to right), top to bottom within each column.

Medium Trees BFS + Column Index Time: O(n) Space: O(n)
1 Understanding the Problem ▾

Imagine holding the tree in front of a window and shining vertical light beams through it. Each beam corresponds to a column. Every node lives in exactly one column. The root lives in column 0. Moving left decrements the column index by 1; moving right increments it by 1. We want all nodes grouped by column, left to right, and within each column ordered top to bottom.

💡 Key Insight

Each node gets a column index determined solely by the path from root to that node: every left-turn subtracts 1, every right-turn adds 1. BFS (level-order traversal) visits nodes top-to-bottom naturally, so appending values in BFS order gives the correct within-column ordering without any extra sorting.

Example tree and its column assignments:

3 col 0 / \ 9 20 9: col -1, 20: col +1 / \ 15 7 15: col 0, 7: col +2

Column groupings for this tree:

-1
[9]
0
[3, 15]
+1
[20]
+2
[7]

Answer: [[9], [3, 15], [20], [7]]

Edge Cases: Empty tree → []. Single node → [[root.val]]. Skewed left tree: all nodes in unique decreasing columns. Two nodes in the same column at the same level → order them left-to-right (BFS handles this by enqueuing left child before right child).

2 The Algorithm: BFS + Column Index Mapping ▾

We combine BFS (breadth-first search) — which naturally processes the tree level by level — with a column index tracker stored alongside each node in the queue. A dictionary maps each column to its accumulated list of values. Because BFS visits nodes in top-to-bottom order, we never need to sort within a column.

1 Edge case check: If root is None, return [].
2 Initialize: A deque starting with (root, 0). A defaultdict(list) called col_map. Two integers min_col and max_col starting at 0.
3 BFS loop: While the queue is non-empty — dequeue (node, col). Append node.val to col_map[col]. Update min_col and max_col. If node.left exists, enqueue (node.left, col−1). If node.right exists, enqueue (node.right, col+1).
4 Build result: Return [col_map[c] for c in range(min_col, max_col+1)]. The range ensures correct left-to-right column ordering without sorting dictionary keys.
🔑 Why BFS, Not DFS?

DFS visits in pre/in/post order — not level by level. Within a single column, a DFS might visit a deep left-branch node before a shallow right-branch node that belongs above it in the same column. With DFS you'd need to store (level, val) pairs and sort within each column. BFS gives the correct top-to-bottom order for free.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every BFS dequeue, a node's column index exactly equals the number of right-branches minus the number of left-branches taken on the path from the root to that node. BFS processes all nodes at depth d before any node at depth d+1, so any two nodes in the same column are appended in strictly increasing depth order — which is exactly top-to-bottom.

Column index correctness: When the BFS enqueues a left child, it uses col − 1; for a right child, col + 1. These offsets accumulate along any root-to-node path: if a node is reached by taking r right-turns and l left-turns, its column is r − l. This uniquely identifies the vertical strip the node falls in.

Top-down ordering proof: BFS processes nodes in non-decreasing order of depth. If node A is at depth d₁ and node B is at depth d₂ in the same column, with d₁ < d₂, then A is dequeued before B. Therefore A's value is appended before B's value in col_map[col]. The result is top-to-bottom.

⚠️ Same Column, Same Level: Left-to-Right

If two nodes share both column and level, BFS still handles it correctly. When processing their common parent, we always enqueue node.left before node.right. This means the left-path node enters the queue first and is dequeued first — giving left-to-right order within the same level and column.

Why track min/max col? Rather than sorting dictionary keys at the end (O(k log k) for k distinct columns), we track the running min and max in O(1) per step. Then we iterate range(min_col, max_col+1) in O(k) to build the output list.

4 The Code (Annotated) ▾

Solution 1 — BFS + Column Index (Interview Pick)

from collections import defaultdict, deque
def verticalOrder(root):
if not root: ← Null root: nothing to traverse
return []
col_map = defaultdict(list) ← col → [values], auto-creates list on first access
queue = deque([(root, 0)]) ← BFS queue: (TreeNode, column_index)
min_col = max_col = 0 ← Track column range to avoid sorting keys later
while queue: ← Process every node exactly once
node, col = queue.popleft() ← deque.popleft() is O(1); list.pop(0) is O(n) — never use list!
col_map[col].append(node.val) ← Record value. BFS order = top-to-bottom ✓
min_col = min(min_col, col) ← Extend range leftward if new leftmost column
max_col = max(max_col, col) ← Extend range rightward if new rightmost column
if node.left:
queue.append((node.left, col-1)) ← Left child is one column to the left
if node.right:
queue.append((node.right,col+1)) ← Right child is one column to the right
return [col_map[c] ← Assemble result column by column, left to right
for c in range(min_col, max_col+1)]
⚠️ Never use list.pop(0) for BFS in Python

Python's list.pop(0) shifts all remaining elements — it's O(n) per operation. Using it in a BFS over n nodes makes the whole algorithm O(n²). Always import and use collections.deque whose popleft() is O(1). This is a common interview mistake that examiners watch for.

Solution 2 — DFS with (level, val) sorting

def verticalOrder_dfs(root):
col_map = defaultdict(list) ← col → [(level, val), ...]
def dfs(node, col, level):
if not node: return
col_map[col].append((level, node.val)) ← Must store level to sort later
dfs(node.left, col-1, level+1)
dfs(node.right, col+1, level+1)
dfs(root, 0, 0)
return [[v for _, v in sorted(col_map[c])] ← Sort each column by level
for c in sorted(col_map.keys()]] ← Sort column keys too
💡 Complexity Summary

BFS (Solution 1): Time O(n) — each node visited once. Space O(n) — queue + col_map.
DFS (Solution 2): Time O(n log n) — sort within each column. Space O(n).
BFS is strictly better or equal on time. For LC #987 (harder variant), DFS with (level, val) pairs is actually required — the additional sort by val breaks BFS's natural ordering assumption.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Decode "Vertical Order"

The word "vertical" signals columns. Draw a small tree and mentally drop vertical lines — nodes in the same line belong to the same column. Assign column 0 to root, then left=parent−1, right=parent+1. This is the encoding that makes grouping possible.

Step 2: Identify the Ordering Constraint

"Top to bottom within each column" → that's level-order, which is exactly what BFS gives. Mention DFS as an alternative but note it requires storing and sorting by depth — extra work. BFS is the cleaner choice here.

Step 3: Choose Data Structures and Code Up

Queue: deque of (node, col) pairs. Map: defaultdict(list). Track min/max col to skip key-sorting at the end. The code writes itself once the data structures are chosen.

Step 4: State Complexity and Edge Cases

Time O(n) — BFS visits each node once. Space O(n) — queue holds at most O(w) items at any time (w = max tree width), col_map holds all n values. Edge cases: null root, all nodes in one column (skewed tree), same-column same-level nodes.

🗣️ What to Say to the Interviewer

"I'll assign each node a column index: root=0, left child=parent_col−1, right=parent_col+1.
For top-to-bottom ordering within each column, BFS is perfect — it processes level by level.
I'll keep a deque of (node, col) pairs, a defaultdict mapping col to values,
and track min_col/max_col so I don't need to sort keys at the end.
Time O(n), Space O(n). Edge case: null root returns empty list."
🔮 Likely Follow-up Questions
  • "What about LC #987 — Vertical Order Traversal II?" → Same approach but with DFS: store (level, val) and sort within each column by (level, val) to break ties by value.
  • "Can you do it in O(n) with DFS?" → No — DFS requires sorting within columns (O(k log k) per column). BFS is the only O(n) approach for LC #314.
  • "What if two nodes are at the same column AND same level?" → BFS enqueues left before right, so they appear left-to-right in the output — correct behavior for this problem.
  • "How does this relate to maximum width of binary tree (LC #662)?" → Same column index trick: width of level = rightmost_col − leftmost_col + 1 at that level.
