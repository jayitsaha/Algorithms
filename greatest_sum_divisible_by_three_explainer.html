LeetCode #1262

Greatest Sum Divisible by Three

Given an integer array nums, return the maximum possible sum of elements chosen from nums such that the sum is divisible by three.

Medium Dynamic Programming Track Best per Mod Time: O(n) Space: O(1)
1 Understanding the Problem ▾

You have an array of non-negative integers. You can pick any subset (including the empty subset, which sums to 0). Your goal: maximize the sum of your chosen subset with the single constraint that the sum must be divisible by 3 (i.e., sum % 3 == 0).

💡 Key Insight

When you add a number to a subset, the new sum's remainder mod 3 is fully determined by the old remainder and the number's remainder: (old_rem + num%3) % 3. So instead of tracking every possible sum value (which can be huge), we only need to track the best sum achievable for each of the three possible remainders: 0, 1, and 2.

✓ Example: nums = [3, 6, 5, 1, 8]

3+6+5+1+8 = 23 (23%3=2, ✗)
3+6+1+8 = 18 (18%3=0 ✓) ← Answer
3+6+5+1 = 15 (15%3=0 ✓)
18 > 15, so Answer = 18

🔑 The Three Buckets

dp[0]: best sum with sum%3==0
dp[1]: best sum with sum%3==1
dp[2]: best sum with sum%3==2
Answer = dp[0] (want rem=0)

Edge cases: If no non-empty subset has sum divisible by 3, dp[0]=0 (the empty set). Single element not divisible by 3 → return 0. All elements are multiples of 3 → take them all. The answer is always ≥ 0.

🧮 Modular Arithmetic Foundation

For any integers a and b: (a + b) % 3 = ((a % 3) + (b % 3)) % 3. The remainder of a sum depends only on the remainders of the summands, not their exact values. This is why 3 states suffice to capture all the information we need — we never need to track the actual sum values in the DP, just which "mod bucket" they land in and the best value in each bucket.

2 The Algorithm: Three-Bucket DP ▾

We maintain an array dp[3] where dp[r] = the maximum sum of any subset processed so far whose sum has remainder r when divided by 3. For each new number, we compute how including it would update each bucket, and take the best option.

1 Initialize: dp = [0, -∞, -∞]. dp[0]=0 because the empty subset has sum 0 (remainder 0 is valid). dp[1]=dp[2]=-∞ because no subset yet gives a positive sum with those remainders. Using -∞ (not 0) prevents phantom "empty set" from polluting buckets 1 and 2.
2 Snapshot: Before processing each number, copy the current dp: new_dp = dp[:]. This is critical — we must compute all updates from the pre-update state to prevent using the same number twice.
3 Compute r = num % 3: The number's remainder determines how it shifts each bucket. Adding num shifts any state with remainder rem to remainder (rem + r) % 3.
4 Update buckets: For each existing state rem ∈ {0,1,2} where dp[rem] != -∞, compute new_rem = (rem+r)%3 and set new_dp[new_rem] = max(new_dp[new_rem], dp[rem]+num). Each update is an O(1) max check.
5 Commit: Set dp = new_dp. Repeat for all numbers. Return dp[0] — the maximum sum with remainder 0.
📐 The Recurrence Relation
Let r = num % 3 for the current number.

For each valid old remainder rem ∈ {0,1,2} (where old dp[rem] > -∞):
  new_dp[(rem + r) % 3] = max(new_dp[(rem + r) % 3], old_dp[rem] + num)

Final answer: dp[0] after all numbers processed

💡 Analogy: The Mod-3 Ledger

Imagine three bank accounts — one for "sums that leave remainder 0", one for remainder 1, one for remainder 2. Each number you add transfers money from one account to another (via modular arithmetic) if doing so improves the balance. The account you never touch keeps its old balance (skipping the number). The answer is the balance in account 0.

3 Why Is This DP? The Invariant & Proof ▾
🔐 Why DP? — Optimal Substructure

The maximum subset sum with remainder j using nums[0..i] is determined by the optimal sums at nums[0..i-1]. Specifically: either we exclude nums[i] (keep the old best for remainder j), or we include it (take the old best for the source remainder that maps to j, then add nums[i]). The optimal choice at step i depends only on the optimal answers from step i-1 — a textbook DP structure.

🔁 Why DP? — Overlapping Subproblems

A naive recursive approach would consider all 2^n subsets. However, many subsets share the same (index, running_remainder) state. For example, many different subsets ending at index i might all have running remainder 1 — we only need to remember the best one. Memoizing on (index, remainder) collapses these overlapping subproblems. Total unique states: n × 3, so total work is O(n).

🔐 The Invariant

After processing nums[0..i], dp[j] = the maximum sum of any subset of nums[0..i] whose sum has remainder j (mod 3). If no such subset exists (beyond the empty set for j=0), dp[j] = -∞. This invariant is maintained by the snapshot + update pattern.

Why -∞ for dp[1] and dp[2] initially (not 0): The empty set has sum 0, giving remainder 0 only. Initializing dp[1]=0 would falsely claim "there's a subset with sum 0 and remainder 1" — which is impossible. The -∞ sentinel says "no valid subset yet." Once a number with remainder 1 or 2 is processed, those sentinels become finite values.

Why snapshot before updating: If we updated dp[0] first (with some number), then computed the update for dp[1] using the already-updated dp[0], we'd effectively be adding the same number twice to build dp[1]. The snapshot copies the pre-update state so all transitions come from the same "generation."

⚠️ Why Not Greedy?

Greedy fails here. Consider [2, 2, 2, 3]. A greedy approach taking the largest first gets 3 (divisible by 3), then might stop or add 2+2+2=6, giving 3+6=9 — but actually taking just 2+2+2=6 gives less, and we'd need to consider 3+2+2+2=9 (also 9). But consider [4, 4, 4]: greedy might take the first 4 (remainder 1), miss 4+4+4=12. The issue: deciding whether to include a number depends on what future numbers exist. DP handles this by tracking all feasible states simultaneously.

Space reduction from O(n×3) to O(3): The tabulation table has n rows, each length 3. But row i only depends on row i-1. So we keep a single length-3 array and update it in-place (with a snapshot), compressing space from O(n) to O(1).

4 The Code (Annotated) ▾

Solution 1 — Tabulation / Bottom-Up DP (Interview Pick)

def maxSumDivThree(nums): ← takes list of non-negative integers
dp = [0, float('-inf'), float('-inf')] ← dp[r] = best sum with sum%3==r; -inf = no valid subset yet
for num in nums: ← O(n): one pass through all numbers
new_dp = dp[:] ← CRITICAL: snapshot before modifying — prevents double-counting
r = num % 3 ← this number's remainder mod 3 (0, 1, or 2)
for rem in range(3): ← consider each existing bucket as a "source"
if dp[rem] == float('-inf'): ← skip impossible states (-inf = no valid subset)
continue
new_rem = (rem + r) % 3 ← adding num shifts remainder by r (modular addition)
new_dp[new_rem] = max(new_dp[new_rem], ← keep the best: either don't include num (current new_dp),
dp[rem] + num) ← or include it (old dp[rem] + num)
dp = new_dp ← commit updated state for this number
return dp[0] ← dp[0] = max sum divisible by 3; always ≥ 0 (empty set)
⚠️ The Snapshot Pattern is Critical

new_dp = dp[:] creates a shallow copy of dp. Without this, updating dp[0] during the inner loop would make the updated value available for computing dp[2] (via rem=1→new_rem=2), effectively adding the same number twice. Always snapshot, then read from snapshot, write to new copy.

Solution 2 — Memoization / Top-Down DP

from functools import lru_cache
def maxSumDivThree_memo(nums): ← top-down: easier to derive from natural recursion
n = len(nums)
@lru_cache(maxsize=None) ← memoize on (index, current_remainder)
def dp(i, rem): ← max additional sum from nums[i..n) given running rem
if i == n: ← base case: no more numbers
return 0 if rem == 0 else float('-inf') ← valid only if accumulated remainder is 0
skip = dp(i + 1, rem) ← option 1: skip nums[i], remainder unchanged
take = nums[i] + dp(i+1, (rem + nums[i]) % 3)← option 2: take nums[i], shift remainder
return max(skip, take) ← always pick the better option
return dp(0, 0) ← start at index 0, running remainder 0
💡 Complexity Summary

Brute Force: Time O(2^n), Space O(n) — check every subset
Memoization (Top-Down): Time O(n×3) = O(n), Space O(n×3) = O(n) — for cache + call stack
Tabulation (Bottom-Up): Time O(n×3) = O(n), Space O(1) — only 3 variables updated per step

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Pattern Signals

"Maximize a subset sum with a divisibility constraint." The subset problem suggests DP (2^n brute force). The modular constraint (÷3) shrinks the state space to just 3 remainders — a classic signal for "track state per mod."

Step 2: Identify States — "What Must I Track?"

"At each step, I need to know the current running sum's remainder mod 3 to decide if adding a number keeps me on track." So state = (index, running_remainder). Index goes 0..n, remainder ∈ {0,1,2} — only 3n total states.

Step 3: Write the Recurrence from Recursion

Start top-down: "I can skip or take nums[i]. If I take it, the remainder shifts by nums[i]%3 mod 3. Base case: if I've processed all numbers, it's valid only if the running remainder is 0." Code this up with lru_cache — it's already correct.

Step 4: Convert to Tabulation + Optimize Space

Bottom-up: process nums left-to-right, maintaining a 3-element dp array. Since each step only depends on the previous step, we don't need the full n×3 table — just 3 values updated with a snapshot. Space drops from O(n) to O(1).

🗣️ What to Say to the Interviewer

"This is a subset sum problem with a mod-3 divisibility constraint. Brute force is O(2^n).
Key observation: when adding a number, the new sum's remainder mod 3 depends only on the
old remainder and the number's remainder. So I only need to track 3 states: dp[0], dp[1], dp[2],
where dp[r] is the maximum sum achievable with remainder r. Initialize dp=[0,-inf,-inf]
(empty set has sum 0, remainder 0). For each number, snapshot the old dp, compute transitions,
and take the max. Return dp[0]. Time O(n), Space O(1)."
🔮 Likely Follow-up Questions
  • "What if divisor is k instead of 3?" → Use dp[k], same O(nk) logic, O(k) space.
  • "What if nums can be negative?" → Same code works. The -inf initialization handles it correctly.
  • "Return the actual subset?" → Track parent pointers in the DP table; increases space to O(n).
  • "What's wrong with dp=[0,0,0] initialization?" → It assumes the empty set has remainder 0, 1, and 2 simultaneously — incorrect. Use -inf for rem=1 and rem=2.

7 Summary & Key Takeaways ▾
Modular constraints shrink the DP state space: track best sum per remainder, not per possible sum value
Initialize dp[0]=0 (empty set) and dp[1]=dp[2]=-∞ (no valid subset yet for those remainders)
Always snapshot dp before updating to prevent using the same number twice in one step
Generalizes to "divisible by k" with a length-k dp array in O(nk) time and O(k) space
The answer is always dp[0] ≥ 0 because the empty set contributes 0 with remainder 0
ApproachTimeSpaceNotes
Brute Force (all subsets) O(2^n) O(n) Correct but exponential
Memoization (top-down DP) O(n) O(n) n×3 states, intuitive to derive
Tabulation (bottom-up) ✓ O(n) O(1) Interview pick: no stack overhead
🎯 When to Recognize "Track Best per Mod" Pattern
  • "Maximize/minimize sum with divisibility constraint" → dp[k] per remainder
  • "Count subsets with sum ≡ r (mod k)" → same state, count instead of max
  • Any running total where only the modular class (not exact value) affects future decisions
  • The state space hint: if constraints reduce to a fixed small set, model them as DP states
