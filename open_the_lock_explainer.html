LeetCode #752

Open the Lock

Find the minimum number of wheel turns to reach the target combination, starting from "0000" and never landing on a deadend combination.

Medium Graph · BFS BFS State Space Time: O(10⁴ + D) Space: O(10⁴ + D)
1 Understanding the Problem ▾

You have a physical combination lock with 4 wheels, each displaying a single digit 0–9. The lock starts at "0000". You want to reach a specific target string like "0202". Each "turn" rotates one wheel by exactly one position, either up or down — with wrapping: after 9 comes 0, and before 0 comes 9.

The catch: some combinations are deadends. If the lock ever displays a deadend, it locks permanently. You must navigate from "0000" to target without ever passing through a deadend state. Return the minimum number of turns, or -1 if it's impossible.

💡 Key Insight — It's a Graph Problem

Each of the 10,000 possible combinations (0000–9999) is a node in a graph. Two nodes are connected by an edge if you can reach one from the other in exactly one wheel turn. Deadends are nodes we cannot visit. We need the shortest path from "0000" to target — and BFS finds shortest paths in unweighted graphs exactly.

2 The Algorithm: BFS on State Space ▾

We treat the lock combinations as an unweighted graph and run Breadth-First Search. BFS is perfectly suited here because: every turn costs exactly 1 (uniform edge weight), and BFS processes states in non-decreasing order of distance from the source, so the first time we reach the target is via the minimum possible turns.

1 Handle edge cases first: Convert deadends list to a set for O(1) lookup. If "0000" is in deadends, return -1. If target equals "0000", return 0.
2 Initialize BFS with pre-poisoned visited: Start queue with ("0000", 0). Set visited = dead_set ∪ {"0000"}. Pre-adding deadends to visited is an elegant trick — BFS will automatically skip them without any extra check.
3 Generate 8 neighbors per state: For each of 4 wheels (indices 0–3), compute two neighbors: turn up (digit + 1) % 10 and turn down (digit - 1 + 10) % 10. Build the new state string by slicing.
4 Goal check and enqueue: If any neighbor equals target → return steps + 1. Otherwise, if neighbor is not in visited → mark it visited and enqueue with steps + 1.
5 Return -1: Queue emptied without reaching target means no valid path exists.
🔑 Why BFS, Not DFS?

DFS explores one path deeply before backtracking — it might find a 50-turn path before discovering a 6-turn path. BFS explores all 1-turn states, then all 2-turn states, then all 3-turn states, etc. The moment BFS first touches the target, it's via the shortest possible path. DFS gives correctness (eventually finds a path) but not optimality (not necessarily the shortest path).

🔑 Neighbor Generation Formula

From state s at wheel index i, two neighbors:

  • Up: neighbor = s[:i] + str((int(s[i]) + 1) % 10) + s[i+1:]
  • Down: neighbor = s[:i] + str((int(s[i]) - 1 + 10) % 10) + s[i+1:]

Example: from "1239", wheel 3 (digit=9): up → "1230" (9+1=10→0), down → "1238" (9-1=8).

3 Why It Works: The BFS Invariant ▾
🔐 The Invariant

At any moment during BFS, every state currently in the queue was reached in exactly k turns (for the current BFS "wave" or "layer"). When we dequeue a state and explore its neighbors, those neighbors are reachable in exactly k+1 turns. This layer-by-layer monotonic expansion is the fundamental correctness guarantee of BFS for unweighted shortest paths.

Why first-visit = minimum distance: Suppose the shortest path to target has length d. BFS will explore all states at distance 1, then 2, ..., then d-1, then finally d. At distance d, it will encounter target for the first time. Since BFS never processes distance d before d-1, and never processes d-1 before d-2, the first encounter with target is guaranteed to be at the minimum distance.

Why the visited set prevents infinite loops: The lock state graph has cycles — you can spin wheel 0 from 0000 to 1000 to 2000... back to 0000. Without a visited set, BFS would loop endlessly. By marking a state visited at the moment of enqueueing (not dequeuing), we also prevent the same state from being added to the queue multiple times by different neighbors — which would cause unnecessary re-processing.

⚠️ Common Mistake: Mark Visited at Dequeue (Wrong!)

If you mark a state visited when you dequeue it, the same state can be added to the queue multiple times — once from each of its neighbors. This is wasteful (blows up queue size) and technically still correct but inefficient. Always mark visited at enqueue time to prevent duplicates. The pre-poisoning of deadends works only because of this approach.

Complexity justification: State space = 10,000 combinations (10⁴). Each state has at most 8 neighbors (4 wheels × 2 directions). Each state is enqueued and processed at most once. Total work: O(8 × 10⁴) = O(10⁴). Building the deadend set costs O(D) where D = |deadends|. Final: O(10⁴ + D) time and space.

4 The Code (Annotated) ▾

Solution 1 — Standard BFS (Interview Pick)

from collections import deque
def openLock(deadends, target): ← deadends: List[str], target: str
dead = set(deadends) ← O(1) lookup instead of O(D) list search
if "0000" in dead: return -1 ← Can't start — start is a deadend
if target == "0000": return 0 ← Already at target — 0 turns
queue = deque([("0000", 0)]) ← BFS queue: (state_string, turns_so_far)
visited = dead | {"0000"} ← Pre-poison: deadends + start both marked
while queue: ← BFS loop: continues until queue empties
state, steps = queue.popleft() ← Dequeue oldest (fewest turns) first
for i in range(4): ← Try each of 4 wheels
d = int(state[i]) ← Current digit at wheel i
for delta in [1, -1]: ← Turn up or turn down
new_d = (d + delta) % 10 ← Wrapping: 9+1→0, 0-1→9 (Python mod is safe)
nb = state[:i]+str(new_d)+state[i+1:] ← String slice to build neighbor
if nb == target: ← Found! BFS guarantees this is minimum
return steps + 1
if nb not in visited: ← Skip deadends + already-seen states
visited.add(nb) ← Mark BEFORE enqueue to prevent duplicates
queue.append((nb, steps+1))← Enqueue with incremented step count
return -1 ← Queue empty = no path exists
⚠️ Why use (d - 1 + 10) % 10 in other languages?

In Python, (-1) % 10 == 9 (Python always returns non-negative modulo), so (d + delta) % 10 with delta=-1 correctly wraps 0 to 9. In Java or C++, -1 % 10 == -1 (remainder, not modulo). The +10 trick (d - 1 + 10) % 10 ensures 0 → 9 wrapping in any language. Python-only code can omit the +10.

Solution 2 — Bidirectional BFS (Advanced Optimization)

def openLock_bidir(deadends, target): ← Expand from both "0000" and target ends
dead = set(deadends)
if "0000" in dead or target in dead: return -1
if target == "0000": return 0
front, back = {"0000"}, {target} ← Two frontiers meeting in the middle
visited = dead | {"0000"}; steps = 0
while front:
if len(front) > len(back): ← Always expand the smaller frontier
front, back = back, front
nxt = set()
for state in front:
for i in range(4):
for delta in [1, -1]:
nb = state[:i]+str((int(state[i])+delta)%10)+state[i+1:]
if nb in back: return steps + 1 ← Frontiers met!
if nb not in visited: visited.add(nb); nxt.add(nb)
front = nxt; steps += 1
return -1
💡 Complexity Summary

Standard BFS: Time O(10⁴ + D), Space O(10⁴ + D) — always visits full layer-by-layer expansion
Bidirectional BFS: In theory O(8^(d/2)) vs O(8^d) — practical speedup when target is far
Wrong approach (DFS): May find a long path before a short one — incorrect for "minimum"

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the BFS State Space Signal

Key trigger phrases: "minimum turns/steps/moves," "starting state," "target state," "forbidden states." When you see all of these, it's almost always BFS on a state space graph. The "state" here is the 4-digit combination string.

Step 2: Define the Graph Explicitly

Say aloud: "I model this as a graph where each of the 10,000 possible combinations is a node. Two nodes connect if they differ by one wheel turn. Deadends are nodes I remove from consideration." This shows you understand the abstraction.

Step 3: Justify BFS Over DFS

Say: "Since every turn costs 1 (uniform weights), BFS guarantees minimum turns. DFS might dive into a 50-turn path before finding a 6-turn path. For weighted graphs I'd use Dijkstra, but here BFS is optimal."

Step 4: Mention the Pre-Poisoning Trick

Say: "An elegant trick: I add all deadends to the visited set before starting BFS. This means I don't need a separate deadend check inside the loop — the visited check handles it automatically."

Step 5 (Bonus): Offer Bidirectional BFS

If asked to optimize: "Bidirectional BFS expands from both start and target simultaneously. When the two frontiers meet, we've found the shortest path. This reduces the search from O(8^d) to O(8^(d/2)) — a major speedup for large d."

🗣️ What to Say to the Interviewer

"I see 'minimum turns' + 'forbidden states' → this is BFS on a state space graph.
State = 4-digit string. 10,000 possible nodes. 8 neighbors per node (4 wheels × 2 directions).
Deadends = nodes I can't visit. I'll pre-add them to visited before BFS starts.
BFS guarantees: first time I reach target = minimum turns. I return steps + 1.
Edge cases: check if "0000" is a deadend → -1. If target == "0000" → 0.
Complexity: O(10⁴ + D) time and space. If needed, bidirectional BFS cuts this further."
🔮 Likely Follow-up Questions
  • "Can you optimize further?" → Bidirectional BFS reduces frontier size
  • "What if turns had different costs?" → Replace BFS with Dijkstra
  • "What if the lock had 8 wheels?" → State space is 10⁸; A* with heuristic needed
  • "Why deque not list?" → deque.popleft() is O(1); list.pop(0) is O(n)
  • "What if target is in deadends?" → Return -1 immediately (add check after dead set creation)

7 Summary & Key Takeaways ▾
BFS guarantees minimum turns — it explores all k-step states before any (k+1)-step state
Pre-poisoning visited with deadends eliminates the need for a separate deadend check in the loop
Mark visited at enqueue time, not dequeue time — prevents duplicate queue entries
The complete state space is exactly 10,000 nodes — BFS is always bounded and terminates
Check "0000" in deadends before starting — this edge case trips many solutions
Bidirectional BFS is the textbook follow-up optimization: explore from both ends simultaneously
ApproachTimeSpaceNotes
DFS (wrong approach) O(10⁴) O(10⁴) No shortest-path guarantee — incorrect for minimum
BFS ✓ (Interview Pick) O(10⁴ + D) O(10⁴ + D) Guaranteed minimum; clean and readable

🎯 When to Recognize BFS State Space
  • "Minimum steps/moves/turns to reach goal state" + all moves cost 1 → BFS
  • State can be encoded as a string, tuple, or integer → each unique state is a node
  • Forbidden/blocked states exist → pre-add to visited set
  • State space is bounded and enumerable → BFS will terminate
  • If edge weights vary → use Dijkstra instead of BFS
