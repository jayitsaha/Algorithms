LeetCode #455

Assign Cookies

Maximize the number of content children — give each child a cookie big enough for their greed factor, each cookie used at most once.

Easy Greedy Sort Both Arrays Time: O(n log n) Space: O(1)
1 Understanding the Problem ▾

You are a parent handing out cookies to children. Each child i has a greed factor g[i] — the minimum cookie size that will make them content. Each cookie j has a size s[j]. A cookie satisfies a child when s[j] ≥ g[i]. You may give each cookie to at most one child. The goal: maximize the number of content children.

💡 Key Insight

To satisfy the most children, always pair the least-greedy unsatisfied child with the smallest cookie that can satisfy them. Using a bigger cookie on an easy-to-please child wastes the bigger cookie — which could have satisfied a greedier child.

✓ Example 1: g=[1,2,3], s=[1,1]

Sort: g=[1,2,3], s=[1,1]
s[0]=1 ≥ g[0]=1 → child 0 ✓
s[1]=1 < g[1]=2 → no match
Answer: 1

✓ Example 2: g=[1,2], s=[1,2,3]

Sort: g=[1,2], s=[1,2,3]
s[0]=1 ≥ g[0]=1 → child 0 ✓
s[1]=2 ≥ g[1]=2 → child 1 ✓
Answer: 2

Edge Cases to consider: empty greed array (0 children → answer 0), no cookies (answer 0), all cookies smaller than every child's greed (answer 0), all children satisfied, single child / single cookie.

2 The Algorithm: Sort Both, Greedy Match ▾

Sort children by greed (ascending) and cookies by size (ascending). Then sweep through cookies with a child pointer. When the current cookie can satisfy the current child, match them and advance both pointers. When a cookie is too small, discard it and try the next (larger) one.

1 Sort both arrays. g.sort() and s.sort() — this ensures we always consider the least-greedy child against the smallest available cookie.
2 Initialize pointers. child = 0 tracks which child we're trying to satisfy. count = 0 tracks how many children are content so far.
3 Iterate over cookies. For each cookie (smallest to largest): if cookie ≥ g[child], it satisfies the current child — count += 1, advance child. The cookie is always consumed (move to next).
4 Return count. Loop ends when all cookies are tried or all children satisfied. Return the total count of content children.
MATCH cookie ≥ g[child] — this cookie is big enough! Increment count, advance child pointer AND consume cookie.
SKIP cookie < g[child] — too small for this child and every greedier child. Discard it, try the next (bigger) cookie. Child pointer stays.
🔑 Why Sorting Enables Greedy

Without sorting, finding "smallest cookie that satisfies current child" would require a full scan — O(m) per child. After sorting both arrays, this check is O(1): just compare the current two pointers. Sorting converts a complex search into a simple scan.

3 Why It Works: The Greedy Invariant ▾
🔐 The Greedy Invariant

At each step: "if the smallest remaining cookie can satisfy the least-greedy remaining child, we should do it immediately." Waiting or using a larger cookie cannot produce a better overall count — it can only waste the larger cookie on an easy child.

Exchange argument (why this is optimal): Suppose an optimal solution uses a cookie C' (larger) for a child A, but leaves a smaller cookie C unused or given to a greedier child B. Since C ≥ g[A] (otherwise nothing satisfies A), we can swap: give C to A and C' to B. The count stays the same or improves (C' is larger and more likely to satisfy B). Therefore, always assigning the smallest sufficient cookie loses nothing.

⚠️ The Critical Discard Rule

When a cookie is too small for the current child, do not skip the child — skip the cookie. The child stays in place; it might be satisfiable by a larger cookie. The small cookie is discarded permanently — it cannot satisfy this child or any greedier child (since children are sorted).

Why cookies are always consumed (j always advances): After comparing a cookie to the current child, the cookie has only two fates — it was assigned to this child (match), or it is too small for this child and every greedier child (skip). Either way, we never need this cookie again. We move to the next.

4 The Code (Annotated) ▾

Solution 1 — Greedy for-loop (Optimal, Interview Pick)

def findContentChildren(g, s):
g.sort() ← Sort greed factors ascending (least greedy child first)
s.sort() ← Sort cookie sizes ascending (smallest cookie first)
child, count = 0, 0 ← child = pointer into g; count = satisfied so far
for cookie in s: ← Try each cookie, smallest first
if child < len(g) and cookie >= g[child]:← guard + satisfaction check
count += 1 ← This child is now content
child += 1 ← Move to next (greedier) unsatisfied child
return count ← Total content children
⚠️ Why child < len(g) guard inside the loop?

Once all children are satisfied (child == len(g)), we stop incrementing — but the for-loop over cookies keeps running. The guard ensures we don't index out of bounds into g. Without it, a Python IndexError occurs when all children are matched but more cookies remain.

Solution 2 — Explicit two-pointer while-loop (clearer intent)

def findContentChildren(g, s):
g.sort(); s.sort() ← Sort both in-place
i = j = 0 ← i = child pointer, j = cookie pointer
while i < len(g) and j < len(s): ← Stop when either array exhausted
if s[j] >= g[i]: ← Cookie j satisfies child i
i += 1 ← Child satisfied — advance child pointer
j += 1 ← Always consume this cookie
return i ← i = how many children were satisfied

Solution 3 — Brute Force O(n × m) — do not use in interviews

def findContentChildren_brute(g, s):
used = [False] * len(s) ← Track which cookies already given out
count = 0
for greed in g: ← For each child
for j in range(len(s)): ← Try every cookie
if not used[j] and s[j] >= greed:
used[j] = True
count += 1
break ← Found a cookie for this child — stop inner loop
return count ← O(n × m) — correct but slow
💡 Complexity Summary

Greedy (optimal): Time O(n log n + m log m) — dominated by sorting. Scan is O(n + m). Space O(1) extra (in-place sort).
Brute Force: Time O(n × m), Space O(m) for the used array.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Greedy Signal

The problem asks "maximize count of satisfied children" where each assignment is independent. There is no "take or skip with future consequences" — just a local comparison. When you see maximize count of matched pairs with threshold conditions, greedy is a strong candidate.

Step 2: State the Greedy Choice

"For the easiest child to satisfy, which cookie should I use? Answer: the smallest cookie that still meets their greed factor." Using a bigger cookie wastes it. This is the greedy property: locally optimal choice leads to globally optimal count.

Step 3: Use Sorting to Make Greedy Efficient

Sort both arrays. Now the "current child" is always the least-greedy remaining, and the "current cookie" is always the smallest remaining. The greedy check becomes a single O(1) comparison per cookie — no inner loops needed.

Step 4: Justify Discarding a Cookie

When a cookie is too small, explain why you can discard it: "This cookie can't satisfy the current child. Since children are sorted by greed, it also can't satisfy any greedier future child. So we can safely discard it and move on."

🗣️ What to Say to the Interviewer

"This is a greedy problem — I want to maximize matched pairs with a threshold.
The greedy insight: give the least-greedy child the smallest cookie that satisfies them.
I'll sort both g and s. Then sweep through cookies with a child pointer.
If cookie >= g[child]: it satisfies that child — count++, advance child.
Always advance the cookie pointer (discard it either way).
Time: O(n log n + m log m). Space: O(1)."
🔮 Likely Follow-up Questions
  • "Prove the greedy is optimal." → Exchange argument (Section 3)
  • "What if cookies can be split?" → Fractional knapsack — greedy by density ratio
  • "Minimize cookies used to satisfy all children?" → Same greedy still optimal; count = min cookies
  • "What if a child has a maximum size they'll accept?" → Add upper bound check — interval matching

7 Summary & Key Takeaways ▾
Sort both arrays — it makes the "find smallest sufficient cookie" step O(1) at every iteration
Always try to satisfy the least-greedy child first with the smallest possible cookie
A cookie too small for the current child is too small for all remaining (greedier) children — safely discard it
The child pointer only advances on a match; the cookie pointer always advances (cookie is always consumed)
The final count equals how far the child pointer advanced through the sorted greed array
