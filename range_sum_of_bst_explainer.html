LeetCode #938

Range Sum of BST

Given a BST and a range [low, high], return the sum of all node values within the range. Exploit the BST structure to prune entire subtrees.

Easy Trees BST Pruning Time: O(n) Space: O(h)
1 Understanding the Problem ▾

We have a Binary Search Tree (BST) and two integers low and high defining a range. We want the sum of every node whose value falls in [low, high] (inclusive).

The naive approach — visit every node and check — works, but a BST gives us a superpower: we know entire subtrees can be impossible. If a node's value is ≤ low, its entire left subtree is even smaller — we can skip it. If a node's value is ≥ high, its entire right subtree is even larger — we can skip it too.

💡 Key Insight

The BST ordering guarantee (left < root < right) lets us prune entire branches based solely on the current node's value. We never need to visit nodes in provably-out-of-range subtrees. This turns an O(n) traversal into something that can skip large portions of the tree in practice.

✓ Example: BST with low=7, high=15

Tree: 10, left=5(left=3,right=7), right=15(left=13,right=18)
In range: 7, 10, 13, 15
Sum = 7 + 10 + 13 + 15 = 45
Pruned: 3 (too small), 18 (too large)

✗ Naive: Visit all 7 nodes

Edge cases: Empty tree (return 0), all nodes in range (return total sum), no nodes in range (return 0), single-node tree, range spans the entire tree.

2 The Algorithm: BST-Aware DFS with Pruning ▾

We use a recursive DFS that exploits the BST ordering at every node to decide which subtrees to visit. Unlike a plain DFS that always goes both left and right, this one intelligently skips directions where no qualifying nodes can exist.

1 Base case — null node: If root is None, return 0. There's nothing here to sum.
2 Prune LEFT subtree if possible: If root.val <= low, the entire left subtree has values even smaller than root.val, which is already ≤ low — all would be out of range. Skip left entirely; recurse only right.
3 Prune RIGHT subtree if possible: If root.val >= high, the entire right subtree has values larger than root.val, which is already ≥ high — all out of range. Skip right; recurse only left.
4 Node is in range — add and recurse both sides: If low <= root.val <= high, add root.val to the total, and recurse on both left and right children (both sides may still have in-range nodes).
5 Return accumulated sum: Each recursive call returns the sum of in-range values in its subtree. The parent adds its own contribution and returns upward.

🌲 Why This Only Works on a BST

In a plain binary tree, a node with value 10 might have a left child of value 100. We can't assume left means smaller. BSTs guarantee that every value in the left subtree is less than the parent, and every value in the right subtree is greater. This ordering property is what enables the pruning shortcut.

3 Why It Works: The Invariant ▾
🔐 The BST Invariant

At any node with value v:
— Every value in its left subtree is strictly less than v.
— Every value in its right subtree is strictly greater than v.
This holds recursively for every node in the tree.

Why the left-prune is correct: If root.val <= low, then root.val itself is either out of range or exactly at the boundary. Since every left subtree value is strictly less than root.val, and root.val <= low, all left subtree values are strictly less than low — none can be in [low, high]. Skipping is safe.

Why the right-prune is correct: Symmetrically, if root.val >= high, every right subtree value is strictly greater than root.val ≥ high, so all are out of range. Skip right.

Why we recurse both when in range: Even if root.val is inside [low, high], we can't prune either child. The left child could be anywhere from 0 to root.val-1 (some in range, some not), and the right child could be anywhere from root.val+1 to ∞ (same). We must check both.

⚠️ The "in range but still prune" Edge Case

When root.val is exactly equal to low, we add it to the sum — but we can still prune the left subtree (all values < low). Similarly, when root.val == high, we add it but prune the right. The conditions <= low and >= high handle these boundary cases correctly.

Complexity proof: In the worst case (range covers entire tree), we visit every node: O(n). In the best case (range is a single value), we visit O(h) nodes following one BST search path: O(log n) for balanced, O(n) worst case. Space is O(h) for the recursion stack: O(log n) balanced, O(n) skewed.

⚠️ Why not just check "if root.val < low or root.val > high: return 0"?

That would skip the current node but still recurse into both children unnecessarily! The pruning power comes from knowing the subtree direction. When root.val <= low, we don't just skip this node — we skip the entire LEFT subtree and only go right (where larger values live). The separate conditions for left vs. right pruning is the key optimization.

Solution 2 — Iterative BFS/DFS with Stack (same complexity, no recursion)

from collections import deque
def rangeSumBST(root, low, high):
total = 0 ← Running sum
queue = deque([root]) ← BFS queue (or use a stack for DFS)
while queue:
node = queue.popleft() ← Process next node
if not node: continue ← Skip null nodes
if low <= node.val <= high: ← In range: add to total
total += node.val
if node.val > low: ← Left child might have in-range values
queue.append(node.left)
if node.val < high: ← Right child might have in-range values
queue.append(node.right)
return total
💡 Complexity Summary

Both approaches: Time O(n) worst case (range covers full tree), Space O(h) for recursion/stack.
In practice: Pruning makes the recursive approach faster when the range is narrow relative to the tree.
Recursive vs Iterative: Recursive is cleaner and directly expresses the BST logic. Iterative avoids call stack overflow for extremely deep trees.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify It's a Tree Problem — Think Recursion

The moment you see a tree problem, your default mental frame is: "Define what my function returns for a single node, handle the base case (None), and combine left + right results." For this problem: "my function returns the sum of in-range nodes in this subtree."

Step 2: Propose the Naive Solution First

"I could do a plain DFS — visit every node, add to sum if in range. That's O(n) time, O(h) space. It works, but ignores the BST structure." Mention this first — it shows you can solve the problem and sets up the optimization.

Step 3: Exploit the BST Property for Pruning

"Wait — this is a BST. The left subtree is always smaller, the right always larger. So if the current node's value is ≤ low, I know the left subtree can't have in-range nodes. I can skip it entirely and only recurse right." This is the aha moment — demonstrate it clearly.

Step 4: Handle the Three Cases Explicitly

Write out the three mutually informative cases: (1) node.val <= low → only recurse right, (2) node.val >= high → only recurse left, (3) otherwise → add to sum and recurse both. This exhausts all possibilities.

🗣️ What to Say to the Interviewer

"This is a BST, so I can prune entire subtrees.
Base case: null node returns 0.
If node.val <= low: everything in the left subtree is also <= low, so I skip left and recurse only right.
If node.val >= high: everything in the right subtree is also >= high, so I skip right and recurse only left.
Otherwise node.val is in [low, high]: add it and recurse both children.
Time O(n) worst case, but often faster due to pruning. Space O(h) for the call stack."
🔮 Likely Follow-up Questions
  • "Can you do it iteratively?" → Yes — use a stack/queue; push only children that could have in-range descendants (same pruning logic).
  • "What if it were a plain binary tree, not a BST?" → Lose pruning; must visit every node. Still O(n) but with no optimization possible.
  • "What if you needed to find the count, not the sum?" → Same algorithm, just return 1 instead of node.val when in range.
  • "What about a very unbalanced tree (like a linked list)?" → Space becomes O(n) due to call stack depth. Iterative approach avoids stack overflow.
