LeetCode #1669

Merge In Between Linked Lists

Remove nodes at indices a through b in list1, and splice list2 in their place.

Medium Linked List Link Manipulation Time: O(n+m) Space: O(1)
1 Understanding the Problem ▾

You have two linked lists — list1 and list2 — and two indices a and b. The task: remove every node from list1 at positions a, a+1, …, b (inclusive), then insert all of list2 in that gap. Return the modified list1.

Think of it like editing text: you highlight a segment (nodes a to b), delete it, then paste in another document (list2). The surrounding text (nodes before a and after b) remains intact.

💡 Key Insight

You only need to find three anchor nodes: (1) the node just before index a — call it prevA; (2) the node just after index b — call it afterB; (3) the tail of list2. Then two pointer reassignments complete the entire merge. No extra data structures, no copying — just pointer surgery.

✗ Without saving afterB first

If you set prevA.next = list2 before finding afterB, you lose the chain to nodes b+1 onward. The rest of list1 becomes unreachable!

✓ Correct order

Find all three anchors first (prevA, afterB, tail2), THEN rewire. All references are stable until you're ready.

Edge cases: a=1 means prevA is the head (node at index 0); b=last means afterB is null — but the problem guarantees b+1 exists, so afterB is never null.

2 The Algorithm: Three-Anchor Pointer Surgery ▾

Every step is just counting and saving pointers. No comparison logic, no recursion, no extra storage. The work splits cleanly into three traversals followed by two rewires.

1 Walk to prevA (index a−1): Start at list1.head (index 0). Advance a−1 times. Stop and save this node as prevA. Its .next currently points to the first node of the removed segment.
2 Walk to afterB (index b+1): Continue from prevA and walk b−a+2 more steps. You pass through the removed segment and land exactly one node past it. Save this as afterB. Why b−a+2? From index a−1, you need to reach b+1: that gap is (b+1)−(a−1) = b−a+2.
3 Walk list2 to its tail: Start at list2's head and advance while .next != None. Save the last node as tail2. This is the node that will bridge back into list1.
4 Rewire seam 1 — attach list2's head: Set prevA.next = list2. This cuts off the removed segment and makes list2 the continuation of list1 after prevA.
5 Rewire seam 2 — attach list1's remainder: Set tail2.next = afterB. This stitches list2's end to the surviving rest of list1. Done.
🔑 The Walk Count Formula

Starting at index 0, to reach index k, you walk exactly k steps. The two loops:
• Loop 1: range(a - 1) — reaches index a−1 (prevA).
• Loop 2: range(b - a + 2) — from index a−1, reaches index b+1 (afterB).
Verify: (a−1) + (b−a+2) = b+1 ✓

3 Why It Works: The Invariant ▾
🔐 The Core Invariant

After the three traversals and before any rewiring: prevA.next = node at index a, afterB = node at index b+1, and tail2.next = None. All other links in both lists are completely untouched. We hold exactly the "handles" we need — no more, no less.

Correctness of the two rewires:

  • prevA.next = list2 — This atomically: (a) detaches nodes a..b from list1 (they become orphans with no incoming references), and (b) attaches list2's head as the successor of prevA. The orphaned nodes are garbage collected by Python's runtime.
  • tail2.next = afterB — This connects list2's last node to the surviving tail of list1 (node at index b+1 onward). Without this, list1 would be cut in half.

Every node in list1 before index a is untouched. Every node in list1 after index b is untouched. Every node in list2 is untouched except tail2's .next pointer. The two rewires are the minimum necessary — and together they are sufficient.

⚠️ Why Order Matters

You must find afterB before you set prevA.next. If you rewire prevA.next first, the pointer to nodes [a..b+1] is gone — you can no longer walk forward to find afterB. Always: gather all anchor nodes first, rewire second.

4 The Code (Annotated) ▾
def mergeInBetween(list1, a, b, list2):
# ── Phase 1: walk to index a-1 ──────────────
prev_a = list1 start at index 0
for _ in range(a - 1): advance a-1 times → land at index a-1
prev_a = prev_a.next
# ── Phase 2: walk to index b+1 ──────────────
after_b = prev_a reuse pointer; will walk b-a+2 more steps
for _ in range(b - a + 2):(b+1)-(a-1) = b-a+2 steps to reach index b+1
after_b = after_b.next
# ── Phase 3: find tail of list2 ─────────────
tail2 = list2 start at list2's head
while tail2.next: stop when .next is None
tail2 = tail2.next
# ── Phase 4: two rewires ─────────────────────
prev_a.next = list2 seam 1: attach list2 head after prevA
tail2.next = after_b seam 2: attach afterB after list2 tail
return list1 head of list1 is unchanged
💡 Why We Return list1 (not a new node)

The head of list1 never changes — it's still the original first node at index 0. Only internal pointers are rewired. So returning list1 (the original head reference) is correct.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1 — Recognize the Pattern

Signals: "linked list", "remove a range", "insert another list". This is a link manipulation problem — no sorting, no searching, no hashing needed. The key skill is identifying which pointer(s) to change.

Step 2 — Draw It First

Before writing code, draw the before/after on a whiteboard. Ask yourself: "Which nodes do I need to remember?" For any in-place cut-and-splice on a singly linked list, you always need: the node just before the cut, the node just after, and the tail of the inserted segment.

Step 3 — Name Your Anchors

Tell the interviewer: "I'll call them prevA, afterB, and tail2. Three traversals to find them, two pointer assignments to connect them." This shows clear, structured thinking.

Step 4 — Verify the Walk Count

Double-check the loop bounds out loud: "To go from index a−1 to index b+1, I need b+1−(a−1) = b−a+2 steps." This explicit verification prevents off-by-one bugs and shows mathematical precision.

🗣️ What to Say to the Interviewer

"The key insight is that I only need to find three specific nodes: the node just before index a, the node just after index b, and the tail of list2. Once I have those, the merge is exactly two pointer reassignments — prevA.next = list2, and tail2.next = afterB. I'll walk list1 once to find both boundary nodes, then walk list2 once for its tail. Time is O(n+m), space is O(1) — just three pointer variables."
⚠️ Top 3 Interview Mistakes
  • Rewiring before finding afterB — Always gather all anchors first.
  • Off-by-one in loop count — Write out the index math explicitly before coding.
  • Forgetting to find tail2 — Without it, afterB never gets reconnected and list1 is cut in half.

Follow-up the interviewer might ask: "What if list2 could be empty?" (Then just remove nodes a..b without inserting anything — prevA.next = afterB.) "What if a=0?" (Then prevA doesn't exist — you'd need a dummy head. But this problem guarantees a ≥ 1.)

7 Summary & Key Takeaways ▾
Identify all anchor nodes before rewiring any pointer
Three anchors are sufficient: prevA (index a−1), afterB (index b+1), tail2
Two pointer assignments complete the entire splice: prevA.next=list2, tail2.next=afterB
Walk count formula: b−a+2 steps from index a−1 to reach b+1
Removed nodes [a..b] become garbage-collected automatically (no manual cleanup)
Time O(n+m), Space O(1) — optimal for this problem class
🎯 When to Use This Pattern
  • Removing a contiguous range of nodes from a linked list
  • Splicing one linked list into another at a specific position
  • Any "cut and reconnect" operation where you need both boundary nodes
  • Problems requiring O(1) space on linked list structure changes
