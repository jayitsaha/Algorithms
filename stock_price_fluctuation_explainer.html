LeetCode #2034

Stock Price Fluctuation

Design a data structure that tracks stock prices at given timestamps, supports price corrections, and answers min/max queries in O(log n).

Medium Trees Hash Map + TreeMap Update: O(log n) Query: O(log n)
1 Understanding the Problem ▾

You're building a stock price tracker. Prices arrive as (timestamp, price) pairs — but they can arrive out of order, and earlier timestamps can be corrected (updated) with a new price. At any moment you need to answer three queries: "What's the current price?" (the price at the latest timestamp seen), "What's the minimum price ever recorded?", and "What's the maximum price ever recorded?"

💡 Key Insight

The critical challenge is that a price correction can invalidate a previously recorded minimum or maximum. If timestamp 1 had price 10 (the max), and we correct it to price 3, the max is no longer 10. A single tracked variable cannot handle this — you need a structure that lets you remove old values and still answer min/max in O(log n).

Example Sequence

update(1, 10) → {1:10}
update(2, 5) → {1:10, 2:5}
current() = 5 (latest ts=2)
maximum() = 10
minimum() = 5
update(1, 3) → {1:3, 2:5}
maximum() = 5 (old max 10 gone!)

✗ Why a Simple Variable Fails

max_price = 10 after update(1, 10)
After update(1, 3): max_price=10? STALE!
Must rescan ALL prices to find true max.
With n timestamps, that's O(n) per query.
We need corrections to be tracked.

Edge cases: Timestamps can arrive in any order. The same timestamp may be updated multiple times — only the most recent price counts. All timestamps and prices are positive integers. At least one update is guaranteed before any query.

2 The Algorithm: Hash Map + Sorted Map in Tandem ▾

We maintain two data structures simultaneously that together answer all queries efficiently:

1 price_map (dict, timestamp → price): A plain hash map. Gives O(1) lookup: "What is the current price at timestamp T?" Essential for knowing the old price before we overwrite it.
2 sorted_map (SortedDict, price → count): A balanced BST sorted by price. Keys are prices, values are how many timestamps currently hold that price. The first key is always the global minimum; the last key is always the global maximum.
3 update(ts, price): (a) If ts already has a price, look it up in price_map, decrement its count in sorted_map (delete if count → 0). (b) Record new price in price_map. (c) Increment count of new price in sorted_map. (d) Update max_ts if ts is larger.
4 current(): Return price_map[max_ts]. We track max_ts separately on every update — O(1).
5 minimum() / maximum(): Return first / last key of sorted_map. Because SortedDict is a balanced BST, these are O(log n) operations.
🔑 Why Store price → count Instead of price → set_of_timestamps?

Multiple timestamps can share the same price (e.g., ts=2 and ts=5 both have price 10). A count is simpler: it tells us how many timestamps currently hold this price. When count reaches 0, we know no timestamp has this price anymore and we safely remove it from the sorted map. No need to track which timestamps — just whether any remain.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At all times: sorted_map[p] = the number of timestamps whose current price is exactly p. The keys of sorted_map are exactly the set of prices that are currently "live" (assigned to at least one timestamp). The first key is the global minimum; the last key is the global maximum — always, immediately after any update.

How each update preserves this invariant:

a Look up old = price_map[ts]. Decrement sorted_map[old]. If 0, delete it. This removes the old price contribution — the invariant count for old decreases by 1, reflecting that one fewer timestamp now has it.
b Set price_map[ts] = price and increment sorted_map[price]. This adds the new price contribution — the invariant count for price increases by 1.

After these two steps, every key in sorted_map has exactly the right count, and every live price is a key. The min and max are correct by construction.

⚠️ Most Common Bug: Forgetting to Remove the Old Price

If you only do sorted_map[price] += 1 without first decrementing the old price's count, the sorted_map overcounts — the old price remains as a key even after it's been replaced. The maximum() query would return stale data. Always look up and handle the old price first.

Edge: same timestamp, same price repeated. update(5, 10) then update(5, 10) again: old=10, sorted_map[10] -=1 then sorted_map[10] +=1 → net no change. Correct — the state hasn't changed.

4 The Code (Annotated) ▾

Solution 1 — Hash Map + SortedDict (Optimal, Interview Pick)

from sortedcontainers import SortedDict ← Balanced BST: O(log n) insert/delete, O(1) min/max key
class StockPrice:
def __init__(self):
self.prices = {} ← timestamp → current price (O(1) lookup)
self.sorted_map = SortedDict() ← price → count of timestamps with this price
self.max_ts = 0 ← Track latest timestamp to answer current()
def update(self, timestamp: int, price: int) -> None:
if timestamp in self.prices: ← Is this a correction of an existing timestamp?
old = self.prices[timestamp] ← What was the old price?
self.sorted_map[old] -= 1 ← One fewer timestamp has this price now
if self.sorted_map[old] == 0: ← No timestamps left with this price?
del self.sorted_map[old] ← Remove it from the BST entirely
self.prices[timestamp] = price ← Record new price for this timestamp
self.sorted_map[price] = self.sorted_map.get(price, 0) + 1← Add/increment in BST
self.max_ts = max(self.max_ts, timestamp)← Update latest timestamp tracker
def current(self) -> int:
return self.prices[self.max_ts] ← O(1): direct hash map lookup with tracked max_ts
def maximum(self) -> int:
return self.sorted_map.peekitem(-1)[0]← Last key of BST = global max price
def minimum(self) -> int:
return self.sorted_map.peekitem(0)[0] ← First key of BST = global min price
💡 Complexity Summary

update(): O(log n) — SortedDict insert/delete is O(log n)
current(): O(1) — direct hash map lookup, max_ts tracked
minimum()/maximum(): O(log n) — BST first/last key
Space: O(n) — n distinct timestamps in price_map; at most n distinct prices in sorted_map

Solution 2 — Two Heaps with Lazy Deletion (no external library)

import heapq
class StockPrice:
def __init__(self):
self.prices = {} ← timestamp → current price (source of truth)
self.min_heap = [] ← (price, ts) — Python's heapq is a min-heap
self.max_heap = [] ← (-price, ts) — negate price for max-heap
self.max_ts = 0
def update(self, timestamp, price): ← Lazy push: don't remove old entries
self.prices[timestamp] = price ← Overwrite; old entries in heaps are now stale
heapq.heappush(self.min_heap, (price, timestamp))
heapq.heappush(self.max_heap, (-price, timestamp))
self.max_ts = max(self.max_ts, timestamp)
def minimum(self): ← Lazy deletion: pop stale entries until valid
while self.min_heap[0][0] != self.prices[self.min_heap[0][1]]:
heapq.heappop(self.min_heap) ← This entry's price was overwritten — skip it
return self.min_heap[0][0]
def maximum(self): ← Same lazy deletion for max heap
while -self.max_heap[0][0] != self.prices[self.max_heap[0][1]]:
heapq.heappop(self.max_heap)
return -self.max_heap[0][0]
⚠️ Lazy Deletion Explained

Python's heapq does not support O(log n) arbitrary removal. When a price is updated, we leave the stale (old price, timestamp) entry in the heap. At query time, we check: does the heap's top entry's price match prices[ts]? If not, it's stale — pop it and check the next. This is O(log n) amortized because each entry is pushed and popped at most once.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify the requirements for each operation

update: needs to handle overwrites → hash map for O(1) lookup. current: needs latest timestamp → track max_ts. min/max: need sorted order that survives updates → sorted structure, not a simple variable.

Step 2: Recognize the "stale data" problem

Say to interviewer: "The tricky part is that when a price is corrected, the old price might have been the minimum or maximum. I can't just keep a variable — I need a structure that lets me remove the old value." This demonstrates you've spotted the core challenge.

Step 3: Propose two solutions

Option A: SortedDict (price → count) — clean O(log n) per operation, easy to reason about. Option B: Two heaps with lazy deletion — uses only standard heapq, amortized O(log n). Prefer Option A for interviews if allowed.

Step 4: Explain the count trick

If the interviewer asks why count: "Multiple timestamps can have the same price. A count tells me how many are still live. When count reaches 0, I remove the price from the sorted structure — it's no longer represented anywhere."

🗣️ What to Say to the Interviewer

"I'll use two structures together.
First, a hash map from timestamp to current price — O(1) lookup.
Second, a SortedDict (balanced BST) mapping price to how many timestamps have it.
For update: if the timestamp exists, decrement old price count in the BST (delete if 0),
then set the new price and increment its count. Also track max_ts for current().
For minimum()/maximum(): first/last key of the sorted BST. O(log n) for all ops."
🔮 Likely Follow-up Questions
  • "What if sortedcontainers is not available?" → Two heaps with lazy deletion (Solution 2)
  • "What if prices are bounded in [1, 10^6]?" → Fenwick tree / segment tree for O(1) min/max
  • "Design the k-th smallest price query too?" → SortedList with order statistics
  • "Thread-safe version?" → Add a lock around update/query critical sections

7 Summary & Key Takeaways ▾
Two structures in tandem: hash map for O(1) ts→price lookup; sorted map for O(log n) min/max queries
Store price→count in the sorted map so one price shared by multiple timestamps is counted correctly
On update: always remove old price count first, then add new price — never add without first subtracting
Track max_ts explicitly for O(1) current() — do not query the sorted map for this
Lazy deletion (heap approach) is valid when no BST library is available: push freely, pop stale entries at query time
This pattern applies to any "maintain a sorted set under updates" problem

🎯 When to Recognize the Hash Map + Sorted Structure Pattern
  • You need O(1) lookup by key AND sorted-order queries (min/max/rank) on values
  • Values can be corrected/updated — not just inserted
  • Multiple keys can share the same value (use frequency count in sorted structure)
  • "Design a class with update and min/max" is the canonical signal for this pattern
