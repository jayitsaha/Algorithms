LeetCode #2536

Increment Submatrices by One

Given an n×n matrix of zeros and queries [r1,c1,r2,c2], add 1 to every cell in the submatrix corner (r1,c1)→(r2,c2) for each query. Return the final matrix.

Medium Prefix Sum 2D Difference Array Time: O(q + n²) Space: O(n²)
1 Understanding the Problem ▾

You start with an n×n grid of zeros. You receive a list of "paint rectangle" operations — each one increments every cell inside a specified axis-aligned rectangle by 1. After applying all operations, what does the grid look like?

💡 Key Insight

The naive approach loops over every cell inside each rectangle — O(n²) per query, O(q·n²) total. For large inputs, this times out. The 2D Difference Array reduces each query to just 4 corner updates, then a single O(n²) prefix-sum sweep reconstructs the full matrix. Total: O(q + n²).

✗ Naive — O(q × n²)

For each query [r1,c1,r2,c2]:
  for r in r1..r2:
    for c in c1..c2:
      mat[r][c] += 1

n=500, q=500 → 62.5M ops

✓ 2D Diff Array — O(q + n²)

For each query: 4 writes (O(1)).
Then one n×n prefix pass.

n=500, q=500 → 250,500 ops
250× faster!

Concrete example: n=3, query [0,0,1,1]. The naive approach touches 4 cells. The 2D difference approach writes to just 4 corner positions in a (n+1)×(n+1) buffer, then a prefix-sum sweep produces the same result.

Edge cases: A single-cell query [r,r,c,c] works correctly — the +1 cancels perfectly at its own borders. Queries that touch the last row/column write to index n, which is why we allocate (n+1)×(n+1).

2 The Algorithm: 2D Difference Array + Prefix Sum ▾

The 2D difference array is the natural 2D extension of the 1D difference array (LeetCode #370). Instead of marking start and end+1 of a range, we mark all four corners of a rectangle. A two-pass prefix sum then recovers the final matrix.

1 Allocate diff[n+1][n+1], all zeros. The extra row and column are "guard" indices — they absorb the r2+1 and c2+1 writes when queries touch the matrix boundary, so we never go out of bounds.
2 For each query [r1,c1,r2,c2], apply 4 corner updates:
diff[r1][c1] += 1 — mark rectangle start
diff[r1][c2+1] -= 1 — cancel to the right
diff[r2+1][c1] -= 1 — cancel below
diff[r2+1][c2+1] += 1 — fix double-cancel at far corner
3 Row-wise prefix sum: For each row r in [0,n): scan left-to-right, diff[r][c] += diff[r][c-1]. After this pass, each cell contains the net horizontal contribution.
4 Column-wise prefix sum: For each column c in [0,n): scan top-to-bottom, diff[r][c] += diff[r-1][c]. After this pass, each cell holds its final answer.
5 Return diff[:n][:n] — the top-left n×n submatrix, discarding the guard row/column.

🔑 Why Four Corners? The Inclusion-Exclusion Idea

Adding +1 at top-left makes it propagate right and down via prefix sums. The -1s at the adjacent borders stop propagation at the rectangle edges. The +1 at the far corner fixes the double-cancellation — the same pattern as the inclusion-exclusion formula for 2D rectangle areas: area(A∪B) = area(A) + area(B) - area(A∩B).

3 Why It Works: The Invariant ▾
🔐 The Invariant

After all queries are encoded and the 2D prefix-sum sweep is complete, diff[r][c] equals exactly the number of queries whose rectangle contained cell (r,c). This holds regardless of query order or overlapping regions.

Mathematical proof sketch: Consider one query with rectangle [r1,c1]→[r2,c2]. We set diff[r1][c1]+=1, diff[r1][c2+1]-=1, diff[r2+1][c1]-=1, diff[r2+1][c2+1]+=1. After the row-wise prefix sum, for row r in [r1, r2]: cell (r,c) accumulates +1 for c in [c1, c2] (the -1 at c2+1 halts rightward propagation). For row r outside [r1, r2], all contributions cancel. After the column-wise prefix sum, the +1 is present only for rows in [r1, r2] (the -1 at r2+1 halts downward propagation). This means exactly the cells inside the rectangle receive +1. Overlapping queries simply add their contributions — prefix sums are linear.

⚠️ Common Mistake: Doing Both Prefix Sums in One Pass

You must complete the entire row-prefix sweep before starting the column-prefix sweep, or vice versa. Interleaving the two dimensions in a single doubly-nested loop produces incorrect results. The 2D prefix sum is separable, but only when each dimension is fully resolved before the other begins.

Overlapping queries: If two rectangles overlap at cell (r,c), both independently encode +1 via their corner markers. The prefix sum naturally sums them — cell (r,c) ends up with 2, which is correct. No special handling needed.

Why (n+1)×(n+1)? When r2=n-1 (last row), we write to diff[n][c1] and diff[n][c2+1] — index n, which requires the guard row. Similarly for c2=n-1. Without the extra dimension, these writes cause an IndexError. We simply ignore row n and column n in the final output by slicing.

4 The Code (Annotated) ▾

Solution 1 — 2D Difference Array (Optimal, Interview Pick)

def rangeAddQueries(n, queries): ← n×n grid size, list of [r1,c1,r2,c2] queries
diff = [[0]*(n+1) for _ in range(n+1)] ← (n+1)×(n+1): guard indices absorb out-of-bounds writes
for r1,c1,r2,c2 in queries: ← each query is O(1): just 4 writes
diff[r1][c1] += 1 ← top-left: start of rectangle
diff[r1][c2+1] -= 1 ← top-right+1: stop rightward propagation
diff[r2+1][c1] -= 1 ← bottom+1-left: stop downward propagation
diff[r2+1][c2+1] += 1 ← far corner: fix the double-cancellation
for r in range(n): ← Pass 1: row-wise prefix sums
for c in range(1, n): ← start at c=1, column 0 is already its own base
diff[r][c] += diff[r][c-1] ← accumulate left-to-right in each row
for c in range(n): ← Pass 2: column-wise prefix sums
for r in range(1, n): ← start at r=1, row 0 is already its own base
diff[r][c] += diff[r-1][c] ← accumulate top-to-bottom in each column
return [row[:n] for row in diff[:n]] ← slice away guard row/col: return n×n result
⚠️ Why Allocate (n+1)×(n+1) Instead of Checking Bounds?

When a query rectangle touches the last row (r2=n-1), we need to write diff[n][c]. Rather than adding if r2+1 < n+1 checks everywhere (which people often forget at corners), we simply allocate one extra row and column. The guard values are never read back — we slice them off with diff[:n][:n] at the end. This is cleaner and avoids a class of subtle bugs.

Solution 2 — Naive Brute Force (O(q·n²))

def rangeAddQueries_naive(n, queries):
mat = [[0]*n for _ in range(n)] ← n×n result
for r1,c1,r2,c2 in queries: ← O(q) outer loop
for r in range(r1, r2+1): ← O(n) rows
for c in range(c1, c2+1): ← O(n) cols — O(n²) per query
mat[r][c] += 1 ← direct increment — correct but too slow
return mat
💡 Complexity Summary

2D Difference Array: Time O(q + n²), Space O(n²). q queries × O(1) corner writes + one O(n²) prefix pass.
Naive Brute Force: Time O(q·n²), Space O(n²). For n=500, q=500: ~62.5M operations vs ~250k for the diff array approach — a 250× speedup.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: State the Brute Force

"The direct approach: for each query, loop over every cell in the rectangle and increment it. This is O(q·n²) — too slow for large inputs. I need a way to process queries in O(1) each."

Step 2: Recognize the Pattern

"This is 'batch range updates, then read all values' — the classic scenario for a difference array. In 1D I'd mark start and end+1. In 2D I mark four corners. The prefix sum then reconstructs the final values."

Step 3: Explain the 4 Corners

"The four updates use inclusion-exclusion: +1 at top-left starts propagation; -1 at two adjacent borders stops propagation at the rectangle edges; +1 at the far corner undoes the double-cancellation. Two prefix-sum passes (one per dimension) recover the result."

Step 4: Mention the Guard Row/Column

"I allocate (n+1)×(n+1) so boundary queries don't go out of bounds. The guard row and column are safe to write to — we just slice them off at the end."

🗣️ What to Say to the Interviewer

"This is a batch range-update problem — the signal is 'apply many rectangle increments, then read the full matrix.' That pattern maps perfectly to a 2D difference array.
I'll create a (n+1)×(n+1) diff array. For each query [r1,c1,r2,c2], I apply 4 O(1) corner updates: +1 at (r1,c1), -1 at (r1,c2+1), -1 at (r2+1,c1), +1 at (r2+1,c2+1).
Then I do two prefix-sum sweeps: first across all rows, then down all columns.
The top-left n×n slice is the answer. Total: O(q + n²) time, O(n²) space."
🔮 Likely Follow-up Questions
  • "What if queries arrive online, and you need to query a single cell immediately?" → 2D Binary Indexed Tree (Fenwick), O(log²n) per update/query.
  • "What if query values are arbitrary (not just +1)?" → Same approach: replace +1 with +val and -1 with -val.
  • "Can you solve this in-place?" → Yes — write the diff to the n×n result matrix itself (using it as the diff buffer), then prefix-sum in-place.
  • "What is the 1D version of this problem?" → LeetCode #370 Range Addition — same idea with 2 markers instead of 4.

7 Summary & Key Takeaways ▾
2D Difference Array = 1D difference array applied in both row and column dimensions
Each query costs O(1): exactly 4 corner writes using the inclusion-exclusion pattern
Allocate (n+1)×(n+1) to safely absorb boundary writes at index n
Two separate prefix-sum sweeps recover the matrix: rows first, then columns
Total complexity O(q + n²) — the two terms are independent; either can dominate
ApproachTimeSpaceNotes
Naive Brute Force O(q·n²) O(n²) TLE for large n, q; intuitive starting point
2D Difference Array ✓ O(q + n²) O(n²) Optimal; the interview answer
2D BIT (online queries) O(q log²n) O(n²) For online query/update mix; overkill here
🎯 When to Recognize the 2D Difference Array Pattern
  • "Add/subtract a value to all cells in an axis-aligned rectangle" — batch, offline
  • "Many range updates, then read all final values" — do NOT need to query after each update
  • Matrix starts at a known state (usually zeros), modified by rectangular operations
  • Need O(1) per update rather than O(area) — for large rectangles, this matters greatly
