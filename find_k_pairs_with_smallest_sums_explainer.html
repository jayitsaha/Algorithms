LeetCode #373

Find K Pairs with Smallest Sums

Given two sorted arrays nums1 and nums2, return the k pairs (u, v) with the smallest sums, where u is from nums1 and v is from nums2.

Medium Heaps Min Heap BFS Time: O(k log k) Space: O(k)
1 Understanding the Problem ▾

You have two sorted arrays. You can pick one number from each — that's a "pair," and its cost is the sum of those two numbers. You want the k cheapest pairs overall.

Example: nums1 = [1, 7, 11], nums2 = [2, 4, 6], k = 3. Visualise all possible pair sums as a grid:

Green cells = the 3 pairs we want. Output: [[1,2], [1,4], [1,6]]

💡 Key Insight

Because both arrays are sorted, the smallest sums are concentrated in the top-left corner of the grid. We explore this grid outward from that corner — always extracting the globally smallest remaining candidate — using a min-heap as our directed frontier.

Edge cases: k may exceed m × n total pairs (return all). Either array may have a single element. Duplicate values in either array are fine — algorithm handles them naturally.

2 The Algorithm: Min Heap BFS ▾

Think of the grid as a search space, with the top-left cell (0, 0) as the starting point. The natural "neighbors" of cell (i, j) are (i+1, j) and (i, j+1) — both have larger or equal sums. A priority queue (min-heap) lets us always expand the neighbor with the smallest sum next. This is Dijkstra-style BFS on the grid.

The key deduplication trick: seed only column 0 for every row, then from each cell only advance in the j-direction. This eliminates duplicate visits.

1 Seed the heap: For each row i in nums1 (up to min(k, m) rows), push (nums1[i] + nums2[0], i, 0). This gives every row its initial minimum-cost entry point.
2 Pop and record: Extract the minimum-sum entry (s, i, j) from the heap. Add [nums1[i], nums2[j]] to the result.
3 Advance along j: If j+1 < len(nums2), push (nums1[i] + nums2[j+1], i, j+1). This moves the frontier one step to the right within the same row.
4 Repeat until we have k pairs or the heap is empty.
🔑 Why Only Push (i, j+1) and Not (i+1, j)?

We already seeded every row at column 0. So (i+1, 0) is already in the heap. If we also pushed (i+1, j), we'd visit (i+1, j) twice — once from initialization and once from the push. By restricting to the j-direction only, each cell is visited exactly once and the algorithm remains O(k log k).

✗ Brute Force (O(mn log mn))

Generate all m×n pairs.
Sort by sum.
Take first k.

Wasteful if k is small and arrays are huge (e.g. k=5, m=n=10,000).

✓ Min Heap BFS (O(k log k))

Explore lazily — only the frontier.
At most k pops, each O(log k).
Heap size stays bounded at O(k).

Only computes what we need.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At any moment, the min-heap contains exactly the current frontier: one entry per row, pointing to that row's smallest sum not yet extracted. Every cell that could potentially be the next global minimum is represented. No smaller unvisited cell exists outside the heap.

Why this guarantees correctness:

  • Initially, column 0 of every row is seeded — the smallest possible sum for each nums1 element.
  • When we pop row i at column j, we immediately push column j+1 of row i. Since nums2 is sorted, nums2[j+1] ≥ nums2[j], so the new entry cannot be smaller than what we just popped.
  • The heap always holds exactly one "next best candidate" per active row. The globally smallest among these is always at the top.
⚠️ Common Mistake: Not Capping Seeds at k

When nums1 has 10,000 elements and k is only 3, seeding all 10,000 rows wastes time and memory. The k-th result can never involve row index ≥ k (since even picking the smallest j=0 for that row gives a sum larger than what's achievable in the first k rows). Always use range(min(k, len(nums1))).

Termination guarantee: After exactly k successful pops (or heap exhaustion), all remaining heap entries have sums ≥ the last extracted sum. Discarding them is correct.

4 The Code (Annotated) ▾

Solution 1 — Min Heap BFS (Optimal, O(k log k) time)

import heapq
def kSmallestPairs(nums1, nums2, k):
if not nums1 or not nums2: return [] ← Guard: empty input → no pairs possible
heap = [] ← Min-heap of (sum, row_i, col_j) tuples
for i in range(min(k, len(nums1))): ← Seed: one entry per row at column 0
heapq.heappush(heap, (nums1[i]+nums2[0], i, 0)) ← Smallest sum for row i
result = []
while heap and len(result) < k: ← Collect until k pairs or heap empty
s, i, j = heapq.heappop(heap) ← Extract globally smallest remaining pair
result.append([nums1[i], nums2[j]]) ← Record this pair in the answer
if j + 1 < len(nums2): ← If there's a next column for this row...
heapq.heappush(heap, (nums1[i]+nums2[j+1], i, j+1)) ← Advance frontier rightward
return result
⚠️ Why store (sum, i, j) not just the sum?

We need i to look up nums1[i] and build the answer pair. We need j to compute the next push (i, j+1). The heap tuple encodes everything needed. Storing only the sum would lose our position in the grid entirely.

Solution 2 — Brute Force Sort (for understanding, O(mn log mn))

def kSmallestPairs_brute(nums1, nums2, k):
pairs = [] ← Will hold every possible pair
for u in nums1: ← O(m) outer
for v in nums2: ← O(n) inner: m×n total pairs generated
pairs.append((u + v, [u, v])) ← Track (sum, pair) for sorting
pairs.sort() ← O(mn log mn) — dominates
return [p for _, p in pairs[:k]] ← Take first k by sum
💡 Complexity Summary

Min Heap BFS: Time O(k log k) — k iterations, each O(log k) heap op. Space O(k) — heap holds at most min(k, m) + k entries.
Brute Force: Time O(mn log mn), Space O(mn) — impractical for large arrays.
When k is tiny: Min Heap BFS is orders of magnitude faster.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize "Sorted Arrays + Find Top K"

When you see sorted input and "find k smallest," think: heap. Sorted order already gives structure — you don't need to scan randomly. A heap lets you greedily pick the next global minimum without generating all pairs first.

Step 2: Propose Brute Force — Identify the Waste

"I could generate all m×n pairs, sort by sum, and return the first k. That's O(mn log mn). But if k is small and m, n are 10,000, I'm generating 100 million pairs for k=5 results. Can I be lazier?"

Step 3: Visualize the Grid, Think BFS

"Lay out sums as a 2D grid: row = index in nums1, column = index in nums2. Because both arrays are sorted, the grid is monotonically increasing right and down. The global minimum is always top-left. I need to visit cells in sum order — that's BFS with a priority queue."

Step 4: Explain the Deduplication Trick

"If from (i, j) I push both (i+1, j) and (i, j+1), I'd visit (i+1, j+1) twice. Solution: seed only column 0 for each row, then from (i, j) push only (i, j+1). Each cell is reached exactly once — from above by initialization, and to the right by the loop."

🗣️ What to Say to the Interviewer

"Both arrays are sorted, so I'll model this as a grid where cell (i,j) has sum nums1[i]+nums2[j].
The minimum is at (0,0) and the grid grows larger going right and down.
I'll use a min-heap seeded with column 0 for each row: (nums1[i]+nums2[0], i, 0).
Each pop gives the current smallest pair. I record it and push (i, j+1) to advance the frontier.
Time is O(k log k): k iterations, each O(log k). Space is O(k) for the heap."
🔮 Likely Follow-up Questions
  • "What if k > m×n?" → Return all pairs; heap exhausts naturally before k pops.
  • "Duplicates in the arrays?" → Algorithm handles correctly; sums may tie but that's fine.
  • "Can you handle 3 sorted arrays?" → Same pattern, more indices. Seed (i,j,k)=0 for each pair.
  • "Related: Kth Smallest in Sorted Matrix?" → Same grid BFS, same heap (#378).

7 Summary & Key Takeaways ▾
Min Heap BFS is the canonical pattern for finding k smallest across multiple sorted sequences
Seed column 0 for each row — every row's minimum is in the heap from the start
From (i, j) push only (i, j+1) — prevents duplicate visits, keeps each cell visited once
Cap seeds at min(k, len(nums1)) — rows beyond k can never contribute to the first k results
Time O(k log k), Space O(k) — far better than brute-force O(mn log mn)
ApproachTimeSpaceNotes
Brute Force (generate all pairs) O(mn log mn) O(mn) Correct; wasteful for large m, n
Min Heap BFS ✓ O(k log k) O(k) Optimal; lazy evaluation of the grid
