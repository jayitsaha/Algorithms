LeetCode #271

Encode and Decode Strings

Design an algorithm to encode a list of strings to a single string, then decode it back — losslessly, for any string content including special characters.

Medium Design Length Prefix Encode O(n) Decode O(n) Space O(n)
1 Understanding the Problem ▾

Imagine you need to transmit a list of arbitrary strings over a channel that only accepts a single string. On the receiving end, you must reconstruct the exact original list — every word, every character, in the right order. The hard part: the strings can contain any character whatsoever, including whatever delimiter you might try to use.

💡 Key Insight

You cannot use a single special delimiter character (comma, pipe, semicolon) because that character might appear inside the strings. The solution is to encode each string's length before the string itself — then the decoder reads exactly that many characters, regardless of their content. No delimiter needed inside the data.

✗ Delimiter Fails

input: ["a,b", "c"]
encoded: "a,b,c"
decoded: ["a","b","c"]
WRONG — split on ","
lost original boundary

✓ Length Prefix Works

input: ["a,b", "c"]
encoded: "3#a,b1#c"
decode: read 3 → "a,b"
read 1 → "c" ✓ Perfect!

Edge Cases: empty list [] → empty string ""; empty string as an element ["", "a"] → "0#1#a"; a word that itself looks like a length prefix (e.g., "3#hi") still decodes correctly because we use exact byte counting, not searching.

🔑 The Mental Model: Network Framing

HTTP chunked transfer encoding, binary file formats (RIFF, PNG), and network protocols all use the same idea: send the frame length before the frame payload. The receiver counts bytes — never searches for a delimiter. This is called length-prefix framing, and it's one of two standard solutions to the serialization delimiter problem. The other is escape character encoding (see Section 4).

2 The Algorithm: Length-Prefix Framing ▾

Each word is wrapped in a header: its character count as a decimal string, then #, then the raw word. Headers are concatenated directly — no separator between words needed.

Encoding ["lint", "code", "you"]
"lint" → 4 # lint
"code" → 4 # code
"you" → 3 # you
Result: 4#lint4#code3#you
E1 Encode — for each string s: emit f"{len(s)}#{s}". Join everything into one output string. The length field tells the decoder exactly how long the following data is.
D1 Decode — initialize: Set read cursor i = 0 and output list result = []. The cursor always points to the start of the next word's length-prefix header.
D2 Find the separator: Scan forward from i until you hit #. Call that position j. The substring s[i:j] contains only decimal digits and represents the word length.
D3 Parse and slice: Convert s[i:j] to integer n. Then extract exactly n characters starting right after #: word = s[j+1 : j+1+n]. Append to result.
D4 Advance cursor: i = j + 1 + n — skip past the header (j+1) and the data (+n). Repeat D2–D4 until i equals len(s).
🔑 Why Content Cannot Corrupt the Header

The length field consists only of digit characters (0–9). The # separator is guaranteed to be non-digit. So scanning forward from i for the first # is unambiguous — digits can never be confused with the separator. Once we have n, we use exact byte counting, so we never look inside the data for anything.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At the start of every decode loop iteration, cursor i points to the first character of a length-prefix header. After one iteration — reading the header and extracting the word — i is advanced to the start of the next header (or to len(encoded) if we're done). This invariant holds throughout, guaranteeing termination and correctness.

Base case: Before the loop starts, i = 0. The encoder always starts with a length-prefix header at position 0, so the invariant holds initially.

Inductive step: Each word's encoded form is len(word) + len("#") + len(word) = len(str(n)) + 1 + n characters. After setting i = j + 1 + n, we've consumed exactly this many characters. The next character (if any) is the start of the following word's header — invariant preserved.

⚠️ Edge Case: Empty String

Empty string "" encodes as "0#". During decode: j points to #, n = 0, slice s[j+1:j+1] = "" — correct. i advances by len("0") + 1 + 0 = 2. No special handling needed anywhere.

Tricky case — word that looks like a header: If a word is "3#hi", it encodes as "4#3#hi". Decode from position i: find first # starting at i → hits the # after 4, so n = 4. Then slice 4 characters: "3#hi". Exactly correct — the embedded # inside the data is never treated as a separator because we jumped past it using byte counting.

4 The Code (Annotated) ▾

Solution 1 — Length-Prefix (Optimal, Interview Pick)

class Codec:
def encode(self, strs: list[str]) -> str: ← takes a list, returns one string
return "".join( ← concatenate all chunks directly
f"{len(s)}#{s}" ← each chunk: decimal length + '#' + raw data
for s in strs ← one chunk per string in the input list
)
def decode(self, s: str) -> list[str]: ← takes one string, returns the list
result, i = [], 0 ← output accumulator + read cursor
while i < len(s): ← loop until all chars consumed
j = s.index('#', i) ← find '#' starting at i (NOT from 0!)
n = int(s[i:j]) ← digits between i and j = word length
result.append(s[j+1 : j+1+n]) ← slice exactly n chars after '#'
i = j + 1 + n ← advance past this word's header + data
return result ← reconstructed list
⚠️ Critical: s.index('#', i) not s.find('#')

Using s.find('#') (no start argument) would always find the first # in the entire string — which after the first word is inside already-decoded data. s.index('#', i) starts the search at i, finding only the current word's separator. This is the single most important implementation detail.

Solution 2 — Escape Character (Alternative, More Complex)

def encode(self, strs): ← escape all '#' in data, join with bare '#'
return "#".join( ← bare '#' = word boundary
s.replace('#', '##') ← escape embedded '#' as '##'
for s in strs
) + "#" ← trailing '#' marks end of list
def decode(self, s): ← scan for non-doubled '#'
result, current, i = [], [], 0
while i < len(s):
if s[i] == '#' and s[i+1:i+2] == '#': ← '##' → escaped '#' in data
current.append('#'); i += 2
elif s[i] == '#': ← bare '#' → word boundary
result.append(''.join(current)); current = []; i += 1
else: current.append(s[i]); i += 1
return result
💡 Complexity Summary

Length-Prefix Encode: O(total chars) — each character appended once.
Length-Prefix Decode: O(total chars) — each character visited once by index() and once by slice.
Space: O(n) for the encoded/decoded output. Length-prefix adds at most 2–3 bytes overhead per word (for the number and #).
Escape approach: Same O(n) but worst-case doubles the string size (all # characters).

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Spot the Hidden Difficulty

"This looks trivial — just join with a comma. But then I realize: what if the strings contain commas? What character is safe to use as a delimiter? The answer is: no single character is guaranteed safe. That's the real challenge."

Step 2: Propose Naïve Approach and Expose Its Failure

"Option A: join with comma. Fails if strings contain commas. Option B: use a non-printable ASCII character or Unicode sentinel. Fragile — what if strings can contain any Unicode? We need a provably correct approach for any content."

Step 3: Apply the Length-Prefix Insight

"Instead of marking where a word ends, I'll mark how long it is. Prefix each word with its length and a separator. The decoder doesn't need to search for a boundary — it reads exactly the right number of bytes."

Step 4: Demonstrate with a Tricky Example

Walk through: ["3#a", "b"] → encoded "3#3#a1#b". Decode: from i=0, find # at j=1, n=3, slice chars 2-4 = "3#a". Advance to 5, find # at 6, n=1, slice char 7 = "b". This proves it handles embedded # correctly.

🗣️ What to Say to the Interviewer

"My approach is length-prefix encoding. For each string, I prepend its length as a decimal number followed by a '#' character, then append the raw string. Joining all these chunks gives the encoded output.

To decode: maintain a cursor i. Find the '#' starting at i — that gives me the length n. Slice exactly n characters after the '#'. Advance i by the header size plus n. Repeat until i reaches the end.

The key correctness point: I always call index('#', i) not find('#') — starting the search from the current cursor position, not from 0. This ensures I don't accidentally land on a '#' inside already-decoded data.

Edge cases: empty strings encode as '0#' and decode back perfectly. Strings containing digits and '#' are handled because once I have n, I use byte counting, not searching."
🔮 Likely Follow-up Questions
  • "What if strings contain null bytes (binary data)?" → Length-prefix still works — we count chars/bytes, never search for a delimiter in the data
  • "Can you do it without storing the length?" → Yes: escape approach — double any '#' in the data, use bare '#' as delimiter
  • "What's the overhead of your encoding?" → O(log n) extra chars per word (for the decimal length), so O(W log max_len) total overhead
  • "How would you handle streaming / very large strings?" → Fixed-width binary length field (4 bytes big-endian int) is more efficient for binary protocols

7 Summary & Key Takeaways ▾
No single delimiter character is safe — strings can contain any character
Length-prefix encoding: store the size of each record before the record itself
Decode by counting bytes, not searching for a delimiter — O(n) and provably correct
Always start index('#', i) from the cursor position i, not from 0
Empty strings encode as "0#" and decode back cleanly — no special casing needed

🎯 When to Recognize the Length-Prefix Pattern
  • "Serialize/deserialize a list of arbitrary strings" → Length prefix
  • "Encode a collection to a single string without a safe delimiter" → Length prefix
  • "Store variable-length records in a file/stream" → Length prefix (binary: 4-byte int)
  • Any problem where data can contain any character → count bytes, don't search for delimiters
