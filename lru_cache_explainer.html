LeetCode #146

LRU Cache

Design a data structure that follows the Least Recently Used (LRU) cache eviction policy with O(1) get and put.

Medium Design Hash Map + Doubly Linked List Time: O(1) Space: O(capacity)
1 Understanding the Problem ▾

An LRU Cache is a fixed-capacity store. When it's full and you insert something new, the item that was used least recently — either read or written — gets evicted. "Recently used" is refreshed on every get and every put.

💡 Key Insight

We need two capabilities simultaneously: O(1) lookup by key (a hash map gives that), and O(1) reordering by recency (a doubly-linked list gives that, given a node pointer). Combining them: the hash map stores pointers to the DLL nodes, so we can find any node instantly and reorder in O(1). Neither structure alone can do both.

✓ What we need

get(key) → O(1) lookup
put(key,val) → O(1) insert
evict LRU → O(1) removal
refresh recency → O(1) reorder

✗ Why naive fails

Array: O(n) shift to reorder
Singly linked: O(n) to unlink
Dict only: no ordering info
Sorted: O(log n) reorder

Concrete example (capacity=2):

put(1,1) → cache: {1}
put(2,2) → cache: {1,2}
get(1) → returns 1; key 1 is now MRU
put(3,3) → cache full! LRU = key 2 (not touched since put). Evict 2. cache: {1,3}
get(2) → returns -1 (was evicted!)
get(3) → returns 3

Edge cases: get on missing key → -1. put on existing key → update value AND refresh recency (don't duplicate!). Capacity 1 evicts on every new key insert.

2 The Algorithm: Hash Map + Doubly Linked List ▾

Two data structures working together. The doubly-linked list (DLL) keeps nodes ordered by recency — head side is MRU (most recently used), tail side is LRU. The hash map maps each key to its DLL node for direct O(1) access.

Hash Map: key → Node pointer

O(1) jump to any node by key. Without this, finding a node in the DLL by key is O(n).

Doubly-Linked List: MRU ←[head]…[tail]→ LRU

Two dummy sentinel nodes (head, tail) eliminate all null-check edge cases. Move-to-front and remove-from-back are always O(1) with direct node pointers.

Combined: O(1) everything

Map gives the node pointer → DLL splices it in O(1). Eviction: tail.prev is always the LRU node. Delete from DLL + delete from map = O(1) eviction.

1 Init: Create sentinel head ↔ tail. Empty hash map. head.next = tail, tail.prev = head.
2 get(key): If not in map → -1. Else: get node, _remove(node), _add_head(node) (mark MRU), return node.val.
3 put(key, val) — existing key: Get node, update node.val, _remove, _add_head. No size change.
4 put(key, val) — new key: Create node. map[key] = node. _add_head(node). If len(map) > cap: lru = tail.prev, _remove(lru), del map[lru.key].
5 _remove(node): node.prev.next = node.next; node.next.prev = node.prev. Two pointer updates. O(1) because we have direct prev/next.
6 _add_head(node): Wire node between head and head.next. Four pointer updates. Always O(1) thanks to sentinels — same code regardless of list size.
🔑 Why sentinel nodes?

head.next is ALWAYS the MRU node. tail.prev is ALWAYS the LRU node. Insertions and deletions are the same few pointer assignments for EVERY node — no "is this the first element?" special cases. The sentinels anchor the list and make the code clean.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every moment: the DLL is ordered from MRU (head.next) to LRU (tail.prev). The hash map and DLL are always in sync — every live key appears in both exactly once. DLL size == map size == number of cached items.

Why move-to-front guarantees correct eviction: Every access (get or put) moves the touched node to the front. The node that goes the longest without being touched drifts to the back. When we need to evict, tail.prev is exactly the correct victim — no scanning required.

Why this is O(1) and not amortized: Every individual operation does a fixed number of pointer updates (≤ 4 for DLL ops) plus one hash map lookup/insert/delete (O(1) average). No loops, no sorting, no scanning. Truly O(1) per call.

⚠️ Update must ALSO refresh recency

When put is called for an existing key, most beginners update the value but forget to call _remove + _add_head. The spec says "put" counts as an access — the node must move to MRU position. Missing this gives wrong eviction order for subsequent operations.

Why store key inside the Node? When evicting tail.prev, we need to delete it from the hash map too. The node must tell us its own key: del map[lru.key]. If we only stored the value, we'd have no way to find the map entry to delete.

4 The Code (Annotated) ▾

Clean Python implementation using a custom Node class and dict:

class Node:
def __init__(self, k=0, v=0):← default args let us create sentinels with Node()
self.key = k← store key so evicted node can remove itself from map
self.val = v
self.prev = self.next = None
class LRUCache:
def __init__(self, capacity: int):
self.cap = capacity
self.map = {} ← key → Node
self.head = Node() ← MRU sentinel
self.tail = Node() ← LRU sentinel
self.head.next = self.tail
self.tail.prev = self.head ← wire sentinels together
def _remove(self, node): ← splice node out of DLL
node.prev.next = node.next ← predecessor skips over node
node.next.prev = node.prev ← successor skips back over node
def _add_head(self, node): ← insert right after head sentinel (= MRU position)
node.next = self.head.next ← node now points to the old first real node
node.prev = self.head ← node points back to head sentinel
self.head.next.prev = node ← old first node now points back to new node
self.head.next = node ← head now points forward to new node (do this LAST)
def get(self, key: int) -> int:
if key not in self.map: ← cache miss
return -1
node = self.map[key] ← O(1) lookup
self._remove(node) ← detach from current position
self._add_head(node) ← re-insert as MRU
return node.val
def put(self, key: int, value: int) -> None:
if key in self.map: ← update existing: change val + refresh recency
node = self.map[key]
node.val = value
self._remove(node)
self._add_head(node)
return
node = Node(key, value) ← new node, key stored inside for eviction
self.map[key] = node
self._add_head(node) ← insert as MRU
if len(self.map) > self.cap:← over capacity
lru = self.tail.prev ← LRU is always just before tail sentinel
self._remove(lru) ← evict from DLL
del self.map[lru.key] ← evict from map using key stored in node
🚫 Bug: Wrong order in _add_head

The four lines in _add_head must execute in order. If you set self.head.next = node before setting self.head.next.prev = node, you've lost the reference to the old first node — its .prev never gets updated. Always update the node's own pointers first, then update the surrounding nodes.

🔑 Alternative: Python OrderedDict

from collections import OrderedDict gives you a dict that tracks insertion order. cache.move_to_end(key, last=False) moves to MRU. cache.popitem(last=True) evicts LRU. This hides the DLL internally — always mention both approaches in interviews, but be ready to implement from scratch.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify Constraints

"Both get and put must be O(1). That's the whole challenge — two different O(1) requirements at once. Hash map handles lookup. What handles the ordering?"

Step 2: Eliminate Alternatives

"A simple array/list doesn't work — moving an element to the front after a get is O(n). A singly-linked list is O(n) to unlink an arbitrary node (need to find its predecessor). A doubly-linked list with direct node pointers: O(1) to unlink, O(1) to insert at front."

Step 3: State the Design

"Two structures: a dict mapping key to the DLL node, and a doubly-linked list ordered MRU to LRU. I'll use sentinel head and tail nodes so insertions and deletions never have null checks. The map stores node pointers so every get/put is O(1)."

Step 4: Trace an Example

"Let me trace capacity=2: put(1,1) → [1]; put(2,2) → [2,1]; get(1) → move 1 to front → [1,2]; put(3,3) → evict 2 (LRU) → [3,1]; get(2) → -1. This validates eviction order before I code."

Step 5: Code the Helpers First

"I'll write _remove and _add_head first, then get and put in terms of them. The helpers are the hard part — once those are right, get and put are just compositions of them."

🗣️ What to Say

"O(1) get and put means hash map for lookup plus a structure that can reorder in O(1).
I need to move any node to the front of an ordered sequence in O(1) — that's a doubly-linked list.
I'll combine them: the dict maps keys to DLL node pointers.

Follow-up questions:

  • Use OrderedDict instead? Yes — mention it, implement from scratch if asked.
  • Thread-safe version? Wrap with threading.RLock in get and put.
  • LFU Cache (LC 460)? Much harder — freq buckets + nested doubly linked lists.

7 Summary & Key Takeaways ▾
Hash map gives O(1) lookup. Doubly-linked list gives O(1) insert/remove at any position given a pointer.
Sentinel (dummy) head and tail nodes eliminate every null-check edge case in pointer manipulation.
Nodes MUST store their own key — the evicted node needs its key to remove itself from the hash map.
Every get AND every put must move the touched node to the MRU position (head side).
Time: O(1) all operations. Space: O(capacity) for DLL + hash map combined.

  • Any cache with O(1) access + O(1) eviction (LRU, MRU, LFU variant)
  • When you need O(1) ordered access AND O(1) key lookup simultaneously
