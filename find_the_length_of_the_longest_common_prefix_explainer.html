LeetCode #3043

Find the Length of the Longest Common Prefix

Given two integer arrays arr1 and arr2, return the length of the longest common prefix shared between any integer from arr1 and any integer from arr2 when compared digit by digit.

Medium Tries Trie Intersection Time: O((m+n)·d) Space: O(m·d)
1 Understanding the Problem ▾

You have two arrays of positive integers. You want to find the longest digit prefix that any number from arr1 shares with any number from arr2. The matching must start from the leftmost (most significant) digit and extend as far as both numbers agree.

💡 Key Insight

Treat each integer as a string of digits. The problem becomes: "what is the longest common prefix between any string in set A and any string in set B?" A Trie encodes all prefixes of set A in one structure, so each set B string can be checked in O(d) time instead of O(|A|·d) time.

✓ arr1=[1,10,100], arr2=[1000]

Pairs & shared prefixes:
(1, 1000) → "1" → len 1
(10, 1000) → "10" → len 2
(100, 1000) → "100" → len 3
Answer: 3

✗ arr1=[1,2,3], arr2=[4,4,4]

All arr1 start with 1/2/3
All arr2 start with 4
No digit match at position 0
Answer: 0

Key constraints & edge cases: All integers are positive (no leading zeros). A number from arr1 could be a strict prefix of a number from arr2 — e.g., "12" vs "123" gives length 2. The maximum digit count d is at most 10 (for 32-bit integers up to ~2 billion). If no pair shares even a single digit, return 0.

🔑 Why Brute Force Fails

Naive approach: compare every pair (arr1[i], arr2[j]) as strings. With m=n=50,000 numbers and d=10 digits, that's 25 billion digit comparisons — far too slow. We need to amortize the comparison cost across all queries.

2 The Algorithm: Trie Intersection ▾

A Trie (prefix tree) is a tree where each edge represents one character. Every path from root to any node represents a prefix of one or more inserted strings. The algorithm has two phases: build and query.

1 Build phase — Insert all arr1 numbers: Convert each number to a digit string, then walk the Trie digit-by-digit. At each step, if the digit node doesn't exist yet, create it. After inserting all arr1 numbers, the Trie encodes every prefix of every number in arr1.
2 Query phase — Query Trie for each arr2 number: For each arr2 number, convert to digits and walk the same Trie. Count how many consecutive digits successfully advance (match). Stop when a digit has no child in the Trie — no arr1 number shares a longer prefix.
3 Track the global maximum: After querying each arr2 number, update best = max(best, length). Return best at the end.
Trie built from arr1 = [1, 10, 100] — all three share digit paths:
root
└── '1' ← shared by "1", "10", "100"
    └── '0' ← shared by "10", "100"
        └── '0' ← only "100"

Query "1000": 1✓ 0✓ 0✓ 0✗ → length = 3
💡 Why This Works in O((m+n)·d)

Inserting m numbers costs O(m·d). Each query costs O(d). Total: O((m+n)·d). Since d ≤ 10, this is effectively O(m+n) — linear in array size. Brute force was O(m·n·d).

3 Why It Works: The Invariant ▾
🔐 The Invariant

After the build phase: a node exists at Trie depth k reachable via digit path d1d2…dk if and only if at least one number in arr1 starts with those k digits.

During a query: at each step we advance only if the current digit is a child of the current Trie node — meaning there exists an arr1 number that matches all digits so far. So length exactly equals the longest common prefix with any arr1 number.

Correctness — why the build encodes all needed prefixes: When inserting "100", we create nodes for '1' (depth 1), '0' (depth 2), '0' (depth 3) in sequence. This simultaneously records that "1", "10", and "100" are all prefixes present in arr1 — without inserting them separately. The Trie inherently stores all prefixes of all inserted strings.

Correctness — why the query gives the right count: The query loop processes digits left-to-right and breaks the moment a digit isn't found. This enforces that the counted digits are contiguous from the start — exactly the definition of a prefix. No suffix or middle match is accidentally counted.

⚠️ Common Mistake: Querying with arr1 numbers

Sometimes people query the Trie with arr1 numbers against arr2 insertions. The result is the same by symmetry, but the conventional approach (insert arr1, query arr2) is cleaner: arr1 is the "dictionary," arr2 numbers are the "lookups." If you swap them, remember to also swap your complexity accounting — m becomes the query count, n the insert count.

Space analysis: In the worst case (no shared prefixes among arr1 numbers), each number creates d unique nodes. Total space: O(m·d). Each node stores at most 10 children (digits 0–9), so each node is O(10) = O(1). Total: O(m·d·10) = O(m·d).

4 The Code (Annotated) ▾

Solution 1 — Trie (Optimal, Interview Pick)

class TrieNode: ← Minimal node: just children dict
def __init__(self):
self.children = {} ← digit char ('0'–'9') → TrieNode
def longestCommonPrefix(arr1, arr2) -> int:
root = TrieNode() ← Root of the digit-prefix Trie
for num in arr1: ← Build phase: insert every arr1 number
node = root ← Each insertion starts fresh at root
for ch in str(num): ← Walk digit-by-digit as characters
if ch not in node.children: ← This digit path doesn't exist yet
node.children[ch] = TrieNode() ← Create the missing node
node = node.children[ch] ← Advance into the child (new or existing)
best = 0 ← Global max prefix length
for num in arr2: ← Query phase: check each arr2 number
node = root ← Always restart from root
length = 0 ← Match length for this query
for ch in str(num): ← Walk query number's digits
if ch in node.children: ← This digit exists in the Trie
node = node.children[ch] ← Advance deeper
length += 1 ← One more prefix character matched
else:
break ← No arr1 number shares a longer prefix
best = max(best, length) ← Update global answer
return best ← Answer: longest common prefix length
⚠️ Why use str(num) instead of digits directly?

Converting to string gives us left-to-right digit characters ('1','0','0' for 100), enabling natural Trie traversal. Extracting digits mathematically (modulo 10) gives them in reverse order and requires extra reversal. String conversion is clean, correct, and idiomatic in Python.

Solution 2 — HashSet of All arr1 Prefixes (O((m+n)·d), no Trie class needed)

def longestCommonPrefix(arr1, arr2) -> int:
prefixes = set() ← Store every prefix string of every arr1 number
for num in arr1:
s = str(num)
for i in range(1, len(s) + 1): ← "1","10","100" for num=100
prefixes.add(s[:i]) ← Add each prefix to the set
best = 0
for num in arr2:
s = str(num)
for i in range(len(s), 0, -1): ← Check longest prefix first → fast exit
if s[:i] in prefixes: ← O(1) hash set lookup
best = max(best, i); break ← Found best for this number
return best
💡 Complexity Summary

Trie: Time O((m+n)·d), Space O(m·d) — explicit tree, elegant pointer structure
HashSet: Time O((m+n)·d), Space O(m·d) — same big-O, but string slicing adds constant overhead
Brute Force: Time O(m·n·d), Space O(1) — too slow for large inputs
d ≤ 10 for 32-bit integers, so d is effectively a small constant.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Transform integers → strings

"Prefix" is a string concept. Immediately convert integers to digit strings — this re-casts the problem as string prefix matching, which is well-understood territory.

Step 2: Propose brute force and find the bottleneck

"Brute force: compare every (arr1[i], arr2[j]) pair as strings. O(m·n·d). With m, n = 50,000, that's 25 billion operations — TLE." The bottleneck is checking every arr1 number for every arr2 query.

Step 3: Spot the Trie signal

The signal: "find the longest common PREFIX across all pairs." Trie is the canonical structure for prefix problems. It encodes all prefixes of arr1 once, so each arr2 query takes O(d) instead of O(m·d).

Step 4: Structure as Build + Query, give complexity

"Build Trie from arr1: O(m·d). Query each arr2 number: O(n·d). Total: O((m+n)·d). Since d ≤ 10, this is linear in array size." Offer the HashSet variant as a simpler alternative with the same complexity.

🗣️ What to Say to the Interviewer

"First observation: integers have digit prefixes — I'll convert to strings.
Brute force compares every pair in O(m·n·d). For 50k arrays, that's too slow.
I'll use a Trie: insert all arr1 numbers digit-by-digit. Then each arr2 number
walks the Trie, counting how many digits match before hitting a missing child.
Build: O(m·d). Query: O(n·d). Total: O((m+n)·d) ≈ linear since d≤10.
Space: O(m·d) for the Trie nodes.
As an alternative, I can collect all arr1 prefixes in a HashSet — same complexity,
simpler code, no custom class needed."
🔮 Likely Follow-up Questions
  • "Can you avoid the Trie class?" → Yes — HashSet of all arr1 prefixes
  • "What if arrays contain very large numbers?" → Same approach; just more digits (still O(d) per insert/query)
  • "What if you need the actual prefix string, not just the length?" → Accumulate digits as you walk the Trie query loop
  • "Can you do it with O(1) space?" → No — must store the arr1 prefixes somewhere (Trie or HashSet)

7 Summary & Key Takeaways ▾
Convert integers to digit strings to enable prefix operations
Build the Trie from arr1 (encodes all prefixes), then query with arr2
Each query walks the Trie digit-by-digit; depth at stop = prefix length
Total time O((m+n)·d) vs O(m·n·d) brute force — linear for d≤10
HashSet alternative: collect all arr1 prefixes, check set membership per arr2 digit prefix

🎯 When to Recognize the Trie Intersection Pattern
  • "Find the longest common prefix between ANY element of array A and ANY element of array B"
  • Large-scale prefix matching (Trie encodes all prefixes in O(m·d))
  • One array is a "dictionary" of known prefixes; the other is "queries" against it
  • Whenever you see "prefix" + "multiple strings" → think Trie
