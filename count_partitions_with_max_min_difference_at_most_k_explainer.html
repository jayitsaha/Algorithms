LeetCode #3117

Count Partitions With Max-Min Difference at Most K

Count the number of ways to partition an array into contiguous subarrays where every subarray has (max − min) ≤ K. Return the count modulo 10&sup9;+7.

Medium Monotonic Queue DP + Monotonic Deques Time: O(n) Space: O(n)
1 Understanding the Problem ▾

You have an array nums of length n and an integer k. You want to split the array into one or more contiguous subarrays (partitions) such that every single partition satisfies: max(partition) − min(partition) ≤ k. Return the total number of valid ways modulo 109+7.

💡 Key Insight

Think of the n−1 gaps between adjacent elements as potential cut points. You either cut or don't cut at each gap — giving 2n−1 possibilities in theory. But most violate the max−min constraint. Dynamic programming lets us count only the valid ones efficiently: dp[i] = ways to partition nums[0..i−1], and we accumulate it by summing dp[j] for all valid partition starting positions j. A monotonic deque maintains the running max and min over the valid window in O(1) amortized.

✓ Valid: nums=[3,6,2,4], k=2

[3]|[6]|[2,4] → 4−2=2 ≤ 2 ✓
[3]|[6]|[2]|[4] → all trivial ✓
Answer: 2

✗ Invalid attempts

[3,6,2,4] → 6−2=4 > 2 ✗
[3]|[6,2,4] → 6−2=4 > 2 ✗
[3,6]|... → 6−3=3 > 2 ✗

Edge cases: n=1 always returns 1. k=0 only allows partitions where all elements are equal. k ≥ overall (max−min) means every split is valid, giving 2n−1 answers.

2 The Algorithm: DP + Two Monotonic Deques ▾

We maintain a sliding window [left, i] that always satisfies max−min ≤ k. As we extend the window to include index i, we push i into a max-deque (decreasing values at back) and a min-deque (increasing values at back). If the window becomes invalid, we advance left until it's valid again. All valid starting points for a partition ending at i are exactly [left, i].

We also keep a prefix sum of dp values so that summing dp[left..i] is an O(1) subtraction.

1 Initialise: dp[0]=1 (empty prefix). prefix[1]=1. max-deque and min-deque empty. left=0.
2 Extend window to i: Pop from back of max-deque while nums[back] ≤ nums[i] (dominated by the new element). Push i. Mirror for min-deque (pop while nums[back] ≥ nums[i]).
3 Shrink from left while invalid: While nums[mx_dq.front] − nums[mn_dq.front] > k: left++. Evict front of either deque if its index fell below left.
4 Compute dp[i+1]: dp[i+1] = prefix[i+1] − prefix[left]. This sums dp[left..i] — the number of ways any valid last-partition ending at index i can start.
5 Update prefix: prefix[i+2] = prefix[i+1] + dp[i+1]. Repeat. Return dp[n] mod 109+7.
📐 The Recurrence Relation
dp[i] = Σ dp[j]  for j in [left, i−1]
where left = smallest j such that max(nums[j..i−1]) − min(nums[j..i−1]) ≤ k

Equivalently:  dp[i] = prefix[i] − prefix[left]
🔑 Why Two Separate Deques?

A single deque can only maintain one ordering (max OR min). To track both the sliding-window maximum and the sliding-window minimum simultaneously, we need two deques: one keeps indices in decreasing-value order (front = max), the other in increasing-value order (front = min). The difference of their fronts gives the current window's spread in O(1).

3 Why It Works: The Invariant ▾
🔐 The Invariant

At the start of every iteration for index i: left is the smallest index such that the subarray nums[left..i] has max−min ≤ k. Any partition ending at i can start at any index in [left, i] and is valid; any start before left produces an invalid partition.

Why the deques maintain max/min correctly:

  • Max-deque (decreasing): when we push index i with value nums[i], we remove all back indices with values ≤ nums[i] — they can never be the window maximum for any future window that includes i. The front always holds the current window's maximum index.
  • Min-deque (increasing): mirror logic. The front always holds the current window's minimum index.
  • When we shrink left, we check if the front of either deque is now out-of-window (index < left) and popleft it.

Why the prefix sum trick is correct: dp[j] represents ways to partition the first j elements. Summing dp[j] for j in [left, i] counts all valid last-partition choices — the last partition starts at position left, left+1, ..., or i (the full array in one piece). Since each dp[j] already accounts for all valid ways to handle elements 0..j−1, the sum gives all valid full-array partitions ending at element i.

⚠️ Off-by-One: 0-indexed nums vs 1-indexed DP

nums is 0-indexed (0 to n−1). dp is 1-indexed: dp[i] = ways to partition nums[0..i−1]. So dp[0]=1 (base case). When processing nums[i], we compute dp[i+1]. Answer = dp[n]. Deque indices point into nums (0-based). Always re-verify which domain each variable lives in.

O(n) amortized: Each index is pushed onto each deque exactly once and popped at most once. Across all n iterations that's O(n) total deque operations — O(1) amortized per step.

4 The Code (Annotated) ▾

Solution 1 — DP + Two Monotonic Deques (Optimal, O(n)) — Interview Pick

from collections import deque
def countPartitions(nums, k): ← entry point
MOD = 10**9 + 7
n = len(nums)
dp = [0] * (n + 1) ← dp[i] = ways to partition nums[0..i-1]
dp[0] = 1 ← base case: empty prefix has exactly 1 way
prefix = [0] * (n + 2) ← prefix[i] = sum(dp[0..i-1])
prefix[1] = 1 ← prefix[1] = dp[0] = 1
mx_dq = deque() ← decreasing deque: front = idx of window max
mn_dq = deque() ← increasing deque: front = idx of window min
left = 0 ← left boundary of valid window
for i in range(n):
while mx_dq and nums[mx_dq[-1]] <= nums[i]:← maintain decreasing order
mx_dq.pop()
mx_dq.append(i) ← i is the new (candidate) max
while mn_dq and nums[mn_dq[-1]] >= nums[i]:← maintain increasing order
mn_dq.pop()
mn_dq.append(i) ← i is the new (candidate) min
while nums[mx_dq[0]] - nums[mn_dq[0]] > k:← window invalid?
left += 1 ← shrink from left
if mx_dq[0] < left: mx_dq.popleft()← evict stale max
if mn_dq[0] < left: mn_dq.popleft()← evict stale min
dp[i+1] = (prefix[i+1] - prefix[left]) % MOD← sum dp[left..i]
prefix[i+2] = (prefix[i+1] + dp[i+1]) % MOD← extend prefix
return dp[n]
⚠️ Why prefix[i+1] − prefix[left] and not something else?

Our prefix is defined as prefix[j] = dp[0]+dp[1]+...+dp[j−1]. The sum of dp[left]+...+dp[i] equals prefix[i+1] − prefix[left]. Drawing a small table (n=3) on paper with all the indices is the best way to convince yourself. This is a classic prefix-sum-off-by-one scenario — commit the definition to memory and re-derive indices from it.

Solution 2 — Brute Force O(n²) (for understanding only)

def countPartitions_brute(nums, k):
MOD, n = 10**9+7, len(nums)
dp = [0]*(n+1); dp[0] = 1
for i in range(1, n+1): ← each ending index (1-based)
mn = mx = nums[i-1]
for j in range(i-1, -1, -1): ← try all starting points in reverse
mn = min(mn, nums[j])
mx = max(mx, nums[j])
if mx - mn > k: break ← can't go further left
dp[i] = (dp[i] + dp[j]) % MOD
return dp[n]
💡 Complexity Comparison

Brute Force: Time O(n²), Space O(n). Inner loop rescans left on every i.
DP + Deques: Time O(n), Space O(n). Each element enqueued/dequeued at most once.
n=105: Brute ≈ 1010 ops (TLE). Deque approach ≈ 3×105 ops.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognise the Counting DP Structure

"We need to count ways to partition — this signals DP. Define dp[i] = ways to partition nums[0..i−1]. Recurrence: dp[i] = ∑ dp[j] for all valid starting positions j of the last partition ending at i. Base case dp[0]=1."

Step 2: Identify the Bottleneck (Inner Loop)

"Naively finding valid j's requires scanning backward — O(n) per i → O(n²). Notice: as i grows, the leftmost valid j (call it left) only moves right — this is a sliding window! We only need to track left."

Step 3: Use Monotonic Deques for O(1) Sliding Window Max/Min

"To check if window is valid, we need max and min in O(1). Classic technique: two monotonic deques. One decreasing (max at front), one increasing (min at front). Evict when window shrinks."

Step 4: Prefix Sum for Range Sum of DP

"dp[i] = ∑ dp[j] for j in [left, i]. Maintain a prefix array so range sum = prefix[i+1] − prefix[left] in O(1). Total: O(n) time and space."

🗣️ What to Say to the Interviewer

"Let me define dp[i] = number of valid partitions of the first i elements.
The recurrence: dp[i] = sum of dp[j] for all valid starting points j of the last partition ending at i.
Valid means max(nums[j..i-1]) - min(nums[j..i-1]) <= k. As i increases, this left boundary only advances right.
I'll use two monotonic deques for O(1) window max and min, and a prefix sum for O(1) range sum of dp values.
Time O(n), Space O(n). Amortized: each element is pushed/popped at most once per deque."

7 Summary & Key Takeaways ▾
Partition-counting with a contiguous-window validity condition → DP with sliding window.
Two monotonic deques (one decreasing for max, one increasing for min) give O(1) amortized window max and min.
Prefix sum of dp values turns the range-sum query dp[left..i] into an O(1) subtraction.
Left boundary is monotonically non-decreasing: each element is enqueued and dequeued at most once → O(n) total.
The trio DP + monotonic deque + prefix sum is a recurring pattern for "count valid partitions with window constraint."
🎯 When to Use This Pattern
  • Counting or optimising partitions with a contiguous-subarray spread constraint.
  • Constraint involves sliding-window max and/or min (spread, ratio, difference).
  • DP recurrence requires a sum over a range of dp values → add a prefix sum array.
  • Valid region is monotone (left only moves right) → amortized O(1) per step with deque.
