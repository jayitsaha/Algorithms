LeetCode #808

Soup Servings

Two soups, A and B, each starting with n ml. Four equally-probable operations drain different amounts. Find: P(A empties first) + 0.5 × P(both empty together).

Medium Dynamic Programming Memoize Probabilities Time: O(n²/625) Space: O(n²/625)
1 Understanding the Problem ▾

We start with n ml of Soup A and n ml of Soup B. We repeatedly pick one of four operations at random (each with probability 1/4):

The 4 Random Operations

Op 1: A−100ml, B−0ml
Op 2: A−75ml, B−25ml
Op 3: A−50ml, B−50ml
Op 4: A−25ml, B−75ml

What We're Computing

answer =
  P(A finishes first)
+ 0.5 × P(A and B together)

NOT P(B first) — asymmetric!

💡 Key Insight: Soup A is Biased to Empty First

Average drain per operation: A loses (100+75+50+25)/4 = 62.5 ml, while B loses (0+25+50+75)/4 = 37.5 ml. A drains 1.67× faster. For very large n, A almost certainly empties first — the answer approaches 1.0 and we can return early.

Concrete small example (n=25, scaled to 1 unit):

  • Op1 gives A=1−4=−3 ≤ 0, B=1 still > 0 → A wins → prob contribution = 1.0 × (1/4)
  • Op2 gives A=1−3=−2 ≤ 0, B=0 → both empty → contribution = 0.5 × (1/4)
  • Op3 gives A=−1 ≤ 0, B=−1 ≤ 0 → both empty → 0.5 × (1/4)
  • Op4 gives A=0, B=−2 ≤ 0 → B empties first, but A=0 so both done → 0.5 × (1/4)

So dp(1,1) = 0.25×(1.0 + 0.5 + 0.5 + 0.5) = 0.25×2.5 = 0.625.

⚠️ Serving can exceed what remains

If A has 30ml and we pick Op1 (drain 100ml), A "goes negative" — we just served the last 30ml. So a − 4 can be negative; we check if a ≤ 0 at the top of the recursion, not max(0, a−4) before recursing. Both work, but the natural approach lets negatives happen and handles them at base cases.

2 The Algorithm: Memoized Top-Down DP ▾

Define dp(a, b) = the answer (probability formula) when we have a units of A and b units of B remaining (after scaling by 25). We memoize every (a, b) pair to avoid recomputation.

1 Scale n by 25: n = ceil(n / 25). Every operation is a multiple of 25ml. In "unit" terms the ops drain (4,0), (3,1), (2,2), (1,3). This shrinks the state space from O(n²) to O((n/25)²).
2 Early-exit for large n: If n ≥ 179 (scaled), return 1.0. The asymmetry is so large that float64 cannot distinguish the true answer from 1.0. This bounds runtime to O(179²) ≈ 32K states maximum.
3 Base cases (terminals): (a ≤ 0 AND b ≤ 0) → return 0.5 (simultaneous). (a ≤ 0) → return 1.0 (A wins). (b ≤ 0) → return 0.0 (B wins).
4 Recurrence: Average the 4 recursive sub-calls — dp(a,b) = 0.25 × [dp(a−4,b) + dp(a−3,b−1) + dp(a−2,b−2) + dp(a−1,b−3)]. Memoize the result.
📐 Recurrence Relation (in scaled units)
dp(a, b) = 0.25 × [ dp(a−4, b) + dp(a−3, b−1) + dp(a−2, b−2) + dp(a−1, b−3) ]

Base: dp(a≤0, b≤0) = 0.5    ← both empty simultaneously
Base: dp(a≤0, b>0) = 1.0    ← A empties first (A wins!)
Base: dp(a>0, b≤0) = 0.0    ← B empties first (no credit)
🔑 Why This Is Dynamic Programming

The same state (a, b) is reachable via many different sequences of operations. For example, dp(2, 3) might be needed by dp(6, 3), dp(5, 4), dp(4, 5), and dp(3, 6) all at once. Without memoization we'd recompute it exponentially many times. With memoization, each of the O(n²) states is computed exactly once — giving polynomial total work.

3 Why It Works: The Invariant ▾
🔐 The Invariant

dp(a, b) always equals the exact probability expression (P(A first) + 0.5×P(both)) for the random process starting from state (a, b). The base cases are ground truth. Each recursive call contributes exactly 1/4 of its result (uniform probability). Memoization never changes a value once set — it only avoids recomputation.

Why 0.5 for simultaneous? The problem statement defines the answer as P(A) + 0.5×P(both). This is a design choice in the problem — not an average. Our base case return 0.5 when both are ≤ 0 encodes this directly. Think of it as "A half-wins" when both empty together.

Why the large-n shortcut is mathematically sound: For large n, by the Central Limit Theorem, the difference (A_drained − B_drained) grows proportionally to n with overwhelming probability. A's expected drain rate is 62.5 vs B's 37.5, so A finishes first with probability that approaches 1 exponentially fast in n. Numerically, for n/25 ≥ 179, the answer exceeds 1 − 10⁻⁹, which float64 rounds to exactly 1.0.

Overlapping subproblems illustrated:

dp(3,3) needs → dp(-1,3)=1.0, dp(0,2)=1.0, dp(1,1)=0.625, dp(2,-0)=0.0 dp(2,2) needs → dp(-2,2)=1.0, dp(-1,1)=1.0, dp(0,0)=0.5, dp(1,-1)=0.0 dp(1,1) appears in both dp(3,3) and dp(2,2) calls → memoized, not recomputed
⚠️ Order of base case checks matters

Always check a ≤ 0 AND b ≤ 0 first, before the individual checks. If you check a ≤ 0 first (alone), you'd return 1.0 even when b is also ≤ 0 — missing the simultaneous case worth only 0.5.

4 The Code (Annotated) ▾

from functools import lru_cache ← Python built-in memoization decorator

if a <= 0: return 1.0 ← A empty, B not: A wins → 1.0
if b <= 0: return 0.0 ← B empty, A not: B wins → 0.0
return 0.25 * ( ← each of 4 ops equally likely (prob 0.25)
dp(a-4, b) + ← Op1: drain 4 units A, 0 B
dp(a-3, b-1) + ← Op2: drain 3A, 1B
dp(a-2, b-2) + ← Op3: drain 2A, 2B
dp(a-1, b-3) ← Op4: drain 1A, 3B
)
return dp(n, n) ← start with equal amounts
⚠️ Common Mistake: Wrong large-n threshold

Some solutions use n >= 500 or n >= 200. The precise threshold is 179 (scaled) — beyond which float64 rounds to 1.0. Larger thresholds waste time on unnecessary computation; smaller might give wrong answers.

Bottom-up tabulation alternative: Fill a 2D table dp[a][b] for 0 ≤ a, b ≤ n. Process in increasing order of (a + b), checking base cases before computing the recurrence. Same asymptotic complexity, avoids Python's recursion stack limit, but requires more careful index management.

💡 Why lru_cache is Perfect Here

Arguments (a, b) are integers — hashable. The recursion terminates because each call has strictly smaller total a+b (or hits a base case). We never need to invalidate the cache. lru_cache(maxsize=None) is an unbounded cache (equivalent to a hash map but with zero boilerplate).

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the DP signal

The problem asks for a probability that depends on a two-dimensional state (remaining A, remaining B) that shrinks monotonically. Each step has random outcomes. This is a textbook case for probability memoization DP — define dp(a, b) = answer from state (a, b).

Step 2: Nail the base cases first

Before writing the recurrence, list all terminal states: (1) both ≤ 0 → 0.5, (2) only A ≤ 0 → 1.0, (3) only B ≤ 0 → 0.0. The order of checks matters — simultaneous must come first. Write these in code before the recurrence line.

Step 3: Spot the scaling optimization

Observe that all operation amounts (100, 75, 50, 25) share GCD = 25. Scale n by 25. This is the critical insight that makes the problem tractable — without it, n up to 10⁸ makes the state space 10¹⁶, completely infeasible.

Step 4: Add the large-n shortcut

After scaling, note that A drains ~1.67× faster. For n ≥ 179, the answer is effectively 1.0. Add this early return to bound the state space to O(179²) regardless of input size.

🗣️ What to Say to the Interviewer

"I recognize this as probability DP — the state (remaining_A, remaining_B) decreases monotonically, and each step branches into 4 equally-likely sub-states. My recurrence is: dp(a,b) = 0.25 × sum of 4 recursive calls. Key observations: all operations are multiples of 25, so I scale n down — this reduces state space from O(n²) to O((n/25)²). For large n (≥179 scaled), the asymmetry means A finishes first with probability 1.0, so I return early. Time complexity is O((n/25)²), bounded by O(179²) ≈ 32K — effectively constant! I'll memoize with @lru_cache to handle overlapping subproblems."
🔎 Pattern Recognition Signals for Probability Memoization
  • Problem asks for a probability (not a count or optimum)
  • Process involves random choices with equal probability
  • State space is 2D and monotonically decreasing toward terminals
  • Multiple paths can reach the same state → overlapping subproblems
  • Operation amounts share a common factor → potential scaling

def soupServings(n):

@lru_cache(maxsize=None)
def dp(a, b):
if a <= 0 and b <= 0: return 0.5 # both empty
if a <= 0: return 1.0 # A first
if b <= 0: return 0.0 # B first
return 0.25*(dp(a-4,b)+dp(a-3,b-1)+dp(a-2,b-2)+dp(a-1,b-3))
return dp(n, n) # dp(2, 2)

7 Summary & Key Takeaways ▾
Scale by GCD of operation amounts (25) to minimize state space from O(n²) to O((n/25)²)
Check simultaneous empty (both ≤ 0) BEFORE individual checks — order matters!
The large-n shortcut (n ≥ 179 scaled) bounds all computation to ~32K states maximum
lru_cache with integer arguments gives zero-boilerplate memoization in Python
