LeetCode #778

Swim in Rising Water

Find the minimum time t such that there exists a path from (0,0) to (n-1,n-1) where every cell along the path has elevation ≤ t.

Hard Graph Dijkstra · Min-Heap Time: O(n² log n) Space: O(n²)
1 Understanding the Problem ▾

You have an n × n grid where each cell holds a unique integer: its elevation. Think of the grid as terrain. Water rises uniformly — at time t, you can swim through any cell whose elevation is ≤ t. Moving between adjacent cells takes zero time, but you can only move into cells already reachable by the current water level.

You start at grid[0][0] (top-left) and want to reach grid[n-1][n-1] (bottom-right). The question: what is the minimum time t needed?

💡 Key Insight

The "time" needed for a path equals the highest elevation on that path — you must wait until the water level reaches every cell along the route. So we want the path that minimizes its maximum cell elevation. This is the classic bottleneck shortest path problem.

Not what we're doing

NOT minimizing hop count
NOT a simple BFS by distance
NOT summing edge weights
Moving = zero time cost

What we actually want

Minimize the maximum elevation along the path
= min over all paths of max(elevations on path)
Solvable with Dijkstra's + min-heap
or Binary Search + BFS

Worked example: For grid = [[0,2],[1,3]], two paths exist:

(0,0)→(0,1)→(1,1): elevations = [0, 2, 3] → max = 3
(0,0)→(1,0)→(1,1): elevations = [0, 1, 3] → max = 3
Both paths have bottleneck 3 (forced by the target cell). Answer = 3.

Edge cases: n=1 returns grid[0][0] (usually 0). If (0,0) has a high elevation, that becomes a lower bound on the answer. The grid is always a permutation of 0..n²−1 so all values are unique.

2 The Algorithm: Dijkstra's Bottleneck Shortest Path ▾

This is a modified Dijkstra's algorithm. In classic Dijkstra's, the "cost" is a sum of edge weights. Here, the cost is the maximum elevation seen so far on the best path to each cell. When we visit a neighbor, the new cost is max(current_cost, neighbor_elevation) — the bottleneck.

1 Push start cell: Initialize the min-heap with (grid[0][0], 0, 0) — tuple of (max_elevation_so_far, row, col). Create an empty visited set.
2 Pop the minimum: Extract (cost, r, c) from the heap. This cell has the globally smallest bottleneck cost among all frontier cells. If already visited, skip.
3 Check for destination: If (r, c) == (n-1, n-1), return cost. By Dijkstra's invariant, this is the minimum possible time.
4 Settle this cell: Add (r, c) to visited. Explore all 4 neighbors. For each unvisited neighbor, compute new_cost = max(cost, grid[nr][nc]). Push (new_cost, nr, nc) onto the heap.
5 Repeat until destination is popped. The heap ordering guarantees we always expand the lowest-bottleneck frontier first.

🔑 Alternative: Binary Search + BFS

Binary search on the answer t in range [0, n²−1]. For each candidate t, run a BFS that only walks cells with elevation ≤ t. If (n-1,n-1) is reachable, try smaller t (search left); otherwise try larger (search right). Time: O(n² · log(n²)) = O(n² log n) — same asymptotic as Dijkstra's.

3 Why It Works: The Invariant ▾
🔐 The Invariant

When a cell (r, c) is popped from the min-heap, the value cost associated with it equals the minimum possible bottleneck to reach that cell from (0,0). No future path can achieve a lower maximum elevation to that cell.

Why the min-heap ordering guarantees this: The heap is sorted by bottleneck cost (max elevation seen). When we pop a cell, it has the globally smallest bottleneck among all unexplored cells. Any other path to this cell would either (a) share this same route, or (b) go through a higher-elevation cell that hasn't been settled yet — and therefore has a higher cost. So the first pop of a cell always gives the optimal cost.

Why max(cost, neighbor_elev) is the right cost function: The "time" needed for a path = the highest elevation on it. When we extend a path to a new neighbor, the new bottleneck is the max of the existing bottleneck and the neighbor's own elevation. This max function is monotone (never decreases as we add cells), which is the key property Dijkstra's requires.

⚠️ Why Not Plain BFS?

BFS finds the path with the fewest hops. But here, fewer hops does NOT mean lower maximum elevation. A direct 2-hop path might cross a high-elevation cell, while a 10-hop path stays on low ground. We need to explore paths by bottleneck cost, not hop count — that's exactly what a min-heap gives us.

Edge case — start or end elevation: The answer is always ≥ max(grid[0][0], grid[n-1][n-1]) since both endpoints must be reachable. The min-heap captures this: the start cell's elevation is the initial heap entry, and when we finally pop the target, its elevation is folded into the cost via the max operation.

4 The Code (Annotated) ▾

Solution 1 — Dijkstra's with Min-Heap (Optimal, Interview Pick)

import heapq ← Python's built-in min-heap module
def swimInWater(grid: list[list[int]]) -> int:
n = len(grid) ← grid is n×n
visited = set() ← tracks permanently settled cells
heap = [(grid[0][0], 0, 0)] ← (bottleneck_cost, row, col). Start cell's elev is its own bottleneck
while heap:
cost, r, c = heapq.heappop(heap) ← pop the cell with globally lowest bottleneck
if (r, c) in visited: ← already settled with a lower cost — skip stale entry
continue
visited.add((r, c)) ← this cell's optimal cost is now final
if r == n-1 and c == n-1: ← reached destination — Dijkstra's guarantees this is optimal
return cost
for dr, dc in [(0,1),(0,-1),(1,0),(-1,0)]: ← 4-directional movement
nr, nc = r + dr, c + dc
if 0 <= nr < n and 0 <= nc < n and (nr, nc) not in visited:
new_cost = max(cost, grid[nr][nc]) ← bottleneck = max of path-so-far and neighbor's elevation
heapq.heappush(heap, (new_cost, nr, nc))
⚠️ Why visited set instead of a dist[] array?

Classic Dijkstra's uses a dist[] array and updates it when a better path is found. Here we use a visited set: once popped, a cell's cost is final, so we simply skip re-popped cells. Both approaches are correct; the visited set is simpler to implement and reason about for this problem.

Solution 2 — Binary Search + BFS (Alternative)

from collections import deque
def swimInWater_bs(grid):
n = len(grid)
def can_reach(t): ← BFS: is (n-1,n-1) reachable using only cells with elev ≤ t?
if grid[0][0] > t: return False ← can't even start at time t
q, seen = deque([(0,0)]), {(0,0)}
while q:
r, c = q.popleft()
if r == n-1 and c == n-1: return True
for dr,dc in [(0,1),(0,-1),(1,0),(-1,0)]:
nr,nc = r+dr, c+dc
if 0<=nr<n and 0<=nc<n and (nr,nc) not in seen and grid[nr][nc]<=t:
seen.add((nr,nc)); q.append((nr,nc))
return False
lo, hi = 0, n*n - 1 ← search space: all possible elevation values
while lo < hi: ← standard binary search on the answer
mid = (lo + hi) // 2
if can_reach(mid): hi = mid ← feasible — try smaller t
else: lo = mid + 1 ← infeasible — need larger t
return lo ← lo == hi: the minimum feasible t
💡 Complexity Summary

Dijkstra's (min-heap): Time O(n² log n), Space O(n²) — single pass, each cell pushed/popped at most once from O(log n²) heap
Binary Search + BFS: Time O(n² log n), Space O(n²) — O(log n²) iterations × O(n²) BFS each
Both are equivalent asymptotically. Dijkstra's typically has better constants in practice.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Bottleneck Pattern

The phrase "minimum time to travel from A to B" combined with "cells are accessible when their value ≤ time" is the signal for a bottleneck path problem. The answer equals the maximum cell value on the optimal (min-bottleneck) path. This is a well-known graph pattern.

Step 2: Propose Binary Search First

"I can binary search on the answer t. For any candidate t, I check with BFS whether the destination is reachable using only cells ≤ t. The search space is [0, n²−1]. This is O(n² log n)." Proposing this shows systematic thinking, even if Dijkstra's is more elegant.

Step 3: Identify the Dijkstra's Mapping

"Alternatively, I can model this as a shortest-path problem where the cost to reach a cell is the maximum elevation on the path, not the sum. Dijkstra's works with any monotone cost function — and max is monotone. So I use a min-heap ordered by this max-cost."

Step 4: State the Key Modification

"The only change from textbook Dijkstra's: instead of new_dist = dist + edge_weight, I use new_cost = max(cost, grid[nr][nc]). The heap still orders by this value, so all of Dijkstra's guarantees apply."

🗣️ What to Say to the Interviewer

"The key observation: the minimum time for any path equals the maximum elevation along it.
So I want to minimize the maximum elevation across all possible paths — a bottleneck shortest path.
I'll use Dijkstra's with a min-heap ordered by (max_elevation_so_far, row, col).
When I push a neighbor, new_cost = max(current_cost, neighbor_elevation).
Dijkstra's invariant guarantees: when I pop the destination for the first time, that cost is optimal.
Time: O(n² log n). Space: O(n²)."
🔮 Likely Follow-up Questions
  • "Can you solve it with Union-Find?" → Yes! Sort all cells by elevation. Add cells one by one; use DSU to union adjacent cells. Return elevation when (0,0) and (n-1,n-1) first connect.
  • "What if you could also move diagonally?" → Add 4 more directions: same algorithm, just 8 neighbors.
  • "What if cell values could repeat?" → Algorithm unchanged; only the visited check matters, not uniqueness.
  • "LeetCode 1631?" → Path With Minimum Effort — same Dijkstra's pattern but cost = max(|diff between adjacent cells|).

7 Summary & Key Takeaways ▾
This is a bottleneck shortest path: minimize the maximum elevation on any path, not the sum
Modified Dijkstra's: cost = max(path_bottleneck, next_cell_elevation) — not a sum
When a cell is popped from the min-heap, its bottleneck cost is permanently optimal (Dijkstra's invariant)
Binary Search + BFS is equally valid — binary search on t, BFS checks reachability
Union-Find approach: sort cells by elevation, union adjacently, return elevation when start and end connect

Brute Force (all paths) Exponential Exponential Not feasible; for intuition only

🎯 When to Recognize the Min-Max Path Pattern
  • "Minimum time/effort such that a path from A to B is possible" → bottleneck path
  • "Minimize the maximum weight/elevation/effort on any path" → Dijkstra's with max cost
  • Grid navigation + threshold constraint that can be binary-searched → Binary Search + BFS
  • Connected components + monotone adding of nodes → Union-Find
