Binary Search for Multiple Queries

Precompute once, answer k queries in O(|s|·log|t|) each — the interviewer's favourite follow-up

Easy / Follow-Up String Processing Binary Search Multi-Query Optimisation Preprocess: O(|t|) Per Query: O(|s|·log|t|)
1 Why Do We Need a Different Approach? ▲

The greedy two-pointer from Solution 1 is O(|t|) per query — optimal for a single check. But the interviewer almost always follows up with:

🎤 The Follow-Up Question

"What if there are k = 10,000 different patterns s₁, s₂, …, sₖ and we want to check each one against the same string t?"

With the greedy approach, each query rescans all of t. For k queries:

Greedy × k Queries

Total: O(k · |t|)
If k=10,000 and |t|=100,000 → 10⁹ operations ✗

Precompute + Binary Search

Setup: O(|t|) once
Each query: O(|s| · log|t|)
Total: O(|t| + k · |s| · log|t|)
If |s|=10, k=10,000, |t|=100,000 → ~1.7M ops ✓
💡 Core Idea

Pay O(|t|) once upfront to build a lookup structure for t. For each query, instead of scanning all of t linearly, binary search for the next valid occurrence of each character of s.

2 The Algorithm: Preprocess + Binary Search ▲

Phase 1 — Preprocess t (done once)

1 Scan t left-to-right, recording every index where each character appears.
2 Store in a dict: char → [sorted list of positions]. Because we insert in index order, the lists are already sorted.

Example: t = "caebcde"

positions = {'a': [1], 'b': [3], 'c': [0, 4], 'd': [5], 'e': [2, 6]}

Phase 2 — Answer Query for Pattern s

1 Start with curr = -1 (the "cursor" — next match must come strictly after this position).
2 For each character c in s, look up positions[c].
3 Use bisect_right(positions[c], curr) to find the index of the first position strictly greater than curr.
4 If that index is past the end of the list → no valid occurrence → return False.
5 Otherwise: curr = positions[c][idx]. Advance the cursor and continue.
6 If all characters of s matched successfully → return True.
🔑 Why bisect_right and not bisect_left?

bisect_right(arr, x) returns the index of the first element strictly greater than x. This is exactly what we need: the next occurrence of character c that comes after curr. Using bisect_left would return the first element ≥ curr, which could match the same position twice if curr itself is in the list.

3 Why It Works: The Ordering Invariant ▲
🔐 The Invariant

After processing character s[k], the variable curr holds the index in t where s[k] was matched. Any subsequent match for s[k+1] must be at an index strictly greater than curr. This enforces the "relative order preserved" requirement directly.

Why bisect_right finds the correct match

The positions list for any character is a sorted array of indices. bisect_right(arr, curr) returns the number of elements ≤ curr — which is exactly the index of the first element > curr. In O(log n) time, we skip all positions that are too early and land on the earliest valid one.

• Empty s: The for-loop doesn't execute → return True. Empty string is a subsequence of anything.
• Character not in positions: char never appears in t → return False immediately.
• No position left after curr: bisect_right returns len(positions[c]) (past end) → return False.
• Greedy correctness still holds: Taking the earliest valid position each time is still optimal — no reason to delay a match when an earlier one preserves order.
4 The Code (Annotated) ▲
from collections import defaultdict
import bisect
# Call once with the large source string t
def preprocess(t: str) -> dict:
positions = defaultdict(list) # char → sorted list of indices
for i, c in enumerate(t): # scan t exactly once — O(|t|)
positions[c].append(i) # lists are sorted automatically
return positions
# Call for EACH of the k queries — O(|s| log |t|) per call
def isSubsequence_multi(s: str, positions: dict) -> bool:
curr = -1 # must find each char AFTER this position
for c in s:
if c not in positions: return False # never in t at all
# first index in positions[c] that is strictly > curr
idx = bisect.bisect_right(positions[c], curr)
if idx == len(positions[c]): return False # exhausted; no match
curr = positions[c][idx] # commit to this match; advance cursor
return True
⚠️ Common Mistake: using bisect_left

If curr = 4 and positions contains 4, bisect_left would return the index OF 4 — the same position we just matched. Using it would allow matching the same position twice, breaking the ordering constraint. Always use bisect_right for "strictly after".

⚠️ Single-query usage

If you only have one query, don't use this. The O(|t|) preprocessing cost makes it break even only when you have multiple queries. For a single check, Solution 1's greedy with O(1) space is strictly better.

5 🎤 Interview Strategy: When to Mention This ▲
Step 1 — Lead with Solution 1

Always start with the greedy two-pointer. It's simpler, O(1) space, and directly optimal for the base problem. Do NOT open with binary search — it over-complicates a straightforward problem.

Step 2 — Proactively Mention the Follow-Up

After presenting Solution 1, volunteer: "For the case of many queries against the same t, we can precompute a positions map and binary search…" This signals strong problem-solving instinct.

Step 3 — Derive the Trade-Off Clearly

Total cost analysis: O(|t| + k·|s|·log|t|) vs O(k·|t|). Binary search wins when k is large AND |s| ≪ |t|. Be able to state when each solution is better — this is where most candidates fall short.

Step 4 — Explain bisect_right Carefully

State explicitly: "I need strictly greater than curr, so I use bisect_right — if I used bisect_left with curr in the list, it would return the same index and break the ordering." Mentioning this subtlety shows depth.

🗣️ What to Say When the Follow-Up Arrives

"With k queries against the same t, re-running O(|t|) gets expensive..."
"I'll preprocess t once: map each char to its sorted list of positions. O(|t|)."
"For each query, I binary search each char's list for the first position > curr."
"bisect_right handles this: finds leftmost position strictly after curr."
"Per query: O(|s| log |t|). Total: O(|t| + k·|s|·log|t|)."
"This beats O(k·|t|) when |s| is small relative to |t|."
💡 Deeper Extensions to Know
  • Space concern: Positions map is O(|t|) — acceptable since t is fixed
  • What if t changes too? Each update to t requires re-building the map — amortized analysis depends on update frequency
  • What if s has repeated chars? Works fine — each repeated char in s triggers its own bisect_right on the same positions list, advancing curr each time

7 Summary & Key Takeaways ▲
Precompute positions map once in O(|t|), amortise across all k queries
bisect_right(positions[c], curr) finds the first index > curr in O(log|t|)
Always use bisect_right (not bisect_left) to enforce strictly-after semantics
Use this solution only when k is large; for single queries, greedy is better
🎯 When to Reach for This Pattern
  • Multiple subsequence checks against the same fixed text
  • Any "find ordered occurrences in sorted indexed structure" problem
  • The text is static but queries are numerous
  • Interviewer asks: "Can you do better if you run this 10,000 times?"
