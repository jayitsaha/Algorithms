LeetCode #304

Range Sum Query 2D — Immutable

Preprocess a 2D matrix so any sub-rectangle sum can be answered in O(1) using the inclusion-exclusion principle.

Medium Prefix Sum 2D Prefix Sum Init: O(m·n) Query: O(1) Space: O(m·n)
1 Understanding the Problem ▾

You're given a 2D integer matrix that never changes after it's created. You'll be called many times with a rectangle defined by its top-left corner (row1, col1) and bottom-right corner (row2, col2). Each call should return the sum of all elements inside that rectangle as fast as possible.

💡 Key Insight

If we precompute a "prefix sum table" where each cell stores the sum of the entire top-left rectangle up to that cell, we can compute any rectangle sum using just four table lookups and three arithmetic operations — constant time, no matter how large the region.

Without Preprocessing: O(m·n) per query

Iterate every cell in the rectangle.
100×100 matrix, 10,000 queries
→ up to 108 operations. Too slow!

With 2D Prefix Sum: O(1) per query

O(m·n) build once. Then each query:
4 lookups + 3 arithmetic ops.
10,000 queries → 40,000 operations. Fast!

Concrete walkthrough with a 3×4 matrix:

Consider the matrix: row 0 = [3, 0, 1, 4], row 1 = [5, 6, 3, 2], row 2 = [1, 2, 0, 1].
Query: sumRegion(1, 1, 2, 3) — rows 1–2, cols 1–3.
Cells: 6 + 3 + 2 + 2 + 0 + 1 = 14. We'll verify this with the prefix table in Section 6.

Edge Cases to consider: query spanning single cell (row1=row2, col1=col2), query covering the whole matrix, matrix of size 1×1, and matrices with negative numbers (the formula handles all uniformly).

2 The Algorithm: 2D Prefix Sum with Inclusion-Exclusion ▾

The idea extends the familiar 1D prefix sum to two dimensions. Define pre[i][j] = sum of all matrix elements in the top-left rectangle from (0,0) to (i-1, j-1). We use a 1-indexed prefix table of size (m+1)×(n+1) so the zero-row and zero-column act as sentinel boundaries.

1 Allocate prefix table: Create pre = [[0]*(n+1) for _ in range(m+1)]. Every cell starts at zero. The extra row/column means pre[0][*] = 0 and pre[*][0] = 0 always, eliminating boundary edge cases.
2 Fill prefix table (row-major): For each matrix cell (i, j), compute:
pre[i+1][j+1] = matrix[i][j] + pre[i][j+1] + pre[i+1][j] - pre[i][j]
This adds the current cell plus the rectangle above (pre[i][j+1]) and the rectangle to the left (pre[i+1][j]), then subtracts the overlap corner (pre[i][j]) counted twice.
3 Answer any query in O(1): For sumRegion(r1, c1, r2, c2):
return pre[r2+1][c2+1] - pre[r1][c2+1] - pre[r2+1][c1] + pre[r1][c1]
Start with the big top-left rectangle; cut away the strip above and the strip to the left; add back the top-left corner that was removed twice.
sumRegion(r1, c1, r2, c2) =
  pre[r2+1][c2+1] ← big rectangle: (0,0) → (r2,c2)
− pre[r1][c2+1] ← strip above: (0,0) → (r1-1, c2)
− pre[r2+1][c1] ← strip left: (0,0) → (r2, c1-1)
+ pre[r1][c1] ← corner added back (double-subtracted)
🔑 Why (m+1)×(n+1) instead of m×n?

With a 1-indexed prefix table, pre[0][*] and pre[*][0] are always 0. This means the query formula pre[r2+1][c2+1] - pre[r1][c2+1] - pre[r2+1][c1] + pre[r1][c1] works correctly even when r1=0 or c1=0 — no special-case branching needed. The +1 offset aligns "sum up to row i" with index i+1.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At all times after construction: pre[i][j] = the sum of every element in the sub-matrix from row 0, col 0 to row i-1, col j-1 (inclusive). It represents "the cumulative sum of the top-left rectangle that ends just before row i and col j."

Why the build recurrence is correct — the inclusion-exclusion for prefix fill:

To compute pre[i+1][j+1] (the sum covering rows 0..i and cols 0..j), we need everything in the row above (pre[i][j+1], covers rows 0..i-1, cols 0..j) plus everything in the column to the left (pre[i+1][j], covers rows 0..i, cols 0..j-1) plus the current cell. But the top-left rectangle pre[i][j] (rows 0..i-1, cols 0..j-1) is included in both of those terms — subtract it once to fix the double-count.

Why the query formula isolates exactly the target rectangle:

Think geometrically. pre[r2+1][c2+1] covers the entire region from (0,0) to (r2,c2). To isolate just (r1,c1) → (r2,c2):

  • Subtract the strip above the query (rows 0..r1-1): that's pre[r1][c2+1]
  • Subtract the strip left of the query (cols 0..c1-1): that's pre[r2+1][c1]
  • The corner (rows 0..r1-1, cols 0..c1-1 = pre[r1][c1]) was subtracted twice — add it back once

What remains is exactly the desired rectangle.

⚠️ The Most Common Off-by-One Bug

Confusing 0-indexed matrix coordinates with 1-indexed prefix table coordinates. The matrix cell matrix[i][j] corresponds to prefix cell pre[i+1][j+1]. In a query sumRegion(r1,c1,r2,c2), always add 1 to get the "bottom-right" inclusive corner: pre[r2+1][c2+1]. The "top-left" corners r1 and c1 are used as-is — no +1 — because they represent the exclusive start boundary of the top-left strips.

Edge-case verification:

Single cell query at (0,0): pre[1][1] - pre[0][1] - pre[1][0] + pre[0][0] = matrix[0][0] − 0 − 0 + 0 = matrix[0][0]. ✓
Whole matrix (r1=c1=0, r2=m-1, c2=n-1): pre[m][n] - 0 - 0 + 0 = pre[m][n]. ✓

4 The Code (Annotated) ▾

Solution 1 — 2D Prefix Sum (Optimal — Interview Pick)

class NumMatrix:
def __init__(self, matrix): ← Called once at construction time
m, n = len(matrix), len(matrix[0]) ← Grid dimensions (rows, cols)
self.pre = [[0]*(n+1) for _ in range(m+1)] ← (m+1)×(n+1) zeros — sentinel borders
for i in range(m): ← Fill row by row
for j in range(n): ← Fill col by col (left to right)
self.pre[i+1][j+1] = ( ← 1-indexed in prefix, 0-indexed in matrix
matrix[i][j] ← current cell value
+ self.pre[i][j+1] ← rectangle strictly above current cell
+ self.pre[i+1][j] ← rectangle strictly left of current cell
- self.pre[i][j] ← subtract top-left corner (double-counted)
)
def sumRegion(self, r1, c1, r2, c2): ← O(1) — just 4 lookups + 3 arithmetic ops
return (
self.pre[r2+1][c2+1] ← sum of entire (0,0)→(r2,c2) rectangle
- self.pre[r1][c2+1] ← remove strip above query: (0,0)→(r1-1,c2)
- self.pre[r2+1][c1] ← remove strip left of query: (0,0)→(r2,c1-1)
+ self.pre[r1][c1] ← add back corner subtracted twice
)
⚠️ Why not store matrix and compute on the fly?

A single brute-force query takes O((r2−r1+1)×(c2−c1+1)) time — potentially O(m·n) per call. With k queries, total cost is O(k·m·n). For 10,000 queries on a 200×200 matrix that's 400 million operations. The prefix sum pays O(m·n) once and reduces each query to O(1): total O(m·n + k). Dramatically better when k is large.

Solution 2 — Row-Only Prefix Sum (partial optimization)

class NumMatrix:
def __init__(self, matrix): ← Precompute prefix sums within each row only
self.row_pre = []
for row in matrix:
rp = [0]
for v in row: rp.append(rp[-1] + v) ← 1D prefix sum for this row
self.row_pre.append(rp)
def sumRegion(self, r1, c1, r2, c2): ← O(m) per query — still linear in rows
total = 0
for r in range(r1, r2+1): ← sum each row's range using row prefix
total += self.row_pre[r][c2+1] - self.row_pre[r][c1]
return total ← Better than brute force, but not O(1)
💡 Complexity Summary

2D Prefix Sum (optimal): Init O(m·n), Query O(1), Space O(m·n)
Row-only prefix sum: Init O(m·n), Query O(m), Space O(m·n)
Brute force: Init O(1), Query O(m·n), Space O(1)
For large k (many queries), the 2D prefix sum dominates. For a single query, brute force is fine.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Spot the "Many Queries on Immutable Data" Pattern

The problem says the matrix never changes and sumRegion will be called many times. This is the classic signal for preprocessing. Say: "Since the matrix is immutable, I should precompute something that makes each query O(1), trading space for time."

Step 2: Extend 1D Prefix Sums to 2D

"In 1D, prefix[i] = sum of first i elements; range sum(l,r) = prefix[r+1] − prefix[l]. In 2D, pre[i][j] = sum of top-left rectangle. Rectangle query = inclusion-exclusion of four corner lookups."

Step 3: Draw the Inclusion-Exclusion Diagram

Draw on paper: "The big rectangle minus the strip above, minus the strip to the left. The top-left corner gets subtracted twice, so I add it back once. This is exactly the formula: pre[r2+1][c2+1] − pre[r1][c2+1] − pre[r2+1][c1] + pre[r1][c1]."

Step 4: Justify the (m+1)×(n+1) Table Size

"I'll use a 1-indexed prefix table with sentinel zeros on row 0 and column 0. This makes the formula work uniformly for any query including those starting at row 0 or column 0 — no if-else boundary guards."

🗣️ What to Say to the Interviewer

"The matrix is immutable and sumRegion is called many times — classic preprocessing problem.
I'll build a 2D prefix sum table in O(m·n) where pre[i][j] = sum of top-left rectangle to (i-1,j-1).
Build: pre[i+1][j+1] = matrix[i][j] + pre[i][j+1] + pre[i+1][j] - pre[i][j].
Query: pre[r2+1][c2+1] - pre[r1][c2+1] - pre[r2+1][c1] + pre[r1][c1].
Init: O(m·n). Each query: O(1). Space: O(m·n)."
🔮 Likely Follow-up Questions
  • "What if the matrix is mutable (updates allowed)?" → 2D Binary Indexed Tree (Fenwick Tree): O(log m · log n) per update and query (#308)
  • "How does this extend to 3D?" → 3D prefix sum: pre[i][j][k] = sum of top-left cuboid; queries need 8 inclusion-exclusion terms
  • "Find the maximum submatrix sum?" → Fix two rows, apply Kadane's 1D on the column sums — O(m²·n)
  • "Count sub-matrices summing to target T?" → Fix two rows, use 1D prefix sum + hash map — O(m²·n) — LeetCode #1074

7 Summary & Key Takeaways ▾
Immutable matrix + many queries → precompute 2D prefix table in O(m·n); answer each query in O(1)
pre[i][j] = sum of top-left rectangle from (0,0) to (i-1,j-1) — the core invariant
Build recurrence uses inclusion-exclusion: current cell + above + left − corner (to fix double-count)
Query formula: big rectangle − strip above − strip left + corner (same inclusion-exclusion principle)
Use (m+1)×(n+1) with sentinel zeros: eliminates all boundary conditions from both build and query formulas
For mutable matrices, upgrade to 2D Binary Indexed Tree (O(log m · log n) per operation)
