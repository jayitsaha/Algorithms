LeetCode #167

Two Sum II — Input Array Is Sorted

Given a 1-indexed sorted array, find two numbers that add up to the target. Return their 1-indexed positions. Each input has exactly one solution.

Medium Two Pointers Opposite Direction Time: O(n) Space: O(1)
1 Understanding the Problem ▾

You are given a sorted array of integers and a target sum. Find exactly two numbers in the array that add up to the target, and return their 1-based indices (the problem says the first index is 1, not 0). You are guaranteed exactly one valid answer exists.

💡 The Key Observation

Because the array is sorted, small values are on the left and large values are on the right. If our current sum is too small, we need a bigger value — move the left pointer right. If the sum is too big, we need a smaller value — move the right pointer left. This is the core of the two-pointer technique.

Consider numbers = [2, 7, 11, 15], target = 9. Walking through small examples:

✓ What we're finding

numbers[0] + numbers[1] = 2 + 7 = 9
target = 9 ✓
Answer: [1, 2] (1-indexed)

✗ What we don't want

Check every pair: O(n²) pairs
Brute force is too slow for large arrays
The sorted property gives us O(n)

Critical constraint: The problem says you must use only O(1) extra space. No hash map allowed (that would be Two Sum I). The sorted property is the key that lets us avoid extra memory.

⚠️ 1-Indexed Output

The problem returns 1-indexed positions. At the end, add 1 to each 0-indexed pointer value. A common mistake is forgetting this and returning [0, 1] instead of [1, 2].

2 The Algorithm: Opposite-Direction Two Pointers ▾

Place one pointer at the far left (smallest element) and another at the far right (largest element). Compute their sum and squeeze inward based on the comparison with target.

1 Initialize pointers: left = 0, right = len(numbers) - 1. These point to the smallest and largest elements respectively.
2 Compute current sum: s = numbers[left] + numbers[right]. This is the sum of the two candidates we're currently examining.
3 Compare with target:
• If s == target → found! Return [left+1, right+1].
• If s < target → sum too small; we need a bigger number → left += 1.
• If s > target → sum too big; we need a smaller number → right -= 1.
4 Repeat: Keep looping while left < right. The problem guarantees exactly one answer exists, so we will always find it before the pointers cross.

Decision at Each Step

Sum < Target
Sum is too small. Move left pointer right to get a bigger left value.
⟶
Sum > Target
Sum is too big. Move right pointer left to get a smaller right value.
Sum = Target
Found! Return [left+1, right+1] (1-indexed).
🔑 Why Can't We Miss the Answer?

Every time we move a pointer, we eliminate all pairs containing the old pointer position with any remaining element on the other side. We can do this safely because we've already determined those pairs can't produce the target sum (they're either all too big or all too small). The answer always survives this elimination.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every step, the answer (if it exists) lies within the subarray numbers[left..right]. We never eliminate the correct pair from consideration — we only eliminate pairs that cannot be the answer.

Proof sketch for correctness:

  • When s < target, every pair (left, x) where x ≤ right gives a sum ≤ s < target (since numbers[x] ≤ numbers[right] for a sorted array). So no pair using the current left can be the answer. Safe to advance left.
  • When s > target, by symmetric logic, every pair (x, right) where x ≥ left gives a sum ≥ s > target. So no pair using the current right can be the answer. Safe to retreat right.

Termination: With each step, the window [left, right] shrinks by exactly 1. Since the problem guarantees a solution, we'll find it before left == right (a single element can't pair with itself).

⚠️ Why This Doesn't Work on an Unsorted Array

The invariant relies on sorted order. If we advance left because the sum is too small, we need to know that numbers[left+1] > numbers[left] — so the new sum is larger. In an unsorted array, moving left might give us an even smaller number and the logic breaks. Sorting first and then applying this technique on an unsorted array costs O(n log n).

4 The Code (Annotated) ▾

Solution 1 — Two Pointers (Optimal, Interview Pick)

def twoSum(numbers: List[int], target: int) -> List[int]:
left, right = 0, len(numbers) - 1 ← start at both ends of the sorted array
while left < right: ← pointers must be distinct (can't use same element twice)
s = numbers[left] + numbers[right] ← sum of current candidates
if s == target: ← found the pair!
return [left + 1, right + 1] ← convert 0-indexed → 1-indexed
elif s < target: ← sum too small: need a larger number
left += 1 ← move left pointer right (next bigger value)
else: ← sum too big: need a smaller number
right -= 1 ← move right pointer left (next smaller value)
⚠️ Why not left += 1; right -= 1 when equal?

When we find s == target, we return immediately. We don't need to check if there are other pairs — the problem guarantees exactly one solution. Some beginners write unnecessary continuation logic here.

Solution 2 — Binary Search (O(n log n), for reference)

def twoSum(numbers: List[int], target: int) -> List[int]:
for i, num in enumerate(numbers): ← iterate through each element as the first
complement = target - num ← what value do we need as the second element?
lo, hi = i + 1, len(numbers) - 1 ← binary search in the remainder of array
while lo <= hi: ← standard binary search loop
mid = (lo + hi) // 2
if numbers[mid] == complement:
return [i + 1, mid + 1] ← 1-indexed answer
elif numbers[mid] < complement: lo = mid + 1
else: hi = mid - 1

Solution 3 — Brute Force (O(n²), never use in interview)

def twoSum(numbers, target): ← check every pair — O(n²) time, O(1) space
for i in range(len(numbers)): ← outer loop: pick first element
for j in range(i+1, len(numbers)): ← inner loop: pick second element
if numbers[i] + numbers[j] == target:
return [i+1, j+1] ← found — but very slow!
💡 Complexity Summary

Two Pointers (optimal): Time O(n) — each element is visited at most once. Space O(1) — only two pointer variables.
Binary Search: Time O(n log n) — n outer iterations, O(log n) per binary search. Space O(1).
Brute Force: Time O(n²). Space O(1) — but too slow for large inputs.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Sorted Signal

The problem says "sorted array" and "O(1) extra space." These two constraints together scream two pointers. If it were unsorted with O(n) space allowed, you'd use a hash map (Two Sum I). The sorted property replaces the hash map.

Step 2: Place Pointers at Extremes

Start at both ends. The smallest element is on the left; the largest is on the right. Their sum is the "range" you're working within. Moving left inward increases the sum; moving right inward decreases it — this is your control mechanism.

Step 3: State the Invariant Aloud

Tell the interviewer: "At each step, if the answer exists, it must be within the current window [left, right]. Each move eliminates at least one candidate that cannot be the answer." This shows you understand correctness, not just mechanics.

Step 4: Handle the 1-Indexed Return

Don't forget: the problem is 1-indexed. When you return, do [left+1, right+1]. Mention this to the interviewer so they know you read the problem carefully — it's a common trap.

🗣️ What to Say to the Interviewer

"I notice the array is sorted and we need O(1) space. That combination points to two pointers.
I'll put one pointer at the start (smallest value) and one at the end (largest value).
If their sum equals the target, I'm done — return those indices (1-indexed).
If the sum is too small, I need a bigger value, so I move the left pointer right.
If the sum is too big, I need a smaller value, so I move the right pointer left.
Each step eliminates at least one candidate that can't be the answer.
Since a solution is guaranteed, I'll find it in O(n) time with O(1) space."
🔮 Likely Follow-up Questions
  • "What if the array is not sorted?" → Use hash map (Two Sum I, #1) — O(n) time/space
  • "Three Sum?" → Fix one element, run two pointers on remainder (#15)
  • "Four Sum?" → Fix two elements, run two pointers on remainder (#18)
  • "What if there are multiple answers?" → Collect all valid pairs, still O(n)
  • "What if the target doesn't exist?" → Return [] when pointers cross
