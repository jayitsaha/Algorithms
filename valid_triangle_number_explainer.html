LeetCode #611

Valid Triangle Number

Given an integer array nums, return the number of triplets chosen from the array that can make triangles if we take them as side lengths of a triangle.

Medium Mathematics & Geometry Sort + Two Pointers Time: O(n²) Space: O(1)
1 Understanding the Problem ▾

A triangle is valid when each pair of sides can "close" — meaning no single side is so long that the other two can't reach each other. Formally: a + b > c, a + c > b, and b + c > a must all hold. That's three conditions to check for every triplet.

💡 Key Insight: Sort to Reduce 3 Conditions to 1

If we sort so that a ≤ b ≤ c, then b + c > a and a + c > b are automatically satisfied (since c alone is already ≥ a and ≥ b). The only condition that can fail is a + b > c. Three checks collapse into one!

✓ Valid: [3, 4, 6]

3 + 4 = 7 > 6 ✓
Sides close together

✗ Invalid: [1, 2, 10]

1 + 2 = 3 < 10 ✗
Tiny sides, giant third

✗ Degenerate: [2, 4, 6]

2 + 4 = 6 = 6 ✗
Need STRICT > not ≥

Edge Cases: Arrays shorter than 3 → 0 triangles. Any zero in the triplet → impossible (0 + b ≤ c for any c ≥ b). Duplicate values like [2,2,3] → valid triangle (2+2=4 > 3). The answer can be large but fits in a standard int for LeetCode constraints.

2 The Algorithm: Fix-Largest + Two-Pointer Sweep ▾

After sorting, we iterate k from n−1 down to 2, treating nums[k] as the largest side. For each k, two pointers left=0 and right=k−1 sweep the prefix to count valid pairs.

1 Sort nums ascending. This ensures that for indices i < j < k, we have nums[i] ≤ nums[j] ≤ nums[k], and only the condition nums[i] + nums[j] > nums[k] can fail.
2 Fix k = n−1 down to 2. nums[k] is the candidate largest side. For each k, we need to count pairs (left, right) in the prefix [0..k−1] where nums[left] + nums[right] > nums[k].
3 Case: nums[left] + nums[right] > nums[k]. This means nums[left+1], nums[left+2], ..., nums[right−1] are all at least nums[left], so pairing ANY of them with nums[right] also exceeds nums[k]. That's (right − left) valid pairs. Add it, then shrink right.
4 Case: nums[left] + nums[right] ≤ nums[k]. Even the smallest remaining value (nums[left]) paired with nums[right] is too small. Advance left to try a larger value.
5 Repeat until left meets right, then move to the next k. The entire inner loop runs O(n) per k value, giving O(n²) total.
🔑 The Batch Counting Step

When nums[left] + nums[right] > nums[k], the valid "left partner" indices with right are: left, left+1, …, right−1. Count = right − left. We credit all of them at once and then move right inward. This O(1) batch is what converts a naïve O(n³) algorithm into O(n²).

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every moment during the two-pointer sweep for fixed k: all triplets that have been counted are genuinely valid, and every valid triplet with nums[k] as the largest side will eventually be counted — none are skipped.

Correctness of batch count: When nums[left] + nums[right] > nums[k], consider any index m with left ≤ m < right. Since the array is sorted, nums[m] ≥ nums[left], so nums[m] + nums[right] ≥ nums[left] + nums[right] > nums[k]. Every pair (m, right) with m in [left, right−1] is valid. Adding (right − left) is exact.

No skipped pairs: When nums[left] + nums[right] ≤ nums[k], nums[left] is too small — even pairing it with a larger right value won't help for this specific right (the pair only gets harder as right decreases). We correctly advance left. Symmetrically, once right moves down, we don't need to re-examine pairs with the old right value alongside the new left, because those were already counted or correctly skipped.

⚠️ Why Not Fix the Smallest Side?

If we fix the smallest (i) and two-pointer for j and k, we'd need pairs where nums[j] + nums[k] > nums[i]. Since nums[i] is the smallest, this is almost always true, and the batch-counting trick doesn't work cleanly in the same direction. Fixing the LARGEST side gives the clean "either all left-of-right pass, or none" property.

4 The Code (Annotated) ▾

Solution 1 — Sort + Two Pointers (Optimal, Interview Pick)

def triangleNumber(nums: list[int]) -> int:
nums.sort() ← O(n log n); reduces 3 triangle conditions to 1
n = len(nums)
count = 0 ← Running count of valid triplets
for k in range(n-1, 1, -1): ← k is the LARGEST side; k starts at n-1 and goes to 2
left, right = 0, k-1 ← Two pointers span everything to the left of k
while left < right: ← Continue until pointers meet
if nums[left] + nums[right] > nums[k]:
count += right - left ← ALL pairs (left..right-1, right) valid; batch count
right -= 1 ← Shrink right; its contribution is fully accounted for
else:
left += 1 ← nums[left] too small to partner with any right; advance
return count ← Total valid triangles found
⚠️ Off-by-one: why right − left, not right − left + 1?

When nums[left] + nums[right] > nums[k], the valid left indices for this right value are: left, left+1, ..., right−1. That's exactly (right−1) − left + 1 = right − left values. We do NOT count index right itself as "left" because we need left < right (three distinct indices). So the count is right − left, not right − left + 1.

Solution 2 — Brute Force O(n³)

def triangleNumber_brute(nums):
nums.sort() ← Still sort so only one condition needed
count = 0
n = len(nums)
for i in range(n-2): ← Fix smallest side
for j in range(i+1, n-1): ← Fix middle side
for k in range(j+1, n): ← Largest side
if nums[i]+nums[j] > nums[k]:
count += 1
return count
💡 Complexity Summary

Sort + Two Pointers (optimal): Time O(n² + n log n) = O(n²), Space O(1)
Brute Force: Time O(n³), Space O(1) — TLE for n > ~1000
Why O(n²) is optimal here: Output can be O(n²) triplets (e.g., all-equal array [k,k,k,...] gives C(n,3) = O(n³) triangles but the COUNT can still be computed in O(n²))

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Reduce the Problem

"Triangle needs a+b > c, a+c > b, b+c > a. If I sort the array so a ≤ b ≤ c, only a+b > c can fail. Now I need to count sorted triplets satisfying one inequality."

Step 2: State the Brute Force

"Three nested loops: fix i, j, k with i < j < k and check nums[i]+nums[j] > nums[k]. O(n³) — too slow for n=1000."

Step 3: Spot the Two-Pointer Opportunity

"If I fix k (largest side) and set two pointers left=0, right=k−1 — when nums[left]+nums[right] > nums[k], ALL pairs from left to right−1 with nums[right] are also valid (sorted order guarantees this). I can count them in O(1) and shrink right. Otherwise advance left. Inner loop is O(n), so total O(n²)."

Step 4: Handle Edge Cases

Zero sides always fail (0+b=b≤c). Arrays with fewer than 3 elements return 0. Duplicates work naturally — sort puts them adjacent and the comparison handles them correctly.

🗣️ What to Say to the Interviewer

"I'll sort the array first. Once sorted, a triplet (i,j,k) with i<j<k is a valid triangle iff nums[i]+nums[j]>nums[k].
I'll fix k from right to left, and run two pointers left=0, right=k-1 on the prefix.
If nums[left]+nums[right]>nums[k]: all right-left pairs with nums[right] as second element are valid — count+=right-left, shrink right.
Else: nums[left] is too small even for this right, advance left.
O(n log n) sort + O(n²) sweep = O(n²), O(1) extra space."
🔮 Likely Follow-up Questions
  • "What about floating-point sides?" → Same algorithm; comparison operators handle floats, no changes needed
  • "Can you enumerate all valid triplets?" → Must return the pairs explicitly; still O(n²) outer loop but may output O(n³) triplets
  • "How is this related to 3Sum?" → Same "sort + fix one + two-pointer" template, but instead of target sum we have an inequality condition
  • "Is O(n log n) possible?" → Not known for general case; O(n²) is the accepted optimal
