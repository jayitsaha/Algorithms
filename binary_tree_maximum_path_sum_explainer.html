LeetCode #124

Binary Tree Maximum Path Sum

Find the maximum sum of any path in the tree, where a path is any sequence of connected nodes (does not need to pass through root).

Hard Trees DFS Postorder Time: O(n) Space: O(h)
1 Understanding the Problem ▾

A path in a binary tree is a sequence of nodes where each adjacent pair shares an edge. The path must not revisit the same node. Crucially, the path does not need to pass through the root — it can start and end anywhere in the tree, as long as it goes in one continuous direction (no branching).

💡 Key Insight: Every Path Has a Turning Node

Every valid path in a binary tree has exactly one highest (turning) node — the topmost node on the path. At that node, the path may come from the left subtree, go through the node itself, and continue into the right subtree. Below the turning node, the path only goes one direction. This structure lets us evaluate every possible path by considering each node as a potential turning point.

At every node, we face two separate questions simultaneously:

  • As the turning point: What is the best path that uses this node as its apex? (May include left arm + node + right arm.) This updates the global best.
  • As a contributor: What is the best one-sided arm I can offer my parent? (Only one direction — the path cannot branch at me from my parent's perspective.)

Important edge cases: All node values may be negative — the answer is the single least-negative node value. Single-node tree is valid. A path of length 1 (a single node) is always valid. You cannot skip nodes in the middle of a path.

2 The Algorithm: Postorder DFS — Two Roles per Node ▾

We perform a postorder DFS (Left → Right → Node). Each recursive call simultaneously: (a) updates the global answer by treating the current node as a turning point, and (b) returns the best one-sided arm it can contribute to its parent.

1 Base case: If the node is None, return 0. A null child contributes nothing to any arm.
2 Get left arm: Call dfs(node.left) and clamp to 0: left = max(0, dfs(node.left)). If the left subtree's best path is negative, we simply don't include it — starting fresh at this node is better.
3 Get right arm: Same with node.right. Both arms are now non-negative values (or 0 if we skip a subtree).
4 Update global answer: The best path through this node as the turning point = left + node.val + right. This covers all 4 types: single node (both arms = 0), left arm only (right = 0), right arm only (left = 0), and full bridge (both > 0). Update the global max.
5 Return best arm upward: The parent can only extend us in one direction, so return node.val + max(left, right). The parent's max(0, ...) call will decide whether to use us at all.
Four Path Types Through Any Node (all covered by left + node + right with max(0) clamping)
🔑 The Two-Jobs Trick

Each recursive call does two jobs at once. The return value (one-sided arm) serves the parent. The side-effect (updating global ans) captures the best path using this node as its turning point. These are computed simultaneously — no second pass needed.

3 Why It Works: The Invariant ▾
🔐 The Invariant

dfs(node) returns the maximum one-sided arm rooted at node: the best sum of a downward path starting from this node, guaranteed ≥ 0 (thanks to the parent's max(0, ...) call). The global ans accumulates the best turning-point path seen across all nodes.

Why clamping arms to 0 is correct: A path may "start fresh" at any node — it is not required to include a parent. max(0, dfs(child)) models the decision: "include this subtree in my path or don't." If the subtree's best contribution is negative, we choose not to include it (treat it as 0).

Why postorder is required: To evaluate a node as a turning point, we need both subtree arms already computed. Postorder processes all descendants before their ancestor, so when we reach a node, both arms are already known and correct.

Why every path is considered exactly once: Every path has a unique topmost (turning) node. Our DFS visits every node exactly once and evaluates it as a potential turning node. Since every node is visited, every path's turning node is evaluated, so every path is considered. No path is double-counted because a path has exactly one turning node.

⚠️ Critical Mistake: Returning the Two-Sided Path Value to Parent

A common bug: returning left + node.val + right to the parent instead of node.val + max(left, right). The value left + node.val + right represents a path that branches at this node — the parent cannot extend it further without creating a fork. Only return the single best arm upward; reserve the two-sided value for the global answer update.

All-negative trees: We initialize ans = float('-inf'), not 0. The problem guarantees at least one node, so the answer must be at least one node's value (possibly very negative). The clamping max(0, arm) only affects arms passed between recursive calls, not the global answer itself.

4 The Code (Annotated) ▾

Solution 1 — Postorder DFS (Optimal · Interview Pick)

def maxPathSum(root: TreeNode) -> int:
ans = float('-inf') ← -inf handles all-negative trees; must be at least one node
def dfs(node): ← Returns best ONE-SIDED arm from this node downward (≥0)
nonlocal ans ← Allows mutation of ans from outer scope
if not node: ← Base case: null child contributes 0 to any arm
return 0
left = max(0, dfs(node.left)) ← Best left arm; 0 means "skip this subtree"
right = max(0, dfs(node.right)) ← Best right arm; 0 means "skip this subtree"
ans = max(ans, left + node.val + right) ← Covers all 4 path types through this turning point
return node.val + max(left, right) ← ONE-SIDED arm: parent picks left or right, not both
dfs(root) ← Trigger the traversal (return value unused)
return ans ← Global max across all n turning points
💡 How max(0, arm) Collapses All 4 Cases

When left=0 and right=0: candidate = node.val (single node path). When only left>0: candidate = left + node.val. When only right>0: candidate = node.val + right. When both>0: candidate = left + node.val + right. One formula handles all four cases gracefully.

Solution 2 — Brute Force Enumeration (O(n²)) — for comparison only

def maxPathSum_brute(root): ← O(n²): for each node as turning point, compute both arms from scratch
def best_arm(node): ← Max one-sided path sum starting at node
if not node: return 0
return node.val + max(0, best_arm(node.left), best_arm(node.right))
if not root: return 0
# For root as turning point:
through_root = max(0,best_arm(root.left)) + root.val + max(0,best_arm(root.right))
# Recurse into subtrees:
return max(through_root, maxPathSum_brute(root.left), maxPathSum_brute(root.right))
# O(n²): best_arm is called O(n) times, each O(n) — fix by merging into one DFS
⚠️ The nonlocal / List-Trick Gotcha

In Python 3, nonlocal ans is clean. In Python 2 (no nonlocal), use ans = [float('-inf')] and access via ans[0]. In interviews using Python 3, nonlocal is preferred for clarity. Some interviewers use a class wrapper: self.ans = float('-inf') — all three are acceptable.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Clarify the Problem Definition

Ask: "Does the path need to go through the root?" (No.) "Can values be negative?" (Yes — crucial for initialization.) "Is a single node a valid path?" (Yes.) "Must the path be contiguous?" (Yes.) These clarifications show you understand the subtleties.

Step 2: Identify the "Turning Node" Insight

"Every path has a topmost node — the turning point. If I fix that node, the best path through it uses the best left arm and best right arm below it. This suggests postorder DFS: compute subtree answers first, then use them at the parent."

Step 3: Design the Return Value

"My function has two jobs: update the global answer with the two-sided path (left + me + right), and return the one-sided arm to the parent. These are different values computed at the same node."

Step 4: Handle Negatives Carefully

"I clamp arms to 0 using max(0, ...) so negative subtrees are simply skipped. But I initialize ans = -inf (not 0) because the answer must include at least one actual node — even if all nodes are negative."

🗣️ What to Say to the Interviewer

"Every path has a unique turning node — the highest point on the path.
I'll do postorder DFS: recurse into both children, then process the current node.
At each node, I collect left arm = max(0, dfs(left)) and right arm = max(0, dfs(right)).
The max(0, ...) lets me skip negative subtrees — the path can start fresh anywhere.
I update a global ans with left + node.val + right — that's the turning-point candidate.
I return node.val + max(left, right) — the best one-sided arm I can offer my parent.
Time O(n), Space O(h) — each node visited once; stack depth equals tree height."
🔮 Likely Follow-up Questions
  • "Print the actual path, not just the sum?" → Track parent pointers or reconstruct by re-running DFS from the max node
  • "What if all values are negative?" → ans = -inf ensures the max single node is returned; our algorithm already handles it
  • "Space complexity for a skewed tree?" → O(n) — the recursion stack depth equals the tree height, which is O(n) for a skewed tree
  • "What about a general tree (not binary)?" → Same idea: sort children arms descending, keep top 2 for turning-point, top 1 for arm returned upward

7 Summary & Key Takeaways ▾
Every path has exactly one turning (topmost) node — the DFS considers each node in that role
Postorder DFS: children must be processed first to provide arms for the parent's evaluation
max(0, arm) skips negative subtrees cleanly — the path may start fresh at any node
Return value (arm) serves the parent; global ans update serves the answer — two jobs per call
Initialize ans = float('-inf') — not 0 — to handle all-negative trees correctly
ApproachTimeSpaceNotes
Brute Force — separate arm computation per node O(n²) O(h) Correct but slow; easy to derive then optimize
Postorder DFS — one pass ✓ O(n) O(h) Optimal; h = O(log n) balanced, O(n) skewed
🎯 When to Recognize the "Max Path Through Node" Pattern
  • "Find max/min sum/length path in binary tree (any start, any end)"
  • "Optimize something over all paths in the tree"
  • Recursive function needs a return value for parent AND a side effect for global answer
  • Clamping arms to 0 allows "start fresh" anywhere — signals this pattern
