LeetCode #646

Maximum Length of Pair Chain

Given an array of pairs [a, b] where a < b, find the longest chain where each pair's left element is strictly greater than the previous pair's right element.

Medium Dynamic Programming Greedy LIS on Intervals Greedy: O(n log n) Time, O(1) Space DP: O(n²) Time, O(n) Space
1 Understanding the Problem ▾

You have a list of pairs [a, b] where a < b always. A chain is a sequence of pairs where each pair's left value is strictly greater than the previous pair's right value. Your goal: find the maximum length such chain, picking any subset of pairs in any order.

💡 Key Insight

Think of each pair as an interval [a, b] on a number line. Two pairs can be chained if they don't overlap — and specifically if the second one starts strictly after the first one ends. The "maximum chain" is exactly the Maximum Non-Overlapping Intervals problem. You can sort and reorder freely — pairs are not pinned to positions.

✓ Valid chain of length 3

[1,2] → [3,4] → [5,6]
3 > 2 ✓   5 > 4 ✓
Each starts after the previous ends.

✗ Invalid: [1,5] then [5,7]

5 is NOT > 5 — need strictly greater
[1,5] then [6,7] would work (6 > 5)
Equal endpoints don't count!

Why can we reorder? The problem says we pick any subset and arrange them however we want. There's no positional constraint — this is purely about finding the maximum compatible subset of intervals.

Edge cases: Single pair → answer is 1. All pairs overlapping → answer is 1. Pairs with negative values → same algorithm works. Pairs given in any order → must sort first.

2 The Algorithm: Two Approaches — DP and Greedy ▾

Approach A: Greedy — "Earliest Deadline First"

Sort pairs by their right (end) value. Then scan left-to-right: whenever a pair's left value is strictly greater than our current chain_end, greedily take it. This "earliest finish first" heuristic maximizes future choices.

1 Sort by right endpoint: pairs.sort(key=lambda p: p[1]). Earliest-ending pairs come first. This is the key invariant of the greedy: at each step, the pair we consider ends as early as possible.
2 Initialize: chain_end = -∞ (chain is empty), count = 0.
3 Scan each pair [a, b]: If a > chain_end, this pair extends the chain. Take it: count += 1, chain_end = b. Otherwise, skip — it overlaps with our current chain, and our already-chosen pair ends earlier anyway.
4 Return count.

Approach B: DP (LIS-style) — Tabulation

Sort by end value. Define dp[i] = length of the longest chain ending exactly at pair i. For each pair i, look back at all pairs j < i: if pairs[j][1] < pairs[i][0], we can extend. Take the maximum.

🔑 Why Sort by End (Not Start)?

If you sort by start value, greedily picking the first compatible pair might take one that ends very late, blocking many future pairs. Sorting by end ensures: the pair we evaluate ends as soon as possible, leaving the maximum room for the future. For DP, it ensures that when we compute dp[i], all valid predecessors j have already been computed.

⚠️ DP vs Greedy — Two Valid Paths

Both are correct. The DP approach (O(n²)) directly mirrors the LIS recurrence — it's excellent for interview pattern recognition. The Greedy approach (O(n log n)) is optimal and concise. In interviews, mention DP first to show pattern recognition, then optimize to greedy. The walkthrough below shows the greedy.

3 Why It Works: DP Invariant & Greedy Proof ▾

Why is this DP? To find the longest chain ending at pair i, we need to know the longest chains ending at all pairs j < i — that's optimal substructure. A brute-force recursion would recompute the same subproblems repeatedly — that's overlapping subproblems. DP eliminates duplication.

📐 DP Recurrence (LIS on Intervals)
Sort pairs by right endpoint.
dp[i] = longest chain ending exactly at pair i (base case: 1)

For each i, for each j < i where pairs[j][1] < pairs[i][0]:
    dp[i] = max(dp[i], dp[j] + 1)

Answer = max(dp[i]) over all i
🔐 The Greedy Invariant

After processing k pairs and taking count of them, chain_end equals the minimum possible right endpoint of any chain of length count. By always choosing the earliest-ending compatible pair, we guarantee that chain_end is as small as possible — creating the most opportunity for future pairs.

Greedy proof by exchange argument: Suppose an optimal solution picks pair X at some step, but we pick pair Y instead (both compatible, but Y ends earlier — Y was processed first in sorted order). We can swap X for Y: since Y ends ≤ X's end, any pair that follows X also follows Y. So the swap doesn't decrease the chain length. Therefore, always picking the earliest-ending compatible pair is safe — it never leads to a suboptimal result.

💡 State Machine View

Imagine a cursor sitting at chain_end. Each pair either moves the cursor forward (extend chain) or is rejected (overlap). The greedy minimizes each cursor advance, ensuring the cursor stays as far left as possible for maximum future extensions. Every advance moves the cursor to the new right endpoint.

Edge case correctness: Single pair → loop runs once, a > -∞ is always true, returns 1. All overlapping → only the first (earliest-ending) pair is taken, returns 1. Negative pairs → works identically.

4 The Code (Annotated) ▾

Solution 1 — Greedy (Interview Pick): O(n log n) time, O(1) space

def findLongestChain(pairs):
pairs.sort(key=lambda p: p[1]) ← Sort by RIGHT endpoint — greedy ordering
chain_end = float('-inf') ← Empty chain; -inf means any pair can start it
count = 0 ← Length of the chain we've built so far
for a, b in pairs: ← Scan pairs in sorted order
if a > chain_end: ← Strictly greater — this pair starts after chain ends
count += 1 ← Extend the chain
chain_end = b ← Update the cursor: chain now ends at b
return count ← Maximum chain length
⚠️ Strictly Greater: use a > chain_end, NOT a >= chain_end

The problem requires the left of the next pair to be strictly greater than the right of the previous pair. Pairs like [1,5] and [5,7] do NOT form a valid chain — 5 is not strictly greater than 5. Use >, not >=.

Solution 2 — DP Tabulation (Bottom-Up): O(n²) time, O(n) space

def findLongestChain_dp(pairs):
pairs.sort(key=lambda p: p[1]) ← Sort by end; ensures valid ordering for DP
n = len(pairs)
dp = [1] * n ← dp[i] = longest chain ending at pair i; every pair alone = 1
for i in range(1, n): ← For each pair, compute its best chain length
for j in range(i): ← Check all previous pairs as possible predecessors
if pairs[j][1] < pairs[i][0]: ← j can directly precede i (no overlap, strictly less)
dp[i] = max(dp[i], dp[j]+1) ← Extend j's chain with i; take the best extension
return max(dp) ← Best chain tail across all pairs

Solution 3 — Memoization (Top-Down): O(n²) time, O(n) space

from functools import lru_cache
def findLongestChain_memo(pairs):
pairs.sort(key=lambda p: p[1]) ← Sort once; indices now correspond to sorted order
n = len(pairs)
@lru_cache(maxsize=None) ← Cache results; repeated calls return instantly
def dp(i): ← dp(i) = longest chain starting from index i
best = 1 ← pairs[i] alone is a chain of length 1
for j in range(i+1, n): ← Try every later pair as the next link
if pairs[j][0] > pairs[i][1]: ← j can follow i (strictly non-overlapping)
best = max(best, 1 + dp(j)) ← chain(i) = 1 + best chain starting at j
return best
return max(dp(i) for i in range(n)) ← Try every starting pair; return global max
💡 Complexity Summary

Greedy: Sort O(n log n) + linear scan O(n) = O(n log n) time. O(1) extra space (or O(n) if sort allocates).
DP Tabulation: Two nested loops = O(n²) time. O(n) space for dp array.
Memoization: n states × up to n transitions = O(n²) time. O(n) cache + O(n) recursion stack.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Pattern

Signal words: "pairs/intervals", "chain", "next starts after previous ends", "maximum length". This is the interval scheduling pattern — or equivalently, LIS applied to 2D intervals. The fact you can freely reorder pairs confirms sorting is the first move.

Step 2: Brute Force to DP

"My first instinct: try all subsets. That's O(2ⁿ) — too slow. Notice subproblems repeat: 'longest chain ending at pair i' depends on 'longest chains ending at j < i'. So DP: dp[i] = max(dp[j] + 1) for compatible j. Sort by end, run O(n²) DP."

Step 3: Optimize to Greedy

"I notice the DP has a greedy structure: since pairs are sorted by end, the first compatible predecessor is always the best — we don't need to look at all j. This simplifies to: greedily take any pair that starts after chain_end, updating chain_end each time. O(n log n)."

Step 4: Validate with an Example

Walk through [[3,4],[1,2],[2,5],[5,6],[4,8]] by hand: sorted → [[1,2],[3,4],[2,5],[5,6],[4,8]]. Take [1,2] (count=1, end=2). Take [3,4] (count=2, end=4). Skip [2,5] (2≤4). Take [5,6] (count=3, end=6). Skip [4,8] (4≤6). Answer: 3.

🗣️ What to Say to the Interviewer

"I'll sort the pairs by their right endpoint — classic interval scheduling move.
I track chain_end, the right endpoint of the last chosen pair.
For each pair [a, b] in sorted order: if a > chain_end, the pair can extend the chain.
I take it greedily, increment count, and update chain_end = b.
If a ≤ chain_end, it overlaps — skip. The already-chosen pair ends earlier, which is better.
This greedy works by an exchange argument: earliest-finish-first never blocks future pairs.
Time: O(n log n). Space: O(1). I can also do it as O(n²) DP if you want the recurrence."
🔮 Likely Follow-up Questions
  • "Minimum pairs to remove to make a valid chain?" → n − greedy_count (Non-overlapping Intervals #435)
  • "Return the actual chain, not just its length?" → Track which pairs were greedily selected
  • "What if we allow equal endpoints (a ≥ prev_right)?" → Change > to >=
  • "Connect to LIS?" → Same dp[i]=max(dp[j]+1) recurrence, but with interval compatibility instead of numeric ordering

7 Summary & Key Takeaways ▾
Sort by right endpoint — the universal first step for all interval scheduling problems
Greedy: always take the earliest-ending compatible pair; skip overlapping ones
DP recurrence: dp[i] = max(dp[j]+1) for j where pairs[j][1] < pairs[i][0] — identical to LIS
Strictly greater than (>), not ≥ — touching endpoints are NOT a valid chain
n − greedy_count = minimum pairs to remove (Non-overlapping Intervals, #435)
ApproachTimeSpaceNotes
Brute Force (all subsets) O(2ⁿ) O(n) Try every subset; impractical
DP Tabulation (LIS-style) O(n²) O(n) Classic, shows pattern clearly; good interview start
Memoization (Top-Down) O(n²) O(n) Derives naturally from recursion; same complexity
Greedy (Sort + Scan) ✓ O(n log n) O(1)* Optimal; interview pick. *O(n) if sort allocates.
