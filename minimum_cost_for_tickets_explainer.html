LeetCode #983

Minimum Cost For Tickets

You have travel plans on specific days of a year. Buy 1-day, 7-day, or 30-day passes at given costs. Find the minimum total cost to cover all travel days.

Medium Dynamic Programming Min Across Passes Time: O(365) Space: O(365)
1 Understanding the Problem ▾

You're planning trips on certain days of the year. You know exactly which days you'll travel — let's say days 1, 4, 6, 7, 8, and 20. You can buy three types of travel passes: a 1-day pass at costs[0], a 7-day pass at costs[1] (covers any 7 consecutive days), and a 30-day pass at costs[2]. You can buy passes on any day, and passes cover consecutive days starting from that day. The question: what's the minimum total cost to ensure you're covered on every travel day?

💡 Key Insight

Every calendar day is either a travel day (must be covered by some active pass) or a non-travel day (coverage irrelevant). On non-travel days you never need to buy a pass — the cost through that day is identical to the cost through the previous day. On travel days you must pick the cheapest option: buy a 1-day, 7-day, or 30-day pass, each of which lets you "rewind" the appropriate number of days in the dp array.

Example Input

days = [1, 4, 6, 7, 8, 20]
costs = [2, 7, 15]
1-day=$2, 7-day=$7, 30-day=$15

Optimal Answer = $11

$2 day-1 pass on day 1
$7 week pass on day 4 (covers 4–10)
$2 day-1 pass on day 20
2 + 7 + 2 = $11

Edge cases to consider: If all days are clustered in a single week, one 7-day pass might beat all dailies. If travel days span 30 consecutive days, a 30-day pass might win. If days are scattered far apart, individual daily passes are cheapest. The DP considers all three options automatically at every travel day.

2 The Algorithm: Bottom-Up DP Over Calendar Days ▾

We define dp[d] as the minimum cost to cover all travel days from day 1 through day d. We fill this table from d = 1 to d = last_travel_day (at most 365). The key is distinguishing travel from non-travel days.

1 Build a set of travel days for O(1) membership testing. This lets us distinguish every calendar day as travel or non-travel in constant time.
2 Base case: dp[0] = 0. Zero days, zero cost. We allocate the dp array up to last_travel_day + 1.
3 Non-travel day: dp[d] = dp[d-1]. No pass is needed; the cost is identical to the day before. This is the "skip" transition.
4 Travel day — evaluate all three pass options:
• 1-day pass: costs[0] + dp[d-1] (only covers today)
• 7-day pass: costs[1] + dp[max(0, d-7)] (covers today back to d-6, so we need min cost only through d-7)
• 30-day pass: costs[2] + dp[max(0, d-30)] (covers today back to d-29)
5 Take the minimum of all three options. Store in dp[d].
6 Return dp[last_travel_day]. Non-travel days after the last travel day add nothing, so we stop there.
📐 The Recurrence Relations
if d not in travel_days:
    dp[d] = dp[d-1]

if d in travel_days:
    dp[d] = min(
        costs[0] + dp[d-1],
        costs[1] + dp[max(0, d-7)],
        costs[2] + dp[max(0, d-30)]
    )
🔑 Why "reach back" for 7-day and 30-day passes?

A 7-day pass purchased on or before day d covers day d if bought any time during [d-6, d]. The cheapest way to be covered on day d using a 7-day pass is to buy it exactly on day d — this means days [d-6, d-1] are covered "for free" by that pass. The cost we add is whatever we spent through day d-7, not d-1. Hence dp[max(0, d-7)].

3 Why Is This DP? Optimal Substructure & Overlapping Subproblems ▾
🔐 Optimal Substructure

The minimum cost to cover all travel days through day d depends on the optimal cost through day d-1, d-7, and d-30. If those sub-solutions are optimal, we can extend them with one pass choice to get an optimal solution for day d. You cannot skip optimizing subproblems — a globally optimal solution must contain optimal solutions to its subproblems.

🔐 Overlapping Subproblems

Without memoization, a naive recursive approach would recompute "min cost through day 5" over and over: once when computing a 1-day pass option for day 6, once when computing 7-day pass for day 12, etc. DP caches dp[d] after the first computation, so each value is computed exactly once — O(365) total work.

Why not Greedy? A greedy approach ("buy the cheapest pass right now") fails because a 7-day pass might look expensive today but saves money if you travel 4 more times this week. The benefit of a multi-day pass depends on future travel patterns — you need to account for what's coming, which requires dynamic programming over the full timeline.

The invariant: At any point, dp[d] holds the provably minimum cost to ensure all travel days from 1 to d are covered. Once computed, it never needs revision — we always process days in increasing order and each dp[d] depends only on earlier (already-computed) entries.

4 The Code (Annotated) ▾

Solution 1 — Tabulation / Bottom-Up DP (Interview Pick)

def mincostTickets(days, costs):
travel = set(days) ← O(1) membership check: "is day d a travel day?"
last = days[-1] ← Days is sorted; last travel day bounds our loop
dp = [0] * (last + 1) ← dp[0]=0 base case; dp[1..last] filled below
for d in range(1, last + 1): ← Iterate every calendar day, 1 through last travel day
if d not in travel: ← Non-travel day: no pass needed
dp[d] = dp[d - 1] ← Carry forward yesterday's minimum cost
else: ← Travel day: evaluate all three pass options
dp[d] = min( ← Take cheapest of three alternatives
costs[0] + dp[d - 1], ← 1-day pass: covers only today; reach back 1 day
costs[1] + dp[max(0, d - 7)], ← 7-day pass: covers days d-6 through d; reach back 7
costs[2] + dp[max(0, d - 30)] ← 30-day pass: covers d-29 through d; reach back 30
)
return dp[last] ← Minimum cost to cover all travel days
⚠️ Why max(0, d-7) — don't just use d-7!

When d is small (say, d=3) and we try a 7-day pass, d-7 = -4. In Python, negative indices wrap around arrays — dp[-4] returns a wrong value near the end of the array. max(0, d-7) clamps the lookback to index 0, correctly representing "the 7-day pass covers everything from day 1, so no prior cost needed."

Solution 2 — Top-Down Memoization (same O(n) complexity, easier to derive)

from functools import lru_cache
def mincostTickets(days, costs):
travel = set(days)
@lru_cache(None) ← Cache results: each (day) computed at most once
def dp(d): ← Returns min cost from day d onward to end
if d > days[-1]: return 0 ← Base case: past last travel day, no more cost
if d not in travel: return dp(d + 1) ← Non-travel: skip ahead to next day
return min( ← Travel day: pick cheapest pass
costs[0] + dp(d + 1), ← 1-day: skip 1 day forward
costs[1] + dp(d + 7), ← 7-day: skip 7 days forward
costs[2] + dp(d + 30) ← 30-day: skip 30 days forward
)
return dp(days[0]) ← Start from first travel day; days before it are free
💡 Tabulation vs. Memoization: Which to Use in an Interview?

Tabulation (bottom-up): Fills dp array day-by-day. O(last_day) time and space. No recursion stack. Cleaner to trace through. Present this first in interviews.

Memoization (top-down): Recursive — naturally mirrors the recurrence relation. Easier to derive from scratch since it literally encodes "what choices do I have at day d?" Often better for thinking through the problem initially. Propose it as the derivation path, then optimize to tabulation.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize "Minimum Cost + Multiple Choices" = DP

The problem asks for minimum total cost and at each step you choose among options whose benefit spans different time windows. Choosing a 7-day pass today affects whether you need to buy anything for the next 6 days. This future-affecting decision structure is the hallmark of DP, not Greedy.

Step 2: Define the State Clearly

"What information do I need at each step?" Answer: just the day number. dp[d] = minimum cost to cover all travel days up through day d. Single-dimensional state — clean and simple.

Step 3: Derive Transitions — The "Reach-Back" Insight

If I'm on day d and I buy a 7-day pass, it covers days [d-6, d]. That means all those days between d-6 and d are free — I only need to have paid for coverage through day d-7. So the transition is: costs[1] + dp[d-7]. Think of it as the pass "absorbs" 6 days behind today.

Step 4: Handle the Non-Travel Days — Don't Forget This!

Many candidates jump straight to the travel-day formula and forget that non-travel days also need to be filled: dp[d] = dp[d-1]. If you miss this, your dp array has gaps and lookbacks like dp[d-7] return wrong values.

🗣️ What to Say to the Interviewer

"This is a DP problem — at each travel day I have three choices, and the cost of a multi-day pass
depends on future travel frequency, so Greedy won't work.
I'll define dp[d] = minimum cost to cover all travel days through day d.
Base case: dp[0] = 0.
Non-travel days: dp[d] = dp[d-1], no pass needed.
Travel days: dp[d] = min of three options — costs[0]+dp[d-1], costs[1]+dp[max(0,d-7)],
costs[2]+dp[max(0,d-30)]. The max(0,...) prevents negative indexing.
Return dp[last_travel_day]. Time O(365), Space O(365) — effectively constant."
🔮 Likely Follow-up Questions
  • "Can you do it with O(1) space?" → Use a sliding deque/window of the last 30 dp values instead of the full array.
  • "What if there are more pass types?" → Generalize: for each (duration, cost) pair, add cost + dp[max(0, d-duration)] to the min comparison.
  • "What if travel days go up to 10^9?" → Index by travel day ordinal (event-based DP) instead of calendar day to avoid sparse arrays.

7 Summary & Key Takeaways ▾
Non-travel days always carry forward: dp[d] = dp[d-1]. No pass, no cost delta.
On travel days, "reach back" by the pass duration — a 7-day pass on day d needs only dp[d-7] before it.
max(0, d-k) is essential to prevent negative index access for small values of d.
The DP array size is bounded by O(365) — the year has 365 days at most. Effectively O(1) space complexity.
Greedy fails here because multi-day passes require knowing how often you'll travel in the next k days.
Both tabulation and memoization give identical O(365) time complexity; prefer tabulation in interviews.

🎯 When to Recognize the "Min Across Passes" DP Pattern
  • "Minimum cost to cover events/tasks" over a timeline with multiple pass durations
  • Some days require action, others don't — "carry forward" is the non-action transition
  • Multiple coverage durations (1, 7, 30) — each lookback different distances in dp array
  • Answer is a single optimum at the end of a 1D array — classic bottom-up DP shape
