LeetCode #64

Minimum Path Sum

Given an m×n grid filled with non-negative numbers, find a path from top-left to bottom-right minimizing the sum. You can only move right or down at each step.

Medium Dynamic Programming Min From Up or Left Time: O(m·n) Space: O(1) in-place
1 Understanding the Problem ▾

You stand at the top-left corner of a grid of numbers. You want to reach the bottom-right corner. Every cell has a cost. At each step, you may move right or down only — never left, never up. You want the path that minimizes the total cost accumulated along the way.

💡 Key Insight

Every interior cell can be reached from exactly two places: the cell directly above, or the cell directly to its left. The cheapest way to reach any cell is therefore the cheaper of those two predecessors, plus the cell's own cost. This single observation is the entire algorithm — applied cell by cell, left-to-right, top-to-bottom.

At (1,0): right(5) vs down(4) → down
At (2,0): right(2) → right, right
Path: 1+1+4+2+1 = 9 ✗ (suboptimal)

Edge cases to consider: 1×1 grid (answer is just that cell), 1×n grid (only one path — must go right every step), m×1 grid (must go down every step), grid with large values (no overflow in Python, but watch in Java/C++), all-zero grid (answer = 0).

2 The Algorithm: 2D Bottom-Up DP (Tabulation) ▾

We build a DP table where dp[i][j] = minimum total cost of any valid path from (0,0) to (i,j). We fill it row by row, left to right, using previously computed neighbors.

📐 The Recurrence Relations
dp[i][j] = grid[i][j] + min(dp[i-1][j], dp[i][j-1])

— Boundary cases —
i=0, j=0: dp[0][0] = grid[0][0] (base: start here)
i=0, j>0: dp[0][j] = grid[0][j] + dp[0][j-1] (first row: only from left)
i>0, j=0: dp[i][0] = grid[i][0] + dp[i-1][0] (first col: only from above)

Why is this Dynamic Programming?

Optimal substructure: The optimal path to (i,j) must pass through either (i-1,j) or (i,j-1). Whichever is cheaper is the "last step" of the optimal path. So dp[i][j] depends on optimal solutions to strictly smaller subproblems — this is the definition of optimal substructure.

Overlapping subproblems: A naive DFS would recompute the minimum path to (1,1) many times — once for every path from (0,0) that reaches (1,1) on its way to the destination. DP computes each cell exactly once and caches it.

1 Set up (no separate array needed): We can store dp values directly in the grid to achieve O(1) space. grid[i][j] will be updated to mean dp[i][j].
2 Base case: grid[0][0] is already correct — it's the starting cost. Skip it.
3 First row: Each cell can only come from the left. grid[0][j] += grid[0][j-1]. This turns the first row into a running prefix sum.
4 First column: Each cell can only come from above. grid[i][0] += grid[i-1][0]. This turns the first column into a running prefix sum.
5 Interior cells: grid[i][j] += min(grid[i-1][j], grid[i][j-1]). The updated neighbors to the left and above already hold their dp values — so this is safe.
6 Answer: Return grid[m-1][n-1] — after filling, this holds the minimum path cost to the bottom-right corner.
🔑 Why NOT Greedy?

A greedy algorithm always picks the cheaper immediate neighbor. But the "cheaper step now" might lead into expensive territory later. DP avoids this by considering all valid orderings implicitly: min(from_above, from_left) encodes both options at every cell — no future exploration needed because every predecessor's dp value is already the globally optimal path to reach it.

3 Why It Works: The Invariant ▾
🔐 The Invariant

After we process cell (i, j) in left-to-right, top-to-bottom order, grid[i][j] holds the exact minimum cost of any valid path from (0,0) to (i,j). This invariant holds for every cell at the time we access it as a predecessor.

Why the fill order guarantees correctness: When we compute dp[i][j], we need dp[i-1][j] (cell above, in the previous row — already fully processed) and dp[i][j-1] (cell to the left, already processed in this row). Both are ready. There's no circular dependency.

Proof by induction:

Base: dp[0][0] = grid[0][0] — trivially correct, only one way to be at start.
First row: dp[0][j] = grid[0][j] + dp[0][j-1] — only path is straight right, so this is the only and optimal path cost.
First column: dp[i][0] = grid[i][0] + dp[i-1][0] — only path is straight down.
Interior: Assuming dp[i-1][j] and dp[i][j-1] are both optimal, the optimal path to (i,j) must have arrived from one of them. min picks the better option, and adding grid[i][j] gives the exact optimal cost to (i,j). ∎

⚠️ Mutation Warning

The in-place approach overwrites the input grid. Always ask in interviews: "May I modify the input?" If the answer is no, create a separate dp = [[0]*n for _ in range(m)] array and write dp values there, reading from the original grid. Same O(mn) time, O(mn) space instead of O(1).

4 The Code (Annotated) ▾

Solution 1 — In-Place Bottom-Up Tabulation (Interview Pick ✓)
O(m·n) time, O(1) space

def minPathSum(grid: list[list[int]]) -> int:
m, n = len(grid), len(grid[0]) ← m rows, n columns
for i in range(m): ← process each row top-to-bottom
for j in range(n): ← within row, process left-to-right
if i == 0 and j == 0: ← top-left: no predecessors, leave as-is
continue
elif i == 0: ← first row: predecessor is only to the left
grid[i][j] += grid[i][j-1] ← running sum: total cost arriving from left
elif j == 0: ← first col: predecessor is only above
grid[i][j] += grid[i-1][j] ← running sum: total cost arriving from above
else: ← interior: choose cheaper of two predecessors
grid[i][j] += min(grid[i-1][j], grid[i][j-1]) ← THE recurrence
return grid[m-1][n-1] ← bottom-right now = min path sum

Solution 2 — Top-Down Memoization (for understanding)
O(m·n) time, O(m·n) space (cache) + O(m+n) stack

from functools import lru_cache
def minPathSum(grid: list[list[int]]) -> int:
m, n = len(grid), len(grid[0])
@lru_cache(maxsize=None) ← cache answers so we never recompute dp(i,j)
def dp(i, j): ← returns min cost to reach (i,j) from (0,0)
if i == 0 and j == 0: return grid[0][0] ← base case
if i == 0: return grid[i][j] + dp(i, j-1) ← first row: only from left
if j == 0: return grid[i][j] + dp(i-1, j) ← first col: only from above
return grid[i][j] + min(dp(i-1,j), dp(i,j-1)) ← general recurrence
return dp(m-1, n-1) ← ask: min cost to reach bottom-right?
💡 Tabulation vs Memoization — Which to Use?

Tabulation (bottom-up): Iterative, no call stack risk, O(1) space possible via in-place. Preferred in interviews.
Memoization (top-down): Follows the recurrence directly (easy to derive from the problem definition), but has recursion overhead and O(mn) space always.
In an interview: State the recurrence → implement tabulation → mention in-place optimization.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Pattern

Keywords: "grid," "move right or down only," "minimum cost/sum." These together almost always signal 2D DP. The movement restriction means each cell has at most two predecessors — perfect for simple recurrences.

Step 2: Define the DP State

Ask: "What information do I need at each cell?" Answer: the minimum cost to reach this cell. So dp[i][j] = min cost from (0,0) to (i,j). State is the cell coordinates — 2D DP.

Step 3: Write the Recurrence Aloud

Since we can only arrive from above or from the left: dp[i][j] = grid[i][j] + min(dp[i-1][j], dp[i][j-1]). State boundary cases: first row only from left; first column only from above.

Step 4: Decide Space Optimization

Base approach: O(mn) space for a separate dp array. Optimization: modify grid in-place for O(1). Ask "may I modify input?" first. Could also use O(n) with a 1D dp array (rolling row), though in-place is cleaner here.

🗣️ What to Say to the Interviewer

"I see a grid where I can only move right or down — classic 2D DP setup.
I'll define dp[i][j] as the minimum cost to reach cell (i,j) from (0,0).
Recurrence: dp[i][j] = grid[i][j] + min(dp[i-1][j], dp[i][j-1]).
Boundaries: first row comes only from the left; first column only from above.
May I modify the input grid? If yes, I'll do in-place modification — O(1) extra space.
I fill left-to-right, top-to-bottom, so both predecessors are always ready.
Time: O(m·n). Space: O(1) in-place. Return grid[m-1][n-1] as the answer."
🔮 Likely Follow-up Questions
  • "Can you also move left or up?" → Now it's shortest path on a weighted graph — use Dijkstra, O(mn log mn)
  • "How do you recover the actual path?" → Store a 'parent' matrix: 'U' or 'L' at each cell indicating which predecessor was chosen; backtrack from (m-1,n-1)
  • "What if the grid has negative numbers?" → DP still works; min() handles negatives correctly
  • "Can you reduce space further?" → Yes: use a 1D dp[n] array, update it row by row in-place

7 Summary & Key Takeaways ▾
Every cell is reachable from at most two predecessors: up or left. Border cells have only one.
The full recurrence is: dp[i][j] = grid[i][j] + min(dp[i-1][j], dp[i][j-1])
Fill order (left→right, top→bottom) guarantees all predecessors are computed before they're needed
Modify the grid in-place to achieve O(1) extra space — always ask permission first
Greedy fails here; DP's exhaustive subproblem decomposition finds the true optimum
The pattern extends to: Unique Paths (sum instead of min), Triangle (1D grid), Dungeon Game (reverse fill)
ApproachTimeSpaceNotes
Brute Force DFS O(2^(m+n)) O(m+n) All 2^(m+n-2) paths explored — exponential
Memoization (top-down) O(m·n) O(m·n) Recursive; cache + call stack
Tabulation (separate dp) O(m·n) O(m·n) Iterative; preserves original grid
In-place Tabulation ✓ O(m·n) O(1) Best; mutates grid — ask first
🎯 When to Use "Min From Up or Left" DP
  • "Grid, move right or down, minimize total cost" → this exact pattern
  • "Number of distinct paths" in same grid → same DP, use addition instead of min
  • "Obstacles in grid" → same DP, set dp = infinity for blocked cells
  • Any 2D optimization where each cell has limited predecessors → 2D tabulation
