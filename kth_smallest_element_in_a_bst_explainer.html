LeetCode #230

Kth Smallest Element in a BST

Given the root of a binary search tree and an integer k, return the k-th smallest value (1-indexed) of all the values of the nodes in the tree.

Medium Trees In-order Traversal Time: O(H + k) Space: O(H)
1 Understanding the Problem ▾

We're given a Binary Search Tree (BST) and an integer k. The task: find the k-th smallest value (k is 1-indexed, so k=1 means the minimum).

The naive approach — collect all values, sort them, return index k-1 — works but ignores everything BSTs give us for free. The key question is: can we leverage the BST structure to avoid sorting?

💡 The Core Insight: BST In-order = Sorted

A Binary Search Tree guarantees: Left subtree values < Node value < Right subtree values, recursively at every level. This means in-order traversal (Left → Node → Right) always visits nodes in sorted ascending order. The k-th node visited is exactly the k-th smallest — no sorting needed!

Example Tree (k = 3)

5 / \ 3 6 / \ 2 4 / 1

In-order visit order (sorted!)

Visit 1 → 1st smallest
Visit 2 → 2nd smallest
Visit 3 → 3rd smallest ← answer
Visit 4 → 4th smallest
Visit 5 → 5th smallest

Edge Cases: k = 1 (return leftmost node), k = total nodes (return rightmost node), single-node tree (return that node's value), completely skewed tree (degenerate BST behaving like a linked list).

2 The Algorithm: Iterative In-Order with Early Exit ▾

We perform an iterative in-order DFS using an explicit stack. As we visit nodes in sorted order, we decrement k. When k reaches zero, we've found our answer and stop immediately — no need to traverse the rest of the tree.

1 Initialize: Empty stack, pointer curr = root. The stack simulates the recursion call stack — it holds nodes we descended past that we still need to process.
2 Descend left: While curr is not None, push curr onto the stack and move curr = curr.left. We're sinking to the leftmost node — the current local minimum. Each pushed node is "saved" to process after its left subtree is exhausted.
3 Visit (pop) the node: Pop from stack → this is the next smallest unvisited node. Decrement k. If k == 0, this is the k-th smallest — return its value immediately.
4 Pivot right: Set curr = popped.right. Now go back to step 2, sinking left into the right subtree. We repeat until the answer is found or the tree is exhausted.
🔑 Iterative vs. Recursive — Why Iterative?

Both are O(H + k) time and O(H) space. The iterative version wins in two ways: (1) True early exit — we return the instant k hits 0, without unwinding any call stack frames. (2) Interview follow-up readiness — if the BST is frequently modified and we query k-th smallest often, we can convert this to an augmented BST approach (store left subtree count per node) without changing the traversal framework.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At the moment we pop a node and decrement k, the nodes we have already popped-and-processed are exactly the (original_k − current_k) smallest values in the BST, in sorted ascending order. This is guaranteed by the BST property combined with Left-Root-Right traversal order, which is maintained by the stack.

Why in-order gives sorted output: The BST property says every value in the left subtree is less than the current node, which is less than every value in the right subtree — at every single node. In-order (Left → Root → Right) respects this ordering at each level, recursively. The result is always globally sorted.

Why the explicit stack preserves this: The stack holds the "pending ancestors" — nodes we descended past when going left, which haven't been processed yet. When we pop, we process the smallest unvisited node among all remaining candidates. Pivoting right explores larger values. The stack correctly encodes the in-order future.

⚠️ Critical Condition: while curr or stack

After popping a node and pivoting right, if the right child is None, curr becomes None. But the stack still has ancestors to process! The outer loop condition curr or stack (not just curr) keeps things alive. A common mistake is using only while curr which exits too early when backtracking through a None right child.

⚠️ 1-indexed vs 0-indexed: Off-by-One Trap

k is 1-indexed: k=1 means the 1st smallest (minimum). The check k -= 1; if k == 0: return handles this correctly. Do NOT check k == 1 before decrementing, or k == 0 before decrementing — both are off by one.

4 The Code (Annotated) ▾

Solution 1 — Iterative In-Order with Early Exit (Optimal, Interview Pick)

def kthSmallest(root, k):
stack = [] ← Explicit stack replaces call stack; holds pending ancestors
curr = root ← Pointer to node we're currently sinking into
while curr or stack: ← Continue while unexplored tree or pending backtrack nodes
while curr: ← Sink all the way left (find current local minimum)
stack.append(curr) ← Save: we'll process this after its left subtree is done
curr = curr.left ← Go left — everything left is smaller than curr
curr = stack.pop() ← Backtrack: this is the next node in sorted order
k -= 1 ← "Visit" this node — decrement the countdown
if k == 0: ← We just visited the k-th smallest — done!
return curr.val ← Early exit: no need to traverse remaining nodes
curr = curr.right ← Now explore right subtree (next larger values)

Solution 2 — Recursive In-Order (Simpler to write, harder to early-exit cleanly)

def kthSmallest(root, k):
self.k = k ← Mutable counter shared across recursive calls
self.result = None ← Store answer when found
def inorder(node):
if not node or self.result is not None:← Base case: null node OR already found answer
return
inorder(node.left) ← Recurse left (smaller values first)
self.k -= 1 ← Visit this node
if self.k == 0: ← k-th visit: record answer
self.result = node.val; return
inorder(node.right) ← Recurse right (larger values)
inorder(root)
return self.result
💡 Complexity Summary

Iterative (Optimal): Time O(H + k) — H to reach leftmost node, then k pops. Space O(H) for the stack.
Recursive: Same O(H + k) / O(H), but call stack overhead. For H = O(log n) balanced, this is efficient.
Brute Force (collect all): Time O(n), Space O(n) — traverses and stores everything. Correct but wasteful.
Note: H = O(log n) for balanced BST, O(n) for worst-case skewed tree.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Fire the BST Pattern Recognition

Any problem asking "find k-th smallest/largest in a BST" should immediately trigger: "In-order traversal visits BST nodes in sorted order." State this observation aloud — it shows you know the fundamental BST property cold.

Step 2: Propose the Brute Force

"The simplest approach: in-order traversal to collect all values into a list, then return list[k-1]. O(n) time and O(n) space. But we're collecting n values when we only need k of them."

Step 3: Optimize with Early Exit

"Instead of collecting, I maintain a counter and stop the moment the counter reaches k. I'll use iterative in-order — explicit stack — so I can return immediately without unwinding recursion. Time O(H + k), space O(H)."

Step 4: Volunteer the Follow-Up Proactively

"If the BST is modified frequently (inserts/deletes) and k-th smallest is queried many times, I'd augment each node with a left_count (size of left subtree). Then each query is O(H) by binary-searching the counts — no traversal needed."

🗣️ What to Say to the Interviewer

"BSTs have a key property: in-order traversal visits nodes in ascending sorted order.
So I'll do an iterative in-order with an explicit stack and a countdown k.
I push nodes as I sink left; when I pop, I decrement k.
The moment k hits zero, I return that node's value — early exit, no more traversal.
Time is O(H + k): H to reach the leftmost node, then k pops to count up.
Space is O(H) for the stack — O(log n) balanced, O(n) worst-case skewed."
🔮 Likely Follow-up Questions
  • "What if the BST is modified frequently?" → Augment nodes with left subtree size; binary search on counts for O(H) queries
  • "Find the k-th largest?" → Reverse in-order (Right → Root → Left), same technique
  • "Find all values in range [lo, hi]?" → In-order with pruning (skip left if node < lo, skip right if node > hi)
  • "Validate if a tree is a BST?" → In-order must produce strictly increasing values (#98)

7 Summary & Key Takeaways ▾
BST in-order traversal (L→Root→R) always yields sorted ascending order — this is THE defining BST insight for order-statistics problems
Iterative in-order with an explicit stack enables true early exit the moment the k-th node is popped
The outer loop must be while curr or stack — not just while curr — or backtracking through None right children breaks the algorithm
Time O(H + k): H to sink to leftmost node, k pops to count up. For balanced BST, H = O(log n)
Follow-up: augment each node with left subtree size for O(H) per query without full traversal
