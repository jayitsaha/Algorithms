LeetCode #739

Daily Temperatures

Given an array of daily temperatures, return an array where each element is the number of days you must wait until a warmer temperature. If no warmer day exists, put 0.

Medium Monotonic Stack Next Greater Index Difference Time: O(n) Space: O(n)
1 Understanding the Problem ▾

You're given a list of daily temperatures like [73, 74, 75, 71, 69, 72, 76, 73]. For each day, you want to know: how many days do you have to wait until a warmer day arrives? If no warmer day ever comes, the answer is 0.

💡 Key Insight

For each temperature at index i, we need to find the nearest future index j where temps[j] > temps[i]. The answer for day i is then j - i. This is the classic "next greater element" problem, adapted to return differences instead of values.

✓ Example: [73,74,75,71,69,72,76,73]

Day 0: 73 → next warmer Day 1 (74) → wait 1
Day 1: 74 → next warmer Day 2 (75) → wait 1
Day 2: 75 → next warmer Day 6 (76) → wait 4
Day 3: 71 → next warmer Day 5 (72) → wait 2
Day 4: 69 → next warmer Day 5 (72) → wait 1
Day 5: 72 → next warmer Day 6 (76) → wait 1
Day 6: 76 → no warmer day → 0
Day 7: 73 → no warmer day → 0

✗ Naive O(n²) Too Slow

For each day i, scan ahead j = i+1,
i+2, ... until temps[j] > temps[i].

Worst case: [100,99,98,...,1]
Day 0 scans n-1 days → no match
Day 1 scans n-2 days → no match
Total: ~n²/2 comparisons.
For n=100,000 that's 5 billion ops.

Edge cases: single element (output [0]); all same temperatures (all zeros); strictly decreasing temperatures — nothing ever gets warmer, all zeros; strictly increasing — every wait is exactly 1.

2 The Algorithm: Monotonic Decreasing Stack ▾

We maintain a stack of indices whose "warmer future" we haven't found yet — a "waiting room." The temperatures corresponding to these indices form a non-increasing (monotonically decreasing) sequence. Whenever we see a day warmer than the stack top, we immediately know the answer for that colder day.

1 Initialize: Create answer = [0] * n. Zeros are the correct default for any day that never sees a warmer future. Create an empty stack to hold day indices.
2 Scan left to right: For each day i, check whether temps[i] is warmer than the temperature of the index on top of the stack: temps[i] > temps[stack[-1]].
3 Pop and resolve: While the condition holds, pop index prev from the stack. Day prev's answer is i - prev. Keep popping — day i might resolve several colder waiting days.
4 Push current: After all popping, push i onto the stack. Day i is now in the waiting room for its own warmer day.
5 Return: After scanning all days, any indices still on the stack never found a warmer day — their answer stays 0. Return answer.
🔑 The Monotonic Property Explained

The stack always holds indices in increasing order, and their corresponding temperatures in non-increasing order. This is because before pushing day i, we pop everything colder — so everything remaining on the stack is at least as warm as the new entry. This ordering means that when we pop the stack top, we know it is the coldest unresolved day, and no day between it and i could have answered it first.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every step i, the stack contains indices i₁ < i₂ < ... < iₖ such that temps[i₁] ≥ temps[i₂] ≥ ... ≥ temps[iₖ]. Each represents a day that hasn't yet seen a warmer future day. The first index that resolves iₖ is guaranteed to be the nearest warmer day for it.

Why each index is processed at most twice (O(n) total): Each index is pushed exactly once and popped at most once. Even with a while loop inside the for loop, the total number of push + pop operations cannot exceed 2n. This is the amortized analysis argument: any individual step can do many pops, but those pops correspond to elements that were previously pushed and will never be pushed again.

Correctness guarantee: When we pop index prev at step i, day i is the nearest warmer day for prev. Why? Because if any intermediate day k (with prev < k < i) had been warmer than temps[prev], we would have popped prev already at step k. So reaching step i with prev still on the stack means no intermediate day was warmer — day i truly is the nearest.

⚠️ Common Mistake: Storing Temperatures Instead of Indices

The answer requires i - prev — you need to know when the cold day occurred, not just its temperature. If you push temps[i] instead of i, you lose the index information. Always push indices. Retrieve temperature via temps[stack[-1]].

Why remaining stack entries are correct: When the loop ends, any index k still on the stack never encountered a future day where temps[future] > temps[k]. The answer for those days is 0 — and since we initialized answer = [0] * n, we don't need to do anything extra for them.

4 The Code (Annotated) ▾

Solution 1 — Monotonic Stack (Optimal · O(n) · Interview Pick)

def dailyTemperatures(temps): ← Takes list of integers
n = len(temps) ← Number of days
answer = [0] * n ← Default 0: "no warmer day" for all positions
stack = [] ← Holds INDICES of days still waiting for warmth
for i in range(n): ← Scan each day left to right
while stack and temps[i] > temps[stack[-1]]: ← Current day warmer than waiting day?
prev = stack.pop() ← Pop the cold day — its wait is over
answer[prev] = i - prev ← Wait = current index − cold day index
stack.append(i) ← Push current; it's now waiting for ITS warmer day
return answer ← Leftover stack indices stay 0 — already correct
💡 Complexity Summary

Time: O(n) — Each of the n indices is pushed once and popped at most once. Total operations ≤ 2n regardless of the while loop nesting.
Space: O(n) — In the worst case (temperatures in strictly decreasing order), no index is ever popped, and all n indices accumulate on the stack.

Solution 2 — Brute Force (O(n²) · for understanding only)

def dailyTemperatures_brute(temps): ← Correct but too slow for large inputs
n = len(temps)
answer = [0] * n
for i in range(n): ← For every day...
for j in range(i + 1, n): ← ...scan every future day
if temps[j] > temps[i]: ← First warmer day?
answer[i] = j - i ← Record the wait
break ← Stop — only nearest warmer day matters
return answer
5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the "Next Greater Element" Signal

"The question asks: for each element, how far until a larger element appears. This is the canonical next-greater-element pattern. The tool for this is a monotonic stack."

Step 2: State Brute Force and its Flaw

"Naively, for each day I scan forward until I find a warmer temperature — O(n²) worst case when temperatures are decreasing. With n up to 100,000, that's too slow."

Step 3: Derive the Stack Idea

"The O(n²) comes from re-examining days we've already passed. Instead, can I keep a running list of 'unresolved' days? Yes — a stack. When a new day is warmer than the stack top, I can resolve that cold day instantly."

Step 4: Clarify Stack Contents (Indices, Not Temperatures)

"I need to know when the cold day occurred (its index) to compute the wait. So I push indices, not temperatures. The temperature is always accessible as temps[stack[-1]]."

Step 5: Argue O(n) Amortized

"Each index enters the stack once and leaves at most once — at most 2n stack operations total. The while loop doesn't change the asymptotic bound."

🗣️ What to Say to the Interviewer

"This is the next-greater-element pattern — I'll use a monotonic stack.
Initialize answer to all zeros. Scan left to right with index i.
Maintain a stack of day indices waiting for a warmer future.
For each i: while stack non-empty and temps[i] > temps[stack.top],
pop prev and set answer[prev] = i - prev.
Then push i. At the end, remaining stack items stay 0.
O(n) time — each index pushed and popped at most once. O(n) space."
🔮 Likely Follow-up Questions
  • "Next smaller element?" → Flip comparison to <; everything else identical
  • "Previous greater element?" → Scan right-to-left with the same stack logic
  • "Circular array?" → Process the array twice (or use modular indexing) — see #503
  • "Can you do O(1) space?" → Not generally; worst case is strictly decreasing (stack size = n)
  • "What if temperatures can repeat?" → Strictly greater check already handles ties correctly (they wait)

7 Summary & Key Takeaways ▾
The monotonic stack acts as a "waiting room" for days that haven't yet found a warmer future
Stack temperatures are always non-increasing (monotonically decreasing from bottom to top)
Store indices on the stack — you need the index to compute the wait time (i - prev)
O(n) time: each index is pushed once and popped at most once — amortized linear despite the while loop
Initialize answer to zeros — remaining stack entries at the end are automatically correct
Key trigger: "next larger/smaller element" or "wait until condition" → reach for Monotonic Stack
