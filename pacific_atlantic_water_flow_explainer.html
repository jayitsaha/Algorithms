LeetCode #417

Pacific Atlantic Water Flow

Find all cells from which rain water can flow to both the Pacific and Atlantic Oceans.

Medium Graph DFS from Both Oceans Time: O(m·n) Space: O(m·n)
1 Understanding the Problem ▾

You have an m × n grid of heights — imagine a bird's-eye view of an island. The island is flanked by two oceans:

  • Pacific Ocean — touches the top row and left column
  • Atlantic Ocean — touches the bottom row and right column

Rain falls on every cell. Water flows from one cell to an adjacent cell (up/down/left/right) only if the neighbor's height is ≤ the current cell's height (gravity — downhill or flat). Water can also flow off the grid edge into the adjacent ocean.

Return all coordinates [r, c] where rain falling on that cell can eventually reach both the Pacific and Atlantic Oceans.

💡 Key Insight: Reverse the Flow Direction

Simulating forward flow from every cell to the ocean would be O(m²n²) — for each of the m·n cells, a DFS over m·n cells. The trick: flip the question. Instead of asking "can this cell reach the ocean?", ask "which cells can the ocean reach going uphill?" Start from ocean-border cells and flood-fill to neighbors with height ≥ current (uphill). The intersection of the two reachable sets is the answer, and each cell is visited at most once per ocean → O(m·n).

✗ Naive: Forward per-cell

For each of m·n cells:
run DFS toward oceans
revisit many paths
Time: O(m²n²) — TLE

✓ Reverse DFS from borders

Flood uphill from ocean edges
Each cell visited ≤ twice
Intersect two visited sets
Time: O(m·n) — optimal

Edge Cases: 1×1 grid → always in result (borders both oceans). Monotonically increasing rows (every cell drains to Pacific, none to Atlantic). Flat plateau (all heights equal) → every cell reaches both. Empty grid → return [].

2 The Algorithm: Reverse DFS from Both Ocean Borders ▾

We run two independent DFS traversals — one seeded from all Pacific-border cells, one from all Atlantic-border cells — using an uphill condition. Any cell marked by both traversals can drain to both oceans.

1 Initialize two visited sets: pacific and atlantic, each starting empty.
2 Seed Pacific: Add every cell in row 0 (top) and column 0 (left) to pacific and push them onto the DFS stack. These cells directly border the Pacific.
3 DFS uphill for Pacific: From each cell, explore all 4 neighbors. Visit a neighbor only if: (a) it's in bounds, (b) not yet in pacific, and (c) heights[neighbor] ≥ heights[current]. Mark and recurse.
4 Seed Atlantic: Add every cell in row m-1 (bottom) and column n-1 (right) to atlantic. Run the same uphill DFS.
5 Intersect: Return all [r, c] present in both pacific and atlantic. These are the cells from which rain can reach both oceans.

🔑 DFS vs BFS: Both Work Identically

The only difference is the data structure: DFS uses a stack (or recursion), BFS uses a queue. Both visit the same set of cells and produce identical results. Choose BFS in Python to avoid hitting the recursion limit on 200×200 grids. Both are O(m·n) time and space.

3 Why It Works: The Invariant ▾
🔐 The Invariant

After the Pacific DFS: pacific[r][c] = True if and only if there exists a non-decreasing path from (r, c) to some cell on the Pacific border. Equivalently: rain at (r, c) can flow downhill to reach the Pacific. The reverse-flow trick is valid because "water flows from A to B downhill" is logically equivalent to "from B, you can traverse uphill to reach A."

Proof sketch: Forward: water flows X→X₁→X₂→…→border where each step has heights[next] ≤ heights[current]. Reverse: from the border, DFS visits neighbors with heights[neighbor] ≥ heights[current]. These are the same paths, traversed backwards. The visited set in reverse DFS is exactly the set of forward-reachable cells.

Handling plateaus (equal-height cells): Water can flow on flat terrain (heights[neighbor] ≤ heights[current]), so in reverse DFS we allow heights[neighbor] ≥ heights[current] (equal-height neighbors are also valid). The visited set prevents infinite loops on plateaus.

⚠️ The Most Common Bug: Wrong Comparison Direction

In reverse DFS, the condition is heights[nr][nc] >= heights[r][c] — the neighbor must be at least as high as current. Writing <= instead simulates forward flow (downhill), which produces incorrect results. When in doubt: think "can the ocean water physically push uphill to reach this neighbor?" Yes only if neighbor height ≥ current height.

Why we can safely intersect: The two DFS traversals are completely independent. A cell in pacific ∩ atlantic satisfies both reachability conditions simultaneously — it can drain downhill to reach the top/left borders AND can drain downhill to reach the bottom/right borders. No cell is double-counted or missed.

4 The Code (Annotated) ▾

Solution 1 — Recursive DFS (Interview Pick)

def pacificAtlantic(heights): ← heights is m×n matrix
if not heights or not heights[0]: return [] ← guard: empty grid
m, n = len(heights), len(heights[0]) ← rows m, cols n
def dfs(r, c, visited): ← uphill DFS from (r,c)
visited.add((r, c)) ← mark this cell ocean-reachable
for dr, dc in [(0,1),(0,-1),(1,0),(-1,0)]: ← 4 adjacent directions
nr, nc = r + dr, c + dc ← neighbor cell
if (0 <= nr < m ← in bounds row
and 0 <= nc < n ← in bounds col
and (nr, nc) not in visited ← not already marked
and heights[nr][nc] >= heights[r][c]): ← UPHILL: neighbor ≥ current
dfs(nr, nc, visited) ← recurse uphill
pacific = set() ← cells that can drain to Pacific
atlantic = set() ← cells that can drain to Atlantic
for c in range(n): ← seed top row → Pacific
dfs(0, c, pacific)
dfs(m - 1, c, atlantic) ← seed bottom row → Atlantic
for r in range(m): ← seed left col → Pacific, right col → Atlantic
dfs(r, 0, pacific)
dfs(r, n - 1, atlantic)
return [[r, c] for r, c in pacific & atlantic] ← set intersection = answer
⚠️ Python Recursion Limit

On grids up to 200×200, recursive DFS may hit Python's default call-stack limit (~1000). In interviews, mention this and offer the BFS alternative. In production: use sys.setrecursionlimit(200*200+100) or switch to iterative DFS/BFS.

Solution 2 — BFS with deque (Preferred in Python)

from collections import deque
def pacificAtlantic(heights):
m, n = len(heights), len(heights[0])
def bfs(seeds): ← seeds = list of border (r,c) tuples
q = deque(seeds)
vis = set(seeds) ← mark seeds before processing
while q:
r, c = q.popleft()
for dr, dc in [(0,1),(0,-1),(1,0),(-1,0)]:
nr, nc = r+dr, c+dc
if (0<=nr<m and 0<=nc<n
and (nr,nc) not in vis
and heights[nr][nc] >= heights[r][c]):← uphill only
vis.add((nr, nc)) ← mark before enqueue (BFS idiom)
q.append((nr, nc))
return vis
pac_s = [(0,c) for c in range(n)] + [(r,0) for r in range(m)]
atl_s = [(m-1,c) for c in range(n)] + [(r,n-1) for r in range(m)]
pac, atl = bfs(pac_s), bfs(atl_s)
return [[r,c] for r,c in pac & atl]
💡 Complexity Summary

Both solutions: Time O(m·n), Space O(m·n)
Each cell enters the pacific visited set at most once, and the atlantic set at most once → 2·(m·n) work total. The pacific & atlantic set intersection is O(min(|pac|, |atl|)) ⊆ O(m·n).

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Signals

Key signals: "2D grid + reachability from two sources + intersection." When a problem asks "which cells satisfy condition A AND condition B where each condition is a graph reachability problem," think: run two separate traversals, intersect the results.

Step 2: Start with Brute Force

"For each of the m·n cells, I can run a DFS to check if it reaches the Pacific, and another to check the Atlantic — O(m²n²) total. That's 1.6 billion ops on a 200×200 grid. Too slow."

Step 3: The Reverse Insight

"Instead of going from each cell toward the ocean, I can reverse: start AT the ocean and flood uphill. Any cell I can reach from the Pacific border is Pacific-reachable. Same for Atlantic. Run two DFS/BFS passes, then intersect."

Step 4: Uphill Condition

"In the reverse DFS, I visit a neighbor if its height is ≥ current height — because in forward flow, water goes to lower neighbors. Reversing: I go to higher neighbors from the ocean perspective."

🗣️ What to Say to the Interviewer

"I'll use two BFS/DFS passes in reverse. Instead of simulating water flowing from each cell,
I'll start from the ocean borders and flood uphill.
Pacific: seed top row and left column; DFS to neighbors with height ≥ current.
Atlantic: seed bottom row and right column; same DFS.
Return the intersection of both visited sets.
Time O(m·n), Space O(m·n). I'd use BFS in Python to avoid recursion depth issues."
🔮 Likely Follow-up Questions
  • "Can you handle 8-directional flow?" → Extend direction list to 8 tuples
  • "What if there's a third ocean?" → Third DFS pass; intersect all three
  • "What if heights can be negative?" → No change; comparison still works
  • "What's the maximum number of cells in the answer?" → All m·n cells (flat grid)
  • "Optimize space?" → Use 2-bit booleans instead of sets; or in-place bit-marking in heights

7 Summary & Key Takeaways ▾
Reverse the flow: "can cell reach ocean?" becomes "can ocean reach cell going uphill?"
Two DFS/BFS passes — one seeded from Pacific border, one from Atlantic border
Uphill condition in reverse DFS: visit neighbor only if heights[neighbor] ≥ heights[current]
Intersection of both visited sets gives the final answer
Each cell is visited at most once per ocean → total work O(m·n)
Use BFS (iterative deque) in Python to avoid recursion-depth issues on large grids

🎯 When to Recognize Multi-Source Reverse BFS/DFS
  • "Which cells can reach X AND Y?" → Two traversals from X and Y, intersect
  • "Mark all cells reachable from a border" → Seed entire border, BFS inward
  • Flow direction reversal: "A→B downhill" ≡ "B→A uphill"
  • Result = intersection, union, or difference of two reachability sets
