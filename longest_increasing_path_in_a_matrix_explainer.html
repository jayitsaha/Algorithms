LeetCode #329

Longest Increasing Path in a Matrix

Given an m × n integer matrix, return the length of the longest strictly increasing path. You may move in 4 directions (up, down, left, right) — not diagonally.

Hard Dynamic Programming DFS + Memoization Time: O(m·n) Space: O(m·n)
1 Understanding the Problem ▾

You are given a grid of integers. Starting from any cell, you may walk to an adjacent cell (up, down, left, or right) only if that neighbor's value is strictly greater than the current cell's value. You cannot step diagonally or leave the grid. The question: what is the length of the longest such walk?

💡 Key Insight: This is a Longest-Path-in-a-DAG Problem

The "strictly increasing" constraint means you can never revisit a cell — you always move uphill. This transforms the grid into a Directed Acyclic Graph (DAG) where edges point from smaller to larger values. There are no cycles. The answer is the longest path in this DAG — a classic DP problem solvable with DFS + memoization.

✓ Valid Path (length 4)

Matrix:
  9  9  4
  6  6  8
  2  1  1
Path: 1→2→6→9 = length 4

✗ Invalid Moves

9→9: BLOCKED (not strictly >)
6→4: BLOCKED (going downhill)
6→6: BLOCKED (equal, not strictly >)
Off-grid: BLOCKED

Edge Cases: A 1×1 matrix always returns 1. All equal values → no valid moves → answer is 1. A perfectly sorted matrix (e.g., each row/column strictly increasing) gives the longest possible path.

2 The Algorithm: DFS + Memoization (Top-Down DP) ▾

Define dfs(r, c) = the length of the longest strictly increasing path starting at cell (r, c). Compute this recursively for every cell; cache each result. The answer is the maximum over all cells.

1 Cache check: If (r, c) is already in memo, return memo[(r, c)] immediately. No recomputation needed.
2 Initialize best = 1: Every cell by itself is a path of length 1. This is our base case — even if no neighbor qualifies, we return 1.
3 Try all 4 neighbors: For each neighbor (nr, nc) that is in-bounds AND matrix[nr][nc] > matrix[r][c], compute 1 + dfs(nr, nc). Update best to the maximum seen.
4 Memoize and return: Store memo[(r, c)] = best, then return it. Future calls to dfs(r, c) get the cached answer in O(1).
5 Outer loop: Call dfs(r, c) for every cell in the grid. Return the overall maximum. Note: many calls hit the cache immediately, so total work is O(m·n).
🔑 Why No Visited Set is Needed

In standard DFS on graphs, you maintain a visited set to detect cycles. Here, the strictly increasing constraint does that job automatically. If path A→B→C is valid, then C > B > A, so going back from C to A is impossible (A is not strictly greater than C). Cycles cannot form. Omitting the visited set is both correct and necessary — adding one would incorrectly block valid paths that share common cells.

3 Why It Works: The Invariant & Why This is DP ▾
🔐 The Invariant

Once memo[(r, c)] is stored, it holds the exact, final, correct longest path length from (r, c). This is guaranteed because: (1) every recursive call from dfs(r, c) is to a cell with a strictly larger value, and (2) because the graph is acyclic, those recursive calls always terminate and are never circular.

Optimal Substructure: The longest path from (r, c) equals 1 plus the longest path among its valid neighbors. The global answer decomposes into independent per-cell subproblems.

📐 The Recurrence Relation
dfs(r, c) = 1 + max{ dfs(nr, nc) }
for each (nr, nc) adjacent, in-bounds, matrix[nr][nc] > matrix[r][c]
dfs(r, c) = 1 ← base case: no valid neighbor exists

Overlapping Subproblems: Multiple starting cells may have paths converging through the same high-value cell. For example, if cells A, B, and C all have valid moves to cell D, then without memoization, dfs(D) would be recomputed three times. With the cache, it's computed once and O(1) on subsequent accesses.

⚠️ Common Mistake: Adding a Visited Set

Students from graph DFS backgrounds often add visited = set() and mark cells. This is wrong for this problem. The visited set would prevent later starting cells from reusing the cached result at shared "interior" cells, giving incorrect (too-small) answers. The increasing constraint already prevents cycles — trust it.

4 The Code (Annotated) ▾

Solution 1 — DFS + Memoization (Optimal, Interview Pick) ✓

def longestIncreasingPath(matrix): ← takes m×n integer grid
if not matrix or not matrix[0]: return 0 ← guard against empty input
ROWS, COLS = len(matrix), len(matrix[0]) ← grid dimensions
memo = {} ← cache: (r,c) → longest path length from (r,c)
def dfs(r, c): ← returns longest path starting at (r,c)
if (r, c) in memo: return memo[(r, c)] ← cache hit: already computed, skip recomputation
best = 1 ← minimum: just this cell itself
for dr, dc in [(0,1),(0,-1),(1,0),(-1,0)]:← right, left, down, up
nr, nc = r + dr, c + dc
if (0 <= nr < ROWS and 0 <= nc < COLS ← in bounds...
and matrix[nr][nc] > matrix[r][c]):← ...AND strictly greater value
best = max(best, 1 + dfs(nr, nc)) ← 1 (this cell) + best path from neighbor
memo[(r, c)] = best ← cache result BEFORE returning
return best
return max(dfs(r, c) ← try every starting cell...
for r in range(ROWS)
for c in range(COLS)) ← ...return global maximum
⚠️ Why store memo BEFORE returning?

The memoization line memo[(r, c)] = best must come before return best. If we returned first, the cache would never be written, and every future call to dfs(r, c) would recompute from scratch — no speedup at all. Store first, then return.

Solution 2 — Brute Force DFS (No Memoization)

def longestIncreasingPath_brute(matrix):
ROWS, COLS = len(matrix), len(matrix[0])
def dfs(r, c):
best = 1
for dr, dc in [(0,1),(0,-1),(1,0),(-1,0)]:
nr, nc = r + dr, c + dc
if 0<=nr<ROWS and 0<=nc<COLS and matrix[nr][nc]>matrix[r][c]:
best = max(best, 1 + dfs(nr, nc)) ← no cache: recomputes (nr,nc) for every caller
return best
return max(dfs(r, c) for r in range(ROWS) for c in range(COLS))
💡 Complexity Summary

Brute Force DFS: Time O(2^(m·n)) — exponential; each cell can be a neighbor of up to 4 cells, causing massive recomputation.
DFS + Memoization: Time O(m·n) — each of the m·n cells computed exactly once. Each computation visits 4 neighbors in O(1). Space O(m·n) for memo + O(m·n) recursion stack.
Topological Sort (alternative): Also O(m·n) time and space. Iterative BFS on the DAG — avoids Python's recursion stack limit for very large grids.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Model as a Graph Problem

"I see a grid traversal with a directional constraint. I'll model each cell as a node. There's a directed edge from (r,c) to a neighbor only when the neighbor's value is strictly greater. I want the longest path in this directed graph."

Step 2: Recognize No-Cycle Property → DAG → DP

"Key observation: strictly increasing means the graph has no cycles — it's a DAG. Longest path on a DAG is solvable with DP. I'll define dfs(r, c) = longest increasing path starting at (r, c)."

Step 3: Write the Recurrence and Add Memoization

"Recurrence: dfs(r,c) = 1 + max(dfs(neighbor)) over valid neighbors with larger values. Base: no valid neighbor → return 1. Cache each result. Call dfs for all cells, return the max."

Step 4: Analyze Complexity

"With memoization: each of the m·n cells is computed once. Each computation is O(1) (4 neighbor checks). Total: O(m·n) time, O(m·n) space for memo + call stack."

🗣️ What to Say to the Interviewer

"I'll model each cell as a node in a DAG — edges go from smaller to strictly larger neighbors.
Since paths are strictly increasing, there are no cycles. This makes it a longest-path-in-DAG problem.
I'll use DFS with memoization: dfs(r,c) returns the longest increasing path starting at (r,c).
For each cell, I try up to 4 neighbors with larger values, recursing and taking 1 + max result.
I cache each cell's answer — so each of the m·n cells is computed exactly once.
Outer answer: max over all dfs(r,c). Time O(m·n), Space O(m·n)."
🔮 Likely Follow-up Questions
  • "Can you solve it iteratively without recursion?" → Yes — topological sort (Kahn's algorithm) on the DAG using BFS with an in-degree array.
  • "What about diagonal moves?" → Add 4 more direction tuples: (1,1),(1,-1),(-1,1),(-1,-1). Same logic.
  • "What if equal values were allowed (non-decreasing)?" → Cycles become possible. Would need a visited set per DFS path, making it NP-hard (longest path in general graphs is NP-hard). The strictly-increasing constraint is what makes this tractable.
  • "What if we needed the actual path, not just the length?" → Store parent pointers in a second dict alongside the memo; backtrack from the max-value cell.

7 Summary & Key Takeaways ▾
Strictly increasing → no cycles → DAG → DP is applicable
No visited set needed: the strictly-increasing constraint prevents re-entry
Recurrence: dfs(r,c) = 1 + max(dfs(neighbor)) for each valid (larger) neighbor
Memoization ensures each cell is computed exactly once → O(m·n) total
Try every cell as a starting point; answer = max(dfs(r,c)) over all cells

🎯 When to Recognize DFS + Memoization on a Grid
  • Grid traversal with a monotonic constraint (strictly increasing/decreasing)
  • The constraint prevents cycles → DAG structure → DP is applicable
  • Need to try every starting point and find the global optimum
  • Without memo, many starting cells would redundantly compute the same "interior" subpaths
