LeetCode #1448

Count Good Nodes in Binary Tree

Count nodes in a binary tree where no ancestor on the root-to-node path has a strictly greater value.

Medium Trees DFS Preorder Track Max on Path Time: O(n) Space: O(h)
1 Understanding the Problem ▾

A node X is called good if, on the unique path from the root down to X, no node has a value strictly greater than X's value. Equivalently, X's value is greater than or equal to every one of its ancestors.

💡 Key Insight

As we walk any root-to-node path, the only ancestor information we need is the maximum value seen so far. A node is good if and only if node.val ≥ max_so_far. We never need to store the full path — just that single running max.

✓ Good Node

Path: 3 → 4 → 5
Ancestors of 5: {3, 4}
5 ≥ max({3,4}) = 4 → GOOD

✗ Not Good

Path: 3 → 4 → 1
Ancestors of 1: {3, 4}
1 < max({3,4}) = 4 → NOT GOOD

Edge Cases: The root is always good (no ancestors). A single-node tree returns 1. All same-valued nodes are all good (each equals the running max). A strictly decreasing path from root (e.g., 5→4→3→1) has only the root as good.

2 The Algorithm: DFS Preorder + Max Propagation ▾

We perform a preorder DFS — visit the current node, then recurse into left and right children. At every call we carry max_so_far: the maximum node value on the path from the root to the current node's parent.

1 Launch DFS from root with max_so_far = -∞. The sentinel value ensures root always qualifies regardless of its sign.
2 Base case: if node is null, return 0 (no good nodes in an empty subtree).
3 Good-node check: is_good = 1 if node.val ≥ max_so_far else 0.
4 Update max for children: new_max = max(max_so_far, node.val). This "locks in" the current node as an ancestor for its children.
5 Recurse left and right with new_max. Return is_good + left_count + right_count.
node.val ≥ max_so_far → GOOD

This node beats or ties all ancestors. Count it. Update max to node.val (possibly a new record).

node.val < max_so_far → NOT GOOD

An ancestor dominates. Do not count it. Max stays the same — the ancestor's value still rules.

3 Why It Works: The Invariant ▾
🔐 The Invariant

When dfs(node, max_so_far) is called, max_so_far is exactly the maximum value among all ancestors of node on the root-to-node path (or -∞ for the root itself, which has no ancestors).

Proof by induction:

  • Base: Root call passes -∞. Root has no ancestors. Invariant holds trivially.
  • Step: Assume invariant holds at node N with max_so_far = M. When we recurse into a child C, we pass max(M, N.val). The ancestors of C are exactly the ancestors of N plus N itself, whose max is max(M, N.val). Invariant holds for C.

Why this guarantees correctness: A node is good iff node.val ≥ max(ancestor values). Since our invariant guarantees max_so_far holds exactly that maximum, the check is correct. Every node is visited exactly once in DFS, so the count is exact — no double-counting, no missed nodes.

⚠️ Why Preorder, Not Postorder?

We must decide if the current node is good before recursing — because the decision uses max_so_far which flows top-down from ancestors. In postorder, children run first and the ancestor context is already gone. Always use preorder for top-down path problems.

4 The Code (Annotated) ▾

Solution 1 — Recursive DFS (Interview Pick)

def goodNodes(self, root: TreeNode) -> int: ← public API; tree guaranteed non-empty
def dfs(node, max_so_far): ← inner helper; carries context top-down
if not node: ← null child: contributes 0 good nodes
return 0 ← base case: empty subtree
is_good = 1 if node.val >= max_so_far else 0 ← key decision: not beaten by any ancestor?
new_max = max(max_so_far, node.val) ← update max for this node's children
left = dfs(node.left, new_max) ← good count from entire left subtree
right = dfs(node.right, new_max) ← good count from entire right subtree
return is_good + left + right ← total: this node + both subtrees
return dfs(root, float('-inf')) ← -inf: root is always good regardless of sign
⚠️ Why -inf, not 0?

If root had value -5 and we used 0 as initial max, the root would wrongly fail the check (-5 ≥ 0 is false). Starting with float('-inf') guarantees no initial value can block the root.

Solution 2 — Iterative DFS with Explicit Stack

def goodNodes(self, root: TreeNode) -> int: ← avoids Python recursion limit on deep trees
count = 0 ← accumulates good-node total
stack = [(root, float('-inf'))] ← stack of (node, max_on_path) pairs
while stack: ← loop until all nodes processed
node, max_so_far = stack.pop() ← LIFO: get next node and its path context
if node.val >= max_so_far: ← same good-node check
count += 1 ← count it
new_max = max(max_so_far, node.val) ← updated max to pass to children
if node.left: ← only push non-null children
stack.append((node.left, new_max)) ← left child with updated max
if node.right: ← right child with updated max
stack.append((node.right, new_max)) ← same logic, explicit stack instead of call stack
return count ← total good nodes across entire tree
🔑 Complexity

Time O(n) — every node is visited exactly once. Space O(h) — at any point, the call stack (or explicit stack) holds at most h frames where h is tree height. O(log n) for balanced trees, O(n) worst case for a skewed/linear tree.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Signal

The phrase "no ancestor greater than X" is a red flag for top-down DFS with a parameter. Any time a node's property depends on all its ancestors, think: "what single summary of those ancestors do I need?" Here it's the max — one number covers all ancestors.

Step 2: Define the Recursion Signature

Say out loud: "I'll define dfs(node, max_so_far) that returns the count of good nodes in the subtree rooted at node, given that the path max from root to node's parent was max_so_far." This signature encapsulates everything you need.

Step 3: Handle the Base Cases

Null node → 0. Start: dfs(root, -inf). Mention that -inf handles negative root values — interviewer will appreciate this attention to edge cases before they ask.

Step 4: Volunteer the Iterative Version

Mention that for very deep or skewed trees, Python's default recursion limit (~1000) could be hit. Offer an iterative stack version that uses the exact same logic but with an explicit stack of (node, max_so_far) pairs.

🗣️ What to Say to the Interviewer

"Each node is good if its value is ≥ every ancestor's value.
So I only need the maximum ancestor value — not the full path.
I'll DFS preorder and carry that max top-down as a parameter.
dfs(node, max_so_far): if node.val ≥ max_so_far, it's good.
Then I update max to max(max_so_far, node.val) before recursing.
Start with -inf so the root is always good.
Time O(n), Space O(h). If tree is skewed, I can make this iterative
using a stack of (node, max) pairs — same logic, no recursion."
💡 Follow-up Questions to Expect
  • Iterative version? Stack of (node, max_so_far) pairs — identical logic.
  • What if values can be negative? No change — -inf handles it.
  • Count not-good nodes? Just return n - goodNodes(root).
  • BFS instead? Works too — queue of (node, max_so_far). Same O(n)/O(w) where w = max width.
  • Can you define "bad" nodes similarly? Yes: a node is bad if any ancestor is strictly larger.

7 Summary & Key Takeaways ▾
Carry a single running maximum top-down — it summarizes the entire ancestor path in O(1) space per call.
Use preorder DFS: decide if the current node is good before recursing into children.
Initialize max_so_far = float('-inf') so the root is always good, even with negative values.
Time O(n), Space O(h) — optimal, since every node must be visited and the call stack depth equals tree height.
The iterative stack version is a direct translation: push (child, new_max) instead of recursing.
🎯 When to Use "Track Max (or Min) on Path"

This pattern applies whenever a node's answer depends on an aggregate of all its ancestors and that aggregate can be expressed as a single scalar passed top-down:

  • Node property depends on all ancestors (max, min, sum, XOR, product, bitmask...)
  • Decision at a node uses only ancestor info — not descendant info
  • No information flows bottom-up (that would be postorder)
Path SumEasy — top-down DFS carrying running sum; check at leaves
Path Sum IIMedium — same pattern, collect all valid root-to-leaf paths
Sum Root to Leaf NumbersMedium — carry the partial decimal number being formed top-down
Pseudo-Palindromic Paths in a Binary TreeMedium — carry a bitmask of digit parities top-down
Longest ZigZag Path in a Binary TreeMedium — carry current direction and length top-down
Binary Tree Maximum Path SumHard — postorder variant: children return best contribution upward
