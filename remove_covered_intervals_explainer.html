LeetCode #1288

Remove Covered Intervals

Given a list of intervals, remove all intervals that are covered by another interval in the list. Return the number of remaining intervals.

Medium Intervals Sort + Track Max End Time: O(n log n) Space: O(1)
1 Understanding the Problem ▾

An interval [a, b] is covered by interval [c, d] when c <= a AND b <= d — meaning the second interval completely contains the first on the number line. A covered interval adds no unique coverage and can be removed.

💡 Key Insight

After sorting by start ascending (with end descending as a tiebreaker), coverage detection simplifies to a single condition: track the maximum end seen among survivor intervals. If the current interval's end fits within max_end, it is covered — its start is already guaranteed to be valid by the sort order.

2 The Algorithm: Sort + Track Max End ▾

The key insight is that after sorting cleverly, we can reduce a seemingly two-dimensional comparison (check both start and end) to a one-dimensional scan (check only end). Here is the full approach:

1 Sort: Sort intervals by start ascending, then by end descending as a tiebreaker. The start-ascending sort guarantees: when processing interval [a,b], every previously seen interval [c,d] has c <= a. The end-descending tiebreaker guarantees: when two intervals share the same start, the larger (covering) one is processed first.
2 Initialize: max_end = 0 tracks the largest right endpoint among all surviving (non-covered) intervals so far. count = 0 tracks how many survivors we have found.
3 Scan left to right: For each interval [start, end], check end > max_end. If true: this interval extends the right frontier — it cannot be covered. Increment count and update max_end = end. If false: end <= max_end — the interval's right bound fits inside a previously seen interval's range, and by the sort its left bound does too. It's covered — skip it.
4 Return count. Each increment corresponds to exactly one non-covered interval that contributes unique coverage on the number line.
🔑 Why Must We Sort End Descending for Ties?

Suppose input is [[1,4],[1,2]]. If we only sort by start, we might get [1,2] first. Processing [1,2]: 2 > 0 → count=1, max_end=2. Then [1,4]: 4 > 2 → count=2. Wrong! The correct answer is 1 since [1,2] is inside [1,4]. The fix: sort end descending so [1,4] comes first. Processing [1,4]: max_end=4. Then [1,2]: 2 <= 4 → covered. Count = 1. Correct.

⚠️ The Start Check is Free After Sorting

After sorting by start ascending, every interval that comes later has start >= earlier_start. So we never need to explicitly check the start condition for coverage — the sort guarantees it. We only need to check the end.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every step during the scan, max_end equals the largest right endpoint among all surviving (non-covered) intervals processed so far. An interval [a, b] is covered by some earlier survivor [c, d] if and only if c <= a (guaranteed by sort) and d >= b (equivalent to b <= max_end since max_end tracks the largest such d).

Why the end-descending tiebreaker maintains the invariant: When two intervals share the same start — say [s, big] and [s, small] where big > small — the sort ensures [s, big] is processed first. It becomes a survivor with max_end = big. When [s, small] is processed next: small <= big = max_end, so it's correctly identified as covered. Without the tiebreaker, this order could be reversed, causing a false positive.

Greedy correctness: We never need to look ahead. Once an interval is deemed covered, no future interval will "un-cover" it (future intervals start at or after the current one). Once an interval is a survivor, it correctly contributes to max_end for all future comparisons.

⚠️ Common Wrong Approach: Only Comparing Against Adjacent

It is NOT sufficient to check only against the previous interval. An interval might be covered by one two steps back. Example: [[1,8],[2,3],[4,6]]. [4,6] is covered by [1,8], not by [2,3]. The max_end approach handles this correctly because it retains the largest end ever seen.

4 The Code (Annotated) ▾

Solution 1 — Sort + Track Max End (Optimal, Interview Pick)

def removeCoveredIntervals(intervals):
intervals.sort(key=lambda x: (x[0], -x[1])) ← start asc; -end makes larger end sort first on ties
count, max_end = 0, 0 ← survivors found; rightmost boundary of any survivor
for start, end in intervals: ← linear pass over sorted intervals
if end > max_end: ← end exceeds frontier → not covered by anything seen
count += 1 ← this interval is a survivor
max_end = end ← advance the right boundary
# else: end <= max_end → covered, skip ← start already valid by sort; end fits inside → covered
return count ← number of non-covered (surviving) intervals
⚠️ Why lambda x: (x[0], -x[1]) and not just x[0]?

Python's sort is stable and sorts tuples lexicographically. (x[0], -x[1]) sorts by start ascending first; if starts are equal, by -end ascending, which means by end descending. This ensures the longer interval (the potential parent) always precedes the shorter interval (the potential child) when they share a start point.

Solution 2 — Brute Force O(n²)

def removeCoveredIntervals_brute(intervals):
n = len(intervals)
covered = [False] * n ← mark which intervals are covered
for i in range(n): ← for each interval i
for j in range(n): ← check every other interval j
if i == j: continue
a, b = intervals[i]
c, d = intervals[j]
if c <= a and b <= d: ← j covers i
covered[i] = True; break
return sum(1 for c in covered if not c) ← count non-covered intervals
💡 Complexity Summary

Sort + Track Max End: Time O(n log n) sort + O(n) scan = O(n log n). Space O(1) extra (in-place sort)
Brute Force: Time O(n²) nested loops. Space O(n) for covered array
Winner: Sort + Track Max End — linear scan after sort, near-zero extra memory

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Restate and Visualize

Draw intervals on a number line. Ask: "Are endpoints inclusive?" (yes). "Can I have identical intervals?" (yes, one covers the other). Restate: "I need to count intervals that don't fit entirely inside any other interval."

Step 2: Propose Brute Force First

"Naively: for every interval, check if any other interval covers it. O(n²). Correct, but slow. Let me think about how sorting might help."

Step 3: Key Observation — Sort Kills One Dimension

"Coverage requires two conditions: start and end. If I sort by start ascending, the start condition is automatically satisfied by order — I only need to check the end. I'll track the maximum end seen so far."

Step 4: Handle the Same-Start Edge Case

"What if two intervals share the same start? The larger one covers the smaller. I need the larger first, so I add end-descending as a tiebreaker. Now the single scan works cleanly."

🗣️ What to Say to the Interviewer

"I'll sort intervals by start ascending, with end descending for ties.
After this sort, every interval's start is at least as large as all previous intervals' starts.
So coverage reduces to one check: is this interval's end already within max_end?
If yes — covered, skip. If no — survivor, count it and advance max_end.
Total: O(n log n) time, O(1) extra space."
🔮 Likely Follow-up Questions
  • "Return the actual surviving intervals, not just the count?" → Collect intervals where end > max_end
  • "Merge overlapping intervals instead?" → LeetCode 56 — same sort, different merge condition
  • "What if intervals arrive as a stream?" → Need SortedList or interval tree; pre-sorting impossible
  • "Count covered intervals instead?" → Return len(intervals) - count
