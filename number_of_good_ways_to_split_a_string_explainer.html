LeetCode #1525

Number of Good Ways to Split a String

Find the number of indices i where splitting s into s[0..i] and s[i+1..n-1] yields equal counts of distinct characters on both sides.

Medium Hash Tables Prefix-Suffix Distinct Counts Time: O(n) Space: O(k)
1 Understanding the Problem ▾

You are given a string s. A split at index i divides the string into a left part s[0..i] (inclusive) and a right part s[i+1..n-1]. A split is good if both parts contain the same number of distinct characters. Return the total number of good splits.

💡 Key Insight

We don't need to recount distinct characters from scratch at every possible split point — that would be O(n²). Instead, precompute a prefix distinct count (scanning left to right) and a suffix distinct count (scanning right to left), then a single linear pass checks each split point in O(1).

Concrete example: s = "aacaba"

Edge Cases: A split must leave both sides non-empty — i ranges from 0 to n-2. If all characters are identical, every split gives 1 distinct on each side → all n-1 splits are good. Empty or single-character strings have no valid splits.

2 The Algorithm: Prefix + Suffix Distinct Count ▾

The core idea is to separate precomputation from comparison. Two passes build distinct-count arrays, a third pass counts good splits. Alternatively (and more efficiently), we can use a running counter in a single pass.

1 Initialize: Start with right_count = Counter(s) (all characters on the right) and left_count = Counter() (empty left). This lets us treat the "right side" as shrinking while "left side" grows.
2 Walk left to right (i from 0 to n-2): Move character s[i] from right_count to left_count. Update both counters: increment left, decrement right. If right_count[c] hits zero, delete it (to keep len() accurate).
3 Check after each move: Compare len(left_count) vs len(right_count). The len() of a Counter gives the number of keys with nonzero counts — which equals the distinct character count. If equal, increment result.
4 Return result. We never check i = n-1 (loop ends at n-2) so both sides always remain non-empty. The total number of good splits is accumulated in result.

🔑 Why len(Counter) Gives Distinct Count

A Python Counter stores only keys with nonzero counts (if you delete zeros). So len(counter) = number of distinct characters still present. When right_count[c] reaches 0, we delete that key — otherwise len() would overcount. This is the crucial detail that makes the single-pass approach correct.

3 Why It Works: The Invariant ▾
🔐 The Invariant

After processing index i in the main loop: left_count contains exactly the character frequencies of s[0..i], and right_count contains exactly the frequencies of s[i+1..n-1]. Their len() values give the distinct character counts of each partition at split point i.

Why incremental updates are correct: At each step, we move exactly one character from right to left. This perfectly mirrors the definition of a split: s[i] belongs to the left side when split at i. The counters stay in sync with the actual partition without any recomputation.

Why deleting zeros is essential: Consider s = "ab". After moving 'a' to left: left = {a:1}, right should be {b:1}. If we don't delete 'a' from right when it hits 0, right = {a:0, b:1} and len(right) = 2 instead of 1. This false count would prevent detecting a good split. The delete step is what makes len() reliable as a distinct-count proxy.

⚠️ Off-by-One: Valid Split Range is [0, n-2]

A valid split must leave both sides non-empty. The loop runs for i in range(len(s) - 1), so i goes from 0 to n-2. At i=n-2, left gets s[0..n-2] and right has only s[n-1] — both non-empty. At i=n-1, the right side would be empty — that's not a valid split, and the loop correctly excludes it.

Space note: The two counters together hold at most k distinct characters (k ≤ 26 for lowercase English). Space is O(k) — effectively O(1). No prefix/suffix arrays needed, unlike the three-pass approach.

4 The Code (Annotated) ▾

Solution 1 — Running Counter (Interview Pick, O(n) time, O(k) space)

from collections import Counter
def numSplits(s: str) -> int:
right_count = Counter(s) ← all chars start on right; O(n) one-time build
left_count = Counter() ← left begins empty
result = 0
for i in range(len(s) - 1): ← i = 0..n-2; ensures both sides non-empty
c = s[i]
left_count[c] += 1 ← s[i] moves to the left partition
right_count[c] -= 1 ← remove from right partition
if right_count[c] == 0: ← zero-count key would corrupt len()
del right_count[c] ← delete it so len() stays accurate
if len(left_count) == len(right_count):
result += 1 ← equal distinct counts = good split
return result
⚠️ Common Mistake: Forgetting to Delete Zero-Count Keys

If you omit the del right_count[c] step when the count hits 0, the key persists with value 0. len(right_count) then counts that character as still "present" on the right side — giving a wrong distinct count. Always clean up zeros when using len() as a distinct-count proxy for a Counter.

Running Counter (Sol 1): Time O(n), Space O(k) where k ≤ 26 for lowercase → effectively O(1)

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Prefix-Suffix Pattern

Whenever a problem asks "for each split point, compare left vs. right property", think prefix + suffix precomputation. The phrase "count distinct characters on both sides" is the classic trigger: you need boundary-aware aggregation, and precomputing avoids re-scanning.

Step 2: Propose Brute Force First

"A brute force O(n²) approach: for each index i, compute len(set(s[:i+1])) and len(set(s[i+1:])). But this rebuilds both sets from scratch at each split. If we precompute, we can answer each split in O(1)."

Step 3: Explain the Running Counter Trick

"Instead of precomputing arrays, I can keep two running counters. Start with all characters on the right. Walk left to right, moving each character from right to left. The key insight: len(Counter) gives distinct count — but only if we delete zero-count keys."

Step 4: Handle the Zero-Deletion Detail

"When a character's count in right_count drops to 0, I delete that key. Without this, len() would include it as a phantom distinct character. This is the subtle but critical correctness detail."

🗣️ What to Say to the Interviewer

"We're comparing distinct character counts at every split — classic prefix-suffix pattern.
Brute force: O(n²) by rebuilding sets at each split. Can we do better?
Yes — initialize right_count as Counter(all of s), left_count as empty.
Walk left to right. At each step, move s[i] from right to left counter.
Check: len(left_count) == len(right_count) → good split. Increment result if so.
One crucial detail: when right_count[c] hits 0, delete it — or len() gives wrong answers.
Overall: O(n) time, O(k) space where k ≤ 26. Single pass!"
🔮 Likely Follow-up Questions
  • "What if equal frequency (not just equal count of distinct)?" → Compare Counter objects directly, not their lengths
  • "Can you do it without Counter — just sets?" → Yes, maintain two sets; when removing, check if char appears later in right part (harder, needs freq tracking)
  • "What's the space if you use prefix+suffix arrays?" → O(n) for arrays vs O(k) for running counter
  • "What if the string has Unicode?" → Counter/sets handle any hashable type automatically

7 Summary & Key Takeaways ▾
The "distinct count at every split" pattern calls for prefix + suffix precomputation, or a running counter — not repeated full rescans
len(Counter) gives distinct count only when zero-count keys are deleted; always clean up after decrement
Valid split indices are [0, n-2] — loop with range(n-1) to guarantee both sides remain non-empty
Running counter approach uses O(k) space vs O(n) for explicit prefix/suffix arrays
The pattern generalizes: any "split-point property on both sides" can be precomputed in two passes
ApproachTimeSpaceNotes
Brute Force (slice + set) O(n²) O(n) Rebuilds partitions from scratch at every i
Prefix + Suffix Arrays O(n) O(n) Three passes; very readable; O(n) auxiliary arrays
Running Counter ✓ (pick) O(n) O(k) Single pass; O(1) for lowercase English; most elegant
🎯 When to Recognize the Prefix-Suffix Distinct Count Pattern
  • Problem mentions "every split point" or "for each index i, compare left and right"
  • You need a property (count, sum, product) of both sides of a partition
  • Naively rebuilding from scratch gives O(n²) — precompute to reduce to O(n)
  • "Distinct" or "unique" in the problem → use a set or carefully managed Counter
  • Running counter is preferred when you can "move" elements one at a time
