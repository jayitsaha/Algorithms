LeetCode #437

Path Sum III

Given a binary tree and an integer targetSum, return the number of paths where the sum of node values equals targetSum. Paths must go downward (parent → child) but need not start or end at root/leaf.

Medium Trees Prefix Sum + Hash Map Time: O(n) Space: O(n)
1 Understanding the Problem ▾

We have a binary tree and a target integer. We want to count how many downward paths have a node-sum equal to the target. A valid path can start at any node and end at any deeper node — as long as it follows parent-to-child edges (no upward or diagonal jumps).

💡 Key Insight

Every downward path from node A to node B has sum = prefix_sum(B) − prefix_sum(parent of A). If we track prefix sums during DFS using a hash map, we can check "how many ancestors provide a valid path start?" in O(1) per node — turning an O(n²) problem into O(n).

Why this is tricky: Paths starting at any node rules out simple root-to-leaf DP. The naive fix — restart DFS at every node — costs O(n²). The elegant fix is to borrow the prefix-sum trick from the "Subarray Sum Equals K" array problem and adapt it for trees.

Edge Cases: Negative node values (no monotone assumption), target = 0, single-node tree (node.val == target counts as 1 path), all-negative tree, very deep tree (recursion depth — use iterative DFS for very large inputs in production).

2 The Algorithm: DFS + Prefix Sum Hash Map ▾

We adapt the classic "Subarray Sum Equals K" trick (LeetCode #560) to a tree. On a flat array, we track cumulative sums. On a tree, the DFS call path from root to current node IS the array — we track prefix sums of that path in a hash map as we go down, and undo them on the way back up (backtrack).

1 Seed prefix_map[0] = 1: Represents the "empty path before the root." Without this, paths that sum to targetSum starting from the root itself would be silently missed (the lookup would return 0 instead of 1).
2 DFS with running prefix sum: At each node, compute curr_sum = parent_prefix_sum + node.val. This gives the sum from the root all the way down to (and including) the current node.
3 Look up the complement: Check prefix_map[curr_sum − target]. This asks: "how many ancestors have a prefix sum such that cutting from just after them to here gives exactly target?" Each hit is a valid path ending at the current node.
4 Insert current prefix sum: Add prefix_map[curr_sum] += 1 before recursing into children, so children can use current node as a path start.
5 BACKTRACK — remove on return: After both children return, do prefix_map[curr_sum] -= 1. This un-marks the current node so sibling subtrees don't see a stale entry from a different branch.
Core Equation
path_sum(A → B) = prefix_sum(B) − prefix_sum(parent of A)
We want: path_sum == target
→ prefix_sum(B) − prefix_sum(parent of A) == target
→ prefix_sum(parent of A) == prefix_sum(B) − target
→ lookup prefix_map[ curr_sum − target ]
🔑 The {0: 1} Seed Explained

If a path goes all the way from the root to node B, there's no "parent of A" — the path just starts at root. In that case, curr_sum − target == 0. The seed entry prefix_map[0] = 1 ensures this lookup returns 1 (one such "empty ancestor" exists), correctly counting the root-starting path.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At any point during DFS, prefix_map contains exactly the prefix sums of nodes on the current root-to-current-node path, plus the seed {0:1}. The count for each sum tells how many nodes on the current path produced that prefix sum. Nothing from a different branch is ever present.

Why backtracking maintains the invariant: When we return from a subtree, we decrement the current node's prefix sum. If we didn't, the map would retain entries from visited-but-not-current-path nodes. Sibling subtrees would then incorrectly "see" those entries and over-count paths.

Concrete failure without backtrack: Consider a balanced tree. After exploring the entire left subtree, if we forget to clean the map, the right subtree's lookups would find prefix sums from the left branch — creating phantom valid paths that don't exist.

⚠️ Negative Values Don't Break This

Unlike binary search approaches that assume prefix sums are monotone, the hash map technique makes no such assumption. Negative node values simply produce smaller or negative prefix sums — valid hash map keys. The algorithm works identically.

Why every valid path is captured exactly once: A downward path from A to B means A is a proper ancestor of B. At the moment we process B during DFS, A is guaranteed to be on our current root-to-B path — so A's prefix sum is in the map. The lookup at B finds it. No double-counting: each (A, B) pair is checked exactly once, at the moment we process B.

4 The Code (Annotated) ▾

Solution 1 — DFS + Prefix Sum Hash Map (Optimal, O(n), Interview Pick)

from collections import defaultdict
def pathSum(root, targetSum): ← outer function; initializes map and starts DFS
prefix_map = defaultdict(int) ← maps prefix_sum → count of path nodes having it
prefix_map[0] = 1 ← seed: "empty path" prefix=0 exists once
def dfs(node, curr_sum): ← inner DFS; curr_sum = prefix from root to node
if not node: return 0 ← null node: no paths, return 0
curr_sum += node.val ← extend running prefix sum through this node
count = prefix_map[curr_sum - targetSum] ← how many ancestors give valid start for this end?
prefix_map[curr_sum] += 1 ← record this node before recursing (children see it)
count += dfs(node.left, curr_sum) ← recurse left subtree
count += dfs(node.right, curr_sum) ← recurse right subtree
prefix_map[curr_sum] -= 1 ← BACKTRACK: unmark this node before returning to parent
return count ← total paths found in this node's entire subtree
return dfs(root, 0) ← launch DFS with starting prefix sum = 0
🚫 Do NOT Forget the Backtrack Line

prefix_map[curr_sum] -= 1 is non-optional. Removing it makes the map "leak" prefix sums into sibling branches, causing incorrect path counts. This is the single most common bug on this problem. Always pair every insert with a matching decrement on return.

Solution 2 — Brute Force: Double DFS (O(n²), easier to understand first)

def pathSum_brute(root, targetSum):
if not root: return 0
def paths_from(node, remaining): ← count paths starting exactly at this node
if not node: return 0
found = 1 if node.val == remaining else 0
found += paths_from(node.left, remaining - node.val)
found += paths_from(node.right, remaining - node.val)
return found
return (paths_from(root, targetSum) ← try root as start
+ pathSum_brute(root.left, targetSum) ← try all starts in left subtree
+ pathSum_brute(root.right, targetSum)) ← try all starts in right subtree
💡 Complexity at a Glance

Brute Force: O(n²) time — each of n nodes starts a O(n) sub-DFS. O(n) space (call stack).
Prefix Sum: O(n) time — one single DFS pass. O(n) space (hash map + recursion depth).
The hash map trades constant extra space for a factor-of-n time improvement.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Name the hard part — "any starting node"

Root-to-leaf path sums are easy. The difficulty is paths starting anywhere. Say: "I'll first think about how to count paths starting at each node, and see if I can avoid repeating work."

Step 2: Propose brute force and quantify its cost

"Naive: for each node, run DFS counting paths from that node. O(n) nodes × O(n) DFS = O(n²). Correct but quadratic — let's see if we can do O(n)."

Step 3: Connect to Subarray Sum Equals K

"I've seen this pattern on flat arrays: track prefix sums, look up (curr − target) in a hash map. A DFS path from root to current node IS a linear sequence of prefix sums. I can apply the same trick here."

Step 4: Explain the two non-obvious pieces

"Two things need explaining: (1) seed prefix_map[0]=1 so root-starting paths are caught, and (2) decrement after returning from children so the map only reflects the current root-to-node path, not all visited nodes."

🗣️ What to Say to the Interviewer

"I'll start with brute force: try each node as a start — O(n²).
To optimize: I borrow the prefix-sum trick from array problems.
One DFS pass: maintain running prefix sum and a map of sums seen on the path so far.
At each node, lookup (curr_sum − target) — hits are valid path starts.
Seed map with {0:1} to handle root-starting paths.
After children return: decrement map entry — this is the backtrack that keeps the map
scoped to only the current root-to-node path. Final complexity: O(n) time, O(n) space."
🔮 Likely Follow-up Questions
  • "What if paths could go upward or cross branches?" → Much harder: need LCA + prefix sums from root to both endpoints
  • "Return the actual paths, not just the count?" → Maintain a current_path list during DFS; append node on enter, pop on backtrack
  • "Tree has 50,000 nodes — recursion depth concern?" → Convert to iterative DFS with explicit stack; simulate call frames manually
  • "What if the tree is very wide?" → DFS stack depth = tree height, not width — no issue for wide-but-shallow trees

7 Summary & Key Takeaways ▾
Downward path sum = difference of two prefix sums — enables O(1) per-node lookup
Seed prefix_map[0] = 1 to correctly count root-starting paths
Backtrack (decrement) after children return — keeps map scoped to current path only
Works for negative values — no monotone assumption, pure hash map math
O(n) time vs O(n²) brute force — hash map trades space for time
