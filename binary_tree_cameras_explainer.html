LeetCode #968

Binary Tree Cameras

Place the minimum number of cameras in a binary tree such that every node is monitored. A camera placed at a node covers its parent, itself, and its children.

Hard Dynamic Programming Tree DP Post-order 3 States Time: O(n) Space: O(h)
1 Understanding the Problem ▾

You have a binary tree. You can install a camera at any node. A camera placed at node X monitors: the node itself, its parent, and both its children. Your goal is to find the minimum number of cameras needed so that every single node in the tree is monitored by at least one camera.

💡 Key Insight

A camera is most efficient when placed at an internal node (not a leaf), because it can cover both its children AND its parent in one shot. Placing cameras at leaves is wasteful — they can only cover themselves and their parent. The greedy insight: never place a camera at a leaf. Instead, force their parents to hold the camera, which then covers the grandparent too.

Edge Cases: A tree with a single node (1 camera needed, since null children don't count as uncovered). A path of 2 nodes needs 1 camera (at either node, since one covers both). The empty tree is not in the input. Note that null nodes are treated as already-covered (they don't need monitoring).

⚠️ Null = Covered, Not Missing

A critical rule: null children are treated as "covered" (not "uncovered"). This means a camera at a leaf node does NOT need to cover null — it just covers itself and its parent. This shapes the entire 3-state design.

2 The Algorithm: Post-order Tree DP with 3 States ▾

This is a Tree DP problem. We run a post-order DFS (Left → Right → Root), so each node receives full information about its subtrees before making its own decision. Each node returns one of 3 states:

0
NOT Covered
This node is not yet monitored by any camera. Its parent must place a camera.
1
HAS Camera
This node has a camera installed. It covers itself, its children, and its parent.
2
Covered
This node is monitored (by a child's camera) but does not itself have a camera.

Null nodes return state 2 (covered/satisfied) — they don't need monitoring.

1 Post-order DFS: Recursively process left and right subtrees first, then the current node.
2 If either child is uncovered (state 0): Place a camera here! Increment the counter and return state 1 (HAS_CAMERA). This is the greedy move — if a child needs coverage, the parent is the cheapest place to put the camera.
3 If either child has a camera (state 1): This node is now covered. Return state 2 (COVERED) — no camera needed here.
4 Otherwise (both children covered but no camera among them): This node is not yet covered. Return state 0 (NOT_COVERED) and let the parent handle it.
5 Root special case: After the DFS, if the root itself returns state 0 (not covered), place one more camera there. The root has no parent to rescue it.
📐 State Transition Logic (per node)
If left==0 or right==0 → place camera here, return 1 (HAS_CAMERA)
Elif left==1 or right==1 → covered by child, return 2 (COVERED)
Else (both children are COVERED but no camera) → return 0 (NOT_COVERED)
After root: if root returns 0 → cameras += 1
3 Why It Works: The Greedy Invariant ▾
🔐 The Invariant

At each node during post-order traversal: every node in the already-processed subtrees is guaranteed to be covered. We never leave a subtree with an uncovered node — we always pass the "not covered" responsibility upward exactly once, to be resolved by the immediate parent.

Why is this greedy optimal? Consider a chain of nodes A → B → C (A is root, C is leaf). If we process bottom-up:

  • C (leaf) has null children (state 2). C returns state 0: "I'm not covered, parent B must act."
  • B sees C is uncovered → places camera at B. B returns state 1: "I have a camera, which covers C and A."
  • A sees B has a camera → A is covered. A returns state 2.
  • Root is A, state 2 (covered). Done! 1 camera.

If we had placed the camera at C instead, A would remain uncovered (C's camera doesn't reach A). We'd need 2 cameras. By deferring the camera decision upward, we always get maximum coverage per camera.

💡 Why Post-order (not Pre-order)?

Pre-order processes the root first — but a root-first approach can't know whether its children will "rescue" it via a camera. Post-order guarantees full subtree information before a node makes its camera decision. This is the essential difference.

Edge case: null nodes return 2 (covered). If we returned 0 instead, leaf nodes would always think their null children need cameras — they'd always install a camera, which is wasteful. Returning 2 from null means: "I'm fine, don't worry about me."

4 The Code (Annotated) ▾

The optimal solution is a post-order DFS returning one of 3 integer states per node, with a nonlocal camera counter.

cameras = 0← global counter for cameras placed

nonlocal cameras
if node is None: return 2← null = treated as already covered (base case)
left = dfs(node.left)← process left subtree first (post-order)
right = dfs(node.right)← process right subtree
if left == 0 or right == 0:← a child is uncovered → MUST place camera here
cameras += 1← install camera at this node
return 1← state: has_camera (covers parent too)
if left == 1 or right == 1:← a child has a camera → this node is covered!
return 2← state: covered (no camera needed here)
return 0← both children covered but no camera → this node is NOT covered yet
if dfs(root) == 0:← root has no parent; if it's uncovered, place final camera
cameras += 1
return cameras
⚠️ Common Mistake: Checking state 1 before state 0

The order of the if checks matters! Check for state 0 (uncovered) first. If you check for state 1 first, a node with one uncovered child and one camera child would return state 2 (covered) — incorrectly skipping the uncovered child.

📊 Complexity

Time: O(n) — Each node is visited exactly once in the DFS. Space: O(h) — The recursion stack depth equals the tree height h. In the worst case (a path/chain), h = n, so O(n). For a balanced tree, O(log n).

5 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Pattern — Greedy on Trees = Post-order DP

When you see "binary tree + optimization (min/max)" and each node's decision depends on its children's state, think post-order DFS. You need to process leaves first because their state feeds into the parent's decision. This is the Tree DP pattern.

Step 2: Define the Minimal State — What Does a Node Need to Know?

A node needs to report to its parent: "Do you need to rescue me (state 0)?" "Do I have a camera that covers you (state 1)?" or "I'm fine, don't worry about me (state 2)?". Three states are the minimum needed to propagate correct decisions upward.

Step 3: Write Transitions — Greedy Choice at Each Node

Given child states, what should the current node do? The key greedy principle: delay cameras as high as possible. Only place a camera if you MUST (a child is uncovered). Otherwise, propagate the right state upward and let the parent decide.

Step 4: Handle the Root Edge Case

The root has no parent. If the post-order DFS returns state 0 for the root, there's no parent to rescue it — you must place one final camera at the root. Always check the root's return value explicitly.

What to Say to the Interviewer

"This is a tree optimization problem. I see a node covers itself, parent, and children — so coverage propagates upward and downward. I'll use post-order DFS because I need to know the subtree state before deciding what to do at a node. I'll define 3 states per node: 0 = not covered yet, 1 = has a camera, 2 = covered by a child's camera. Base case: null nodes return 2 — they don't need coverage. The greedy rule: if any child returns 0 (uncovered), I MUST place a camera here — that's the earliest optimal moment. If any child returns 1, I'm covered — return 2. Otherwise, return 0 to signal my parent. After DFS, if the root is state 0, I need one more camera there. Time: O(n), Space: O(h) for the recursion stack."

7 Summary & Key Takeaways ▾
Post-order DFS is the key — process children before the parent so decisions flow bottom-up
3 states per node: 0 (not covered), 1 (has camera), 2 (covered by child)
Greedy rule: only place a camera when a child is uncovered (state 0) — never greedily at leaves
Null nodes return state 2 (covered) — they don't need monitoring
Always check the root after DFS — it has no parent to rescue it if it returns state 0
This "return pair/tuple of states" pattern extends to House Robber III and similar Tree DP problems

🎯 When to Recognize the Tree DP / Post-order 3 States Pattern
  • Problem involves a binary tree with an optimization objective (min/max)
  • Each node's decision depends on the state of its children
  • There are a small, fixed number of possible "states" a node can be in
  • The result must propagate upward — post-order is natural
