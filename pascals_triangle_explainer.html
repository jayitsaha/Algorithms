LeetCode #118

Pascal's Triangle

Given an integer numRows, return the first numRows of Pascal's triangle. Each element equals the sum of the two elements directly above it.

Easy Dynamic Programming Sum of Two Above Time: O(n²) Space: O(n²)
1 Understanding the Problem ▾

Pascal's Triangle is one of the most elegant structures in mathematics. You build it row by row: each row begins and ends with 1, and every interior cell is the sum of the two cells directly above it from the previous row.

💡 Key Insight

The rule is beautifully simple: row[i][j] = row[i-1][j-1] + row[i-1][j]. Every row can be computed entirely from the row before it. This is the classic "build from already-known subproblems" pattern of dynamic programming — the current row is our DP state, and the transition is the sum formula.

Edge cases: numRows = 1 returns [[1]]. numRows = 0 returns an empty list (though LeetCode guarantees 1 ≤ numRows ≤ 30). Row i always contains exactly i+1 elements. Each row sums to 2i — a fun bonus fact.

2 The Algorithm: Build Row-by-Row (Bottom-Up DP) ▾

This is textbook bottom-up dynamic programming. We build the answer by filling a table — row by row — where each row depends only on the row before it. There is no "smarter" shortcut: every cell genuinely needs both its parents above to produce the correct value.

1 Seed the table: The result list starts as [[1]]. Row 0 is always a single 1 — it is the base case from which everything else is derived. This is our initial DP state.
2 Loop i from 1 to numRows−1: On each iteration, grab the last completed row: prev = result[-1]. This is the "subproblem result" we stored and are now looking up.
3 Build new_row: Start with [1] (left edge). For each interior index j from 1 to len(prev)−1, append prev[j-1] + prev[j]. Then append a final 1 (right edge).
4 Store and repeat: Append new_row to result. This completed row becomes the prev for the next iteration. After numRows loops, return result.
📐 The Recurrence Relation
row[i][0] = 1                      (left edge — base case)
row[i][j] = row[i-1][j-1] + row[i-1][j]   (interior cells — DP transition)
row[i][i] = 1                      (right edge — base case)
🔑 Why This Is DP, Not Just Simulation

In naive "simulation" you might just apply the rule without thinking about it. But DP has a precise meaning: you store computed subproblem results (each completed row) and look them up to solve larger subproblems (the next row). The "state" is the entire previous row; the "transition" is the sum formula. This is classic tabulation (bottom-up DP) — the output is literally the DP table itself.

3 Why It Works: The Invariant & DP Foundations ▾
🔐 The Invariant

After processing row i, result[i][j] equals the binomial coefficient C(i, j) = i! / (j! × (i−j)!) for all valid j. The recurrence C(n,k) = C(n-1,k-1) + C(n-1,k) is mathematically equivalent to our sum formula — this is Pascal's Identity.

✓ Optimal Substructure

Row i is completely determined by row i−1. To build row 4, you only need row 3. You don't need to recompute rows 0–2. The subproblem answer (prev row) directly produces the larger answer (current row).

✗ Without Memoization (Recursive)

A naive recursive C(n,k) = C(n-1,k-1) + C(n-1,k) recomputes the same cells exponentially. C(5,2) requires C(4,1)+C(4,2), each of which re-expands — exponential blowup without memoization.

⚠️ Why Not Use the Factorial Formula Directly?

You could compute each cell as C(i,j) = i! / (j! * (i-j)!). But for large n, factorials overflow standard integer types, and you'd need careful modular arithmetic. The row-by-row DP approach uses only additions — no overflow, simpler code, and it's what the interviewer expects.

Why edges are always 1: C(i,0) = 1 (one way to choose 0 items from i). C(i,i) = 1 (one way to choose all i items). Interior cells add the "include last item" vs "exclude last item" counts — hence the sum of two parents above.

Symmetry property: Pascal's Triangle is always symmetric: C(n,k) = C(n,n-k). Row n reads the same forwards and backwards. This is a nice sanity check for your implementation.

4 The Code (Annotated) ▾

Solution 1 — Bottom-Up DP / Tabulation (Optimal, Interview Pick)

def generate(numRows: int) -> list[list[int]]:
result = [[1]] ← Seed: row 0 = [1]. Our DP table begins here.
for i in range(1, numRows): ← Build rows 1 through numRows-1
prev = result[-1] ← Fetch the last completed row (our DP "state")
new_row = [1] ← Left edge is always 1 — C(i,0) = 1
for j in range(1, len(prev)): ← Interior positions: j=1 to i-1
new_row.append(prev[j-1] + prev[j]) ← DP transition: sum of two parents above
new_row.append(1) ← Right edge is always 1 — C(i,i) = 1
result.append(new_row) ← Store completed row — becomes "prev" next iteration
return result ← All numRows rows built; the DP table is our answer
⚠️ The Subtle Off-by-One in the Interior Loop

range(1, len(prev)) iterates j from 1 to len(prev)−1 inclusive. For row i, prev has i elements, so this runs i−1 times. Combined with the two edge 1s, new_row has 1 + (i−1) + 1 = i+1 elements — exactly right for row i. Don't accidentally use range(len(prev)−1) or you'll miss the last interior cell.

Solution 2 — Top-Down Memoization (also O(n²))

from functools import lru_cache
def generate(numRows: int) -> list[list[int]]:
@lru_cache(maxsize=None) ← Cache computed cells; avoid re-expansion
def C(n, k): ← C(n,k) = Pascal's entry at row n, col k
if k == 0 or k == n: ← Base cases: edges of the triangle are always 1
return 1
return C(n-1, k-1) + C(n-1, k) ← Pascal's Identity — identical to our row rule
return [[C(i,j) for j in range(i+1)] for i in range(numRows)]
💡 Complexity Summary

Bottom-Up DP: Time O(n²) — fills every cell exactly once. Space O(n²) — stores all rows.
Memoized Recursion: Same O(n²) time and space, but with function call overhead.
Space Optimization (for #119): If you need only the k-th row, keep one rolling array and update right-to-left → O(k) space.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Spot the DP Signal

The problem says "each number is the sum of the two numbers above it." That's a recurrence relation — each value depends on previously computed values. Say: "Since every cell depends only on the row above it, this is a bottom-up DP problem where each row is both a subproblem result and the input to the next subproblem."

Step 2: Define State and Transition Clearly

State: dp[i][j] = value at row i, position j. Transition: dp[i][j] = dp[i-1][j-1] + dp[i-1][j] for interior cells. Base: dp[i][0] = dp[i][i] = 1. Stating these explicitly shows you're thinking in DP terms, not just coding a pattern.

Step 3: Implement Row-by-Row

Seed with [[1]]. Each iteration: grab prev = result[-1], build new row (edges = 1, interior = adjacent pairs summed), append to result. Clean, no edge-case surprises.

Step 4: Proactively Mention the Follow-up

Say: "If you only needed the k-th row (LeetCode #119), I could reduce space to O(k) by maintaining a single array and updating it in-place from right to left." This shows mastery of the space-optimization trick common in 1D DP.

🗣️ What to Say to the Interviewer

"This is a bottom-up DP problem. The state is the entire previous row, and the transition is the sum of adjacent elements.
I'll seed with [[1]], then for each row i I grab prev = result[-1], build new_row with edges=1 and interior = prev[j-1]+prev[j], then append.
Time: O(n²) — every one of the n(n+1)/2 cells is visited exactly once.
Space: O(n²) — we store all rows. If only the last row was needed, I could get that down to O(n)."
🔮 Likely Follow-up Questions

7 Summary & Key Takeaways ▾
Every interior cell = sum of two cells above — the DP recurrence dp[i][j] = dp[i-1][j-1] + dp[i-1][j]
Both edges of every row are always 1 — the base cases that anchor the recurrence
Bottom-up tabulation: build each row from the fully computed previous row, store all
Total work = 1+2+…+n = n(n+1)/2 → O(n²) time. Storing all rows = O(n²) space
Pascal's Triangle IS the DP table — the output IS the filled table
For only the k-th row: update array in-place right-to-left → O(k) space (LeetCode #119)

🎯 When to Recognize the "Sum of Two Above" DP Pattern
  • Problem says "each value is sum/combination of two adjacent values from the row/level above"
  • Output is a 2D structure where each row depends entirely on the previous row
  • Combinatorics problems involving counts in triangular or pyramidal form
  • Whenever "return all rows" — that's the DP table itself as output
