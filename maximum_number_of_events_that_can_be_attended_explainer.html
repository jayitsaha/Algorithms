LeetCode #1353

Maximum Number of Events That Can Be Attended

Given events[i] = [startDay, endDay], attend the maximum number of events. You may attend one event per day — on any day within the event's [start, end] window.

Medium Intervals Sort + Min Heap Time: O(n log n) Space: O(n)
1 Understanding the Problem ▾

Think of a multi-day conference where dozens of talks happen across different date ranges. You want to attend as many talks as possible, but your calendar allows only one talk per day. For each talk, you can choose any day within its scheduled window — but once the window closes, you lose your chance forever.

The key freedom here is flexibility within the window. A talk running from day 2 to day 5 doesn't need to be attended on day 2 — you can attend it on day 3, 4, or 5 too. This is what separates this problem from simpler "one slot per event" problems.

💡 Key Insight

The greedy choice is urgency-first: on each day, among all currently available events, attend the one whose window closes soonest. A talk ending tomorrow is more urgent than one ending next week — if you skip the urgent one, you may lose it forever, while the later talk can always be deferred.

✓ Urgency-First Strategy

events = [[1,2],[2,3],[3,4]]
Day 1: attend [1,2] — ends soonest!
Day 2: attend [2,3] — ends soonest!
Day 3: attend [3,4] — last option
Result: 3 events ✓

✗ Lazy / Random Strategy

events = [[1,2],[2,3],[3,4]]
Day 1: attend [3,4] — plenty of time...
Day 2: attend [2,3]
Day 3: [1,2] is EXPIRED — missed!
Result: 2 events ✗

Edge cases: All events on the same day (you can only attend 1), single-day events (no flexibility — must be attended on that exact day), events so long they span the entire input range, many overlapping events all competing for day 1.

2 The Algorithm: Greedy Day-Sweep with Min-Heap ▾

We simulate a calendar from day 1 to the last event day. On each day we maintain a min-heap of open events sorted by end-day. Think of the heap as your "urgency queue" — the most time-sensitive event is always at the top, ready to be attended.

1 Sort events by start day. This lets us load events into the heap exactly when they become available — as we advance the day counter, we know all events with start ≤ current day should be in the heap.
2 Sweep days 1 → max_end_day. For each day, we do three things: load, prune, attend — in that order.
3 LOAD: Push all events with start ≤ current_day into the min-heap (key = end-day). These are now "open" — their window is active.
4 PRUNE: Pop the heap top while heap[0] < current_day. These events have expired — their last day already passed. Important: events ending today are still valid!
5 ATTEND: If the heap is non-empty, pop the minimum (soonest-ending event) and increment count. We've used today's calendar slot for the most urgent open event.

Trace table for events = [[1,3],[1,2],[3,4],[2,3]] (sorted: [[1,2],[1,3],[2,3],[3,4]]):

Day Heap (end-days) after load+prune Action (greedy attend) Count
1 [2, 3] → attend pops 2 Attend [1,2] (end=2, most urgent) 1
2 [3, 3] → attend pops 3 Attend [1,3] or [2,3] (end=3) 2
3 [3, 4] → attend pops 3 Attend remaining end=3 event 3
4 [4] → attend pops 4 Attend [3,4] (end=4) 4
🔑 Why a Min-Heap?

Among all open events, we always want the one with the smallest end-day (most urgent deadline). A min-heap gives us this in O(log n) per operation. Without the heap, finding the minimum on each day costs O(open events) — which could be O(n) per day and O(n · D) total (TLE for n = 10⁵).

3 Why It Works: The Greedy Invariant ▾
🔐 The Greedy Invariant

At every calendar day, the optimal decision is to attend the open event with the earliest deadline. This is safe because: if we skip an event with an early deadline to attend one with a later deadline, the early event might expire before we get another chance — but the later event always has more time remaining and can be attended on a future day.

Exchange argument (formal proof sketch): Suppose an optimal solution OPT makes a choice on day d that differs from our greedy. OPT attends event B (ending day 7) instead of event A (ending day 3). Our greedy attends A. Consider swapping in OPT: attend A on day d, and move B to any later free day (days d+1 through 7 — all valid since B ends on day 7). The swap is always feasible because B's deadline is later. The total count doesn't change. Therefore our greedy is at least as good as OPT — which means it's optimal.

⚠️ Off-by-One in Pruning: < vs <=

We prune with heap[0] < day (strictly less than). This is critical: an event ending on today (i.e., heap[0] == day) is still attendable today and must NOT be pruned. Using <= would incorrectly discard today's last-chance events, reducing the count.

Why does the day-sweep not miss events? We sort events by start day. As we advance the day counter, we push all events with start ≤ day into the heap. Since start ≤ end always, any event pushed on day d is guaranteed to still have end ≥ d — it's valid right now. The heap only grows with currently-valid events and shrinks via pruning and attendance.

💡 Real-World Analogy: The Conference Attendee

You're at a 5-day conference with 20 talks spread across different date ranges. Every morning you look at your "still available" list and pick the talk whose schedule closes soonest. You never regret this choice — skipping the deadline-sensitive talk today (to attend something more comfortable) is a mistake that can't be undone. The heap is your "urgency inbox."

4 The Code (Annotated) ▾

Solution 1 — Sort + Min-Heap (Optimal, Interview Pick)

import heapq← Python's heapq is a min-heap by default
def maxEvents(events: list) -> int:
events.sort() ← Sort by start day; Python tuples sort lexicographically
n = len(events) ← Total number of events
i = 0 ← Index into sorted events for the "load" phase
heap = [] ← Min-heap of end-days for currently open events
count = 0 ← Number of events successfully attended
max_day = max(e[1] for e in events) ← Last possible day; bounds the sweep
for day in range(1, max_day + 1): ← Sweep each calendar day
while i < n and events[i][0] <= day: ← LOAD: push events that have started
heapq.heappush(heap, events[i][1])← Push end-day (the urgency key)
i += 1 ← Advance the load pointer
while heap and heap[0] < day: ← PRUNE: discard expired events
heapq.heappop(heap) ← heap[0] < day means it ended before today
if heap: ← ATTEND: any event available today?
heapq.heappop(heap) ← Pop soonest-ending event — greedy choice
count += 1 ← Use today's slot
return count ← Maximum events we could attend
⚠️ Pruning Condition: < day vs <= day

An event with end == day is attendable right now (today is its last day). We must use heap[0] < day in the prune loop, NOT <= day. The "attending" step immediately after handles the end == day case correctly.

Solution 2 — Brute Force (O(n² log n), for understanding)

def maxEvents_brute(events): ← TLE on n = 10⁵ (O(n²) in worst case)
events.sort(key=lambda e: e[1]) ← Sort by end day (urgency)
used = set() ← Days already occupied
count = 0
for start, end in events:
for day in range(start, end + 1): ← Try earliest free day in window
if day not in used:
used.add(day) ← Mark day as taken
count += 1
break
return count
💡 Complexity Summary

Sort + Min-Heap: O(n log n) sort + O(D · log n) sweep where D = max_end ≤ 10⁵. Since each event is pushed/popped at most once, total heap ops ≤ 2n. Overall: O((n + D) log n). Space O(n).
Brute Force: O(n log n + n·W) where W = max window length. O(n²) worst case. TLE for n = 10⁵.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify the Interval Scheduling Pattern

Keywords: "events with [start, end]," "one per day," "maximize count." Mentally file this under "interval scheduling." The twist: events have flexible windows — you don't need to pick a specific day upfront. This means we can use a greedy approach.

Step 2: Propose the Greedy Observation

"I'll use a greedy strategy: on each day, attend the open event with the earliest deadline. Any event with a later deadline can always be deferred — but an event ending tomorrow must be taken now or never." Sketch the exchange argument briefly.

Step 3: Choose the Right Data Structure

"I need to efficiently find the open event with the smallest end-day. A min-heap gives me this in O(log n). I'll also sort events by start-day upfront so I can load them into the heap exactly when they become available."

Step 4: Implement Carefully

Outline the three operations per day: LOAD (events[i][0] ≤ day), PRUNE (heap[0] < day, NOT ≤), ATTEND (pop minimum, count++). Mention the pruning off-by-one — this is a common interview stumbling point.

🗣️ What to Say to the Interviewer

"This is an interval scheduling problem with flexible windows — greedy works here.
I'll sort events by start day. Then sweep each day from 1 to max_end.
On each day: load newly available events into a min-heap by end-day,
prune expired events (end < today), then attend the soonest-ending open event.
Time O(n log n), Space O(n). The key off-by-one: prune with < day, not <= day."
🔮 Likely Follow-up Questions
  • "What if events have weights/values?" → Greedy breaks! Need DP (#1751)
  • "Attend up to k events per day?" → Pop up to k items from the heap per day
  • "Minimum rooms for all events?" → Classic Meeting Rooms II (#253)
  • "What if each event must be fully attended?" → Back to standard interval scheduling (non-overlapping)

7 Summary & Key Takeaways ▾
Greedy is provably optimal: always attend the soonest-ending available event (exchange argument)
Sort events by start day to enable efficient LOAD as the day counter advances
Min-heap keyed on end-day gives O(log n) access to the most urgent open event
Prune condition: heap[0] < day (strictly less) — events ending today are still valid!
If events carry values/weights, greedy breaks — switch to DP with binary search (#1751)
Total complexity: O(n log n) sort + O(n log n) heap ops = O(n log n) overall
