LeetCode #505

The Maze II

Find the shortest distance for a ball to roll from start to destination in a maze — the ball rolls until hitting a wall.

Medium Graph Dijkstra's with Rolling Time: O(m·n·log(m·n)) Space: O(m·n)
1 Understanding the Problem ▾

You have a maze represented as a 2D grid where 0 = open space and 1 = wall. A ball starts at start and you want to reach destination. The critical twist: the ball doesn't stop wherever you like — it keeps rolling in a direction until it hits a wall, then it stops.

💡 Key Insight

This is not regular BFS. In BFS, each move costs 1 step. Here, one "move" rolls the ball across many cells until a wall stops it. The cost is the number of cells rolled, not 1. This variable cost makes it a weighted shortest path problem — perfect for Dijkstra's algorithm.

Concrete Example: Ball at (0,4), rolls Left. It passes (0,3), (0,2), (0,1) and hits the left wall at (0,0). One "move" but 4 steps of distance. If instead it rolls Down, it might hit a wall at row 4 — one move, 4 steps. Dijkstra's tracks the total step count to reach each stopping point and always processes the closest one first.

✗ BFS would fail here

BFS treats all moves as cost=1.
Rolling 1 cell = rolling 5 cells = "same".
Returns wrong minimum distances.

✓ Dijkstra finds the true minimum

Priority queue ordered by step count.
Greedily processes nearest stopping point.
Returns true shortest rolling distance.

Edge Cases: start equals destination (return 0), destination is unreachable (return -1), ball can reach destination mid-roll but must stop there exactly (destination counts only if the ball stops there), multiple paths to the same stop point (take minimum).

2 The Algorithm: Dijkstra's with Rolling Simulation ▾

We adapt Dijkstra's shortest path algorithm where each node in the graph is a stopping position (where the ball hits a wall) and each edge weight is the number of cells rolled to get there.

1 Initialize: Create a dist[m][n] grid filled with infinity. Set dist[start] = 0. Push (0, start_row, start_col) into a min-heap (priority queue ordered by distance).
2 Pop the closest unvisited stop: Extract (d, r, c) — the stopping point reachable in minimum d steps so far. If (r, c) is the destination, return d immediately.
3 Skip stale entries: If d > dist[r][c], this is an outdated heap entry — we already found a shorter path to (r,c). Skip it.
4 Simulate rolling in all 4 directions: For each of Up/Down/Left/Right, roll the ball from (r, c) cell by cell until hitting a wall or grid boundary. Count the steps rolled. The final position is the new stopping point.
5 Relax edges: If d + steps_rolled < dist[new_r][new_c], update dist[new_r][new_c] and push (d + steps_rolled, new_r, new_c) into the heap.
6 Return result: If heap empties without reaching destination, return -1 (unreachable).
🔑 Why not just BFS?

Modified BFS with step counting would also work (and is the brute-force approach), but it's O(m²n²) in the worst case. Dijkstra's min-heap processes each stopping position only once at its optimal distance — giving O(mn · log(mn)) efficiency. For large mazes this is a significant improvement.

3 Why It Works: The Invariant + Algorithm Deep-Dive ▾
🔐 The Dijkstra Invariant

When a node (r, c) is popped from the min-heap, its distance dist[r][c] is already the shortest possible. This is Dijkstra's fundamental guarantee: since all edge weights are non-negative (you can't roll a negative number of cells), the heap always processes nodes in order of true minimum distance.

Why the invariant holds here: Rolling always adds positive steps. No "negative roll" exists. Therefore, once we've found the minimum-distance path to a stopping position, any future path to the same position via the heap must be at least as long. The greedy "pop minimum" choice is always correct.

Dijkstra's Algorithm — Deep Dive:

  • Origin: Edsger W. Dijkstra, 1956. Solves single-source shortest path in weighted graphs with non-negative edges.
  • Core invariant: Nodes are processed in non-decreasing order of their true shortest distance. The min-heap ensures this ordering.
  • Why it works: At extraction time, no unvisited node can offer a shorter path to the extracted node — any path through an unvisited node would be at least as long as the current heap minimum.
  • When to recognize: "Shortest path" + "variable edge weights" + "non-negative weights". If weights can be negative, use Bellman-Ford instead.
⚠️ The "rolling" abstraction is the graph edge

In a standard graph problem, edges are given explicitly. Here, we construct the edges on-the-fly: for each stopping position, we simulate rolling in each direction to discover all reachable next stops and their costs. This "simulate rolling" is the edge generation step — a common pattern in maze/grid problems adapted for Dijkstra's.

Edge case correctness: If start == destination, the heap pops (0, start) which matches destination — correctly returns 0. If the destination is never popped from the heap (unreachable), we return -1 after the loop.

4 The Code (Annotated) ▾

Solution 1 — Dijkstra's with Rolling (Optimal, Interview Pick)

import heapq
def shortestDistance(maze, start, destination):
m, n = len(maze), len(maze[0]) ← grid dimensions
dist = [[float('inf')] * n for _ in range(m)] ← all distances start as infinity
dist[start[0]][start[1]] = 0 ← start costs 0 to reach
heap = [(0, start[0], start[1])] ← min-heap: (distance, row, col)
while heap: ← keep going until no more stops to explore
d, r, c = heapq.heappop(heap) ← pop the closest stop
if [r, c] == destination: ← found destination at optimal distance
return d
if d > dist[r][c]: ← stale entry: skip (we already found shorter)
continue
for dr, dc in [(-1,0),(1,0),(0,-1),(0,1)]: ← try all 4 directions
nr, nc, steps = r, c, 0 ← start rolling from current position
while 0 <= nr+dr < m and 0 <= nc+dc < n and maze[nr+dr][nc+dc] == 0:
nr += dr; nc += dc; steps += 1 ← roll one cell, count it
new_d = d + steps ← total distance to this new stop
if new_d < dist[nr][nc]: ← found shorter path to (nr, nc)?
dist[nr][nc] = new_d ← update best known distance
heapq.heappush(heap, (new_d, nr, nc)) ← add to heap for future exploration
return -1 ← destination never reached
⚠️ Why check d > dist[r][c] instead of a visited set?

Dijkstra's can push duplicate entries into the heap (same cell with different distances). When we later pop a stale entry with a higher distance, we skip it. Alternatively, you could use a visited set after the first pop of each cell — both approaches work. The stale-check is slightly simpler.

Solution 2 — BFS with Distance Tracking (Brute Force, O(m²n²))

from collections import deque
def shortestDistance_bfs(maze, start, destination):
m, n = len(maze), len(maze[0])
dist = [[float('inf')] * n for _ in range(m)]
dist[start[0]][start[1]] = 0
queue = deque([start]) ← FIFO queue (not ordered by distance)
while queue:
r, c = queue.popleft()
for dr, dc in [(-1,0),(1,0),(0,-1),(0,1)]:
nr, nc, steps = r, c, 0
while 0<=nr+dr
nr+=dr; nc+=dc; steps+=1
if dist[r][c] + steps < dist[nr][nc]: ← found shorter? update and re-explore
dist[nr][nc] = dist[r][c] + steps
queue.append([nr, nc]) ← may be re-visited when shorter path found
d = dist[destination[0]][destination[1]]
return d if d != float('inf') else -1
💡 Complexity Summary

Dijkstra's (Optimal): Time O(mn · log(mn)), Space O(mn) — heap + dist grid
BFS: Time O(m²n²) worst case (each cell re-enqueued many times), Space O(mn)
Note: m and n are maze rows and columns. The log factor comes from heap operations.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify This as a Weighted Graph Problem

Key signal: "shortest distance" + "variable cost per move" (the ball rolls multiple cells per move). This rules out simple BFS (which assumes unit cost). The graph nodes = stopping positions, edges = rolling paths with step counts as weights.

Step 2: Propose BFS First, Then Optimize

Say: "I can do this with BFS + distance tracking — process each stop and roll in 4 directions. But BFS may re-visit cells, giving O(m²n²). If we use a min-heap (Dijkstra's), we guarantee each stop is processed at its optimal distance — O(mn·log(mn))."

Step 3: Implement the Rolling Simulation

The core is the rolling while-loop: keep moving in a direction until hitting a wall or boundary. This constructs the graph edges on-the-fly. Don't forget to count steps and check bounds at the next position, not the current one.

Step 4: Handle Edge Cases Clearly

If start == destination: return 0. Destination unreachable: return -1. Stale heap entries: check if d > dist[r][c] and skip. Destination must be where the ball stops, not just passes through.

🗣️ What to Say to the Interviewer

"The ball rolls until hitting a wall — this means each 'move' has variable cost based on distance rolled.
Variable-cost shortest path screams Dijkstra's to me.
Nodes are stopping positions; edges are simulated by rolling in each direction until a wall.
I'll use a min-heap ordered by total steps rolled.
Pop the closest stop, check if it's the destination, then simulate rolling in 4 directions.
For each direction, count steps until hitting a wall — that's our edge weight.
Relax: if new distance is shorter, update and push to heap.
If destination is popped, return distance. If heap empties, return -1."

The ball rolls until hitting a wall — each roll is an edge with weight = cells crossed
Variable-cost shortest path on a graph → Dijkstra's with a min-heap
Rolling is simulated on-the-fly: advance cell-by-cell until boundary or wall

🎯 When to Recognize the Dijkstra Pattern in Grid Problems
  • "Shortest path" + "variable cost per step/move" → Dijkstra's, not BFS
  • "Ball rolls until wall" / "object moves until blocked" → rolling simulation + Dijkstra's
  • Grid but costs differ by direction or path type → weighted graph = Dijkstra's
  • Need to find minimum total cost, not minimum number of moves → Dijkstra's
