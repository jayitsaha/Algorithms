LeetCode #2095

Delete the Middle Node of a Linked List

Given the head of a linked list, delete the middle node and return the modified head. The middle is the ⌊n/2⌋ᴰʰ node (0-indexed).

Medium Linked List Fast-Slow Pointers Time: O(n) Space: O(1)
1 Understanding the Problem ▾

You have a singly-linked list and need to remove exactly one node — the middle one. The challenge: linked lists have no indices or built-in len(). You can only move forward, one node at a time.

💡 Key Insight

The "middle" node is defined as the ⌊n/2⌋th node (0-indexed). For n=5, that's index 2 (value 3). For n=4, it's index 2 as well — the second middle for even-length lists. This differs from "Middle of the Linked List" (#876) which returns the first middle.

Example: n=5 (odd)

Input:  1 → 2 → [3] → 4 → 5
Middle: index ⌊5/2⌋ = 2 (value 3)
Output: 1 → 2 → 4 → 5

Example: n=4 (even)

Input:  1 → 2 → [3] → 4
Middle: index ⌊4/2⌋ = 2 (value 3)
Output: 1 → 2 → 4

To delete a node in a singly linked list, you need the node before it. You cannot remove a node directly — you must wire its predecessor's next to skip over it. So the real goal is: find the predecessor of the middle node.

Edge Cases:

  • Single node (n=1): The middle IS the head — return None
  • Two nodes (n=2): ⌊2/2⌋ = index 1, delete the tail — return just the head
  • Very long list: Still O(n) time — the pointer race scales linearly
2 The Algorithm: Fast-Slow Pointer Race ▾

The naive approach is two passes: count the length n, then walk n/2 steps to the predecessor and delete. But we can combine both into one pass using two pointers at different speeds.

Think of it as a race: if fast runs at 2× the speed of slow, when fast reaches the finish line (the end of the list), slow has covered exactly half the distance — arriving at the middle. A third pointer, prev, trails one step behind slow, always ready to perform the deletion.

1 Handle n=1: If head.next is None, the list has one node. Its middle is itself — return None (the list becomes empty).
2 Initialize pointers: slow = head (index 0), fast = head.next (index 1 — one step ahead; this offsets slow's landing position to match ⌊n/2⌋), prev = head (starts same as slow, trails by 1 after first loop body).
3 Run the race: While fast != None and fast.next != None: save prev = slow, then advance slow = slow.next (one step) and fast = fast.next.next (two steps).
4 Delete the middle: When the race ends, slow points at the middle node and prev is its predecessor. Execute prev.next = slow.next — this rewires prev to skip over slow entirely.
5 Return head: The list head is unchanged (we only removed an interior or tail node). Return it.
🔑 Why fast starts at head.next, not head

LeetCode defines middle as index ⌊n/2⌋ (the second middle for even n). If both pointers start at head, slow lands on ⌊(n-1)/2⌋ — the first middle. Starting fast one step ahead shifts the race endpoint, making slow land at ⌊n/2⌋ instead. The initialization is the on/off switch for which middle you find.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every iteration: fast is at position 2k+1 and slow is at position k (where k = number of iterations completed, 0-indexed from the start, with fast offset by 1). When fast can no longer advance 2 steps (fast is None or fast.next is None), slow is at the middle.

Trace for n=5 (nodes at positions 0,1,2,3,4):

  • Init: slow=0, fast=1, prev=0
  • Iter 1: prev=0, slow=1, fast=3 — fast.next=4 ≠ None, continue
  • Iter 2: prev=1, slow=2, fast=5 (None) — loop exits
  • slow=2 = ⌊5/2⌋ ✓   prev=1 is the predecessor ✓

💡 Correct Initialization for This Problem

The most robust approach: use slow = head and fast = head, then enter the loop with while fast.next and fast.next.next (instead of fast and fast.next). This makes slow land at ⌊n/2⌋ correctly for all n. The key: the condition checks two steps ahead of fast. After the loop, slow is the predecessor to the middle, so deletion is slow.next = slow.next.next — no separate prev needed! This is the cleanest solution for this exact problem.

Why prev is still useful: The fast/slow version with prev is intuitive for teaching and generalizes to "delete the middle in a stream where you need the predecessor." Both approaches are O(n) / O(1).

4 The Code (Annotated) ▾

Solution 1 — Elegant One-Pointer Deletion (Optimal) ★ Interview Pick

Use fast.next and fast.next.next as conditions — when the loop ends, slow IS the predecessor to the middle.

def deleteMiddle(head):
if not head or not head.next: ← n=0 or n=1: no middle to delete
return None
slow = fast = head ← both start at head (position 0)
while fast.next and fast.next.next: ← loop while fast can see 2 more nodes
slow = slow.next ← slow advances 1
fast = fast.next.next ← fast advances 2
slow.next = slow.next.next ← slow is now the predecessor; skip its next
return head ← head unchanged; list modified in-place
⚠️ Why "fast.next and fast.next.next" instead of "fast and fast.next"?

The condition fast.next and fast.next.next means "can fast take 2 more hops?" When the loop exits, fast is at the second-to-last node (odd n) or last node (even n). Either way, slow is positioned at the node before the middle — exactly where we need it for deletion via slow.next = slow.next.next.

Solution 2 — Explicit prev pointer (more readable for teaching)

def deleteMiddle(head):
if not head.next: return None ← single node: return empty list
prev, slow, fast = None, head, head ← prev trails 1 behind slow
while fast and fast.next:
prev = slow ← capture predecessor BEFORE advancing slow
slow = slow.next ← slow moves 1
fast = fast.next.next ← fast moves 2
prev.next = slow.next ← prev disconnects slow from the list
return head

Solution 3 — Two-Pass Count (clearest logic, 2× traversal)

def deleteMiddle(head):
n, curr = 0, head
while curr: n += 1; curr = curr.next ← pass 1: count nodes
if n == 1: return None
curr = head
for _ in range(n // 2 - 1): ← walk to predecessor of middle
curr = curr.next
curr.next = curr.next.next ← delete the middle
return head
💡 Complexity Summary

Fast-Slow (Solution 1 & 2): Time O(n), Space O(1) — single pass, 2-3 pointer variables
Two-Pass Count (Solution 3): Time O(n), Space O(1) — two traversals, simpler loop logic
Interview Pick: Solution 1 for elegance; Solution 3 when asked to prove correctness explicitly

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Clarify the Exact Middle Definition

Say: "For even-length lists, is the middle the first or second middle node?" The problem says ⌊n/2⌋ — the second. This detail matters for the pointer initialization. Asking it shows precision.

Step 2: Propose Two-Pass as Brute Force

"Simple approach: count the length n, then walk n//2 - 1 steps from head to reach the predecessor of the middle. Delete it. O(n), O(1), two passes. Correct and easy to verify."

Step 3: Introduce Fast-Slow as One-Pass Optimization

"We can find the predecessor in a single pass using fast-slow pointers. Fast moves at 2× speed. When fast can't take 2 more steps, slow is at the predecessor. I'll use the condition fast.next and fast.next.next."

Step 4: Trace Through Two Examples

Always trace n=5 (odd) and n=2 (edge case). For n=2: slow=0, fast=0; fast.next=1, fast.next.next=None → loop doesn't run → slow.next=slow.next.next → deletes index 1. Correct!

🗣️ What to Say to the Interviewer

7 Summary & Key Takeaways ▾
To delete a node in a singly linked list, you need its predecessor — not the node itself
Fast-slow pointers find the middle in one pass; condition "fast.next and fast.next.next" stops slow at the predecessor
Starting both pointers at head with "while fast.next and fast.next.next" targets ⌊n/2⌋ (LeetCode's definition)
Single-node edge case: return None immediately — the whole list is the "middle"
Fast-slow is a universal linked-list technique: cycle detection, middle finding, kth-from-end, palindrome check
