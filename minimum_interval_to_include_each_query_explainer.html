LeetCode #1851

Minimum Interval to Include Each Query

For each query, find the smallest interval that contains it. Size = right − left + 1.

Hard Advanced Data Structures Sort + Min-Heap Time: O((n+q) log n) Space: O(n+q)
1 Understanding the Problem ▾

You are given n intervals [[l1,r1], [l2,r2], ...] and q integer queries. For each query value x, find the smallest interval (by size = r − l + 1) that contains x — meaning l ≤ x ≤ r. Return an array of answers, one per query, or −1 if no interval contains the query.

💡 Key Insight

If we sort queries and sweep intervals left to right, we can maintain a running min-heap of "currently active" intervals. As the query value increases, we add intervals whose left endpoint has been reached, and lazily evict intervals whose right endpoint has been passed. The heap top always gives the smallest valid interval in O(log n).

Walk-through example: intervals = [[1,4],[2,4],[3,6],[4,4]], queries = [2,3,4,5]

✓ Correct Answers

q=2: intervals [1,4](sz4),[2,4](sz3) cover it → min sz = 3
q=3: [1,4],[2,4],[3,6] cover it → min sz = 3
q=4: all cover it; [4,4] is tiny → min sz = 1
q=5: only [3,6] covers it → min sz = 4

✗ Brute Force (TLE)

For each query scan all n intervals.
O(n × q) = up to 1010 operations.
Fails for n, q = 105.

Constraints: 1 ≤ n, q ≤ 105. Values in [1, 107]. Multiple intervals may overlap. Queries are answered in original order even if processed sorted.

Edge cases: query lies outside all intervals (→ −1); multiple intervals with same minimum size (any is fine); all queries covered by one giant interval; single-point intervals like [4,4].

2 The Algorithm: Offline Sweep with Min-Heap ▾

This is the classic offline interval query pattern: sort both intervals and queries so you can process them in a single left-to-right sweep, maintaining a heap of "active" intervals.

1 Sort intervals by left endpoint. As the query value sweeps right, we add intervals whose start is ≤ current query.
2 Sort queries ascending, keeping original indices. Processing queries in order allows incremental interval addition — we never need to "go back".
3 Min-heap keyed by interval size (r−l+1), storing (size, right_endpoint). The smallest covering interval is always at the top of this heap.
4 For each sorted query: (a) push all intervals with l ≤ query; (b) pop expired intervals (r < query) from heap top; (c) heap[0][0] is the answer, or −1 if empty.
5 Write answer using original index. ans[orig_idx] = result so the final array is in the original query order without extra sorting.
🔐 Why Lazy Deletion?

We could proactively remove expired intervals whenever we advance past their right endpoint — but that would require a separate data structure. Instead, we leave them in the heap and simply skip them when they reach the top. Since each interval is pushed and popped at most once, total heap work is O(n log n) across all queries.

Why min-heap keyed by size, not by right endpoint? We want the smallest interval. If we keyed by right endpoint (earliest ending), we'd get the interval that expires soonest — not the smallest. Keying by size means the answer is always at the top after eviction.

3 Why It Works: The Invariant ▾
🔐 The Invariant

When answering query q, the min-heap contains all intervals with l ≤ q, plus possibly some expired intervals (r < q) that haven't been evicted yet. After lazy eviction, every remaining interval satisfies both l ≤ q AND r ≥ q — i.e., it genuinely contains q.

Why adding by left endpoint is correct: Queries are processed in ascending order. Any interval added while answering a previous smaller query q' still satisfies l ≤ q' ≤ q, so it's still a candidate. We never miss an interval.

Why evicting by right endpoint is safe: An interval with r < q has its right end strictly to the left of q. Since l ≤ r < q, the query value q is outside the interval — it cannot be a valid answer. Discarding it never costs us a correct answer.

Why heap top is optimal: After eviction, all remaining intervals genuinely cover q. The min-heap returns the one with smallest size in O(log n).

⚠️ The Index-Restoration Trick

We sort queries to enable the sweep, but must return answers in the original order. The fix is simple: use enumerate(queries) before sorting, so each query carries its original index. Write ans[orig_idx] = result directly. No extra sorting step needed.

Edge case — all expired: If every interval in the heap has r < q, we pop them all. The heap becomes empty. if heap is False, so we leave ans[orig_idx] = -1 (the default). Correct!

⚠️ Most Common Mistake: Wrong Heap Key

Pushing (r, l, r) instead of (r-l+1, r) makes the heap return the interval ending earliest — not the smallest. For query 4 with intervals [1,4](size 4) and [4,4](size 1), an r-keyed heap returns [1,4] first since both have r=4 but [1,4] was pushed first. Always key by the quantity you want minimized — here, size.

📐 Complexity

Time: O((n+q) log n) — Sorting intervals: O(n log n). Sorting queries: O(q log q). Each of n intervals pushed and popped at most once: O(n log n). Each query: O(log n) heap ops. Overall: O((n+q) log n).

Space: O(n+q) — The heap holds at most n intervals. The sorted queries array is O(q). Answer array is O(q).

Brute force alternative for comparison: for each query, scan all n intervals, track minimum size. Time: O(n × q). For n=q=105, that's 1010 operations — TLE.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Pattern

Signals: "for each of Q queries, find the best interval containing it." This is many-to-many matching. Brute force is O(n×q). The canonical optimization: offline processing + sweep line + heap. When you see "interval + queries + minimize something," think: sort both by value, sweep, heap.

Step 2: Decide What to Sort

Sort intervals by left endpoint — so we add them incrementally as the query value grows. Sort queries ascending with original indices — so we process them left-to-right, adding intervals without revisiting. This is the "offline queries" technique: we sacrifice real-time answering for algorithmic efficiency.

Step 3: Design the Heap Entry

We want the minimum-size interval. So heap key = r - l + 1. We also need the right endpoint to test expiry. Entry: (size, right). Lazy deletion: don't evict proactively — only when a stale entry surfaces at the top during a query. This gives amortized O(n log n) across all evictions.

Step 4: Preserve Original Query Order

Use enumerate(queries) before sorting. Each tuple is (orig_index, value). After sorting by value and getting an answer, write ans[orig_index] = answer. The final array is already in the right order — no extra pass needed.

🗣️ What to Say to the Interviewer

"Brute force is O(n×q) — for each query, scan all intervals. With n,q=10^5 that's 10^10 ops, way too slow. Key insight: if I process queries in sorted order, I can sweep intervals left-to-right and maintain a heap of 'active' intervals — ones whose left endpoint is ≤ current query. I sort intervals by start, sort queries with their original indices, and use a min-heap keyed by interval size. For each query: push all new intervals (l ≤ q), lazily pop expired ones (r < q). Heap top is the answer. Each interval is pushed and popped at most once, so total time is O((n+q) log n)."
💡 Common Follow-Up Questions
  • What if queries are online (arriving in real time)? — Need an interval tree or segment tree; offline processing no longer possible.
  • What if you need the interval itself, not just its size? — Add l to the heap entry: (size, r, l).
  • What if multiple intervals tie on size? — Return any; the heap returns one arbitrarily.
  • Can you reduce space? — No: we need O(n) for the heap and O(q) for the answer array.

7 Summary & Key Takeaways ▾
Sort intervals by left endpoint so you can add them incrementally as the sweep moves right.
Sort queries ascending (keeping original indices) to enable the sweep; restore order via index mapping.
Min-heap keyed by size (r−l+1) gives O(log n) access to the current best covering interval.
Lazy deletion: leave expired intervals in the heap; only pop them when they surface at the top during a query. Each interval is pushed/popped at most once.
Overall O((n+q) log n) — competitive with any comparison-based interval query algorithm.
🎯 When to Use This Pattern
  • "For each query, find the best interval covering it" — offline sweep + heap
  • Both n and q are large (up to 105) — brute force O(nq) is too slow
  • Queries can be processed offline (not streaming) — sort both arrays
  • Lazy deletion is acceptable — no need to remove intervals proactively
