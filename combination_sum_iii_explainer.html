LeetCode #216

Combination Sum III

Find all valid combinations of exactly k numbers from 1–9 that sum to n, using each digit at most once.

Medium Backtracking K Numbers Summing to N Time: O(C(9,k)·k) Space: O(k)
1 Understanding the Problem ▾

You have the digits 1 through 9, each available exactly once. Your task: pick exactly k of them that add up to n. Return every such group. Order within a group doesn't matter — [1,2,4] and [4,2,1] are the same.

✓ k=3, n=7 → [[1,2,4]]

[1,2,4] → 1+2+4 = 7 ✓
[1,3,3] — repeats 3 ✗
[2,2,3] — repeats 2 ✗
Only one valid answer.

✗ k=3, n=9 → 3 answers

[1,2,6] → 9 ✓
[1,3,5] → 9 ✓
[2,3,4] → 9 ✓
[1,4,4] — repeats 4 ✗

💡 Key Insight

The search space is tiny: at most 9 digits, choose exactly k. The maximum number of candidates is C(9,5) = 126 (the worst case). We never need to consider more than 126 combinations — backtracking with pruning visits only valid paths, making this vastly more efficient than naive enumeration.

Edge Cases:

  • If n is too large (e.g., k=2, n=18 — max is 8+9=17), return [].
  • If n is too small (e.g., k=3, n=6 — min is 1+2+3=6), exactly one answer exists.
  • Digits 1–9 used at most once each — no repetition allowed.
  • Each combination is a set — no duplicates in the output.
2 The Algorithm: Backtracking with Pruning ▾

Backtracking is systematic trial-and-error with early abandonment. We build combinations digit-by-digit, and the moment a partial path is doomed, we abandon it and try something else — like erasing a wrong turn in a maze rather than walking all the way to a dead end.

1 Define the recursive signature: backtrack(start, combo, remaining) — start is the smallest digit we can still use, combo is digits chosen so far, remaining is how much more sum we need.
2 Base case — success: If len(combo) == k and remaining == 0, we have a valid combination. Copy it into results and return.
3 Prune — too many numbers: If len(combo) == k but remaining ≠ 0, we've picked the right count but wrong sum. Dead end — return immediately.
4 Prune — overshoot: While looping candidates d from start to 9, if d > remaining, break. Since d only grows, all future digits also overshoot — no need to check them.
5 Choose → Explore → Unchoose: Append d, recurse with start=d+1 (never reuse), then pop d. This three-step rhythm is the heart of every backtracking solution.
backtrack(start, combo, remaining): ┌─ BASE CASE: len==k AND remaining==0 → record combo[:] ├─ PRUNE 1: len==k (but remaining≠0) → return (wrong sum) └─ LOOP d from start to 9: ├─ PRUNE 2: d > remaining → break (overshoot, break not continue) ├─ CHOOSE: combo.append(d) ├─ EXPLORE: backtrack(d+1, combo, remaining-d) └─ UNCHOOSE: combo.pop()
🔑 Why Iterate in Increasing Order?

By always choosing digits ≥ start and passing d+1 as the new start, combinations are automatically generated in sorted order. This means [1,2,4] and [4,2,1] cannot both appear — each unique set is produced exactly once, with zero deduplication overhead.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every recursive call, combo is a strictly increasing sequence of distinct digits from [1,9], with len(combo) < k and sum(combo) < n. This invariant eliminates duplicates, prevents reuse, and ensures every explored path is potentially reachable to a solution.

Why strict increase means no duplicates: If we always pick the next digit strictly greater than the last, every combination we build is sorted. Two different sorted sequences of the same elements are identical — so we can never produce the same combination twice.

Why break (not continue) when d > remaining: We iterate d in increasing order — 1, 2, 3, ... If the current d already overshoots remaining, then d+1, d+2, ... all also overshoot (they're even bigger). Continuing the loop is pointless; breaking exits immediately.

⚠️ The Most Common Mistake

Writing continue instead of break when the digit overshoots. This doesn't cause wrong answers (those branches also fail), but it wastes work checking d+1 through 9 unnecessarily. In an interview, using break demonstrates you understand the ordering property — it's a signal of mastery.

Pruning power: Without pruning, we'd explore 2^9 = 512 subsets. With the count-limit prune, we only explore subsets of size ≤ k. With the overshoot prune, large digits are skipped early. The result: we visit at most O(C(9,k)) paths — as few as 9 paths for k=1 or k=8.

4 The Code (Annotated) ▾

Solution 1 — Backtracking (Optimal, Interview Pick)

def combinationSum3(k: int, n: int) -> list[list[int]]:
result = [] ← collects all valid combinations
def backtrack(start, combo, remaining):
if len(combo) == k and remaining == 0: ← exactly k numbers, sum == n
result.append(combo[:]) ← copy! combo is mutated, not stable
return
if len(combo) == k: ← k numbers but wrong sum → prune
return
for d in range(start, 10): ← try each digit from start to 9
if d > remaining: ← d overshoots, all larger d also overshoot
break ← break (not continue!) — ordering property
combo.append(d) ← CHOOSE: add d to current combination
backtrack(d + 1, combo, remaining - d) ← EXPLORE: d+1 ensures no reuse
combo.pop() ← UNCHOOSE: undo choice, try next d
backtrack(1, [], n) ← start from 1, empty combo, target n
return result
⚠️ Why combo[:] and not combo?

The combo list is a single mutable object shared across all recursive calls. combo.pop() undoes additions after recursion. If you write result.append(combo), you're storing a reference to the same list — by the time the function returns, combo is empty again, and all your "results" point to []. Always copy with combo[:] or list(combo).

Solution 2 — Brute Force via itertools (Pythonic, non-interview)

from itertools import combinations
def combinationSum3(k: int, n: int) -> list[list[int]]:
return [list(c) for c in combinations(range(1, 10), k) ← enumerate all C(9,k) subsets
if sum(c) == n] ← filter: keep only those summing to n
💡 Complexity Summary

Backtracking: Time O(C(9,k) · k) — at most C(9,k) leaves, each costs O(k) to copy. Space O(k) for the recursion call stack (depth = k).
Brute Force: Same time asymptotically, but no pruning — explores every valid-length subset before filtering. Space O(C(9,k) · k) if you store all combos.
In practice: C(9,k) ≤ 126 for all valid k. Both approaches are effectively O(1) given the tight constraints.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Pattern Signal

Keywords: "find all combinations", "exactly k elements", "each used at most once." These signal exhaustive enumeration — backtracking, not greedy or DP. Greedy has no clear local optimum here; DP gives counts, not actual combinations.

Step 2: Define the State

Ask: "What changes between recursive calls?" (1) Which digit to start from next — the start parameter prevents reuse and duplicates. (2) The current partial combination. (3) How much remaining sum we need. These three define the full state.

Step 3: State Base Cases and Prunes Explicitly

Base: len == k AND remaining == 0 → valid answer. Prune 1: len == k but remaining ≠ 0 → wrong sum, stop. Prune 2: d > remaining → break (future digits only larger). Stating these clearly shows structured thinking.

Step 4: Name the Three-Step Template

Every backtracking solution follows: Choose (append d), Explore (recurse), Unchoose (pop d). Naming this template demonstrates pattern mastery and makes your code predictable to the interviewer.

🗣️ What to Say to the Interviewer

"This screams backtracking — we need all combinations, not just one or a count. I'll recurse with three state variables: start digit, current combo, remaining sum needed. Base case: combo length equals k and remaining is zero — record a copy. Prune 1: if we have k numbers but wrong sum, return immediately. Prune 2: if current digit exceeds remaining, break the loop — digits only increase. The start parameter ensures digits are always picked in increasing order, which naturally prevents duplicates without needing a seen-set. Time: O(C(9,k)*k), Space: O(k) for the recursion stack."
🔮 Likely Follow-up Questions
  • "Allow digit reuse?" → Change backtrack(d+1, ...) to backtrack(d, ...) — same digit can be picked again (Combination Sum I, #39)
  • "Arbitrary candidates instead of 1–9?" → Sort candidates first, same template, prune on overshoot
  • "Just count, don't list combinations?" → DP: dp[j][s] = ways to pick j numbers summing to s, without storing the actual combos
  • "Candidates may repeat (like [1,1,2])?" → Sort + skip duplicates at same recursion depth (Combination Sum II, #40)

7 Summary & Key Takeaways ▾
Backtracking = exhaustive search with early abandonment; use when you need ALL valid solutions
The "start" parameter + increasing iteration = free deduplication, no visited-set needed
Use break (not continue) when digit exceeds remaining — ordering makes all future digits worse too
Always copy combo with combo[:] before recording — lists are mutable, references will betray you
Search space is C(9,k) ≤ 126 — this is effectively a constant-time problem under given constraints
