LeetCode #826

Most Profit Assigning Work

Assign jobs to workers to maximize total profit — each worker takes at most one job, but multiple workers can do the same job type.

Medium Greedy Sort + Two Pointers Time: O(n log n + m log m) Space: O(n)
1 Understanding the Problem ▾

You have n jobs. Each job has a difficulty rating and a profit. You also have m workers, each with an ability score. A worker can do a job only if their ability ≥ job difficulty. Each worker does at most one job. Multiple workers can do the same job type. Maximize the total profit across all workers.

💡 Key Insight

Since multiple workers can share the same job, each worker's choice is independent. Every worker should simply take the highest-profit job they are capable of. Sorting both arrays turns this into a clean two-pointer sweep where the best available job only gets better as worker ability increases.

Example: difficulty=[2,4,6,8], profit=[10,30,20,40], workers=[3,5,9]

worker 3 → best job diff≤3: profit 10
worker 5 → best job diff≤5: profit 30
worker 9 → best job diff≤9: profit 40
Total = 10 + 30 + 40 = 80

Running max handles this!

Edge cases: worker ability below all job difficulties → contributes 0. Jobs with identical difficulty but different profit → keep the higher one. All workers with same ability. Single job, single worker. All profits equal.

2 The Algorithm: Greedy Sort + Two Pointers ▾

Sort both arrays so we can sweep workers from weakest to strongest, expanding the set of available jobs as ability grows. Track the running maximum profit seen so far — that's always the best a worker can earn.

1 Pair and sort jobs: Zip difficulty and profit into (diff, profit) tuples, then sort ascending by difficulty. Now jobs[0] is the easiest job.
2 Sort workers: Sort the worker array ascending by ability. Process weakest workers first — each subsequent worker can only access more jobs, never fewer.
3 Sweep with two pointers: Maintain job pointer j and best_profit. For each worker (ability a): advance j while jobs[j].difficulty ≤ a, updating best_profit. Add best_profit to total.
4 Return total: Every worker has been greedily assigned the best job they qualify for. The sum is the answer.
🔑 The Greedy Observation

Once both arrays are sorted, the job pointer j only ever moves right — never resets. For each new (stronger) worker, we simply extend the set of feasible jobs and update the running maximum. This one-pass sweep is O(n + m) after sorting.

3 Why It Works: The Invariant ▾
🔐 The Invariant

After the inner while loop runs for a worker with ability a, best_profit equals the maximum profit among all jobs with difficulty ≤ a. This is maintained for every worker in sorted order.

Why sorting workers guarantees correctness: Workers are processed weakest-to-strongest. Worker i+1 has ability ≥ worker i. Any job available to worker i is also available to worker i+1 — so best_profit can only stay the same or increase. We never need to "go back" in the jobs array.

Why the greedy is optimal here (not just heuristic): Since any number of workers can do the same job, there is no resource conflict. Worker A taking job X does not prevent worker B from also taking job X. Therefore, each worker's choice is fully independent — the globally optimal solution IS giving each worker their individual best. No backtracking or DP needed.

⚠️ When This Greedy Would Fail

If each job could only be assigned to one worker, greedy would fail — you'd need to carefully trade off who gets which job. That's a bipartite matching problem. The key constraint here is "no limit on workers per job," which makes greedy globally optimal.

Running max handles the profit/difficulty ordering mismatch: Harder jobs don't necessarily pay more. A job with difficulty 6 might pay less than one with difficulty 4. By keeping a running maximum (not just looking at the current job), we ensure workers always get the best available deal regardless of job ordering by difficulty.

4 The Code (Annotated) ▾

Solution 1 — Greedy: Sort + Two Pointers (Optimal, Interview Pick)

def maxProfitAssignment(difficulty, profit, worker):
jobs = sorted(zip(difficulty, profit)) ← pair (diff, profit), sort by difficulty asc
worker.sort() ← sort workers by ability asc (weakest first)
total = best_profit = j = 0 ← answer, running max profit, job pointer
for ability in worker: ← process workers weakest → strongest
while j < len(jobs) and jobs[j][0] <= ability:← unlock all jobs this worker can do
best_profit = max(best_profit, jobs[j][1]) ← keep running max (profit may not be monotone)
j += 1 ← advance job pointer (NEVER resets)
total += best_profit ← this worker earns the best available profit
return total
⚠️ Why does j never reset to 0?

Workers are processed in order of increasing ability. Since each new worker is at least as strong as the previous, all jobs already scanned (difficulty ≤ previous ability) remain available. best_profit already holds their maximum profit. Re-scanning them would waste time without changing the result.

Solution 2 — Brute Force: Nested Loops (O(n × m), for reference)

def maxProfitAssignment(difficulty, profit, worker):
total = 0
for a in worker: ← for each worker...
best = 0
for d, p in zip(difficulty, profit): ← scan EVERY job (redundant across workers)
if d <= a: ← worker qualifies
best = max(best, p) ← take if higher profit
total += best ← O(n×m) total — TLE for n,m ≈ 10⁴
return total
💡 Complexity Summary

Greedy (Optimal): Time O(n log n + m log m) for sorting; O(n + m) sweep; Space O(n) for job pairs
Brute Force: Time O(n × m), Space O(1) — correct but TLE when n, m ≈ 10⁴

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Notice Independent Assignments

Read the constraint carefully: "There is no limit on how many workers can be assigned to the same job." This means no resource contention — each worker is fully independent. Immediately think: greedy per worker.

Step 2: State the Brute Force

"For each worker, scan all jobs and pick the max-profit one they can do. O(n × m). Too slow — let's optimize." This frames the problem clearly before improving.

Step 3: Discover the Monotonicity

"If I sort workers by ability, each successive worker can access at least as many jobs as the previous. So as I scan workers left to right, I only ever expand the job set — never shrink it. That's monotone, which means I can use a two-pointer with a running max."

Step 4: Note the Running Max Trick

"Sorting jobs by difficulty doesn't sort them by profit — a harder job might pay less. So I must track best_profit = max over all unlocked jobs, not just the most recently unlocked one."

🗣️ What to Say to the Interviewer

"Since multiple workers can do the same job, there's no conflict — each worker independently picks the best job they qualify for.
Brute force: for each worker scan all jobs — O(n × m). Optimizable with sorting.
Sort jobs by difficulty, sort workers by ability. Use a pointer j into jobs.
For each worker (weakest to strongest): advance j while jobs[j].diff ≤ ability, tracking best_profit = running max.
Add best_profit to total. j never resets — it only moves right (monotone). O(n log n + m log m)."
🔮 Likely Follow-up Questions
  • "What if each job can only go to one worker?" → Bipartite matching / greedy with sorting by profit-to-difficulty ratio
  • "What if workers have skill AND stamina attributes?" → Sort on one dimension, binary search or sweep on the other
  • "How does this change with a cap on total jobs?" → Knapsack-style DP approach
  • "Binary search alternative?" → Build prefix max array over sorted jobs, binary search for each worker's max feasible difficulty index

7 Summary & Key Takeaways ▾
No job scarcity → each worker independently picks best available job → greedy is globally optimal
Sort both arrays so the job pointer only ever moves right — O(n+m) sweep after sorting
Track running max profit, not just the latest job's profit — harder jobs can pay less
The inner while loop's total iterations across all workers is O(n) — amortized constant per worker
Total time O(n log n + m log m), space O(n) for the zipped job pairs

🎯 When to Recognize This Pattern
  • "Assign each agent the best item within their threshold" → sort both, two-pointer sweep
  • "Multiple agents can use the same resource" → assignments are independent → greedy
  • Sorted input with a monotone expanding feasibility window → running max/min + single sweep
  • "Maximize total by pairing from two sorted arrays" → sort + two-pointer or binary search
