LeetCode #729

My Calendar I

Implement a calendar that cannot have double-bookings. Return true if a new event can be added without causing an overlap, false otherwise.

Medium Trees / BST Interval Check Time: O(n log n) avg Space: O(n)
1 Understanding the Problem ▾

You are building a simple calendar. Each event occupies a half-open interval [start, end) — it includes the start time but not the end time. A double-booking occurs when two events share any time in common.

💡 Key Insight: When Do Two Intervals Overlap?

Two intervals [s1, e1) and [s2, e2) overlap if and only if s1 < e2 AND s2 < e1. Equivalently, they do NOT overlap when one ends before the other starts: e1 <= s2 OR e2 <= s1. This overlap test is the core primitive for this entire problem.

✓ No Overlap: [10,20) then [20,30)

[====10..20====)──────────
──────────[====20..30====)
e1=20 <= s2=20 → Adjacent OK → True

✗ Overlap: [10,20) then [15,25)

[====10..20====)──────────
─────[====15..25====)─────
15 < 20 AND 10 < 25 → Overlap → False

Edge Cases: Back-to-back events where end equals next start are not overlapping (half-open intervals). Events arriving in any order are handled by the BST since comparisons are on values, not insertion order. Empty calendar always returns True on first booking.

🔑 What "Half-Open" Means

[10, 20) covers minutes 10 through 19 but NOT minute 20. So [10, 20) and [20, 30) are adjacent — they share no time. The strict inequality e1 <= s2 captures this boundary exactly: if e1 equals s2, it's still not an overlap.

2 The Algorithm: BST Interval Traversal ▾

The challenge is: given a new interval [start, end), quickly determine if it overlaps any already-booked interval, then store it if safe. The key insight is to keep intervals sorted by start time so we can navigate directly to potential conflicts.

🔑 The BST Navigation Rules

At each BST node with interval [ns, ne) and new interval [s, e):
• If e <= ns: new event ends before node starts → go left (no overlap here)
• If s >= ne: new event starts after node ends → go right (no overlap here)
• Otherwise: OVERLAP DETECTED → return False

1 Start at root: If BST is empty, insert the new event as root and return True. The first booking always succeeds.
2 At each node, apply the three-way decision: Does new event end before node? → go left. Does new event start after node ends? → go right. Otherwise → overlap found, return False.
3 Reached a null leaf? There is a valid slot here — insert the new interval as a new BST node. Return True.
4 Propagate upward: The recursive insert returns the (potentially updated) node so the BST structure is maintained through reassignment node.left = insert(node.left, s, e).
💡 Why Only Two Neighbors Can Conflict (SortedList view)

If we keep intervals sorted by start, for new interval [s, e), find its position i in the sorted list. Only the predecessor at i-1 and successor at i can possibly overlap — anything earlier ends before the predecessor, anything later starts after the successor. Two checks suffice.

3 Why It Works: The Invariant ▾
🔐 The BST Ordering Invariant

At any node N with interval [ns, ne): all intervals in N's left subtree have start time < ns, and all in the right subtree have start time ≥ ns. This ordering means: when we decide to go left (because e <= ns), we know no right-subtree interval could start before our event ends — so no right-subtree conflict is possible.

Why we can prune entire subtrees: If e <= ns, we go left. Every right-subtree interval has start ≥ ns ≥ e, meaning they all start at or after our new event ends — guaranteed no overlap. Similarly if s >= ne, we go right and all left-subtree intervals end ≤ ne ≤ s — guaranteed no overlap. The BST ordering lets us safely ignore half the tree at each step.

🔄 Overlap Condition (the universal test)
[s1, e1) overlaps [s2, e2) ⟺ s1 < e2 AND s2 < e1

BST go-left: e <= node.start → safe, no overlap possible on this path
BST go-right: s >= node.end → safe, no overlap possible on this path
BST conflict: everything else → s < node.end AND node.start < e → OVERLAP
⚠️ Worst Case: Sorted Input Creates Unbalanced Tree

If all bookings arrive in sorted order (10, 20, 30, …), each new interval goes to the right child of the last, creating a linked list — O(n) per operation. For guaranteed O(log n), use Python's SortedList from sortedcontainers which maintains balance automatically.

4 The Code (Annotated) ▾

Solution 1 — Manual BST (Interview Pick — shows understanding)

class Node: ← BST node stores one booked interval
def __init__(self, s, e):
self.start, self.end = s, e
self.left = self.right = None ← Children start null
class MyCalendar:
def __init__(self):
self.root = None ← Empty calendar to start
def book(self, start: int, end: int) -> bool:
result = self._insert(self.root, start, end)
if result is None: return False ← None signals a conflict was found
self.root = result ← Update root (needed for first insert)
return True
def _insert(self, node, s, e):
if node is None: ← Found an empty slot → insert here
return Node(s, e)
if e <= node.start: ← New event ends before node starts → go left
node.left = self._insert(node.left, s, e)
if node.left is None: return None ← Propagate conflict upward
elif s >= node.end: ← New event starts at/after node ends → go right
node.right = self._insert(node.right, s, e)
if node.right is None: return None ← Propagate conflict upward
else: ← Neither condition → OVERLAP DETECTED
return None ← Signal conflict (do not insert)
return node ← Return node to rebuild BST path
⚠️ Why e <= node.start (not <)?

The intervals are half-open [start, end). If e == node.start, our event ends exactly where the node starts — they are adjacent, not overlapping. Using <= correctly places adjacent events in the left subtree as non-conflicting. Similarly s >= node.end handles the symmetric case.

Solution 2 — SortedList (Practical, guaranteed O(log n))

from sortedcontainers import SortedList ← Always sorted; bisect in O(log n)
class MyCalendar:
def __init__(self):
self.cal = SortedList()
def book(self, start, end):
i = self.cal.bisect_left((start, end)) ← Position in sorted order
if i > 0 and self.cal[i-1][1] > start:← Predecessor ends after our start → conflict
return False
if i < len(self.cal) and self.cal[i][0] < end:← Successor starts before our end → conflict
return False
self.cal.add((start, end))
return True

Solution 3 — Brute Force (Simplest, propose first in interviews)

class MyCalendar:
def __init__(self): self.events = []
def book(self, start, end):
for s, e in self.events: ← Check every stored event — O(n)
if s < end and start < e: ← Universal overlap test
return False
self.events.append((start, end))
return True
💡 Complexity Summary

Brute Force: Time O(n) per call, Space O(n)
Manual BST: Time O(log n) avg / O(n) worst (unbalanced), Space O(n)
SortedList: Time O(log n) guaranteed, Space O(n)
For n calls total: Brute Force O(n²), BST O(n log n) avg, SortedList O(n log n) always.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Nail the Overlap Condition First

Before touching data structures, state clearly: "[s1,e1) and [s2,e2) overlap iff s1 < e2 AND s2 < e1." Verify on the board with [10,20) vs [20,30) (adjacent, no overlap → e1=20 not < s2=20, FALSE → correct). This shows rigor.

Step 2: Propose Brute Force

"Store events in a plain list. For each book(start, end): scan all stored events and apply the overlap test. O(n) per call. For 1000 calls that's 10⁶ operations — workable but can we do better?"

Step 3: Identify the Optimization

"If we keep events sorted by start time, a new [s,e) can only conflict with events that start before e. Among those, only the one with the largest start (the predecessor) could overlap from the left side. And only the successor (smallest start ≥ s) could overlap from the right. Two checks instead of n."

Step 4: Implement the BST

"A BST naturally maintains sorted order. At each node: if new event ends before node starts, go left; if new event starts after node ends, go right; otherwise conflict. This is a clean three-way split that I can code recursively."

Step 5: Mention the Trade-off

"The manual BST is O(log n) average but O(n) worst if input is sorted. Python's SortedList solves this with a balanced structure. In production I'd use SortedList; in this interview I'll implement BST to show understanding."

🗣️ What to Say to the Interviewer

"Overlap check: [s1,e1) and [s2,e2) conflict iff s1 < e2 AND s2 < e1.
Brute force: store list of events, scan all on each book(). O(n) per call.
To optimize: maintain a BST sorted by start. At each node, go left if e<=node.start,
go right if s>=node.end, else return False (overlap). O(log n) avg per call.
Worst case for BST: sorted input → O(n). SortedList from sortedcontainers fixes this."
