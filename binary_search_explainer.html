LeetCode #704

Binary Search

Given an integer array nums sorted in ascending order and an integer target, return the index of target, or -1 if it is not present. Must run in O(log n) time.

Easy Binary Search Classic lo, hi, mid Time: O(log n) Space: O(1)
1 Understanding the Problem ▾

You have a sorted array of distinct integers and a target value. Your job is to find the target's index — or report -1 if absent. The key constraint is O(log n) time, which immediately rules out a simple left-to-right scan.

💡 Key Insight

Because the array is sorted, checking any single element immediately tells you which half of the remaining candidates the target could be in — if it exists at all. Each comparison lets you discard half the remaining search space. After just 20 comparisons you can search 1,000,000 elements.

✓ Sorted — Binary Search works

nums = [-1, 0, 3, 5, 9, 12]
target = 9
Probe index 2 (val=3): too small → go right
Probe index 4 (val=9): MATCH → return 4

✗ Unsorted — Binary Search breaks

nums = [9, -1, 3, 5, 0, 12]
target = 9
Probe mid (val=5) — is target left or right?
→ Can't tell → algorithm invalid!

Edge Cases to consider: empty array (return -1 immediately), single-element array (check that one element), target smaller than nums[0] (hi shrinks below lo quickly), target larger than nums[-1] (lo grows past hi quickly), target equals nums[0] or nums[-1] (boundary correctness).

🔐 The Core Guarantee (Invariant)

If the target exists in the array, it is always somewhere inside [lo, hi] at the start of every iteration. The algorithm never eliminates a position that could contain the target. This invariant is what makes binary search provably correct.

2 The Algorithm: Classic lo / hi / mid ▾

We maintain two index pointers — lo (left boundary, purple) and hi (right boundary, amber) — that bracket the region where the target could still be. Each step we probe the midpoint and tighten the window.

1 Initialize: lo = 0, hi = len(nums) - 1. Target, if it exists, must be somewhere in nums[0 .. n-1]. Both endpoints are inclusive.
2 Loop while lo ≤ hi: When lo == hi there is still one candidate to check. When lo > hi the search space is empty — no candidates remain. The ≤ (not <) handles the single-element case.
3 Compute mid = lo + (hi - lo) // 2: The floor midpoint of the current window. We use this form (not (lo+hi)//2) to avoid integer overflow in 32-bit languages like Java and C++. In Python both are equivalent.
4 Three-way compare nums[mid] vs target:
• nums[mid] == target → found, return mid
• nums[mid] < target → target is strictly to the right → lo = mid + 1
• nums[mid] > target → target is strictly to the left → hi = mid - 1
5 Return -1: Loop exited with lo > hi — the entire array was searched and target was never found.

Why mid + 1 and mid - 1 (not just mid)? We already know nums[mid] is not the target. Re-including mid in the next window wastes a step and — critically — when lo == hi causes an infinite loop because the window never shrinks.

⚠️ The Most Common Bug: Infinite Loop

Writing lo = mid or hi = mid causes an infinite loop when lo == hi: mid computes to lo (== hi), the condition is false, nothing moves, forever. The ±1 is not optional — it ensures strict progress each iteration.

3 Why It Works: The Invariant ▾
🔐 The Search Space Invariant

At the start of every iteration: if target exists in nums, it must be at some index in [lo, hi]. No update to lo or hi ever excludes a position where the target could be hiding.

Proof by induction on the invariant:

  • Base case (initialization): lo=0, hi=n−1 covers the entire array. If target is anywhere in nums, it's in [0, n−1]. Trivially true.
  • Inductive step — case nums[mid] < target: Array is sorted. All indices ≤ mid have values ≤ nums[mid] < target. So target (if it exists) is at some index in [mid+1, hi]. Setting lo = mid+1 preserves the invariant.
  • Inductive step — case nums[mid] > target: Symmetrically, all indices ≥ mid have values ≥ nums[mid] > target. Target (if present) must be in [lo, mid−1]. Setting hi = mid−1 preserves the invariant.
  • Termination: Each step either finds the target or strictly shrinks [lo, hi] by at least 1. When lo > hi, the search space is empty — if target existed, the invariant would have directed us to it; since we never found it, it's absent.

Search space halves every step (n = 8 example)

Step 0: [0..7] — 8 candidates
Step 1: 4 candidates remain
Step 2: 2 candidates remain
Step 3: 1 candidate — found or return -1
💡 Why Exactly O(log n)?

After k iterations the search window has size at most ⌈n / 2k⌉. The loop exits when this drops below 1, so k ≤ log₂(n). In practice: n=1 → 0 iterations; n=2 → 1; n=1,000,000 → at most 20. Compare this to linear search's worst-case n=1,000,000 comparisons.

4 The Code (Annotated) ▾

Solution 1 — Iterative Binary Search (Optimal, Interview Pick)

def search(nums: list[int], target: int) -> int:
lo, hi = 0, len(nums) - 1 ← Both endpoints inclusive; target in [lo, hi] at all times
while lo <= hi: ← '≤' not '<': lo==hi is still 1 valid candidate
mid = lo + (hi - lo) // 2 ← Overflow-safe floor midpoint of [lo, hi]
if nums[mid] == target: ← Hit!
return mid
elif nums[mid] < target: ← mid and everything left of it is too small
lo = mid + 1 ← Eliminate [lo..mid]; search [mid+1..hi]
else: ← nums[mid] > target → everything right is too large
hi = mid - 1 ← Eliminate [mid..hi]; search [lo..mid-1]
return -1 ← lo > hi: empty window, target absent
⚠️ Why lo + (hi - lo) // 2 and not (lo + hi) // 2?

In Python integers are arbitrary-precision, so overflow is not a concern here. But in Java/C++ with 32-bit ints, lo + hi overflows when both exceed ~1 billion. The form lo + (hi - lo) // 2 is always safe since hi - lo is never larger than the array length. Building this habit in Python will save you in language-agnostic interviews.

Solution 2 — Brute Force Linear Scan (for comparison)

def search_linear(nums: list[int], target: int) -> int:
for i, v in enumerate(nums): ← Check every element left-to-right; O(n)
if v == target:
return i ← Found at index i
return -1 ← Scanned entire array, not found

Solution 3 — Recursive Binary Search (elegant but O(log n) space)

def search_recursive(nums, target, lo=0, hi=None):
if hi is None: hi = len(nums) - 1 ← Default hi on first call
if lo > hi: return -1 ← Base case: empty search space
mid = lo + (hi - lo) // 2
if nums[mid] == target: return mid
if nums[mid] < target: return search_recursive(nums, target, mid+1, hi)
else: return search_recursive(nums, target, lo, mid-1)
💡 Complexity Summary

Linear Scan: Time O(n), Space O(1) — doesn't exploit sorted order
Iterative Binary Search: Time O(log n), Space O(1) — optimal; use in interviews
Recursive Binary Search: Time O(log n), Space O(log n) — call stack depth; correct but iterative preferred

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Signal

"Sorted array" + "find element" + "O(log n) time" — this is the exact fingerprint of Binary Search. State this immediately: "The sorted property lets me eliminate half the candidates per step, which is O(log n)." Don't propose linear scan as your opening idea.

Step 2: State Your Invariant

Say out loud: "I'll maintain lo and hi such that if target exists, it's always within [lo, hi]. I start with lo=0, hi=n-1 covering the full array." Stating the invariant proves you understand correctness, not just the template.

Step 3: Nail the Off-by-One Details

Say explicitly: "I use while lo ≤ hi, not <, because when lo equals hi there's still one candidate. I set lo = mid+1 and hi = mid-1 — not mid — to exclude the already-checked element and guarantee the window strictly shrinks."

Step 4: Handle Edge Cases Verbally

Walk through edge cases before coding: "Empty array: lo=0, hi=-1, loop never runs → -1. Target < nums[0]: hi immediately shrinks below lo. Target > nums[-1]: lo grows past hi. All handled automatically by the invariant."

🗣️ What to Say to the Interviewer

"The sorted array lets me do binary search — O(log n).
I'll track a search window [lo, hi]. Invariant: target, if present, is always in this window.
At each step: compute mid, compare nums[mid] to target.
If equal → return mid. If mid too small → target is right, lo = mid + 1.
If mid too large → target is left, hi = mid − 1.
When lo > hi the window is empty → return −1. No target found."

7 Summary & Key Takeaways ▾
Binary Search requires sorted input — without it you can't make elimination decisions
Each step eliminates half the remaining candidates → O(log n) total comparisons
Use while lo ≤ hi (not <) — lo == hi is still one valid candidate to check
Always set lo = mid + 1 and hi = mid - 1 — including mid again risks infinite loops
Use lo + (hi - lo) // 2 for overflow safety in 32-bit integer languages
The invariant — "if target exists, it's always in [lo, hi]" — is what makes the algorithm provably correct
