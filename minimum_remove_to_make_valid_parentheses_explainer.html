LeetCode #1249

Minimum Remove to Make Valid Parentheses

Remove the minimum number of parentheses so the string is valid. Return any valid result.

Medium Stacks Stack Index Tracking Time: O(n) Space: O(n)
1 Understanding the Problem ▾

A parentheses string is valid when every ( has a matching ) to its right, and every ) has a matching ( to its left, with no crossing. Letters in the string are always fine — they never affect validity.

The problem gives us a string that may contain unmatched parentheses — either a ( that never gets closed, or a ) that arrives before any open bracket. We must remove the fewest possible characters to achieve validity. Any valid minimum result is accepted.

💡 Key Insight

We only ever need to remove parentheses — never letters. There are exactly two kinds of "bad" parentheses: unmatched ) (a close with no preceding open) and unmatched ( (an open with no following close). Every other character survives into the result.

✓ Valid: "a(b(c)d)"

Outer '(' at 1 ↔ ')' at 7
Inner '(' at 3 ↔ ')' at 5
All letters pass through
→ No removal needed

✗ Invalid: "a(b)c)d("

'(' at 1 ↔ ')' at 3 ✓ (matched)
')' at 5 has no preceding '(' ✗
'(' at 7 has no following ')' ✗
→ Remove idx 5 and idx 7

Edge Cases: empty string → return ""; all letters → return as-is; all unmatched opens "(((" → remove all; already valid → return unchanged; deeply nested like "((()))" → all matched, no change.

2 The Algorithm: Stack Index Tracking ▾

The core idea: scan the string once, using a stack of indices (not characters) to track which opening parentheses are still unmatched. When we see a ), we try to match it with the most recent unmatched ( — if one exists, pop it (matched pair, keep both). If the stack is empty, this ) is unmatched — mark it for removal. After scanning, any index still on the stack is an unmatched ( — mark those too. Build the result by skipping marked positions.

1 Initialize: stack = [] (indices of pending opens) and remove = set() (indices to delete).
2 Scan left-to-right: For each (i, c). Letters: do nothing. '(': push i onto stack. ')': proceed to step 3.
3 Match or mark: If stack is non-empty, stack.pop() — the most recent unmatched ( is now matched. If stack is empty, remove.add(i) — no available open for this ).
4 Drain stack: remove.update(stack) — any index still on stack is an unmatched (, add all to remove.
5 Build result: ''.join(c for i, c in enumerate(s) if i not in remove) — one pass, O(1) set lookup per character.
🔑 Why Track Indices, Not Just Count?

If we only track a count of unmatched parens, we know how many to remove but not which ones. In "((a)", removing the first or second ( both give valid results — either is fine. But for correctness we need specific positions to skip during output construction. The stack gives us exactly that: the indices of unmatched opens at the end are the specific chars we must drop.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At any point during the left-to-right scan, the stack contains exactly the indices of all opening parentheses that have been seen but not yet matched. The remove set contains exactly the indices of all closing parentheses that have been seen and found unmatched. Together, these two sets precisely capture every "broken" parenthesis found so far.

Why greedy matching is optimal: When we encounter a ) and the stack has an available (, we should always match them. The alternative — refusing to match — would leave both the ( and this ) unmatched, requiring us to remove both later. Greedy matching never increases the count of removals; it can only decrease it or keep it the same.

Why leftover stack items must be removed: Any index remaining in the stack after the full scan is an opening parenthesis that found no closing partner anywhere in the string. There is no future character that can match it — the scan is done. It cannot be part of any valid pair, so it must be removed. This is both necessary and sufficient.

Minimality: We mark an index for removal only when it's genuinely unmatched. We never mark a matched pair. Every unmatched parenthesis contributes exactly one index to the remove set. Since a string with k unmatched parentheses requires removing at least k characters, and we remove exactly k, the solution is optimal.

⚠️ Common Trap: Modifying the String In-Place

Strings are immutable in Python. Even in languages where you can modify in-place, deleting characters one-by-one from a string shifts all subsequent characters — that's O(n) per deletion → O(n²) total for k deletions. The correct approach: accumulate removal indices in a set, then build the output in a single O(n) pass.

⚠️ Why a set for remove, not a list?

The final pass checks i not in remove for every index — n times total. With a list, each membership check is O(n) → O(n²) overall. With a set, each check is O(1) → O(n) overall. Always use a set for repeated membership testing against a fixed collection.

Solution 2 — Two-Pass Counter (O(n) time, no stack/set overhead)

def minRemoveToMakeValid(s: str) -> str:
# Pass 1: left-to-right — remove unmatched ')'
open_cnt = 0; res = []
for c in s:
if c == '(': open_cnt += 1; res.append(c) ← Pending open; keep it for now
elif c == ')':
if open_cnt: open_cnt -= 1; res.append(c) ← Match found
# else: unmatched ')' — simply skip it
else: res.append(c)
# Pass 2: right-to-left — remove open_cnt unmatched '('
ans = []
for c in reversed(res):
if c == '(' and open_cnt > 0: open_cnt -= 1 ← Skip excess opens
else: ans.append(c)
return ''.join(reversed(ans))
💡 Complexity Summary

Stack Index Tracking: Time O(n), Space O(n) — stack + set each at most n elements
Two-Pass Counter: Time O(n), Space O(n) — same output array; avoids explicit stack/set
Interview recommendation: Use Stack Index Tracking — it demonstrates the stack invariant clearly and is easier to explain and verify for correctness in real time.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify the Pattern Signal

Keywords: "parentheses," "valid," "remove minimum." These three together almost always mean stack. The LIFO property of a stack naturally models nested bracket structure — the last-opened bracket must be the first-closed.

Step 2: Enumerate What Makes a Paren "Bad"

Two distinct cases: (a) ) with no preceding unmatched ( — detectable instantly during left-to-right scan. (b) ( with no following ) — detectable only after the entire string is scanned. Both need removal.

Step 3: Decide to Track Indices

The problem asks to produce the result string, not just a count. We need to know which characters to keep. Push indices onto the stack (not the ( character itself), so the leftover stack indices tell us exactly which characters to skip.

Step 4: Build Output Efficiently

Collect all indices to remove in a set. Then build result with a generator comprehension: skip any index in the set, include all others. Total time: O(n) for scan + O(n) for build = O(n).

🗣️ What to Say to the Interviewer

"I'll use a stack to track indices of unmatched opening parens.
Scanning left to right: '(' → push its index. ')' → if stack is non-empty, pop (matched pair, keep both); if stack is empty, this ')' is unmatched, so I add its index to a remove set.
After the scan, any index still in the stack is an unmatched '(' — I add those to remove too.
Finally, I build the result string, skipping any index in the remove set.
Time O(n), Space O(n). This gives minimum removals because we only mark truly unmatched characters."
🔮 Likely Follow-up Questions
  • "O(1) extra space?" → Two-pass counter (no stack/set); output array is unavoidable
  • "Also '[]' and '{}'?" → Push (char, idx) pair; on )/]/} pop only if top matches expected open bracket
  • "Minimum additions (not removals)?" → LeetCode #921 — count unmatched using same stack logic
  • "All possible minimum-removal results?" → LeetCode #301 — BFS/DFS over removal subsets (much harder)
