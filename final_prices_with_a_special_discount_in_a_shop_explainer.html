LeetCode #1475

Final Prices With a Special Discount in a Shop

For each item, find the first item to its right with price ≤ its price and subtract it as a discount. Return the array of final prices.

Easy Monotonic Stack Next Smaller Element Time: O(n) Space: O(n)
1 Understanding the Problem ▾

You're in a shop with items priced prices[0], prices[1], ..., prices[n-1]. Each item has a special discount rule: scan the items to the right of item i, and the first item j where prices[j] ≤ prices[i] gives a discount equal to prices[j]. If no such j exists, no discount.

In other words, result[i] = prices[i] - (first prices[j] where j > i and prices[j] ≤ prices[i]), or just prices[i] if no such j exists.

💡 Key Insight

This is the classic "Next Smaller or Equal Element" problem applied to every index. For each position, we need the nearest right neighbor whose value is ≤ the current value. A monotonic (decreasing) stack answers all these queries simultaneously in one O(n) left-to-right pass — no nested loops needed.

Edge cases: Single element (no discount possible). All strictly increasing (no discounts at all). All strictly decreasing (every item except the last is discounted by the next). All equal values (each item is discounted by the next — equal prices qualify). Empty array (return empty).

2 The Algorithm: Monotonic Stack (Next Smaller or Equal) ▾

A monotonic stack is a stack that maintains elements in a sorted order (either always increasing or always decreasing). Here we use a non-increasing stack: items on the stack have prices in non-increasing order from bottom to top. When we see a price that is smaller than or equal to the top, we know the current item IS the discount for the top — pop and record it.

1 Initialize: Copy prices into result (default: no discount applied). Create an empty stack that stores indices of items waiting for a discount.
2 Scan left to right: For each index i, ask: "Can prices[i] serve as a discount for any items currently on the stack?"
3 Pop and apply: While the stack is non-empty AND prices[stack[-1]] ≥ prices[i], pop index j and set result[j] = prices[j] - prices[i]. Item i is the first smaller-or-equal right neighbor of item j.
4 Push current index: After all eligible pops, push i onto the stack. This item hasn't found its discount yet and will wait.
5 No discount for remainder: After the full scan, any indices still on the stack found no smaller-or-equal right neighbor. They keep their original price (already correct from step 1).
🔑 Why This is O(n), Not O(n²)

Each index is pushed onto the stack exactly once and popped at most once. So across the entire algorithm, the total number of push + pop operations is at most 2n. Even though the while-loop may pop multiple items in one iteration of the outer for-loop, the amortized cost per element is O(1). Total time: O(n).

3 Why It Works: The Invariant ▾
🔐 The Stack Invariant

At any moment during the scan, every index on the stack is a candidate that has not yet found its first smaller-or-equal right neighbor. When a new element arrives, it can only be a discount-giver for elements whose price is ≥ it. By scanning left-to-right and maintaining this property, we guarantee that the first time an element gets a discount assigned is from the nearest valid right neighbor.

Why popping gives the first (not any) discount: When item i is processed, the items it can discount are exactly those on the stack with prices ≥ prices[i]. Because we scan left-to-right and maintain the ordering, the top of the stack is always the most recently pushed index — meaning it's the closest index to i that still needs a discount. When we pop that index, we've correctly found its first valid right neighbor (item i), because if there had been a closer one, it would have already popped this index earlier.

Why equal prices are included: The problem says a discount applies when prices[j] ≤ prices[i]. "Less than or equal" means same-priced items to the right also count. If we used strict less-than only, we'd miss valid discounts for equal-priced items.

⚠️ Common Mistake: Using > Instead of >=

The pop condition must be prices[stack[-1]] ≥ prices[i] (include equals). Using only > would fail on inputs like [5, 5, 5], where the first 5 should get a discount of 5 (from the second 5), giving result [0, 0, 5]. With strict >, no pops happen and you'd incorrectly return [5, 5, 5].

Why this is called "Next Smaller Element": This is a well-known algorithmic pattern. The standard version finds the next strictly smaller element. This problem is a slight variant (next smaller-or-equal), but the stack direction and structure are identical. The pattern appears in: Daily Temperatures (next warmer day), Stock Span (previous larger day), Histogram (largest rectangle), and many more.

4 The Code (Annotated) ▾

Solution 1 — Monotonic Stack (Optimal — Interview Pick)

def finalPrices(prices: list[int]) -> list[int]:
result = prices[:] ← Copy: unfound discounts stay as original price
stack = [] ← Stack of indices waiting for a discount-giver
for i, price in enumerate(prices): ← i=current index, price=prices[i]
while stack and prices[stack[-1]] >= price: ← Current is discount for stack top?
j = stack.pop() ← Pop the index that found its discount
result[j] = prices[j] - price ← Apply: original_price - first_smaller_right
stack.append(i) ← Push current — still looking for its discount
return result ← Remaining stack entries keep original price (no discount)
⚠️ Why push indices, not values?

We store indices in the stack because when a discount is found, we need to update result[j] — we need the position, not just the price. If we stored values, we'd lose the index. Always push indices in monotonic stack problems and look up values via prices[stack[-1]]. This is a universal pattern.

Solution 2 — Brute Force O(n²)

def finalPrices_brute(prices: list[int]) -> list[int]:
n = len(prices)
result = prices[:]
for i in range(n): ← For each item
for j in range(i + 1, n): ← Scan every element to its right
if prices[j] <= prices[i]: ← First valid discount?
result[i] -= prices[j] ← Apply discount
break ← Only first valid discount counts
return result
💡 Complexity Summary

Monotonic Stack: Time O(n) — each index pushed once and popped at most once. Space O(n) worst case (all prices strictly increasing, nothing gets popped).
Brute Force: Time O(n²), Space O(1) extra. Use to verify your stack solution on small inputs.
Interview tip: Brute force first to show correctness, then optimize. The monotonic stack is the expected O(n) solution.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Pattern from Problem Language

Key phrase: "for each item, find the first item to its right that satisfies a comparison condition." This is a next-element query for all positions — the strongest signal for a monotonic stack. If you see "first to the right / left" with a comparison, think stack immediately.

Step 2: Start with Brute Force

"I'd start with nested loops: for each i, scan j from i+1 and stop at the first valid j. O(n²) time, O(1) space. Correct but too slow for large inputs." This shows you can solve it correctly before optimizing.

Step 3: Explain the Stack Insight

"The key observation: as I scan left-to-right, each new element might be the answer to previous elements. A monotonic stack lets me efficiently track which previous elements are still looking for their 'first smaller' neighbor. When I find an answer, I pop and record it."

Step 4: Handle the Edge Case Explicitly

Call out the ≥ vs > distinction: "The discount applies when prices[j] is less than or equal — so equal prices count. My pop condition needs to be prices[stack[-1]] ≥ prices[i], not strictly greater."

🗣️ What to Say to the Interviewer

"This is a 'next smaller or equal element' problem — a classic monotonic stack use case.
Brute force: O(n²) nested loops, find first valid j for each i.
Optimal: scan left-to-right with a decreasing stack of indices.
For each price, pop stack entries with price greater or equal — they found their discount.
Set result[j] = prices[j] - prices[i] for each popped j, then push i.
Remaining stack entries at the end keep original price (no discount found).
Time: O(n) amortized — each index pushed and popped at most once.
Space: O(n) for the stack."
🔮 Likely Follow-up Questions
  • "What if we want the last (not first) smaller element?" → Scan right-to-left, or collect all valid j's and take the rightmost
  • "What if we want strictly smaller (not ≤)?" → Change condition to prices[stack[-1]] > prices[i]
  • "Can you do this in-place?" → Yes: modify prices[j] directly during pop, no separate result array
  • "Daily Temperatures — is this the same?" → Yes! Next warmer day = next greater element, same stack structure

7 Summary & Key Takeaways ▾
A monotonic decreasing stack answers all "next smaller element" queries in one O(n) pass
Always push indices (not values) — you need the position to update the result array
Each index is pushed once and popped at most once — O(n) amortized even with the while loop
Use ≥ (not >) in the pop condition — equal prices count as valid discounts here
Indices left on stack at the end had no valid right neighbor — keep their original price
