LeetCode #827

Making A Large Island

You are given an n×n binary grid. You are allowed to change at most one 0 to 1. Return the size of the largest island after applying this operation.

Hard Graph DFS Label Islands + Try Flip Time: O(n²) Space: O(n²)
1 Understanding the Problem ▾

You have an n×n binary grid of 0s (sea) and 1s (land). Cells connected in 4 directions (up/down/left/right) form islands. You may flip exactly one 0 to a 1 — or flip none if the grid is already all 1s. Return the size of the largest island you can create.

💡 Key Insight

Flipping a 0 to 1 doesn't add just one cell — it acts as a bridge that can merge several nearby islands into one. If a sea cell at (r, c) has neighbors from three distinct islands of sizes 5, 3, and 2, flipping it yields an island of size 5+3+2+1=11. The challenge is computing this quickly for every candidate 0-cell.

Edge cases to consider:

  • Grid is all 1s — answer is n² (the entire grid is one island; no flip needed)
  • Grid is all 0s — answer is 1 (flip any single cell)
  • A 0-cell whose multiple neighbors belong to the same island — must not double-count
  • 1×1 grid with a single 1 — answer is 1
2 The Algorithm: Label Islands + Try Flip ▾

Think of this like a land surveyor. First, they stamp every connected parcel with a lot number and record its total area. Then, they stand on each empty field and ask: "If I develop this plot, which adjacent lots do I connect, and what's the combined acreage?" The key is doing the surveying once and looking up results O(1) per query.

1 Start label counter at 2. Grid values are 0 or 1 — using labels 2, 3, 4, … avoids any ambiguity. island_size = {} will map label → cell count.
2 Phase 1 — label every island via DFS: For each cell with value 1 (unlabeled land), flood-fill with the current label (overwriting grid cells in-place), count cells, store in island_size[label], increment label.
3 Initialize result = max(island_size.values(), default=0). This handles the all-ones case: if no 0 exists, Phase 2 never updates result, so we need the correct baseline.
4 Phase 2 — try each 0-cell (r, c): Collect a set of unique island labels from its ≤ 4 neighbors. Sum their sizes + 1 (for the flipped cell). Update result.
5 Return result. This is the maximum island size achievable with at most one flip.
⚠️ Critical: Use a Set for Neighbor Labels

An island can wrap around the corner of a 0-cell, making two of its four neighbors carry the same label. If you just sum neighbor values without deduplication, you count that island's area twice. Always collect neighbor labels into a set first, then sum island_size[l] for l in seen.

3 Why It Works: The Invariant ▾
🔐 The Core Invariant

After Phase 1: every land cell holds a label ≥ 2, and island_size[label] equals the exact number of cells in that island. This is a static, correct snapshot — it never changes during Phase 2.

Why the size sum is always correct when we flip (r, c):

Flipping grid[r][c] from 0 to 1 creates a new island consisting of: the single flipped cell + every island reachable from it via a 4-directional step. Because islands are disjoint by definition (DFS labeling ensures each cell belongs to at most one island), their sizes add without overlap. The set-of-labels deduplication is the only subtlety: two neighbors pointing to the same label is not two separate contributions — it's the same island approached from two sides.

Why Phase 1 overwrites the grid with labels:

We need a way to ask "which island does neighbor (nr, nc) belong to?" in O(1). Storing the answer directly in the grid cell — overwriting 1 with its island's label — is the most space-efficient approach. After labeling, a cell value of 0 still means sea; any value ≥ 2 means "island with this label." No additional data structure is needed.

💡 Why Starting Labels at 2 Matters

If we used label=1 for the first island, we'd be writing 1 into cells that already contained 1. The outer scan in Phase 1 checks if grid[r][c] == 1 to find unlabeled land. If the first island gets label=1, its cells look identical to unlabeled cells — the scan might re-enter them. Starting at 2 (or any value > 1) keeps the three states distinct: 0=sea, 1=unlabeled-land, ≥2=labeled-island.

Handling the all-ones edge case: When every cell is 1, the outer loop in Phase 2 (if grid[r][c] == 0) finds nothing, so it never updates result. Without the initialization result = max(island_size.values()), we'd return 0. The initialization captures the pre-existing largest island, giving the correct answer without any flip.

4 The Code (Annotated) ▾
def largestIsland(grid): n×n binary grid, modified in-place
n = len(grid) grid is always square (n×n)
island_size = {} label (int ≥ 2) → cell count
label = 2 # labels start above 0 and 1 avoids collision with grid values
def dfs(r, c, lbl): flood-fill from (r,c), marking cells lbl
if r < 0 or r >= n or c < 0 or c >= n: bounds guard
return 0
if grid[r][c] != 1: return 0 0=sea or ≥2=already labeled → skip
grid[r][c] = lbl mark visited + store island id in cell
return (1 + dfs(r+1,c,lbl) + dfs(r-1,c,lbl) count this cell + recurse all 4 dirs
+ dfs(r,c+1,lbl) + dfs(r,c-1,lbl))
# ── Phase 1: label every connected island ──
for r in range(n):
for c in range(n):
if grid[r][c] == 1: found unlabeled land
island_size[label] = dfs(r, c, label) DFS floods island, returns size
label += 1 next island gets next integer label
# ── Phase 2: try flipping each sea cell ──
result = max(island_size.values(), default=0) baseline: best existing island (all-1s case)
for r in range(n):
for c in range(n):
if grid[r][c] == 0: candidate flip cell
seen = set() unique island labels among neighbors
for nr,nc in [(r+1,c),(r-1,c), check all 4 neighbors
(r,c+1),(r,c-1)]:
if 0 <= nr < n and 0 <= nc < n: bounds check
if grid[nr][nc] > 1: neighbor is labeled land
seen.add(grid[nr][nc]) set deduplicates same-island neighbors
sz = 1 + sum(island_size[l] for l in seen)+1 for the flipped cell
result = max(result, sz) update global maximum
return result largest achievable island
⚠️ Common Bug: Forgetting the Set

The most frequent mistake is writing sum(island_size[grid[nr][nc]] for ...) without deduplication. Consider a U-shaped island: its two ends are both neighbors of the cell at the bottom of the U. You'd add that island's size twice, inflating the answer. The fix is one line: seen = set().

🔎 Complexity Analysis

Time O(n²): Phase 1 — each cell visited at most once by DFS (marking prevents re-visits). Phase 2 — each cell visited once, 4-neighbor check is O(1). Both phases linear in grid size.

Space O(n²): island_size has at most O(n²) entries (trivially, since there are n² cells). Recursion stack depth up to O(n²) in worst case (all-1 grid = one giant snake-island). Grid is modified in-place so no extra n×n array.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Start with Brute Force and Name the Problem

The naive approach: for each 0-cell, flip it, run BFS/DFS to find the resulting island size, track max, undo the flip. Time: O(n²) cells × O(n²) BFS each = O(n⁴). For n=500 that's 62.5 billion operations — clearly TLE. State this immediately to show you think about complexity first.

Step 2: Name the Repeated Work

Why is brute force slow? Because every flip re-explores the same islands that don't change between flips. This is the key insight: island sizes are static. If we precompute them, we pay the DFS cost once instead of O(n²) times.

Step 3: Design the Labeling Scheme

To look up "which island does this neighbor belong to?", we need a cell → island-ID mapping. The elegant trick: overwrite grid values with integer labels ≥ 2 during DFS. The grid itself becomes our lookup table. No extra array needed.

Step 4: Anticipate the Deduplication Edge Case

Before the interviewer has to point it out, bring up: "When trying a 0-cell, I'll collect neighbor labels into a set — because the same island can appear as multiple neighbors if it wraps around the 0-cell. A set prevents double-counting."

Step 5: Mention the All-Ones Edge Case

Initialize result to the max value in island_size before Phase 2. This gracefully handles grids with no 0s — Phase 2 never triggers, and you return the largest pre-existing island.

🗣️ What to Say to the Interviewer

"Let me start with brute force: for every 0, flip it, BFS for island size, track max. That's O(n^4) — too slow for n=500. The bottleneck is re-running BFS for every candidate flip. But island sizes don't change between flips, so I can precompute them once. Here's my plan: Phase 1 — DFS over the grid. Each time I find an unlabeled 1-cell, I flood-fill it with a unique label (starting at 2, above grid values), count cells, store in island_size[label]. Overwrites the grid in-place. Phase 2 — For each 0-cell, check its 4 neighbors. Collect unique island labels in a set (to avoid double-counting a wrapping island), sum their sizes plus 1, update result. Edge case: no 0s — initialize result to max(island_size.values()). Time: O(n^2). Space: O(n^2) for the dict and recursion stack."
💡 Follow-Up Questions to Expect
  • What if we can flip k zeros? — BFS/Union-Find becomes necessary; label trick still useful for Phase 1.
  • Can you avoid mutating the grid? — Yes: use a separate label_grid[n][n] array for labels.
  • What if 8-directional adjacency? — Expand neighbor list to 8 directions; algorithm is identical.
  • Stack overflow concern for large grids? — Convert DFS to iterative BFS (use a deque). Same O(n²) complexity.

7 Summary & Key Takeaways ▾
Two-phase approach: label once with DFS (O(n²)), then query each 0-cell in O(1) per neighbor — total O(n²).
Labels start at 2 to distinguish from grid values 0 and 1 — the grid itself becomes the O(1) island-lookup table.
Set of neighbor labels prevents double-counting when one island borders the same 0-cell from multiple directions.
Initialize result = max existing island size to handle all-ones grids where no flip is needed.
Can adapt to iterative BFS for DFS to avoid recursion stack overflow on very large all-land grids.
