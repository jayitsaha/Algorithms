LeetCode #757

Set Intersection Size At Least Two

An integer interval [a, b] represents the set of integers from a to b. Find the minimum set S of integers such that for every interval in a list, at least two integers from S lie in the interval.

Hard Greedy Sort by End Time: O(n log n) Space: O(n)
1 Understanding the Problem ▾

We have a list of integer intervals — e.g., [1,3], [2,5], [4,6]. We need to build the smallest possible set S of integers such that every interval contains at least 2 members of S.

Think of it as: we're placing "pins" on a number line, and every interval must catch at least 2 pins. We want to minimize total pins used.

💡 Key Insight

The constraint "at least 2" (not "at least 1") makes this fundamentally harder than the classic interval piercing problem. A single element only satisfies half the requirement for each interval. Crucially, an interval must contribute 2 distinct integers — you can't count the same number twice.

✗ S = {2} — Fails

Interval [1,3]: 2 ∈ S ✓ but only 1 element
Need AT LEAST 2 per interval.
This set does not satisfy the problem.

✓ S = {2, 3} — Works

[1,3]: {2,3} both in range → 2 elements ✓
[2,5]: {2,3} both in range → 2 elements ✓
[4,6]: only {none} — STILL FAILS!

Worked example: For intervals [[1,3],[3,7],[8,9]], the minimum set is {1,2,3,8,9} with size 5. Each interval gets exactly 2 elements: [1,3] → {1,2} or {2,3}, [3,7] → {3, anything 4-7} — but we already have 3, so add a second. [8,9] → needs {8,9}. A clever greedy can do this in 5 total.

Edge Cases: A single-element interval [a,a] is impossible to satisfy (you can't fit 2 distinct integers in a range of size 1) — but the problem guarantees a ≤ b, so the minimum interval size is 2. An interval of size exactly 2, like [3,4], forces us to include both 3 and 4.

2 The Algorithm: Greedy — Sort by End, Place Rightmost ▾

This is a classic greedy interval coverage problem solved by sorting intervals by their right endpoint and greedily placing integers as far right as possible (within the interval) to maximize coverage of future intervals.

🔑 Why Sort by End?

An interval with an earlier right endpoint is "tighter" — it leaves fewer options for future intervals to share its coverage. By handling tight intervals first, we ensure their constraints are satisfied. Numbers we place near the right end of a tight interval are most likely to also fall inside later, wider intervals.

1 Sort by end, break ties by start descending. Sort intervals by right endpoint ascending. For ties in right endpoint, sort by left endpoint descending — this ensures a narrower interval (larger start) comes before a wider one with the same end, so we handle the harder constraint first.
2 Maintain a running "tail" — last two chosen integers. Track the two rightmost chosen integers as p and q (where q is the largest). These are the two numbers most likely to cover future intervals.
3 For each interval, count how many of {p, q} fall inside it. If both p and q are ≥ the interval's start, we need 0 new integers (the interval already has 2 elements from S). If only q is inside, we need 1 more. If neither, we need 2 new integers.
4 Add minimum needed integers as far right as possible. Place new integers at the rightmost positions of the current interval (end, end-1). This maximizes the chance they'll satisfy future intervals too — "greedy right placement."
5 Return total count. The size of S is the answer. We don't need to enumerate S explicitly — just count how many integers we added.
Decision Tree per Interval:
0 covered:
Neither p nor q is inside the interval. → Add 2 new integers (at end and end−1). New p=end−1, q=end.
1 covered:
Only q (the larger) is inside. → Add 1 new integer at end. New p=q, q=end.
2 covered:
Both p and q are inside. → Add 0 integers. S already covers this interval.
⚠️ Why Only q Could Be Inside, Not p?

Since intervals are sorted by end, and p < q, if p is inside the interval (p ≥ start), then q (which is larger) is also ≥ start. So the only asymmetric case is: q ≥ start but p < start — meaning only the larger element is covered. This happens when p "fell off" an earlier, narrower interval's left boundary.

3 Why It Works: The Invariant ▾
🔐 The Invariant

After processing each interval in sorted order, the two most recently added integers (q and p, with p < q) are the rightmost elements of S. Any future interval that starts at or before q already contains q — meaning it needs at most 1 more integer. Any future interval that starts at or before p already contains both p and q — meaning it is fully satisfied.

Why placing integers at the rightmost positions is optimal: When we must add a new integer for the current interval, we place it as far right as possible (at the interval's end). Any future interval [a, b] has its left boundary a ≥ a', the left boundary of the current interval (since we process in non-decreasing order of right endpoints). A right-placed integer is guaranteed to be reachable by more future intervals than a left-placed one.

Why sorting by end with ties broken by start descending works: Consider two intervals with the same end — say [1,5] and [3,5]. If we process [1,5] first, we'd place integers at 4,5. Then [3,5] is covered because 4≥3 and 5≥3. If we processed [3,5] first (narrower), we'd also place at 4,5 and [1,5] is still covered. However, if intervals were [1,5] and [4,5], placing at 4,5 covers both. The descending start tie-break ensures narrower intervals (harder constraints) come first, preventing a case where a wider interval's placement doesn't cover the narrower one.

💡 Proof Sketch

Assume an optimal solution O and our greedy solution G. G never places more integers than O in any prefix of intervals (exchange argument). If G adds an integer at position r (right end), and O uses position r' < r instead, we can swap r' for r in O — it still satisfies the current interval and is more likely to satisfy future ones. Hence G is optimal.

Edge: tight intervals [a, a+1]. A size-2 interval forces us to include both a and a+1 (there's no flexibility). The greedy places them at a+1 and a (end, end-1), which is exactly {a, a+1}. No alternative exists, so this is always globally optimal.

4 The Code (Annotated) ▾

Solution 1 — Greedy (Sort by End + Rightmost Placement) — Optimal, O(n log n)

def intersectionSizeTwo(intervals):
# Sort by end asc; for same end, sort by start desc ← Narrower intervals (larger start, same end) come first
intervals.sort(key=lambda x: (x[1], -x[0]))
p, q = -1, -1 # Two rightmost integers in S so far ← Sentinels: neither is inside any real interval initially
count = 0 ← Total size of S (our answer)
for a, b in intervals: ← Process each interval in sorted order
if p >= a: ← Both p and q are inside [a,b] (since p < q, if p>=a then q>=a too)
pass # Already satisfied — 0 new integers needed
elif q >= a: ← Only q is inside [a,b]; p < a (fell outside left boundary)
p = q; q = b ← Place 1 new integer at b (rightmost). Shift: old q becomes p.
count += 1
else: ← Neither p nor q is inside [a,b]. Need 2 new integers.
p = b - 1; q = b ← Place at b-1 and b (rightmost two positions in [a,b])
count += 2
return count ← Size of the minimum set S
⚠️ Common Mistake: Placing at Left Instead of Right

Beginners often place new integers at the leftmost positions (a, a+1). This is wrong! Placing at the RIGHT end (b-1, b) maximizes the chance these integers fall inside future intervals (which start at ≥ some value but end later). Left-placed integers are quickly "left behind" by narrower future intervals.

⚠️ Why -x[0] in Sort Key?

For intervals with the same right endpoint, we process narrower intervals (larger start = smaller range) first. Consider [1,5] and [3,5]: if we process [3,5] first and add integers at 4,5, then [1,5] gets both 4≥1 and 5≥1 — satisfied. If instead we process [1,5] first and add 4,5, then [3,5] gets 4≥3 and 5≥3 — also satisfied. In this example both orderings work. But for [4,5] and [1,5]: adding 4,5 covers [4,5] with p=4, q=5 and [1,5] with p=4≥1 — OK. The descending start tie-break guarantees safety in general by always handling the tightest constraint first.

Greedy (Optimal): Time O(n log n) for sort + O(n) for scan = O(n log n). Space O(1) extra (sort in-place).

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Greedy Interval Pattern

Key signals: "intervals," "minimum set," "coverage requirement." This looks like interval piercing (find minimum points that hit all intervals) — but with a twist: you need 2 per interval instead of 1. That twist changes the solution from a single-pass to a two-element tracking approach.

Step 2: Why Sort by End?

In all interval greedy problems, sorting by right endpoint is the key move. The interval that ends soonest is most "critical" — if we don't place integers there now, we'll be forced to use extra integers later. By handling early-ending intervals first, we make decisions that defer to future flexibility.

Step 3: Track Only 2 Values

"I don't need to track the entire set S — I only need to track the two rightmost integers, because only they can possibly satisfy future intervals. Older integers are too small to fall inside future intervals (future intervals start at higher values on average)."

Step 4: Three Cases for Each Interval

For each interval [a, b], check how many of {p, q} are ≥ a. If both: add 0. If only q: add 1 at b (update p=q, q=b). If neither: add 2 at b-1 and b (p=b-1, q=b). Always add at the rightmost position for maximum future coverage.

🗣️ What to Say to the Interviewer

"This is a greedy interval problem. I'll sort by right endpoint to process tightest intervals first.
I'll track p and q — the two largest integers in S so far.
For each interval [a, b]: if p ≥ a, both are inside — no new integers needed.
If only q ≥ a, I need one more — I place it at b (rightmost, for max future coverage).
If neither is inside, I need two — I place at b-1 and b.
Time O(n log n) from sort. Space O(1) extra. Result is total count added."
🔮 Likely Follow-up Questions
  • "What if k=1 instead of 2?" → Classic interval piercing: sort by end, track 1 rightmost. O(n log n).
  • "What if k=3 or arbitrary k?" → Track k rightmost integers. Add at b, b-1, ..., b-(needed-1).
  • "Return the actual set, not just the count?" → Replace count with a list; append the new integers.
  • "Minimum number of arrows to burst all balloons?" → Exact same pattern but k=1.

7 Summary & Key Takeaways ▾
Sort intervals by right endpoint (ascending) — tight intervals must be satisfied first
Track only p and q (two rightmost chosen integers) — these are the only ones that can help future intervals
Always place new integers at the rightmost positions (b, b-1) — maximizes future coverage
Three cases: 0 covered → add 2; 1 covered (q only) → add 1; both covered → add 0
Tie-break: same end → sort by start descending (handle narrower first)
