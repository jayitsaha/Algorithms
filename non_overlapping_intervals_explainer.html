LeetCode #435

Non-overlapping Intervals

Given an array of intervals, return the minimum number of intervals you need to remove to make the rest non-overlapping.

Medium Intervals Greedy · Sort by End Time: O(n log n) Space: O(1)
1 Understanding the Problem ▾

You're given a list of intervals on a number line. Some of them may overlap — they share a stretch of time. Your goal is to find the minimum number of intervals to delete so the rest are all non-overlapping.

💡 Key Insight — Flip the Question

Don't think "what is the minimum to remove?" Think instead: "what is the maximum number I can KEEP?" Because removals = total − kept, maximizing kept automatically minimizes removals. This reframe transforms a confusing minimization into a clean maximization — one solvable with a classic greedy trick.

✗ Input: may overlap

[1,3], [2,4], [3,5], [0,6]
[2,4] overlaps [1,3]
[0,6] overlaps everything
Must remove some.

✓ Output: max non-overlapping

Keep: [1,3], [3,5]
They share only endpoint 3
(touching = NOT overlapping)
Removals: 2 → Answer: 2

Overlap definition: Intervals A=[a1,a2] and B=[b1,b2] overlap if a1 < b2 AND b1 < a2. Touching at a single endpoint (e.g., [1,3] and [3,5]) does not count as overlapping.

Edge cases: empty array → 0; all identical intervals → n−1; already all non-overlapping → 0; single interval → 0.

2 The Algorithm: Greedy — Sort by End Time ▾

This problem is equivalent to the Activity Selection Problem — a classic greedy problem. The insight: always prefer the interval that ends earliest, because it leaves the most room for future intervals on the timeline.

1 Sort by end time: intervals.sort(key=lambda x: x[1]). After sorting we process intervals in order of when they finish — "earliest finisher first".
2 Initialize: count = 0 (removals so far) and prev_end = -infinity (no interval kept yet — anything can start here).
3 No overlap (start ≥ prev_end): The current interval starts at or after the last kept interval ended. Keep it! Set prev_end = current_end.
4 Overlap (start < prev_end): Conflict! We must remove one. Remove the current interval (which ends later, since we sorted by end). Increment count. Do NOT update prev_end.
5 Return count: Every removal we counted was the minimum necessary decision. The total is the answer.
🔑 Why Sort by End, NOT by Start?

Sorting by start fails. Example: [[0,10],[1,2],[2,3]] sorted by start keeps [0,10] first (it starts earliest), then rejects [1,2] and [2,3] — that's 2 removals. But the optimal is to remove only [0,10] and keep [1,2] and [2,3] — just 1 removal. Sorting by end greedily keeps the interval that "frees up" the most future space.

⚠️ Which Interval Do We Remove in a Conflict?

When two intervals overlap, we always remove the current one (which ends later in sorted order). We keep prev_end unchanged — the earlier-ending interval remains our boundary. Keeping the earlier-ending interval can never be worse for future choices: it frees up more of the timeline.

3 Why It Works: The Greedy Invariant ▾
🔐 The Invariant

At every step, prev_end holds the minimum possible end time achievable for the same number of kept intervals. In other words, the greedy selection up to this point is optimal: no other choice of which intervals to keep could yield a smaller prev_end with the same count.

Proof sketch (exchange argument): Suppose an optimal solution keeps interval X at some position, but the greedy algorithm keeps interval Y instead, where Y ends no later than X (Y.end ≤ X.end). We can swap X for Y in the optimal solution and the result remains valid — Y.end ≤ X.end means no future interval that was compatible with X becomes incompatible with Y. The swap preserves or improves the solution.

Why smaller prev_end is always better: A smaller prev_end means more future intervals on the timeline can start before the boundary. More future options = easier to avoid conflicts. Greedily picking the smallest possible prev_end at each step maximizes future flexibility.

⚠️ The >= vs > Bug — Most Common Mistake

Intervals that touch at a single point like [1,3] and [3,5] are not overlapping by this problem's definition. The non-overlap check must be start >= prev_end (greater-than-or-equal), NOT start > prev_end. Using strict inequality will incorrectly count touching intervals as overlapping, over-counting removals.

Equal end times: If two intervals have the same end time and one must be removed, it doesn't matter which — prev_end ends up the same value either way. The greedy still works correctly.

4 The Code (Annotated) ▾

Solution 1 — Greedy: Sort by End (Optimal, Interview Pick)

def eraseOverlapIntervals(intervals): ← input: list of [start, end] pairs
intervals.sort(key=lambda x: x[1]) ← sort by END time — the greedy insight
count = 0 ← removals so far
prev_end = float('-inf') ← end of last KEPT interval; -inf = nothing kept yet
for start, end in intervals: ← process in sorted order
if start >= prev_end: ← no overlap: starts at or after last kept ended
prev_end = end ← keep it; advance our boundary
else: ← overlap: start < prev_end
count += 1 ← remove current (later-ending) interval
return count ← minimum removals needed
⚠️ Why don't we update prev_end when removing?

When we remove the current interval, prev_end stays pointing to the last kept interval's end. We removed the current (which ends later) precisely because it conflicts with the kept one. The kept interval remains our reference for future comparisons.

Solution 2 — DP: Longest Non-overlapping Chain (O(n²))

def eraseOverlapIntervals_dp(intervals): ← O(n²) time, O(n) space
if not intervals: return 0
intervals.sort() ← sort by START for this DP variant
n = len(intervals)
dp = [1] * n ← dp[i] = max non-overlapping ending at i
for i in range(1, n):
for j in range(i): ← check all prior intervals
if intervals[j][1] <= intervals[i][0]:
dp[i] = max(dp[i], dp[j] + 1) ← extend chain: j then i is valid
return n - max(dp) ← total - max we can keep = min to remove

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Reframe from Removals to Kept

"Minimum removals = n − maximum kept. So I'll maximize the number of non-overlapping intervals I can keep."

Step 2: Recognize Activity Selection

"This is the classic Activity Selection Problem. Select the maximum number of non-conflicting intervals. Greedy: sort by end time and always pick the earliest-finishing compatible interval."

Step 3: Justify the Greedy

"By exchange argument: if we keep an interval that ends later when we could have kept one ending sooner, we can always swap without making things worse — the earlier-ending interval is at least as good for future compatibility."

Step 4: Implement and Check Edge Cases

"Sort by end. Track prev_end = -∞. For each interval: if start ≥ prev_end → keep (update prev_end). Else → count removal. Edge: touching intervals (start == prev_end) are valid and should be kept."

🗣️ What to Say to the Interviewer

"I'll reframe: minimum removals = n − maximum kept intervals.
This is Activity Selection — greedy: sort by end time, take the earliest-finishing compatible interval.
Sort by end. Track prev_end (end of last kept interval), count removals.
For each interval: if start >= prev_end, no overlap, keep it and update prev_end.
Otherwise: overlap — remove the current (it ends later). count++. prev_end stays.
Key subtlety: >= not > since touching at a point is not overlapping.
Time O(n log n) for sort, O(n) for scan. Space O(1) extra."

7 Summary & Key Takeaways ▾
Reframe "minimum removals" as "maximum kept" — they are equivalent but the greedy solution flows from the latter.
Sort intervals by END time. This is the Activity Selection greedy principle: earliest finisher first maximizes future options.
When two intervals conflict, remove the current (later-ending) one. Keep prev_end pointing to the earlier-ending kept interval.
Use start >= prev_end (not >) — touching at an endpoint is NOT an overlap by this problem's definition.
Time O(n log n) dominated by sort; Space O(1) — only two scalar variables beyond the sorted input.
🎯 When to Recognize This Pattern
  • Problem involves intervals, events, meetings, or activities with start/end times
  • Goal is to maximize compatible selections OR minimize conflicts/removals
  • Keywords: "non-overlapping", "maximum non-conflicting", "minimum to remove"
  • After reframing: "pick the most items with no pairwise conflict" → Activity Selection
  • Always ask: sort by start or end? End → maximize kept. Start → merge/cover problems.
