LeetCode #761

Special Binary String

Rearrange the special substrings of s in descending lexicographic order to produce the largest possible valid result.

Hard Recursion Recursive Decomposition Time: O(n² log n) Space: O(n log n)
1 Understanding the Problem ▾

A special binary string is defined with two rules. First, every prefix must have at least as many 1s as 0s. Second, the total count of 1s equals the total count of 0s. Equivalently:

Special(s) means: s = "" OR 1 Special 0 OR Special + Special
Read: a special string is either empty, or it wraps a special string between a 1 and 0, or it's two special strings concatenated.
🔑 The Parentheses Analogy

Map 1 → ( and 0 → ). A special binary string is exactly a valid parenthesization. The "balance" (running count of 1s minus 0s) never goes negative, and ends at 0. This analogy is your compass for the whole problem.

✓ Special Strings

"" → empty, trivially special
10 → 1 + "" + 0
1100 → 1 + (10) + 0
1010 → (10) + (10)
110100 → 1 + (1010) + 0

✗ Not Special

01 → starts with 0 (balance goes -1)
110 → counts: 2 ones, 1 zero
1001 → prefix "100" has balance -1
11 → two ones, no zeros at all

What the problem asks: You are allowed to swap any two adjacent special substrings. Find the ordering of swaps that produces the lexicographically largest result, and return that string.

💡 Key Insight

Since you can swap any adjacent special substrings (not just once), you can effectively freely reorder the top-level maximal special substrings in any order you want. So the problem reduces to: decompose into maximal special pieces at each level, maximize each piece recursively, then sort pieces descending to get the largest concatenation.

Edge cases: Empty string → return "". A string that is itself one maximal special string (like "10") → inner is empty, return "10". Multiple segments already in optimal order → still correct (sort is stable for equal keys).

2 The Algorithm: Recursive Decomposition + Greedy Sort ▾

The function makeLargestSpecial(s) takes a special binary string and returns the lexicographically largest rearrangement. It works in three phases:

1 Scan with a balance counter. Initialize balance=0, i=0. Walk j from left to right: +1 on '1', -1 on '0'. Each time balance hits 0, the substring s[i..j] is one complete maximal special segment.
2 Strip the outer wrapper and recurse. Every maximal segment has the form 1 + inner + 0. Extract inner = s[i+1:j]. Call makeLargestSpecial(inner) recursively. Then re-wrap: '1' + result + '0'. Advance i = j+1.
3 Sort descending and join. Collect all reconstructed segments. Sort them in reverse lexicographic order. Concatenate. This gives the maximum possible string at this level.

Worked trace on s = "11011010":

j=0: '1'→bal=1 j=1: '1'→bal=2 j=2: '0'→bal=1 j=3: '1'→bal=2
j=4: '1'→bal=3 j=5: '0'→bal=2 j=6: '1'→bal=3 j=7: '0'→bal=2
Hmm, balance never reaches 0 → entire "11011010" is ONE segment!
inner = s[1:7] = "101101"
Recurse on "101101":
j=0: '1'→1 j=1: '0'→0 ← seg1 = s[0:1]="10", inner=""
j=2: '1'→1 j=3: '1'→2 j=4: '0'→1 j=5: '0'→0 ← seg2 = s[2:5]="1101"
Recurse on "1101" inner = "10": makeLargestSpecial("10") = "10"
seg2 = "1" + "10" + "0" = "1100"
Sort ["10","1100"] desc → ["1100","10"] (since "1100" > "10")
Return "110010"
Back at top level: wrap → "1" + "110010" + "0" = "11100100"
Output: "11100100"
🧠 Why Descending Sort Maximizes the Result

For any two segments A and B, we want A + B if A + B > B + A lexicographically. Since every segment starts with '1' and ends with '0', comparing A and B directly gives the same ordering. This is the same greedy argument as LeetCode #179 (Largest Number) — sort by the "better prefix first" criterion.

3 Why It Works: The Invariant ▾
🔐 The Core Invariant

makeLargestSpecial(s) always receives a valid special binary string and always returns a valid special binary string that is the lexicographically largest rearrangement. This invariant holds at every recursive level because:

1. Swapping adjacent specials preserves validity. If A and B are both special (balanced, never negative prefix-balance), then both A + B and B + A are valid concatenations of specials — still balanced globally, still valid. So we can freely reorder them.

2. The balance-zero decomposition is exhaustive and non-overlapping. The balance counter starting from any position i=0 will hit zero at exactly the right boundary of each maximal special substring. There's no ambiguity — this gives a unique decomposition at each level.

3. Recursion correctly maximizes inner content. By induction: assume the recursive call on any inner string of length < n returns the maximum. Then we process each inner correctly, re-wrap, and sort. The base case (empty string) trivially returns the maximum (itself).

4. Descending sort is optimal. For any two adjacent segments, placing the lexicographically larger one first strictly improves the result. So sorting descending is a globally optimal greedy choice — no permutation can do better.

⚠️ Common Mistakes

Sorting by length only: Fails for equal-length segments with different content, e.g., "1100" (sorted: 11-00) vs "1010" (sorted: 10-10). "1100" > "1010" lexicographically, so length alone doesn't work — compare the full string.

Forgetting to re-wrap: After recursing on the inner content, you MUST put the '1' and '0' back. The outer wrapper is essential for the parent level's sort comparison.

Off-by-one in slice: Inner is s[i+1:j], not s[i:j+1]. The segment is s[i..j] (inclusive), so inner excludes both endpoints.

4 The Code (Annotated) ▾

Solution 1 — Recursive Decomposition (Optimal · Interview Pick)

def makeLargestSpecial(s: str) -> str:
count, i = 0, 0 ← balance and start-of-segment pointer
specials = [] ← will hold all maximized top-level segments
for j, c in enumerate(s): ← scan every character with index j
count += 1 if c == '1' else -1 ← '1' opens (+1), '0' closes (-1) — parentheses analogy
if count == 0: ← balance back to 0: s[i..j] is a complete maximal segment
inner = makeLargestSpecial(s[i+1:j]) ← strip outer 1 and 0, recurse on inner content
specials.append('1' + inner + '0') ← re-wrap: put the outer 1 and 0 back
i = j + 1 ← advance segment start to after this segment
specials.sort(reverse=True) ← sort all segments desc: largest first
return ''.join(specials) ← concatenate → lexicographically largest result
⚠️ Key Detail: s[i+1:j] not s[i:j+1]

When balance hits 0 at position j, the complete segment is s[i] through s[j] (inclusive). s[i] is the opening '1' and s[j] is the closing '0'. The inner content to recurse on is everything strictly between them: s[i+1:j] (Python slice is exclusive on the right, so this gives s[i+1] through s[j-1]). We then manually add the '1' and '0' back after recursion.

Solution 2 — Brute Force (for understanding, do NOT use in interviews)

from itertools import permutations
def makeLargestSpecial_brute(s: str) -> str:
# Collect top-level segments using same balance scan
count, i, segs = 0, 0, [] ← same segment-finding logic
for j, c in enumerate(s):
count += 1 if c == '1' else -1
if count == 0: segs.append(s[i:j+1]); i = j+1 ← do NOT recurse on inner
best = ""
for perm in permutations(segs): ← O(k!) — catastrophic for k>8
best = max(best, ''.join(perm)) ← check every ordering
return best
💡 Complexity Summary

Recursive (Optimal): O(n² log n) time — O(log n) recursion depth for balanced inputs, each level scans O(n) and sorts O(k log k) segments where total length ≤ n. Space O(n log n) for stack frames and string slices.
Brute Force: O(k! · n) — factorial in the number of top-level segments. Infeasible for any real input.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Anchor on the Parentheses Analogy

"A special binary string with 1=( and 0=) is exactly a valid parenthesization. This immediately tells me how to detect boundaries: a running balance that returns to 0 marks the end of a segment. I'll use this to decompose the string."

Step 2: Identify What Can Be Freely Reordered

"Adjacent special substrings can be swapped freely because concatenating two specials in any order produces another valid special sequence. So I have complete freedom to reorder top-level maximal segments — the question is just which order to choose."

Step 3: Recognize the Recursive Substructure

"Each maximal segment is '1 + inner + 0' where inner is also special. I can't swap things across different outer wrappers — only within the same level. So I must recurse: maximize inner before comparing outer wrappers. Same problem, smaller size — classic recursive decomposition."

Step 4: Apply the Greedy Sort

"After recursing, I sort segments descending. Is this safe? Yes — for any A and B, if A > B lex then A+B > B+A lex (since both start with '1'). Descending sort is the globally optimal choice. This is the same proof as LeetCode 179."

🗣️ What to Say to the Interviewer

"I recognize 1=(, 0=) — this is a balanced parentheses structure.
I'll use a balance counter to decompose s into maximal special segments.
Each segment has form '1 + inner + 0'. I recurse on the inner to maximize it.
Then re-wrap, collect all segments, sort them descending, and join.
Descending sort is correct because for any A > B lex, A+B > B+A lex (both start with '1').
Base case: empty string returns itself.
Time: O(n² log n), Space: O(n log n)."
🔮 Likely Follow-up Questions
  • "What is the recursion depth?" → O(log n) for balanced strings (like "111...000"), O(n/2) worst case for deeply chained "10 10 10..." at every level.
  • "Can you prove the sort is correct?" → Compare A+B vs B+A. Since both A and B start with '1', lex comparison of A and B determines which concatenation wins. Same argument as #179.
  • "What if two segments have equal lex value?" → They're identical strings, so any order gives the same result. The sort is stable; outcome is unchanged.
  • "Can this be done iteratively?" → Yes, with an explicit stack, but the recursive form is cleaner and shows the structure more clearly.
