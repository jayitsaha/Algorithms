LeetCode #1027

Longest Arithmetic Subsequence

Given an integer array nums, return the length of the longest arithmetic subsequence in nums. A subsequence is an arithmetic sequence if consecutive elements have the same difference.

Medium Dynamic Programming dp[i][diff] length Time: O(n²) Space: O(n²)
1 Understanding the Problem ▾

An arithmetic sequence is a list where consecutive elements differ by the same constant. For example, [3, 6, 9, 12] has common difference +3, and [10, 7, 4, 1] has common difference −3. A subsequence preserves order but does not require consecutive elements — we may skip elements.

So the challenge is: pick the largest set of indices i₁ < i₂ < ... < iₖ from nums such that nums[i₂]−nums[i₁] = nums[i₃]−nums[i₂] = ··· = constant, and return k.

💡 Key Insight

An arithmetic sequence is completely characterized by its last element and its common difference. If we know the chain ending at index i with difference d, we can extend it whenever we find a new element that is exactly d greater than nums[i]. This screams DP: track the best chain length for every (position, difference) pair.

✓ Example 1: nums = [3, 6, 9, 12]

Subsequence: [3, 6, 9, 12]
Differences: +3, +3, +3
Common diff: 3
Length: 4 ← answer

⚡ Example 2: nums = [9, 4, 7, 2, 10]

Subsequence: [4, 7, 10]
Indices: 1, 2, 4 (skip index 3)
Common diff: 3
Length: 3 ← answer

Edge Cases: A single element trivially forms a sequence (length 1). Any two elements form a sequence (length 2) — so the answer is always ≥ 2 given the constraints. The difference can be 0 (all equal), negative, or positive. Duplicate values are valid (difference 0 sequences).

2 The Algorithm: Why This Is Dynamic Programming ▾

Before seeing the code, understand why DP is the right tool here, and not greedy or brute force.

🔑 Two Pillars of DP for This Problem

1. Optimal Substructure: The longest arithmetic subsequence ending at index i with difference d is exactly 1 plus the longest such subsequence ending at some earlier index j with the same d, where nums[i] − nums[j] = d. Big problem solved by smaller problems.

2. Overlapping Subproblems: Many pairs (i, d) arise repeatedly as we scan. For example, the chain info at index 3 with diff +3 is needed when processing index 4, index 5, index 7, etc. Storing these avoids exponential re-computation.

Why brute force is O(n³): We could fix a start index and a difference, then greedily scan right. But this reconsiders the same chains from scratch each time — no reuse. DP stores each chain's length so later indices can extend it in O(1).

📐 The Recurrence Relation
State: dp[i][d] = length of longest arithmetic subsequence ending at index i with common difference d

Transition: For each pair (j, i) with j < i:
    d = nums[i] − nums[j]
    dp[i][d] = dp[j].get(d, 1) + 1

Base case: dp[i] = {} initially (empty — no chains known yet)
Answer: max over all i, d of dp[i][d] (initialized to 2)
1 Initialize: dp = [dict() for _ in range(n)]. Each dp[i] is a hash map: difference → chain length. Start with empty dicts. ans = 2 (minimum possible).
2 Outer loop — choose endpoint i: for i in range(1, n). For each index i, we build the best chains that end here.
3 Inner loop — look back at j: for j in range(i). Compute d = nums[i] − nums[j]. If dp[j] has an entry for d, extend that chain. Otherwise start a new length-2 chain.
4 Update dp[i][d]: dp[i][d] = dp[j].get(d, 1) + 1. The default of 1 means "j alone is a chain of length 1; adding i makes it length 2."
5 Track the global best: ans = max(ans, dp[i][d]) after each update. Return ans.
⚠️ Why Use a Dict Per Position (Not a 2D Array)?

Values can range from 0 to 500, giving differences from −500 to +500 (1001 possible values). A 2D array of size n × 1001 works but requires an offset to handle negatives. A dict per position is cleaner: it auto-handles negatives, only stores seen differences, and is idiomatic Python. In practice, each dp[i] has at most i entries.

3 Why It Works: The Invariant ▾
🔐 The Invariant

After processing all pairs (j, i) for a fixed i, dp[i][d] contains the exact length of the longest arithmetic subsequence that ends at index i and has common difference d. This is guaranteed for all previously processed positions too.

Why examining all j < i is necessary and sufficient: The element just before index i in the optimal chain can be at any position j < i — not necessarily adjacent. So we must check all of them. But dp[j][d] already captures the best chain ending at j, so we don't re-examine all of j's history — that work is already done and stored.

Why dp[j].get(d, 1) + 1 is correct: If dp[j][d] exists, it gives the length of the longest chain ending at j with difference d. Adding i extends it by 1. If it doesn't exist, j has no known chain with d — j itself is a single element (length 1), and i makes it length 2. The default of 1 handles this base case elegantly without any if-else guard.

Coverage guarantee: Every possible arithmetic subsequence must have some pair of consecutive elements (j, i) that we examine in the double loop. When we find this pair, we correctly build on the chain stored at dp[j]. No valid subsequence is missed.

💡 Analogy: Chain Handoff

Think of each index as a runner in a relay race. Runner i scans all earlier runners j and asks: "Were you running at pace d = nums[i] − nums[j]? If so, pass me your baton." dp[j][d] tells runner i exactly how many legs of the race j's team has already run at that pace. Runner i adds one more leg and reports the new total.

✓ What Makes This DP (Not Greedy)

Each dp[i][d] depends on ALL earlier dp[j][d] values — not just the most recent. A greedy "always pick the nearest element with the right difference" would miss chains that skip over nearer elements for longer-term gains.

✗ Why Greedy Fails

nums = [1, 5, 2, 3, 4]. Greedy with d=1 starting at 1 picks [1, 2, 3, 4] (length 4). But greedy might fixate on a local match and miss the globally optimal chain if it starts with suboptimal pair choices.

4 The Code (Annotated) ▾

Solution 1 — DP Tabulation / Bottom-Up (Interview Pick ✓)

def longestArithSeqLength(nums: list) -> int:
n = len(nums) ← number of elements in the input
dp = [dict() for _ in range(n)] ← dp[i] maps difference → chain length ending at i
ans = 2 ← any 2 elements form a valid sequence; lower bound is 2
for i in range(1, n): ← i is the current endpoint being built toward
for j in range(i): ← j is the candidate "previous element" in the chain
d = nums[i] - nums[j] ← common difference if i directly follows j
dp[i][d] = dp[j].get(d, 1) + 1 ← extend j's chain (default 1 = j alone) by 1
ans = max(ans, dp[i][d]) ← keep track of the globally longest chain seen
return ans ← longest arithmetic subsequence length found
⚠️ Critical: Why get(d, 1) and Not get(d, 0)?

If dp[j] has no entry for difference d, it means j is a lone element in a chain of this difference — but j itself still counts as length 1. Adding nums[i] next gives length 2. Using default 0 would give length 1, failing to count j. This is the most common bug students write on first attempt.

Solution 2 — DP Memoization / Top-Down

def longestArithSeqLength(nums: list) -> int:
from functools import lru_cache
n = len(nums)
@lru_cache(maxsize=None)
def dp(i, d): ← longest AP ending at index i with difference d
best = 1 ← base: i alone is length 1
for j in range(i): ← look back at all earlier positions
if nums[i] - nums[j] == d: ← if j can be the element before i in this sequence
best = max(best, dp(j, d) + 1) ← recurse into j's chain and extend
return best
ans = 2
for i in range(n):
for j in range(i):
ans = max(ans, dp(i, nums[i] - nums[j]))
return ans
🔑 Bottom-Up vs Top-Down: Trade-offs

Tabulation (bottom-up) — preferred: Iterative, no recursion stack overhead, naturally fills subproblems in dependency order. In Python, avoids hitting the default 1000-frame recursion limit on large inputs. Time O(n²), Space O(n·D) where D = distinct differences per position.

Memoization (top-down): Closer to the recurrence in code structure. Useful if you think recursively. Same asymptotic complexity. Risk: Python recursion limit for n ≥ 500.

Solution 3 — Brute Force O(n³) (for understanding; never use in interviews)

def longestArithSeqLength_brute(nums): ← O(n³) — illustrates why DP is needed
n, ans = len(nums), 2
for i in range(n): ← fix start of sequence
for j in range(i+1, n): ← fix second element → determine difference
d, length, last = nums[j]-nums[i], 2, nums[j]
for k in range(j+1, n): ← greedily extend from (i,j) with diff d
if nums[k] - last == d:
length += 1; last = nums[k]
ans = max(ans, length)
return ans ← re-computes every chain from scratch: O(n³) total
5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the DP Pattern

"This asks for the longest subsequence with a structural property — exactly the shape of LIS. The key word is 'longest,' the constraint is 'preserve order,' and the property is 'constant difference.' That trio is a DP signal."

Step 2: Define the State Carefully

"For LIS, state is dp[i] = length of longest increasing subsequence ending at i. Here, the common difference is also part of the state — I need dp[i][d] = length of longest arithmetic subsequence ending at i with difference d."

Step 3: Write the Transition

"For each pair (j < i), d = nums[i]−nums[j]. Extend dp[j][d] by 1. Default to 1 if dp[j] has no chain for d. Update global answer."

Step 4: Choose Data Structure

"Differences can be negative and span a wide range. Dict per position is cleaner than a 2D array with offset indexing. Each dict has at most i entries — total memory is O(n²) in the worst case."

🗣️ What to Say to the Interviewer

"This looks like a DP problem — I need the longest subsequence with a structural constraint.
State: dp[i][d] = length of longest AP ending at index i with common difference d.
Transition: for each j < i, d = nums[i]−nums[j]; dp[i][d] = dp[j].get(d, 1) + 1.
The default of 1 handles the case where j has no prior chain for this difference.
I track the global max as I go. Time O(n²), Space O(n²). Ready to code?"
🔮 Likely Follow-up Questions
  • "Can you do better than O(n²)?" → For bounded values (0–500), technically yes with sorted structures, but O(n²) is the standard/expected for general case.
  • "What if there are duplicates?" → Handled naturally — difference 0 is a valid arithmetic sequence.
  • "Count all arithmetic subsequences (not just the longest)?" → LeetCode #446 "Arithmetic Slices II" — same dp[i][d] framework, but sum counts instead of max length.
  • "Return the actual subsequence?" → Store parent pointers: parent[i][d] = j. Backtrack from the max cell.

7 Summary & Key Takeaways ▾
State is (position, difference) — both must be tracked to characterize a chain
dp[i][d] = dp[j].get(d, 1) + 1 for every j < i where nums[i]−nums[j]=d
Default of 1 (not 0) in get() is critical — j alone is already length 1
Use a dict per position — differences span a wide signed range
Initialize ans = 2 — any two elements form an arithmetic subsequence
Pattern generalizes: replace "difference" with any computable key property

  • "Longest subsequence with constant property between consecutive elements" → dp[i][property]
  • Need to group states by a derived key (difference, ratio, XOR) → dict per position
  • Every pair (i, j) contributes to a state — O(n²) DP is the right complexity class
