LeetCode #654

Maximum Binary Tree

Given an array of integers with no duplicates, build the "maximum binary tree": the root is always the largest element, the left subtree is built from elements to its left, and the right subtree from elements to its right.

Medium Divide and Conquer Find Max, Build Left/Right Time: O(n²) worst Space: O(n)
1 Understanding the Problem ▾

You are given an array like [3, 2, 1, 6, 0, 5]. Build a binary tree following three rules recursively: (1) The root of any subtree is the maximum element in the current subarray. (2) Everything to the LEFT of that maximum forms the left subtree (by the same three rules). (3) Everything to the RIGHT of that maximum forms the right subtree. An empty subarray produces a null node.

💡 Key Insight

This problem's description is already the algorithm written in English. "Root = max, left subtree = left part, right subtree = right part" — that's a recursive function. Recognize this self-similarity and the implementation writes itself.

Input: [3, 2, 1, 6, 0, 5]

max = 6 at index 3 → root
left [3,2,1] → left subtree
right [0,5] → right subtree
Each part is solved the same way

Resulting Tree Structure

    2 0

Edge cases to consider: Empty array → return None. Single element → leaf node with no children. Already-sorted array → maximally skewed tree of depth n (worst case O(n²)). All elements are guaranteed distinct by the problem.

🔑 Why is the result unique?

Since all elements are distinct, each subarray has exactly one maximum. There is no choice to make at any step — the algorithm is deterministic. Given an input, there is exactly one valid output tree.

2 The Algorithm: Recursive Divide and Conquer ▾

The maximum element acts as a pivot — it divides the array into two independent halves. After placing the root, the left and right subproblems share no data and can be solved independently. This is the hallmark of Divide and Conquer.

1 Base case: If the subarray is empty (lo > hi), return None. This handles missing children of leaf nodes.
2 Find the max: Linear scan through nums[lo..hi] to find index max_i of the largest element. This is O(n) per call.
3 Create root: root = TreeNode(nums[max_i]). This node is the root of the subtree for the range [lo..hi].
4 Recurse left: root.left = build(lo, max_i - 1). Build the left subtree from everything left of the max.
5 Recurse right: root.right = build(max_i + 1, hi). Build the right subtree from everything right of the max.
6 Return root: The fully assembled node (with both subtrees attached) is returned to the parent call, which attaches it as a child.
// Recursion tree for [3, 2, 1, 6, 0, 5] build([3,2,1,6,0,5]) max=6 (idx 3) → root=6 ├─L build([3,2,1]) max=3 (idx 0) → root=3 │ ├─L build([]) → None (lo>hi, empty left of idx 0) │ └─R build([2,1]) max=2 (idx 0) → root=2 │ ├─L build([]) → None │ └─R build([1]) → leaf 1 └─R build([0,5]) max=5 (idx 1) → root=5 ├─L build([0]) → leaf 0 └─R build([]) → None (lo>hi, empty right of idx 5)
🧭 Connection to Quick Sort

This algorithm is structurally the same as Quick Sort using the array maximum as the pivot: pick max, partition around it, recurse on each half. Quick Sort produces a sorted array from the process; Maximum Binary Tree captures the structure of that recursive partitioning as an actual tree.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every recursive call, build(lo, hi) returns the root of the maximum binary tree built from exactly nums[lo..hi]. This invariant holds at the base case (empty → None, single element → leaf) and by induction holds at every larger range because both recursive calls satisfy the same invariant on their smaller inputs.

Why the max must be the root: The problem defines the root as the largest element. Within any subarray, there is exactly one largest element (distinct elements guaranteed). Choosing it satisfies the root constraint for this level.

Why splitting at max_i is correct: Elements at indices < max_i appear to the left of the max in the original array — they belong in the left subtree by definition. Elements at indices > max_i appear to the right — they belong in the right subtree. Splitting at max_i preserves this positional relationship precisely. No element can "cross" to the wrong side.

Why recursion always terminates: Each call receives a range that is strictly smaller than its parent call (we always exclude the max element itself). Eventually ranges shrink to 0 or 1 element, both of which are base cases.

⚠️ Common Misconception: "Is this a BST?"

No. A BST requires left < root < right by value. Here, values on the left or right can be anything relative to each other. For example, node 3 has right child 2 (2 < 3 is fine in a BST, but here it's because 2 appears after 3 in the original array's left subarray [3,2,1]). The constraints are purely about position in the original array, not BST value ordering.

Worst case analysis: If input is ascending-sorted (e.g., [1,2,3,4,5]), the max is always the last element. Left subarray gets all remaining elements; right subarray is always empty. Depth = n, each level scans O(n) → O(n²) total. A random input gives expected depth O(log n) → O(n log n) average.

4 The Code (Annotated) ▾

Solution 1 — Recursive D&C with Index Bounds (Interview Pick)

class TreeNode:
def __init__(self, val=0, left=None, right=None):
self.val, self.left, self.right = val, left, right
def constructMaximumBinaryTree(nums): ← entry point; delegates to the helper
def build(lo, hi): ← builds tree from nums[lo..hi] inclusive
if lo > hi: ← base case: empty subarray → null child
return None
max_i = lo ← assume first element is the max index
for i in range(lo, hi + 1): ← scan the entire current window
if nums[i] > nums[max_i]: ← found a bigger element?
max_i = i ← update our best-so-far index
root = TreeNode(nums[max_i]) ← create this subtree's root
root.left = build(lo, max_i - 1) ← recurse on elements LEFT of max
root.right = build(max_i + 1, hi) ← recurse on elements RIGHT of max
return root ← return the assembled node
return build(0, len(nums) - 1) ← start with the full array range
⚠️ Why index bounds instead of slicing?

Python slicing (nums[:idx], nums[idx+1:]) allocates a new list for each call. In a worst-case balanced tree of depth log n, the total extra memory is O(n log n). In the worst case (sorted input, depth n), slicing allocates O(n²) bytes. Using index bounds (lo, hi) into the original array is O(1) overhead per call and O(n) total for the recursion stack.

Solution 2 — Clean Slice Version (simpler to read, more space)

def constructMaximumBinaryTree(nums): ← more readable but uses O(n²) space worst-case
if not nums: ← base case: empty list
return None
idx = nums.index(max(nums)) ← two O(n) passes: max(), then index()
root = TreeNode(nums[idx]) ← root = maximum element
root.left = constructMaximumBinaryTree(nums[:idx]) ← new list: O(n) copy
root.right = constructMaximumBinaryTree(nums[idx+1:]) ← new list: O(n) copy
return root
💡 Complexity Summary

Index-bound recursion: Time O(n²) worst / O(n log n) avg · Space O(n) stack
Slice recursion: Same time · Space O(n²) worst (copies at each level)
Monotonic Stack (O(n)): Process left-to-right, maintain decreasing stack. Each element pops smaller elements (they become its left child) and then becomes the right child of the remaining top. Builds the tree in one pass — mention as a bonus optimization.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Read the Problem as Pseudocode

The problem statement literally says: root = max, left subtree = left subarray, right subtree = right subarray. That's a recursive function. Read it aloud: "Base case is empty array. Otherwise find max, make it root, recurse left, recurse right." You're already done.

Step 2: Name the Pattern (Divide and Conquer)

Tell the interviewer: "I see a Divide and Conquer pattern. The max element divides the array into two independent halves. After placing the root, neither half interacts with the other — I can solve each completely independently."

Step 3: Write the Skeleton, Then Fill In

Write the recursive skeleton first: if lo > hi: return None; find max; create root; root.left = build(lo, max_i-1); root.right = build(max_i+1, hi); return root. The structure comes before the details.

Step 4: Analyze Complexity and Offer Optimization

State O(n²) worst case upfront (sorted input). Mention O(n log n) average. Then offer: "If O(n) is needed, there's a monotonic decreasing stack solution that builds the tree in one left-to-right pass." This shows depth of knowledge.

🗣️ What to Say to the Interviewer

"The problem description is already the recursive algorithm: root is the max, left subtree from the left part, right subtree from the right part.
I'll write a helper build(lo, hi) using index bounds — avoids extra space from slicing.
Base case: lo > hi returns None. Otherwise, scan for the max index, create the root, recurse left on [lo, max_i-1] and right on [max_i+1, hi].
Time: O(n) per level × depth. Worst case (sorted input): O(n²). Average: O(n log n).
Space: O(n) for the tree + O(n) recursion stack.
If you want O(n) time, there's a monotonic stack approach — want me to walk through that?"
🔮 Likely Follow-up Questions
  • "Can you do O(n)?" → Monotonic decreasing stack; left-to-right single pass
  • "Maximum Binary Tree II?" → LeetCode #998: insert a new last element into the right spine
