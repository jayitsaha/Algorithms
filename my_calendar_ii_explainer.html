LeetCode #731

My Calendar II

Implement a calendar that allows double-booking but not triple-booking. Return true if an event can be added, false if it would cause a triple overlap.

Medium Trees / Intervals Two Lists Time: O(n) per call Space: O(n)
1 Understanding the Problem ▾

You are building a calendar booking service. Each call to book(start, end) requests a new event covering the half-open interval [start, end). The rule: at most two events may overlap at any instant. A third simultaneous event is a "triple booking" and must be rejected.

Think of a meeting room that has two projectors — two meetings can run simultaneously (double-booking), but a third would break things.

Core Interval Overlap Formula

Two intervals [a,b) and [c,d) overlap if and only if a < d AND c < b. Their intersection is [max(a,c), min(b,d)), which is non-empty when max(a,c) < min(b,d).

Key Insight

Instead of counting overlaps per point (expensive), maintain a separate list of double-booked regions. A new event is rejected iff it overlaps any region in that list — because that would turn a double into a triple. If accepted, any intersection with existing single bookings becomes a new double-booked region.

2 The Algorithm — Two Lists ▾

Maintain two lists: calendar (all accepted bookings) and overlaps (all double-booked regions — where exactly two events coincide). Each book call does three things in order:

1Triple-check: Scan overlaps. If the new event intersects ANY double-booked region, adding it would triple-book that region — return False.
2Expand doubles: Scan calendar. For each existing event, compute its intersection with the new event. Any non-empty intersection is a new double-booked region — add it to overlaps.
3Accept: Append the new event to calendar. Return True.
Naive: Count overlaps per point

For each new booking, scan all existing pairs and find how many overlap the new interval. O(n²) per call, hard to implement correctly.

Two Lists: Track double regions

Precompute all pairwise intersection regions. Triple-checking becomes a single linear scan over that list. O(n) per call — clean and correct.

Critical Order: Check BEFORE Update

You MUST check overlaps for triple conflicts before you start adding new entries to it. Doing them in the wrong order introduces spurious rejections. The invariant: by the time you update overlaps, you've already confirmed no triple will occur.

3 Why It Works: The Invariant ▾
The Invariant

After every successful book call: overlaps contains exactly the union of all pairwise intersections of the events currently in calendar. Equivalently, every point in any interval in overlaps is currently covered by at least 2 calendar events, and no point in calendar is covered by more than 2.

Proof that checking overlaps catches all triple bookings:

Suppose a new event N overlaps with region R in overlaps. By the invariant, R is covered by at least 2 existing events A and B. Now N also covers R. So R is covered by A, B, and N simultaneously — a triple booking. Correctly rejected.

Proof that not overlapping overlaps means safe:

If N doesn't intersect any interval in overlaps, then for every point p that N covers, at most 1 existing event already covers p (because if 2 did, their intersection would be in overlaps and would include p). So adding N makes that at most 2 — still within limit.

Edge Cases

Adjacent intervals [10,20) and [20,30): max(10,20)=20, min(20,30)=20 → 20 < 20 is false. Not overlapping. Half-open intervals handle this automatically.

Same event booked twice, e.g. [10,20) twice: Second call checks overlaps (empty after first). Then it finds intersection with [10,20) in calendar → adds [10,20) to overlaps. Books it. A third call for [10,20) would then find [10,20) in overlaps and be rejected. Correct behavior.

4 The Code (Annotated) ▾

Solution 1 — Two Lists (Interview Pick) — O(n) per call, O(n) space

class MyCalendarTwo:// Booking system class
def __init__(self):
self.calendar = [] // All accepted [start,end) bookings
self.overlaps = [] // All double-booked regions (pairwise intersections)
def book(self, start, end):// Called with each new event [start,end)
for a, b in self.overlaps:// Step 1: Check for triple-booking risk
if start < b and a < end:// Standard interval overlap: a<d AND c<b
return False// New event intersects a double-booked region → triple → reject
for a, b in self.calendar:// Step 2: Find new double-booking regions
lo = max(start, a)// Intersection start = the later of the two starts
hi = min(end, b)// Intersection end = the earlier of the two ends
if lo < hi:// If intersection is non-empty (positive length)
self.overlaps.append((lo, hi))// Record this new double-booked region
self.calendar.append((start, end))// Step 3: Accept the booking
return True// Booking succeeded
Common Mistake — Wrong Order

Many candidates scan calendar first, update overlaps, then check overlaps. This is wrong: you'd see an overlap you JUST added from the current event and reject the booking falsely. Always check overlaps (triple-conflict test) FIRST, then expand overlaps with new intersections.

Solution 2 — Segment Tree (Advanced, O(log n) per call)

# Dynamic segment tree with lazy propagation
class MyCalendarTwo:
def __init__(self): self.tree = {} // {node: max_bookings_in_range}
def update(self, lo, hi, l, r, node): // Increment [lo,hi) in tree
if lo >= r or hi <= l: return 0 // No overlap with query range
if lo <= l and r <= hi:// Fully inside: count this node
self.tree[node] = self.tree.get(node,0) + 1
return self.tree[node]
mid = (l + r) // 2
left = self.update(lo,hi,l,mid,2*node)
right = self.update(lo,hi,mid,r,2*node+1)
return self.tree.get(node,0) + max(left, right)
def book(self, s, e):
if self.update(s,e,0,10**9,1) >= 3: // Would it triple-book?
self.update(s,e,0,10**9,1) // Undo (can't easily undo in this simple form)
return False
return True

5 Interview Strategy ▾
Step 1: Recognize the Pattern

When you see "allow k overlaps but not k+1," think: maintain k-1 lists of pre-computed overlap zones. For k=2 here, one overlaps list suffices. This pattern extends: My Calendar III (find max k) uses a difference array for O(n log n) total.

Step 2: Start from My Calendar I

My Calendar I rejects all overlaps — it scans calendar for any intersection. Here, we allow the first overlap but reject the second. So add ONE more tracking layer: calendar for all events, overlaps for double regions. The new event only violates the rule if it hits overlaps (not just calendar).

Step 3: State the Invariant

Tell the interviewer: "I'll maintain overlaps as the union of all pairwise intersections. The check becomes: does the new event intersect any double-booked region?" This shows precise thinking about what the data structure represents.

Step 4: Mention the Order Bug

Proactively say "I check overlaps FIRST before updating it — otherwise I'd see my own newly-added intersections and reject valid events." Interviewers love when candidates anticipate and explain subtle ordering issues.

What to Say

"I'll use two lists: calendar for all accepted events, and overlaps for the double-booked regions — every point where two events already intersect. For each new event, I first check if it intersects any interval in overlaps. If yes, that region would become triple-booked, so I return false. Otherwise, I compute all intersections with calendar events and add them to overlaps, then add the event to calendar and return true. The invariant: overlaps always contains exactly the pairwise intersections. Time is O(n) per call, O(n) space total."
Follow-up: My Calendar III (LeetCode 732)

Asks for the max k across all bookings. Use a difference array with a sorted map: increment at start, decrement at end, track running max. O(n log n) total using a TreeMap. The two-lists trick doesn't generalize directly — you'd need k-1 lists for k-overlap limit.

7 Summary & Key Takeaways ▾
Maintain two lists: calendar (all bookings) and overlaps (all double-booked regions).
A new event is rejected iff it intersects any interval in overlaps — that would be a triple booking.
Always check overlaps FIRST, then update overlaps with new intersections — order matters.
Intersection formula: [max(a,c), min(b,d)) — valid only when max < min.
This generalizes: for k-booking limit, maintain k-1 levels and check the deepest list.
When to Use This Pattern
  • Interval scheduling with bounded overlap count (k = 2, 3, ...)
  • Any "calendar booking" problem where you must limit concurrent events
  • When you need to know WHERE overlaps occur, not just whether they exist
