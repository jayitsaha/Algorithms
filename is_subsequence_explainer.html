Is Subsequence

Check if every character in s appears in t in the same relative order

Easy String Processing Two Pointers (Greedy Match) Time: O(|t|) Space: O(1)
1 Understanding the Problem ▲

Given two strings s and t, return True if s is a subsequence of t.

A subsequence means you can form s by deleting some (or zero) characters from t — but you must keep the remaining characters in their original order.

Think of it like highlighting letters in a long sentence: you can skip letters freely, but you must read the page left-to-right.

✓ is a subsequence
s = "ace"
t = "abcde"
a
b
c
d
e

Indices 0 → 2 → 4 — order preserved ✓

✗ NOT a subsequence
s = "aec"
t = "abcde"
a
b
c
d
e

'e' is at index 4, but 'c' must come AFTER it — impossible!

💡 Key Insight

We don't need to try every possible way to match characters. Because s must appear in order, we can greedily match from left to right. The moment we find a match in t, we take it — it's never beneficial to delay.

⚠️ Subsequence ≠ Substring

A substring requires characters to be contiguous ("abc" in "xabcd"). A subsequence only requires the characters to appear in order — they can be spread apart with gaps. This problem is about subsequence.

2 The Algorithm: Two-Pointer Greedy Match ▲

We use a single pointer i into s (the pattern we want to match), and scan through every character of t with a for-loop. Each character of t either advances i (match!) or is skipped.

1 Initialize i = 0. This is our "next character of s to find" pointer.
2 Scan every character char in t from left to right.
3 If char == s[i] (and i < len(s)) — match found! Advance i += 1.
4 If no match — skip char and continue to the next character in t. i stays put.
5 After the full scan: if i == len(s), every character of s was found in order → True. Otherwise False.

Why Not Brute Force?

ApproachIdeaTimeSpace
Generate All Subsequences of t Check if s is among them O(2^|t|) O(2^|t|)
Nested Loops For each char in s, linearly scan remaining t O(|s|·|t|) O(1)
Greedy Two-Pointer ✓ Single pass through t, advance s-pointer on match O(|t|) O(1)
💡 Why Taking the Earliest Match is Always Optimal

Suppose character 'c' appears at both position 2 and position 7 in t. Should we skip position 2 and use position 7?

No — using position 2 is always at least as good. It matches the same character AND leaves more of t available (positions 3–end) for the remaining characters of s. Waiting can only reduce our options. This is the greedy exchange argument.

3 Why It Works: The Invariant ▲
🔐 The Loop Invariant

At any point during the loop, i is the length of the longest prefix of s that has been matched as a subsequence within the portion of t already scanned — using the earliest possible positions greedy.

Proof by Exchange Argument

Claim: Greedily taking the leftmost available match for each character is always optimal.

Proof: Suppose an optimal solution matches s[0] at position p in t, but there exists an earlier position q < p where t[q] = s[0]. If we swap to use position q instead, we still match s[0], but now positions q+1 … |t|-1 remain — a superset of positions p+1 … |t|-1. So the greedy choice is never worse. By induction, it extends to all characters.

Edge Cases — All Handled Naturally

• s = "" (empty): i=0 already equals len(s)=0 before the loop even starts → returns True. Empty string is a subsequence of everything.
• t = "" and s is non-empty: The for loop doesn't execute. i=0 ≠ len(s) → returns False.
• len(s) > len(t): Can't match more characters than t has. i simply never reaches len(s) → returns False.
• s == t: Every character matches in order → i increments every step → returns True.
4 The Code (Annotated) ▲

Solution 1 — Optimal: Two-Pointer Greedy Match ← Interview Pick

# i: pointer into s — which character of s we need to match next
def isSubsequence(s: str, t: str) -> bool:
i = 0 # start at beginning of s
for char in t: # scan every character of t exactly once
if i < len(s) and char == s[i]: # guard: i in bounds + match
i += 1 # matched! advance to next needed character in s
return i == len(s) # True if all chars of s were matched
⚠️ Why the i < len(s) Guard?

Once i reaches len(s), we've matched everything — but the for-loop still continues scanning t. Without the guard, s[i] would raise an IndexError on the next iteration. Alternatively, you could break early for a micro-optimisation, but the guard is cleaner and Pythonic.

Solution 2 — Follow-Up: Binary Search for Multiple Queries

If we need to check many patterns against the same large t, precompute every character's positions and binary-search per lookup.

from collections import defaultdict
import bisect
def preprocess(t: str) -> dict:
positions = defaultdict(list) # char → sorted list of indices in t
for i, c in enumerate(t): # O(|t|) once
positions[c].append(i)
return positions
def isSubsequence_multi(s: str, positions: dict) -> bool:
curr = -1 # last matched position in t (must find AFTER this)
for c in s:
if c not in positions: return False # char never appears in t
# find first occurrence of c strictly after position curr
idx = bisect.bisect_right(positions[c], curr)
if idx == len(positions[c]): return False # no more c's left
curr = positions[c][idx] # advance past this match
return True
🔑 When Does Binary Search Win?

For a single query, the O(|t|) greedy is unbeatable. The binary search version pays off when you have k queries against the same t: O(|t|) preprocessing once, then O(|s| log |t|) per query instead of O(|t|). Total cost: O(|t| + k·|s|·log|t|) vs O(k·|t|) — binary search wins when |s| ≪ |t|.

Greedy Two-PointerO(|t|)O(1)Single query

5 🎤 Interview Strategy: How to Arrive at This ▲
Step 1 — Recognise the Pattern

The words "subsequence" and "relative order preserved" are your signals. Any time you need to match characters across two strings in order, think: one pointer per string, advance the pattern pointer only on a match.

Step 2 — Define What Each Pointer Means

Pointer i into s: "which character am I looking for next?" The for-loop over t plays the role of j: "where am I currently scanning?" This two-role separation is the key design decision.

Step 3 — Articulate the Greedy Argument

When you find a match, take it immediately. Waiting for a later occurrence of the same character in t can only reduce what's available for subsequent characters. This is the exchange argument, and it's worth stating aloud.

Step 4 — Anticipate the Follow-Up

Almost every interviewer who asks this problem will follow up with: "What if there are 10,000 queries against the same t?" Answer: precompute a position map for each character in t, then binary search per character of s. Mentioning this proactively earns bonus points.

🗣️ What to Say to the Interviewer

"Since s must appear in t in relative order, I can use a single pointer into s..."
"I'll scan t left-to-right. Each time I see t[j] == s[i], I advance i."
"The greedy choice is correct — taking the earliest match leaves the most of t available."
"This is O(|t|) time and O(1) space — one linear scan."
"For the follow-up with k queries: precompute char → positions, binary search each char of s."
💡 Follow-Up Questions to Expect
  • What if you have 10⁹ queries against the same t? → Binary search / precompute
  • Count the number of distinct subsequences in t that equal s? → DP (hard variant)
  • Find the minimum window in t containing s as a subsequence? → Sliding window
  • Shortest Way to Form String? → Count min copies of s needed

7 Summary & Key Takeaways ▲
Single pointer i into s + one linear scan of t = O(|t|) time, O(1) space
Greedy: always take the earliest possible match — never delay
Empty s → always True; empty t with non-empty s → always False
For k queries against same t: precompute positions + binary search

  • One string significantly shorter than the other
