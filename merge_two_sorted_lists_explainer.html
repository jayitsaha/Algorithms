LeetCode #21

Merge Two Sorted Lists

You are given the heads of two sorted linked lists list1 and list2. Merge them into one sorted list by splicing together their nodes. Return the head of the merged list.

Easy Linked List Recursion Time: O(m+n) Space: O(m+n)
1 Understanding the Problem ▾

You have two linked lists that are already sorted in ascending order. Your job is to weave them together into one single sorted list — without creating any new nodes. Just re-wire the existing next pointers to produce the merged result.

💡 Key Insight

At any moment, the problem reduces to one binary decision: which list's current head is smaller? That smaller node belongs next in the merged result. Then we recursively ask the same question on the rest of that list vs the other list. The problem structure mirrors the data structure: linked lists are recursive, and so is the merge.

✓ Example

list1: 1 → 2 → 4
list2: 1 → 3 → 4
─────────────────
Result: 1→1→2→3→4→4

Edge Cases

[] + [1,2] = [1,2]
[] + [] = []
[5] + [1] = [1,5]
Duplicates are fine!

Key constraint: no new nodes. You must re-link existing nodes by re-wiring their next pointers. No copying of values, no creating new ListNode objects. This makes in-place pointer manipulation the only valid approach.

2 The Algorithm: Recursive Merge ▾

We treat the merge as a chain of local decisions. Each recursive call makes one choice — which head wins — then delegates the rest to a smaller version of the same problem.

1 Base case — one list is null: If l1 is None, return l2. If l2 is None, return l1. The other list is already sorted — return it as-is. This terminates the recursion.
2 Compare the two heads: If l1.val ≤ l2.val, then l1's head is the next node in the merged list. Otherwise l2's head wins.
3 Wire the winner's next: Set winner.next = merge(winner.next, loser). We advance only the winner's pointer; the loser is passed forward unchanged. This is the recursive call.
4 Return the chosen head: Return the winning node. The caller will wire this node as its own next — building the chain from the bottom of the recursion upward.
5 Recursion unwinds: As the call stack collapses, each frame sets one next pointer and returns its head. The fully merged list is assembled in reverse-call order — bottom up.

Recursion tree for list1=[1,2,4], list2=[1,3,4]:

merge([1,2,4], [1,3,4])1≤1 → pick l1(1) → l1.next = merge([2,4],[1,3,4])
↓
merge([2,4], [1,3,4])2 > 1 → pick l2(1) → l2.next = merge([2,4],[3,4])
↓
merge([2,4], [3,4])2≤3 → pick l1(2) → l1.next = merge([4],[3,4])
↓
merge([4], [3,4])4 > 3 → pick l2(3) → l2.next = merge([4],[4])
↓
merge([4], [4])4≤4 → pick l1(4) → l1.next = merge(null,[4])
↓
merge(null, [4]) ← BASE CASE: return [4]
↑ unwind: 4→4 → 3→4→4 → 2→3→4→4 → 1→2→3→4→4 → 1→1→2→3→4→4
⚠️ We only advance ONE pointer per call

Each recursive call consumes exactly one node (the winner's head). The other list is passed forward intact. This guarantees termination: at each level one list shrinks by one, so after m+n calls one list is empty and we hit the base case.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every recursive call, both input lists are still sorted ascending — because we only advance one pointer at a time, and a sorted list minus its head is still sorted. The function merge(l1, l2) always returns a correctly sorted merged list of its two sorted inputs. This invariant holds at every level of the recursion.

Proof by structural induction:

Base: If either list is null, we return the other — which is already sorted. Trivially correct.

Inductive step: Assume merge(l1.next, l2) correctly returns a sorted list (smaller input — induction hypothesis holds). Since l1.val ≤ l1.next.val (sorted input) and l1.val ≤ l2.val (we chose l1 as the winner), we know l1.val is ≤ every value in the merged result. Setting l1.next = merge(l1.next, l2) gives a sorted list starting at l1. QED.

⚠️ Stack Depth is O(m+n)

Each of the m+n calls pushes one frame onto the call stack. For lists with millions of nodes, this causes a stack overflow. The iterative approach with a dummy head uses O(1) space and handles arbitrarily large inputs. Always mention this trade-off in interviews — it shows production awareness.

Why base case is safe: When one list runs out, we return the other list in its entirety — O(1) operation. The remaining list is already fully sorted, so no further processing is needed. No element is ever lost.

4 The Code (Annotated) ▾

Solution 1 — Recursive Merge (Interview Pick, elegant)

def mergeTwoLists(l1, l2):
if l1 is None: ← Base case: l1 exhausted; return l2 as-is
return l2
if l2 is None: ← Base case: l2 exhausted; return l1 as-is
return l1
if l1.val <= l2.val: ← l1's head is smaller or equal; it wins
l1.next = mergeTwoLists(l1.next, l2) ← Wire l1.next to rest of merge
return l1 ← Return l1 as head of this sub-merge
else: ← l2.val < l1.val; l2's head wins
l2.next = mergeTwoLists(l1, l2.next) ← l1 unchanged; advance l2
return l2 ← Return l2 as head of this sub-merge
⚠️ Forgetting the return after wiring

A very common mistake: writing l1.next = mergeTwoLists(l1.next, l2) but omitting return l1. Without the return, the function returns None and the chain breaks. Every branch must both wire .next AND return its chosen head to the caller.

Solution 2 — Iterative with Dummy Head (O(1) space, production-preferred)

def mergeTwoLists(l1, l2):
dummy = ListNode(0) ← Sentinel: avoids special-casing the head
cur = dummy ← cur = tail of growing result list
while l1 and l2: ← While both lists still have nodes
if l1.val <= l2.val:
cur.next = l1 ← Attach l1 node
l1 = l1.next ← Advance l1
else:
cur.next = l2 ← Attach l2 node
l2 = l2.next ← Advance l2
cur = cur.next ← Move result tail forward
cur.next = l1 or l2 ← Attach remaining list (already sorted)
return dummy.next ← Skip sentinel; first real node is the answer
💡 Why the dummy node?

Without the dummy, we'd need to special-case "what is the head of the result list?" — an awkward if-statement before the loop. The dummy absorbs that complexity: cur always starts somewhere valid. The real answer is dummy.next. The dummy node itself is discarded.

🔑 Complexity Summary

Recursive: Time O(m+n), Space O(m+n) — one call per node, call stack depth = m+n
Iterative: Time O(m+n), Space O(1) — only 3 pointer variables, no stack growth
Both visit every node exactly once and do O(1) work per node.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Merge Pattern

Two sorted sequences → one sorted sequence = merge. This is half of merge sort. Because both inputs are sorted, the next smallest element is always at one of the two heads — you never need to look backward.

Step 2: Identify the Recursive Subproblem

"If I pick the smaller head, what do I still need?" → Merge the rest of that list with the other list in full. Same problem, smaller input. The recursive structure is obvious once you see it.

Step 3: State Base Cases First

Always say base cases out loud before the recursion: "If l1 is null, I return l2. If l2 is null, I return l1." This grounds the recursion and prevents infinite loops — interviewers notice when you articulate this clearly.

Step 4: Proactively Offer the O(1)-Space Version

"The recursive version uses O(m+n) call stack space. I can also solve this iteratively with a dummy head for O(1) space, which is safer for very long lists." This shows you think beyond just making the tests pass.

🗣️ What to Say to the Interviewer

"Two sorted lists → this is a merge problem, like the merge step in merge sort.
Key insight: the next merged node is always the smaller of the two current heads.
Recursively: merge(l1, l2) = smaller_head, with smaller_head.next = merge(rest, other).
Base cases: null list → return the other list.
This is O(m+n) time, O(m+n) space due to recursion depth.
For O(1) space: iterative with a dummy head node — same time complexity."
