LeetCode #567

Permutation in String

Given strings s1 and s2, return true if s2 contains a permutation of s1 as a contiguous substring.

Medium Sliding Window Frequency Count Window Time: O(n) Space: O(1)
1 Understanding the Problem ▾

We have two strings s1 and s2. We need to determine: does any contiguous substring of s2 use exactly the same characters (with the same frequencies) as s1, just possibly in a different order?

A permutation of a string is any rearrangement of its characters. "abc", "bca", "cab" are all permutations of each other — they have the same letters, just in different orders. We are not asked to generate permutations (that would be factorial in s1's length); we just need to know if any one of them appears as a window in s2.

💡 Key Insight

A contiguous substring of s2 is a permutation of s1 if and only if it has the same length as s1 and the same character frequencies as s1. This means we can check by comparing frequency counts — no need to generate or sort permutations.

✓ s1="ab", s2="eidbaooo"

s2[3..4] = "ba"
need: a×1, b×1
window "ba": b×1, a×1
Frequencies match → True

✗ s1="ab", s2="eidboaoo"

All windows of length 2:
"ei","id","db","bo","oa","ao","oo"
None has a×1, b×1 together
→ False

Edge cases to consider: s1 longer than s2 (always False), s1 length 1 (just check if that character appears), identical strings (always True), all-same-character strings like "aaa" (only True if s1 is also all 'a's of the same length), and s1 containing duplicate characters — our frequency approach handles all of these naturally.

2 The Algorithm: Fixed-Size Sliding Window ▾

We slide a window of exactly k = len(s1) characters across s2, one position at a time. At each position, we check whether the window's character frequencies match s1's frequencies. To avoid recomputing from scratch at each position, we do incremental updates: add the new character entering from the right, remove the character leaving from the left.

1 Quick-fail on length: If len(s1) > len(s2), there cannot be any window of size k in s2 — return False immediately.
2 Build frequency arrays: Create need[26] from s1 and window[26] from the first k characters of s2. Both are fixed-size arrays indexed by ord(c) - ord('a').
3 Count initial matches: Count how many of the 26 character slots already agree: matches = sum(1 for i in range(26) if window[i] == need[i]). If matches == 26, the first window itself is a permutation — return True.
4 Slide and update: For each new right index r from k to len(s2)-1: add s2[r] (the incoming character) to window, then remove s2[r-k] (the outgoing character). After each change, update the matches counter by checking whether the match status of the affected slot changed.
5 Check every step: If matches == 26 after any slide, return True. After all slides, return False.
🔑 The Matches Counter: Why It's O(1) Per Step

Naively, comparing two 26-element arrays costs O(26). The matches counter eliminates that: we track how many slots currently agree as a single integer. Each character addition or removal can only affect one slot — so we check: "did this slot match before? Does it match now?" and adjust by ±1. Only two checks per character event, never a full comparison. True O(1) per step.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every point in the algorithm: window[c] always equals exactly the count of character c in the current window s2[left..right]. The matches counter always equals the number of character indices i (out of 26) for which window[i] == need[i]. When matches == 26, all character frequencies match — the window is a permutation of s1.

Why incremental updates preserve correctness: When the window slides one step, exactly two characters change — the one entering the right edge and the one leaving the left edge. All other 22+ characters' counts are unchanged. So the matches counter only needs at most 2 adjustments per slide. The window invariant is maintained because we always add before removing (or the order is consistent), keeping window[] accurate.

Why window size is always exactly k: We initialize with exactly k characters and slide both left and right pointers forward together — left follows right by exactly k positions. So the window never grows or shrinks below k. This is the key difference from a variable-size sliding window.

⚠️ Fixed vs Variable Window: Knowing Which to Use

This problem requires a fixed-size window because we need a substring of exactly len(s1) characters. Variable-size windows (expand right, shrink left when invalid) are used when you're optimizing the window's length — e.g., "longest substring without repeating characters" or "shortest window containing all characters of t." The trigger for fixed-size is: the target length is specified by the problem.

Correctness on edge cases: If s1 has a character that appears multiple times (e.g., s1="aab"), need[a]=2. The window must also have exactly 2 'a's. If the window has 3 'a's, need[a]=2 ≠ window[a]=3, that slot doesn't match — correctly detected. If the window has 1 'a', same: 2 ≠ 1 — also detected. Our counter approach handles multiplicities exactly.

4 The Code (Annotated) ▾

Solution 1 — Fixed-Size Sliding Window with Match Counter (Optimal)

def checkInclusion(s1: str, s2: str) -> bool:
if len(s1) > len(s2): ← Cannot fit s1 into s2 at all
return False
k = len(s1) ← Fixed window size
need = [0] * 26 ← Target: frequency of each char in s1
window = [0] * 26 ← Current: frequency of each char in window
for c in s1: ← Build need[] from s1
need[ord(c) - 97] += 1
for c in s2[:k]: ← Seed window with first k chars of s2
window[ord(c) - 97] += 1
matches = sum(1 for i in range(26) if window[i] == need[i])
← Count how many of 26 slots currently agree
if matches == 26: return True ← First window is already a permutation!
for r in range(k, len(s2)): ← Slide: r is the new right-edge index
ri = ord(s2[r]) - 97 ← Index of incoming character
if window[ri] == need[ri]: matches -= 1 ← Was matched before adding? Will now overshoot
window[ri] += 1
if window[ri] == need[ri]: matches += 1 ← Does adding it hit the target exactly?
li = ord(s2[r - k]) - 97 ← Index of outgoing (leftmost) character
if window[li] == need[li]: matches -= 1 ← Was matched before removing? Will now be under
window[li] -= 1
if window[li] == need[li]: matches += 1 ← Does removing it hit the target exactly?
if matches == 26: return True ← All 26 slots agree → permutation found!
return False ← Exhausted all windows, none matched
⚠️ Why Check-Before-Modify then Check-After-Modify?

The pattern is: check old state → modify → check new state. We must know whether the slot was matched before the change (to undo that match if needed) and whether it matches after the change (to record a new match). Checking after-then-modifying, or modifying then checking the old value, produces incorrect match counts. The 4-line structure per character (check, modify, check, optional adjust) is the canonical form for the matches-counter pattern.

Solution 2 — Counter Comparison (Simpler to Code)

from collections import Counter
def checkInclusion(s1: str, s2: str) -> bool:
k = len(s1)
if k > len(s2): return False
need = Counter(s1) ← Target frequency map
win = Counter(s2[:k]) ← Initial window frequency map
if win == need: return True
for r in range(k, len(s2)):
win[s2[r]] += 1 ← Add incoming character
out_c = s2[r - k] ← Character leaving the window
win[out_c] -= 1 ← Remove outgoing character
if win[out_c] == 0: del win[out_c] ← Keep Counter clean (avoid spurious 0 keys)
if win == need: return True ← O(k) comparison but k ≤ 26 letters so O(1)
return False
💡 Which version to code in an interview?

Start with the Counter version — it reads almost like pseudocode and is far easier to write correctly under pressure. The match-counter version is marginally faster in practice but harder to get right on a whiteboard. Tell the interviewer: "I can also use a 26-element array with a match counter to avoid the Counter comparison overhead — same O(n) asymptotic, smaller constant." Implement whichever they prefer.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Reframe the Problem

"A permutation = same characters, same counts. So I need a contiguous substring of s2 with the same character frequencies as s1. That substring must have exactly len(s1) characters. This is a fixed-size window problem."

Step 2: State the Brute Force

"Brute force: extract every substring of s2 with length len(s1), compute its frequency map, compare to s1's frequency map. O(n · k) time. For n=10^4 and k=10^3 this is 10^7 operations — probably acceptable but let's optimize."

Step 3: Propose the Sliding Window

"Instead of recomputing the frequency map from scratch, I maintain it as I slide. Each slide: add one character, remove one character. Total O(n) operations. I'll compare the window map to the target map each step — O(26) = O(1). Total O(n)."

Step 4: Mention the Matches Optimization

"To avoid even the O(26) comparison each step, I track an integer 'matches' = number of character slots that currently agree. On each slide, only the two affected slots need rechecking. When matches == 26, all slots agree — done."

🗣️ What to Say to the Interviewer

"Permutation means same character frequencies — I don't need to generate permutations.
I want a window in s2 of size len(s1) with matching frequencies. Fixed sliding window.
I'll build two 26-element arrays: need[] from s1, window[] from the first k chars of s2.
Track 'matches': how many of 26 slots currently agree. Slide one step at a time:
add incoming char (update window[], recheck that slot), remove outgoing char (same).
Return True when matches==26. Time O(n+k)=O(n). Space O(1) — two fixed arrays."

7 Summary & Key Takeaways ▾
A permutation = same character frequencies — never generate permutations explicitly
Fixed window size k = len(s1): the window never grows or shrinks
Incremental update: add incoming char, remove outgoing char — O(1) per slide
Match counter trick: check-before-modify, then check-after-modify — four lines per char
Two 26-element arrays = O(1) space regardless of input length
ApproachTimeSpaceNotes
Brute Force (recompute each window) O(n · k) O(k) Correct but slow for large k
Counter comparison (sliding) O(26n) O(1) Simple code; O(26) comparison per step
Match counter (fixed array) ✓ O(n) O(1) O(1) per step; interview pick
🎯 When to Use Fixed-Size Sliding Window
  • "Does s2 contain a permutation/anagram of s1?" → Fixed window of size len(s1)
  • "Maximum/minimum/sum over all windows of size k?" → Fixed window over array or string
  • "Find all occurrences of pattern (or its permutations) in text?" → Fixed window with freq check
  • Window length is given or directly derivable from constraints → Fixed-size template
  • Window length must be optimized → Variable-size template (different problem class)
