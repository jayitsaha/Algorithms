LeetCode #1438

Longest Continuous Subarray With Absolute Diff ≤ Limit

Find the longest contiguous subarray where max − min ≤ limit, using two monotonic deques to track range in O(1) per step.

Medium Monotonic Queue Two Deques for Range Time: O(n) Space: O(n)
1 Understanding the Problem ▾

Given an integer array nums and an integer limit, return the size of the longest contiguous subarray such that the absolute difference between any two elements in the subarray is at most limit.

Since we need the max absolute difference over all pairs in the window, and the max absolute diff always equals max(window) − min(window), the condition simplifies to: keep window valid while max − min ≤ limit.

💡 Key Insight

The only two values that determine whether a window is valid are its maximum and minimum. If we can track these two values as the window expands and contracts in O(1) per step, we get an O(n) sliding window. Two monotonic deques — one for max (decreasing), one for min (increasing) — do exactly this.

✓ Valid window: [4, 7, 3], limit=4

max = 7, min = 3
diff = 7 − 3 = 4 ≤ 4 ✓
Length = 3 → answer = 3

✗ Invalid window: [8, 2], limit=4

max = 8, min = 2
diff = 8 − 2 = 6 > 4 ✗
Window must shrink

Edge cases to consider: Single element (always valid, answer ≥ 1); all equal elements (entire array is valid); limit = 0 (only equal neighbors count); strictly increasing input where adjacent diff > limit (answer = 1); n = 1 trivially returns 1.

2 The Algorithm: Sliding Window + Two Monotonic Deques ▾

We maintain a variable-size sliding window [left, right] and two deques that always tell us the current window's max and min in O(1):

  • max_dq: stores indices in decreasing order of values. The front is always the index of the window's current maximum.
  • min_dq: stores indices in increasing order of values. The front is always the index of the window's current minimum.

1 Expand right: For each new index right, pop from the back of max_dq while nums[back] ≤ nums[right] (those elements can never be max while right is in window). Pop from back of min_dq while nums[back] ≥ nums[right]. Then append right to both.
2 Check validity: Window is valid iff nums[max_dq[0]] − nums[min_dq[0]] ≤ limit. The deque fronts give max and min in O(1).
3 Shrink left: If invalid, increment left. After incrementing, pop the front of each deque if its stored index fell below left (it's now out of window). Repeat until valid.
4 Update answer: After validating, result = max(result, right − left + 1). Every element is pushed and popped from each deque at most once → O(n) total.
🔑 Why Deques Beat Simple Variables

With plain cur_max and cur_min variables, expanding right is fine, but shrinking left could evict the current max or min — we'd need O(n) to re-scan for the new max/min. Deques solve this: when the front element leaves, the next-best candidate is already queued up at the new front. No re-scan needed.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At any point, both deques contain only indices within [left, right]. max_dq's values form a strictly decreasing subsequence so its front is always the maximum of the current window. min_dq's values form a strictly increasing subsequence so its front is always the minimum.

Why back-popping is safe: Before inserting index right into max_dq, we pop any back index j where nums[j] ≤ nums[right]. Index j < right, so j will always coexist in the window with right. Since nums[right] ≥ nums[j], index j can never be the window maximum while right is present — discarding it loses no information. Same logic applies to min_dq with ≥.

Why front-popping on shrink is safe: When left advances to left+1, any deque front whose stored index equals the old left is now outside the window. We pop it. The new front becomes the next-best candidate, and the invariant holds again.

Amortized O(n): Each of the n elements is appended to each deque exactly once and removed at most once. Total operations across all n iterations: O(n) per deque → O(n) overall.

⚠️ Classic Bug: Forgetting to Evict Stale Fronts

After incrementing left, you must check if max_dq[0] < left: max_dq.popleft() (and same for min_dq) before re-checking the diff. Forgetting this means the deque front is a ghost from outside the window, and your max/min will be wrong.

4 The Code (Annotated) ▾

Solution 1 — Two Monotonic Deques (Optimal, Interview Pick)

from collections import deque
def longestSubarray(nums, limit):
max_dq = deque() # indices, decreasing values ← front = current window max index
min_dq = deque() # indices, increasing values ← front = current window min index
left = result = 0 ← left pointer; track best answer
for right, val in enumerate(nums): ← expand window: right moves every step
while max_dq and nums[max_dq[-1]] <= val: ← back dominated? evict it (val is at least as large)
max_dq.pop()
max_dq.append(right) ← right becomes new back of max_dq
while min_dq and nums[min_dq[-1]] >= val: ← back dominated? evict it (val is at least as small)
min_dq.pop()
min_dq.append(right) ← right becomes new back of min_dq
while nums[max_dq[0]] - nums[min_dq[0]] > limit: ← window invalid: max - min too large
left += 1 ← shrink from the left
if max_dq[0] < left: max_dq.popleft() ← evict stale max if it left the window
if min_dq[0] < left: min_dq.popleft() ← evict stale min if it left the window
result = max(result, right - left + 1) ← window valid: update best length
return result ← longest valid subarray length
⚠️ Why ≤ for max_dq and ≥ for min_dq?

Using ≤ in max_dq means we pop equal elements from the back, keeping only the rightmost among equals. This is correct and slightly preferred because the rightmost equal element stays in the window longest as left advances. Using strict < also works — it just keeps more equal elements in the deque, which is harmless.

Solution 2 — SortedList (Cleaner Code, O(n log n))

from sortedcontainers import SortedList
def longestSubarray(nums, limit):
sl = SortedList() ← ordered structure; O(log n) insert/remove
left = result = 0
for right, val in enumerate(nums):
sl.add(val) ← O(log n) insert; maintains sorted order
while sl[-1] - sl[0] > limit: ← max = sl[-1], min = sl[0] in O(1)
sl.remove(nums[left]) ← O(log n) remove by value
left += 1
result = max(result, right - left + 1)
return result

Solution 3 — Brute Force (O(n²), for clarity only)

def longestSubarray(nums, limit):
n, result = len(nums), 0
for i in range(n): ← try every left boundary
cur_max = cur_min = nums[i]
for j in range(i, n): ← extend right as far as valid
cur_max = max(cur_max, nums[j])
cur_min = min(cur_min, nums[j])
if cur_max - cur_min > limit: break ← once invalid, can only get worse
result = max(result, j - i + 1)
return result
💡 Complexity Summary

Two Deques (optimal): Time O(n), Space O(n) — each element enters/exits each deque at most once
SortedList: Time O(n log n), Space O(n) — O(log n) per insert/remove
Brute Force: Time O(n²), Space O(1) — only for understanding; fails on large inputs

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Pattern — "Longest Subarray with Constraint"

Whenever a problem asks for the longest subarray satisfying a monotone constraint, think sliding window. The constraint here (max − min ≤ limit) is monotone: if a window violates it, making it larger only makes things worse. So we can greedily extend right and shrink left when needed.

Step 2: Identify the Bottleneck — Tracking Range After Shrink

The trick is that when we shrink from the left, we might remove the current max or min. Naively, we'd re-scan the window: O(n) per shrink → O(n²) total. We need a data structure that gives us the new max/min instantly after the left element leaves.

Step 3: Propose Two Monotonic Deques

A monotonic deque (like in Sliding Window Maximum, LC #239) tracks the running maximum of a window. We need two of them: one for max (decreasing), one for min (increasing). The front of each is always the max/min of the current window. Each element enters and exits each deque at most once → O(n).

Step 4: Mention the SortedList Alternative

If the interviewer wants simpler code at the cost of O(n log n): use Python's SortedList. Insert/remove in O(log n), but sl[-1] − sl[0] gives the range in O(1). Great fallback if you blank on deque invariants under pressure.

🗣️ What to Say to the Interviewer

"This is a sliding window problem: find the longest subarray satisfying a range constraint.
I'll use two pointers — expand right greedily, shrink left when max − min exceeds limit.
The challenge: tracking max and min as the window changes. Re-scanning is O(n²).
Instead, I'll maintain two monotonic deques: one decreasing for max, one increasing for min.
Front of each deque is always the window's max/min. Each element enters/exits at most once → O(n).
When I shrink left, I pop stale deque fronts (indices < left) before checking validity again."
🔮 Likely Follow-up Questions
  • "Return the actual subarray?" → Record (left, right) at the point where result updates

  • "What if limit changes per query?" → Segment tree / sparse table for range max/min queries

from collections import deque
def longestSubarray(nums, limit):
max_dq, min_dq = deque(), deque()
left = result = 0
for right, val in enumerate(nums):
while max_dq and nums[max_dq[-1]] <= val: max_dq.pop()
max_dq.append(right)
while min_dq and nums[min_dq[-1]] >= val: min_dq.pop()
min_dq.append(right)
while nums[max_dq[0]] - nums[min_dq[0]] > limit:
left += 1
if max_dq[0] < left: max_dq.popleft()
if min_dq[0] < left: min_dq.popleft()
result = max(result, right - left + 1)
return result
