LeetCode #56

Merge Intervals

Given an array of intervals where intervals[i] = [start_i, end_i], merge all overlapping intervals and return an array of the non-overlapping intervals that cover all the intervals in the input.

Medium Intervals Sort + Extend Time: O(n log n) Space: O(n)
1 Understanding the Problem ▾

You are given a list of intervals on a number line. Two intervals overlap when one begins before the other ends. Your task: collapse every set of overlapping intervals into a single interval, returning the minimum number of intervals that together cover exactly the same range as the original input — with no overlaps between any two.

💡 Key Insight

If we sort intervals by start time, overlapping intervals become adjacent in the array. We never need to look forward or backward more than one step — a simple left-to-right scan is enough. This turns an O(n²) pair-checking problem into an O(n log n) sort-then-scan.

✗ The containment pitfall

Input: [[1,10],[2,3]]
[2,3] is inside [1,10].
Naive: merged[-1][1] = 3 ← WRONG
Correct: merged[-1][1] = max(10,3) = 10
Always use max() not plain assignment.

Edge Cases: Empty input (return []), single interval (return as-is), all intervals overlapping into one giant interval, all intervals disjoint (return sorted copy), intervals touching at one point like [1,3] and [3,5] — these share the endpoint so they merge to [1,5] with the standard ≤ check.

2 The Algorithm: Sort Then Scan and Extend ▾

After sorting by start time, we maintain a "current merge window" — the last element of our result. For each new interval, we ask one question: does it overlap the current window? If yes, extend the window. If no, commit the window and start a new one.

1 Sort by start: intervals.sort(key=lambda x: x[0]). After sorting, any interval that overlaps with the previous one has its start between the previous start and end. Overlaps become strictly adjacent — no skipping needed.
2 Seed the result: merged = [intervals[0]]. Start with the first interval as the initial merge window. The loop begins at index 1.
3 For each remaining interval [s, e]: Compare s against merged[-1][1] (the end of the current merge window).
4 If s ≤ merged[-1][1] (overlap or touch): Extend the window. Set merged[-1][1] = max(merged[-1][1], e). The max is critical — it handles the containment case where the new interval ends before the current window does.
5 Else (gap): merged.append([s, e]). The current window is complete. Start a fresh window with the new interval.
6 Return merged: After scanning all intervals, the result list holds the minimum set of non-overlapping intervals covering all the input.
🔑 Why Sort by Start (Not End)?

If we sorted by end time, non-overlapping intervals could become adjacent in the sorted order and trigger false merges. Sorting by start is the only ordering that guarantees "all intervals that could overlap with the current window appear immediately after it in the list." This is the structural backbone of the algorithm.

⚠️ In-Place Sort Modifies the Input

intervals.sort(...) mutates the caller's list. In interviews, say: "I'm sorting in-place for O(log n) sort-stack space; if immutability matters I'd use sorted(intervals, key=lambda x: x[0]) which returns a new list." The rest of the algorithm (writing to merged) doesn't touch the input.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every step of the scan, merged is a sorted list of non-overlapping intervals that together cover exactly the same range as all the input intervals processed so far. The last element merged[-1] is the "active window" still being extended.

Why sorting makes overlaps adjacent: After sorting by start time, suppose interval B overlaps with interval A (B.start ≤ A.end). Then B.start ≥ A.start (sort order), and B appears right after A in the list. No interval C can be between A and B in sorted order while overlapping A but not being adjacent to it, because C.start ≥ A.start and C.start ≤ B.start — C would either also overlap A (appear before B) or not (appear after B). Overlapping intervals form a contiguous block.

Why max() is the right choice, not assignment: If [2,3] arrives when the current window is [1,10], we have: 2 ≤ 10 (overlap). If we did merged[-1][1] = 3, the window shrinks from [1,10] to [1,3] — catastrophically wrong. max(10, 3) = 10 correctly preserves the wider end. The merged window can only grow, never shrink.

Why no backward scan is needed: Once we commit an interval to merged (by opening a new window), no future interval can overlap with it. Because we sorted by start, all future intervals have start ≥ current start ≥ the committed interval's end. The algorithm is strictly one-directional — O(n) after the sort.

⚠️ The Touching-Endpoint Question

The condition s ≤ merged[-1][1] (≤, not <) means that [1,3] and [3,5] merge into [1,5]. This is the standard interpretation: closed intervals include their endpoints, and sharing an endpoint counts as overlapping. If a problem uses open intervals or requires strict overlap, change to s < merged[-1][1]. Always clarify with the interviewer.

4 The Code (Annotated) ▾

Solution 1 — Sort + Scan (Optimal, Interview Pick) ✓

def merge(intervals): ← takes list of [start, end] pairs
intervals.sort(key=lambda x: x[0]) ← sort by start time; makes overlaps adjacent
merged = [intervals[0]] ← seed result; loop starts at index 1
for s, e in intervals[1:]: ← unpack each subsequent interval
if s <= merged[-1][1]: ← overlap or touch? new start ≤ current end
merged[-1][1] = max(merged[-1][1], e) ← extend; max handles containment case
else: ← gap: new start > current end
merged.append([s, e]) ← commit current; start new merge window
return merged ← minimum non-overlapping intervals
💡 Why [intervals[0]] and not []?

Seeding merged with the first element lets the loop body safely do merged[-1] without any empty-list guard. The loop starts at intervals[1:], so we process all intervals exactly once. If we started with merged = [], we'd need an extra if not merged check inside the loop — unnecessary complexity.

Solution 2 — Brute Force (conceptual only, O(n²))

def merge_brute(intervals): ← O(n²) — do not use in interviews
intervals.sort(key=lambda x: x[0]) ← still sort first for correctness
result = []
for i, (s, e) in enumerate(intervals):
if result and s <= result[-1][1]: ← overlap check (same as optimal)
result[-1][1] = max(result[-1][1], e)
else:
result.append([s, e])
return result ← actually same as optimal! Sort is key.
💡 The Real Brute Force Would Be...

The true O(n²) brute force would check every pair (i, j) for overlap without sorting. With sorting, the "brute force" and "optimal" are actually the same linear scan — the sort is what makes it efficient. The only way to avoid the O(n log n) sort is if intervals arrive pre-sorted, making the full algorithm O(n).

💡 Complexity

Time: O(n log n) — dominated by the sort; the scan is O(n).
Space: O(n) for the output array merged; O(log n) stack for the sort (in Python's Timsort).

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Interval Signal

The words "overlapping intervals" and "merge" are the clearest signal of the interval pattern. The mental trigger is: "Can I sort these to make overlaps adjacent?" Yes — sort by start time. This insight should come immediately.

Step 2: Draw a Number Line

Literally sketch 3–4 intervals on a number line on paper (or whiteboard). Ask yourself: "After sorting, how do I know if two adjacent intervals overlap?" Answer: their starts and ends tell me. This grounds the abstract problem in geometry.

Step 3: State the Key Decision

"At each step, there are exactly two cases: overlap (extend) or gap (start new group). That's it. The ≤ check and max() extension are the two lines that handle all sub-cases including containment and touching."

Step 4: Prove the max() Case Explicitly

"Consider [[1,10],[2,3]]. After sorting, 2 ≤ 10 so we overlap. If I wrote merged[-1][1] = 3, the result would be [1,3] — losing the [4,10] range. That's why I need max(10, 3) = 10."

🗣️ What to Say to the Interviewer

"I'll sort intervals by start time — O(n log n).
After sorting, overlapping intervals are adjacent, so I can do a single linear scan.
I'll keep a result list seeded with the first interval.
For each subsequent interval: if its start is ≤ the current end, they overlap — I extend using max() to handle containment. Otherwise, I append a new interval.
Final result: O(n log n) time, O(n) space. The key subtlety is using max(), not assignment, when extending."
🔮 Likely Follow-up Questions
  • "Insert a new interval into an already-merged list?" → LeetCode #57 — binary search + merge; or linear scan for O(n)
  • "Minimum meeting rooms needed?" → LeetCode #253 — sort start/end separately, or min-heap on end times
  • "What if intervals arrive as a data stream?" → Use a sorted structure (SortedList or BST) for O(log n) insertion
  • "Can you do it without sorting?" → Only if intervals are pre-sorted; otherwise O(n log n) is optimal

🎯 When to Recognize the Interval Sort + Extend Pattern

  • "Count non-overlapping intervals" → Sort by end + greedy skip
  • "Minimum rooms / resources for all intervals" → Sort + min-heap on end times
