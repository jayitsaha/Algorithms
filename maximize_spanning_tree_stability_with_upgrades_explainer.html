Maximize Spanning Tree Stability with Upgrades

Given a weighted undirected graph and k upgrade tokens, maximize the minimum edge weight in any spanning tree — the "stability" of the most stable spanning tree achievable.

Hard Graph Binary Search on Answer Kruskal's MST Time: O(E log E · log W) Space: O(N + E)
1 Understanding the Problem ▾

You have a weighted undirected graph with n nodes and m edges. Each edge has a stability weight. You have k upgrade tokens — each token lets you increase any edge's weight by a fixed amount delta. You want to find a spanning tree (connecting all nodes) that maximizes the minimum edge weight in that tree. The answer is the maximum achievable "minimum edge" across all possible uses of upgrades and all possible spanning trees.

💡 Key Insight

The "maximize the minimum" phrasing is the strongest signal in this problem. It screams Binary Search on Answer: instead of directly constructing the optimal spanning tree, guess a target minimum weight T and ask: "Can I use at most k upgrades to make a spanning tree exist where every edge weighs ≥ T?" Binary search over T.

The feasibility question: For a given target T, an edge is "viable" if its original weight ≥ T (free to use), or if its weight + delta ≥ T (costs one upgrade token). Use Kruskal's algorithm considering only viable edges — if a spanning tree forms using ≤ k upgrades, T is feasible.

Edge cases: If the graph is already disconnected (no spanning tree possible at all), return -1. If k = 0, the answer is the minimum edge of a maximum spanning tree (found by standard Kruskal in descending order). If all edges can be upgraded past any T, the answer grows unboundedly (though in practice weights are bounded).

2 The Algorithm: Binary Search + Kruskal's MST ▾

The approach has two interlocking parts: Binary Search narrows down the target minimum weight T, and a modified Kruskal's checks feasibility for each T by greedily building a spanning tree.

1 Set search bounds: lo = min possible edge weight (or 0), hi = max edge weight + k * delta. These are the extremes of what T could be. Binary search between them.
2 Binary search on T: Compute mid = (lo + hi) // 2. Call feasible(mid). If feasible, lo = mid + 1 (try higher). Otherwise hi = mid - 1 (try lower). Repeat until lo > hi. Answer = lo - 1.
3 Classify edges for target T: An edge with weight w is "free" if w ≥ T (no upgrade needed). It is "upgradeable" if w + delta ≥ T (costs one token). Otherwise it's unusable for this T.
4 Greedy Kruskal (modified): First, try to build a spanning tree using only FREE edges (sorted by weight descending, so we prefer stronger edges). Then, fill any remaining disconnected components using UPGRADEABLE edges, counting each one used. If we connect all n nodes using ≤ k upgrades, feasible = True.
5 Union-Find (DSU): Use path-compressed Union-Find to efficiently track connected components during Kruskal's. Each union operation merges two components. A spanning tree needs exactly n-1 unions.
🔑 Why Binary Search Works Here

The feasibility function is monotone: if target T is achievable, then any T' < T is also achievable (lower bar = easier). This monotonicity property is precisely the condition required for binary search to work. The search finds the boundary between feasible and infeasible.

📐 Feasibility Check Formula
For target T:
free_edges = {(u,v,w) : w ≥ T} ← no upgrade needed
upgrade_edges = {(u,v,w) : w+delta ≥ T, w < T} ← costs 1 token
Run Kruskal on free_edges first, then on upgrade_edges.
Count upgrades_used ≤ k AND components == 1 → feasible(T) = True
3 Why It Works: The Invariant ▾
🔐 The Binary Search Invariant

After every iteration, the invariant is: every T ≤ lo-1 is feasible, and every T ≥ hi+1 is infeasible. When lo > hi, the answer is lo-1 (the largest feasible T).

Why Kruskal's is valid here: Kruskal's algorithm works because greedily picking the highest-weight edge that doesn't form a cycle produces an optimal (maximum weight) spanning tree. The modification: we run two passes — first with free edges (no cost), then with upgrade edges (with cost) — and this greedy order minimizes the number of upgrades needed to span the graph.

⚠️ Greedy Order for Upgrades

We must try free edges first, then upgrade edges. If we used an upgrade when a free edge could have served, we'd waste upgrade tokens. Kruskal's greedy order naturally handles this — free edges (higher effective weight for feasibility) come first, upgrade edges fill in the gaps.

Why we don't need to choose WHICH edges to upgrade: For a given T, an edge either CAN be upgraded to meet T (costs 1 token) or it CANNOT. We just need to know: is there a spanning tree where the non-free edges number ≤ k? The greedy Kruskal minimizes upgrades used, so if any valid plan exists, the greedy finds one.

💡 The Monotonicity Proof

If feasible(T) = True, then for any T' < T: every edge that was usable for T is still usable for T' (since a lower threshold lets MORE edges qualify). So feasible(T') = True as well. This strict monotonicity is why binary search converges to the exact boundary.

4 The Code (Annotated) ▾

Solution 1 — Binary Search + Kruskal's MST (Optimal, Interview Pick)

def maxStability(n, edges, k, delta):
# Union-Find helpers
def find(p, x): ← Path-compressed find
while p[x] != x: p[x] = p[p[x]]; x = p[x]
return x
def union(p, rank, a, b): ← Union by rank
a, b = find(p, a), find(p, b)
if a == b: return False ← Already connected — skip
if rank[a] < rank[b]: a, b = b, a
p[b] = a; rank[a] += (rank[a] == rank[b]) ← Merge smaller under larger
return True
def feasible(target): ← Can we build ST with min weight ≥ target?
p = list(range(n)); rank = [0] * n
free = [(u,v,w) for u,v,w in edges if w >= target] ← No upgrade needed
upg = [(u,v,w) for u,v,w in edges if w+delta >= target > w] ← Costs 1 token
upgrades_used = 0
for u,v,w in sorted(free, key=lambda e:-e[2]): ← Kruskal: free edges first (desc weight)
union(p, rank, u, v)
for u,v,w in sorted(upg, key=lambda e:-e[2]): ← Then upgrade edges (desc weight)
if union(p, rank, u, v):
upgrades_used += 1 ← Count each upgrade actually used
roots = len(set(find(p, i) for i in range(n)))
return roots == 1 and upgrades_used <= k ← Spanning tree formed within budget
lo = min(w for _,_,w in edges)
hi = max(w for _,_,w in edges) + k * delta ← Max possible T after k upgrades
while lo <= hi:
mid = (lo + hi) // 2
if feasible(mid): lo = mid + 1 ← T is achievable; try higher
else: hi = mid - 1 ← T too high; try lower
return hi # = lo - 1 after loop ← Largest feasible T
⚠️ Why return hi (not lo)?

After the binary search loop, lo = hi + 1. The last feasible T is lo - 1 = hi. Returning hi (rather than lo) gives the correct answer — the maximum T for which the spanning tree is still achievable.

Solution 2 — Brute Force: Try All Upgrade Combinations (Exponential)

from itertools import combinations
def maxStabilityBrute(n, edges, k, delta):
best = 0
for num_upgrades in range(k+1): ← Try 0, 1, ..., k upgrades
for chosen in combinations(range(len(edges)), num_upgrades): ← Every C(m,i) subset
new_edges = [list(e) for e in edges]
for idx in chosen:
new_edges[idx][2] += delta ← Apply upgrade to chosen edges
best = max(best, max_spanning_min(n, new_edges)) ← Run max-min MST
return best ← O(2^m * m log m) — only for tiny graphs
💡 Complexity Summary

Binary Search + Kruskal: O(E log E · log W) where W = weight range.
Binary search: O(log W) iterations. Each feasibility check: O(E log E) for sorting + O(E α(N)) for DSU.
Brute Force: O(2^E · E log E) — infeasible for large inputs.
Space: O(N + E) for DSU + edge lists.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Spot "Maximize the Minimum" — This Means Binary Search

Any time you see "maximize the minimum" or "minimize the maximum," binary search on the answer is almost always the right approach. The answer space forms a sorted range; binary search the boundary. Say this explicitly to the interviewer.

Step 2: Formulate the Feasibility Check

Ask: "If I fix a target T, what does it take to check if it's achievable?" For T: (1) edges with w ≥ T are free, (2) edges with w+delta ≥ T cost 1 upgrade, (3) others are unusable. Build a spanning tree using these — if it exists with ≤ k upgrades, T is feasible.

Step 3: Choose Kruskal's for the Inner Check

The feasibility check is a spanning tree problem with two tiers of edges. Kruskal's greedy works perfectly: use free edges first, then upgrade edges (minimizing cost). Union-Find gives O(α(n)) per operation. This is the standard MST toolkit.

Step 4: Set Binary Search Bounds Carefully

lo = min edge weight (smallest achievable answer — no upgrades), hi = max edge weight + k*delta (upper bound — upgrade every edge maximally). Binary search between these. After the loop, hi = answer.

🗣️ What to Say to the Interviewer

"I notice this asks to maximize the minimum edge weight — a classic binary search on answer signal.
I'll binary search on the target T: lo = min weight, hi = max weight + k * delta.
For each T, I check: can I form a spanning tree where every edge has weight ≥ T, using at most k upgrades?
To check: free edges (weight ≥ T) are used first in Kruskal's, then upgrade edges (weight + delta ≥ T) as needed.
If we connect all nodes with ≤ k upgrades used, T is feasible. Return hi after binary search.
Time: O(E log E · log W), Space: O(N+E)."
🔮 Likely Follow-up Questions
  • "What if upgrades have different costs?" → Turn into a weighted coverage problem; use Dijkstra-like approach on upgrade budget.
  • "What if we want the maximum spanning tree total weight instead?" → Standard Kruskal's max-MST (sort descending, no binary search needed).
  • "Can you upgrade edges more than once?" → Each upgrade still costs 1 token; k tokens → k total upgrades (possibly same edge multiple times). Adjust delta calculation.
  • "What if the graph might be disconnected?" → Check roots == 1 in feasibility; return -1 if even at T = min weight, no spanning tree exists.

7 Summary & Key Takeaways ▾
"Maximize the minimum" → Binary Search on Answer (the most important recognition)
Feasibility = can we build a spanning tree with min edge ≥ T using ≤ k upgrades?
Two-pass Kruskal: free edges first, then upgrade edges — minimizes token cost greedily
Union-Find with path compression + union by rank gives near-O(1) per merge
After binary search: answer is hi (= lo-1), the last feasible threshold
Monotonicity of feasibility is what justifies binary search — always verify this property first

  • A feasibility check is easy to implement (spanning tree existence)
  • The feasibility function is monotone in the answer parameter
