LeetCode #700

Search in a Binary Search Tree

Given the root of a BST and an integer val, find and return the node whose value equals val (returning the entire subtree rooted there). Return null if it does not exist.

Easy Trees BST Property Time: O(h) Space: O(h)
1 Understanding the Problem ▾

You are given a Binary Search Tree (BST) and a target integer val. Find the node whose value equals val and return the entire subtree rooted at that node. If no such node exists, return null.

The return value is a TreeNode, not a boolean — you hand back the subtree root, which includes that node's left and right children too. This is important: if val=2 exists in the tree, returning node(2) implicitly includes all of node(2)'s descendants.

💡 Key Insight: The BST Property is a Built-In GPS

A Binary Search Tree guarantees: every value in the left subtree is strictly less than the current node's value, and every value in the right subtree is strictly greater. This holds recursively for every subtree — not just the root. This invariant means at each node you can immediately eliminate one entire half of the remaining tree. You never need to look both ways.

The BST ordering rule, visualised:

LEFT subtree  → all values < root.val
RIGHT subtree → all values > root.val
Example tree with root=4: {1,2,3} are left; {5,6,7} are right
Search val=2: 2<4 → go left. 2==2 → found!
✓ Found: val=2, tree=[4,2,7,1,3]

root=4: 2 < 4 → go LEFT
node=2: 2 == 2 → MATCH
Return subtree [2,1,3]

✗ Not Found: val=5, tree=[4,2,7,1,3]

root=4: 5 > 4 → go RIGHT
node=7: 5 < 7 → go LEFT
node=None → return null

Edge Cases: tree is empty (root=null → return null), target equals the root value (O(1) answer), target lives at the deepest leaf, target doesn't exist anywhere (return null after exhausting a branch).

2 The Algorithm: Compare and Go Left/Right ▾

At every node we make exactly one three-way comparison. The BST property removes all ambiguity — there is never a reason to explore both children simultaneously.

1 Null base case: If curr is None, we have fallen off the tree. The value doesn't exist here — return None.
2 Match: If curr.val == val, this is our target node. Return curr — the caller gets the subtree rooted here.
3 Go Left: If val < curr.val, the BST property guarantees the right subtree contains only values ≥ curr.val > val. Ignore the right entirely — follow curr.left.
4 Go Right: If val > curr.val, the left subtree contains only values ≤ curr.val < val. Ignore the left — follow curr.right.

Complete decision table at each node:

curr is None→return None   (not found)
curr.val == val→return curr   (found!)
val < curr.val→go LEFT     (smaller values there)
val > curr.val→go RIGHT    (larger values there)
🔑 Iterative Variant — O(1) Space

The recursive version above uses O(h) call-stack space. Translating to iterative: use a curr pointer and a while curr: loop. Move curr = curr.left or curr.right each iteration. When the loop exits (curr is null), the value wasn't found. This is the preferred form when tree height could be large (degenerate / very deep tree).

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every step of the search, the following is always true: if val exists anywhere in the original BST, it exists within the current subtree being examined. We never accidentally discard the subtree that contains the answer.

Why we can safely prune entire subtrees: The BST property is a global recursive guarantee, not just a local one. When we say val < curr.val, we know every single node reachable through curr.right has a value ≥ curr.val > val. There is no node hiding on the right that could equal val. We don't need to check — we can skip it completely, safely.

Why this yields O(h) complexity: Each comparison moves exactly one level deeper in the tree and eliminates all nodes on the other side of that level. In the worst case (searching for a missing value or the deepest leaf) we traverse from root to leaf — h levels. For a balanced BST, h = O(log n), giving O(log n) search. For a degenerate (skewed) BST, h = O(n), giving O(n) worst case — the same as an unsorted list.

⚠️ Common Trap: Using This Logic on a Non-BST

This "go left if smaller, go right if larger" logic only works if the tree is a valid BST. On a plain binary tree with no ordering guarantee, you must visit every node (DFS or BFS), giving O(n). Always confirm in an interview: "Is this a BST or just a binary tree?" — they look the same in code but require completely different search strategies.

Edge case handling by the invariant:

  • Empty tree (root=null): Invariant holds vacuously — nothing to search → return null immediately.
  • Target = root.val: Match on the first comparison. O(1) — single node visited.
  • Target not in tree: We eventually hit a null child. Invariant is maintained all the way down — we only stopped because there is genuinely nothing left to search.
4 The Code (Annotated) ▾

Solution 1 — Recursive (clean, interview-friendly)

def searchBST(root, val):
if not root: ← Null node: val is not in this subtree. Return None, not False.
return None
if root.val == val: ← Found the node. Return it (and its subtree) to the caller.
return root
if val < root.val: ← Target smaller: BST guarantees right side has no answer.
return searchBST(root.left, val)
return searchBST(root.right, val)← Target larger: BST guarantees left side has no answer.
⚠️ Return the node, not root.val

The problem asks for the subtree rooted at the matched node. Returning root (the TreeNode) is correct. Returning root.val would be wrong — it loses all the children. Also, never return False for "not found": the return type is Optional[TreeNode], so use None.

Solution 2 — Iterative (O(1) space — preferred for very deep trees)

def searchBST(root, val):
curr = root ← Start at root. We'll walk curr down the tree.
while curr: ← Keep going as long as curr is not null.
if curr.val == val: ← Match! Return this node (and its subtree).
return curr
elif val < curr.val: ← Go left: target is smaller, right side is useless.
curr = curr.left
else: ← Go right: target is larger, left side is useless.
curr = curr.right
return None ← Loop exited (curr is null): val not found in the tree.
💡 Complexity Summary

Recursive: Time O(h), Space O(h) — each recursive frame sits on the call stack.
Iterative: Time O(h), Space O(1) — single pointer, no extra memory.
h = O(log n) for a balanced BST; h = O(n) worst case (degenerate / sorted-input tree).
In practice, mention iterative when the interviewer asks "can you do better on space?"

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Spot the BST Signal

Any problem that says "Binary Search Tree" (not just "binary tree") is hinting that you should exploit ordering. Ask yourself: "What is the BST property buying me here?" For search, it eliminates one entire subtree per comparison — that is the whole win.

Step 2: Mention the Brute-Force First

"If this were a plain binary tree, I'd have to DFS and check every node — O(n). But because it's a BST, I can exploit the left < root < right ordering to skip one subtree at every level. That gives O(h)." This shows you understand why BST search is efficient.

Step 3: Code Recursive, Offer Iterative

Write the clean 4-line recursive solution. Then say: "If the tree could be very deep or degenerate, I can convert to an iterative while-loop that uses O(1) space instead of O(h) call-stack space." Code it if asked.

Step 4: Qualify Your Complexity

Never just say "O(log n)" for a BST problem. Always say: "O(h) where h is the tree height — O(log n) for a balanced BST, O(n) for a degenerate one." This shows depth of understanding.

🗣️ What to Say to the Interviewer

"Since this is a BST, I can use the ordering property to cut the search space in half at each node.
At each node: if the value matches, I return that node.
If my target is smaller than the current value, it can only be in the left subtree — go left.
If it's larger, it can only be in the right subtree — go right.
Base case: null node means not found, return null.
Time O(h), Space O(h) recursive. I can make it O(1) space with an iterative version."
🔮 Likely Follow-up Questions

7 Summary & Key Takeaways ▾
BST search is binary search on a pointer-linked tree — same O(log n) average principle
Three-way comparison at each node: equal → return, smaller → left, larger → right
Return the node (subtree root), not just the value — the caller may need its children
Recursive is elegant; iterative (while loop) saves O(h) call-stack space
Always qualify: "O(h) = O(log n) balanced, O(n) degenerate"

🎯 When to Recognise the BST "Compare and Go Left/Right" Pattern
  • Problem explicitly says "Binary Search Tree" (not just "binary tree")
  • You need to find, insert, delete, or validate a specific value by magnitude
  • You need O(h) instead of O(n) — the ordering is the performance lever
  • Each decision is a three-way comparison: match / go-left / go-right
