Find Minimum in Rotated Sorted Array

A sorted array was rotated between 1 and n times. Find the minimum element in O(log n) time.

Medium Binary Search Find Pivot Point Time: O(log n) Space: O(1)
1 Understanding the Problem ▾

Start with a sorted array: [1, 2, 3, 4, 5, 6, 7]. A rotation picks a point and moves the front chunk to the back. Rotating 4 positions gives [5, 6, 7, 1, 2, 3, 4]. We need to find the minimum element (1) in O(log n) time — we can't just scan linearly.

After rotation, the array forms two ascending sub-sequences separated by a "drop". The minimum is always the first element of the lower right sub-sequence — the exact point where the value drops. The amber cells are "large" (left half), the red cell is the minimum (pivot), and blue cells are "small" (right half).
💡 Key Insight

A rotated sorted array consists of two sorted ascending runs. The minimum is the first element of the second (smaller) run — the pivot where the drop occurs. Binary search can locate this drop in O(log n) by deciding at each midpoint which run we're in.

✓ Rotated: [4, 5, 6, 7, 0, 1, 2]

Left run: [4,5,6,7] — all large
Right run: [0,1,2] — all small
Drop at index 4 → minimum = 0

No Rotation: [1, 2, 3, 4, 5, 6, 7]

Entire array is one ascending run
No drop — nums[hi] is always ≥ nums[mid]
Algorithm returns nums[0] = 1 ✓

Edge Cases: Single element → return it directly. Array not rotated → minimum at index 0. Rotated by n → same as not rotated. All elements are distinct (no duplicates in this problem).

2 The Algorithm: Binary Search on the Pivot ▾

The elegant observation: at any midpoint, compare nums[mid] to nums[hi]. Because the array has two sorted runs, this single comparison tells us unambiguously which run mid falls in, and therefore which side the minimum is on.

nums[mid] > nums[hi] → mid in LEFT run

Mid's value exceeds hi's value, so mid is in the large left run. The minimum is strictly to the right of mid. Move lo = mid + 1 (safe to exclude mid).

nums[mid] ≤ nums[hi] → mid in RIGHT run

Mid is in the small right run (or the array isn't rotated). Mid could be the minimum itself! Move hi = mid (keep mid in the search window).

1 Initialize pointers: lo = 0, hi = len(nums) - 1. The search window covers the entire array.
2 Compute mid each iteration: mid = (lo + hi) // 2. This is the candidate we'll inspect to decide which half to keep.
3 Compare and eliminate: If nums[mid] > nums[hi], the minimum is right of mid → set lo = mid + 1. Otherwise, minimum is at mid or left → set hi = mid. Window shrinks by at least half each step.
4 Return when lo == hi: At this point, only one candidate remains. It must be the minimum. Return nums[lo].
🔑 Why Compare Mid to Hi (not lo)?

If we compared nums[mid] to nums[lo], then when nums[mid] > nums[lo], mid could be in either run — ambiguous! But comparing to nums[hi] is always clean: since nums[hi] is at the rightmost position of a two-run array, if nums[mid] > nums[hi], mid is definitely in the large left run. No ambiguity.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At the start of every iteration, the minimum element is guaranteed to be within nums[lo..hi] (inclusive). The window always contains the answer and only shrinks — never jumps over the minimum.

Case 1 — nums[mid] > nums[hi]: The entire sub-array from lo to mid (inclusive) is in the left (large) run. All values from lo to mid are larger than nums[hi], which is larger than the minimum in the right run. Therefore, neither mid nor anything to its left can be the minimum. Setting lo = mid + 1 is safe — the invariant holds.

Case 2 — nums[mid] ≤ nums[hi]: Mid is in the right (small) run, or the whole visible window is sorted. Mid could itself be the pivot (the minimum). So we cannot set hi = mid - 1. Setting hi = mid preserves mid in the window — invariant holds.

Termination: Each step strictly reduces hi - lo by at least 1 (since mid < hi when lo < hi). The loop exits when lo == hi. One candidate remains — the minimum.

⚠️ The Classic Mistake: Using hi = mid - 1

When nums[mid] ≤ nums[hi], setting hi = mid - 1 can eliminate the actual minimum. Example: [2, 1]. lo=0, hi=1, mid=0. nums[0]=2 > nums[1]=1 → lo=1. Done correctly: returns 1. But if we had mid=1 in a different state and used hi = mid-1 = 0 when nums[1] was the answer, we'd skip it. Always use hi = mid.

The non-rotated case is automatic: When the array is fully sorted, nums[mid] ≤ nums[hi] is always true (right run spans everything). hi keeps moving left until it reaches lo=0. Result: nums[0], which is the minimum. Correct!

4 The Code (Annotated) ▾

Solution 1 — Binary Search on Pivot (Interview Pick, O(log n) time)

def findMin(nums: list[int]) -> int:
lo, hi = 0, len(nums) - 1 ← Search window = entire array
while lo < hi: ← Stop when window has one element left
mid = (lo + hi) // 2 ← Integer midpoint; no overflow risk in Python
if nums[mid] > nums[hi]: ← Mid is in the large left run
lo = mid + 1 ← Min is strictly right of mid; safe to skip mid
else: ← Mid is in the small right run (or array sorted)
hi = mid ← Min could BE mid; keep it in window
return nums[lo] ← lo==hi; only candidate = the minimum
⚠️ Common Mistake: Using nums[lo] in the Comparison

Many beginners write if nums[mid] > nums[lo]. This is wrong! When mid is larger than lo, mid could still be in either run. The right comparison is always nums[mid] vs nums[hi] — it provides an unambiguous answer about which run mid belongs to.

Solution 2 — Linear Scan (Brute Force, O(n))

def findMin(nums: list[int]) -> int:
minimum = nums[0] ← Start with first element as candidate minimum
for num in nums: ← Scan every element — O(n)
minimum = min(minimum, num) ← Update if smaller found
return minimum ← Or just: return min(nums)
💡 Complexity Summary

Binary Search (optimal): Time O(log n) — halves search space each step. Space O(1) — just two pointers.
Linear Scan (naive): Time O(n) — visits every element. Space O(1).
Always choose Binary Search in interviews. The problem explicitly asks for O(log n).

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the O(log n) + Sorted Signal

Any problem involving a sorted (or partially sorted) structure with an O(log n) requirement screams binary search. Don't hesitate — announce this pattern immediately: "Since the input is sorted (even if rotated), and we need O(log n), I'll use binary search."

Step 2: Draw the Two-Run Structure

On paper or a whiteboard, sketch an example: [4,5,6,7,0,1,2]. Draw an arrow where the "drop" happens. Say: "The array has two ascending runs separated by a pivot. The minimum is the first element of the second run." This shows structural insight.

Step 3: Identify the Binary Decision

For binary search to work, you need a condition that unambiguously eliminates half the array. The key: "Compare nums[mid] to nums[hi]. If mid exceeds hi, mid is in the large left run → minimum is right. Otherwise, mid is in the small right run → minimum is mid or left."

Step 4: Get the Loop Boundary Right

Use while lo < hi (loop exits when one element remains). Use hi = mid (not mid-1) when mid could be the answer. Practice this on [2,1] to verify: lo=0, hi=1, mid=0, nums[0]=2 > nums[1]=1 → lo=1. Return nums[1]=1. Correct!

🗣️ What to Say to the Interviewer

"The problem needs O(log n) on a sorted-but-rotated array, so I'll binary search.
The rotated array has two ascending runs. The minimum is where the 'drop' happens.
I'll track lo and hi pointers and compute mid each iteration.
If nums[mid] > nums[hi], mid is in the large left run — minimum is right of mid, so lo = mid+1.
Otherwise mid is in the small right run — minimum is at mid or left, so hi = mid (not mid-1).
When lo == hi, we have one candidate left — that's the minimum.
Time O(log n), Space O(1)."
🔮 Likely Follow-up Questions
  • "What if there are duplicates?" → LeetCode #154. Add elif nums[mid] == nums[hi]: hi -= 1. O(log n) average, O(n) worst case.
  • "Find a target value in rotated array?" → #33. Find which half is sorted, check if target is in it, then binary search that half.
  • "Count how many times it was rotated?" → The index of the minimum element = rotation count.
  • "What if the array might not be rotated at all?" → Algorithm handles it naturally: no-rotation means one sorted run, hi converges to lo=0, returns nums[0].
