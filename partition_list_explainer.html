LeetCode #86

Partition List

Given a linked list and a value x, partition it so all nodes less than x come before nodes greater than or equal to x, preserving the original relative order in each group.

Medium Linked List Two Lists Time: O(n) Space: O(1)
1 Understanding the Problem ▾

You have a linked list and a pivot value x. Your task is to rearrange the nodes so that all nodes with values strictly less than x appear before all nodes with values greater than or equal to x.

The critical constraint — one that beginners often miss — is that relative order must be preserved within each group. If nodes 1, 2, 2 all appear less than x in the original list, they must appear in that same order (1 → 2 → 2) in the output, not rearranged. This is what makes it a partition, not a sort.

💡 Key Insight

Think of it as splitting one list into two: a "less" list (all nodes where val < x) and a "greater-or-equal" list (all nodes where val ≥ x), then attaching the tail of the less list to the head of the gteq list. Since we traverse and append in original order, relative ordering is automatically preserved — no extra tracking needed.

Input: head=[1→4→3→2→5→2], x=3

1 < 3 ✓ 4 ≥ 3 ✓ 3 ≥ 3 ✓
2 < 3 ✓ 5 ≥ 3 ✓ 2 < 3 ✓
Mixed order — needs rearranging

Output: [1→2→2→4→3→5]

Less group: 1→2→2 (original order!)
GtEq group: 4→3→5 (original order!)
Groups joined: [less] → [gteq]

Edge cases to consider: empty list (return null), single node, all values < x (gteq list is empty), all values ≥ x (less list is empty), value exactly equal to x (goes into ≥ group), duplicates equal to x.

2 The Algorithm: Two Dummy Lists ▾

We maintain two sub-lists simultaneously using dummy head nodes and tail pointers. Walk the original list once, routing each node to the correct sub-list. At the end, join the two sub-lists together.

1 Create two dummy sentinels: less_dummy and gteq_dummy. These are placeholder nodes that never hold real data — they just give us a stable starting point for each sub-list. Also create tail pointers less_tail and gteq_tail, both initially pointing to their respective dummies.
2 Walk the original list with curr: For each node, check curr.val < x. If true, append to the less-list (less_tail.next = curr; less_tail = curr). If false, append to the gteq-list. Advance curr = curr.next each iteration.
3 Null-terminate the gteq-list: gteq_tail.next = None. Without this, the last node in the gteq group still has its old .next pointer from the original list — this can create a cycle or dangling pointer in the merged result.
4 Join and return: less_tail.next = gteq_dummy.next stitches the less-list's tail to the gteq-list's real head (skipping the dummy). Return less_dummy.next — the real head of the merged list.

🔑 Why Dummy Heads?

Without a dummy head, you'd have to special-case the very first node of each sub-list: "if the list is empty, set the head pointer; otherwise set tail.next." With a dummy head, you always have a node to append to. Every iteration is identical: tail.next = node; tail = node. The code is cleaner, shorter, and less error-prone. Dummy heads are the standard idiom for building linked lists from scratch.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At any point during the traversal, the "less" sub-list contains exactly those nodes seen so far with val < x, in their original relative order. Likewise the "gteq" sub-list contains those with val ≥ x in original order. This invariant is maintained trivially: we visit nodes left-to-right and always append — never insert in the middle, never skip, never reorder.

Why relative order is guaranteed: We visit nodes strictly left-to-right (original order). When we append to a sub-list, we always append to the tail, not the head or middle. So the sub-list grows in the same order we encounter nodes — any node that appeared earlier in the original list appears earlier in its sub-list. This is a structural guarantee, not something we need to check.

Why the cycle-prevention step (gteq_tail.next = None) is critical: Consider the last node in the original list — say it has value 5 (≥ x). It gets appended to the gteq-list. But its .next pointer still points to whatever followed it originally — which is null in this case, but in other inputs, that original .next might point to a node now in the less-list. After we connect the two lists, the last gteq-node's stale pointer could create a cycle. Always null-terminate before connecting.

⚠️ Classic Bug: Stale .next Pointers

This is the most common mistake with linked list partitioning. After you append a node to a sub-list, its .next still points to its old neighbor in the original list. This is fine while building — you advance curr = curr.next before moving on. But at the end, the last node of the gteq-list has a stale .next. Null-terminate it: gteq_tail.next = None. Then connect: less_tail.next = gteq_dummy.next. Order matters — null-terminate first, then connect.

Space O(1): We're only rewiring existing nodes' .next pointers. No new nodes are allocated except for the two dummy sentinels, which are constant. The space is O(1) regardless of input size.

4 The Code (Annotated) ▾

Solution 1 — Two Dummy Lists (Optimal, O(n) time, O(1) space)

def partition(head, x): ← head: input list; x: partition value
less_dummy = ListNode(0) ← sentinel for the "less than x" sub-list
gteq_dummy = ListNode(0) ← sentinel for the "≥ x" sub-list
less_tail = less_dummy ← tail pointer for less-list (starts at dummy)
gteq_tail = gteq_dummy ← tail pointer for gteq-list (starts at dummy)
curr = head ← walker starts at the real head
while curr: ← process every node exactly once, O(n) total
if curr.val < x: ← strict less-than: equal goes to gteq-list
less_tail.next = curr ← splice onto less-list tail
less_tail = less_tail.next ← advance tail to the newly added node
else: ← curr.val >= x
gteq_tail.next = curr ← splice onto gteq-list tail
gteq_tail = gteq_tail.next ← advance tail to the newly added node
curr = curr.next ← advance walker to next original-list node
gteq_tail.next = None ← CRITICAL: null-terminate to prevent cycles
less_tail.next = gteq_dummy.next ← stitch: less-list tail → gteq-list real head
return less_dummy.next ← skip dummy sentinel, return real head
⚠️ Order of Final Two Lines Matters

Always null-terminate the gteq-list first (gteq_tail.next = None), then connect (less_tail.next = gteq_dummy.next). If you reverse the order, you might accidentally introduce a cycle before severing the old pointer. Think: "sever before connect."

Solution 2 — Collect Values + Rebuild (Brute Force)

def partition_brute(head, x):
less_vals, gteq_vals = [], [] ← collect values into two Python lists
curr = head
while curr:
if curr.val < x: less_vals.append(curr.val) ← partition while collecting
else: gteq_vals.append(curr.val)
curr = curr.next
dummy = ListNode(0) ← rebuild the list from scratch
tail = dummy
for v in less_vals + gteq_vals:
tail.next = ListNode(v) ← allocate new nodes — O(n) extra space
tail = tail.next
return dummy.next
💡 Complexity Summary

Two Dummy Lists: Time O(n), Space O(1) — rewire pointers only. Interview pick.
Collect + Rebuild: Time O(n), Space O(n) — allocates n new nodes. Acceptable as a starting point, then optimize.
Both are O(n) time but the dummy-list approach uses constant space.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Spot the "Stable Partition" Signal

The phrase "preserve relative order" is the signal. It means you cannot swap nodes around — you need to collect them in order. This immediately rules out any in-place swap approach. You need to build the output sequence in traversal order.

Step 2: Propose the Two-List Idea

"I'll build two separate sub-lists as I traverse. One collects nodes with val < x, the other collects val ≥ x. At the end, I join them. Since I traverse left-to-right and append to the tail, order is automatically preserved."

Step 3: Proactively Mention Dummy Heads

"I'll use dummy sentinel nodes for each sub-list. This eliminates the need to check whether the sub-list is empty before appending — I always do tail.next = node; tail = node. It makes the code uniform and edge-case-free."

Step 4: Remember to Null-Terminate

"One subtle point: before joining the lists, I must set gteq_tail.next = None. Otherwise the last gteq node still has its original .next, which may point somewhere unexpected — a potential cycle."

🗣️ What to Say to the Interviewer

"I'll use two dummy-headed sub-lists — one for nodes less than x, one for nodes >= x.
I walk the original list once. Each node goes to whichever sub-list it belongs to.
I track tail pointers for O(1) append, so the loop body is always the same two lines.
After the loop: null-terminate the gteq-list, then stitch its head to the less-list's tail.
Return less_dummy.next. Time O(n), Space O(1) — no new nodes allocated."
🔮 Likely Follow-up Questions
  • "Can you do it without extra space?" → Already O(1) — only dummy nodes are extra.
  • "What if we needed 3 partitions (val < x, val == x, val > x)?" → Add a third dummy list for equal-to-x nodes; join all three at end.
  • "What if the list is doubly linked?" → Same logic, but also update prev pointers when relinking.
  • "What does this remind you of?" → Odd Even Linked List (#328), Quick Sort's partition step.

Two dummy sentinels eliminate null-handling on first-node insertions
Appending to the tail preserves relative order automatically
Always null-terminate gteq_tail.next BEFORE connecting the lists
Return less_dummy.next (skip the sentinel, not less_dummy itself)
