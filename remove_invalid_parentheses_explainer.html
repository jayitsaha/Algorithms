LeetCode #301

Remove Invalid Parentheses

Remove the minimum number of invalid parentheses to make the input string valid. Return all possible results in any order.

Hard Backtracking BFS Time: O(n·2ⁿ) Space: O(n·2ⁿ)
1 Understanding the Problem ▾

Given a string of letters and parentheses, we need to remove the fewest possible parentheses so that every ( is matched with a ), and nesting is correct. The catch: there may be many different ways to remove the minimum — we must return all of them.

💡 Key Insight

Two constraints interact: minimum removals (greedy constraint) + all distinct results (exhaustive search constraint). Neither greedy alone (gives one answer) nor naive backtracking (ignores minimum) works. We need BFS — which groups solutions by removal count and stops at the first level with valid strings.

✗ Invalid: "()())()"

Balance: 1,0,1,0,-1 ← negative!
extra ')' at index 4
min removals: 1

✓ All valid results

"()()()" — remove index 4
"(())()" — remove index 1
Both length 6; 1 removal each

What counts as valid? Scan left-to-right with a running balance. Add 1 for (, subtract 1 for ). If balance ever goes negative → invalid immediately. If balance is nonzero at end → invalid. Letters are ignored.

Edge cases: Empty string → [""]. Already valid → [original]. All parens mismatched like "(((" → [""]. Letters only like "abc" → ["abc"].

2 The Algorithm: BFS Level-by-Level Removal ▾

Think of each string as a state. Children are strings formed by removing one paren. BFS processes all states at "distance 1" before "distance 2." The first BFS level that yields valid strings is the answer — all answers at that level needed exactly the same minimum number of removals.

1 Seed queue with original string. Create a visited set to deduplicate — removing different positions can yield the same string.
2 Dequeue one string, run validity check. If valid: add to results and set found = True. If invalid: generate children (unless found is True).
3 Generate children by trying to remove each ( or ) position. Skip letters — they're never removed. Only enqueue strings not in visited.
4 After processing the full BFS level: if found = True, return results. Otherwise, continue to the next level (one more removal).

Level 1 found valid strings — STOP. Return {"()()()", "(())()"}.
🔑 Alternative: DFS + Pruning (Optimal)

Pre-count excess ( and ) separately. DFS with budgets left_rem and right_rem. At each position: keep the char, or remove it if budget allows. Dedup by skipping all-but-first of consecutive same parens (e.g., "(((": only remove leftmost). No visited set, smaller memory footprint.

3 Why It Works: The Invariant ▾
🔐 The BFS Invariant

Every string at BFS depth d was formed from the original by removing exactly d parentheses. BFS exhausts depth d before depth d+1. Therefore, the first depth where any valid string appears equals the minimum number of removals needed. All valid strings at that depth are minimum-removal answers.

Why the visited set prevents duplication: Removing s[1] from "(()" gives "()" — same as removing s[0] gives "()" . Without a visited set, we'd explore "()" multiple times, leading to duplicate results and exponential wasted work. The visited set collapses these identical states into one.

Why if found: continue (not break): When we find the first valid string at level d, other valid strings from the same level may still be in the queue. Using break would miss them. Using continue keeps processing current-level strings (collecting more valid ones) while preventing child generation for the next level.

⚠️ The "Same Level" Trap

The key insight that trips beginners: found is set during a level but controls the next level's generation. All strings currently in the queue at level d must still be checked for validity — only their children (level d+1) must not be enqueued. The if found: continue line achieves exactly this.

Validity is O(n): One scan. Counter starts at 0. ( increments. ) decrements. If counter goes below 0, return False immediately. If counter != 0 at end, return False. Otherwise True. Letters leave counter unchanged.

4 The Code (Annotated) ▾

Solution 1 — BFS (Interview Pick, clear correctness argument)

from collections import deque
def removeInvalidParentheses(s: str):
def is_valid(t): ← O(n) balance checker
count = 0
for c in t:
if c == '(': count += 1
elif c == ')': count -= 1
if count < 0: return False ← early exit: unmatched ')'
return count == 0 ← must be fully balanced at end
queue = deque([s]) ← BFS starts from original
visited = {s} ← deduplicate identical strings
found = False ← True once we hit the min-removal level
result = []
while queue:
cur = queue.popleft() ← process next candidate
if is_valid(cur):
result.append(cur) ← valid at this depth → collect
found = True ← mark min removal level reached
if found: continue ← don't go deeper — critical!
for i in range(len(cur)):
if cur[i] not in '()': continue ← never remove letters
nxt = cur[:i] + cur[i+1:] ← string without position i
if nxt not in visited:
visited.add(nxt)
queue.append(nxt) ← enqueue for next BFS level
return result
⚠️ Why "if found: continue" prevents wrong answers

Suppose the valid answers are at depth 1. If we generate depth-2 children from any string, those depth-2 candidates are also processed by the while loop. Some might be valid (they removed 2 parens), but they're NOT minimum-removal answers. The if found: continue gate ensures no depth-2 candidates are ever enqueued once we've confirmed depth 1 has valid strings.

Solution 2 — DFS + Pruning (Avoids visited set, more efficient)

def removeInvalidParentheses(s: str):
# Step 1: count excess '(' and ')' separately
left_rem = right_rem = 0
for c in s:
if c == '(': ← tentatively count open
left_rem += 1
elif c == ')':
if left_rem: left_rem -= 1 ← close matches open → cancel
else: right_rem += 1 ← no match → this ')' is excess
result = set()
def dfs(idx, l_rem, r_rem, l_cnt, r_cnt, path):
if idx == len(s): ← processed all characters
if l_rem==0 and r_rem==0 and l_cnt==r_cnt:
result.add(path) ← all excess removed, balanced
return
c = s[idx]
# Always keep letters
if c not in '()':
dfs(idx+1, l_rem, r_rem, l_cnt, r_cnt, path+c)
return
# Option A: skip (remove) — dedup by only skipping first in a run
if c=='(' and l_rem>0 and (idx==0 or s[idx-1]!='('):
dfs(idx+1, l_rem-1, r_rem, l_cnt, r_cnt, path) ← remove this '('
if c==')' and r_rem>0 and (idx==0 or s[idx-1]!=')'):
dfs(idx+1, l_rem, r_rem-1, l_cnt, r_cnt, path) ← remove this ')'
# Option B: keep the paren
if c == '(':
dfs(idx+1, l_rem, r_rem, l_cnt+1, r_cnt, path+c)
elif r_cnt < l_cnt: ← only keep ')' if there's an open '('
dfs(idx+1, l_rem, r_rem, l_cnt, r_cnt+1, path+c)
dfs(0, left_rem, right_rem, 0, 0, "")
return list(result)
💡 Complexity Summary

BFS: Time O(n × 2ⁿ) — up to 2ⁿ unique strings, each validity-checked in O(n). Space O(n × 2ⁿ) for visited set.
DFS + Pruning: Time O(2ⁿ) — far better in practice due to budget-based pruning. Space O(n) call stack + output.
In practice, both are fast for n ≤ 25 (typical interview constraints).

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify the Exhaustive Search Requirement

"The problem asks for ALL valid results, not just one. This rules out greedy. Combined with 'minimum removals,' we need a search that controls depth (= number of removals). BFS controls depth naturally."

Step 2: Formulate the State Space

"Model each string as a node. Children: remove one paren at any index. BFS level = number of removals. Stop at first level with valid strings. This is guaranteed minimum."

Step 3: Address the Deduplication Problem

"Removing different indices can yield the same string — e.g., removing index 0 or index 2 from '((' both give '('. Use a visited set so identical strings are processed only once."

Step 4: Mention the DFS Optimization If Asked

"An alternative: pre-count exactly how many '(' and ')' are excess. DFS with those budgets. Skip consecutive same-char removals to deduplicate. No visited set — saves O(n × 2ⁿ) memory."

🗣️ What to Say to the Interviewer

"The key constraint is minimum removals AND all results. BFS gives minimum naturally.
Start with original string in queue. Process level by level — each level removes one more paren.
First level with valid strings = minimum removal level → collect all valid strings there, return.
Use visited set to avoid duplicate strings — crucial for correctness and performance.
The 'if found: continue' pattern processes the current level completely before stopping.
Time: O(n × 2^n), Space: O(n × 2^n). For n ≤ 25, this is acceptable."
🔮 Likely Follow-up Questions
  • "What if we need minimum additions too?" → Count unmatched ( at end of scan — those need closing chars appended.
  • "DFS instead of BFS?" → Yes, pre-count excess, DFS with budgets, dedup by skipping consecutive runs.
  • "Why not try every subset?" → 2ⁿ subsets × O(n) check = O(n × 2ⁿ). BFS does similar work but stops early.
  • "Strings with only letters?" → The input is valid already. Length check at start, return [s] immediately.

7 Summary & Key Takeaways ▾
BFS is the right choice when you need "all minimum-removal solutions" — it groups solutions by removal count
The visited set collapses duplicate strings from different removal orders into a single state
"if found: continue" processes the current BFS level fully without generating the next level
Never remove letters — they're immutable; only remove ( and ) characters
DFS + pre-counted budgets avoids the visited set and uses O(n) call-stack memory instead of O(n·2ⁿ)

🎯 When to Use BFS for Backtracking
  • "Remove minimum X to achieve property Y, return ALL solutions" → BFS by removal count
  • "Shortest path / fewest operations in unweighted state space" → BFS always optimal
  • "All states equidistant from source" → BFS level = distance
  • When state space has heavy duplication → add visited set
