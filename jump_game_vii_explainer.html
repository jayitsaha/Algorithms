LeetCode #1871

Jump Game VII

Given a binary string s and integers minJump and maxJump, determine if you can reach the last index starting from index 0, jumping only onto '0' cells within [minJump, maxJump] distance.

Medium Greedy BFS with Range Time: O(n) Space: O(n)
1 Understanding the Problem ▾

You stand at index 0 of a binary string (e.g., "011010"). Your goal: reach the last index. From any position i, you can jump to position j if all three conditions hold:

  • minJump <= j - i <= maxJump — the jump distance is within the allowed range
  • s[j] == '0' — you can only land on '0' cells ('1' cells are walls)
  • s[0] is guaranteed to be '0' — you start there
💡 Key Insight

Instead of thinking "where can I jump to from i?", reverse the question: "which earlier positions can jump to me at j?" A position i can reach j exactly when j - maxJump <= i <= j - minJump. So for each j, we only need to check if any reachable index exists in the window [j-maxJump, j-minJump]. Track this with a sliding count — no expensive inner loop needed.

✓ Reachable: s="011010", min=2, max=3

0 → 3 (jump +3, s[3]='0') ✓

→ False

Edge Cases: Single character (already at last index → True). Last index is '1' (always False). minJump == maxJump allows exactly one jump distance. Large maxJump spanning the whole string (check if any reachable position exists in window).

2 The Algorithm: Greedy BFS with Sliding Window Count ▾

We sweep left-to-right, maintaining pre — the count of reachable positions that fall in the current "can jump to j" window. As j advances, the window slides: one index enters at the right edge, one exits at the left edge. Checking "any reachable?" costs O(1): just check pre > 0.

1 Initialize: reach[0] = True, pre = 0. The reach array tracks which indices we can land on. pre is the running count of reachable indices currently in the sliding window.
2 For each j from 1 to n-1: Add entering index. If j - minJump >= 0, add reach[j - minJump] to pre. This index just entered the window — it's exactly minJump steps behind j, so a jump of exactly minJump from it reaches j.
3 Check reachability: If pre > 0 AND s[j] == '0', mark reach[j] = True. There's at least one reachable position in the window that can jump to j, and j is a valid landing spot.
4 Slide: Remove exiting index. If j - maxJump >= 0, subtract reach[j - maxJump] from pre. This index has now fallen outside the window — it's too far back to jump to j+1 with a distance of at most maxJump.
5 Return reach[n-1]. After the sweep, the boolean at the last index tells us whether we can reach it.
🔑 Why pre Counts Instead of Scanning the Window

For each j, the naive approach scans every index in [j-maxJump, j-minJump] — O(maxJump) per index, O(n×maxJump) total. By maintaining pre as a rolling sum that increments/decrements exactly once per index (when it enters/exits the window), we do O(1) per j and O(n) total. Each index is added to pre at most once and removed at most once.

3 Why It Works: The Invariant ▾
🔐 The Invariant

When we process index j (after the "add entering" step, before the "remove exiting" step), pre equals exactly the number of reachable indices in the range [j-maxJump, j-minJump]. This is precisely the set of positions that can jump to j within the allowed distance range.

Why the window boundaries are derived correctly: A position i can jump to j iff minJump <= j - i <= maxJump. Rearranging both inequalities: j - maxJump <= i <= j - minJump. So the exact set of valid sources for j is the window [j-maxJump, j-minJump].

Why the window slides correctly as j advances by 1: Moving from j to j+1, the window shifts: the new right edge is (j+1) - minJump = j - minJump + 1, so index j - minJump + 1 enters (we add it in step 2 of the next iteration). The new left edge is (j+1) - maxJump = j - maxJump + 1, so index j - maxJump exits (we subtract it in step 4 of the current iteration). One add and one subtract per step — exactly correct.

⚠️ Critical: Order of Operations Must Be Exact

For each j: first add reach[j-minJump] to pre (index enters window), then check pre > 0 and mark reach[j], then subtract reach[j-maxJump] from pre (index exits window). If you subtract before checking, you'll miss the current j's window state. If you add after checking, you'll miss j's contribution. The order is not arbitrary.

Why pre -= reach[j-maxJump] instead of pre -= 1: Position j-maxJump may never have been reachable. reach[j-maxJump] is a bool (True=1, False=0). We only decrement pre if that position was actually reachable. Using -=1 unconditionally would make pre go negative and corrupt all future checks.

4 The Code (Annotated) ▾

Solution 1 — Sliding Window Count (Optimal, O(n) time · Interview Pick)

def canReach(s: str, minJump: int, maxJump: int) -> bool:
n = len(s) ← cache length once
reach = [False] * n ← reach[i]: can we land at index i?
reach[0] = True ← start position — always reachable
pre = 0 ← count of reachable positions in sliding window
for j in range(1, n): ← sweep every index from 1 to n-1
if j - minJump >= 0: ← guard: don't index below 0
pre += reach[j - minJump] ← this index just entered the window (reach is 0 or 1)
if pre > 0 and s[j] == '0': ← someone in window can jump here, AND j is open
reach[j] = True ← mark j as reachable
if j - maxJump >= 0: ← guard: don't index below 0
pre -= reach[j - maxJump] ← this index just exited the window — slide left edge
return reach[-1] ← did we ever reach the last index?
⚠️ Common Interview Mistake: Using a deque instead of a count

It's tempting to store reachable indices in a deque and check deque[0] >= j - maxJump. That works correctly but adds implementation complexity. The count trick (pre) is simpler: since reach[] is boolean, adding/subtracting it is equivalent to adding/subtracting 1 only when that index was reachable. One line of code does both "is this index reachable" and "add or skip".

Solution 2 — BFS (Brute Force, O(n × maxJump))

from collections import deque
def canReach_bfs(s, minJump, maxJump):
n, q = len(s), deque([0]) ← BFS queue starting at index 0
visited = {0} ← avoid revisiting indices
while q: ← process level by level
i = q.popleft() ← take next reachable position
for j in range(i+minJump, min(i+maxJump+1, n)): ← try all jump targets
if s[j]=='0' and j not in visited: ← valid unvisited target
if j==n-1: return True ← early exit on success
visited.add(j); q.append(j)
return (n-1) in visited ← check final result
💡 Complexity Summary

Sliding Window (Optimal): Time O(n), Space O(n) — one add and one subtract per index
BFS (Brute Force): Time O(n × maxJump), Space O(n) — can TLE when maxJump is large
Key insight: reach[] booleans act as 0/1 integers — arithmetic on them gives us pre for free

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize It as Reachability

"Can we reach the end?" signals BFS or DP. The binary string acts like a grid — '0' cells are walkable, '1' cells are walls. The jump range [minJump, maxJump] is the edge condition.

Step 2: Start with BFS (and state the complexity)

"From each reachable index i, enqueue all j in [i+min, i+max] where s[j]='0'. BFS guarantees we find the end if reachable. Complexity: O(n × maxJump) — can be quadratic."

Step 3: Observe the Range Dependency

"For each j, the question is: is any index in [j-maxJump, j-minJump] reachable? This is a range query — sliding window handles it in O(1) with a count variable."

Step 4: Code the Window Updates Carefully

"Add reach[j-minJump] to pre first (enters window), then check pre>0, then subtract reach[j-maxJump] (exits window). Since reach is boolean, arithmetic on it is 0 or 1 automatically."

🗣️ What to Say to the Interviewer

"I'll start with BFS from index 0. For each reachable position i, try all j in [i+minJump, i+maxJump]
where s[j]='0'. This is O(n × maxJump) — correct but potentially quadratic.

To optimize: for each j, the set of positions that can jump to j is exactly [j-maxJump, j-minJump].
Instead of scanning that range, I maintain 'pre' = count of reachable positions in that window.
Slide the window: add reach[j-minJump] when it enters, subtract reach[j-maxJump] when it exits.
If pre > 0 and s[j]='0', mark j reachable. O(n) time, O(n) space."
🔮 Likely Follow-up Questions
  • "Can you reduce space to O(1)?" → No — we need the full reach[] to answer window queries. The window can span up to maxJump positions, all of which we need to track.
  • "What if cells have costs and you want minimum cost path?" → Dijkstra or DP with deque (Jump Game VI #1696)
  • "What if you can jump backwards?" → BFS with visited set (Jump Game III #1306)
  • "How is this different from Jump Game II?" → Jump II allows any distance up to arr[i]; here it's a fixed range [min, max] on a binary string

7 Summary & Key Takeaways ▾
Reachability over a jump range = sliding window count — not a scan of every predecessor
Reverse the question: "who can reach j?" has a clean window [j-maxJump, j-minJump]
pre counts reachable positions in the window using bool arithmetic (True=1, False=0)
Order matters: add-entering → check → subtract-exiting for each j
pre -= reach[...] not pre -= 1 — never subtract if position was unreachable
