LeetCode #724

Find Pivot Index

Return the leftmost pivot index where the sum of all elements to its left equals the sum of all elements to its right.

Easy Prefix Sum Left Sum == Right Sum Time: O(n) Space: O(1)
1 Understanding the Problem ▾

Given an array of integers nums, find the pivot index — a position i where the sum of every element strictly to its left equals the sum of every element strictly to its right. The element at the pivot itself is not counted on either side.

💡 Key Insight

Think of a see-saw: you are looking for the fulcrum — the exact position where both sides balance perfectly. The pivot itself sits at the balance point but doesn't tip either side.

✓ Pivot found: [1, 7, 3, 6, 5, 6]

Pivot at index 3 (value = 6)
Left: 1 + 7 + 3 = 11
Right: 5 + 6 = 11
11 == 11 → Return 3

✗ No pivot: [1, 2, 3]

i=0: left=0, right=5 → fail
i=1: left=1, right=3 → fail
i=2: left=3, right=0 → fail
→ Return -1

Important edge cases:

  • Index 0 as pivot: Left sum is 0 (no elements), right sum is everything else
  • Last index as pivot: Right sum is 0 (no elements), left sum is everything else
  • Single element: Always a pivot — left=0, right=0
  • Multiple pivots: Return the leftmost one (scan left to right)
  • All zeros: Every index is a pivot — return 0
  • Negative numbers: The equation still holds exactly
2 The Algorithm: One-Pass Prefix Sum ▾

The naive approach — for each index, sum everything left and sum everything right — is O(n²). We can eliminate the inner loops using a clever mathematical relationship between left sum, right sum, and the total.

🔑 The Core Equation

For any index i: left_sum + nums[i] + right_sum = total
Therefore: right_sum = total − left_sum − nums[i]

We only need to track left_sum as we iterate — right_sum falls out for free from the equation. No second inner loop required.

1 Compute total: Sum all elements once. This takes O(n) and gives us our global reference point.
2 Initialize left_sum = 0: Before we look at index 0, there are no elements to its left — so the left sum is zero.
3 Scan left to right: For each index i, derive right_sum = total − left_sum − nums[i]. If left_sum == right_sum, return i.
4 Advance left_sum: Add nums[i] to left_sum after the pivot check. This maintains the invariant for the next index.
5 Return -1: If no pivot is found after scanning the entire array, return -1.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At the start of iteration i, left_sum always equals the sum of nums[0..i-1] — every element strictly to the left of index i. We maintain this by adding nums[i] at the end of each iteration, not the beginning.

Mathematical proof of correctness:

Define for index i:

  • L(i) = sum of nums[0..i-1] (elements strictly left of i)
  • R(i) = sum of nums[i+1..n-1] (elements strictly right of i)
  • total = L(i) + nums[i] + R(i) for all i

Solving: R(i) = total − L(i) − nums[i]

Index i is a pivot iff L(i) == R(i), i.e., L(i) == total − L(i) − nums[i].

Since our invariant tracks L(i) exactly via left_sum, we compute R(i) in O(1) — no second pass needed.

⚠️ Update Order Is Critical

The check left_sum == total − left_sum − nums[i] must happen before you add nums[i] to left_sum. If you update first, you've incorrectly included nums[i] in the left side — the invariant breaks and you'll miss every pivot.

Why O(1) space? We store exactly two numbers throughout: total (computed once) and left_sum (updated incrementally). The right sum is derived on the fly — it never needs its own variable permanently.

4 The Code (Annotated) ▾

Solution 1 — One-Pass Prefix Sum (Optimal, O(n) time, O(1) space) — Interview Pick

def pivotIndex(nums: list[int]) -> int:
total = sum(nums) ← Compute full sum once: O(n). This is our global reference.
left_sum = 0 ← Before index 0, no elements are to the left.
for i, num in enumerate(nums): ← i = index, num = nums[i]
right_sum = total - left_sum - num ← Derive right sum in O(1) from the equation.
if left_sum == right_sum: ← Pivot condition: both sides balance.
return i ← Leftmost pivot — return immediately.
left_sum += num ← Advance left_sum AFTER check to maintain invariant.
return -1 ← Exhausted all indices — no pivot exists.
💡 Why This Is the Interview Pick

Two passes total: one to compute the total (unavoidable), one to find the pivot. Space is O(1) — just two integers live at any time. The derivation equation is the crux — write it on the board first, then the code follows naturally.

Solution 2 — Brute Force (O(n²) time, O(1) space)

def pivotIndex(nums: list[int]) -> int:
for i in range(len(nums)): ← Try every index as a candidate pivot.
if sum(nums[:i]) == sum(nums[i+1:]): ← Recompute both sides from scratch: O(n) each.
return i
return -1
⚠️ Why Brute Force Fails at Scale

Each of n iterations recomputes two sums — each O(n). Total: O(n²). On a 10,000-element array that's 100 million operations vs. ~20,000 for the prefix sum approach. Always present brute force first, then offer to optimize.

Solution 3 — Prefix Sum Array (O(n) time, O(n) space)

def pivotIndex(nums: list[int]) -> int:
n = len(nums)
prefix = [0] * (n + 1) ← prefix[i] = sum(nums[0..i-1])
for i in range(n):
prefix[i+1] = prefix[i] + nums[i] ← Build prefix array.
total = prefix[n] ← Total is the last entry.
for i in range(n):
left = prefix[i] ← O(1) left sum lookup.
right = total - prefix[i+1] ← O(1) right sum lookup.
if left == right: return i
return -1

Pedagogically cleaner — it shows the prefix array concept explicitly. But uses O(n) space. Solution 1 is strictly better since we can derive the same information without the array.

💡 Complexity Summary

Brute Force: Time O(n²), Space O(1)
Prefix Array: Time O(n), Space O(n)
One-Pass (Optimal): Time O(n), Space O(1)

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Clarify the Problem

Confirm: "The pivot element is excluded from both sums?" (Yes.) "If multiple pivots exist, return leftmost?" (Yes.) "Can we have negative numbers?" Check constraints — LeetCode says -1000 ≤ nums[i] ≤ 1000. Important: negatives don't break the equation.

Step 2: Propose Brute Force First

"For each index, I could recompute both sums from scratch — O(n²). But I can do better by precomputing the total and maintaining a running left sum."

Step 3: Derive the Key Equation

"If left_sum + nums[i] + right_sum = total, then right_sum = total − left_sum − nums[i]. I can compute right_sum in O(1) at every step without another inner loop."

Step 4: Code the Invariant Carefully

"I start left_sum at 0, check the pivot condition, then add nums[i] to left_sum. The key is the update happens AFTER the check — this preserves the invariant that left_sum = sum of everything strictly to the left."

🗣️ What to Say to the Interviewer

"I'll compute total = sum(nums) upfront — O(n), unavoidable.
Then I'll scan left to right with left_sum starting at 0.
At each index i, right_sum = total minus left_sum minus nums[i] — O(1).
If left_sum equals right_sum, I've found the pivot — return i immediately.
Otherwise I add nums[i] to left_sum and continue.
Overall: O(n) time, O(1) space. Just two variables throughout."

The pivot equation — right_sum = total − left_sum − nums[i] — is the entire insight
Compute total once (unavoidable O(n)); derive right_sum in O(1) each step
Always check the pivot condition BEFORE updating left_sum (invariant!)

Pivot at index 0: left_sum = 0 naturally (no elements left of it)
Multiple pivots: scan left to right and return on first match — guaranteed leftmost
