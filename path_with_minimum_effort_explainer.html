LeetCode #1631

Path With Minimum Effort

Find a path from top-left to bottom-right of a height matrix that minimises the maximum absolute difference between adjacent cells on the path.

Medium Graph Dijkstra on Max Edge Time: O(m·n·log(m·n)) Space: O(m·n)
1 Understanding the Problem ▾

You have a 2D grid where every cell stores an elevation. Starting at the top-left corner (0,0) you want to reach the bottom-right corner (m-1, n-1), moving only up/down/left/right one step at a time.

The "effort" of a path is the maximum absolute height difference between any two consecutive cells on that path. Your goal is to find the path where this worst-case single step is as small as possible — a classic minimax objective.

💡 Key Insight

We are not minimising the sum of differences — we are minimising the maximum single step. This transforms into a modified shortest-path problem: define the "distance" to any cell as the minimum possible maximum edge weight on any path to it. Replace the usual dist + edge relaxation with max(dist, edge).

✓ Optimal Path: [[1,2,2],[3,8,2],[5,3,5]]

(0,0)→(1,0)→(2,0)→(2,1)→(2,2)
Steps: |1-3|=2, |3-5|=2, |5-3|=2, |3-5|=2
Max step = 2 → Answer = 2

✗ Wrong Path: going through the "8" cell

(0,0)→(0,1)→(1,1)→(2,1)→(2,2)
Steps: |1-2|=1, |2-8|=6, |8-3|=5, |3-5|=2
Max step = 6 → Suboptimal!

Edge Cases: A 1×1 grid returns 0 (start = goal). All cells at equal height → answer is 0. Large height jumps (up to 106) must be handled without overflow (Python ints are arbitrary precision, so no issue).

2 The Algorithm: Modified Dijkstra on Max Edge ▾

Classical Dijkstra finds the path where edge weights sum is minimised. Here, we redefine the "distance" to cell (r,c) as the minimum possible maximum effort to reach it. The only change to Dijkstra is the relaxation formula.

Classical Dijkstra (sum)

dist[v] = Σ edge weights

Relax: dist[v] = dist[u] + w(u,v)

Use when: minimize total cost

Modified Dijkstra (max)

dist[v] = max edge on path

Relax: dist[v] = max(dist[u], w(u,v))

Use when: minimize bottleneck

1 Initialize: Create dist[m][n] = ∞ everywhere, except dist[0][0] = 0. Push (effort=0, r=0, c=0) into a min-heap.
2 Pop minimum: Extract the entry with the smallest effort. If it is the goal (m-1, n-1), return immediately — the heap guarantees this is optimal.
3 Skip stale entries: The heap may contain outdated entries. If popped_effort > dist[r][c], we already found a better path — skip this entry.
4 Relax neighbours: For each 4-directional neighbour, compute new_effort = max(current_effort, |heights[r][c] - heights[nr][nc]|). If this improves dist[nr][nc], update and push.
5 Repeat until the heap is empty or the goal is found. Every cell is settled at most once with its optimal effort.
🔑 Why Not BFS?

BFS is correct for unweighted or equal-cost graphs. Here, different edges have different height differences. A cell discovered later via a low-effort path should beat one found earlier via a high-effort path. Dijkstra's priority queue handles this by always processing the cell with the smallest accumulated effort next.

3 Why It Works: The Invariant ▾
🔐 The Invariant

When a cell is popped from the min-heap, dist[r][c] holds the true minimum possible maximum edge weight on any path from (0,0) to (r,c). This value is final — it will never be improved again.

Why the invariant holds: The heap is ordered by effort. When we pop a cell with effort e, every remaining heap entry has effort ≥ e. The max operation is monotone: extending any path by one more edge can only keep the max the same or increase it. Therefore no future exploration can produce a path to this cell with effort less than e.

The monotonicity requirement: Standard Dijkstra requires non-negative edge weights so that dist[u] + w ≥ dist[u] (adding weight never decreases cost). Here, we need max(e, w) ≥ e — always true since max of two values is ≥ either. The proof transfers directly.

⚠️ Common Mistake: Forgetting the Stale-Entry Check

When we update dist[nr][nc] to a better value, the old heap entry for that cell is not removed — Python's heapq has no decrease-key operation. We leave stale entries in the heap and detect them with: if eff > dist[r][c]: continue. Omitting this check causes cells to be reprocessed needlessly (or incorrectly).

Handling edge cases: A 1×1 grid — (0,0) is the goal, so we return dist[0][0] = 0 immediately on the first pop. Uniform heights — all edges have weight 0, and the answer is 0 throughout.

4 The Code (Annotated) ▾

Solution 1 — Modified Dijkstra (Optimal, Interview Pick)

import heapq
def minimumEffortPath(heights): ← Input: m×n grid of elevations
m, n = len(heights), len(heights[0])
dist = [[float('inf')]*n ← Min effort to reach each cell; ∞ = unreached
for _ in range(m)]
dist[0][0] = 0 ← Start costs 0 effort
heap = [(0, 0, 0)] ← Min-heap: (effort, row, col)
dirs = [(0,1),(0,-1),(1,0),(-1,0)]
while heap: ← Until heap is empty
eff, r, c = heapq.heappop(heap) ← Pop cell with smallest effort
if r == m-1 and c == n-1: ← Reached goal? Return immediately — this is optimal
return eff
if eff > dist[r][c]: ← Stale entry: better path already found, skip
continue
for dr, dc in dirs: ← Explore all 4 neighbours
nr, nc = r+dr, c+dc
if 0 <= nr < m and 0 <= nc < n:
edge = abs(heights[r][c]-heights[nr][nc]) ← Cost of this single step
new_eff = max(eff, edge) ← KEY: take max, not sum. Worst step so far.
if new_eff < dist[nr][nc]:← Relaxation: better path found?
dist[nr][nc] = new_eff
heapq.heappush(heap, (new_eff, nr, nc))
return dist[m-1][n-1] ← Fallback (should not be reached for a connected grid)
⚠️ Why max(eff, edge) and not eff + edge?

The problem defines effort as the maximum step, not the total. This single word — max vs sum — is the entire crux of the problem. Changing it to eff+edge solves the classic "cheapest path" problem instead. In interviews, always restate the objective clearly: "I am minimising the maximum edge, not the sum."

Solution 2 — Binary Search + BFS (Alternative)

from collections import deque
def minimumEffortPath(heights): ← Binary search on the answer value
m, n = len(heights), len(heights[0])
def canReach(limit): ← BFS: can we reach goal using only edges ≤ limit?
vis = [[False]*n for _ in range(m)]
q = deque([(0,0)]); vis[0][0] = True
while q:
r, c = q.popleft()
if r == m-1 and c == n-1: return True
for dr,dc in [(0,1),(0,-1),(1,0),(-1,0)]:
nr,nc = r+dr,c+dc
if 0<=nr
if abs(heights[r][c]-heights[nr][nc]) <= limit:
vis[nr][nc]=True; q.append((nr,nc))

lo, hi = 0, max(max(r) for r in heights) ← Search range: 0 to max height value
while lo < hi: ← Standard binary search
mid = (lo + hi) // 2
if canReach(mid): hi = mid ← mid works? try smaller
else: lo = mid + 1 ← mid too small? need at least mid+1
return lo
💡 Complexity Summary

Modified Dijkstra: Time O(m·n·log(m·n)), Space O(m·n) — each cell pushed at most once per neighbour
Binary Search + BFS: Time O(m·n·log H) where H = max height value, Space O(m·n)
In practice Dijkstra is faster (fewer BFS passes); Binary Search is easier to reason about under interview pressure.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify the Minimax Objective

The phrase "minimum of the maximum" is a strong signal. Minimax objectives on graphs almost always reduce to a modified shortest-path problem. The key question is: what does "distance" mean in this modified setting?

Step 2: Redefine Distance

Define dist[r][c] = "minimum possible maximum edge weight on any path from (0,0) to (r,c)." The relaxation then becomes new_eff = max(dist[r][c], edge_weight). If this is better, update and push. The rest of Dijkstra stays unchanged.

Step 3: Justify Correctness

Dijkstra is correct when the cost function is monotone. Here: max(e, w) ≥ e always — adding more steps never decreases the maximum. That is the monotonicity Dijkstra requires, so the correctness proof carries over directly.

Step 4: Offer the Binary Search Alternative

"There's also a Binary Search approach: binary search on the effort value (0 to max height), and for each candidate use BFS to check if a path exists with all edges below that threshold. O(m·n·log H). Easier to derive if Dijkstra doesn't come to mind immediately."

🗣️ What to Say to the Interviewer

"The key observation: effort = max single step, not sum. This is a minimax path problem.
I'll use Dijkstra but redefine the relaxation: new_eff = max(current_eff, |height_diff|).
Initialize dist[0][0]=0, all others=infinity. Min-heap on effort.
Pop minimum: if it's the goal, return immediately (Dijkstra guarantee = optimal).
Skip stale entries. For each valid neighbour, compute new_eff. If improved, push to heap.
Correctness: max is monotone (extending a path never decreases the maximum step).
Time O(m*n*log(m*n)). Space O(m*n)."
🔮 Likely Follow-up Questions
  • "What if you also want to print the actual path?" → Track a parent pointer matrix alongside dist; reconstruct by backtracking from (m-1,n-1)
  • "Can you solve this with Union-Find?" → Yes: sort all edges by weight, add edges in order until (0,0) and (m-1,n-1) are connected; the last added edge weight is the answer
  • "What if edges had negative weights?" → Dijkstra fails; but height differences are always ≥ 0, so this doesn't apply here
  • "How does this differ from Swim in Rising Water (#778)?" → Same minimax structure, Dijkstra applies there too; difference is edge vs node-based cost

7 Summary & Key Takeaways ▾
Minimax path = Dijkstra with max(dist, edge) instead of dist + edge in the relaxation
Dijkstra is valid here because max is monotone — extending a path never decreases its maximum step
The stale-entry check (if eff > dist[r][c]: continue) is required because heapq has no decrease-key
The first pop of the goal cell gives the optimal answer — return immediately
Binary Search + BFS is a valid alternative: easier to derive, slightly different time complexity (log H vs log(m·n))

🎯 When to Recognise the "Dijkstra on Max Edge" Pattern
  • Problem says "minimize the maximum" or "minimize the bottleneck" along a path
  • Grid or graph traversal where the cost is defined by the worst single edge
  • Edge weights are non-negative (required for Dijkstra's monotonicity argument)
  • "Swim in rising water" / "safest path" / "path with max-min value" — all same pattern
