LeetCode #1425

Constrained Subsequence Sum

Given an integer array nums and an integer k, return the maximum sum of a non-empty subsequence where the difference between consecutive chosen indices is at most k.

Hard Monotonic Queue DP + Monotonic Deque Time: O(n) Space: O(n)
1 Understanding the Problem ▾

We pick a non-empty subset of indices from nums such that consecutive chosen indices are at most k apart. We want to maximise the total sum of the chosen elements. Crucially, we can pick as few as one element — and the subsequence must be contiguous in terms of the index gap, not in terms of the actual positions.

💡 Key Insight: DP + Sliding Window Max

Define dp[i] = the maximum subsequence sum that ends at index i. To extend to index i, we can jump from any prior index j where i - k ≤ j < i — but only if that previous dp value was positive (a negative prefix only hurts). So:

dp[i] = nums[i] + max(0, max(dp[i-k..i-1]))

The answer is max(dp). The inner "max over a sliding window of size k" is exactly what a Monotonic Deque computes in O(1) amortized time.

✓ Optimal: nums=[10,2,-10,5,20], k=2

Pick indices 0,1,3,4
Gaps: 1,2,1 — all ≤ k=2 ✓
Sum: 10+2+5+20 = 37

✗ Invalid: pick indices 0,3,4

Gap 3-0 = 3 > k=2 ✗
Constraint violated.
Not a valid subsequence.

Edge cases: single element (answer = that element); all negatives (answer = max single element, since we must pick at least one); k ≥ n (any subset valid, reduces to a variant of max subarray); nums contains zeros (zeros are safe to include or skip — the dp naturally handles this).

2 The Algorithm: DP + Monotonic Deque ▾

We build the DP table from left to right. At each index i we need the maximum dp value over the window [i-k, i-1] — this is the classic sliding window maximum problem, solved in O(1) amortized using a monotonic deque that stores indices in decreasing order of their dp values.

1 Initialise: dp = nums[:]. Each index starts as a single-element subsequence. Create an empty deque — it will hold indices in order of decreasing dp value.
2 Evict expired (front): Pop from the deque front any index j where i - j > k. That index is now too far back — connecting from it to i would violate the gap constraint.
3 Extend (use window max): If the deque is non-empty and dp[deque[0]] > 0, add it to dp[i]: that is, dp[i] += dp[deque[0]]. We only extend when the predecessor's running sum is positive — otherwise, starting fresh at i gives a better answer.
4 Evict dominated (back): Pop from the deque back any index j where dp[j] ≤ dp[i]. Index i is both newer and at least as large — it will always be preferred over j for any future query.
5 Push current index to the back. The deque now reflects the window max for future iterations.
6 Return max(dp) — the largest "ending at index i" value across all positions is the global answer.
📐 The Recurrence Relation
dp[i] = nums[i] + max(0, max(dp[j] for j in range(max(0, i-k), i)))
answer = max(dp)
🔑 Why max(0, …)? — Connection to Kadane's Algorithm

If the best reachable predecessor has a negative total, extending it would reduce dp[i] below just using nums[i] alone. Clamping the predecessor contribution at 0 lets the subsequence "restart" at index i. This is structurally identical to Kadane's Algorithm, but restricted to a window of size k instead of unbounded history.

3 Why It Works: The Invariant ▾
🔐 The Deque Invariant

At any moment, the deque stores indices j₁ < j₂ < … < jₘ such that dp[j₁] > dp[j₂] > … > dp[jₘ] — increasing indices, strictly decreasing dp values. The front always holds the index with the maximum dp value in the current valid window.

Why back-eviction is safe: When processing index i, if some back index j has dp[j] ≤ dp[i], then for any future query at index i' > i: both j and i may be in the window [i'-k, i'-1], but i has a higher (or equal) dp value and a later index. Since we want the maximum, j is completely dominated and will never be the answer — safe to discard permanently.

Why front-eviction is safe: When i - deque[0] > k, the front index is too far from i. Any future index i' > i is even further from that front index. So it will never be a valid predecessor for any future query. Remove it permanently.

Amortized O(1): Every index enters the deque exactly once (via append) and exits exactly once (via front-eviction or back-eviction). The total work across all n iterations is O(n) — amortized O(1) per step.

⚠️ Critical: Order of Operations

The sequence must be: evict-front → use-front → evict-back → push-current. If you push current before evicting the back, you compare dp[i] against itself. If you use the front before evicting expired entries, you may use an out-of-window predecessor. Order matters.

⚠️ All-Negatives Edge Case

When all nums values are negative, every dp[i] = nums[i] — we never extend because the window max is always negative. The final max(dp) = max(nums), the least-negative element. This is correct: the constraint says non-empty, so we must pick at least one element. The algorithm handles this naturally since dp[deque[0]] > 0 will always be False.

4 The Code (Annotated) ▾

Solution 1 — DP + Monotonic Deque (Optimal — O(n) time, Interview Pick)

from collections import deque← O(1) popleft from both ends
def constrainedSubsetSum(nums, k):
dp = nums[:] ← dp[i] = max subseq sum ending at i; baseline = single element
dq = deque() ← monotonic deque of indices, decreasing dp values front-to-back
for i in range(len(nums)): ← process every index left to right
while dq and i - dq[0] > k:← STEP 1: evict expired front (gap too large)
dq.popleft() ← that index can never be a valid predecessor for i or any future index
if dq and dp[dq[0]] > 0: ← STEP 2: use window max only if it's positive (otherwise restart)
dp[i] += dp[dq[0]] ← extend: chain the best predecessor's running sum into dp[i]
while dq and dp[dq[-1]] <= dp[i]:← STEP 3: evict dominated back entries
dq.pop() ← current index i is newer and at least as good; back is useless
dq.append(i) ← STEP 4: push current index; deque stays decreasingly sorted by dp
return max(dp) ← answer: the best "ending at i" value across all i
⚠️ Bug Trap: dp[i] += dp[dq[0]] Modifies dp[i] In-Place

We initialise dp = nums[:] so dp[i] starts as nums[i]. When we do dp[i] += dp[dq[0]], the result is nums[i] + dp[best_predecessor] — correct. Do NOT write dp[i] = nums[i] + dp[dq[0]] after already modifying dp[i] elsewhere, as that would be redundant. Keep it simple: initialise to nums, then add the predecessor if helpful.

Solution 2 — Plain DP, O(nk) — Brute Force (Good Starting Point)

def constrainedSubsetSum_brute(nums, k):
n = len(nums)
dp = nums[:] ← same DP definition
for i in range(1, n):
best = 0 ← clamp at 0: restart if no positive predecessor
for j in range(max(0,i-k), i):← scan k prior indices — O(k) per iteration
best = max(best, dp[j])← find the window maximum naively
dp[i] += best ← extend if predecessor was positive
return max(dp)
🔬 Algorithm Deep-Dive: Monotonic Deque

What it is: A double-ended queue (deque) maintained so that stored values are always monotonically ordered (increasing or decreasing). For this problem we maintain decreasing dp values, so the front is always the current window maximum.

Core invariant: The deque holds indices in increasing order; their dp values are in strictly decreasing order. Front = window max, always.

Amortized O(1): Each of the n indices is pushed exactly once and popped exactly once (either from front when expired, or from back when dominated). Total pops across all n iterations ≤ n. Hence O(n) total.

When to recognize it: Any time a DP recurrence requires the max (or min) over a sliding window of the form [i-k, i-1], replace the O(k) scan with a monotonic deque for O(1) per step. Classic problems: Sliding Window Maximum (#239), Jump Game VI (#1696), this problem.

💡 Complexity Summary

Brute Force DP: Time O(nk), Space O(n) — fine when k is small (k=1 → O(n))
Monotonic Deque DP: Time O(n), Space O(n) — optimal; deque is at most O(k) ≤ O(n)
Max-Heap DP (alternative): Time O(n log n), Space O(n) — simpler to code; not optimal

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognise the DP Structure

"Subsequence with adjacent index gap ≤ k" immediately suggests DP. Define dp[i] = best subsequence sum ending at i. The transition looks at most k steps back.

Step 2: Write the Brute-Force O(nk) Version First

"For each i, scan the k prior dp values for the maximum. dp[i] = nums[i] + max(0, max_over_window). This is O(nk) — too slow when k ~ n. But it shows the structure clearly."

Step 3: Name the Bottleneck — Sliding Window Max

"The inner loop is just 'sliding window maximum of size k' over the dp array. I know that's solvable in O(1) amortized with a monotonic deque — same technique as LeetCode 239."

Step 4: Implement the Deque Carefully

"The deque stores indices with decreasing dp values. At each i: (1) evict expired front, (2) use front if positive, (3) evict dominated back, (4) push i. That's it — O(n) overall."

🗣️ What to Say to the Interviewer

"I'll define dp[i] = max subsequence sum ending at index i.
Transition: dp[i] = nums[i] + max(0, max dp[j] for j in [i-k, i-1]).
The max(0,...) lets us restart if all predecessors have negative sums.
Naive inner max is O(k) per step → O(nk) total.
But 'sliding window max' is the classic monotonic deque problem (LC 239).
I'll maintain a deque of indices with decreasing dp values.
At each i: evict expired front (gap > k), use front if dp > 0,
evict dominated back (dp[back] <= dp[i]), push i.
Total: O(n) time, O(n) space. Answer is max(dp)."
🔮 Likely Follow-up Questions
  • "k = 1?" → standard Kadane's algorithm on adjacent elements only
  • "k ≥ n?" → same code works; deque never evicts from the front
  • "What is the actual subsequence?" → add a parent array; trace back from the index where dp is max
  • "Can you do it with a heap?" → yes, O(n log n) with lazy deletion; deque is strictly better
  • "Minimum sum instead of maximum?" → flip the deque to increasing order; mirror image

7 Summary & Key Takeaways ▾
Define dp[i] = max subsequence sum ending at i — the answer is max(dp)
Transition: dp[i] = nums[i] + max(0, sliding window max of dp over [i-k, i-1])
The max(0,…) clamp allows "restarting" — identical to Kadane's trick
Monotonic deque gives O(1) amortized window max: each index pushed/popped exactly once
Operation order is critical: evict-front → use-front → evict-back → push-current
All-negative inputs: dp[i] = nums[i] always (no positive predecessor); answer = max(nums)

🎯 When to Recognise This Pattern
  • DP where dp[i] depends on max or min of dp[j] over a fixed window [i-k, i-1] → Monotonic Deque
  • "Subsequence" + "adjacent gap ≤ k" constraint → DP + windowed max/min

  • If k=1, it degenerates to Kadane's. If k≥n, any subset is valid.
