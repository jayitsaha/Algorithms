LeetCode #13

Roman to Integer

Given a roman numeral string s, return its integer value.

Easy Math Subtract if Smaller Before Time: O(n) Space: O(1)
1 Understanding the Problem ▾

Roman numerals use seven symbols. Normally you add them left to right from largest to smallest. But six special two-symbol combinations are subtractive — a smaller symbol before a larger one means you subtract the smaller.

I1
V5
X10
L50
C100
D500
M1000

The 6 subtraction pairs:

IV
5 − 1 = 4
IX
10 − 1 = 9
XL
50 − 10 = 40
XC
100 − 10 = 90
CD
500 − 100 = 400
CM
1000 − 100 = 900
💡 Key Insight

You don't need to memorize all 6 pairs. Notice that in every pair, the first symbol has a smaller value than the second. So one universal rule covers them all: if the current symbol's value is less than the next symbol's value, subtract it; otherwise add it.

✓ "MCMXCIV" = 1994

M → next M > C: +1000
C → next M (1000>100): −100
M → next X: +1000
X → next C (100>10): −10
C → next I: +100
I → next V (5>1): −1
V → last: +5 = 1994

✗ Naive Mistake: Just Add All

M+C+M+X+C+I+V
= 1000+100+1000+10
+100+1+5 = 2216
Wrong! Missing subtractions.

Edge cases: single character (return its value), all-additive like "VIII"=8, all-subtractive pairs, and the maximum "MMMCMXCIX"=3999.

2 The Algorithm: Left-to-Right with Look-Ahead ▾

We scan s from left to right. At each position i, we compare the value of s[i] to the value of s[i+1]. If current is smaller, subtract; otherwise add. The last character is always added (no "next" exists).

1 Build the lookup map: val = {'I':1, 'V':5, 'X':10, 'L':50, 'C':100, 'D':500, 'M':1000}. This is fixed-size (7 entries), so O(1) space.
2 Initialize total = 0. This will accumulate the integer value.
3 For each index i from 0 to len(s)-1: look up val[s[i]]. If i+1 < len(s) AND val[s[i]] < val[s[i+1]], then subtract val[s[i]] from total (subtraction pair). Otherwise add val[s[i]] to total.
4 Return total. One pass, no special-casing of individual pairs — the single "smaller before larger" rule handles all 6 subtraction cases automatically.
🔑 The Alternative: Right-to-Left Scan

An equally valid approach: scan from right to left, tracking prev (the last symbol seen). If current < prev, subtract current (it was "before" a larger one in the original order). Otherwise add. prev = max(prev, current) tracks the threshold. Same O(n)/O(1) complexity — some people find this direction more natural.

3 Why It Works: The Invariant ▾
🔐 The Invariant

After processing position i, total holds the correct integer representation of the Roman numeral string s[0..i]. We have applied exactly the right sign (+/-) to every character seen so far.

Why "current < next" is the only condition we need: In any valid Roman numeral, a smaller symbol can only appear before a larger one in one of the 6 sanctioned subtraction pairs. The problem guarantees valid input, so whenever we see val[s[i]] < val[s[i+1]], it must be one of those pairs. Subtracting gives the net contribution of the pair.

Worked example — "CM" encodes 900:

  • At C (i=0): val[C]=100, val[M]=1000. 100 < 1000 → subtract → total = 0 − 100 = −100
  • At M (i=1): val[M]=1000, no next. Add → total = −100 + 1000 = 900 ✓

Why the last symbol is always safe to add: There's no character after it, so the "next" comparison is false (guarded by i+1 < len(s)). In valid Roman numerals, the final character is never the subtractive half of a pair — a pair's smaller symbol is always immediately followed by the larger symbol it's paired with.

⚠️ Why Not Check All 6 Pairs Explicitly?

Some solutions hard-code: if s[i:i+2] in ('IV','IX','XL','XC','CD','CM'): total -= val[s[i]]. This works but requires maintaining the pairs list. The "compare next value" approach is shorter, more elegant, and generalizes to any valid Roman system with subtraction conventions — no list to maintain.

4 The Code (Annotated) ▾

Solution 1 — Left-to-Right, Compare Next (Optimal · Interview Pick)

def romanToInt(s: str) -> int:
val = {'I':1,'V':5,'X':10,'L':50, ← Fixed 7-symbol lookup map (O(1) space)
'C':100,'D':500,'M':1000}
total = 0 ← Running integer accumulator
for i in range(len(s)): ← Scan left-to-right, one character at a time
if (i + 1 < len(s) and ← Bounds check: is there a "next" character?
val[s[i]] < val[s[i+1]]):← Subtraction pair: current is smaller than next
total -= val[s[i]] ← Subtract (e.g. I in IX: −1, C in CM: −100)
else:
total += val[s[i]] ← Add normally (current ≥ next, or last symbol)
return total ← Final integer answer
⚠️ Critical: Bounds Check Before Comparing Next

The check i + 1 < len(s) MUST come before val[s[i+1]]. Python evaluates and short-circuit left-to-right, so if the bounds check fails (last character), the second condition is never evaluated and no IndexError occurs. Forgetting this check is the most common bug in this problem.

Solution 2 — Right-to-Left Scan (Elegant Alternative)

def romanToInt(s: str) -> int:
val = {'I':1,'V':5,'X':10,'L':50,'C':100,'D':500,'M':1000}
total = val[s[-1]] ← Start with the last character (always added)
for i in range(len(s)-2,-1,-1): ← Scan right-to-left, from second-to-last to first
if val[s[i]] < val[s[i+1]]: ← If current smaller than the one to its RIGHT = subtract
total -= val[s[i]] ← Subtraction pair (no bounds check needed — i+1 always valid)
else:
total += val[s[i]] ← Normal addition
return total
💡 Complexity Summary

Both solutions: Time O(n), Space O(1)
n = number of characters in s (max 15 for values 1–3999)
The 7-symbol map is constant-size — no dynamic allocation based on input
Practically: This problem runs in O(1) since Roman numerals encoding values ≤ 3999 have at most 15 characters.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Understand the Rules First (Don't Rush to Code)

Recite the 7 symbols and their values. Acknowledge the 6 subtraction pairs. This signals systematic thinking. The interviewer is watching how you decompose the problem.

Step 2: Notice the Unifying Pattern

Say: "I notice that in every subtraction pair — IV, IX, XL, XC, CD, CM — the first symbol has a smaller value than the second. So instead of checking all 6 pairs, I can use a single rule: if current value < next value, subtract; else add."

Step 3: Trace Through an Example

Trace "MCMXCIV" = 1994 on the whiteboard before writing code: M(+1000), C(−100 because next is M), M(+1000), X(−10 because next is C), C(+100), I(−1 because next is V), V(+5). 1000−100+1000−10+100−1+5 = 1994. ✓

Step 4: State Complexity and Mention Alternative

"O(n) time, O(1) space since the symbol map has fixed 7 entries. There's also a right-to-left variant that's equally clean — track the previous character's value and subtract when current < previous-right."

🗣️ What to Say to the Interviewer

"The key observation: in all 6 Roman subtraction pairs, the first symbol is always smaller than the second.
So I scan left-to-right. For each symbol: compare its value to the next symbol's value.
If current < next → subtract (it's the subtractive half of a pair).
Otherwise → add it. The last symbol always gets added — no comparison needed.
This handles all 6 pairs with one rule. O(n) time, O(1) space."
🔮 Likely Follow-up Questions
  • "Integer to Roman?" (LC #12) → Greedy: maintain a list of (value, symbol) pairs in descending order; repeatedly use largest symbol that fits.
  • "What if input could be invalid?" → Validate: no symbol repeats more than 3 times (4+ is invalid), only 6 valid subtraction pairs.
  • "Why does this work for CM but not IC?" → IC is not valid Roman. The problem guarantees valid input, so we don't need to defend against invalid combinations.
  • "Can you do it without a dictionary?" → Yes, with an if-elif chain mapping char to value — but a dict is cleaner and O(1) lookup.

🎯 When to Recognize the "Subtract if Smaller Before" Pattern
  • Encoding systems where a symbol's sign depends on its neighbor
  • "Look-ahead" scan: decide current element's contribution based on next element
  • Any left-to-right linear conversion with context-dependent rules
  • The look-ahead idiom also appears in temperature problems ("next warmer day") and stock spans
