LeetCode #4

Median of Two Sorted Arrays

Given two sorted arrays nums1 and nums2, return the median of the two sorted arrays combined. The run time complexity must be O(log(m+n)).

Hard Binary Search Binary Search on Partition Time: O(log(min(m,n))) Space: O(1)
1 Understanding the Problem ▾

The median is the middle value that divides a sorted dataset into two equal halves. For an odd total number of elements, the median is the single middle value. For an even total, it is the average of the two middle values.

💡 Key Insight

We don't need to merge the arrays. We need to find a partition point in each array such that: (1) the combined left halves and right halves are equal in size, and (2) every element in the combined left half is ≤ every element in the combined right half. Binary searching for the correct partition in the smaller array gives us O(log(min(m,n))).

Odd total → single median

nums1 = [1, 3]
nums2 = [2]
merged = [1, 2, 3]
Median = 2.0

Even total → average two middles

nums1 = [1, 2]
nums2 = [3, 4]
merged = [1, 2, 3, 4]
Median = (2+3)/2 = 2.5

The core idea: Cut both arrays at positions i and j so the left portions together have exactly ⌊(m+n+1)/2⌋ elements. When the cut is at the right place, the biggest element in the left half equals the median (odd total) or the average with the smallest element in the right half (even total).

Edge cases to consider: one array is empty (answer is the other's median), all of one array is smaller than all of the other, arrays of length 1, ties and duplicate values, both arrays of equal length.

2 The Algorithm: Binary Search on Partition ▾

We binary-search on how many elements from nums1 go into the left half. Once that count i is fixed, the count from nums2 is automatically j = half_len - i. We look for the i where the boundary elements "cross over" correctly.

1 Always search the shorter array. Swap so nums1 is shorter. Binary search range is [0, m] where m = len(nums1). This keeps j in valid range for all i values.
2 Pick the cut in nums1. i = (lo + hi) // 2 — take i elements from nums1 for the left half. Then j = half_len - i elements from nums2. half_len = (m + n + 1) // 2 (the +1 handles odd totals correctly).
3 Read the four boundary values. n1L = nums1[i-1] or −∞ if i=0. n1R = nums1[i] or +∞ if i=m. Similarly n2L = nums2[j-1] and n2R = nums2[j] with sentinels.
4 Check validity: n1L ≤ n2R AND n2L ≤ n1R. If both hold, the partition is correct. If n1L > n2R, we took too many from nums1 (hi = i-1). If n2L > n1R, we took too few (lo = i+1).
5 Return the median. max_left = max(n1L, n2L). For odd total, return max_left. For even total, return (max_left + min(n1R, n2R)) / 2.
🔑 Why Binary Search Works Here

As i increases (more elements from nums1 on the left), n1L increases monotonically and n2R decreases monotonically. There is exactly one crossing point where n1L transitions from ≤ n2R to > n2R — binary search locates this in O(log m) steps.

3 Why It Works: The Invariant ▾
🔐 The Partition Invariant

At the valid partition: every element in the combined left half ≤ every element in the combined right half, and the left half has exactly ⌊(m+n+1)/2⌋ elements. Since both arrays are already individually sorted, only the four boundary values need to be compared — all other elements on each side are guaranteed ordered by the existing sort.

Why only four comparisons? If n1L ≤ n2R and n2L ≤ n1R, then: every element to the left of the cut in nums1 is ≤ n1L ≤ n2R ≤ every element to the right of the cut in nums2, and similarly for nums2's left side. The sorted order within each array propagates the inequality through all elements — we only need to check where the two arrays "meet at the boundary."

Why does the binary search always terminate? The function of n1L - n2R is monotonically non-decreasing as i increases (more elements from nums1 means n1L gets larger while n2R gets smaller). There's at most one valid partition; binary search finds it or proves it doesn't exist (impossible with valid input).

✓ Valid — Median Found
n1L ≤ n2R
n2L ≤ n1R
✗ n1L > n2R — i too large
Set hi = i - 1
(Take fewer from nums1)
⚠️ The ±∞ Sentinels

When i = 0, nums1 contributes nothing to the left half. We set n1L = -∞ so the condition n1L ≤ n2R is trivially satisfied. When i = m, all of nums1 is on the left; we set n1R = +∞ so n2L ≤ n1R is trivially satisfied. This unifies all edge cases without special-casing.

Why search the smaller array? We need j = half_len - i to satisfy 0 ≤ j ≤ n. With m ≤ n, every value of i in [0..m] keeps j in a valid range. If we searched the larger array, small i values could make j negative.

4 The Code (Annotated) ▾

Solution 1 — Brute Force: Merge and Find (O((m+n)log(m+n)) time, O(m+n) space)

def findMedianSortedArrays_brute(nums1, nums2):
merged = sorted(nums1 + nums2) ← Concatenate both arrays and sort
n = len(merged)
if n % 2 == 1: ← Odd: single middle element
return float(merged[n // 2])
return (merged[n//2-1] + merged[n//2]) / 2 ← Even: average of two middles

Solution 2 — Two-Pointer Without Merge Array (O(m+n) time, O(1) space)

def findMedianSortedArrays_linear(nums1, nums2):
m, n = len(nums1), len(nums2)
total = m + n; mid = total // 2
i = j = 0; prev = curr = 0
for k in range(mid + 1): ← Advance to the (mid+1)-th element
prev = curr ← Keep second-to-last for even case
if i < m and (j >= n or nums1[i] <= nums2[j]):
curr = nums1[i]; i += 1
else:
curr = nums2[j]; j += 1
if total % 2 == 1: return float(curr)← Odd: the element we just moved to
return (prev + curr) / 2 ← Even: average prev and curr

Solution 3 — Binary Search on Partition (Optimal, O(log(min(m,n))) — Interview Pick ✓)

def findMedianSortedArrays(nums1, nums2):
if len(nums1) > len(nums2): ← Ensure nums1 is shorter (we search it)
nums1, nums2 = nums2, nums1
m, n = len(nums1), len(nums2)
half_len = (m + n + 1) // 2 ← Size of left half (biased +1 handles odd totals)
lo, hi = 0, m ← i can be 0..m (how many from nums1 go left)
while lo <= hi:
i = (lo + hi) // 2 ← Mid: take i elements from nums1 for left half
j = half_len - i ← j elements from nums2 complete the left half
n1L = nums1[i-1] if i > 0 else float('-inf')← Left edge of nums1 (or -inf)
n1R = nums1[i] if i < m else float('inf') ← Right edge of nums1 (or +inf)
n2L = nums2[j-1] if j > 0 else float('-inf')← Left edge of nums2 (or -inf)
n2R = nums2[j] if j < n else float('inf') ← Right edge of nums2 (or +inf)
if n1L <= n2R and n2L <= n1R: ← Perfect partition!
max_left = max(n1L, n2L) ← Largest element in left half
if (m + n) % 2 == 1: ← Odd total: median IS the left half's max
return float(max_left)
return (max_left + min(n1R, n2R)) / 2 ← Even: avg two middles
elif n1L > n2R: ← Took too many from nums1
hi = i - 1 ← Move cut left in nums1
else: ← n2L > n1R: took too few from nums1
lo = i + 1 ← Move cut right in nums1
⚠️ Why (m + n + 1) // 2 with the extra +1?

For odd total (e.g. m+n=5): (5+1)//2 = 3. The left half gets 3 elements — the extra one is the median itself. For even total (e.g. m+n=6): (6+1)//2 = 3. Both halves get 3 elements, and we average the max-left and min-right. The +1 before flooring ensures correct behavior in both cases with a single formula.

💡 Complexity Summary

Brute Force (merge+sort): O((m+n)log(m+n)) time, O(m+n) space — never use if log constraint given
Two-Pointer (count to median): O(m+n) time, O(1) space — linear, better but not optimal
Binary Search on Partition: O(log(min(m,n))) time, O(1) space — optimal, only acceptable solution

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Notice the O(log) Constraint — Binary Search Required

The problem explicitly requires O(log(m+n)). Any O(m+n) or worse solution is invalid. Logarithmic complexity on sorted data means binary search. The question is: what are we binary-searching on?

Step 2: Reframe from "Find Middle Element" to "Find Correct Partition"

The median splits the merged dataset into two equal halves. Instead of asking "which index is the median?" ask "how should I split both arrays so left half = right half in size, and max(left) ≤ min(right)?" This reformulation reveals a binary-searchable structure.

Step 3: Identify the One Degree of Freedom

Once we decide how many elements from nums1 go to the left half (i), everything is determined: the count from nums2 is j = half_len - i. So i is our single binary search variable. Range: [0, m] where m is the shorter array length.

Step 4: State the Validity Condition Clearly

"The partition is valid when nums1[i-1] ≤ nums2[j] AND nums2[j-1] ≤ nums1[i]. This ensures every left-half element is ≤ every right-half element. If nums1[i-1] is too large, move i left. If nums2[j-1] is too large, move i right." Say this out loud — it shows you understand the invariant.

Step 5: Handle Boundaries with Sentinels

Mention that when i=0 (nothing from nums1 on left), you use -∞ for n1L, and when i=m (all of nums1 on left), you use +∞ for n1R. This avoids special-casing and keeps the validity condition uniform.

🗣️ What to Say to the Interviewer

"The O(log) constraint means binary search. I'll reformulate the problem: instead of finding the median directly, I'll find a partition of both arrays where everything on the left ≤ everything on the right, and the halves are equal in size.

I'll binary search on i — the number of elements from the shorter array nums1 in the left half. Then j = half_len - i comes from nums2. The partition is valid when nums1[i-1] ≤ nums2[j] and nums2[j-1] ≤ nums1[i]. I adjust lo/hi based on which condition fails.

For edge cases (i=0 or i=m), I use ±infinity sentinels. Once valid, the median for odd total is max(n1L, n2L), and for even it's average with min(n1R, n2R). Time: O(log(min(m,n))), Space: O(1)."
🔮 Likely Follow-up Questions
  • "What if arrays aren't sorted?" → Sort first: O((m+n)log(m+n)); or use quickselect O(n) avg
  • "Find the kth smallest in two sorted arrays?" → Generalize: binary search eliminating k/2 candidates per step
  • "Median of K sorted arrays?" → Merge with min-heap O(N log k), then find median
  • "Can you do this recursively?" → Yes, binary elimination of k/2 elements recursively

7 Summary & Key Takeaways ▾
The median = the element(s) at the boundary of a 50/50 partition of all elements
Binary search on i (cut position in shorter array) — O(log(min(m,n)))
j is determined by i: j = half_len - i (total left-half size is fixed)
Validity: n1L ≤ n2R AND n2L ≤ n1R — only four boundary values need checking
Use ±∞ sentinels when cut is at the very start or end of an array
Always search the shorter array to keep j in a valid range
half_len = (m+n+1)//2 handles both odd and even totals uniformly
ApproachTimeSpaceNotes
Brute Force (merge + sort) O((m+n)log(m+n)) O(m+n) Violates the O(log) constraint
Two-Pointer (count to mid) O(m+n) O(1) Linear — better, still not acceptable
Binary Search on Partition ✓ O(log(min(m,n))) O(1) Optimal — the correct interview answer
🎯 When to Recognize Binary Search on Partition
  • Two (or more) sorted arrays + O(log) time required → binary search on partition
  • "Find the kth smallest element across two sorted arrays" → same technique, generalized
  • Any problem where choosing a split in one structure determines a split in another
  • When merging would be trivial but too slow, and you need to short-circuit with monotonic condition
