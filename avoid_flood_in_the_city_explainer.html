LeetCode #1488

Avoid Flood in The City

Given a schedule of rain days and sunny days, choose which lake to drain each sunny day so no lake ever floods. Return a valid drain schedule, or [] if impossible.

Medium Greedy Greedy Empty Selection Time: O(n log n) Space: O(n)
1 Understanding the Problem ▾

You're given an array rains of length n. Each day is one of two types:

  • rains[i] > 0: Lake rains[i] gets rained on and becomes full. If it was already full — the city floods. Return [] immediately.
  • rains[i] = 0: Sunny day. You must pick exactly one lake and drain it (empty it). Choose wisely!

Return any valid answer array where rain days have -1 and sunny days have the lake number you chose to drain. If no valid schedule exists, return [].

💡 Key Insight

Sunny days are limited free actions. When a lake is about to be rained on a second time, you need to have already drained it on some previous sunny day. The greedy choice: when the second rain arrives on lake k, find the earliest available sunny day in the window between the two rains, and retroactively assign it to drain lake k. Earliest = most future flexibility.

✓ Solvable: [1, 2, 0, 0, 2, 1]

Day 0: rain lake 1 → full
Day 1: rain lake 2 → full
Day 2: sunny → ans[2] = ? (decide later)
Day 3: sunny → ans[3] = ? (decide later)
Day 4: rain lake 2 again! → assign day 2 to drain lake 2
Day 5: rain lake 1 again! → assign day 3 to drain lake 1
→ ans = [-1,-1,2,1,-1,-1] ✓

✗ Impossible: [1, 2, 0, 1, 2]

Day 0: rain lake 1 → full
Day 1: rain lake 2 → full
Day 2: sunny → only 1 slot
Day 3: rain lake 1 again!
Day 4: rain lake 2 again!
Can't drain both 1 & 2 with only 1 sunny.
→ [] (impossible)

Edge cases: All sunny days (drain lake 1 as placeholder). No sunny days but a lake rains twice — always impossible. A sunny day after all rains — assign lake 1 (harmless). Two rains of same lake with zero sunnies between — immediately impossible.

2 The Algorithm: Greedy with Sorted-Set Look-ahead ▾

We scan left to right. Sunny days are collected into a sorted set. When a rain threatens a full lake, we binary-search for the earliest usable sunny day and retroactively assign it. This deferred assignment is the heart of the greedy strategy.

1 Initialize: full = {} (lake → day it last rained), sunny = SortedList() (available sunny day indices), ans = [-1]*n.
2 On a sunny day (rains[i] = 0): Add i to sunny. Set ans[i] = 1 as a placeholder (lake 1 is harmless, it'll be overwritten if needed).
3 On a rain day for lake r — first rain: Lake r is not in full, so no conflict. Record full[r] = i. ans[i] = -1.
4 On a rain day for lake r — second rain (conflict!): Binary-search sunny for the earliest index j where full[r] < j < i. If found: set ans[j] = r, remove j from sunny, update full[r] = i. If NOT found: return [].
5 Return ans: Any remaining sunny days keep the placeholder value 1. The full schedule is valid.

🔑 Why a Sorted Set + Binary Search?

When lake r gets its second rain on day i, the valid drain window is (full[r], i) — strictly between the two rains. We need the earliest available sunny index in this open interval. A SortedList supports O(log n) bisect_right(full[r]) to find that index, and O(log n) remove(j) to consume it. Total: O(n log n) time.

3 Why It Works: The Greedy Invariant ▾
🔐 The Invariant

After processing day i, every lake that is currently "full" either: (a) has no future rain scheduled (fine for now), or (b) will be assigned a drain action on the earliest available sunny day in its conflict window the moment that second rain arrives. No full lake is ever left unprotected against an imminent rain.

Why picking the earliest sunny day is optimal (Exchange Argument): Suppose lake k conflicts on day i and we have sunny days j1 < j2 both in the valid window. We pick j1 (earliest). Could this hurt us? Only if some future lake k' needs j1 specifically. But k' would need a sunny day in its own window — if j1 works for k', so does j2 (since j2 > j1 is also in the window if j1 was). So assigning j1 to k and leaving j2 for k' is never worse than the reverse.

Impossibility is tight: If no sunny day exists in the conflict window (last_rain[r], today), no amount of future planning can help — the conflict has already occurred. We must return [] immediately.

⚠️ Common Mistake: Using Any Sunny Day

A greedy that picks the most recent or latest available sunny day in the window is also correct, but picking a day outside the window (before last_rain[r]) would drain the lake before it was even full — which does nothing. Always restrict the search to strictly after the last rain of that lake.

4 The Code (Annotated) ▾

Solution 1 — Greedy + SortedList (Optimal — Interview Pick)

from sortedcontainers import SortedList ← O(log n) insert, remove, bisect
def avoidFlood(rains):
n = len(rains)
ans = [-1] * n ← default: rain days stay -1
full = {} ← lake → day of its most recent rain
sunny = SortedList() ← sorted indices of unused sunny days
for i, r in enumerate(rains):
if r == 0: ← sunny day: collect now, assign later
sunny.add(i) ← O(log n) insert
ans[i] = 1 ← placeholder; overwritten if assigned
elif r in full: ← lake r is already full — conflict!
# Need a sunny day strictly after full[r] and strictly before i
idx = sunny.bisect_right(full[r])← first sunny index > last rain of r
if idx == len(sunny) or sunny[idx] >= i:← no valid sunny in window → impossible
return []
j = sunny[idx] ← earliest valid sunny day
ans[j] = r ← retroactively assign: drain lake r on day j
sunny.remove(j) ← consume this sunny day, O(log n)
full[r] = i ← lake r refills today
else: ← first rain on lake r
full[r] = i ← record that lake r is now full
return ans ← leftover sunny days keep placeholder 1
⚠️ Why check sunny[idx] ≥ i, not just idx == len(sunny)?

bisect_right(full[r]) finds the first sunny index strictly after full[r]. But we also need that index to be strictly before today's rain day i. If sunny[idx] ≥ i, the found day is today or in the future — we can't drain the lake "after" the second rain already hits. So we return [].

Solution 2 — Brute Force (O(n²), for understanding only)

def avoidFlood_brute(rains): ← O(n²) — TLE for n=10⁵
ans = [1] * len(rains) ← start with all sunny = drain lake 1
full = {} ← tracks currently full lakes
for i, r in enumerate(rains):
if r == 0: ← sunny day: nothing to do here
continue
ans[i] = -1 ← rain day output is -1
if r in full: ← conflict: lake r already full
found = False
for j in range(full[r]+1, i): ← O(n) scan backward in window
if rains[j] == 0 and ans[j] != -1:
ans[j] = r; found = True; break
if not found: return []
full[r] = i
return ans
💡 Complexity Summary

Greedy + SortedList (Optimal): Time O(n log n), Space O(n)
Brute Force: Time O(n²), Space O(n) — fails for n=10⁵
Alternative (no sortedcontainers): Pre-collect all sunny indices sorted, then use Python's built-in bisect module for the same O(log n) binary search. Mark consumed sunnies in a set. Same complexity, stdlib only.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify the Scheduling Structure

Key signals: "free action on sunny days," "lakes fill and must be drained," "limited resources with deadlines." This is a scheduling problem. Rains create deadlines, sunny days are the slots. Greedy scheduling: match each deadline to the earliest available slot in its window.

Step 2: Propose Brute Force

"When a second rain hits lake k, I can scan backward through its window to find the first unused sunny day and assign it there. O(n) per conflict, O(n²) overall. Correct but too slow for n=10⁵."

Step 3: Optimize with Binary Search

"The backward scan is slow because we're searching linearly. If I maintain sunny day indices in a sorted data structure, I can binary-search for the first index strictly after last_rain[r] in O(log n). SortedList from sortedcontainers, or pre-sort + bisect from stdlib."

Step 4: State the Greedy Correctness

"Greedy choice: always pick the earliest available sunny day in the window. Exchange argument: using earliest leaves later sunny days for future conflicts — it's never worse than picking later. Prove: if j1 < j2 both valid for k, swapping them doesn't help any other lake."

🗣️ What to Say to the Interviewer

"I'll scan left to right, maintaining two structures:
1) full: dict mapping lake → day it last rained
2) sunny: SortedList of available sunny day indices

On a sunny day: add index to sunny, set ans[i]=1 placeholder.
On a rain day for lake r:
- If r not in full: first rain, record full[r]=i.
- If r in full: conflict! Binary-search sunny for earliest j > full[r] and j < i.
If found: ans[j]=r, remove j from sunny, update full[r]=i.
If not found: return [] — impossible.
Return ans. Time O(n log n), Space O(n)."
🔮 Likely Follow-up Questions
  • "Can you do it without sortedcontainers?" → Pre-collect all sunny indices in sorted order (one pass), then use bisect.bisect_right. Mark used indices in a set. O(n log n) time, stdlib only.
  • "What if we must drain optimally (minimize total sunny days used)?" → Our greedy naturally minimizes usage — we only drain when a conflict forces it.
  • "What if a lake can hold multiple rain units before flooding?" → Extension: track a counter per lake; conflict only when counter overflows capacity.
  • "Why does picking the earliest sunny day guarantee optimality?" → Exchange argument: no lake needs a specific earlier sunny day if we could've used a later one for it too.
