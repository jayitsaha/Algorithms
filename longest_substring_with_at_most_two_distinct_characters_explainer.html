LeetCode #159

Longest Substring with At Most Two Distinct Characters

Given a string s, return the length of the longest substring that contains at most two distinct characters.

Medium Sliding Window Hash Map ≤ 2 Keys Time: O(n) Space: O(1)
1 Understanding the Problem ▾

We have a string and must find the longest contiguous portion (substring) that uses no more than two different letters. For "eceba", the substring "ece" (length 3) is the best: it contains only 'e' and 'c'. The full string uses 4 distinct characters — that violates the constraint.

💡 Key Insight

Think of a sliding frame that glides along the string. The frame can expand freely as long as it holds at most 2 distinct letters. The instant a 3rd letter enters, the left wall of the frame must slide right until the intruder falls out. We track the largest frame width ever seen.

✓ Valid window: "ece"

Characters: {e, c} = 2 distinct ✓
Length = 3
This is the answer for "eceba"

✗ Invalid window: "eceb"

Characters: {e, c, b} = 3 distinct ✗
Exceeds the limit of 2
Must shrink from the left

Edge cases: If the string has 0 or 1 distinct characters, the entire string is a valid answer. If all characters are distinct (e.g., "abc"), the window always holds exactly 2 chars, and we return 2 (or 1 if length is 1). Empty string returns 0.

🔑 Substring vs Subsequence

A substring is a contiguous slice of characters — no skipping. "ce" is a substring of "eceba" (positions 1–2), but "ea" is NOT a substring (there is no adjacent "ea"). We cannot pick non-adjacent characters.

2 The Algorithm: Variable Sliding Window with Last-Index Map ▾

This is the variable-size sliding window pattern. Two pointers (left, right) define the current window. We also keep a hash map char_map that maps each character currently in the window to its most recent index. When the map grows beyond 2 keys, the window is invalid and we evict the character whose last occurrence is leftmost.

1 Initialize: char_map = {}, left = 0, result = 0.
2 Expand right: For each index right, record char_map[s[right]] = right (last seen position). This adds the new char or updates its index if it was already present.
3 Check validity: If len(char_map) > 2, the window has 3 distinct chars — invalid. Find leftmost = min(char_map.values()). That character's last occurrence is the furthest left, so it's the one that "leaves" the window first as we shrink.
4 Evict and shrink: del char_map[s[leftmost]] and set left = leftmost + 1. Now the map has exactly 2 keys again.
5 Update result: result = max(result, right - left + 1). This safely fires after every step — the window is always valid at this point.

Why store last index instead of count? When we need to evict a character, we must know exactly where to set left. The last index of the evicted character tells us: "everything up to and including this position is outside our new window." We set left = last_index + 1 directly — no scanning needed.

⚠️ if vs while

We add exactly one character per iteration. The map size can overshoot by at most 1 (from 2 to 3). A single if suffices to restore validity — no loop needed. Using while is correct but wastes a condition check. Smart interviewers notice.

3 Why It Works: The Invariant ▾
🔐 The Invariant

Every time we evaluate right - left + 1, the substring s[left..right] contains at most 2 distinct characters. The invariant is maintained by restoring len(char_map) ≤ 2 before any measurement.

Why we never miss the optimal window:

  1. right visits every possible window-end exactly once.
  2. For each window-end, we shrink left to the minimum valid position — any larger left would give a shorter window. So we find the longest valid window ending at each right.
  3. We keep a running maximum — so the global best is never missed.

Why evict the character with the minimum last-index? When 3 chars coexist, exactly one of them must be removed to restore the 2-char constraint. The "oldest" character — the one whose most recent appearance is leftmost — is the correct choice: advancing left past its position guarantees it exits the window while keeping the other two characters as long as possible (maximizing future window length).

💡 Analogy: Hotel Room Limit

A hotel corridor has a rule: at most 2 different guests per active zone. As new guests check in from the right, the moment a 3rd distinct guest appears, you force out the guest who checked in earliest (their room is leftmost). You record the longest stretch where the rule was satisfied.

Edge case proof: If the string has ≤ 2 distinct characters, len(char_map) never exceeds 2 — we never shrink and return the full length n. If n=0, the for loop doesn't execute and we return 0 correctly.

4 The Code (Annotated) ▾
def lengthOfLongestSubstringTwoDistinct(s: str) -> int:
char_map = {} ← char → last index it appeared in current window
left = 0 ← left boundary (inclusive) of sliding window
result = 0 ← longest valid window length seen so far
for right in range(len(s)): ← right boundary expands by one each iteration
char_map[s[right]] = right ← add/update char. If new: 3rd key may appear.
if len(char_map) > 2: ← window invalid: 3 distinct chars
leftmost = min(char_map.values()) ← index of char we'll evict (O(k)=O(1))
del char_map[s[leftmost]] ← remove that char from the map
left = leftmost + 1 ← shrink window past evicted char's last index
result = max(result, right - left + 1) ← window valid; record if best
return result ← best length seen across all valid windows
⚠️ Common Mistake: Using counts instead of last indices

If you store counts instead of last indices, eviction requires walking left forward one step at a time in a while loop until the evicted char's count hits zero. That's O(n) per eviction in the worst case. Last-index storage makes eviction O(1): set left = leftmost + 1 in one step.

💡 Space is truly O(1)

char_map holds at most 3 entries at any moment — we evict immediately after the 3rd appears. Even if the alphabet is huge (Unicode), our active map is bounded by our constraint: ≤ 2+1 = 3 keys. Space does not grow with input length.

Time complexity: O(n) — right moves forward n times. left only moves forward (amortized O(1) per step). min(char_map.values()) is O(k) where k ≤ 3 — constant. Total: O(n).

Space complexity: O(1) — map size is bounded by 3 regardless of input.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Sliding Window Signal

"Longest substring with at most K [some constraint]" is almost always a variable sliding window problem. The key words are: contiguous, substring, longest/shortest, at most/exactly K. Whenever you see these together, think two pointers.

Step 2: Define validity and choose your data structure

Valid window = at most 2 distinct chars. Track with a hash map. The critical design decision: store last indices (not counts) so that eviction is O(1) — you jump left directly rather than scanning.

Step 3: Code it up in 8 lines

The solution is compact. Start with empty map, left=0, result=0. Loop on right: update map, check size, evict if needed, update result. Return result.

Step 4: Proactively generalize to K distinct

Before the interviewer asks: "This trivially generalizes to K distinct characters — just replace 2 with k. That's LeetCode #340." This shows you understand the pattern, not just the problem.

🗣️ What to Say to the Interviewer

"I see 'longest substring with constraint' — that's a sliding window. Left and right pointers define the window. I'll use a hash map from char to its last-seen index. Right expands greedily each step. When the map hits 3 keys, I find the entry with the minimum value — that's the character whose last occurrence is leftmost — evict it, and jump left to min_index + 1. Then record window size. Time: O(n). Space: O(1) — the map is bounded by 3 entries. This generalizes to K distinct by changing the threshold from 2 to k."
🎯 Follow-Up Questions to Expect
  • Generalize to K distinct characters? → Change 2 to k. O(n) time, O(k) space (bounded).
  • Can you avoid the min(values) call? → Use OrderedDict or a deque for O(1) eviction — but since k≤3 it doesn't matter asymptotically.
  • What if the string has Unicode characters? → Hash map still works; key space is larger but map size is still bounded by k+1.
  • What if we want exactly 2 distinct? → (exactly K) = (at most K) - (at most K-1). LeetCode #992.

7 Summary & Key Takeaways ▾
Variable sliding window: right expands freely, left only shrinks when the 2-distinct constraint breaks
Store last-seen index (not count) in the map — enables O(1) eviction via min(values)
The map holds at most 3 entries at any moment — truly O(1) space regardless of string length
Use if not while — we add exactly one character per step, so the map overshoots by at most 1
Generalizes to K distinct: change the threshold from 2 to k (LeetCode #340, "Fruit Into Baskets" is this exact problem disguised)

Brute Force (all substrings) O(n²) O(1) Check every [i,j] pair — too slow for large inputs
