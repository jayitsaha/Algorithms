LeetCode #94

Binary Tree Inorder Traversal

Given the root of a binary tree, return the inorder traversal of its node values.

Easy Trees DFS Inorder Time: O(n) Space: O(h)
1 Understanding the Problem ▾

Tree traversal means visiting every node in a defined order. There are three classic depth-first orders — preorder, inorder, postorder — distinguished by when the root is visited relative to its children. Inorder visits: Left subtree first, then the root, then the right subtree.

💡 Key Insight: Why Inorder Matters

For a Binary Search Tree (BST), inorder traversal always yields values in strictly ascending (sorted) order. This is the single most important property of inorder in interview problems. Any time a problem says "BST" and "sorted" in the same breath, think inorder.

Consider the tree [1, null, 2, 3] — root is 1, right child is 2, and 2's left child is 3. The three traversal orders visit the same nodes differently:

Preorder
1→ 2→ 3 Root → Left → Right → output [1, 2, 3]
Inorder ★
1→ 3→ 2 Left → Root → Right → output [1, 3, 2]
Postorder
3→ 2→ 1 Left → Right → Root → output [3, 2, 1]
✓ Inorder rule applied step by step

Visit node 1: left(1)=null → visit 1
Then go right to 2
Visit node 2: left(2)=3 → go to 3
Visit node 3: left(3)=null → visit 3
Back to 2: right(3)=null → visit 2
Result: [1, 3, 2]

Edge Cases

root = null → return []
Single node → [val]
Left-skewed → deepest first
Right-skewed → root first
All nodes same val → OK

Memory trick: "IN" = root is in-between left and right. "PRE" = root is before both. "POST" = root is after both. Once you remember this, you'll never mix them up.

2 The Algorithm: Iterative Stack Simulation ▾

Recursion naturally expresses inorder traversal, but the iterative version using an explicit stack is the interview-preferred answer. It makes the implicit call stack explicit and avoids recursion depth limits on pathological inputs. The key idea: "dive left, pop to visit, then pivot right."

1 Initialize: Create an empty stack and set curr = root. These are our only state variables: curr tracks where we're exploring, and stack holds nodes we've entered but haven't visited yet.
2 Dive left (push): While curr is not null, push curr onto the stack and go left: curr = curr.left. This traces the entire left spine of the current subtree. We push each node as a promise: "I'll visit you after your left subtree is done."
3 Pop and visit: When curr is null (no more left children), pop the top of the stack into curr. Append curr.val to result. This node's entire left subtree has been visited — now we visit the node itself.
4 Pivot right: Set curr = curr.right. If a right child exists, the next iteration will push its left spine. If null, the next outer loop iteration will pop the next pending node from the stack.
5 Terminate: The outer loop runs while curr or stack. When both are exhausted — curr is null and the stack is empty — every node has been visited. Return result.
🔑 Recursive Alternative (mention first in interviews)

def dfs(node): dfs(node.left); result.append(node.val); dfs(node.right) — Three lines. Directly encodes the definition. O(n) time, O(h) space (call stack). Perfect first answer; then offer iterative as follow-up. For a balanced tree h=O(log n); for a right-skewed tree h=O(n).

3 Why It Works: The Invariant ▾
🔐 The Core Invariant

At any point during the loop: every node on the stack has had its entire left subtree already processed, but has not been visited itself yet, and its right subtree is untouched. The stack is a list of "owed visits" in LIFO order — the shallowest unpaid debt is on top.

Why pushing the left spine works: Inorder demands we visit the leftmost descendant before anything else. By pushing every node along the left spine onto the stack, we ensure the last-pushed (bottommost-leftmost) node is popped first. LIFO stack + left-first pushing = correct inorder sequencing.

Why popping then going right gives correct order: When we pop node X, it means: "all of X's left subtree nodes were pushed and popped before X." So X.left subtree is fully visited. Visiting X now and then pivoting to X.right gives exactly: Left ✓ → Root ✓ → Right. That's inorder at every node.

Why curr or stack is the right stop condition: After popping a node and setting curr = node.right, curr may be a new node not yet on the stack. If we only checked stack, we'd exit too early. Both must be exhausted simultaneously.

⚠️ Most Common Mistake: Visiting on Push

If you call result.append(curr.val) inside the inner while curr loop (i.e., on push), you get preorder traversal — root before children. For inorder, always append on pop. The push just means "remember to visit this node after its left side is done."

What happens with a null root? curr = None and stack = [] → outer loop condition None or [] is False → loop never runs → return []. Zero special-case handling required.

4 The Code (Annotated) ▾

Solution 1 — Iterative Stack (Interview Pick)

def inorderTraversal(root): ← accepts TreeNode root, may be None
result, stack = [], [] ← result accumulates output; stack is our explicit call stack
curr = root ← curr = current node being explored (starts at root)
while curr or stack: ← continue while work remains (curr to explore OR stack to drain)
while curr: ← inner loop: descend the left spine
stack.append(curr) ← push: "visit me after my left subtree is done"
curr = curr.left ← go left; if None, inner loop exits
curr = stack.pop() ← backtrack: all of curr's left subtree is processed
result.append(curr.val) ← VISIT: record this node (left done, now root)
curr = curr.right ← pivot to right subtree; inner loop will push its left spine
return result ← every node visited exactly once in inorder
⚠️ Why curr or stack not just while stack?

After line 9 (curr = curr.right), if the right child is not null it hasn't been pushed yet. A loop guarded only by while stack would exit early. The curr check keeps us alive as long as there's an unexplored node, even if the stack happens to be empty at that instant.

Solution 2 — Recursive DFS (Cleaner, same complexity)

def inorderTraversal(root):
result = []
def dfs(node): ← inner function, captures result by closure
if not node: return ← base case: null node → do nothing
dfs(node.left) ← recurse left first
result.append(node.val) ← visit root after left is fully done
dfs(node.right) ← recurse right after visiting root
dfs(root)
return result
💡 Complexity Summary

Time — both approaches: O(n). Every node is pushed once, popped once, visited once.
Space: O(h) where h = tree height. Balanced: O(log n). Skewed (all left or all right): O(n).
Morris Traversal: O(1) space by temporarily threading the tree. Same O(n) time. Very advanced; rarely expected unless interviewer specifically asks about O(1) space.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: State the Traversal Order Immediately

Say: "Inorder = Left → Root → Right. For a BST this produces sorted output — worth noting even if this is a general tree problem." This flags your pattern knowledge right away.

Step 2: Give the Recursive Solution First

Write the recursive version — it's 5 lines and directly mirrors the definition. Say: "O(n) time, O(h) space. Simple and correct. If the interviewer wants iterative or O(1) space, I can do that too."

Step 3: Explain the Iterative Stack Pattern

Walk through verbally: "I simulate the call stack explicitly. Push each node as I descend left. When I hit null, pop to visit, then pivot right. Loop condition is curr-or-stack." Draw a short trace on whiteboard — this makes it concrete.

Step 4: Note Morris Traversal if O(1) Space is Mentioned

Mention briefly: "Morris traversal achieves O(1) space by threading right-child pointers temporarily. It's O(n) time but complex — I'd implement it only if explicitly required." Shows breadth without getting stuck.

🗣️ What to Say to the Interviewer

"Inorder traversal: Left → Root → Right. For a BST this gives ascending order.
Recursive solution: base case null, recurse left, append val, recurse right. O(n)/O(h).
For iterative: use an explicit stack. Track curr as the node we're exploring.
Inner loop: push curr, go left, repeat until null.
Then pop → visit (append val) → go right.
Outer loop condition: curr or stack — both must be exhausted to stop.
Same time and space: O(n) time, O(h) space."

7 Summary & Key Takeaways ▾
Inorder = Left → Root → Right; root is visited "in" between its children
For any BST, inorder traversal always yields values in ascending sorted order
Recursive solution (3 lines) is the most natural; propose it first in interviews
Iterative: push left spine, pop to visit, pivot right — controlled by curr or stack
Loop condition must be curr or stack, not just stack, to avoid early exit
Space = O(h): O(log n) balanced tree, O(n) worst-case skewed tree
