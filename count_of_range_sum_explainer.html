LeetCode #327

Count of Range Sum

Given an integer array nums and integers lower, upper, return the count of range sums sum(i,j) that lie in [lower, upper] inclusive.

Hard Sorting Merge Sort + Range Count Time: O(n log n) Space: O(n)
1 Understanding the Problem ▾

A range sum sum(i, j) is the sum of all elements from index i to j inclusive in nums. We need to count how many such sums fall within the closed interval [lower, upper]. For an array of length n, there are O(n²) possible (i,j) pairs to check.

💡 Key Insight: Range Sums are Prefix Sum Differences

Define pre[k] = nums[0] + nums[1] + … + nums[k-1] (with pre[0] = 0). Then sum(i, j) = pre[j+1] - pre[i]. So the problem becomes: count pairs (i, j) with i < j+1 such that lower ≤ pre[j] - pre[i] ≤ upper. This is a pair-counting problem on the prefix-sum array!

Concrete example: nums = [-2, 5, -1], lower = -2, upper = 2.

Prefix sums: pre = [0, -2, 3, 2]. We check all pairs (i < j):

✓ Valid Range Sums in [-2, 2]

pre[1]-pre[0] = -2-0 = -2 ✓
pre[3]-pre[0] = 2-0 = 2 ✓
pre[3]-pre[2] = 2-3 = -1 ✓
Answer: 3

✗ Out-of-Range Sums

pre[2]-pre[0] = 3-0 = 3 > 2 ✗
pre[2]-pre[1] = 3-(-2) = 5 > 2 ✗
pre[3]-pre[1] = 2-(-2) = 4 > 2 ✗
3 pairs rejected

Why brute force fails: For n = 105, checking all O(n²) = 1010 pairs times out. We need to count pairs faster using the structure of sorted prefix sums.

Edge cases: Empty array → 0. Single element: count 1 if nums[0] ∈ [lower, upper]. All negatives. lower = upper (exact match). Large values → use Python (arbitrary precision) or long in Java to avoid overflow.

2 The Algorithm: Modified Merge Sort on Prefix Sums ▾

We run a modified merge sort on the prefix sum array. Normal merge sort just sorts — our version also counts valid cross-pairs during each merge step. The magic: when merging two already-sorted halves, we can count all valid pairs in O(n) using two monotone pointers.

1 Build prefix sums: Construct pre[0..n] where pre[0] = 0 and pre[k] = pre[k-1] + nums[k-1]. Length n+1 array. Now sum(i, j) = pre[j+1] - pre[i].
2 Divide: Split the prefix-sum subarray at mid. Recurse on left half and right half independently. Each recursive call returns a sorted subarray AND accumulates counts for pairs within it.
3 Count cross-pairs during merge: Left half is sorted, right half is sorted. For each element val in the right half (iterating left to right), maintain two pointers lo_ptr and hi_ptr into the left half that track the valid window where val - upper ≤ left[i] ≤ val - lower. Count += hi_ptr - lo_ptr.
4 Standard merge: After counting, perform the ordinary merge to produce a sorted merged array. This sorting is essential — it enables efficient two-pointer counting in ancestor merge steps.
5 Aggregate: Counts from all merge steps accumulate. Every valid (i, j) pair is counted at exactly one merge level — when i first appears in a left half and j in a right half.
🔑 Why Two Pointers Never Backtrack

As we iterate val through the sorted right half (increasing), the valid window for left-half elements shifts right: [val - upper, val - lower] moves right with val. So lo_ptr and hi_ptr only ever advance — never retreat. Total pointer moves per merge level: O(n). Total: O(n log n).

Brute Force: O(n²)

For each j: scan all i < j
No structure to skip any
n=10⁵ → ~5×10⁹ ops
→ Time Limit Exceeded

Merge Sort: O(n log n)

Sorted left → monotone window
Two ptrs advance only forward
O(n) per level × O(log n) levels
→ Passes every test case!

3 Why It Works: The Invariant ▾
🔐 The Invariant

After each call merge_sort(arr) returns, two guarantees hold: (1) the subarray is sorted in ascending order, and (2) all valid pairs where BOTH endpoints are within this subarray have been counted. The "count" step at merge time handles exactly the cross-pairs: those with one endpoint in the left half, the other in the right.

Correctness — every pair counted exactly once: A pair (pre[i], pre[j]) with i < j gets counted at exactly the level where pre[i] is in the left half and pre[j] is in the right half. This happens at precisely one merge level, determined by the smallest common ancestor in the recursion tree. No pair is skipped or double-counted.

Why the window condition works: We want lower ≤ val - left[i] ≤ upper, rearranged as val - upper ≤ left[i] ≤ val - lower. Since the left half is sorted:
— lo_ptr: first index where left[lo_ptr] ≥ val - upper
— hi_ptr: first index where left[hi_ptr] > val - lower (one past last valid)
— Count += hi_ptr - lo_ptr

⚠️ Critical Order: Count BEFORE Merge

Counting must happen while the two halves are still separate. Once you merge them into one sorted array, you can no longer tell which elements came from the left half vs. right half. If you swap the order (merge first, count after), the algorithm breaks completely.

Why sorting is needed at all: Without sorting the left half, for each val in the right you'd need O(n) scan to count valid left elements — that's O(n) per element, O(n²) total per level, defeating the purpose. Sorting collapses the valid left elements into a contiguous window, enabling O(1) count per element using pointer positions.

4 The Code (Annotated) ▾

Solution 1 — Merge Sort on Prefix Sums (Optimal, O(n log n))

def countRangeSum(nums, lower, upper): ← entry point
n = len(nums)
pre = [0] * (n + 1) ← prefix sum array, one longer than nums
for i in range(n):
pre[i+1] = pre[i] + nums[i] ← build: pre[k] = sum of first k elements
count = [0] ← mutable container to accumulate count across calls
def merge_sort(arr): ← sorts arr IN-PLACE while counting cross-pairs
if len(arr) <= 1: ← base case: 0 or 1 element, trivially sorted
return
mid = len(arr) // 2 ← split at midpoint
left = arr[:mid] ← copy left half
right = arr[mid:] ← copy right half
merge_sort(left) ← recursively sort left; also accumulates sub-counts
merge_sort(right) ← recursively sort right
# COUNT cross-pairs (BEFORE merging)
lo_ptr = hi_ptr = 0 ← two ptrs into sorted left half
for val in right: ← val = pre[j] for each j in right partition
while lo_ptr < len(left) and \
left[lo_ptr] < val - upper: ← left[i] too small: val - left[i] > upper
lo_ptr += 1 ← skip: this left element gives sum > upper
while hi_ptr < len(left) and \
left[hi_ptr] <= val - lower: ← left[i] still in range: val - left[i] >= lower
hi_ptr += 1 ← include: this left element gives valid sum
count[0] += hi_ptr - lo_ptr ← all left elements in [lo_ptr, hi_ptr) are valid
# STANDARD MERGE (after counting)
i = j = k = 0 ← three-pointer standard merge
while i < len(left) and j < len(right):
if left[i] <= right[j]:
arr[k] = left[i]; i += 1
else:
arr[k] = right[j]; j += 1
k += 1
while i < len(left):
arr[k] = left[i]; i += 1; k += 1 ← flush remaining left
while j < len(right):
arr[k] = right[j]; j += 1; k += 1 ← flush remaining right
merge_sort(pre) ← kick off merge sort on the prefix sum array
return count[0] ← accumulated count across all merge steps
⚠️ Pointer Conditions Explained

For each val = pre[j] (right half), valid left elements pre[i] satisfy: val - upper ≤ pre[i] ≤ val - lower.
lo_ptr advances while left[lo_ptr] < val - upper (strictly less = not yet in window).
hi_ptr advances while left[hi_ptr] <= val - lower (≤ = still in window, advance past).
Count = hi_ptr - lo_ptr: elements at indices [lo_ptr, hi_ptr) are all valid.

Solution 2 — Brute Force (O(n²), for verification)

def countRangeSum(nums, lower, upper):
count = 0
for i in range(len(nums)): ← start of range
s = 0
for j in range(i, len(nums)): ← end of range
s += nums[j] ← accumulate sum(i..j)
if lower <= s <= upper: ← check in range
count += 1
return count ← TLE for n > 10⁴
💡 Complexity Summary

Merge Sort: Time O(n log n) — O(n) per level × O(log n) levels. Space O(n) for the prefix-sum array and temp copies.
Brute Force: Time O(n²), Space O(1). Correct but TLE for n > ~10⁴.
Alternative (SortedList/BIT): Also O(n log n) using from sortedcontainers import SortedList for online insertion + range counting. Same complexity, arguably simpler code.

4b 🔬 Algorithm Deep-Dive: Counting During Merge ▾

Origin: Merge sort was invented by John von Neumann (1945). The technique of counting cross-pair statistics during the merge step is a classic application of divide-and-conquer, popularized in courses like Kleinberg & Tardos "Algorithm Design" (Chapter 5 — Counting Inversions).

📐 Recurrence Relation
T(n) = 2·T(n/2) + O(n) ← O(n) counting + O(n) merge per level
→ Master Theorem Case 2: T(n) = O(n log n)

Core invariant: After merge_sort(arr[lo..hi]) returns, the subarray is sorted AND all valid pairs with BOTH endpoints in [lo..hi] are counted. The merge step counts exactly the "cross-pairs" missed by either sub-call.

This pattern applies to any "count pairs (i<j) satisfying f(arr[i], arr[j])" problem:

  • Count Inversions — arr[i] > arr[j] with i < j. Condition: is right[j] < left[i]?
  • Reverse Pairs (#493) — nums[i] > 2·nums[j] with i < j. Condition: is left[i] > 2·right[j]?
  • Count Smaller Numbers After Self (#315) — per-element counts, same merge trick.
  • This problem — lower ≤ pre[j] - pre[i] ≤ upper. Condition: is left[i] ∈ [val-upper, val-lower]?

When to recognize this pattern in an interview: Problem asks "count pairs (i, j) with i < j satisfying some comparison of arr[i] and arr[j]." If you can express the condition as "is arr[i] in some window defined by arr[j]?" and sorting one relative to the other doesn't invalidate the pair's meaning, you can count during merge.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Prefix Sum Reduction

"sum(i,j) = pre[j+1] - pre[i]. So I need pairs (i < j) where lower ≤ pre[j] - pre[i] ≤ upper. That's a pair-counting problem on the prefix sum array of length n+1."

Step 2: Identify the Bottleneck

"Brute force checks all O(n²) pairs — TLE. For each j, I want to efficiently count how many i < j have pre[i] in a certain window. This screams 'use structure of sorted elements'."

Step 3: "Count pairs during merge" pattern

"Classic divide-and-conquer pattern: merge sort, count cross-pairs at each merge. If I sort both halves, for each val in right I two-pointer the left half's window. Both pointers advance monotonically → O(n) per level → O(n log n) total."

Step 4: Derive the window bounds

"lower ≤ val - left[i] ≤ upper. Rearranging: val - upper ≤ left[i] ≤ val - lower. lo_ptr: find first left[i] ≥ val - upper. hi_ptr: find first left[i] > val - lower. Count = hi_ptr - lo_ptr."

🗣️ What to Say to the Interviewer

"Step 1: Build prefix sums. sum(i,j) = pre[j+1] - pre[i]. Problem becomes: count pairs in prefix array.
Step 2: Brute force O(n²) won't work. Need to count valid i for each j efficiently.
Step 3: Modified merge sort — sort prefix sums, count valid cross-pairs during merge.
Step 4: For each right-half element val, binary window in sorted left half. Two pointers advance monotonically.
Result: O(n log n) time, O(n) space. Every pair counted at exactly one merge level."
🔮 Likely Follow-up Questions
  • "Alternative to merge sort?" → SortedList + bisect, or Binary Indexed Tree. Both O(n log n).
  • "Count inversions?" → Same pattern, simpler condition: count right[j] < left[i].
  • "Reverse Pairs?" → LeetCode #493. Condition: left[i] > 2·right[j]. Nearly identical code.
  • "Handle integer overflow?" → In Java/C++, prefix sums may exceed int. Use long. Python is fine.
