LeetCode #264

Ugly Number II

Find the nth ugly number — a positive integer whose only prime factors are 2, 3, or 5.

Medium Math Three Pointers for 2·3·5 Time: O(n) Space: O(n)
1 Understanding the Problem ▾

An ugly number is a positive integer whose only prime factors are 2, 3, and 5. The sequence of ugly numbers begins: 1, 2, 3, 4, 5, 6, 8, 9, 10, 12, 15, 16, 18, 20, 24, 25, ... Given n, return the nth ugly number (1-indexed).

💡 Key Insight

Every ugly number beyond 1 is produced by multiplying a smaller ugly number by 2, 3, or 5. Rather than testing every positive integer (wasteful!), we can generate ugly numbers in sorted order by maintaining three streams — one for each factor.

✓ Ugly Numbers (first 12)

4 = 2×2
5 = 1×5
6 = 2×3
8 = 4×2 = 2³
9 = 3×3 = 3²
10 = 2×5
12 = 4×3
15 = 3×5
16 = 8×2 = 2⁴

✗ NOT Ugly (have other prime factors)

7 = 7 (prime 7)
11 = 11 (prime 11)
13 = 13 (prime 13)
14 = 2×7
21 = 3×7
22 = 2×11
26 = 2×13
35 = 5×7

7, 11, 13, ... skip over these!

The three multiplicative streams of ugly numbers look like this:

×2 stream
2, 4, 6, 8, 10,
12, 16, 18, 20,
24, ...
×3 stream
3, 6, 9, 12, 15,
18, 24, 27, 30,
36, ...
×5 stream
5, 10, 15, 20, 25,
30, 40, 45, 50,
60, ...

Our goal: merge these three streams into one sorted sequence — removing duplicates (6 appears in both ×2 and ×3 streams) — and return the nth element.

Edge Cases: n=1 → always 1. The sequence grows slowly; for large n, numbers can exceed 32-bit integers.

2 The Algorithm: Three-Pointer Generation ▾

We maintain an array of ugly numbers and three pointers — one for each factor. Each pointer points to the last ugly number it has already "used" to generate a candidate. At every step, we pick the smallest of three candidates, add it to the array, then advance any pointer(s) that produced that minimum.

1 Seed: ugly = [1]. The first ugly number is 1 by definition (no prime factors). Set p2 = p3 = p5 = 0 — all three pointers start at index 0, meaning the next candidates are ugly[0]×2=2, ugly[0]×3=3, ugly[0]×5=5.
2 Compute candidates: At each iteration compute c2 = ugly[p2]×2, c3 = ugly[p3]×3, c5 = ugly[p5]×5. These are the three candidates for the next entry in the sorted ugly-number sequence.
3 Pick minimum: next_ugly = min(c2, c3, c5). The globally smallest candidate is the next ugly number in sorted order. Append it to ugly.
4 Advance pointer(s): For each pointer whose candidate equals next_ugly, increment it. This is the deduplication step — when two pointers produce the same value (e.g., 6 from ×2 and ×3), advance both so neither inserts a duplicate.
5 Repeat steps 2–4 until ugly has n elements. Return ugly[n-1].
🔑 Analogy: Three Tape Merges

Imagine three tapes (streams), each infinite and sorted:
Tape A: 1×2, 2×2, 3×2, 4×2, 5×2, ... = 2, 4, 6, 8, 10, ...
Tape B: 1×3, 2×3, 3×3, ... = 3, 6, 9, 12, ...
Tape C: 1×5, 2×5, ... = 5, 10, 15, ...
You read heads from each tape and always write the smallest one to your output. The three pointers are those tape read-heads. This is a classic k-way sorted merge.

3 Why It Works: The Invariant ▾
🔐 The Invariant

After generating ugly[0..k], pointer p2 points to the smallest index i such that ugly[i]×2 has not yet been placed in the sequence. Identically for p3 (factor 3) and p5 (factor 5). Together, the three candidates represent the "frontier" of each multiplicative stream.

Completeness (no ugly number is skipped): Every ugly number U > 1 has a smallest prime factor in {2,3,5}. Suppose it's 2. Then U/2 is also ugly (it has no other prime factors). So U will appear as ugly[i]×2 exactly when p2 reaches index i where ugly[i] = U/2. By induction on the value of U, every ugly number appears. The same logic holds for factors 3 and 5.

No duplicates: Some values can be produced by multiple streams (e.g., 6 = 2×3 = 3×2, and 30 = 2×15 = 3×10 = 5×6). Advancing all matching pointers in the same step means neither stream will re-generate that value at the next iteration. The separate if (not elif) check handles this.

Sorted order: Since pointers only advance (never go back), ugly[pX] only increases, so ugly[pX]×factor only increases over time. We always pick the minimum of three non-decreasing sequences, so the output is non-decreasing.

⚠️ The Most Common Bug: Using elif Instead of if

If you write if c2 == next: p2+=1 elif c3 == next: p3+=1 elif c5 == next: p5+=1, then when c2 = c3 = 6 only p2 advances. The next iteration gives c3 = ugly[p3]*3 = 1*3 = 3... wait, that example isn't right. More precisely: once 6 is picked, next iteration p3 still points to ugly[0]=1, so c3 = 1*3 = 3 — no, p3 was at 1 (pointing to ugly[1]=2), so c3 = 2*3 = 6 again, and 6 gets inserted twice. Always use three separate if statements.

4 The Code (Annotated) ▾

Solution 1 — Three-Pointer Generation (Optimal · O(n) time, O(n) space)

def nthUglyNumber(n: int) -> int:
ugly = [1] ← Seed list; 1 is ugly by convention
p2 = p3 = p5 = 0 ← All pointers start at index 0 (value ugly[0]=1)
while len(ugly) < n: ← Generate until we have n ugly numbers
c2 = ugly[p2] * 2 ← Next candidate from factor-2 stream
c3 = ugly[p3] * 3 ← Next candidate from factor-3 stream
c5 = ugly[p5] * 5 ← Next candidate from factor-5 stream
nxt = min(c2, c3, c5) ← Globally smallest = next in sorted order
ugly.append(nxt) ← Add to the sequence
if nxt == c2: p2 += 1 ← Advance ×2 pointer if it contributed
if nxt == c3: p3 += 1 ← Advance ×3 pointer if it contributed (NOT elif!)
if nxt == c5: p5 += 1 ← Advance ×5 pointer if it contributed (NOT elif!)
return ugly[n - 1] ← 1-indexed: the nth ugly number is at index n-1
⚠️ Critical: Three Separate if-statements, NOT elif

When multiple candidates tie for the minimum (e.g., ugly number 6 can come from both ×2 and ×3 streams), all matching pointers must advance in the same step. Using elif only advances one, causing duplicates in the sequence. Always use three independent if checks.

Solution 2 — Min-Heap (Simpler to code, but O(n log n))

import heapq
def nthUglyNumber(n: int) -> int:
heap = [1] ← Min-heap starts with 1
seen = {1} ← Track visited to avoid duplicates
cur = 1
for _ in range(n): ← Extract n times; the last is the answer
cur = heapq.heappop(heap) ← Always get the smallest ugly number
for f in (2, 3, 5): ← Generate next from all three factors
if cur * f not in seen: ← Only add if not already in heap
heapq.heappush(heap, cur * f)
seen.add(cur * f) ← Mark as seen to prevent duplicate inserts
return cur
💡 Complexity Comparison

Three-Pointer: Time O(n), Space O(n). No heap overhead. Each ugly number is generated exactly once. This is the interview pick.

Min-Heap: Time O(n log n) due to heap operations, Space O(n) for heap + seen set. Easier to reason about for beginners; also generalizes easily to arbitrary prime factors (Super Ugly Number).

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Pattern

The problem asks for the nth number in an infinite structured sequence. Signals: "only prime factors are...", "find nth in sequence", no constraints on n being small. This is a sequence generation problem, not a search problem — don't test each integer.

Step 2: Brute-Force First

Naively: iterate integers 1, 2, 3, ... and check if each is ugly (divide by 2/3/5 repeatedly; if the result is 1, it's ugly). This is O(n × log M) where M is the nth ugly number. Works but slow for large n.

Step 3: Observe the Structure

Key question: "Where do ugly numbers come from?" Answer: every ugly number >1 is a product of a smaller ugly number and one of {2, 3, 5}. So we don't need to test — we can generate them by multiplication. The challenge: generate them in sorted order without duplicates.

Step 4: Three Sorted Streams

Think of multiplying every ugly number by 2, by 3, by 5 — each gives a sorted sequence. To merge three sorted sequences, use three pointers (one per sequence) and always advance the one with the smallest current head — just like merging k sorted arrays. Handle ties by advancing all matching pointers simultaneously.

🗣️ What to Say to the Interviewer

"My first thought is brute force — check every integer and test if it's ugly. That's O(k log k) per test where k is the candidate, too slow for large n. Better observation: every ugly number is just a smaller ugly number times 2, 3, or 5. So I can generate them in order. I'll keep three pointers — p2, p3, p5 — each pointing into my built ugly[] array. At each step: candidates are ugly[p2]*2, ugly[p3]*3, ugly[p5]*5. I pick the minimum, append it, then advance ALL pointers that tied for minimum. That handles deduplication — for example 6 = 2*3 = 3*2, so both p2 and p3 advance when 6 is chosen. This runs in O(n) time and space."
🎯 Follow-Up Questions They Might Ask
  • Super Ugly Number (LC #313): Given a list of primes, same approach but with k pointers instead of 3. Three-pointer generalizes cleanly.
  • Space optimization: Can we do O(1) space? No efficient closed-form exists; the sequence doesn't have a simple formula. O(n) is essentially required.
  • What's ugly[1690]? The 1690th ugly number is 231 = 2,147,483,648. The values grow quickly — for large n, they exceed 32-bit integers.

7 Summary & Key Takeaways ▾
Every ugly number is generated by multiplying a smaller ugly number by 2, 3, or 5 — so we can generate the sequence without testing each integer.
Three pointers act as read-heads on three infinite sorted streams; always picking the minimum keeps output sorted.
When two or more pointers tie (produce equal minimum), advance ALL of them — this is the deduplication step. Never use elif.
Time: O(n). Space: O(n). We compute each of the n ugly numbers exactly once.
The min-heap variant is easier to code and generalize but runs in O(n log n). Use three-pointer in interviews when asked for O(n).
🎯 When to Use This Pattern
  • Problem asks for nth number in a sequence defined by multiplicative rules
  • "Only prime factors are..." constraint
  • Need to merge multiple sorted streams in O(1) extra space per pointer
  • You need to avoid duplicates from overlapping multiplicative streams
