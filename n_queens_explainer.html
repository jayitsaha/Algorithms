LeetCode #51

N-Queens

Place n queens on an n×n chessboard so no two queens attack each other. Return all distinct board configurations.

Hard Backtracking Column / Diagonal Tracking Time: O(n!) Space: O(n²)
1 Understanding the Problem ▾

The N-Queens problem asks: on an n×n chessboard, place exactly n queens so that no queen attacks another. In chess, a queen attacks everything in its row, column, and both diagonals. So two queens conflict if they share the same row, column, or diagonal.

💡 Key Insight: One Queen Per Row

Since a queen attacks its entire row, you can place at most one queen per row. This means we must place exactly one in each of the n rows. The problem becomes: "for row 0, which column? for row 1, which column? ..." — a sequential decision problem with n choices per level, pruned by conflict detection.

For n=4, there are exactly 2 solutions:

Three types of conflict to track:

  • Same column: two queens in column c
  • Same main diagonal ↘: squares where row − col is equal
  • Same anti-diagonal ↗: squares where row + col is equal
🔑 The Diagonal Identity Trick

Every square (r, c) belongs to exactly one main diagonal (identified by r − c, which ranges from −(n−1) to n−1) and one anti-diagonal (identified by r + c, ranging from 0 to 2n−2). If two queens share this integer ID, they're on the same diagonal. Store these IDs in Python sets for O(1) conflict checks — no 2D grid needed.

Edge cases: n=1 → one solution. n=2 and n=3 → no solution exists. Output is a list of boards; each board is a list of n strings (each string is a row, with 'Q' for the queen and '.' elsewhere).

2 The Algorithm: Backtracking with Set-Based Conflict Detection ▾

We recurse row by row. At each row, try every column. Before placing, verify against three sets. If safe, place (add to sets), recurse to the next row, then remove (backtrack) to try the next column.

1 Base case: If row == n, all rows filled. Convert the placement[] array (column chosen per row) into board strings and record as a valid solution.
2 Try each column: For col in 0…n−1, check: col ∈ cols (column conflict), row−col ∈ diag (main diagonal conflict), row+col ∈ anti (anti-diagonal conflict). Any hit → skip this column.
3 Place: Add identifiers to the three sets. Record placement[row] = col. Recurse to row + 1.
4 Backtrack (CRITICAL): After recursion returns, remove the queen's identifiers from all three sets. This restores the state so we can try the next column without phantom conflicts.
5 Repeat: After exhausting all columns in a row, return to the caller. The full decision tree is explored; only complete conflict-free placements become results.

⚠️ Why Backtracking, Not BFS or DP?

We need all valid configurations. Backtracking is the natural fit: it exhaustively explores every branch of the decision tree, prunes dead branches early (conflict detection), and collects every valid leaf (complete board). BFS wastes memory storing all partial states simultaneously. DP doesn't apply because we aren't reusing overlapping sub-problems in a way that allows memoization — each row's valid columns depend on the specific placement history.

3 Why It Works: The Invariant ▾
🔐 The Invariant

Before placing a queen in row r, the three sets contain exactly the column IDs, main-diagonal IDs, and anti-diagonal IDs of all queens placed in rows 0…r−1. Therefore, checking these sets guarantees the new queen does not conflict with any previously placed queen.

Why three sets are necessary and sufficient: Two queens at (r1,c1) and (r2,c2) attack each other if and only if one of these holds:

  • c1 == c2 → same column → caught by cols
  • r1 − c1 == r2 − c2 → same main diagonal ↘ → caught by diag
  • r1 + c1 == r2 + c2 → same anti-diagonal ↗ → caught by anti

We never check rows because the algorithm places exactly one queen per row by construction — two queens in the same row is impossible.

Completeness: Every column is tried in every row, so every branch of the decision tree is visited. No valid configuration is skipped. Combined with the invariant, every board found is genuinely conflict-free.

⚠️ The Backtrack Step Is Critical

After recursion returns, we must remove the queen's identifiers before trying the next column. Forgetting this leaves "phantom queens" in the sets, causing future columns to be falsely blocked. This is the defining mistake in broken backtracking implementations — always verify your undo mirrors your do.

⚠️ Common Mistake: Keeping a 2D "attacked" Grid

Beginners sometimes maintain an n×n boolean grid and mark every attacked square when placing a queen. This costs O(n) per placement to update AND requires complex undo logic. The three-set approach gives O(1) check, O(1) add, O(1) remove — dramatically simpler and faster.

Solution 2 — Bitmask Backtracking (same O(n!), ~3× faster in practice)

def solveNQueens(n):
results, placement = [], [-1] * n
def bt(row, cols, diag, anti): ← integers as bitmasks; bit k = column k taken
if row == n:
results.append(['.' * placement[r] + 'Q' + '.' * (n-1-placement[r]) for r in range(n)])
return
avail = ((1 << n) - 1) & ~(cols | diag | anti) ← bits=1 → safe columns
while avail:
bit = avail & (-avail) ← isolate lowest set bit
col = bit.bit_length() - 1 ← column index
placement[row] = col
bt(row+1, cols|bit, (diag|bit)>>1, (anti|bit)<<1) ← shift diagonals per row
avail &= avail - 1 ← clear lowest bit (implicit backtrack)
bt(0, 0, 0, 0)
return results
💡 Complexity Summary

Set Backtracking: Time O(n!), Space O(n) recursion + O(n²) for stored boards
Bitmask Backtracking: Same asymptotic; bitwise ops are ~3× faster than set operations in practice
Why O(n!): Row 0 has n column choices, row 1 at most n−1, … row n−1 at most 1. Diagonal pruning reduces this further; O(n!) is the upper bound.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Backtracking Signal

"The problem asks for ALL valid configurations, not just one or an optimum. Each decision (column choice per row) can be validated instantly, and we can undo it easily. That's the hallmark of backtracking: explore, validate, recurse, undo."

Step 2: Collapse the Search Space

"Naively we'd try all n^n placements. But since queens attack rows, exactly one queen per row. So the search becomes: choose a column for row 0, then row 1, …, row n−1. That's at most n! states, and diagonal pruning cuts it further."

Step 3: Design O(1) Conflict Detection

"I need three checks: column, main diagonal, anti-diagonal. Key trick: the integers r−c and r+c uniquely identify diagonals. Store occupied IDs in Python sets. Adding and removing is O(1) — no matrix needed, no complex undo."

Step 4: Write and Verify the Backtrack

"Place (three adds + record column), recurse, then remove (three discards). The undo step is symmetric with the place step — a common interview check: 'Does your backtrack exactly mirror your placement?'"

🗣️ What to Say to the Interviewer

"This is a classic backtracking problem — we want all valid placements.
Key observation: one queen per row, so I'll recurse row by row choosing a column.
For conflict detection, I'll use three sets: occupied columns, r-c diagonal IDs, r+c anti-diagonal IDs.
O(1) check each time. Place: add identifiers. Recurse. Backtrack: discard identifiers.
Time O(n!) with significant diagonal pruning. Space O(n) for sets + O(n²) per solution stored."

7 Summary & Key Takeaways ▾
One queen per row — reduces search from n^n to at most n!
Three sets (cols, diag, anti) give O(1) conflict detection — no 2D grid needed
Diagonal identity: r−c for ↘ diagonals, r+c for ↗ diagonals — the core trick
Backtrack = place-recurse-remove; the remove step must mirror the place step exactly
Build board strings only at base case — defer expensive O(n²) work until success
ApproachTimeSpaceNotes
Brute force (all n^n placements) O(n^n) O(n²) No pruning, impractical
Set Backtracking ✓ O(n!) O(n²) Clean, interview-ready
Bitmask Backtracking O(n!) O(n) Fastest; complex to explain
