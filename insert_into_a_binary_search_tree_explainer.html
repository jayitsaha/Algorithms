LeetCode #701

Insert into a Binary Search Tree

Given a BST root and a value to insert, return the root of the BST after inserting the value. It is guaranteed the value does not exist in the original BST.

Medium Trees BST Property Time: O(h) Space: O(h)
1 Understanding the Problem ▾

A Binary Search Tree (BST) enforces one strict invariant: for every node N, all values in N's left subtree are smaller than N, and all values in N's right subtree are larger. We need to insert a new value while keeping this property intact for every node in the tree.

💡 Key Insight

The BST property gives you a GPS for every comparison: if the value you're inserting is less than the current node, go left — that's the only region where it can legally live. If greater, go right. When you eventually fall off the edge of the tree (reach a null pointer), you've found the one and only correct position. There is no ambiguity at any step.

✓ Inserting 5 into [4,2,7,1,3]

At 4: 5 > 4 → go right
At 7: 5 < 7 → go left
Left child of 7 = null
Place 5 as left child of 7 ✓

✗ Why we can't skip the descent

If we attach 5 anywhere else,
e.g. right child of 4 directly —
then 5 < 7 violates BST order
in the right subtree of 4. ✗

Edge Cases to consider: empty tree (return a single-node tree), value less than all existing nodes (ends up at far-left leaf), value greater than all existing nodes (far-right leaf), inserting into a skewed (linked-list-like) BST where depth = n.

2 The Algorithm: BST Guided Descent ▾

Think of traversing the BST like navigating a highway using exit signs: at each node, the sign says "smaller values → exit left, larger values → stay right." You follow the signs until the road ends — that's your destination. The recursive solution mirrors this naturally: each recursive call "follows the sign" and the base case is "road ends — place node here."

1 Base case — null reached: If root is None, we've found the exact insertion point. Return a brand new TreeNode(val). This return value propagates back up the call stack and is assigned to the appropriate child pointer of the parent.
2 Navigate left: If val < root.val, the new value must live in the left subtree (any position in the right subtree would violate BST order). Recursively call insertIntoBST(root.left, val) and assign the result back to root.left.
3 Navigate right: If val > root.val, mirror of step 2. Recurse into root.right and assign result back to root.right. (Equal values are impossible — the problem guarantees val is not in the tree.)
4 Return root unchanged: After the recursive call completes, return root. The root's value didn't change — only one child pointer (either left or right) may have been updated by the recursive assignment. Returning root preserves the full subtree structure above this call.
🔑 Why new values always become leaves

Internal nodes already "own" their position — they were inserted first and their value determined where they sit. A new value, guaranteed to differ from all existing values, gets compared at each internal node and sifted downward until it reaches an empty slot — which is always a leaf position. This is a fundamental property of BST insertion: simple insertion never displaces existing nodes.

3 Why It Works: The BST Invariant ▾
🔐 The Invariant

At every node during traversal: the new value can only legally reside in the subtree we recurse into. When we go left because val < node.val, placing val anywhere in the right subtree would break BST order (val would be smaller than node.val but in the right subtree — invalid). The left subtree is the only valid region. This remains true at every level, all the way down to the insertion point.

Proof by induction on tree height: Base case (empty subtree): returning TreeNode(val) trivially satisfies BST property. Inductive step: assume BST property holds for all subtrees of height < h. If we insert into root.left (because val < root.val), the returned subtree still satisfies BST by hypothesis. Since val < root.val, val is smaller than root — placing it anywhere in root's left subtree preserves root's own BST constraint. Symmetric argument for right.

Why we don't need to restructure: Unlike AVL or Red-Black trees, a plain BST insert never rotates or rebalances. It only creates one new node at a leaf. All ancestor nodes remain unchanged — we just update one child pointer along the path from root to insertion point.

⚠️ The Return-Value Pattern is Critical

In the recursive solution, root.left = insertIntoBST(root.left, val) looks redundant — after all, if root.left already existed we just recurse into it, and if it was None we return a new node. The key: when root.left was None, the return value is the new TreeNode. Without the assignment, the parent's .left pointer would remain None — the new node would be created but immediately garbage-collected. The assignment wires the new node into the tree.

4 The Code (Annotated) ▾

Solution 1 — Recursive (Interview Pick, O(h) time, O(h) space)

def insertIntoBST(root, val): ← root is the current subtree root (None = empty slot)
if root is None: ← Base case: we've reached an empty child slot
return TreeNode(val) ← Create the new leaf and return it to the parent caller
if val < root.val: ← BST rule: val must live in the left subtree
root.left = insertIntoBST(root.left, val) ← Recurse left; stitch returned subtree back to root.left
else: ← val > root.val (equal is impossible, guaranteed)
root.right = insertIntoBST(root.right, val)← Recurse right; stitch returned subtree back to root.right
return root ← Return this node (unchanged) so parent links hold
⚠️ Don't forget to reassign the child pointer

Writing insertIntoBST(root.left, val) without root.left = ... discards the newly created node. The recursive call creates TreeNode(val) but the parent's .left still points to None. Always write root.left = insertIntoBST(root.left, val) — it's a no-op when root.left already exists, and critical when it was None.

Solution 2 — Iterative (O(h) time, O(1) extra space)

def insertIntoBST(root, val):
if not root: return TreeNode(val) ← Empty tree → new node IS the root
curr = root ← Walk down the tree without call stack overhead
while True:
if val < curr.val: ← Should go left
if curr.left is None: ← Found the insertion slot
curr.left = TreeNode(val); break ← Attach directly, no return-value juggling needed
curr = curr.left ← Keep descending
else: ← Should go right (val > curr.val)
if curr.right is None: ← Found the insertion slot
curr.right = TreeNode(val); break ← Attach directly
curr = curr.right ← Keep descending
return root ← Original root is always unchanged
💡 Complexity Summary

Recursive: Time O(h), Space O(h) call stack (h = height of the BST)
Iterative: Time O(h), Space O(1) — no call stack, just a pointer walking down
Balanced BST: h = O(log n) → O(log n) per insertion
Skewed BST: h = O(n) → O(n) worst case (degenerate into linked list)

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the BST Pattern

Signal: "BST" + "insert/find/delete." The BST property eliminates half the tree at every step — this means O(h) not O(n). Think: "I don't need to touch the entire tree. The structure guides me." This is the BST Guided Descent pattern.

Step 2: Define the Recursive Structure

"My function takes a subtree root and inserts val into it, returning the (possibly updated) subtree root." State the base case explicitly: "If root is None, I've found the slot — return a new node." Then state the recursion: "Otherwise, go left or right based on comparison."

Step 3: Highlight the Subtle Detail

Interviewers love when you catch the return-value assignment: "I must write root.left = insertIntoBST(root.left, val) — not just call the function. Without the assignment, the new leaf is created but the parent's pointer stays null."

Step 4: Offer the Iterative Solution

Proactively say: "The recursive solution is O(h) space for the call stack. If stack space is a concern, the iterative version walks down with a curr pointer and directly mutates the child — O(1) extra space." This shows depth of understanding.

🗣️ What to Say to the Interviewer

"Since this is a BST, the property tells me exactly where to look at every node.
I'll use recursive descent: base case is when root is None — that's the insertion slot.
If val < root.val I recurse left and assign the result back to root.left.
If val > root.val I recurse right and assign the result back to root.right.
Then I return root (unchanged). New values always become leaves.
Time O(h), space O(h). I can do O(1) space with an iterative approach."
🔮 Likely Follow-up Questions
  • "What if the BST is skewed?" → O(n) worst case; mention AVL/Red-Black for guaranteed O(log n)
  • "Write the iterative version." → Walk with curr pointer, check left/right null before descending
  • "What if duplicates are allowed?" → Clarify spec: could insert left, insert right, skip, or store list at node
  • "How do you delete from a BST?" → Three cases: leaf (remove), one child (bypass), two children (replace with in-order successor/predecessor)

7 Summary & Key Takeaways ▾
BST property gives a unique insertion path — no ambiguity, no backtracking needed
New values always become leaf nodes — simple insertion never restructures the tree
Recursive: return the updated subtree upward so all parent links stay valid
Iterative saves O(h) call-stack space by directly mutating child pointer at the leaf
Performance depends on height: O(log n) balanced, O(n) skewed — use self-balancing trees for guarantees
