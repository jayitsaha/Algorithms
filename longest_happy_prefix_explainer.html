LeetCode #1392

Longest Happy Prefix

Find the longest string that is both a proper prefix and a proper suffix of s — the KMP failure function in disguise.

Hard String Processing KMP Failure Function Time: O(n) Space: O(n)
1 Understanding the Problem ▾

A happy prefix is a string that is simultaneously a proper prefix and a proper suffix of s. "Proper" means it cannot be the entire string s itself — it must be strictly shorter.

We want the longest such string (or "" if none exists).

💡 Key Insight

This is exactly what the KMP failure function (also called the "partial match table" or "prefix function") computes at its last position. The failure function at index i gives the length of the longest proper prefix of s[0..i] that is also a suffix of s[0..i]. We just need that value for index n-1 (the whole string).

✓ Example: s = "level"

Prefix "l" == Suffix "l" ✓
Prefix "le" vs Suffix "el" ✗
Prefix "lev" vs Suffix "vel" ✗
Answer: "l"

✓ Example: s = "aaaa"

"a" == suffix "a" ✓
"aa" == suffix "aa" ✓
"aaa" == suffix "aaa" ✓
Answer: "aaa"

Edge cases: A single character (no proper prefix/suffix → return ""). All same characters like "aaa" → answer is "aa" (length n−1). No match possible like "abcd" → return "".

⚠️ "Proper" is Critical

The prefix/suffix must be shorter than the whole string. For "aaa", the answer is "aa" (length 2), NOT "aaa" (length 3 — that's the whole string).

2 The Algorithm: KMP Failure Function ▾

The KMP failure function (prefix function) fail[i] stores the length of the longest proper prefix of s[0..i] that is also a suffix. We build this table in O(n) and read off fail[n-1].

1 Initialize: Create array fail of length n filled with zeros. Set k = 0 (length of current matching prefix). fail[0] = 0 always (single char has no proper prefix).
2 Iterate i from 1 to n−1: Try to extend the current match: if s[i] == s[k], increment k and set fail[i] = k.
3 On mismatch (s[i] != s[k]): Don't reset to zero! Fall back via k = fail[k-1] to try a shorter prefix. Keep falling back until k == 0 or a match is found.
4 If k == 0 and still mismatch: Set fail[i] = 0 — no proper prefix matches here.
5 Answer: Return s[:fail[n-1]] — the first fail[n-1] characters of s.

The fallback via fail[k-1] is the non-obvious key step. When we have a mismatch, we don't abandon all progress. We know that s[0..k-1] matches s[i-k..i-1]. The failure function tells us the longest proper prefix of s[0..k-1] that's also a suffix — so we can try extending that shorter match instead of restarting from zero.

🔬 KMP Failure Function — Algorithm Deep-Dive

Origin: Knuth, Morris, and Pratt (1977). The failure function was designed to avoid redundant character comparisons in string pattern matching. It preprocesses the pattern in O(m) so that searching a text of length n takes O(n+m) total — no quadratic worst case.

Core Invariant: At every index i, fail[i] = length of the longest proper prefix of s[0..i] that equals a suffix of s[0..i]. This is always in the range [0, i].

Why O(n) time despite the nested loop? The pointer k can increase by at most 1 per outer iteration (when there's a match), and it can only decrease via fallbacks. Since k can never go below 0, the total number of decreases across all iterations is bounded by the total number of increases — which is at most n. So the total work is O(n) amortized.

Why it works: The fallback k = fail[k-1] says: "I know the first k characters match. The longest border (prefix=suffix) of those k characters has length fail[k-1]. So I can restart matching from there." This exploits the self-similarity structure of the string.

When to recognize it: Any problem asking about the relationship between prefixes and suffixes of a single string. Signals: "longest prefix that is also a suffix", "does string have a period", "pattern matching without backtracking".

3 Why It Works: The Invariant ▾
🔐 The Invariant

At the start of each iteration (processing index i), k holds the length of the longest proper prefix of s[0..i-1] that is also a suffix. Equivalently, s[0..k-1] == s[i-k..i-1].

Base case: At i = 1, k = 0 (no prefix of a single character). The invariant holds trivially.

Inductive step (match): If s[i] == s[k], then the border of length k for s[0..i-1] can be extended by one character. So fail[i] = k + 1. We increment k to maintain the invariant for the next iteration.

Inductive step (mismatch): We know s[0..k-1] == s[i-k..i-1]. We want the next shorter border of s[0..k-1], which has length fail[k-1]. This is valid because s[0..fail[k-1]-1] is both a prefix and suffix of s[0..k-1] — meaning it's also a suffix of s[0..i-1]. We try extending this shorter match next.

💡 Why the Fallback Chain Terminates

Each fallback strictly decreases k (since fail[k-1] < k always — a border is strictly shorter than the string). So the chain k → fail[k-1] → fail[fail[k-1]-1] → ... must reach 0 in finite steps. At k == 0 with a mismatch, fail[i] = 0 (no border) and we move on.

Correctness of the answer: fail[n-1] gives the length of the longest proper prefix of the entire string s that equals a suffix. This is exactly the longest happy prefix. We return s[:fail[n-1]].

4 The Code (Annotated) ▾
def longestPrefix(s: str) -> str: — function signature
n = len(s) — cache length to avoid repeated calls
fail = [0] * n — failure function array; fail[i] = length of longest border of s[0..i]
k = 0 — k = length of current candidate border being tracked
for i in range(1, n): — start from 1 (fail[0] is always 0)
while k > 0 and s[i] != s[k]: — mismatch: fall back to next shorter border
k = fail[k - 1] — key KMP step: skip to the longest border of the current border
if s[i] == s[k]: — match! we can extend the border by one character
k += 1 — border grows: s[0..k-1] == s[i-k+1..i]
fail[i] = k — record: longest border of s[0..i] has length k
return s[:fail[n - 1]] — the border of the whole string = the longest happy prefix
⚠️ Common Mistake: Resetting k to 0 on Mismatch

Many beginners write else: k = 0 instead of the while k > 0: k = fail[k-1] fallback. This is wrong! It turns an O(n) algorithm into O(n²) and produces incorrect results for strings like "aabaabaab". The whole power of KMP is in the fallback chain.

🔐 Brute Force Comparison

A naive O(n²) approach would try every prefix length from n-1 down to 1 and check if s[:L] == s[n-L:]. Each comparison is O(L), giving O(n²) total. For n = 100,000 that's 10 billion operations — KMP does it in 100,000.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Pattern

The problem says "prefix that equals a suffix." This is the defining property of a border in string algorithms. Whenever you see this, think KMP failure function. The signals: "longest prefix = suffix", "pattern has period", "overlap of prefix and suffix".

Step 2: Establish the Brute Force Baseline

State: "A naive approach tries all lengths from n−1 down to 1, comparing s[:L] with s[n-L:]. This is O(n²). For n up to 10⁵, we need O(n)."

Step 3: Connect to KMP

State: "I recognize this as the KMP prefix function / failure function. It computes, for each index i, the longest proper prefix of s[0..i] that equals a suffix — which is precisely what we need at index n−1."

Step 4: Explain the Fallback

The interviewer will likely probe: "Why k = fail[k-1] instead of k = 0?" Answer: "When we have a mismatch at length k, we don't abandon everything. We fall back to the longest border of the first k characters — which is already a suffix of what we matched — and try extending that shorter prefix instead."

Step 5: Prove O(n) Amortized

State: "k can increase by at most 1 per outer iteration (n total increases). Each fallback strictly decreases k. So total fallbacks ≤ total increases ≤ n. The while loop across all iterations runs O(n) total — not O(n) per outer iteration."

🗣️ What to Say to the Interviewer

"The key observation is that 'longest prefix that equals a suffix' is exactly the border of a string, and the KMP failure function computes this for every prefix in O(n) time. I'll build the failure function: maintain k = current border length. For each character, if it extends the border, increment k. On mismatch, fall back via fail[k-1] — this is the critical non-obvious step that gives us O(n) amortized complexity. The answer is s[:fail[n-1]]. Time: O(n). Space: O(n) for the fail array. Follow-up variants: 'Does the string have a period?' → check if n % (n - fail[n-1]) == 0. 'String matching in text?' → KMP search using this same fail array."

7 Summary & Key Takeaways ▾
The longest happy prefix IS the KMP failure function value at position n−1
The fallback k = fail[k-1] is the crucial non-obvious step — never reset to 0
The algorithm is O(n) amortized because k increases by at most 1 per step and decreases are bounded by increases
Borders of a string form a "chain": each is a border of the previous, accessible via the fail array
Same fail array powers full KMP string search — this problem is essentially the preprocessing step
🎯 When to Recognize KMP Failure Function
  • Problem involves "prefix that equals a suffix" of a single string
  • String pattern matching without backtracking
  • "Does the string consist of repeating units?"
  • Finding overlaps between string and its reverse (palindrome applications)
  • O(n) requirement for what would naively be O(n²)
