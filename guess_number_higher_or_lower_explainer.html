LeetCode #374

Guess Number Higher or Lower

A number is picked from 1 to n. Use the guess(num) API — which returns -1, 1, or 0 — to find it in O(log n) calls.

Easy Binary Search Classic Binary Search Time: O(log n) Space: O(1)
1 Understanding the Problem ▾

Imagine a classic "higher or lower" guessing game. Someone secretly picks a number between 1 and n. You make a guess; they reply "too high", "too low", or "correct!" The goal is to find the number using as few guesses as possible.

In code, the reply is encoded by a pre-defined guess(num) API. The API contract is:

guess(num) returns -1

Your guess is too high.
The pick is below num.
→ Narrow search to [lo, num-1]

guess(num) returns 1

Your guess is too low.
The pick is above num.
→ Narrow search to [num+1, hi]

guess(num) returns 0 → Correct! Return num.
💡 Key Insight

Each call to guess() eliminates roughly half the remaining candidates. If you always probe the midpoint of your current range, you halve the problem size each call. Starting with n numbers, after k calls you have at most n/2k left. You need at most ⌈log2(n)⌉ calls — that is O(log n).

Worked example — n=16, hidden pick=11:

Guess 8
→ 1 (too low) · new range: [9, 16]
Guess 12
→ -1 (too high) · new range: [9, 11]
Guess 10
→ 1 (too low) · new range: [11, 11]
Guess 11
→ 0 (found!) · done in 4 guesses

Edge cases: n=1 (only number is 1, single guess suffices), pick=1 (lowest boundary), pick=n (highest boundary), n up to 231−1 (integer overflow risk with naive mid formula).

2 The Algorithm: Classic Binary Search ▾

Binary search is the systematic application of "halve the search space every step." We maintain two boundary pointers, lo and hi, representing the live candidate range [lo, hi]. Every iteration we probe the midpoint and shrink the range based on the oracle's response.

1 Initialize boundaries. lo = 1 (smallest possible pick), hi = n (largest possible pick). The secret number is guaranteed to be in [1, n].
2 Compute the midpoint safely. mid = lo + (hi - lo) // 2. This always equals (lo + hi) // 2 mathematically but avoids 32-bit overflow when lo + hi exceeds INT_MAX (critical in Java/C++, good habit in Python).
3 Call guess(mid) and branch. If it returns 0 → return mid. If −1 (too high) → set hi = mid − 1, excluding mid and everything above. If 1 (too low) → set lo = mid + 1, excluding mid and everything below.
4 Repeat while lo ≤ hi. The loop terminates because every iteration strictly shrinks the range (by at least 1). Since the problem guarantees a valid pick, we always find it before the range empties.
🔑 Binary Search on Implicit Sorted Space

Most binary search problems show you an explicit sorted array. Here, the "array" is the implicit sequence 1, 2, 3, …, n — you never see it, but guess() acts as the comparison function. Recognizing that you can binary-search any monotonically ordered space, even an implicit one, is the key insight this problem teaches.

⚠️ Common Mistake: Confusing -1 and 1

guess() returns -1 if your guess is higher than the pick, and 1 if it is lower. Many beginners instinctively flip these. Remember it from the API-author's perspective: "your guess minus the pick" — if you overshot, the difference is positive, so the signal is negative (they're saying "come down"). Mnemonic: -1 = too high, lower your guess; 1 = too low, raise your guess.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every iteration, the secret pick is always within [lo, hi]. This invariant never breaks. Initially, lo=1 and hi=n, so it holds trivially. When we set hi = mid−1, we've just been told mid is too high — the pick is below mid, so [lo, mid−1] still contains it. When we set lo = mid+1, the pick is above mid, so [mid+1, hi] still contains it. Every update preserves the invariant.

Termination proof: The range size is hi − lo + 1. Each iteration reduces it by at least half (we eliminate the mid element plus roughly half the remaining). After k iterations the size is at most ⌈n / 2k⌉. When this reaches 1 (i.e., lo = hi), that single value must be the pick — and guess(lo) will confirm it with 0.

Why O(log n) and not O(n): The range starts at n and halves each step. It takes log2(n) halvings to reduce it to 1. For n = 109, that is about 30 iterations — compared to up to a billion for a linear scan. This is the exponential leverage binary search provides.

⚠️ Integer Overflow — The Hidden Bug

In Java or C++: if lo = 1_500_000_000 and hi = 2_000_000_000, then lo + hi = 3_500_000_000 which overflows a signed 32-bit int (max 2,147,483,647). The result wraps to a negative number — catastrophic. Always use lo + (hi - lo) / 2. Python is safe (arbitrary-precision ints), but write it correctly anyway as interview habit.

4 The Code (Annotated) ▾

Solution 1 — Classic Binary Search (O(log n) · Interview Pick)

def guessNumber(n: int) -> int: ← n is upper bound; pick ∈ [1, n]
lo, hi = 1, n ← Initialize search space to full [1, n]
while lo <= hi: ← Loop until range is empty (guaranteed to find)
mid = lo + (hi - lo) // 2 ← Midpoint (overflow-safe; same as (lo+hi)//2)
result = guess(mid) ← Ask the oracle: is mid the pick?
if result == 0: ← 0 = exact match → found it
return mid
elif result == -1: ← -1 = mid too high → search lower half
hi = mid - 1 ← Exclude mid and everything above
else: ← result == 1; mid too low → search upper half
lo = mid + 1 ← Exclude mid and everything below

Solution 2 — Brute Force Linear Scan (O(n) · Understand Only)

def guessNumber_brute(n: int) -> int:
for num in range(1, n + 1): ← Try 1, 2, 3, … in order
if guess(num) == 0: ← Found it
return num ← Worst case: n calls (pick = n)
💡 Complexity Comparison

Brute Force: O(n) time — up to n API calls. For n = 109 that is 1 billion calls — not acceptable.
Binary Search: O(log n) time, O(1) space — at most 30 calls for n = 109. A factor of 33 million better.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Pattern

The numbers 1 to n are implicitly sorted. The guess() API tells you which direction to search. This is the textbook binary search setup: ordered space + directional oracle = binary search. Say this out loud immediately.

Step 2: Start with Brute Force (Always)

"I could scan linearly from 1 to n — O(n) calls. For n up to a billion, that's a billion API calls. That's too slow. I need O(log n), which suggests binary search." Then pivot to the optimal.

Step 3: Verify the API Contract

Say explicitly: "Let me make sure I have the API mapping right: -1 means my guess is too high so I set hi = mid−1; 1 means too low so I set lo = mid+1; 0 means found." This prevents the -1/1 flip error and shows careful reading.

Step 4: Mention Overflow (Bonus Points)

"I'll compute mid as lo + (hi − lo) // 2 to avoid integer overflow for large n — important in Java or C++." This small detail shows senior-level awareness.

Step 5: State Complexity and Wrap Up

"Time complexity: O(log n) — we halve the search space each call. Space: O(1) — just two pointers and a mid variable. Optimal for this problem."

🗣️ What to Say to the Interviewer

"The numbers 1 to n are implicitly sorted. guess() tells me direction, so I'll binary search.
Initialize lo=1, hi=n. Loop while lo <= hi:
mid = lo + (hi - lo) // 2 [overflow-safe midpoint]
Call guess(mid):
0 → return mid
-1 → too high, hi = mid - 1
1 → too low, lo = mid + 1
Time O(log n), Space O(1). For n=10^9: at most ~30 API calls."
🔮 Likely Follow-up Questions
  • "What is the minimum number of guesses guaranteed for n?" → ⌈log2(n+1)⌉. For n=10: 4 guesses worst case.
  • "How is First Bad Version the same problem?" → isBadVersion() is the oracle; first bad = lower boundary; same binary search template (#278).
  • "Can you do it recursively?" → Yes: base case lo==hi, return lo. Recursive case: compute mid, recurse on half. O(log n) stack frames.
  • "What if the answer space is infinite?" → Exponential search: double hi until too high, then binary search in that range.

7 Summary & Key Takeaways ▾
Binary search halves the search space each step — O(log n) vs O(n) for linear scan
guess(-1) means your guess is too HIGH; guess(1) means too LOW — map carefully
Invariant: the pick is always inside [lo, hi] at every iteration
Use lo + (hi − lo) // 2 to compute mid — prevents 32-bit integer overflow
Recognize this as "binary search on implicit sorted space with oracle" — generalizes widely

🎯 When to Recognize Classic Binary Search
  • "Find a hidden/target value in an ordered space using a comparison oracle" → Binary search on value
  • "Minimize the maximum X" or "maximize the minimum X" → Binary search on the answer
  • "First true / last false in a monotone predicate" → Binary search the boundary
  • Sorted array + target → Standard binary search on index
