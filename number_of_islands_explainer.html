LeetCode #200

Number of Islands

Given an m×n grid of '1's (land) and '0's (water), count the number of islands — groups of adjacent (horizontal/vertical) land cells.

Medium Graph DFS Mark Visited Time: O(m×n) Space: O(m×n)
1 Understanding the Problem ▾

You have a 2D grid where every cell holds either '1' (land) or '0' (water). An island is any group of land cells that are connected horizontally or vertically — diagonal connections don't count. Count how many distinct islands exist.

💡 Key Insight

Model the grid as a graph: every land cell is a node, and two land cells share an edge if they are horizontally or vertically adjacent. Counting islands is exactly counting connected components. The classic trick: once you discover an island, immediately "sink" its entire body by setting every cell to '0' so you never count it again.

Edge cases to watch for: entirely water grid (answer 0), entirely land grid (answer 1), single cell, long thin snake-shaped islands, and very large grids that test recursion depth.

2 The Algorithm: DFS "Sink the Island" ▾

We perform a two-level traversal. The outer level is a linear scan of every grid cell. The inner level is a depth-first search that "floods" an entire island the moment we discover it.

1 Scan left-to-right, top-to-bottom. Iterate r from 0 to rows−1 and c from 0 to cols−1.
2 Detect a new island. When grid[r][c] == '1', this cell has not been visited yet (all previously visited land has been sunk). Increment count.
3 Launch DFS to sink the island. Call dfs(r, c), which recursively visits all 4-directional neighbours. At each cell, if it is still '1', set it to '0' (sink) and recurse.
4 DFS base cases — stop when: (a) the position is out of bounds, or (b) the cell is already '0' (water or already sunk).
5 Continue outer scan. All sunk cells look like water now, so the scanner skips them automatically. Only fresh, unvisited land increments the counter.

🔑 Why "Sink" Instead of a Visited Set?

Setting visited cells to '0' is O(1) per cell and requires zero additional memory — we reuse the existing grid as our visited marker. The tradeoff is that we modify the input. Always mention this to an interviewer: "I'm mutating the grid to avoid a separate visited array. If the input must be preserved, I'd use a boolean visited[m][n] array instead — same complexity."

3 Why It Works: The Invariant ▾
🔐 The Invariant

At any point during the outer scan, every land cell the scanner has already passed is '0' — either it was originally water, or it was sunk during a DFS triggered by an earlier discovery. Therefore the only '1' cells that remain are ones that belong to islands we have not yet discovered. The counter increments exactly once per connected component.

Why DFS reaches every cell in the island: By definition, every cell in an island can be reached from any other cell in that island by following horizontal and vertical adjacencies. DFS exhaustively follows all valid neighbours, so it will reach every cell in the connected component — none are missed.

Why we never double-count: The moment dfs(r, c) sets grid[r][c] = '0', the cell becomes indistinguishable from water. The outer scanner will see '0' and skip it. There is no code path that increments count a second time for an already-sunk cell.

⚠️ Python Recursion Depth on Large Grids

A fully-land grid of 300×300 could produce a DFS call stack 90,000 frames deep — Python's default limit is 1000, causing a RecursionError. In production (or if the interviewer asks about robustness), switch to an iterative BFS with a deque. Mention this proactively; interviewers respect candidates who anticipate edge cases.

Edge case correctness: All-water grid → no '1' found → count stays 0. All-land grid → scanner hits (0,0), count becomes 1, DFS sinks every cell, scanner finds nothing else → count = 1. Single-cell '1' → count = 1, DFS sinks it immediately.

⚠️ Critical: Check Bounds Before Accessing grid[r][c]

The bounds check must come first. If we checked grid[r][c] before checking bounds, an out-of-range index would throw an IndexError. Python does not silently ignore negative indices the way you might expect in this context — grid[-1][0] would access the last row, not trigger an error, giving wrong results.

Solution 2 — Iterative BFS (no recursion depth risk)

from collections import deque
def numIslands(grid):
if not grid: return 0
rows, cols, count = len(grid), len(grid[0]), 0
for r in range(rows):
for c in range(cols):
if grid[r][c] == '1':
count += 1
grid[r][c] = '0' ← sink before enqueue (avoid dup entries)
q = deque([(r, c)])
while q:
row, col = q.popleft()
for dr, dc in [(1,0),(-1,0),(0,1),(0,-1)]:
nr, nc = row+dr, col+dc
if 0<=nr<rows and 0<=nc<cols \
and grid[nr][nc]=='1':
grid[nr][nc]='0' ← sink immediately when enqueuing
q.append((nr, nc))
return count
💡 Complexity Summary

Both DFS and BFS: Time O(m·n) — each cell is visited at most once (set to '0' after first visit).
Space O(m·n) — DFS: implicit call stack up to m·n deep. BFS: deque holds up to O(min(m,n)) entries per level in practice, worst case O(m·n).
If input must be preserved: Use a separate visited = [[False]*cols for _ in range(rows)] — same complexity.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize "Connected Components in a Grid"

"Each land cell is a graph node. Adjacency is horizontal/vertical only. Counting islands = counting connected components. Classic DFS/BFS pattern."

Step 2: Decide How to Track Visited Cells

"Option A: Mutate the grid — set visited land to '0'. O(1) space, but destroys input. Option B: Boolean visited array — same asymptotic space, preserves input. I'll use mutation and mention the tradeoff."

Step 3: Write DFS with Clear Base Cases

"Base case 1: out-of-bounds → return. Base case 2: cell is '0' → return. Recursive case: set to '0', then recurse in 4 directions. Bounds check must come before grid access."

Step 4: Proactively Address Recursion Depth

"For very large all-land grids, recursion depth could hit Python's 1000-frame limit. I'd use iterative BFS with a deque for production use, or call sys.setrecursionlimit(rows*cols+10)."

🗣️ What to Say to the Interviewer

"I'll model this as a graph problem — land cells are nodes, adjacent land cells share edges.
Counting islands = counting connected components. I'll use DFS.
Scan every cell. When I find a '1' that hasn't been visited, I increment count and launch DFS.
DFS sinks the entire island by setting cells to '0' — this marks them visited in O(1) space.
Base cases: out-of-bounds or already '0'. Check bounds before accessing grid values.
Time: O(m*n). Space: O(m*n) for the call stack. I'd use BFS for large inputs to avoid stack overflow."

7 Summary & Key Takeaways ▾
Islands = connected components in a grid graph; DFS/BFS finds and marks entire components in one pass
Sink visited land cells to '0' — O(1) visited-marking with no extra data structure
Every cell is processed at most once: total time is O(m·n) regardless of island count or shape
Check bounds BEFORE accessing grid values to avoid IndexError or wrong-row wrap-around
Mention recursion depth risk in interviews; offer iterative BFS as the production-safe alternative
