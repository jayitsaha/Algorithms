LeetCode #55

Jump Game

Given an integer array nums where each element is your maximum jump length, starting at index 0, determine if you can reach the last index.

Medium Greedy Track Max Reachable Time: O(n) Space: O(1)
1 Understanding the Problem ▾

You start at index 0 of an array. Each element nums[i] tells you the maximum number of steps you can jump forward from position i — you can jump anywhere from 1 to nums[i] positions (you don't have to use all your jump power). The goal: can you reach the last index?

Think of the array as a series of platforms over a chasm. Each platform is labeled with how far you can leap from it. You need to know whether it's possible to cross — not which exact route to take.

💡 Key Insight

You don't need to simulate all jump sequences. Instead, maintain one number: the farthest index you can reach right now. At each position i, if i is within reach, update the max reach using i + nums[i]. If you ever step onto an index that's beyond your current max reach, you're stranded forever — no prior jump could have gotten you there.

✓ Reachable: [2, 3, 1, 1, 4]

i=0: reach = max(0, 0+2) = 2
i=1: reach = max(2, 1+3) = 4 ✓
4 ≥ 4 (last idx) → True

✗ Stranded: [3, 2, 1, 0, 4]

i=0..3: reach stays at 3
i=4: 4 > max_reach(3) — stuck!
Can't reach here → False

Edge cases: Single element [0] → already at the last index, return True. [0, 1, 2] → stuck at index 0 immediately, return False. First element is large (e.g. [100, 0, 0, ...]) → max reach jumps to 100, return True instantly. All zeros → only reachable if n=1.

2 The Algorithm: Track Max Reachable (Greedy) ▾

This is a textbook greedy problem. The locally optimal choice at each position: extend the reachable frontier as far as possible. We never need to revisit positions or try different jump lengths — the running maximum captures everything.

1 Initialize max_reach = 0. We're standing at index 0, so the farthest we can reach starts at 0. Also set n = len(nums).
2 Iterate: for i in range(n). Visit every index from left to right.
3 Stranded check: if i > max_reach → return False. If the current index is beyond our reach, no previous jump could have gotten us here. We're permanently stuck.
4 Update: max_reach = max(max_reach, i + nums[i]). From index i, jumping the maximum nums[i] steps reaches i + nums[i]. Take the max to keep our best reach so far.
5 Early exit: if max_reach ≥ n-1 → return True. If we can already reach the last index, we're done. No need to continue.
6 After the loop → return True. If we completed all iterations without getting stranded, the last index was reachable all along.

🔑 Why Greedy Is Correct Here

Greedy works when the local optimum leads to a global optimum. Here: at each reachable position, the best possible jump is nums[i] steps. We always want the maximum reach — there's no scenario where a shorter jump from any position would enable reaching farther overall. The running max is a "union" of all reachable jump ranges simultaneously.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every iteration, max_reach equals the maximum index reachable from any position we could have visited from index 0. Formally: max_reach = max(j + nums[j]) for all j ≤ i where j itself was reachable. This invariant holds because: (1) we only update when i ≤ max_reach (i.e., i is reachable), and (2) we always take the max to include all prior information.

Why "i > max_reach" means failure: If max_reach < i, it means every reachable position j < i has been processed, and none of them could reach beyond max_reach. Index i — and everything beyond — is cut off from index 0. No different choice of earlier jumps would change this, because max_reach already represents the best possible outcome from every reachable position seen so far.

Why one pass suffices: The array has a left-to-right dependency — to reach index k, you must have been able to reach some earlier index first. Processing left-to-right and maintaining a running max means we see all dependencies in the right order. No backtracking needed.

⚠️ Common Mistake: "Jump Exactly nums[i] Steps"

Each element is the maximum jump length, not the exact jump. You can jump any distance from 1 to nums[i]. This is why we use i + nums[i] as the furthest reach, not a specific next index. A 0 at position i means you're stuck if you land there — but you may have already passed index i via a longer earlier jump.

💡 Intuition via Analogy

Imagine you're watering plants with a hose that can reach max_reach meters. Each time you stand at a new tap (reachable index), you extend the hose farther if the tap has better pressure (larger nums[i]). You can water any plant within reach — you don't need to walk to each plant. The "stuck" case is when the next plant is beyond your hose's maximum reach even with all taps combined.

4 The Code (Annotated) ▾

Solution 1 — Greedy (Optimal) — Interview Pick

def canJump(nums: list[int]) -> bool:
max_reach = 0 ← Farthest index reachable from any visited position so far
n = len(nums) ← Cache length; used in loop and exit check
for i in range(n): ← Visit every index left to right
if i > max_reach: ← Are we beyond what's reachable? Stranded!
return False
max_reach = max(max_reach, i + nums[i]) ← Extend reach by jumping nums[i] from here
if max_reach >= n - 1: ← Already proven last index is reachable
return True ← Early exit — no need to continue
return True ← Loop completed without getting stranded → reachable
⚠️ Can I simplify and just return max_reach ≥ n-1 at the end?

Yes! You can remove the early exit and just write return max_reach ≥ n-1 after the loop. The early exit is purely an optimization — it avoids processing remaining elements when the answer is already known. For interviews, showing the early exit demonstrates you think about efficiency. Both versions are correct.

Solution 2 — Dynamic Programming (O(n²)) — for understanding

def canJump(nums: list[int]) -> bool:
n = len(nums)
dp = [False] * n ← dp[i] = True if index i is reachable from 0
dp[0] = True ← We start at index 0 by definition
for i in range(1, n): ← For each index, check if any prior index can reach it
for j in range(i): ← Check all indices before i
if dp[j] and j + nums[j] >= i: ← j is reachable AND can jump far enough
dp[i] = True
break ← Found one valid predecessor — no need to check more
return dp[n - 1] ← Is the last index reachable?

Solution 3 — Brute Force DFS (O(2ⁿ)) — shows exponential blowup

def canJump(nums: list[int]) -> bool:
def dfs(i: int) -> bool: ← Can we reach the end starting from index i?
if i >= len(nums) - 1: ← Base: at or past last index
return True
for jump in range(1, nums[i] + 1): ← Try every possible jump length
if dfs(i + jump): ← Recurse on the jump destination
return True
return False ← No jump from here leads to success
return dfs(0)
💡 Complexity Summary

DFS Brute Force: Time O(2ⁿ) — exponential branching at each step; Space O(n) call stack
DP (Tabulation): Time O(n²) — two nested loops; Space O(n) for dp array
Greedy (Optimal): Time O(n) — single pass; Space O(1) — just one variable

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Read the Question Type

"This asks 'can you reach?' — a reachability question, not a shortest-path question. This is a critical distinction. Reachability often admits greedy or simple DP. If it asked for the minimum number of jumps, I'd use BFS or greedy with boundary tracking."

Step 2: Propose the Brute Force

"The naive approach: DFS from index 0, trying all possible jump lengths at each position. This correctly checks every path but runs in O(2ⁿ) due to branching. I can add memoization to cache per-index results and bring it to O(n²)."

Step 3: Identify the Greedy Opportunity

"Key observation: I don't care which specific path reaches index k — just whether any does. At each reachable position, the best I can do is jump nums[i] steps. The running maximum of i + nums[i] over all reachable positions tells me everything about reachability. That's O(n) and O(1) space."

Step 4: Handle the "Stranded" Edge Case

"The only way we can fail is if we reach an index that no prior jump could have reached — when i > max_reach. This is the definitive failure condition. I check this before updating max_reach so we don't count an unreachable position as contributing."

🗣️ What to Say to the Interviewer

"I'll start with brute-force DFS — try all jump sequences from index 0. O(2^n), can memoize to O(n^2).
But the real insight is greedy: I only need to track max_reach — the farthest I can reach from
any position I've visited. For each index i, if i > max_reach, I'm stranded — return False.
Otherwise update max_reach = max(max_reach, i + nums[i]).
If max_reach reaches n-1, return True early. Single pass, O(n) time, O(1) space.
The greedy works because taking the maximum jump at each reachable position dominates
all shorter options — there's no benefit to jumping less from any reachable index."
🔮 Likely Follow-up Questions
  • "What's the minimum number of jumps to reach the end?" → Jump Game II (#45) — greedy with two boundary pointers (current vs next reach)
  • "Work backwards instead?" → Yes: track the "leftmost good index" from the right; propagate reachability backwards
  • "What if jumps can go backwards?" → Now it's a graph problem; need BFS/DFS with visited set
  • "What if nums[i] is the exact distance, not max?" → Cycle detection problem — use fast/slow pointer or visited tracking

7 Summary & Key Takeaways ▾
Track the maximum reachable index — not individual jump paths
From any reachable index i, the farthest you can extend is i + nums[i]
If current index i exceeds max_reach, you're stranded — return False immediately
Single pass O(n), O(1) space — the greedy dominates DP and DFS for this problem
Reachability questions often admit greedy; minimum-hops questions need BFS/boundary tracking

🎯 When to Recognize Track Max Reachable
  • Problem asks "can you reach X?" with variable-size forward moves
  • Each element defines a maximum range, not a fixed step
  • You need reachability, not the shortest path (if minimum hops needed → BFS)
  • Zero values in the array create potential blockades to detect
  • Interval coverage problems often use the same frontier-extension logic
