LeetCode #662

Maximum Width of Binary Tree

Return the maximum width of the tree. Width of one level = length between leftmost and rightmost non-null nodes, including null nodes between them.

Medium Trees BFS Level Order Track Index Per Level Time: O(n) Space: O(n)
1 Understanding the Problem ▾

Picture the binary tree drawn on a grid. At each row (level), draw a horizontal line from the leftmost non-null node to the rightmost non-null node. The "width" is the length of that line — counting every position (including nulls in between). Find the widest row.

💡 Key Insight

Width is about positions, not node count. A level like [5, null, null, 9] has 2 nodes but width 4 because there are two null "holes" between them. We need a way to measure the gap without materializing null nodes.

Tree: [1, 3, 2, 5, 3, null, 9]

Level 0: [1] → width = 1
Level 1: [3, 2] → width = 2
Level 2: [5, 3, ∅, 9] → width = 4
Answer: 4

Naive counting (wrong!)

Level 2: 3 nodes
Node count = 3
Missing null hole = 4
Must count positions, not nodes

Edge Cases: Single node (width = 1). All right-skewed chain — every level has one node, width = 1. Very deep trees — virtual indices can overflow 32-bit integers, so index normalization per level is essential.

The virtual index trick: Treat the tree like a heap stored in an array: root = index 0, left child of node i = 2i, right child = 2i+1. Width at a level = rightmost_index − leftmost_index + 1. Nulls between two real nodes will naturally create index gaps that the formula captures.

2 The Algorithm: BFS + Virtual Index Tracking ▾

We use level-order BFS and store each node alongside its virtual index — the index it would occupy in a complete binary tree array. At every level, the width is just the difference between the last and first virtual index plus one.

1 Initialize: Push (root, 0) onto the BFS queue. Set max_width = 0.
2 Per-level setup: Snapshot level_size = len(queue). Peek at the front to get first_idx — the leftmost index on this level.
3 Normalize: For each node dequeued, compute idx -= first_idx. This shifts all indices so the leftmost starts at 0, preventing exponential growth in deep trees.
4 Enqueue children: Left child gets index 2 * idx, right child gets 2 * idx + 1. Only enqueue if the child node exists.
5 Compute width: After processing all nodes on a level, width = last_idx + 1 (since first is normalized to 0). Update max_width.
🔑 Why Normalize? The Overflow Problem

At depth d, raw indices reach 2^d. For a tree of depth 50, that's 2^50 — overflows 32-bit integers and even some 64-bit ones. Subtracting the level's minimum index resets relative positions to small numbers. Since width is a difference, subtracting the same base from both ends leaves the result unchanged: (R − base) − (L − base) = R − L.

3 Why It Works: The Invariant ▾
🔐 The Invariant

Every node in the queue carries a virtual index consistent with the heap-array numbering: if a node has index i, its left child is always 2i and right child is 2i+1. This faithfully models the position a node would occupy among all null and non-null positions in a complete binary tree at the same depth.

Why the width formula is exact: At any level, the leftmost non-null node has the smallest virtual index L, the rightmost has the largest R. Between them, positions L, L+1, ..., R are all valid heap-array slots at this depth — some occupied by real nodes, some by nulls. Width = R − L + 1 counts them all, which is exactly what the problem demands.

Why normalization preserves correctness: We subtract the same constant (the level's first index) from every index on a level. The leftmost normalized index becomes 0. The rightmost becomes R − L. Width = (R − L) − 0 + 1 = R − L + 1. Unchanged.

⚠️ Common Mistake: Forgetting to Normalize Before Enqueueing Children

Normalize before computing child indices. If you normalize after, the children are enqueued with the pre-normalization (potentially huge) index values, and the overflow problem reappears one level later. Always: idx -= first_idx, then queue.append((child, 2*idx)).

Skewed tree edge case: A fully right-skewed tree (each node only has a right child) produces nodes with indices 0, 1, 3, 7, 15, ... At each level there is exactly one node, so L = R and width = 1 at every level. Correct — with normalization, these reset to 0 each level anyway.

4 The Code (Annotated) ▾

Solution 1 — BFS with Virtual Index (Optimal, Interview Pick)

from collections import deque
def widthOfBinaryTree(root) -> int:
if not root: return 0 ← handle empty tree
max_width = 0
queue = deque([(root, 0)]) ← store (node, virtual_index) pairs
while queue:
level_size = len(queue) ← snapshot: how many nodes at this depth
_, first_idx = queue[0] ← peek: leftmost index on this level
last_idx = first_idx ← will track rightmost (updated as we process)
for _ in range(level_size):
node, idx = queue.popleft() ← dequeue next node at this depth
idx -= first_idx ← NORMALIZE: prevent exponential index growth
last_idx = idx ← rightmost index is the last one we process
if node.left:
queue.append((node.left, 2*idx)) ← left child = 2i
if node.right:
queue.append((node.right, 2*idx + 1)) ← right child = 2i+1
max_width = max(max_width, last_idx + 1)← first_idx normalized to 0, so width = last+1
return max_width
⚠️ Why last_idx + 1 (not last_idx - first_idx + 1)?

After normalization, the first index on this level is always 0. So first_idx (normalized) = 0 and the formula simplifies to last_idx - 0 + 1 = last_idx + 1. If you forget normalization happened, you might write last_idx - first_idx + 1 and double-subtract — careful!

Solution 2 — DFS with Leftmost Index Map (O(h) space)

def widthOfBinaryTree(root) -> int:
left_most = {} ← depth → leftmost index seen at that depth
ans = [0] ← mutable container (or use nonlocal in nested fn)
def dfs(node, depth, idx):
if not node: return
if depth not in left_most: ← preorder: first visit = leftmost (we go left first)
left_most[depth] = idx
ans[0] = max(ans[0], idx - left_most[depth] + 1) ← update max width for this depth
dfs(node.left, depth + 1, 2 * idx) ← left before right ensures leftmost recorded first
dfs(node.right, depth + 1, 2 * idx + 1)
dfs(root, 0, 1) ← 1-based index avoids 0*2=0 edge case at root
return ans[0]
💡 Complexity Summary

BFS approach: Time O(n) — each node enqueued and dequeued once. Space O(n) — queue holds up to one full level (~n/2 nodes at the last level of a complete tree).
DFS approach: Time O(n). Space O(h) — call stack depth equals tree height. O(log n) for balanced, O(n) worst case skewed.
In interviews, BFS is preferred — it's more intuitive for level-based problems.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize "Per Level" → BFS

The problem talks about the width of each level. Any time a problem asks for a property per level in a binary tree, your first instinct should be BFS. Level-order traversal processes exactly one level per outer loop iteration.

Step 2: Spot the Gap Problem

Width includes nulls between non-null nodes. Counting non-null nodes per level is wrong. You need to measure positions. Ask yourself: "how do I assign unique horizontal positions to nodes without materializing nulls?"

Step 3: Invent Virtual Indexing

Recall the heap/array representation of binary trees: node at index i → left child 2i, right child 2i+1. This gives every position (real or null) a unique number. Width = rightmost_index − leftmost_index + 1.

Step 4: Proactively Flag Overflow

At depth d, indices reach 2^d. For a tree of depth 50, that overflows most integer types. Propose normalization immediately — subtract the first index of each level from all indices on that level. This demonstrates you think about production quality, not just correctness on small inputs.

🗣️ What to Say to the Interviewer

"Since we need the width of each level, I'll use BFS.
The tricky part is that width includes null gaps — I can't just count non-null nodes.
I'll assign each node a virtual index: root = 0, left child of i = 2i, right child = 2i+1.
Width at a level is rightmost_index minus leftmost_index plus one — this counts null gaps too.
One gotcha: indices grow as 2^depth, risking overflow. I'll normalize at each level
by subtracting the leftmost index, so values stay small.
Time O(n), Space O(n) for the queue."
🔮 Likely Follow-up Questions
  • "Can you do DFS instead?" → Yes: preorder DFS, record leftmost index per depth in a dict, compute width per node visit. O(h) space.
  • "What if you use 1-based indexing?" → Same formula, avoids 0*2=0 confusion at root's children; personal preference.
  • "What's the worst-case tree?" → A perfect binary tree of depth d — width = 2^d at the last level. Still O(n) since we visit each node once.
  • "Could a complete binary tree have a narrower last level?" → Yes! The second-to-last level can be wider if the last level has few nodes far apart.

7 Summary & Key Takeaways ▾
Virtual indexing (parent i → children 2i, 2i+1) measures positional gaps without materializing null nodes
Width = rightmost_index − leftmost_index + 1 at a level, which correctly counts null "holes" between real nodes
Normalize each level by subtracting its first index — prevents 2^depth overflow in deep trees
Normalize BEFORE computing children's indices, so child indices also inherit the small base
BFS naturally isolates each level; snapshot len(queue) before inner loop to process exactly one level

🎯 When to Recognize the "Track Index Per Level" Pattern
  • "Width/span/range of a tree level" → virtual index + BFS
  • "Position of a node among all possible positions at its depth" → heap numbering
  • "Check if binary tree is complete" → indices must be contiguous (gap = not complete)
  • Any level property that depends on structural position, not just node values
