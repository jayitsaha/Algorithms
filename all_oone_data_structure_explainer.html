LeetCode #432

All O`one Data Structure

Design a data structure that supports inc(key), dec(key), getMaxKey(), and getMinKey() — all in O(1) time.

Hard Data Structure Design DLL + Hash Map Time: O(1) all ops Space: O(n)
1 Understanding the Problem ▾

We need a data structure that tracks string counts (like a frequency counter) but additionally needs to report the key with the maximum count and the key with the minimum count — all in constant time O(1).

💡 Key Insight

A plain hash map gives O(1) inc/dec but finding the min/max requires a full scan — O(n). A sorted structure (heap, balanced BST) gives O(log n) for all ops. The trick to achieve true O(1) for all four operations is to maintain a doubly-linked list sorted by frequency, where each node groups all keys that currently have the same count.

✓ What We Need (O(1) all)

inc("a") — count "a" by 1
inc("b") — count "b" by 1
inc("a") — count "a" by 1
getMaxKey() → "a" (count=2)
getMinKey() → "b" (count=1)

✗ Naive HashMap approach fails

inc/dec → O(1) ✓
getMaxKey → O(n) scan ✗
getMinKey → O(n) scan ✗

Can't find min/max without scanning all keys.

Edge Cases to consider: getMaxKey() or getMinKey() on an empty structure should return "". Decrementing a key with count 1 should remove it from tracking entirely. All keys start at count 0 (not tracked until first inc call).

2 The Algorithm: Doubly-Linked List + Hash Map ▾

We combine two data structures: a doubly-linked list (DLL) sorted by frequency (ascending from head to tail), and a hash map that maps each key to the DLL node containing it.

DLL Node stores: frequency count + set of all keys with that count + prev/next pointers
key_to_node map: "a" → node(freq=2), "b" → node(freq=1), "c" → node(freq=3) ...
DLL order: DUMMY_HEAD ↔ node(1, {"b"}) ↔ node(2, {"a"}) ↔ node(3, {"c"}) ↔ DUMMY_TAIL
Min key: Any key in DUMMY_HEAD.next.keys (lowest freq node, O(1) access)
Max key: Any key in DUMMY_TAIL.prev.keys (highest freq node, O(1) access)
1 Initialize: Create two sentinel nodes — head (dummy min) and tail (dummy max) — linked together. Create an empty key_to_node hash map.
2 inc(key): Find the key's current node (or treat as having freq=0). The new freq is curr_freq + 1. If the next node has exactly curr_freq + 1, move the key there; otherwise insert a new node with that freq immediately after. Remove key from old node (and delete old node if its key set becomes empty). Update key_to_node[key].
3 dec(key): Same logic in reverse. Find current node. New freq is curr_freq - 1. If new freq = 0, remove key entirely (del key_to_node[key]). Otherwise, if the prev node has exactly curr_freq - 1, move key there; else insert a new node before current. Clean up empty nodes.
4 getMinKey(): Return any key from head.next.keys. Since DLL is sorted ascending, head.next is always the minimum-frequency node. Return "" if head.next is tail.
5 getMaxKey(): Return any key from tail.prev.keys. Since DLL is sorted ascending, tail.prev is always the maximum-frequency node. Return "" if tail.prev is head.
🔑 Why Doubly Linked (not singly)?

When decrementing, we need to insert a new node before the current node, which requires access to curr.prev. With a singly linked list we'd need to scan from the beginning to find the predecessor — destroying O(1). The doubly-linked structure lets us splice nodes in constant time from both directions.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At all times, the doubly-linked list is sorted in strictly ascending order of frequency from head.next to tail.prev. Every key maps in key_to_node to exactly the one DLL node whose freq matches the key's current count. No two distinct DLL nodes share the same frequency.

Why inc/dec maintain this invariant: When a key's count changes from f to f+1, we look at curr.next. If curr.next.freq == f+1, the target bucket already exists — we add the key there (no resorting needed, adjacency is preserved). If curr.next.freq > f+1 (a gap), we insert a new node between curr and curr.next — maintaining sorted order. The same logic applies symmetrically for dec. Each operation touches at most 2 nodes and requires O(1) pointer manipulation.

Why min/max are always O(1): The sentinel head is always the left boundary, and tail is always the right boundary. Their immediate neighbors are the frequency-minimum and frequency-maximum nodes respectively. Since these pointers are maintained by every inc/dec, reading min/max is just pointer dereference — no traversal.

⚠️ The "set" inside each node matters

Each DLL node holds a Python set of keys (not a list). When we need to remove a key from a node, set removal is O(1). When getMaxKey()/getMinKey() asks for "any" key, we use next(iter(node.keys)) to grab one in O(1). Any ordered structure (list) would require O(n) removal.

Empty node cleanup: After moving a key away from a node, if that node's key set becomes empty, we must remove the node from the DLL immediately. Otherwise the invariant breaks — we'd have a frequency node with no keys, potentially being returned as the min/max candidate.

4 The Code (Annotated) ▾

Solution 1 — DLL + Hash Map (Optimal, O(1) all operations)

class Node:← DLL node: stores a frequency + all keys at that freq
def __init__(self, freq):
self.freq = freq ← This node represents all keys with count = freq
self.keys = set() ← Python set: O(1) add/remove/any-element
self.prev = self.next = None
class AllOne:
def __init__(self):
self.head = Node(0) ← Sentinel min-boundary (freq=0, never real)
self.tail = Node(float('inf')) ← Sentinel max-boundary
self.head.next = self.tail ← Empty list: head ↔ tail
self.tail.prev = self.head
self.key_to_node = {} ← key → DLL node containing that key
def _insert_after(self, node, new_node):← Splice new_node into DLL after node
new_node.prev = node
new_node.next = node.next
node.next.prev = new_node
node.next = new_node ← 4 pointer updates, all O(1)
def _remove_node(self, node):← Unlink node from DLL (when its keys set empties)
node.prev.next = node.next
node.next.prev = node.prev
def inc(self, key):
curr = self.key_to_node.get(key, self.head)← Unknown key? Treat as at sentinel (freq=0)
new_freq = curr.freq + 1
if curr.next.freq == new_freq:← Bucket for new_freq already exists right after?
curr.next.keys.add(key) ← Yes! Just move key there
self.key_to_node[key] = curr.next
else:← No bucket with new_freq yet — create one
new_node = Node(new_freq)
new_node.keys.add(key)
self._insert_after(curr, new_node) ← Inserts right after curr, preserving sort order
self.key_to_node[key] = new_node
curr.keys.discard(key) ← Remove key from old node (discard is safe if not present)
if not curr.keys and curr is not self.head:← Old node empty AND not sentinel? Delete it
self._remove_node(curr)
def dec(self, key):
curr = self.key_to_node[key] ← key must exist (problem guarantees this)
new_freq = curr.freq - 1
if new_freq == 0: ← Count hits 0 → remove key entirely
del self.key_to_node[key]
elif curr.prev.freq == new_freq:← Bucket for new_freq already exists right before?
curr.prev.keys.add(key)
self.key_to_node[key] = curr.prev
else:← Create new bucket before curr
new_node = Node(new_freq)
new_node.keys.add(key)
self._insert_after(curr.prev, new_node)← Insert after curr.prev = before curr
self.key_to_node[key] = new_node
curr.keys.discard(key)
if not curr.keys:
self._remove_node(curr)
def getMaxKey(self):
if self.tail.prev is self.head: return ""← Empty structure
return next(iter(self.tail.prev.keys)) ← Any key at max-freq node
def getMinKey(self):
if self.head.next is self.tail: return ""← Empty structure
return next(iter(self.head.next.keys)) ← Any key at min-freq node
⚠️ Why use curr.keys.discard() not curr.keys.remove()?

In inc, the new key may not yet be in curr.keys when curr is the sentinel head (for a brand-new key). discard silently does nothing if the element isn't present, while remove would raise KeyError. The curr is not self.head guard also prevents us from removing the sentinel node itself.

💡 Complexity Summary

All four operations: O(1) — pointer manipulation only, no loops, no comparisons beyond immediate neighbors.
Space: O(n) — one DLL node per distinct active frequency level, one hash map entry per distinct key.
In the worst case there are n distinct frequency levels (after n different inc/dec patterns), so DLL length ≤ n.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify the Bottleneck

"inc and dec are naturally O(1) with a plain hash map. The hard part is min/max in O(1). I need to keep keys ordered by their count at all times — the question is what structure allows ordered insertion AND O(1) min/max access."

Step 2: Evaluate Alternatives

"A heap gives O(log n) inc/dec because we'd need to bubble elements. A sorted array/tree similarly needs O(log n) inserts. A doubly-linked list sorted by frequency lets me do O(1) splicing because I'm always moving a key to an adjacent frequency — either the neighboring node already exists or I create one right next to me."

Step 3: Design the Node Structure

"Each DLL node stores: (1) the frequency count, (2) a set of all keys at that frequency, (3) prev/next pointers. The hash map maps each key to its current DLL node for O(1) lookup. Sentinels avoid null checks."

Step 4: Walk Through inc Carefully

"For inc(key): get the current node (sentinel if new). New freq = curr.freq + 1. If curr.next already has this freq, add key there. Otherwise, create a new node between curr and curr.next. Remove key from curr, delete curr if empty."

🗣️ What to Say to the Interviewer

"The naive approach — hash map for counts — gives O(1) inc/dec but O(n) for min/max.
To get O(1) everywhere, I need the keys sorted by frequency at all times with O(1) insert.
I'll use a doubly-linked list where each node holds a frequency and a set of keys at that count.
The list is sorted ascending: head→tail goes low freq to high freq.
A separate hash map maps each key to its DLL node for O(1) location.
On inc: move key to freq+1 bucket (possibly creating it). On dec: move to freq-1 bucket (or remove if zero).
getMax returns any key from tail.prev.keys; getMin from head.next.keys. Both O(1)."
🔮 Likely Follow-up Questions
  • "What if you need ALL keys at max frequency?" → Return the entire tail.prev.keys set — already O(1) access, O(k) to iterate.
  • "How do you handle concurrent modifications?" → Need locks around DLL updates (not thread-safe as-is).
  • "Can you do this with a sorted container instead of DLL?" → Yes: SortedList from sortedcontainers gives O(log n) — simpler code but not truly O(1).
  • "LFU Cache uses a similar structure — how?" → LFU tracks per-frequency key sets with same DLL trick, plus an LRU ordering within each frequency bucket (#460).

7 Summary & Key Takeaways ▾
A DLL sorted by frequency enables O(1) min/max by keeping sentinel head/tail boundaries
Each DLL node holds a set of all keys at that frequency for O(1) key operations
On inc: move key one step right in DLL (create node only if gap exists)
On dec: move key one step left (delete key entirely if new_freq = 0)
Always delete empty DLL nodes immediately to maintain invariant
Sentinels (head/tail) eliminate null pointer edge cases in pointer manipulation

🎯 When to Recognize the DLL + Hash Map Pattern
  • "Design a data structure with O(1) min/max on counts/frequencies" → DLL + HashMap
  • "LRU/LFU cache" → ordered DLL for eviction + HashMap for O(1) lookup
  • "Maintain ordering AND O(1) access to both ends" → Doubly linked sentinel structure
  • "Group elements by a dynamic numeric key that changes incrementally" → DLL of buckets
