LeetCode #47

Permutations II

Given a collection of numbers that might contain duplicates, return all possible unique permutations in any order.

Medium Backtracking Sort + Skip Duplicates Time: O(n · n!) Space: O(n)
1 Understanding the Problem ▾

A permutation is an arrangement of all the elements of a sequence in some order. For [1, 2, 3] there are 3! = 6 permutations. When the input has duplicates — say [1, 1, 2] — naively generating all 3! = 6 arrangements gives repeats. The unique ones are only 3: [1,1,2], [1,2,1], [2,1,1].

💡 Key Insight

If we sort the input first, duplicate values land adjacent to each other. At any recursion level, if we're about to pick the same value that the previous sibling already tried at this position, we skip it — the entire subtree it would explore is structurally identical to the one just explored. This eliminates duplicates before they're ever generated.

✗ Without dedup: [1, 1, 2]

Pick 1a: → [1,1,2], [1,2,1]
Pick 1b: → [1,1,2], [1,2,1] (dup!)
Pick 2: → [2,1,1]
6 results, 2 duplicates

✓ With dedup: [1, 1, 2]

Pick 1 (first): → [1,1,2], [1,2,1]
Skip 1 (same as prev sibling!) ← pruned
Pick 2: → [2,1,1]
3 unique results, no waste

Edge Cases: all elements identical ([2,2,2] → only 1 permutation); single element; all distinct elements (falls back to LeetCode #46 behavior, 0 pruning needed); negative numbers (sort still works).

2 The Algorithm: Backtracking with Sort + Skip ▾

The backbone is backtracking: at each recursive call we choose one unused number, place it in the next position, recurse deeper, then undo the choice (backtrack) and try the next option. The skip condition is what makes this efficient for duplicates.

1 Sort the input. Sorting places duplicate values adjacent to each other. This is non-negotiable — the skip condition depends on it.
2 Track which indices are used. A boolean array used[i] marks which positions in the sorted input have already been placed in the current path. Two elements with the same value get distinct indices.
3 Skip duplicate siblings. Before trying index i, check: is nums[i] == nums[i-1] and not used[i-1]? Both conditions must hold. If so, skip — this value at this position was already explored by the previous sibling.
4 Choose → recurse → backtrack. Mark used[i] = True, append nums[i] to path, recurse. On return: pop from path, set used[i] = False. This restores state for the next sibling.
5 Base case: When len(path) == n, copy the path and record it. Copying is critical — we never append a live reference to a list we'll mutate later.
🔑 The Skip Condition Dissected

i > 0 and nums[i] == nums[i-1] and not used[i-1]

The not used[i-1] part is subtle. If used[i-1] is True, then nums[i-1] is an ancestor in our current path — using nums[i] after it creates a genuinely different arrangement (like putting the second 1 after the first). We must NOT skip in that case. We only skip when used[i-1] is False, meaning it's a sibling whose entire subtree was already explored.

Skip condition at position 0 for sorted [1, 1, 2]
Try index 0 → value 1
→
Try index 1 → value 1 (SKIP: same as prev sibling)
→
Try index 2 → value 2
Only the first occurrence of each value is explored at each level. Identical subtrees are never duplicated.
3 Why It Works: The Invariant ▾
🔐 The Invariant

At every position in the recursion, among all elements with the same value, we always pick the leftmost one whose predecessor is already in the current path (or is absent). Concretely: we use nums[i] at this position only if either (a) nums[i] is a distinct new value, or (b) nums[i-1] is already an ancestor in our path. This gives each group of identical values a canonical ordering and prevents two different calls from generating the same sequence.

Why completeness holds: The skip only fires when used[i-1] is False (sibling, not ancestor). If used[i-1] is True (nums[i-1] IS in our path), we do NOT skip nums[i] — this creates a genuinely different arrangement. So every distinct permutation is reachable.

Why uniqueness holds: Two permutations are identical iff they pick the same value sequence at each position. Equal values are adjacent after sorting. The skip condition ensures that at any given position, each value is tried at most once. Therefore each unique value sequence is generated exactly once.

⚠️ Most Common Wrong Approach

Writing i > 0 and nums[i] == nums[i-1] without and not used[i-1] is incorrect. This would skip nums[i] even when nums[i-1] is an ancestor, preventing valid arrangements like [1, 1, 2] from being generated (the inner 1 would be wrongly skipped at position 1 even though the outer 1 is already placed).

All-same edge case ([2, 2, 2]): At position 0, try index 0 (value 2). Indices 1 and 2 are skipped (same value, prev sibling unused). At position 1, index 1 (value 2) — used[0]=True, so not skipped! At position 2, only index 2 remains. Result: exactly [2,2,2]. Correct!

4 The Code (Annotated) ▾

Solution 1 — Sort + Skip Backtracking (Optimal · Interview Pick)

def permuteUnique(nums): ← input may contain duplicates
nums.sort() ← CRITICAL: sort so duplicates become adjacent
result = [] ← collect all unique permutations here
used = [False] * len(nums) ← per-index, not per-value: tracks current path
def backtrack(path): ← path = partial permutation being built
if len(path) == len(nums): ← base case: all n elements placed
result.append(path[:]) ← path[:] = copy! path will be mutated later
return
for i in range(len(nums)): ← try every index as the next element
if used[i]: ← skip if this index is already in current path
continue
if i > 0 and nums[i] == nums[i-1] \ ← same value as previous index...
and not used[i-1]: ← ...and prev is a sibling (not ancestor) → SKIP
continue
used[i] = True ← CHOOSE: mark this index as in-use
path.append(nums[i]) ← add its value to the partial permutation
backtrack(path) ← EXPLORE: recurse to fill next position
path.pop() ← BACKTRACK: remove value
used[i] = False ← BACKTRACK: un-mark index
backtrack([]) ← start with empty path
return result
⚠️ Why path[:] and not just path?

Python lists are passed by reference. result.append(path) stores a pointer to the same list that backtracking keeps mutating. Every recorded permutation would become the final state of path — all wrong. path[:] (or list(path)) takes a snapshot that is immutable to future changes.

Solution 2 — Brute Force with Set Deduplication

from itertools import permutations ← stdlib generates all n! permutations as tuples
def permuteUnique(nums):
return [list(p) for p in set(permutations(nums))] ← set removes tuple duplicates

Correct but wasteful: generates all n! permutations before deduplication. When there are k identical elements, this does k! unnecessary work. The sort+skip approach prunes those branches immediately.

💡 Complexity Summary

Sort + Skip (Interview Pick): O(n · n!) time worst case (all distinct), O(n) recursion depth + path. Much better when duplicates exist — with all-same input, only 1 permutation is ever built.
Brute Force + Set: O(n · n!) time always, O(n · n!) space.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize Backtracking

"We need to generate all valid arrangements — this is an enumeration problem. Backtracking is the canonical approach: choose a position, fill it, recurse, undo. I start from LeetCode #46 (Permutations without duplicates) as the base template."

Step 2: Note the Duplicate Problem

"With duplicates, a set at the end works but wastes O(n · n!) work generating redundant branches. Better: prune at the source. If two siblings would start with the same value, the second subtree is identical to the first — skip it."

Step 3: Derive the Skip Condition

"Sort first so equal values are adjacent. In the loop, skip index i if: (1) nums[i] equals nums[i-1] AND (2) nums[i-1] is not in our current path (used[i-1]=False). Condition 2 is crucial — it distinguishes a duplicate sibling from a valid deeper placement."

Step 4: State Complexity

"Worst case O(n · n!) when all elements are distinct — no pruning fires. With duplicates, the effective branching factor shrinks significantly. Space is O(n) for the recursion depth and path."

🗣️ What to Say to the Interviewer

"I'll use backtracking: at each step choose an unused element, recurse, undo.
To handle duplicates: first sort the array so equal values are adjacent.
In the for loop, I skip index i if nums[i]==nums[i-1] AND used[i-1] is False.
That second condition is key: if i-1 is already in my path, i is a valid deeper choice.
But if i-1 is a sibling (not in path), placing i would duplicate the subtree — skip.
Base case: path length equals n — copy and record. Time O(n·n!) worst case."

7 Summary & Key Takeaways ▾
Sort first — always. Without it, the skip condition is meaningless.
Skip when: same value as prev AND prev is a sibling (not in current path).
Always copy the path when recording: result.append(path[:]).
used[] is indexed (not value-based) — two equal values have different indices.
With all-same input, only 1 permutation is generated instead of n! — maximum pruning.

🎯 When to Use Sort + Skip Duplicates
  • "Generate all unique [permutations / subsets / combinations] from an array with duplicates"
  • You're doing backtracking and the input contains repeated values
  • You want to avoid a post-processing deduplication step
  • Output size is much smaller than n! because of duplicates
