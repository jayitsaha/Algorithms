LeetCode #253

Meeting Rooms II

Given an array of meeting time intervals, find the minimum number of conference rooms required.

Medium Intervals Greedy + Min-Heap Time: O(n log n) Space: O(n)
1 Understanding the Problem ▾

You're given a list of meetings with start and end times. You need to book conference rooms so every meeting can happen. Meetings that overlap in time need different rooms. The question: what is the fewest rooms that suffices?

💡 Key Insight

The minimum rooms required equals the maximum number of meetings simultaneously active at any moment. We don't need to figure out room assignments — just count the peak concurrent overlap.

Walk-through example: intervals = [[0,30], [5,10], [15,20]]

At time 5–10, both Room 1 and Room 2 are active simultaneously → 2 rooms. Room 2 finishes at 10, then [15,20] can reuse it. Answer = 2.

✗ Naive: Pairwise Comparison

Check all pairs for overlap.
O(n²) — too slow for large n.
Hard to track which rooms are free.

✓ Optimal: Min-Heap of End Times

Sort by start. Heap tracks active rooms.
O(n log n) time, O(n) space.
Heap root = room freeing up earliest.

Edge cases: No meetings → 0 rooms. Single meeting → 1 room. All disjoint → 1 room (keep reusing). All overlapping → n rooms (one per meeting).

2 The Algorithm: Greedy + Min-Heap ▾

The core idea: when a new meeting starts, check if any room is already free. The best candidate to reuse is the room whose current meeting ends earliest. A min-heap of end times always surfaces this room in O(log n) time.

1 Sort by start time. Process meetings in chronological order. This is the greedy ordering — we handle meetings in the sequence they actually arrive, so our room decisions are always made with correct timing context.
2 Initialize a min-heap. Push the first meeting's end time into the heap. The heap stores end times of all currently occupied rooms. The smallest element (heap root) = the room that becomes free the soonest.
3 For each subsequent meeting: If heap[0] ≤ start (the earliest room finishes before or when we start), pop its end time and push our new end time — we're reusing that room. Otherwise, push our end time without popping — we need a new room.
4 Return len(heap). After all meetings are processed, each heap entry represents one occupied room. The heap size is the total number of rooms needed.
🔑 Why Min-Heap Root is Always the Right Check

We only need to ask: "Is there ANY free room?" If the earliest-ending room (heap root) isn't free yet, then certainly no other room is free either — all others end even later. We never need to inspect other rooms. This is why one comparison is sufficient.

💡 Sweep Line Alternative (also O(n log n))

Create two events per meeting: (start, +1) and (end, -1). Sort all events (ties: ends come before starts). Scan left to right maintaining a running count. Return the peak count. This approach is equally valid and often easier to reason about for correctness proofs.

3 Why It Works: The Invariant ▾
🔐 The Invariant

After processing the k-th meeting, the min-heap contains exactly the end times of all meetings that are still running at the moment the (k+1)-th meeting will begin. The heap size equals the number of rooms currently in use.

Proof of no over-allocation: We only push a new entry (allocate a new room) when start < heap[0]. Since the heap root is the minimum end time, all other rooms end even later. No room is free — a new room is genuinely required. We never add rooms unnecessarily.

Proof of no under-allocation: When we pop+push (reuse a room), the condition is start ≥ heap[0]. The room we reuse ended at or before our start time — no overlap. All meetings placed in the same "room slot" over time are strictly sequential: each starts after the previous one ends.

⚠️ Sorting by Start Time is Non-Negotiable

Without sorting, a later-starting meeting could be processed before an earlier one. We might conclude a room is "free" when it actually still has a meeting running. The greedy decision (check heap root vs. current start) is only correct when meetings are processed in chronological start order.

Boundary — when end == start: We use heap[0] <= start. A room ending at time 10 and a meeting starting at time 10 do not overlap — the room is available at time 10. Using <= correctly allows this reuse. If the problem treated intervals as exclusive (open end), you'd use strict <.

4 The Code (Annotated) ▾

Solution 1 — Greedy + Min-Heap (Optimal, Interview Pick)

import heapq ← Python's built-in min-heap (heapq is always a min-heap)
def minMeetingRooms(intervals): ← list of [start, end] intervals
if not intervals: ← guard: 0 meetings = 0 rooms needed
return 0
intervals.sort(key=lambda x: x[0]) ← sort by start time — greedy ordering
heap = [] ← min-heap of end times of active rooms
heapq.heappush(heap, intervals[0][1]) ← Room 1: push first meeting's end time
for start, end in intervals[1:]: ← process meetings 2..n in sorted order
if heap[0] <= start: ← earliest room ends at or before our start?
heapq.heappop(heap) ← yes: free that room (remove old end time)
heapq.heappush(heap, end) ← always push: reuse freed room OR new room
return len(heap) ← heap size = rooms in use = answer
⚠️ Why always push (even when we reused)?

Whether we popped or not, we always push the new meeting's end time. If we popped first (reuse), the pop removes the old end time and the push adds the new one — net size stays the same. If we didn't pop (new room), the push increases the size by 1. This elegant pattern handles both cases in a single heappush.

Solution 2 — Sweep Line / Event Sorting

def minMeetingRooms_sweep(intervals): ← treat each start/end as an event
events = []
for start, end in intervals:
events.append((start, 1)) ← +1: a meeting starts, need a room
events.append((end, -1)) ← -1: a meeting ends, room freed
events.sort(key=lambda x: (x[0], x[1])) ← sort by time; ties: -1 ends before +1 starts
rooms = max_rooms = 0
for _, delta in events: ← scan all events in time order
rooms += delta ← +1 adds concurrent meeting; -1 frees one
max_rooms = max(max_rooms, rooms) ← track peak concurrent count
return max_rooms ← peak = minimum rooms required
💡 Complexity Summary

Min-Heap: Time O(n log n) — sort once, then n heap ops each O(log n). Space O(n) for heap.
Sweep Line: Time O(n log n) — sort 2n events. Space O(n) for events list.
Both are asymptotically optimal. Min-Heap is more interview-common; Sweep Line is elegant for proofs.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Reframe as Peak Overlap

Recognize: "minimum rooms" = "maximum simultaneous meetings." The answer isn't about assigning meetings to rooms — it's about finding the moment when the most meetings are running at once. That maximum value is the answer.

Step 2: Propose Brute Force

"I could scan all n² pairs, detect overlaps, and simulate assignments. That's O(n²). Or I could scan every time point — O(T) where T is time range, which could be huge. Neither is acceptable."

Step 3: Identify the Greedy Opportunity

"If I sort by start time, I process meetings chronologically. When meeting k starts, I want to know: is any room free? The only room that could be free is the one ending soonest. That's a min query — perfect for a min-heap of end times."

Step 4: Code Cleanly and Explain the Invariant

Sort → init heap with first end → loop: check heap[0] vs start → pop if free → always push → return heap size. Proactively mention the ≤ vs < boundary question to show you think about edge cases.

🗣️ What to Say to the Interviewer

"I'll sort meetings by start time, then use a min-heap to track end times of occupied rooms.
For each meeting: if the smallest end time in the heap is ≤ our start, that room is free — pop old end, push new end (reuse). Otherwise just push (new room).
At the end, heap size = rooms used = minimum rooms needed.
Time O(n log n) — sort plus n heap operations. Space O(n) for the heap."
🔮 Likely Follow-up Questions
  • "Also output which meetings go in each room?" → Track a room ID alongside each end time in the heap
  • "Solve it without a heap?" → Yes — sweep line with events, shown above
  • "Meeting Rooms I?" → Sort by start, check consecutive pairs for overlap — just return True/False
  • "What if meetings have priorities?" → Max-heap for priorities, min-heap for scheduling — two heaps

7 Summary & Key Takeaways ▾
Sort by start time first — greedy processes meetings in arrival order
Min-heap of end times gives O(log n) access to "earliest-freeing room"
Pop + Push = reuse room; Push only = new room — one clean loop handles both
Final heap size = answer (rooms in use at the peak)
Boundary: heap[0] <= start allows same-time reuse (end == start is not overlap)
