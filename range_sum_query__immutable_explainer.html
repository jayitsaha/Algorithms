LeetCode #303

Range Sum Query - Immutable

Given an integer array nums, handle multiple sumRange(left, right) queries returning the sum of elements between indices left and right (inclusive).

Easy Prefix Sum Array Time: O(1) query Space: O(n)
1 Understanding the Problem ▾

We have a fixed array (immutable — it never changes after initialization) and we will be asked to answer many range sum queries: "what is the sum of elements from index left to index right?"

The keyword is multiple queries. If there were just one query, summing up elements between left and right takes O(n) time — fine. But with thousands of queries, doing O(n) work per query becomes O(n × Q) total, which is unacceptably slow for large inputs.

💡 Key Insight

Pay a one-time O(n) preprocessing cost to build a prefix sum array. Then every query is answered in O(1) with a single subtraction: prefix[right+1] − prefix[left]. The array never changes (immutable), so this precomputed data stays valid forever.

Edge cases to consider: single-element query where left == right (formula still works: prefix[left+1] − prefix[left] = nums[left]), negative numbers (subtraction handles them automatically), and the full-array query (left=0, right=n-1).

2 The Algorithm: Prefix Sum (Cumulative Running Total) ▾

The idea is to precompute a running total of all elements up to each position. This cumulative array — the prefix sum array — encodes all possible left-endpoint sums, letting us cancel them out at query time.

1 Build prefix array (constructor, once): Allocate prefix[0..n] with prefix[0] = 0 as a sentinel. Then for each index i: prefix[i+1] = prefix[i] + nums[i]. After this loop, prefix[k] stores the sum of nums[0] through nums[k-1].
2 Answer each query in O(1): For sumRange(left, right), return prefix[right+1] − prefix[left]. This equals sum(0..right) minus sum(0..left-1), leaving exactly nums[left..right]. One subtraction, no loop.
3 Why size n+1 with sentinel? The sentinel prefix[0] = 0 handles left=0 uniformly. Without it, we would need an if-branch: "if left==0, return prefix[right+1], else return prefix[right+1] − prefix[left]". The sentinel eliminates this special case cleanly.
📐 The Core Formula
sumRange(L, R) = prefix[R+1] − prefix[L]
where prefix[k] = nums[0] + nums[1] + … + nums[k−1], and prefix[0] = 0
🔑 Why This Works: Cancellation of Shared Prefix

Both prefix[R+1] and prefix[L] share the sum of nums[0..L-1]. Subtracting cancels that shared portion, leaving only nums[L..R]. It is the same technique as computing averages from a running total — a pattern that appears everywhere in statistics, signal processing, and database query optimization.

3 Why It Works: The Invariant ▾
🔐 The Invariant

After construction, prefix[k] stores the sum of exactly the first k elements (indices 0 through k-1). Equivalently: prefix[k] = nums[0] + nums[1] + … + nums[k-1], for all k from 0 to n. This invariant is established during construction and never violated (the array is immutable).

Algebraic proof of the query formula:

prefix[R+1] − prefix[L]
= (nums[0]+…+nums[R]) − (nums[0]+…+nums[L-1])
= nums[L] + nums[L+1] + … + nums[R] ← shared prefix cancels out

The sentinel prefix[0] = 0 is a deliberate base case. When L=0, the formula becomes prefix[R+1] − prefix[0] = prefix[R+1] − 0, which is just prefix[R+1] — correct. Without the sentinel you would need to handle L=0 separately, making the code messier.

Negative numbers: The invariant holds regardless of sign. prefix stores the actual sum (which can be negative), and subtraction of two potentially negative numbers still gives the correct difference. No special handling needed.

⚠️ Off-by-one Is the #1 Trap Here

Two common mistakes: (1) Wrong formula — writing prefix[right] − prefix[left] instead of prefix[right+1] − prefix[left] will exclude nums[right] from the sum. (2) Wrong array size — using a size-n prefix array (without the +1) and then indexing prefix[right+1] will go out of bounds for right = n-1. Always use size n+1 and the formula prefix[right+1] − prefix[left].

4 The Code (Annotated) ▾

Solution 1 — Prefix Sum (Interview Pick ★) — O(n) init, O(1) query, O(n) space

class NumArray:
def __init__(self, nums: list[int]):
n = len(nums) ← number of elements
self.prefix = [0] * (n + 1) ← size n+1; prefix[0]=0 is the sentinel
for i in range(n): ← build cumulative sums, left to right
self.prefix[i+1] = self.prefix[i] + nums[i] ← running total
def sumRange(self, left: int, right: int) -> int:
return self.prefix[right+1] - self.prefix[left] ← O(1): one subtraction
⚠️ Why prefix[right+1] and not prefix[right]?

prefix[k] = sum of indices 0..k-1 (the first k elements). So prefix[right+1] = sum of indices 0..right, which includes nums[right]. Using prefix[right] would give sum of 0..right-1, incorrectly excluding the rightmost element. The +1 is load-bearing.

Solution 2 — Brute Force (O(1) init, O(n) per query)

class NumArray:
def __init__(self, nums):
self.nums = nums ← just store the array, no preprocessing
def sumRange(self, left, right):
return sum(self.nums[left:right+1]) ← O(n) per query — too slow for many calls
💡 Complexity Comparison

Brute Force: Init O(1) · Query O(n) · Total for Q queries: O(nQ)
Prefix Sum: Init O(n) · Query O(1) · Total for Q queries: O(n + Q)
With n=10,000 and Q=10,000: brute force = 108 ops; prefix sum = 20,000 ops — a 5,000x speedup.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Spot the "Multiple Queries" Signal

The problem says "handle multiple queries." That is the trigger for prefix sum thinking. Brute-force per query is O(n) and fine for one query, but with Q queries you get O(nQ). Ask: "How many queries are expected?" — this confirms you need preprocessing and shows you think about real-world scale.

Step 2: State the Trade-Off Explicitly

"I can trade O(n) space and O(n) one-time preprocessing to make every subsequent query O(1). Since the array is immutable, the prefix sum is computed once and remains valid forever — there's no invalidation problem." Mentioning immutability shows you understand why this approach is safe here.

Step 3: Derive the Formula Live

Do not just recall the formula — derive it on the whiteboard. "I want sum(L..R). If I know sum(0..R) and sum(0..L-1), I subtract them and the shared prefix cancels. So I store prefix[k] = sum of first k elements. Then sumRange(L,R) = prefix[R+1] − prefix[L]." Derivation is more impressive than memorization.

Step 4: Call Out the Sentinel Choice

"I'll use a size-(n+1) array with prefix[0]=0. This means the formula works uniformly even when L=0, avoiding a special-case branch." This is a small but telling sign of attention to clean API design and edge-case awareness.

🗣️ What to Say to the Interviewer

"The problem mentions multiple queries on an immutable array — classic prefix sum setup.
I'll precompute a prefix array where prefix[k] = sum of nums[0..k-1].
Then sumRange(L, R) = prefix[R+1] - prefix[L]. That's O(1) per query after O(n) init.
I'll use size n+1 with prefix[0]=0 as a sentinel — handles L=0 without special cases.
Space is O(n) for the prefix array, which is acceptable since input itself is O(n)."

7 Summary & Key Takeaways ▾
Prefix sum = precomputed running totals — pay O(n) once, answer each query in O(1)
Core formula: sumRange(L, R) = prefix[R+1] − prefix[L]
Use a size n+1 array with prefix[0]=0 sentinel — avoids special-case for L=0
Only works directly on immutable arrays; mutable arrays need Fenwick Tree or Segment Tree
The prefix sum pattern generalizes to 2D, XOR, products, modular arithmetic, and difference arrays

🎯 When to Recognize the Prefix Sum Pattern
  • "Sum of elements in a range?" → Prefix sum
  • "Multiple queries on fixed data?" → Preprocess with prefix sum
  • "Count subarrays with target sum?" → Prefix sum + hash map
  • "Range updates, single-point queries?" → Difference array (inverse of prefix sum)
  • "Running statistics (running average, running XOR)?" → Prefix variant
