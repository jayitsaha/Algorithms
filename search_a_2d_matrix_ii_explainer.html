LeetCode #240

Search a 2D Matrix II

Search for a value in an m×n matrix where each row and each column is sorted in ascending order.

Medium Binary Search Start from Corner Time: O(m+n) Space: O(1)
1 Understanding the Problem ▾

You are given an m × n integer matrix with two special properties: every row is sorted left-to-right in ascending order, and every column is sorted top-to-bottom in ascending order. Given a target, determine whether it exists in the matrix.

💡 Key Insight

The top-right corner is special: it is simultaneously the maximum of its row and the minimum of its column. This means at every comparison we can eliminate an entire row OR an entire column — no wasted comparisons, no ambiguity.

✓ What we can exploit

Each row: 1 < 4 < 7 < 11 < 15
Each col: 1 < 2 < 3 < 10 < 18
Top-right = row max & col min
One comparison → one elimination
O(m+n) total comparisons max

✗ What doesn't work

Brute force: O(m×n) — too slow
Binary search per row: O(m log n)
Flatten to 1D: rows don't chain!
Top-left: both dirs increase (ambig)
Need to use BOTH sort properties

Edge cases to consider: empty matrix (m=0 or n=0), 1×1 matrix, target smaller than smallest element (matrix[0][0]), target larger than largest (matrix[m-1][n-1]), target present multiple times (duplicates are fine — correctness holds).

2 The Algorithm: Walk from Top-Right Corner ▾

We start at the top-right corner (row 0, col n−1) and walk towards the target using the sorted structure to eliminate one row or column per step. The corner is the unique position where we can make a decisive, unambiguous binary choice at every step.

1 Initialize: Set row = 0, col = n−1. This is the top-right corner — the largest value in row 0 and the smallest value in column n−1.
2 Loop while in bounds (row < m and col >= 0): Read curr = matrix[row][col]. If curr == target, return True immediately.
3 If curr > target: The target cannot be in this column — every cell below is even larger (col is sorted top-down). Eliminate column: col -= 1 (move left).
4 If curr < target: The target cannot be in this row — every cell to the left is even smaller (row is sorted left-right). Eliminate row: row += 1 (move down).
5 If we exit the loop (out of bounds), the target does not exist. Return False.
Decision Tree at Each Step

Starting at top-right corner: max of row 0, min of column n−1

← col−−
curr > target
eliminate entire column
↓ row++
curr < target
eliminate entire row

Each step moves strictly left or strictly down — we never revisit a cell. At most m+n−1 steps.

🔑 Why Top-Right? Why Not Top-Left?

At top-left (matrix[0][0]): if curr < target, you could go right OR down — no clear choice, must explore both. At top-right: curr > target means only col decreases can help (rows below are larger); curr < target means only row increases can help (columns left are smaller). A clean binary decision every single time. Bottom-left is equally valid by symmetry.

3 Why It Works: The Invariant ▾
🔐 The Invariant

Throughout the search, the following is always true: if the target exists in the matrix, it must be in the sub-rectangle spanning rows [row..m−1] and columns [0..col]. Every cell outside this region has been definitively ruled out. We never eliminate a cell that could contain the target.

Proof that "curr > target → eliminate column" is safe: Column is sorted top-to-bottom, so matrix[row][col] ≤ matrix[row+1][col] ≤ ... ≤ matrix[m-1][col]. If curr > target, then all cells at or below [row][col] in column col are ≥ curr > target. None can be the target. Column col can be discarded entirely.

Proof that "curr < target → eliminate row" is safe: Row is sorted left-to-right, so matrix[row][0] ≤ matrix[row][1] ≤ ... ≤ matrix[row][col]. If curr < target, all cells at or left of [row][col] in row row are ≤ curr < target. None can be the target. Row row can be discarded entirely.

⚠️ This is NOT the same as LeetCode #74

In #74 (Search a 2D Matrix), rows chain — the first element of row i+1 is larger than the last of row i. That allows treating the matrix as a flat sorted array and doing binary search in O(log(m×n)). In #240, this chaining does NOT hold — rows are independent. The corner-walk approach is needed. Do not confuse these two problems in an interview.

Complexity proof: Each step either decrements col or increments row. Col starts at n−1 and can decrease at most n−1 times. Row starts at 0 and can increase at most m−1 times. Total steps ≤ (m−1) + (n−1) = m+n−2. Therefore time complexity is O(m+n). Space is O(1) — only two pointer variables.

4 The Code (Annotated) ▾

Solution 1 — Corner Walk (Interview Pick, O(m+n) time, O(1) space)

def searchMatrix(matrix, target):
m, n = len(matrix), len(matrix[0]) ← dimensions: m rows, n columns
row, col = 0, n - 1 ← start at top-right corner
while row < m and col >= 0: ← both bounds must hold (AND, not OR)
curr = matrix[row][col] ← value at current pivot
if curr == target: ← exact match
return True
elif curr > target: ← pivot too large → kill this column
col -= 1 ← move left
else: ← pivot too small → kill this row
row += 1 ← move down
return False ← exhausted all candidates
⚠️ Critical: AND not OR in while condition

We need row < m AND col >= 0. Each elimination moves one pointer toward its boundary. Moving left (col−−) can push col below 0. Moving down (row++) can push row past m−1. We stop as soon as either goes out of range — using AND keeps looping only while BOTH are valid. If we used OR, we'd read out-of-bounds memory.

Solution 2 — Binary Search per Row (O(m log n), simpler to derive)

import bisect
def searchMatrix(matrix, target):
for row in matrix: ← each row is independently sorted
pos = bisect.bisect_left(row, target) ← O(log n) binary search in this row
if pos < len(row) and row[pos] == target:← bounds check then value check
return True
return False ← target not in any row
💡 Complexity Summary

Corner walk (optimal): Time O(m+n), Space O(1). At most m+n−2 comparisons.
Binary search per row: Time O(m log n), Space O(1). Better when n >> m and matrix is very wide.
Brute force scan: Time O(m×n) — mention only to then immediately optimize.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Double-Sort Structure

The keyword signal is "each row AND each column sorted." This is distinct from #74 where the whole matrix is sorted row-major. Here, rows are independent. Confirm: "Are rows chained? Can I do a flat binary search?" — answer is no for this problem.

Step 2: Start with the Simpler Approach

"Each row is sorted, so I can binary search each row — O(m log n). Correct, but can I also use column sorting?" Proposing this shows structured thinking. Only then offer the corner insight.

Step 3: Ask Which Cell Connects Both Sortings

"To use both row and column sorting, I need a position where one direction uses row-order and the other uses column-order with opposite directions. The top-right corner gives exactly that: going left stays in the same row (row-sort), going down stays in the same column (col-sort)."

Step 4: State the Elimination Argument

"At top-right: if curr > target, everything in this column below is larger — I eliminate the entire column in one comparison. If curr < target, everything left in this row is smaller — I eliminate the entire row. At most m+n comparisons. O(m+n) time."

🗣️ What to Say to the Interviewer

"I'll use the sorted structure by starting at the top-right corner.
If curr == target, done. If curr > target, target can't be in this column — move left.
If curr < target, target can't be in this row — move down.
Each step eliminates one row or one column — at most m+n steps. O(m+n) time, O(1) space."
🔮 Likely Follow-up Questions
  • "What if the matrix is fully sorted row-major?" → Treat as flat array, binary search in O(log(m×n)) — LeetCode #74
  • "Why can't you do better than O(m+n)?" → Information-theoretic lower bound for this structure — can be proven formally
  • "What if you start from bottom-left?" → Equally valid: left decreases, up decreases too — mirror of top-right logic
  • "Can this handle duplicate values?" → Yes — duplicates slow practical performance but don't break the elimination logic
