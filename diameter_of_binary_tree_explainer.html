LeetCode #543

Diameter of Binary Tree

Given the root of a binary tree, return the length of the diameter — the longest path between any two nodes (measured in edges, not nodes). The path need not pass through the root.

Easy Trees DFS Postorder Time: O(n) Space: O(h)
1 Understanding the Problem ▾

The diameter of a binary tree is the length of the longest path between any two nodes. The path does not need to pass through the root — it can start and end anywhere. Length is counted in edges, not nodes.

💡 Key Insight

Every path in a binary tree has a highest node — the node where the path changes direction from going left-down to going right-down (or one endpoint is the ancestor of the other). At that highest node, the path length equals left subtree depth + right subtree depth. The answer is the maximum of this across every node in the tree.

Edge cases: Empty tree → 0. Single node (no edges) → 0. Skewed chain of n nodes → n−1 edges. A balanced tree where the longest path is entirely in one subtree.

2 The Algorithm: Postorder DFS with Global Max ▾

We use a postorder DFS — process left child, then right child, then the current node. This ensures we have both children's depths before computing the candidate diameter at the current node. We maintain a global diameter variable that tracks the maximum path length seen so far.

1 Base case: dfs(None) returns 0. A null node contributes zero depth to any path.
2 Recurse left: L = dfs(node.left) — get the maximum depth reachable going into the left subtree.
3 Recurse right: R = dfs(node.right) — get the maximum depth reachable going into the right subtree.
4 Update global max: diameter = max(diameter, L + R). The path through this node has L + R edges.
5 Return depth to parent: return 1 + max(L, R). The deepest path from this node downward is 1 edge to the better child, plus that child's depth. We take max (not sum) because a path going up to the parent can only go through one side.
🔑 Two Different Values Per Node

The value returned (1 + max(L,R)) and the value used to update the answer (L + R) are different. The return value is the longest single arm downward. The update value is the longest path through this node, which uses both arms. This distinction is the heart of the algorithm.

3 Why It Works: The Invariant ▾
🔐 The Invariant

After dfs(node) returns, the returned value equals the maximum number of edges on any downward path starting from node. The global diameter variable holds the longest path seen across all nodes visited so far.

Why every path is captured: Any path in a binary tree has a unique highest node (the "turning point"). We call dfs on every node, so every possible turning point is evaluated. At each turning point, we compute L + R, which is exactly the length of the longest path passing through it as its highest node.

Why 1 + max(L, R) is the correct depth: From any node, the furthest you can reach in a single direction is 1 edge to the best child, plus whatever that child can reach. The 1 counts the edge from current node to child. We cannot use L + R here because a parent can only approach this node from one direction — it can then continue to one side, not both.

⚠️ Classic Mistake: Only Checking the Root

Returning depth(root.left) + depth(root.right) is a common bug. This only evaluates the path through the root. If the longest path lies entirely in a subtree (e.g., a left subtree that is a long zigzag), this approach misses it. You must check every node.

Complexity: Each node is visited exactly once in the DFS — O(n) time. The recursion stack uses O(h) space where h is the tree height (O(log n) balanced, O(n) worst-case skewed).

⚠️ Python Scoping: Why self.diameter?

Inside a nested function in Python, assigning to a bare diameter variable creates a new local variable, hiding the outer one. Two clean solutions: (1) use self.diameter on the Solution object, or (2) declare nonlocal diameter if using a list or mutable wrapper. Both are acceptable in interviews — self.diameter is most common.

💡 Complexity Summary

Postorder DFS (Sol 1): Time O(n) — one pass. Space O(h) call stack.
Brute Force (Sol 2): Time O(n²) — depth() called once per node from every ancestor. Space O(h).
Iterative postorder: Time O(n), Space O(n) — uses explicit stack; eliminates recursion depth risk.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Pattern

"Longest path between two nodes in a tree" → every path has a highest node. This means I need to evaluate every node as a potential turning point. DFS on the whole tree is the right tool.

Step 2: Propose Brute Force

"A naive approach: for every node, separately compute its left and right subtree depths and sum them. Recurse on subtrees for candidates there too. This calls depth() repeatedly — O(n²) time."

Step 3: Identify the Inefficiency

"The wasteful part is computing depth separately. If my DFS function both returns the depth AND updates the global max in a single traversal, I compute each node's depth exactly once — O(n)."

Step 4: State the Two-Value Trick

"The key insight: the value I return (depth = 1+max(L,R)) is different from the value I use to update the answer (candidate = L+R). The depth goes up the recursion; the candidate updates the global max."

🗣️ What to Say to the Interviewer

"At each node, the longest path through it equals left-depth + right-depth.
My DFS returns the node's depth to its parent, but also updates a global max
with left + right at each call. One pass = O(n). I'll use self.diameter to share
state across recursive calls. Base case: null node → depth 0."
🔮 Likely Follow-up Questions
  • "What if recursion overflows on very deep trees?" → Use iterative postorder with an explicit stack.
  • "Return the actual path, not just the length?" → Track parent pointers; when updating diameter, record the two endpoints.
  • "N-ary tree?" → Same idea: sort children depths, top two are the best arms. The rest is identical.
  • "Longest path in a general graph?" → NP-hard in general. With a DAG it's solvable via topological sort + DP.

7 Summary & Key Takeaways ▾
The diameter at any node = left_depth + right_depth. Check every node as a potential turning point.
Postorder DFS (left → right → node) naturally gives us children's answers before we process the parent.
Return 1 + max(L, R) to parent (single arm), but use L + R to update the global max (full path through here).
Use self.diameter or nonlocal to share the global max across recursive calls in Python.
Time O(n) — one visit per node. Space O(h) — proportional to tree height for the call stack.
🎯 When to Use This Pattern
  • Any "longest path" or "maximum path sum" problem in a binary tree
  • When the path is allowed to change direction at some internal node
  • When the DFS must return a different value (depth) than what it contributes to the answer (path length)
  • General signal: "compute a global maximum from local, bottom-up information in a tree"
