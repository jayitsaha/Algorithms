LeetCode #1048

Longest String Chain

Find the longest chain of words where each word differs from the next by exactly one inserted character.

Medium Dynamic Programming Sort by Length + DP Time: O(n·L²) Space: O(n·L)
1 Understanding the Problem ▾

You're given a list of words. Word A is a predecessor of word B if you can insert exactly one letter anywhere in A (including start or end) to get B. A string chain is a sequence w₁, w₂, …, wₖ where each word is a predecessor of the next. We want the longest such chain possible using words from our list.

💡 Key Insight: Flip the Direction

Instead of asking "what letter can I add to A to get B?", flip it: ask "what single letter can I remove from B to get A?" For a word of length L, there are exactly L possible single-deletions to try. If any gives a word that exists in our input, we found a chain link.

Critical constraint: Two words can only be chain neighbors if their lengths differ by exactly 1. Words of the same length can never be adjacent in a chain. This means sorting by length is a natural first step — predecessors are always shorter.

🔐 Why This is Dynamic Programming

Optimal substructure: The longest chain ending at word W equals 1 + the longest chain ending at W's best predecessor. We can express the answer to a bigger problem in terms of answers to smaller problems.

Overlapping subproblems: Multiple longer words might share the same predecessor (e.g., "bca" and "bda" both have "ba" as a predecessor). We compute dp["ba"] once and reuse it for both — this is the memoization payoff.

2 The Algorithm: Sort by Length + Bottom-Up DP ▾

We maintain a hash map dp where dp[word] = length of the longest chain ending at that word. We process words in increasing length order (bottom-up), guaranteeing all predecessors are computed before we need them.

1 Sort words by length (ascending). Predecessors are always shorter, so this gives us topological order — we always compute shorter words before longer ones that depend on them.
2 Initialize dp = {} and best = 1. Each word will get a DP value. The global answer is at least 1 (every single word is a chain of length 1).
3 For each word: set dp[word] = 1 as the base case. Then generate all L possible predecessors by deleting one character at each position.
4 For each deletion candidate prev: if prev exists in our dp map, update dp[word] = max(dp[word], dp[prev] + 1).
5 Track the global maximum after processing each word. Return it as the final answer.

⚠️ Why Sort BEFORE Running DP?

If we process words in random order, we might reach word "bca" before word "ba" — meaning dp["ba"] wouldn't exist yet, and we'd incorrectly compute dp["bca"] = 1. Sorting ensures the dependency graph is traversed in topological order (no cycles, no stale lookups).

3 Why It Works: The Invariant ▾
🔐 The DP Invariant

When we compute dp[word], for every word shorter than word that appears in our input and is a valid predecessor, its dp value is already correct and final. Sorting by length guarantees this.

Why deletion = insertion (symmetry): "A is a predecessor of B" means insert one letter into A to get B. Flip it: delete one letter from B to get A. These are the same relationship viewed from different ends. By trying all |B| single deletions from B, we enumerate every possible predecessor of B.

Why the dependency graph is a DAG: Since a predecessor is always strictly shorter, the dp values form a dependency tree pointing from longer words to shorter words. No cycles are possible (a word can't be its own predecessor). Processing in length order is traversing this DAG in topological order.

Correctness of the max: We take max over all valid predecessors because the chain can extend from any of them. We want the longest chain ending at word, so we pick the predecessor with the longest chain and extend it by 1.

⚠️ Common Mistake: Sorting Doesn't Mean All Inputs Are Usable

After sorting, not every shorter word in the list is a predecessor of every longer word. Only specific ones where exactly one character deletion matches. The if prev in dp check filters this correctly — only words that both exist in our input AND have been computed get used.

Edge case — word not in dp: If prev = word[:i]+word[i+1:] is not in dp, it either (a) doesn't exist in our input, or (b) hasn't been computed yet. Case (b) cannot happen since shorter words are always processed first. So not in dp safely means "this predecessor doesn't exist as an input word."

⚠️ "if prev in dp" vs "if prev in word_set"

Using if prev in dp is elegant: the dp map only contains words we've already processed (all shorter words at this point). Using word_set would also work, but you'd need dp.get(prev, 0) as a safety fallback — using dp directly is cleaner and self-documenting.

🔐 Complexity Justification

Time O(n·L²): For each of n words (max length L), we generate L candidate predecessors. Each predecessor takes O(L) time to create via string slicing. Total: O(n × L × L) = O(n·L²). With L ≤ 16 per problem constraints, this is extremely fast even for large n.

Space O(n·L): The dp hash map stores n words as keys, each up to L characters long. Hashing a string takes O(L). Total map memory: O(n·L).

5 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Longest-Chain DP Signal

The phrase "longest chain" is a major signal for DP. Think of Longest Increasing Subsequence (LIS): we're looking for the longest sequence where each element "extends" from a previous one. Here "extends" means "insert one character." Same shape, different domain.

Step 2: Identify the Dependency Direction

Each word depends on shorter words (its predecessors). This means if we sort by length and process shortest-first, we always have the information we need when we need it. This is exactly bottom-up DP.

Step 3: Flip from Insertion to Deletion

The "insert one letter" check is expensive to enumerate (26 letters × L+1 positions = 26L candidates per pair). Instead, deletion gives exactly L candidates per word. For each, string-slice to get the (L-1)-character candidate. O(L²) per word — much better.

Step 4: Code the Recurrence

dp[word] = 1 + max(dp[prev]) over all valid predecessors. Use a hash map not an array (words are strings). Track global max as you go. Total time O(n·L²), space O(n·L).

What to Say to the Interviewer

"I recognize this as a longest-chain DP — same structure as LIS. Each word's chain length depends only on shorter words (predecessors). So I'll sort by length to process dependencies first (bottom-up order). For each word, I try removing each character one at a time — this gives me all possible predecessors in O(L²). I check if each predecessor exists in our dp map and take the max. Time: O(n·L²), Space: O(n·L). With L≤16 per constraints, this is fast."
💡 Connection to LIS (Longest Increasing Subsequence)

In LIS: sort numbers, for each number find all smaller numbers that could precede it, take max chain + 1.
Here: sort words by length, for each word find all shorter words that could precede it (via deletion), take max chain + 1. The DP shape is identical — only the "can X precede Y?" check differs.

⚠️ Follow-Up Questions to Expect
  • Can you reconstruct the actual chain (not just the length)? — Store predecessor pointers in dp.
  • What if words can have length differences > 1? — The problem changes; multi-hop chains become relevant.
  • What if the input is huge? — Discuss parallelizing by length buckets or external sorting.

7 Summary & Key Takeaways ▾
Sort by length first — this gives topological order for the DP (shorter words computed before longer words that depend on them)
Flip "insert" to "delete" — trying all L single-char deletions from a word enumerates all its possible predecessors in O(L²)
Use a hash map dp (word → chain length), not an array — keys are strings, and hash maps give O(L) lookup (hashing takes O(L))
This is structurally identical to LIS (Longest Increasing Subsequence) — sort + "extend from best predecessor" is the universal longest-chain DP template
With L ≤ 16 per constraints, O(n·L²) ≈ O(256n) — effectively linear in n; works comfortably even for n = 1000
🎯 When to Recognize This Pattern (Signals)
  • Problem asks for "longest chain/path/sequence" where each element extends from a previous one
  • Each element has a natural "size" or "rank" (string length, number value) defining dependency direction
  • Checking "can X precede Y?" is O(polynomial) but not O(1) — the hash map DP avoids recomputing
  • Multiple elements may share the same predecessor — this is the overlapping subproblems that justify DP
