LeetCode #643

Maximum Average Subarray I

Find a contiguous subarray of length k with the maximum average value, and return that average.

Easy Sliding Window Maintain Window Sum Time: O(n) Space: O(1)
1 Understanding the Problem ▾

You are given an integer array nums and an integer k. Find the contiguous subarray of length exactly k that has the maximum average, and return that average as a float.

💡 Key Insight

Since the window size k is fixed, maximizing the average is identical to maximizing the sum — you only need to divide by k once at the very end. This unlocks a powerful optimization: you never recompute a window from scratch. Instead you slide it: subtract the element leaving on the left, add the element entering on the right.

✓ Example: nums=[1,12,−5,−6,50,3], k=4

Window [0..3]: 1+12−5−6 = 2 → avg 0.50
Window [1..4]: 12−5−6+50 = 51 → avg 12.75
Window [2..5]: −5−6+50+3 = 42 → avg 10.50
Answer: 12.75 (window indices 1–4)

✗ Naive: recompute each window

For each start i in 0..n-k:
  sum = nums[i]+...+nums[i+k-1]
O(k) per window × O(n) windows
= O(n·k) — wasteful overlap!

Edge Cases: k = 1 (single-element max), k = n (only one possible window — the entire array), all elements are negative (result is the "least negative" window), duplicate maximum sums (either index is acceptable).

2 The Algorithm: Fixed-Size Sliding Window ▾

The sliding window technique exploits the fact that two consecutive windows of size k share k−1 elements. Instead of re-summing those shared elements, we carry the sum forward and update it with a single addition and subtraction — O(1) per step instead of O(k).

Window 1 (i=0..3)
[1, 12, −5, −6]
sum = 2
Window 2 (i=1..4) ← Best
[12, −5, −6, 50]
sum = 51 → avg 12.75
Window 3 (i=2..5)
[−5, −6, 50, 3]
sum = 42
1 First window: Compute window_sum = sum(nums[0:k]). Set max_sum = window_sum. This is O(k) and only happens once.
2 Slide right: For each i from k to n−1: window_sum += nums[i] − nums[i−k]. nums[i] is the new element entering from the right; nums[i−k] is the element leaving from the left. One O(1) update per step.
3 Track best: After each update, max_sum = max(max_sum, window_sum). We compare sums — not averages — since k is the same for every window.
4 Return: return max_sum / k. Divide by k exactly once. All n−k+1 windows have been checked in O(n) total time.
🔑 The Sliding Mechanism

When the window moves one step right: the right edge moves forward (include nums[i]) and the left edge moves forward (exclude nums[i−k]). The outgoing index is always exactly k positions behind the incoming index: outgoing = i − k.

3 Why It Works: The Invariant ▾
🔐 The Invariant

After each iteration with right-edge index i, window_sum equals exactly nums[i−k+1] + nums[i−k+2] + … + nums[i] — the sum of the k elements ending at position i. This invariant is maintained by the update rule: add the incoming right element, subtract the outgoing left element.

Algebraic proof of the update rule:

old_sum = nums[i−k] + nums[i−k+1] + … + nums[i−1]
new_sum = nums[i−k+1] + nums[i−k+2] + … + nums[i]
new_sum = old_sum − nums[i−k] + nums[i] ✓

Completeness guarantee: The loop runs from i=k to i=n−1, giving us n−k iterations. Together with the first window (indices 0 to k−1), we have examined all n−k+1 possible windows. No window is skipped. Therefore, the maximum is guaranteed.

⚠️ Off-by-one: Which element leaves?

When the right edge is at index i, the window covers [i−k+1, i]. The element that just left is at index i−k — one step further left than the new window's left edge. If you accidentally use nums[i−k+1] (the new window's leftmost), you'll silently corrupt the sum and get wrong answers on every test case after the first window.

Why compare sums, not averages? For two windows A and B of the same size k: avg(A) > avg(B) if and only if sum(A) > sum(B). Division by k is a monotone function — it preserves ranking. Comparing integer (or float) sums avoids repeated floating-point division, which is both faster and more numerically stable.

4 The Code (Annotated) ▾

Solution 1 — Fixed Sliding Window (Optimal) — Interview Pick

def findMaxAverage(nums: list, k: int) -> float:
window_sum = sum(nums[:k]) ← O(k) setup: sum the first window [0..k-1]
max_sum = window_sum ← Initialize best sum to the first window's sum
for i in range(k, len(nums)): ← i is the new right edge entering the window
window_sum += nums[i] - nums[i-k]← Slide: add incoming [i], remove outgoing [i-k]
max_sum = max(max_sum, window_sum)← Did this window beat our best? Update if so
return max_sum / k ← Divide once here — never inside the loop
⚠️ Why not divide by k inside the loop?

Dividing by k at each step works but is unnecessary because dividing by a positive constant preserves order. More importantly, integer addition/subtraction is exact; repeated floating-point division accumulates rounding error. Divide once at the end for correctness and clarity.

Solution 2 — Brute Force (O(n·k)) for comparison

def findMaxAverage_brute(nums: list, k: int) -> float:
max_avg = float('-inf') ← Handles all-negative inputs gracefully
for i in range(len(nums) - k + 1): ← Every valid window start position
avg = sum(nums[i:i+k]) / k ← Recomputes k-sum from scratch — O(k) work per window
max_avg = max(max_avg, avg) ← Track best average seen
return max_avg ← O(n·k) total — fine for small inputs, TLE for large
💡 Complexity Summary

Sliding Window: Time O(n) — O(k) one-time setup + O(1) per slide × n−k slides. Space O(1).
Brute Force: Time O(n·k) — O(k) sum per window × n−k+1 windows. Space O(k) for the slice.
For n=100,000 and k=50,000: sliding window ≈ 100,000 ops; brute force ≈ 5,000,000,000 ops.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Spot the Pattern Signals

"Contiguous subarray" + "of length k" (fixed!) + "maximum" = textbook fixed-size sliding window. The fixed-size constraint is the differentiator: if k could vary, you'd need a variable window. Since it's fixed, the left pointer tracks exactly k behind the right pointer at all times.

Step 2: Offer Brute Force First

"Simplest approach: for each starting index i, compute sum of nums[i..i+k−1] and divide by k. O(n·k) time. But adjacent windows share k−1 elements — we're re-adding them every time. There must be a smarter way."

Step 3: Derive the Optimization

"Instead of recomputing, maintain a running sum. Slide the window: when the right edge moves to i, subtract nums[i−k] (leaving the left) and add nums[i] (entering the right). One O(1) update per step — O(n) total."

Step 4: Mention the Divide-at-End Trick

"Since k is constant, maximizing the sum maximizes the average. I'll compare sums throughout and divide by k once at the end. This avoids repeated floating-point operations and is cleaner code."

🗣️ What to Say to the Interviewer

"Fixed-size window k suggests sliding window — adjacent windows share k−1 elements.
Brute force recomputes each window from scratch: O(n·k). Too slow.
Optimization: maintain a running sum. Slide: add nums[i], remove nums[i−k].
Initialize: sum first k elements → max_sum = window_sum.
Slide i from k to n−1, update window_sum and max_sum each step.
Return max_sum / k at the end — divide only once.
Time O(n), Space O(1)."
🔮 Likely Follow-up Questions
  • "What if you also need to return the starting index?" → Track best_start = i − k + 1 when max_sum updates
  • "What if k is not fixed — you want the subarray with maximum average of any length ≥ k?" → Binary search on the answer + prefix averages (#644 Maximum Average Subarray II)
  • "Maximum sum subarray of any size?" → Kadane's Algorithm (#53 — entirely different pattern)
  • "Average of every window of size k (not just max)?" → Same sliding window; store each avg in result array
