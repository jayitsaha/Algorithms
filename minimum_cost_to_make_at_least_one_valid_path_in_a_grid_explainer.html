LeetCode #1368

Minimum Cost to Make at Least One Valid Path in a Grid

Given an m×n grid with direction signs, find the minimum cost to change signs so a valid path from (0,0) to (m-1,n-1) exists.

Hard Graph 0-1 BFS Time: O(m·n) Space: O(m·n)
1 Understanding the Problem ▾

You have an m × n grid where each cell contains a direction sign — 1 (right), 2 (left), 3 (down), or 4 (up). Following these arrows as written, you may or may not reach the bottom-right corner (m-1, n-1) from the top-left (0,0). You can change any sign for a cost of 1. Find the minimum number of sign changes needed to guarantee at least one valid path.

💡 Key Insight

Following an existing sign costs 0; overriding it to go a different direction costs 1. This naturally models a weighted graph where edge weights are only 0 or 1 — exactly the sweet spot for 0-1 BFS (a deque instead of a full priority queue). We want the shortest (cheapest) path from (0,0) to (m-1,n-1) in this 0/1-edge-weight graph.

Concrete example (3×3 grid):

Grid: [[1,1,1],[2,2,2],[3,3,3]] — row 0 points right, row 1 points left, row 2 points down.

Following the arrows: (0,0)→(0,1)→(0,2)…then stuck (arrow points right, but we are at column edge). We'd need to change the (0,2) sign to go down (cost 1), then change (1,2) to go down (cost 1), and so on. The algorithm finds the optimal sequence of changes that minimises total cost.

Naive: Try all subsets of sign changes

For an m×n grid, 2^(m·n) subsets — completely infeasible even for a 5×5 grid.

Optimal: Model as 0-1 weighted graph

Each cell is a node; free edges (cost 0) follow the sign, paid edges (cost 1) go all other 3 directions. Run 0-1 BFS → O(m·n).

🔐 Formal Edge Model

From cell (r, c), there are 4 possible moves (right, left, down, up). The edge to the neighbor that the existing sign points to has weight 0; the other three neighbors each have weight 1. We seek the minimum-weight path from (0,0) to (m-1,n-1).

2 The Algorithm: 0-1 BFS ▾

0-1 BFS is a specialised shortest-path algorithm for graphs where every edge has weight 0 or 1. It uses a deque (double-ended queue): cost-0 neighbours are pushed to the front, cost-1 neighbours to the back. This maintains the deque in non-decreasing cost order — analogous to Dijkstra but O(V+E) because we never need a heap.

Cost-0 Move (Free)

Neighbour pointed to by existing sign.
→ Push to FRONT of deque.
Cost stays the same.

Cost-1 Move (Change Sign)

Any other valid neighbour.
→ Push to BACK of deque.
Cost increases by 1.

1 Initialise: dist[r][c] = ∞ for all cells; dist[0][0] = 0. Push (0, 0, 0) [cost, row, col] to deque.
2 Pop from front of deque. If cost > dist[r][c], skip (stale entry).
3 For each of the 4 directions from (r, c): compute edge cost (0 if it matches the sign, 1 otherwise). If dist[r][c] + edge_cost < dist[nr][nc], update dist and push to front (cost 0) or back (cost 1).
4 Terminate when deque is empty. Return dist[m-1][n-1].
⚠️ Why Not Dijkstra?

Dijkstra with a min-heap would give O(m·n · log(m·n)), which is fine but suboptimal. Because weights are only 0 or 1, the deque trick maintains sorted order in O(1) per operation → O(m·n) total. For m,n ≤ 100, this means ~10,000 operations vs ~130,000 for Dijkstra — 13× faster asymptotically.

🔐 Why Not Standard BFS?

Standard BFS assumes all edges have the same weight. Here weights are 0 or 1, so a plain BFS would process some nodes before their optimal cost is known. The deque trick is the fix: zero-cost edges at the front ensure we always pop the minimum-cost node next.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At any point during 0-1 BFS, the deque is sorted in non-decreasing order of cost. Specifically, it contains at most two distinct cost values — the current minimum cost k and k+1. Nodes with cost k are in the front section, nodes with cost k+1 are in the back.

Why this guarantees correctness:

  • We always pop the minimum-cost unvisited node (like Dijkstra).
  • When we push a cost-0 neighbour to the front, it still has cost k — the front remains sorted.
  • When we push a cost-1 neighbour to the back, it has cost k+1 — the back stays at k+1.
  • Once the front section (cost k) is exhausted, the back section (cost k+1) becomes the new front — this is exactly the BFS level-by-level progression.

Edge cases handled:

  • 1×1 grid: Already at destination, cost = 0.
  • All arrows already valid: Every move is cost-0, total cost = 0.
  • Every move requires a change: Cost = m + n - 2 (the Manhattan distance along the direct path).
💡 Analogy: The Traffic Sign Route

Think of navigating a city where streets all have one-way signs. If you follow the existing sign, it's free (green light). If you want to go the "wrong way," you pay a toll of $1. 0-1 BFS finds the cheapest route — always choosing free roads first before paying for direction changes.

4 The Code (Annotated) ▾
from collections import deque// deque supports O(1) appendleft and append
def minCost(grid):// main function
m, n = len(grid), len(grid[0])// grid dimensions
# dir: sign -> (dr, dc)
DIRS = [(0,1),(0,-1),(1,0),(-1,0)]// right, left, down, up (1-indexed in grid)
dist = [[float('inf')]*n for _ in range(m)]// min cost to reach each cell, start at ∞
dist[0][0] = 0// source cell has cost 0
dq = deque([(0, 0, 0)]) # (cost, row, col)// initialise deque with source
while dq:// process until deque empty
cost, r, c = dq.popleft()// pop cheapest node (front)
if cost > dist[r][c]:// stale entry: a cheaper path was found already
continue// skip — do not reprocess
for d, (dr, dc) in enumerate(DIRS):// try all 4 neighbours
nr, nc = r + dr, c + dc// neighbour coordinates
if 0 <= nr < m and 0 <= nc < n:// bounds check
# grid sign is 1-indexed: sign d+1 points to DIRS[d]
edge = 0 if grid[r][c] == d+1 else 1// free if sign matches, else cost 1
new_cost = cost + edge// total cost to reach neighbour
if new_cost < dist[nr][nc]:// found cheaper path to neighbour
dist[nr][nc] = new_cost// update dist table
if edge == 0:// free move → front of deque (keep cost sorted)
dq.appendleft((new_cost, nr, nc))
else:// paid move → back of deque
dq.append((new_cost, nr, nc))
return dist[m-1][n-1]// answer: min cost to reach bottom-right
⚠️ Common Mistake: 1-indexed Signs

The problem uses signs 1, 2, 3, 4 (not 0-indexed). Direction d in DIRS corresponds to sign value d+1. Off-by-one here means every move costs 1 when it should cost 0 — the algorithm still finds a path but with inflated cost. Always double-check the sign-to-direction mapping.

🔐 Stale Entry Check

Because a node can be added to the deque multiple times (with different costs), we use if cost > dist[r][c]: continue to skip outdated entries. This is the same "lazy deletion" trick used in Dijkstra — it avoids maintaining a complex visited set.

Complexity:

  • Time: O(m·n) — each cell is processed at most once, and each edge examined at most twice.
  • Space: O(m·n) — for the dist table and deque (at most m·n entries).
5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognise "minimum changes = minimum cost"

Any time you see "minimum changes to achieve a goal," think of it as a cost-minimisation problem on a graph. The question is: what does a node represent? What does an edge weight represent?

Step 2: Model as a graph with 0/1 edge weights

Cells are nodes. From each cell there are 4 edges — one for each direction. The edge to the cell the current sign points to costs 0 (free); all other edges cost 1 (we'd have to change the sign). This is the crucial modelling insight.

Step 3: Choose the right shortest-path algorithm

Options: BFS (all equal weights — no), Dijkstra (general weights — yes but O(n log n)), 0-1 BFS (0/1 weights only — yes and O(n)). Since weights are only 0 or 1, use 0-1 BFS with a deque.

Step 4: Implement and handle the 1-indexed signs

DIRS = [(0,1),(0,-1),(1,0),(-1,0)]. Sign value d+1 corresponds to DIRS[d]. Edge cost = 0 if grid[r][c] == d+1, else 1.

🗣️ What to Say to the Interviewer

"Let me think of this as a graph problem. Each cell is a node, and from each cell I have 4 possible moves. If the existing arrow already points in that direction, moving there costs nothing — I don't need to change any sign. Otherwise, I'd be overriding the sign, which costs 1. So the edge weights are only 0 or 1. For 0/1-weighted graphs, I can use 0-1 BFS — it's essentially Dijkstra but with a deque instead of a heap, giving O(m·n) time instead of O(m·n log m·n). The algorithm: initialise dist to infinity, dist[0][0]=0. Push (0,0,0) to deque. Repeatedly pop the front, skip stale entries, and for each of 4 directions: if free, push to front; if cost 1, push to back. Return dist[m-1][n-1]. One gotcha: the signs are 1-indexed. Direction index d corresponds to sign d+1."
💡 Follow-up: What if costs are 0 to K?

If sign changes had varying costs (not just 0 or 1), you'd need Dijkstra with a min-heap. If costs were bounded integers 0 to K, you could use a K-bucket queue for O((V+E)·K). The 0-1 BFS deque trick only works when edge weights are exactly {0, 1}.

7 Summary & Key Takeaways ▾
Model the problem as a shortest-path on a 0/1-weighted graph: free edge = follow sign, paid edge = change sign.
0-1 BFS uses a deque: appendleft for cost-0 edges, append for cost-1 edges — keeps the deque sorted without a heap.
Time complexity O(m·n) beats Dijkstra's O(m·n · log(m·n)) by exploiting the {0,1} weight constraint.
Skip stale deque entries with the lazy check: if cost > dist[r][c]: continue.
Remember the 1-indexed sign encoding: sign value d+1 corresponds to direction index d in DIRS.
🎯 When to Use 0-1 BFS
  • Edge weights are exactly 0 or 1 (or can be mapped to {0,1})
  • You want shortest/cheapest path from source to destination
  • Full Dijkstra would work but is overkill — 0-1 BFS is faster and simpler
