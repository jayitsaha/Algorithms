Longest Balanced Subarray II

Given an integer array of +1s and -1s, find the length of the longest subarray whose sum equals zero — but with point updates: elements can change after the initial query.

Hard Advanced Data Structures Segment Tree Time: O((n + q) log n) Space: O(n)
1 Understanding the Problem ▾

You are given an array where every element is either +1 or -1. Your goal is to find the longest contiguous subarray with a sum of zero — meaning equal numbers of +1s and -1s.

The twist that makes this "II" (and Hard): after answering the initial query, you must also handle point updates — individual elements get flipped, and each time you must re-answer: "what is the longest balanced subarray now?"

💡 Key Insight

Map +1 → +1 and -1 → -1, build a prefix-sum array. Subarray [l, r] has sum zero iff prefix[r] == prefix[l-1]. The longest such pair of equal prefix values gives the answer. With updates, we need a data structure that can track the first occurrence of each prefix value and answer max-gap queries dynamically — that's where the Segment Tree shines.

Concrete example (static):

Array: [+1, -1, +1, +1, -1, -1]

Prefix sums (with sentinel 0 at index 0):
P = [0, 1, 0, 1, 2, 1, 0]

We look for the farthest pair of equal values in P. P[0]=0 and P[6]=0 → gap = 6. So the entire array is balanced.

🔑 Prefix Sum Reduction

Subarray sum[l..r] = 0 ⟺ prefix[r] − prefix[l−1] = 0 ⟺ prefix[r] = prefix[l−1]. So the problem reduces to: "Find the maximum r − l among all pairs (l, r) with prefix[l] = prefix[r]."

Edge cases: Empty array → 0. All same sign → 0. Updates that flip values change the prefix sum of every element from that index onward — a naive rebuild would be O(n) per update.

2 The Algorithm: Segment Tree on Prefix-Sum History ▾

The optimal solution combines three ideas:

1 Prefix-Sum Transform: Convert the problem. Replace each element with its contribution to the running sum, then append a sentinel 0 at position 0. Now "balanced subarray" becomes "equal prefix values at two positions."
2 Range Offset: Prefix sums lie in [−n, +n]. Shift by +n so all values are in [0, 2n], usable as array indices.
3 First-Occurrence Map (Static) or Segment Tree (Dynamic): For the static case, a hash map stores the first index where each prefix value appears. For each subsequent position r with prefix[r] = v, the answer candidate is r − first[v]. With updates, maintain a Segment Tree over prefix values; each node stores the earliest and latest indices that achieved a particular prefix value, letting us query the max-gap in O(log n).
4 Point Update Handling: When element i flips from +1 to −1 (or vice), all prefix sums from index i onward shift by ±2. This is a range update on prefix sums. To avoid rebuilding, the Segment Tree is indexed by (value, index) pairs: update by removing old (prefix[i], i) entries and inserting new ones lazily. With coordinate compression this becomes O(log n) per update.

Brute-force approach for contrast: For each pair (i, j) check if subarray [i, j] sums to zero — O(n²) time, unacceptable with updates.

⚠️ Common Misconception

People assume a simple hash map suffices for the dynamic version. It doesn't — a flip at index i changes prefix[i], prefix[i+1], …, prefix[n] all at once. You'd need to re-scan the entire right portion, which is O(n) per update. The Segment Tree collapses this to O(log n).

3 Why It Works: The Invariant ▾
🔐 The Invariant

At all times, seg_tree[v] stores the pair (first_idx[v], last_idx[v]) — the earliest and latest positions in the original array (including sentinel position 0) where prefix sum equals v. The answer is always max over all v of (last_idx[v] − first_idx[v]).

Why this guarantees correctness:

A subarray from index l+1 to r (1-indexed) sums to zero iff prefix[l] = prefix[r]. The length of this subarray is r − l. We want to maximize this over all valid (l, r) pairs. Since the Segment Tree tracks the minimum and maximum index for each prefix value, querying it for the global max gap covers every valid pair.

Invariant under updates: When element at index i is updated (flipped), the prefix sums at indices i, i+1, …, n all change by the same delta Δ ∈ {−2, +2}. We can represent this lazily: a "global offset" that shifts all prefix values in a range. The Segment Tree propagates these lazy tags in O(log n), restoring the invariant efficiently.

💡 Why a Segment Tree and Not a Balanced BST?

A balanced BST (like a sorted map) could store (value, index) pairs and answer "find max index − min index for same value" in O(log n) per query. But range updates (shifting all values ≥ i by ±2) on a BST require O(n log n) in the worst case. The Segment Tree with lazy propagation handles these in O(log n) because it operates on contiguous index ranges, not value ranges.

Edge case correctness: The sentinel prefix value 0 at position 0 ensures we capture subarrays starting at the beginning of the array (e.g., if prefix[5] = 0, the subarray [1..5] is captured). Empty subarrays (same index twice) are never considered since we track the minimum and maximum as separate positions.

Solution 1 — Static: Prefix Sum + Hash Map (Interview baseline)

first = {0: 0} ← sentinel: prefix sum 0 first seen at index 0 (before array starts)
prefix = 0 ← running prefix sum
best = 0 ← best answer so far
for i, v in enumerate(nums, 1): ← 1-indexed for clean indexing
prefix += v ← accumulate; v is +1 or -1
if prefix in first: ← we've seen this sum before...
best = max(best, i - first[prefix]) ← ...subarray from first[prefix]+1 to i has sum zero
else: ← first time seeing this sum...
first[prefix] = i ← ...record earliest index (greedy: first occurrence maximizes gap)
return best
⚠️ Why store only the FIRST occurrence?

We want to maximize the gap r − l for equal prefix values. A later occurrence of the same value can only help as the right endpoint. The left endpoint should be as early as possible. So we record the first time we see each prefix value and never update it.

⚠️ Cross-Boundary Gap Insight

You might worry: what if the minimum index is in the left subtree and the maximum index (for the same value) is in the right? That can't happen — the Segment Tree is indexed by prefix value, not by array position. Each leaf corresponds to one unique prefix value. Both the min and max index for that value are stored together at that leaf. So no cross-boundary issue arises for same-value pairs.

🔑 Complexity

Static build: O(n log n) — insert n+1 prefix values into the Segment Tree.
Per update: O(log n) — remove old prefix values and insert new ones for affected indices.
Space: O(n) — Segment Tree of size 4·(2n+1).

4b 🔬 Algorithm Deep-Dive: Segment Tree ▾
Segment Tree — Origin & Overview

The Segment Tree is a divide-and-conquer data structure developed in the 1970s–80s for computational geometry and range-query problems. It supports both range queries (e.g., min, max, sum over [l, r]) and point or range updates in O(log n) time with O(n) space. The key idea: represent a range [0, n−1] as a binary tree where each node covers a sub-range, and leaf nodes represent single elements.

Core Invariant: Each internal node [lo, hi] stores the aggregate (min, max, sum, etc.) of all elements in its range. When a single element changes, only O(log n) ancestors need updating — one per level of the tree.

Why It Works (structural proof sketch):

1 The tree has depth ⌈log₂ n⌉. Any point update touches at most one node per level (the one whose range contains the updated index), so O(log n) updates total.
2 Any range query [l, r] decomposes into O(log n) disjoint nodes that exactly cover [l, r]. Their aggregates are combined bottom-up.
3 Lazy propagation: instead of pushing updates to all O(n) leaves immediately, each node carries a "pending tag" applied only when the node is accessed. This amortizes range-update costs to O(log n).

Generalization for this problem: We index the Segment Tree by prefix-sum value (not array index). Each leaf stores the set of array positions that achieved that prefix value — specifically just the min and max of those positions, since that's all we need for the max-gap answer. This is sometimes called a "value-indexed segment tree" or "inverse segment tree."

💡 When to Recognize Segment Tree

The problem says: range queries + point/range updates, both needed efficiently. Specifically: "find the maximum/minimum/sum over [l, r]" AND "change element i, re-query." If both must be O(log n), think Segment Tree (or Fenwick Tree for simpler aggregates like sum). The Segment Tree is more general — use it when you need custom merge operations (like tracking both min and max simultaneously).

Each node stores (min_idx, max_idx) — the earliest and latest array positions where a prefix-sum value in that node's range was observed. The answer is always the root's max_idx − min_idx (or equivalently the global max of leaf-level max_idx − min_idx for leaves with more than one observation).

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Simplify Without Updates

Start by ignoring updates. The static problem is: "longest subarray summing to zero, elements are ±1." Classic prefix-sum insight: prefix[r] = prefix[l] implies subarray [l+1..r] has sum 0. Hash map to store first occurrence → O(n). Explain this clearly first.

Step 2: Identify What Breaks With Updates

When element at index i flips by Δ, prefix[i], prefix[i+1], …, prefix[n] all shift by Δ. The hash map stores O(n) entries all of which might change. Naively, O(n) per update — unacceptable for many updates.

Step 3: Think "Range Update + Range Query" = Segment Tree

The signal: you need to efficiently handle a batch of simultaneous updates (all suffixes shift) AND re-answer a global aggregation query (max gap among equal values). This is the canonical Segment Tree setup. Index the tree by prefix-value, store (min_idx, max_idx) at leaves, merge upward.

Step 4: Bound the Value Range

With n elements each ±1, prefix sums lie in [−n, +n]. There are 2n+1 distinct possible values — a manageable number. Use this as your segment tree size. Mention coordinate compression if the interviewer asks about memory.

🗣️ What to Say to the Interviewer

"Let me start with the static case. Subarray sum = 0 with ±1 elements means equal counts of +1 and −1. Using prefix sums, this is equivalent to finding two indices with equal prefix values — classic O(n) with a hash map storing first occurrences. Now with updates: flipping element i shifts all prefix sums from i onward by ±2. The hash map breaks because all suffix entries need updating. I need a structure that handles range-updates-on-values + global-max-gap queries. A Segment Tree indexed by prefix value, storing (min_idx, max_idx) at each value bucket, gives O(log n) per update and O(1) for the global answer at the root."
💡 Follow-Up: What if elements can be arbitrary integers?

If elements aren't constrained to ±1, prefix sums could reach arbitrary values. You'd need coordinate compression (sort and discretize the prefix values) or a balanced BST (like SortedList in Python). The Segment Tree approach still works but requires knowing all prefix values upfront (offline processing) or using a dynamic Segment Tree (pointer-based, O(log n) per new value).

7 Summary & Key Takeaways ▾
Reduce "subarray sum = 0" to "equal prefix values" — this is the universal first step for sum-based subarray problems
For the static case, a hash map storing first occurrences gives O(n) time and O(n) space
For the dynamic case (with point updates), a Segment Tree indexed by prefix-value gives O(log n) per update and O(log n) per query
The Segment Tree's key invariant: each node stores (min_idx, max_idx) for prefix values in its range; the root answer is the global max gap
Recognize the Segment Tree pattern when you see: range queries + point/range updates both needed in O(log n)
