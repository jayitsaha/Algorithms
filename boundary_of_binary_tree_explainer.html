LeetCode #545

Boundary of Binary Tree

Return the values of the boundary of a binary tree in anti-clockwise direction starting from the root — left boundary, leaves, then right boundary in reverse.

Medium Trees Left Boundary + Leaves + Right Time: O(n) Space: O(h)
1 Understanding the Problem ▾

The boundary of a binary tree is the set of nodes forming the visible "silhouette" when you look at the tree from outside. Imagine tracing the tree's outline anti-clockwise starting at the root: you go down the left edge, collect all bottom (leaf) nodes left-to-right, then climb up the right edge.

💡 Key Insight

The boundary decomposes into exactly three non-overlapping parts: the left boundary (top-down, excluding leaves), all leaf nodes (left-to-right), and the right boundary (bottom-up, excluding leaves). Solve each part independently — then concatenate.

Exact definitions:

L Left Boundary: Start at root.left. At each node, prefer left child; if none, go right. Stop before you reach a leaf. Add each non-leaf node top-down.
F Leaves: Every node with no children, in left-to-right order (DFS). This includes the leftmost and rightmost leaves — but NOT the root even if it is a leaf (the root is handled specially).
R Right Boundary: Start at root.right. Prefer right child; if none, go left. Stop before a leaf. Collect top-down into a temp list, then reverse it for bottom-up (anti-clockwise) order.
Example Tree

     1
    / \
   2 3
  / \ / \
  4 5 6 7

Output: [1, 2, 4, 5, 6, 7, 3]

Root: [1]
Left bd: [2] (stop before leaf 4)
Leaves: [4, 5, 6, 7]
Right bd: [3] reversed

Edge Cases: Single root node — root IS the only leaf, return [root.val]. Tree with only left subtree — no right boundary to collect. Skewed tree — one of the three parts may be empty, which is fine.

2 The Algorithm: Three-Pass DFS ▾

We make three separate traversals, each collecting one piece of the boundary. This is simpler than trying to do it in one pass — separation of concerns eliminates tricky conditionals.

0 Define is_leaf(node): Returns True if a node has no children. This helper is called in every pass to decide whether to include a node.
1 Add root.val first (always). Then check if root itself is a leaf (single-node tree) — if so, return immediately.
2 Left boundary pass: Iterative loop from root.left. While node exists: if NOT leaf, append. Descend: prefer node.left, else node.right. Stop when node is None.
3 Leaves pass: DFS traversal of entire tree. At each node: if it is a leaf, append its value. Otherwise recurse left then right.
4 Right boundary pass: Iterative loop from root.right. Same logic as left but prefer node.right first. Collect into right_bd, then res.extend(reversed(right_bd)).
Key Concept: Why Leaves are Separate

If we included leaves in the left/right boundary loops, the leftmost and rightmost leaves would appear twice — once in the boundary loop and once in the leaves pass. By explicitly stopping at leaves, we achieve a clean three-way partition where every boundary node appears in exactly one group.

3 Why It Works: The Invariant ▾
The Invariant

At any moment after the three passes: every node on the tree's outer perimeter appears in exactly one of {left boundary, leaves, right boundary}. Internal nodes (with both a left and right ancestor shielding them) never appear in any pass.

Why mutual exclusion holds:

  • Left boundary pass: includes ONLY non-leaf nodes on the leftmost path — guaranteed to miss all leaves
  • Leaves pass: includes ONLY nodes with no children — cannot include left/right boundary nodes (which are non-leaves by definition)
  • Right boundary pass: includes ONLY non-leaf nodes on the rightmost path — symmetric to left

Why all perimeter nodes are captured: A node is "visible" from the outside if and only if it lies on the leftmost path, is a leaf, or lies on the rightmost path. Any interior node has both a left child and a right child blocking it from view — these nodes are correctly excluded from all three passes.

Common Mistake: Off-by-one on the Root

The root is always in the result (index 0). The left boundary starts at root.left, not at root. The right boundary starts at root.right, not at root. Starting a pass at root instead of root's child double-counts the root.

Common Mistake: Wrong Preference Direction

Left boundary must prefer the LEFT child (go right only if left is absent). Right boundary must prefer the RIGHT child (go left only if right is absent). Swapping these preferences gives the wrong interior path, not the true boundary.

4 The Code (Annotated) ▾
def boundaryOfBinaryTree(root):
if not root: return []← empty tree guard
def is_leaf(n):← helper used in all 3 passes
return not n.left and not n.right
res = [root.val]← root is always the first boundary node
if is_leaf(root): return res← single-node tree: done
# ── Pass 1: Left Boundary (top-down, no leaves) ──
node = root.left← start at left child, not root
while node:
if not is_leaf(node):← skip leaves; leaves pass handles them
res.append(node.val)
node = node.left if node.left else node.right← prefer left; fallback right
# ── Pass 2: All Leaves (left-to-right DFS) ──
def add_leaves(n):
if not n: return
if is_leaf(n):← base case: found a leaf
res.append(n.val); return
add_leaves(n.left)← left before right → left-to-right leaf order
add_leaves(n.right)
add_leaves(root.left)← leaves in left subtree
add_leaves(root.right)← leaves in right subtree
# ── Pass 3: Right Boundary (collect top-down, then reverse) ──
right_bd = []← temp list to collect before reversing
node = root.right← start at right child, not root
while node:
if not is_leaf(node):
right_bd.append(node.val)← collect forward (top-down)
node = node.right if node.right else node.left← prefer right; fallback left
res.extend(reversed(right_bd))← reverse = bottom-up = anti-clockwise
return res
Complexity Analysis

Time: O(n) — Pass 1 visits at most O(h) nodes. Pass 2 (leaves DFS) visits all n nodes once. Pass 3 visits at most O(h) nodes. Total: O(n).
Space: O(h) — The recursive add_leaves call stack is at most O(h) deep. For a balanced tree h = O(log n); for skewed h = O(n).

5 Interview Strategy: How to Arrive at This ▾
Step 1: Draw it — decompose visually

Literally draw a tree and trace the anti-clockwise boundary with your finger. You'll naturally see three distinct segments: left spine going down, bottom leaves going right, right spine going up. Name them explicitly to your interviewer.

Step 2: Define "leaf" precisely

The critical boundary between "left boundary" and "bottom leaves" is the leaf node. A node with even one child is NOT a leaf. State this clearly: "I'll use an is_leaf() helper to stop boundary passes before leaves."

Step 3: Propose three helpers, not one complex traversal

A single pass trying to handle all three cases becomes error-prone. Three focused passes are easier to reason about, test, and explain. Each is 4-6 lines. Interviewers appreciate clean decomposition.

Step 4: Trace through the example before coding

For the tree in the problem: root=1, left bd=[2], leaves=[4,5,6,7], right bd reversed=[3]. Verify: [1,2,4,5,6,7,3]. This confirms your logic before you write a line of code.

What to Say to the Interviewer

"I'll decompose the boundary into three non-overlapping parts:
the left boundary (top-down, non-leaves only),
all leaf nodes (left to right via DFS),
and the right boundary (non-leaves, collected top-down then reversed).
The key helper is is_leaf(), which separates these three groups cleanly.
Each pass is O(n) at worst, but combined they each visit every node at most once,
so total time is O(n) and space is O(h) for the recursion stack."
Potential Follow-Up Questions
  • Can you do this iteratively without recursion? — Yes: use a stack for the leaves pass.
  • What if the tree is very skewed (like a linked list)? — Space becomes O(n) due to stack depth. An iterative solution avoids this.
  • How does this change if "boundary" excludes the root? — Remove res = [root.val] and start left pass from root instead.

7 Summary & Key Takeaways ▾
The boundary decomposes into three independent, non-overlapping groups: left boundary, all leaves, right boundary reversed.
The is_leaf() predicate is the linchpin — it separates which nodes belong to which pass and prevents double-counting.
Left boundary prefers left child; right boundary prefers right child. Both are iterative loops that stop before leaves.
Right boundary is collected top-down, then reversed to get anti-clockwise (bottom-up) order.
Time O(n) — total work across all three passes is proportional to n. Space O(h) for recursion stack in leaves pass.
When to Recognize This Pattern
  • Problem mentions "outline", "silhouette", "boundary", or "perimeter" of a tree
  • Need to collect left-spine nodes AND leaf nodes AND right-spine nodes separately
  • Anti-clockwise or clockwise enumeration of tree nodes
  • "Right side view" or "Left side view" are closely related simpler variants
