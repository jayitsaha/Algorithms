LeetCode #1335

Minimum Difficulty of a Job Schedule

Schedule n jobs in order across exactly d days (≥1 per day). Each day's cost = max difficulty of its jobs. Minimize total cost.

Hard Dynamic Programming dp[day][job] with Max Time: O(d·n²) Space: O(d·n)
1 Understanding the Problem ▾

You have a list of job difficulties and must schedule them across exactly d days. Two hard constraints: jobs must be done in the given order (no reordering), and each day must have at least one job. Each day's cost equals the maximum difficulty among its jobs. Minimize the total.

💡 Key Insight

Because jobs must stay in order, every valid schedule splits the job array into exactly d contiguous non-empty chunks. The problem reduces to: how do we partition an array into d consecutive groups to minimize the sum of group-maxima?

✓ Example: jobs=[6,5,4,3,2], d=2

Day 1: [6,5,4,3] → max=6
Day 2: [2] → max=2
Total = 6+2 = 8 ✓ (optimal)

✗ Invalid: cannot reorder

Day 1: [2,1] ← impossible!
Jobs must be contiguous
blocks in their original order.

Edge cases: If n < d → return -1 (impossible to assign ≥1 job per day). If d == 1 → answer is max(jobDifficulty). If d == n → sum of all difficulties.

⚠️ Why brute force is infeasible

There are C(n−1, d−1) ways to split n jobs into d groups. For n=300, d=10 that is hundreds of billions of partitions. We must identify overlapping subproblems and cache their solutions.

2 The Algorithm — Why is This DP? ▾

Dynamic Programming requires two properties. Both are present here.

🔐 Optimal Substructure

The minimum cost to schedule jobs 0..j in i days is built from the minimum cost to schedule jobs 0..k in i−1 days (for any valid cut-point k), plus the cost of today's chunk (jobs k+1..j). Optimal choices compose: the best suffix is independent of how the prefix was split.

🔐 Overlapping Subproblems

Many different parent states ask: "what's the cheapest way to do jobs 0..k in 2 days?" Memoizing dp[day][last_job_idx] ensures each subproblem is solved exactly once.

State definition: dp[day][j] = minimum total difficulty to schedule exactly the first j+1 jobs (indices 0 through j) across exactly day days.

📐 The Recurrence Relations
dp[1][j] = max(jobs[0 .. j]) ← base case: 1 day, do all jobs 0..j

dp[day][j] = min over k from (day-2) to (j-1) of:
    dp[day-1][k] + max(jobs[k+1 .. j])

Answer: dp[d][n-1]
1 Guard: if n < d, return -1 immediately — can't give each day at least one job.
2 Base case (day=1): dp[1][0] = jobs[0]. Then for j=1..n-1: dp[1][j] = max(dp[1][j-1], jobs[j]) — scanning prefix maxima in O(n).
3 Fill days 2..d: For each j from day-1 to n-1, scan k from j-1 down to day-2. Maintain a running maximum of jobs[k+1..j] as the window grows (k moves left). Update dp[day][j] = min(dp[day][j], dp[day-1][k] + cur_max).
4 Answer: Return dp[d][n-1] — all n jobs scheduled in exactly d days.
3 Why It Works: The Invariant ▾
🔐 The Invariant

When we compute dp[day][j], it holds the minimum possible total cost of any valid schedule that uses exactly day days and finishes with job index j as the last job of day day. All jobs 0 through j have been assigned.

Why enumerating all k is complete: Every valid schedule for day i ending at job j must have some day i-1 ending at some job k, where i-2 ≤ k < j (to guarantee ≥1 job per day on each prior day). We try every such k, so we cannot miss the optimal split.

Why scanning k right-to-left is efficient: As k decreases from j-1 to day-2, the window jobs[k+1 .. j] grows by adding jobs[k+1]. The maximum can only stay the same or increase. We maintain it with cur_max = max(cur_max, jobs[k+1]) — O(1) per step — giving O(n) for the entire inner loop per (day, j).

⚠️ The Right-to-Left Trick

If you iterated k left-to-right, you would have a shrinking window whose max is expensive to recompute (requires a deque or sparse table). The right-to-left scan gives a growing window — a simple running max suffices. This reduces the algorithm from O(d·n³) to O(d·n²).

Why the answer sits at dp[d][n-1]: We want all n jobs scheduled in exactly d days. The last day always ends at job n-1. The first index of the DP's "day" dimension is 1, so the answer is dp[d][n-1].

4 The Code (Annotated) ▾

Solution 1 — Bottom-Up DP (Interview Pick), O(d·n²) time, O(d·n) space

def minDifficulty(jobDifficulty, d): ← jobDifficulty array + number of days d
n = len(jobDifficulty)
if n < d: return -1 ← Impossible: can't assign ≥1 job per day
INF = float('inf')
dp = [[INF] * n for _ in range(d + 1)] ← dp[0] unused; dp[1..d][0..n-1]
dp[1][0] = jobDifficulty[0] ← Day 1 ending at job 0: cost = that job
for j in range(1, n): ← Build base case: running prefix max
dp[1][j] = max(dp[1][j-1], jobDifficulty[j]) ← 1 day, jobs 0..j: cost = max
for day in range(2, d + 1): ← Fill each subsequent day
for j in range(day - 1, n): ← j ≥ day-1: prior days need at least day-1 jobs
cur_max = 0 ← Running max of today's job window
for k in range(j - 1, day - 3, -1):← k from j-1 down to day-2; window grows
cur_max = max(cur_max, jobDifficulty[k + 1]) ← jobs[k+1] joins today's window
if dp[day-1][k] < INF: ← Skip unreachable states
dp[day][j] = min(
dp[day][j],
dp[day-1][k] + cur_max) ← prev days cost + today's max
return dp[d][n-1] if dp[d][n-1] < INF else -1
⚠️ Why range(j-1, day-3, -1) for the k loop?

k is where day (day-1) ends. Today (day) handles jobs k+1 through j. We need: (a) today has ≥1 job → k ≤ j-1; (b) prior days have ≥(day-1) jobs total → k ≥ day-2. Python's range is exclusive on the stop, so range(j-1, day-3, -1) gives k from j-1 down to day-2 inclusive.

Solution 2 — Top-Down Memoization (easier to derive)

from functools import lru_cache
def minDifficulty(jobDifficulty, d):
n = len(jobDifficulty)
if n < d: return -1
@lru_cache(maxsize=None)
def dp(days_left, start): ← days_left days remain; start from job 'start'
if days_left == 1: ← Base: 1 day left → do all remaining jobs
return max(jobDifficulty[start:])
res, cur_max = float('inf'), 0
for end in range(start, n - days_left + 1):← Leave enough jobs for remaining days
cur_max = max(cur_max, jobDifficulty[end])
res = min(res, cur_max + dp(days_left - 1, end + 1))
return res
return dp(d, 0)
💡 Complexity Summary

Both approaches: Time O(d · n²) — d days, n starting positions, n inner choices.
Bottom-up space: O(d · n). Reducible to O(n) using two rolling rows.
Top-down space: O(d · n) memo + O(d) call stack depth.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Partition Signal

"Jobs in order" + "split into exactly d groups" = contiguous partitioning. Every partition problem with an optimization objective is a DP candidate.

Step 2: Define the State

State needs to capture (a) how many days we've used, (b) how far into the job list we've gone. State = (day, last_job_idx). Value = minimum cost.

Step 3: Write the Recurrence Verbally

"To know dp[day][j], try every cut-point k. Yesterday ended at job k; today does jobs k+1 through j with cost = max(jobs[k+1..j])." Convert this to code.

Step 4: Spot and Fix the Bottleneck

Naively computing max(jobs[k+1..j]) for every k is O(n) → O(d·n³) total. Fix: scan k right-to-left, maintain a running max. Inner loop becomes O(n) amortized → O(d·n²) total.

🗣️ What to Say to the Interviewer

"Jobs must be in order and we need exactly d contiguous groups — classic partition DP.
State: dp[day][j] = min cost to schedule jobs 0..j in exactly day days.
Recurrence: dp[day][j] = min over k of (dp[day-1][k] + max(jobs[k+1..j])).
To avoid recomputing the max, I scan k right-to-left, tracking a running max O(1).
Base case: dp[1][j] = max(jobs[0..j]). Answer: dp[d][n-1].
Time: O(d·n²). Space: O(d·n), optimizable to O(n) with rolling arrays."
🔮 Likely Follow-up Questions
  • "Reduce space to O(n)?" → Use two 1D arrays: prev_day and cur_day.
  • "What if cost per day is sum, not max?" → Use prefix sums; O(d·n) time.
  • "What if jobs can be done in any order?" → Different problem; greedy sort may apply.
  • "What if we want the schedule, not just the cost?" → Backtrack through the dp table.

7 Summary & Key Takeaways ▾
Jobs must stay in order → contiguous partition → DP, not greedy
State: dp[day][j] = min cost for jobs 0..j in exactly day days
Recurrence: try all cut-points k; today's cost = max(jobs[k+1..j])
Right-to-left k scan + running max turns O(d·n³) → O(d·n²)
Guard: n < d → return -1; base case: 1 day = prefix max
