LeetCode #346

Moving Average from Data Stream

Design a class that computes the moving average of the last size numbers in a stream of integers.

Easy Queues Queue with Size Limit Time: O(1) Space: O(size)
1 Understanding the Problem ▾

A moving average (also called a rolling or sliding window average) looks at only the most recent size values in a stream. As each new number arrives, the window slides forward — dropping the oldest value and incorporating the newest. The average is recomputed over this fresh window.

💡 Key Insight

You don't need to resum the whole window from scratch on every call. Maintain a running sum: when a new value arrives, add it to the sum; when the oldest value is evicted, subtract it. The average is then window_sum / len(queue) in O(1) time — no matter how large the window is.

Think of a stock ticker displaying the average close price of the last 5 trading days. As day 6 arrives, day 1 ages out. You don't add all 5 prices each time — you adjust the total by adding the new price and subtracting the one that just left the window.

2 The Algorithm: Bounded Queue + Running Sum ▾

We maintain a FIFO queue capped at size elements and a running sum variable. Each call to next(val) performs only O(1) work, regardless of window size.

1 Initialise: Store the capacity size, create an empty deque, and set window_sum = 0. The deque will hold the current window's values; the sum mirrors their total.
2 Append new value: Add val to the back of the queue (this is the "in" end of the FIFO). Also add val to window_sum.
3 Evict if over capacity: If len(queue) > size, pop the element from the front (the oldest value) and subtract it from window_sum. The window is now back within capacity.
4 Return average: Compute window_sum / len(queue). Dividing by len(queue) (not by size) correctly handles the warm-up period when fewer than size values have arrived.
🔑 Why deque and Not a Plain List?

collections.deque offers O(1) popleft() — removing from the front is a pointer update. A regular Python list's pop(0) is O(n) — it shifts every element one position left. For any queue-based solution, always reach for deque.

❌ Brute Force — Recompute Each Call

Keep last size values in a list.
Each call: sum(window) from scratch.
Time per call: O(size).
For size=10 000 and 10 000 calls: 108 ops.

✓ Optimal — Running Sum

Maintain sum alongside the queue.
Each call: one add + one conditional sub.
Time per call: O(1).
Same answer, any window size, zero extra work.

3 Why It Works: The Invariant ▾
🔐 The Invariant

After every call to next(val), window_sum is exactly equal to the sum of all values currently stored in the queue. The queue always holds the most recent min(calls_made, size) values — never more, never fewer.

Why the invariant holds — inductive proof sketch:

  • Base case: Before any call, queue = [], window_sum = 0. Trivially equal.
  • Step — no eviction: We append val to both the queue and window_sum simultaneously. Equal before → equal after.
  • Step — with eviction: We also remove old = popleft() from both the queue and window_sum simultaneously. Equal before → equal after.

Why the average is correct: Because window_sum = sum(queue) and len(queue) is the exact count of values in the current window, the ratio window_sum / len(queue) is by definition the mean of the window.

⚠️ Common Mistake: Dividing by size Always

During the warm-up phase (fewer than size calls made), dividing by size gives a deflated answer. Example: size=3, first call next(6) → window = [6], sum = 6. Correct avg = 6/1 = 6.0. If you divide by size: 6/3 = 2.0 — wrong. Always use len(self.queue).

Why O(1) space is impossible: We must remember the oldest value to subtract it when it leaves the window. Without storing all size values, we cannot know what to subtract. So O(size) space is optimal — and tight.

4 The Code (Annotated) ▾

Solution 1 — Bounded Deque + Running Sum (Optimal, O(1) per call)

from collections import deque ← O(1) popleft; plain list.pop(0) is O(n)
class MovingAverage:
def __init__(self, size: int): ← Called once; sets up the window
self.size = size ← Maximum number of values to remember
self.queue = deque() ← Holds current window values (oldest at front)
self.window_sum = 0 ← Always = sum(queue) — the invariant
def next(self, val: int) -> float:
self.queue.append(val) ← New value enters at the back (newest end)
self.window_sum += val ← Keep sum in sync: sum(queue) += val
if len(self.queue) > self.size: ← Queue grew beyond capacity?
self.window_sum -= self.queue.popleft() ← Evict oldest; subtract from sum
return self.window_sum / len(self.queue) ← len ≤ size; handles warm-up correctly
⚠️ Order of Operations Matters

Append the new value before checking length. This way, if the queue was exactly at capacity, the new append makes it one over, triggering the eviction. If you check length first and the queue is full, you'd evict before adding — and miss the addition.

Solution 2 — Circular Buffer (Alternative O(1), fixed pre-allocated space)

class MovingAverage:
def __init__(self, size: int):
self.size = size
self.buf = [0] * size ← Pre-allocated ring; no dynamic allocation
self.head = 0 ← Next slot to overwrite (wraps around)
self.count = 0 ← How many values are currently in the window
self.window_sum = 0
def next(self, val: int) -> float:
self.window_sum -= self.buf[self.head] ← Subtract old value in this slot (0 initially)
self.buf[self.head] = val ← Overwrite slot with new value
self.window_sum += val ← Add new value to sum
self.head = (self.head + 1) % self.size ← Advance head; wrap at capacity
self.count = min(self.count + 1, self.size) ← Track actual window size (caps at size)
return self.window_sum / self.count
💡 Complexity Summary

Recompute each call (brute force): Time O(size) per call, Space O(size)
Deque + running sum: Time O(1) per call, Space O(size) — deque grows up to size
Circular buffer: Time O(1) per call, Space O(size) fixed — no dynamic resizing ever

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify the Pattern Signals

Key phrases: "last size elements", "data stream", "moving/rolling/sliding average". This screams bounded FIFO queue. The "design a class" framing tells you to think about persistent state between calls — what must the object remember?

Step 2: Start With Brute Force

"I could store the last size values in a list and sum them each call — O(size) per call, O(size) space. Correct. But for large windows called frequently, this is wasteful — we're re-adding values that haven't changed."

Step 3: Discover the Running Sum Trick

"The new sum equals old sum + new value − evicted value. If I track the sum explicitly, I can update it in O(1). A queue tells me which value to evict — it's always the oldest, sitting at the front."

Step 4: Mention the Circular Buffer for Bonus Points

"For a production-grade system, I'd pre-allocate a fixed array of size size and use a wrapping head pointer. This avoids dynamic allocation entirely. The head advances modulo size, overwriting the oldest slot each time."

🗣️ What to Say to the Interviewer

"I need to maintain the last 'size' values and return their average on each call.
Brute force: keep a list, sum it each time — O(size) per call.
Optimal: track a running sum. On append, add val; if window is full, also subtract the evicted oldest.
I'll use a deque for O(1) popleft(). Divide by len(queue) — not by size —
because the window may not be full during the first few calls.
Invariant: window_sum always equals sum(queue). Time O(1), Space O(size)."
🔮 Likely Follow-up Questions
  • "What if values can be negative?" → Same logic — addition and subtraction are sign-agnostic.
  • "How would you handle thread safety?" → Wrap the deque operations in a mutex/lock.
  • "Can you avoid a deque?" → Yes — circular buffer with a fixed array and modulo head.
  • "Moving median instead of average?" → Two heaps (max + min) for O(log n) per call (#295).
  • "Moving max instead of average?" → Monotonic deque for O(1) amortised per call (#239).

7 Summary & Key Takeaways ▾
A FIFO queue naturally models a sliding window — oldest element at front, newest at back
The running sum trick reduces recompute O(size) → O(1): add new, subtract evicted
Always use deque for queue operations — popleft() is O(1), list.pop(0) is O(n)
Divide by len(queue), not by size — window may not be full during warm-up
Circular buffer pre-allocates fixed space — no dynamic memory; a senior-level mention
O(size) space is optimal — you must remember values to know what to subtract when they leave
