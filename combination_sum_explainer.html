LeetCode #39

Combination Sum

Given an array of distinct integers and a target, find all unique combinations that sum to the target. Each number may be used unlimited times.

Medium Backtracking Unlimited Reuse · Start Index Time: O(N^(T/M)) Space: O(T/M)
1 Understanding the Problem ▾

Given an array of distinct integers candidates and a target, return all unique combinations where the chosen numbers sum to target. You may use the same candidate unlimited times. The solution set must contain no duplicate combinations (so [2,2,3] and [3,2,2] count as the same combination — only one should appear).

💡 Key Insight

This is a search problem — we must explore all ways to "fill" the target sum. At each step we make a decision: pick a candidate (possibly again) and reduce the remaining budget. The word "all" combinations signals backtracking — we cannot find just one answer, we must exhaustively enumerate. The key trick that prevents duplicates is a start index: choices are always non-decreasing, so [3,2,2] is never generated because once we pass 2 and choose 3, we never go back.

✓ candidates=[2,3,6,7], target=7

[2,2,3] → 2+2+3 = 7 ✓
[7] → 7 = 7 ✓
Output: [[2,2,3],[7]]

✗ Not valid or duplicate

[3,2,2] — dup of [2,2,3]
[2,3,2] — dup of [2,2,3]
[1,6] — 1 not in candidates

Edge Cases: target equals a single candidate ([7] alone); no combination sums to target (empty output); all candidates larger than target (every branch pruned immediately); target achievable only with many reuses (e.g., target=10, candidates=[3] → no solution; target=9, candidates=[3] → [3,3,3]).

2 The Algorithm: Backtracking with Start Index ▾

Backtracking is systematic trial and error on a decision tree. We build a combination element by element. At each node in the tree, we try adding each available candidate, recurse deeper, then undo our choice (backtrack) and try the next option. This covers every possibility without generating duplicates.

1 Sort candidates: Sorting enables early loop termination — once a candidate exceeds the remaining budget, all larger ones do too. This prunes entire subtrees without exploring them.
2 Define backtrack(start, path, remaining): start = leftmost index still available; path = numbers chosen so far; remaining = how much budget is left to fill.
3 Base case — success: If remaining == 0, path is a valid combination. Append a copy (path[:]) to results and return.
4 Loop from start to end: For each candidates[i], if it exceeds remaining, break (sorted → all further ones also exceed). Otherwise: append it, recurse with start=i (same i, allowing reuse), then pop it.
5 The "start=i" vs "start=i+1" toggle: Passing i lets us pick the same element again. Passing i+1 would prohibit reuse. This single parameter is the only difference between Combination Sum I and II.

🔑 Why "start=i" Enables Reuse and Prevents Duplicates

After choosing candidates[i], we recurse with start=i — the same position — so the next call can pick candidates[i] again. This gives unlimited reuse. But we never go backward to index < i, so [3,2,2] is impossible: once we've committed to picking 3 (i=1), the available candidates are {3,6,7} — 2 (i=0) is gone forever.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At any recursive call with backtrack(start, path, remaining): the elements in path are in non-decreasing order, and remaining equals target minus sum(path). This two-part invariant guarantees: (1) every multiset appears in sorted order, so no two distinct paths yield the same multiset; (2) remaining is always the exact budget left, so the base case remaining==0 is sound.

Uniqueness proof: Two paths represent the same combination if and only if they are the same multiset. Since every path is built in non-decreasing order (the start index forbids going backward), two paths that are the same multiset must be identical sequences. Therefore the set of paths generated is in bijection with the set of valid multisets — no duplicates.

Pruning correctness: After sorting, if candidates[i] > remaining, then for all j > i, candidates[j] >= candidates[i] > remaining. Adding any of them would overshoot. Breaking at position i is equivalent to removing all subtrees rooted at i, i+1, ..., n-1 — all of which are provably empty of solutions. No valid combination is discarded.

⚠️ Most Common Bug: Appending the Reference, Not a Copy

When remaining == 0, you must write results.append(path[:]) — not results.append(path). Since backtracking mutates the same list via append/pop, any saved reference will point to the eventual empty list at the end of execution. Every "saved" result becomes []. Always copy with path[:] or list(path).

Edge case correctness: If candidates is empty, the for loop never executes and we return []. If the smallest candidate exceeds target, the prune fires in the very first call and we return [] immediately. If target equals a single candidate, we find it in the first branch at depth 1.

4 The Code (Annotated) ▾

Solution 1 — Backtracking with Sort & Prune (Optimal, Interview Pick)

def combinationSum(candidates, target):
candidates.sort() ← Sort enables early break when candidate > remaining
results = []
def backtrack(start, path, remaining):
if remaining == 0: ← Base case: hit target exactly
results.append(path[:]) ← Copy! Not path — that reference will be mutated later
return
for i in range(start, len(candidates)): ← Only candidates from index 'start' onward
if candidates[i] > remaining: ← Prune: sorted → all further candidates also too large
break
path.append(candidates[i]) ← Choose: add this candidate
backtrack(i, path, ← Explore: start=i (not i+1!) → allows reusing same element
remaining - candidates[i])
path.pop() ← Unchoose: restore path for the next iteration
backtrack(0, [], target) ← Kick off: all candidates available, empty path, full target
return results
⚠️ start=i vs start=i+1 — Critical Difference

The only modification needed to solve Combination Sum II (each element once) is passing i+1 instead of i in the recursive call, plus skipping adjacent duplicate elements. That single character difference separates "unlimited use" from "each element once."

Solution 2 — Backtracking without Sort (Brute Force baseline)

def combinationSum_brute(candidates, target):
results = []
def backtrack(start, path, remaining):
if remaining == 0:
results.append(path[:]); return
if remaining < 0: return ← Prune AFTER overshooting (not before — wasteful)
for i in range(start, len(candidates)):
path.append(candidates[i])
backtrack(i, path, remaining - candidates[i])
path.pop()
backtrack(0, [], target)
return results ← Correct but enters overshooting calls — slower in practice
💡 Complexity

Time: O(NT/M) where N = number of candidates, T = target, M = smallest candidate. In the worst case (small candidates, large target) the tree is deep and wide.
Space: O(T/M) for the recursion call stack depth (max depth = target / min_candidate). The output space is additional and depends on the number of solutions.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Backtracking Signal

The word "ALL combinations / arrangements / subsets" is the strongest backtracking signal. If the problem asked "does any combination exist?" or "how many combinations?", you could use DP or greedy. But "find all" means you must enumerate — backtracking is the right tool.

Step 2: Frame the Decision Tree

Ask: "What do I choose at each step?" Here: which candidate to add next. "What's my state?" The remaining budget and what's in the path. "When do I stop?" When remaining hits 0 (success) or a candidate exceeds remaining (failure). This framing maps directly to the function signature.

Step 3: Address Duplicates Explicitly

Say to the interviewer: "I'll use a start index to enforce non-decreasing order on my choices. This ensures [2,3] and [3,2] are treated as the same combination and only [2,3] is generated. No visited set or deduplication post-processing needed."

Step 4: Mention the Sort Optimization

"I'll sort candidates first. Then in the loop, as soon as a candidate exceeds the remaining budget, I break — since sorted order means all subsequent candidates also exceed it. This prunes entire subtrees and avoids recursing into hopeless branches."

🗣️ What to Say to the Interviewer

"This asks for ALL combinations — that signals backtracking over a decision tree.
At each step I choose a candidate to add to my current path, recurse, then undo.
To prevent duplicates, I pass a start index — choices are always non-decreasing,
so [3,2,2] is never generated. For unlimited reuse I pass start=i (same index),
not i+1. I'll sort first and break when a candidate exceeds remaining — prunes subtrees.
Time: O(N^(T/M)), Space: O(T/M) for recursion depth."
🔮 Likely Follow-up Questions
  • "Each element used at most once?" → Combination Sum II: pass i+1, skip duplicate values
  • "Find exactly k numbers summing to n?" → Combination Sum III: add depth parameter
  • "Count the combinations, don't list them?" → Combination Sum IV: DP (unbounded knapsack)
  • "What if candidates can repeat?" → Combination Sum II variant; sort + skip duplicates
  • "Candidates can be negative?" → The break optimization is invalid; use remaining < 0 check instead

7 Summary & Key Takeaways ▾
Backtracking = choose + explore + unchoose. The path is mutated in place; pop after the recursive call restores it.
The start index prevents duplicates: choices are always non-decreasing, so each multiset is generated exactly once.
Passing start=i (same index) enables unlimited reuse. Changing to i+1 gives "each element at most once."
Sort candidates first, then break the loop when a candidate exceeds remaining — prunes entire subtrees.
Always copy the path (path[:]) when saving to results — the reference will be mutated by subsequent backtracking.
ApproachTimeSpaceNotes
Backtracking (no sort/pruning) O(N^(T/M)) + extra O(T/M) Enters overshooting branches
Backtracking + sort + break ✓ O(N^(T/M)) O(T/M) Interview pick; prunes early
🎯 When to Recognize This Pattern
  • "Find ALL combinations / subsets that satisfy a condition" → backtracking
  • "Elements can be reused unlimited times" → pass same index (start=i) on recurse
  • "No duplicate combinations" → enforce non-decreasing order via start index (no visited set needed)
  • "Only COUNT the combinations (not list them)" → switch to DP (unbounded knapsack)
