LeetCode #97

Interleaving String

Given strings s1, s2, and s3, return true if s3 is formed by an interleaving of s1 and s2.

Medium Dynamic Programming 2D DP on Both Strings Time: O(m·n) Space: O(m·n) → O(n)
1 Understanding the Problem ▾

An interleaving of two strings means merging them into one while preserving the relative order of characters from each source string. Think of shuffling two separate decks of cards: each card keeps its position relative to the other cards from its own deck, but cards from the two decks can be interleaved in any way.

💡 Key Insight

At every position in s3, the next character was contributed by either s1 or s2. This binary choice at each of the (m+n) positions — combined with the constraint that relative order within each source must be preserved — creates a 2D search space: (i chars used from s1) × (j chars used from s2).

Concrete example: s1 = "aab", s2 = "axy", s3 = "aaxaby"

s3: a a x a b y
from: s1 s2 s2 s1 s1 s2 ← one valid assignment
s1 chars used: a(s1[0]), a(s1[1]), b(s1[2]) — order: 0→1→2 ✓
s2 chars used: a(s2[0]), x(s2[1]), y(s2[2]) — order: 0→1→2 ✓
✓ Valid Interleaving

s1="aab", s2="axy"
s3="aaxaby"
len check: 3+3=6 ✓
Relative orders preserved ✓

✗ Not an Interleaving

s1="aab", s2="axy"
s3="aayxab"
s2 chars appear as: y,x
but s2="axy" → must be a,x,y ✗

Edge cases: Both empty → True (trivially). One empty → reduces to substring equality check. Lengths don't sum to len(s3) → immediate False (most important quick-exit).

2 The Algorithm: 2D Tabulation DP ▾

We build a 2D boolean table dp[i][j] where the entry is "Can the first i characters of s1 and first j characters of s2 together form the first i+j characters of s3?"

Notice the elegant constraint: once we fix i and j, we know exactly which position in s3 we're matching: index i + j - 1 (0-indexed). There's no ambiguity.

1 Quick fail: If len(s1) + len(s2) ≠ len(s3), return False. Total character count must match.
2 Create table: dp = [[False]*(n+1) for _ in range(m+1)]. Set dp[0][0] = True: zero chars from both → zero chars of s3 trivially.
3 Fill first column (j=0, only using s1): dp[i][0] = dp[i-1][0] and s1[i-1] == s3[i-1]. Can s1's first i chars match s3's first i chars, using s1 alone?
4 Fill first row (i=0, only using s2): dp[0][j] = dp[0][j-1] and s2[j-1] == s3[j-1]. Same idea for s2.
5 Fill interior (i≥1, j≥1): k = i+j-1. dp[i][j] = (dp[i-1][j] and s1[i-1]==s3[k]) or (dp[i][j-1] and s2[j-1]==s3[k]). Either take from s1 (came from row above) or from s2 (came from cell to the left).
6 Return dp[m][n]: Have we consumed all m chars from s1 and all n chars from s2 in a valid way?
🔑 The Two-Transition Rule

Every interior cell (i, j) can be reached via exactly two predecessor cells:
• From (i-1, j): we just took s1[i-1] from s1. Valid iff that char equals s3[i+j-1].
• From (i, j-1): we just took s2[j-1] from s2. Valid iff that char equals s3[i+j-1].
The cell is True if either predecessor is True and the character matches.

3 Why It Works: Invariant & DP Deep Dive ▾

Why is this Dynamic Programming? Two pillars must both be present:

Optimal Substructure

Whether s1[0..i-1] + s2[0..j-1] can form s3[0..i+j-1] depends directly and only on whether s1[0..i-2] + s2[0..j-1] could form s3[0..i+j-2] (plus a character match). Larger valid interleavings are built from smaller valid ones.

Overlapping Subproblems

The cell dp[2][3] is a dependency for both dp[3][3] and dp[2][4]. Without caching, recursive approaches recompute every (i,j) pair from multiple parent nodes — exponential blowup: O(2^(m+n)).

📐 The Recurrence Relation
Base: dp[0][0] = True
Edge: dp[i][0] = dp[i-1][0] and s1[i-1] == s3[i-1]
Edge: dp[0][j] = dp[0][j-1] and s2[j-1] == s3[j-1]
Interior (i≥1, j≥1), k = i+j-1:
  dp[i][j] = (dp[i-1][j] and s1[i-1]==s3[k])
             or (dp[i][j-1] and s2[j-1]==s3[k])
🔐 The Invariant

dp[i][j] = True if and only if there exists at least one valid way to interleave s1[0..i-1] and s2[0..j-1] to produce exactly s3[0..i+j-1].

This invariant is maintained inductively: dp[0][0] satisfies it trivially, and the recurrence propagates it correctly by extending only from previously-valid states with a matching character.

⚠️ Why Greedy Fails Completely

Consider s1="ab", s2="ba", s3="abba". At position 0, s3[0]='a'. Both s1[0]='a' and s2[1]='b' exist. Greedy picks s1. Now we need s3[1]='b'. s1[1]='b' — greedy picks it again. Now s1 is exhausted but s2="ba" must match s3[2..3]="ba" — and it does! OK so far... but if s3="baba" instead, greedy taking from s2 first also works. The issue is that at SOME inputs, greedy gets stuck with no backtracking. DP avoids this by exploring all valid paths implicitly through the table.

4 The Code (Annotated) ▾

Solution 1 — Tabulation / Bottom-Up DP (Interview Pick)

def isInterleave(s1: str, s2: str, s3: str) -> bool:
m, n = len(s1), len(s2) ← m rows (s1 length), n cols (s2 length)
if m + n != len(s3): return False ← lengths must add up — hardest fail-fast
dp = [[False]*(n+1) for _ in range(m+1)] ← (m+1)×(n+1) table of False
dp[0][0] = True ← base: empty prefixes → empty s3 ✓
for i in range(1, m+1): ← first column: only s1 characters used
dp[i][0] = dp[i-1][0] and s1[i-1] == s3[i-1] ← chain must be unbroken
for j in range(1, n+1): ← first row: only s2 characters used
dp[0][j] = dp[0][j-1] and s2[j-1] == s3[j-1] ← same idea for s2
for i in range(1, m+1): ← fill interior row by row
for j in range(1, n+1): ← every cell in this row
k = i + j - 1 ← index in s3 we are now matching
from_s1 = dp[i-1][j] and s1[i-1] == s3[k] ← extend a valid i-1,j state
from_s2 = dp[i][j-1] and s2[j-1] == s3[k] ← extend a valid i,j-1 state
dp[i][j] = from_s1 or from_s2 ← True if either valid path exists
return dp[m][n] ← consumed all chars from both strings?
⚠️ The k = i+j-1 Indexing Trick

At cell (i,j) we've already consumed i characters from s1 and j from s2, so we've covered positions 0 through i+j-1 in s3. The next character we're adding to match is at index i+j-1 (because we're "finishing" that character with this cell). Off-by-one errors here are the #1 bug in this problem.

Solution 2 — Memoization / Top-Down DP

def isInterleave(s1: str, s2: str, s3: str) -> bool:
m, n = len(s1), len(s2)
if m + n != len(s3): return False
from functools import lru_cache
@lru_cache(maxsize=None) ← caches every (i,j) pair automatically
def dp(i, j): ← i = chars used from s1, j = from s2
if i == m and j == n: return True ← consumed all → success
k = i + j ← current position in s3 (no -1 here: 0-indexed already)
if i < m and s1[i] == s3[k]: ← try consuming s1[i]
if dp(i+1, j): return True ← short-circuit: return as soon as a path works
if j < n and s2[j] == s3[k]: ← try consuming s2[j]
if dp(i, j+1): return True
return False ← neither choice worked from here
return dp(0, 0)
💡 Space-Optimized Tabulation — O(n) Space

We only ever look at row i-1 when computing row i. So we can reuse a single 1D array, updating it in place:

dp[j] in this row = result for (i, j)
dp[j] before update = result for (i-1, j) ← "from_s1"
dp[j-1] already updated = result for (i, j-1) ← "from_s2"

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify the Choice Structure

"At each position k in s3, I must take the next character from either s1 or s2 — as long as it matches s3[k]. Two choices at every step suggests either backtracking or DP." Mention both and note DP wins here.

Step 2: Show Brute Force = O(2^(m+n)), then Spot Overlap

"Recursively, I'd try both choices. But dp(2,3) is reachable from dp(1,3)→take-s1 AND from dp(2,2)→take-s2. The state (i,j) = how many chars consumed from each string is computed multiple times. Memoize!"

Step 3: Define State Clearly

"State is (i, j): i chars consumed from s1, j from s2. Position in s3 is fully determined: i+j. So I need an (m+1)×(n+1) table. That's O(mn) — perfectly acceptable."

Step 4: Write Recurrence, Code, Mention Optimization

"dp[i][j] = (dp[i-1][j] and s1[i-1]==s3[i+j-1]) or (dp[i][j-1] and s2[j-1]==s3[i+j-1]). I fill base cases first, then interior row by row. Space can be reduced to O(n) with a rolling array."

🗣️ What to Say to the Interviewer

"Quick sanity check: if len(s1)+len(s2) != len(s3), return False immediately.
Then I notice: at each position in s3 I take from either s1 or s2.
State is (i, j) = how many chars consumed from each. Position in s3 = i+j.
This gives a 2D DP table. dp[i][j] = true iff i chars of s1 and j chars of s2
can form s3[0..i+j-1]. Recurrence: either extend from (i-1,j) via s1 or (i,j-1) via s2.
Fill base cases (edges), then interior. Time O(mn), Space O(n) with rolling array."
🔮 Likely Follow-up Questions
  • "Can you do O(n) space?" → Yes — rolling 1D array. dp[j] before update = dp[i-1][j]; dp[j-1] = dp[i][j-1] since already updated left-to-right.
  • "What if we have 3 source strings?" → Extend to 3D DP: dp[i][j][k], O(l·m·n) time and space.
  • "How does this relate to Edit Distance?" → Both are 2D DP on two strings. Edit Distance cells hold integers (min cost); this problem holds booleans (reachable).
  • "Could you reconstruct one valid interleaving?" → Yes — backtrack through the dp table, at each (i,j) check which predecessor was True and record the choice.

7 Summary & Key Takeaways ▾
State (i, j) fully encodes the subproblem — position in s3 is always i + j
Two transitions into every cell: from (i-1, j) via s1, or from (i, j-1) via s2
k = i+j-1 is the exact s3 index being matched at cell (i, j)
Space reduces from O(m·n) to O(n) with a rolling 1D array
Greedy fails: local matches can block globally valid paths — DP explores all paths implicitly
Memoization and tabulation both run O(m·n); tabulation has no stack overhead
🎯 When to Recognize This Pattern (2D DP on Two Strings)
  • Problem involves two strings and at each step you pick from one or the other
  • Position in the "result" equals the sum of positions in the two sources
  • Relative order within each source must be preserved (subsequence/merge structure)
  • Greedy choice fails — local matching doesn't guarantee global validity
  • Problem asks for existence (bool), count, or optimal cost — all fit this template
🧠 Mnemonic: "Two Roads into Every Room"

Visualize the DP table as a city grid. You start at (0,0) and want to reach (m,n). You can move right (take from s2) or down (take from s1). But a move is only allowed if the character at your current source position matches s3 at the combined depth. Room (i,j) is "reachable" if at least one of the two roads leading into it was passable. DP computes reachability for every room efficiently.
