LeetCode #289

Game of Life

Apply Conway's Game of Life rules to a 2D board simultaneously for all cells. Update in-place without using extra space.

Medium Matrix Encode States In-place Time: O(m×n) Space: O(1)
1 Understanding the Problem ▾

Conway's Game of Life is a cellular automaton. The board is a 2D grid of cells, each either alive (1) or dead (0). Every cell checks its 8 neighbors (horizontal, vertical, diagonal) and transitions according to four rules — all simultaneously.

The Four Rules:

  1. Live cell, < 2 live neighbors → dies (underpopulation)
  2. Live cell, 2 or 3 live neighbors → survives to next generation
  3. Live cell, > 3 live neighbors → dies (overpopulation)
  4. Dead cell, exactly 3 live neighbors → becomes alive (reproduction)
💡 Key Insight: The "Simultaneous" Constraint

Every cell's next state is computed from the original board, before any updates take effect. If you update cell (0,1) and then use that new value when computing (0,2)'s neighbors, you corrupt the result. The entire board must change at once — this is the real challenge of this problem.

✗ Sequential (Wrong)

Update (0,1). Then when computing (0,2), we read the NEW value of (0,1) as a neighbor. That's using the next generation's data — wrong answer!

✓ Simultaneous (Correct)

Encode the transition in the cell value. Neighbor-counting always recovers the original state. Decode at the end.

Edge Cases: Border cells have fewer than 8 neighbors — out-of-bounds positions are skipped. An all-dead board stays dead. An all-alive board has interior cells with 8 live neighbors (die by overpopulation) and border cells with varying counts.

2 The Algorithm: Encode States In-place ▾

Instead of copying the board, we use extra integer values to encode the transition directly into each cell. The trick: cells in the grid only use values 0 and 1 — we can use 2 and 3 as "encoded intermediate states" that still let us recover the original state during the scan.

The Four States:

0
Was dead → stays dead
1
Was alive → stays alive
2
Was alive → will die
3
Was dead → will be born

Original alive = v in (1, 2) · Original dead = v in (0, 3)

1 Pass 1 — Scan and Encode: For each cell, count live neighbors using board[nr][nc] in (1, 2) (recovers original aliveness even if cell was already encoded). Apply rules: if alive cell must die → mark 2; if dead cell must be born → mark 3. Alive-stays (1→1) and dead-stays (0→0) remain unchanged.
2 Pass 2 — Decode: After all cells have been scanned, do a second sweep. Remap: state 2 → 0 (dying → dead), state 3 → 1 (born → alive). States 0 and 1 are already correct final values. Done in O(mn) time, O(1) space.
🔑 Why check board[nr][nc] in (1, 2)?

After we encode some cells, their values become 2 (alive-dying) instead of 1. If we naively check == 1, we'd miss these cells and undercount live neighbors. Checking in (1, 2) says "was this cell originally alive?" — which is always correct regardless of encoding order.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At any point during Pass 1, for any cell with encoded value v: v in (1, 2) always and only returns True if the cell was originally alive. This is the bridge between the original board and the in-progress encoded board.

Why? We only assign 2 to cells that were 1 (alive). We only assign 3 to cells that were 0 (dead). So: originally-alive cells are always {1, 2}, and originally-dead cells are always {0, 3}. The mapping is bijective — no confusion possible.

Why two passes? During Pass 1, some cells still hold encoded values (2, 3). A cell to the right hasn't been visited yet, so if we decode early, we might change a cell's value before a later cell reads it as a neighbor. We wait until all neighbor-counting is done, then decode safely.

⚠️ Most Common Bug: Using == 1 for neighbor detection

Writing if board[nr][nc] == 1: live += 1 silently undercounts neighbors because cells already encoded as 2 (alive-dying) won't be counted. The bug is subtle: your code runs, produces a result, but it's wrong for any cell processed after a dying neighbor. Always use board[nr][nc] in (1, 2).

Alternative — Copy approach (O(mn) space, simpler): Snapshot the whole board first, then compute all next states from the snapshot. Easier to understand, but uses O(mn) extra space. Always worth mentioning in an interview as the "correct simple solution" before proposing the in-place trick.

4 The Code (Annotated) ▾

Solution 1 — Encode States In-place (Optimal, O(1) extra space) — Interview Pick

def gameOfLife(board): ← modifies board in-place; no return value
m, n = len(board), len(board[0]) ← grid dimensions (rows × cols)
dirs = [(-1,-1),(-1,0),(-1,1),(0,-1), ← all 8 neighbor direction offsets (dr, dc)
(0,1),(1,-1),(1,0),(1,1)]
for r in range(m): ← Pass 1: scan every cell row by row
for c in range(n):
live = 0 ← count original live neighbors
for dr, dc in dirs:
nr, nc = r + dr, c + dc
if 0 <= nr < m and 0 <= nc < n \← skip out-of-bounds neighbors
and board[nr][nc] in (1, 2): ← 1=alive-stays, 2=alive-dying; BOTH were originally alive!
live += 1
if board[r][c] == 1: ← cell is currently alive (not yet encoded)
if live < 2 or live > 3: ← rules 1&3: underpopulation or overpopulation
board[r][c] = 2 ← encode: was alive, will die (still looks alive to neighbors)
elif board[r][c] == 0: ← cell is currently dead (not yet encoded)
if live == 3: ← rule 4: exactly 3 neighbors → reproduction
board[r][c] = 3 ← encode: was dead, will be born (looks dead to neighbors)
for r in range(m): ← Pass 2: decode all intermediate states
for c in range(n):
if board[r][c] == 2: board[r][c] = 0 ← dying → finally dead
if board[r][c] == 3: board[r][c] = 1 ← born → finally alive
⚠️ Why don't we need to encode "stays alive" (1→1)?

A cell that stays alive keeps value 1 — which is already the correct final state. We only encode transitions (death and birth) because those are the cells whose interim encoded values (2, 3) would be misread by subsequent neighbor scans without encoding. "No change" cells are self-consistent.

Encode In-place: Time O(m×n) — two full passes · Space O(1) — no extra memory

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Spot "Simultaneous Update" as the Core Problem

The phrase "simultaneously" is the key signal. It means the next state of every cell depends on the current board state, not the partially-updated one. This is the entire difficulty — counting neighbors is trivial.

Step 2: Propose the Naive Solution First

"The simplest approach: copy the entire board. Compute next states from the copy, write into the original. O(mn) space, O(mn) time. Correct and readable. Should I code this up first, then optimize?"

Step 3: Identify the Space Bottleneck

"The only reason we need a copy is to remember 'was each cell originally alive or dead?' Can we encode that information inside the cell's value itself? The board only uses 0 and 1 — we have room for extra states."

Step 4: Design the Encoding Scheme

"State 2 = alive-dying (originally alive, still looks alive to neighbors via in (1,2)). State 3 = dead-born (originally dead, still looks dead via not in (1,2)). After one full scan, decode: 2→0, 3→1."

🗣️ What to Say to the Interviewer

"The key challenge is simultaneous update — I can't overwrite a cell before its neighbors have all read its original value.
Simple approach: snapshot the board in a copy. Read neighbors from copy, write to original. O(mn) space.
Optimal approach: encode transitions with extra states — 2 for alive→dies, 3 for dead→born.
During neighbor counting, check board[nr][nc] in (1, 2) to recover original aliveness.
One pass to encode, one pass to decode. O(mn) time, O(1) space."
🔮 Likely Follow-up Questions
  • "What if the board is infinite?" → Represent only live cells as a set of (r,c) tuples; iterate only neighbors of live cells
  • "What if only a few cells are alive (sparse board)?" → Same sparse representation — hash set; O(live_cells × 8) per step
  • "Can you do it with just 2 bits per cell instead of 2 states?" → Yes: use bit 0 = current state, bit 1 = next state; board[r][c] & 1 for original
  • "What if the board doesn't fit in memory?" → Process row by row, keeping 3 rows (prev, current, next) in memory at a time

7 Summary & Key Takeaways ▾
The "simultaneous update" requirement is the real challenge — all cells must see the original board, not a partially-updated one
Encode transitions with extra states: 2 = alive→dying, 3 = dead→born. States 0 and 1 need no encoding (already correct)
Invariant: board[r][c] in (1, 2) always recovers original aliveness, regardless of how many neighbors have already been encoded
Two passes: scan+encode (Pass 1), then decode (Pass 2). Total O(m×n) time, O(1) space
In interviews: always propose the O(mn) copy solution first, then offer the in-place encoding as an optimization

🎯 When to Recognize "Encode States In-place"
  • Problem says "in-place" and cells are binary (0/1) — room for extra integer states
  • Cells must update based on their neighbors' original values, not updated ones
  • "Simultaneously" appears in the problem description
  • You need to distinguish "was X, becomes Y" vs "was X, stays X" without a copy
  • Matrix problems where a second pass can decode the intermediate state
