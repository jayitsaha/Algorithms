LeetCode #25

Reverse Nodes in k-Group

Given the head of a linked list, reverse every k consecutive nodes. Leave a trailing group of fewer than k nodes in their original order.

Hard Linked List Reversal Time: O(n) Space: O(1)
1 Understanding the Problem ▾

You have a singly linked list and a number k. Walk through the list in chunks of exactly k nodes. For each chunk that is a full k nodes long, reverse the order of those nodes in-place. If the last chunk has fewer than k nodes, leave it unchanged.

💡 Key Insight

The reversal of k nodes is just the classic Reverse Linked List applied repeatedly. The real challenge is the pointer surgery: you must stitch each reversed group cleanly back into the surrounding list without losing track of what came before and what comes next.

Input: 1→2→3→4→5, k=2

Group 1: [1, 2] — full, reverse it
Group 2: [3, 4] — full, reverse it
Group 3: [5] — only 1 < 2, skip

Output: 2→1→4→3→5

Group 1 reversed: [2, 1]
Group 2 reversed: [4, 3]
Remainder unchanged: [5]

Input: 1→2→3→4→5, k=3

Group 1: [1, 2, 3] — full, reverse
Group 2: [4, 5] — only 2 < 3, skip

Output: 3→2→1→4→5

Group 1 reversed: [3, 2, 1]
Remainder unchanged: [4, 5]

Edge cases to consider: k=1 (no reversal needed, list unchanged), k equals list length (whole list reversed), list length not divisible by k (trailing remainder kept as-is), single-node list.

2 The Algorithm: Dummy-Head Iterative Reversal ▾

We iterate through the list with a dummy head to avoid special-casing the real head. Each iteration runs three tightly defined phases: Check, Reverse, and Connect.

1 Dummy head setup: Create a dummy node with dummy.next = head. Track prev_tail = dummy — this is the tail of the most recently completed reversed group. Initially nothing is completed, so it starts at dummy.
2 Phase 1 — Check: Walk k steps forward from prev_tail. If we hit None at any step, fewer than k nodes remain. Return dummy.next immediately — the remainder is left untouched.
3 Phase 2 — Reverse: Save group_start = prev_tail.next (this node will become the group tail after reversal). Run the classic 3-pointer reversal exactly k times. Afterwards: prev = new group head, cur = first node of the next group.
4 Phase 3 — Connect: Three pointer assignments stitch everything together: (1) prev_tail.next = prev — attach previous segment to the new head; (2) group_start.next = cur — attach the new tail to the next segment; (3) prev_tail = group_start — advance prev_tail to the tail of this now-complete group.
🔑 Why the Dummy Head is Essential

Without dummy, reversing the first group creates a new head and you'd need a special case to update the head pointer. With dummy, prev_tail = dummy treats the first group the same as every subsequent group — prev_tail.next = prev just works. At the end, dummy.next is the new real head.

3 Why It Works: The Invariant ▾
🔐 The Loop Invariant

At the start of every iteration: prev_tail is the last node of the already-processed, correctly-ordered prefix. Everything to its left is finalized and correct. prev_tail.next is the first unprocessed node.

Why the 3-pointer reversal is correct: Starting with prev=None, cur=group_start and running k steps of (nxt=cur.next; cur.next=prev; prev=cur; cur=nxt) flips k pointers. After k steps, prev is the node that was k-th (now the group's new head) and cur is the node that was (k+1)-th — exactly the start of the next group. The original group_start now has next=None, making it the group's new tail.

Why the stitching is provably correct: After reversal we have four anchors:

  • prev_tail — last confirmed node of the processed prefix
  • prev — new head of this reversed group
  • group_start — new tail of this reversed group
  • cur — first node of the next group

Two assignments reconnect the list: prev_tail.next = prev (prefix → new head) and group_start.next = cur (new tail → next group). The invariant is then restored by setting prev_tail = group_start.

⚠️ The Critical Order of Operations

You MUST save group_start = prev_tail.next BEFORE starting the reversal loop. During reversal, cur walks forward and prev_tail.next is not yet updated. If you forget to save group_start, you lose the reference to the group tail (the old group head) and cannot complete the stitching step.

4 The Code (Annotated) ▾

Solution 1 — Iterative with Dummy Head (Optimal — Interview Pick)

def reverseKGroup(head, k):
dummy = ListNode(0) ← Fake node before head. Eliminates head-update special case.
dummy.next = head
prev_tail = dummy ← Tail of last completed group. Starts at dummy (nothing complete yet).
while True:
# Phase 1: verify k nodes exist
check = prev_tail ← Walk k steps to ensure a full group is available
for _ in range(k):
check = check.next
if check is None: ← Hit null before k steps → remainder group, stop
return dummy.next
# Phase 2: reverse k nodes
group_start = prev_tail.next ← SAVE this — becomes the group tail after reversal
prev, cur = None, group_start ← Classic reversal initialization
for _ in range(k): ← Reverse exactly k pointer links
nxt = cur.next ← Save next before overwriting cur.next
cur.next = prev ← Flip: point current node backward
prev = cur
cur = nxt ← After k steps: prev=new head, cur=next group start
# Phase 3: stitch back
prev_tail.next = prev ← Connect previous segment to reversed group's new head
group_start.next = cur ← Connect reversed group's new tail to next segment
prev_tail = group_start ← Advance: group_start is now the tail of this completed group
⚠️ Common Bug: Missing group_start.next = cur

After the reversal loop, group_start.next still points wherever it pointed before — which is now in the middle of the reversed group, creating a cycle or dangling pointer. The line group_start.next = cur is mandatory. Easy to forget because group_start now looks like an "internal" node but it's actually the tail.

Solution 2 — Recursive (elegant, O(n/k) stack depth)

def reverseKGroup(head, k):
cur, count = head, 0 ← Count k nodes to verify enough exist
while cur and count < k:
cur = cur.next; count += 1
if count < k: ← Base case: fewer than k nodes remain
return head ← Return as-is (the remainder)
prev, cur = None, head ← Reverse first k nodes in place
for _ in range(k):
nxt = cur.next
cur.next = prev
prev = cur; cur = nxt
head.next = reverseKGroup(cur, k) ← head is now the tail; attach result of recursion
return prev ← prev is the new head of this reversed group
💡 Complexity Summary

Iterative: Time O(n), Space O(1) — optimal
Recursive: Time O(n), Space O(n/k) — O(n/k) stack frames (one per group)
Collect-and-rebuild: Time O(n), Space O(n) — convert list to array, reverse chunks, rebuild — simplest to code but disqualified by O(1) space requirement

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Core Pattern

Signals in the problem: "linked list", "reverse", "k consecutive nodes", "in-place". This is Reverse Linked List #206 applied repeatedly. If you can reverse a whole list, you can reverse a segment. The novelty here is the stitching that reconnects segments.

Step 2: Decompose into Two Sub-Problems

Sub-problem A — How to reverse k nodes: standard 3-pointer reversal (prev=None, cur=start, run k steps). Sub-problem B — How to reconnect: need four anchors — (prev_tail, new_head=prev, new_tail=group_start, next_segment=cur). Wire: prev_tail→new_head, new_tail→next_segment.

Step 3: Mention the Dummy Head Pattern

Tell the interviewer: "I'll use a dummy node before the list. This lets prev_tail start at dummy, so the first group is handled identically to all subsequent groups — no special-casing the head update."

Step 4: Name the Three Phases Explicitly

Walk through the loop structure: (1) Check phase — walk k steps to confirm a full group. (2) Reverse phase — 3-pointer reversal x k. (3) Connect phase — three pointer assignments, advance prev_tail. This shows structured thinking.

🗣️ What to Say to the Interviewer

"I'll build on the standard linked list reversal. I use a dummy node before the head.
In a loop: first I walk k steps to confirm k nodes exist — if not, return.
Then I reverse those k nodes in-place with a 3-pointer approach.
After reversal: prev holds the new head, group_start (saved before reversing) holds the new tail, cur is the next segment start.
Three wiring steps reconnect everything. Then I advance prev_tail and repeat.
Time O(n), Space O(1)."
🔮 Likely Follow-up Questions
  • "Swap Nodes in Pairs (#24)?" → Same problem, k=2 hardcoded — simpler version
  • "What if k=1?" → Each group has 1 node; reversal is a no-op; list unchanged
  • "Can you do it recursively?" → Yes: reverse first k, recurse on rest, set head.next to result. O(n/k) stack.
  • "What if you want to reverse only alternate groups?" → Track group index; skip reversal for odd groups
  • "What if the remainder should ALSO be reversed?" → Remove the early return; always reverse even incomplete groups

7 Summary & Key Takeaways ▾
Always use a dummy head for in-place linked list restructuring — it eliminates head-update special cases
The three phases per group are: Check (verify k nodes), Reverse (3-pointer flip), Connect (4-anchor stitching)
Save group_start BEFORE reversal — it becomes the group tail and needs explicit wiring to the next segment
After reversing k nodes: prev = new head, group_start = new tail, cur = next segment start
Each node is visited at most twice (once in check, once in reversal) → O(n) time, O(1) space
The remainder is handled cleanly: the check phase returns before any reversal when fewer than k nodes remain
ApproachTimeSpaceNotes
Iterative + Dummy ✓ O(n) O(1) Interview pick; handles all edge cases cleanly
Recursive O(n) O(n/k) Elegant; stack depth = number of full groups
Collect + Rebuild O(n) O(n) Convert to array, reverse chunks, rebuild list
