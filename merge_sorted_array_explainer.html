LeetCode #88

Merge Sorted Array

Merge two sorted arrays nums1 and nums2 in-place into nums1, which has enough trailing space to hold the result.

Easy Two Pointers Merge from End Time: O(m+n) Space: O(1)
1 Understanding the Problem ▾

You are given two sorted integer arrays. nums1 has size m+n — the first m elements are real, the trailing n elements are zeros (placeholder space). nums2 has exactly n elements. Your task: merge both into nums1 in non-decreasing order. No new array allowed — everything must happen in-place.

💡 Key Insight

The n trailing zeros in nums1 are empty seats reserved exactly for this merge. If we fill those seats from the rightmost position backward — always placing the larger of the two current tails — we never overwrite any real data that we haven't yet read. This turns an in-place merge (normally requiring O(n) extra space) into an O(1) space operation.

✗ Naive: copy then sort

nums1 = [1,2,3,0,0,0]
nums2 = [2,5,6]
→ Copy nums2 into zeros
→ nums1.sort()
O((m+n) log(m+n)) — wastes sorted property

✓ Optimal: three pointers, fill from end

p=2 (tail of nums1 reals)
q=2 (tail of nums2)
k=5 (write cursor, rightmost)
O(m+n) time, O(1) space

Concrete trace (nums1=[1,2,3,_,_,_], nums2=[2,5,6]): Compare tails: 3 vs 6 → place 6 at k=5. Then 3 vs 5 → place 5 at k=4. Then 3 vs 2 → place 3 at k=3. Then 2 vs 2 → place 2 (nums2) at k=2. Main loop ends (nums2 exhausted). nums1's remaining [1,2] are already in positions 0 and 1 — no work needed. Result: [1,2,2,3,5,6].

Edge cases: If n=0, nums2 is empty — nums1 is already sorted, do nothing. If all of nums1's real elements are smaller than nums2 (e.g., [1,2,3,_,_,_] vs [4,5,6]), the main loop exhausts p first, and the mop-up while q >= 0 copies the rest of nums2.

2 The Algorithm: Three-Pointer Merge from End ▾

The algorithm uses three pointers that all start at their respective tails and move leftward. The write cursor k fills slots right-to-left; the read cursors p and q track the unread portion of each array. At every step, we pick the maximum of the two current read positions and write it to k.

1 Initialize: p = m - 1 (last real element of nums1), q = n - 1 (last element of nums2), k = m + n - 1 (last slot of nums1 — the rightmost empty seat).
2 Main loop (while p ≥ 0 and q ≥ 0): Compare nums1[p] and nums2[q]. Place the larger at nums1[k]. Decrement the pointer for whichever array we drew from, and always decrement k.
3 Mop-up remaining nums2: After the main loop, if q ≥ 0, copy nums2[0..q] → nums1[0..q]. These are the smallest elements — they belong at the front. If only nums1 elements remain (p ≥ 0 but q < 0), they're already in-place — nothing to do.

🔑 Why Backward Merging Avoids Overwrites

When we write to nums1[k], we need that slot to be either an empty zero or a real element we have already placed. Since k starts at m+n-1 and p starts at m-1, the write cursor always leads (or ties) the read cursor: at any point k ≥ p. So writing to nums1[k] never clobbers nums1[p] — the read position is always to the left of or at the write position.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every point in the algorithm, nums1[k+1 .. m+n-1] contains the (m+n-1-k) largest elements from the combined arrays, in sorted order. Equivalently: the right portion of nums1 is always a correct suffix of the final merged result. The unread portions nums1[0..p] and nums2[0..q] together supply exactly the elements still needed.

Why placing the maximum preserves the invariant: Both input arrays are sorted ascending, so their tails (nums1[p] and nums2[q]) are the largest unread elements in each. The maximum of these two is the largest unread element globally. Placing it at nums1[k] — the next position to the left of the already-placed suffix — maintains the sorted property.

No overwrite conflict: We proved k ≥ p always. More precisely, at step t we have written t elements total and have moved k down by t, while p has moved down by at most t (since we also advance q). So k ≥ p holds throughout, and a write to nums1[k] never destroys an unread nums1[p].

⚠️ The Mop-Up Bug — Most Common Interview Mistake

Forgetting the while q >= 0 copy is the #1 error. Example: nums1=[4,5,6,_,_,_], nums2=[1,2,3]. The main loop exhausts p first (6,5,4 all placed), leaving q=2 pointing to nums2[2]=3. Without mop-up, positions 0-2 of nums1 still hold [4,5,6] — the wrong answer. Always add the mop-up for nums2.

💡 Why the nums1 mop-up is a no-op

If only nums1 elements remain (q < 0, p ≥ 0), those elements are sitting at nums1[0..p]. They are the smallest remaining elements, and positions 0..p are exactly where they should end up. nums1[i] = nums1[i] for all i — nothing to copy. This is the "free" part of merging in-place.

4 The Code (Annotated) ▾

Solution 1 — Brute Force: Copy then Sort

def merge(nums1, m, nums2, n):
nums1[m:m+n] = nums2 ← Overwrite trailing zeros with nums2 values
nums1.sort() ← Python Timsort: O((m+n) log(m+n)) — correct but slow

Solution 2 — Optimal: Merge from End (Interview Pick)

def merge(nums1, m, nums2, n):
p = m - 1 ← Last real element index in nums1
q = n - 1 ← Last element index in nums2
k = m + n - 1 ← Rightmost slot in nums1 (our write cursor)
while p >= 0 and q >= 0: ← Loop while both have unread elements
if nums1[p] > nums2[q]: ← Compare current tails
nums1[k] = nums1[p] ← nums1's tail wins → write it to slot k
p -= 1 ← Retreat nums1 read pointer
else: ← nums2's tail wins (or tie — doesn't matter)
nums1[k] = nums2[q] ← Write nums2 tail to slot k
q -= 1 ← Retreat nums2 read pointer
k -= 1 ← Always move write cursor left after a placement
while q >= 0: ← Mop-up: nums2 still has elements remaining
nums1[k] = nums2[q] ← All leftover nums2 are smaller than placed elements
q -= 1
k -= 1
💡 Complexity Summary

Brute Force (copy + sort): Time O((m+n) log(m+n)), Space O(1) — sorts in-place via Timsort
Merge from End: Time O(m+n) — each of the m+n elements is placed exactly once; Space O(1) — three integer pointers only
Interview Pick: Merge from End — optimal on both dimensions

⚠️ Tie-Breaking: > vs >=

When nums1[p] == nums2[q], placing either is correct — the values are identical. The code uses strict > so ties go to the else branch (nums2). Flipping to >= routes ties to nums1 — both produce valid merged arrays. Don't overthink it; just be consistent.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: State the Naive Approach

"The easiest approach: copy nums2 into the trailing zeros of nums1, then call sort(). That's O((m+n) log(m+n)). It works, but it ignores that both arrays are already sorted — we're throwing away useful structure."

Step 2: Identify the Forward-Merge Problem

"Can I merge from the front like a standard merge? I'd write to nums1[0], but nums1[0] already holds a real element. I'd need to shift everything over or use extra storage — no good for O(1) space."

Step 3: Recognize the Empty-Seats Observation

"But those n trailing zeros are empty seats! If I fill from the right, I write into the zeros first. By the time my write cursor reaches real nums1 data, I've already moved the read pointer past it. No conflict."

Step 4: Don't Forget the Mop-Up

"After the main loop, if nums2 has leftover elements (q ≥ 0), they must be ≤ everything placed — copy them to the front. If only nums1 has leftovers, they're already in-place — skip."

🗣️ What to Say to the Interviewer

"nums1 has m+n space, m real elements, n trailing zeros.
Naive: copy nums2 in, sort. O((m+n) log(m+n)). Can we do O(m+n)?
Key insight: fill from the end — the zeros are scratch space.
Three pointers: p = m-1 (nums1 tail), q = n-1 (nums2 tail), k = m+n-1 (write).
Each iteration: place max(nums1[p], nums2[q]) at nums1[k]; decrement the source and k.
Loop until one pointer goes below 0.
Mop-up: if q >= 0, copy nums2[0..q] → nums1[0..q]. Nums1 leftovers: already in-place.
Time O(m+n), Space O(1)."
🔮 Likely Follow-up Questions
  • "What if nums1 has no trailing space?" → Need O(m+n) extra array; copy both arrays in and merge.
  • "Merge k sorted arrays?" → Min-heap of (value, array_idx, elem_idx), O(N log k) (#23).
  • "What if arrays contain duplicates?" → The algorithm handles this naturally; ties are fine.
  • "Can m or n be 0?" → Yes. n=0: do nothing. m=0: nums1 = nums2 after mop-up.

7 Summary & Key Takeaways ▾
Fill from the END — the trailing zeros are reserved scratch space, so no real data gets clobbered
Three pointers: p (nums1 tail, starts m-1), q (nums2 tail, starts n-1), k (write cursor, starts m+n-1)
Each element is placed exactly once → O(m+n) time; three pointers only → O(1) space
Mop-up only needed for leftover nums2 — remaining nums1 elements are already in correct positions
Tie-breaking is irrelevant — either choice produces a valid sorted result

🎯 When to Recognize This Pattern
  • "Merge two sorted arrays in-place, one has extra capacity" → Three-pointer merge from end
  • "Write backward to avoid overwriting unread data" → Signature of merge-from-end
  • "One array has n trailing zeros or null slots" → Use them as write buffer
  • Any in-place merge where a read-write conflict prevents forward merging
