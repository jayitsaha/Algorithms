LeetCode #340

Longest Substring with At Most K Distinct Characters

Given string s and integer k, return the length of the longest substring that contains at most k distinct characters.

Medium Sliding Window Hash Map ≤ K Keys Time: O(n) Space: O(k)
1 Understanding the Problem ▾

A substring is a contiguous slice of a string — "ece" in "eceba" is a substring, but "ea" is not (non-contiguous). We want the longest such slice that uses at most k different characters. The window can repeat characters freely; only the count of distinct characters must stay ≤ k.

💡 Key Insight

The window should grow greedily: expand right as long as distinct characters ≤ k. The instant we hit k+1 distinct, we shrink from the left — evicting the leftmost character — until the window is valid again. The longest valid window seen is our answer.

✓ Valid window (k=2): "ece"

map = {e:2, c:1}
distinct = 2 ≤ k=2
length = 3 ← answer

✗ Invalid window: "eceb"

map = {e:2, c:1, b:1}
distinct = 3 > k=2
Must shrink left →

Edge cases: k = 0 → answer is 0 (no chars allowed). k ≥ total distinct chars in s → answer is len(s). Empty string → 0. Single character, any k ≥ 1 → 1. All same characters "aaaa" → len(s).

2 The Algorithm: Variable Sliding Window + Hash Map ▾

We maintain a window defined by two pointers left and right, and a hash map char_count tracking how many times each character appears in the current window. The map's key count equals the number of distinct characters — that's what we constrain.

1 Initialize: left = 0, result = 0, char_count = {}. The window is empty and sits at the start of the string.
2 Expand right: Advance right one position. Add s[right] to char_count (increment its count). The window now includes this character.
3 Shrink if over budget: While len(char_count) > k, decrement char_count[s[left]]. If the count reaches 0, delete the key entirely (so the map's length drops). Advance left += 1. Repeat until ≤ k distinct.
4 Update result: The window [left, right] is now guaranteed valid. Update result = max(result, right - left + 1).
5 Return: After right has scanned all of s, return result — the length of the longest valid window we ever saw.
Window trace for s = "eceba", k = 2
right=0 "e" → {e:1} — 1 distinct ✓ len=1
right=1 "ec" → {e:1,c:1} — 2 distinct ✓ len=2
right=2 "ece" → {e:2,c:1} — 2 distinct ✓ len=3 ← BEST
right=3 "eceb" → {e:2,c:1,b:1} — 3 distinct ✗ SHRINK
→ shrink "eb" → {e:1,b:1} — 2 distinct ✓ len=2
right=4 "eba" → {e:1,b:1,a:1} — 3 distinct ✗ SHRINK
→ shrink "ba" → {b:1,a:1} — 2 distinct ✓ len=2
🔑 Why "at most" means a variable window

Fixed-size windows always slide by exactly 1. Variable windows stay as large as possible — we only shrink when forced. This greediness is correct because a longer valid window is always at least as good as a shorter one.

3 Why It Works: The Invariant ▾
🔐 The Invariant

After every complete iteration (expand then shrink-loop), the window [left, right] always satisfies len(char_count) ≤ k. The result is updated after the shrink-loop, so it only ever stores the size of a valid window.

Why the shrink-loop restores validity: Each shrink removes one occurrence of s[left]. When count drops to 0, that key is deleted, reducing len(char_count) by 1. We repeat until the map has ≤ k keys. The invariant is re-established before we record the result.

Why we never miss the optimum: right visits every index once. For each right, left is as small as possible (window as large as possible) while keeping ≤ k distinct. Any shorter window ending at right would be suboptimal. Since we check every right, we can't miss the global optimum.

⚠️ The Delete-on-Zero Trap

When a character's count reaches 0 after decrement, you must delete it from the map. Leaving char_count[c] = 0 means len(char_count) still counts that character as distinct. The shrink-loop will keep running past where it should stop, giving a shorter answer than optimal — or an infinite loop in degenerate cases.

Time complexity O(n): Each character is added once (right always moves right, n total increments) and removed at most once (left never moves right past right, n total decrements). Total work = O(2n) = O(n).
Space complexity O(k): After each shrink the map has ≤ k keys. During a shrink it briefly has k+1 before one is deleted — still O(k).

4 The Code (Annotated) ▾

Solution 1 — Variable Sliding Window (Optimal, interview pick)

def lengthOfLongestSubstringKDistinct(s: str, k: int) -> int:
if k == 0: return 0 ← Edge case: 0 distinct chars allowed → only empty window is valid
char_count = {} ← char → count inside window; len(char_count) = distinct chars
left = result = 0 ← left boundary of window; best length found so far
for right in range(len(s)): ← right expands one step per iteration; never goes backwards
c = s[right] ← current character entering the window
char_count[c] = char_count.get(c, 0) + 1← add c to window; get(c,0) handles first occurrence gracefully
while len(char_count) > k: ← window has too many distinct chars — must shrink
lc = s[left] ← character at left boundary being evicted
char_count[lc] -= 1 ← one fewer occurrence of lc in window
if char_count[lc] == 0: ← lc fully evicted from window...
del char_count[lc] ← ...remove the key so distinct count drops by 1
left += 1 ← advance left boundary; window shrinks by 1
result = max(result, right - left + 1) ← window is valid here; record its size if best
return result ← longest valid window length across all positions of right
⚠️ While vs If for the shrink loop

Using if instead of while would only shrink by one step. If k=1 and we have a window "aaa" then encounter 'b', we'd need to evict all three 'a's before 'b' becomes the sole distinct character. An if would leave 2 'a's in the map with 2 distinct chars, violating the invariant.

Solution 2 — Brute Force: All Substrings (O(n²) time)

def lengthOfLongestSubstringKDistinct(s: str, k: int) -> int:
result = 0
for i in range(len(s)): ← try each starting index
seen = {} ← distinct char count for s[i..j]
for j in range(i, len(s)): ← extend substring rightward
seen[s[j]] = seen.get(s[j], 0) + 1
if len(seen) > k: break ← too many distinct → this start is exhausted
result = max(result, j - i + 1)
return result
💡 Complexity Comparison

Sliding Window (Optimal): Time O(n) — left and right each move at most n steps total. Space O(k).
Brute Force: Time O(n²) — O(n) starting positions × O(n) inner scan. Space O(k).
Fixed array (ASCII-only): Replace dict with int[256]; same asymptotic but better cache locality for small alphabets.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Pattern Recognition

"Longest substring" + an upper-bound constraint = sliding window. The words "at most k distinct" tell you the constraint and what to track. A hash map will count distinct chars in O(1) per operation.

Step 2: Start with Brute Force

"Naively I'd try all O(n²) substrings, extend until distinct > k, track the max. That's O(n²). I can do better with a sliding window that avoids re-scanning the left portion."

Step 3: Explain the Two-Pointer Design

"Right always moves forward one step. The map tells me instantly if the window is over-budget. When it is, I shrink from left until the map is back to ≤ k keys. Both pointers move at most n steps total → O(n)."

Step 4: Mention the Delete-on-Zero Rule

"Critical detail: when a character's count hits 0, I delete it from the map. Otherwise len(map) is wrong and I'll over-shrink or under-shrink. This is the most common bug in sliding window problems."

🗣️ What to Say to the Interviewer

"This is a variable sliding window. I'll use a hash map to track character counts.
Right expands one step at a time. When the map exceeds k keys, I shrink from left:
decrement count, delete key if it hits zero, advance left — repeat until ≤ k distinct.
After each right step the window is valid, so I update the max length.
Edge case: k=0 → return 0. Both pointers move at most n steps → O(n) time, O(k) space.
Shall I code it?"

7 Summary & Key Takeaways ▾
Variable sliding window: expand right freely, shrink left only when constraint is violated
Hash map tracks char counts; len(map) = distinct chars in window — O(1) to check
Always delete a key when its count hits 0 — otherwise distinct count is wrong
Each character enters and exits the window at most once → O(n) total time
Use a while-loop (not if) for shrinking — may need multiple left advances per right step
ApproachTimeSpaceNotes
Brute Force (all substrings) O(n²) O(k) Try every start, extend until k+1 distinct
Sliding Window + HashMap ✓ O(n) O(k) Optimal; handles any alphabet size
Sliding Window + Fixed Array O(n) O(1)=O(256) Better constants; only for bounded alphabets
🎯 When to Recognize the Variable Sliding Window
  • "Longest / shortest contiguous substring satisfying [constraint]"
  • Constraint is an upper or lower bound (not fixed) → window size varies
  • Need to track a property of the window contents → use a map or counter
  • O(n) expected → each element enters/exits window once
