LeetCode #35

Search Insert Position

Given a sorted array of distinct integers and a target value, return the index if found, or the index where it would be inserted to maintain sorted order.

Easy Binary Search Lower Bound Time: O(log n) Space: O(1)
1 Understanding the Problem ▾

You have a sorted array of distinct integers and a target. The task: find where target "belongs." If it is already in the array, return its current index. If it is not there, return the index where you would insert it to keep the array sorted.

💡 Key Insight

This is a lower bound search — find the first index i such that nums[i] >= target. Whether target exists or not, when binary search exits, lo always lands exactly on this index. No special "found vs not found" branching is needed at the end.

✓ Target Found: nums=[1,3,5,6], target=5

5 exists at index 2
Binary search hits it exactly → 2

✗ Target Missing: nums=[1,3,5,6], target=2

2 is not in the array
1 < 2 < 3 → insert at index 1

Edge cases to consider: target smaller than all elements (return 0), target larger than all (return n), single-element array where target equals or doesn't equal the element.

What "insert position" means for nums=[1,3,5,6]
target = 0
→ insert at index 0 (before all elements)
target = 2

2 The Algorithm: Binary Search — Lower Bound ▾

Binary search works by repeatedly halving the search space. At each step, we compare the middle element to target and eliminate the half that can't contain the answer. The key elegance here: when binary search exits, lo is the exact answer whether the target was found or not.

1 Initialize pointers: lo = 0, hi = len(nums) - 1. The active search window is the closed interval [lo, hi] — both endpoints are included.
2 Compute mid: mid = lo + (hi - lo) // 2. The subtraction form avoids integer overflow in languages with fixed-width integers (habit from C/Java; in Python this doesn't matter, but show you know it).
3 Compare and move: If nums[mid] == target, return mid. If nums[mid] < target, the answer is strictly to the right → set lo = mid + 1. If nums[mid] > target, the answer is at mid or to the left → set hi = mid - 1.
4 Return lo: When lo > hi, the loop ends. At this point, lo is the first index where nums[lo] >= target — the exact insert position. No additional branching needed.
🔑 Why Return lo, Not hi or hi+1?

When the loop exits, lo == hi + 1 (they crossed). The lo pointer only advances when nums[mid] < target — meaning everything below lo is definitely too small. So lo sits at the first position that is not less than target. That is precisely the insert position. hi sits one below — last position too small. Either lo or hi + 1 works, but lo is cleaner.

Why O(log n): Each iteration cuts the window in half. Starting from n elements, after k iterations we have n/2ᵏ. We stop when this reaches 1, so k = log₂(n). Space is O(1) — only three integer variables: lo, hi, mid.

3 Why It Works: The Invariant ▾
🔐 The Binary Search Invariant

At all times during the loop, the answer (the insert position) lies in the range [lo, hi + 1]. Initially this is [0, n], covering all valid positions. Each step eliminates half the range while preserving this invariant. When lo > hi, the range collapses to a single point: lo. That is the answer.

Why each branch preserves the invariant:

  • nums[mid] == target → exact match at mid; return it immediately.
  • nums[mid] < target → target cannot be at mid or any index below mid (array is sorted). So set lo = mid + 1. The insert position is still somewhere in [lo, hi+1] = [mid+1, hi+1].
  • nums[mid] > target → target (if present) is at index ≤ mid; the insert position could be at mid (if target goes right before nums[mid]). Set hi = mid - 1. Range is now [lo, mid].
⚠️ Classic Bug: hi = mid Instead of hi = mid - 1

With the while lo <= hi form, always use hi = mid - 1 when narrowing left. Using hi = mid risks an infinite loop: when lo == hi == mid, we'd check the same element forever without making progress. The lo <= hi form requires strict reduction on both sides.

Boundary proofs: If target is larger than all elements, lo keeps advancing until lo = n = len(nums). That is the correct insert position (append at end). If target is smaller than all, hi retreats below 0 on the first iteration, and lo stays at 0. Correct (prepend at front).

Termination guarantee: Each iteration either returns early (exact match) or strictly shrinks the window (lo increases OR hi decreases). The window cannot stall, so the loop always terminates in at most ⌈log₂(n+1)⌉ iterations.

4 The Code (Annotated) ▾

Solution 1 — Binary Search (Optimal, O(log n) time, O(1) space) — Interview Pick

def searchInsert(nums, target): ← nums is sorted, all distinct
lo, hi = 0, len(nums) - 1 ← closed interval [lo, hi], both inclusive
while lo <= hi: ← loop while window is non-empty
mid = lo + (hi - lo) // 2 ← midpoint; (hi-lo)//2 avoids integer overflow
if nums[mid] == target: ← exact match: target is in the array
return mid ← return its current index
elif nums[mid] < target: ← mid is too small; answer is to the right
lo = mid + 1 ← eliminate mid and everything left of it
else: ← nums[mid] > target; answer is at mid or left
hi = mid - 1 ← eliminate everything right of mid
return lo ← lo = first index where nums[lo] >= target
⚠️ Why lo, Not hi or mid?

When the loop ends: lo == hi + 1. lo is the first index that was never eliminated because it was "too small." That makes it exactly the first index ≥ target. hi = lo − 1, which is the last index that is strictly less than target — off by one. Writing hi + 1 also works but is needlessly verbose.

Solution 2 — Linear Scan (Brute Force, O(n) — mention to show contrast)

def searchInsert(nums, target):
for i, n in enumerate(nums): ← scan left to right through sorted array
if n >= target: ← first element >= target is the insert position
return i ← handles both exact match (n==target) and gap
return len(nums) ← target > all elements; insert at the end
💡 Complexity Summary

Binary Search (optimal): Time O(log n), Space O(1) — always use this
Linear Scan: Time O(n), Space O(1) — correct but ignores sorted order
Python built-in: bisect.bisect_left(nums, target) — O(log n), but implement manually in interviews

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Spot the Binary Search Signal

The problem says sorted array + find position. That is the canonical signal for binary search. Sorted input means we can eliminate half the candidates at every step — O(log n) is achievable and expected. If an interviewer says "sorted" and you write O(n), you've missed the point.

Step 2: Reframe as "First Index Where nums[i] >= target"

Whether or not target is in the array, the answer is the same: the first position that is not less than target. Recognizing this unifies "found" and "not found" into a single clean case — you don't need a post-loop check to see if target actually exists.

Step 3: Propose Brute Force, Then Optimize

"A linear scan finds the first element ≥ target in O(n). But the array is sorted, so I can use binary search for O(log n). Let me implement that." This shows methodical thinking even for an Easy problem.

Step 4: Implement Carefully and Narrate

Call out: "I use inclusive bounds [lo, hi]. I compute mid as lo + (hi-lo)//2. When nums[mid] is less than target, I move lo right; when greater, hi left; when equal, return immediately. On loop exit, lo is the answer — no extra branch needed."

🗣️ What to Say to the Interviewer

"The sorted array tells me to use binary search for O(log n).
I'll maintain a closed window [lo, hi]. Each step I check the midpoint.
If nums[mid] equals target: found it, return mid.
If nums[mid] is less than target: answer is to the right, set lo = mid + 1.
If nums[mid] is greater: answer is here or left, set hi = mid - 1.
When lo exceeds hi, lo is the first index where nums[lo] >= target.
That's the insert position — handles both found and not-found in one return."

7 Summary & Key Takeaways ▾
Sorted array + "find a position" = binary search. Always O(log n).
Lower bound: find first index i where nums[i] >= target. That is both the found-index and the insert-position in one.
When the loop exits, lo is the answer — no separate "found vs not found" check needed.
Use mid = lo + (hi - lo) // 2 to avoid integer overflow (good habit from C/Java).
This lower-bound template is the foundation for Find First/Last Position, First Bad Version, and many medium/hard binary search variants.

🎯 When to Recognize Binary Search (Lower Bound)
  • Input array is sorted (ascending or descending)
  • Looking for a value or the first position satisfying a condition
  • O(n) is too slow; O(log n) is expected or hinted
  • "First index where condition is true" → lower bound (lo = mid + 1 when false)
  • "Last index where condition is true" → upper bound (hi = mid when true, different form)
