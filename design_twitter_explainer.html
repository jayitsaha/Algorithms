LeetCode #355

Design Twitter

Design a simplified Twitter: users can post tweets, follow/unfollow, and retrieve the 10 most recent tweets from their news feed.

Medium Design Hash Maps + Heap Merge postTweet: O(1) getNewsFeed: O(F log F)
1 Understanding the Problem ▾

We implement a Twitter class with four methods. Three are trivial — the fourth is not:

  • postTweet(userId, tweetId) — store a tweet, O(1) expected
  • follow(followerId, followeeId) — record a social link, O(1)
  • unfollow(followerId, followeeId) — remove a social link, O(1)
  • getNewsFeed(userId) — return the 10 most recent tweet IDs across everyone the user follows, plus their own tweets, sorted newest-first
💡 Key Insight

Each user's tweet history is already sorted by time. Getting a merged feed is exactly the "Merge K Sorted Lists" problem. A max-heap on timestamps lets us always pick the globally newest tweet without loading all tweets into memory at once.

Data Structures

tweets: Dict[uid → list[(ts, tweetId)]]
follows: Dict[uid → Set[followeeId]]
self.ts: int # global clock

Merge K Sorted Lists

Each user's list = sorted by timestamp
Heap holds 1 entry per followee
Pop → emit → push next from same user

Edge Cases: User with no tweets (skip when seeding heap). Fewer than 10 total tweets (return what exists). User unfollows someone they never followed (no-op). User's own tweets always appear in their own feed.

2 The Algorithm: HashMap + Max-Heap Merge ▾

The design splits cleanly into write path (fast, O(1)) and read path (the interesting part — lazy heap merge).

tweets HashMap
userId → list of (timestamp, tweetId). Append on post. Tail = most recent.
follows HashMap
userId → Set of followeeIds. Set gives O(1) follow/unfollow/membership.
Max-Heap (on read)
Holds one entry per followee: (−ts, tweetId, userId, listIndex). Created fresh each getNewsFeed call.
1 postTweet(uid, tid): Increment global self.ts. Append (ts, tid) to tweets[uid]. O(1).
2 follow / unfollow: follows[followerId].add(followeeId) or .discard(followeeId). O(1).
3 getNewsFeed — seed phase: Build followees set = follows[uid] | {uid} (always include self). For each followee who has tweets, push their most recent tweet onto the heap as (−ts, tweetId, uid, lastIndex).
4 getNewsFeed — drain phase: Pop up to 10 times. Each pop yields the globally newest unprocessed tweet. After popping entry at index i for user u, if i > 0 push tweets[u][i-1] back in — lazily advancing that user's pointer.
5 Return result: The collected list of up to 10 tweetIds, in newest-first order.
🔑 Why This Is "Merge K Sorted Lists" (#23)

Each user's tweet list is a sorted sequence (by timestamp). We never merge all K lists up front — that would be O(total tweets). Instead, we keep a "frontier" on the heap: the next candidate from each list. The heap always picks the globally newest, and we lazily advance only the pointer we just consumed. Heap size = number of followees F, so each pop/push is O(log F), not O(total tweets).

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every point in the drain loop, the heap contains exactly one entry per followee who still has unprocessed tweets, and that entry is the most recent unprocessed tweet from that followee. Therefore the heap's root is always the globally most recent available tweet.

Why this is correct: Suppose tweet T is in the top 10 but is not on the heap. That means some more-recent tweet from the same user has been processed. But processed tweets are appended to result — so T's user has already been advanced past T. Since we advance the pointer only one step at a time, T would have been pushed onto the heap when its predecessor was popped. Contradiction — T must have been on the heap at the right moment.

Global timestamp guarantees strict ordering: Every call to postTweet increments self.ts before storing. So two tweets from different users or the same user always get different timestamps, and higher timestamp = strictly more recent. Negating in the heap converts Python's min-heap into a max-heap by recency.

⚠️ Always Include Self in Followees

The problem says "users can see their own tweets in their feed." Don't rely on each user having called follow(self). Use follows[uid] | {uid} (set union) in getNewsFeed to unconditionally include the requesting user's own tweets.

⚠️ Use discard, Not remove

set.remove(x) raises KeyError if x is absent. The problem allows calling unfollow on someone never followed. set.discard(x) silently does nothing — the correct choice here.

4 The Code (Annotated) ▾

Solution 1 — HashMap + Max-Heap Merge (Optimal, Interview Pick)

import heapq ← Python's built-in min-heap module
from collections import defaultdict
class Twitter:
def __init__(self):
self.ts = 0 ← Global monotonic clock; every tweet gets a unique timestamp
self.tweets = defaultdict(list) ← uid → [(ts1, id1), (ts2, id2), ...] oldest first, newest at tail
self.follows = defaultdict(set) ← uid → {followeeId, ...}; set = O(1) add/remove/lookup
def postTweet(self, userId: int, tweetId: int) -> None:
self.ts += 1 ← Advance clock; guarantees unique, strictly increasing timestamps
self.tweets[userId].append((self.ts, tweetId)) ← Append to tail; most recent tweet is always at index len−1
def getNewsFeed(self, userId: int) -> list[int]:
heap = []
followees = self.follows[userId] | {userId} ← Always include self (set union); handles "own tweets in feed"
for uid in followees: ← Seed heap: one entry per followee
if self.tweets[uid]: ← Skip users with no tweets (empty list)
i = len(self.tweets[uid]) - 1 ← Index of most recent tweet (tail)
ts, tid = self.tweets[uid][i]
heapq.heappush(heap, (-ts, tid, uid, i))← Negate ts: min-heap acts as max-heap; tuple ordering ensures stability
result = []
while heap and len(result) < 10: ← Collect at most 10; stop if heap exhausted
neg_ts, tid, uid, i = heapq.heappop(heap) ← Pop globally most recent tweet
result.append(tid)
if i > 0: ← If this user has more (older) tweets...
i -= 1
ts, tid2 = self.tweets[uid][i]
heapq.heappush(heap, (-ts, tid2, uid, i))← ...push the next older one (lazy advance)
return result ← Newest-first, up to 10 items
def follow(self, followerId: int, followeeId: int) -> None:
self.follows[followerId].add(followeeId) ← Idempotent: safe to call twice
def unfollow(self, followerId: int, followeeId: int) -> None:
self.follows[followerId].discard(followeeId) ← discard (not remove) avoids KeyError if not following
💡 Complexity Summary

postTweet: O(1) amortized · O(T) space total for all tweets
follow/unfollow: O(1) · O(U·F) space for follow graph
getNewsFeed: O(F log F) to seed + O(10 log F) to drain = O(F log F)
where F = number of followees, T = total tweets, U = users

Solution 2 — Brute Force (Collect-all, Sort)

def getNewsFeed_brute(self, userId):
all_tweets = [] ← Collect ALL tweets from ALL followees
for uid in self.follows[userId] | {userId}:
all_tweets.extend(self.tweets[uid]) ← Could be millions of tweets
all_tweets.sort(reverse=True) ← Sort all T tweets: O(T log T)
return [tid for _, tid in all_tweets[:10]] ← Take first 10

O(T log T) per call — works for small inputs but does not scale. Mention this first, then optimize.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Clarify the API Contract

Ask: "Is getNewsFeed always returning exactly 10, or up to 10?" "Does a user see their own tweets without following themselves?" "Are tweet IDs unique globally?" These show you think about spec before coding.

Step 2: Pick Data Structures for O(1) Writes

"postTweet and follow/unfollow must be fast. I'll use a dict of lists for tweets (append = O(1)) and a dict of sets for follows (add/discard = O(1)). A global counter gives each tweet a sortable timestamp."

Step 3: Recognize getNewsFeed as Merge K Sorted Lists

"Getting the feed means merging F sorted tweet histories. I'll name-drop 'Merge K Sorted Lists' — that's the classic heap pattern. Seed the heap with each followee's most recent tweet, then drain it lazily."

Step 4: Implement the Heap Tuple Carefully

The tuple (−ts, tweetId, userId, listIndex) must include userId and listIndex so we know which user's list to advance after a pop. Mention this explicitly to show you've thought about the implementation.

🗣️ What to Say to the Interviewer

"I'll use two hashmaps: tweets maps userId to a list of (timestamp, tweetId) pairs,
and follows maps userId to a set of followeeIds.
postTweet is O(1) list append. follow/unfollow are O(1) set operations.
getNewsFeed is Merge K Sorted Lists — I'll use a max-heap.
Seed it with the most recent tweet from each followee (including self).
Each pop gives the globally newest tweet; then push that user's next tweet.
Collect up to 10. Total O(F log F) where F = number of followees."
🔮 Likely Follow-up Questions
  • "What if a user follows millions of people?" → O(F log F) seed is the bottleneck. Real Twitter pre-computes feeds ("fan-out on write") and stores them in Redis sorted sets.
  • "How do you handle tweet deletion?" → Mark deleted lazily — skip tombstoned tweets during heap drain. Or remove from list (O(n) but infrequent).
  • "Can you paginate the feed?" → Pass a cursor (last seen timestamp); restart heap drain from that point.
  • "What about private accounts?" → Add a permissions check before seeding the heap from that user's list.

7 Summary & Key Takeaways ▾
getNewsFeed is Merge K Sorted Lists — recognize and name-drop this in interviews
Use a global monotonic counter for timestamps; negate for Python's min-heap to simulate max-heap
Store tweets oldest-first, read from the tail — avoids O(n) reverse
Heap tuple includes (−ts, tweetId, userId, listIndex) so you can lazily advance per-user pointers
Use set.discard (not remove) for unfollow; use set union | {userId} in getNewsFeed for self-inclusion
Heap size = F (followees), not T (total tweets) — the key to scalability
MethodBrute ForceHeap Merge
postTweetO(1)O(1)
follow/unfollowO(1)O(1)
getNewsFeedO(T log T)O(F log F)
Space (total)O(T + U·F)O(T + U·F)

T = total tweets, F = followees per user, U = users. Since F << T in real systems, the heap approach wins decisively.

🎯 When to Use This Pattern
  • Any system with multiple independently-sorted streams to merge (activity feeds, log aggregation, ranked results)
  • Need top K globally from F sorted sources without loading all data
  • Lazy/on-demand access patterns — heap advances pointers only when needed
Merge K Sorted ListsHard — The exact same heap pattern on linked list nodes; the canonical form of this algorithm
Find K Pairs with Smallest SumsMedium — Heap merge of row-sorted 2D pairs; same lazy push-next structure
Kth Largest Element in a StreamEasy — Simpler heap design; maintains min-heap of size K
Top K Frequent ElementsMedium — HashMap + heap ranking; same two-structure combination
LRU CacheMedium — Another classic design: O(1) get/put with doubly linked list + hashmap
Smallest Range Covering Elements from K ListsHard — Advanced heap merge; tracks min across K sorted lists simultaneously
