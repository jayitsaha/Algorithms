LeetCode #1673

Find the Most Competitive Subsequence

Given an integer array nums and a positive integer k, return the most competitive subsequence of nums of size k.

Medium Monotonic Queue Monotonic Stack with Size Time: O(n) Space: O(k)
1 Understanding the Problem ▾

A subsequence preserves the relative order of selected elements but does not require them to be contiguous. From [3,5,2,6], valid subsequences of size 2 include [3,5], [3,2], [3,6], [5,2], [5,6], and [2,6].

A subsequence A is more competitive than B if, at the first position where they differ, A has a smaller value. This is exactly lexicographic comparison, like dictionary order: [2,6] beats [3,2] because at position 0, 2 < 3.

💡 Key Insight

We want the lexicographically smallest subsequence of exactly size k. Greedily: at every position in our result, we want the smallest possible value. If we see a smaller element, we should prefer it over a larger one already chosen — but only if we have enough elements remaining to still reach k total.

Key observations:

  • Must pick exactly k elements in original order.
  • "Most competitive" = lexicographically smallest among all size-k subsequences.
  • You can discard an element only if enough elements remain to still fill k positions.
  • At index i with current stack size s, remaining elements = n - i. We need at least k - s more. Can pop if n - i > k - s.

Edge cases: k = n (take all, no popping possible), k = 1 (return minimum), array already sorted ascending (stack fills monotonically), array sorted descending (every element triggers popping of previous ones).

2 The Algorithm: Monotonic Stack with Size Constraint ▾

We maintain a monotonic non-decreasing stack with a maximum size of k. As we scan left to right, we greedily replace larger elements with smaller ones when possible — constrained by the size requirement.

1 Initialize: empty stack [], precompute n = len(nums).
2 For each element num at index i: Enter the while-loop pop phase.
3 Pop condition (all three must hold): stack is non-empty, stack[-1] > num (current is better), AND n - i > k - len(stack) (room to pop: enough future elements remain to still reach k).
4 Push if not full: If len(stack) < k, push current element. If the stack is already full (size = k), skip.
5 Return stack — contains exactly k elements forming the most competitive subsequence.
🔑 The "Room to Pop" Formula

At index i: remaining elements including current = n - i. Slots still needed if we pop one = k - (len(stack) - 1) = k - len(stack) + 1. Pop is safe when remaining > slots needed after pop, i.e. n - i > k - len(stack).

Intuitively: len(stack) + (n - i) > k — total "available" elements exceeds k, so we can afford to discard one.

⚠️ Common Mistake: Wrong Pop Condition

Using n - i >= k - len(stack) (with >=) would pop when remaining exactly equals needed — leaving the stack one element short. The condition must be strictly greater than (>) because we need at least one additional future element to replace the popped one.

3 Why It Works: The Invariant ▾
🔐 The Invariant

After processing index i, the stack holds the most competitive valid subsequence of its current size, considering only elements from indices 0 through i, subject to the constraint that enough elements remain to grow to k by the end.

Why greedy popping is correct: Suppose the stack contains [..., x] and we encounter y < x. At the position where x sits in the result, having y instead strictly improves the subsequence (smaller value at that position). All earlier positions are unaffected. So popping x and eventually using y there is always optimal — provided we can still reach k elements.

Why the size guard is necessary: If we pop too aggressively, we might run out of elements. If we have exactly k - len(stack) elements remaining (including current), every single one must be taken. Popping would create an unfillable gap.

Why we skip when len(stack) == k: We need exactly k elements. Once full, new (larger) elements at the end simply aren't part of the optimal answer — they'd only extend past size k. Smaller elements at the end might replace top, but only via the pop-then-push path.

💡 Analogy: Curating a Queue for a Competition

You're selecting k contestants in order for a quiz bowl. Each contestant has a score. You want the team to look best when scores are read left to right. As each new contestant arrives, if they score lower than your last-picked contestant AND you'd still be able to pick k total after this swap — drop the last-picked contestant in favor of the newcomer. Never trade out when you'd be left short-handed.

Note: The result stack is NOT guaranteed to be monotonically increasing. The size constraint can force us to stop popping even when a smaller element arrives. For example in nums=[3,5,2,6,4,1,8], k=3: the 1 at index 5 cannot pop the 2 at index 0 (would need 3 more elements but only 2 remain).

Solution 1 — Monotonic Stack with Size Constraint (Optimal, Interview Pick)

n = len(nums) ← Cache n for the pop-guard formula
for i, num in enumerate(nums): ← Scan every element with its index
while (stack ← Guard 1: non-empty (nothing to pop otherwise)
and stack[-1] > num ← Guard 2: top is worse (current is smaller)
and n - i > k - len(stack)):← Guard 3: room to pop — enough remaining

if len(stack) < k: ← Only push if we still need more elements
stack.append(num) ← Add current to result
return stack ← Exactly k elements, most competitive
⚠️ Why strictly > in the pop guard?

If n - i == k - len(stack), we have exactly as many elements left (including num) as slots to fill. After a pop, we'd need k - len(stack) + 1 elements from n - i available — but n - i would then be short by exactly one. We must keep the top and take every remaining element. Using >= here is a classic off-by-one bug.

💡 Complexity Summary

Monotonic Stack (Optimal): Time O(n) — each element is pushed and popped at most once. Space O(k) — stack holds at most k elements.
Brute Force: Time O(C(n,k)·k) — exponential. Impractical for n=10000, k=5000. Use only to verify correctness on tiny inputs.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Pattern

Key triggers: "lexicographically smallest" + "subsequence" + "size k" + array input. These canonically signal monotonic stack. The phrase "most competitive" is a synonym for "lexicographically smallest." Any problem asking you to minimize a sequence's first differing position follows this template.

Step 2: Brute Force First

"I could try all C(n,k) subsequences and compare lexicographically. That's exponential — O(C(n,k)·k). For large n this is completely infeasible. I need a greedy O(n) approach."

Step 3: Derive the Greedy

"For the result to be optimal, position 0 should be as small as possible. If I see a smaller element, I should use it for that position instead of a larger one already chosen. This is monotonic stack logic — pop larger, push smaller."

Step 4: Add the Size Constraint

"The extra challenge is we need exactly k elements. So I can't pop freely — I must check that after the pop, enough elements remain to reach k. The guard is: n - i > k - len(stack)."

🗣️ What to Say to the Interviewer

"This is a greedy monotonic stack problem. I want the lex-smallest subsequence of size k.
I'll scan left to right with a stack. For each element, if it's smaller than the top AND I can still reach k elements total after popping, I pop the top — it's suboptimal at that position.
The size guard is n - i > k - len(stack): remaining elements exceed remaining needed slots.
Only push if stack size is less than k. At the end, return the stack.
Time O(n), Space O(k). Identical in structure to Remove K Digits (LC 402), just with exact-size constraint."
🔮 Likely Follow-up Questions
  • "k = n?" → Pop guard n - i > k - len(stack) always fails because n - i = k - len(stack) at every step. Return full array.
  • "k = 1?" → Stack keeps shrinking to just the minimum element seen so far (that has room to exist).
  • "How is this different from Remove K Digits?" → Remove K Digits removes at most k elements; this selects exactly k. Same pop logic, different termination.
  • "Can you generalize?" → Create Maximum Number (LC 321) extends this to picking from two arrays — same stack technique applied twice then merging.

7 Summary & Key Takeaways ▾
Monotonic stack + size guard gives the lex-smallest subsequence of size k in O(n)
Pop condition: stack non-empty AND current < top AND n - i > k - len(stack)
Push condition: len(stack) < k — only add if we still need more elements
Each element pushed and popped at most once → O(n) total time, O(k) space
The result stack may NOT be monotone — size constraint can block necessary pops
This is "Remove K Digits" with an exact-size constraint instead of at-most-k
