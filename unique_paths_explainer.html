LeetCode #62

Unique Paths

Count distinct paths a robot can take from top-left to bottom-right of an m×n grid, moving only right or down.

Medium Dynamic Programming 2D Grid DP Time: O(m·n) Space: O(n)
1 Understanding the Problem ▾

A robot sits at the top-left corner (0, 0) of an m × n grid and wants to reach the bottom-right corner (m-1, n-1). The only moves allowed are right or down. How many distinct routes can it take?

💡 Key Insight

Because you can only move right or down, every cell (r, c) can only be entered from the cell above it (r-1, c) or from the cell to its left (r, c-1). So the number of paths to (r, c) is exactly the sum of paths to those two cells. That single observation is the entire algorithm.

✓ Example: m=3, n=7 → 28

Total steps: (3-1)+(7-1) = 8
Down moves needed: 2
Right moves needed: 6
Choose 2 downs from 8: C(8,2) = 28 ✓

⚠ Edge Cases

m=1 or n=1: answer=1
(straight line, zero choices)
m=1, n=1: at goal, answer=1
Both constraints: 1 ≤ m,n ≤ 100

Why not greedy? Greedy needs a "locally best" choice at each step, but every right/down move is equally valid — there's no cost to minimize. We're counting possibilities, not optimizing, so DP is the natural fit.

Small concrete trace (3×3 grid): The top row (row 0) has 1 path each — the robot can only go right. The left column (col 0) has 1 path each — only down. Cell (1,1): arrived from (0,1) or (1,0) = 1+1 = 2 paths. Cell (2,2): arrived from (1,2) or (2,1) = 3+3 = 6 paths. The pattern builds naturally.

2 The Algorithm: Sum From Top and Left ▾

We build a DP table where dp[r][c] = number of distinct paths from (0,0) to (r,c), filled row-by-row left-to-right. Each interior cell sums its two neighbors. The space-optimized version uses a single 1D row.

1 Create dp array of length n, all 1s. This represents the top row: traveling only right from (0,0), there is exactly one way to reach any cell in that row. Column 0 of every other row stays 1 forever (only reached by going straight down) — the inner loop starts at c=1, so col 0 is never overwritten.
2 For each interior row r (1 to m-1) and each interior column c (1 to n-1): perform dp[c] += dp[c-1]. Before the update, dp[c] holds the previous row's value (paths from above). dp[c-1] has already been updated this row (paths from left). Their sum is the answer for this cell.
3 Return dp[n-1]. After all rows are processed, the last element holds the count of paths to the bottom-right corner.
📐 The Recurrence Relation
dp[r][c] = dp[r-1][c] + dp[r][c-1]
Base: dp[0][c] = 1 for all c  |  dp[r][0] = 1 for all r
1D form: dp[c] += dp[c-1]   (dp[c] = old row's value = "from above")

🔑 Mathematical Shortcut: Combinatorics

Every valid path takes exactly (m-1) down steps and (n-1) right steps, for (m+n-2) total steps. We just pick which (m-1) of those steps go down — the rest are right. Answer = C(m+n-2, m-1). For m=3, n=4: C(5,2) = 10. This gives O(1) space and O(m+n) time. In Python: from math import comb; return comb(m+n-2, m-1).

3 Why It Works: The Invariant & Why This is DP ▾
🔐 The DP Invariant

After processing cell (r, c), dp[r][c] contains the exact number of distinct right-down paths from (0,0) to (r,c). This invariant is established for row 0 by initialization, and for all other cells by the recurrence — which is only applied after both dependency cells are finalized.

Why is this Dynamic Programming? Two pillars must hold — and both do:

1 Optimal Substructure: The path count to (r, c) depends only on two sub-answers: paths to (r-1, c) and paths to (r, c-1). If those are correct, the current cell's answer is guaranteed correct. No other cells are needed.
2 Overlapping Subproblems: A naive recursive approach splits into f(r-1,c) and f(r,c-1), both of which recursively call shared sub-cells. For example, f(1,1) is needed by both f(2,1) and f(1,2). Without memoization, the tree blows up exponentially — O(2^(m+n)) calls versus O(m·n) unique subproblems.
⚠️ Why the 1D In-Place Update is Correct

In the inner loop, when we compute dp[c] += dp[c-1]:
• dp[c] before update = value from the previous row = "paths from above"
• dp[c-1] was just updated in this row's left-to-right pass = "paths from left"
So the in-place addition correctly computes the 2D recurrence without a 2D array. The key is that we scan left to right — dp[c-1] is always updated before dp[c] in the same row.

Why row 0 and col 0 are 1: In row 0, the robot started at (0,0) and moved only right to get there — zero choices, one path. Same for col 0: only moved down. These boundary values anchor the recurrence. Every other cell's value is built from these anchors outward, row by row.

Filling order guarantees correctness: We fill left-to-right, top-to-bottom. When we compute dp[r][c], both dp[r-1][c] (same column, previous row) and dp[r][c-1] (same row, already updated) are complete. This "topological order" with respect to the dependency graph ensures no circular dependencies.

4 The Code (Annotated) ▾

Solution 1 — 1D DP, Space-Optimized (Interview Pick)

def uniquePaths(m: int, n: int) -> int: ← m rows, n columns
dp = [1] * n ← Row 0: all 1s (base case — only one path along top row)
for r in range(1, m): ← Process each row after the first
for c in range(1, n): ← Column 0 stays 1 (never updated); start at col 1
dp[c] += dp[c - 1] ← dp[c] (from above) += dp[c-1] (from left). In-place!
return dp[n - 1] ← Bottom-right cell = answer
⚠️ Why not dp[-1] instead of dp[n-1]?

dp[-1] also works in Python and returns the last element. However, dp[n-1] is more explicit and readable — it makes clear you are returning the value at the last column index, not accidentally negatively indexing a different cell.

Solution 2 — Full 2D DP (Easiest to Understand)

def uniquePaths(m: int, n: int) -> int:
dp = [[1] * n for _ in range(m)] ← Initialize entire m×n table to 1 (covers base cases)
for r in range(1, m): ← Skip row 0 (all already 1)
for c in range(1, n): ← Skip col 0 (all already 1)
dp[r][c] = dp[r-1][c] + dp[r][c-1] ← The classic recurrence: above + left
return dp[m-1][n-1] ← Answer is bottom-right cell

Solution 3 — Top-Down Memoization

from functools import lru_cache
def uniquePaths(m: int, n: int) -> int:
@lru_cache(maxsize=None) ← Cache each (r,c) so we never recompute it
def dp(r, c): ← Returns path count from (0,0) to (r,c)
if r == 0 or c == 0: return 1 ← Base case: top row or left column → exactly 1 path
return dp(r-1, c) + dp(r, c-1) ← Recurrence: from above + from left
return dp(m-1, n-1) ← Query the bottom-right target

Solution 4 — Combinatorics (O(1) Space)

from math import comb
def uniquePaths(m: int, n: int) -> int:
return comb(m + n - 2, m - 1) ← C(m+n-2, m-1): choose which (m-1) of (m+n-2) steps go down
💡 Complexity Summary

2D DP: Time O(m·n), Space O(m·n)
1D DP (optimal DP): Time O(m·n), Space O(n)
Memoization: Time O(m·n), Space O(m·n) + call stack O(m+n)
Combinatorics: Time O(m+n), Space O(1) — Python handles big integers natively

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the DP Signal

"Grid + count paths + restricted monotone moves" is a textbook DP pattern. The restriction to right/down means subproblems don't overlap cyclically — each cell has exactly two possible predecessors, which is the ideal DP shape.

Step 2: Define the State Clearly

"I'll define dp[r][c] = number of distinct paths from (0,0) to (r,c). I need to know this count for every cell, and I'll build it up from the top-left."

Step 3: Derive the Recurrence

"Since I can only come from above or from the left: dp[r][c] = dp[r-1][c] + dp[r][c-1]. Base case: row 0 and col 0 are all 1 because there's only one straight-line path to each."

Step 4: Propose Space Optimization

"I can compress from O(m·n) to O(n) space by using a single 1D array and updating it row-by-row: dp[c] += dp[c-1]. Before the update, dp[c] holds the previous row's value — that's my 'from above' input. Then mention the O(1) combinatorial formula as a bonus."

🗣️ What to Say to the Interviewer

"I observe the robot can only move right or down — so each cell can only be reached
from the cell above it or the cell to its left. That gives me: paths(r,c) = paths(r-1,c) + paths(r,c-1).
Base case: the entire first row and first column are all 1s — straight-line paths only.
I'll fill the table row-by-row, left-to-right, and return dp[m-1][n-1].
Time O(m·n), Space O(m·n). I can reduce space to O(n) with a 1D rolling array.
Bonus: this is also C(m+n-2, m-1) combinatorially — O(1) space with Python's comb()."
🔮 Likely Follow-up Questions
  • "What if some cells are blocked?" → LeetCode #63: set dp[r][c] = 0 for obstacle cells; same structure
  • "What if cells have costs and you want the minimum cost path?" → LeetCode #64: replace + with min; dp[r][c] = min(dp[r-1][c], dp[r][c-1]) + cost[r][c]
  • "Can you solve in O(1) space without using the formula?" → No; 1D array is O(n), true O(1) needs combinatorics
  • "What if the robot can also move up or left?" → No longer monotone; use BFS/DFS with cycle handling — DP doesn't directly apply

7 Summary & Key Takeaways ▾
Each cell is reachable only from above or from the left — that constraint creates a clean recurrence
Base case: entire top row and left column are 1 (single forced path to each)
Recurrence: dp[r][c] = dp[r-1][c] + dp[r][c-1]
1D rolling array compresses O(m·n) space to O(n): dp[c] += dp[c-1] in-place
Combinatorial shortcut: C(m+n-2, m-1) — same answer, O(1) space, O(m+n) time
The DP value at (r,c) is also the (r,c) entry of Pascal's Triangle (offset by one)

🎯 When to Recognize 2D Grid DP
  • "Count paths in a grid with restricted moves (right/down)" → Sum from top and left
  • "Minimum cost to reach bottom-right" → Same structure, use min instead of sum
  • "Can you reach the destination with obstacles?" → Set blocked cells to 0
  • Monotone movement (no backward steps) → tabulation always applies cleanly
