LeetCode #1004

Max Consecutive Ones III

Given a binary array nums and an integer k, return the maximum number of consecutive 1s in the array if you can flip at most k 0s.

Medium Sliding Window Count Zeros ≤ K Time: O(n) Space: O(1)
1 Understanding the Problem ▾

You have a binary array — only 0s and 1s — and a "flip budget" of k. You may turn at most k zeros into ones. The question: what is the longest run of consecutive ones you can create?

💡 Key Insight — The Reframe

Flipping a 0 is equivalent to allowing it inside your window of ones. So the problem becomes: find the longest contiguous subarray that contains at most k zeros. You never actually flip anything — you just find the best window.

✗ What we're NOT doing

Trying every subarray:
for i in range(n):
  for j in range(i, n):
   count zeros [i..j]
O(n²) — TLE for n=10⁵

✓ What we ARE doing

Sliding window:
expand right greedily
shrink left when zeros > k
record window size
O(n) — each element processed twice

Worked example: nums = [1,1,0,0,1,1,1,0,1], k = 2.

The window [1,1,0,0,1,1,1] (indices 0–6) contains 2 zeros, which is exactly our k=2 budget, giving length 7. After index 7 (another zero), the window would have 3 zeros — too many. We slide left until zeros drop back to 2. The best window in this example has length 7.

Edge cases: k = 0 means no flips — just find the longest existing run of 1s. k ≥ (total zeros in array) means we can flip everything, so the answer is n. All-zeros array with k = n returns n. Single-element arrays trivially return 1.

2 The Algorithm: Variable Sliding Window ▾

Maintain a window [left, right] representing the current candidate subarray. Expand right greedily one step at a time. Only shrink left when the window becomes invalid (more than k zeros). Record the window size after each expansion.

1 Initialize: left = 0, zeros = 0, max_len = 0. Empty window, zero budget used, no best yet.
2 Expand right: For each index right, include nums[right] in the window. If it's a 0, increment zeros (consuming one flip budget).
3 Shrink left (if needed): While zeros > k, the window is invalid. Move left forward. If the element we just removed was a 0, decrement zeros (restoring one flip budget).
4 Record maximum: After step 3, window [left, right] is guaranteed valid (≤ k zeros). Update max_len = max(max_len, right - left + 1).
5 Repeat: Advance right and repeat steps 2–4 for every element. Return max_len.

🔑 Why "at most k zeros" is the same as "flip at most k zeros"

Any contiguous subarray with exactly z zeros (where z ≤ k) can be made all-ones by flipping those z zeros. So the length of that subarray equals the length of the consecutive ones sequence we'd get after flipping. We want the maximum such length across all valid windows.

3 Why It Works: The Invariant ▾
🔐 The Invariant

After every iteration (after the while-shrink loop), the window [left, right] always contains at most k zeros. The variable zeros exactly equals the count of zeros in nums[left..right]. We never record the window length while it is invalid.

Why left only moves forward: The sliding window technique works because we never need to move left backward. All windows with a smaller starting index and the same ending index have already been seen (on previous iterations of the outer for-loop). Moving left backward would only increase the window, potentially making it invalid again — it can never reveal a better answer than the maximum we've already recorded.

Why we never miss the optimal window: For any optimal window [i*, j*] with ≤ k zeros, when right = j*, our left is at most i* (we only shrink when forced). So the window length right - left + 1 ≥ j* - i* + 1 is at least as large as the optimal. Since we record the maximum at every step, we capture it.

⚠️ Common Mistake: Using if instead of while

When zeros > k, one shrink step might remove a 1 (not a 0), so zeros is still > k. An if stops after one step and records an invalid window. The while loop keeps shrinking until the invariant is restored — which might take several steps if the window has many leading 1s.

Time complexity proof: Each element is added to the window exactly once (when right reaches it) and removed at most once (when left passes it). Total operations ≤ 2n → O(n). Space is O(1) — we only store three integers.

4 The Code (Annotated) ▾

Solution 1 — Variable Sliding Window (Optimal · Interview Pick)

def longestOnes(nums: list, k: int) -> int:
left = 0 ← Left boundary; only moves forward (shrink side)
zeros = 0 ← Count of zeros currently inside window [left, right]
max_len = 0 ← Best valid window length seen so far
for right in range(len(nums)):← Expand window: right is the new right boundary
if nums[right] == 0: ← Incoming element is a 0 → consume one flip budget
zeros += 1
while zeros > k: ← Window invalid: too many zeros. Must shrink left.
if nums[left] == 0: ← Element being evicted is a 0 → restore flip budget
zeros -= 1
left += 1 ← Move left boundary one step right
max_len = max(max_len, ← Window is now valid. Record its size.
right - left + 1)
return max_len ← Best length found across all valid windows
⚠️ Why "zeros -= 1" only runs when nums[left] == 0?

When we shrink the window by advancing left, we're removing nums[left] from the window. If that element was a 1, zeros count is unchanged. If it was a 0, we're giving back a flip budget. The if-check correctly handles both cases without unconditionally decrementing zeros.

Solution 2 — Brute Force O(n²) (for understanding only)

def longestOnes_brute(nums, k):
n, best = len(nums), 0
for i in range(n): ← Try every possible starting index
zeros = 0
for j in range(i, n): ← Extend right as long as valid
if nums[j] == 0: zeros += 1
if zeros > k: break ← Budget exceeded → stop extending from this i
best = max(best, j - i + 1)
return best ← O(n²) total — correct but too slow
💡 Complexity Summary

Sliding Window: Time O(n) — each element enters and exits the window at most once. Space O(1) — only three variables.
Brute Force: Time O(n²) — nested loops over every subarray start/end. Space O(1).
The sliding window is strictly optimal for this problem. No O(n log n) approach exists.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Reframe the Problem

"Flip at most k zeros to maximize consecutive ones" reframes to "find the longest subarray with at most k zeros." This equivalence is the entire key. Say it out loud to the interviewer — it shows you understand the problem deeply.

Step 2: Recognize the Pattern

Signals pointing to sliding window: binary/integer array, "at most k" budget, maximize length, contiguous subarray. This is the classic variable-size sliding window: expand right freely, shrink left when a constraint is violated.

Step 3: Define the Window's Validity Condition

A window is valid when it contains ≤ k zeros. A window is invalid when it contains > k zeros. Track zeros (integer counter). Valid → record length. Invalid → shrink.

Step 4: Apply the Sliding Window Template

The template: left=0, state=initial; for right in range(n): add nums[right]; while invalid: remove nums[left], left++; record right-left+1. This template solves a huge class of "longest subarray with property" problems.

🗣️ What to Say to the Interviewer

"First insight: flipping k zeros is equivalent to finding the longest subarray with at most k zeros.
I'll use a variable sliding window with left and right pointers.
For each new element at right: if it's a 0, increment zeros counter.
While zeros exceeds k, advance left — if we're removing a 0, decrement zeros.
After each step, record right - left + 1. Return the maximum.
Time O(n), space O(1). Want me to code it up?"
🔮 Likely Follow-up Questions
  • "What if k = 0?" → Every zero immediately triggers a shrink, so window never contains a 0. Answer is the longest existing run of 1s. Works correctly.
  • "What if k ≥ total zeros?" → The while-shrink never triggers (zeros never exceeds k). Window grows to include the entire array. Answer is n.
  • "What is the minimum window that already contains at least k ones?" → Completely different problem — use a fixed window or binary search.
  • "What if elements are 0, 1, or 2 and you can flip 2s to 0s?" → Generalize: track count of "bad elements" instead of zeros. Same template.
  • "Longest Subarray of 1s After Deleting One Element (LC #1493)?" → Same problem with k=1 but requires exactly one deletion. Adjust condition.
