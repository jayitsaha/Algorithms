LeetCode #1584

Min Cost to Connect All Points

Find the minimum total cost to connect all n points using Manhattan distance edges — the classic Minimum Spanning Tree problem on a complete graph.

Medium Graph Prim's Algorithm MST Time: O(n²) Space: O(n)
1 Understanding the Problem ▾

You receive n points on a 2D plane: points = [[x₁,y₁],[x₂,y₂],...]. You can connect any two points at a cost equal to their Manhattan distance: |x₁−x₂| + |y₁−y₂|. The goal is to connect every point into one connected component using the minimum total cost.

💡 Key Insight

"Connect ALL nodes with minimum total edge weight" = Minimum Spanning Tree (MST). Every pair of points is implicitly connected (complete graph Kn). We need the MST of this complete graph where edge weights are Manhattan distances. The MST uses exactly n−1 edges and guarantees every node is reachable.

✗ Wrong Goal: Shortest Path

Dijkstra/BFS finds min cost
between exactly TWO nodes.
Doesn't connect all n nodes.
Not what we want here.

✓ Right Goal: Spanning Tree

MST connects ALL n nodes

No redundant edges (no cycles).
Minimum total cost guaranteed.

Example walkthrough: points = [[0,0],[2,2],[3,10],[5,2],[7,0]]. Manhattan distances between every pair form 10 possible edges. The MST picks the 4 cheapest non-cycle-forming edges to connect all 5 nodes.

Edge cases to consider: Single point → return 0. Two points → return their Manhattan distance. Points on the same horizontal/vertical line (distance = |x₁−x₂| or |y₁−y₂| only). Duplicate points → distance 0 (free edge).

2 The Algorithm: Prim's MST (Dense Graph Variant) ▾

For dense graphs — where every pair of nodes has an edge, as here — array-based Prim's runs in O(n²) time and O(n) space. Heap-based Prim's or Kruskal's would be O(n² log n) because sorting/heapifying O(n²) edges dominates. Array Prim's avoids materializing the edge list entirely.

🔑 Algorithm Deep-Dive: Prim's MST (1957)

Named after: Robert Prim; independently discovered by Vojtech Jarník (1930) and Edsger Dijkstra (1959).

Core Invariant: min_dist[j] always holds the cheapest known single edge from any node currently in the MST to the unvisited node j. Picking the minimum is always safe (Cut Property).

MST Cut Property: For any partition of nodes into two sets S and V−S, the minimum-weight edge crossing the cut belongs to every MST. Each Prim's step exploits this: S = current MST nodes, V−S = unvisited nodes.

Recognize this algorithm when: "minimum total edge cost to connect all nodes" — especially when the graph is dense (many edges).

1 Initialize: min_dist = [∞, ∞, ..., ∞]. Set min_dist[0] = 0 (start at node 0, costs nothing to "enter"). in_mst = [False]*n. total_cost = 0.
2 Pick cheapest frontier: Scan all unvisited nodes. Find the index u with the smallest min_dist[u]. This is the cheapest bridge from the MST into unvisited territory.
3 Add to MST: Set in_mst[u] = True. Add min_dist[u] to total_cost. The edge that achieved min_dist[u] is now permanently in the MST.
4 Update neighbors: For every unvisited node v, compute Manhattan distance d = |points[u][0]−points[v][0]| + |points[u][1]−points[v][1]|. If d < min_dist[v], update: we found a cheaper bridge to v.
5 Repeat n times (one iteration per node). After all n nodes are added, return total_cost — this is the MST cost.

3 Why It Works: The Invariant ▾
🔐 The Prim's Invariant

At every step of the algorithm, min_dist[j] = the cheapest single edge connecting any MST node to unvisited node j. When we pick the minimum, we are guaranteed — by the MST Cut Property — that this is the correct edge to add. No future edge can be cheaper and still avoid a cycle.

The Cut Property (informal proof): Imagine S = all nodes currently in the MST, and T = all unvisited nodes. The cheapest edge crossing this cut (S → T) MUST appear in every MST of the graph. If it didn't, we could swap it in and get a cheaper spanning tree — contradicting that we already have an MST. Prim's finds exactly this minimum edge at every step.

Why exactly n−1 edges? A spanning tree connecting n nodes has exactly n−1 edges — any fewer and not all nodes are connected, any more and there's a cycle (not a tree). Our loop runs n times but the first iteration adds node 0 at cost 0 (a free "seed"), so we add n−1 real edges to total_cost.

⚠️ Prim's vs Dijkstra's — They Look Identical But Are Different

Both algorithms maintain a "distance" array and greedily pick the minimum. The critical difference:

  • Dijkstra: dist[v] = min(dist[v], dist[u] + edge_weight(u,v)) — cumulative path cost from source
  • Prim's: min_dist[v] = min(min_dist[v], edge_weight(u,v)) — cheapest SINGLE edge into v

Adding dist[u] to total_cost in Prim's gives the MST cost. Doing the same in Dijkstra gives shortest-path distances. Mixing these up is the #1 interview mistake for this problem.

Kruskal's as an alternative: Sort all O(n²) edges, then greedily add non-cycle-forming edges using Union-Find. O(n² log n) time and O(n²) space — worse than Prim's for this problem. Use Kruskal's when you have a sparse graph with an explicitly given edge list.

4 The Code (Annotated) ▾

Solution 1 — Prim's MST, Dense O(n²) · Interview Pick

def minCostConnectPoints(points):
n = len(points) ← number of points / nodes
in_mst = [False] * n ← True once node is committed to MST
min_dist = [float('inf')] * n ← cheapest known bridge from MST to each node
min_dist[0] = 0 ← start at node 0; costs 0 to "add" our seed
total_cost = 0 ← MST total cost accumulator
for _ in range(n): ← repeat exactly n times (one per node)
# Pick cheapest unvisited node
u = -1
for i in range(n): ← O(n) scan to find minimum
if not in_mst[i] and (u == -1 or min_dist[i] < min_dist[u]):
u = i ← best unvisited candidate so far
in_mst[u] = True ← commit: add u to MST
total_cost += min_dist[u] ← add cheapest edge cost into u
for v in range(n): ← update all unvisited neighbors
if not in_mst[v]: ← skip already-committed nodes
d = abs(points[u][0] - points[v][0]) + abs(points[u][1] - points[v][1])
↑ Manhattan distance from new MST node u to v
if d < min_dist[v]: ← found a cheaper bridge to v?
min_dist[v] = d ← update: u→v is now the best known edge to v
return total_cost ← MST complete; return total edge cost
💡 Complexity Analysis

Time: O(n²) — Outer loop: n iterations. Inner pick-scan: O(n). Inner update-loop: O(n). Total: O(n × 2n) = O(n²). This is optimal for a complete graph with O(n²) edges — you cannot do better without inspecting every edge at least once.
Space: O(n) — Only in_mst and min_dist arrays of size n. No adjacency matrix or edge list ever stored.

Solution 2 — Kruskal's with Union-Find · O(n² log n)

def minCostConnectPoints(points): ← Kruskal's: sort all edges, greedy add
n = len(points)
edges = [] ← will hold O(n²) edges
for i in range(n):
for j in range(i+1, n): ← every pair, no duplicates
d = abs(points[i][0]-points[j][0]) + abs(points[i][1]-points[j][1])
edges.append((d, i, j)) ← (cost, from, to)
edges.sort() ← O(n² log n) — sorts all edges cheapest-first
parent = list(range(n)) ← Union-Find: each node is its own root
def find(x): ← path-compressed find
while parent[x] != x:
parent[x] = parent[parent[x]] ← path halving for efficiency
x = parent[x]
return x
cost, used = 0, 0
for d, u, v in edges: ← process cheapest edge first
if find(u) != find(v): ← different components? adding this won't form a cycle
parent[find(u)] = find(v) ← merge components (union)
cost += d; used += 1
if used == n - 1: break ← MST done: n−1 edges added
return cost
⚠️ Why Prim's Wins for This Problem

Kruskal's needs to materialize all O(n²) edges — for n=1000, that's 499,500 edges stored in memory (O(n²) space). Sorting them adds another O(n² log n) overhead. Prim's computes distances on-the-fly: only O(n) space, O(n²) time. Always prefer Prim's when the graph is dense (complete or near-complete).

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Spot the MST Signal

Key words: "minimum cost", "connect ALL points/cities/nodes", implicit or explicit edge weights. Distinguish from shortest-path problems (those connect only two specific nodes, not all of them). If you need to reach EVERY node at minimum total cost → MST.

Step 2: Identify Graph Density

This problem has a complete graph Kn — every pair of points has an edge. With n=1000 points, that's ~500K edges. This is "dense". For dense graphs: Prim's O(n²). For sparse graphs (explicit edge list with E << n²): Kruskal's or heap-Prim's at O(E log V).

Step 3: Propose Both, Then Optimize

Start: "Kruskal's — sort all n² edges, then Union-Find. O(n² log n) time, O(n²) space." Then: "We can do better. Prim's with an array avoids the edge list entirely. O(n²) time, O(n) space. Let me code that."

Step 4: Name the Theorem

Mention the Cut Property to show depth: "Prim's is correct because the Cut Property guarantees the minimum-weight edge crossing the frontier between MST nodes and unvisited nodes always belongs to every MST." This signals graph theory fundamentals.

🗣️ What to Say to the Interviewer

"This asks me to connect all n points with minimum total cost — that's an MST problem.
Since every pair of points has a valid connection (Manhattan distance), we have a complete graph.
For complete graphs, array-based Prim's is optimal: O(n²) time, O(n) space.
I maintain min_dist[i] = cheapest known edge from any MST node to unvisited node i.
Each round: pick the unvisited node with minimum min_dist, commit it, update neighbors.
After n rounds, all nodes are in the MST. Return accumulated total_cost."
🔮 Likely Follow-up Questions
  • "Can you solve with Kruskal's?" → Yes, but O(n² log n) and O(n²) space. Works but slower for dense graphs.
  • "What if some pairs can't be connected?" → Skip those in distance computation; check if MST is possible at end.
  • "What if we only need k components?" → Stop early after n−k edges are added (Minimum Spanning Forest).
  • "Euclidean distance instead of Manhattan?" → Same algorithm, different distance formula. Change to sqrt((x1-x2)²+(y1-y2)²).
  • "Single point?" → n=1, loop runs once, adds min_dist[0]=0. Return 0.

7 Summary & Key Takeaways ▾
"Connect ALL nodes with minimum total cost" = Minimum Spanning Tree (MST)
Dense complete graph (Kn): use array-based Prim's — O(n²) time, O(n) space, no edge list
Sparse explicit-edge graph: use Kruskal's + Union-Find — O(E log E) time
The MST Cut Property proves Prim's greedy choice is always correct
min_dist[i] = cheapest SINGLE edge into i (not cumulative — unlike Dijkstra's dist[i])
MST always has exactly n−1 edges; any more creates cycles, any fewer leaves nodes disconnected
ApproachTimeSpaceBest For
Kruskal's + Union-Find O(n² log n) O(n²) Sparse graphs, explicit edge list
Prim's Array ✓ (this problem) O(n²) O(n) Dense / complete graphs
Prim's + Min-Heap O(E log V) O(E) Sparse graphs, fast key-decrease
🎯 When to Recognize the MST Pattern
  • "Minimum cost to connect all cities/nodes/points" → MST
  • Network design: roads, cables, pipes with minimum total length → MST
  • n nodes, need exactly n−1 connections, all reachable → MST
  • NOT MST: "shortest path from A to B" → Dijkstra / BFS
