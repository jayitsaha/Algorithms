Count Complete Tree Nodes

Given a complete binary tree, count nodes in O(log²n) — faster than visiting every node.

Easy Trees Left/Right Height Compare Time: O(log²n) Space: O(log n)
1 Understanding the Problem ▾

A complete binary tree has all levels fully filled except possibly the last, which is filled from left to right with no gaps. This structure constraint is what allows an O(log²n) solution — we never need to visit every node.

Key Insight

The structure guarantees that one of the two subtrees at every node is always a perfect binary tree — and we can compute its size instantly with bit shifting. We skip perfectly-filled halves in O(1), never visiting their individual nodes.

Complete tree (valid input)

All levels full except last.
Last level: strictly left-to-right.
No gaps allowed between filled positions.

NOT complete (invalid for this problem)

Gap on the last level (not left-packed).
Right child present but left child missing.
Arbitrary missing nodes anywhere.

Edge cases: empty tree → 0; single node → 1; perfect tree of height h → (1<<h)−1 (the fast path fires immediately at the root, returning in O(log n) without any recursion).

2 The Algorithm: Left/Right Height Comparison ▾

We measure two spine heights at each node. Their relationship tells us in O(log n) whether the current subtree is perfect — no traversal needed.

1 Compute left-spine height (lh): From the current node, repeatedly follow left children until falling off. Count every node visited. This costs O(log n) — tree height is log n.
2 Compute right-spine height (rh): From the same node, repeatedly follow right children until falling off. Count nodes visited. Also O(log n).
3 If lh == rh → perfect subtree: The entire subtree is a perfect binary tree containing exactly (1<<lh)−1 = 2lh−1 nodes. Return this directly — no recursion into children required.
4 If lh > rh → not perfect, recurse: Return 1 + countNodes(root.left) + countNodes(root.right). Thanks to completeness, one of the two recursive calls will always immediately hit the fast path (its subtree is perfect).

Why measure spines specifically?

In any complete binary tree, the left spine always reaches the deepest level. The right spine reaches the deepest level only when the tree is perfect. Equal spine lengths is therefore a precise O(log n) test for "is this subtree perfect?"

3 Why It Works: The Invariant ▾
The Invariant

In any subtree of a complete binary tree: lh == rh if and only if every position in that subtree is filled (it is a perfect binary tree). Conversely, if any node is absent, the left spine will always be strictly longer than the right spine.

Proof sketch: The left spine reaches the deepest level by the definition of completeness. The right spine reaches the deepest level if and only if the last level is entirely full — i.e., the tree is perfect. So lh == rh is equivalent to the subtree being perfect.

Why O(log²n)?

  • Each function call does O(log n) work (two spine walks, each bounded by tree height).
  • Each call recurses into at most one child that is not yet a perfect tree — the other child is resolved by the formula in O(1).
  • The non-perfect subtree has at most half the remaining nodes, so recursion depth is O(log n).
  • Total: O(log n) recursive levels × O(log n) work per level = O(log²n).
Why not brute-force DFS (O(n))?

For n = 10^9 nodes, O(n) means a billion operations. O(log²n) means only ~900 operations (30²). The problem statement explicitly asks for better than O(n) — brute-force DFS fails this constraint even though it produces the correct answer.

Solution 2 — Left/Right Height Compare — OPTIMAL (O(log²n)) — Interview Pick

def _height(node, left: bool) -> int: ← walk one spine, count nodes visited
h = 0 ← start counting at 0
while node:
h += 1 ← count this node
node = node.left if left else node.right ← follow chosen spine
return h ← O(log n): spine length ≤ tree height
def countNodes(root) -> int:
if not root: ← base case
return 0
lh = _height(root, left=True) ← left spine: 1→2→4 gives lh=3
rh = _height(root, left=False) ← right spine: 1→3 gives rh=2
if lh == rh: ← equal heights → this subtree is perfect!
return (1 << lh) - 1 ← 2^lh - 1 nodes in a perfect tree of height lh
return 1 + countNodes(root.left) \ ← not perfect: recurse both children
+ countNodes(root.right) ← one child always resolves immediately
Why (1 << lh) - 1 and not just (1 << lh)?

A perfect binary tree of height lh (where height = node count on spine) has exactly 2lh − 1 nodes — not 2lh. Verify: height 1 (leaf only) → 2^1−1=1 node. Height 2 → 2^2−1=3 nodes. Height 3 → 2^3−1=7 nodes. The bit shift (1<<lh) = 2^lh is fast and avoids floating-point; subtract 1 for the correct count. Returning (1<<lh) without the −1 would overcount by exactly 1 at every fast-path return.

Solution 3 — Binary Search on Last Node (O(log²n), same asymptotic complexity)

def countNodes(root) -> int: ← different framing: find index of rightmost node on last level
if not root: return 0
h = 0; node = root ← measure tree height via left spine
while node.left: h += 1; node = node.left
lo, hi = 0, (1 << h) - 1 ← last level has 2^h positions, indexed 0 to 2^h-1
while lo < hi:
mid = (lo + hi + 1) // 2
if node_exists(root, h, mid): lo = mid ← binary search the last level
else: hi = mid - 1
return (1 << h) - 1 + lo + 1 ← (full levels above last) + (filled spots on last level)
5 Interview Strategy: How to Arrive at This ▾
Step 1: Clarify "complete" vs "full" vs "perfect"

Ask: "Is this guaranteed to be a complete binary tree — all levels full except last, last fills left-to-right?" Confirm the distinction: complete ≠ full (every node has 0 or 2 children) ≠ perfect (all leaves at same level, every level full). Only completeness unlocks the O(log²n) trick.

Step 2: State the brute force — O(n) DFS

"The naive solution: recursively count 1 + left + right. O(n) time, O(log n) stack space. Correct, but the problem asks for better than O(n). We need to exploit the complete tree structure."

Step 3: Exploit structure — one subtree is always perfect

"In a complete tree, if left and right spine heights are equal, the whole subtree is perfect — count with a formula. Otherwise recurse. Key: only one recursive path is ever non-perfect at each level. Depth = O(log n), work per level = O(log n) → O(log²n) total."

Step 4: Use bit shift for integer power of 2

"I use (1<<lh)−1 not 2**lh−1. Bit shift avoids float operations, is guaranteed O(1) in any language, and signals you know low-level integer tricks. The interviewer will notice this."

What to Say to the Interviewer

"I notice this is a complete binary tree — that's the key constraint.
Brute force DFS is trivial in O(n), but we can do better by exploiting structure.
I measure left and right spine heights at each node — both O(log n).
If they match: perfect subtree → (1<<lh)−1 nodes, returned in O(1).
If they differ: recurse both children — but one is always perfect, resolved instantly.
Recursion depth is O(log n). Work per level is O(log n). Total: O(log²n)."
Likely Follow-up Questions
  • "What's the index of the last node?" → Binary search on the last level, same O(log²n)
  • "Insert a node into a complete binary tree?" → Locate the next empty slot via last-node index, O(log²n)
  • "What if it's not a complete tree, just any binary tree?" → Fall back to O(n) DFS — no structural shortcut exists
  • "Can you do it iteratively (O(1) extra space)?" → Yes — simulate the recursion stack iteratively using node pointers

7 Summary & Key Takeaways ▾
Complete binary tree: last level fills left-to-right. This structural guarantee enables sub-linear counting.
Left spine height == right spine height is an O(log n) test for "is this subtree perfect?"
Perfect subtree node count = (1 << height) − 1. Use bit shift, never float exponentiation.
Each recursion level resolves at least one subtree in O(1). Depth is O(log n). Total: O(log²n).
Space complexity is O(log n) — the recursion stack is bounded by tree height, not node count.
