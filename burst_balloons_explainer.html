LeetCode #312

Burst Balloons

Given n balloons with values nums[i], burst them all to maximize coins. Bursting balloon i earns nums[left] × nums[i] × nums[right] using the still-existing neighbors at burst time.

Hard Dynamic Programming Interval DP · Last Burst Time: O(n³) Space: O(n²)
1 Understanding the Problem ▾

You have a row of balloons, each with a positive integer value nums[i]. When you burst balloon i, you earn nums[left] × nums[i] × nums[right] coins — where left and right are the still-existing neighbors at that moment. Once burst, neighbors collapse together. The goal: choose a burst order that maximizes total coins.

⚠️ Why "First Burst" Thinking Breaks Down

If you ask "which balloon should I burst first?", the state after each burst changes dynamically — neighbors collapse, creating a new array each time. Subproblem boundaries are unstable. This makes it impossible to define clean, reusable subproblems without exponential blowup.

Concrete Example: nums = [3, 1, 5, 8]. Add virtual boundary balloons with value 1 at each end → padded = [1, 3, 1, 5, 8, 1].

✗ Forward: messy subproblems

Burst 3: 1×3×1 = 3 coins
→ remaining: [1, 5, 8, 1]
Burst 1: 1×1×5 = 5 coins
→ remaining: [1, 5, 8, 1]
... neighbors change each time

✓ Backward: "last burst" is stable

If k is LAST burst in (l, r):
neighbors of k = l and r (fixed!)
coins = nums[l]×nums[k]×nums[r]
left half + right half: independent

💡 The Core Insight: Think About the LAST Balloon Burst

Instead of "which to burst first?", ask "which is the last to burst in a given interval?" If balloon k is the last burst in range (l, r) (open interval), then at its moment of bursting, its only neighbors are the permanent boundaries l and r. This gives a clean, non-overlapping decomposition into two sub-intervals (l, k) and (k, r) — both solved independently. This is the signature of Interval DP.

Edge Cases to consider: single balloon (n=1), all same values, very large values (up to 35, so product up to 35³ × n), nums = [1] (trivial).

2 The Algorithm: Interval DP — Last Burst Pivot ▾

Key setup: Pad nums with boundary sentinels: nums = [1] + nums + [1]. Now indices run 0 to n+1. Define dp[l][r] = maximum coins from bursting all balloons strictly inside the open interval (l, r), treating l and r as permanent boundaries.

1 Pad with 1s: Prepend and append 1. This makes every balloon's left and right boundary well-defined without out-of-bounds checks. The "virtual" boundary balloons are never burst.
2 Define state dp[l][r]: Maximum coins from bursting all original balloons in the open interval (l, r). Base case: if r − l < 2 (no balloon strictly inside), dp[l][r] = 0.
3 Recurrence — try each k as last burst: For every k in (l, r): coins = nums[l] × nums[k] × nums[r] + dp[l][k] + dp[k][r]. Take the max over all k.
4 Fill by increasing length: To compute dp[l][r], we need dp[l][k] and dp[k][r], which have smaller lengths. Iterate length from 2 upward — guaranteeing all needed sub-intervals are already solved.
5 Return dp[0][n+1]: The full original array sits inside the interval (0, n+1). This is the answer.
📐 The Recurrence Relations
dp[l][r] = max over k in (l, r) of: nums[l] × nums[k] × nums[r] + dp[l][k] + dp[k][r] Base case: dp[l][r] = 0 when r - l < 2 (no balloon inside) Answer: dp[0][n+1] where n = original nums length
🔑 Why dp[l][k] and dp[k][r] Are Independent

Because k is the last balloon burst in (l, r), it has NOT been burst when we compute dp[l][k] or dp[k][r]. Balloon k acts as a permanent right boundary for dp[l][k] and a permanent left boundary for dp[k][r]. Neither sub-interval shares any balloon — they're truly independent. This is why we can add them safely.

3 Why It Works: The DP Invariant ▾
🔐 The Invariant

At every point during computation, dp[l][r] correctly holds the maximum coins obtainable by optimally bursting every balloon strictly inside the open interval (l, r), given that balloons at positions l and r are permanent guardians (never burst). This is maintained by processing intervals in increasing order of length.

Optimal Substructure: Once we commit to k as the last burst in (l, r), the problem decomposes into two completely independent sub-problems. The optimal total = optimal for (l, k) + optimal for (k, r) + the final burst of k. No information from one sub-interval "leaks" into the other — they're separated by k, which hasn't been burst yet.

Overlapping Subproblems: Many outer intervals share the same inner interval. For example, dp[0][3] and dp[1][4] both need dp[1][3]. Without memoization or bottom-up filling, we'd recompute it separately each time. There are O(n²) unique intervals, each needing O(n) pivot tries → O(n³) total with DP.

Why greedy fails: Greedy (always burst highest or lowest value first) ignores the compounding effect. Bursting a low-value balloon early can set up a high-value triple-product later. Example: for [3,1,5,8], optimal solution coins = 167 (burst 1 last-ish to multiply with 3 and 5 — yielding 15 extra coins that greedy would miss).

⚠️ Why Open Intervals (Strictly Inside)?

We define dp[l][r] for the open interval (l, r) — meaning l and r are not burst themselves. This is intentional: l and r serve as the "still alive" neighbors when balloon k is burst. If we defined it over the closed interval [l, r], we'd need separate handling for the boundaries — the open-interval formulation keeps everything uniform and elegant.

4 The Code (Annotated) ▾

Solution 1 — Bottom-Up Tabulation (Optimal / Interview Pick)

def maxCoins(nums: list[int]) -> int:
nums = [1] + nums + [1] ← Add boundary sentinels; never burst them
n = len(nums) ← n = original_length + 2
dp = [[0] * n for _ in range(n)] ← dp[l][r] = max coins inside open interval (l,r)
for length in range(2, n): ← Iterate interval sizes: 2 = smallest non-empty
for l in range(n - length): ← Left boundary; must leave room for r = l+length
r = l + length ← Right boundary of this interval
for k in range(l + 1, r): ← k = last balloon burst in (l, r)
coins = (nums[l] * nums[k] * nums[r] ← Final burst: l and r still present as neighbors
+ dp[l][k] + dp[k][r]) ← Sub-intervals already computed (smaller length)
dp[l][r] = max(dp[l][r], coins) ← Keep the best last-burst pivot
return dp[0][n - 1] ← Full interval (0, n-1) = entire original array
⚠️ Why iterate by length instead of by l or r?

When filling dp[l][r], we access dp[l][k] and dp[k][r] for k in (l, r). Both have strictly smaller interval lengths (r − l). By iterating length from 2 upward, all smaller lengths are guaranteed filled before we need them. If we iterated naively by l then r, we'd read dp[l][k] before it was computed.

Solution 2 — Top-Down Memoization (same complexity, clearer recursion)

def maxCoins(nums: list[int]) -> int:
nums = [1] + nums + [1] ← Same padding
n = len(nums)
from functools import lru_cache
@lru_cache(maxsize=None) ← Auto-memoize: cache results by (l, r)
def dp(l, r): ← Returns max coins in open interval (l, r)
if r - l < 2: return 0 ← No balloon inside → 0 coins
return max(
nums[l] * nums[k] * nums[r] ← Burst k last; l and r are alive neighbors
+ dp(l, k) + dp(k, r) ← Recurse: left and right sub-intervals
for k in range(l + 1, r) ← Try every possible last pivot
)
return dp(0, n - 1) ← Full padded interval
💡 Complexity Summary

Brute Force (all orderings): Time O(n!), Space O(n) — too slow for n > 12
Tabulation (bottom-up): Time O(n³), Space O(n²) — O(n²) states × O(n) pivots each
Memoization (top-down): Time O(n³), Space O(n²) — same states, plus call stack
With n ≤ 500, n³ ≈ 125M operations — feasible in ~1 second

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Pattern Signals

Key signals for Interval DP: (1) problem involves a sequence, (2) you process all elements in some order, (3) adjacent elements interact (cost depends on neighbors), (4) the "first action" approach gives shifting boundaries. When these signals appear together, think Interval DP with "last action" framing.

Step 2: State the Stuck Point Explicitly

"If I ask which balloon to burst first, the answer changes the neighbors for every subsequent burst — subproblems are unstable. Can I flip the question?" Then propose: "What if I ask which balloon to burst last in a given range? Its neighbors would be the fixed boundaries of that range." This shows you understand the core difficulty.

Step 3: Define State and Recurrence Out Loud

Define dp[l][r] = max coins from the open interval (l, r). Explain the padding trick. Write the recurrence: dp[l][r] = max over k of nums[l]×nums[k]×nums[r] + dp[l][k] + dp[k][r]. Justify why dp[l][k] and dp[k][r] are independent (k hasn't been burst yet when solving them).

Step 4: Choose Bottom-Up and Explain Ordering

Propose tabulation over memoization for interview (no call stack overhead). Explain why you iterate by length: "I need smaller intervals before larger ones, so length-first guarantees that." Complexity: O(n³) time, O(n²) space. Mention that for n ≤ 500, this is well within limits.

🗣️ What to Say to the Interviewer

"My initial thought: simulate bursting in order. But each burst changes neighbors, making subproblems unstable — greedy won't work.

Flip the perspective: which balloon is burst LAST in a given interval? If k is last in (l, r), its neighbors are always the interval boundaries l and r — stable! So the coins from that burst = nums[l] × nums[k] × nums[r].

Before k is burst, everything else in (l, k) and (k, r) was already burst. These are independent sub-intervals. dp[l][r] = max over k of (nums[l]×nums[k]×nums[r] + dp[l][k] + dp[k][r]).

I'll pad with 1s, iterate length 2 to n, fill bottom-up. Answer = dp[0][n-1]. Time O(n³), Space O(n²)."
🔮 Likely Follow-up Questions
  • "Reconstruct the optimal burst order?" → Track the best k for each dp[l][r] in a parent table; recurse to recover order
  • "What if bursting costs a fixed fee?" → Subtract fee per burst: coins = nums[l]×nums[k]×nums[r] - fee + dp[l][k] + dp[k][r]
  • "Same problem in a circle?" → Break the circle at a fixed point; apply same DP with adjusted indices
  • "What's this pattern called?" → Interval DP; related: Matrix Chain Multiplication, Strange Printer

🎯 When to Recognize Interval DP
  • Sequence (array/string) where you process all elements in some order
  • Cost depends on adjacent/neighboring elements — order matters
  • "First action" framing makes subproblem boundaries shift → try "last action"
  • Optimal for a range = optimal for two sub-ranges + some combining cost
  • Time complexity hint: O(n³) is expected for n ≤ 500
