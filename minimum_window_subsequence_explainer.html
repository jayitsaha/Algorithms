LeetCode #727

Minimum Window Subsequence

Given strings s1 and s2, find the minimum-length contiguous substring of s1 that contains s2 as a subsequence — characters in order, not necessarily contiguous.

Hard Sliding Window Two Pointers Time: O(|s1|·|s2|) Space: O(1)
1 Understanding the Problem ▾

We need to find the shortest contiguous substring of s1 that contains s2 as a subsequence. A subsequence means all characters of s2 appear in order within the window — but they don't need to be adjacent; the window just has to be contiguous in s1.

💡 Key Insight: Subarray vs Subsequence

This is NOT the same as Minimum Window Substring (#76). That problem needs all characters present in any order. Here, characters of s2 must appear in their exact left-to-right order within the chosen window. Order is preserved; gaps between characters are allowed.

✓ s1="abcdebdde", s2="bde"

"bcde" (idx 1–4): b at 1, d at 3, e at 4 ✓
"bdde" (idx 5–8): b at 5, d at 6, e at 8 ✓
Both length 4 — first one wins.
Answer: "bcde" (length 4)

✗ Common confusion

Cannot cherry-pick b at idx 1,
d at idx 6, e at idx 8 — those
chars are not a contiguous window!
We need a contiguous substr of s1.

Edge Cases: s2 longer than s1 → "". s2 = "" → "". s1 = s2 → s1. No valid window exists → "".

🔑 Subsequence vs Substring — the Critical Distinction

Substring: "abc" in "xabcx" — characters must be contiguous.
Subsequence: "ace" in "abcde" — characters in order, gaps allowed.
This problem: find the shortest substring (contiguous) of s1 that contains s2 as a subsequence (in-order, gapped).

2 The Algorithm: Forward-Backward Two-Pointer Sweep ▾

The key idea is a two-phase sweep per anchor position. From each anchor in s1 we scan forward to find where s2 is fully matched (right boundary), then scan backward to find the tightest left boundary. This gives the shortest window ending at that right point.

Phase 1: Forward →
From anchor i, advance through s1 matching s2 left-to-right. Stop when all of s2 is consumed. Record right end.
→
Phase 2: Backward ←
Walk backward from right end, matching s2 right-to-left. Greedily assigns latest positions, giving the tightest left boundary.
→
Phase 3: Update
Compare window length. If shorter, update best. Next anchor = found start + 1.
1 Initialize: i = 0 (scan position in s1), best = "". We scan s1 left to right.
2 Phase 1 (Forward): Reset j = 0. Advance i through s1; when s1[i] == s2[j], increment j. Always increment i. Stop when j == len(s2) (all matched) or i exhausts s1.
3 Phase 2 (Backward): Save end = i. Reset j = len(s2)-1, decrement i by 1. Walk left: when s1[i] == s2[j], decrement j. Always decrement i. Stop when j < 0. Then start = i + 1.
4 Update & advance: Window is s1[start:end]. Update best if shorter. Set i = start + 1 and repeat — the next anchor begins right after the start we found.
⚠️ Why Not Standard Sliding Window?

Classic sliding window works when we only care about character presence (a frequency map). Here, order matters — s2 must appear as a subsequence in a specific left-to-right order. The backward pass is the key optimization: it finds the rightmost valid start for each discovered right endpoint, eliminating redundant searches.

3 Why It Works: The Invariant ▾
🔐 The Invariant

After the backward scan completes at start: the substring s1[start:end] is the shortest valid window that ends exactly at position end-1. No shorter window ending there can contain s2 as a subsequence — the backward scan matched s2's characters as late (rightward) as possible, greedily shrinking the left side.

Why the backward pass finds the minimum left boundary: In the forward pass we consume s2 characters greedily from the left — each match happens at the earliest available position. This gives us a valid right endpoint end. Starting from end, the backward pass matches s2 from right-to-left: each time s1[i] == s2[j] we take that match (the rightmost occurrence). This greedily assigns each s2 character to the latest possible s1 position, making the window as tight as possible.

Why next anchor = start + 1: Any window starting before or at start ending at the same end would only be longer. But a new anchor at start + 1 might find an entirely different (and possibly shorter) window ending further right. We don't skip anything — we just avoid re-examining windows we know are suboptimal.

⚠️ Off-By-One: Why start = i + 1?

In the backward scan loop, when s1[i] == s2[j] we decrement j, then we always decrement i — even on the final match. So when the loop exits (j < 0), i has been decremented one extra time. The true start of the window is i + 1, not i. This is the most common bug when implementing this algorithm.

Correctness guarantee: Every minimal window (the shortest for each possible right endpoint) is inspected exactly once. Since the global minimum must be minimal for some right endpoint, we will find it.

4 The Code (Annotated) ▾

Solution 1 — Two-Pointer Forward-Backward (Optimal, Interview Pick)

def minWindow(s1: str, s2: str) -> str:
m, n = len(s1), len(s2) ← cache lengths
i, best = 0, "" ← i = current scan position; best = shortest window
while i < m: ← outer loop over anchor positions
j = 0 ← reset s2 pointer for each anchor
while i < m and j < n: ← Phase 1: forward scan
if s1[i] == s2[j]: ← s1's current char matches next needed s2 char?
j += 1 ← yes: advance s2 pointer
i += 1 ← always advance s1 pointer
if j < n: break ← s2 never fully matched — no more windows possible
end = i ← i is one past the right end; save it
j = n - 1 ← reset to last s2 char for backward pass
i -= 1 ← step back to last matched s1 position
while j >= 0: ← Phase 2: backward scan — tighten left boundary
if s1[i] == s2[j]: ← right-to-left match of s2?
j -= 1 ← consume s2 char from the right
i -= 1 ← always step left (even on match)
start = i + 1 ← i is one past the left end; actual start = i+1
if not best or end-start < len(best):← shorter window?
best = s1[start:end] ← update best
i = start + 1 ← next anchor: just after where this window began
return best ← "" if no window found
⚠️ The start = i + 1 Bug

In the backward loop, i decrements after every iteration — including after the last match. When j hits -1 and the loop exits, i is one position too far left. Always write start = i + 1, never start = i. Verify on paper: for "bcde" the backward pass should yield start=1, not start=0.

Solution 2 — DP Table O(|s1|·|s2|) time and space

def minWindow_dp(s1: str, s2: str) -> str:
m, n = len(s1), len(s2)
# dp[i][j] = start index in s1 such that s1[dp[i][j]:i]
# contains s2[0:j] as a subsequence, or -1 if none
dp = [[-1]*( n+1) for _ in range(m+1)]
for i in range(m+1): dp[i][0] = i ← empty s2 matched: start=current pos
best = ""
for i in range(1, m+1):
for j in range(1, n+1):
if s1[i-1] == s2[j-1]:
dp[i][j] = dp[i-1][j-1] ← chars match: inherit start from diag
else:
dp[i][j] = dp[i-1][j] ← no match: skip s1[i], propagate start
if dp[i][n] != -1:
window = s1[dp[i][n]:i]
if not best or len(window) < len(best):
best = window
return best
💡 Complexity Summary

Brute Force (all substrings): Time O(|s1|²·|s2|), Space O(|s1|) — too slow
Two-Pointer (Optimal for interviews): Time O(|s1|·|s2|) worst case, Space O(1)
DP Table: Time O(|s1|·|s2|), Space O(|s1|·|s2|) — useful for extensions/variants

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Clarify the Subsequence Requirement

Immediately say: "This requires s2 as a subsequence — characters in order, gaps allowed — not just all characters present. That rules out the standard sliding-window approach from Minimum Window Substring."

Step 2: Propose Brute Force

"Try every pair (start, end) in s1, check if the window contains s2 as a subsequence. O(|s1|²×|s2|) — correct but too slow."

Step 3: Recognize the Forward-Backward Optimization

"Key observation: once I find a right endpoint via forward matching, I can find the tightest left boundary by scanning backward — matching s2 from right-to-left. This eliminates O(|s1|) redundant inner loops."

Step 4: Handle the Off-By-One before Coding

"In the backward loop I decrement i unconditionally — including on the final match — so the real start is i+1, not i. I'll verify this on my test case before finalizing."

🗣️ What to Say to the Interviewer

"I'll anchor at each position i in s1 and run a forward scan to match s2 left-to-right, giving me a right endpoint.
Then I scan backward from that endpoint matching s2 right-to-left to get the tightest left boundary — start.
The window is s1[start:end]. I compare with best, then advance i to start+1 for the next candidate.
Time O(|s1|·|s2|) in the worst case, O(1) space. This is the interview-preferred solution."
🔮 Likely Follow-up Questions
  • "Can you solve with DP?" → Yes: dp[i][j] = earliest start in s1 such that s1[start:i+1] contains s2[0:j+1]. O(m×n) time and space.
  • "How does this differ from Minimum Window Substring?" → #76 uses a frequency map; here order matters so we need the sequential forward-backward sweep.
  • "What if s2 can overlap with itself?" → Already handled — we return the lexicographically first shortest window when lengths tie (first found wins).
  • "Can you do O(n log n)?" → Not simply; O(mn) is the standard optimal for this problem class.

7 Summary & Key Takeaways ▾
Subsequence ≠ Substring: order is preserved, gaps allowed; but the window must be contiguous in s1
Forward scan finds the right boundary; backward scan tightens the left — two passes, two directions
start = i + 1 after backward scan — i is always decremented one extra time on the final match
Next anchor = start + 1 — guarantees no valid shorter window is skipped
O(1) space — no frequency map, no DP table; two pointers and two integers suffice

Brute Force (all substrings) O(|s1|²·|s2|) O(|s1|) Check every window for subsequence
Two-Pointer Fwd-Bwd ✓ O(|s1|·|s2|) O(1) Interview pick; forward+backward sweeps

🎯 When to Recognize the Forward-Backward Pattern
  • "Shortest/minimum window containing X as a subsequence" → Forward-Backward Two-Pointer
  • Standard sliding window fails because order matters → think sequential subsequence matching
  • Right boundary easy (greedy forward match); left needs tightening → backward scan
  • Next anchor starts one past where previous window began
