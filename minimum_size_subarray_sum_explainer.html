LeetCode #209

Minimum Size Subarray Sum

Given an array of positive integers and a target, find the minimal length of a contiguous subarray whose sum is ≥ target. Return 0 if no such subarray exists.

Medium Sliding Window Shrink When Sum ≥ Target Time: O(n) Space: O(1)
1 Understanding the Problem ▾

You have an array of positive integers and a target value. You need to find the shortest contiguous subarray (a slice of consecutive elements) whose elements add up to at least the target. Return its length, or 0 if no such subarray exists.

💡 Key Insight

Because all numbers are positive, adding more elements to a window always increases the sum, and removing elements always decreases it. This monotonic property lets a sliding window work perfectly: once the window sum hits the target, we can shrink from the left to find the tightest possible valid window, without fear of missing anything.

Example: target=7, nums=[2,3,1,2,4,3]

[2,3,1,2] → sum=8 ≥ 7, len=4
[1,2,4] → sum=7 ≥ 7, len=3
[4,3] → sum=7 ≥ 7, len=2
Minimum: 2

Why brute force hurts

Check all (l, r) pairs → O(n²).
For n=10⁵, that's 10¹⁰ operations.
Exceeds time limit. Need O(n).

Edge Cases to consider: target exceeds the total array sum (return 0), single element ≥ target (return 1), all elements equal to target/2 so the whole array is required, and duplicate values in the array (handled naturally by the sum).

2 The Algorithm: Variable Sliding Window ▾

We maintain a variable-size window defined by two pointers — left and right. The right pointer sweeps forward expanding the window; whenever the current window sum reaches the target, we aggressively shrink from the left to find the smallest valid window at this right endpoint.

1 Initialise: left = 0, curr_sum = 0, min_len = infinity.
2 Expand right: For each index right from 0 to n-1, add nums[right] to curr_sum.
3 Shrink while valid: While curr_sum ≥ target, update min_len = min(min_len, right - left + 1), then subtract nums[left] from curr_sum and advance left.
4 Return result: If min_len is still infinity, no valid subarray exists — return 0. Otherwise return min_len.
🔑 "While" not "If" — the crucial detail

We shrink with a while loop, not an if statement. After removing nums[left], the window might still sum to ≥ target. Since all values are positive, each additional shrink step could reveal a shorter valid window. We must exhaust all possible shrinks before advancing right.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every point in the outer loop, curr_sum equals the exact sum of nums[left..right]. The inner while loop exhausts every valid (left, right) pair for the current right endpoint — from widest to tightest — before right advances further.

Why we never miss the optimal window: Suppose the optimal answer is the subarray from index L to index R. When right reaches R, the inner while loop kicks in and shrinks left from its current position. Because all values are positive, the sum only decreases as we shrink. We will reach left = L (recording length R - L + 1) and possibly go further still — but we will never skip it.

Why O(n) despite the nested while loop: The left pointer only moves forward. Each element is added to curr_sum exactly once when right passes it, and subtracted at most once when left passes it. Total operations ≤ 2n → O(n).

⚠️ Why this ONLY works with positive numbers

The sliding window is valid here because all values are positive — extending the window always increases the sum, shrinking always decreases it. If negative numbers were allowed, removing an element might actually increase the sum. The monotonic property breaks completely, and you would need a different approach (e.g., prefix sums with a monotonic deque, as in LeetCode #862 — the hard variant of this exact problem).

4 The Code (Annotated) ▾

Solution 1 — Sliding Window (Optimal, O(n) time — Interview Pick)

def minSubArrayLen(target: int, nums: List[int]) -> int:
left = 0 ← Left boundary of the window, starts at index 0
curr_sum = 0 ← Sum of all elements currently in the window
min_len = float('inf') ← Best valid window size seen; infinity = "not found yet"
for right in range(len(nums)): ← Expand window by advancing right pointer
curr_sum += nums[right] ← Include the new right element into the window
while curr_sum >= target: ← Window is valid — keep shrinking to find tightest fit
min_len = min(min_len, ← Record this window's length BEFORE shrinking
right - left + 1)
curr_sum -= nums[left] ← Remove leftmost element from window sum
left += 1 ← Shrink window from the left
return 0 if min_len == float('inf') else min_len ← 0 means no valid window found
⚠️ Common Mistake: Shrink before recording

Always record the length before removing nums[left]. If you shrink first and then compute length, you're measuring a window that might no longer meet the target. The order must be: record → subtract → advance left.

Solution 2 — Brute Force (O(n²) time — mention first, then optimise)

def minSubArrayLen(target, nums):
n, min_len = len(nums), float('inf')
for l in range(n): ← Try every possible left boundary
total = 0
for r in range(l, n): ← Extend right until sum reaches target
total += nums[r]
if total >= target:
min_len = min(min_len, r - l + 1)
break ← Smallest r for this l; no need to look further right
return 0 if min_len == float('inf') else min_len
💡 Complexity Summary

Sliding Window: Time O(n) — each element touched at most twice. Space O(1).
Brute Force: Time O(n²) — all (left, right) pairs. Space O(1).
Prefix Sum + Binary Search: Time O(n log n). Space O(n). A useful middle-ground when you want to reduce from O(n²) but can't achieve O(n).

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Spot the Pattern Signals

"Contiguous subarray" + "positive integers" + "optimise length against a sum threshold" — these three signals together almost always mean sliding window. Explicitly verifying the positive-integers constraint is what distinguishes this from the hard variant (#862).

Step 2: Propose Brute Force First

"The straightforward approach is O(n²): for every left index, extend right until the sum ≥ target and record the length. Correct, but too slow for n=10⁵. I can do this in O(n) with a sliding window."

Step 3: Justify the Sliding Window

"Since all values are positive, the window sum is monotonically increasing as we expand and monotonically decreasing as we shrink. This means for any right endpoint, we can find the tightest valid left endpoint by greedily shrinking — no backtracking needed."

Step 4: Handle Edge Cases Out Loud

"If no valid window is found, min_len stays infinity — return 0. A single element ≥ target is handled automatically: right=0 adds it, the while fires, records length 1, left overshoots right, and then the inner loop exits since curr_sum drops to 0."

🗣️ What to Say to the Interviewer

"All numbers are positive — that's the key constraint that unlocks O(n).
I'll use two pointers: right always moves forward adding to the window sum.
Whenever the sum hits the target, I've found a valid window. I record its length,
then shrink from the left — because the values are positive, shrinking always decreases
the sum, so I know exactly when to stop. I keep shrinking until the sum drops below target.
Each element is added once and removed at most once: O(n) total, O(1) space."
🔮 Likely Follow-up Questions
  • "What if the array contains negative numbers?" → Sliding window breaks. Use prefix sums + monotonic deque for O(n), or binary search on prefix sums for O(n log n). See LeetCode #862.
  • "Can you solve it in O(n log n) if you can't remember the sliding window trick?" → Yes: prefix sums array + binary search for smallest r such that prefix[r] - prefix[l] ≥ target.
  • "What's the maximum size subarray with sum ≤ target?" → Same window structure, but the "valid" condition flips — you expand while valid and record before shrinking.
  • "Minimum window substring?" → LeetCode #76 — same shrink-when-valid pattern, but the "valid" condition tracks character frequencies instead of a sum.

7 Summary & Key Takeaways ▾
All-positive arrays have monotonic sum — adding always increases, removing always decreases. This is the foundation of the entire algorithm.
Despite the nested while loop, each element is touched at most twice (add + remove), giving O(n) total time.
Use a while loop to shrink, not an if — after one shrink the window might still be valid, revealing an even shorter solution.
Record the window length BEFORE shrinking. The window is valid at that moment; measuring after removal gives the wrong size.
Initialise min_len to infinity (not n+1) so you can distinguish "no valid window found" from "entire array is the answer".
🎯 When to Use This Pattern
  • Problem asks for the shortest or longest contiguous subarray
  • Array contains all positive (or all non-negative) integers
  • Constraint involves a sum threshold (≥ or ≤ some target)
  • O(n) time is required — O(n²) brute force is too slow
  • You want to minimise or maximise window size subject to a sum condition
🔑 The Core Template (Memorise This)
