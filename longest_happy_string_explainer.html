LeetCode #1405

Longest Happy String

Given integers a, b, c, construct the longest possible string using at most a 'a's, b 'b's, and c 'c's such that no substring "aaa", "bbb", or "ccc" exists.

Medium Heaps Max Heap Greedy Time: O(n) Space: O(1)
1 Understanding the Problem ▾

A happy string (also called a "diverse string" in some variants) is a string that never contains three or more consecutive identical characters. We have a budget of a copies of 'a', b copies of 'b', and c copies of 'c'. We don't have to use them all — just build the longest valid string possible.

💡 Key Insight

Always greedily pick the character with the highest remaining count — but never if it would create three in a row. When you can't use the top character, temporarily use the second-most abundant character for one step to break the streak, then resume picking the top character.

✓ Valid: a=1, b=1, c=7 → "ccbccacc"

✗ Invalid: "cccbcca"

Edge cases to consider: All zero counts (empty string), only one character type (e.g., a=5, b=0, c=0 → "aa" — max 2 of any single type), all equal counts (e.g., a=2, b=2, c=2 → alternate freely), one count much larger than others (greedy must use fallback character repeatedly).

🔑 Why Maximum Consecutive Run = 2

The constraint bans "aaa", "bbb", "ccc" — so any character can appear at most twice consecutively. This means at each decision point we either continue a run of 1 (making it 2) or start a new run — and we must interrupt any current run of 2 before adding more of the same character.

2 The Algorithm: Max Heap Greedy ▾

At each step, we make a single greedy choice: pick the character with the highest remaining supply, as long as it won't create a triple. A max-heap (priority queue by count) gives us the highest-count character in O(log k) time per step, where k=3 (number of distinct characters).

1 Initialize the heap: For each character (a, b, c) with a non-zero count, push (-count, char) into a min-heap. Negating the count simulates a max-heap in Python's heapq (which is a min-heap by default).
2 Pop the max-count character: Use heapq.heappop to get the character with the highest remaining count.
3 Triple check: Would appending this character create "three in a row"? Check: len(res) >= 2 and res[-1] == ch and res[-2] == ch. If YES, go to step 4. If NO, go to step 5.
4 Fallback — use the second character: Pop the second-highest-count character. Append it, decrement its count, push it back. Push the original top character back unchanged. If there's no second character, stop — we're stuck.
5 Normal — append the top character: Append it, decrement count, push back if count > 0. Loop back to step 2.

3 Why It Works: The Greedy Invariant ▾
🔐 The Greedy Invariant

At every step, the character with the highest remaining count should be placed as often as safely possible. Choosing a lower-count character when the top is safe would waste the most "urgent" supply — the character with the largest count is most likely to get stranded (become unusable if only it remains).

Why greedy is optimal (exchange argument): Suppose an optimal solution at some step uses character X when character Y has a higher count and could be used safely. Swapping X for Y in that position either maintains or increases the resulting string length (because Y has more "fuel" left). So always picking the max-count character when safe is at least as good as any alternative.

Why the fallback (second character) is safe: When we must use the second character, we use exactly one of it — just enough to break the current streak. We then immediately return to the most-available character. We never usemore interruption characters than necessary.

Termination is correct: The only time we stop early is when the only remaining character would create a triple and no alternative exists. At that point, no further progress is possible regardless of strategy — we've truly found the longest achievable string.

⚠️ Why Not Always Append 2 at Once?

A tempting optimization: "if I'm going to pick 'c', why not add 'cc' immediately?" This can break the algorithm — you might add "cc" when the last character was already 'c', creating "ccc". The heap approach is cleaner: one character at a time, always checking the triple condition freshly.

4 The Code (Annotated) ▾

Solution 1 — Max Heap Greedy (Optimal, Interview Pick)

import heapq
def longestDiverseString(a: int, b: int, c: int) -> str:
heap = [] ← Will hold (-count, char) pairs
for cnt, ch in [(a,'a'),(b,'b'),(c,'c')]:
if cnt > 0: ← Skip zero-count chars (nothing to place)
heapq.heappush(heap, (-cnt, ch))← Negate: Python min-heap becomes max-heap on count
res = [] ← Build result as list; join at end (faster than += str)
while heap:
cnt, ch = heapq.heappop(heap) ← Pop highest-count char (most negative = highest count)
cnt = -cnt ← Un-negate to get the actual count
if len(res)>=2 and res[-1]==ch and res[-2]==ch:
← Triple check: last 2 chars == this char?
if not heap: ← No fallback available → we're stuck, stop
break
cnt2, ch2 = heapq.heappop(heap) ← Pop second-best character
cnt2 = -cnt2
res.append(ch2) ← Append second char to break the streak
if cnt2 - 1 > 0: ← Push second char back if it has remaining supply
heapq.heappush(heap, (-(cnt2-1), ch2))
heapq.heappush(heap, (-cnt, ch)) ← Push the top char back (we didn't use it this step)
else:
res.append(ch) ← No triple risk: safely append the top character
if cnt - 1 > 0: ← Push back with count decremented by 1
heapq.heappush(heap, (-(cnt-1), ch))
return ''.join(res) ← Join list to string
⚠️ Common Mistake: Checking Only the Last Character

A frequent bug: writing res[-1] == ch instead of the full triple check. This incorrectly prevents "ab" being followed by "b" (which is fine — only "bb" followed by "b" is bad). You must check BOTH res[-1] == ch AND res[-2] == ch, and also guard with len(res) >= 2.

Solution 2 — Simple Sort Greedy (cleaner, since only 3 chars)

def longestDiverseString(a: int, b: int, c: int) -> str:
counts = {'a': a, 'b': b, 'c': c}
res = []
while True:
# Sort 3 chars by count desc — O(1) since only 3 items
order = sorted(counts, key=lambda x: -counts[x])
placed = False
for ch in order:
if counts[ch] == 0: continue
if len(res)>=2 and res[-1]==ch and res[-2]==ch: continue
res.append(ch); counts[ch] -= 1
placed = True; break ← Pick first valid in sorted order
if not placed: break ← No valid placement found: stop
return ''.join(res)
💡 Complexity Summary

Heap Greedy: Time O(n log 3) = O(n) where n = a+b+c; Space O(3) = O(1) (heap has at most 3 items)
Sort Greedy: Time O(n) — sorting 3 items each step is O(1); Space O(1)
Both are optimal. The heap version generalizes to k character types (O(n log k)) — a common interview follow-up.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Spot the Greedy Signal

"We want to maximize string length with a local constraint (no triple). Maximization + local constraint with independent choices → think greedy. Ask: what's the optimal choice at each step?"

Step 2: Identify the Greedy Choice

"To maximize length, always deplete the character with the most remaining count. Using it often prevents getting stuck with a large pile of one character we can't safely place. This is the core greedy insight."

Step 3: Handle the Triple Constraint

"When the top character would create a triple, we use the second-most abundant for exactly one step, then resume. This is the minimum interruption needed — one character to break the streak, then back to greedy."

Step 4: Choose Data Structure

"We repeatedly need 'the highest count character right now' — that's a max-heap. With only 3 characters, even a simple sort works, but propose the heap for generalizability to k character types."

🗣️ What to Say to the Interviewer

"My approach is greedy with a max-heap. At each step I pick the character with the most remaining count.
If that would create 'aaa'/'bbb'/'ccc', I instead use the second-most abundant character for one step.
If I can't even do that (only one type remains and it would triple), I stop — that's the optimal length.
A max-heap gives me the most-frequent character in O(log k) per step.
Total time: O(n log k) where n = total chars, k = character types. Space: O(k).
This generalizes: if you have k character types with the same constraint, the algorithm is identical."
🔮 Likely Follow-up Questions
  • "What if there are k character types?" → Same heap, O(n log k) — identical approach
  • "What if max consecutive run is m (not 2)?" → Track a streak counter; use fallback when streak hits m
  • "Is the result unique?" → No — multiple valid longest happy strings may exist
  • "Prove optimality." → Exchange argument: swapping to a lower-count char when top is safe can only hurt length

🎯 When to Recognize the Max Heap Greedy Pattern
  • "Maximize length/count" with a "no consecutive runs of k" constraint → Greedy + Heap
  • Scheduling tasks with "cooldown between same type" → Task Scheduler pattern
  • "Rearrange so no two adjacent elements are the same" → Reorganize String pattern
  • Repeatedly need "most frequent available item" → Max Heap
