LeetCode #425

Word Squares

Given a set of words (without duplicates), return all the word squares you can build from them. A word square is a sequence of k words where the k-th row and column read the same string.

Hard Tries Trie + Backtracking Time: O(n·L·26^L) Space: O(n·L)
1 Understanding the Problem ▾

A word square is a grid of words where reading across (rows) and reading down (columns) gives the same sequence. Formally: square[i][j] = square[j][i] for all i, j. Every row word must equal the column word at the same index.

✓ Valid: ["ball","area","lead","lady"]

b a l l
a r e a
l e a d
l a d y
Col 0="ball", Col 1="area" ✓

✗ Invalid: ["ball","area","lead","sage"]

b a l l
a r e a
l e a d
s a g e
Col 0="bals" ≠ "ball" ✗

💡 Key Insight: Symmetry Determines Future Prefixes

Once you place k words in rows 0…k-1, the k-th column is already partially written. The word you place in row k must start with that partial column. This prefix constraint is what makes backtracking with a Trie so powerful: instead of checking all n words for each row, you instantly retrieve only words matching the required prefix.

Concrete example — placing row 2 with "ball" in row 0 and "area" in row 1:

Column 2 so far: ball[2] = 'l', area[2] = 'e'
Required prefix for row 2: "le"
get_words("le") → ["lead"] ← only this word qualifies

Edge Cases: Empty list → return []. All length-1 words → each word is its own 1×1 square. Words with no shared prefixes → all branches prune immediately.

2 The Algorithm: Trie-Guided Backtracking ▾

We combine two techniques: a Trie with word lists at every node (for O(L) prefix queries) and recursive backtracking (to explore all valid orderings). The Trie does the heavy lifting of pruning — without it, we'd scan all n words for each row position.

1 Build the Trie: Insert every word. At each node, store the complete word list of all words passing through that node. This is the key difference from a standard Trie: each internal node knows every word in its subtree.
2 Outer loop: Try every word as row 0. For each starting word, call backtrack(step=1, square=[word]).
3 Derive prefix constraint: At row k, the required prefix = square[0][k] + square[1][k] + … + square[k-1][k]. This is the k-th column read from rows already placed.
4 Query Trie: get_words(prefix) walks the Trie and returns the pre-stored word list at that node. O(L) walk, O(1) return. If node absent → return [] → prune this branch immediately.
5 Recurse and backtrack: For each candidate word: append to square → recurse(step+1) → pop. Base case: step == n → record the completed square.
🔑 Prefix Formula

prefix = "".join(square[i][step] for i in range(step))

This extracts column step from the first step rows already in the square. It tells us exactly what the next word must begin with.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At all times during backtracking with k words placed: for every pair i,j < k, square[i][j] = square[j][i]. The partial square is always "symmetric" for its filled region.

Why it stays true: When placing word W as row k, we enforce that W's characters 0…k-1 match column k's characters 0…k-1. That is: W[j] = square[j][k] for j < k. Since square[j][k] = (the k-th character of word in row j) = word_j[k], and we need W[j] = word_j[k], the prefix check enforces exactly this. After placing W, square[k][j] = W[j] = word_j[k] = square[j][k] for all j < k. Symmetry is preserved for row/col k.

⚠️ Why Word Lists at Every Node (Not Just Leaves)?

Standard Trie: words stored only at leaves. To get all words with prefix "ba", you'd DFS the subtree rooted at node "ba" — O(subtree size) per query.

Enhanced Trie: every node stores its complete word list, built at insert time. Query cost is O(L) to traverse + O(1) to read the list. Total insert cost: O(n·L) entries, O(n·L) total space. This is a classic time-space trade.

Why backtracking finds all squares: The recursion tree explores every possible combination of words for each row position, subject to the prefix constraint. Because we try every word as row 0 in the outer loop, and backtracking explores all valid completions from each starting point, we are guaranteed to enumerate all valid word squares without duplicates (since words are distinct and ordering matters).

⚠️ Common Mistake: backtrack(0, []) instead of backtrack(1, [word])

Starting at step=0 with empty square works too, but then the prefix for step=0 is "" — meaning all words are candidates. It's equivalent but slightly cleaner to seed the square with each word and start at step=1. Either way is correct; the outer loop handles row 0 selection.

Alternative — Prefix Hash Map (simpler code, same complexity):

from collections import defaultdict
prefix_map = defaultdict(list) → prefix → list of words
for word in words: → precompute all prefixes
for i in range(len(word)+1): → word[0:0]="" to word[0:n]=word
prefix_map[word[:i]].append(word) → O(n·L²) space (vs O(n·L) for Trie)
⏱️ Complexity

Trie + Backtrack: Time O(n·L·26^L), Space O(n·L). The 26^L factor is the theoretical branching factor — in practice, the Trie prunes most branches to near-zero. Prefix Map: Same time complexity, Space O(n·L²) for storing all prefix strings explicitly.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize "Find ALL valid arrangements" → Backtracking

Any problem asking for all valid configurations (not just one, not a count) with complex constraints almost always calls for backtracking. The word square symmetry constraint linking rows and columns is the "complex constraint" signal here.

Step 2: Derive the Prefix Constraint Before Coding

The single most important insight: at row k, the first k characters of the word are already determined by column k. Write out a small example: place "ball" and "area", then ask "what must row 2 start with?" Trace ball[2]='l', area[2]='e' → prefix "le". This is the pruning key.

Step 3: Choose the Lookup Structure — Trie vs Hash Map

Mention both: a prefix hash map (defaultdict of prefix → word list) is simpler to implement. A Trie is more memory-efficient but harder to code. In an interview, the hash map approach is often accepted and easier to get right under pressure. Mention the Trie as the "textbook optimal".

Step 4: Code the Backtracking Template

Base case: step == n → append copy. Recursive: compute prefix, get candidates, for each: append → recurse → pop. The append/pop pattern guarantees clean state restoration. Always append a copy (square[:]) not the list itself.

🗣️ What to Say to the Interviewer

"I need to find all valid word squares. Since I need ALL valid configurations, I'll use backtracking.
The key observation: at row k, the first k characters are determined by column k of the placed words.
So I can prune candidate words using a prefix query. I'll precompute a prefix map — or better, a Trie — so I can look up words by prefix in O(L) time.
The backtracking itself is standard: place a word, recurse to the next row, undo.
Time: O(n·L·26^L) theoretical, heavily pruned in practice. Space: O(n·L²) for prefix map, O(n·L) for Trie."
🔮 Likely Follow-up Questions
  • "Why store words at every Trie node?" → Avoids subtree DFS; O(1) return after O(L) path walk.
  • "What if words have different lengths?" → Group by length, apply algorithm per group.
  • "Can you count squares without enumerating them?" → Same backtracking, increment counter at base case.
  • "Hash map vs. Trie trade-off?" → Hash map: O(n·L²) space, simpler code. Trie: O(n·L) space, cleaner prefix sharing. Same time complexity.

7 Summary & Key Takeaways ▾
Word square symmetry (square[i][j]=square[j][i]) means each new row's prefix is fully determined by the partial column beneath the diagonal
Store the complete word list at every Trie node — this makes prefix queries O(L) walk + O(1) return, avoiding subtree DFS
Backtracking template: append → recurse → pop. Always append a copy (square[:]) at the result collection step
Outer loop must try every word as row 0 — the square can start with any word in the vocabulary
Aggressive pruning: if get_words(prefix) returns [], the for-loop has zero iterations — backtrack immediately, no wasted work
Prefix hash map is a simpler alternative to Trie (same time complexity, uses O(n·L²) space instead of O(n·L))
ApproachTimeSpaceNotes
Brute force (check all n-permutations) O(n! · L) O(n) Completely intractable
Backtrack + linear scan (no Trie) O(n^n · L) O(n) Better but too slow for n≥20
Prefix HashMap + Backtrack O(n·L·26^L) O(n·L²) Simple; interviewer-friendly
Trie + Backtrack ✓ O(n·L·26^L) O(n·L) Optimal; standard interview answer
🎯 When to Recognize Trie + Backtracking

  • "Which words start with X?" repeated many times during search → Trie pays off
  • Backtracking where candidate lists can be precomputed from a shared structure → store at Trie nodes
