LeetCode #287

Find the Duplicate Number

Given an array of n+1 integers where each value is in [1, n], find the duplicate number. Must use O(1) extra space and cannot modify the array.

Medium Math Floyd Cycle Detection Time: O(n) Space: O(1)
1 Understanding the Problem ▾

You have an array of n+1 integers where every value is in the range [1, n]. By the Pigeonhole Principle, at least one value must appear more than once — there are n+1 elements but only n distinct possible values. The problem guarantees exactly one value is duplicated (though it might appear more than twice).

💡 Key Insight

Think of each array value as a pointer: standing at index i, you "follow the pointer" to index nums[i]. Because values are in [1,n] and the array has indices 0..n, index 0 is a unique "entry node" that nothing points back to. The duplicate value means two different indices point to the same next index, which creates a cycle. Finding the cycle's entrance = finding the duplicate.

Why naive approaches fail

Sort: O(n log n) — too slow
Hash Set: O(n) space — forbidden
Modify array: forbidden by constraint
XOR trick: only works for "missing number"

Floyd's approach wins

Time: O(n) — linear scan only
Space: O(1) — just two pointers
Array unchanged — read-only access
Handles value appearing 3+ times

Example walkthrough (nums = [1,3,4,2,2]): Follow the pointer chain starting from index 0: index 0 → nums[0]=1 → index 1 → nums[1]=3 → index 3 → nums[3]=2 → index 2 → nums[2]=4 → index 4 → nums[4]=2 → index 2 → nums[2]=4 → ... The sequence visits index 2 twice. Index 2 is the "cycle entrance" — and 2 is the duplicate value.

⚠️ Why index 0 is special

Values are in [1,n], so no value ever equals 0. This means index 0 can never be visited by following a pointer (nothing points to it). It acts as the "head" of our implicit linked list — outside the cycle, guaranteed. Starting our traversal at index 0 puts us on the "tail" (ρ part) before entering the cycle.

2 The Algorithm: Floyd's Tortoise & Hare ▾

Floyd's Cycle Detection runs in two phases. Phase 1 finds a point inside the cycle. Phase 2 locates the exact cycle entrance — which equals the duplicate number.

1 Initialize: Set slow = nums[0] and fast = nums[nums[0]]. Both start by following pointer(s) from index 0. Slow takes one step, fast takes two.
2 Phase 1 — Find intersection: Repeat slow = nums[slow] (one step) and fast = nums[nums[fast]] (two steps) until slow == fast. They are guaranteed to meet inside the cycle (because fast laps slow in the cycle).
3 Phase 2 — Find entrance: Reset slow = 0. Keep fast at the Phase 1 meeting point. Now advance both by ONE step each iteration: slow = nums[slow], fast = nums[fast]. When slow == fast, that value is the cycle entrance = the duplicate number.
4 Return: Return slow (equals fast at this point) — it holds the duplicate number.
🔑 Why Phase 2 Works (the math)

Let ρ = tail length (steps from index 0 to the cycle entrance), λ = cycle length. After Phase 1, the meeting point is at offset φ inside the cycle, where φ ≡ ρ (mod λ). So the distance from the meeting point back to the entrance is ρ steps (going around). Resetting slow to index 0 and advancing both by 1 per step: slow takes ρ steps to reach the entrance; fast also takes exactly ρ steps from the meeting point to go around to the entrance. They arrive simultaneously.

3 Why It Works: The Invariant ▾
🔐 The Core Invariant

The array is a functional graph: every node (index) has exactly one outgoing edge (to nums[i]). The n+1 nodes map into indices 1..n, so by Pigeonhole, at least two nodes share the same successor. This shared successor is the duplicate, and it's also the cycle's entrance. Floyd's algorithm exploits the cycle structure to locate this node in O(n) time.

Phase 1 proof that slow and fast always meet: Once fast enters the cycle, it's chasing slow (which may not yet be in the cycle). Fast gains 1 step per iteration. The "distance" between them decreases by 1 each round. Eventually fast laps slow — they meet in at most λ additional iterations after both are in the cycle. Meeting is guaranteed.

Why meeting ≠ entrance: The meeting point in Phase 1 is generally NOT the cycle entrance — fast has gone around the cycle one or more times and "lap-intersected" slow somewhere inside. We can't directly read the duplicate from the meeting point index value. Phase 2 resets slow to 0 and uses the ρ-distance symmetry to find the entrance specifically.

⚠️ Do NOT try to use meeting point directly as the answer

After Phase 1, the meeting point's index (e.g., index 4 in our example) is NOT necessarily the duplicate. The ENTRANCE of the cycle (e.g., index 2) is the duplicate. That's why Phase 2 is essential. A common bug is returning fast after Phase 1 instead of completing Phase 2.

Edge case — duplicate appears many times: If the value 2 appears 3 or more times, multiple edges converge to index 2, making the cycle shorter. Floyd's algorithm is entirely agnostic to how many times the duplicate appears — it finds the cycle entrance regardless of cycle length.

4 The Code (Annotated) ▾

Solution 1 — Floyd's Cycle Detection Interview Pick · O(n) time · O(1) space

def findDuplicate(nums: list[int]) -> int:
# Phase 1: Find intersection point inside cycle
slow = nums[0] ← slow follows one pointer from the head (index 0)
fast = nums[nums[0]] ← fast follows two pointers (jumps two steps)
while slow != fast: ← keep moving until tortoise meets hare
slow = nums[slow] ← slow: one step at a time
fast = nums[nums[fast]] ← fast: two steps at a time
# Phase 2: Find cycle entrance = duplicate value
slow = 0 ← reset slow to the very beginning (index 0, not nums[0])
while slow != fast: ← advance both at speed 1 until they meet again
slow = nums[slow] ← slow: one step from index 0 toward the entrance
fast = nums[fast] ← fast: one step from meeting point toward the entrance
return slow ← slow == fast == cycle entrance == duplicate number
⚠️ Critical: Phase 2 resets slow to 0, NOT to nums[0]

In Phase 2, we set slow = 0 (the index itself), not slow = nums[0]. This is because 0 is our "virtual head" — the node that has no in-pointer. We need slow to travel exactly ρ steps to reach the entrance. The loop body then immediately does slow = nums[slow] = nums[0], so the first step correctly follows the head pointer. Getting this wrong (setting slow = nums[0]) shifts slow by one, breaking the symmetry.

Solution 2 — Binary Search on Value Range O(n log n) time · O(1) space · Good fallback

def findDuplicate(nums: list[int]) -> int:
lo, hi = 1, len(nums) - 1 ← binary search over the value space [1..n], not indices
while lo < hi:
mid = (lo + hi) // 2
count = sum(x <= mid for x in nums) ← count elements ≤ mid
if count > mid: ← more values ≤ mid than there are valid numbers 1..mid → duplicate is ≤ mid
hi = mid ← narrow search to left half
else:
lo = mid + 1 ← duplicate is in right half (count ≤ mid means no excess on left)
return lo

Binary search on the answer value, not array index. Pigeonhole tells us: if there are more than mid array values that are ≤ mid, the duplicate must be in [1, mid]. O(n log n) total, O(1) space. Propose this first if you draw a blank on Floyd's, then optimize.

💡 Complexity Summary

Floyd's Cycle Detection (optimal): Time O(n), Space O(1)
Binary Search on Value: Time O(n log n), Space O(1)
Hash Set: Time O(n), Space O(n) — fails the space constraint
Sort (in-place): Time O(n log n), Space O(1) — fails the no-modify constraint

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Call Out the Constraints Immediately

Say: "The key constraints here are O(1) space and no array modification — that rules out hash set and sort. I need an approach that reads the array without storing extra data." This shows you're reading the problem carefully, not just jumping to an answer.

Step 2: Propose Binary Search as a Valid First Solution

"I can binary search on the value range. Pick mid. Count how many values are ≤ mid. If count > mid, by Pigeonhole the duplicate is ≤ mid. This is O(n log n) and O(1) space — it satisfies the constraints. Want me to implement this, or should I look for O(n)?"

Step 3: Surface the Linked List Analogy

"There's a beautiful O(n) approach. Since values are in [1,n], every value is a valid next-index. We can treat the array as a linked list: node i points to nums[i]. The duplicate means two nodes share the same successor — that creates a cycle. This is Linked List Cycle II. Floyd's Tortoise and Hare finds the cycle entrance in O(n), O(1)."

Step 4: Explain Both Phases Clearly

Draw the ρ shape on a whiteboard. "Phase 1: slow moves 1 step, fast moves 2 steps — they meet somewhere inside the cycle. Phase 2: reset slow to index 0 (not nums[0]), advance both by 1. The distance from index 0 to the entrance equals the distance from the meeting point to the entrance — so they arrive together." Mention the math only if asked.

🗣️ What to Say to the Interviewer

"Constraints rule out hash set (O(n) space) and in-place sort (modifies array).
First, I could binary search the value range in O(n log n), O(1) space.
But we can do O(n): values in [1,n] means each value is a valid index pointer.
This creates a linked-list structure. The duplicate → two nodes share a successor → cycle.
Phase 1: slow=1 step, fast=2 steps — they meet inside the cycle.
Phase 2: reset slow to index 0, advance both by 1 — they meet at the cycle entrance.
Cycle entrance = the duplicate number. O(n) time, O(1) space."
🔮 Likely Follow-up Questions
  • "Prove Phase 2 works mathematically." → Tail length ρ: slow travels ρ from 0 to entrance; fast travels ρ from meeting point around cycle to entrance. They converge because meeting point is ρ steps from entrance.
  • "What if the duplicate appears 3 times?" → Algorithm still works. Multiple edges into the cycle node just make the cycle shorter, but Floyd's finds the entrance regardless.
  • "Can you find ALL missing AND duplicate numbers?" → Different problem. For each value v: if nums[nums[v]-1] == v and position wrong → duplicate/missing. Or use cyclic sort variant (#448, #442).
  • "Linked List Cycle II (#142)?" → Same Floyd's algorithm but on an actual linked list. This problem is the array analog.
