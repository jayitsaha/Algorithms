LeetCode #323

Number of Connected Components in an Undirected Graph

Given n nodes (0 to n-1) and a list of undirected edges, return the number of connected components.

Medium Graph Union-Find Time: O(n · α(n)) Space: O(n)
1 Understanding the Problem ▾

Imagine n islands numbered 0 to n−1. Each edge [u, v] is a bridge connecting island u and island v. A connected component is a group of islands where you can reach any island from any other by crossing bridges. The question: how many isolated groups are there?

💡 Key Insight

Start with n independent components (one per node). Each edge we process either links two different components — merging them into one (count − 1) — or links two nodes already in the same component (cycle, count unchanged). The answer is how many components survive after all edges are processed.

✓ Example 1: n=5, edges=[[0,1],[1,2],[3,4]]

Group A: 0—1—2
Group B: 3—4
→ Answer: 2

✓ Example 2: n=5, edges=[[0,1],[1,2],[2,3],[3,4]]

Group A: 0—1—2—3—4
All in one group
→ Answer: 1

Edge cases: No edges → answer = n (each node isolated). Self-loops → same component, skip. Duplicate edges → union is idempotent, safe. n = 0 → return 0. n = 1 → return 1 always.

2 The Algorithm: Union-Find (Disjoint Set Union) ▾

Union-Find (also called DSU — Disjoint Set Union) is the ideal data structure for connectivity queries. It maintains a forest of trees where each tree represents one connected component, and every tree has a unique root (the component's representative).

1 Initialize: Set parent[i] = i for all nodes — each node is its own root. Set rank[i] = 0. Set components = n.
2 Find (path compression): To find the root of node x, follow parent pointers up the chain. On the way back, set every intermediate node's parent directly to the root — this "flattens" the tree so future lookups cost O(1).
3 Union (by rank): For each edge (u, v), find both roots. If they differ — attach the shorter tree under the taller one (preserving O(log n) max height) and decrement components. If they're the same root, the nodes are already connected — skip.
4 Return components: After processing all edges, components is your answer.
🔬 Union-Find Deep-Dive

Origin: Bernard Galler and Michael Fischer (1964). The near-linear amortized bound via path compression + union by rank was proved by Tarjan (1975) and later refined to the inverse-Ackermann bound O(α(n)) by Hopcroft and Ullman.

Core invariant: Two nodes u, v are connected ↔ find(u) == find(v). The parent array encodes a forest; every tree corresponds to exactly one connected component; every tree has exactly one root (a node that is its own parent).

Why rank? Without it, n successive unions can create a chain 0→1→2→...→n (height n, find = O(n)). Attaching the shorter tree under the taller guarantees height ≤ log₂(n) before compression. Combined with compression, amortized cost per operation = O(α(n)) ≈ O(1) in practice.

Recognize when: Problem uses words like "group," "cluster," "connected," "reachable," "merge," or asks "are u and v in the same group?"

💡 Alternative: BFS/DFS

Build an adjacency list, then run DFS from each unvisited node, counting fresh DFS calls. Time: O(n + e). Space: O(n + e) — must store the graph. Union-Find only needs O(n) space and processes edges in one pass without building the graph. In interview settings, mention BFS/DFS first as the "obvious" approach, then upgrade to Union-Find for the O(n) space win.

3 Why It Works: The Invariant ▾
🔐 The Invariant

After processing k edges, components = n − (number of successful merges). A "successful merge" is a union call where find(u) ≠ find(v). We start with n; every merge reduces by 1. The final value is the number of components.

Case analysis for correctness:

  • find(u) ≠ find(v): u and v live in different trees. Merging them is exactly the right thing — we're connecting two separate components. Component count correctly decreases by 1.
  • find(u) == find(v): u and v are already reachable from each other. Adding edge (u,v) creates a cycle within the same component. We do nothing — component count stays the same. Correct.

Since both cases are handled correctly, and we process every edge exactly once, the final components value is always correct.

⚠️ Common Bug: Path Compression Order

In the recursive find, path compression happens on the return call: parent[x] = find(parent[x]). This sets every ancestor of x directly to the root in one pass. Some beginners write a loop that only updates one level — that works but is slower. The recursive version is clean and correct; watch out for Python's default recursion limit (sys.setrecursionlimit) on very large inputs.

Rank proof sketch: A tree with rank k has at least 2^k nodes (proved by induction). Therefore, any tree of n nodes has rank ≤ log₂(n). Path compression further reduces find to nearly O(1), giving the overall O(n · α(n)) bound.

4 The Code (Annotated) ▾

Solution 1 — Union-Find with Path Compression + Union by Rank (Interview Pick)

def countComponents(n, edges): ← n nodes, edges = list of [u, v] pairs
parent = list(range(n)) ← parent[i] = i: every node is its own root
rank = [0] * n ← rank[i] = height of tree rooted at i
components = n ← start: n isolated nodes = n components
def find(x): ← find root representative of x
if parent[x] != x: ← x is not its own root
parent[x] = find(parent[x]) ← path compression: point x directly to root
return parent[x] ← return the root
def union(u, v): ← try merging components of u and v
nonlocal components ← allow modification of outer counter
ru, rv = find(u), find(v) ← find roots (with path compression)
if ru == rv: return ← same component → cycle edge, skip
if rank[ru] < rank[rv]: ru, rv = rv, ru ← ensure ru is the "taller" tree
parent[rv] = ru ← attach shorter tree under taller
if rank[ru] == rank[rv]: rank[ru] += 1 ← only grow height when ranks were equal
components -= 1 ← two components merged into one
for u, v in edges: ← process every edge exactly once
union(u, v)
return components ← count of connected groups remaining
⚠️ nonlocal vs. list trick

Python requires nonlocal to assign to an outer variable inside a nested function. If you prefer, use a mutable container: comps = [n] and write comps[0] -= 1 — no nonlocal needed. Both are valid; nonlocal is more readable.

Solution 2 — DFS/BFS (O(n + e) time, O(n + e) space)

def countComponents(n, edges):

adj[u].append(v); adj[v].append(u) ← undirected: add both directions
visited = set(); components = 0
def dfs(node):
visited.add(node)
for nb in adj[node]:
if nb not in visited: dfs(nb)
for i in range(n):
if i not in visited: ← unvisited node = start of new component
dfs(i); components += 1 ← explore entire component, count it
return components

5 🎤 Interview Strategy ▾
Step 1: Recognize the Pattern

Keywords: "undirected graph," "connected components," "how many groups." These are the canonical Union-Find signals. Any time you see merging, grouping, or reachability without needing exact paths, DSU is the right tool.

Step 2: Propose BFS/DFS First

Say: "The straightforward approach is DFS — build an adjacency list and count how many times I need to start a fresh DFS. That's O(n + e) time and space." Then offer to optimize: "But Union-Find handles this in O(n) space and one pass, without building the adjacency list."

Step 3: Write the DSU Template from Memory

The template is always the same three pieces: (a) initialize parent/rank/count, (b) recursive find with path compression, (c) union by rank that decrements count on success. Practice this until you can write it in 90 seconds.

Step 4: Trace n=5, edges=[[0,1],[1,2],[3,4]]

components=5. Edge (0,1): roots 0,1 differ → merge, components=4. Edge (1,2): find(1) compresses to root 0; roots 0,2 differ → merge, components=3. Edge (3,4): roots 3,4 differ → merge, components=2. Answer: 2.

🗣️ What to Say to the Interviewer

"I notice this is a classic connected-components problem. My first instinct is DFS — O(n + e) time and space.

For a cleaner O(n) space solution, I'd use Union-Find. Start with components = n. For each edge, find both roots. If they differ, merge (attach smaller rank under larger) and decrement components. Path compression ensures future finds are nearly O(1).

Final time complexity: O(n · α(n)) — effectively linear. Space: O(n) for parent and rank arrays."
💡 Follow-Up Questions
  • Dynamic edge insertion? Union-Find handles it naturally — just call union() as each new edge arrives.
  • Edge deletion? Hard for DSU. Would need an offline approach or Link-Cut Trees.
  • Track component sizes? Add a size array, update on union. Useful for "largest component" queries.
  • Directed graph? Use Tarjan's SCC (Strongly Connected Components) algorithm instead.

7 Summary & Key Takeaways ▾
Union-Find solves connected components in O(n · α(n)) ≈ effectively O(n), using only O(n) space — no adjacency list required.
Start with components = n. Each successful union (different roots) decrements by 1. The counter is your answer.
Path compression (parent[x] = find(parent[x])) flattens the tree on every lookup — future finds cost nearly O(1).
Union by rank always attaches the smaller tree under the larger, keeping tree height at O(log n) before compression.
The DSU template transfers unchanged to dozens of problems: redundant connections, MST (Kruskal), island merging, account merging, and more.
🎯 When to Use This Pattern
  • Counting connected components in any graph
  • Detecting cycles in an undirected graph (find(u) == find(v) before union = cycle)
  • Kruskal's MST: greedily add cheapest edge connecting two different components
  • Any grouping/merging problem: islands, accounts, friend circles, pixel regions
  • Online queries: "is u connected to v?" answered in O(α(n)) after preprocessing
