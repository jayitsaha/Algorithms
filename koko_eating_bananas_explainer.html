LeetCode #875

Koko Eating Bananas

Find the minimum eating speed k so Koko can finish all banana piles within h hours.

Medium Binary Search BS on Answer (Speed) Time: O(n log m) Space: O(1)
1 Understanding the Problem ▾

Koko has n piles of bananas. The guards will return in h hours. Each hour, Koko picks one pile and eats at most k bananas from it. If a pile has fewer than k bananas, she finishes the whole pile in one hour. She never starts a second pile in the same hour. We want the minimum k that lets her finish everything in time.

💡 Key Insight: Monotone Feasibility Over Speed

If Koko can finish all piles at speed k, she can certainly finish at any speed > k — going faster only helps. This monotone step-function over k is the exact signature of Binary Search on the Answer. Instead of searching the piles, we binary search over possible speeds.

Speed k=3 on piles=[3,6,7,11], h=8

pile 3: ceil(3/3)=1 hr
pile 6: ceil(6/3)=2 hr
pile 7: ceil(7/3)=3 hr
pile11: ceil(11/3)=4 hr
Total = 10 hrs > 8 ✗ Too slow

Speed k=4 on piles=[3,6,7,11], h=8

pile 3: ceil(3/4)=1 hr
pile 6: ceil(6/4)=2 hr
pile 7: ceil(7/4)=2 hr
pile11: ceil(11/4)=3 hr
Total = 8 hrs = 8 ✓ Works!

The search space: k must be at least 1 (must make some progress) and at most max(piles) (at that speed, every pile finishes in exactly 1 hour — n hours total, which works since h ≥ n is guaranteed). So k lives in [1, max(piles)].

⚠️ Critical Detail: Koko Eats Only One Pile Per Hour

Even if k=100 and a pile has 3 bananas, Koko eats those 3 in one hour and does not continue to another pile that hour. The unused eating capacity (97 bananas worth) is wasted. This "one pile per hour" rule is why each pile costs exactly ceil(pile/k) hours.

Integer ceiling division: Instead of math.ceil(p/k) (floating point), use (p + k - 1) // k. For p=11, k=4: (11+3)//4 = 14//4 = 3. This is exact, fast, and avoids precision bugs for large values (piles can be up to 109).

2 The Algorithm: Binary Search on Speed ▾

Brute force tries every k from 1 to max(piles) — O(n · m) time. Binary search cuts this to O(n log m) by halving the candidate range each iteration. At each candidate speed, we pay O(n) for the feasibility check, but only do O(log m) checks total.

1 Set search bounds: lo = 1 (minimum conceivable speed), hi = max(piles) (guaranteed to work in n hours). Answer is in [lo, hi].
2 Pick mid: mid = (lo + hi) // 2. This is our candidate speed to test. Picking the midpoint halves the range each iteration.
3 Feasibility check: Compute total_hours = sum(ceil(p/mid) for p in piles). If total_hours ≤ h, speed mid is feasible (Koko finishes in time).
4 Narrow the range: If feasible → set hi = mid (keep mid as a candidate, try slower). If infeasible → set lo = mid + 1 (mid is too slow, must go faster).
5 Terminate: Loop exits when lo = hi. At that point, lo is the minimum feasible speed — return it.
🔑 The "Leftmost True" Binary Search Template

The feasibility predicate looks like: [F, F, F, ..., F, T, T, T, ..., T] over speeds [1..max]. We want the leftmost T. The exact template:

lo, hi = 1, max(piles)
while lo < hi:
  mid = (lo + hi) // 2
  if feasible(mid): hi = mid
  else: lo = mid + 1
return lo

3 Why It Works: The Invariant ▾
🔐 Loop Invariant

At every point in the binary search, the answer lies in [lo, hi]. When we set hi = mid (feasible), the answer is ≤ mid so it stays in [lo, mid]. When we set lo = mid + 1 (infeasible), the answer is > mid so it stays in [mid+1, hi]. The range strictly shrinks each iteration until lo = hi = answer.

Why hi = mid (not mid - 1) when feasible? Because mid itself might be the answer! If we wrote hi = mid - 1, we'd skip mid and miss it. The asymmetry — hi = mid vs lo = mid + 1 — ensures we always include the last feasible point.

Termination proof: Each iteration, either lo increases (by at least 1) or hi decreases (by at least 1), since mid is always < hi when lo < hi (integer floor). So hi - lo strictly decreases each iteration. Since it starts finite (max(piles) - 1), it reaches 0 in O(log m) steps.

⚠️ Why Not Return hi?

At loop exit, lo = hi, so lo and hi are the same value — either return works. But return lo is the conventional form for "leftmost True" binary search, and it makes the intent clear: lo has been pushed to the first feasible speed throughout the entire search.

💡 Binary Search on Answer — Pattern Recognition

Reach for this pattern whenever you see: "Find the minimum X such that [some condition involving X]" where:
(1) X has a clear lower and upper bound
(2) feasible(X) is monotone (once True, always True for larger X)
(3) feasible(X) can be computed efficiently (O(n) or better)

This transforms an O(max) brute force into O(log(max)) feasibility checks.

4 The Code (Annotated) ▾

Brute Force — Linear Scan (O(n · m), too slow for large inputs)

def minEatingSpeed(piles, h):
for k in range(1, max(piles) + 1): ← Try every speed from 1 to max
total = sum((p + k - 1) // k ← ceil(p/k) per pile, summed
for p in piles)
if total <= h: ← First feasible k is our answer
return k

Time: O(n · m) where m = max(piles). For m = 109, this is 109 iterations — TLE.

Binary Search on Speed — Optimal (Interview Pick)

def minEatingSpeed(piles, h):
lo, hi = 1, max(piles) ← Answer is in [1, max(piles)]
while lo < hi: ← Loop until range collapses
mid = (lo + hi) // 2 ← Candidate speed to test
hours = sum((p + mid - 1) // mid ← ceil(p/mid) for each pile
for p in piles) ← Total hours at speed mid
if hours <= h: ← Feasible — mid works, try slower
hi = mid ← Keep mid (might be answer); shrink right
else: ← Infeasible — mid is too slow
lo = mid + 1 ← mid ruled out; try speeds above mid
return lo ← lo = hi = minimum feasible speed
⚠️ hi = mid vs. hi = mid - 1

When feasible, we write hi = mid, NOT hi = mid - 1. Why? Because mid itself could be the answer. Setting hi = mid - 1 would skip it. The paired loop condition while lo < hi (not ≤) ensures we don't infinite-loop, since when lo = hi the loop exits and we return lo safely.

💡 (p + k - 1) // k is Exact Ceiling Division

For pile p, speed k: math.ceil(p/k) uses floating point and can give wrong answers for large integers. The formula (p + k - 1) // k is purely integer arithmetic. It works by asking: "how many groups of k fit in p, rounding up?" Add k-1 before dividing to round up rather than down.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Spot the "Minimum X satisfying a condition" Frame

The moment you see "find the minimum [number/speed/capacity/days]" — pause and ask: is the feasibility of this number monotone? If yes (once feasible, always feasible for larger values), it's a classic binary search on the answer. This problem is asking for minimum k, k is in a bounded range — perfect match.

Step 2: Confirm Monotonicity Out Loud

"If Koko can finish at speed 5, she can definitely finish at speed 6 — eating faster only helps, never hurts. So feasible(k) goes from False to True exactly once. This monotone predicate is the license to binary search."

Step 3: Define the Feasibility Check

"Given candidate k, how many hours does she need? For each pile p, she needs ceil(p/k) hours — that's the integer ceiling. Sum over all piles. If sum ≤ h, k is feasible. This check is O(n)."

Step 4: Apply the Leftmost-True Template

lo=1, hi=max(piles), while lo<hi: mid=(lo+hi)//2, if feasible → hi=mid, else → lo=mid+1. Return lo. Mention you use (p+k-1)//k instead of float ceil for safety.

🗣️ What to Say to the Interviewer

"I notice we're minimizing k where feasibility is monotone over k.
The answer lies in [1, max(piles)] — at k=max, she finishes in n hours.
I'll binary search over k, checking feasibility in O(n) each iteration.
Feasibility: sum of ceil(pile/k) for all piles must be at most h.
I use (p+k-1)//k to avoid floating-point ceiling issues.
Total: O(n log m) time, O(1) space — where m = max(piles)."
🔮 Follow-up Questions to Expect
  • "What if Koko can eat from multiple piles in one hour?" → total hours = ceil(sum(piles)/k), much simpler
  • "What's the minimum number of days to ship packages?" → Capacity to Ship (#1011), identical template
  • "Why not just binary search on the index in sorted piles?" → The answer is a speed value, not a pile index — BS is on the value space
  • "Can you do better than O(n log m)?" → Unlikely without structural assumptions; O(n log m) is the standard for this class

7 Summary & Key Takeaways ▾
Binary search on the answer (speed k), not on the input array (piles)
Feasibility is monotone: feasible(k) = [F...F T...T] — perfect for binary search
Hours for pile p at speed k = ceil(p/k) = (p+k-1)//k (exact integer arithmetic)
Template: lo=1, hi=max; while lo<hi: hi=mid if feasible else lo=mid+1; return lo
Time O(n log m), Space O(1) — optimal for this problem class

🎯 When to Use Binary Search on Answer
  • "Find minimum/maximum [number] such that [condition holds]"
  • The condition is monotone over the candidate value
  • The candidate space has a clear min and max
  • A feasibility check function exists that is O(n) or better
  • The answer space is a value (speed, capacity, days) not an index
