LeetCode #1937

Maximum Number of Points with Cost

Pick one cell per row from a grid to maximize total points, paying the absolute column distance as a penalty for each row transition.

Medium Dynamic Programming Optimize with Prefix Max Time: O(m·n) Space: O(n)
1 Understanding the Problem ▾

You have an m × n grid where points[r][c] is the score of picking cell (r, c). You must pick exactly one cell per row, traveling row by row from top to bottom. Your total score is the sum of all picked cell values, minus a transition penalty equal to the absolute column difference between consecutive picks.

Formally: if you pick column c_r in row r, your total score is:

score = Σ points[r][c_r] − Σ |c_r − c_{r+1}|
💡 Key Insight

The penalty couples every consecutive pair of row decisions — you cannot optimize each row in isolation. This coupling makes it a DP problem. The further trick is that the naive O(n²) transition per row can be reduced to O(n) using a left-to-right and right-to-left prefix-max sweep that "bakes" the penalty in incrementally.

✓ Optimal — Total = 9

Row 0: col 2 → +3
Row 1: col 1 → +5, |1−2|=1 → net +4
Row 2: col 0 → +3, |0−1|=1 → net +2
Total: 3 + 4 + 2 = 9

✗ Greedy Per-Row Max Fails

Row 0: col 2 max → 3
Row 1: col 1 max → 5 − 1 = 4
Row 2: col 0 max → 3 − 1 = 2
Lucky here but penalty can destroy greedy!

Edge Cases: single row (no penalty, max of row), single column (always col 0, no penalty ever), m=1 row (answer = max of that row), uniform grid (penalty determines all), very large n requiring O(n) per row.

2 The Algorithm: DP + Prefix-Max Optimization ▾

We maintain a 1D array dp[j] meaning: the best total score achievable by picking any path that ends at column j of the current row. For each new row, we update dp using an O(n) sweep instead of an O(n²) nested loop.

1 Base case: dp[j] = points[0][j] for all j. First row has no predecessor, no penalty.
2 Left sweep (left → right): Build left[j] where left[j] = max(left[j−1] − 1, dp[j]). This represents: the best dp value reachable at column j if we're allowed to approach only from the left. The −1 per step encodes the one-unit penalty for each column traveled rightward.
3 Right sweep (right → left): Build right[j] where right[j] = max(right[j+1] − 1, dp[j]). Same idea but approaching from the right — each step leftward costs 1 penalty unit.
4 Update dp: dp[j] = points[row][j] + max(left[j], right[j]). The penalty is already embedded in left/right — no explicit abs() needed.
5 Answer: return max(dp) after processing all rows. The best column in the final row holds the global optimum.
📐 Recurrence Relations
dp[j] = points[0][j]                                 // base case
left[j] = max(left[j−1] − 1,  dp[j])   // from left
right[j] = max(right[j+1] − 1,  dp[j]) // from right
dp_new[j] = points[r][j] + max(left[j], right[j])
🔑 Why −1 Per Step Encodes the Penalty

Moving from column k to column j costs |j−k|. For a leftward approach (k ≥ j), dp[k] − (k−j) = dp[k] − k + j. Taking max over k ≥ j of (dp[k] + k) and subtracting j is what the right sweep computes incrementally (each step adds 1 to the "−k" part). The left sweep is symmetric. This is the classic "max of linear functions" optimization.

3 Why It Works: Invariant & Why This is DP ▾
🔐 The DP Invariant

After processing row r, dp[j] holds the maximum total score (cell values minus all transition penalties) for any path from row 0 through any cells that ends at column j of row r. This invariant is maintained by construction at each row update.

Why Dynamic Programming? Two pillars:

1. Optimal Substructure. The optimal path to (r, j) always consists of an optimal path to some cell (r−1, k) plus the single-step transition cost |j−k|. If the path to (r−1, k) were suboptimal, we could substitute a better path and improve the overall result. So optimal subproblems compose into optimal solutions.

2. Overlapping Subproblems. When computing the best predecessor for column j=2 and j=4, both consider the dp value at k=1 in the previous row. Without memoization, a recursive approach would recompute dp[r−1][1] for every different j in row r. DP computes each (row, col) pair once and reuses it.

Why greedy fails: Picking the per-row maximum ignores that a large cell in column 0 imposes heavy penalties on future rows if they must reach far columns. Only DP can balance immediate gain vs. long-term penalty cost.

⚠️ The O(n²) Naive DP — Correct But Too Slow

The textbook transition: dp_new[j] = points[r][j] + max(dp[k] − |j−k| for k in range(n)) is correct and O(n) per cell, giving O(m·n²) total. For n up to 10⁵ and m up to 10⁵, this is 10¹⁰ operations — way too slow. The prefix-max trick reduces each row to O(n), giving O(m·n) total.

Proof that left[j] = max over k ≤ j of (dp[k] − (j−k)): By induction. Base: left[0] = dp[0] = dp[0] − (0−0). Step: Assume left[j−1] = max_{k≤j−1}(dp[k]−(j−1−k)). Then left[j] = max(left[j−1]−1, dp[j]) = max(max_{k≤j−1}(dp[k]−(j−k)), dp[j]−0) = max_{k≤j}(dp[k]−(j−k)). QED.

4 The Code (Annotated) ▾

Solution 1 — DP + Prefix-Max Sweeps (Interview Pick, Optimal)

def maxPoints(points: list[list[int]]) -> int:
m, n = len(points), len(points[0])
dp = points[0][:] ← Base case: copy first row (no penalty)
for r in range(1, m): ← Process each row after the first
left = [0] * n
left[0] = dp[0] ← Leftmost: no left predecessor
for j in range(1, n): ← Left sweep: propagate with penalty
left[j] = max(left[j-1] - 1, dp[j]) ← Extend left reach OR start fresh at j
right = [0] * n
right[n-1] = dp[n-1] ← Rightmost: no right predecessor
for j in range(n-2, -1, -1): ← Right sweep: propagate with penalty
right[j] = max(right[j+1] - 1, dp[j]) ← Extend right reach OR start fresh at j
dp = [points[r][j] + max(left[j], right[j])
for j in range(n)] ← New dp: cell value + best predecessor (penalty embedded)
return max(dp) ← Best endpoint in final row = global answer
⚠️ Why Does max(left[j-1] - 1, dp[j]) Work?

left[j−1] is the best value reachable at column j−1 from any column to its left. Moving one step right costs 1 unit of penalty, so the contribution to column j from the left is left[j−1] − 1. Alternatively, we can just "start fresh" at column j with no lateral travel, giving dp[j]. We take the max of both. Over many iterations, this propagates the best-left-predecessor to every column in O(n).

Solution 2 — Top-Down Memoization (shows the recurrence clearly)

from functools import lru_cache
def maxPoints(points):
m, n = len(points), len(points[0])
@lru_cache(maxsize=None)
def dp(r, j): ← Best total ending at row r, col j
if r == 0: return points[0][j] ← Base case
best = max(dp(r-1, k) - abs(j - k) ← Try every predecessor column
for k in range(n)) ← O(n) per call = O(m*n^2) total
return points[r][j] + best
return max(dp(m-1, j) for j in range(n))
🚨 Memoization TLEs — Use Only to Understand the Recurrence

The memoized solution is O(m·n²): each of the m·n states calls O(n) predecessors. For n=10⁵, this is 10¹⁰ operations — LeetCode will TLE. Study the memoization to understand what dp(r, j) represents, then implement Solution 1 (tabulation + prefix-max) for the actual submission.

💡 Complexity Summary

Solution 1 (Tabulation + Prefix Max): Time O(m·n), Space O(n)
Solution 2 (Memoization): Time O(m·n²), Space O(m·n) — TLE for large inputs
Brute Force: Time O(n^m) — exponential, infeasible

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the DP Signal

"Choosing one element per row with inter-row cost is a classic DP setup. The penalty |c_r − c_{r+1}| means row r's decision depends on row r−1's — I can't optimize each row independently. Optimal substructure holds: best path through (r, j) = best path through some (r−1, k) plus transition."

Step 2: Write the Naive Recurrence

"dp[j] = points[r][j] + max over k of (dp_prev[k] − |j−k|). This is O(n) per cell, O(n²) per row, O(m·n²) total. Too slow for n=10⁵."

Step 3: Split the Absolute Value

"Notice |j−k| = (j−k) when k ≤ j (left approach) and (k−j) when k ≥ j (right approach). For left approach: max_{k≤j}(dp[k] − (j−k)) = max_{k≤j}(dp[k] + k) − j. I can compute this running max in one left-to-right pass, decrementing by 1 each step."

Step 4: Implement Prefix-Max Sweeps

"Two O(n) sweeps: left[j] = max(left[j−1]−1, dp[j]), then right[j] = max(right[j+1]−1, dp[j]). New dp[j] = points[r][j] + max(left[j], right[j]). Total O(m·n), O(n) space."

🗣️ What to Say to the Interviewer

"Row decisions couple via penalty — that's the DP signal. State: dp[j] = best total ending at col j of this row.
Naive transition is O(n²). But |j-k| splits into left/right cases, each solvable in one linear pass.
Left sweep: left[j] = max(left[j-1]-1, dp[j]). Right sweep: right[j] = max(right[j+1]-1, dp[j]).
New dp[j] = points[r][j] + max(left[j], right[j]). O(m*n) time, O(n) space."
🔮 Likely Follow-up Questions
  • "Reconstruct the optimal path?" → Store argmax column chosen per row in a choices[] array, then backtrace.
  • "What if penalty is min(|j−k|, C) capped at C?" → Run left/right sweeps but stop decrementing once you hit −C.
  • "What if penalty is |j−k|²?" → Non-linear distance; prefix-max no longer works. Use divide-and-conquer DP or SMAWK.
  • "Can we pick from all rows in both directions?" → Same DP structure, just process bidirectionally.

7 Summary & Key Takeaways ▾
Inter-row penalty |c_r − c_{r+1}| couples consecutive row decisions — DP is required
Naive DP transition is O(n) per cell → O(m·n²) total — TLE for large inputs
Split |j−k| into left-approaching and right-approaching cases, each solvable in O(n)
Each prefix-max sweep decrements by 1 per step — this IS the penalty being baked in
Only O(n) space needed: we can overwrite dp[] each row since we only need the previous
Pattern generalizes to any DP with max(prev_state − linear_distance) transition

Brute Force O(n^m) O(m) Exponential — infeasible

🎯 When to Recognize "Optimize with Prefix Max" in DP
  • DP transition: dp_new[j] = f(j) + max_k(dp_old[k] − |j−k|)
  • Naive version is O(n) per cell (check all previous states)
  • Penalty is linear in distance (not quadratic or other)
  • The phrase "absolute column difference" in a DP context is a strong hint
