LeetCode #63

Unique Paths II

Count distinct paths from top-left to bottom-right of a grid containing obstacles — robot can only move right or down.

Medium Dynamic Programming Grid DP with Obstacles Time: O(m·n) Space: O(n)
1 Understanding the Problem ▾

You have a robot starting at the top-left corner of an m × n grid. It wants to reach the bottom-right corner. The robot can only move right or down — never left or up. Some cells are marked as obstacles (1 in the input); the robot cannot enter those cells. Your task: count how many distinct valid paths exist.

💡 Key Insight

Because you can only arrive at a cell from above or from the left, the number of paths to any cell is simply the sum of paths to the cell above it and paths to the cell to its left. An obstacle instantly sets that cell's count to 0, and that zero propagates forward — blocking any path that would have run through it.

2 The Algorithm: Bottom-Up Grid DP (Tabulation) ▾

We build a DP table of the same dimensions as the grid. Each entry dp[r][c] answers: "How many distinct paths from (0,0) to (r,c) exist without hitting an obstacle?" We fill the table systematically so that every cell's dependencies are already computed before we need them.

📐 Recurrence Relations
dp[r][c] = 0                                      if obstacleGrid[r][c] == 1 dp[r][c] = dp[r-1][c] + dp[r][c-1]                  otherwise dp[0][0] = 1                                      (base case — no obstacle) out-of-bounds cells = 0                          (boundary condition)
1 Guard clause: If the start cell is an obstacle, there are 0 paths — return immediately. (The robot can't even start.)
2 Initialize dp[0][0] = 1. There is exactly one way to be at the start: just be there. All other cells start at 0 (we'll fill them in).
3 Fill first row (r=0): Each cell can only be reached from the left. Copy the left neighbor's value — unless the current cell is an obstacle, in which case set 0 (and all cells to the right also become 0 by cascade).
4 Fill first column (c=0): Same logic vertically — each cell can only be reached from above. One obstacle in the left column makes everything below it 0.
5 Fill interior cells (row by row, left to right): For each cell: if obstacle → 0; else dp[r][c] = dp[r-1][c] + dp[r][c-1]. Both dependencies are already computed (above: previous row; left: previous column in this row).
6 Return dp[m-1][n-1] — the bottom-right cell's value is the final answer.
🔑 Space Optimization to O(n)

Since dp[r][c] only depends on dp[r-1][c] (cell above) and dp[r][c-1] (cell to the left), we can maintain a single 1D array and overwrite it row by row. dp[c] before overwriting = "from above"; dp[c-1] already overwritten = "from left." This cuts space from O(m·n) to O(n).

3 Why Is This Dynamic Programming? The Invariant ▾
🔐 The Invariant

When we compute dp[r][c], every cell it depends on — dp[r-1][c] and dp[r][c-1] — has already been finalized. The row-by-row, left-to-right fill order enforces this. The DP table is always in a consistent state: every filled cell holds the exact, correct count of paths to that position.

Optimal Substructure: "The number of paths to (r,c)" depends perfectly on "the number of paths to (r-1,c)" and "the number of paths to (r,c-1)." These subproblems are fully independent of each other — the paths that reach (r-1,c) have nothing to do with those that reach (r,c-1). Summing them gives the exact correct count without any double-counting, because every path to (r,c) passes through exactly one of those two cells as its last step.

Overlapping Subproblems: Consider a naive recursion — count(r, c) = count(r-1, c) + count(r, c-1). The call count(2, 2) spawns count(1, 2) and count(2, 1), each of which spawns count(0, 2), count(1, 1), etc. Without memoization, a 5×5 grid already involves thousands of redundant calls. DP computes each cell exactly once.

⚠️ Why Not Greedy?

A greedy approach (e.g., "always move right when possible") finds one path but misses others entirely. Counting requires exploring all valid combinations. DP is necessary precisely because we need to aggregate — not optimize — over all possible choices.

Obstacle propagation: The elegance of setting dp[r][c] = 0 for obstacles is that it propagates correctly through the formula. Any downstream cell that could only be reached through an obstacle cell will sum 0 + something or 0 + 0 — never going negative or incorrect. No special case logic is required beyond that single assignment.

4 The Code (Annotated) ▾

Solution 1 — Bottom-Up DP / Tabulation (Interview Pick, O(m·n) space)

def uniquePathsWithObstacles(obstacleGrid): ← Input: m×n grid, 0=free, 1=obstacle
m, n = len(obstacleGrid), len(obstacleGrid[0]) ← Grid dimensions
if obstacleGrid[0][0] == 1: ← Can't start → immediate return 0
return 0
dp = [[0]*n for _ in range(m)] ← DP table: same shape, all zeros
dp[0][0] = 1 ← Base case: 1 way to be at the start
for c in range(1, n): ← Fill top row: reachable only from left
dp[0][c] = 0 if obstacleGrid[0][c] else dp[0][c-1] ← Obstacle → 0, else inherit left
for r in range(1, m): ← Fill left column: reachable only from above
dp[r][0] = 0 if obstacleGrid[r][0] else dp[r-1][0] ← Obstacle → 0, else inherit above
for r in range(1, m): ← Fill interior row by row
for c in range(1, n): ← Left to right within each row
if obstacleGrid[r][c] == 1: ← Obstacle: forcibly 0, no paths through here
dp[r][c] = 0
else: ← Sum paths from above and from left
dp[r][c] = dp[r-1][c] + dp[r][c-1]
return dp[m-1][n-1] ← Answer: paths to bottom-right corner
⚠️ Common Mistake: Forgetting the First-Row/Column Cascade

If there is an obstacle at (0, 2) in the top row, not only is dp[0][2] = 0 but all cells to its right must also be 0 — they can only be reached from the left and the left is now 0. The formula dp[0][c] = 0 if obstacle else dp[0][c-1] handles this automatically — after the obstacle, dp[0][c-1] = 0, so every subsequent cell inherits 0 through the chain.

Solution 2 — Top-Down DP / Memoization (intuitive)

from functools import lru_cache ← Automatic caching decorator
def uniquePathsWithObstacles(obstacleGrid):
m, n = len(obstacleGrid), len(obstacleGrid[0])
@lru_cache(maxsize=None) ← Cache result for each (r, c) call
def dp(r, c): ← Returns # paths from (0,0) to (r,c)
if r < 0 or c < 0: return 0 ← Out of bounds: no paths
if obstacleGrid[r][c] == 1: return 0 ← Obstacle: no paths through here
if r == 0 and c == 0: return 1 ← Base case: at the start
return dp(r-1, c) + dp(r, c-1) ← Recurse: paths from above + from left
return dp(m-1, n-1) ← Ask: how many paths to bottom-right?

Solution 3 — Space-Optimized Tabulation (O(n) space)

def uniquePathsWithObstacles(obstacleGrid):
n = len(obstacleGrid[0])
dp = [0] * n ← 1D rolling array — represents current row
dp[0] = 1 if not obstacleGrid[0][0] else 0 ← Start cell initialization
for row in obstacleGrid: ← Process each row in order
for c in range(n):
if row[c] == 1: ← Obstacle wipes this slot to 0
dp[c] = 0
elif c > 0: ← Add left neighbor (already updated this row)
dp[c] += dp[c-1] ← dp[c] still = above value; dp[c-1] = left
return dp[-1] ← Last slot = bottom-right answer
💡 How dp[c] += dp[c-1] does both directions at once

When we process row r at column c: dp[c] before overwriting holds the value from row r-1 (paths from above). dp[c-1] was just updated in this same row-pass (paths from left). So dp[c] += dp[c-1] adds them together in-place — equivalent to the 2D recurrence, using only O(n) memory.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify the Pattern

Signals: grid navigation, restricted moves (right/down only), "count distinct paths." The restriction to right/down means each cell has at most 2 predecessors — a classic setup for 2D grid DP. Obstacles modify the counting but don't change the fundamental structure.

Step 2: Define the State

dp[r][c] = number of distinct paths from (0,0) to (r,c) that avoid all obstacles. Ask yourself: "What information do I need to make a decision at each cell?" Just the position — no additional state.

Step 3: Write the Recurrence

Every path to (r,c) must pass through (r-1,c) or (r,c-1) as its last step. So dp[r][c] = dp[r-1][c] + dp[r][c-1]. Obstacle? Override with 0. This is the whole algorithm.

Step 4: Handle Base Cases

Start cell = 1 (if no obstacle). First row: each cell = left neighbor (can't come from above). First column: each cell = cell above (can't come from left). After any obstacle in row/column, everything beyond is 0 — the formula propagates this automatically.

Step 5: Mention Space Optimization

Impress with: "Since we only look at the previous row and the current row's left neighbor, we can use a single 1D array updated in-place — reducing space from O(m·n) to O(n)." Derive it live: dp[c] += dp[c-1] before overwriting handles both directions.

🗣️ What to Say to the Interviewer

"This is a grid DP problem. I can only move right or down, so each cell has exactly
two possible predecessors: the cell above it and the cell to its left.
I'll define dp[r][c] = number of paths from start to (r,c) avoiding obstacles.
Base case: dp[0][0] = 1. Obstacle cells get 0.
Recurrence: dp[r][c] = dp[r-1][c] + dp[r][c-1] if no obstacle, else 0.
Fill row by row. Answer is dp[m-1][n-1]. Time O(m·n), space reducible to O(n)."
🔮 Likely Follow-up Questions
  • "What if moves include up and left too?" → Becomes a graph problem (BFS/DFS); no longer DP.
  • "Minimize total cost instead of count paths?" → Minimum Path Sum (#64): change + to min.
  • "Find the actual path, not just count?" → Backtrack from dp[m-1][n-1]: always step to the neighbor with the larger dp value.
  • "Can obstacles appear dynamically?" → BFS with mutable visited set; DP requires static grid.

7 Summary & Key Takeaways ▾
dp[r][c] = sum of paths from above + from left; obstacles force that cell and all dependents to 0
Fill order: top-left corner → first row → first column → interior (row by row, left to right)
Guard clause: if start or end is an obstacle, return 0 before any DP work
Space compresses from O(m·n) to O(n) with a single rolling 1D array
The recurrence is identical to Unique Paths I — obstacles just set certain cells to 0
