LeetCode #560

Subarray Sum Equals K

Given an integer array nums and an integer k, return the total number of contiguous subarrays whose sum equals k.

Medium Prefix Sum Hash Map Time: O(n) Space: O(n)
1 Understanding the Problem ▾

A subarray is a contiguous slice of an array — elements at consecutive positions. The problem asks: how many such slices have elements that add up to exactly k? The twist is that nums can contain negative numbers, zeros, and positives all mixed together.

💡 Key Insight

Any subarray sum from index i to j equals prefix_sum[j] − prefix_sum[i−1]. So instead of checking every (i, j) pair — which is O(n²) — we can ask: "how many earlier prefix sums have the value prefix_sum[j] − k?" That single lookup per element reduces the entire algorithm to O(n) using a hash map.

Example: nums=[1,1,1], k=2

Prefix sums: [1, 2, 3]
[1,1] at indices 0..1 → sum=2 ✓
[1,1] at indices 1..2 → sum=2 ✓
Answer: 2

Edge cases to consider: Array of all zeros with k=0 (every non-empty subarray counts, result is n*(n+1)/2), a single element equal to k, negative numbers that cancel each other out, and k=0 with mixed positive/negative values. The algorithm handles all of these naturally.

⚠️ Common Misread: "Subarray" vs "Subsequence"

A subarray must be contiguous — elements at adjacent indices. A subsequence allows gaps. This problem requires contiguous elements only. If elements could be non-contiguous, we'd need DP.

2 The Algorithm: Prefix Sum + Hash Map ▾

We maintain a running prefix sum and a frequency map of all prefix sums seen so far. At each position, we look up whether prefix_sum − k has appeared before. Each occurrence of that value corresponds to a subarray ending here that sums to k.

📐 The Prefix Sum Identity
sum(nums[i..j]) = prefix[j] − prefix[i−1] = k
⟹ prefix[i−1] = prefix[j] − k
So count how many earlier j had prefix[j] = current_prefix − k
1 Initialize: Set count = 0 and prefix_sum = 0. Seed the hash map freq = {0: 1}. The value 0 with count 1 is a "phantom" prefix sum representing the state before we read any element — critical for detecting subarrays that start at index 0.
2 Extend prefix sum: For each element, do prefix_sum += num. This gives the cumulative sum from the start up to the current position.
3 Look up complement — BEFORE inserting: Check if prefix_sum − k is in freq. Add freq.get(prefix_sum − k, 0) to count. Each entry in freq for that key represents a distinct earlier starting point for a valid subarray ending here.
4 Record current prefix sum: Increment freq[prefix_sum] by 1. Future elements can now look this up to discover subarrays that start right after the current position.
5 Return count: After one complete pass, count holds every valid subarray found.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every iteration when processing index j, freq contains the frequency of every prefix sum seen from index -1 (the phantom zero) through index j−1. The lookup freq[prefix_sum − k] therefore gives the exact count of starting positions that produce a subarray ending at j with sum equal to k.

Mathematical justification: For a subarray [i+1 .. j] to sum to k, we need prefix[j] − prefix[i] = k, equivalently prefix[i] = prefix[j] − k. Since we process left-to-right and query the map before inserting the current prefix sum, the map only ever contains prefix sums for indices 0 through j−1. So every hit in the map corresponds to a unique, valid, strictly-earlier starting boundary.

Why the sentinel {0: 1} is essential: Without it, we'd miss subarrays starting at index 0. The "phantom" prefix sum of 0 represents the state before reading any element. When prefix_sum equals k at index j, we need prefix[−1] = 0 to be in the map. The sentinel provides exactly that.

⚠️ Order Matters: Look Up BEFORE Inserting

If you insert prefix_sum into freq first, then query for prefix_sum − k, you might match the current position against itself. For example, if k = 0 and you insert prefix_sum=5 then look up 5−0=5, you'd count a zero-length subarray, which is invalid. Always query first, then insert.

Duplicate prefix sums are handled correctly: If prefix_sum = 7 appears 3 times in the map (at 3 different earlier indices), then looking up freq[7] returns 3. This correctly counts 3 distinct subarrays — one ending at the current position, each starting right after one of those 3 earlier positions. The multiplication is implicit in the frequency count.

4 The Code (Annotated) ▾

Solution 1 — Prefix Sum + Hash Map (Optimal O(n)) — Interview Pick

def subarraySum(nums, k):
count = 0 ← Final answer: total valid subarrays found
prefix_sum = 0 ← Running cumulative sum, reset to 0 before any elements
freq = {0: 1} ← Sentinel: prefix sum "0" seen once (before index 0)
for num in nums: ← Single left-to-right pass
prefix_sum += num ← Extend prefix sum to include current element
count += freq.get(prefix_sum - k, 0) ← How many earlier positions had (prefix - k)? Each is a valid subarray ending here
freq[prefix_sum] = freq.get(prefix_sum, 0) + 1 ← Record this prefix for future lookups
return count ← Total subarrays whose sum equals k
⚠️ Why .get(key, 0) and not direct dict access?

If prefix_sum − k has never appeared as a prefix sum, accessing freq[prefix_sum − k] raises a KeyError. Using freq.get(key, 0) safely defaults to 0 — meaning no valid subarray ends here from that particular prefix. This also applies when inserting: freq.get(prefix_sum, 0) + 1 handles first-time entries without an explicit if-check.

Solution 2 — Brute Force O(n²) (baseline for comparison)

def subarraySum_brute(nums, k):
count, n = 0, len(nums)
for i in range(n): ← Every possible starting index
total = 0
for j in range(i, n): ← Extend subarray from i to j
total += nums[j] ← Incrementally build subarray sum
if total == k: count += 1 ← Count valid subarray
return count
💡 Complexity Summary

Prefix Sum + Hash Map: Time O(n), Space O(n) — one pass, one hash map with at most n+1 entries.
Brute Force (nested loops): Time O(n²), Space O(1) — times out for n=20,000.
Sliding Window: Time O(n), Space O(1) — but ONLY works when all numbers are non-negative. Fails here!

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Clarify — Are there negative numbers?

This is the pivot question. If only non-negative: sliding window works in O(n). If mixed or negative: you need prefix sums. Always ask! It signals you know both approaches and understand when each applies.

Step 2: Propose brute force first

"Naively, I check every (i,j) pair and compute the subarray sum — O(n²) time, O(1) space. Correct but too slow for large n. Let me think of something better."

Step 3: Introduce prefix sums

"Any subarray sum [i..j] = prefix[j] − prefix[i−1]. So I'm hunting for pairs where prefix[j] − prefix[i−1] = k, or equivalently prefix[i−1] = prefix[j] − k. This looks exactly like Two Sum — instead of finding two elements summing to target, I'm finding two prefix sums with a specific difference."

Step 4: Map out the one-pass solution

"I'll scan left to right, maintaining prefix_sum. At each position, I look up freq[prefix_sum − k] to count valid subarrays ending here. Then I store prefix_sum in freq. Seed freq with {0:1} to handle subarrays starting at index 0."

🗣️ What to Say to the Interviewer

"Can nums contain negative numbers? [Yes] Good, that rules out sliding window.
Let me think in terms of prefix sums. Any subarray [i..j] sum = prefix[j] - prefix[i-1].
I need this to equal k, so I need prefix[i-1] = prefix[j] - k.
This is Two Sum! Use a hash map: store how often each prefix sum has appeared.
One pass: extend prefix_sum, look up freq[prefix_sum - k], add to count, then insert.
Seed freq={0:1} for subarrays starting from index 0.
Time O(n), Space O(n). Want me to code it up?"
🔮 Likely Follow-up Questions
  • "What if you need the actual subarrays, not the count?" → Store start indices instead of counts. O(n) time but output can be O(n²).
  • "What about the longest subarray summing to k?" → Store the first index where each prefix sum appeared; update max_len = max(max_len, j − freq[prefix_sum − k]). (#325)
  • "Count subarrays with sum divisible by k?" → Use prefix sum modulo k. (#974)
  • "2D version — submatrix sum equals k?" → Fix two row boundaries, reduce each column to a 1D prefix sum. (#363)

7 Summary & Key Takeaways ▾
Any subarray sum = prefix[j] − prefix[i], enabling O(n) search via hash maps
Seed freq = {0: 1} — the phantom prefix-zero — to catch subarrays starting at index 0
Always query freq[prefix_sum − k] BEFORE inserting prefix_sum — order prevents self-matching
Sliding window fails for negative numbers — prefix + hash map is the general solution
This is Two Sum applied to prefix sums — the same "store seen values, look up complement" pattern
Handles k=0, all-zeros arrays, and negative elements equally well without special cases

🎯 When to Recognize the Prefix Sum + Hash Map Pattern
  • "Count subarrays with sum/XOR/product equal to target" + negatives allowed → prefix + hash map
  • "Subarray sum divisible by k?" → Use prefix mod k in the map
  • Any problem that smells like "find pairs with a specific difference" → it's Two Sum in disguise
  • Whenever sliding window can't shrink reliably (due to negatives), switch to prefix sums
