LeetCode #93

Restore IP Addresses

Given a string s containing only digits, return all valid IPv4 addresses that can be formed by inserting exactly three dots. Return them in any order.

Medium Backtracking 4 Parts, Valid Range Time: O(1) Space: O(1)
1 Understanding the Problem ▾

A valid IPv4 address looks like 192.168.1.1: four non-empty numeric parts separated by dots. Given a raw digit string, we must figure out every legal way to insert exactly three dots.

Each part must satisfy all three of these rules simultaneously:

✓ Value 0–255
"0", "192", "255" all fine
✓ Length 1–3 digits
One, two, or three characters per part
✗ No leading zeros
"01", "001", "010" all invalid
💡 Key Insight

The problem reduces to: place exactly 3 dots somewhere in the string to create 4 valid parts. This is a constraint-satisfaction enumeration — try all placements, discard illegal ones, collect the legal results. Backtracking is perfect because we can detect invalid states early and cut entire branches.

✓ "25525511135" — 2 valid IPs

"255.255.11.135" ✓
"255.255.111.35" ✓

✗ Common traps in this input

"2.55.255.11135" ✗ (last part too long)
"255.25.51.1135" ✗ (last part > 255)

Edge Cases to consider:

  • String length < 4: impossible — can't have four non-empty parts → return []
  • String length > 12: impossible — four parts of 3 digits each = 12 max → return []
  • "0000" → exactly one answer: "0.0.0.0"
  • "1111111111111" (13 chars) → impossible, return []
2 The Algorithm: Backtracking with Pruning ▾

We use a recursive backtracking function backtrack(start, parts) where start is our current index into the string and parts is the list of segments chosen so far.

1 Base case: If len(parts) == 4, we've placed all 4 parts. Check: does start == len(s)? If yes, all characters are consumed — this is a valid IP. Add ".".join(parts) to results. Return either way (no more recursion needed).
2 Pruning check: Count parts_left = 4 - len(parts) and remaining = len(s) - start. If remaining < parts_left (not enough digits to fill remaining parts) OR remaining > 3 * parts_left (too many digits, can't absorb them all), this branch is dead — return early.
3 Try each segment length 1, 2, 3: For each length, extract the candidate segment s[start : start+length]. Validate: no leading zero (if length > 1, segment[0] must not be '0'), and integer value ≤ 255. If either check fails, break (not continue!) since longer segments will also fail.
4 Choose → Explore → Unchoose: Append the valid segment to parts, recurse with start + length, then pop the segment back. This is the classic backtracking triple: choose, explore all possibilities down that path, undo the choice.
🔑 Why "break" Instead of "continue" on Violation

When we find that a 2-digit segment has a leading zero (e.g., "0X"), we know a 3-digit segment starting at the same position ("0XY") also has a leading zero. When a 2-digit value exceeds 255 (impossible actually since 99 max, but for 3-digit: e.g., "256"), a 3-digit starting here would be even larger. In both cases, all longer segments fail too — so we break out of the length loop, not just skip to the next length.

The three pruning conditions at each call:

✗ Too few remaining
remaining < parts_left
Can't make enough non-empty parts
✗ Too many remaining
remaining > 3 × parts_left
Can't absorb all digits in valid segments
✗ Invalid segment
Leading zero in multi-digit,
or integer value > 255
3 Why It Works: The Invariant ▾
🔐 The Invariant

At every call to backtrack(start, parts), the variable start equals exactly sum(len(p) for p in parts). In other words, we have consumed exactly the characters that are assigned to completed parts. No character is reused, none is skipped. Each recursive call extends this invariant by one more valid segment.

Completeness — why no valid IP is missed: The only choices at each step are segment lengths 1, 2, or 3. Since any valid IP segment has length 1, 2, or 3 (a 4-digit segment would be at minimum 1000, which exceeds 255), the loop over lengths 1–3 covers all possibilities. The base case checks we used all characters. Together these guarantee completeness.

Soundness — why only valid IPs are produced: A result is appended only when four conditions simultaneously hold: (a) exactly 4 parts, (b) entire string consumed, (c) each part has no leading zero unless it is "0" itself, (d) each part's integer value ≤ 255. All four are enforced before any append. No invalid string can reach the result list.

⚠️ Common Mistake: Checking Value Without Checking Leading Zero

int("01") == 1, which passes the ≤ 255 test! The leading zero check is separate and must come first: if length > 1 and segment[0] == '0': break. If you only check the integer value, you will incorrectly include segments like "01", "001", "010" in your results.

Constant time bound: The string length is bounded: valid inputs have 4–12 characters. Each recursive call tries at most 3 lengths. The recursion depth is at most 4 (one level per IP part). So the total calls are bounded by 3^4 = 81. This is O(1) with respect to input size — the algorithm terminates in constant time regardless of the specific digit string.

4 The Code (Annotated) ▾

Solution 1 — Backtracking (Optimal, Interview Pick)

def restoreIpAddresses(s: str) -> list[str]:
result = [] ← Accumulate all valid IPs here
def backtrack(start: int, parts: list): ← start = current index; parts = segments chosen so far
if len(parts) == 4: ← Base case: exactly 4 parts placed
if start == len(s): ← Must have consumed the entire string
result.append(".".join(parts)) ← Valid IP! Join parts with dots
return ← Stop recursing (4 parts is max)
parts_left = 4 - len(parts) ← How many more parts we still need
remaining = len(s) - start ← How many characters are left unassigned
if remaining < parts_left \ ← Not enough digits to fill remaining parts
or remaining > 3 * parts_left: ← Too many digits, can't absorb in valid segs
return ← Prune this branch early
for length in range(1, 4): ← Try segments of length 1, 2, 3
if start + length > len(s): break ← Don't read past end of string
seg = s[start : start + length] ← Extract candidate segment
if length > 1 and seg[0] == '0': ← Leading zero? (only "0" itself is allowed)
break ← Longer segs from same pos also have leading zero
if int(seg) > 255: ← Value exceeds IP octet max
break ← Longer segs will be even larger
parts.append(seg) ← CHOOSE: commit this segment
backtrack(start + length, parts) ← EXPLORE: recurse with advanced start
parts.pop() ← UNCHOOSE: backtrack to try next length
backtrack(0, []) ← Kick off from index 0, empty parts
return result
⚠️ The Subtle Backtracking Pattern: parts.append / parts.pop

This is the canonical choose → explore → unchoose template. The parts list is mutated in place (not copied). After the recursive call returns (having explored all sub-paths from this choice), we pop() to undo the choice — leaving parts exactly as it was before we entered this iteration of the loop. This allows the next iteration (next length) to start fresh.

Solution 2 — Iterative (Three Nested Loops)

Since exactly 3 dots must be placed, we can use three nested for-loops to enumerate all dot positions explicitly — clean, no recursion:

def restoreIpAddresses(s: str) -> list[str]:
n, result = len(s), []
def ok(part): ← Validation helper: no leading zero, value ≤ 255
return (len(part) <= 3
and int(part) <= 255
and (part[0] != '0' or len(part) == 1))
for i in range(1, min(4, n)): ← First dot: after 1, 2, or 3 chars
for j in range(i+1, min(i+4, n)): ← Second dot: 1-3 chars after first
for k in range(j+1, min(j+4, n)): ← Third dot: 1-3 chars after second
p1,p2,p3,p4 = s[:i],s[i:j],s[j:k],s[k:]
if all(ok(p) for p in [p1,p2,p3,p4]):← All 4 parts valid?
result.append(f"{p1}.{p2}.{p3}.{p4}")
return result
💡 Complexity Summary

Backtracking: Time O(1) — at most 81 branch points × depth 4. Space O(1) stack + O(k) output where k ≤ 2^32 but practically ≤ 19 valid IPs per input.
Iterative: Same O(1) — at most 3×3×3 = 27 valid loop bodies. Simpler to reason about; less generalizable.
For the interview, mention backtracking since it generalizes to variable numbers of parts (e.g., IPv6).

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Pattern — Enumeration with Constraints

"Return all valid …" immediately signals backtracking or combinatorial enumeration. The fixed structure (exactly 4 parts) and per-part constraints (0–255, no leading zero) mean pruning will eliminate most branches efficiently. This is not DP — we need the actual strings, not a count.

Step 2: Define State and Choices

State = (current index in string, list of parts built). Choice = segment length (1, 2, or 3). Transition = validate segment, append to parts, advance index. Base case = 4 parts built and index at end of string.

Step 3: Enumerate Pruning Conditions Aloud

Show thoroughness by listing all three: (a) too few chars remaining, (b) too many chars remaining, (c) invalid segment (leading zero or value > 255). Mention that on violation, we break the length loop — not just skip — because longer segments at the same position always fail for the same reason.

Step 4: Offer the Iterative Alternative

Since the structure is exactly 3 dots, mention "I could also use three nested loops — one per dot position — and validate all 4 segments. Same O(1) complexity, no recursion. But backtracking generalizes better if the part count changes."

🗣️ What to Say to the Interviewer

"I need to place exactly 3 dots into the digit string to form 4 valid IP parts.
I'll use backtracking — at each step I pick a segment of length 1, 2, or 3 from the current position.
Before each recursive call I prune: if too few or too many chars remain for the parts left, abort.
Each candidate segment must pass two checks: no leading zero (if multi-digit), and value at most 255.
When both checks fail, I break — not continue — since longer segments fail for the same reason.
Base case: 4 parts built and the full string consumed → valid IP, add to results.
The recursion depth is at most 4, with 3 choices each, so worst case is 81 paths — O(1) time."
🔮 Likely Follow-up Questions
  • "What if it were IPv6?" → 8 groups, hex digits, 0–FFFF. Same backtracking with different validation.
  • "Can you do it without recursion?" → Three nested for-loops (dot positions i, j, k).
  • "Is '010.10.10.10' valid?" → No — '010' has a leading zero despite being value 10.
  • "What about '0.0.0.0'?" → Valid! Input "0000", all parts are "0", no leading zero issue.
  • "How many valid IPs can a 12-char string produce at most?" → At most 19 (provable upper bound).

7 Summary & Key Takeaways ▾
IP addresses have fixed structure: 4 non-empty parts separated by exactly 3 dots
Backtracking tries segment lengths 1, 2, 3 at each position; recurses on valid ones
Three pruning conditions catch dead branches before full recursion: too few/many chars, invalid segment
Leading zero check is separate from value check — int("01") passes ≤ 255 but "01" is still invalid
Use break (not continue) when a violation is found — longer segments at same position also fail
Classic backtracking triple: parts.append (choose) → backtrack() (explore) → parts.pop() (unchoose)
Time complexity is O(1): search space bounded by 3^4 = 81 paths; input bounded to 4–12 chars
ApproachTimeSpaceNotes
Naive (all splits, validate after) O(n²) O(1) No early pruning; generates many invalid splits first
Iterative (3 nested loops) O(1) O(1) Simple; only works for fixed 4-parts structure

🎯 When to Use This Backtracking Pattern
  • Problem asks to "generate all valid X" from a string — enumeration with constraints
  • Fixed or bounded number of groups/slots to fill from a linear sequence
  • Each slot has a small number of valid options (here: 1–3 digit lengths)
  • Constraints can be checked incrementally — prune bad prefixes without generating full solutions
