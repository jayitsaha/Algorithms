LeetCode #1472

Design Browser History

Implement BrowserHistory — visit a new URL, go back N steps, go forward N steps. Visiting a new URL erases all forward history.

Medium Design List + Two Pointers Time: O(1) back/forward Space: O(n)
1 Understanding the Problem ▾

You're building the navigation core of a web browser. Three operations: visit(url) navigates to a new page, back(steps) moves backward through history, and forward(steps) moves forward. The nuanced rule: when you visit() while in the middle of your history, all pages ahead of you are permanently erased — just like a real browser.

💡 Key Insight

Think of browser history as a tape with a read-head at position curr. You slide the head left (back) and right (forward). When you visit a new URL, you write to the tape at curr and cut off everything after it. No data needs to move — only the head position changes on navigation.

Edge Cases: back(steps) when steps exceeds available history — clamp to index 0. forward(steps) when at the end — clamp to last index. Steps can be arbitrarily large; just saturate at the boundary.

2 The Algorithm: Dynamic Array + Current Index Pointer ▾

Store all URLs in a flat Python list (self.history) and maintain a single integer self.curr pointing to the currently-viewed page. The end of the list is an implicit second pointer. Together these give O(1) navigation — back and forward are just integer arithmetic, no data movement.

1 Initialize: history = [homepage], curr = 0. The tape has one entry; the head is at position 0.
2 visit(url): Slice history = history[:curr+1] (truncate forward history), then append(url) and set curr = len(history) - 1.
3 back(steps): curr = max(0, curr - steps). Clamp prevents going before the start. Return history[curr].
4 forward(steps): curr = min(len(history) - 1, curr + steps). Clamp prevents going past the end. Return history[curr].
🔑 Why Not Two Stacks?

The intuitive approach (a back-stack and a forward-stack) is O(steps) for back/forward — each step pops from one stack and pushes to the other. The single-list + index approach does the same navigation in O(1) because arithmetic replaces data movement. Propose stacks first to show awareness, then optimize.

⚠️ Two-Stack Cost Analysis

Two stacks is correct but suboptimal. back(100) requires 100 pop-push operations. The array approach does: curr = max(0, curr - 100) — one operation. For large histories navigated frequently, the difference compounds severely.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At all times: history[0..curr] (inclusive) contains the reachable past URLs in chronological order. history[curr+1..end] contains forward URLs accessible via forward(). curr always points to the URL you are currently viewing.

Why visit() preserves the invariant: history[:curr+1] keeps exactly the "past + current" slice. Appending the new URL then sets curr to len-1, making the new page the current. The forward slice becomes empty — correct, because you can't go forward from a freshly visited page.

Why clamping is correct: max(0, curr - steps) saturates at the array's lower bound. min(len-1, curr + steps) saturates at the upper bound. These are the exact semantics the problem specifies: "you can only return to the minimum of steps you can go back."

⚠️ Classic Mistake: Forgetting the Truncation

If you just append on visit() without first truncating, then going back and visiting creates stale forward data. Example: visit A → visit B → visit C → back(1) → you're at B. Now visit D. Without truncation history = [A, B, C, D] with curr=3, but forward() from D could return C — a URL that was supposed to be erased.

Proof — no stale forward data: After visit(), curr = len(history)-1 always. Since curr equals the last valid index, there is no history[curr+1] — the forward slice is provably empty. forward(any) immediately saturates at curr. Invariant holds.

4 The Code (Annotated) ▾

Solution 1 — Array + Index Pointer (Optimal, Interview Pick)

class BrowserHistory:
def __init__(self, homepage: str):
self.history = [homepage] ← tape starts with just the homepage
self.curr = 0 ← read-head at index 0 (the only page)
def visit(self, url: str) -> None:
self.history = self.history[:self.curr + 1] ← truncate: wipe everything after curr
self.history.append(url) ← add new URL at end of remaining tape
self.curr = len(self.history) - 1 ← move head to the new last position
def back(self, steps: int) -> str:
self.curr = max(0, self.curr - steps) ← slide head left, clamp at 0
return self.history[self.curr] ← return the page we landed on
def forward(self, steps: int) -> str:
self.curr = min(len(self.history)-1, self.curr+steps) ← slide right, clamp at last index
return self.history[self.curr] ← return the page we landed on
💡 Complexity Summary

visit(): O(k) for the slice where k = curr (Python creates a new list). Append is O(1) amortized.
back() / forward(): O(1) — pure arithmetic + O(1) array access.
Space: O(n) where n = total distinct URLs visited (not counting erased forward history).

Solution 2 — Two Stacks (Brute Force, O(steps) navigation)

class BrowserHistory:
def __init__(self, homepage: str):
self.back_st = [homepage] ← all history including current (top = current)
self.fwd_st = [] ← forward history (top = closest forward)
def visit(self, url: str) -> None:
self.back_st.append(url) ← push new page onto back stack
self.fwd_st.clear() ← erase all forward history (O(1) via clear)
def back(self, steps: int) -> str:
while steps > 0 and len(self.back_st) > 1: ← can't pop last element (homepage)
self.fwd_st.append(self.back_st.pop()) ← move one page from back → forward
steps -= 1
return self.back_st[-1] ← top of back stack = current page
def forward(self, steps: int) -> str:
while steps > 0 and self.fwd_st:
self.back_st.append(self.fwd_st.pop()) ← move one page from forward → back
steps -= 1
return self.back_st[-1]
⚠️ Two Stacks: O(steps) per back/forward, not O(1)

Every back(k) call requires k individual pop+push operations. The array approach does the same in O(1) by adjusting a single index. Mention both in interviews — propose stacks for clarity, then optimize to the array approach.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Enumerate Operations and Target Complexities

List all three operations: visit(), back(), forward(). Ask: "What complexity should each be?" For this problem, back/forward should be O(1) since they're frequent. visit() can be O(n) amortized. This drives the design choice.

Step 2: Propose Two Stacks — then Identify the Bottleneck

"First approach: back-stack and forward-stack. visit() is O(1) — push new URL, clear forward stack. But back(steps) is O(steps) — I pop one URL per step. That's not O(1)." Show you can analyze complexity critically.

Step 3: Reframe — Navigation Is Position Arithmetic

Key insight: "I don't need to physically move data between structures. If I record all URLs in a flat array and track my position with an integer, back/forward become arithmetic — subtract or add, then clamp. O(1) guaranteed."

Step 4: Walk Through the visit() Wipe Carefully

The tricky bit: "On visit(), I must truncate the array at curr+1 before appending. Otherwise stale forward URLs remain accessible. Let me walk through: back(1) sets curr to N-2, then visit(url) does history[:N-1] + [url]. Old page at N-1 is gone."

🗣️ What to Say to the Interviewer

"I'll store URLs in a list with a curr index as my current position.
For visit(): first history = history[:curr+1] — this erases forward history.
Then append the new URL and set curr = len(history) - 1.
For back(steps): curr = max(0, curr - steps) — arithmetic, no data moved.
For forward(steps): curr = min(len-1, curr + steps) — same idea.
back() and forward() are O(1). visit() is O(curr) for the slice, amortized O(1)."
🔮 Likely Follow-up Questions
  • "Can you make visit() also O(1)?" → Use a doubly linked list: visit creates a new node, sets next=null (cuts forward), sets node as current tail.
  • "What if you need to support bookmarks?" → Add a hashmap: URL → list of indices where it was visited.
  • "Design with tabs?" → Each tab is its own BrowserHistory instance stored in a list indexed by tab number.
  • "How to handle very large history (memory limit)?" → Bounded deque; evict oldest entries when capacity exceeded.

7 Summary & Key Takeaways ▾
One flat list + one index integer gives O(1) back/forward — no data movement
visit() must truncate at curr+1 before appending — wipes forward history correctly
Clamping: max(0, curr-steps) and min(len-1, curr+steps) handles all boundary cases
Two-stack approach is O(steps) per navigation — propose it, then pivot to array
The "curr pointer" is the two-pointer pattern applied to a single growing array

🎯 When to Recognize the List + Index Pointer Pattern
  • Need "current position" in a sequence with backward/forward navigation
  • "History" where entering a new state erases the future states
  • O(1) position changes required, no data movement acceptable
  • Undo/redo systems, text editor cursors, playlist current-track pointers
