LeetCode #1429

First Unique Number

Design a data structure that supports adding numbers from a stream and efficiently querying the first number that has appeared exactly once.

Medium Queues Queue + Hash Map add: O(1) showFirstUnique: O(1) amortized
1 Understanding the Problem ▾

You're given an initial array of numbers and will receive a stream of new numbers one at a time via add(value). You must design a class where showFirstUnique() returns the first number (by insertion order) that has appeared exactly once so far. If no such number exists, return -1.

💡 Key Insight

Two things need to happen simultaneously: we need to know how many times each number has appeared (a hash map handles this in O(1)), and we need to preserve insertion order to know which unique number came first (a queue handles this). The elegant trick is lazy eviction: we never proactively remove a number from the queue when it becomes a duplicate — we wait until it surfaces at the front and check then.

Edge cases: all numbers become duplicates (return -1), a single-element stream (unique until a duplicate arrives), numbers can be added many more times than they appear in the initial array, and the queue may become empty if every number gets a duplicate added.

2 The Algorithm: Queue + Hash Map (Lazy Eviction) ▾

We maintain two data structures in parallel. The queue preserves insertion order; the hash map is the ground truth for counts. On every query, we peel duplicates off the front of the queue until we find a number whose map count is exactly 1.

1 Initialize: Create an empty deque and an empty counts dict. Process each number in nums by calling add() — this keeps initialization and stream updates consistent.
2 add(value): If value is not yet in counts, enqueue it (first occurrence only — prevents queue bloat). Then always increment counts[value]. If count goes from 1→2, the number just lost its uniqueness, but we do NOT touch the queue.
3 showFirstUnique(): Peek the front of the queue. If counts[front] == 1, return it — it's the answer. If counts[front] > 1, this number became a duplicate after being enqueued; pop it and repeat. If the queue empties, return -1.
4 Amortized analysis: Each value is enqueued at most once and dequeued at most once. Even if a single showFirstUnique() call pops k elements, those k will never be popped again. Total pops across all calls ≤ total enqueues ≤ N, so the amortized cost per call is O(1).
🔑 Why a deque, not a list or a set?

A Python list supports O(1) append to back but O(n) removal from front (list.pop(0) shifts all elements). A set loses insertion order entirely. A deque (double-ended queue) gives O(1) for both appendleft/append and popleft/pop, making it the perfect structure when we need FIFO with efficient front access.

3 Why It Works: The Invariant ▾
🔐 The Invariant

The counts map is always an exact record of how many times each number has been seen. The queue contains every number ever added, in insertion order, but may include numbers that have since become duplicates. The correct answer is always the front-most element of the queue whose count equals exactly 1.

Correctness of lazy eviction: When add(x) causes counts[x] to go from 1 to 2, x is now a duplicate. It may still be in the queue. But when any future showFirstUnique() reaches x at the front, it checks the map and finds counts[x] > 1, so it skips and pops x. The map is the source of truth; the queue only provides ordering.

Correctness of ordering: Numbers are enqueued only on their first occurrence. Since a deque maintains FIFO order and we never re-insert, the relative order of elements in the queue matches their first-insertion order exactly. So the "first unique by insertion" is always correctly identified as the leftmost element in the queue with count 1.

⚠️ Common Trap: Enqueuing Every Occurrence

If you naively do self.queue.append(value) every single time add() is called — including duplicates — the algorithm still works (duplicates get skipped during queries), but the queue grows unnecessarily large. After n adds to the same value, the queue has n copies. The optimization: only enqueue when value not in counts, keeping the queue size at most equal to the number of distinct values.

Why O(1) amortized, not O(n) per call: In the worst case, showFirstUnique() pops every element in the queue before finding one with count 1 (or returning -1). But those elements, once popped, are gone forever. The total number of pops across all calls is bounded by the total number of enqueues, which is at most the number of distinct values seen. Spreading this cost across all operations gives O(1) amortized.

4 The Code (Annotated) ▾

Solution 1 — Queue + Hash Map (Optimal, O(1) amortized)

from collections import deque ← deque = double-ended queue, O(1) at both ends
class FirstUnique:
def __init__(self, nums):
self.queue = deque() ← Maintains insertion order of distinct values
self.counts = {} ← Ground truth: num → how many times seen
for num in nums: ← Process initial array same as stream values
self.add(num) ← Reuse add() — keeps logic DRY and consistent
def showFirstUnique(self) -> int:
while self.queue: ← Keep peeking until unique found or queue empty
front = self.queue[0] ← Peek without popping — O(1) for deque
if self.counts[front] == 1: ← Count is 1 → still unique → this is our answer
return front
self.queue.popleft() ← Duplicate at front: lazily evict it
return -1 ← Queue empty: every number is a duplicate
def add(self, value) -> None:
if value not in self.counts: ← First time seeing this value
self.queue.append(value) ← Enqueue only on first occurrence (optimization)
self.counts[value] = self.counts.get(value, 0) + 1 ← Always increment count
⚠️ Why peek (queue[0]) instead of pop immediately?

We use self.queue[0] to peek the front without removing it. If the front element has count 1, we return it and leave it in the queue — it must stay there so the next query can find it again quickly. We only call popleft() when we confirm the front is a duplicate and we're discarding it permanently.

Solution 2 — Brute Force (O(n) per query, for contrast)

from collections import Counter
class FirstUnique:
def __init__(self, nums):
self.nums = list(nums) ← Store everything in a plain list
def showFirstUnique(self) -> int: ← O(n) — rebuilds counter on every call
counts = Counter(self.nums) ← O(n) scan to count all values
for num in self.nums: ← Second O(n) scan to find first unique
if counts[num] == 1: ← Check frequency
return num
return -1
def add(self, value) -> None:
self.nums.append(value) ← O(1) add, but every query pays O(n)
💡 Complexity Summary

Queue + Hash Map (optimal): add() O(1), showFirstUnique() O(1) amortized, Space O(d) where d = distinct values
Brute Force: add() O(1), showFirstUnique() O(n), Space O(n)
Key trade-off: The optimal solution pays a small extra memory cost (the deque) in exchange for constant-time queries — a classic time-space optimization.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Pattern

Key signals: "design a data structure", "stream of numbers", "first unique". "First" combined with a stream means insertion order matters → think Queue. "Unique" means frequency tracking → think Hash Map. These two signals together almost always mean Queue + Hash Map.

Step 2: Propose Brute Force First

"Simplest approach: store all numbers in a list, rebuild a Counter on every query. add() is O(1), but showFirstUnique() is O(n). With m queries that's O(nm) total — too slow. Let's precompute."

Step 3: Identify the Two Needs

"I need (1) frequency counts in O(1) → hash map, and (2) insertion-order tracking in O(1) → queue. If I combine them: add to both on each add(), query front of queue on showFirstUnique(). But I need to handle numbers that were unique and then became duplicates."

Step 4: Introduce Lazy Eviction + Prove Amortized O(1)

"When a number becomes a duplicate, I don't remove it from the queue eagerly — that would be O(n) to scan for it. Instead I leave it. When querying, I pop the front if its count >1. Each element is popped at most once → amortized O(1) across all calls." This is the insight that closes the interview.

🗣️ What to Say to the Interviewer

"This needs insertion order AND frequency — Queue + Hash Map.
add(): if first time, enqueue. Always increment counts.
showFirstUnique(): peek queue front; if count==1, return it.
If count>1, pop (lazily evict) and peek again. Return -1 if empty.
Each element enqueued once, dequeued at most once → amortized O(1)."
🔮 Likely Follow-up Questions
  • "What if we also need to remove numbers from the stream?" → Doubly linked list + hash map (like LRU), O(1) removal from middle
  • "First unique character in a string stream?" → Same exact pattern with chars (#387)
  • "Can you do it with strict O(1) worst-case (not amortized)?" → Not easily — requires more complex bookkeeping; amortized is the accepted answer
  • "What if we need the k-th unique number?" → Pop k-1 unique numbers from a temporary queue, return the k-th

7 Summary & Key Takeaways ▾
Queue (deque) preserves insertion order; hash map tracks counts — together they give O(1) amortized queries
Lazy eviction: only remove duplicates from the queue front when a query arrives, not eagerly on add()
Only enqueue on first occurrence — keeps queue size bounded to distinct values, not total insertions
Amortized O(1): each element enters and leaves the queue at most once across all operations
Return -1 when the queue empties — every inserted number has been seen more than once

🎯 When to Use Queue + Hash Map
  • "First/oldest element satisfying property P?" → Queue for order + Map for property
  • Data streams where you need insertion order AND a per-element attribute simultaneously
  • Any "evict-when-invalid" pattern — lazy removal is O(1) amortized; eager removal would cost O(n)
  • When the property of elements can degrade over time (unique → duplicate)
