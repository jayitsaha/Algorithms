LeetCode #759

Employee Free Time

Given K employees each with a sorted list of non-overlapping work intervals, find all time windows where no employee is working — the common free time for everyone.

Hard Advanced Data Structures Merge Intervals Time: O(N log K) Space: O(K)
1 Understanding the Problem ▾

You have K employees, each with a personal work schedule — a sorted list of non-overlapping time intervals. The task: find every gap of time where absolutely no employee is working. We want the gaps in the union of all work intervals.

💡 Key Insight

Free time = gaps between merged working intervals. If you combine every employee's schedule into one big sorted list and merge all the overlapping/touching segments, any gap between consecutive merged blocks means nobody is working during that time. It doesn't matter who is or isn't working — only whether the gap exists in the union.

Example: 3 Employees

Emp 1: [1,3], [6,7]
Emp 2: [2,4]
Emp 3: [2,5], [9,12]
Union: [1,5] ∪ [6,7] ∪ [9,12]

✓ Answer: Free Time Gaps

[5, 6] ← between [1,5] and [6,7]
[7, 9] ← between [6,7] and [9,12]

(No gap before t=1 or after t=12)

Key observations:

  • Each employee's own schedule is already sorted and non-overlapping.
  • We do NOT care which employee is working — only that someone is.
  • Gaps before all work starts or after all work ends are not included (only finite gaps).
  • N = total intervals across all employees; K = number of employees (heap size).
⚠️ Common Misconception: Union vs. Intersection

Free time is NOT when every employee is simultaneously free at the same moment (that would be the intersection of free periods). It is simply the gaps in the union of all their work intervals — any window where nobody happens to be working.

2 The Algorithm: K-Way Merge with Min-Heap ▾

Since every employee's schedule is already sorted, we can merge all K schedules simultaneously using a min-heap (priority queue) — always picking the globally smallest start-time interval at each extraction. This is the classic K-way merge pattern, exactly like merging K sorted linked lists. Once intervals arrive in sorted order, identifying gaps is trivial.

1 Seed the heap: Push the first interval from each of the K employees as a tuple (start, end, emp_index, interval_index). The heap orders by start time. Cost: O(K log K).
2 Bootstrap prev_end: Pop the globally earliest interval (the heap minimum). Set prev_end = interval.end. Push the next interval from that same employee into the heap.
3 Merge loop: While the heap is non-empty, pop the smallest-start interval. If interval.start > prev_end, a gap exists — record [prev_end, interval.start] as free time. Update prev_end = max(prev_end, interval.end). Push the next interval from this employee.
4 Return result: Each of the N intervals is pushed and popped once (O(log K) per operation) → O(N log K) total. The heap never exceeds K entries.
🔑 Why Heap Instead of Sort All?

Flattening and sorting all N intervals costs O(N log N). The heap approach costs O(N log K). Since K ≤ N, heap is never worse and can be dramatically better: 100 employees with 10,000 intervals each → log(100) = 7 comparisons vs log(1,000,000) = 20. Heap also avoids materializing the full sorted list in memory — only K entries live in the heap at any time.

💡 Alternative: Flatten + Sort (Simpler, O(N log N))

Collect all intervals into one list, sort by start time, then sweep to merge and detect gaps. Perfectly correct and often sufficient. The heap approach is strictly better when K is much smaller than N. In interviews, propose flatten+sort first, then optimize to heap.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every point in the loop, prev_end equals the rightmost boundary of all working intervals processed so far. When the next popped interval starts after prev_end, it means no work interval covered the window [prev_end, next.start] — that window is completely free.

Why the heap guarantees correctness: A min-heap on start times ensures we always process intervals in non-decreasing order of their start — the same global order we would get from sorting all N intervals. This is critical: since we process the earliest-start interval first, the moment we see one that starts after prev_end, we know nothing could bridge the gap. Any interval processed before had an earlier start; any interval still in the heap starts no earlier than current (heap invariant) and thus cannot fill in earlier time.

Why max(prev_end, interval.end)? When two intervals overlap (e.g., [1,5] and [2,3]), the second is nested inside the first. We must not shrink prev_end to 3 — the work zone still extends to 5. Taking the maximum prevents falsely shrinking the coverage.

⚠️ Edge Case: Touching Intervals

If intervals touch but don't overlap — e.g., [1,5] and [5,8] — the condition next.start > prev_end evaluates to 5 > 5 which is False. No free-time gap is recorded (correctly, since [5,5] is a zero-duration gap). Touch ≠ gap.

Why no gap before t=0 or after last interval? We initialize prev_end with the first interval's end and never compare anything before it. We stop when the heap empties and never record anything after the last interval. The "finite gaps" constraint is enforced naturally.

4 The Code (Annotated) ▾

Solution 1 — K-Way Merge with Min-Heap (Optimal, Interview Pick)

import heapq
def employeeFreeTime(schedule): ← schedule: list[list[Interval]]
heap = [] ← min-heap: (start, end, emp_i, iv_i)
for i, emp in enumerate(schedule): ← seed one entry per employee
iv = emp[0]
heapq.heappush(heap, (iv.start, iv.end, i, 0))
result = []
# Bootstrap: pop the globally earliest interval
s, e, ei, ii = heapq.heappop(heap) ← first interval overall
if ii + 1 < len(schedule[ei]): ← push this employee's next interval
nxt = schedule[ei][ii + 1]
heapq.heappush(heap, (nxt.start, nxt.end, ei, ii + 1))
prev_end = e ← rightmost work boundary seen so far
while heap:
s, e, ei, ii = heapq.heappop(heap)
if s > prev_end: ← gap! nobody working in [prev_end, s]
result.append(Interval(prev_end, s))
prev_end = max(prev_end, e) ← extend coverage (handles nested intervals)
if ii + 1 < len(schedule[ei]): ← push employee's next interval
nxt = schedule[ei][ii + 1]
heapq.heappush(heap, (nxt.start, nxt.end, ei, ii + 1))
return result
⚠️ Why Two-Phase Initialization?

We pop the first interval before the loop to avoid needing prev_end = None with a conditional inside the loop. An alternative is prev_end = -float('inf') — then the first iteration's check s > -inf is always True, but we would add a spurious gap from negative infinity. The two-phase approach is cleaner.

Solution 2 — Flatten and Sort (Simpler, O(N log N))

def employeeFreeTime(schedule):
all_ivs = sorted( ← flatten all intervals
[iv for emp in schedule for iv in emp],
key=lambda x: x.start ← sort globally by start
)
result = []
prev_end = all_ivs[0].end ← bootstrap with first interval
for iv in all_ivs[1:]:
if iv.start > prev_end: ← gap found
result.append(Interval(prev_end, iv.start))
prev_end = max(prev_end, iv.end) ← extend coverage
return result
💡 Complexity Summary

K-Way Heap (Optimal): Time O(N log K), Space O(K) heap + O(G) result
Flatten + Sort: Time O(N log N), Space O(N) flat list
N = total intervals; K = employees; G = number of gaps found

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Patterns

You see "intervals" + "find gaps in their union" → Merge Intervals pattern. You see "K sorted sources" + "process globally in order" → K-way merge with min-heap. Both signals fire together here.

Step 2: Propose Simple Approach First

"A straightforward approach: collect all intervals, sort by start time, then sweep and merge — record gaps. O(N log N) time, O(N) space. Simple and correct. But we can do better because the schedules are pre-sorted."

Step 3: Optimize with Min-Heap

"Each employee's schedule is already sorted. I can use a min-heap for K-way merge — seed with one interval per employee, always pop the minimum-start, push the next from that employee. O(N log K) time, O(K) space. Log K instead of log N because the heap stays size K."

Step 4: Explain the Tuple Trick

"Python heapq needs comparable elements. Interval objects aren't natively comparable, so I push tuples (start, end, emp_index, interval_index). Tuples compare lexicographically, so the heap sorts by start first (which is what we want)."

🗣️ What to Say to the Interviewer

"Free time means gaps in the union of all work intervals — not intersection.
Since each employee's schedule is already sorted, I'll use a min-heap for K-way merge.
Push one interval per employee to start. Pop the globally earliest-start interval each time.
If its start is greater than prev_end, record [prev_end, start] as free time.
Update prev_end = max(prev_end, end). Push the employee's next interval. Repeat.
Time O(N log K), Space O(K) — heap stays size K, never O(N)."
🔮 Likely Follow-up Questions
  • "Find when at least M employees are all free?" → Count active workers; free when count falls to 0
  • "Schedules are NOT pre-sorted?" → Sort each employee's list first: O(N log N) pre-sort
  • "N is huge but K = 3?" → Heap gives log 3 ≈ 1.6 per step vs log N — enormous savings
  • "Merge K sorted arrays (integers, not intervals)?" → Same heap pattern as LeetCode #23

7 Summary & Key Takeaways ▾
Free time = gaps in the union of all work intervals (not intersection)
K pre-sorted lists → K-way min-heap merge: O(N log K) instead of O(N log N)
Heap always yields the globally smallest start — identical to sorting, but without the O(N) memory
Track prev_end; take max on overlap; record gap when next.start > prev_end
Heap size is exactly K throughout — grows no larger than one entry per employee
