LeetCode #410

Split Array Largest Sum

Split an integer array into k non-empty contiguous subarrays to minimise the largest subarray sum.

Hard Binary Search Binary Search on Answer Time: O(n log S) Space: O(1)
1 Understanding the Problem ▾

You have an integer array and exactly k allowed pieces. Each piece must be a contiguous subarray — you can't reorder elements. The cost of any split scheme is the sum of the largest piece. You want to minimise that cost by choosing where to place the k−1 dividers.

💡 Key Insight

Instead of asking "which of the exponentially many k-way splits is best?", flip the question: "Given a budget limit X, can we split into at most k pieces so that no piece exceeds X?" This feasibility check is O(n) greedy. We binary-search over all candidate values of X to find the minimum that is feasible.

Concrete example — nums=[7,2,5,10,8], k=2:

✗ Sub-optimal (3-piece attempt)

[7,2] | [5,10] | [8]
Would need k=3 pieces
Not allowed — k=2 only

✓ Optimal Split (k=2)

[7,2,5] | [10,8]
Sums: 14, 18
Largest = 18 — minimum possible

Why the answer lives in [max(nums), sum(nums)]? The minimum answer is at least max(nums) — whichever piece contains the largest element must accommodate it entirely. The maximum answer is sum(nums) — using one big piece. The true answer lives inside this interval. Crucially, if limit X is feasible, limit X+1 is also feasible — a monotone structure perfect for binary search.

🔑 The Monotone Property

Define feasible(X) = "can we split into ≤ k pieces with each piece sum ≤ X?" If True for X, it stays True for all larger X. If False for X, it stays False for smaller X. This exact monotone property enables binary search to zero in on the minimum feasible X in O(log S) iterations.

2 The Algorithm: Binary Search on the Answer ▾

The paradigm is "Binary Search on the Answer" (also called "BS on Sum" or "Parametric Search"). Instead of searching an array for a target, we binary-search the space of possible answers — here, the range of "largest subarray sum" values.

1 Set search range. lo = max(nums) — the tightest possible limit (a single element can't be split). hi = sum(nums) — the loosest limit (one piece). The answer is somewhere in between.
2 Bisect left loop. While lo < hi, set mid = (lo + hi) // 2 and call feasible(mid).
3 Feasibility check (greedy). Walk nums left to right. Keep a running sum. If adding the next element would exceed mid, start a new piece (pieces += 1, reset running sum). If pieces needed ever exceeds k → return False.
4 Narrow range. feasible(mid) == True → answer is at most mid, so hi = mid. feasible(mid) == False → mid is too tight, so lo = mid + 1.
5 Return lo. When lo == hi, we've converged to the minimum feasible limit — the answer.

⚠️ Why Greedy Works in the Feasibility Check

When we greedily extend a piece as far as possible before cutting, we use the fewest cuts possible for a given limit. If even the greedy minimum number of pieces exceeds k, no rearrangement of cuts can do better. This "earliest-cut" greedy is optimal for feasibility checking because we're packing left to right with a fixed bound per bin.

3 Why It Works: The Invariant ▾
🔐 The Loop Invariant

At every iteration: lo is the smallest value we haven't yet confirmed infeasible (i.e., it might still be the answer), and hi is the smallest value we've confirmed is feasible. When lo == hi, they converge to the minimum feasible value — that's the answer.

Why the answer space is monotone: feasible(X) is a step function that jumps from False to True exactly once — at the minimum feasible X (our answer). Below that point: always False. At and above: always True. Binary search on a monotone boolean function converges to the jump point in O(log(hi − lo)) steps.

Why bisect-left (hi = mid when feasible)? We want the smallest X where feasible(X) is True. If mid works, the answer is mid or lower — never higher. So we narrow the upper bound to mid, keeping mid as a candidate. When mid fails, nothing ≤ mid can work, so lo jumps to mid+1.

⚠️ Classic Off-by-one Bug

Using hi = mid - 1 when feasible would discard mid as a candidate, potentially skipping the answer. Using lo = mid (not mid+1) when infeasible creates an infinite loop when lo+1 == hi. The exact correct template for "find minimum feasible": feasible → hi = mid; infeasible → lo = mid + 1.

Edge cases:

  • k == 1: Only one piece allowed. lo = max(nums), hi = sum(nums). feasible(hi) always True → converges to sum(nums). Correct.
  • k ≥ n: Each element gets its own piece. lo = hi = max(nums). No iterations — immediate answer. Correct.
  • Single element: lo = hi = nums[0]. Immediate answer. Correct.
  • All same values: Binary search still works — monotone property holds regardless of value distribution.
4 The Code (Annotated) ▾

Solution 1 — Binary Search on the Answer (Optimal, Interview Pick)

def splitArray(nums, k): ← nums = array, k = number of pieces
def feasible(limit): ← inner helper: O(n) greedy feasibility check
pieces, curr = 1, 0 ← start with 1 piece, running sum = 0
for num in nums: ← scan left to right
if curr + num > limit: ← adding num would overflow this piece
pieces += 1 ← cut here; start a new piece
curr = 0 ← reset running sum for the new piece
if pieces > k: ← needed more pieces than allowed
return False ← early exit — this limit is too tight
curr += num ← add element to current piece
return True ← finished within k pieces → limit works
lo, hi = max(nums), sum(nums) ← search space: [tightest, loosest] bounds
while lo < hi: ← bisect-left loop; exits when lo == hi
mid = (lo + hi) // 2 ← candidate "largest sum" limit
if feasible(mid): ← can we split within this limit?
hi = mid ← feasible: narrow upper bound (keep mid as candidate)
else:
lo = mid + 1 ← infeasible: must use a larger limit
return lo ← lo == hi = minimum feasible limit = answer
⚠️ Why lo = max(nums) not max(nums) - 1?

Any single element max(nums) must be fully inside one piece — you can't split within an element. So no limit below max(nums) can ever be feasible. Starting lo there skips impossible values without missing any answer.

Solution 2 — DP O(n²k) (for conceptual understanding)

def splitArray_dp(nums, k): ← slower, but shows the problem's structure
n = len(nums)
pre = [0] * (n + 1) ← prefix sums for O(1) range sum queries
for i in range(n): pre[i+1] = pre[i] + nums[i]
INF = float('inf')
dp = [[INF]*(k+1) for _ in range(n+1)] ← dp[i][j] = min-max-sum for first i elems in j pieces
dp[0][0] = 0 ← base: 0 elements, 0 pieces = cost 0
for i in range(1, n+1):
for j in range(1, k+1):
for m in range(j-1, i): ← try all split points: last piece = nums[m..i-1]
last_sum = pre[i] - pre[m] ← sum of the last piece
dp[i][j] = min(dp[i][j], max(dp[m][j-1], last_sum))
return dp[n][k]
💡 Complexity Summary

Binary Search on Answer: Time O(n log S) where S = sum(nums). Space O(1).
DP: Time O(n²k), Space O(nk). Practical only for small n and k.
When n ≥ 1000: BS dominates. log(10⁹) ≈ 30 iterations × O(n) = fast.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognise "Minimise the Maximum"

The phrase "minimise the largest subarray sum" is the canonical trigger for Binary Search on the Answer. Also triggers: "split into k workers to minimise maximum load", "capacity to ship within D days", "Koko eating bananas". All encode the same monotone feasibility structure.

Step 2: Pin Down the Search Range

Lower bound: every element must fit in some piece → lo = max(nums). Upper bound: no cuts at all → hi = sum(nums). Say this explicitly in the interview — it shows you understand the problem's constraints.

Step 3: Write the Greedy Feasibility Check

"Given limit X, I'll greedily fill each piece as far as possible. When adding the next element would exceed X, start a new piece. If pieces needed exceeds k, infeasible." This is the heart of the solution — get it right and the rest is template code.

Step 4: Apply the Bisect-Left Template

while lo < hi: mid=(lo+hi)//2; hi=mid if feasible else lo=mid+1; return lo. Memorise this template for "minimum feasible X" problems. For "maximum feasible X", use bisect-right.

🗣️ What to Say to the Interviewer

"This is a 'minimise the maximum' problem — the key pattern for binary search on the answer.
The answer must lie between max(nums) (we can't split individual elements) and sum(nums) (one big piece).
I'll binary search that range. For each candidate limit mid, I check feasibility greedily:
scan left to right, accumulate a running sum; if adding the next element would exceed mid, cut there.
Count pieces — if we need more than k pieces, mid is too tight; otherwise feasible.
Total: O(n log S) time, O(1) space. Let me code it up."
🔮 Likely Follow-up Questions
  • "What if k > n?" → Answer = max(nums). Each element is its own piece. lo starts at max(nums) and never moves.
  • "DP approach?" → O(n²k) with prefix sums. Fine for small inputs; mention it but recommend BS.
  • "How many iterations does the binary search take?" → O(log(sum−max)) ≈ O(log(n × max_val)) ≈ 30 for typical constraints.
  • "Related problem: Capacity to Ship Packages (#1011)?" → Identical structure — ship capacity instead of limit, days instead of k. Same feasibility check template.

7 Summary & Key Takeaways ▾
Binary Search on the Answer = flip an optimisation problem into a sequence of feasibility checks
Search range is always [max(nums), sum(nums)] for "split into k pieces" problems
Feasibility check is O(n) greedy — extend each piece until overflow, then cut
Bisect-left template: hi=mid when feasible, lo=mid+1 when not, return lo
Total O(n log S) time, O(1) space — the gold-standard solution for this problem class

🎯 When to Recognise "Binary Search on Answer"
  • "Minimise the maximum …" or "Maximise the minimum …"
  • "Split into k groups / workers / days / ships" with a cost on each group
  • "Find the smallest X such that X is achievable" — and feasibility is O(n) to check
  • The answer is a number in a computable range with a monotone feasibility property
