LeetCode #2452

Words Within Two Edits of Dictionary

Given query words and a dictionary, return all queries that differ from some dictionary word by at most 2 character substitutions.

Medium Tries Trie + Edit Distance Time: O((D+Q)·W) Space: O(D·W)
1 Understanding the Problem ▾

You are given a list of query words and a dictionary. All words have the same length. For each query, decide: does it differ from any dictionary word by at most 2 character substitutions?

A substitution means changing exactly one character at one position. Two words are "within 2 edits" if you can change 0, 1, or 2 characters of one to match the other. Insertions and deletions are not counted — only substitutions, because all words have equal length.

💡 Key Insight

When all strings have the same length, edit distance reduces to Hamming distance — just count positions where the characters differ. We only care whether this count is ≤ 2.

✓ Query Accepted (≤ 2 edits)

query = "adc"
dict = "abc"
pos 0: a=a ✓ | pos 1: d≠b ✗ | pos 2: c=c ✓
edits = 1 ≤ 2 → INCLUDE

✗ Query Rejected (> 2 edits)

query = "xyz"
dict = "abc"
pos 0: x≠a ✗ | pos 1: y≠b ✗ | pos 2: z≠c ✗
edits = 3 > 2 → EXCLUDE

🔍 Why a Trie?

All words share the same length and use lowercase letters. A Trie groups dictionary words by shared prefix. When traversing for a query, we can prune entire branches the moment we accumulate more than 2 mismatches — any word deeper in that subtree would also exceed the limit.

Edge cases: A query identical to a dictionary word (0 edits) is always included. Empty dictionary means no queries are accepted. Query length is guaranteed equal to dictionary word lengths.

2 The Algorithm: Trie with DFS Edit-Distance Pruning ▾

We store all dictionary words in a Trie. For each query, we DFS through the Trie character by character. At each level, we compare the query's character against the branch we're exploring. The mismatch count is passed downward; branches exceeding the threshold are immediately abandoned.

1 Build Trie: Insert every dictionary word. Each node stores children (dict of char → node) and is_end (True if a complete word ends here).
2 For each query: Run DFS from the Trie root with idx=0 (position in query) and edits=0 (mismatches so far).
3 Prune immediately: At any DFS call, if edits > 2, return False. This node and all its descendants are eliminated in O(1).
4 Match characters: For each child character ch, compute new_edits = edits + (ch != query[idx]). Recurse with idx+1.
5 Terminal check: At idx == len(query), return node.is_end. If True, this path represents a complete dictionary word within 2 edits. Early-exit on first True.

💡 Trie Pruning vs Brute Force

Brute force: O(Q × D × W) — every query checks every dictionary word fully. Trie: when many words share a prefix that causes early mismatch accumulation, entire subtrees are pruned. The more common prefixes exist, the bigger the speedup. In the worst case (no shared prefixes), both are equivalent.

3 Why It Works: The Invariant ▾
🔐 The Core Invariant

At every DFS call at depth d, the parameter edits equals the exact Hamming distance between the first d characters of the query and the first d characters of any dictionary word traced by the current Trie path.

Why pruning is safe: Hamming distance is monotonically non-decreasing as we add more characters. Each new character either matches (edits stays same) or mismatches (edits increases by 1). It can never decrease. So if edits > 2 at depth d, then for ALL dictionary words in this Trie subtree, their Hamming distance with the query is already > 2, even before looking at remaining characters.

Why the terminal check is correct: At depth W (word length) with node.is_end = True, we've compared all positions and found a complete dictionary word. The accumulated edits value is the exact Hamming distance between the full query and that word.

Pruning Correctness

edits_at_depth_d ≤ edits_at_W

Contrapositive:
edits_at_d > 2
→ edits_at_W > 2
→ safe to prune all descendants

Match Correctness

idx == W
AND node.is_end == True
AND edits ≤ 2

→ Hamming dist ≤ 2
→ query is valid ✓

⚠️ Early Exit Correctness

The problem only asks whether a match exists, not to find all matches. As soon as DFS returns True on any path, we immediately propagate True upward and stop exploring other branches. This is safe because we just need existence, not enumeration.

4 The Code (Annotated) ▾

Solution 1 — Trie + DFS Pruning (Interview Pick)

class TrieNode:
def __init__(self):
self.children = {} dict: char → TrieNode child
self.is_end = False True when a complete dict word ends here
class Solution:
def twoEditWords(self, queries, dictionary):
root = TrieNode() empty root; no character
for word in dictionary: build Trie from all dict words
node = root
for ch in word:
if ch not in node.children:
node.children[ch] = TrieNode()
node = node.children[ch]
node.is_end = True mark terminal node for this word
def dfs(node, query, idx, edits):
if edits > 2: PRUNE: exceeded threshold; skip subtree
return False
if idx == len(query): consumed all query chars: check is_end
return node.is_end
for ch, child in node.children.items():
new_edits = edits + (ch != query[idx])+1 on mismatch
if dfs(child, query, idx + 1, new_edits):
return True early exit: found one match, done!
return False no child branch led to a valid match
return [q for q in queries if dfs(root, q, 0, 0)]
⚠️ Common Mistake: Over-counting or Under-counting Edits

The expression edits + (ch != query[idx]) uses Python's boolean-to-int coercion: True = 1, False = 0. This cleanly adds 1 for mismatch, 0 for match. Don't use if/else — this is idiomatic Python and preferred in interviews.

Solution 2 — Brute Force Baseline

def twoEditWords(self, queries, dictionary):
def hamming(a, b): count position-wise mismatches
return sum(x != y for x, y in zip(a, b))
return [q for q in queries
if any(hamming(q, d) <= 2 for d in dictionary)]

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Simplify Edit Distance

Notice all words have equal length. Full edit distance (Levenshtein) isn't needed — only substitutions matter. This is Hamming distance. Immediately simplifies the problem.

Step 2: State the Brute Force Explicitly

For each query, compare against all D dictionary words using a position-wise mismatch count. O(Q · D · W). This is your correct starting point — state it before optimizing.

Step 3: Identify the Redundant Work

If two dictionary words share a 3-mismatch prefix with a query, you're checking both of them even though you could have stopped at the prefix. A Trie lets you share this prefix check.

Step 4: Propose the Trie + DFS Design

Insert all dictionary words into a Trie. For each query, DFS with a mismatch counter. Prune at edits > 2. Return True on first leaf with edits ≤ 2 and is_end. O((D+Q)·W) amortized.

Step 5: Prove Correctness with the Invariant

Hammer distance is monotonically non-decreasing: a prefix with > 2 mismatches means the full word will also have > 2. This makes the pruning safe. State this explicitly.

🗣️ What to Say to the Interviewer

"Since all words have the same length, this is Hamming distance — I just count mismatching positions. Brute force: O(Q × D × W) — each query vs every dict word. Observation: words sharing a common prefix have shared mismatch-checking work. A Trie exploits this — once a prefix causes edits > 2, I prune the entire subtree. I'll build a Trie from the dictionary in O(D·W), then DFS for each query in O(W) on average with pruning. The key invariant: Hamming distance only increases as we go deeper — so pruning early is always safe. Early exit once the first match is found."
💡 Follow-Up Questions to Expect
  • Different word lengths? — Use full Levenshtein DP; Trie approach still works but tracking state is harder.
  • At most k edits (not 2)? — Change the pruning condition from edits > 2 to edits > k. Same algorithm.
  • Worst case for Trie? — All words start with distinct characters (no shared prefixes). Trie gives no benefit; same as brute force.
  • Can we query the Trie online (one at a time)? — Yes! Build Trie once, then each query is independent. Build cost amortized over many queries.

7 Summary & Key Takeaways ▾
Equal-length words mean edit distance = Hamming distance — just count position-wise mismatches
A Trie groups dictionary words by shared prefix, enabling pruning: one mismatch-check covers many words
The DFS mismatch counter is monotonically non-decreasing — this is what makes aggressive pruning provably safe
Early exit (return True immediately on first match) avoids exploring duplicate/redundant branches
State brute force first in interviews; then explain the Trie pruning optimization and its amortized benefit
🎯 Pattern Recognition Signals
  • Strings of equal length + bounded number of substitutions → Hamming distance + Trie
  • "Within k edits of dictionary" with fixed-length words → Trie DFS with counter k
  • Multiple queries against a fixed dictionary → build once, query many (Trie amortizes build cost)
  • When you see "prune search space based on accumulated cost" → DFS + early termination
