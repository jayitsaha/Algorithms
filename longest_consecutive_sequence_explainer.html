LeetCode #128

Longest Consecutive Sequence

Given an unsorted array of integers, return the length of the longest consecutive elements sequence. Must run in O(n) time.

Medium Hash Tables Hash Set + Sequence Start Check Time: O(n) Space: O(n)
1 Understanding the Problem ▾

Given an unsorted array of integers — say [100, 4, 200, 1, 3, 2] — find the length of the longest sequence of consecutive integers that can be formed from the array's elements. Order in the original array doesn't matter; only which values exist.

The sequence 1, 2, 3, 4 can be formed from the example (all four values are present), giving a length of 4. The lone values 100 and 200 each form sequences of length 1. So the answer is 4.

💡 Key Insight

A number n is the start of a new sequence if and only if n - 1 is NOT in the array. By only counting sequences from their starts, we process every element at most twice — giving us O(n). The hash set provides O(1) membership checks to make this efficient.

✓ Example: [100, 4, 200, 1, 3, 2]

Sequences present:
[100] → length 1
[1, 2, 3, 4] → length 4 ← best
[200] → length 1
Answer: 4

Edge Cases to Handle

[] → 0 (empty array)
[7] → 1 (single element)
[1,1,2,2] → 2 (duplicates)
[-2,-1,0,1] → 4 (negatives OK)
[9,5,1,3,7] → 1 (no consec pairs)

What "consecutive" means here: integers differing by exactly 1. The problem is NOT asking for the longest increasing subsequence in array order — it's about which integer values exist, regardless of their positions.

2 The Algorithm: Hash Set + Sequence Start Check ▾

The trick: use a hash set for O(1) lookups, then only "start counting" a sequence at its unique beginning. Every sequence has exactly one start — the element whose predecessor is absent. We skip all middle elements.

1 Build hash set: num_set = set(nums). This costs O(n) time and O(n) space. Duplicates are deduplicated. From here, "is X in the array?" runs in O(1) average time.
2 Identify sequence starts: For each n in num_set, check: is n - 1 in the set? If YES → skip (n is in the middle of someone else's sequence). If NO → n is the unique start of a new sequence.
3 Count forward from each start: From a confirmed start n, walk cur = n, n+1, n+2 ... as long as each next value is in the set. Count the length of this run.
4 Track global maximum: After each sequence is fully counted, best = max(best, length). Return best when all elements have been processed.

Decision at each number n:

🔑 Why This Is O(n): Amortized Analysis

The outer loop visits each of the n unique values once: O(n). The inner while-loop, across all invocations, can visit each element at most once total — because sequences are non-overlapping and we only count from starts. Total inner-loop iterations ≤ n. Combined: O(n + n) = O(n).

3 Why It Works: The Invariant ▾
🔐 The Core Invariant

Every maximal consecutive sequence has exactly one element where n - 1 is absent: its start. By counting only from these starts, every sequence is processed exactly once — never missed, never double-counted.

Proof of completeness (no sequence is missed): Let S = {a, a+1, ..., a+k} be any maximal consecutive sequence in the array. By "maximal," a-1 is not in the array. So when the outer loop reaches a, the check a-1 not in num_set is True, and we count S in full. Every sequence is found.

Proof of no double-counting: For any non-start element m (where m-1 is in the set), we immediately skip it. m is only counted during the sequence that starts at a < m. Each element contributes to at most one sequence's count.

⚠️ Why Sorting Doesn't Meet the Requirement

The intuitive solution is to sort the array and scan for consecutive runs: O(n log n). This works correctly but fails the O(n) constraint. With a hash set, we replace sorted order with O(1) membership queries — getting the same logical scanning without sorting. This is the core trade-off to articulate in an interview.

Duplicate handling: Since we build a set, duplicates are automatically removed. [1, 1, 2, 2, 3] → {1, 2, 3}. The sequence 1→2→3 has length 3, correctly ignoring the duplicate values.

💡 Intuition: The "Secretary" Analogy

Imagine each integer is an employee, and consecutive employees form a team. We ask each employee: "Is there someone with employee ID one less than you?" If yes, that person is your supervisor — you're in the middle of a chain, so wait. If no, you're the team leader — count how many are in your chain. Every team gets counted exactly once, by its leader.

4 The Code (Annotated) ▾

Solution 1 — Hash Set + Sequence Start Check (Optimal · Interview Pick)

def longestConsecutive(nums: list[int]) -> int:
num_set = set(nums) ← O(n): build set, deduplicate, O(1) lookups
best = 0 ← Global max (0 handles empty input)
for n in num_set: ← Iterate SET (not list) to avoid reprocessing duplicates
if n - 1 not in num_set: ← Only start counting from sequence starts
cur = n ← cur walks forward through the sequence
length = 1 ← Start element counts as length 1
while cur + 1 in num_set:← Extend while next consecutive exists
cur += 1 ← Advance pointer
length += 1 ← Tally one more element in this run
best = max(best, length) ← Sequence done — update global max
return best ← 0 if empty input; longest sequence otherwise
⚠️ Why `for n in num_set` not `for n in nums`?

If nums = [1, 1, 1, 2, 3] and we iterate nums, we process 1 as a start three times, each time counting 1→2→3. The answer is still correct (best = 3), but we do O(duplicate_count × sequence_length) redundant work. Iterating the set — where 1 appears once — keeps it strictly O(n).

Solution 2 — Sort and Linear Scan (O(n log n) · Simpler)

def longestConsecutive(nums: list[int]) -> int:
if not nums: return 0
nums = sorted(set(nums)) ← Sort unique values; O(n log n)
best = cur_len = 1 ← Start at length 1
for i in range(1, len(nums)): ← Compare each element to its predecessor
if nums[i] == nums[i-1] + 1: ← Consecutive → extend current run
cur_len += 1
else: ← Gap → save best, reset
best = max(best, cur_len)
cur_len = 1
return max(best, cur_len) ← Handle the final run (don't forget!)
💡 Complexity Summary

Hash Set (optimal): Time O(n) · Space O(n) — interview answer
Sort + Scan: Time O(n log n) · Space O(n) — propose this first as stepping stone
Brute Force: Time O(n³) — for each element, search for consecutive chain — never do this

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Restate and Clarify

"I need the length of the longest run of consecutive integers in an unsorted array. This is about which values exist, not their positions." Ask: "Are there duplicates?" (Yes — they don't extend sequences.) Ask: "Can values be negative?" (Yes — hash set handles this fine.)

Step 2: Propose the O(n log n) Approach

"The natural approach: sort the array, then scan for consecutive runs — O(n log n). This is correct and easy to implement. But the problem asks for O(n), so I need to find a way to avoid sorting."

Step 3: Arrive at the O(n) Insight

"What did sorting give me? The ability to check 'is n+1 present?' cheaply. A hash set gives me the same in O(1). The remaining issue is avoiding redundant counting — I solve this by only counting from sequence starts (where n-1 is absent)."

Step 4: Prove the Time Complexity

"The outer loop visits n elements. The inner while-loop, summed over all outer iterations, visits each element at most once — because sequences don't overlap and we only enter the while-loop at starts. Total: O(n). This is the amortized argument."

🗣️ What to Say to the Interviewer

"I'll start by building a hash set of all numbers — O(n) time and space.
Then I iterate through the set. For each n, I check: is n-1 in the set?
If yes, n is in the middle of a longer sequence — skip it to avoid double-counting.
If no, n is a unique sequence start — I count forward while n, n+1, n+2 ... exist in the set.
I track the max length seen. At the end I return it.
Time O(n) — each element is processed at most twice across all iterations.
Space O(n) for the hash set. This handles duplicates, negatives, and empty input."
🔮 Likely Follow-up Questions
  • "Return the actual sequence, not just the length?" → Track best_start alongside best; reconstruct with range(best_start, best_start + best).
  • "What if numbers are huge but sparse?" → Same approach works perfectly — hash set is index-independent.
  • "Alternative approach?" → Union-Find: union n with n+1 when both exist; largest component = answer. Also O(n α(n)).
  • "Longest arithmetic subsequence?" → Harder — use DP with a hash map tracking sequence lengths per difference (#1218).
