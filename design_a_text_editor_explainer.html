LeetCode #2296

Design a Text Editor

Design a text editor with a movable cursor that can add/delete characters and return the last k characters to the left of the cursor.

Hard Design Two Stacks Time: O(k) per op Space: O(n)
1 Understanding the Problem ▾

You need to build a simplified text editor from scratch — one that supports a movable cursor. The cursor sits at some position in a string of characters. You must implement four operations:

A addText(text) — Insert text at the cursor position, moving the cursor to the end of the inserted text.
D deleteText(k) — Delete up to k characters immediately to the left of the cursor. Return the actual number deleted.
← cursorLeft(k) — Move the cursor k positions to the left. Return the last 10 characters to the left of the cursor.
→ cursorRight(k) — Move the cursor k positions to the right. Return the last 10 characters to the left of the cursor.
💡 Key Insight

Everything that matters happens around the cursor. Characters to the left are added/deleted. The cursor moves left/right. If we split the text at the cursor into two pieces — "what's to the left" and "what's to the right" — every operation becomes a simple push/pop on two stacks.

Example: Text = "abc|de"

Left stack: ['a','b','c'] (top='c')
Cursor: | (between c and d)
Right stack: ['e','d'] (top='d')

addText("x") → "abcx|de"
deleteText(2) → "a|de" (deleted b,c)
cursorLeft(1) → cursor before c
Last 10 left: "abc" → returns "abc"

✗ Naive Approach: Plain String

Store as string + cursor index
addText: insert at idx → O(n) shift
deleteText: slice → O(n) copy
cursorLeft/Right: O(1) move, O(k) slice

Too slow for large strings!

Edge Cases: cursor at start (can't move left further), cursor at end, deleting more than available chars, k larger than text length, empty editor.

2 The Algorithm: Two-Stack Cursor Split ▾

Split the text at the cursor into two stacks. The cursor lives between them. Every operation is O(k) — no O(n) shifts needed.

1 addText(text): Push each character of text onto the LEFT stack. After inserting, the cursor is at the end of the new text — left stack holds all chars up to (and including) the new text, right stack unchanged.
2 deleteText(k): Pop up to k characters from the LEFT stack (stop if empty). Count actual deletions and return it. Since left stack's top is the character immediately left of cursor, this correctly deletes leftward.
3 cursorLeft(k): Pop up to k characters from LEFT stack, push each onto RIGHT stack. The cursor moves left. Then return the last min(10, len(left)) characters from the left stack (joined).
4 cursorRight(k): Pop up to k characters from RIGHT stack, push each onto LEFT stack. The cursor moves right. Return the last 10 characters from the left stack.
🔑 Why The Right Stack Stores Characters in Reverse

The right stack stores characters in reverse order of how they appear in the text. The character immediately to the right of the cursor is at the top of the right stack. When we pop from right and push to left (cursor moves right), the character that was just to the right becomes the character just to the left — exactly correct! No reversal needed at any point.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At all times: left stack (bottom-to-top) + right stack (top-to-bottom) = the full text, left-to-right. The cursor position is implicitly defined by the sizes of the two stacks. len(left) = cursor index. This invariant is preserved by every single operation.

Proof for each operation:

  • addText: Pushes to left → left grows, right unchanged. Full text = left+right still holds. Cursor moved right by |text| positions.
  • deleteText: Pops from left → left shrinks. The deleted character is simply removed from the concatenation. ✓
  • cursorLeft: Pop from left, push to right. One char moves from the end of left to the beginning of right. The full text (left+right) is unchanged, but cursor moved one left. ✓
  • cursorRight: Pop from right, push to left. One char moves from the beginning of right to the end of left. Full text unchanged, cursor moved one right. ✓
⚠️ Common Trap: Boundary Conditions

When moving left and already at the start (len(left) == 0), you cannot pop further — the cursor is already at position 0. Similarly for right. Always clamp with min(k, len(stack)) iterations. The problem says "move min(k, available)" which the loop naturally handles by checking if left (truthy when non-empty).

Why returning last 10 from the left is correct: After any cursor movement, the left stack holds all characters to the left of the new cursor position (bottom-to-top). The last (rightmost) of these are at the top of the left stack. So we read left[-min(10,len(left)):] which gives us exactly the 10 characters immediately preceding the cursor, in correct left-to-right order.

4 The Code (Annotated) ▾

Solution 1 — Two Stacks (Optimal, O(k) per operation) — Interview Pick ✓

class TextEditor:
def __init__(self):
self.left = [] ← chars before cursor, top = immediately left of cursor
self.right = [] ← chars after cursor, top = immediately right of cursor
def addText(self, text: str) -> None:
for ch in text: ← each char is pushed left-to-right
self.left.append(ch) ← push onto left stack; cursor advances after each char
def deleteText(self, k: int) -> int:
deleted = 0 ← count actual deletions (may be less than k)
while self.left and k > 0:
self.left.pop() ← remove char immediately left of cursor
deleted += 1
k -= 1
return deleted
def cursorLeft(self, k: int) -> str:
while self.left and k > 0:
self.right.append(self.left.pop()) ← char moves: left-of-cursor → right-of-cursor
k -= 1
return ''.join(self.left[-10:]) ← last 10 chars of left = 10 before new cursor pos
def cursorRight(self, k: int) -> str:
while self.right and k > 0:
self.left.append(self.right.pop()) ← char moves: right-of-cursor → left-of-cursor
k -= 1
return ''.join(self.left[-10:]) ← same: last 10 of left after moving right
⚠️ Why self.left[-10:] works correctly

Python list slicing with [-10:] returns the last (up to) 10 elements. If the left stack has fewer than 10 elements, it simply returns all of them — no IndexError, no special-casing needed. The join converts them to a string in the correct left-to-right order since they're stored bottom-to-top (which is left-to-right for our text).

Solution 2 — Doubly Linked List (Alternative Approach)

class TextEditor: ← DLL approach: O(k) moves, O(1) add/delete at cursor
def __init__(self):
self.text = [] ← use list as DLL simulation (simpler in Python)
self.pos = 0 ← cursor position: index into text
def addText(self, text: str) -> None:
for ch in text:
self.text.insert(self.pos, ch) ← O(n) insert — fine for interviews, not for production
self.pos += 1
def _get_left(self) -> str:
start = max(0, self.pos - 10)
return ''.join(self.text[start:self.pos])
💡 Complexity Comparison

Two Stacks: O(k) per cursorLeft/Right (each step is a pop+push), O(|text|) for addText, O(k) for deleteText, O(10) = O(1) for get-last-10.
Array with index: O(n) for addText/deleteText (shifting), O(k) for cursor moves. Much worse for frequent insertions.
Rope data structure: O(log n) all operations — used in real editors (VSCode uses a Piece Tree variant). Beyond interview scope.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Design Problem Type

"This is a data structure design problem. My first question: what operations need to be fast?" Insert/delete/move all happen relative to the cursor. The cursor is the central abstraction — everything is defined relative to it.

Step 2: Propose Naive, Identify Bottleneck

"A naive approach: store text as a string with a cursor index. addText needs to shift O(n) characters rightward. That's the bottleneck." Now think: "what if I don't store text as one contiguous piece? What if I split at the cursor?"

Step 3: The Two-Stack Insight

"I split at the cursor into TWO pieces. Left piece: chars before cursor. Right piece: chars after cursor. Now: add = append to left (O(1)), delete = pop from left (O(1)), move left = pop from left, push to right (O(k)), move right = reverse. No shifting!"

Step 4: Map Each Operation

Walk through each of the 4 operations with the two-stack model. Verify correctness with a small example. "Let me trace: start empty, addText('ab') → left=['a','b'], cursorLeft(1) → pop 'b' from left, push to right → left=['a'], right=['b']. Returns 'a'. Correct!"

🗣️ What to Say to the Interviewer

"The key observation is that the cursor splits the text into two independent halves.
I'll use two stacks: left holds chars before cursor (top = char just left of cursor),
right holds chars after cursor in reverse (top = char just right of cursor).
addText pushes each character onto left — O(|text|) total.
deleteText pops from left — O(k).
cursorLeft/Right transfers chars between stacks — O(k).
Returning last 10: slice the top of left stack — O(10) = O(1).
This gives O(k) per operation, O(n) space total."
🔮 Likely Follow-up Questions
  • "What's the real-world equivalent?" → Gap buffer data structure — used in Emacs, classic text editors. Two stacks is functionally a gap buffer.
  • "Can you support undo/redo?" → Add a history stack. Before each mutation, push current state (or the inverse operation) onto an undo stack.
  • "Can you make cursor movement O(1)?" → Rope / Piece Table. O(log n) insert/delete, O(log n) movement. VSCode uses a variant.
  • "What if text has Unicode (multi-byte chars)?" → Work at the grapheme cluster level, not byte level. Python handles this naturally with str indexing.

7 Summary & Key Takeaways ▾
The cursor naturally splits text into two halves — model them as two stacks
Left stack top = character immediately left of cursor (delete target, display source)
Right stack top = character immediately right of cursor (reversed storage)
Moving cursor k left/right = k stack transfers — O(k), not O(n)
Returning last 10 left = slice top 10 of left stack = O(1) — no traversal needed
This is the "Gap Buffer" data structure — classic text editor foundation

🎯 When to Recognize the Two-Stack / Gap-Buffer Pattern
  • "Design a data structure with a cursor/pointer that moves through a sequence"
  • Operations happen at or near a specific position — not random access
  • Need to efficiently insert/delete at the current position
  • The split-at-cursor insight: "what if I model left and right of position separately?"
