LeetCode #2115

Find All Possible Recipes from Given Supplies

Given recipes with their required ingredients and a list of initial supplies, return all recipes you can make — including chains where one recipe is an ingredient for another.

Medium Graph Topological Sort Time: O(V+E) Space: O(V+E)
1 Understanding the Problem ▾

You run a kitchen. You start with some base ingredients (the supplies list — think "pantry items"). You have a cookbook: each recipe in recipes[i] requires all ingredients in ingredients[i]. The critical twist: a finished recipe can itself be used as an ingredient for another recipe.

The question: which recipes can you actually cook? A recipe is cookable if every one of its ingredients is either available in your initial supplies, or is itself a recipe you can make.

💡 Key Insight

This is a dependency resolution problem. Think of npm packages: to install A, you first need B and C. If B also needs D, the order is D → B → A. Topological Sort processes each node only after all its dependencies are satisfied — that's exactly the "when can I make this recipe?" logic.

Concrete example walkthrough:

recipes=["bread","sandwich","burger"] | supplies=["yeast","flour","meat"]

yeast ✓ + flour ✓ → bread ✓ can make
bread + meat ✓ → sandwich ✓ can make
sandwich + bread → burger ✓ can make
✓ Makeable chain

yeast, flour → bread
bread + meat → sandwich
sandwich + bread → burger
Output: ["bread","sandwich","burger"]

✗ Blocked recipe

If "pizza" needs "dough"
and "dough" isn't a supply
or a makeable recipe,
pizza is blocked forever.

Edge cases: Circular dependency (recipe A needs B, B needs A — cycle, never makeable). Recipe needing an unknown ingredient (not in supplies, not a recipe). Recipe with zero ingredients (always makeable — check if this is intended). Empty supplies list.

2 The Algorithm: Kahn's BFS Topological Sort with Initial Set ▾

Model this as a directed graph (DAG): draw an edge ingredient → recipe for each requirement. This means "ingredient must be available before recipe can be made." Then run Kahn's Algorithm — a BFS topological sort starting from nodes with no prerequisites.

The "initial set" twist: supplies are already available, so we pre-seed the queue with all supplies rather than only zero-indegree recipe nodes. This lets us skip building edges for supply→recipe connections (supplies don't "come from" anywhere — they just exist).

1 Convert supplies to a set for O(1) lookup. Build a recipe_set to distinguish recipes from supplies when dequeuing.
2 Build the graph: For each recipe and its ingredients, create edges ingredient → recipe and increment indegree[recipe] — but only for ingredients NOT in supplies (supply ingredients are already satisfied).
3 Seed the queue: Enqueue all supplies. Also enqueue any recipe whose indegree is already 0 (its ingredients are all supplies). These can be processed immediately.
4 BFS loop: Dequeue an item. If it's a recipe, add it to result. Then, for every recipe that listed this item as an ingredient, decrement its indegree by 1. If any recipe's indegree reaches 0, enqueue it — all its requirements are now confirmed available.
5 Return result: All recipes we dequeued are guaranteed makeable. Anything never dequeued had an unsatisfied dependency (cycle or missing ingredient).
🔑 Kahn's Algorithm — The Indegree Invariant

Indegree of a node = number of its incoming edges still "active" (unsatisfied prerequisites). Kahn's works by: (1) start from zero-indegree nodes (nothing blocks them), (2) when we process a node, remove its outgoing edges — this may bring other nodes to zero, which we then enqueue. We visit each edge and node exactly once: O(V + E).

3 Why It Works: The Invariant ▾
🔐 The Invariant

Every item in the BFS queue is unconditionally available: either it was a supply (given), or every ingredient it required has already been dequeued and confirmed available before it. Therefore, any recipe we dequeue can genuinely be made.

Why indegree = 0 means "all ingredients ready": We initialize indegree[recipe] to count only non-supply ingredients (supply ingredients are already satisfied at initialization). Each time a non-supply ingredient becomes available (gets dequeued), we decrement the indegree of every recipe needing it. Indegree hitting 0 means every non-supply ingredient has been confirmed available — combined with supplies being pre-available, the recipe is now fully satisfiable.

Why cycles never produce a result: If recipe A needs B and B needs A, they form a cycle. A's indegree can only reach 0 if B is dequeued first, but B's indegree can only reach 0 if A is dequeued first. Neither ever gets enqueued. The algorithm handles cycles implicitly — no explicit cycle detection needed.

⚠️ Common Mistake: Not Skipping Supply Edges

Some implementations build edges for ALL ingredients, including supplies. This is wrong: it would initialize indegree[bread] = 2 for bread ← [yeast, flour]. But since yeast and flour are supplies (immediately available), bread should start with indegree = 0 and be enqueued right away. Always skip edges where the ingredient is in supplies_set.

Why only recipes appear in the output (not supplies): Supplies get dequeued (as BFS seeds), but they were given to us before we started cooking — they're not "recipes we made." We filter using recipe_set: only items that are in the recipes list get added to the result.

4 The Code (Annotated) ▾

Solution 1 — Kahn's Topological Sort (Optimal, Interview Pick)

from collections import defaultdict, deque
def findAllRecipes(recipes, ingredients, supplies):
supplies_set = set(supplies) ← O(1) lookup: is this item freely available?
recipe_set = set(recipes) ← O(1) lookup: is this item a recipe (not supply)?
graph = defaultdict(list) # item → recipes ← adjacency: "item unlocks these recipes"
indegree = {r: 0 for r in recipes} ← count of unsatisfied non-supply ingredients per recipe
for recipe, ings in zip(recipes, ingredients):
for ing in ings:
if ing not in supplies_set: ← skip if already available (supply)
graph[ing].append(recipe) ← ing must be ready before recipe
indegree[recipe] += 1 ← one more unsatisfied prerequisite
# Seed: all supplies + recipes already at indegree 0
queue = deque(supplies) ← supplies are immediately available
for r in recipes:
if indegree[r] == 0: ← all ingredients are supplies → ready now
queue.append(r)
result = []
while queue:
item = queue.popleft() ← next confirmed-available item
if item in recipe_set: ← only collect recipes, not supplies
result.append(item)
for dependent in graph[item]: ← propagate: this item is now available
indegree[dependent] -= 1 ← one fewer unsatisfied prerequisite
if indegree[dependent] == 0: ← all prerequisites now satisfied!
queue.append(dependent) ← ready to make this recipe
return result
⚠️ Why defaultdict(list) instead of dict with .get()?

When we look up graph[item] for a supply item (which has no entries), a regular dict would raise KeyError. defaultdict(list) returns an empty list by default — the for loop over an empty list is a no-op. Safe and clean.

Solution 2 — DFS with Memoization (alternative approach)

def findAllRecipes(recipes, ingredients, supplies):
supply_set = set(supplies)
ing_map = {r: ingredients[i] for i,r in enumerate(recipes)}← recipe → its ingredients
memo = {} # False=visiting/blocked, True=makeable
def can_make(item):
if item in supply_set: return True ← base case: it's a supply
if item not in ing_map: return False ← unknown item → can't make
if item in memo: return memo[item] ← cached: avoids re-work AND detects cycles
memo[item] = False # mark as "currently visiting"← cycle guard: if we revisit mid-DFS, return False
result = all(can_make(ing) for ing in ing_map[item])← all ingredients must be makeable
memo[item] = result
return result
return [r for r in recipes if can_make(r)]
💡 Complexity Summary

Kahn's BFS (Sol 1): Time O(V+E), Space O(V+E) — V = all unique items, E = total ingredient relationships. Each node and edge processed once.
DFS Memo (Sol 2): Time O(V+E), Space O(V) for memo + O(V) recursion stack depth. Same asymptotic, but BFS preferred (no stack overflow risk on deep chains, easier to trace in interviews).

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Pattern

Key signals in the problem: "X requires Y," "chain of dependencies," "find what's possible given starting conditions." These scream Topological Sort. Specifically: "given some starting set, find all reachable nodes" → Kahn's with initial set seeding.

Step 2: Model as a Graph

Say aloud: "I'll draw a directed graph where each edge goes from ingredient to recipe — meaning 'ingredient must be ready before recipe can be made.' Supplies have no prerequisites, so they're my starting nodes."

Step 3: Explain Indegree

"I'll count the indegree of each recipe — but only for non-supply ingredients, since supplies are already satisfied. Indegree 0 means all prerequisites are met. Kahn's BFS propagates availability: when a node is processed, it decrements the indegrees of nodes that depended on it."

Step 4: Handle the "Initial Set" Correctly

"The key difference from standard Course Schedule: I seed the queue with all supplies — they don't need any processing, they're just available. This avoids having to model supply→recipe edges at all."

🗣️ What to Say to the Interviewer

"This is a dependency resolution problem — I'll use topological sort.
Model: edge from ingredient to recipe meaning ingredient must come first.

Build indegrees counting only non-supply dependencies.
Seed BFS queue with all supplies and any recipe already at indegree 0.
BFS loop: dequeue item; if it's a recipe, add to result; then
decrement dependents' indegrees and enqueue those reaching 0.

Time O(V+E), Space O(V+E). Cycles are handled automatically —
cyclic nodes never reach indegree 0, never get enqueued."
🔮 Likely Follow-up Questions
  • "What if there's a circular dependency?" → Already handled. Cyclic nodes never reach indegree 0.
  • "What if a recipe has 0 ingredients?" → indegree[recipe] = 0 initially → enqueued right away → always makeable.
  • "Can you detect which recipes are blocked by cycles vs. missing ingredients?" → Yes, with extra bookkeeping — track which unresolved recipes point to non-existent items vs. to other unresolved recipes.
  • "What if recipes can depend on each other AND cycles exist?" → Same algorithm handles it — cycles just permanently block those recipe nodes.

7 Summary & Key Takeaways ▾
Model "X requires Y" as directed edge Y → X (Y unlocks X)
Indegree = unsatisfied prerequisites; initialize counting only non-supply deps
Seed the BFS with ALL supplies — they're available from the start with no waiting
Cycles are automatically excluded: cyclic nodes never reach indegree 0
Only add recipes to result when dequeued, not supplies (filter by recipe_set)
Convert supplies to a set first for O(1) ingredient membership checks
