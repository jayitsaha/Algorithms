LeetCode #847

Shortest Path Visiting All Nodes

Find the shortest path length that visits every node in an undirected graph. You may start and stop at any node, revisit nodes and edges.

Hard Dynamic Programming BFS + Bitmask State Graph Traversal Time: O(2ⁿ · n²) Space: O(2ⁿ · n)
1 Understanding the Problem ▾

You have an undirected graph of n nodes (0 to n−1). The adjacency list graph[i] lists all nodes directly connected to node i. Your goal: find the minimum number of edges you need to traverse to visit every node at least once. You may start at any node, end at any node, and revisit nodes and edges freely.

💡 Key Insight

This is NOT a standard shortest-path problem. The challenge is tracking which nodes you've visited so far, because two paths that arrive at the same node via different subsets of visited nodes are fundamentally different situations. We need a state that captures both where we are and what we've seen.

✓ Example: graph = [[1,2,3],[0],[0],[0]]

Star graph: 0 links to 1,2,3

Path: 1 → 0 → 2 → 0 → 3
Edges: 4 ← ANSWER

Or: 0 → 1 → 0 → 2 → 0 → 3
Edges: 5 (worse, started wrong)

✗ Why standard BFS fails

Standard BFS finds shortest path between TWO nodes.

Here we need to visit ALL nodes, possibly backtracking.

The graph has cycles; "visited" per normal BFS would block revisiting needed nodes.

Critical observation: The constraint n ≤ 12 is a massive hint. With 12 nodes, there are only 2¹² = 4096 possible subsets of visited nodes. This is small enough to encode as a bitmask integer and use it as part of the BFS state.

Edge cases: n=1 → answer is 0 (already visited everything). Disconnected components → not possible per problem statement (graph is guaranteed to have all nodes reachable).

2 The Algorithm: Multi-Source BFS on (Node, Mask) State Space ▾

The insight is to expand the state space. Instead of BFS on the original graph (where nodes are just 0..n−1), we do BFS on an augmented graph where each "node" is a pair (current_node, visited_mask). This augmented graph has n × 2ⁿ nodes — with n=12 that's only 49,152 states.

🔐 Bitmask State Encoding

We represent the set of visited nodes as an integer. Bit i = 1 means node i has been visited.

visited={0,1}→mask = 0b0011 = 3
visited={2,3}→mask = 0b1100 = 12
all 4 visited→mask = 0b1111 = 15 = (1<<4)−1
set bit i→new_mask = mask | (1<<i)
1 Multi-source initialization: Since we can start at any node, add ALL starting states (i, 1<<i, dist=0) to the BFS queue simultaneously. Mark them all as seen. This is called "multi-source BFS."
2 BFS expand: Dequeue state (node, mask, dist). For each neighbor nei of node, compute new_mask = mask | (1<<nei).
3 Goal check: If new_mask == full_mask (all bits set), we've visited every node — return dist + 1 immediately. BFS guarantees this is the minimum.
4 Deduplication: Only enqueue (nei, new_mask) if that pair hasn't been seen before. Mark it as seen before enqueuing (not after dequeuing) to avoid adding the same state multiple times.
📐 State Space Size
States = n × 2ⁿ
Transitions per state ≤ n (each node has at most n−1 neighbors)
Total work = O(n² × 2ⁿ)
n=12 → 12 × 12 × 4096 = 589,824 operations ← very fast
3 Why It Works: The Invariant ▾
🔐 The BFS Shortest-Path Invariant

In BFS on an unweighted graph, the first time we discover a state, we discover it via the shortest path to that state. This is BFS's core property — it processes states in order of non-decreasing distance.

Why does the (node, mask) state space work? Because two paths that arrive at the same node with the same visited-mask are interchangeable. They can reach exactly the same future states from here. The shorter one strictly dominates.

Formally: if path A arrives at (node=2, mask=0111) in 3 steps, and path B arrives at the same state in 5 steps, then any completion of path B can be replicated by path A with fewer total steps. We can safely discard path B.

⚠ Why Not Dijkstra?

Since all edges have weight 1 (unweighted graph), BFS is sufficient and faster. Dijkstra adds O(log n) overhead per operation. Only switch to Dijkstra if edges had different weights.

💡 Why n ≤ 12 Matters

The bitmask has 2ⁿ possible values. For n=12: 4,096. For n=20: 1,048,576 (1M — still ok). For n=30: 1 billion — too slow. The constraint n ≤ 12 is the problem telling you: "bitmask DP or BFS is the intended solution."

Why multi-source BFS? We're asking: "starting from the optimal start node, what's the minimum distance?" We don't know which start node is optimal, so we try all of them simultaneously. Multi-source BFS finds the answer in a single pass — equivalent to adding a virtual source node connected to all real start nodes at distance 0.

4 The Code (Annotated) ▾
from collections import deque// deque gives O(1) popleft (list.pop(0) is O(n))
def shortestPathLength(graph):
n = len(graph)// number of nodes
if n == 1: return 0// single node: trivially visited already
full_mask = (1 << n) - 1// e.g. n=4 → 0b1111=15; all nodes visited
queue = deque()
seen = set()// set of (node, mask) pairs we've already processed
for i in range(n):// MULTI-SOURCE: seed from every possible start
queue.append((i, 1 << i, 0))// (start_node, only_bit_i_set, dist=0)
seen.add((i, 1 << i))// mark seen NOW, before dequeueing
while queue:
node, mask, dist = queue.popleft()// FIFO → processes shorter paths first
for nei in graph[node]:// try all neighbors
new_mask = mask | (1 << nei)// set bit nei (mark neighbor as visited)
if new_mask == full_mask:// all nodes now visited?
return dist + 1// YES — return immediately; BFS guarantees minimum
if (nei, new_mask) not in seen:// new state?
seen.add((nei, new_mask))// mark before enqueuing
queue.append((nei, new_mask, dist + 1))// +1 edge
⚠ Common Mistake: Single Start Node

Starting BFS from only node 0 is wrong. The optimal path might start from node 1 or node 3. Always seed all n start states. In the star-graph example, starting from a leaf (nodes 1,2,3) is better than starting from the hub (node 0).

⚠ Common Mistake: Marking Seen After Dequeue

If you mark (nei, new_mask) as seen only when you dequeue it (not when you enqueue it), the same state can be added to the queue many times, causing exponential queue growth. Always mark before enqueueing.

Alternative: Bitmask DP (Bottom-Up)

def shortestPathLength_dp(graph):
n = len(graph)
full = (1 << n) - 1
# dp[mask][node] = min steps to be at node with visited=mask
INF = float('inf')
dp = [[INF] * n for _ in range(1 << n)]
for i in range(n): dp[1 << i][i] = 0// base: start at node i, visited={i}
# Use BFS-like order or relax via priority queue
# For unweighted, BFS (above) is simpler and identical complexity
for mask in range(1 << n):
for node in range(n):
if not (mask >> node & 1): continue// node must be in mask
if dp[mask][node] == INF: continue
for nei in graph[node]:
nm = mask | (1 << nei)
if dp[nm][nei] > dp[mask][node] + 1:
dp[nm][nei] = dp[mask][node] + 1
return min(dp[full])
💡 BFS vs DP — Which to Use?

Both have the same asymptotic complexity. BFS is simpler and faster in practice for unweighted graphs — no 2D array allocation, terminates early on first full-mask state. Use BFS in interviews. DP is useful if you need to reconstruct the path or if edges have weights (use Dijkstra instead of BFS then).

5 Interview Strategy: How to Arrive at This ▾
Step 1: Read the Constraints First

Immediately notice n ≤ 12. Any time you see n ≤ 20 with "visit all nodes/cities/cells", think bitmask. The problem is essentially asking for the "Shortest Hamiltonian Path" variant, which is NP-hard in general but solvable with bitmask DP/BFS for small n.

Step 2: Define the State

"What information do I need at each step to determine what I can do next?" Answer: (1) which node am I currently at, and (2) which nodes have I already visited. Encode (2) as a bitmask. State = (node, visited_mask). Total states = n × 2ⁿ.

Step 3: Choose BFS (not DFS, not Dijkstra)

Unweighted graph + minimum edges = BFS. DFS would explore all paths unnecessarily. Dijkstra adds unneeded overhead since edge weights are all 1. BFS on the augmented state space is the right choice.

Step 4: Handle the "Any Start" Constraint

Multi-source BFS: add all (i, 1<<i, 0) states to the initial queue. This elegantly handles "you may start at any node" without needing to run separate BFS from each start.

Step 5: Goal and Termination

Goal: new_mask == (1<<n) - 1. Return immediately when found — BFS guarantees optimality. The "early termination" on goal state makes this fast in practice.

7 Summary & Key Takeaways ▾
State = (node, visited_mask) — both pieces are needed; mask alone loses position, position alone loses history
Multi-source BFS: seed ALL start nodes at distance 0 since any node can be the starting point
n ≤ 12 (or ≤ 20) with "collect all" → immediately think bitmask; 2¹² = 4096 is tiny
Mark state as seen BEFORE enqueueing — prevents O(exponential) queue growth
BFS on unweighted graphs guarantees shortest path — return immediately on first full-mask hit
full_mask = (1<<n)−1 is the goal; OR-ing bits into mask adds newly visited nodes
💡 When to Use BFS + Bitmask
  • Graph problem where you need to "collect" or "visit" all nodes/keys/items
  • Small n (≤ 20) — makes 2ⁿ states tractable
  • You may revisit nodes/edges (standard BFS's visited array would block this)
  • Finding minimum steps/cost to reach a goal involving a subset
  • "Start at any node" — use multi-source BFS initialization
