LeetCode #95

Unique Binary Search Trees II

Given an integer n, return all structurally unique BSTs which have exactly n nodes with values 1 to n. Return the answer in any order.

Medium Dynamic Programming Generate Trees Recursively Time: O(4ⁿ/n^1.5) Space: O(4ⁿ/n^1.5)
1 Understanding the Problem ▾

A Binary Search Tree (BST) enforces one rule: for every node, all values in its left subtree are smaller, and all values in its right subtree are larger. Given the integers 1 through n, we want to find every distinct shape of BST that satisfies this rule using each of these n values exactly once.

💡 Key Insight

If we choose value i as the root, the BST property forces values 1…i-1 into the left subtree and values i+1…n into the right subtree. The two subtrees are completely independent subproblems — any valid left subtree can be paired with any valid right subtree.

For n = 3, there are exactly 5 unique BSTs (the 3rd Catalan number):

1 \ 2 \ 3
Root = 1
1 \ 3 / 2
Root = 1
2 / \ 1 3
Root = 2
3 / 1 \ 2
Root = 3
3 / 2 / 1
Root = 3

Why 5? The count follows the Catalan sequence: C(1)=1, C(2)=2, C(3)=5, C(4)=14, C(5)=42… The formula is C(n) = C(2n, n) / (n+1). This grows exponentially, which is why the output size dominates complexity.

Edge Cases: n=1 returns a single node [TreeNode(1)]. n=0 conventionally returns [None] (empty tree). The recursive base case handles both.

2 The Algorithm: Divide-and-Conquer with Memoization ▾

We define a helper generate(lo, hi) that returns a list of all BST roots whose nodes contain values lo through hi. To solve the full problem, we call generate(1, n).

1 Check memo cache: If (lo, hi) is already cached, return it immediately. Same range always produces same trees — no need to recompute.
2 Base case: If lo > hi, the range is empty — return [None]. This represents "one valid option: an empty subtree." Returning [] would be wrong (see warning box in Section 4).
3 Try each root: For every value r in [lo, hi], recursively compute lefts = generate(lo, r-1) and rights = generate(r+1, hi).
4 Cartesian product: For every pair (L, R) from lefts × rights, create TreeNode(r, left=L, right=R) and add it to results. This ensures every combination is captured.
5 Cache and return: Store memo[(lo, hi)] = results and return. Any future call with the same range reuses this instantly.
📐 The Recurrence
generate(lo, hi) = [None] if lo > hi generate(lo, hi) = union over r in [lo..hi] of: { TreeNode(r, L, R) | L in generate(lo, r-1), R in generate(r+1, hi) }
🔑 Why Range [lo, hi] Fully Characterizes the Subproblem

Because the values 1..n are consecutive integers, any range [lo, hi] uniquely determines which set of values goes in a subtree. Two different calls with the same (lo, hi) produce identical tree lists — with the same concrete node values — making memoization completely valid.

3 Why It Works: The Invariant & DP Argument ▾
🔐 The Invariant

At every level of recursion, generate(lo, hi) returns exactly the complete list of all BST roots such that each tree contains nodes valued lo, lo+1, …, hi and satisfies the BST property at every node. This invariant is maintained by induction: base case returns the only valid empty tree; inductive step combines valid subtrees via Cartesian product.

Why the Cartesian product is complete and correct: Once root r is fixed, the BST property forces the partition: left child subtree must be a BST over {lo..r-1} and right child subtree must be a BST over {r+1..hi}. These two sets are disjoint. Any valid left subtree can be independently combined with any valid right subtree. So every distinct BST with root r is exactly one element of the Cartesian product. Summing over all root choices gives every distinct BST over [lo, hi].

Why overlapping subproblems exist: When generating all BSTs for n=4, generate(1,2) gets called as the left subtree when root=3, AND generate(2,3) gets called as the right subtree when root=1. Without memoization, these computations repeat. The number of distinct (lo, hi) pairs is O(n²), so memoization reduces from exponential calls to O(n²) unique computations.

⚠️ Shared Node References

Because the memo cache stores actual tree node references, the same node objects may appear as subtrees of multiple returned roots. This is safe in LeetCode (trees are not mutated), but in production code that modifies trees after generation, you would need to deep-copy nodes when combining.

4 The Code (Annotated) ▾

Solution 1 — Recursion with Memoization (Interview Pick)

def generateTrees(n: int):
memo = {} ← Cache: (lo, hi) → list of tree roots
def generate(lo, hi):
if (lo, hi) in memo: ← Already solved this range — instant return
return memo[(lo, hi)]
if lo > hi: ← Empty range: one option is "no node here"
return [None] ← [None] not [] — lets leaf nodes form
trees = []
for root_val in range(lo, hi + 1): ← Try every value as root
lefts = generate(lo, root_val - 1)← All BSTs for left subtree values
rights = generate(root_val + 1, hi)← All BSTs for right subtree values
for L in lefts: ← Cartesian product of left × right
for R in rights:
node = TreeNode(root_val, L, R) ← Build one valid BST
trees.append(node)
memo[(lo, hi)] = trees ← Store result for reuse
return trees
return generate(1, n)
⚠️ Critical: Why [None] Instead of []?

If lo > hi returned [], then for L in lefts would iterate zero times — meaning nodes with no left (or no right) child would never be created. Returning [None] says "one valid option: attach nothing here." The outer loops then run once with L=None or R=None, correctly creating leaf nodes and nodes with one child.

Solution 2 — Pure Recursion (No Memoization, conceptually simpler)

def generateTrees(n: int):
def generate(lo, hi):
if lo > hi: ← Same base case
return [None]
trees = []
for r in range(lo, hi + 1): ← Same logic, no cache
for L in generate(lo, r - 1):
for R in generate(r + 1, hi):
trees.append(TreeNode(r, L, R))
return trees ← Recomputes overlapping ranges each time
return generate(1, n)
💡 Complexity Summary

The nth Catalan number C(n) = C(2n,n)/(n+1) governs both output size and work.
n=1: 1 tree · n=2: 2 trees · n=3: 5 trees · n=4: 14 trees · n=5: 42 trees
Time: O(n² · C(n)) — n² distinct (lo,hi) pairs, each doing proportional-to-output work.
Space: O(n · C(n)) for all tree nodes stored in memo. Output size is unavoidable.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize "Return ALL" — Not Counting

The problem says "return all structurally unique BSTs" — not "how many." This signals enumeration, not DP counting. Compare to #96 (Unique BSTs) which only counts and runs in O(n²). Here we must build actual tree nodes, so complexity is bounded by output size.

Step 2: Use the BST Property to Decompose

Once you pick a root, the BST property forces the partition for free. No choice is made about which values go left vs right — the constraint does it. This is the key insight that makes recursion clean: generate(lo, root-1) and generate(root+1, hi) are the only valid subproblems.

Step 3: Identify Overlapping Subproblems → Add Memoization

Notice generate(2, 4) might be needed as the right subtree of root=1, and also as the left subtree of root=5 (for larger n). The range (lo, hi) uniquely characterizes the result. Add memo = {} keyed on (lo, hi) to avoid exponential redundancy.

Step 4: Nail the Base Case Explanation

Proactively explain the [None] vs [] distinction. Most interviewers will probe this. Say: "I return [None] not [] so that the Cartesian product loops still execute — otherwise nodes with empty left or right subtrees would never be created."

🗣️ What to Say to the Interviewer

"This is an enumeration problem, not a counting problem — I need to build all actual trees.
Core idea: for each value r as root, BST property forces left={lo..r-1} and right={r+1..hi}.
So I define generate(lo, hi) = all BSTs with nodes lo through hi.
Base case: lo > hi → return [None], representing an empty subtree slot.
For each root r: lefts = generate(lo, r-1), rights = generate(r+1, hi).
Cartesian product: for each (L, R) pair, create TreeNode(r, L, R).
Add memoization on (lo, hi) to avoid recomputing identical subranges.
Time is O(n² * C(n)) where C(n) is the nth Catalan number — unavoidable given output size."
🔮 Likely Follow-up Questions
  • "Just count them, not generate?" → LeetCode #96, classic DP O(n²): dp[i] = sum of dp[j-1]*dp[i-j] for j in 1..i
  • "Values aren't 1..n?" → Sort the values first, then use sorted indices as lo/hi
  • "What if trees can be modified after generation?" → Deep-copy nodes when building; don't share references from memo
  • "Can you serialize all trees to JSON?" → Yes, DFS each returned root — same O(C(n)) output

7 Summary & Key Takeaways ▾
Choose root r → BST property forces left={lo..r-1}, right={r+1..hi} automatically
Return [None] not [] as the base case — enables the Cartesian product to create leaf nodes
Memoize on (lo, hi) — the pair fully characterizes which set of trees to generate
Output size = Catalan number C(n) ≈ 4ⁿ/n^1.5 — exponential, unavoidable
Pattern: "Return ALL" + structural decomposition → divide-and-conquer + Cartesian product
Distinguish from #96 (count only, pure O(n²) DP) — this requires building actual nodes
ApproachTimeSpaceNotes
Pure Recursion (no memo) O(n² · C(n)) + redundancy O(C(n) · n) Correct but recomputes subranges
Recursion + Memoization ✓ O(n² · C(n)) O(n · C(n)) Optimal; avoids redundant work
🎯 When to Recognize the Generate-Trees Pattern
  • "Return all" or "enumerate all" structures — not just count
  • A structural constraint (BST, balanced, sorted) partitions the problem by root choice
  • Left and right subproblems become independent once a root is fixed
  • Final step is always "combine via Cartesian product"
  • Output count is Catalan-number-like → recursion + memoization is the standard tool
