LeetCode #2290

Minimum Obstacle Removal to Reach Corner

Find the minimum number of obstacles to remove to travel from the top-left to the bottom-right corner of a grid.

Hard Graph 0-1 BFS Time: O(m·n) Space: O(m·n)
1 Understanding the Problem ▾

You're given an m × n binary grid where 0 means an empty cell (free to walk through) and 1 means an obstacle (you must remove it to pass, costing 1 removal). You start at (0, 0) and need to reach (m-1, n-1). You can move up, down, left, or right. What is the minimum number of obstacles to remove?

💡 Key Insight

This is a shortest-path problem with edge weights of either 0 or 1. Moving into an empty cell costs 0; moving into an obstacle cell costs 1 (you remove it). We want the cheapest path from corner to corner — and 0-1 BFS solves this in O(m·n) instead of Dijkstra's O(m·n·log(m·n)).

✗ Naive: Try All Paths

Exponential paths in an m×n grid.
Cannot enumerate them all.
DFS finds A path, not the BEST one.

✓ Optimal: 0-1 BFS

Process cheapest cell first.
0-cost neighbors → front of deque.
1-cost neighbors → back of deque.
O(m·n) — each cell processed once.

Edge cases: If (0,0) itself is an obstacle (value 1), that counts as a removal if you start there — but per constraints, grid[0][0] == 0 and grid[m-1][n-1] == 0 are guaranteed. A 1×1 grid returns 0. A path consisting entirely of 0-cells returns 0 removals.

2 The Algorithm: 0-1 BFS with a Deque ▾

0-1 BFS (also called "Deque BFS" or "Double-Ended Queue BFS") is an elegant extension of standard BFS that handles edge weights of exactly 0 or 1. Instead of a priority queue (like Dijkstra), we use a deque:

  • 0-cost neighbors go to the front of the deque (they're as cheap as the current node)
  • 1-cost neighbors go to the back of the deque (they cost one more)

This maintains the deque as a monotonic cost structure — items at the front always have cost ≤ items at the back, just like Dijkstra's priority queue but with O(1) push operations instead of O(log n).

1 Initialize: Create a dist matrix of size m×n filled with ∞. Set dist[0][0] = 0. Push (cost=0, row=0, col=0) to a deque.
2 Pop from front: Take the element with the smallest known cost from the front. If its stored cost is stale (greater than current dist), skip it — we already found a better path.
3 Explore 4 neighbors: For each valid neighbor (nr, nc), compute new_cost = d + grid[nr][nc]. If new_cost < dist[nr][nc], update it and enqueue: appendleft if grid[nr][nc]==0, append if grid[nr][nc]==1.
4 Done: When the deque empties, dist[m-1][n-1] holds the minimum obstacle removals needed to reach the corner.
🔑 Why Not Regular BFS?

Regular BFS works when all edges have equal cost. Here, walking into an empty cell costs 0 and an obstacle costs 1 — so BFS would wrongly treat them as equivalent. 0-1 BFS maintains a "sorted" deque by exploiting the binary weight structure, giving us Dijkstra-quality results at BFS speed.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At any point during 0-1 BFS, the front of the deque always contains nodes with the minimum cost seen so far. The deque never has more than two distinct cost levels at once: the current level and the current level + 1. This is why we can use a deque instead of a sorted priority queue.

Why pushing 0-cost to front maintains correctness: If moving to cell B from current cell A costs 0, then dist[B] = dist[A]. B is "just as far" as A in terms of cost, so it deserves to be processed before any 1-cost neighbors. Placing B at the front ensures this.

Why pushing 1-cost to back is safe: Moving into an obstacle increases cost by exactly 1. All previously enqueued items have cost ≤ current cost. By appending to the back, we ensure the obstacle cell gets processed after all cheaper or equal-cost cells.

⚠️ The Stale Entry Trick

We may push the same cell multiple times with different costs. When we pop (d, r, c), we check if d > dist[r][c]. If so, we already found a cheaper path and this entry is stale — skip it. This is the same "lazy deletion" trick used in Dijkstra's algorithm.

Proof of optimality (by contradiction): Suppose the algorithm returns cost k but the true minimum is k' < k. Then there's an optimal path with k' removals. Every cell on this path would be processed with the correct (minimum) cost before reaching the destination, because at each step, the front of the deque has the minimum cost. Contradiction.

4 The Code (Annotated) ▾

Solution 1 — 0-1 BFS (Optimal, O(m·n) time)

from collections import deque
def minimumObstacles(grid):
m, n = len(grid), len(grid[0]) ← grid dimensions
dist = [[float('inf')] * n ← dist[r][c] = min obstacles to reach (r,c)
for _ in range(m)]
dist[0][0] = 0 ← start with zero cost at source
dq = deque([(0, 0, 0)]) ← (cost, row, col); source at front
while dq:
d, r, c = dq.popleft() ← always pop from front (cheapest first)
if d > dist[r][c]: ← stale entry: already found cheaper path
continue
for dr, dc in [(0,1),(0,-1),(1,0),(-1,0)]:
nr, nc = r + dr, c + dc ← compute neighbor coordinates
if 0 <= nr < m and 0 <= nc < n:← bounds check
nd = d + grid[nr][nc] ← 0 if empty, 1 if obstacle
if nd < dist[nr][nc]: ← found a cheaper path to neighbor
dist[nr][nc] = nd
if grid[nr][nc] == 0: ← free cell: put at front (same cost level)
dq.appendleft((nd, nr, nc))
else: ← obstacle: put at back (one cost level up)
dq.append((nd, nr, nc))
return dist[m-1][n-1] ← min obstacles to reach bottom-right
⚠️ Why check nd < dist[nr][nc] before enqueuing?

Without this guard, we'd push duplicate entries every time a cell is reached, even if we already have a better cost for it. This would balloon the deque size. The check ensures we only enqueue a cell when we genuinely improve its distance estimate.

Solution 2 — Dijkstra with a Min-Heap (O(m·n·log(m·n)) time)

import heapq
def minimumObstacles(grid):
m, n = len(grid), len(grid[0])
dist = [[float('inf')] * n for _ in range(m)]
dist[0][0] = 0
heap = [(0, 0, 0)] ← min-heap by cost
while heap:
d, r, c = heapq.heappop(heap)
if d > dist[r][c]: continue ← stale check same as 0-1 BFS
for dr, dc in [(0,1),(0,-1),(1,0),(-1,0)]:
nr, nc = r + dr, c + dc
if 0 <= nr < m and 0 <= nc < n:
nd = d + grid[nr][nc]
if nd < dist[nr][nc]:
dist[nr][nc] = nd
heapq.heappush(heap, (nd, nr, nc))
return dist[m-1][n-1]
💡 Complexity Summary

0-1 BFS: Time O(m·n), Space O(m·n) — optimal! Use this in interviews.
Dijkstra / Min-Heap: Time O(m·n·log(m·n)), Space O(m·n) — works, but strictly slower.
DP (no backtracking): Does not work — path can go in any direction including up/left, so simple DP cannot capture all optimal paths.

5 🔬 Algorithm Deep-Dive: 0-1 BFS ▾
🔬 0-1 BFS — The Algorithm

Origin: A specialization of Dijkstra's algorithm for graphs where edges have weights of only 0 or 1. The key observation is that when only binary weights exist, a sorted priority queue is overkill — a deque suffices.

Core Invariant: The deque holds nodes in non-decreasing cost order. At any moment, all nodes in the deque have cost either current_min or current_min + 1. This is guaranteed because we only ever add cost 0 (front) or cost 1 (back).

Why it beats Dijkstra here: Dijkstra uses a min-heap with O(log n) push/pop per operation. With binary weights, we can maintain the cost ordering with O(1) deque operations — giving us the same guaranteed-minimum-first processing at lower cost.

Generalization: If costs were 0 and k (not 1), you'd use k queues (dial's algorithm). If costs are arbitrary non-negative integers, use Dijkstra. 0-1 BFS sits in the sweet spot for binary-weighted graphs.

When to recognize it: Problem involves a grid/graph with exactly two edge weights (0 and 1) AND you want the shortest/cheapest path. Classic signals: "minimum removals", "minimum flips", "minimum cost path" in a binary grid.

6 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize It's a Shortest-Path Problem

The signal: "minimum cost to travel from source to destination through a grid." This screams shortest path. But which algorithm? The grid is binary — cells are 0 (free) or 1 (cost to remove). Edge weights are 0 or 1 — not arbitrary.

Step 2: Match to the Right Tool

Equal weights → BFS. Non-negative arbitrary weights → Dijkstra. Binary weights (0 or 1) → 0-1 BFS. This is the specific tool designed for this exact structure. If you don't remember 0-1 BFS, Dijkstra works and is acceptable — mention that 0-1 BFS would be optimal.

Step 3: Set Up the State Space

State = cell position (r, c). Cost to reach = number of obstacles removed. We need a dist matrix initialized to infinity, and a deque seeded with the start (0, 0, 0).

Step 4: The Key Deque Logic

When exploring a neighbor: if it's an empty cell (cost 0), push to front (appendleft) — it's at the same cost level. If it's an obstacle (cost 1), push to back (append) — it costs one more. Always update dist only when you found a cheaper path.

🗣️ What to Say to the Interviewer

"This is a shortest-path problem on a grid with binary edge weights — 0 for empty cells and 1 for obstacles. The optimal tool here is 0-1 BFS, which uses a deque instead of a priority queue.

The key idea: when we move into a free cell, the cost stays the same, so we push it to the front. When we move into an obstacle, the cost increases by 1, so we push it to the back. This maintains the deque in sorted order with O(1) operations, giving us O(m·n) time versus Dijkstra's O(m·n·log(m·n)).

We also maintain a dist matrix to track the minimum cost to each cell, and we skip stale deque entries using the standard 'if d > dist[r][c]: continue' guard."
💡 Follow-up Questions to Expect
  • "What if obstacles had cost 2 instead of 1?" → Use Dijkstra (costs > 1 break the deque invariant)
  • "What if you could teleport between certain cells for free?" → BFS with 0-cost teleport edges, still 0-1 BFS
  • "What if the grid is huge but sparse?" → Same algorithm; complexity stays O(m·n)
  • "Can you do it with less space?" → The dist matrix is necessary; O(m·n) space is optimal

8 Summary & Key Takeaways ▾
0-1 BFS handles binary-weighted graphs in O(m·n) — better than Dijkstra's O(m·n·log(m·n))
Key trick: 0-cost neighbors go to the front (appendleft), 1-cost to the back (append)
The deque naturally stays in non-decreasing cost order — same guarantee as a priority queue, O(1) operations
Always use a stale-entry check: if d > dist[r][c], skip — don't reprocess cells we've already improved
Signal phrases: "minimum removals", "minimum flips", "binary cost", "shortest path in binary grid" → think 0-1 BFS
🎯 When to Use 0-1 BFS
  • Graph/grid with edge weights of exactly 0 or 1
  • "Minimum cost path" where cost is binary (pass-through vs. pay-to-enter)
  • Need O(m·n) time — can't afford the log factor of Dijkstra
  • Problem involves choosing whether to use a resource at 0 or 1 additional cost
