LeetCode #547

Number of Provinces

Given an n×n adjacency matrix isConnected where cities are directly or indirectly connected, return the number of provinces (connected components).

Medium Graph Union Find Time: O(n²·α(n)) Space: O(n)
1 Understanding the Problem ▾

You have n cities. The input is an n×n adjacency matrix: isConnected[i][j] = 1 means city i and city j share a direct road. A province is a group of cities where every pair is reachable from each other — directly or through a chain of roads. Your task: count the total number of provinces.

💡 Key Insight

A province is exactly a connected component in graph theory. The question reduces to: "How many connected components exist in this undirected graph?" Two classic tools solve this: Union-Find (merge pairs as you scan the matrix) or DFS/BFS (flood-fill from each unvisited city).

✓ Example: 2 Provinces

Cities 0, 1, 2, 3
0↔1 (direct road)
0↔2 (direct road)
→ Province A = {0, 1, 2}
City 3: no connections
→ Province B = {3}
Answer: 2

✗ Transitivity Trap

If 0↔1 and 1↔2
but NO direct 0↔2 edge,
0 and 2 are STILL in the
same province — connected
through city 1!
Order matters!

Edge cases: n=1 → always 1 province. All cities connected → 1 province. Zero connections → n provinces. The diagonal is always 1 (city connected to itself); ignore it. Matrix is symmetric (undirected graph).

2 The Algorithm: Union-Find (Disjoint Set Union) ▾

We use a parent array where every city starts as its own root. When we discover two connected cities, we union their sets. After scanning all edges, count distinct roots = count provinces.

1 Initialize: parent[i] = i for all cities. count = n. Each city is an independent province.
2 Scan upper triangle: For all pairs (i, j) where j > i, check if isConnected[i][j] == 1.
3 Find roots: Call path-compressed find(i) and find(j) to get their component representatives.
4 Union if different: If find(i) != find(j), set parent[find(j)] = find(i) and do count -= 1.
5 Return count: After all pairs, count is exactly the number of provinces.
find — Path Compression
Chase parent pointers to reach the root. On the way back, point each node directly to the root — flattening the tree for future calls.
def find(x): if parent[x] != x: parent[x] = find(parent[x]) return parent[x]
union — Merge Components
Get both roots. If different, attach one tree under the other and reduce the province count by 1. If same root, cities already share a province — skip.
ri, rj = find(i), find(j) if ri != rj: parent[rj] = ri count -= 1
3 Why It Works: The Invariant ▾
🔐 The Invariant

At all times, count equals the number of distinct connected components. Every union call either merges two separate components (count decreases by 1) or does nothing (roots already equal — no double-count). find only restructures pointers; it never changes which component a node belongs to.

Proof sketch: Initially n cities, n provinces. Each edge isConnected[i][j] = 1 says i and j should be in the same province. If they already are (same root), no change needed. If not, merging is correct — exactly 1 fewer province. After all O(n²) pairs are processed, every transitively-connected city shares a root. The final count is exact.

Path compression safety: Pointing a node directly to its root never changes which root it belongs to — just how many hops it takes to get there. Component identity is preserved.

⚠️ Only Scan the Upper Triangle

The matrix is symmetric: isConnected[i][j] == isConnected[j][i]. Scanning all cells processes each edge twice. Use j > i to avoid this. Union-Find won't double-decrement (the root check prevents it), but scanning the full matrix wastes half the iterations. Upper triangle is cleaner and faster.

⚠️ Must Check ri != rj Before Decrementing

If cities 0 and 2 were already joined via city 1, their roots are the same. Decrementing count without checking would give a wrong answer — you'd subtract 1 for a merge that already happened. Always guard with the root equality check.

5 🔬 Algorithm Deep-Dive: Union-Find (DSU) ▾

Origin: Union-Find (also called Disjoint Set Union / DSU) was introduced by Galler and Fischer in 1964, with near-linear time analysis by Tarjan in 1975. It efficiently answers: "Are these two elements in the same group?" and "Merge these two groups."

🔐 Core Invariant

Every node has a parent pointer. Following parent pointers eventually reaches the root, which is the representative of the component. Two nodes are in the same component if and only if they share a root: find(a) == find(b).

Two optimizations give near-O(1) amortized per operation:

1 Path Compression: While finding the root of x, set every ancestor of x to point directly to the root. This halves the tree height on each call, so future lookups are almost instant.
2 Union by Rank: Always attach the smaller (lower-rank) tree under the taller one. This prevents chains from forming. Without this, a sequence of bad unions can degrade to O(n) per operation.

Ackermann analysis: With both optimizations, any sequence of m operations on n elements runs in O(m·α(n)), where α grows so slowly it never exceeds 4 for any practical input size. Effectively O(1) per operation.

💡 When to Recognize Union-Find

Signal phrases: "connected components", "number of groups/islands/clusters", "are A and B in the same group?", "edges added dynamically", "cycle detection in undirected graph". Key advantage over DFS: handles dynamic edge insertion without a full re-scan.

6 🎤 Interview Strategy ▾
Step 1: Reframe as Connected Components

The word "province" is a disguise. The moment you see "group of connected X" with a matrix or edge list, think: connected components problem. The graph has n nodes and edges where isConnected[i][j] == 1.

Step 2: Propose Both Approaches Briefly

Mention DFS (flood-fill, intuitive) and Union-Find (more scalable, handles dynamic updates). Then pick Union-Find to show breadth. Explain path compression and why you scan only the upper triangle.

Step 3: Code find() Carefully

Interviewers check: path compression in find(), root equality guard before decrementing count, correct parent update direction (parent[rj] = ri, not the other way by mistake). Practice writing this from memory.

Step 4: State Complexity Confidently

Time: O(n²) dominated by the matrix scan. Space: O(n) for the parent array. If they ask about Union-Find overhead, mention O(n²·α(n)) ≈ O(n²) — the matrix scan is the bottleneck.

🗣️ What to Say to the Interviewer

"This is a connected-components problem — each province is one component. I'll use Union-Find with path compression: initialize parent[i]=i for each city, then scan the upper triangle of the matrix. When isConnected[i][j]=1, I union i and j: find their roots, and if different, merge and decrement count. Time is O(n²) for the scan, space is O(n) for the parent array. I could also do DFS flood-fill from each unvisited city — also O(n²). Both are correct; Union-Find scales better for dynamic edge insertion."
⚠️ Common Follow-up Questions

Q: What if edges are added one at a time (online)?
Union-Find handles this perfectly — just call union(i, j) as each edge arrives. DFS would need a full re-scan of the adjacency matrix each time, making it O(n²) per edge addition versus O(α(n)) with Union-Find.

Q: How would you detect if adding an edge creates a cycle?
Before doing union(i, j), check if find(i) == find(j). If they share a root, the edge would create a cycle — it's already there. This is the cycle detection use case for Union-Find (e.g., LeetCode 684 Redundant Connection).

Number of Provinces = Number of connected components in the graph encoded by isConnected
Union-Find: scan O(n²) matrix, merge components in O(α(n)) each — total O(n²)
Path compression makes repeated find() calls near-instant by flattening tree structure
