LeetCode #1489

Find Critical and Pseudo-Critical Edges in MST

Given a weighted undirected connected graph, find all critical edges and pseudo-critical edges in its minimum spanning tree (MST).

Hard Graph Union-Find MST Time: O(E² · α(V)) Space: O(V)
1 Understanding the Problem ▾

You are given a weighted undirected graph with n nodes and a list of edges (each with a weight). The graph has a Minimum Spanning Tree (MST) — a subset of edges that connects all nodes with the minimum total weight. There may be multiple valid MSTs if some edges share the same weight.

💡 Key Insight

An edge is critical if removing it makes the MST impossible or forces a higher cost MST. An edge is pseudo-critical if it appears in at least one valid MST, but not all of them — i.e., it can be forced in without raising the cost, but isn't strictly required.

🔴 Critical Edge

Removing this edge increases the MST cost (or disconnects the graph).

Test: Build MST without this edge.
If MST weight > original → CRITICAL.

🟡 Pseudo-Critical Edge

Forcing this edge in still gives a valid MST (same cost).

Test: Build MST with this edge forced.
If MST weight = original → PSEUDO-CRITICAL.

Example walkthrough: Consider 4 nodes, edges: (0,1,w=1), (1,2,w=1), (2,3,w=2), (0,3,w=2), (0,2,w=3). The MST has total weight 4. Edge (0,1,1) is critical — removing it forces cost > 4. Edge (1,2,1) is also critical. Edges (0,3,2) and (2,3,2) are pseudo-critical — either can complete the MST at cost 4, but neither is uniquely required.

Key challenge: The MST may not be unique. We must test each edge's role across all possible MSTs, not just one of them.

Edge Cases: Parallel edges (same endpoints, different weight), graphs with a unique MST (all edges have distinct weights → every MST edge is critical), minimum graph (3 nodes, 3 edges = triangle).

2 The Algorithm: Include / Exclude Testing + Kruskal's MST ▾

The core technique is "test each edge in two ways" against a baseline MST weight. We use Kruskal's algorithm (Union-Find) to efficiently build the MST. Here is the full plan:

1 Sort edges by weight — Kruskal's algorithm always considers edges in non-decreasing weight order. This greedy strategy works because adding the cheapest available edge that doesn't create a cycle always progresses toward a valid MST.
2 Compute base MST weight — Run Kruskal's on all edges (no exclusions). Record the total weight base_mst. This is our reference. If a later MST has higher cost, we know something is wrong.
3 For each edge e, test CRITICAL: Run Kruskal's on all edges except e. If the resulting MST weight is greater than base_mst (or the graph becomes disconnected), then e is critical — no other edge can compensate.
4 For each non-critical edge e, test PSEUDO-CRITICAL: Force edge e into the MST first (preconnect its two endpoints), then complete the MST greedily with the remaining edges. If the total cost equals base_mst, then e participates in at least one valid MST — it is pseudo-critical.
5 Collect results: Return two lists — all critical edge indices and all pseudo-critical edge indices (from the original edge list numbering, not sorted order). Keep original indices alongside edges while sorting.
🔑 Kruskal's MST Algorithm (The Engine)

Sort all edges by weight. Iterate through them. For each edge (u, v, w): if u and v are in different components (find their roots — if different), union them and add the edge to MST. This greedy approach always yields a minimum spanning tree. Uses Union-Find for near-O(1) component queries.

3 Why It Works: The Invariant + Union-Find Deep Dive ▾
🔐 The Invariant: MST Cost Comparison

For any edge e:
• Critical if: MST_without(e).cost > base_mst OR graph disconnected without e
• Pseudo-critical if: NOT critical AND MST_with_e_forced.cost == base_mst
• Neither if: MST_with_e_forced.cost > base_mst (e is never in any MST)

Why does excluding prove criticality? If the MST without edge e costs more, it means e was the only cheapest path connecting two components at that point in Kruskal's. Any replacement path costs more — so e is irreplaceable.

Why does forcing prove pseudo-criticality? By pre-adding edge e, we guarantee it's in the spanning tree. If Kruskal's can still reach base_mst cost from there, it means e is consistent with at least one optimal MST. If the cost were higher, e would contradict the optimal — meaning no valid MST includes it.

🔐 Union-Find Data Structure

Origin: Tarjan (1975). Tracks which nodes belong to the same connected component.
Core invariant: Each component has a root; find(x) returns the root with path compression. union(a, b) merges two components by rank.
Why it works: Path compression flattens the tree — after a find, all nodes on the path point directly to root. Union by rank prevents tall trees. Together: nearly O(1) amortized per operation (actually O(α(n)) — inverse Ackermann, effectively constant).
Recognize when: "Connected components", "detect cycle in undirected graph", "offline MST (Kruskal's)", "dynamic connectivity".

Edge case: ties in weight. When edges have equal weight, Kruskal's may choose different edges, resulting in multiple valid MSTs. That's exactly why the exclude/include testing framework works — it checks membership across all possible MSTs structurally, not just one arbitrary MST.

⚠️ Why Index Preservation Matters

After sorting edges by weight, the original edge indices shift. Always carry the original index with each edge as a tuple (w, u, v, orig_idx). The problem asks for original indices in the output, not sorted positions.

4 The Code (Annotated) ▾

Solution — Include/Exclude Edge Testing with Kruskal's + Union-Find (Interview Pick)

def findCriticalAndPseudoCriticalEdges(n, edges):
# Step 1: attach original index to each edge before sorting
indexed = [(w,u,v,i) for i,(u,v,w) in enumerate(edges)] ← preserve original index i
indexed.sort() ← sort by weight (Kruskal's order)
E = len(indexed)
def make_uf(n): ← fresh Union-Find for n nodes
parent = list(range(n)) ← each node is its own root initially
rank = [0] * n
return parent, rank
def find(p, x): ← find root with path compression
while p[x] != x: p[x] = p[p[x]]; x = p[x] ← two-step compression (halves path)
return x
def union(p, r, a, b): ← union by rank, returns True if merged
ra, rb = find(p,a), find(p,b)
if ra == rb: return False ← already connected → adding would create cycle
if r[ra] < r[rb]: ra, rb = rb, ra ← merge smaller rank under larger
p[rb] = ra ← rb's root now points to ra
if r[ra] == r[rb]: r[ra] += 1
return True
def kruskal(skip=-1, force=-1): ← skip or force an edge index
p, r = make_uf(n)
cost = 0; used = 0
if force != -1: ← forced edge: union endpoints first
w,u,v,_ = indexed[force]
union(p, r, u, v); cost += w; used += 1
for i,(w,u,v,_) in enumerate(indexed): ← greedy edge selection
if i == skip or i == force: continue ← respect skip/force constraints
if union(p, r, u, v): cost += w; used += 1 ← add edge if it merges two components
return cost if used == n-1 else float('inf') ← inf if graph disconnected
base = kruskal() ← Step 2: compute reference MST weight
critical, pseudo = [], []
for i, (_,_,_,orig) in enumerate(indexed): ← test each edge
if kruskal(skip=i) > base: ← Step 3: exclude test (critical?)
critical.append(orig)
elif kruskal(force=i) == base: ← Step 4: include test (pseudo-critical?)
pseudo.append(orig)
return [critical, pseudo]
⚠️ Critical Order: Test Critical BEFORE Pseudo-Critical

An edge that fails the skip test (increases cost → critical) should NOT be tested for pseudo-criticality. The elif ensures this: we only run the force test when the edge is NOT critical. If you ran both tests independently, you could mislabel a critical edge as also pseudo-critical.

💡 Complexity Summary

Time: O(E log E) to sort + O(E²·α(V)) for E rounds of Kruskal's where each Kruskal's run is O(E·α(V)). Since E ≥ V−1 in a connected graph, this is O(E²·α(V)) overall. For the problem constraints (n ≤ 100, E ≤ 200), this is fast enough.
Space: O(V) for the Union-Find per Kruskal's run.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Understand the Definitions Precisely

Say: "A critical edge must appear in every valid MST. A pseudo-critical edge appears in some but not all MSTs. This tells me I need to test edge membership across the space of all possible MSTs — not just one."

Step 2: Recognize This as an MST Classification Problem

The signals: weighted graph, minimum cost spanning tree, edge classification. Immediately think Kruskal's or Prim's. Kruskal's is better here because we can easily skip/force individual edges during the sort-and-union process.

Step 3: Propose the Include/Exclude Framework

"For each edge e: (a) Try building MST without e — if cost increases, it's critical. (b) Try forcing e first — if total cost equals base MST, it's pseudo-critical. Both tests use Kruskal's, so I run O(E) Kruskal's calls total."

Step 4: Implement Union-Find with Path Compression + Union by Rank

Show you know the standard implementation. Mention that you need a fresh Union-Find for each Kruskal's call — you can't reuse state between tests. The factory function make_uf(n) handles this cleanly.

🗣️ What to Say to the Interviewer

"I'll compute a baseline MST using Kruskal's — sort edges by weight, then greedily pick edges that connect different components via Union-Find.
For each edge, I'll run two tests:
1. Skip test: exclude this edge and recompute MST. If cost rises → critical.
2. Force test: pre-include this edge, then complete MST. If cost equals baseline → pseudo-critical.
I need to track original indices since I sort the edge list.
Total: O(E²·α(V)) time, O(V) space — acceptable for the given constraints."
🔮 Likely Follow-up Questions
  • "Can you use Prim's instead of Kruskal's?" → Yes, but skipping/forcing edges is harder with Prim's. Kruskal's is more natural here.
  • "What if we need to handle edge deletions dynamically?" → Would need offline algorithms or link-cut trees for O(log n) per update.
  • "All edges unique weights — what changes?" → Every MST edge becomes critical, no pseudo-critical edges. MST is unique with distinct weights.
  • "Can you do better than O(E²)?" → Yes: advanced matroid intersection or sensitivity analysis of MSTs can do O(E·α(V)+E log E), but this is beyond typical interviews.

7 Summary & Key Takeaways ▾
Sort edges by weight first — Kruskal's algorithm depends on processing cheapest edges first
Always preserve original indices when sorting — the problem asks for them in the output
Critical test: MST cost rises when edge is excluded → edge is irreplaceable
Pseudo-critical test: MST cost unchanged when edge is forced → edge fits at least one MST
Test critical BEFORE pseudo — use elif so critical edges are never re-tested
Create a fresh Union-Find for each Kruskal's call — state does not carry over between tests
Path compression + union by rank → near O(1) per operation (O(α(n)) amortized)
