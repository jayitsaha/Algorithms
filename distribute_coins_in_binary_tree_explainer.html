LeetCode #979

Distribute Coins in Binary Tree

Find the minimum number of moves to give every node exactly one coin. One move = carrying one coin across one edge.

Medium Trees Post-order DFS Time: O(n) Space: O(h)
1 Understanding the Problem ▾

You are given a binary tree. Each node holds some number of gold coins. The total number of coins equals the number of nodes — so there are exactly enough coins to give one to each node. Your goal: find the minimum number of moves to redistribute so every node ends up with exactly 1 coin. One "move" means carrying one coin across one edge (parent to child, or child to parent — both cost 1 move).

💡 Key Insight: Think in Terms of Edge Flow

Instead of simulating individual coin movements, think about each edge as a pipe. For each edge, some number of coins must flow through it (either up or down). The total number of coins crossing an edge = the moves that edge contributes. We just need to figure out how many coins cross each edge — which is simply |excess coins in the subtree below that edge|.

Example: Tree [3, 0, 0]

root=3, left=0, right=0
Left subtree: 0 coins, 1 node
→ edge carries 1 coin downward
Right subtree: 0 coins, 1 node
→ edge carries 1 coin downward
Answer: 2 moves

The "Excess" Formula

excess(subtree) =
  coins_in_subtree
  − nodes_in_subtree

If positive → subtree has surplus, sends coins up
If negative → subtree has deficit, receives coins
moves for edge = |excess|

Edge cases to consider: A single node with val=1 → 0 moves. A leaf with val=0 → adds 1 move to its parent edge. A leaf with val=5 → sends 4 coins upward, 4 moves on that edge. Tree of all zeros except one node with n coins → that node must distribute all others.

2 The Algorithm: Post-order Excess Flow DFS ▾

We traverse the tree in post-order (left → right → parent). At each node, we compute how many excess coins its subtree can supply to the parent. The absolute value of this number is added to our running move count — those are the coins that must physically cross the edge to the parent.

1 Base case (null node): Return 0. A null child contributes no coins and no demand — it's neutral.
2 Recurse left, then right (post-order): Call dfs(node.left) and dfs(node.right). Each call returns the net excess of that subtree — how many coins it can spare (positive) or needs (negative).
3 Count the edge moves: moves += abs(left_excess) + abs(right_excess). Each child's excess represents coins flowing through the respective edge — whether they flow up or down is irrelevant, only the quantity matters.
4 Return this node's own excess: return node.val - 1 + left_excess + right_excess. We subtract 1 to "reserve" one coin for this node. The children's excess is added because surplus or deficit flows up to the parent.
🌳 Why Post-order? Why not Pre-order?

Post-order processes children before parents. We need to know what the left and right subtrees need or supply before we can compute a node's total excess. Pre-order would process the parent before its children, so we wouldn't have that information yet. Post-order = bottom-up = children answer first.

Analogy: Think of each subtree as a factory. A factory with more raw material than it needs ships the surplus up the supply chain. A factory that's short requisitions material from above. The distribution center (parent) counts all shipments and requisitions — those are the moves.

3 Why It Works: The Invariant ▾
🔐 The Invariant

The return value of dfs(node) always equals (total coins in this subtree) − (number of nodes in this subtree). This is the signed count of coins that must cross the edge from node to its parent. Taking the absolute value gives the number of edge crossings (moves) that edge contributes.

Proof by induction:

Base case: dfs(null) = 0. A null subtree has 0 coins and 0 nodes. 0 − 0 = 0. Correct.

Inductive step: Assume dfs(L) = (L.coins − L.nodes) and dfs(R) = (R.coins − R.nodes). Then:

dfs(node) = node.val − 1 + dfs(L) + dfs(R)
= node.val − 1 + (L.coins−L.nodes) + (R.coins−R.nodes)
= (node.val + L.coins + R.coins) − (1 + L.nodes + R.nodes)
= total_subtree_coins − total_subtree_nodes

This is exactly the net flow that must cross the edge above this node. QED.

⚠️ Common Misconception: Direction of Flow

You might think coins flowing down (parent to child) cost differently than coins flowing up (child to parent). They don't — each is 1 move per coin per edge. That's why we take |excess|, not just excess. Positive excess means coins flow up; negative means they flow down. Either way, |excess| moves happen on that edge.

Why is this minimum? Every coin that needs to cross an edge must cross it exactly once — there is no shortcut in a tree. In a graph, a coin might take a detour, but trees have exactly one path between any two nodes. So the flow computed is unavoidable — it IS the minimum.

4 The Code (Annotated) ▾

Optimal solution — one post-order DFS, O(n) time, O(h) space.

def distributeCoins(root): entry point; root = TreeNode
moves = 0 global counter — total edge crossings
def dfs(node): inner helper; returns excess of subtree
nonlocal moves capture outer var so we can mutate it
if not node: base case: null child
return 0 no coins, no demand — neutral
left = dfs(node.left) post-order: recurse LEFT subtree first
right = dfs(node.right) then recurse RIGHT subtree
moves += abs(left) + abs(right) |left edge flow| + |right edge flow|
return node.val - 1 + left + right excess: own coins minus 1 for self, plus children's net
dfs(root) trigger traversal; root has no parent edge, discard return
return moves total minimum moves
⚠️ Why discard the root's return value?

The root has no parent edge. Its excess (positive or negative) doesn't contribute any moves — there's no edge above it to carry coins through. The root's leftover excess just sits at the root. We only add to moves when a child returns a value to its parent (which has an edge).

🔑 Complexity Analysis

Time: O(n) — each node is visited exactly once.
Space: O(h) — the call stack holds at most h frames, where h is tree height. O(log n) for balanced trees, O(n) worst case (skewed).

Alternative: BFS won't work cleanly here — BFS processes nodes level by level (top-down), but we need bottom-up aggregation. We'd need to store results for all children before processing a parent, which effectively simulates post-order anyway and adds complexity. The recursive DFS is cleanest.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify the Structure

It's a binary tree problem. The answer involves counting something across edges. When you see "aggregate over tree edges" or "minimum cost across a tree," think DFS. Because we need children's info before parents, think post-order.

Step 2: Reframe as Edge Flow

The key insight is switching from "where does each coin go?" to "how many coins cross each edge?" In a tree, there's exactly one path between any two nodes, so each coin crossing an edge is unavoidable and contributes exactly 1 move. Count flows, not paths.

Step 3: Define the Recursive Function

What should the recursive function return? The net excess of a subtree: (coins in subtree) − (nodes in subtree). Positive = sends coins up to parent. Negative = needs coins from parent. Either way, |excess| moves happen on the edge to parent.

Step 4: Implement and Verify

Write the recurrence: excess = node.val - 1 + dfs(left) + dfs(right). Add abs(left) + abs(right) to moves each call. Verify with examples: [3,0,0] → moves=2. [0,3,0] → left returns 2, moves+=2, root returns 0+2−1=1, moves+=1 → total 3.

🗣️ What to Say to the Interviewer

"I notice this is a tree redistribution problem. Rather than simulating which coins move where, I'll think about each edge: how many coins must physically cross it? In a tree, there's only one path between any two nodes, so coin flow is deterministic. If I define the excess of a subtree as (coins in subtree) minus (nodes in subtree), that's the net number of coins that must cross the edge above it — positive means surplus sent upward, negative means deficit filled from above. So I'll use post-order DFS: process children first, accumulate |left_excess| + |right_excess| into a global moves counter, then return this node's own excess upward. Time O(n), space O(h) for the recursion stack."
🎯 Follow-up Questions to Expect
  • Can you do it iteratively? — Yes, using an explicit stack for post-order simulation.
  • What if coins can only move in one direction? — Much harder; track directions explicitly, no longer O(n).
  • What if the tree is a general graph? — Still O(n) but need to root it and handle cycles.
  • How would you test edge cases? — Single node val=1 (0 moves), all zeros at leaves except root, chain-shaped tree.

7 Summary & Key Takeaways ▾
Use post-order DFS whenever a tree node's answer depends on aggregating results from its children (bottom-up).
Model the problem as edge flow: each edge carries |excess| coins, where excess = (subtree coins) − (subtree nodes).
The recursive helper returns a signed excess. Add |left| + |right| to moves at every node — those are edge crossings.
Discard the root's return value — the root has no parent edge, so its excess costs no moves.
Time O(n) — one DFS visit per node. Space O(h) — call stack depth equals tree height.
