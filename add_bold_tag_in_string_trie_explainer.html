1 What is a Trie? (From Zero) ▼

A Trie (pronounced "try", from retrieval) is a tree data structure for storing strings. Unlike a hash set where you store whole words, a Trie stores strings character by character — each edge represents one character, and paths from the root spell out words.

🔑 The Core Idea

Imagine a dictionary organized as a tree. The root is empty. Each branch adds one letter. When you reach a node marked as "end of word", the path from root to that node spells a complete word. Words sharing a common prefix share the same path up to where they diverge.

Real-world analogy: Think of your phone's autocomplete. As you type "c-a-t", the system walks down a tree: root → c → a → t. At "t" it knows "cat" is a word. If you type "c-a-r", it branches at "a" → goes to "r" instead of "t". The prefix "ca" was shared, saving space.

Why not just use a list or set?

  • List: To check if "cat" is a prefix of anything, you'd scan every word → O(W) per check
  • Set: Fast lookup for exact words, but can't do "does any word START with 'ca'?" efficiently
  • Trie: Walk the tree character by character. At any node you know: "Is this a prefix? Is this a full word?" — O(L) per query where L is word length
💡 When to Use a Trie
  • Searching for multiple patterns in a string simultaneously
  • Prefix matching — "find all words starting with X"
  • Autocomplete / spell-check systems
  • When you'd otherwise call str.startswith() for many words → Trie does them all at once

⚠️ Common Misconception

Node labels don't store the full word — each node just represents the last character added. The path from root spells the word. The green "a" at the bottom-left isn't "a", it's the end of root → a → a → a = "aaa".

2 How Trie Operations Work ▼

A Trie node has exactly two things:

  1. children — a dictionary mapping characters to child TrieNodes
  2. is_end — a boolean marking "a complete word ends at this node"
class TrieNode:
def __init__(self):
self.children = {} # char → TrieNode
self.is_end = False # does a word end here?

Insert a word: Walk from root. For each character, if the child doesn't exist, create it. At the last character, mark is_end = True.

def insert(root, word):
node = root # Start at root
for ch in word: # Walk character by character
if ch not in node.children:
node.children[ch] = TrieNode() # Create if missing
node = node.children[ch] # Move down
node.is_end = True # Mark end of word

Search / Walk: Follow the same path. If at any point a character isn't in children, the word (or prefix) doesn't exist. If you reach the end and is_end is True, the whole word exists.

def search(root, word):
node = root
for ch in word:
if ch not in node.children:
return False # Path doesn't exist
node = node.children[ch]
return node.is_end # True only if full word

🔑 Key Advantage for This Problem

For "Add Bold Tag", we need to check if any word from the dictionary starts at position i in the string. Without a Trie, we'd try every word separately → O(W·L) per position. With a Trie, we walk once from position i, and the Trie simultaneously checks all words that could match → O(L) per position. If 3 words share the prefix "aa", we only walk "aa" once, not three times.

3 The Algorithm: Trie + Boolean Array ▼

We combine a Trie (for fast multi-word matching) with a boolean array (for merging intervals):

1 Build the Trie: Insert all dictionary words into a Trie.
2 Scan string with Trie: For each position i in s, walk the Trie from root following s[i], s[i+1], .... Whenever we hit a node with is_end = True, mark all characters in that range as bold.
3 Build result: Scan the bold[] array. Insert <b> at the start of each bold region and </b> at the end. Overlapping/adjacent words are automatically merged!
💡 Why This Beats Brute Force

Without Trie (Solution 1): For each word, call s.find(word) repeatedly → O(n·W·L) total.
With Trie (Solution 2): For each position in s, do one Trie walk → O(n·L) total.
When W (word count) is large, Trie is dramatically faster because shared prefixes are walked only once.

4 Why It Works: The Invariant ▼
🔐 The Invariant

After scanning position i, bold[k] = True for every index k that is covered by at least one dictionary word starting at or before position i. The Trie guarantees we find every dictionary word starting at i in a single walk.

Trie walk correctness: Starting at position i, we follow s[i], s[i+1], ... in the Trie. Every time we reach an is_end node, we've found a word that starts at i. If we hit a dead end (character not in children), no longer words can match from i.

Boolean array merging: If "aaa" marks indices [0,1,2] and "aab" marks [1,2,3], both set bold[1] = bold[2] = True. The overlapping region is automatically merged — we just check where bold regions start and end.

⚠️ The Trie Walk Doesn't Stop at First Match

When we find a word at is_end, we don't stop — we keep walking! "aaa" contains "aa" as a prefix. If both "aa" and "aab" are in the dictionary, one walk from position 0 finds "aa" at depth 2 then keeps going to find "aab" at depth 3. This is critical for correctness.

5 The Code (Annotated) ▼
class TrieNode: # Each node in the Trie
def __init__(self):
self.children = {} # char → TrieNode mapping
self.is_end = False # True if a word ends here
def addBoldTag(s, words): # Main function
root = TrieNode() # Create empty Trie root
for word in words: # Phase 1: Build Trie
node = root
for ch in word: # Walk/create path
if ch not in node.children:
node.children[ch] = TrieNode()
node = node.children[ch]
node.is_end = True # Mark word boundary
n = len(s)
bold = [False] * n # Phase 2: Boolean mask
for i in range(n): # Try each start position
node = root # Reset to Trie root
for j in range(i, n): # Walk forward in s
if s[j] not in node.children: # Dead end in Trie
break
node = node.children[s[j]] # Move to next Trie node
if node.is_end: # Found a complete word!
for k in range(i, j + 1): # Mark range bold
bold[k] = True
res = [] # Phase 3: Build output
for i in range(n):
if bold[i] and (i == 0 or not bold[i-1]):
res.append("<b>") # Start of bold region
res.append(s[i])
if bold[i] and (i == n-1 or not bold[i+1]):
res.append("</b>") # End of bold region
return "".join(res)
⚠️ Why We Don't Break After Finding a Word

After if node.is_end: mark bold, we continue the inner loop. If dictionary has "aa" and "aab", starting at position 1 in "aaabbcc" we find "aa" (marks [1,2]) then keep walking to find nothing at 'b' (Trie doesn't have "aab..." from 'b'). But if the example were different and "aab" branched further, we'd catch that too.

6 🎤 Interview Strategy ▼
Step 1: Recognize the Pattern

"Multiple dictionary words to search in a string" → This screams Trie. Any time you see a dictionary of patterns to match simultaneously, think Trie or Aho-Corasick.

Step 2: Start with the Naive Solution

First mention: "I could search each word with .find() and mark ranges." Complexity is O(n·W·L). Then say: "I can optimize the matching phase using a Trie."

Step 3: Explain the Trie Advantage

"By building a Trie from dictionary words, I walk the string once from each position. Shared prefixes are explored only once, reducing matching to O(n·L)."

Step 4: Explain Bold Merging

"I use a boolean array — each index is marked if any word covers it. Overlapping/adjacent bold regions merge automatically when I scan for transitions."

🗣️ What to Say to the Interviewer

"I'll build a Trie from the dictionary words. Then for each starting position in s, I walk the Trie following characters in s. Every time I reach a node marked as end-of-word, I mark those indices bold. After processing all positions, I scan the bold array and insert tags at transitions. This runs in O(n·L) for matching plus O(W·L) to build the Trie, much better than checking each word individually."

8 Trie Deep-Dive: Complexity, Variants & Interview Patterns ▼

Space complexity of a Trie:

  • Worst case: O(Σ · N) where Σ is alphabet size and N is total nodes. With a dict (like Python), only present edges are stored.
  • Array-based: Each node stores a fixed array of 26 pointers. Faster but wastes space if most are null.
  • HashMap-based (our approach): Each node stores only existing children. More memory-efficient for sparse alphabets.

Aho-CorasickMulti-pattern streamingTrie + failure links (like KMP). Finds all dictionary matches in O(n + matches)
💡 Trie vs Hash Set: When to Choose Which
  • Hash Set wins when you only need exact word lookup and don't care about prefixes
  • Trie wins when you need prefix queries, or need to walk character-by-character matching against multiple patterns simultaneously
  • This problem uses the Trie's key strength: simultaneous multi-pattern matching by walking the string and the Trie in lockstep

🔑 The "Trie Walk" Pattern

The pattern used in this problem — "for each position in the input, walk the Trie and trigger actions at end-nodes" — is the core Trie application pattern. Variations include: finding longest prefix match (stop at deepest is_end), finding all matches (collect every is_end), and counting distinct prefixes.

9 Summary & Key Takeaways ▼
A Trie stores strings character-by-character as tree paths. Shared prefixes share tree paths, saving time and space.
Insert = walk + create. Search = walk + check is_end. Both are O(L) where L is word length.
Trie + Walk pattern: For each position in s, walk the Trie following characters. At each is_end node, a dictionary word was found.
Boolean array = free interval merge. Multiple overlapping words just set the same bold indices to True. Contiguous True regions become one tag pair.
O(n·L) beats O(n·W·L) when W is large — the Trie eliminates per-word scanning.
