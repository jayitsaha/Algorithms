LeetCode #1642

Furthest Building You Can Reach

Given buildings with heights, a supply of bricks, and ladders, find the index of the furthest building you can reach moving left to right.

Medium Heaps Greedy + Min-Heap Time: O(n log k) Space: O(k)
1 Understanding the Problem ▾

You are a traveller moving left to right across a skyline of buildings. Each building has a height. You start at index 0 and want to go as far right as possible.

When stepping from building i to i+1:

  • If heights[i+1] <= heights[i] — downhill or flat, move for free, no resource consumed.
  • If heights[i+1] > heights[i] — must climb up by diff = heights[i+1] - heights[i]. Pay with either diff bricks or one ladder (ladders cover any gap size).
💡 Key Insight

A ladder is universally more valuable than bricks for the same gap — it costs exactly one ladder regardless of gap size. Therefore, ladders should cover the largest gaps, and bricks should cover the smallest. The challenge: we must decide this online (left to right), without knowing future gaps.

✓ Optimal: Ladder on Biggest Climb

heights=[4,2,7,6,9,14,12]
bricks=5, ladders=1
Climbs: +5 (idx 2), +3 (idx 4), +5 (idx 5)
Ladder on first +5 → free
Bricks: 3+5 = 8 needed, only 5
→ reach index 4

✗ Suboptimal: Ladder Wasted on Small Climb

Use ladder on +3 (smaller climb)
Pay 5 bricks for the first +5
Then need 5 bricks for +5 at idx 5
0 bricks remain → can't afford it
→ reach index 4 same, but risky

Edge Cases: No upward climbs at all (reach the end for free); all climbs are equal (ladder covers arbitrary one, bricks cover rest); bricks = 0 and ladders cover every climb; single building (return index 0).

2 The Algorithm: Greedy + Min-Heap ▾

The strategy is elegant: always assign a ladder to each new climb, then immediately "downgrade" the smallest ladder assignment to bricks whenever we exceed our supply. This ensures ladders always end up on the k largest gaps seen so far.

1 Walk left to right. Compute diff = heights[i+1] - heights[i]. If diff <= 0, step over for free — continue.
2 Push diff onto a min-heap. Conceptually, this tentatively assigns a ladder to this gap. The heap stores the gap sizes of all current ladder assignments.
3 If heap size > ladders, we've over-committed. Pop the minimum from the heap (the weakest ladder claim) and pay for it with bricks: bricks -= heappop(heap).
4 If bricks < 0 after the deduction, we can't afford the transition. Return i — the current building, which is the furthest we reached.
5 If the loop completes without running out of resources, we reached the last building. Return len(heights) - 1.
🪜 Ladder (min-heap)

The heap holds the k largest climbs. Heap size = ladders used. The heap minimum is the "least deserving" ladder — first to be replaced when we're over-committed.

🧱 Bricks (leftover)

Bricks cover every climb that gets evicted from the ladder heap. Each eviction costs exactly the gap size in bricks. When bricks run out, the journey ends.

🔐 The Heap Invariant

At every step, the min-heap contains exactly the k largest positive gaps seen so far, where k = number of ladders. This invariant is maintained after each push-and-possibly-pop pair. The heap minimum is O(1) to read, O(log k) to remove.

3 Why It Works: The Greedy Exchange Argument ▾
📐 Invariant Maintained

After processing each transition, our assignment has ladders on the k largest climbs and bricks on everything else. Any assignment that differs from this either uses more bricks or fewer bricks — but never reaches further.

Formal justification (exchange argument): Suppose an optimal assignment A* differs from our greedy G at some point. In A*, a ladder covers a small gap X, and bricks cover a larger gap Y. Swapping them gives A* with ladder on Y and bricks on X — this costs fewer bricks overall (bricks_needed decreases by Y - X). So A* was not worse than G, and after all such swaps, A* converges to G. Our greedy is therefore optimal.

💡 Why Min-Heap and Not Max-Heap?

We need to find the smallest gap currently assigned a ladder — the one most worth replacing with bricks when we go over the limit. A min-heap surfaces the minimum in O(1) and removes it in O(log k). A max-heap would give the largest, which is the one we least want to remove.

Edge case correctness:

  • diff <= 0: skipped entirely, never enters heap logic.
  • Last building reached: the loop finishes; return len(heights) - 1 is correct.
  • Failure on first climb: return 0 (started at index 0, couldn't reach index 1).
4 The Code (Annotated) ▾
import heapq— Python's built-in min-heap library
def furthestBuilding(heights, bricks, ladders):
heap = []— min-heap: gap sizes where ladders are tentatively assigned
for i in range(len(heights) - 1):— iterate over every consecutive pair
diff = heights[i+1] - heights[i]— height change (negative = downhill)
if diff <= 0:— flat or downhill: no resource needed
continue
heapq.heappush(heap, diff)— tentatively assign a ladder here
if len(heap) > ladders:— over-committed: too many ladders in use
bricks -= heapq.heappop(heap)— swap: pay bricks for smallest ladder gap
if bricks < 0:— can't afford the bricks swap → stuck here
return i— return current index (NOT i+1, couldn't reach it)
return len(heights) - 1— survived all transitions → at the last building
⚠️ Common Mistake: Off-by-One in Return Value

When bricks < 0, we return i — the building we are currently standing on. We tried to move from i to i+1 but failed. Returning i+1 is wrong — we never reached it. Returning i-1 is also wrong — we did reach i.

⚠️ Common Mistake: Order of Operations

Always: push first, then check if len(heap) > ladders, then pop and deduct bricks, then check if bricks < 0. Reversing any two of these four steps produces incorrect results.

Complexity: Time O(n log k) — n iterations, each with at most one push and one pop into a heap of size ≤ k+1. Space O(k) — the heap holds at most k+1 elements at any moment (we pop immediately when it exceeds k).

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognise the Greedy Opportunity

Two resources with different costs and a goal to maximise distance. The key observation: ladders are uniformly more valuable for large gaps. If you could see all gaps upfront, you'd sort them and assign ladders to the biggest ones. Since we can't see the future, we must decide online.

Step 2: Think "Maintain Top-K Online"

We want ladders to always cover the k largest gaps seen so far. This is a classic sliding/growing top-k problem — solved perfectly by a min-heap of size k. Push every element, pop when size exceeds k.

Step 3: Connect the "Pop" to Brick Payment

Every time we pop the minimum gap, we're saying: "This gap no longer gets a ladder — pay for it with bricks." The pop value IS the brick cost. Decrement bricks by the popped value. If bricks go negative, the journey is over.

Step 4: Clarify the Return Value

State this explicitly to the interviewer: "When bricks go negative at transition i → i+1, we return i because that's the last building we successfully stood on." Many candidates trip here; naming it shows depth.

🗣️ What to Say to the Interviewer

"I notice ladders are strictly better than bricks for large gaps, so they should cover the biggest climbs. I'll walk left to right and greedily assign a ladder to each climb, using a min-heap to track which gaps have ladders. When I've used more ladders than I have, I swap the smallest ladder claim to bricks — that's the heappop. If bricks go negative, I can't make that jump. The answer is i (current building), not i+1, since I failed the transition from i to i+1. Time: O(n log k), Space: O(k) for the heap."
🎯 Pattern Recognition Signals
  • "K resources of type A, unlimited of type B" + "A is uniformly better" → greedy top-k with min-heap
  • "Maintain k best elements online (left to right)" → min-heap of size k, push-then-pop
  • "Furthest/maximum reach with limited budget" → greedy walk with failure check

7 Summary & Key Takeaways ▾
Ladders always cover the k largest gaps — min-heap of size k maintains this invariant dynamically
Push every positive diff; pop the minimum when heap exceeds ladder count
When bricks go negative, return i (current building) — not i+1
Downhill and flat transitions are always free — handled by the diff ≤ 0 check
Time: O(n log k) — one push + one pop per step, each O(log k)
Space: O(k) — heap holds at most k+1 elements at any moment
ApproachTimeSpaceNotes
Brute Force (try all assignments) O(2^n) O(n) Infeasible — exponential
Binary Search on Answer O(n log n) O(n) Binary search on reach index; verify greedily
Min-Heap Greedy ✓ O(n log k) O(k) Single pass; optimal; interview pick
🎯 When to Reach for Min-Heap + Greedy
  • "K items of scarce resource A; unlimited of B; A saves more on large instances" → min-heap size k
  • "Maintain top-k elements in a growing stream" → push all, pop minimum when oversized
  • "Furthest you can go" or "maximum reach" with a running cost → greedy walk + failure check
