LeetCode #162

Find Peak Element

A peak element is strictly greater than its neighbors. Find the index of any peak element in O(log n) time.

Medium Binary Search Compare with Neighbor Time: O(log n) Space: O(1)
1 Understanding the Problem ▾

A peak element is an element strictly greater than its neighbors. The array has imaginary sentinels nums[-1] = nums[n] = -∞ at both ends, meaning the first and last elements only need to beat one neighbor. You must return the index of any peak — not necessarily the largest element.

💡 Key Insight

You only need to find a peak — not the peak. This subtle distinction unlocks binary search: the moment you find any position where the array goes downhill, you know a peak is on that side. You don't need to scan the whole array. The -∞ sentinels guarantee at least one peak always exists.

✓ Peak at index 2: [1, 3, 5, 4, 2]

nums[2] = 5
left: nums[1] = 3 < 5 ✓
right: nums[3] = 4 < 5 ✓
→ index 2 is a valid peak

✓ Multiple valid peaks: [1, 3, 2, 5, 4]

Index 1: 3 > 1 and 3 > 2 → peak
Index 3: 5 > 2 and 5 > 4 → peak
Either index (1 or 3) is accepted!

Key constraint: nums[i] != nums[i+1] — no adjacent equal elements. This rules out plateaus and is precisely what makes binary search provably correct: every midpoint has a clear slope direction.

2 The Algorithm: Binary Search on Slope Direction ▾

We're not searching for a target value — we're searching using slope direction. At every midpoint, we look at the slope from mid to mid+1. If the array goes uphill there, the peak is further right. If it goes downhill, the peak is at or to the left of mid. Either way we cut the search space in half.

1 Initialize: lo = 0, hi = n-1. The loop invariant is established: a peak is guaranteed to exist in [lo, hi]. (True for the whole array initially.)
2 Compute midpoint: mid = (lo + hi) // 2. Compare nums[mid] with nums[mid+1] — just one comparison, just the right neighbor.
3 Uphill right? If nums[mid] < nums[mid+1], slope goes up toward mid+1. A peak must exist in [mid+1, hi]. Set lo = mid + 1. Discard left half safely.
4 Downhill or at peak? Otherwise (nums[mid] >= nums[mid+1]), slope goes down from mid or mid is higher. A peak exists in [lo, mid]. Set hi = mid. Note: mid is included!
5 Converge: Loop runs while lo < hi. When lo == hi, the window is a single element — the loop invariant guarantees it's a peak. Return lo.
🔑 Why Only One Comparison (Right Neighbor)?

We only compare nums[mid] vs nums[mid+1]. We skip the left neighbor because the loop invariant already guarantees a peak is inside [lo, hi]. The slope direction tells us which half contains it — we don't need to verify both sides of mid at every step. The invariant does that work for us.

3 Why It Works: The Invariant ▾
🔐 The Loop Invariant

At the start of every iteration, at least one peak element exists within [lo, hi]. This holds initially (every array has a peak). Each pointer move preserves it. When lo == hi, that single index is the guaranteed peak.

Why the invariant holds after "uphill right" move (lo = mid + 1):
nums[mid] < nums[mid+1] means the array goes up at index mid. Consider the sub-array starting at mid+1. Because the array is finite, it can't keep going up forever. At some point it must go down (or reach the right boundary, which is -∞). That turning point — where it stops going up — is a peak. So [mid+1, hi] contains a peak.

Why the invariant holds after "downhill" move (hi = mid):
nums[mid] >= nums[mid+1] means mid is higher than its right neighbor. Either mid is already a peak (left neighbor from invariant is satisfied by induction), or a peak exists in [lo, mid-1]. In both cases, [lo, mid] contains a peak. We keep mid in the window because it might be the peak itself.

⚠️ Critical: Why hi = mid (not mid - 1)?

In standard binary search, when we know the target isn't at mid, we safely do hi = mid - 1. Here, we can't discard mid — it might be a peak (it's higher than its right neighbor). Using hi = mid - 1 would lose the answer. Always hi = mid for "Compare with Neighbor" binary search.

💡 The Fog-on-a-Mountain Analogy

You're hiking in thick fog and can only see one step ahead. Rule: "always walk uphill." You're guaranteed to reach a peak — you can't keep going uphill forever on a finite mountain. Binary search is the efficient version: instead of walking one step, you teleport to the middle and ask "which way is uphill?" — halving your remaining search space each time.

4 The Code (Annotated) ▾

Solution 1 — Binary Search, Compare with Neighbor (Optimal — Interview Pick)

def findPeakElement(nums: list[int]) -> int:
lo, hi = 0, len(nums) - 1 ← Invariant: a peak exists in [lo, hi]
while lo < hi: ← Loop until window collapses to one element
mid = (lo + hi) // 2 ← Floor division; safe from overflow in Python
if nums[mid] < nums[mid + 1]: ← Slope goes UP to the right
lo = mid + 1 ← Peak in [mid+1, hi]; discard left half
else: ← Slope goes DOWN from mid (or mid > mid+1)
hi = mid ← Peak in [lo, mid]; keep mid — it could be the answer
return lo ← lo == hi; this is the guaranteed peak
⚠️ Two Common Off-By-One Mistakes

Mistake 1: Writing hi = mid - 1 — this discards mid, which could be the peak when it's greater than mid+1. Always hi = mid.
Mistake 2: Loop condition lo <= hi instead of lo < hi — with hi = mid, this can cause an infinite loop when lo == hi. Use strict lo < hi.

Solution 2 — Linear Scan (Brute Force, O(n))

def findPeakElement(nums: list[int]) -> int:
n = len(nums) ← Treat as if nums[-1] = nums[n] = -inf
for i in range(n): ← Check every index; O(n)
left = nums[i-1] if i > 0 else float('-inf') ← Left sentinel at boundary
right = nums[i+1] if i < n-1 else float('-inf') ← Right sentinel at boundary
if nums[i] > left and nums[i] > right: ← Strictly greater than both?
return i ← Return first peak found
💡 Complexity Summary

Binary Search (Optimal): Time O(log n) — halves search space each iteration; Space O(1)
Linear Scan (Brute Force): Time O(n) — visits each element once; Space O(1)
Recursive Binary Search: Time O(log n); Space O(log n) — call stack; prefer iterative

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Clarify the Problem

Ask: "Do I return any peak or the global maximum?" (Any peak.) "Are adjacent elements always different?" (Yes.) This second constraint is essential — it's what prevents plateaus and makes slope direction unambiguous. Without it, binary search breaks.

Step 2: Propose Linear Scan First

"I can scan linearly in O(n), checking each element against its neighbors. Return the first peak found. This is correct but the problem explicitly requires O(log n), so I need to do better."

Step 3: Motivate Binary Search via Slope

"This array isn't sorted, but I don't need the array sorted. I need to eliminate half the candidates per step. Looking at the slope at mid: if nums[mid] < nums[mid+1], the array is going uphill, so a peak must be to the right. If downhill, a peak is at or to the left of mid. Either way I cut the space in half."

Step 4: State the Invariant Explicitly

"The loop invariant is: a peak exists within [lo, hi]. Both pointer moves preserve this invariant. When lo == hi, that's my peak. And I use hi = mid (not mid-1) because mid itself could be the peak — I never prematurely discard it."

🗣️ What to Say to the Interviewer

"Binary search, but not on a target value — on slope direction.
Invariant: a peak always exists in [lo, hi].
At each midpoint: if nums[mid] < nums[mid+1], slope goes up → peak in right half, set lo = mid+1.
Otherwise → peak at mid or left, set hi = mid (keep mid! it could be the peak).
When lo == hi, the window has one element — it's the peak. Return lo.
O(log n) time, O(1) space."
🔮 Likely Follow-up Questions
  • "What if elements could be equal (plateau)?" → Three-way comparison (check both neighbors); binary search becomes trickier.
  • "Find peak in a 2D grid?" → Binary search on columns, for each column take row maximum (LC #1901).
  • "Find peak in a mountain array with no adjacent equals?" → Same technique; then binary search ascending/descending halves for target (LC #1095).
  • "How many peaks are in the array?" → Can't do better than O(n) — must check every element.

7 Summary & Key Takeaways ▾
Binary search works on unsorted arrays when the slope direction determines which half holds the answer
Invariant: a peak is always in [lo, hi] — both pointer moves preserve this guarantee
Use hi = mid (not mid - 1) because mid could itself be the peak when downhill
Use loop condition lo < hi (not lo <= hi) to avoid infinite loops with hi = mid
Finding any peak is sufficient — resist the urge to find the global maximum

🎯 When to Recognize "Compare with Neighbor" Binary Search
  • Problem asks for a local maximum or minimum, not global
  • Array has a monotone structure in sub-regions (mountain / valley shape)
  • "Find any valid answer" — not "find the best answer"
  • O(log n) time constraint is given explicitly
  • Adjacent elements are guaranteed distinct (nums[i] != nums[i+1])
