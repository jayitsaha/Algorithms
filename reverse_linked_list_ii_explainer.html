LeetCode #92

Reverse Linked List II

Given the head of a singly linked list and two integers left and right (1-indexed), reverse the portion of the list from position left to position right, and return the modified list.

Medium Linked List Reverse Sublist [m, n] Time: O(n) Space: O(1)
1 Understanding the Problem ▾

You have a singly linked list and two 1-indexed positions left and right. Reverse only the nodes between those positions (inclusive), then return the head of the modified list. You must accomplish this in-place — no extra data structure, and ideally in a single pass.

💡 Key Insight

Think of the list as three segments: a prefix (nodes 1 to left−1), the sublist (nodes left to right, which gets reversed), and a suffix (nodes right+1 to end). Only the arrows inside the sublist change direction. The prefix and suffix are completely untouched — we just need to reconnect them to the newly reversed sublist.

Input: 1→2→3→4→5, left=2, right=4

prefix: [1]
sublist: [2, 3, 4] ← reverse this
suffix: [5]

Output: 1→4→3→2→5

prefix: [1] (unchanged)
sublist: [4, 3, 2] ← reversed!
suffix: [5] (unchanged)

Edge Cases:

  • left == right: No reversal needed — return as-is.
  • left == 1: The head itself is inside the sublist. The head changes — a dummy node before the list handles this cleanly.
  • right == n (last node): The suffix is empty. Still works normally.
  • Single node list: left == right == 1 — return unchanged.
🔑 Why In-Place Reversal of a Sublist is Tricky

Reversing an entire linked list is straightforward: walk forward, flip arrows. But reversing a sublist requires you to also carefully reattach the reversed chain to the prefix (before left) and suffix (after right). You need to track four key nodes: the node before the sublist starts, the first node of the sublist, the last node of the sublist, and the node after the sublist ends.

2 The Algorithm: One-Pass "Head Insertion" Reversal ▾

The elegant approach uses head insertion: instead of fully reversing the sublist then reconnecting, we iteratively "pull" each node of the sublist to the front of the growing reversed portion. No explicit save-and-reconnect step needed.

1 Dummy head: Create a dummy node before the head. prev = dummy. This sentinel removes the special case when left == 1 (the head itself changes).
2 Walk to anchor: Advance prev exactly (left − 1) times. Now prev is the last node of the prefix — the stable anchor just before the reversal zone. Set curr = prev.next (the first node of the sublist). curr will not move again.
3 Head insertions: Repeat (right − left) times. Each iteration: save nxt = curr.next, unhook nxt from its position (curr.next = nxt.next), then insert nxt right after prev (nxt.next = prev.next then prev.next = nxt). After each step, the reversed portion grows by one node at the front.
4 Return dummy.next: The dummy always points to the true head of the list, whether or not the head was inside the sublist.
💡 The Mental Model: "Sliding a Card to the Front"

Imagine dealing playing cards face-up. You have a partial row and want to reverse positions 2 through 4. Instead of lifting all three out and flipping the stack, you keep sliding the next card to the front of the group: first card 3 slides in front of card 2 (giving 3,2), then card 4 slides in front of card 3 (giving 4,3,2). Two slides for three cards = (right−left) slides for (right−left+1) cards. curr is the original card 2 — it stays put as the tail of the reversed group.

3 Why It Works: The Invariant ▾
🔐 The Invariant

After iteration k of the inner loop, the nodes from position left to position left + k are in reversed order immediately following prev, and curr remains at the original first node of the sublist (now the tail of the reversed portion). prev never moves from its anchor position.

Tracing one iteration step in detail: Suppose before the iteration the list looks like:
... → prev → [front of reversed] → curr → nxt → ...

Step A: nxt = curr.next ← save the node we are about to pull out
Step B: curr.next = nxt.next ← curr skips over nxt (nxt is detached from chain)
Step C: nxt.next = prev.next ← nxt points at the old front (it will precede it)
Step D: prev.next = nxt ← attach nxt as the new front of the reversed portion
After: ... → prev → nxt → [front] → curr → ...

Why exactly (right − left) iterations: curr starts as the first node of the sublist (position left). After every insertion, it is naturally the tail of the growing reversed portion — we never need to insert curr itself. The loop only inserts the nodes that follow it. There are (right − left) such nodes (positions left+1 through right).

⚠️ The Order of Steps B, C, D Cannot Be Swapped

If you do Step D before Step C, you lose the reference to prev.next (the old front). If you do Step C before Step B, curr.next still points to nxt, so Step B would overwrite the wrong thing. Always: save → detach → re-point → insert. Think of it as removing a card from the middle of a row and sliding it to the front — you must pick it up before you can move anything else.

The dummy node proof: When left == 1, prev is the dummy node. After reversal, dummy.next points to whatever node was inserted last (the original rightmost node of the sublist, which is now the new head). dummy.next always correctly reflects the true head — even if the head changed.

4 The Code (Annotated) ▾

Solution 1 — One-Pass Head Insertion (Optimal, Interview Pick)

def reverseBetween(head, left, right):
dummy = ListNode(0) ← sentinel before head; handles left==1 gracefully
dummy.next = head
prev = dummy ← anchor node, stays at position left-1 forever
for _ in range(left - 1): ← walk prev exactly (left-1) hops
prev = prev.next
curr = prev.next ← first node of sublist; becomes the tail of reversed portion
for _ in range(right - left): ← NOT right-left+1; curr is already placed
nxt = curr.next ← node to pull to front — SAVE first or it's lost
curr.next = nxt.next ← unlink nxt from its current position
nxt.next = prev.next ← nxt will precede current reversed-front
prev.next = nxt ← insert nxt as new front of reversed portion
return dummy.next ← true head (unchanged if left>1; new head if left==1)
⚠️ Common Mistake: Looping (right − left + 1) Times

The sublist has (right − left + 1) nodes total. But curr starts as the first node and naturally ends up as the tail after all the insertions — you never insert curr itself. So the loop only runs (right − left) times. Looping one extra time would over-rotate the sublist, producing the wrong order.

Solution 2 — Brute Force: Collect, Slice-Reverse, Rebuild

def reverseBetween(head, left, right):
if not head or left == right: return head ← quick exits
nodes, curr = [], head
while curr: ← collect all nodes into a list
nodes.append(curr); curr = curr.next
l, r = left - 1, right - 1 ← convert to 0-indexed
nodes[l:r+1] = nodes[l:r+1][::-1] ← reverse the subarray slice
for i in range(len(nodes)-1): nodes[i].next = nodes[i+1] ← restitch .next
nodes[-1].next = None
return nodes[0]
💡 Complexity Summary

One-Pass Head Insertion (Optimal): Time O(n), Space O(1) — three pointers, one forward pass
Brute Force (Collect & Reverse): Time O(n), Space O(n) — extra list for all n nodes
In interviews, the O(1) space solution is always expected for linked list problems.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Signals

"Linked list + reverse a portion + in-place + single pass" → this is the classic Reverse Sublist [m,n] pattern. Mention dummy head immediately: "I'll use a sentinel node before head to avoid the left==1 edge case."

Step 2: Offer Brute Force First

"I could collect all nodes into an array, reverse the slice from index left−1 to right, then restitch pointers. O(n) time and O(n) space. Let me see if I can get O(1) space." Always confirm the interviewer wants the optimization before spending time on it.

Step 3: Draw the Pointer Diagram

Draw: prev → [front of reversed] → curr → nxt → ... and show one iteration. Explain verbally: "I save nxt, unlink it, then insert it right after prev. Three pointer changes per step, exactly (right−left) steps." Interviewers love when you draw this — it shows you understand the mechanics before coding.

Step 4: State the Invariant

"At all times, curr is the tail of the growing reversed portion. prev is the fixed anchor. After (right−left) iterations, the full sublist is reversed and connected." This shows you've thought about correctness, not just the mechanics.

🗣️ What to Say to the Interviewer

"I'll use a dummy node before the head — this makes left==1 a non-issue.
Walk prev to position left−1. Then curr = prev.next is the sublist's first node.
curr will stay put — it becomes the tail of the reversed sublist.
Loop (right−left) times: each time, pull curr.next to just after prev.
Three pointer lines per step: save nxt, curr.next skips nxt, nxt.next = prev.next, prev.next = nxt.
Return dummy.next. Time O(n), Space O(1), single pass."
🔮 Likely Follow-up Questions
  • "Reverse entire list?" → LeetCode #206; same technique with left=1, right=n, or even simpler 3-variable reverse.
  • "Reverse in groups of k?" → LeetCode #25; apply this sublist reversal every k nodes, repeatedly.
  • "What if it's a doubly linked list?" → Also update prev pointers. Same overall structure.
  • "Can you do it without dummy node?" → Yes: check if left==1 before entering, handle head change manually. Dummy is cleaner.
  • "Reverse every other k-group?" → Track which groups to reverse vs. skip; apply sublist reversal selectively.

7 Summary & Key Takeaways ▾
Dummy head before the list — always use it for any linked list surgery; eliminates left==1 special case
prev is the fixed anchor at position left−1; it never moves during the reversal loop
curr starts as sublist head and becomes the tail — it also never moves after initialization
Loop exactly (right − left) times: you insert (right−left) nodes; curr is already in place
Three pointer lines per iteration — order is critical: save nxt, skip, re-point, insert
O(1) space: only three extra pointers (dummy, prev, curr, nxt); no data structure needed
