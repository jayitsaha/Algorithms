LeetCode #472

Concatenated Words

Given an array of strings words, return all the concatenated words in the given list of words.

Hard Tries Trie + DP Time: O(n·L²) Space: O(n·L)
1 Understanding the Problem ▾

You're given a list of unique, non-empty strings (words). A concatenated word is a word that can be formed entirely by joining two or more other words from the same list together. Return all such words.

💡 Key Insight

A word w is "concatenated" if there exists at least one split point where the prefix is a dictionary word AND the remaining suffix can also be recursively decomposed into dictionary words — using a minimum of 2 pieces total. This is exactly the Word Break problem applied to every word as a candidate, with the twist that the word itself cannot be used as a single piece.

✓ Concatenated: "catsdog"

"cats" + "dog" → both in dict ✓
"cat" + "s"??? → "s" not in dict ✗
First split works → YES

✗ Not Concatenated: "cat"

"c"+"at"? → "c" not in dict
"ca"+"t"? → "ca" not in dict
Cannot split into 2+ words → NO

The crucial constraint: The word itself cannot count as one of its own pieces. We need at least two other dictionary words. So "cat" cannot satisfy its own split using "cat" as a single piece — even if "cat" is in the dictionary.

Edge cases to consider: A word repeated like "catcat" = "cat"+"cat" is valid if "cat" is in the list. Empty strings (excluded by constraints). A word that has many valid splits (e.g., "aaaa" from {"a","aa","aaa"}) — DP handles all combinations efficiently without exponential blowup.

2 The Algorithm: Sort + Trie-Guided Word Break DP ▾

We combine two classic techniques: a Trie for fast prefix matching, and bottom-up DP (Word Break style) for checking whether a word can be decomposed. The key elegance is sorting by length and inserting lazily.

1 Sort by word length (shortest first). A concatenated word is always strictly longer than each of its component words. By sorting, when we test word W, all shorter words are already in the Trie and available as components.
2 Initialize DP array: dp[0..n] where dp[i] = True means word[0:i] is fully coverable by dictionary words. Set dp[0] = True (base case: empty prefix needs zero pieces).
3 For each reachable position i (dp[i]=True), walk the Trie starting at character word[i]. At each Trie node where is_end=True, mark dp[j+1]=True (j = current position). This propagates reachability forward through all valid dictionary words starting at i.
4 Check dp[n] for the final answer. We skip the case where the entire word itself is found as a single Trie match (i=0, j=n-1) to enforce the ≥2 pieces constraint. With the sort+lazy-insert approach, this case can't even occur because the word is not yet in the Trie.
5 Insert word into Trie AFTER testing. This is the core trick — a word is never in the Trie when it's being tested, so it cannot "use itself" as a component. After testing, insert it so future (longer) words can use it.
🔑 Why Trie Instead of a Hash Set?

With a hash set, you'd probe word[i:j] for every (i,j) pair — O(L) probes per position, O(L) positions = O(L²) probes each costing O(L) for the substring — total O(L³) per word. With a Trie, we walk character-by-character from each position i — O(L) Trie walk per position, O(L) positions = O(L²) character operations per word. The Trie also provides early termination: if a character isn't in the Trie, we stop immediately instead of continuing to try longer substrings.

3 Why It Works: The Invariant ▾
🔐 The DP Invariant

At any point, dp[i] = True means and only means: "the substring word[0:i] can be exactly partitioned into one or more words from the current Trie dictionary." dp[n] = True means the entire word is coverable, and since the word itself is not in the Trie at test time, it must be covered by ≥2 separate pieces.

Why sorting by length is the right invariant to maintain: Suppose word W can be concatenated using components C1 and C2. Both C1 and C2 are strictly shorter than W (since they're non-empty and together form W). By sorting shortest-first and only inserting after testing, when W is tested, both C1 and C2 are guaranteed to already be in the Trie — because they were processed (and inserted) before W.

Why dp[0] = True is the correct seed: The empty prefix doesn't require any dictionary words to "cover" it — vacuously, zero words cover zero characters. This is the starting point from which all positions propagate. Think of it as: "if I'm at position 0, I've used zero words so far, which is a valid starting state."

Why the DP correctly handles all combinations: The nested loop structure (outer over reachable positions i, inner Trie walk from i) is equivalent to exploring all possible first-word choices for each sub-suffix, in a bottom-up manner. No combination is missed, and no combination is revisited — each position is processed exactly once.

⚠️ Common Mistake: Inserting Before Testing

If you insert "cats" into the Trie, then test "cats", the Trie walk from i=0 will find "cats" as a complete word ending at j=3, setting dp[4]=True, then return dp[4]=True — claiming "cats" is concatenated (using itself as a single piece). This is wrong. The sort+post-insert strategy avoids this entirely. Alternatively, track piece count and require ≥2, or remove the word before testing and re-insert after.

Proof sketch: By induction on word length. Base: all length-1 words tested against an empty Trie — dp stays False (no components), correctly not concatenated. Inductive step: assume all words of length < L are correctly handled. When testing word W of length L, the Trie contains exactly all words shorter than W. The DP finds a partition of W into such words if and only if one exists. QED.

4 The Code (Annotated) ▾

Solution 1 — Trie + DP (Optimal, Interview Pick)

class TrieNode:
def __init__(self):
self.children = {} ← char → TrieNode mapping
self.is_end = False ← True if a complete word ends here
def findAllConcatenatedWordsInADict(words):
words.sort(key=len) ← Shortest first: components always precede candidates
root = TrieNode() ← Empty Trie, grows as we process each word
result = []
def insert(word):
node = root
for c in word:
if c not in node.children:
node.children[c] = TrieNode()
node = node.children[c]
node.is_end = True ← Mark end of this word in Trie
def canForm(word):
n = len(word)
dp = [False] * (n + 1) ← dp[i] = word[0:i] coverable by Trie words
dp[0] = True ← Base: empty prefix is trivially coverable
for i in range(n): ← Each reachable starting position
if not dp[i]: continue ← Skip unreachable positions (pruning)
node = root ← Begin Trie walk from root at position i
for j in range(i, n): ← Extend match character by character
if word[j] not in node.children: break ← Trie dead end: no word starts this way
node = node.children[word[j]]
if node.is_end: ← Found a complete Trie word ending at j
if i == 0 and j == n - 1: continue ← Entire word as one piece: skip (need ≥2)
dp[j + 1] = True ← Mark the position after this word as reachable
return dp[n] ← True = entire word covered by ≥2 Trie words
for word in words: ← Process shortest first (sorted above)
if canForm(word): ← Test BEFORE inserting into Trie
result.append(word)
insert(word) ← Now insert so future longer words can use it
return result
⚠️ The i==0 and j==n-1 Guard

This guard prevents the word from being "covered" by itself as a single piece in the Trie walk. With the sort+lazy-insert strategy, this guard is actually unnecessary (the word isn't in the Trie yet), but it's excellent defensive programming — it makes the intent explicit and handles edge cases if the strategy were ever changed.

Solution 2 — DFS + Memoization + Set (Simpler to derive in an interview)

def findAllConcatenatedWordsInADict(words):
word_set = set(words) ← O(1) membership test
memo = {} ← Cache (suffix, is_first_call) → bool
def can_form(s, first=True):
if (s, first) in memo: return memo[(s, first)]
if not first and s in word_set: ← Suffix is a whole word AND at least 1 piece used
return True
for i in range(1, len(s)): ← Try each possible first-piece length
prefix = s[:i]
if prefix in word_set and can_form(s[i:], False):
memo[(s, first)] = True
return True
return memo.setdefault((s, first), False)
return [w for w in words if can_form(w)]
💡 Complexity Summary

Trie + DP (optimal): Time O(n·L²), Space O(n·L) for Trie storage + O(L) per DP array
DFS + Memo + Set: Time O(n·L²) amortized, Space O(n·L) for memo + O(L) call stack
Brute Force (no memo): Time O(n·2^L) — exponential, fails on L > 30
where n = number of words, L = max word length

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Reduce to a Known Problem

Immediately state: "This is Word Break (#139) applied to every word. For each candidate, I want to know if it can be decomposed into ≥2 other words from the list." This frames the problem correctly and shows pattern recognition.

Step 2: Name the Constraint and Solve It Elegantly

"The tricky part: the word cannot use itself as a component. I'll sort by length and only insert words into the Trie after testing them. Since a concatenated word must be strictly longer than its parts, sorting guarantees all valid components are already in the Trie."

Step 3: Choose the Right Data Structure

"For prefix lookup, a Trie is more efficient than probing a hash set with every substring — O(L) Trie walk vs. O(L) substrings each taking O(L) time. The Trie also prunes dead branches: if the Trie has no path for a prefix, we stop immediately."

Step 4: Code the DP Transition Cleanly

"dp[0]=True. For each reachable i, walk the Trie. Every time I hit an is_end node at position j, I set dp[j+1]=True. At the end, dp[n] tells me if the full word is reachable." Walk through the example on the whiteboard with "catsdog".

🗣️ What to Say to the Interviewer

"This is Word Break applied to every word in the list. Let me sort by length first.
For each word: run DP where dp[i] means 'word[0:i] is covered by Trie words'.
From each True dp[i], walk the Trie — whenever I hit is_end at j, set dp[j+1]=True.
Test the word BEFORE inserting it into the Trie — this prevents self-use naturally.
If dp[len(word)] is True, it's concatenated. Time O(n·L²), Space O(n·L)."
🔮 Likely Follow-up Questions
  • "What if the list has duplicates?" → Deduplicate first; duplicates don't add new components.
  • "Can you return the actual partitions, not just the words?" → Store backpointers for each dp[i]=True.
  • "What if we need exactly K words in the concatenation?" → Track piece count in DP state: dp[i][k] = True means word[0:i] covered by exactly k pieces.
  • "Why not use DFS+memo instead of DP?" → Both are O(L²) per word. DFS+memo is easier to derive; Trie+DP has better constants and pruning.
