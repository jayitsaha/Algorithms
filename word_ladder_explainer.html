LeetCode #127

Word Ladder

Find the length of the shortest transformation sequence from beginWord to endWord, changing exactly one letter per step, where each intermediate word must exist in wordList.

Hard Graph BFS Wildcard Patterns Time: O(N·L²) Space: O(N·L)
1 Understanding the Problem ▾

You are given a beginWord, an endWord, and a list of valid dictionary words. Find the minimum number of steps to transform beginWord into endWord, where each step changes exactly one letter and every intermediate word must be in the dictionary. Return 0 if no such path exists.

✓ Path Found — return 5

beginWord = "hit"
endWord = "cog"
wordList: ["hot","dot","dog","lot","log","cog"]

hit → hot → dot → dog → cog
4 hops + start = length 5

✗ No Path — return 0

Same input but wordList
has no "cog":
["hot","dot","dog","lot","log"]

endWord unreachable
→ Return 0

💡 Key Insight

This is a shortest path problem on an implicit graph. Treat each word as a graph node. Two nodes share an edge if the words differ by exactly one letter. Since all edges cost equally (one hop), BFS on this graph yields the minimum number of transformations. The challenge is finding "neighbors" efficiently without O(N²) pairwise comparisons.

Edge Cases: endWord not in wordList → return 0 immediately (impossible). beginWord == endWord → problem doesn't ask this. Empty wordList → return 0. Single-character words → every pair is either identical or 1-hop neighbors.

2 The Algorithm: BFS with Wildcard Pattern Buckets ▾

The core problem is that finding all words that differ from a current word by exactly one letter naively requires comparing against all N dictionary words — an O(N·L) operation per node. Repeated across all N nodes, this gives O(N²·L). We need a better way to enumerate neighbors.

🔑 The Wildcard Bucket Optimization

For word "hot" of length 3, generate L=3 wildcard patterns: "*ot", "h*t", "ho*". Pre-map each pattern to all dictionary words that match it. Then finding all 1-letter neighbors of any word is just L dictionary lookups — O(L) lookups, each amortized O(1). Total neighbor enumeration per node: O(L²).

1 Fast-fail check: If endWord is not in wordList, return 0 immediately. No path can ever lead to a word outside the dictionary.
2 Build wildcard buckets: Iterate over every word in wordList. For each word, generate L wildcard patterns (one per character position). Append each word to buckets[pattern]. This costs O(N·L²) time and O(N·L) space.
3 BFS initialization: Create a queue with (beginWord, 1). Create a visited set containing beginWord. Distance starts at 1 (the start word counts as step 1).
4 BFS expansion: Pop the front of the queue. Generate L wildcard patterns for the current word. For each pattern, look up all words in the bucket. For any unvisited word: if it's endWord, return dist + 1. Otherwise mark visited and enqueue.
5 Return 0 if queue exhausted: If BFS completes without ever seeing endWord, no transformation sequence exists.

3 Why It Works: The BFS Invariant ▾
🔐 The BFS Shortest-Path Invariant

In BFS on an unweighted graph, when a node is first dequeued at distance d, it is impossible to reach that node in fewer than d hops. This is because BFS exhausts all level-d nodes before exploring any level-(d+1) node. Therefore the first time we encounter endWord, we have found the shortest path to it.

Why wildcard buckets are correct: Two words of equal length differ by exactly one letter if and only if they share at least one wildcard pattern. "hot" and "dot" both produce the pattern "*ot". Conversely, if two words share NO wildcard pattern, they must differ at two or more positions. So the bucket lookup is a mathematically exact encoding of the "1-letter-different" neighbor relation.

Why DFS would fail for shortest path: DFS might discover the path hit→hot→dot→dog→log→cog (length 6) before finding hit→hot→dot→dog→cog (length 5). DFS gives a path, not the shortest path. BFS's level-by-level guarantee is essential.

⚠️ Critical: Mark visited when enqueuing, not dequeuing

If a word is added to the queue but only marked visited when processed, the same word can be added multiple times from different neighbors at the same BFS level. This doesn't affect correctness (same shortest distance will be found), but it multiplies work and can cause exponential blowup in dense graphs. Always add to visited set at the moment you enqueue.

beginWord doesn't need to be in wordList: It's the start node — we never need to "reach" it through dictionary hops. Only intermediate words and endWord must be in wordList. The wildcard buckets are built exclusively from wordList, so beginWord naturally won't appear as a neighbor (unless it happens to be in the list).

4 The Code (Annotated) ▾

Solution 1 — BFS with Wildcard Buckets (Optimal, Interview Pick)

from collections import defaultdict, deque ← deque for O(1) popleft; defaultdict for buckets
def ladderLength(beginWord, endWord, wordList):
if endWord not in wordList: return 0 ← Fast fail: unreachable target
L = len(beginWord) ← All words guaranteed same length
buckets = defaultdict(list) ← Pattern → [matching words]
for word in wordList: ← Pre-process: build wildcard buckets
for i in range(L): ← L patterns per word
buckets[word[:i]+'*'+word[i+1:]].append(word) ← e.g. "hot"→"*ot","h*t","ho*"
queue = deque([(beginWord, 1)]) ← BFS queue: (word, distance)
visited = {beginWord} ← Mark start visited immediately
while queue: ← BFS main loop
word, dist = queue.popleft() ← O(1) dequeue from front
for i in range(L): ← Try all L wildcard positions
pattern = word[:i] + '*' + word[i+1:] ← Wildcard at position i
for neighbor in buckets[pattern]: ← All words sharing this pattern
if neighbor == endWord: ← Check BEFORE visiting — early exit
return dist + 1 ← BFS guarantee: shortest path
if neighbor not in visited: ← Skip already-visited words
visited.add(neighbor) ← Mark BEFORE enqueuing!
queue.append((neighbor, dist+1))← Add with incremented distance
return 0 ← Queue exhausted: no path
⚠️ Why check endWord separately before visiting?

If you add endWord to the visited set and queue like other words, you'd need one extra dequeue step to process it — that's correct but wastes one iteration. Checking neighbor == endWord before the visited check gives an early return the instant endWord is first discovered, regardless of whether it's in visited. This is a clean, safe optimization.

Solution 2 — Brute Force BFS (O(N²·L), educational reference)

def ladderLength_brute(beginWord, endWord, wordList):
word_set = set(wordList) ← O(1) membership test
if endWord not in word_set: return 0
queue = deque([(beginWord, 1)])
visited = {beginWord}
while queue:
word, dist = queue.popleft()
for candidate in word_set: ← O(N) per node — the bottleneck
if candidate not in visited:
diffs = sum(a!=b for a,b in zip(word,candidate)) ← O(L)
if diffs == 1:
if candidate == endWord: return dist + 1
visited.add(candidate)
queue.append((candidate, dist+1))
return 0
💡 Complexity Comparison

Wildcard BFS: Time O(N·L²) — building buckets + BFS with L² neighbor lookup per node
Brute Force BFS: Time O(N²·L) — BFS × comparing each candidate × word length
Bidirectional BFS: Time O(N·L²) same asymptotic, but ~2× faster in practice
With N=5000, L=5: Wildcard ~125K ops vs Brute ~125M ops

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Shortest-Path Structure

Key signals: "shortest transformation", "minimum steps", "each step changes one thing". This is BFS on an implicit graph. Nodes = words, edges = 1-letter-different pairs, edge weights = uniform. Shortest unweighted path = BFS.

Step 2: Name the Bottleneck Before Coding

Don't jump to BFS immediately. Ask: "How do I find neighbors efficiently?" The naive O(N·L) comparison per node scales to O(N²·L) total. That's the bottleneck. Solving neighbor enumeration IS the problem.

Step 3: Propose Wildcard Buckets

"Pre-process the dictionary: for each word, generate L wildcard patterns (replace each character with '*'). Group words by pattern. Now finding neighbors is L dict lookups — O(L²) per word vs O(N·L)." This shows you can trade preprocessing space for query speed.

Step 4: Mention Bidirectional BFS as Follow-up

If asked about scaling: "BFS searches a sphere of radius d from the start. Bidirectional BFS searches two spheres of radius d/2 — one from start, one from end — meeting in the middle. This squares the reduction in search space: O(b^d) → O(b^(d/2))."

🗣️ What to Say to the Interviewer

"This is a shortest path problem — words are nodes, and two words connect if they differ by exactly one letter. Equal edge weights means BFS.
The key challenge is neighbor enumeration. I'll preprocess the dictionary into wildcard buckets: replace each character position with '*' and group matching words. Then for any word, I look up L patterns to find all 1-letter neighbors in O(L²).
BFS guarantees the first time I reach endWord is via the shortest path. I'll mark words visited when enqueuing to prevent duplicates.
Time: O(N·L²). Space: O(N·L). Early exit if endWord not in wordList."
🔮 Likely Follow-up Questions
  • "Return all shortest paths?" → Word Ladder II (#126): BFS to build parent pointers at each level, DFS to extract paths
  • "What if transformations can change 2 letters?" → Generate all (L choose 2) pairs of wildcard positions per word
  • "N = 10⁶ words?" → Bidirectional BFS dramatically reduces search space
  • "DNA mutations?" → Minimum Genetic Mutation (#433) — exact same algorithm, alphabet = {A,C,G,T}

7 Summary & Key Takeaways ▾
Model the problem as shortest path on an implicit graph — "minimum steps" signals BFS
Wildcard buckets reduce neighbor-finding from O(N·L) to O(L²) per node
Mark nodes visited when enqueuing (not dequeuing) to prevent duplicate queue entries
endWord not in wordList → return 0 immediately; saves the entire BFS
Use deque.popleft() for O(1) BFS — never list.pop(0) which is O(N)
Bidirectional BFS is the natural extension for extreme scale
