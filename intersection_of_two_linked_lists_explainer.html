LeetCode #160

Intersection of Two Linked Lists

Given the heads of two singly linked-lists headA and headB, return the node at which the two lists intersect. Return null if the two linked lists have no intersection.

Easy Linked List Two Pointers (Switch Heads) Time: O(m+n) Space: O(1)
1 Understanding the Problem ▾

Two singly linked lists can share a tail — like two rivers merging into one. At some point, both lists flow through the exact same sequence of nodes down to the end. We want to find where they first join.

💡 Key Insight: Reference Equality, Not Value Equality

"Intersection" means the same physical node object in memory — not a node with the same value. Two nodes that both contain 8 but live in separate lists are NOT an intersection. We check pointer identity (a is b in Python), not a.val == b.val.

A: 4→ 1→ 8→ 4→ 5→ null
B: 5→ 6→ 1→ 8→ 4→ 5→ null

Purple = same physical nodes in memory. Answer: the node with value 8.

Why Lengths May Differ

List A: 5 total nodes (2 private + 3 shared). List B: 6 total nodes (3 private + 3 shared). Both counts include the shared tail — so they can differ by however many private prefix nodes each list has.

✗ No-Intersection Case

A: 2 → 6 → 4 → null
B: 1 → 5 → null
They never merge. Return null. The algorithm elegantly handles this: both pointers reach null simultaneously after m+n total steps.

Edge cases: empty list (return null), single-node lists that happen to be the same node, the entire list A equals list B (everything is "shared"), and no intersection at all.

2 The Algorithm: Two-Pointer Switch Heads ▾

Place one pointer at each list head. Advance both one step per iteration. When a pointer falls off the end (reaches null), redirect it to the head of the other list. Keep going until the two pointers point to the same node — that's the intersection. If there's no intersection, both pointers reach null simultaneously and we return null.

1 Initialize: a = headA, b = headB. Both pointers start at their respective list heads.
2 Loop while not equal: While a != b, advance both pointers. Equality check uses Python's is semantics (pointer identity), not value comparison.
3 Redirect on null: a = a.next if a else headB — if a is null, jump to headB; otherwise just advance. Same logic for b. Each pointer crosses exactly once.
4 Return: When a == b, return a. This is the intersection node, or null if they met at the end with no shared node.
🔑 The Core Insight: Equal Total Distance Synchronizes Pointers

Let m = length of A, n = length of B, c = length of shared tail. After the redirect, pointer A travels (m − c) private A nodes + redirect + (n − c) private B nodes = m+n−c steps to reach the intersection. Pointer B does the same total. They're perfectly synchronized at that point and land on the exact same node.

Alternative — Hash Set: Walk all of A, storing each node reference in a set. Then walk B until you find a node already in the set. O(m+n) time, O(m) space. Easy to implement but uses memory. The two-pointer trick does it in O(1) space.

3 Why It Works: The Invariant ▾
🔐 The Invariant: Both Pointers Travel Identical Total Distance

Denote: m = length of A, n = length of B, c = shared suffix length (0 if no intersection).

  • Pointer A's path: A-private (m−c nodes) → redirect → B-private (n−c nodes) → arrives at intersection. Total: m − c + n − c = m + n − 2c steps before intersection, + c more after = m+n−c steps total to reach intersection.
  • Pointer B's path: B-private (n−c nodes) → redirect → A-private (m−c nodes) → arrives at intersection. Total: same arithmetic → m+n−c steps total.

Both reach the intersection node at exactly the same step. They're synchronized — they land on the same node simultaneously.

No-intersection (c = 0): Both pointers walk m+n total steps (all of A then all of B for pointer A; all of B then all of A for pointer B). After m+n steps, both are at null. The loop exits because null == null (both are the same Python None object).

⚠️ Common Trap: Redirecting at Last Node vs. At Null

The redirect must happen when the pointer is null (already fell off the end) — not when it's at the last node. The correct expression is a.next if a else headB: if a is currently null, redirect to headB; otherwise move to a.next. If you check a.next instead of a, you redirect one step too early and the pointers never sync correctly.

Why not compare values? Two separate nodes could have the same value 8 but be different nodes. Python's == on objects defaults to identity comparison (same as is) for custom objects without __eq__, which is exactly what we want — but be aware of this if working in languages where == might compare struct fields.

4 The Code (Annotated) ▾

Solution 1 — Two-Pointer Switch Heads (Optimal, O(m+n) time, O(1) space)

def getIntersectionNode(headA, headB):
a, b = headA, headB ← Two runners, each at their own list head
while a != b: ← Loop until same node, or both null
a = a.next if a else headB ← Advance a; if a is null, redirect to headB
b = b.next if b else headA ← Advance b; if b is null, redirect to headA
return a ← Intersection node, or None if no intersection
⚠️ Why "if a else headB" — not "if a.next else headB"?

When a is at the last node, a.next is None — if you redirect here, you skip advancing past the last node and jump one position early. The redirect should happen only after a has already become None. The ternary a.next if a else headB means: "if a is currently not null, give me a.next; otherwise give me headB."

Solution 2 — Hash Set (Brute Force, O(m+n) time, O(m) space)

def getIntersectionNode(headA, headB):
visited = set() ← Store node references (memory addresses) from A
node = headA
while node: ← Walk all of A, recording each node
visited.add(node) ← Python set uses id(node) for hash → identity check
node = node.next
node = headB
while node: ← Walk B, check if we've seen this node in A
if node in visited: return node ← First B node that was also in A → intersection
node = node.next
return None ← Finished B without a match → no intersection
💡 Complexity Summary

Two-Pointer Switch Heads: Time O(m+n), Space O(1) — interview pick
Hash Set: Time O(m+n), Space O(m) — clearer but wastes memory
Length Difference: Time O(m+n), Space O(1) — equivalent but 2× the code

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Clarify "Intersection" Means Reference, Not Value

Say: "Just to confirm — we're looking for the same node object in memory, not just a node with the same value, right?" This shows you understand the subtlety and prevents a wrong approach. Also ask if you can modify the lists (usually: no).

Step 2: Propose Hash Set as Brute Force

"The simple approach: store all of A's nodes by reference in a hash set, then scan B until I find a node in the set. O(m+n) time, O(m) space. Correct, but uses O(m) extra memory." This shows you can solve it — now offer to do better.

Step 3: Identify the Root Problem

"The challenge with two pointers is that the lists may be different lengths, so they'd hit the shared section at different times — they'd be out of sync. If I could align them, one step at a time they'd land on the same shared node simultaneously."

Step 4: Arrive at the Switch Trick

"One way to equalize: after pointer A finishes A, continue it into B; after B finishes B, continue it into A. Both now travel exactly m+n total steps. The length-difference offset cancels out — they'll meet at the intersection on the second pass through, or both hit null simultaneously."

🗣️ What to Say to the Interviewer

"The core challenge: lists can be different lengths, so naive two pointers go out of sync.
Key insight: if I redirect each pointer to the other head when it hits null, both pointers
travel exactly m+n steps before meeting. The offset disappears — they land on the
intersection node simultaneously, or both land on null if no intersection.
Three lines of actual logic. Time O(m+n), Space O(1)."
🔮 Likely Follow-up Questions
  • "What if there's a cycle in one of the lists?" → Different problem (Floyd's cycle detection #142); the switch-head trick assumes acyclic lists.
  • "Can you do it by computing lengths?" → Yes: find both lengths, advance the longer list's pointer by the difference, then step in lockstep until equal. Same O(m+n) time, O(1) space but more code.
  • "What if nodes have duplicate values?" → Doesn't matter. We compare by reference, not value.
  • "Find the length of the intersection?" → Once you have the intersection node, count remaining nodes to null.

Redirect each pointer to the other list's head when it hits null — total travel equalizes to m+n for both
Intersection is reference equality (same node object), not value equality
No-intersection is free: both pointers land on null at the same time after m+n steps
The entire solution is 5 lines — 3 of which are the setup and return
