LeetCode #1293

Shortest Path in a Grid with Obstacles Elimination

Find the minimum steps from top-left to bottom-right of an m×n grid, allowed to eliminate at most k obstacles along the way.

Hard Graph BFS with State Time: O(m·n·k) Space: O(m·n·k)
1 Understanding the Problem ▾

You start at the top-left cell (0, 0) of an m × n grid. Each cell is either empty (0) or an obstacle (1). You want to reach the bottom-right cell (m-1, n-1) using the fewest steps. At each step you move in one of four directions (up, down, left, right). You have a special budget: you can walk through at most k obstacles as if they were open cells.

💡 The Core Insight — State Expansion

Standard BFS tracks which cell you've been to. Here, reaching the same cell with different remaining elimination budgets creates genuinely different situations — one might let you find a shortcut the other can't. So the BFS state must be (row, col, k_remaining), not just (row, col).

Edge Cases:

  • If k ≥ m + n - 2: the Manhattan distance path has at most m+n-3 cells (excluding start/end); we can eliminate all of them. Answer = m + n - 2.
  • If the grid is 1×1: you're already at the destination. Return 0.
  • If the destination is unreachable even with k eliminations: return -1.
2 The Algorithm: BFS over a 3D State Space ▾

We reshape this problem into a standard unweighted shortest-path problem — just with a bigger "graph." Each node is a triple (row, col, k_remaining). Edges connect adjacent nodes where moving to a free cell keeps k unchanged, and moving into an obstacle decrements k by 1 (and is only allowed if k ≥ 1).

1 Manhattan shortcut: If k ≥ m + n - 2, return m + n - 2 immediately. We can always blast straight through every obstacle on the direct diagonal path.
2 Initialize BFS: Create a queue with the starting state (steps=0, r=0, c=0, k_remaining=k). Create a visited set containing the initial state (0, 0, k).
3 BFS loop: Dequeue the front state. If (r, c) is the destination, return the step count — BFS guarantees this is minimal. Otherwise, explore all 4 neighbors.
4 For each in-bounds neighbor (nr, nc): Compute new_k = remaining - grid[nr][nc]. If new_k ≥ 0 and the state (nr, nc, new_k) has not been visited, mark it visited and enqueue (steps+1, nr, nc, new_k).
5 Return -1 if the queue empties without reaching the destination — no valid path exists.
🔑 Why BFS, Not DFS or Dijkstra?

Every step has the same cost (exactly 1). Using an obstacle elimination doesn't add distance — it only changes your remaining budget. Since all edges are weight-1, BFS naturally processes states in non-decreasing order of steps. The first time the destination state is dequeued, it was reached via the fewest steps. Dijkstra would work but is slower (O(m·n·k·log(m·n·k))). DFS has no shortest-path guarantee.

3 Why It Works: The Invariant ▾
🔐 The BFS Level-Order Invariant

BFS processes states in strict non-decreasing order of distance from the source. All states at distance d are processed before any state at distance d+1. Therefore, the first time the goal state (m-1, n-1, any k) is dequeued, it was reached via the minimum number of steps — there is no shorter path not yet considered.

Why the 3D visited set is correct: If we have already enqueued state (r, c, k_rem=5) and later would enqueue (r, c, k_rem=3), we must allow the second one — the k=3 path may have taken a different route with different future options. However, if we've already enqueued (r, c, k_rem=3) and later see (r, c, k_rem=3) again, that is a true duplicate — same position, same budget, arrived in the same or more steps (BFS order) — skip it.

Why we mark visited BEFORE enqueueing: If we only mark on dequeue, the same state may be added to the queue multiple times. For a state with many predecessors, this could blow up queue size exponentially. Mark on enqueue to ensure each (r, c, k) is queued at most once.

⚠️ Most Common Bug

Using (r, c) — without k — as the visited key. This causes the algorithm to prematurely block valid paths. Example: if you visit (2,3) with k=0 first (found via a long detour), you block a later visit with k=2, which could have used its budget to reach the goal much faster via a straight line. Always include k in the visited state.

Complexity proof: The total number of distinct states is m × n × (k+1). Each state is enqueued and dequeued at most once. Each dequeue examines 4 neighbors. Total operations: O(4 × m × n × k) = O(m·n·k).

4 The Code (Annotated) ▾

Solution 1 — BFS with 3D State (Optimal, Interview Pick)

from collections import deque
def shortestPath(grid, k): ← grid is m×n; k = max eliminations allowed
m, n = len(grid), len(grid[0])
if k >= m + n - 2: ← Shortcut: can blast straight through Manhattan path
return m + n - 2 ← Minimum possible steps; answer in O(1)
# state = (steps, row, col, k_remaining)
queue = deque([(0, 0, 0, k)]) ← Start: 0 steps, at (0,0), k budget left
visited = {(0, 0, k)} ← Mark start state visited immediately
while queue:
steps, r, c, remaining = queue.popleft() ← FIFO: popleft() ensures BFS level-order
if r == m - 1 and c == n - 1: ← Reached bottom-right? First dequeue = shortest
return steps
for dr, dc in [(0,1),(0,-1),(1,0),(-1,0)]: ← 4 directions: right, left, down, up
nr, nc = r + dr, c + dc
if 0 <= nr < m and 0 <= nc < n: ← Bounds check before accessing grid
new_k = remaining - grid[nr][nc] ← grid=0→k unchanged; grid=1→k decrements by 1
if new_k >= 0 and (nr, nc, new_k) not in visited: ← Can afford it and state is unseen
visited.add((nr, nc, new_k)) ← Mark before enqueue to prevent queue duplicates
queue.append((steps + 1, nr, nc, new_k)) ← One more step to reach (nr, nc)
return -1 ← Queue exhausted — destination unreachable
⚠️ The Elegant One-Liner: new_k = remaining - grid[nr][nc]

Since grid[nr][nc] is exactly 0 (open) or 1 (obstacle): moving into an open cell subtracts 0 (budget unchanged); moving into an obstacle subtracts 1 (uses one elimination). If the result is negative, we can't afford it. This eliminates the need for an if/else — a single subtraction handles both cases.

Solution 2 — Brute Force DFS with Backtracking (conceptual only)

def shortestPath_dfs(grid, k): ← Explores all paths — exponential time
m, n = len(grid), len(grid[0])
best = [float('inf')]
def dfs(r, c, steps, remaining, seen):
if r == m-1 and c == n-1: ← Found destination — check if shortest
best[0] = min(best[0], steps); return
if steps >= best[0]: return ← Prune: can't beat current best
for dr, dc in [(0,1),(0,-1),(1,0),(-1,0)]:
nr, nc = r+dr, c+dc
if 0<=nr<m and 0<=nc<n and (nr,nc) not in seen:
new_k = remaining - grid[nr][nc]
if new_k >= 0: ← Can afford the move
seen.add((nr,nc))
dfs(nr, nc, steps+1, new_k, seen)
seen.remove((nr,nc)) ← Backtrack — allow revisit in other paths
dfs(0, 0, 0, k, {(0,0)})
return best[0] if best[0] != float('inf') else -1
💡 Complexity at a Glance

BFS with 3D State: Time O(m·n·k), Space O(m·n·k) — each state visited at most once
DFS Brute Force: Exponential — visits same cell many times from different directions
A* (heuristic BFS): O(m·n·k·log) — faster in practice, more complex to implement

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: "Shortest path on a grid" → BFS reflex

Say this out loud: "Minimum steps on a grid with unit cost → BFS." This pattern recognition is the first unlock. You don't need Dijkstra because every step costs exactly 1, regardless of whether an obstacle is eliminated.

Step 2: What makes two paths different?

Ask: "Can two paths reach the same cell in different situations that affect what I can do next?" Here, yes — same cell, different remaining k. So (row, col) is not enough. The state must capture (row, col, k_remaining).

Step 3: Mention the Manhattan shortcut

Before coding, say: "If k ≥ m+n-2, we can eliminate every obstacle on the direct path — answer is always m+n-2." This edge-case catch is easy points and shows engineering maturity.

Step 4: State complexity clearly and confidently

The state space is m rows × n columns × (k+1) possible remaining budgets = O(m·n·k) states. BFS visits each at most once. Time and space are both O(m·n·k).

🗣️ What to Say to the Interviewer

"Minimum steps on a grid = BFS. The twist is the elimination budget k.
Standard BFS only tracks (row, col), but here the same cell with different
remaining k represents genuinely different situations — so I'll expand
the state to (row, col, k_remaining) and run BFS over this 3D space.

Edge case first: if k ≥ m+n-2, answer is just m+n-2.
Otherwise: BFS, mark (r,c,k) visited on enqueue.
new_k = remaining - grid[nr][nc] handles obstacle/empty elegantly.
First time we dequeue the goal = shortest path.

🔮 Likely Follow-up Questions
  • "What if obstacles have different weights to eliminate?" → Use Dijkstra with a priority queue (weighted edge = elimination cost).
  • "Can you reduce memory?" → Use a 3D boolean array instead of a set for O(1) lookups; same asymptotic space.
  • "What if you can regenerate k over time?" → Add regeneration as another state dimension or use Bellman-Ford.
  • "Could A* be faster?" → Yes, in practice; Manhattan heuristic admissible. But BFS is simpler and correct for this.

7 Summary & Key Takeaways ▾
Minimum steps on a grid = BFS (reflexive pattern recognition)
When a resource changes path optimality, include it in the BFS state
State is (row, col, k_remaining) — the 3D space is the key innovation
Manhattan shortcut: if k ≥ m+n-2, answer is m+n-2 (handle in O(1))
new_k = remaining - grid[nr][nc] handles obstacle/empty in one elegant line
Mark state visited on enqueue (not dequeue) to prevent duplicate queue entries
