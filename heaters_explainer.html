LeetCode #475

Heaters

Given house positions and heater positions on a number line, find the minimum radius so every house is covered by at least one heater.

Medium Binary Search Binary Search Closest Time: O((m+n) log n) Space: O(1)
1 Understanding the Problem ▾

Imagine houses and heaters scattered along a single road (a number line). Every heater warms all houses within distance r in both directions. We want the smallest possible radius r such that no house is left cold.

💡 Key Insight

For each house, find its nearest heater. That distance is the minimum radius a heater needs to reach that house. The global answer is the maximum of all per-house minimum distances — because r must satisfy the hardest house, and that's the one furthest from any heater.

✓ Correct Framing

Per house: dist to nearest heater
Answer: max of all per-house dists
→ radius covers the worst house

✗ Common Confusion

Don't use dist to any heater
Don't take the min of min-dists
→ must use max over all houses

Edge cases: house exactly at a heater (dist 0), heaters entirely to one side, single heater covering all, heaters and houses unsorted (sort heaters first). Positions can be very large (up to 109).

2 The Algorithm: Sort + Binary Search Closest ▾

The brute force scans all heaters for each house — O(m×n). The key optimization: sort heaters once, then for each house use binary search (O(log n)) to find the exact boundary where heaters transition from smaller to larger than the house. The nearest heater is always one of the two at that boundary.

1 Sort heaters: O(n log n) upfront. This is the investment that enables fast neighbor lookup for every house.
2 Binary search per house: bisect_left(heaters, house) returns index pos — the first heater ≥ house. The right neighbor is at heaters[pos]; the left neighbor is at heaters[pos-1].
3 Compute distances to both neighbors: right_dist = heaters[pos] - house (if pos < n), left_dist = house - heaters[pos-1] (if pos > 0). Use float('inf') when a neighbor is out of bounds.
4 Update global max: The nearest heater distance for this house is min(left_dist, right_dist). Update ans = max(ans, nearest_dist). After all houses, ans is the answer.
🔑 Why bisect_left gives the right neighbors

bisect_left(sorted_arr, x) returns the leftmost index where x can be inserted and keep the array sorted. This means arr[pos] is the first element ≥ x (right neighbor) and arr[pos-1] is the last element < x (left neighbor). One of these two is always the nearest element to x in a sorted array.

Alternative — Binary Search on the Answer: Binary search the radius r from 0 to max_position. For each candidate r, check if all houses are covered in O((m+n) log n). This is O((m+n) log V) where V ≈ 2×109 — about 31 outer iterations, each O((m+n) log n). Correct, but slightly slower than the direct approach.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every iteration, ans holds the minimum radius needed to cover all houses processed so far. For each processed house, the nearest heater has been identified and its distance recorded. ans is the maximum of these distances — the radius that covers the hardest house seen so far.

Why the max is the answer: All heaters share one radius r. If house A needs r ≥ 3 and house B needs r ≥ 5, we need r ≥ 5. Setting r = max over all houses is both necessary (smaller r leaves some house uncovered) and sufficient (every house's nearest heater is within that max distance).

Why binary search finds the nearest neighbor: After sorting, the nearest heater to any position x is always either the largest heater ≤ x or the smallest heater ≥ x. These are exactly the two elements at the bisect_left boundary. Checking both and taking the minimum is provably optimal — no other heater can be closer.

⚠️ Must Check Both Neighbors — Not Just One Side

A common mistake: only checking the right neighbor (the one found at pos). But for house=3 with heaters=[1,4]: right neighbor 4 is distance 1, left neighbor 1 is distance 2. If you only checked the left, you'd report 2 instead of the correct 1. Always compute both and take min.

4 The Code (Annotated) ▾

Solution 1 — Sort + Binary Search (Optimal, Interview Pick)

from bisect import bisect_left
def findRadius(houses, heaters): ← both arrays may be unsorted
heaters.sort() ← sort once: O(n log n); enables binary search
ans = 0 ← running max of per-house min-distances
for house in houses: ← process each house: m iterations total
pos = bisect_left(heaters, house) ← index of first heater >= house; O(log n)
right = heaters[pos] - house if pos < len(heaters) else float('inf') ← right neighbor dist
left = house - heaters[pos-1] if pos > 0 else float('inf') ← left neighbor dist
ans = max(ans, min(right, left)) ← nearest heater dist; update global max
return ans ← minimum radius covering all houses
⚠️ Why float('inf') as default distance?

When pos == 0, no heater is to the left of the house. Using float('inf') ensures that min(right, inf) = right — only the valid right neighbor matters. Similarly when pos == len(heaters) (house is rightmost), min(inf, left) = left. The sentinel handles both boundary cases without any if-guard.

Solution 2 — Brute Force O(m×n)

def findRadius_brute(houses, heaters):
ans = 0
for house in houses: ← O(m) houses
nearest = min(abs(house - h) for h in heaters) ← O(n) scan per house
ans = max(ans, nearest) ← correct but O(m*n) total — TLE at m,n=25000
return ans
💡 Complexity Summary

Brute Force: Time O(m×n), Space O(1) — TLE for m,n ≈ 25,000
Sort + Binary Search (optimal): Time O((m+n) log n), Space O(1) or O(n) for sort buffer
BS on Answer: Time O((m+n) × log(2×109)) ≈ O((m+n)×31), Space O(1)

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the minimax structure

The phrase "minimum radius such that all houses are covered" is a minimax signal. Translate: for each house, compute the minimum radius needed (distance to nearest heater), then take the max. This framing immediately clarifies the problem.

Step 2: Propose brute force

"Scan all heaters for each house, take the min distance, track the max. O(m×n). With m, n up to 25,000, that's 625M operations — TLE."

Step 3: Spot the binary search opportunity

"The bottleneck is finding the nearest heater. If heaters are sorted, I can use binary search to find the two neighbors in O(log n). Sort once upfront — then O(log n) per house. Total: O((m+n) log n)."

Step 4: Explain bisect_left carefully

"bisect_left gives the insertion point — the first element ≥ the house. The right neighbor is at that index, the left neighbor is one before. Check both, take the min distance. Use float('inf') to handle boundary cases gracefully."

🗣️ What to Say to the Interviewer

"For each house I need the distance to its nearest heater. The answer is the max of these.
Brute force: O(m*n). Optimization: sort heaters, binary search per house → O((m+n) log n).
bisect_left gives the insertion point: right neighbor at pos, left neighbor at pos-1.
Check both, take min. Use inf for missing neighbors. Update running max. Return it."
🔮 Likely Follow-up Questions
  • "What if heaters are already sorted?" → Skip sort; O(m log n) total
  • "Binary search on the radius?" → Yes — check feasibility per radius in O((m+n) log n), outer BS is O(log V) iterations
  • "What if houses are in 2D?" → Nearest-neighbor in 2D; k-d tree or priority queue
  • "Minimum number of heaters to cover all houses given radius r?" → Greedy interval covering problem

7 Summary & Key Takeaways ▾
Sort heaters once — enables O(log n) nearest-neighbor lookup per house via bisect_left
bisect_left returns the insertion point: right neighbor at pos, left neighbor at pos-1
Always check BOTH neighbors and take the minimum to find the true nearest heater
Use float('inf') when a neighbor is out of bounds — min() naturally ignores it
Answer = max of all per-house nearest-heater distances (minimax structure)
Pattern generalizes: "cover all items at minimum uniform cost" → sort + binary search closest
