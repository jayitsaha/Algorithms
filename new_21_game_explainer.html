LeetCode #837

New 21 Game

Alice draws numbers randomly from 1 to maxPts, accumulating score until she reaches at least k. Find the probability her final score is at most n.

Medium Dynamic Programming Sliding Window Probability Time: O(n) Space: O(n)
1 Understanding the Problem ▾

Alice plays a card game. She starts with a score of 0 and draws a random integer from 1 to maxPts (each equally likely), adding it to her score. She keeps drawing until her score is at least k, then she stops. We want to find: what is the probability that Alice's final score is at most n?

💡 Key Insight

Once Alice's score crosses k, she stops. But "crossing k" on different draws lands her at different final scores — anywhere from k to k + maxPts - 1. We want the probability she lands in the "good zone" [k, n] rather than the "too high zone" [n+1, k+maxPts-1].

✓ Valid ending: score in [k, n]

n=10, k=1, maxPts=10:
Draw once: score 1–10
All ≤ n=10 → P = 1.0

✗ Bad ending: score > n

n=6, k=1, maxPts=10:
Draw once: scores 7–10 are over n
P(score ≤ 6) = 0.6

Edge cases worth stating early: If k == 0, Alice never draws (already stopped at 0, and 0 ≤ n always). If n >= k + maxPts - 1, even Alice's worst possible draw lands within the limit — in both cases, return 1.0 immediately.

⚠️ Subtle Distinction: Stopping Condition vs Winning Condition

Alice stops when score ≥ k. But we want final score ≤ n. Both thresholds matter. A score of 5 with k=3, n=7 is a valid winner. A score of 8 with n=7 is a loss — even though Alice stopped correctly when hitting k=3, her last draw pushed her too far.

2 The Algorithm: DP with Sliding Window Sum ▾

Define dp[i] as the probability that Alice's score passes through exactly i while she is still actively drawing (score < k). We build this table from left to right. The clever trick: instead of summing maxPts prior values naively (O(n·maxPts)), we maintain a running window sum that slides one step right per iteration — bringing us from O(n·maxPts) to O(n).

1 Edge-case fast return: If k == 0 or n >= k + maxPts - 1, return 1.0. This avoids unnecessary computation and handles degenerate inputs cleanly.
2 Initialize: dp[0] = 1.0 (Alice starts at 0 with certainty). Set wSum = 1.0 (the window sum, initially holding just dp[0]). Set result = 0.0.
3 Fill dp[1] through dp[n]: For each score i, compute dp[i] = wSum / maxPts. Each of the maxPts draw values contributes equally, so we just divide the total predecessor probability by maxPts.
4 Maintain the window: If i < k (still-drawing score), add dp[i] to wSum. This makes it a source for future draws. If i >= k (stopped score), add to result instead (do NOT add to wSum — stopped states can't draw again).
5 Slide the window: If i >= maxPts, subtract dp[i - maxPts] from wSum. The score at i - maxPts can no longer contribute to dp[i+1] or beyond — it's too far back for any single draw of size 1..maxPts to bridge the gap.
📐 The Recurrence Relation
dp[0] = 1.0 ← base case

dp[i] = wSum / maxPts, i ∈ [1, n]

where wSum = Σ dp[j] for j ∈ [max(0, i-maxPts), min(k-1, i-1)]

Maintained as sliding window: after dp[i] computed—
  if i < k → wSum += dp[i] (add new drawing state)
  if i ≥ maxPts → wSum -= dp[i - maxPts] (remove expired state)
3 Why It Works: The DP Invariant ▾
🔐 The Invariant

At each iteration, wSum exactly equals the sum of dp[j] for all j in the window [max(0, i-maxPts), i-1] where j < k. This represents the total probability mass currently in "drawing" states that can contribute one draw to reach score i.

Why is this DP?
Optimal Substructure: The probability of reaching score i depends only on the probability of reaching scores i-maxPts through i-1 (in drawing states). This is an exact recurrence — no approximation.
Overlapping Subproblems: Each dp[j] is used as a predecessor for multiple future scores. Without memoization/tabulation, we'd recompute it for each of those futures.

⚠️ Why Don't We Add dp[i] to wSum When i ≥ k?

When Alice reaches score i ≥ k, she stops drawing immediately. A stopped score cannot produce future draws — it's a terminal state. If we added it to wSum, we'd be pretending Alice could draw more cards from a stopped position, which would overcount probabilities and give wrong answers.

Why divide by maxPts? From any drawing state j (where j < k), Alice draws a random integer from 1 to maxPts. Each value is equally probable at 1/maxPts. So the contribution from state j to state i = j + draw is dp[j] × (1/maxPts). Summing all valid j values gives dp[i] = wSum / maxPts.

Proof the sliding window is correct: Score j can contribute to score i only if i - j ∈ [1, maxPts], i.e., j ∈ [i-maxPts, i-1]. Also, j must be a drawing state (j < k). As i increases by 1, the window shifts right by 1: we add the new left bound dp[i-1] (if i-1 < k) and remove the one that's fallen off the left edge dp[i-maxPts-1].

4 The Code (Annotated) ▾

Solution 1 — Top-Down Memoization (intuitive, O(n·maxPts) naive or O(n) with prefix)

from functools import lru_cache
def new21Game_memo(n, k, maxPts):
@lru_cache(maxsize=None)
def prob(score): ← P(final ≤ n | currently at score)
if score >= k: ← Stopped! Check if within limit
return 1.0 if score <= n else 0.0
# Still drawing: try all maxPts draws
return sum(prob(score + d) for d in range(1, maxPts+1)) / maxPts← O(maxPts) per call
return prob(0) ← Start from 0

Solution 2 — Bottom-Up DP with Sliding Window (Optimal, O(n) time) — Interview Pick

dp[0] = 1.0 ← Base case: start at 0 with certainty
wSum = 1.0 ← Sliding window sum (initially only dp[0])
result = 0.0
for i in range(1, n + 1): ← Fill dp left to right
dp[i] = wSum / maxPts ← Each predecessor contributes 1/maxPts of its probability

wSum += dp[i]
else: ← Stopped score in [k,n]: valid final outcome
result += dp[i]
if i >= maxPts: ← Slide window: score i-maxPts is now unreachable from i+1
wSum -= dp[i - maxPts]
return result ← Sum of dp[k..n] = P(final score ≤ n)
⚠️ Common Bug: Wrong Edge Condition

Some write n >= k + maxPts instead of n >= k + maxPts - 1. The maximum final score is (k-1) + maxPts — Alice was at score k-1 (last drawing state), drew the maximum value maxPts. So the edge condition needs n >= k-1 + maxPts = k + maxPts - 1.

💡 Complexity Summary

Memoization (naive): Time O(n·maxPts), Space O(n) — inner sum loop dominates
Sliding Window DP: Time O(n), Space O(n) — wSum makes each iteration O(1)
The transformation from O(n·maxPts) to O(n) is the classic "prefix sum" optimization applied to a probability recurrence.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify the State Space

"The game has a natural score-based state. The probability of reaching each score depends only on probabilities of previous scores. That's a DP signal — define dp[i] = P(Alice's score passes through i while still drawing)."

Step 2: Write the Naive Recurrence

"dp[i] = (1/maxPts) × sum(dp[j] for j in [i-maxPts, i-1] where j < k). Each drawing state j can produce score i by drawing (i-j). The 1/maxPts is because all draws are equally likely."

Step 3: Recognize the Sliding Window

"The sum of the last maxPts dp values moves exactly one step forward per i. That's a sliding window! Maintain wSum with +/- operations instead of re-summing. O(n·maxPts) → O(n)."

Step 4: Be Careful About Which States Enter the Window

"Only drawing states (j < k) enter wSum. Stopped states (j ≥ k) are terminal — they can't produce further draws, so dp[j ≥ k] must never enter the window."

🗣️ What to Say to the Interviewer

"Let me handle edge cases first: k=0 or n ≥ k+maxPts-1 → return 1.0.
For the general case, I'll use DP. dp[i] = probability of passing through score i while still drawing.
Base: dp[0] = 1. Recurrence: dp[i] = (sum of dp[j] for drawing predecessors) / maxPts.
The sum is over a sliding window of size maxPts — I can maintain it in O(1) per step.
Key invariant: only drawing states (i < k) enter the window; stopped states don't.
Final answer: sum(dp[k..n]). Time O(n), Space O(n)."
🔮 Likely Follow-up Questions
  • "Can you do it in O(1) space?" → Hard — we need dp[i-maxPts] to shrink wSum. Can't easily use a rolling window without storing maxPts values.
  • "What if draws aren't uniform?" → Must weight each predecessor by its draw probability. The simple wSum shortcut breaks.
  • "What if draws are without replacement?" → Becomes a different problem; state space needs to track which cards remain.
  • Related: Knight Probability in Chessboard (#688) — Same idea: DP where each state's probability is the average of reachable predecessor states.

7 Summary & Key Takeaways ▾
dp[i] = probability of Alice's score passing through exactly i while still drawing
The recurrence dp[i] = wSum / maxPts uses a sliding window over the last maxPts drawing states
Stopped states (i ≥ k) never enter wSum — they are terminal and cannot produce draws
The sliding window removes expired predecessors (i - maxPts) that can no longer contribute
Final answer = sum of dp[k..n]; don't forget edge cases k=0 or n ≥ k+maxPts-1 → 1.0
ApproachTimeSpaceNotes
Memoization (no window) O(n·maxPts) O(n) Correct but too slow; TLE on large inputs

🎯 When to Recognize This Pattern: Sliding Window Probability DP
  • "Probability of reaching target X" with uniform random jumps of size 1..k
  • dp[i] depends on the sum of dp[i-k .. i-1] — identical window structure
  • Stopping condition creates boundary: states below it contribute to future draws, states above it do not
  • The optimization is always: sum-over-window → sliding-window-sum
