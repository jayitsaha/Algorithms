LeetCode #986

Interval List Intersections

Given two sorted lists of closed intervals, return all intervals that belong to both lists — their intersection.

Medium Intervals Two Pointers Time: O(m+n) Space: O(k) output
1 Understanding the Problem ▾

Imagine two people's shared calendar — Alice's busy slots and Bob's busy slots — each listed as sorted, non-overlapping intervals. You want to find every time window when both are simultaneously busy. That's this problem exactly.

💡 Key Insight

Two closed intervals [a, b] and [c, d] intersect if and only if max(a, c) ≤ min(b, d). The intersection, when it exists, is [max(a,c), min(b,d)] — the tighter bound on each side. After extracting the intersection, advance the pointer whose interval ends first — that interval is exhausted and cannot contribute to future overlaps.

Constraints and edge cases: Either list empty → return []. Lists of different lengths — handled naturally (loop ends when either is exhausted). Touching endpoints like [1,2] ∩ [2,5] = [2,2] — valid, since intervals are closed (endpoints included). The lists are already sorted and guaranteed non-overlapping within each list.

2 The Algorithm: Two-Pointer Linear Merge ▾

Think of merging two sorted sequences — like the merge step in merge sort — but instead of selecting the smaller element, we're checking for overlap. Because both lists are sorted, we never need to look backward: once a pointer advances, that interval is done forever.

1 Initialize: Set pointer i = 0 into firstList (A) and j = 0 into secondList (B). Create an empty result list.
2 Compute candidate intersection: lo = max(A[i][0], B[j][0]) (the later start) and hi = min(A[i][1], B[j][1]) (the earlier end). If lo ≤ hi, record [lo, hi] in results.
3 Advance the pointer that ends first: If A[i][1] ≤ B[j][1], increment i. Otherwise increment j. The interval that ends earlier is "spent" — all future intervals in the other list start no earlier than it, so it can't contribute new overlaps.
4 Repeat until exhausted: Continue while i < len(A) and j < len(B). When either list runs out, remaining intervals in the other list have no partner — stop and return results.
🔑 Why Advance the Earlier-Ending Pointer?

If A[i] ends at position 5 and B[j] ends at 8, then the next interval A[i+1] starts after 5 (since A's intervals don't overlap each other). Any future overlap between A[i+1] and B[j] will be detected in the next iteration with B[j] still in place. Advancing A now is safe — A[i] is fully processed.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every iteration, all intervals A[0..i-1] and B[0..j-1] have been fully processed. Any intersection they could produce has already been recorded. The current pair (A[i], B[j]) is the only pair whose intersection remains to be discovered involving either interval.

Completeness (we miss nothing): Suppose A[i] and B[j+1] overlap. Then B[j+1] starts after B[j] ends. If A[i] overlaps B[j+1] but not B[j] (no intersection with B[j]), then A[i][0] > B[j][1], meaning lo > hi in the current step. We advance j (B ends first since A[i] hasn't even started yet relative to B[j]). We correctly continue to B[j+1] with A[i] still in place and find the overlap. No intersection is skipped.

The intersection formula: Geometrically, [a,b] ∩ [c,d] is the set of points ≥ both start points and ≤ both end points. That's [max(a,c), min(b,d)]. Non-empty iff max(a,c) ≤ min(b,d). This formula works for all four overlap cases shown in Section 1.

⚠️ Common Mistake: Advancing by Start Instead of End

A common error is advancing the pointer with the smaller start instead of the smaller end. This fails when a long interval (small start, large end) overlaps multiple short intervals in the other list. By advancing on end, we guarantee we don't miss any of those overlaps.

⚠️ Common Mistake: Using lo < hi (strict) for Closed Intervals

Closed intervals include their endpoints. [1,3] and [3,7] share the single point 3, giving intersection [3,3]. Using lo < hi (strict) misses this. Always use lo ≤ hi for closed intervals.

4 The Code (Annotated) ▾

Solution 1 — Two Pointers (Optimal, Interview Pick)

def intervalIntersection(firstList, secondList):
A, B = firstList, secondList ← short aliases for readability
i, j, res = 0, 0, [] ← i→A, j→B, res=output accumulator
while i < len(A) and j < len(B):← stop when either list is exhausted
lo = max(A[i][0], B[j][0]) ← intersection start: later of the two starts
hi = min(A[i][1], B[j][1]) ← intersection end: earlier of the two ends
if lo <= hi: ← valid overlap (≤ not < — closed intervals!)
res.append([lo, hi]) ← record intersection
if A[i][1] <= B[j][1]: ← which interval ends first?
i += 1 ← A[i] ends first → advance A's pointer
else:
j += 1 ← B[j] ends first → advance B's pointer
return res ← all k intersections, O(k) output space
💡 Complexity Summary

Two Pointers: Time O(m+n) — each of the m+n intervals is visited exactly once. Space O(k) where k = number of intersections (output only; O(1) auxiliary).
Brute Force: Time O(m·n) — every pair is checked. Space O(1) auxiliary + O(k) output.

Solution 2 — Brute Force (for reference, O(m·n))

def intervalIntersection_brute(A, B):

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Pattern Signal

Two sorted lists + "find what they share" = two-pointer merge. The sorted constraint is the key signal — it's the same insight as merge sort's merge step, but for overlap detection instead of element ordering. Brute force ignores sorted order and wastes O(m·n) — mention this first.

Step 2: Derive the Intersection Formula

Draw [a,b] and [c,d] on a number line. Their shared region must start after both start (take the max of starts) and end before both end (take the min of ends). Write lo = max(a,c), hi = min(b,d). They overlap iff lo ≤ hi. Emphasize ≤ not < for closed intervals.

Step 3: Justify Advancing the Earlier-Ending Pointer

"After recording any overlap, I advance whichever interval ends first — it's done. All future intervals in the other list start no earlier than this point, so the shorter interval can't contribute to new overlaps. This is the greedy decision that keeps us at O(m+n)."

Step 4: State Complexity

Each pointer advances at most m or n times. Each step is O(1). Total: O(m+n). Space: O(1) auxiliary (two pointers, two variables), plus O(k) for the output list which we can't avoid without changing the return type.

🗣️ What to Say to the Interviewer

"Both lists are sorted, so I can use two pointers — one into each list — and merge through them linearly.
At each step: lo = max of starts, hi = min of ends.
If lo ≤ hi, I've found an intersection [lo, hi] — append it.
Then I advance whichever pointer's interval ends first — that interval is spent.
This runs in O(m+n) time and O(1) extra space (plus O(k) for the output)."
🔮 Likely Follow-up Questions
  • "What if the lists aren't sorted?" → Sort first (O(m log m + n log n)), then same algorithm.
  • "What if intervals within a list can overlap?" → Merge each list first (#56), then intersect.
  • "Find free times (gaps) for both people?" → Same structure; collect non-overlapping segments instead.
  • "Three calendars?" → Intersect first two, then intersect result with third list.

Two sorted interval lists → two-pointer linear merge in O(m+n)
Intersection of [a,b] and [c,d] is [max(a,c), min(b,d)] — valid when lo ≤ hi
