LeetCode #1499

Max Value of Equation

Given points sorted by x-coordinate, find max of y_i + y_j + |x_i - x_j| where x_j - x_i ≤ k and i < j.

Hard Monotonic Queue Sliding Window Max Time: O(n) Space: O(n)
1 Understanding the Problem ▾

You are given n points sorted strictly by x-coordinate: points[i] = [xi, yi]. For every valid pair (i, j) where i < j and xj − xi ≤ k, compute yi + yj + |xi − xj| and return the maximum value found across all such pairs.

💡 The Key Algebraic Simplification

Since the points are sorted by x and we only consider i < j, we know xj ≥ xi, so |xi − xj| = xj − xi (no absolute value needed). The expression becomes:

yi + yj + xj − xi = (yi − xi) + (yj + xj)

This splits into two independent parts: one depending only on the left point i, and one only on the right point j. For each j, we just need the maximum (yi − xi) among all valid left points!

Concrete Example: k=4

points = [[1,3],[2,0],[5,10],[6,-10]]
Pair [1,3] & [2,0]: 3+0+1 = 4
Pair [1,3] & [5,10]: 3+10+4 = 17
Pair [2,0] & [5,10]: 0+10+3 = 13
Pair [5,10] & [6,-10]: 10-10+1 = 1
Answer: 17

Same example, reformulated

[1,3]: y−x = 2   y+x = 4
[2,0]: y−x = -2   y+x = 2
[5,10]: y−x = 5   y+x = 15
[6,-10]: y−x = -16   y+x = -4
Best: 2 + 15 = 17 ✓

The sliding window insight: As we scan j from left to right, valid left partners (with xj − xi ≤ k) form a shrinking window. When a point falls outside the window, it can never be a valid partner for j or any future point. This is the classic sliding window maximum problem, solvable with a monotonic deque in O(n).

Edge cases: All y values negative — formula still works, take the best among all valid pairs. Very large k — effectively no distance constraint, just track the global max (y−x). Single pair — return its value directly. Points very close together (xj−xi = 0) — valid, |diff|=0.

2 The Algorithm: Monotonic Deque Optimization ▾

We maintain a monotonically decreasing deque of (y−x, x) pairs. The deque front always holds the maximum (y−x) among all valid left candidates within x-distance k of the current point.

The Core Formula Split
y_i + y_j + x_j - x_i = (y_i - x_i) + (y_j + x_j)
Store left term in deque Compute per point j
1 Initialize: Empty deque dq, answer ans = -infinity. The deque stores (y−x, x) pairs in decreasing order of y−x (front = maximum).
2 Evict stale from front: While dq[0].x is too far left (xj − xi > k), pop from the front. That point can never be a valid left partner for j or any future point (x only increases).
3 Query and update answer: If deque is non-empty, the front holds the current maximum (yi−xi). Compute ans = max(ans, front.val + yj + xj).
4 Evict dominated from back: While dq[-1].val ≤ yj − xj, pop from the back. The back entry is dominated: current j has equal or higher score AND a larger x (stays valid longer). Back entry can never be optimal.
5 Insert current point: Push (yj−xj, xj) onto the back. Continue to next point.
🔑 Why the Deque Is Monotonically Decreasing

We want the deque front to always be the maximum y−x among valid points. When we add a new point with value vnew, any older back entry with value vold ≤ vnew is dominated: it contributes less AND expires sooner (smaller x). Removing it is permanently safe — whenever it would have been chosen, vnew would be chosen instead. This strict dominance maintains correctness.

3 Why It Works: The Invariant ▾
🔐 The Deque Invariant

At the start of processing point j, the deque contains exactly the "useful" subset of all previous points: (1) within x-distance k of j (valid for pairing), and (2) not dominated by any later point in the deque. The front always holds the maximum (y−x) among all currently valid left candidates.

Front eviction proof: The input is sorted by x, so xj ≥ xj-1. If xj − xi > k for the front entry, then for any future j' (xj' ≥ xj), xj' − xi will be even larger. The front can never be a valid partner again. Eviction is safe and permanent.

Back eviction proof: Suppose back entry B has value vB and current point C has value vC ≥ vB. For any future right partner j': C contributes vC ≥ vB to the equation, AND C stays valid at least as long as B (since xC ≥ xB). So whenever B would be chosen as a left partner for j', C would be chosen instead with an equal or better result. B is permanently useless.

⚠️ Critical: Update Answer Before Inserting Current Point

Always: evict front → query front → evict back → insert current. If you insert before querying, you'd pair point j with itself (same point), which is invalid. The problem requires i ≠ j with i < j. The current point must not be in the deque when you query it for the current j's answer.

Why O(n) total time: Each of the n points is appended to the deque exactly once. Each point is also removed at most once (either from front for staleness, or from back for dominance). Total deque operations ≤ 2n = O(n). The outer loop is O(n). Combined: O(n) amortized.

4 The Code (Annotated) ▾

Solution 1 — Monotonic Deque (Optimal, O(n)) — Interview Pick

from collections import deque
def findMaxValueOfEquation(points, k):
dq = deque() ← stores (y−x, x) pairs; front = max y−x among valid points
ans = float('-inf') ← running maximum of the full equation value
for xj, yj in points: ← iterate all points left to right (x already sorted)
while dq and xj - dq[0][1] > k: ← evict front: x-distance exceeds k → permanently stale
dq.popleft() ← O(1) pop from front of deque
if dq: ← if any valid left partner exists in window...
ans = max(ans, dq[0][0] + yj + xj) ← (y_i−x_i) + (y_j+x_j) = full equation value
while dq and dq[-1][0] <= yj - xj: ← evict back: dominated by current point
dq.pop() ← current has higher score + won't expire before the back
dq.append((yj - xj, xj)) ← insert current point as a candidate for future right partners
return ans ← maximum over all valid (i,j) pairs within k
⚠️ Why store (y−x, x) not just (y−x)?

We need the x-coordinate to check whether a stored point is still within k-distance when a new j arrives. Without x, the front eviction step (xj - dq[0][1] > k) is impossible. The pair stores both the contribution value (y−x) and the "timestamp" (x) needed for validity checks.

Solution 2 — Brute Force (O(n²), for understanding)

def findMaxValueOfEquation_brute(points, k):

💡 Complexity Summary

Monotonic Deque: Time O(n) amortized (each point pushed & popped at most once), Space O(n) for deque
Brute Force: Time O(n²), Space O(1) — TLE for n = 10⁵
Sorted Map / BST: O(n log n) using a sorted container to track max y−x with x-range queries — valid alternative but more code

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Use the Sorted Property

The problem says points are sorted by x. For i < j, xj ≥ xi, so remove the absolute value: |xi−xj| = xj−xi. Rewrite: yi + yj + xj − xi = (yi−xi) + (yj+xj). This algebraic split is the key unlock.

Step 2: Recognize Sliding Window Maximum

For each j, we need max (yi−xi) over all i where xj−xi ≤ k. As j increases, the valid window of i's shifts right (old points expire). We want the running maximum over a sliding window — a textbook monotonic deque application.

Step 3: Code the Deque Template

Monotonic deque for sliding window max: store (value, position), maintain decreasing order. Front eviction = window boundary check, Back eviction = dominance check. The deque front is always the current window maximum. O(n) total.

Step 4: Nail the Order of Operations

For each point j: (1) evict stale from front, (2) query front for best left partner, (3) evict dominated from back, (4) push current. Never swap steps 2 and 4 or you'll pair a point with itself.

🗣️ What to Say to the Interviewer

"Points are sorted by x, so for i < j, the absolute value disappears.
Rearranging: (y_i - x_i) + (y_j + x_j). For each j, I need max (y_i - x_i)
over all i with x_j - x_i ≤ k — that's sliding window maximum.
I'll use a monotonic deque storing (y-x, x) in decreasing value order.
Per point: evict stale front, query front for ans, evict dominated back, push current.
Time O(n) amortized — each point pushed and popped at most once."
🔮 Likely Follow-up Questions
  • "What if points weren't sorted by x?" → Sort first: O(n log n) total still optimal.
  • "What if we need the actual pair, not just the max value?" → Store indices in the deque alongside x; track which pair gave the max when updating ans.
  • "What if k = infinity?" → No front eviction needed. Just track running max (y−x) linearly. O(n) one-pass trivial.
  • "Can you do it with a priority queue?" → Yes: max-heap of (y−x, x), but lazy deletion needed when x is out of range → O(n log n). Deque is strictly better.

7 Summary & Key Takeaways ▾
Sorted input lets you remove the absolute value: |xi−xj| = xj−xi for i < j
Algebraic split (yi−xi) + (yj+xj) turns the problem into sliding window maximum
Monotonic deque gives O(1) amortized window max: front eviction for staleness, back eviction for dominance
Always process in order: evict front → query → evict back → insert. Never pair a point with itself.
Total O(n) time: each of n points enters and exits the deque at most once, so 2n deque ops total
🎯 When to Use Monotonic Deque Optimization
  • Objective can be algebraically split into "left contribution" + "right contribution"
  • Left contributions have a sliding validity window (x-distance, time, index constraint)
  • You need the running maximum (or minimum) over that window
  • Input is sorted or naturally processed in order (arrays, time series, sorted events)
