LeetCode #77

Combinations

Given two integers n and k, return all possible combinations of k numbers chosen from the range [1, n].

Medium Backtracking Choose K from N Time: O(k · C(n,k)) Space: O(k)
1 Understanding the Problem ▾

You have the numbers 1 through n. Your task: pick exactly k of them and report every possible selection. Order does not matter — {1,2} and {2,1} are the same combination and should only appear once in the output.

💡 Key Insight

This is a pure combinatorics enumeration problem: generate all C(n,k) subsets of size k from {1, 2, …, n}. The trick is to always choose from numbers greater than or equal to the last chosen number, which automatically prevents duplicates without needing any "visited" bookkeeping.

Edge Cases: k=1 produces n singletons. k=n produces exactly one combination (all numbers). The problem guarantees 1 ≤ k ≤ n ≤ 20. With n=20, k=10, C(20,10) = 184,756 — manageable but not trivial.

2 The Algorithm: Backtracking with a Start Pointer ▾

We use backtracking — a recursive search that builds a partial solution step by step, undoing ("backtracking") each choice after fully exploring it, and cutting branches early when they can't possibly lead to a valid answer.

The central idea: our recursive function receives a start pointer and the current partial combination. We try adding each number from start to n, recurse, then undo. Because we always recurse with start = i + 1, every combination we produce is in strictly increasing order — no visited set needed.

1 Base case: If the current combination has exactly k numbers, save a copy to results and return. This leaf of the recursion tree is a valid answer.
2 Pruning: Compute need = k − len(current) (slots remaining) and limit = n − need + 1 (last valid start). The loop only runs from start to limit. If we started beyond limit, there wouldn't be enough numbers left to fill k slots — wasted work eliminated.
3 Choose → Recurse → Backtrack: For each candidate i in [start, limit]: (a) append i to current, (b) recurse with start = i + 1, (c) pop i off current. This three-step pattern is the backbone of every backtracking solution.

3 Why It Works: The Invariant ▾
🔐 The Core Invariant

At every recursive call, the current combination is in strictly increasing order, and start is always strictly greater than every number already in current. This single invariant simultaneously guarantees: (1) no number is used twice, and (2) no combination appears more than once.

Why no duplicates? Every combination we produce is a strictly increasing sequence — it has a unique canonical form. Two different execution paths that produce the same set would need to produce the same sorted sequence. But since we enforce increasing order structurally (via the start pointer), no two paths can produce the same output.

Why no reuse? We recurse with i + 1, not i. So every future element in the combination must be strictly larger than i. The number i is used at most once per path.

Why does the pruning bound work? If we're at start with need slots remaining, the pool available is [start, start+1, …, n], size n − start + 1. If this pool has fewer than need elements, even taking everything won't fill k slots. So we set limit = n − need + 1 and loop only up to limit — this is baked into the loop range, not a separate if-check.

⚠️ The Two Most Common Bugs

Bug 1 — Forgetting to copy: result.append(current) appends a reference. All stored entries will be the same (empty) list at the end. Always use result.append(current[:]).

Bug 2 — Forgetting to pop: If you don't call current.pop() after recursing, current keeps growing across different branches. The undo step is not optional.

4 The Code (Annotated) ▾

Solution 1 — Backtracking with Pruning (Optimal, Interview Pick)

def combine(n: int, k: int) -> list[list[int]]:
result = [] ← accumulate all valid combinations here
def backtrack(start: int, current: list):
if len(current) == k: ← base case: we've chosen k numbers
result.append(current[:]) ← COPY current — not a reference!
return
need = k - len(current) ← how many more numbers we still need
limit = n - need + 1 ← last valid first number (need numbers must follow)
for i in range(start, limit + 1): ← only try candidates that leave room
current.append(i) ← CHOOSE: add i to the combination
backtrack(i + 1, current) ← RECURSE: next number must be > i
current.pop() ← BACKTRACK: undo the choice
backtrack(1, []) ← start: pick from 1, combo empty
return result
⚠️ Why limit = n - need + 1?

If we pick i and still need need more numbers, those must come from {i+1, i+2, …, n} — a pool of size n − i. We need n − i ≥ need − 1 (the remaining need after picking i), i.e., i ≤ n − need + 1. This is the loop upper bound. Without it, the algorithm still produces correct results — but wastes time on branches that can't complete.

Solution 2 — Itertools One-Liner (Pythonic, not the interview answer)

from itertools import combinations
def combine(n: int, k: int) -> list[list[int]]:
return [list(c) for c in combinations(range(1, n+1), k)] ← same logic, stdlib
💡 Complexity

Time: O(k · C(n,k)) — we build k-element lists for each of C(n,k) combinations.
Space: O(k) call stack depth + O(k · C(n,k)) for the output (which must be stored anyway).
Pruning savings: Without pruning, we'd visit all C(n,0)+C(n,1)+…+C(n,k) nodes; pruning skips branches where the pool runs dry.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize "Enumerate All" → Backtracking

"The problem asks for ALL combinations — enumeration problems where we need every valid answer are the canonical backtracking signal. Greedy/DP find one optimal answer; backtracking finds all of them. Also a dead giveaway: exponential output size means we need exponential time — no way around it."

Step 2: Design the Recursive State

"My state is: start (smallest number I may still pick) and current (what I've picked so far). The start pointer replaces a visited set — if I always pick in increasing order, I never revisit and never produce duplicates."

Step 3: Write the Three-Part Pattern

"Every backtracking solution looks like: (1) base case — save and return; (2) loop over candidates; (3) inside loop: append → recurse → pop. This pattern is identical for Subsets, Permutations, and Combination Sum — memorize the skeleton."

Step 4: Add Pruning (shows mastery)

"The loop bound is range(start, n − need + 2) rather than range(start, n + 1). This prunes branches where the remaining numbers can't fill k slots — reduces constant factor significantly on large inputs."

🗣️ What to Say to the Interviewer

"This asks for ALL combinations of size k from [1,n] — classic backtracking enumeration.
I'll write a recursive function that takes 'start' and builds the combo one number at a time.
Start pointer ensures strictly increasing order → no duplicates, no visited set needed.
Base case: combo has k elements → save a copy and return.
Pruning: loop only up to n-need+1 where need = k - len(current).
Time: O(k · C(n,k)), Space: O(k) stack depth."

7 Summary & Key Takeaways ▾
"Enumerate all" → backtracking. This is the canonical signal — greedy or DP find one answer, backtracking finds all.
The start pointer maintains strictly increasing order, which prevents duplicate combinations with zero overhead (no visited set).
Always append a copy (current[:]) at the base case — appending the live list gives you empty slices when backtracking finishes.
Always pop() after recursing — the undo step is non-negotiable. Shared mutable state must be restored.
Pruning bound: limit = n − need + 1. Bake it directly into the loop range instead of an if-guard — same effect, cleaner code.
The Choose → Recurse → Backtrack skeleton is identical for Combinations, Subsets, Permutations, and Combination Sum.

🎯 When to Use This Pattern
  • "Return all combinations / subsets of size k" → backtracking with start pointer
  • "No repeated numbers in one combination" → recurse with i+1 (not i)
  • "Order doesn't matter" → start pointer enforces canonical order automatically
  • Result set is exponential in size → we need exponential time → backtracking is optimal
