LeetCode #278

First Bad Version

You are a product manager and the latest product has n versions. All versions after the first bad version are also bad. Find the first bad version with minimum API calls.

Easy Binary Search Lower Bound Time: O(log n) Space: O(1)
1 Understanding the Problem ▾

You have n software versions numbered 1 through n. At some point a bug was introduced — call that version bad. The key rule: once a version is bad, every subsequent version is also bad. The version history always looks like: Good, Good, ..., Bad, Bad, ..., Bad.

You're given an API isBadVersion(v) that returns True if version v is bad. Your goal: find the first bad version using as few API calls as possible, because each call is expensive (imagine a network roundtrip or a full build).

💡 Key Insight

The isBadVersion answers form a monotone sequence: F F F F … T T T T. We need to find the leftmost True. A monotone predicate over a sorted range is the exact trigger for Binary Search (Lower Bound variant). Instead of scanning linearly (O(n) calls), binary search finds the boundary in O(log n) calls.

✓ Example: n=8, first bad=5

Version: 1 2 3 4 5 6 7 8
isBad?: F F F F T T T T
Answer: 5 (first True)

⚠ Why linear scan fails

n can be 2,147,483,647
Linear: up to 2B API calls
Binary: only ≤ 31 API calls

Edge cases to consider: version 1 is the first bad (answer = 1), version n is the only bad one (answer = n), n = 1 (only one version, must be bad). The constraint guarantees at least one bad version exists.

2 The Algorithm: Binary Search — Lower Bound ▾

We maintain a search window [lo, hi] that always contains the first bad version. Each iteration we check the midpoint and halve the window.

1 Initialize: Set lo = 1, hi = n. The first bad version is in [1, n] by definition.
2 Compute midpoint: mid = lo + (hi - lo) // 2. This overflow-safe formula avoids integer overflow when lo and hi are both near 2^31.
3 Check isBadVersion(mid): If True → first bad is at mid or to the left. Set hi = mid (keep mid in range — it could be the answer!). If False → first bad is strictly after mid. Set lo = mid + 1 (safely exclude mid).
4 Converge: Loop while lo < hi. When they meet, lo == hi is the only remaining candidate — that's the first bad version. Return lo.

↓

search [lo, mid]
mid may be the answer
isBadVersion(mid) = False
↓
lo = mid + 1
search [mid+1, hi]
mid is definitely good
🔑 Lower Bound Template

This is the "find first True" template. Critical rule: when condition(mid) is True, use hi = mid (not mid - 1), because mid itself might be the answer. The loop exits at lo == hi with a single guaranteed-correct candidate.

3 Why It Works: The Invariant ▾
🔐 The Loop Invariant

At the start of every iteration: the first bad version is guaranteed to be within [lo, hi]. The interval strictly shrinks each iteration. When lo == hi, the invariant tells us that single point is the first bad version.

Proof the invariant is maintained:

  • Base case: lo=1, hi=n. The first bad version is definitely in [1, n]. ✓
  • Inductive step — isBad(mid) = True: The first bad is at mid or earlier. Setting hi = mid preserves the invariant: first bad is in [lo, mid]. ✓
  • Inductive step — isBad(mid) = False: Mid is good, so first bad is strictly after mid. Setting lo = mid+1 preserves the invariant: first bad is in [mid+1, hi]. ✓
  • Termination: Each step reduces hi - lo by at least 1 (since mid < hi when lo < hi). After at most ⌈log₂ n⌉ steps, lo == hi.
⚠️ Classic Bug: hi = mid - 1 when isBad is True

If you write hi = mid - 1 when mid is bad, you exclude mid from the search — but mid might itself be the first bad version! The correct lower bound update is hi = mid. The loop still terminates: when lo == mid (only two candidates), mid-as-hi doesn't advance, but setting lo = mid+1 on a False would exit the loop. When mid is True, hi = mid = lo, so lo < hi becomes False and the loop exits.

Overflow note: When n = 2,147,483,647 (2^31 - 1), lo + hi can overflow a 32-bit signed integer to a negative number. Python integers don't overflow, but in Java/C++ this is a real bug. Always use lo + (hi - lo) // 2.

4 The Code (Annotated) ▾

Solution 1 — Binary Search, Lower Bound (Optimal — Interview Pick)

def firstBadVersion(n: int) -> int:
lo, hi = 1, n ← search the full range [1, n]
while lo < hi: ← while more than one candidate remains
mid = lo + (hi - lo) // 2 ← midpoint, overflow-safe formula
if isBadVersion(mid): ← ask the API — this is the only expensive call
hi = mid ← bad: first bad is ≤ mid; keep mid in range!
else: ← good: first bad is strictly after mid
lo = mid + 1 ← safely exclude mid, it's confirmed good
return lo ← lo == hi, the one remaining candidate is the answer
⚠️ Why return lo (not mid)?

When the loop exits, lo == hi. Both variables point to the same version — the first bad version. We return lo. Never return mid; mid is a local variable computed inside the loop and is out of scope once we exit. Both lo and hi are valid to return (they're equal).

Solution 2 — Linear Scan (Brute Force — O(n) API calls, will TLE)

def firstBadVersion(n: int) -> int:
for v in range(1, n + 1): ← check every version from 1 to n
if isBadVersion(v): ← first True we see is the answer
return v ← up to n calls — TLE when n ~ 2^31
💡 Complexity Summary

Binary Search: Time O(log n) — at most ⌈log₂ n⌉ API calls. Space O(1).
Linear Scan: Time O(n) — up to 2,147,483,647 calls for max n. Space O(1).
For n = 2^31 − 1: binary search needs ≤ 31 calls; linear scan needs up to 2 billion.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify the Monotone Property

"The problem says all versions after the first bad one are bad. So isBadVersion answers are F,F,...,T,T,...,T — a monotone sequence. Monotone predicate + find boundary = Binary Search."

Step 2: Pick the Right Variant

"I need the first True — that's the Lower Bound variant. Key rule: when mid is True, set hi=mid (not mid-1), because mid itself might be the answer. When mid is False, set lo=mid+1 to safely exclude it."

Step 3: Handle the Overflow Concern Proactively

"n can be up to 2^31 − 1 in the constraints. If I write (lo+hi)//2, the sum may overflow in Java/C++. I'll use lo + (hi-lo)//2 instead. Even in Python where there's no overflow, this shows awareness."

Step 4: Minimize API Calls as the Core Goal

"The entire point is that isBadVersion is expensive. Binary search makes only O(log n) calls. For the maximum n, that's 31 calls vs 2 billion. I should mention this explicitly."

🎯 When to Use This Pattern
  • The predicate is monotone: once True, always True (or once False, always False)
  • You need the first position where condition is True (lower bound)
  • Calling the predicate directly is expensive — minimize calls
  • The search space is huge (linear scan would TLE)
