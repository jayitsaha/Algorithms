LeetCode #465

Optimal Account Balancing

Given a list of transactions, return the minimum number of transactions needed to settle all debts.

Hard Dynamic Programming Bitmask DP Time: O(3ⁿ) Space: O(2ⁿ)
1 Understanding the Problem ▾

You have a group of friends who split expenses throughout a trip. Each transaction [from, to, amount] means person from paid person to the given amount. After all is done, some people are owed money (creditors, positive net balance) and some owe money (debtors, negative net balance). The goal: settle all debts using the fewest possible transactions.

💡 Key Insight

The original transactions don't matter — only each person's net balance does. After collapsing everyone to a single "owes/is-owed" number, the problem becomes: partition the non-zero balances into groups that each sum to zero, then minimize the number of payments within those groups (each zero-sum group of size k needs exactly k−1 transactions).

Given Transactions

0→1: $10
1→0: $1
1→2: $5
0→3: $5

Net Balances (what matters)

Person 0: +10 −1 −5 = +4 (creditor)
Person 1: +1 −10 +5 = −4 (debtor)
Person 2: +5 = +5 (creditor)
Person 3: −5 = −5 (debtor)

With net balances [+4, −4, +5, −5]: we can settle 0→1 ($4) and 3→2 ($5) — just 2 transactions. The balances form two zero-sum pairs, so each pair needs exactly 1 transaction.

Edge cases: All balances already zero (answer = 0); only two non-zero people with opposite balances (answer = 1); a single creditor collecting from many debtors (answer = number of debtors); n people with all same magnitude (may pair perfectly).

2 The Algorithm: Bitmask DP on Debt Subsets ▾

After reducing to net balances, we work with a list of non-zero values (at most ~12 people practically). A bitmask represents which subset of these balances we are considering. We precompute for every subset: does it sum to zero? And the key recurrence: the minimum transactions to settle a zero-sum subset of k people is (k − 1).

1 Compute net balances: Build a map from person → net balance by processing all transactions. Then extract only the non-zero balances into a list debts of length n.
2 Define dp[mask]: The minimum number of transactions to settle exactly the subset of people indicated by mask. There are 2ⁿ masks. Only masks whose elements sum to zero are "settleable" independently.
3 Base case: dp[0] = 0 (empty set, no transactions needed). All other masks start at ∞.
4 Enumerate zero-sum sub-masks: For each non-zero mask m, iterate through all non-empty sub-masks s of m. If s is zero-sum (its elements sum to zero), the minimum transactions to settle s is popcount(s) − 1. Update: dp[m] = min(dp[m], dp[m ^ s] + popcount(s) − 1).
5 Answer: dp[(1 << n) − 1] — the minimum transactions to settle everyone.
📐 The Recurrence
dp[mask] = min over all non-empty zero-sum sub-masks s of mask:
    dp[mask ^ s] + popcount(s) − 1

popcount(s) = number of bits set in s = size of the zero-sum group
A zero-sum group of k people settles in exactly k−1 transactions
🔑 Why k−1 transactions for a zero-sum group of size k?

Think of it as a chain: person 1 pays person 2, person 2 pays person 3, ..., person k-1 pays person k. That's k−1 payments and it fully settles all k balances. This is always optimal — you can't do better than k−1 for a group where everyone has a non-zero balance. A group of 2 needs 1 payment, a group of 3 needs 2, etc.

Sub-mask enumeration trick: To enumerate all non-empty sub-masks of mask m, use the classic idiom: for s = m; s > 0; s = (s-1) & m. This visits every subset in O(3ⁿ) total work (each element is either in the outer mask, the sub-mask, or neither — 3 choices per element).

3 Why It Works: The Invariant ▾
🔐 Why is This Dynamic Programming?

Optimal Substructure: The optimal way to settle the full set decomposes into settling independent zero-sum subgroups. Once we find a zero-sum subset s, its settlement is independent of the remaining people — no shared transactions. So the total cost = cost(s) + cost(rest).

Overlapping Subproblems: The same subset mask can appear as a "remaining debt group" in many different decomposition paths. Without memoization (the dp array), we'd recompute the same subset costs exponentially many times.

Why only zero-sum subsets? A group of people can settle among themselves without involving outsiders if and only if their net balances sum to zero — otherwise, money would have to flow in or out of the group, which contradicts settling "among themselves."

💡 The Invariant

At any point in the DP, dp[mask] represents the minimum number of transactions to completely settle all debts among the people indicated by mask, assuming their balances sum to zero (otherwise dp[mask] = ∞). The invariant is maintained because we only update dp[mask] by combining a zero-sum sub-mask with an already-solved remainder.

Greedy doesn't work: You might think "always match the biggest creditor with the biggest debtor." This greedy is locally optimal but not globally. Example: balances [+3, +3, −4, −2]. Greedy: match +3 with −4 (one payment, +3 left for −2 → −2 leftover won't close), resulting in sub-optimal paths. The DP finds the partition into zero-sum groups first, then handles each optimally.

⚠️ DFS Backtracking Alternative

There's a popular DFS approach where you try each creditor paying each debtor, recurse, and track the minimum. This is correct but harder to analyze. The Bitmask DP is cleaner for interviews: it encodes ALL subset states explicitly and is provably O(3ⁿ). Both approaches have exponential worst-case complexity because the problem is NP-hard in general.

4 The Code (Annotated) ▾

Solution 1 — Bitmask DP (Interview Pick, O(3ⁿ) time)

def minTransfers(transactions):
bal = {} ← net balance map: person → amount owed/owing
for u, v, w in transactions: ← process each transaction [from, to, amount]
bal[u] = bal.get(u, 0) - w ← u paid out w → net decreases
bal[v] = bal.get(v, 0) + w ← v received w → net increases
debts = [v for v in bal.values() if v != 0] ← only non-zero balances matter
n = len(debts) ← number of people with non-zero balance
INF = float('inf')
dp = [INF] * (1 << n) ← dp[mask] = min transactions to settle mask group
dp[0] = 0 ← base case: empty group needs 0 transactions
for mask in range(1, 1 << n): ← iterate every non-empty subset
# precompute sum of this subset
total = sum(debts[i] for i in range(n) if mask & (1 << i))
sub = mask ← start sub-mask enumeration at mask itself
while sub: ← enumerate all non-empty sub-masks of mask
# sum of sub-mask
s_sum = sum(debts[i] for i in range(n) if sub & (1 << i))
if s_sum == 0: ← this sub-group can settle among themselves!
k = bin(sub).count('1') ← how many people in this zero-sum group
dp[mask] = min(dp[mask], dp[mask ^ sub] + k - 1) ← k people need k−1 transactions
sub = (sub - 1) & mask ← classic sub-mask enumeration step
return dp[(1 << n) - 1] ← answer: settle the full set (all bits on)
⚠️ Why (sub - 1) & mask?

This is the classical bit trick for enumerating all submasks of a mask. sub - 1 flips the lowest set bit of sub to 0 and sets all bits below it to 1. ANDing with mask removes any bits not in the original mask. The loop terminates when sub reaches 0 (empty set). Total iterations across all masks is exactly 3ⁿ — each element is in mask-only, sub-only, or neither.

Solution 2 — DFS Backtracking (Simpler to derive, same worst case)

def minTransfers(transactions):
bal = {} ← same net balance computation
for u, v, w in transactions:
bal[u] = bal.get(u, 0) - w
bal[v] = bal.get(v, 0) + w
debts = [v for v in bal.values() if v != 0]
def dfs(idx): ← try to settle from person at index idx
while idx < len(debts) and debts[idx] == 0:
idx += 1 ← skip already-settled people
if idx == len(debts): return 0 ← all settled! 0 more transactions needed
res = float('inf')
for j in range(idx + 1, len(debts)): ← try each other person as the settlement partner
if debts[j] * debts[idx] < 0: ← only if one is positive, other negative
debts[j] += debts[idx] ← idx pays/receives from j, clearing idx
res = min(res, 1 + dfs(idx + 1)) ← 1 transaction + recursively settle rest
debts[j] -= debts[idx] ← backtrack: undo the settlement
return res
return dfs(0)
💡 Complexity Summary

Bitmask DP: Time O(3ⁿ), Space O(2ⁿ) — n ≤ 12 in practice (constraint: ≤12 people)
DFS Backtracking: Time O(n!) worst case (trying every permutation), Space O(n) stack
Both are exponential; n ≤ 12 makes them feasible. DP is faster in practice due to memoization.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Simplify the Problem

"First, I notice the original transaction list is irrelevant — all that matters is each person's net balance. I'll compute net balances in O(T) where T = number of transactions. Then I only care about the non-zero balances."

Step 2: Recognize the Partition Problem

"The sub-problem is: partition the non-zero balances into groups that each sum to zero. Each zero-sum group of k people needs exactly k−1 transactions. I want to minimize the total by finding the best partition — this screams exponential search over subsets."

Step 3: Recognize n ≤ 12 → Bitmask DP is Feasible

"With at most 12 people, we have at most 2¹² = 4096 subsets. Bitmask DP over all subsets, using the sub-mask enumeration trick, gives O(3¹²) ≈ 531,441 operations — totally feasible. This is the classic subset-sum DP pattern."

Step 4: State, Recurrence, Base Case

"State: dp[mask] = min transactions to settle the people in mask (if mask is zero-sum). Recurrence: try every zero-sum sub-mask s of mask: dp[mask] = dp[mask XOR s] + popcount(s) − 1. Base: dp[0] = 0."

🗣️ What to Say to the Interviewer

"The key observation is that we only need net balances, not the original transactions.
Once I have the net balances, the problem is: find the minimum way to partition
non-zero balances into zero-sum groups, where each group of k people costs k−1 transactions.
Since there are at most 12 people, I can use Bitmask DP over all 2^n subsets.
For each subset, I find all zero-sum sub-masks using the (s-1)&mask trick and update.
Total time: O(3^n) with O(2^n) space — both around 500K for n=12."
🔮 Likely Follow-up Questions
  • "What if n is larger, say 100?" → Problem becomes NP-hard. Approximate with greedy (always match largest creditor ↔ largest debtor).
  • "Why not just greedy?" → Greedy finds local matches but may miss a globally cheaper partition. Example: [+3, −2, −1, +2, −3, +1] — greedy uses 5, optimal uses 3.
  • "Explain the sub-mask enumeration." → Each element either is in mask (not sub), is in both (sub), or is in neither. 3 choices → 3ⁿ total.
  • "What does popcount(s)−1 represent?" → A chain of k−1 one-to-one payments fully settles k people: A→B→C...→last. Optimal lower bound for any zero-sum group.

7 Summary & Key Takeaways ▾
Reduce all transactions to net balances first — the original graph doesn't matter
A zero-sum group of k people always settles in exactly k−1 transactions (chain payment)
Bitmask DP over subsets: dp[mask] = min transactions to settle the masked group
Sub-mask enumeration (s-1)&mask runs in O(3ⁿ) total — each element has 3 states
Greedy (max creditor ↔ max debtor) is O(n log n) but suboptimal — exponential is required for optimal
ApproachTimeSpaceNotes
Greedy (max-match) O(n log n) O(n) Fast but NOT optimal — misses better partitions
DFS Backtracking O(n!) O(n) Correct; simpler to code; pruning helps in practice
Bitmask DP ✓ O(3ⁿ) O(2ⁿ) Optimal & elegant; clearly O(3ⁿ) complexity proof
🎯 When to Recognize Bitmask DP
  • Small n (≤ 20): each element can be "in" or "out" of a subset
  • "Partition into groups" with an optimization objective
  • Traveling Salesman: visited cities as a bitmask
  • Assignment problems: assigned workers as a bitmask
  • Any problem where state = "which subset of elements have been processed"
