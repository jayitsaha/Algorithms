LeetCode #443

String Compression

Compress an array of characters in-place using run-length encoding. Return the new length. Must use only constant extra space.

Medium Two Pointers Read-Write Pointers Time: O(n) Space: O(1)
1 Understanding the Problem ▾

You are given an array of characters (e.g., ['a','a','b','b','c','c','c']). Compress it in-place using run-length encoding: replace each run of consecutive identical characters with the character followed by the count (only if count > 1). Return the new length of the array.

💡 Key Insight

The tricky constraint is O(1) extra space in-place. You cannot build a new string — you must overwrite the original array. Because compression always makes the output shorter or equal in length, a write pointer that lags behind a read pointer guarantees we never overwrite characters we still need to read.

Example 1: Compresses nicely

Input: ['a','a','b','b','c','c','c']
Output: ['a','2','b','2','c','3']
Return: 6

Example 2: Count=1 — no digit written

Input: ['a']
Output: ['a']
Return: 1 (count=1 means skip the digit)

Important subtlety: If a run is longer than 9, the count is multi-digit. For a run of 12, you write '1' then '2' — each digit as a separate character element in the array. This is what makes the write pointer logic slightly non-trivial.

Edge Cases: Single character (count=1, write just the char), runs of length > 9 (must write each digit separately), array of all-same characters, alternating characters (no compression possible), array of length 1.

2 The Algorithm: Read-Write Two Pointers ▾

We maintain three positions in the array simultaneously. Think of it like a cassette editor: a read head scanning forward, a write head writing compressed output behind it, and a run-start anchor marking where the current group of identical characters began.

1 Initialize: write = 0 (next position to write output), i = 0 (read pointer). We will advance i through the array to scan each character.
2 Find a run: At position i, record group_start = i. Move i forward while chars[i] stays the same. When we stop, the run length is count = i - group_start.
3 Write the character: chars[write] = chars[group_start], then write += 1. This records the character that ran.
4 Write the count (if > 1): Convert count to a string. For each digit character, write it to chars[write] and increment write. Skip this step entirely if count is 1.
5 Repeat: group_start = i is set automatically at the top of the outer loop. Continue until i reaches the end. Return write as the new length.

🔑 Why Write Never Overtakes Read

Each run of k characters produces at most 1 + len(str(k)) output characters. For k=1 that is 1 character (not 2). For k=2..9 it is 2 characters (not >2). For k=100 it is 4 characters (c, 1, 0, 0) for 100 reads. In all cases output size ≤ input size, so write ≤ read always holds.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At the start of each outer loop iteration, chars[0..write-1] contains the correctly compressed prefix of all runs processed so far, and chars[i..n-1] contains the unprocessed suffix. The write pointer always satisfies write ≤ i.

Proof that write ≤ i always holds: A run of k characters starting at index a occupies positions a to a+k-1 in the input. Compressing it produces 1 + (k>1 ? len(str(k)) : 0) characters. For k=1: 1 ≤ 1. For k≥2: 1+len(str(k)) ≤ k (since 1+1=2≤2, 1+1=2≤9, 1+2=3≤99, etc.). So write advances by at most as many positions as read for every run. By induction: write ≤ i at every step.

What about count = 1? We write only the character, not the digit '1'. This is intentional per the problem spec, and it also ensures that single-character arrays are handled correctly: input ['a'] → output ['a'] with length 1.

⚠️ Multi-Digit Count Trap

For a run of 100 characters, the count "100" is three separate characters: '1', '0', '0'. You must iterate over str(count) and write each digit individually. A common mistake is trying to write the integer directly — but array elements must be single characters.

Why the inner while-loop approach works cleanly: while i < n and chars[i] == chars[group_start]: i += 1 — when this exits, every character from group_start to i-1 is identical. The count is i - group_start. This naturally handles runs of any length without any special cases.

4 The Code (Annotated) ▾

Solution 1 — Read-Write Two Pointers (Optimal: O(n) time, O(1) space) — Interview Pick

def compress(chars): ← chars is the original array, modified in-place
write = 0 ← Next position to write compressed output
i = 0 ← Read pointer that scans left to right
n = len(chars) ← Cache original length (doesn't change)
while i < n: ← Process every character in the array
group_start = i ← Anchor: remember where this run began
while i < n and chars[i] == chars[group_start]:← Advance until run ends
i += 1 ← Skip past identical chars in this run
chars[write] = chars[group_start] ← Always write the character
write += 1 ← Advance write pointer past the character
count = i - group_start ← How many times did this char repeat?
if count > 1: ← Only write count for runs of 2 or more
for digit in str(count): ← Each digit is a separate character element
chars[write] = digit ← Write '1','2',... one at a time
write += 1 ← Advance past the digit
return write ← New length of the compressed array
⚠️ Common Mistake: Forgetting to always write the character

Some beginners write the character only when count > 1, skipping it for single-char runs. That is wrong. You always write the character at chars[write]. Only the count digit is conditional on count > 1.

Solution 2 — Brute Force with extra space (O(n) time, O(n) space)

def compress_brute(chars): ← Uses extra space — NOT acceptable per constraints
result = [] ← Extra O(n) list — fails the O(1) requirement
i = 0
while i < len(chars):
ch, count = chars[i], 0
while i < len(chars) and chars[i] == ch:
i += 1; count += 1
result.append(ch)
if count > 1: result.extend(str(count))
chars[:len(result)] = result ← Copy back — still O(n) space was used
return len(result)
💡 Complexity Summary

Read-Write Two Pointers (Optimal): Time O(n) — each character read exactly once; digits written in O(log k) per run but sum to O(n) total. Space O(1) — only 3 integer variables.
Brute Force: Time O(n), Space O(n) — conceptually simpler but violates the in-place constraint.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the O(1) Space Signal

"In-place modification with constant extra space" is the classic signal for a read-write two-pointer approach. One pointer reads (the natural scan), another writes compressed output. This works because run-length compression can only shrink or maintain length — so the write pointer never overtakes the read pointer.

Step 2: Name the Three Roles Before Coding

Identify: (1) read pointer i — scanning right through the array, (2) group_start anchor — remembering where the current run began so you can compute its length as i - group_start, (3) write pointer — where compressed output lands. Naming these before writing code signals clear thinking.

Step 3: Handle Multi-Digit Count Proactively

Bring up the multi-digit count before the interviewer asks: "For a run of 12, the count '12' becomes two characters '1' and '2'. So I iterate over str(count) and write each digit character individually." This shows careful attention to edge cases.

Step 4: Prove the Safety Invariant

The interviewer will ask: "How do you know the write pointer doesn't overwrite unread data?" Answer: "A run of k characters produces at most 1 + len(str(k)) output characters, which is always ≤ k. So write advances by at most as much as read for every run — they can never cross."

🗣️ What to Say to the Interviewer

"The key constraint is O(1) space. Since compression can't increase length, I'll use read and write pointers.
I scan with i to find runs — when chars[i] differs from chars[group_start], the run has ended.
I write the character, then if count > 1, I write each digit of the count as a separate character.
The tricky part is multi-digit counts: '12' becomes chars[write]='1', chars[write+1]='2'.
The write pointer is always ≤ read pointer, so in-place modification is safe.
Return write, which is the new length."
