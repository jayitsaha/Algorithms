LeetCode #1314

Matrix Block Sum

Given an m×n integer matrix mat and an integer k, return a matrix answer where answer[i][j] equals the sum of all mat[r][c] for i−k ≤ r ≤ i+k and j−k ≤ c ≤ j+k (clamped to matrix bounds).

Medium Prefix Sum 2D Prefix Sum Time: O(m·n) Space: O(m·n)
1 Understanding the Problem ▾

For every cell (i, j) in the matrix, compute the sum of all values within a square window of radius k centered at that cell — clamped to the matrix edges. Every output cell reads an overlapping sub-rectangle of the input.

💡 Key Insight

Adjacent cells share enormous amounts of their query rectangle. If we naively re-sum each window, we repeat work exponentially as k grows. The fix: precompute a 2D prefix sum table in one O(m·n) pass, then answer every rectangle query in exactly 4 table lookups — O(1) each.

Concrete example (3×3 matrix, k=1):

✗ Naive (O(m·n·k²))

Each of 9 cells reads up to
(2·1+1)² = 9 values.
9 × 9 = 81 additions total.
For k=100: 9 × 201² ≈ 363k ops.

✓ Prefix Sum (O(m·n))

Build prefix table: 9 additions.
Each cell: exactly 4 lookups.
9 × 4 = 36 lookups total.
For k=100: same 36 lookups.

Edge cases: When k ≥ m or k ≥ n, every cell's window covers the entire matrix. The clamping via max(0, i−k) and min(m−1, i+k) handles all border cells uniformly — no special cases needed.

2 The Algorithm: 2D Prefix Sum ▾

2D prefix sum is a direct extension of 1D prefix sums. Instead of "sum of a subarray," we answer "sum of a sub-rectangle" in O(1) after O(m·n) preprocessing. The trick is a padded (m+1)×(n+1) table that bakes in boundary zeros.

1 Allocate P[(m+1)][(n+1)] filled with zeros. The extra row and column at index 0 act as a border of zeros, eliminating boundary-check special cases from both the build and query formulas.
2 Build — fill using inclusion-exclusion: P[i][j] = mat[i-1][j-1] + P[i-1][j] + P[i][j-1] - P[i-1][j-1]. We add the current cell plus the "above" prefix strip plus the "left" prefix strip, subtracting the top-left block that was counted twice.
3 For each output cell (i, j), compute the clamped window: r1 = max(0, i−k), c1 = max(0, j−k), r2 = min(m−1, i+k), c2 = min(n−1, j+k). These are the actual row and column bounds that stay inside the matrix.
4 Query — 4-corner formula: ans[i][j] = P[r2+1][c2+1] − P[r1][c2+1] − P[r2+1][c1] + P[r1][c1]. This is the 2D inclusion-exclusion: start with the full bottom-right rectangle, remove the two overshooting strips, add back the doubly-removed corner.
📐 The Two Key Formulas
Build: P[i][j] = mat[i-1][j-1] + P[i-1][j] + P[i][j-1] - P[i-1][j-1]
Query: ans[i][j] = P[r2+1][c2+1] - P[r1][c2+1] - P[r2+1][c1] + P[r1][c1]
where r1=max(0,i-k), c1=max(0,j-k), r2=min(m-1,i+k), c2=min(n-1,j+k)
🔑 Why the +1 Padding?

The prefix table is 1-indexed relative to mat. P[i][j] stores the sum of mat[0..i-1][0..j-1]. Allocating (m+1)×(n+1) and initializing to zero means P[0][*] = P[*][0] = 0 automatically, so the build formula works for all (i,j) without any "if i==0" guards.

3 Why It Works: The Invariant ▾
🔐 The Invariant

After building, P[i][j] equals the sum of all elements in the rectangle mat[0..i-1][0..j-1] — the "top-left" sub-matrix ending at row i−1 and column j−1. This invariant holds for every (i,j) from (0,0) to (m,n).

Why the build recurrence is correct: Think of each new cell as extending existing rectangles. P[i-1][j] covers the wide column strip above row i-1. P[i][j-1] covers the wide row strip left of column j-1. Their union double-counts the top-left block P[i-1][j-1], so subtract it. Add the new cell. Exactly the correct sum — by induction, every P[i][j] is correct if all previous entries are correct (base case: P[0][*] = P[*][0] = 0).

Why the query formula is correct: P[r2+1][c2+1] = sum of the mega-rectangle from (0,0) to (r2,c2). We overshoot above r1 and left of c1. Subtract P[r1][c2+1] (strip above r1) and P[r2+1][c1] (strip left of c1). But the top-left block P[r1][c1] was in both strips — it was subtracted twice, so add it back once. Result: sum of exactly mat[r1..r2][c1..c2].

⚠️ The Most Common Off-by-One: Forgetting +1 in the Query

When querying the window boundary (r2, c2), you need P[r2+1][c2+1] — NOT P[r2][c2]. Because P is 1-indexed, mat[r2][c2] maps to P[r2+1][c2+1]. Forgetting the +1 silently drops the entire bottom row and rightmost column of every window — the algorithm produces wrong answers with no error.

When window covers entire matrix (k ≥ m,n): r1=0, c1=0, r2=m-1, c2=n-1. Query becomes P[m][n] - P[0][n] - P[m][0] + P[0][0] = P[m][n] - 0 - 0 + 0 = P[m][n] = total sum. Correct.

4 The Code (Annotated) ▾

Solution 1 — 2D Prefix Sum (Optimal, O(m·n) time · Interview Pick)

def matrixBlockSum(mat, k):
m, n = len(mat), len(mat[0]) ← Matrix dimensions
# ── Phase 1: Build prefix table ──
P = [[0]*(n+1) for _ in range(m+1)] ← (m+1)×(n+1) zeros; border row/col = 0
for i in range(1, m+1): ← 1-indexed to align with padded table
for j in range(1, n+1):
P[i][j] = (mat[i-1][j-1] ← Current cell value (0-indexed in mat)
+ P[i-1][j] ← Prefix strip above (entire columns 0..j-1)
+ P[i][j-1] ← Prefix strip to left (entire rows 0..i-1)
- P[i-1][j-1]) ← Remove top-left block counted twice
# ── Phase 2: Answer queries ──
ans = [[0]*n for _ in range(m)] ← Output matrix, same shape as mat
for i in range(m):
for j in range(n):
r1, c1 = max(0, i-k), max(0, j-k) ← Top-left window corner, clamped ≥ 0
r2, c2 = min(m-1, i+k), min(n-1, j+k)← Bottom-right, clamped ≤ last row/col
ans[i][j] = (P[r2+1][c2+1] ← Sum of (0,0)→(r2,c2) rectangle
- P[r1][c2+1] ← Subtract strip above r1
- P[r2+1][c1] ← Subtract strip left of c1
+ P[r1][c1]) ← Add back doubly-subtracted corner
return ans
⚠️ Why P[r2+1][c2+1] and not P[r2][c2]?

The prefix table is 1-indexed: P[i][j] covers mat[0..i-1][0..j-1]. To include mat[r2][c2], you need P[r2+1][c2+1]. Similarly, to "stop before row r1," you use P[r1] (which covers rows 0 through r1-1). This consistent +1 shift makes the formula correct throughout.

Solution 2 — Brute Force (O(m·n·(2k+1)²), simple but slow)

def matrixBlockSum(mat, k):
m, n = len(mat), len(mat[0])
ans = [[0]*n for _ in range(m)]
for i in range(m):
for j in range(n):
for r in range(max(0,i-k), min(m,i+k+1)): ← Row window (note min uses m not m-1)
for c in range(max(0,j-k), min(n,j+k+1)):← Col window
ans[i][j] += mat[r][c] ← Re-sum for EVERY cell (wasteful overlap)
return ans
💡 Complexity Summary

2D Prefix Sum (optimal): Time O(m·n) · Space O(m·n) for prefix table
Brute Force: Time O(m·n·(2k+1)²) · Space O(1) extra — breaks at large k
Insight: The prefix table is the same size as the output — so in practice we trade O(m·n) space for O(1)-per-query rectangle lookups

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Spot the Repeated Rectangle Query

Ask yourself: "Am I computing many sums over overlapping regions?" — yes. Every adjacent pair of output cells shares almost the entire window. This pattern — many overlapping range queries — is the classic signal for prefix sums.

Step 2: Anchor to 1D Knowledge

"I know 1D prefix[i] = sum of arr[0..i-1], so subarray sum[l,r] = prefix[r+1] - prefix[l]. In 2D, P[i][j] = sum of mat[0..i-1][0..j-1], and rectangle sum = 4-corner formula." Derive, don't memorize.

Step 3: Draw the Inclusion-Exclusion Diagram

Sketch a rectangle. P[r2+1][c2+1] is the big box. Minus top strip. Minus left strip. Plus corner (counted out twice). This diagram is the proof — sketch it for the interviewer.

Step 4: Handle Clamping Cleanly

"I use max(0, i-k) and min(m-1, i+k) to clamp the window — no special cases needed for border cells. The zero-padded border in the prefix table handles the rest."

🗣️ What to Say to the Interviewer

"Each output cell needs the sum of a rectangle in mat — and adjacent cells' rectangles heavily overlap.
So I'll precompute a 2D prefix table P where P[i][j] = sum of mat[0..i-1][0..j-1].
Build: P[i][j] = mat[i-1][j-1] + P[i-1][j] + P[i][j-1] - P[i-1][j-1]. One pass, O(m·n).
Query: for cell (i,j), clamp window to (r1,c1)→(r2,c2), then
ans = P[r2+1][c2+1] - P[r1][c2+1] - P[r2+1][c1] + P[r1][c1]. Four lookups, O(1).
Total O(m·n) time and O(m·n) space — k doesn't affect runtime at all."
🔮 Likely Follow-up Questions
  • "Can you reduce space to O(1)?" → Not for this formulation — the prefix table is inherently O(m·n). Mention that in-place modification of mat is unsafe since cells depend on each other.
  • "What if k varies per query?" → Same prefix table answers any k — build once, query with any k value.
  • "What about a circular window instead of square?" → Harder; prefix sums don't directly apply; BFS or convolution is needed.
  • "Extend to 3D?" → 3D prefix sum, 8-corner inclusion-exclusion (8 terms).

7 Summary & Key Takeaways ▾
2D prefix sum answers any rectangle query in O(1) — build once, query forever
Pad with an extra row/col of zeros — eliminates all boundary special cases in build AND query
Build: mat cell + above strip + left strip − doubly-counted corner
Query: bottom-right − top strip − left strip + double-subtracted corner (4 lookups)
Clamp with max/min — handles all border cells without if-guards
Total O(m·n) regardless of k — k only affects the clamped window bounds, not the query cost
