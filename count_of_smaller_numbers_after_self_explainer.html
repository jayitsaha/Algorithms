LeetCode #315

Count of Smaller Numbers After Self

Given an integer array nums, return counts[i] = number of elements to the right of nums[i] that are strictly smaller.

Hard Advanced Data Structures BIT + Coordinate Compression Time: O(n log n) Space: O(n)
1 Understanding the Problem ▾

For every position i, count how many elements to the right of nums[i] are strictly less than nums[i]. The result is a new array of the same length. This is the "inversion count per element" problem.

💡 Key Insight

Process the array right to left. Whenever we land on nums[i], all elements to its right have already been "registered." If we maintain a data structure that counts how many registered elements fall below a threshold, we can answer the query in O(log n).

Example: nums = [5, 2, 6, 1]

i=0: nums[0]=5, right=[2,6,1]
  smaller: 2, 1 → count=2
i=1: nums[1]=2, right=[6,1]
  smaller: 1 → count=1
i=2: nums[2]=6, right=[1]
  smaller: 1 → count=1
i=3: nums[3]=1, right=[]
  smaller: none → count=0
Answer: [2, 1, 1, 0]

Why Brute Force is Too Slow

Nested loops: O(n²)
n=10,000 → 100M ops
n=100,000 → 10B ops
LeetCode constraint: n ≤ 100,000
Need: O(n log n) approach

Edge Cases: single element → [0]; all same values → all 0s (no strictly smaller); strictly increasing array → all 0s; strictly decreasing array → [n-1, n-2, ..., 1, 0]; values can be negative (requires coordinate compression, not offset arithmetic).

2 The Algorithm: BIT + Coordinate Compression ▾

A Binary Indexed Tree (Fenwick Tree) supports two operations in O(log n): point update (add 1 at a position) and prefix sum query (count of elements at positions 1..i). We use it as a dynamic histogram of seen values. But BIT indices must be in [1, n], so we first map values to ranks.

1 Coordinate Compression: Sort unique values. Assign each value a rank (1, 2, 3, ...) based on sorted position. Now every value in nums can be looked up as a BIT index. Example: nums=[5,2,6,1] → sorted_unique=[1,2,5,6] → ranks: {1:1, 2:2, 5:3, 6:4}.
2 Scan Right to Left: For each i from n-1 down to 0, look up rank r = rank[nums[i]].
3 Query BIT: Call query(r-1) to get the count of already-registered values with rank less than r. These are exactly the right-side elements smaller than nums[i]. Store this as counts[i].
4 Update BIT: Call update(r) to register nums[i] as seen. Future queries from elements to the left will include it in their count if appropriate.
🔑 Alternative: Modified Merge Sort

During the merge step of merge sort, if an element from the right half is placed before k remaining left-half elements (because it is smaller), those k left elements each gain +1. This naturally counts inversions. Same O(n log n) time, O(n) space; no extra data structure needed but requires tracking original indices through recursion.

3 🔬 Algorithm Deep-Dive: BIT & Merge Sort ▾
Binary Indexed Tree (Fenwick Tree) — Origin & Invariant

Origin: Peter Fenwick, 1994. Designed for dynamic prefix sum computation over a frequency table. Enables both update and query in O(log n) using a flat array of size n.

Core invariant: bit[i] stores the cumulative sum of positions [i - lowbit(i) + 1 .. i], where lowbit(i) = i & (-i) isolates the lowest set bit. This partitions the array into non-overlapping intervals whose sizes are powers of 2, mirroring a binary tree structure implicit in the index.

Why it works: Every prefix sum [1..i] decomposes into O(log i) non-overlapping intervals, each stored in a BIT cell. Stripping the lowest set bit of i moves to the "left neighbor" covering the previous segment. Adding the lowest set bit moves to the "parent" for updates. Together, these traverse the implicit tree in O(log n).

When to recognize it: Any problem with interleaved point updates and prefix sum queries — counting inversions, finding order statistics, counting elements smaller/larger seen so far. Key signal: "dynamic histogram" or "count elements satisfying a threshold dynamically."

Modified Merge Sort — Counting During Merge

Core idea: In a standard merge sort we compare the smallest remaining elements of two halves. Modified: when an element from the right half "wins" (is smaller, gets placed first), all remaining left-half elements are larger — so they each have one more "smaller element to the right."

Invariant: When merging sorted halves [L] and [R], right_count tracks how many right-half elements have already been placed. When we take left[l], counts[original_index_of_left[l]] += right_count. This is valid because both halves are already sorted — all those right elements were originally to the right of all left elements in the original array.

Trade-off vs BIT: Merge sort has no coordinate compression step and slightly better cache behavior. BIT is more modular (easy to extend to "count in range [lo,hi]") and more recognizable as a general technique. Both should be in your toolkit.

4 The Code (Annotated) ▾

Solution 1 — BIT + Coordinate Compression (Optimal, Interview Pick)

from bisect import bisect_left
def countSmaller(nums):
sorted_u = sorted(set(nums)) ← unique values in sorted order
rank = {v: i+1 for i,v in enumerate(sorted_u)} ← 1-indexed rank for each value
n, m = len(nums), len(sorted_u)
bit = [0] * (m + 1) ← BIT of size m; index 0 unused
def update(i): ← add 1 at rank i, propagate up
while i <= m:
bit[i] += 1
i += i & (-i) ← add lowest set bit to move to parent
def query(i): ← prefix sum [1..i]: count of elements with rank ≤ i
s = 0
while i > 0:
s += bit[i]
i -= i & (-i) ← strip lowest set bit: move to left neighbor
return s
counts = [0] * n
for i in range(n-1, -1, -1): ← right to left: right elements already registered
r = rank[nums[i]]
counts[i] = query(r - 1) ← count of seen elements with rank < r (strictly smaller)
update(r) ← register current element so left neighbors can count it
return counts
⚠️ Why query(r-1) and not query(r)?

We want strictly smaller elements: only those with rank strictly less than r. query(r) would also count elements equal to nums[i] that were seen earlier (there may be duplicates). Using r-1 ensures strict inequality.

Solution 2 — Modified Merge Sort (Alternative, also O(n log n))

def countSmaller(nums):
counts = [0] * len(nums)
indexed = list(enumerate(nums)) ← pair (orig_index, value) so we can update counts[orig_index]
def merge_sort(arr):
if len(arr) <= 1: return arr
mid = len(arr) // 2
left = merge_sort(arr[:mid])
right = merge_sort(arr[mid:])
merged, l, r = [], 0, 0
while l < len(left) and r < len(right):
if left[l][1] > right[r][1]: ← left element is larger: right[0..r-1] are all smaller than it
counts[left[l][0]] += len(right) - r ← all remaining right elements are smaller
merged.append(left[l]); l += 1
else:
merged.append(right[r]); r += 1
merged += left[l:] + right[r:]
return merged
merge_sort(indexed)
return counts
⚠️ Why counts[left[l][0]] += len(right) - r?

When left[l] is placed (it is larger), all right elements from index r to the end are smaller than left[l] and were originally to its right. That is len(right)-r elements. We add them all at once — O(1) per element in the left half — because the right half is sorted and all remaining right elements are smaller.

Solution 3 — Brute Force O(n²) (reference only, TLE)

def countSmaller(nums):
n = len(nums)
return [sum(nums[j]<nums[i] for j in range(i+1,n)) for i in range(n)]
5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: State the Brute Force

"For each i, scan everything to the right: O(n²). For n=10⁵ this is 10 billion operations — not acceptable."

Step 2: The Key Observation

"If I process right to left, all right-side elements are already 'known' when I reach position i. I need: how many known values are smaller than current? This is a dynamic prefix count — O(log n) with a BIT."

Step 3: Introduce Coordinate Compression

"BIT indices must be [1..n]. Values can be negative or huge. I'll compress: sort unique values, assign ranks 1..k. Now I index the BIT by rank. Compression: O(n log n) for sorting."

Step 4: Offer the Merge Sort Alternative

"A second approach: modified merge sort. During the merge step, count how many right-half elements precede a left-half element — those are exactly the smaller elements originally to its right. Same complexity, no extra data structure."

🗣️ What to Say to the Interviewer

"Brute force is O(n²) — scan everything to the right for each element.
Better: process right to left. For each element, I need count of already-seen values that are smaller.
I'll use a BIT (Fenwick Tree) for O(log n) point update + prefix sum query.
Since values aren't in [1,n], I coordinate-compress: sort unique values, assign ranks.
Then: query(rank-1) = count of smaller seen; update(rank) = register current.
Total: O(n log n). Alternative: modified merge sort — track inversions during merge step.
Space O(n) for the BIT, rank map, and counts array."
🔮 Likely Follow-up Questions
  • "Count LARGER numbers to the right?" → query(total_seen) - query(rank) gives elements with rank > r
  • "Count inversions in the whole array?" → Sum of all counts[i] is the total inversion count
  • "What if values up to 10⁹?" → Compression handles this; BIT size = distinct values ≤ n
  • "Count elements in range [lo, hi] to the right?" → query(rank[hi]) - query(rank[lo]-1)
  • "Can you use a BST?" → Python's sortedcontainers.SortedList: bisect gives position = count smaller, then insert

7 Summary & Key Takeaways ▾
Process right to left: all right-side elements are "already seen" when you reach position i
Coordinate compression maps arbitrary values to compact ranks [1..k], enabling BIT indexing
BIT query(rank-1) gives count of strictly smaller seen elements in O(log n)
BIT update(rank) registers current element in O(log n) for future left-side queries
Merge sort alternative: count inversions during the merge step, tracking original indices
Sum of all counts[i] = total inversions in the array (useful follow-up fact)

🎯 When to Recognize BIT + Coordinate Compression
  • "Count inversions" or "smaller/larger to the left/right" — dynamic order statistics
  • Interleaved updates (register an element) and prefix queries (how many ≤ threshold so far)
  • Values outside [1, n] — always compress before using BIT
  • "Online" queries where you cannot sort the input first without losing position information
