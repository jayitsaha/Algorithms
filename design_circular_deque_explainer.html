LeetCode #641

Design Circular Deque

Implement a double-ended queue backed by a fixed-size circular array. All six operations — insert/delete at front and rear, peek, isEmpty, isFull — must run in O(1).

Medium Design Array + Modulo Time: O(1) all ops Space: O(k)
1 Understanding the Problem ▾

A deque (double-ended queue) is a line where people can join or leave from either the front or the back. A circular deque stores elements in a fixed-size ring: when you reach one end of the array, you wrap back to the other end instead of going out of bounds.

💡 Key Insight

You never need to move data. Use two integer pointers — front and rear — that say "where the live region starts and ends." To add at the front, step front one slot backwards (wrapping with modulo). To add at the rear, step rear one slot forwards. O(1) because it's just arithmetic on indices.

The six operations you must implement:

Mutating (return True/False)

insertFront(value)
insertLast(value)
deleteFront()
deleteLast()

Query (read-only)

getFront() → value or -1
getRear() → value or -1
isEmpty() → bool
isFull() → bool

Critical edge cases:

  • Insert into full deque: return False without modifying state
  • Delete from empty deque: return False without modifying state
  • First insert ever: both front and rear start at 0 — do NOT move the pointer before writing or you'll misalign them
  • One element left: front == rear; both deleteFront and deleteLast remove the same slot
2 The Algorithm: Fixed Array + Circular Pointer Wrap ▾

Allocate an array of exactly k slots once in __init__. Maintain four pieces of state: the array, front (index of first live element), rear (index of last live element), and size (count of live elements). Every operation is a pointer move + one array read or write.

1 Init: q = [0]*k, front = rear = 0, size = 0, cap = k. No element exists yet — pointers are parked at index 0.
2 insertLast(v): If full → False. If not empty, advance rear: rear = (rear+1)%k. Write q[rear] = v. Increment size. Return True.
3 insertFront(v): If full → False. If not empty, step front back: front = (front-1+k)%k. Write q[front] = v. Increment size. Return True.
4 deleteFront(): If empty → False. Advance front forward: front = (front+1)%k. Decrement size. Return True. (No data erasure needed — the slot is just no longer in the live range.)
5 deleteLast(): If empty → False. Step rear back: rear = (rear-1+k)%k. Decrement size. Return True.
6 getFront / getRear: If empty → -1. Otherwise, q[front] or q[rear]. Single array read, O(1).
🔑 The Wrap-Around Formulas
Moving forward (insertLast, deleteFront): new_idx = (idx + 1) % k Moving backward (insertFront, deleteLast): new_idx = (idx - 1 + k) % k Why "+ k"? Prevents negative modulo in C++/Java. In Python, (-1) % 5 = 4 naturally, but writing "+ k" signals intent clearly.

Why track size instead of comparing front == rear? Both an empty and a completely full deque can have front == rear. A size counter disambiguates the two in O(1) without any extra computation.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every moment: q[front] holds the front element, q[rear] holds the rear element, and the live elements occupy slots front, (front+1)%k, ..., rear (in circular order) — exactly size of them. Inserting extends this range by one slot; deleting shrinks it by one slot.

Why O(1) for everything? Each operation performs exactly one modulo addition, at most one array read or write, and one increment/decrement to size. No loops. No shifting. The ring illusion is achieved entirely through arithmetic on two integers.

First insert correctness: When size == 0, both front and rear sit at index 0. If the first call is insertLast(v), we write at q[0] — good. If we advanced rear first, it would jump to 1, and then front(0) != rear(1) for a one-element deque — a bug. The guard if not isEmpty() before moving the pointer fixes this.

⚠️ Common Mistake: Moving Pointer Before First Insert

A beginner might write rear = (rear+1)%k; q[rear] = v unconditionally. This makes the very first element land at index 1, leaving index 0 unused and making getFront return the wrong slot. Always check if not isEmpty() before moving the pointer in insertFront and insertLast.

⚠️ Common Mistake: Using front == rear as Empty Check

When size == k and the deque is full, front can also equal rear (after exactly k elements wrap around). Always use size == 0 for empty and size == k for full.

💡 Complexity Analysis

Time: O(1) per operation — exactly 1 modulo op, 1 array access, 1 size update.
Space: O(k) for the pre-allocated array. After __init__, zero additional memory is ever allocated.

🔵 Alternative: Doubly Linked List

A doubly linked list also achieves O(1) for all 6 ops. Pros: no capacity limit, natural "wrap" via head/tail pointers. Cons: O(k) extra space for node pointers (2 pointers per node), no cache locality, no random access. For a fixed-capacity deque, the array approach is faster in practice due to cache efficiency.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1 — Identify the bottleneck

Naive: use Python's collections.deque. Interviewer wants you to implement it. Key insight: inserting at front of a regular array is O(n) due to shifting. A circular design eliminates shifting entirely.

Step 2 — Pick your data structure

Fixed array beats linked list for this problem: O(1) random access lets getFront/getRear be a single array read. Pre-allocate once, never again.

Step 3 — Design the state

You need: the array, front, rear, size, capacity. Resist the temptation to drop size and use front == rear for empty/full — that breaks when the deque wraps fully around.

Step 4 — Derive the modulo formula on the fly

Draw a ring of 4 slots on paper. "front is at 0. I want to insertFront. Where should the new front be?" → index 3 (one step left, wrapping). That's (0-1+4)%4 = 3. Demonstrate this live on your whiteboard.

Step 5 — Handle the first-insert edge case

Start empty: front=rear=0. Call insertLast(5). If you advance rear first → rear becomes 1, but front is still 0. Now getFront reads q[0]=0 (wrong). Fix: only move the pointer when the deque is non-empty.

🗣️ What to Say

"I'll use a fixed array of size k with two integer pointers — front and rear — plus a size counter. For insertFront I step front backwards: (front - 1 + k) % k. For insertLast I step rear forwards: (rear + 1) % k. I track size separately to tell empty from full, since both cases can have front == rear. One edge case: the very first insert should NOT move the pointer — I guard that with 'if not isEmpty()'. Every operation is O(1) and no memory is allocated after init."

Likely follow-ups:

  • "Implement with a doubly linked list" — head/tail pointers, O(1) inserts/deletes, no modulo needed
  • "Make it thread-safe" — wrap each mutation in a lock; or use CAS for lock-free variant
  • "Support dynamic resizing" — double array on full; copy elements to new array resetting front=0
  • "Where is this used in real systems?" — I/O ring buffers, monotonic deque for sliding window max, BFS queues

7 Summary & Key Takeaways ▾
Pre-allocate array of size k once; never reallocate — all ops stay O(1)
Use two integer pointers (front, rear) plus a size counter as the only state
Wrap pointers with modulo: forward = (idx+1)%k, backward = (idx-1+k)%k
Use size counter (not front==rear) to distinguish empty from full
Skip pointer movement on the first insert — write at current position
No data ever shifts — only two integers (front, rear) change on each mutation
