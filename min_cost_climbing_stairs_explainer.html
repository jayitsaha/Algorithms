LeetCode #746

Min Cost Climbing Stairs

Given an integer array cost where cost[i] is the cost of step i, find the minimum cost to reach the top of the floor. You can start from index 0 or 1, and from each step you can climb 1 or 2 steps.

Easy Dynamic Programming 1-D DP Min of Two Predecessors Time: O(n) Space: O(1)
1 Understanding the Problem ▾

Imagine a staircase where every step has a toll booth. You pay the toll when you step on that stair, then can jump either 1 or 2 stairs forward. Your goal is to escape the staircase — reach the "top," which is one position past the last stair — for as little total toll as possible. You can start at stair 0 or stair 1 (both free to enter).

💡 Key Insight

Every stair can only be reached from exactly one of two predecessors: the stair one back, or the stair two back. The cheapest cost to reach stair i is therefore cost[i] + min(cheapest way to reach i-1, cheapest way to reach i-2). This "min of two predecessors" shape is one of the most common DP patterns — once you see it here, you will recognise it immediately in House Robber, Climbing Stairs, Delete and Earn, and many more.

Edge Cases: Exactly 2 steps (answer = min(cost[0], cost[1])), all same values (any path), a single giant spike in the middle (jump over it via a 2-step), alternating high-low values.

2 Why Is This Dynamic Programming? ▾

DP applies when a problem has both of these structural properties simultaneously. Both hold here strongly:

1 Optimal Substructure: The cheapest path to the top must pass through either stair n-1 or stair n-2. The cheapest way to reach each of those is itself a sub-problem with the same shape. So the global optimum is built from local optima — each sub-solution feeds the next.
2 Overlapping Subproblems: Naive recursion recomputes the same states exponentially. dp(5) is called from both dp(6) and dp(7). Without caching, this is O(2ⁿ). With memoization or tabulation, each state is computed exactly once — O(n) total.
📐 The Recurrence Relation
dp[0] = cost[0]
dp[1] = cost[1]
dp[i] = cost[i] + min(dp[i-1], dp[i-2])  for i ≥ 2

Answer = min(dp[n-1], dp[n-2])  — the top has no toll; exit from the last or second-to-last stair

State meaning: dp[i] is the minimum total cost to step ON stair i — including all tolls paid on the way there, plus the toll at stair i itself. Once computed, any later state can use it without worrying about how we arrived.

Why not greedy? Greedy picks the cheapest immediate next step. A cheap stair now might funnel you into an expensive one with no escape. DP considers all possible paths implicitly through the recurrence — the minimum propagates forward with full foresight.

3 Why It Works: The Invariant ▾
🔐 The Invariant

After computing dp[i], it holds the exact minimum cost of landing on stair i, accounting for the entire path from the start. Every future state that reads dp[i] can trust it unconditionally. We never need to revise a computed value — this is the "optimal substructure" guarantee in action.

Why base cases are just cost[0] and cost[1]: You begin on step 0 or 1 for free — there are no predecessor stairs to have paid tolls on. So the accumulated cost of reaching stair 0 is purely cost[0], and similarly for stair 1. The recurrence only applies from stair 2 onward.

Why the answer is min(dp[n-1], dp[n-2]) and not dp[n-1] alone: The "top" is a virtual finish line with no toll. You can jump to it with a 1-step from stair n-1, or a 2-step from stair n-2. Both are valid exits — the answer is whichever of those accumulated costs is smaller.

⚠️ Common Confusion: Does the top floor have a cost?

No. The top is just the destination — stairs 0 through n-1 have tolls, but the top (index n, or "one past the end") does not. If you mistakenly add a cost[n] entry, your answer will be wrong. The final answer is min(dp[-1], dp[-2]) in Python — the two final stair costs, not a nonexistent top toll.

Space optimization — why O(1) is possible: Each dp[i] references only dp[i-1] and dp[i-2]. No earlier values are ever needed again. We can throw them away immediately, keeping only two rolling variables — prev2 and prev1 — and updating them in sync. Space falls from O(n) to O(1) with zero loss of correctness.

4 The Code (Annotated) ▾

Solution 1 — Bottom-Up Tabulation, O(1) Space (Optimal, Interview Pick)

def minCostClimbingStairs(cost):
prev2 = cost[0] ← dp[0] = cost[0]: base case, start freely at stair 0
prev1 = cost[1] ← dp[1] = cost[1]: base case, start freely at stair 1
for i in range(2, len(cost)): ← Fill dp[2], dp[3], … in order
curr = cost[i] + min(prev1, prev2)← Pay toll at i, add cheapest way to arrive (1-step or 2-step back)
prev2 = prev1 ← Slide: old prev1 becomes the new "two back"
prev1 = curr ← New curr becomes the new "one back"
return min(prev1, prev2) ← Exit from last or second-to-last stair (top has no toll)
⚠️ Variable Swap Order Is Critical

You must set prev2 = prev1 BEFORE prev1 = curr. If you reverse the order, prev2 gets the new curr value instead of the old prev1, silently corrupting the next iteration. This is one of the most common bugs on this exact problem during interviews.

Solution 2 — Top-Down Memoization (Equally valid, easier to derive)

from functools import lru_cache
def minCostClimbingStairs(cost):
n = len(cost)
@lru_cache(None) ← Decorator auto-caches results by argument value
def dp(i): ← Returns: min cost to have stepped ON stair i
if i <= 1: return cost[i] ← Base case: indices 0 and 1 cost just their own toll
return cost[i] + min(dp(i-1), dp(i-2)) ← Same recurrence as tabulation, computed lazily
return min(dp(n-1), dp(n-2)) ← Top = min of the two final exits

Solution 3 — Brute Force Recursion (for understanding the recurrence tree only)

def minCostClimbingStairs(cost):
def rec(i):
if i <= 1: return cost[i]
return cost[i] + min(rec(i-1), rec(i-2)) ← Identical recurrence — but no cache means O(2ⁿ) recomputation
n = len(cost)
return min(rec(n-1), rec(n-2))
💡 Complexity Summary

Brute Force: Time O(2ⁿ), Space O(n) call stack
Memoization (top-down): Time O(n), Space O(n) cache + stack
Tabulation O(1) space (bottom-up): Time O(n), Space O(1) — optimal for interviews

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Spot the DP Signal

"The problem asks for minimum cost — an optimization goal. Each position involves a choice (1-step or 2-step). The choice today affects future options. And the same subproblem (reaching stair k) appears from multiple paths — overlapping subproblems. This is DP."

Step 2: Define the State Precisely

"Let dp[i] = minimum total cost of stepping ON stair i. I pay cost[i] when I land, plus whatever it cost to get there. Base cases: dp[0] = cost[0] and dp[1] = cost[1] — I can start at either for free."

Step 3: Write the Recurrence

"To reach stair i I must have come from stair i-1 or i-2. So: dp[i] = cost[i] + min(dp[i-1], dp[i-2]). The answer is min(dp[n-1], dp[n-2]) since the top floor has no toll — I just need to reach it."

Step 4: Optimize Space to O(1)

"Since each step only looks back 2, I only need two rolling variables. Replace the full array with prev2 and prev1. Update order matters: advance prev2 = prev1 first, then prev1 = curr."

🗣️ What to Say to the Interviewer

"This is DP — minimization with choices at each step, and subproblems overlap.
State: dp[i] = min cost to step ON stair i (inclusive of all prior tolls).
Base cases: dp[0]=cost[0], dp[1]=cost[1] — start freely at either.
Recurrence: dp[i] = cost[i] + min(dp[i-1], dp[i-2]).
Answer: min(dp[n-1], dp[n-2]) — top floor has no toll, exit from last or second-to-last.
Space: only look back 2 positions → O(1) with two rolling variables prev2, prev1."
🔮 Likely Follow-up Questions
  • "What if you can jump 1, 2, or 3 steps?" → dp[i] = cost[i] + min(dp[i-1], dp[i-2], dp[i-3]); keep three rolling vars
  • "What if you must start from step 0 only?" → Remove the base case for step 1; track only from dp[0]
  • "Print the actual path, not just the cost" → Store parent pointers during fill, trace back at the end
  • "How is this like Fibonacci?" → Climbing Stairs (#70) is pure Fibonacci; this version adds per-step costs

7 Summary & Key Takeaways ▾
dp[i] = cost[i] + min(dp[i-1], dp[i-2]) — the canonical "min of two predecessors" recurrence
Base cases dp[0] = cost[0], dp[1] = cost[1] — you start freely at either stair
Answer = min(dp[n-1], dp[n-2]) — the top floor is toll-free; exit from the cheaper of the last two
Space compresses to O(1) by rolling just prev2 and prev1 forward — update prev2 first, then prev1
Top-down (memoization) and bottom-up (tabulation) give identical results; bottom-up avoids call stack overhead
ApproachTimeSpaceNotes
Brute Force Recursion O(2ⁿ) O(n) Good for understanding the recurrence; never use in practice
Memoization (top-down) O(n) O(n) Natural derivation from recursion; cache eliminates recomputation
Tabulation O(1) ✓ O(n) O(1) Optimal; two rolling variables, no recursion overhead
🎯 When to Recognize the "Min of Two Predecessors" DP Pattern
  • Linear sequence where each element can be reached from 1 or 2 positions back
  • Optimization goal (min/max) that accumulates over choices along the path
  • Fibonacci-shaped dependency: dp[i] depends on exactly dp[i-1] and dp[i-2]
  • No branching state: the structure is the same at every step, just the values differ
