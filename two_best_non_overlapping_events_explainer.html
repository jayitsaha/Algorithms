LeetCode #2054

Two Best Non-Overlapping Events

Given events with start, end, and value, pick at most two non-overlapping events to maximize total value.

Medium Intervals Sort + Binary Search Time: O(n log n) Space: O(n)
1 Understanding the Problem ▾

You are given a list of events. Each event has a start time, an end time, and a value. You may attend at most two events. Two events can both be attended only if they do not overlap — one must end strictly before the other begins (end < start, not end ≤ start).

Your goal: choose at most 2 non-overlapping events to maximize the sum of their values. You can attend just one if that gives the best answer.

💡 Key Insight

For each event i (as the "second" event), the optimal first event is the one with the highest value that ends strictly before event i starts. If we sort events by end time and precompute a prefix-maximum of values, we can answer "best ended event before time T" in O(log n) via binary search — giving O(n log n) overall.

✓ Valid: Two Non-Overlapping Events

Event A: [1, 3, val=4]
Event B: [4, 6, val=3]
B starts day 4 > A ends day 3
Combined value = 4 + 3 = 7

✗ Invalid: Overlapping Events

Event A: [1, 5, val=10]
Event B: [3, 7, val=8]
B starts day 3 < A ends day 5
Cannot attend both — must pick one

Edge cases: Only one event (answer = its value). All events overlap pairwise (answer = max single value). An event ends on day X and another starts on day X — they overlap (end is NOT strictly less than start). Multiple events with same start/end times.

2 The Algorithm: Sort + Prefix-Max + Binary Search ▾

The core strategy: fix one event as the "current" event, then binary search for the best compatible partner that already ended. We set this up with two sorted arrays and a prefix-max precomputation.

1 Sort events by start time. Iterate each event as the "current" (potentially second) event. We want to find the best event that ended before this one starts.
2 Build a second list sorted by end time. Extract just the end times into an array ends[]. This sorted list enables binary search for "which events ended before my start?"
3 Build a prefix-max array over the end-sorted list. prefix_max[i] = maximum value among all events in positions 0..i of the end-sorted list. This converts "best ended event by position i" into an O(1) lookup.
4 For each event (sorted by start): (a) Update answer with solo value. (b) Binary search ends[] for the last index where end < current start. (c) If found, update answer with current value + prefix_max at that index.
🔑 Why Two Separate Sorted Arrays?

We sort by start time to iterate events in "chronological start" order — treating each as a candidate second event. We need a separate sort by end time to binary search "what already ended before my start." These two orderings serve fundamentally different purposes — you cannot do both with a single sort.

Worked example with events = [[1,3,4],[2,4,2],[3,5,1],[4,6,3]]:

End-sorted: [[1,3,4],[2,4,2],[3,5,1],[4,6,3]]
ends: [3, 4, 5, 6]
prefix_max: [4, 4, 4, 4] (event[0] value=4 dominates all)

Event [1,3,4]: solo=4. bisect_left(ends,1)-1 = -1. No partner. ans=4
Event [2,4,2]: solo=4. bisect_left(ends,2)-1 = -1. No partner. ans=4
Event [3,5,1]: solo=4. bisect_left(ends,3)-1 = -1. No partner. ans=4
Event [4,6,3]: solo=4. bisect_left(ends,4)-1 = 0. pm[0]=4. 4+3=7. ans=7

3 Why It Works: The Invariant ▾
🔐 The Invariant

At any point when processing event with start = S, prefix_max[j] equals the maximum value of any single event whose end time is ≤ ends[j]. Since ends[] is sorted, bisect_left(ends, S) - 1 gives the rightmost position where end < S — the best possible non-overlapping first event for any "second event" starting at S.

Why does "end strictly less than start" matter? If event A ends on day 3 and event B starts on day 3, they are considered overlapping (both claim day 3). So we need end < start, not end ≤ start. bisect_left(ends, S) finds the first index with ends[i] ≥ S, so subtracting 1 gives the last index with ends[i] < S. This is the correct strict inequality.

⚠️ Common Off-By-One Trap

Using bisect_right(ends, start - 1) also works for integer times (finds last index ≤ start-1, i.e., < start). But bisect_left(ends, start) - 1 is cleaner and works for floating point too. Don't use bisect_right(ends, start) — that would include events ending ON the start day, violating the non-overlap condition.

Why is prefix_max non-decreasing? As we move right through end-sorted events, we only ever see the same or higher prefix maximums (we take max, never min). This means a single index lookup is all we need — no range queries or segment trees required for this problem.

💡 Analogy: The Best Colleague

Imagine you're scheduling the second meeting of your day. You want the most valuable meeting that already finished before yours starts. Instead of calling everyone to ask "are you done yet?", you've pre-sorted all meetings by end time and pinned the running best on a board — one glance (binary search) tells you the answer.

4 The Code (Annotated) ▾

Solution 1 — Sort + Prefix-Max + Binary Search (Optimal · Interview Pick)

from bisect import bisect_left
def maxTwoEvents(events): ← Takes [[start,end,val], ...]
events.sort(key=lambda e: e[0]) ← Sort by START to iterate as "second event"
end_sorted = sorted(events, key=lambda e: e[1])← Separate list sorted by END for binary search
ends = [e[1] for e in end_sorted] ← Extract end times: the array we binary-search
prefix_max = [0] * len(end_sorted) ← Will hold running max values by end time
prefix_max[0] = end_sorted[0][2] ← Base: earliest-ending event's value
for i in range(1, len(end_sorted)): ← Build left-to-right
prefix_max[i] = max(prefix_max[i-1], end_sorted[i][2]) ← Running max of values
ans = 0
for start, end, val in events: ← Try each event as "current" (solo or second)
ans = max(ans, val) ← Solo attendance: just this event
idx = bisect_left(ends, start) - 1 ← Last event ending STRICTLY before our start
if idx >= 0: ← If a compatible partner exists
ans = max(ans, val + prefix_max[idx]) ← Paired value: current + best partner
return ans
⚠️ Why not just sort once by start?

If we only had one sort (by start), when we reach event i we'd need to know "which of events 0..i-1 has the highest value AND ends before event i starts?" Checking all previous events is O(n) per query = O(n²) total. Sorting by end time separately lets us binary search this in O(log n).

Solution 2 — Min-Heap (Sweep Line, Alternative O(n log n))

import heapq
def maxTwoEvents(events):
events.sort() ← Sort by start time
heap = [] ← Min-heap of (end_time, value)
best_ended = 0 ← Best value among all finished events
ans = 0
for start, end, val in events:
while heap and heap[0][0] < start: ← Pop events that finished before this starts
best_ended = max(best_ended, heapq.heappop(heap)[1]) ← Track best ended
ans = max(ans, val, val + best_ended) ← Solo or paired with best ended
heapq.heappush(heap, (end, val)) ← Add this event for future pairing
return ans
💡 Complexity Summary

Brute Force O(n²): Try all pairs — TLE on large inputs.
Sort + Binary Search O(n log n): Optimal. Clean prefix-max precomputation.
Heap sweep O(n log n): Equally optimal. More "online" / streaming style.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Pattern

Key signals: "intervals," "non-overlapping," "pick at most k" → scheduling/interval problem. "Maximize sum of two" → we're choosing a pair, not just one. Sorting + binary search is the classic tool for interval problems.

Step 2: Propose Brute Force

"I'd try every pair (i, j), check non-overlap condition end_i < start_j, track max sum. O(n²) — too slow for n up to 10⁵. I need to reduce each query from O(n) to O(log n)."

Step 3: Key Observation — Fix One, Search for the Other

"If I fix one event as the 'current' (second) event, I only need the best event that ended before my start — a maximum-value query over a range. Sort by end time + prefix max makes this O(log n) per event."

Step 4: Design the Precomputation

"Sort events by end time. Build ends[] and prefix_max[]. For current event with start=S: bisect_left(ends, S) - 1 gives the rightmost index where end < S. prefix_max[idx] gives the best partner value in O(1)."

🗣️ What to Say to the Interviewer

"I see intervals + non-overlapping + pick at most 2 — scheduling problem.
Brute force: O(n²) try all pairs. Too slow.

Key insight: fix one event as current. Binary search for the best partner that ended before I start.

Setup: sort events by START to iterate. Separately sort by END, extract end times.
Build prefix_max[i] = max value in end_sorted[0..i]. One binary search per event = O(log n).

For each event: ans = max(ans, val, val + prefix_max[bisect_left(ends, start)-1])
Total time O(n log n), space O(n)."
🔮 Likely Follow-up Questions
  • "What if you can pick at most k events?" → DP[k][i] = best value using k events from first i sorted by end. Binary search for predecessor. O(kn log n).
  • "What if overlap means within d days?" → Change condition: end + d < start. Adjust binary search accordingly.
  • "Can you solve it without binary search?" → Yes: heap / two pointers if you sort differently. Same O(n log n).
  • "What if you want exactly 2 events?" → Same algo but only update ans from pairs (skip solo update or handle separately).

7 Summary & Key Takeaways ▾
Sort by start time to iterate events as "current event" — treating each as potential second in a pair
A separate sort by end time + prefix-max array enables O(log n) "best compatible partner" queries
Non-overlap means end strictly less than start (end < start) — bisect_left(ends, start) - 1 gives this correctly
Always consider solo attendance — the optimal answer might be a single very high-value event
Two sorted arrays for two purposes: start-sort for iteration, end-sort + prefix-max for binary search

🎯 When to Recognize This Pattern

  • "Best compatible partner for each element" → Prefix/suffix max + binary search
  • Input is intervals with values → Sort by start or end, precompute extremes
  • "Maximize sum of two items with a compatibility constraint" → Fix one, binary search for the other
