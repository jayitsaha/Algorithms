LeetCode #208

Implement Trie (Prefix Tree)

Design a data structure supporting insert, search, and startsWith operations on strings, exploiting shared prefixes for O(m) lookup time.

Medium Tries Node with Children Map insert: O(m) search: O(m) Space: O(ALPHA·n)
1 Understanding the Problem ▾

A Trie (from retrieval, rhymes with "try") is a tree in which every path from the root to a marked node spells out a stored word. Each node branches into children — one per character that can follow — and all words sharing a prefix travel the same initial path. This structural sharing is what makes prefix queries blazing fast.

We need three operations:

  • insert(word) — Walk from root, creating nodes as needed, one per character. Mark the final node is_end = True.
  • search(word) — Walk the path. Return True only if every character's node exists and the final node has is_end = True.
  • startsWith(prefix) — Walk the path. Return True if every character exists — the final node's is_end doesn't matter.
💡 Key Insight

The difference between search and startsWith is exactly one flag: is_end. Both do the same walk. Search demands node.is_end == True at the end. Prefix check returns True regardless of that flag, as long as the path exists at all.

✓ Why Tries Beat Hash Sets for Prefix Queries

startsWith("ap") → O(2). Done.
Completely independent of #words stored.
Shared prefix = shared nodes = shared computation.

✗ Hash Set Cannot Answer Prefix Queries in O(m)

Exact lookup word in s is O(m). Fine.
But "any word starting with X?" needs scanning every stored word. O(total chars).
No structural sharing — no prefix leverage.

Example Trie after inserting "app", "apple", "ape":

root
 └─ a
     └─ p
         ├─ p [is_end=True]  ← "app"
         │   └─ l
         │       └─ e [is_end=True]  ← "apple"
         └─ e [is_end=True]  ← "ape"

search("app")   → walk a→p→p, check is_end=True  → True
search("ap")    → walk a→p,   check is_end=False → False
startsWith("ap")→ walk a→p,   path exists        → True
search("apt")   → walk a→p, no 't' child         → False

Edge cases: Inserting the same word twice is harmless — is_end just stays True. A node can simultaneously be a word-end (is_end=True) and have children, e.g. the second 'p' in "app" also leads to 'l' for "apple". The empty string: walks zero steps, returns root's is_end (which is False unless explicitly inserted).

2 The Algorithm: TrieNode + Walk Pattern ▾

Every Trie node stores two things: a children map (character → TrieNode) and a boolean is_end flag. The Trie itself holds only the root — an empty anchor node. Every operation is a walk from root, one character at a time.

1 TrieNode: children = {} (grows lazily) + is_end = False. Alternative for ASCII-only: children = [None] * 26 indexed by ord(c) - ord('a'). Dict is safer for Unicode.
2 insert(word): node = root. For each char c: if c not in node.children, create new TrieNode. Descend: node = node.children[c]. After loop: node.is_end = True.
3 search(word): node = root. For each c: if c not in children, return False. Descend. After loop: return node.is_end — not True! This is the critical distinction.
4 startsWith(prefix): Identical to search except the final return is True (not node.is_end). If we consumed all prefix characters without falling off, some stored word starts with this prefix.
🔑 Unify with a _walk() Helper

All three operations share the same "follow the path, bail if missing" logic. Extract:
def _walk(self, s) -> TrieNode | None — returns the node at end of path, or None if path missing.
Then: search returns node is not None and node.is_end. startsWith returns node is not None. Cleaner and fewer bugs.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At any node N reachable from root by the path of characters c₁c₂…cₖ, the substring c₁c₂…cₖ is a common prefix of every word stored in the subtree rooted at N. Equivalently: the Trie stores every inserted word's prefix exactly once, shared across all words that begin with it.

This invariant is established trivially during insert: we create a child node for character c only when it doesn't exist yet. We never reorganize. The tree's shape is always exactly the set of all prefixes of all inserted words, with shared prefixes sharing nodes.

Why search is correct: If word W was inserted, every character in W was either found (already existed from a shared prefix) or created. The path must exist. is_end = True at the final node was set during insert, so we know a complete word ends there — not just a prefix of something longer.

Why startsWith is correct: If every character of prefix P is traversable from root, then at some point during a prior insert, a word beginning with P was processed (it created or reused those nodes). That word is still in the Trie. So a word starting with P exists.

⚠️ The is_end Flag Is Non-Negotiable

Without is_end, you can't distinguish between "app" as a word and "app" as merely the prefix of "apple". If you insert only "apple" then call search("app"), the walk a→p→p succeeds (the path exists), but the node at the second 'p' was not marked is_end — correctly returning False. Removing is_end would make search indistinguishable from startsWith.

Dual-role nodes: A node can have is_end = True AND have children simultaneously. For example, after inserting "app" and "apple", the node for the second 'p' in "app" has is_end = True (word "app") and also has child 'l' (continuing to "apple"). Both properties are independent and coexist perfectly.

4 The Code (Annotated) ▾
class TrieNode: ← one node in the tree
def __init__(self):
self.children = {} ← char → TrieNode; grows on demand
self.is_end = False ← True only if a word ends exactly here
class Trie:
def __init__(self):
self.root = TrieNode() ← universal anchor; no letter of its own
def insert(self, word: str) -> None:
node = self.root ← start every walk from root
for c in word: ← one edge per character
if c not in node.children: ← path missing? create it
node.children[c] = TrieNode()
node = node.children[c] ← descend (whether new or existing)
node.is_end = True ← mark: a complete word ends here
def search(self, word: str) -> bool:
node = self.root
for c in word:
if c not in node.children: ← fell off the tree → word was never inserted
return False
node = node.children[c]
return node.is_end ← path exists — but is it a real word?
def startsWith(self, prefix: str) -> bool:
node = self.root
for c in prefix:
if c not in node.children:
return False
node = node.children[c]
return True ← path exists → some word has this prefix
⚠️ Classic Bug: return True in search()

Returning True (not node.is_end) at the end of search is the #1 mistake. Example: insert "apple", then search("app") — the walk succeeds because "app" is a valid prefix path, but "app" was never a word. Returning True would give wrong answer. Always return node.is_end.

🔑 Array-based Children (26 slots)

For lowercase a–z only: self.children = [None]*26. Access by self.children[ord(c)-ord('a')]. Faster cache access, no hashing — but wastes 26 pointers per node regardless of how sparse the branching is. Dict is more memory-efficient for sparse alphabets or Unicode.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify the Pattern

The problem says "prefix" and gives you startsWith as an operation. Any time prefix lookup is a first-class requirement, Trie is the answer. The signal: O(m) prefix queries that a hash set can't provide without O(total) scanning. Name the data structure explicitly: "This is a classic Trie problem."

Step 2: Define the Node Structure

Say: "Each node needs to know which characters can follow it — I'll use a dict — and a boolean to record whether a complete word ends here." Clarify: "Lowercase English only? Then I could use a 26-element array. For safety/Unicode I'll use a dict." This shows you know both options.

Step 3: Explain the Unified Walk

All three methods are the same pattern: walk from root character-by-character, return early if a child is missing. The only difference is the final return value. Articulate this: "Insert creates nodes; search returns node.is_end; startsWith returns True." This is the crux.

Step 4: Complexity and Trade-offs

Time: O(m) per operation, independent of number of stored words. Space: O(ALPHA × total_characters_inserted) worst case. Mention the array-vs-dict trade-off. If asked about memory: "A compressed Trie (Patricia) merges single-child chains into one node."

🗣️ What to Say to the Interviewer

"I see we need prefix-based lookups, so a Trie is the right structure.
Each TrieNode holds a children dict and an is_end flag.
All three methods share the same walk: start at root, one step per character.
insert creates missing nodes and sets is_end=True on the last node.
search does the same walk but returns node.is_end — not True — at the end.
startsWith is identical to search but returns True regardless of is_end.
Time is O(m) per op; space is O(ALPHA times total characters inserted)."
🔑 Likely Follow-Up Questions
  • Delete a word: Walk to end, unset is_end. Optionally prune orphaned leaf nodes bottom-up (DFS).
  • Count words with prefix: Store a count at each node, increment on insert, check at prefix end.
  • Autocomplete top-k: DFS from node at prefix end, collect all is_end paths. Use a heap for top-k by frequency.
  • Word Search II (Boggle): Build Trie from word list, then DFS on grid guided by Trie to prune early.

7 Summary & Key Takeaways ▾
A Trie encodes words as root-to-leaf paths. Shared prefixes share nodes — this is the core space and time benefit.
Every operation (insert, search, startsWith) is the same pattern: walk from root one character at a time, bail early if a child is missing.
is_end flag distinguishes a complete word from a mere prefix. Search checks it; startsWith ignores it.
insert creates nodes that don't exist; search and startsWith are purely read operations with O(1) extra space.
All three operations are O(m) time where m = word/prefix length, independent of how many words are stored.
Use a dict for sparse/Unicode alphabets; use a 26-element array for dense lowercase-ASCII workloads.
🎯 When to Reach for a Trie
  • You need O(m) "does any word start with X?" queries (hash sets can't)
  • Autocomplete / typeahead — DFS from prefix end to collect completions
  • Word search on a grid (Boggle / Word Search II) — Trie prunes DFS branches early
  • Spell checkers — validate prefixes while typing, word-by-word
  • IP routing — longest prefix match via a binary Trie on bit representation
