LeetCode #1011

Capacity To Ship Packages Within D Days

Find the minimum weight capacity of a ship so that all packages are shipped within the given number of days.

Medium Binary Search BS on Answer Time: O(n log S) Space: O(1)
1 Understanding the Problem ▾

You have an array weights — packages on a conveyor belt that must be loaded in order (you cannot reorder them). A ship leaves once per day, carrying some contiguous prefix of the remaining packages, limited by a weight capacity C. Your goal: find the minimum C that allows shipping all packages in at most days days.

💡 Key Insight

The feasibility of a capacity is monotone: if capacity C lets you ship in D days, then any capacity > C is also feasible (more room, never worse). If C fails, any capacity < C also fails. This No...No...Yes...Yes pattern is exactly what binary search needs. We binary-search the capacity value itself, not the array.

Search bounds:

  • Lower bound = max(weights): If the ship can't carry the heaviest single package, it can never load it. No capacity below this can work.
  • Upper bound = sum(weights): Load everything on day 1. Always feasible. The answer is no larger than this.

Edge cases: D = 1 (capacity must = sum), D ≥ n (put one package per day; capacity = max), single package (capacity = weights[0]), all weights equal.

2 The Algorithm: Binary Search on Capacity ▾

We binary-search the answer space [max(weights), sum(weights)]. For each candidate capacity mid, we run a greedy simulation: pack packages left-to-right, starting a new day whenever the next package would exceed mid. If total days needed ≤ D, capacity is feasible and we try smaller; otherwise we must go larger.

1 Set bounds: lo = max(weights), hi = sum(weights). The minimum feasible capacity is somewhere in [lo, hi].
2 Binary search loop: While lo < hi, compute mid = (lo + hi) // 2 and run the feasibility check with capacity = mid.
3 Feasibility check (greedy): Initialize days_needed = 1, load = 0. For each package weight w: if load + w > capacity, start new day (days_needed += 1, load = 0). Then load += w. Return days_needed ≤ days.
4 Narrow search: If feasible → hi = mid (keep mid as a candidate; try to find something smaller). If infeasible → lo = mid + 1 (mid definitely too small; exclude it).
5 Return lo: When lo == hi, we have the smallest feasible capacity. Return it.

🔑 Algorithm Pattern: Binary Search on Answer (BS on Answer)

This pattern appears whenever: (1) The answer is a numeric value in a bounded range, (2) Feasibility is monotone (works for X implies works for X+1 or X-1), (3) Feasibility can be checked in O(n) time. Classic template: lo=lower, hi=upper; while lo<hi: mid=(lo+hi)//2; hi=mid if feasible(mid) else lo=mid+1; return lo. This finds the leftmost True in a [False...False...True...True] sequence.

3 Why It Works: The Invariant ▾
🔐 The Binary Search Invariant

At every iteration, the true answer lies in [lo, hi]. More precisely: lo - 1 is always infeasible (if applicable), and hi is always feasible. When lo == hi, this single remaining value is the minimum feasible capacity.

Why the greedy simulation is optimal: Since packages must be shipped in order, the greedy approach of packing as many as possible each day is optimal. No reordering or "smarter" packing can do better — with a fixed order and capacity, greedy minimizes the number of days used. If greedy fails, no strategy can succeed.

Why lo = max(weights): A package of weight w cannot be loaded if w > capacity. The ship is literally too small. Any capacity below max(weights) fails immediately in the simulation (we'd try to load a package heavier than the limit). So [lo, hi] is tight: no valid answer exists below lo.

Why hi = mid (not mid-1) when feasible: We want the minimum. When mid is feasible, it's a candidate answer — we cannot eliminate it. We set hi = mid to keep it in scope while still narrowing the range. When lo == hi, the surviving value is confirmed minimum.

⚠ Off-by-One: The Most Common Binary Search Bug

There are two standard "find leftmost" templates. This problem uses: mid = (lo+hi)//2 (round down), hi = mid if feasible, lo = mid+1 if not. Rounding down with hi = mid guarantees progress because when lo+1=hi, mid=lo, and we either set hi=lo (loop ends) or lo=lo+1=hi (loop ends). Using mid = (lo+hi+1)//2 with these update rules would create an infinite loop.

Total complexity: The feasibility check is O(n). The binary search runs O(log(sum - max)) = O(log S) iterations where S is the sum of weights. Total: O(n log S). Space is O(1) — no auxiliary arrays needed.

4 The Code (Annotated) ▾

Solution 1 — Binary Search on Capacity (Optimal, Interview Pick)

def shipWithinDays(weights: list, days: int) -> int:
lo, hi = max(weights), sum(weights) ← bounds: heaviest package to carry-all-at-once
while lo < hi: ← until search space collapses to 1 value
mid = (lo + hi) // 2 ← candidate capacity (round down for leftmost-true template)
if feasible(weights, days, mid): ← can we ship within D days at this capacity?
hi = mid ← mid works; keep it as candidate, try smaller
else: ← mid too small
lo = mid + 1 ← mid definitely fails; smallest possible is mid+1
return lo ← lo == hi == minimum feasible capacity
def feasible(weights, days, capacity) -> bool:
days_needed = 1 ← always use at least one shipping day
current_load = 0 ← weight loaded on today's ship
for w in weights: ← process packages in order (cannot reorder!)
if current_load + w > capacity: ← adding this package would overflow today's ship
days_needed += 1 ← start a new shipping day
current_load = 0 ← reset — new day starts empty
current_load += w ← load this package (onto reset or existing day)
return days_needed <= days ← True if finished within allowed days
⚠ Subtle Bug: current_load = 0 then current_load += w

When we overflow and start a new day, we reset current_load = 0 THEN add w. This correctly places the overflowing package on the new day. A common bug is to set current_load = w after overflow — that's equivalent but breaks if we also add w after the if block. Keep the structure consistent: reset to 0, then fall through to the unconditional +=.

Solution 2 — Brute Force Linear Scan (for reference)

def shipWithinDays_brute(weights, days):
for capacity in range(max(weights), sum(weights)+1): ← try every possible capacity
if feasible(weights, days, capacity): ← first feasible = minimum
return capacity ← O(S) iterations × O(n) each = O(n·S)
💡 Complexity Comparison

Binary Search: O(n · log S) time, O(1) space (S = sum − max)
Brute Force: O(n · S) time, O(1) space
With n=500, max=500: S could be ~249,500. Binary search: 500 × 18 = 9,000 ops. Brute: 500 × 249,500 = 124,750,000 ops. Binary search is >13,000× faster.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Spot "Minimize X Such That Constraint Holds"

The problem says "find the minimum weight capacity." This phrasing — minimize a value that makes a constraint (ship in D days) satisfied — is the classic signal for binary search on the answer. Recognize it immediately.

Step 2: Verify Monotonicity Out Loud

"If capacity C lets me ship everything in D days, then capacity C+1 also works — I have more room. If C fails, C-1 also fails. The feasibility function is monotone." Say this explicitly; it justifies the binary search to the interviewer.

Step 3: State the Search Space Bounds

"Lower bound: max(weights) — must be able to carry the heaviest single package. Upper bound: sum(weights) — ship everything on day 1. My answer is in [max, sum]." Tight bounds reduce iterations.

Step 4: Write the Feasibility Check First

Implement feasible(weights, days, capacity) as a standalone function. Greedy: accumulate load, start new day on overflow. Test it with a small example (capacity=6, expect 3 days for [3,2,2,4,1,4]). Then wrap in binary search.

🗣 What to Say to the Interviewer

"This asks for a minimum value satisfying a constraint — binary search on the answer.
Search space: [max(weights), sum(weights)]. Both bounds are tight and meaningful.
For any candidate capacity mid, I greedily simulate O(n): pack left-to-right, new day on overflow.
If days_needed ≤ days: feasible, try smaller (hi=mid).
If days_needed > days: infeasible, must go larger (lo=mid+1).
When lo==hi, return it. Total: O(n log S) time, O(1) space."
🔮 Likely Follow-Up Questions
  • "Can packages be split across days?" → Then answer = ceil(sum/days) — no search needed
  • "Can packages be reordered?" → Becomes bin-packing (NP-hard in general), but if sorted desc + greedy it's different
  • "What's the time complexity more precisely?" → O(n · log(sum(w) − max(w)))
  • "How does this relate to Koko Eating Bananas?" → Identical pattern: BS on eating speed, greedy check on hours needed

7 Summary & Key Takeaways ▾
Binary-search the capacity value (the answer), not the packages array
Bounds: lo = max(weights) (minimum viable), hi = sum(weights) (always works)
Feasibility = greedy O(n) simulation: pack left-to-right, new day on overflow
Monotone predicate: feasible(C) implies feasible(C+1) — this is what enables BS
Template: hi=mid when feasible, lo=mid+1 when not, return lo when lo==hi
Total complexity: O(n log S) time, O(1) space — very efficient

🎯 When to Recognize "Binary Search on Answer"
  • Problem asks for "minimum X such that constraint Y" or "maximum X such that constraint Y"
  • X is a numeric value in a contiguous bounded range
  • Feasibility of X is monotone (works for X ⇒ works for X+1 in one direction)
  • Feasibility check can be done in O(n) or O(n log n) time
