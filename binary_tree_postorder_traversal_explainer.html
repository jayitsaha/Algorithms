Binary Tree Postorder Traversal

Given the root of a binary tree, return the postorder traversal of its nodes' values.

Easy Trees DFS: Postorder Stack with Reverse Time: O(n) Space: O(n)
1 Understanding the Problem ▾

A tree traversal visits every node exactly once. The order you visit nodes defines the traversal type. The word "postorder" tells you exactly when the root is processed: after both subtrees. Post = last.

💡 Key Insight

Postorder answers: "Process all children before their parent." This is the natural order for computing tree heights, deleting a tree (free children before parent), or evaluating an expression tree (compute operands before the operator).

All three DFS traversals compared:

TraversalOrderUse CaseOutput (example)
PreorderRoot → L → RSerialize tree, copy tree[1,2,4,5,3]
InorderL → Root → RBST sorted output[4,2,5,1,3]
Postorder ✓L → R → RootDelete tree, evaluate expressions[4,5,2,3,1]

Edge cases to consider: Empty tree (return empty list), single-node tree (return [root.val]), left-skewed tree (all nodes go to left), right-skewed tree.

2 The Algorithm: Two Approaches ▾

Approach A — Recursive (simplest): The call stack naturally handles the traversal for us. Base case: null node returns empty list. Otherwise recurse left, recurse right, append root last.

A1 Base case: If root is None, return []. Every recursion needs to stop somewhere.
A2 Recurse left: left = postorder(root.left) — fully traverse the left subtree, collecting its postorder values.
A3 Recurse right: right = postorder(root.right) — fully traverse the right subtree.
A4 Return root last: return left + right + [root.val]. The root value goes at the end — that's the "post" in postorder.

Approach B — Iterative: Stack + Reverse (interview pick): Interviewers often ask for the iterative version. The key insight: postorder [L, R, Root] reversed is [Root, R, L], which is almost preorder with left and right swapped.

B1 Initialize: stack = [root], result = []. Push root onto the stack to start.
B2 Pop and collect: While stack is not empty, pop the top node and append its value to result. This collects values in Root→Right→Left order.
B3 Push children (left first, then right): Push the left child first so it goes to the bottom. Push right child second so it sits on top and is processed next (LIFO).
B4 Reverse at the end: Return result[::-1]. The Root→Right→Left list, reversed, becomes Left→Right→Root — which is postorder!
🔑 Why the Reverse Trick is Elegant

Postorder: [L, R, Root]
Its reverse: [Root, R, L]
That's just preorder with left/right swapped — easy to implement with a stack! Do it, collect the reversed result, then flip it at the end. You turn the hardest traversal into a simple modification of the easiest one.

3 Why It Works: The Invariant ▾
🔐 The Invariant (Iterative Approach)

At any point during the stack loop, the result list holds node values in Root→Right→Left order for all nodes processed so far. The stack contains the "frontier" — nodes waiting to be processed. After the loop ends, reversing the entire result yields correct postorder for every node in the tree.

Why push left before right? A stack is LIFO — the last item pushed is the first one popped. We want to process the RIGHT child before the LEFT (to build Root→R→L in result). So we push LEFT first (it goes deeper), then push RIGHT (it sits on top and pops next). Counter-intuitive: push order is reversed from process order.

Why reversal gives postorder: Root→R→L contains every node exactly once. Reversing a total sequence [a₁, a₂, ..., aₙ] yields [aₙ, ..., a₂, a₁]. The dual of Root→R→L is L→R→Root. This is a provable identity — it always holds regardless of tree shape.

⚠️ Common Confusion: Push Order vs. Process Order

You push left before right, but right is processed first. This is the LIFO paradox. The push order is the opposite of the intended process order. Always reason about which node will be on top of the stack when you pop — that's the node you'll process next.

For the recursive approach: The invariant is simpler — Python's call stack maintains the traversal order automatically. Each function call for a node completes only after both child calls complete, which naturally enforces Left→Right→Root ordering via return-value concatenation.

Edge cases are automatically handled: If a node has no left child, we don't push anything for left. If no right child, same. A null check guards each push. An empty tree returns an empty list before the loop even starts.

4 The Code (Annotated) ▾

Solution 1 — Recursive (Clean and Elegant)

def postorderTraversal(root): ← Takes TreeNode root, returns list[int]
if not root: ← Base case: null node → empty list
return []
left = postorderTraversal(root.left) ← Fully recurse left subtree first
right = postorderTraversal(root.right) ← Then fully recurse right subtree
return left + right + [root.val] ← Root LAST → that's postorder
⚠️ Why + [root.val] and not [root.val] + ?

The root must come after both subtrees. left + right + [root.val] appends root at the end. If you write [root.val] + left + right, that's preorder (root first). If you write left + [root.val] + right, that's inorder. The position of root.val is exactly what distinguishes the three traversals.

Solution 2 — Iterative: Stack + Reverse (Interview Pick)

def postorderTraversal(root): ← Iterative version using explicit stack
if not root: ← Null root edge case
return []
stack, result = [root], [] ← Stack starts with root; result collects
while stack: ← Keep going until all nodes are processed
node = stack.pop() ← LIFO: pop top node (last pushed)
result.append(node.val) ← Collect value — building Root→R→L order
if node.left: ← Push left first (goes to bottom)
stack.append(node.left)
if node.right: ← Push right second (sits on top → pops next)
stack.append(node.right)
return result[::-1] ← Reverse: Root→R→L becomes L→R→Root = postorder!
💡 Complexity Summary

Both approaches — Time: O(n) — every node is visited exactly once.
Recursive — Space: O(h) where h = tree height. O(log n) for balanced, O(n) worst case (skewed tree).
Iterative — Space: O(n) — stack can hold up to n nodes (e.g., complete binary tree at the last level). Plus O(n) for the result list.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Define Postorder Clearly

Say: "Postorder visits: Left subtree, Right subtree, then Root — the root is processed last." Draw the example tree on paper (or board) and manually write out the expected output. This prevents any ambiguity and shows you understand the problem.

Step 2: Offer the Recursive Solution First

"The recursive approach is clean: recurse left, recurse right, append root. It's O(n) time, O(h) space. The call stack handles the traversal order automatically." Write this solution first — it's fast, correct, and shows you know recursion.

Step 3: Explain the Iterative Trick When Asked

"If we need iterative, I notice postorder [L,R,Root] is the reverse of [Root,R,L]. That second sequence is preorder with left/right swapped — easy with a stack. Push left before right so right pops first, collect in reversed order, then reverse at the end."

Step 4: Trace Through the Push-Order Reasoning

Walk through: "I pop root (1), append 1. Push left (2) then right (3). Stack = [2,3]. Right (3) is on top — I pop 3 next, append 3. Continue until empty, then reverse [1,3,2,5,4] → [4,5,2,3,1]." Tracing it aloud proves the logic works.

🗣️ What to Say to the Interviewer

"Postorder means Left, Right, Root — the root is processed last.
Recursive: base case null returns empty. Return postorder(left) + postorder(right) + [root.val].
O(n) time, O(h) space where h is tree height.

For iterative: postorder reversed is Root→Right→Left.
Use a stack. Pop each node, append to result, push left then right.
Right is on top so it's popped next → result builds Root→R→L order.
Reverse at the end → Left→Right→Root = postorder.
Time O(n), Space O(n)."
🔮 Likely Follow-up Questions
  • "Iterative without the reversal?" → Use collections.deque with appendleft instead of append, push right before left
  • "All three traversals iteratively?" → Generalise with a (node, visited) tuple on the stack
  • "Preorder vs Inorder vs Postorder — when to use which?" → Preorder=serialize, Inorder=BST sorted, Postorder=delete/evaluate
  • "Morris traversal (O(1) space)?" → Uses threaded binary tree pointer manipulation — advanced topic

7 Summary & Key Takeaways ▾
Postorder = Left → Right → Root; the root is always processed last
Postorder reversed = Root → Right → Left (a modified preorder — easy with a stack)
Iterative trick: push left before right so right is on top and processed first
result[::-1] is the final transformation that yields true postorder
Postorder is essential for: deleting trees, evaluating expression trees, computing subtree aggregates bottom-up
ApproachTimeSpaceNotes

🎯 When to Recognize the Postorder Pattern
  • "Process children before their parent" → Postorder
  • "Delete/cleanup a tree" → Postorder (free children before parent)
  • "Evaluate an expression tree" → Postorder (compute operands before operator)
  • "Compute subtree aggregate: size, height, sum, path" → Postorder (bottom-up)
  • "Verify or modify every subtree before checking at current node" → Postorder
