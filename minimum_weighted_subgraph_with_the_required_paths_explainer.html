LeetCode #2203

Minimum Weighted Subgraph With the Required Paths

Given a directed weighted graph, find the minimum-weight subgraph containing paths from src1 → dest and src2 → dest.

Hard Graph Dijkstra Time: O((V + E) log V) Space: O(V + E)
1 Understanding the Problem ▾

You have a directed weighted graph with n nodes and a list of edges [from, to, weight]. You are given two source nodes src1 and src2, and a destination node dest. You need to find a connected subgraph (subset of edges) such that:

  • There is a directed path from src1 to dest
  • There is a directed path from src2 to dest
  • The total weight of selected edges is minimised

Return the minimum total weight, or -1 if no such subgraph exists.

💡 Key Insight

Both paths must reach dest. At some point, they will either travel completely independently, or they will merge at some intermediate node v and share the tail v → dest. The optimal answer is: for every possible meeting node v, compute dist(src1→v) + dist(src2→v) + dist(v→dest). The minimum over all v is the answer.

Edge Cases: If src1 or src2 cannot reach dest at all (disconnected), return -1. When src1 == dest or src2 == dest, their contribution is 0. The meeting node v could be src1, src2, or dest itself — those degenerate cases are automatically handled by the formula.

2 The Algorithm: Dijkstra from 3 Sources ▾

The formula min over v of: d1[v] + d2[v] + d3[v] requires three distance arrays:

  • d1[v] = shortest distance from src1 to v (on the original graph)
  • d2[v] = shortest distance from src2 to v (on the original graph)
  • d3[v] = shortest distance from v to dest = shortest distance from dest to v on the reversed graph
1 Build two adjacency lists: the original directed graph and its reversal (reverse every edge's direction). Both store [neighbor, weight] pairs.
2 Run Dijkstra from src1 on the original graph → produces d1, shortest distances from src1 to all nodes.
3 Run Dijkstra from src2 on the original graph → produces d2, shortest distances from src2 to all nodes.
4 Run Dijkstra from dest on the reversed graph → produces d3, shortest distances from dest backwards to all nodes. On the reversed graph, a path from dest to v corresponds to a forward path from v to dest in the original.
5 Iterate every node v: if all three distances are finite, compute d1[v] + d2[v] + d3[v]. Take the minimum. If no valid v exists, return -1.
🔑 Why the Reversed Graph for d3?

We need the shortest path from node v to dest. Dijkstra only computes distances from a single source. By reversing all edges and running Dijkstra from dest, we get: reversed-distance from dest to v = original forward distance from v to dest. This elegant trick lets us compute "cost to reach dest from any node" with a single Dijkstra run instead of n runs.

3 Why It Works: The Invariant ▾
🔐 The Invariant

In any minimum-weight subgraph where src1 and src2 both reach dest, the two paths must share a common tail. Specifically, there always exists some node v (the last "splitting point" when traversing backwards from dest) where the two paths merge. Before v each path is independent; from v onward they share the same route to dest. Trying every candidate v and evaluating the shared-tail formula is therefore complete — no valid subgraph can have a smaller cost.

Correctness sketch: Suppose the optimal subgraph has paths P1 (src1→dest) and P2 (src2→dest). Let v be the first node (scanning from dest backward) that both P1 and P2 pass through. Then: cost of subgraph ≥ cost(src1→v in P1) + cost(src2→v in P2) + cost(v→dest in shared tail). But d1[v], d2[v], d3[v] are the shortest possible costs for each leg. So our formula gives a lower bound — and that lower bound is attainable by constructing the optimal paths. Therefore the formula is exact.

⚠️ What if all three meet at dest?

When v = dest, d3[dest] = 0, and the formula reduces to d1[dest] + d2[dest] — the cost of two fully independent optimal paths with no shared edges. This is the degenerate case (no merging), which our approach handles naturally since we iterate all nodes including dest itself.

4 The Code (Annotated) ▾

Solution 1 — Dijkstra from 3 Sources (Optimal, Interview Pick)

import heapq
from collections import defaultdict
def minimumWeight(n, edges, src1, src2, dest):
g = defaultdict(list) ← original graph: g[u] = [(v, w), ...]
rg = defaultdict(list) ← reversed graph: rg[v] = [(u, w), ...]
for u, v, w in edges: ← build both adjacency lists simultaneously
g[u].append((v, w)) ← original direction u → v
rg[v].append((u, w)) ← reversed: v → u (so dest → v works for d3)
def dijkstra(src, graph): ← standard Dijkstra, returns dist array
INF = float('inf') ← sentinel for "unreachable"
dist = [INF] * n ← dist[v] = shortest known dist from src to v
dist[src] = 0 ← distance from src to itself is 0
pq = [(0, src)] ← min-heap: (dist, node)
while pq: ← process nodes in order of increasing distance
d, u = heapq.heappop(pq) ← pop node with smallest tentative dist
if d > dist[u]: continue ← stale entry: already found shorter path
for v, w in graph[u]: ← relax each outgoing edge
nd = d + w ← new candidate distance through u
if nd < dist[v]: ← improvement found
dist[v] = nd ← update dist table
heapq.heappush(pq, (nd, v)) ← push updated candidate
return dist ← dist[v] = min cost from src to v (INF if unreachable)
d1 = dijkstra(src1, g) ← shortest from src1 to every node
d2 = dijkstra(src2, g) ← shortest from src2 to every node
d3 = dijkstra(dest, rg) ← reversed: shortest from dest backward to every node
(i.e., d3[v] = cost of v→dest in original graph)
ans = float('inf') ← best cost across all candidate meeting nodes
for v in range(n): ← try every node as the meeting point
ans = min(ans, d1[v] + d2[v] + d3[v]) ← inf + anything = inf (invalid paths ignored)
return ans if ans < float('inf') else -1 ← -1 if no valid subgraph
⚠️ Common Mistake: Forgetting the Reversed Graph

Beginners try to run Dijkstra from every node v to dest — that's O(n × (V+E) log V) which TLEs. The key insight is that running Dijkstra from dest on the reversed graph gives you ALL "v → dest" distances in a single O((V+E) log V) pass.

Solution 2 — Brute Force (TLE on large inputs)

# For each node v, run full Dijkstra from v to dest separately.
# Time: O(n * (V+E) log V) — far too slow for n up to 10^5.
# Only practical for tiny graphs or as a verifier.
💡 Complexity Summary

Optimal (3× Dijkstra): Time O((V + E) log V) — three Dijkstra passes; Space O(V + E) — two adjacency lists + three dist arrays.
Brute Force: Time O(n × (V + E) log V) — one Dijkstra per candidate meeting node; hopelessly slow.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Structure

"Two sources, one destination, directed weighted graph — this is a shortest-path problem. The key constraint is that BOTH sources must reach dest. Ask: 'Do the paths have to share any edges?' The answer is: no, but they may share edges optimally."

Step 2: Formalize the Observation

"Any valid subgraph has two paths converging on dest. Sketch out what those paths look like — they either run fully separate, or they merge at some node v before reaching dest. That merge point v is the key."

Step 3: Derive the Formula

"For each candidate meeting node v, the cost is: d(src1→v) + d(src2→v) + d(v→dest). I need all three for every v. The first two come from Dijkstra on src1 and src2. The third requires all v→dest costs — I get that by reversing the graph and running Dijkstra from dest."

Step 4: Code It Up

"Build original graph + reversed graph. Run Dijkstra 3 times. Iterate all nodes, compute the sum, take the minimum. Handle the -1 case when the minimum is still infinity."

🗣️ What to Say to the Interviewer

"Both paths must reach dest. Think of the 'Y-shape': two branches converge at some meeting node v, then share the path v → dest. The cost is d1[v] + d2[v] + d3[v].
I'll run Dijkstra from src1 and src2 on the original graph to get d1 and d2.
For d3 — cost from every node to dest — I reverse all edges and run Dijkstra from dest.
Then I iterate every node v, sum the three distances, and return the minimum.
Time complexity: O((V + E) log V) for three Dijkstra runs. Space: O(V + E)."
🔮 Likely Follow-up Questions
  • "What if the graph has 3 sources?" → Run Dijkstra from all 3 sources + 1 reversed from dest; formula becomes d1[v]+d2[v]+d3[v]+d4[v].
  • "Can you handle negative weights?" → Replace Dijkstra with Bellman-Ford; complexity rises to O(VE).
  • "What if the graph is undirected?" → Reversed graph = original graph (no need for rg). Just run Dijkstra twice.
  • "How do you verify the subgraph is connected?" → If any d is INF, the sum is INF, and we return -1 — handled automatically.

def minimumWeight(n, edges, src1, src2, dest):
g, rg = defaultdict(list), defaultdict(list)
for u, v, w in edges:
g[u].append((v, w)); rg[v].append((u, w))
d1 = dijkstra(src1, g) # shortest from src1
d2 = dijkstra(src2, g) # shortest from src2
d3 = dijkstra(dest, rg) # reversed → cost v→dest
ans = float('inf')
for v in range(n):
ans = min(ans, d1[v] + d2[v] + d3[v])
return ans if ans < float('inf') else -1

7 Summary & Key Takeaways ▾
Run Dijkstra 3 times: from src1, from src2, and from dest on the reversed graph
The "reversed graph trick" converts "v → dest" into "dest → v reversed" — one Dijkstra instead of n
For each candidate meeting node v: cost = d1[v] + d2[v] + d3[v]
Minimum over all v is the answer; infinity means no valid subgraph exists → return -1
Pattern generalises to k sources: run k+1 Dijkstra passes (k forward + 1 reversed from dest)
ApproachTimeSpaceNotes
Brute Force (Dijkstra from every v) O(n·(V+E) log V) O(V+E) TLE for n ≥ 10³
3× Dijkstra ✓ (Interview Pick) O((V+E) log V) O(V+E) Optimal; elegant reversed-graph trick
🎯 When to Recognize the "Dijkstra from 3 Sources" Pattern
  • Multiple sources must reach a single destination in a weighted directed graph
  • Need to find a minimum-cost shared subgraph (paths may merge)
  • Any "v → dest" cost query → reverse the graph, run Dijkstra from dest
  • Optimal paths form a Y-shape or star converging at one meeting point
