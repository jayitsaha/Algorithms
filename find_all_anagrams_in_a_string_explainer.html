LeetCode #438

Find All Anagrams in a String

Given strings s and p, return all start indices in s where a substring is an anagram of p.

Medium Sliding Window Frequency Count Time: O(n) Space: O(1)
1 Understanding the Problem ▾

Given string s and pattern p, find the starting index of every window in s whose characters are a rearrangement of p. "cba" and "bac" and "abc" are all anagrams of "abc" — same letters, different order.

💡 Key Insight

A window of length len(p) is an anagram of p if and only if it has the exact same character frequencies. Instead of recomputing the full frequency count for every window (O(n·m)), we slide a fixed-size window and update the count incrementally — one character added, one removed — paying O(1) per step.

2 The Algorithm: Fixed-Size Sliding Window ▾

The naive approach checks every window of length m = len(p) against p from scratch, costing O(n·m). The trick is to notice that when we slide one step right, only one character enters and one leaves — 98% of the window is unchanged. We maintain counts incrementally.

We track a need counter: the number of distinct characters whose count in the window does not yet match their count in p. When need == 0, the window is an anagram.

1 Quick-fail: If len(p) > len(s), return [] — no window of size m can fit.
2 Build p_count: Count frequencies for each character in p. Set need = len(p_count) — the number of distinct characters we must satisfy.
3 Slide right: For each new character added to the window, update s_count. If that character's count in s now exactly equals its count in p, decrement need (satisfied!). If it just over-shot the target coming from an exact match, increment need (broken).
4 At full window size: If need == 0, record left in results. Then remove the leftmost character: if removing breaks an exact count match, increment need. Advance left by 1.
5 Repeat until right reaches end of s. Return collected indices.
🔑 The "need" Counter — True O(1) Match Detection

Comparing two frequency maps takes O(26) per window. The need counter compresses this into a single integer. We only update it when a count crosses the exact target threshold — either satisfying it (need−−) or breaking it (need++). This makes each add/remove operation O(1) regardless of alphabet size.

✗ Brute Force O(n·m)

n windows × m chars each:
Count window chars → O(m)
Compare to p_count → O(26)
Total: O(n·m) — TLE at n=10⁴

✓ Sliding Window O(n)

First window → O(m)
Each of n−m slides: O(1)
Match check per slide: O(1)
Total: O(n) — handles 10⁵ easily

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every point, need equals the number of distinct characters in p whose count in the current window is not exactly equal to their count in p. A window is an anagram if and only if need == 0.

Why "over-representation" is handled correctly: Say p needs 2 'a's. If our window has 3, we've over-shot. need only changes when crossing the threshold: it decremented when s_count['a'] went 1→2 (exactly matched), and it does NOT re-increment when going 2→3 (still over). When sliding removes an 'a' going 3→2, no need change (still over). Going 2→1 breaks the match, so need increments. This is exactly correct — over-counting is recognized when it eventually drops below the target.

Why the window stays exactly size m: We maintain [left, right] with right − left + 1 = m once the window is full. After every addition of right, if the window is full, we immediately slide left forward. This is unconditional — no shrink logic needed. The window size is always fixed.

⚠️ Critical Order: Record Match BEFORE Removing Left

At each full window: first check need == 0 and record left if match, then remove s[left] from the window. The window [left, right] is valid right now. After sliding, [left+1, right] is the next window — we haven't processed the new right yet.

4 The Code (Annotated) ▾

Solution 1 — Sliding Window with Need Counter (Interview Pick, O(n) time)

def findAnagrams(s: str, p: str) -> list[int]:
if len(p) > len(s): return [] ← Impossible to fit p in s
p_count, s_count = {}, {} ← Target freq map; live window freq map
for c in p: ← Build p_count once, O(m)
p_count[c] = p_count.get(c, 0) + 1
need = len(p_count) ← Distinct chars not yet satisfied
result, left = [], 0
for right in range(len(s)): ← Expand right edge one step at a time
c = s[right] ← New character entering window
s_count[c] = s_count.get(c, 0) + 1 ← Add to window
if c in p_count and s_count[c] == p_count[c]:← Exactly satisfied this char
need -= 1
if right - left + 1 == len(p): ← Window is now full size
if need == 0: ← All chars matched → anagram!
result.append(left)
lc = s[left] ← Char leaving the window
if lc in p_count and s_count[lc] == p_count[lc]:← About to break exact match
need += 1
s_count[lc] -= 1 ← Remove from window
left += 1 ← Slide left forward
return result
⚠️ Why Check lc's count BEFORE decrementing?

We check s_count[lc] == p_count[lc] before doing s_count[lc] -= 1. If we decremented first, the condition would check the wrong value. The question is: "Is this removal going to break a currently satisfied count?" — we must ask before the change.

Solution 2 — Counter Comparison (Cleaner, slightly slower in practice)

from collections import Counter
def findAnagrams(s: str, p: str) -> list[int]:
if len(p) > len(s): return []
pc = Counter(p)
wc = Counter(s[:len(p)]) ← Initial window
result = [0] if wc == pc else [] ← Check first window
for i in range(len(p), len(s)): ← Slide right edge
wc[s[i]] += 1 ← Add new right char
wc[s[i - len(p)]] -= 1 ← Remove old left char
if wc[s[i - len(p)]] == 0: ← Clean zero entries for == to work
del wc[s[i - len(p)]]
if wc == pc: result.append(i-len(p)+1)← O(26) comparison each time
return result
💡 Which to Use in an Interview?

Solution 1 (need counter) is the interview pick. It shows deeper understanding: O(1) match detection vs O(26) in Solution 2. Mention Solution 2 first as the intuitive approach, then optimize to Solution 1. Both are technically O(n) but Solution 1 has strictly better constants.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify the Fixed-Window Signal

We're looking for substrings of length exactly len(p). Fixed-length windows → fixed-size sliding window. Unlike "longest substring without repeating," we never need to grow or shrink conditionally. Every window is the same size.

Step 2: State the Brute Force

"For each of n−m+1 windows, count frequencies and compare to p. O(n·m) — too slow for n,m up to 10⁴. We're repeating work because consecutive windows share m−1 characters."

Step 3: Observe the Redundancy

"When we slide one step: one character enters, one leaves. I should update the count map incrementally rather than recomputing it from scratch."

Step 4: Introduce the Need Counter

"Instead of comparing maps each step (O(26)), I'll track how many character targets are currently unsatisfied — 'need'. Adding/removing updates need only at exact threshold crossings. O(1) per step."

Step 5: Trace Through Example

Walk s="cbaebabacd", p="abc" by hand. Show need reaching 0 at index 0 (window "cba") and index 6 (window "bac"). Confirm need stays > 0 at all other positions.

🗣️ What to Say to the Interviewer

"I see fixed-length substrings — classic fixed sliding window. Brute force is O(n·m).
Sliding window: maintain a frequency count for the window, update incrementally.
To avoid O(26) map comparison each step, I'll use a 'need' counter — the number
of chars whose window count doesn't match p. It changes only when a count crosses
the exact threshold. need==0 means anagram found. O(n) time, O(1) space (26-char alphabet)."
🔮 Likely Follow-up Questions
  • "Just need to know if any anagram exists?" → Same algorithm, return bool (#567 Permutation in String)
  • "How is this related to Valid Anagram (#242)?" → #242 checks one full-string window; #438 slides that check.
  • "Minimum Window Substring (#76)?" → Same need counter, but window shrinks when need==0 to minimize size.
  • "What if p has Unicode?" → Hash map already handles any chars. Fixed array [0]*26 only works for lowercase.
  • "What's the space complexity?" → O(1) — at most 26 keys in each map for lowercase English.

7 Summary & Key Takeaways ▾
Fixed-size window: no shrink logic — after adding right, always advance left if window is full
The "need" counter compresses O(26) map comparison into O(1) per slide
need changes only at exact threshold crossings — adding satisfied (+need−−) or removing satisfied (+need++)
Always record the match BEFORE removing the leftmost character
Space is O(1) — at most 26 keys per map for lowercase English
