LeetCode #256

Paint House

There are n houses in a row. Each house can be painted red, green, or blue — no two adjacent houses may share the same color. Given an n×3 cost matrix, find the minimum total cost to paint all houses.

Medium Dynamic Programming Min of Other Colors Time: O(n) Space: O(1)
1 Understanding the Problem ▾

Imagine n houses on a street. A painting crew can paint each house Red, Green, or Blue. Painting house i with color j costs costs[i][j] dollars. The only rule: no two neighboring houses can have the same color. Minimize total painting cost across all houses.

💡 Key Insight

To find the cheapest way to paint house i red, you only need to know: "what was the cheapest way to paint house i−1 in any non-red color?" You don't need the full coloring history — just three running numbers (one per color for the previous house). This is the "min of other colors" recurrence.

Worked Example

costs = [[17, 2, 17],
          [16, 16, 5],
          [14, 3, 19]]

Optimal: G(2) → B(5) → G(3)
Total = $10

Why Greedy Fails

At house 0: pick Green ($2) ✓
At house 1: skip Green, pick Blue ($5) ✓
At house 2: skip Blue, cheapest is Green ($3) ✓
Lucky here, but greedy can fail on
adversarial inputs — DP is always correct.

Edge cases: n=1 (min of the 3 costs directly), two houses with tied costs (any valid pair works), extreme cost differences that penalize locally cheap choices.

🔑 Why Greedy Can Fail

Imagine house 0 costs [1, 100, 100], house 1 costs [100, 1, 100], house 2 costs [100, 100, 1]. Greedy always picks the local minimum that isn't the same as the prior house — but ANY valid path here costs 1+1+1=3. They're all equal, so greedy works. Now change house 2 to [100, 99, 1]. Greedy might pick G→R→B=1+100+1=102, missing G→B→G which isn't available. The point: greedy can be forced into a locally poor choice by the constraint. DP always tracks the true optimum for each terminal state.

2 The Algorithm: Rolling DP — Min of Other Colors ▾

We process houses left to right, maintaining three running totals r, g, b — the cheapest total cost to paint all houses up to and including the current one, with the current house painted that color.

1 Base case: Set r, g, b = costs[0]. House 0 has no predecessor, so the minimum cost for each terminal color is just the painting cost itself.
2 Iterate houses 1 through n−1: For each row costs[i], extract the three painting costs as cr, cg, cb.
3 Simultaneous update — min of other colors: New red = cr + min(g, b). New green = cg + min(r, b). New blue = cb + min(r, g). All three updates use the OLD values of r, g, b — use tuple assignment.
4 Return min(r, g, b): After all houses, the answer is the minimum across the three possible terminal colors.
📐 Recurrence Relations
dp[i][R] = costs[i][R] + min(dp[i-1][G], dp[i-1][B])
dp[i][G] = costs[i][G] + min(dp[i-1][R], dp[i-1][B])
dp[i][B] = costs[i][B] + min(dp[i-1][R], dp[i-1][G])

The elegant symmetry: every color's new DP value is its own cost plus the minimum of the OTHER two colors' previous DP values. No look-ahead needed — every decision is locally computable from the previous row only.

3 Why It Works: DP Invariant & Correctness ▾
🔐 The DP Invariant

After processing house i, r (resp. g, b) holds the minimum possible total painting cost for a valid coloring of houses 0…i where house i is colored red (resp. green, blue). All adjacency constraints are baked into the value — there is no invalid coloring history that could produce a smaller number.

Why is this Dynamic Programming?

Optimal Substructure: The cheapest way to paint house i red depends on the cheapest valid coloring of houses 0…i−1 that ends in non-red. We don't need the actual colors chosen for houses 0…i−2 — only the three summary values. The subproblems compose perfectly: the min over {g_prev, b_prev} gives the exact tightest lower bound for non-red endings.

Overlapping Subproblems: Naive recursion — "try all 3 choices per house, recurse backward" — has 3^n leaves. Many subproblems are repeated: e.g., "min cost to paint houses 0..3 ending in green" would be recomputed by the red, blue, and green branches of house 4. DP computes each (i, color) pair exactly once, reducing to O(n) time.

⚠️ The Simultaneous Update Trap

When computing new_r, new_g, new_b, you MUST use the OLD r, g, b from the previous house. If you write r = cr + min(g, b) then g = cg + min(r, b), the second line uses the NEWLY updated r — which is wrong. Solution: use Python tuple assignment r, g, b = (cr+min(g,b), cg+min(r,b), cb+min(r,g)) — the right-hand side is fully evaluated before any assignment occurs.

Proof by induction: Base (i=0): r=costs[0][R], trivially optimal with no neighbors. Inductive step: assume dp values are correct for i−1. For house i painted red, the cheapest valid predecessor is min(dp[i-1][G], dp[i-1][B]) — correct by hypothesis and the adjacency constraint. Adding costs[i][R] gives the true minimum for (i, Red). The max/min operation never overestimates because both options are considered, and never underestimates because the only allowed predecessors are the non-red ones. □

4 The Code (Annotated) ▾

Solution 1 — Space-Optimized Tabulation (Optimal, O(n) time / O(1) space) — Interview Pick

def minCost(costs: list[list[int]]) -> int:
r, g, b = costs[0] ← Base case: house 0, no predecessor constraint
for cr, cg, cb in costs[1:]: ← Unpack each subsequent house's three costs
r, g, b = (cr + min(g, b), ← New r: paint red + cheapest non-red predecessor
cg + min(r, b), ← New g: paint green + cheapest non-green predecessor
cb + min(r, g)) ← New b: paint blue + cheapest non-blue; all use OLD r,g,b
return min(r, g, b) ← Minimum over all terminal colors is the answer
⚠️ Why Tuple Assignment on Lines 4–6?

Python evaluates the ENTIRE right-hand side tuple before assigning. So all three expressions cr+min(g,b), cg+min(r,b), cb+min(r,g) use the OLD values of r, g, b. Only then does Python assign the new values simultaneously. Without tuple unpacking you'd need: nr = cr+min(g,b); ng = cg+min(r,b); nb = cb+min(r,g); r,g,b = nr,ng,nb.

Solution 2 — Memoization Top-Down (intuitive, O(n) time / O(n) space)

from functools import lru_cache
def minCost(costs: list[list[int]]) -> int:
n = len(costs)
@lru_cache(maxsize=None) ← Cache (house, color) pairs — 3n unique states
def dp(i, c): ← Min cost for houses 0..i with house i = color c
if i == 0: return costs[0][c] ← Base: first house, just its cost
return costs[i][c] + min( ← Recurse: current cost + cheapest valid predecessor
dp(i-1, j) for j in range(3) if j != c)
return min(dp(n-1, c) for c in range(3)) ← Best terminal color
💡 Complexity Summary

Tabulation (bottom-up): Time O(n), Space O(1) — just three variables, no table.
Memoization (top-down): Time O(n), Space O(n) — call stack depth n, cache stores 3n entries.
Brute Force (all 3^n paths): Time O(3^n) — completely infeasible for large n.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the DP Signals

Ask yourself: sequence of decisions? Yes (n houses). Fixed choices per step? Yes (3 colors). Adjacent constraint? Yes (no same-color neighbors). Minimize total cost? Yes. These four signals together strongly indicate DP over a state per position. The state space is small: only 3 colors × n positions.

Step 2: Define State and Transition

State: dp[i][c] = min cost to paint houses 0..i with house i = color c. Transition: for each color c at house i, the best predecessor is any color ≠ c at house i−1. So dp[i][c] = costs[i][c] + min(dp[i-1][j] for j ≠ c). With only 3 colors, "j ≠ c" means exactly the other two.

Step 3: Identify Space Optimization

Notice dp[i] only depends on dp[i-1] — not any earlier row. So instead of an n×3 table, keep just three variables r, g, b (the previous row). This reduces space from O(n) to O(1). Always worth mentioning — shows you think about constant factor optimization.

Step 4: Watch the Simultaneous Update

When updating r, g, b in-place, use Python's tuple assignment so all three expressions see the OLD values. This is a common bug to flag proactively: "I'll be careful to update all three simultaneously, not sequentially."

🗣️ What to Say to the Interviewer

"I see a sequence of n decisions, each with 3 choices and an adjacency constraint — classic DP signals.
State: dp[i][c] = min cost to reach house i painted color c, respecting all prior constraints.
Transition: dp[i][c] = costs[i][c] + min of the other two colors at dp[i-1].
Base case: dp[0][c] = costs[0][c]. Answer: min(dp[n-1][0..2]).
Since each row only depends on the previous, I can use just three rolling variables — O(1) space.
I'll update them simultaneously with tuple assignment to avoid using partially-updated values."
🔮 Likely Follow-up Questions
  • "What if there are k colors instead of 3?" → O(n×k²) naive; O(n×k) with running min + second-min trick (Paint House II, #265)
  • "Can you return the actual color sequence, not just the cost?" → Keep a parent/choice table, backtrack from terminal min state
  • "What if houses are arranged in a circle?" → Fix house 0's color, run DP on houses 1..n-1 for each choice — 3 linear passes
  • "What if some houses are already painted?" → Only include unpainted houses' costs; for painted ones, force that color's "cost" to 0 and others to infinity

7 Summary & Key Takeaways ▾
DP state = (house index, color) — only 3 states per step; full history is irrelevant once we have the three summary values
Recurrence: dp[i][c] = costs[i][c] + min of the OTHER two colors from dp[i−1]
Space optimization: rolling three variables (r, g, b) instead of an n×3 table — O(1) space
Simultaneous tuple update prevents the classic bug of using a newly-updated variable in the same step
Return min(r, g, b) after all houses — pick whichever terminal color is cheapest

Brute Force (try all paths) O(3^n) O(n) Completely infeasible

🎯 When to Use This DP Sub-Pattern
  • Sequence of decisions, each with a small fixed set of choices (k options)
  • Adjacent choices are constrained (can't repeat same choice)
  • Goal: minimize or maximize total cost
  • Each row only depends on the previous row → rolling variable optimization
