LeetCode #279

Perfect Squares

Given an integer n, return the minimum number of perfect square numbers that sum to n.

Medium Dynamic Programming Unbounded Knapsack Time: O(n√n) Space: O(n)
1 Understanding the Problem ▾

A perfect square is an integer that is the square of another integer: 1, 4, 9, 16, 25, 36 … Every positive integer can be written as a sum of perfect squares — Lagrange's Four-Square Theorem guarantees the answer is always ≤ 4. But we want the fewest squares possible.

💡 Key Insight: This is Coin Change in Disguise

Treat perfect squares as coin denominations {1, 4, 9, 16, 25, …} and n as the target amount. Each denomination can be used as many times as we want (unbounded). We want the minimum number of coins. This is identically Coin Change (LeetCode #322), just with a specific coin set.

✓ n = 12 → Answer: 3

4 + 4 + 4 = 12 ✓ (3 squares)
9 + 1 + 1 + 1 = 12 (4 squares)
1×12 = 12 (12 squares)
Minimum: 3 (three 4s)

n = 13 → Answer: 2

4 + 9 = 13 ✓ (2 squares)
1 + 12 impossible (12 not square)

Minimum: 2 (one 4, one 9)

Edge Cases: n=1 → 1 (just 1²). n=4 → 1 (just 2²). n=3 → 3 (three 1²s). n=7 → 4 (four 1²s plus ... wait: 4+1+1+1=7, so 4). The answer is always in {1, 2, 3, 4} by Lagrange's theorem.

2 The Algorithm: Why This Is DP (Unbounded Knapsack) ▾

Before jumping to code, let's establish why this is a DP problem and not greedy.

Pillar 1: Optimal Substructure

The minimum squares for n depends on optimal answers to smaller sub-problems. If we use square s², we need the optimal solution for n - s². The global optimum is built from local optima.

Pillar 2: Overlapping Subproblems

Naïve recursion recomputes the same sub-targets repeatedly. dp[8] is needed when computing both dp[9] (via 1) and dp[12] (via 4) and dp[17] (via 9). Without memoization: exponential time.

⚠️ Why Greedy Fails: n = 12

Greedy always picks the largest perfect square ≤ remaining: 12 → pick 9, left with 3 → pick 1, 1, 1 → total 4 squares. But 4+4+4=12 uses only 3 squares. Greedy's local best (pick 9 now) leads to a globally suboptimal result. DP considers ALL choices at each step.

📐 The Recurrence Relation
dp[0] = 0 (base case: 0 squares to make 0)

dp[i] = min( dp[i − s²] + 1 )
            for all s = 1, 2, 3, … where s² ≤ i

i.e. try squares {1, 4, 9, 16, …} each ≤ i, take the minimum

Unbounded vs 0/1 Knapsack: In 0/1 knapsack, each item is used at most once (outer loop = items, inner = capacity, iterate capacity backward). In unbounded knapsack, items are reusable: outer loop = target values 1..n, inner loop = all valid squares. Because the inner loop's sub-target (i − s²) is always smaller than i, reusing a square is automatically handled — no duplication.

1 Initialize dp[0..n] = ∞, set dp[0] = 0. ∞ means "not yet reached." dp[0] = 0 anchors the recurrence — zero squares needed to reach zero.
2 For each target i from 1 to n: try every perfect square s² ≤ i. For each: "if we spend one copy of s² now, the remaining is i − s², which already has an optimal solution dp[i − s²]."
3 Update: dp[i] = min(dp[i], dp[i − s²] + 1). The +1 accounts for the one square s² we just used. We take the minimum across all valid squares for this target i.
4 Return dp[n]. After filling the entire table, dp[n] is exactly the minimum number of perfect squares summing to n.
3 Why It Works: The Invariant ▾
🔐 The Invariant

After the outer loop completes iteration i, dp[i] holds the exact minimum number of perfect squares that sum to exactly i. This is always correct because: (a) every dp[j] for j < i was already correctly finalized before we computed dp[i], and (b) since s² ≥ 1 always, i − s² < i always — so we only ever look at smaller, already-solved sub-problems.

Why ordering matters (outer=targets, inner=squares): When computing dp[i], we reference dp[i − s²] which has already been finalized (i − s² < i). This is exactly the bottom-up order that makes DP work. If we had done it with 0/1 knapsack ordering (outer=squares, inner=targets), each square would only be used once — but we need reuse.

Correctness of min(): We take the minimum over all possible "last square used." For each choice of final square s², the rest of the solution is forced to be optimal by induction (dp[i − s²] is already the minimum). The overall minimum is thus the minimum of all these optimal sub-problems plus 1.

💡 Lagrange's Four-Square Theorem

Every positive integer can be expressed as the sum of at most four perfect squares. This means dp[n] ∈ {1, 2, 3, 4} always. We can use this mathematically (Legendre's three-square theorem gives an O(√n) check), but the DP gives the exact answer efficiently at O(n√n).

Trace for n=3: dp[0]=0. dp[1]: try 1²=1, dp[0]+1=1 → dp[1]=1. dp[2]: try 1²=1, dp[1]+1=2 → dp[2]=2. dp[3]: try 1²=1, dp[2]+1=3; try 2²=4 > 3, skip → dp[3]=3. Correct: 1+1+1=3.

Trace for n=4: dp[4]: try 1²=1, dp[3]+1=4; try 2²=4, dp[0]+1=1 → min=1 → dp[4]=1. Correct: 4=2², just one square.

4 The Code (Annotated) — Both Tabulation & Memoization ▾

Solution 1 — Bottom-Up Tabulation DP (Interview Pick)

def numSquares(n: int) -> int:
dp = [float('inf')] * (n + 1) ← dp[i]=min squares for i; init ∞ (unreached)
dp[0] = 0 ← base: 0 squares needed to reach 0
for i in range(1, n + 1): ← fill each target 1..n in order
s = 1 ← start with smallest square s²=1
while s * s <= i: ← try each perfect square ≤ i
dp[i] = min(dp[i], dp[i-s*s] + 1) ← use s²: cost 1 + optimal for remainder
s += 1 ← next square to try (s²=(s+1)²)
return dp[n] ← minimum squares summing to n
⚠️ Why float('inf') not 0 for initialization?

We're minimizing, so we start each dp[i] at the worst possible value (infinity). Any valid sum of squares produces a finite count that replaces it. Initializing to 0 would make dp[i] appear "solved" with zero squares — wrong! The only true 0 is dp[0] = 0 (base case).

Solution 2 — Top-Down Memoization DP (Same O(n√n) complexity, recursive mindset)

from functools import lru_cache
def numSquares(n: int) -> int:
@lru_cache(maxsize=None) ← Python's built-in memo cache
def dp(rem): ← min squares to reach exactly rem
if rem == 0: return 0 ← base case: 0 remaining → 0 more squares
best = float('inf') ← worst case for this sub-problem
s = 1
while s * s <= rem: ← try each square ≤ rem
best = min(best, dp(rem - s*s) + 1) ← recurse; lru_cache avoids recomputation
s += 1
return best
return dp(n)
🔑 Tabulation vs Memoization: Which to Use?

Tabulation (bottom-up): Iterative. No recursion stack. Fills every cell 0..n. Better for large n (no stack overflow).
Memoization (top-down): Recursive. Only computes sub-problems actually needed. Easier to derive directly from the recurrence relation. Both are O(n√n) time, O(n) space.

💡 Complexity Summary

Tabulation DP: Time O(n√n) — n targets, up to √n squares each. Space O(n).
Memoization DP: O(n√n) time, O(n) space (cache + call stack).
Brute Force (no memo): O(n^(n/2)) — exponential, TLE for n > ~30.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Pattern Signals

"Minimum count of items from a fixed set that sum to a target, items reusable" → Unbounded Knapsack / Coin Change. Perfect squares are just a specific coin set: {1, 4, 9, 16, …}. This mapping is the key unlock.

Step 2: Propose Brute Force + Its Failure

"Recursively try all combinations — exponential because we recompute identical sub-targets. For example, dp[4] is computed when processing dp[5], dp[8], dp[13], dp[20] … all redundantly. Need memoization or bottom-up table."

Step 3: Define the DP State Precisely

"Let dp[i] = minimum perfect squares summing to exactly i. Base: dp[0]=0. Recurrence: dp[i] = min(dp[i−s²]+1) for all s with s²≤i. Build bottom-up i=1..n."

Step 4: State Complexity Clearly

"Outer loop runs n times. Inner loop runs O(√i) ≤ O(√n) times. Total: O(n√n) time, O(n) space. This is optimal for the DP approach — a mathematical shortcut exists (Legendre's theorem) but DP is expected in interviews."

🗣️ What to Say to the Interviewer

"This is min-count unbounded knapsack — perfect squares are my coins, n is the target.
I'll define dp[i] as the minimum squares summing to i. dp[0]=0.
For each i, I try every square s² ≤ i: dp[i] = min(dp[i], dp[i−s²] + 1).
This is bottom-up, so each dp[j] for j < i is already finalized when I need it.
Final answer: dp[n]. Time O(n√n), Space O(n)."
🔮 Likely Follow-up Questions
  • "Can you reduce space?" → No: dp[i] depends on dp[i−1], dp[i−4], dp[i−9]… we need all prior values.
  • "What if each square could only be used once?" → 0/1 Knapsack: outer=items, inner=targets, iterate backward.
  • "Is there an O(√n) math approach?" → Yes, using Legendre's three-square theorem + checking perfect squares. But DP is the expected interview answer.
  • "How does this differ from Coin Change?" → Structurally identical; only the set of denominations differs.

7 Summary & Key Takeaways ▾
Perfect Squares = Coin Change with coins {1, 4, 9, 16, …} — same DP structure
dp[i] = min(dp[i − s²] + 1) for all s with s² ≤ i
Build bottom-up: outer loop = targets 1..n, inner = squares — enables reuse (unbounded)
Greedy (always pick largest square) fails — DP considers ALL choices, takes global minimum
Answer always in {1, 2, 3, 4} by Lagrange's Four-Square Theorem

🎯 When to Recognize Unbounded Knapsack
  • "Minimum count of reusable items summing to target" → Unbounded Knapsack (this problem)
  • "Number of ways to reach target with reusable items" → Unbounded Knapsack (count variant)
  • Items are reusable + minimizing quantity → outer loop = targets, inner = items
  • Items are single-use + minimizing/counting → 0/1 Knapsack (outer = items, inner = targets backward)
