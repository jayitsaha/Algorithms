LeetCode #309

Best Time to Buy and Sell Stock with Cooldown

Given an array of stock prices, find the maximum profit with unlimited transactions but a 1-day cooldown after every sell.

Medium Dynamic Programming State Machine Time: O(n) Space: O(1)
1
Understanding the Problem
▼

You are given an integer array prices where prices[i] is the price of a stock on day i. You may complete as many transactions as you like — buy one share and sell one share, repeatedly — with one constraint: after you sell, you must wait one full day (cooldown) before buying again.

The Cooldown Constraint

If you sell on Day 3, you cannot buy on Day 4. The earliest you can buy again is Day 5. This one-day gap is what separates this problem from the simpler "unlimited transactions" variant (#122).

Example walkthrough: prices = [1, 2, 3, 0, 2]

Optimal strategy: Buy Day 0 ($1) → Sell Day 1 ($2) → Cooldown Day 2 → Buy Day 3 ($0) → Sell Day 4 ($2) = profit of $1 + $2 = $3.

Naive attempt (wrong)

Buy 0 → Sell 2 ($2 profit) → Cooldown 3 → Buy 4?
Stock is $2 and it's the last day. Can't sell. Profit = $2. Suboptimal!

Optimal (correct)

Buy 0 → Sell 1 ($1) → Cooldown 2 → Buy 3 ($0 free!) → Sell 4 ($2) = $3 total profit.

Key Insight — 3 States Capture All Possibilities

On any given day, you are in exactly one of three states: HOLD (own a stock), SOLD (just sold today, entering cooldown), or REST (no stock, not in cooldown — free to act). These three states are exhaustive and mutually exclusive.

Important Constraint

You can only hold one stock at a time. You cannot buy again while already holding. You also cannot buy the day immediately after a sell.

The Algorithm — State Machine DP
▼

The core insight is modeling the problem as a state machine. Each day we transition between states based on what action we take. We track the maximum possible profit for each state at each day.

States explained:

  • HOLD[i] — The max profit achievable if you OWN a stock at the end of day i.
  • SOLD[i] — The max profit achievable if you JUST SOLD today (end of day i in sold/cooldown state).
  • REST[i] — The max profit achievable if you're FREE (no stock, no cooldown) at end of day i.
Recurrence Relations
hold[i] = max(hold[i-1],             ← keep holding
               rest[i-1] - price[i])  ← buy today (from REST only)

sold[i] = hold[i-1] + price[i]     ← sell today (from HOLD)

rest[i] = max(rest[i-1],            ← stay resting
               sold[i-1])           ← cooldown complete

Base cases (day 0):

  • hold = -prices[0] — We buy on day 0, spending that amount.
  • sold = -infinity — Impossible to have sold without buying first.
  • rest = 0 — Start with no stock, no profit, free to act.

Final answer: max(sold, rest) — we never want to end holding unrealized value.

Why It Works — The Invariant
▼
The Core Invariant

At every step, hold, sold, and rest represent the maximum profit achievable while being in that state on that day. This invariant holds after initialization and is preserved by the recurrence.

Optimal Substructure: The best outcome tomorrow depends only on the best outcomes in each state today. We don't need to know how we got into a state — only what the max profit is for being in it. This is the hallmark of DP.

Overlapping Subproblems: Without memoization, a brute-force recursive solution would recompute "what's the best outcome starting from day 3 in REST state?" many times. The state machine DP computes each (day, state) pair exactly once.

How the cooldown is enforced:

  • You can only enter HOLD by buying from REST — not from SOLD.
  • After a sell, you enter SOLD, which feeds into REST (not directly back into HOLD).
  • This means there is always at least one REST day between selling and buying again — that's the 1-day cooldown, structurally enforced by the state graph.
Without this structure

If you allowed buying directly from SOLD state, you'd remove the cooldown. The DP would then solve the unconstrained "#122" problem instead.

With this structure

The path HOLD → SOLD → REST → HOLD guarantees at least one REST day between any sell and the next buy. Cooldown enforced by graph topology.

Space Optimization

Since each state depends only on the previous day's values, we don't need an O(n) array. Three variables suffice — O(1) space. We just have to be careful to save all previous values before computing new ones, or the updates would corrupt each other.

The Code (Annotated)
▼

Optimal Solution — O(n) time, O(1) space:

def maxProfit(prices: list[int]) -> int:
if not prices: return 0← guard: empty list → 0 profit
hold = -prices[0]← bought on day 0; profit = -cost
sold = float('-inf')← can't have sold without buying; -inf = impossible
rest = 0← starting state: no stock, no cooldown, 0 profit
for price in prices[1:]:← process days 1..n-1
prev_hold = hold← snapshot before any update
prev_sold = sold
prev_rest = rest
hold = max(prev_hold, prev_rest - price)← keep holding OR buy from REST today
sold = prev_hold + price← sell today; only valid from HOLD
rest = max(prev_rest, prev_sold)← stay resting OR cooldown complete from yesterday's sell
return max(sold, rest)← never end in HOLD (unrealized gains)
Why prev_hold and not hold?

We compute all three states simultaneously. If we used the updated hold to compute sold, we'd be computing "sell using today's hold" instead of "sell using yesterday's hold." Saving snapshots (prev_hold etc.) ensures all updates use the previous day's values, not the current day's partially-computed values.

Why max(sold, rest) and not max(sold, rest, hold)?

Ending while still holding a stock means the profit is unrealized — you can't spend it. It is always at least as good (usually better) to sell before the last day rather than remain in HOLD. Including hold in the final max could never give a strictly better answer.

Alternative: Top-Down DP with Memoization

from functools import lru_cache
def maxProfit(prices):
n = len(prices)
@lru_cache(maxsize=None)
def dp(day, state): ← state: 0=REST, 1=HOLD, 2=SOLD
if day >= n: return 0← base: no more days
if state == 0: ← REST: can buy or skip
return max(dp(day+1, 1) - prices[day], dp(day+1, 0))
elif state == 1: ← HOLD: can sell or keep
return max(dp(day+1, 2) + prices[day], dp(day+1, 1))
else: ← SOLD: must rest next day
return dp(day+1, 0)
return dp(0, 0) ← start on day 0 in REST state

Interview Strategy
▼
Pattern Recognition Signal

When you see "unlimited transactions + constraint between actions" (cooldown, transaction fee, k-limit), reach for State Machine DP. The constraint between actions forces you to model transitions explicitly.

Step 1 — Identify all states

Ask: "What are the distinct situations I can be in?" Here: holding a stock, having just sold, or being free to buy. These states must be mutually exclusive and exhaustive.

Step 2 — Write legal transitions

For each state, enumerate what you can do. HOLD → sell (goes to SOLD) or keep (stays HOLD). SOLD → must cooldown (goes to REST). REST → buy (goes to HOLD) or skip (stays REST).

Step 3 — Write the recurrences

Express each state's value as a max over all ways to reach it. This is your DP transition.

Step 4 — Find base cases

Initialize for day 0. Use -infinity for impossible states. Derive initial values logically, not by guessing.

Step 5 — Optimize space

If each step only needs the previous step's values, reduce from O(n) arrays to O(1) variables. Remember to snapshot before updating.

Interview Script — What to Say Out Loud

"I notice this has unlimited transactions but a cooldown constraint — that's a classic state machine DP pattern."
"On any given day I'm in one of three states: HOLD, SOLD, or REST."
"HOLD means I own a stock. SOLD means I just sold and am entering cooldown. REST means I'm free to buy."
"The key constraint — I can only buy from REST, not from SOLD — enforces the 1-day cooldown."
"My recurrences are: hold = max(hold, rest - price); sold = hold + price; rest = max(rest, sold)."
"I need to snapshot the previous values before updating, or the updates will use stale data."
"The answer is max(sold, rest) — I never want to end holding."
"Time: O(n). Space: O(1) since I only look back one step."
Follow-Up Questions

Q: What if cooldown is 2 days? Add a REST2 state between SOLD and REST. REST becomes REST2, and REST2 flows into REST (free). Still O(n) time, O(1) space.

Q: What if cooldown is k days? Use a circular buffer or list of k REST states. Or pass cooldown as a parameter in the top-down DP.

Q: What if there's a transaction fee? Deduct fee on sell: sold = hold + price - fee. Everything else stays the same (#714).

Summary & Key Takeaways
▼
Model "action + constraint" stock problems as state machines with 3 states: HOLD, SOLD, REST. The graph topology enforces the cooldown automatically.
The recurrences express max profit for each state using only the previous day's values — no O(n) arrays needed (O(1) space).
Always snapshot previous values (prev_hold, prev_sold, prev_rest) before computing the new ones, or you corrupt the results.
The answer is max(sold, rest), never including hold — ending while holding means unrealized profit that could always be locked in by selling earlier.
Initialized sold = -infinity represents an impossible state (haven't bought yet). This propagates correctly through the DP, keeping impossible states from "winning."
Extending to k-day cooldown: add k-1 intermediate REST states. Extending to a fee: subtract fee in the sold recurrence. The pattern scales cleanly.

When to apply State Machine DP:

  • There is a constraint between consecutive actions (cooldown, fee, transaction limit)
  • The constraint can be modeled as allowed/forbidden transitions between a small number of states
  • You want maximum/minimum over all valid action sequences
