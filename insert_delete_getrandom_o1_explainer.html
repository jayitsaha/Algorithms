LeetCode #380

Insert Delete GetRandom O(1)

Design a data structure that supports insert, remove, and getRandom — all in average O(1) time.

Medium Design Array + Hash Map Swap Time: O(1) avg Space: O(n)
1 Understanding the Problem ▾

We need to design a data structure — a class — that exposes three methods, each running in O(1) average time:

  • insert(val) — Add val if not already present. Return True if inserted, False if it was a duplicate.
  • remove(val) — Remove val if present. Return True if removed, False if not found.
  • getRandom() — Return a uniformly random element from all currently stored values.

This is a "design" problem, which means the question is really: "What combination of data structures gives us all three properties simultaneously?"

💡 Key Insight

No single data structure achieves all three in O(1). A hash set gives O(1) insert/remove but O(n) getRandom. An array gives O(1) getRandom but O(n) remove (due to shifting). The solution: use both — an array for random access, a hash map for O(1) lookup, and a clever "swap-with-last + pop" trick to achieve O(1) deletion from the array.

✗ Hash Set Alone Fails

✓ Array + HashMap Wins

Edge Cases: Inserting a duplicate should return False without modifying state. Removing a non-existent element should return False. Removing the last element in the array (no actual swap needed — the target is already at the tail). Calling getRandom on a one-element set always returns that element. All of these should be handled gracefully.

2 The Algorithm: Array + Hash Map with Swap-and-Pop ▾

We maintain two structures in lockstep: lst (a list) stores all current values, and val_to_idx (a dict) maps each value to its current index in lst. They are always in sync.

I insert(val): Check val in val_to_idx in O(1) — if found, return False. Otherwise, lst.append(val) (O(1) amortized), then record val_to_idx[val] = len(lst) - 1. The array grows by one; the map gets one new entry. Return True.
R remove(val): If val not in val_to_idx, return False. Otherwise: (1) look up del_idx = val_to_idx[val], (2) grab last = lst[-1], (3) write lst[del_idx] = last (move last element into the gap), (4) update val_to_idx[last] = del_idx, (5) lst.pop() removes the now-duplicate tail, (6) del val_to_idx[val] erases val from the map. Return True.
G getRandom(): return random.choice(lst). Because the array is always dense (no holes, always filled from index 0 to len-1), every stored value is equally reachable by index. Python's random.choice selects a random index in O(1).
🔑 Why the Swap Trick Works

Deleting from the middle of an array normally requires shifting all elements to the right — O(n) work. But since this is a set (unordered), position in the array doesn't carry meaning. So we exploit that freedom: move the last element to fill the gap left by the deleted element, then trim the tail. The only bookkeeping needed is updating the moved element's index in the hash map — a single O(1) write.

Special case to note: If the value to remove is already the last element (del_idx == len(lst) - 1), then last == val. The swap is a no-op. But we must be careful: when we do val_to_idx[last] = del_idx, and last == val, we'd be updating an entry we're about to delete. In practice the code works fine — we delete it immediately after. Just be aware of this when explaining to an interviewer.

3 Why It Works: The Invariant ▾
🔐 The Invariant

Before and after every operation: for every value v in the set, lst[val_to_idx[v]] == v. The array and map are perfectly synchronized. If you know the value, the map tells you its index. If you look at any index in the array, the map has a matching entry. This two-way consistency is what makes all three operations correct.

Trace of remove(2) restoring the invariant:

Before: lst = [1, 2, 3], map = {1:0, 2:1, 3:2}. We delete val=2 at del_idx=1. last=3 at index 2.

  • Write lst[1] = 3 → lst becomes [1, 3, 3] (temporarily)
  • Update map[3] = 1 → now map says 3 lives at index 1 ✓
  • lst.pop() → lst becomes [1, 3] — tail removed
  • del map[2] → 2 fully erased from both structures

After: lst = [1, 3], map = {1:0, 3:1}. Invariant restored: lst[0]=1↔map[1]=0, lst[1]=3↔map[3]=1. Clean.

⚠️ Common Bug: Forgetting to Update the Moved Element

A very common mistake is to write the swap in the array but forget to update the map for the moved element. Without val_to_idx[last] = del_idx, the map still thinks last is at the old index (now gone), and future operations on last will read from the wrong index. This breaks the invariant silently — your code will fail on removal of elements that were moved during a previous deletion.

Why getRandom is truly uniform: The array is always dense — no gaps. Elements occupy indices 0 through len-1 contiguously. Picking a uniformly random index in [0, n-1] maps to each element with probability exactly 1/n. There's no bias from sparse regions or empty slots.

4 The Code (Annotated) ▾

Solution 1 — Array + HashMap Swap (Optimal · O(1) average · Interview Pick)

import random
class RandomizedSet:
def __init__(self):
self.lst = [] ← Dense value array; enables O(1) random access by index
self.val_to_idx = {} ← val → current index in lst; enables O(1) lookup and delete
def insert(self, val: int) -> bool:
if val in self.val_to_idx: ← O(1) hash lookup — is val already in the set?
return False ← Set semantics: duplicates rejected without modification
self.lst.append(val) ← Append to tail — O(1) amortized (list doubles capacity when full)
self.val_to_idx[val] = len(self.lst) - 1 ← Record that val now lives at the last index
return True
def remove(self, val: int) -> bool:
if val not in self.val_to_idx:← Not in the set — nothing to do
return False
del_idx = self.val_to_idx[val]← Where in the array does val live?
last = self.lst[-1] ← Grab the tail element — it will plug the gap
self.lst[del_idx] = last ← Overwrite val's slot with last — O(1), no shifting
self.val_to_idx[last] = del_idx← Update map: last has moved to del_idx
self.lst.pop() ← Remove now-duplicate tail in O(1)
del self.val_to_idx[val] ← Erase val from the map — it no longer exists in the set
return True
def getRandom(self) -> int:
return random.choice(self.lst)← Uniform random from dense array — O(1)
⚠️ What if val is the last element?

If val == last (the element to remove is at the tail), then lst[del_idx] = last writes the same value back (no-op), and val_to_idx[last] = del_idx writes the same index back (also no-op). Then lst.pop() removes it and del val_to_idx[val] cleans up the map. The logic works correctly without any special-casing — but be aware of this when explaining your code.

Solution 2 — HashSet Only (Brute force — fails getRandom requirement)

class RandomizedSetBrute:
def __init__(self): self.s = set()
def insert(self, val): ← O(1) ✓
if val in self.s: return False
self.s.add(val); return True
def remove(self, val): ← O(1) ✓
if val not in self.s: return False
self.s.discard(val); return True
def getRandom(self): ← O(n) ✗ — list() conversion is linear!
return random.choice(list(self.s))

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Decompose the Operations

Ask what each operation individually needs. insert/remove require O(1) existence test and O(1) add/remove — that's a hash map or set. getRandom requires O(1) random access by index — that's an array. The insight is that no single structure satisfies all three, so you need to combine two.

Step 2: Identify the Conflict — Array Delete is O(n)

The only problem: removing from the middle of an array is O(n) due to shifting. Ask yourself: "Does the array need to stay ordered?" No — it's just a pool for random sampling. So you can avoid shifting entirely by using the swap-with-last trick.

Step 3: Define and Commit to the Invariant

State explicitly: "I'll keep val_to_idx[v] always equal to v's current index in lst." Every operation must restore this invariant before returning. This makes the code verifiable and shows you design data structures rigorously.

Step 4: Walk Through the Remove Edge Case

Proactively say: "One edge case is when the element to remove is already at the tail. Let me trace through: del_idx equals the last index, last equals val, we overwrite the slot with itself, update the map (same value), pop, and delete from map. Correct!" Preempting this earns bonus points.

🗣️ What to Say to the Interviewer

"Let me think about what each operation needs individually.
insert and remove need O(1) existence checks — that's a hash map.
getRandom needs O(1) random access by index — that's an array.
Neither structure alone satisfies all three, so I'll use both together.
The tricky part is achieving O(1) removal from the array.
Key insight: the array doesn't need to stay sorted or ordered — it's just a pool.
So I'll swap the target element with the last element, update the map for the moved element, then pop the tail — O(1) total.
I'll maintain the invariant: val_to_idx[v] always equals the current index of v in lst.
I also need to handle removing the last element — the swap with itself is a no-op, still correct."
🔮 Likely Follow-up Questions
  • "What if duplicates are allowed?" → LeetCode #381: store a set of indices per value in the map; remove from any one index.
  • "What if you need weighted random?" → Extend the pool so each value appears proportional to its weight; or use random.choices(lst, weights=wts).
  • "Thread safety?" → Wrap all three methods with a threading lock (threading.Lock).
  • "Worst-case vs average-case?" → Python list.append is O(n) worst-case when the underlying buffer doubles, but O(1) amortized. Hash map lookup is O(n) worst-case with hash collisions, O(1) average.

7 Summary & Key Takeaways ▾
When one data structure can't satisfy all operations, combine two — each covering the other's weakness
The swap-with-last + pop trick achieves O(1) deletion from an unordered array by exploiting that order doesn't matter
Maintain a bidirectional invariant: lst[i] and val_to_idx[v] must always agree — any operation that breaks this is a bug
getRandom is O(1) only because the array is always dense — no gaps, uniform index distribution
The removing-last-element edge case requires no special code — the swap-with-self is a benign no-op
