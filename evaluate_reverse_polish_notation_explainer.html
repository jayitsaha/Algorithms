LeetCode #150

Evaluate Reverse Polish Notation

Evaluate an arithmetic expression given in postfix (RPN) notation — operators follow their operands.

Medium Stacks Stack Operand/Operator Time: O(n) Space: O(n)
1 Understanding the Problem ▾

You are given an array of strings representing an arithmetic expression in Reverse Polish Notation (RPN), also called postfix notation. Each token is either an integer (possibly negative) or one of the operators +, -, *, /. Return the integer result of the expression.

💡 Key Insight: Operators Come AFTER Their Operands

In normal (infix) math we write 3 + 4. In RPN we write 3 4 + — the operator follows both operands. This elegantly eliminates any need for parentheses or operator precedence rules. Wherever you see an operator, it acts on the two most recently encountered numbers that haven't been consumed yet.

✗ Infix (ambiguous without parens)

2 + 1 * 3 — ambiguous!
Is it (2+1)*3 = 9
or 2+(1*3) = 5?
Needs parens + precedence rules

✓ RPN (completely unambiguous)

["2","1","+","3","*"]
= (2+1)*3 — unambiguous
["2","1","3","*","+"]
= 2+(1*3) — also unambiguous

Walk through ["4","13","5","/","+"] — this represents 4 + (13÷5):

  1. See "4" → number: push 4. Stack: [4]
  2. See "13" → number: push 13. Stack: [4, 13]
  3. See "5" → number: push 5. Stack: [4, 13, 5]
  4. See "/" → operator: pop 5 (b=right), pop 13 (a=left), compute int(13/5)=2, push 2. Stack: [4, 2]
  5. See "+" → operator: pop 2 (b=right), pop 4 (a=left), compute 4+2=6, push 6. Stack: [6]
  6. End of tokens → return stack.pop() = 6

Edge Cases: single token like ["42"] → return 42 (push then pop). Negative numbers like "-3" start with "-" but have digits — they're numbers, not operators. Division truncates toward zero: int(7/2)=3, int(-7/2)=-3 (not -4!).

2 The Algorithm: Single Operand Stack ▾

For each token, make a binary decision — is it a number or an operator?

Token
Is it in {+, -, *, /}?
→
NO → Number push int(token) onto stack
|
YES → Operator pop b, pop a → push(a OP b)
1 Initialize an empty operand stack. This is our working memory — numbers sit here until an operator needs them.
2 Scan each token left-to-right. For each token, branch on whether it's a number or an operator.
3 Token is a number? Convert from string to int and push it. It waits on the stack for a future operator to consume it.
4 Token is an operator? Pop b (first pop = right/top operand), then pop a (second pop = left operand). Compute a OP b. Push the result back. This is the key step — the result becomes an operand for the next operator!
5 After processing all tokens, exactly one value remains on the stack. Pop and return it — that's the final answer.
🔑 Why Stack Is the Natural Fit

In RPN, an operator always needs the two most recently seen, not-yet-consumed numbers. A stack's LIFO property delivers exactly this: the top two elements are always the most recently pushed. No other data structure maps as cleanly to this "last in, first used" requirement. This is also why most CPU instruction sets natively support stack-based arithmetic.

⚠️ Pop Order Is Critical!

First pop = b (right operand). Second pop = a (left operand). For addition and multiplication (commutative) this doesn't matter. But for subtraction and division it does! Expression 6 2 - means 6−2=4. If you pop in the wrong order you get 2−6=−4. Always: b = stack.pop(); a = stack.pop(); push(a OP b).

3 Why It Works: The Invariant ▾
🔐 The Stack Invariant

At any point during the left-to-right scan, the stack contains exactly the evaluated results of the complete sub-expressions we've processed so far, in order from oldest (bottom) to newest (top). When an operator arrives, it always finds exactly 2 values ready — its two operands. When all tokens are processed, exactly 1 value remains: the total result.

Why a valid RPN always terminates with exactly 1 value: Assign each token a "balance": number = +1, operator = −1. A valid RPN expression has (a) every prefix with balance ≥ 1 (stack never goes empty mid-computation), and (b) total balance = exactly 1 (ends with 1 result). This is a well-known combinatorial property of valid postfix expressions — it's why compilers and calculators that use RPN internally (like HP calculators and the JVM bytecode evaluator) are guaranteed to terminate correctly.

Why int(a/b) not a//b for division: Python's // operator floors toward negative infinity. -7 // 2 = -4. But the problem (and virtually all arithmetic systems) define division as truncating toward zero. int(-7 / 2) = int(-3.5) = -3. For positive numbers both behave identically; only negative dividends or divisors differ.

🚨 Python Floor Division Trap

-1 // 2 in Python gives -1 (floor), but the correct answer per LeetCode is 0 (truncate toward zero). This is a submission killer. Always: int(a / b). Alternatively: math.trunc(a / b).

4 The Code (Annotated) ▾

Solution 1 — Operand Stack (Optimal · Interview Pick)

def evalRPN(tokens: list[str]) -> int:
stack = [] ← Operand stack: holds integers waiting to be consumed by operators
ops = {'+', '-', '*', '/'} ← O(1) set lookup; avoids elif chain; handles "-3" (negative number, not in set)
for token in tokens: ← Single left-to-right pass: O(n)
if token not in ops: ← Token is a number (int, possibly negative)
stack.append(int(token)) ← Convert string→int, push; waits for a future operator
else: ← Token is an operator
b = stack.pop() ← RIGHT operand: top of stack (most recently pushed number)
a = stack.pop() ← LEFT operand: second from top
if token == '+': stack.append(a + b)← Addition (commutative, order irrelevant)
elif token == '-': stack.append(a - b)← Subtraction: a (left) minus b (right) — ORDER MATTERS
elif token == '*': stack.append(a * b)← Multiplication (commutative)
else: stack.append(int(a / b))← Division: truncate toward zero (not floor!)
return stack.pop() ← Valid RPN always ends with exactly 1 value on stack
⚠️ Detecting Numbers vs Operators

Using token not in ops is clean and handles negative numbers like "-3" — they start with "-" but are NOT in the operator set {'+','-','*','/'}. Alternative: try: stack.append(int(token)) in a try/except ValueError block — also correct.

Solution 2 — Dispatch Table (elegant, same complexity)

import operator
def evalRPN(tokens: list[str]) -> int:
DISPATCH = {
'+': operator.add, ← operator.add(a, b) = a + b
'-': operator.sub, ← operator.sub(a, b) = a - b
'*': operator.mul, ← operator.mul(a, b) = a * b
'/': lambda a,b: int(a/b) ← No operator.truediv — need int() wrapper
}
stack = []
for t in tokens:
if t in DISPATCH:
b, a = stack.pop(), stack.pop() ← One-liner: b=top (right), a=second (left)
stack.append(DISPATCH[t](a, b))
else:
stack.append(int(t))
return stack[-1] ← Or stack.pop(); stack[-1] avoids the pop but is equivalent
💡 Complexity for Both Solutions

Time O(n): One pass through n tokens; each push/pop is O(1).
Space O(n): Worst case: all tokens are numbers (no operators), stack grows to n. In realistic balanced expressions, stack stays small — proportional to the depth of the expression tree.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Pattern Signal

"We need to defer numbers until an operator asks for them, and each operator always needs the two most recently deferred numbers. The phrase 'most recently seen, not yet consumed' is the stack fingerprint. This is a textbook operand-stack problem."

Step 2: State the Invariant Before Coding

"I'll maintain: the stack always contains the partial results of all complete sub-expressions evaluated so far. When I hit an operator, the top two elements are exactly its two operands."

Step 3: Call Out the Two Gotchas

"Before I code, two things worth mentioning: (1) Pop order — first pop is the right operand, second is left. This matters for subtraction and division. (2) Division: int(a/b) not a//b — the problem requires truncation toward zero, not Python's floor."

Step 4: Handle Negative Numbers Cleanly

"I detect numbers with token not in {'+','-','*','/'}. This elegantly handles negative numbers like '-3' — they're strings starting with '-' but they're not in the 4-element operator set."

🗣️ What to Say to the Interviewer

"I'll use a stack of operands. For each token: if it's a number, push. If it's an operator, pop two values — the first pop is the right operand b, the second is the left operand a. Compute a OP b, push the result.
Two things worth calling out before coding: pop order matters for subtraction and division, and I'll use int(a/b) not a//b to truncate toward zero as the problem requires.
After all tokens, the stack has exactly one value — the final answer. Time O(n), Space O(n)."
