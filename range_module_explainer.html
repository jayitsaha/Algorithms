LeetCode #715

Range Module

Design a data structure that tracks ranges of numbers on a number line. Support addRange, removeRange, and queryRange on half-open intervals [left, right).

Hard Design TreeMap Intervals Time: O(n log n) per op Space: O(n)
1 Understanding the Problem ▾

Think of a coverage tracker for a streaming service. The number line represents timestamps. You can mark a range as watched, unmark a range, and check whether a range is fully covered. Ranges can overlap, merge, and split dynamically.

💡 Key Insight

All intervals are half-open [left, right): left is included, right is excluded. We only need to store the boundaries of tracked ranges in sorted order. A sorted map start → end perfectly represents the entire state.

The three operations illustrated:

addRange(10, 20) → tracked: [10,20)
addRange(30, 45) → tracked: [10,20), [30,45)
queryRange(10,20) → True (fully covered by [10,20))
queryRange(20,25) → False (gap between 20 and 30)
removeRange(14,20)→ tracked: [10,14), [30,45) (split!)
addRange(18, 35) → tracked: [10,14), [18,45) (merge!)

Why "half-open"? Half-open intervals compose cleanly: [a,b) + [b,c) = [a,c). Adjacent intervals with no gap between them can merge by simply joining at the shared boundary point.

Edge Cases: Adding a range that fully encloses existing ones (absorb all), removing a range that splits one interval into two, querying partially overlapping ranges, consecutive adds that should merge into one contiguous block.

2 The Algorithm: Sorted Map of Intervals ▾

We maintain a sorted dictionary mapping interval start → interval end. Python's SortedDict from sortedcontainers provides O(log n) binary search to find neighboring intervals. At all times, stored intervals are non-overlapping, non-adjacent, and sorted.

🔑 Why Sorted Map, Not List or Set?

We need to efficiently find the interval just before a given point (floor query) and iterate forward from there. A sorted map gives us bisect_right(x) - 1 for the floor in O(log n), then O(k) to scan k overlapping intervals — total O((k+1) log n) per operation.

addRange(left, right) — merge overlapping intervals:

1 Find left neighbor: Use bisect_right(left) - 1 to find the rightmost interval starting at or before left. If it reaches into our new range (its end ≥ left), absorb it: expand left to its start and right to max of both ends.
2 Absorb rightward: Scan forward while the next interval's start ≤ right (touches or overlaps). Expand right to max of both ends. Delete all absorbed intervals.
3 Insert merged result: Store ranges[left] = right. Invariant is restored.

removeRange(left, right) — cut and save tails:

1 Find overlapping intervals: Same floor query. Scan forward while start < right.
2 Save tails before deleting: If an overlapping interval starts before left, save [start, left). If it ends after right, save [right, end). These are the portions outside the removal zone.
3 Delete, then re-insert tails: Remove all overlapping intervals, then write back any saved tails.

queryRange(left, right) — single floor lookup:

1 Floor query: Find the rightmost interval with start ≤ left. If none exists, return False.
2 Single check: Return True iff that interval's end ≥ right. Since intervals are non-overlapping, if this one doesn't cover [left, right), nothing can.
⚠️ Critical: Adjacent Intervals Must Merge

When checking overlap in addRange, use start <= right (not < right). Intervals [10,20) and [20,30) are adjacent (20 is the boundary) — they must merge to [10,30). Using strict < would leave them as two separate entries, breaking the non-adjacent invariant.

3 Why It Works: The Invariant ▾
🔐 The Invariant

After every operation, self.ranges contains non-overlapping, non-adjacent, sorted half-open intervals that exactly represent all tracked points. No two entries touch or cross; every tracked point belongs to exactly one stored interval.

Why queryRange is O(log n): Since intervals never overlap and are sorted, there is at most one stored interval that could contain [left, right) — the one with the largest start ≤ left. One binary search + one comparison. Done.

Why addRange maintains the invariant: The sweep in step 2 is exhaustive — we keep absorbing forward while any interval's start ≤ right (touching counts as overlapping due to half-open semantics). After the sweep, the merged interval has strictly the largest possible start and end, with no overlapping neighbors remaining.

Why removeRange maintains the invariant: We save both possible tails before deleting. The left tail (if any) ends at left, the right tail (if any) starts at right. Since the removal zone is [left, right), neither tail overlaps it or each other.

✓ Valid Map State

{5:10, 20:35, 50:70}
Intervals: [5,10) [20,35) [50,70)
Each sorted, non-overlapping
No adjacency (10≠20, 35≠50)

✗ Invalid (never stored)

{5:15, 10:25}
[5,15) overlaps [10,25)
Must merge to [5,25)
addRange always merges!

4 The Code (Annotated) ▾

Solution 1 — SortedDict (Optimal, Interview Pick)

from sortedcontainers import SortedDict
class RangeModule:
def __init__(self):
self.ranges = SortedDict() ← sorted map: start → end for each [start,end)
def addRange(self, left: int, right: int) -> None:
i = self.ranges.bisect_right(left) - 1 ← index of rightmost start ≤ left
if i >= 0:
s, e = self.ranges.keys()[i], self.ranges.values()[i]
if e >= left: ← left neighbor overlaps or touches our range
left = s ← absorb: expand left boundary
right = max(right, e) ← absorb: expand right boundary
del self.ranges[s] ← remove absorbed interval
while self.ranges: ← absorb any overlapping intervals to the right
j = self.ranges.bisect_left(left)
if j >= len(self.ranges): break
s, e = self.ranges.keys()[j], self.ranges.values()[j]
if s > right: break ← next interval starts beyond our right — stop
right = max(right, e) ← extend our right to cover absorbed interval
del self.ranges[s] ← remove absorbed interval
self.ranges[left] = right ← insert merged result
def removeRange(self, left: int, right: int) -> None:
i = self.ranges.bisect_right(left) - 1
to_del, save = [], []
start_scan = i if i >= 0 else 0 ← begin from left neighbor (may extend into [left,right))
for j in range(start_scan, len(self.ranges)):
s, e = self.ranges.keys()[j], self.ranges.values()[j]
if s >= right: break ← no more overlapping intervals
if e <= left: continue ← left neighbor ends before our range — skip
to_del.append(s) ← mark for deletion
if s < left: save.append((s, left)) ← left tail: portion before removal zone
if e > right: save.append((right, e)) ← right tail: portion after removal zone
for s in to_del: del self.ranges[s]
for s, e in save: self.ranges[s] = e ← re-insert preserved tails
def queryRange(self, left: int, right: int) -> bool:
i = self.ranges.bisect_right(left) - 1 ← rightmost interval starting ≤ left
if i < 0: return False ← no interval starts at or before left
e = self.ranges.values()[i]
return e >= right ← that interval's end must cover right endpoint
⚠️ Common Mistake: Forgetting the Left Neighbor

Beginners only check intervals starting at or after left. But an existing interval [5, 15) will overlap a new range [10, 20) even though its start (5) is before our new left (10). Always check the floor interval (rightmost start ≤ left) first.

Solution 2 — Pure Sorted List (no external library)
If you can't use sortedcontainers, maintain a sorted list of [start, end] pairs and use bisect manually. Same logic, slightly more verbose index arithmetic. O(n) insert/delete in worst case due to list shifting, but often acceptable.

import bisect
class RangeModuleList:
def __init__(self): self.ivs = [] ← sorted list of [start, end] pairs
def addRange(self, l, r):
ivs, nl, nr = self.ivs, l, r
i = bisect.bisect_left(ivs, [l]) ← first interval with start ≥ l
if i > 0 and ivs[i-1][1] >= l: ← left neighbor overlaps
i -= 1; nl = ivs[i][0]; nr = max(nr, ivs[i][1])
j = i
while j < len(ivs) and ivs[j][0] <= nr:
nr = max(nr, ivs[j][1]); j += 1
self.ivs[i:j] = [[nl, nr]] ← splice: replace absorbed entries with merged one
def queryRange(self, l, r):
i = bisect.bisect_right(self.ivs, [l, float('inf')]) - 1
return i >= 0 and self.ivs[i][1] >= r
💡 Complexity Summary

SortedDict (Solution 1): O(log n) per query, O((k+1)·log n) per add/remove where k = intervals absorbed
Sorted List (Solution 2): O(log n) query, O(n) add/remove worst case (list shifting)
Space: O(n) where n = number of distinct tracked intervals

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify the Core Need

Ask yourself: "What operations do I need, and how do I find overlapping intervals efficiently?" Three operations need O(log n) lookup by value — that screams sorted structure with binary search. A hash map won't work because we need range queries.

Step 2: Choose Sorted Map Over Sorted Set

We need to store pairs (start, end) and look them up by start. A sorted map (dict keyed by start) lets us find the floor in O(log n) and store the paired end value. If the interviewer doesn't allow external libs, mention bisect on a list of pairs.

Step 3: Design Each Operation

Start with queryRange (simplest — floor + one check), then addRange (find all overlapping, merge), then removeRange (find all overlapping, cut out the middle, re-insert tails). Each shares the same "floor lookup + forward scan" pattern.

Step 4: Handle Adjacency Carefully

Walk through the example where two intervals meet exactly: [10,20) and [20,30). Show that the condition for merging in addRange must use ≤ (not <) so adjacent intervals collapse. This shows you understand the half-open interval semantics.

🗣️ What to Say to the Interviewer

"I need to store a dynamic set of non-overlapping intervals sorted by start.
I'll use a SortedDict mapping start → end. All ops share a floor lookup pattern.
queryRange: find largest start ≤ left, check if its end ≥ right. O(log n).
addRange: find floor, absorb all intervals touching [left,right), insert merged. O((k+1)logn).
removeRange: find overlapping intervals, save any left/right tails outside [left,right),
delete overlapping entries, re-insert tails. Same complexity.
Key invariant: after every op, stored intervals are non-overlapping, non-adjacent, sorted."
🔮 Likely Follow-up Questions
  • "What if you can't import sortedcontainers?" → Use bisect on a sorted list of [start,end] pairs
  • "How would you serialize/restore the module?" → Just serialize the sorted list of intervals
  • "How would you count total tracked numbers?" → Sum (end - start) for all intervals in O(n)
  • "What if intervals can be 64-bit integers?" → No change — we store boundaries, not bitmaps

7 Summary & Key Takeaways ▾
Store intervals in a sorted map (start → end); never store overlapping or adjacent pairs
All three operations share the same "floor lookup via bisect_right(x)-1" entry point
addRange merges by absorbing the left neighbor and then sweeping rightward
removeRange saves left/right tails before deleting, then re-inserts them
queryRange needs only a single floor lookup + one bound check — O(log n)
Adjacency condition: use ≤ not < when checking overlap to merge touching intervals
