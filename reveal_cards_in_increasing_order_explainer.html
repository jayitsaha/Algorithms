LeetCode #950

Reveal Cards In Increasing Order

Given an integer array deck, reorder it so that when cards are revealed one-by-one using the ritual (reveal top, move next to bottom), they appear in increasing order.

Medium Queues Simulate Reverse Process Time: O(n log n) Space: O(n)
1 Understanding the Problem ▾

Imagine a dealer performing a magic trick. They hold a deck of cards face-down and follow an exact ritual: flip the top card face-up (reveal it), then take the new top card and move it to the bottom of the deck. Repeat until all cards are revealed. The audience sees cards appearing one at a time.

The puzzle: given a list of numbers, arrange a deck so that following this ritual reveals the numbers in increasing order. If you want the audience to see 2, 3, 5, 7, 11 in that order, what should the deck look like before you start?

💡 Key Insight

Going forward is hard — each card placement affects all future positions. Instead, figure out which positions get revealed in which order by simulating the ritual on an index queue. Then assign sorted cards to those positions in order. Working from the "what positions are revealed when" question makes the solution fall out naturally.

The Reveal Ritual (Forward)

deck = [A, B, C, D, E]
reveal A → move B to bottom
reveal C → move D to bottom
reveal E → move B to bottom
reveal D → reveal B
Revealed: A, C, E, D, B

✓ Want Increasing Reveals

sorted = [2, 3, 5, 7, 11]
want position 0 revealed 1st → 2
want position 2 revealed 2nd → 3
want position 4 revealed 3rd → 5
want position 3 revealed 4th → 7
want position 1 revealed 5th → 11

Edge Cases: Single card (trivially correct), two cards (first reveals, second moves then reveals), all same values (any arrangement works), and deck of length 1 (a degenerate case handled by the if-check before rotation).

2 The Algorithm: Index Queue Simulation ▾

We exploit the fact that the reveal ritual is fully deterministic. By running it on a queue of indices (not card values), we learn exactly which positions will be revealed first, second, third, etc. Then we assign the sorted values to those position slots.

1 Sort the deck ascending — [2, 3, 5, 7, 11, 13, 17]. These values will be assigned to positions in this order: smallest goes to the first-revealed position.
2 Build an index queue — deque([0, 1, 2, ..., n-1]). This represents the position slots, ordered as they appear "on top" of our virtual deck of positions.
3 For each sorted card, pop the front index: that index is the next position to be revealed. Place this card at result[index]. This is the "reveal" step.
4 Rotate one index: pop the new front index and move it to the back of the queue. This mirrors "move top card to bottom" — that position is skipped for now and will come around again later.
5 Return result — the array now holds cards in exactly the arrangement that produces increasing reveals when the ritual is applied.

🔑 Why Index Queue Works

The index queue undergoes exactly the same "reveal and rotate" operations as a real deck would during the ritual. By running these operations on indices instead of values, we discover the precise order that positions get revealed — without needing to know card values at all. Assigning sorted cards to those positions in order is then trivially correct.

3 Why It Works: The Invariant ▾
🔐 The Invariant

After processing the k-th sorted card, result[i] holds the k-th smallest value at the position that will be k-th revealed in any real run of the ritual. The index queue at each step faithfully replicates the state of the deck positions — so every popleft() call yields exactly the next position to be revealed.

Why the rotation is isomorphic to the real ritual: In the actual reveal, after revealing a card the dealer moves the new top card to the bottom. In our simulation, after popping the reveal-position index we pop the next index and push it to the back. These are the same structural operation on the same ordered sequence — just performed on indices rather than values.

Why sort ascending matters: The first position popped from the queue is the one revealed first. We want the first reveal to show the smallest number. So we assign sorted[0] (smallest) to the first-popped position, sorted[1] to the second-popped position, and so on. This one-to-one assignment of sorted order to reveal order is the core correctness guarantee.

⚠️ Common Mistake: Trying to Simulate Forward Without the Index Queue

Many beginners try to directly simulate the reveal process and fill in card values as they go. This fails because placing one card changes the relative position of all remaining cards — you'd need to re-simulate from scratch after each placement, giving O(n²) complexity at minimum. The index queue sidesteps this entirely by computing all reveal positions upfront.

The guard if index_queue: prevents trying to rotate when only one card remains. For the last card, there's nothing to move to the bottom — the if-check handles this edge case cleanly.

4 The Code (Annotated) ▾

Solution 1 — Index Queue Simulation (Optimal, Interview Pick)

from collections import deque ← Need O(1) popleft; list.pop(0) is O(n)
def deckRevealedIncreasing(deck):
n = len(deck) ← Save length; we'll iterate exactly n times
deck.sort() ← Sort ascending: smallest card revealed first
result = [0] * n ← Output slots; we'll fill these by position
index_queue = deque(range(n)) ← deque([0,1,...,n-1]): positions in "deck order"
for card in deck: ← Walk sorted values smallest to largest
result[index_queue.popleft()] = card ← Front index = next revealed position → assign card
if index_queue: ← Don't rotate if this was the last card
index_queue.append(index_queue.popleft())← Move new front to back: "skip" step of ritual
return result ← Arranged for increasing reveals
⚠️ Why Two popleft() Calls Per Iteration?

The first popleft() grabs the position to reveal this card (and assigns it). The second popleft() followed by append() moves the new top to the bottom — exactly mirroring "take the new top card and put it at the deck's bottom." Two different operations, both using popleft(), but serving different purposes.

Solution 2 — Brute Force with List (O(n²), for conceptual clarity)

def deckRevealedIncreasing_brute(deck):
n = len(deck)
deck.sort() ← Same sorting step
indices = list(range(n)) ← Regular list — pops are O(n)
result = [0] * n
for card in deck:
result[indices.pop(0)] = card ← O(n) shift! That's the bottleneck
if indices:
indices.append(indices.pop(0)) ← O(n) shift again — O(n²) total
return result
💡 Complexity Summary

Index Queue (Optimal): Time O(n log n) — sort dominates. All queue ops O(1) each × n iterations = O(n). Space O(n) for result array + index queue.
Brute Force List: Time O(n²) — O(n) for each pop(0) and n iterations. Fine for understanding; fails at n=500+.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Trace the Ritual by Hand

On a small example (3 cards), manually simulate the reveal ritual on indices [0, 1, 2]. Write down which index is revealed first, second, third. You'll get: 0, 2, 1. This gives you the "reveal order" without needing card values.

Step 2: Connect Reveal Order to Card Assignment

"If position 0 is revealed first and I want increasing reveals, position 0 gets the smallest card. Position 2 is revealed second, so it gets the second-smallest. Position 1 is last, so it gets the largest." The mapping is clear once you know the reveal order.

Step 3: Generalize to Queue Simulation

Instead of manually tracing, use a queue of indices. Apply the exact same ritual operations (pop front → rotate new front to back) to the queue. This computes the reveal order efficiently in O(n) total queue operations.

Step 4: State the Complexity and Data Structure Choice

Emphasize: "I use a deque specifically so that both popleft and append are O(1). A regular list would make popleft O(n), degrading the whole solution to O(n²)." This shows you know your data structures.

🗣️ What to Say to the Interviewer

"The key question is: in what order are positions revealed? If I can answer that, I just
assign sorted values to those positions in that order.
I'll simulate the ritual on a deque of indices [0..n-1], applying the same reveal+skip
mechanics but on positions, not values. Sort deck ascending. For each sorted card:
pop front index from deque — that's the reveal position — assign card there.
Then rotate: pop new front, push to back. This is O(n) queue work + O(n log n) sort.
Choosing deque over list is critical: popleft is O(1) on deque, O(n) on list."
🔮 Likely Follow-up Questions
  • "What if we want decreasing order?" → Sort descending; same simulation
  • "What if k cards move to the bottom before each reveal?" → Rotate k times per iteration
  • "Can you do it in O(n) time?" → Not without a better-than-comparison sort; O(n log n) is tight here
  • "What is the reveal order permutation for n = power of 2?" → It's the bit-reversal permutation (worth knowing!)

7 Summary & Key Takeaways ▾
Simulate the ritual on indices, not values — to find the reveal order of positions without knowing card values
Two operations per card: popleft() to reveal + popleft()/append() to rotate the "skip" card to the back
Must use deque for O(1) popleft — a list would make each pop O(n), blowing up to O(n²) total
Sorting (O(n log n)) dominates — the queue simulation itself is O(n) with O(1) per operation
Pattern signal: any problem where items are "revealed and skipped cyclically" → queue simulation
ApproachTimeSpaceNotes
Brute Force (list.pop(0)) O(n²) O(n) Correct but slow — list shifts on every pop
Index Queue (deque) ✓ O(n log n) O(n) Optimal; sort dominates, queue is O(n) total
🎯 When to Recognize This Pattern
  • "Elements are revealed/used and some move to the end cyclically" → queue simulation
  • "Find the arrangement producing a target output order" → simulate on indices, then assign
  • "Every other item (or every k-th item) is skipped" → deque with rotate
  • Josephus-style circular elimination → queue pop + append
