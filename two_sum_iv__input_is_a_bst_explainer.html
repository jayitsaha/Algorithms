LeetCode #653

Two Sum IV — Input is a BST

Given the root of a BST and an integer k, return true if there exist two elements in the BST such that their sum equals k.

Easy Trees In-order + Two Pointers Time: O(n) Space: O(n)
1 Understanding the Problem ▾

You are given a Binary Search Tree (BST) and a target integer k. Your task: determine whether there exist two distinct nodes whose values sum to exactly k.

This is the classic Two Sum problem in disguise — except the data lives in a tree rather than an array. The key question is whether we can exploit the BST's sorted structure instead of exhaustively trying every pair in O(n²).

💡 Key Insight

A BST visited via in-order traversal (left → node → right) yields values in strictly increasing sorted order. Once you have a sorted sequence, Two Sum is trivially solved in O(n) with two pointers converging from both ends. The entire algorithm is: in-order traversal → sorted array → two-pointer scan.

✓ Found Pair — k=9

BST values: 2, 3, 4, 6, 7
In-order: [2, 3, 4, 6, 7]
L=2, R=7 → 2+7=9 ✓ → True

✗ No Pair Found — k=28

BST values: 2, 3, 4, 6, 7
In-order: [2, 3, 4, 6, 7]
Max sum = 6+7=13 < 28 → False

Edge Cases: Single-node tree (can't form a pair → always False), k that requires using the same node twice (disallowed — L must be strictly less than R), negative node values, and k that's smaller than the two smallest elements or larger than the two largest.

2 The Algorithm: In-Order Traversal + Two Pointers ▾

The optimal solution fuses two classical sub-algorithms: (1) BST in-order DFS to extract a sorted sequence, and (2) the two-pointer "close in from both ends" technique for finding a target pair sum in a sorted array.

1 In-order DFS: Traverse the BST recursively — go left first, then append the current node's value to a list, then go right. Because of the BST property (left < root < right at every subtree), this yields values in ascending sorted order.
2 Initialize two pointers: Set L = 0 (pointing to the smallest element) and R = len(nums) - 1 (pointing to the largest). The key invariant: if any valid pair exists, it lies within the window [L, R].
3 Converge pointers: Compute s = nums[L] + nums[R]. If s == k → found, return True. If s < k → the sum is too small; move L right (use a bigger number). If s > k → the sum is too big; move R left (use a smaller number).
4 Terminate: Loop while L < R. When they meet or cross (L ≥ R), all pairs have been checked and no solution exists — return False. Using L < R (not L != R) prevents ever picking the same element twice.
Brute Force — O(n²)

For each node u:
  DFS for node v where
  u.val + v.val == k

Ignores BST structure entirely.

Optimal — O(n)

In-order → sorted list
Two pointers on sorted list
Both phases are O(n).

Fully leverages BST property.

🔑 Alternative: Hash Set DFS (O(n) time, O(n) space)

During any DFS traversal, for each node with value v: check if k - v is in a seen set (meaning the complement was already visited). If yes → True. Otherwise add v to the set. Same O(n)/O(n) complexity; works even on non-BST binary trees. Simpler code, but doesn't demonstrate BST knowledge. Mention both in an interview.

3 Why It Works: The Invariants ▾
🔐 BST In-Order Invariant

At any node in a BST: every value in the left subtree is strictly less than the node's value, and every value in the right subtree is strictly greater. Visiting left → root → right thus always processes values in ascending order. This is the defining property of BSTs — it's what makes them useful for ordered lookups.

🔐 Two-Pointer Correctness Invariant

After each step, the following holds: if there exists a valid pair (a, b) with a + b = k, then a = nums[i] for some i ≥ L and b = nums[j] for some j ≤ R. In other words, no valid pair ever falls outside the current [L, R] window.

Why moving L right is safe when sum < k: We have nums[L] + nums[R] < k. For any index i < L, nums[i] ≤ nums[L] (sorted), so nums[i] + nums[R] ≤ nums[L] + nums[R] < k. No element to the left of L can help — we can safely discard all pairs involving nums[L] and advance L.

Why moving R left is safe when sum > k: By symmetric argument: nums[L] + nums[j] ≥ nums[L] + nums[R] > k for all j > R (since arr is sorted). Discard all pairs involving nums[R] and move R left.

⚠️ The "Same Node Twice" Trap

The problem says "two elements" — implying two distinct nodes. If k is exactly twice some node's value (e.g., node=5, k=10), you must not count it as (5, 5). The condition while L < R ensures L and R are always at different indices, so we never pair a node with itself. This is the correct guard.

Why in-order gives exactly sorted output: Consider any two nodes A and B in the BST where A is an ancestor of B. If B is in A's left subtree, then B is visited before A (in-order processes left first). If B is in A's right subtree, B is visited after A. BST property ensures left subtree values < root < right subtree values, so every earlier visit produces a smaller value. Induction gives us a fully sorted output.

4 The Code (Annotated) ▾

Solution 1 — In-Order + Two Pointers (Interview Pick, O(n) time)

def findTarget(root, k): ← root = BST root node, k = target sum
nums = [] ← Collects BST values in sorted order
def inorder(node): ← Inner helper: DFS left → append → right
if not node: return ← Base case: null node means leaf's child
inorder(node.left) ← Recurse into left (smaller values) first
nums.append(node.val) ← Append node in correct sorted position
inorder(node.right) ← Recurse into right (larger values) after
inorder(root) ← Execute traversal; nums is sorted after this
L, R = 0, len(nums) - 1 ← L at smallest, R at largest
while L < R: ← Stop when L meets R (no more distinct pairs)
s = nums[L] + nums[R] ← Try current smallest + largest
if s == k: return True ← Perfect match — we're done
elif s < k: L += 1 ← Too small: increase by moving L right
else: R -= 1 ← Too big: decrease by moving R left
return False ← All pairs exhausted, no solution
⚠️ Why not check s < 0 or guard against missing nodes?

BST values can be negative, so a negative sum is perfectly valid — don't add any "s < 0" guard. The only condition that matters is how s compares to k. Also: inorder handles all null checks via if not node: return, so there's no risk of NoneType errors in the traversal.

Solution 2 — Hash Set DFS (O(n) time, simpler derivation)

def findTarget(root, k):
seen = set() ← Stores all node values seen during DFS
def dfs(node):
if not node: return False
if k - node.val in seen: ← Complement already visited? Pair found!
return True
seen.add(node.val) ← Record this val for future complement checks
return dfs(node.left) or dfs(node.right) ← Search both subtrees
return dfs(root)
💡 Complexity Summary

In-Order + Two Pointers: Time O(n), Space O(n) for the nums array + O(h) recursion stack where h = tree height. Best demonstrates BST understanding.
Hash Set DFS: Time O(n), Space O(n) for the set. Works on any binary tree. Simpler to code.
BST Iterator Approach (advanced): Time O(n), Space O(h) — uses two lazy iterators (one ascending, one descending) stepping through the BST without storing all values.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the "Two Sum" Pattern

The question asks: "do two elements sum to k?" That's Two Sum. In a sorted array you'd use two pointers (O(n)). In an unsorted array you'd use a hash set (O(n)). Here, data is in a BST — which is an implicit sorted structure. Signal: "Can I extract a sorted array cheaply?"

Step 2: Propose the Brute Force

"Naive: for every node, search the entire tree for its complement. O(n²) — too slow. Or flatten tree to list naively (ignoring BST property) then use hash set — O(n) time but misses the BST angle."

Step 3: Exploit BST = Sorted Sequence

"BST in-order gives sorted array in O(n). Once sorted, two-pointer Two Sum is also O(n). Combined: O(n) time, O(n) space. This is optimal since we must read all nodes at least once."

Step 4: Offer the Hash Set Alternative

"Alternatively, hash set DFS: for each node v, check if k-v is in the set, then insert v. O(n)/O(n), and works even on non-BSTs. If I don't need to demonstrate BST knowledge, this is simpler to code in an interview."

🗣️ What to Say to the Interviewer

"This is Two Sum on a BST. Key: BST in-order traversal = sorted sequence.
Plan: (1) in-order DFS to collect values into sorted list nums. O(n).
(2) Two pointers L=0, R=len-1. While L < R:
sum too small → L++; sum too big → R--; equal → return True.
(3) If loop ends → return False.
Time: O(n). Space: O(n) for the list + O(h) recursion stack.
Alternatively, hash set DFS works in one pass — I can code either."
🔮 Likely Follow-up Questions
  • "Can you do it in O(h) space instead of O(n)?" → Yes: two BST iterators (one forward, one reverse), each using O(h) stack space, step in-sync like two pointers
  • "What if it's just a binary tree, not a BST?" → In-order doesn't give sorted; use hash set DFS instead
  • "Find ALL pairs that sum to k?" → Collect full in-order list, run two pointers collecting all pairs where sum == k (don't return early)
  • "Three Sum in a BST?" → In-order to sorted list, then run 3Sum algorithm on the list
