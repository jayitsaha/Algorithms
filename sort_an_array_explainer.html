LeetCode #912

Sort an Array

Given an array of integers nums, sort the array in ascending order and return it — without using any built-in functions which run in O(n log n) time or better.

Medium Sorting Classic Merge Sort Time: O(n log n) Space: O(n)
1 Understanding the Problem ▾

You're given an array of integers (which may include negatives, duplicates, and zeros) and must return it sorted in ascending order. The catch: you cannot call sorted(), .sort(), or any built-in sort that runs in O(n log n) or better. You must implement a real sorting algorithm from scratch.

💡 Key Insight

The constraint "no built-in sort, n up to 5×10⁴" rules out O(n²) sorts (Bubble, Insertion, Selection) — they would time out. The expected solution is Merge Sort: the canonical divide-and-conquer sort that guarantees O(n log n) in all cases and is also stable. It's the safest interview answer.

Input (unsorted)

[5, 2, 3, 1]
[5, 1, 1, 2, 0, 0]
[-4, 1, 0, -3, 2]

Output (sorted ascending)

[1, 2, 3, 5]
[0, 0, 1, 1, 2, 5]
[-4, -3, 0, 1, 2]

Edge Cases to consider: empty array (return as-is), single element (already sorted), all identical elements (duplicates balanced fine), all negatives, and already-sorted input (Merge Sort still runs in O(n log n), unlike nearly-sorted inputs where Insertion Sort shines).

2 The Algorithm: Classic Merge Sort (Divide & Conquer) ▾

Merge Sort is a divide-and-conquer algorithm. It splits the array in half, recursively sorts each half, then merges the two sorted halves back into one. The key insight: merging two already-sorted arrays is O(n) using two pointers — we simply always pick the smaller of the two front elements.

1 Base case: If the array has 0 or 1 element, it's already sorted — return it immediately. This is where all recursion branches terminate.
2 Divide: Find midpoint mid = len(arr) // 2. Slice into left = arr[:mid] and right = arr[mid:]. These create new subarrays (copies), not views.
3 Conquer: Recursively call merge_sort(left) and merge_sort(right). By the inductive invariant, when these return, both halves are fully sorted.
4 Merge: Use two pointers — i for left, j for right. Always pick min(left[i], right[j]) and append to output. After either pointer is exhausted, drain the other half's remaining elements (they're already in order).
5 Write back: Update the original array in-place via nums[:] = merge_sort(nums) — the slice assignment mutates the existing object without rebinding the reference.

🔑 Why Merge Sort is the Interview Pick

Merge Sort guarantees O(n log n) in all cases — no worst-case traps unlike Quick Sort (which degrades to O(n²) on adversarial input without randomization). It is also stable (equal elements maintain relative order). The tradeoff: O(n) auxiliary space for the temp arrays. For reliability + correctness in interviews, Merge Sort wins.

3 Why It Works: The Invariant ▾
🔐 The Invariant

After merge_sort(arr) returns, the array arr is sorted in non-decreasing order. This holds at every level of the recursion — from single-element base cases (trivially sorted) up through the full array.

Why the merge step is correct: Before merging, both left and right are sorted (by the recursive invariant). At each step of the two-pointer merge, we compare only left[i] and right[j] — these are the smallest unplaced elements in each half. The smaller of the two is therefore the globally smallest remaining element. Picking it always gives the correct next output element.

Why O(n log n) — the intuition: The recursion tree has depth log₂(n) — you can halve n at most log₂(n) times before reaching size 1. At each depth level, every element is touched exactly once during merging — so total work per level is O(n). Multiply: O(log n) levels × O(n) per level = O(n log n).

⚠️ Common Mistake: Forgetting to Drain the Remaining Half

When one pointer reaches the end of its subarray, the while loop exits — but the other half may still have elements. Since that half is sorted, all remaining elements are already in the right relative order. Simply do result += left[i:] and result += right[j:] to drain both (one will be empty). Skipping this produces a truncated result.

Stability proof: In the comparison left[i] <= right[j], we use <= (not <). When values are equal, we take from the left half first. Since left comes before right in the original array, equal elements maintain their original relative order — the definition of a stable sort.

Solution 1 — Classic Merge Sort (Interview Pick)

def sortArray(nums): ← Entry point; returns sorted nums
def merge_sort(arr): ← Recursive divide-and-conquer helper
if len(arr) <= 1: ← Base case: 0 or 1 element is sorted
return arr
mid = len(arr) // 2 ← Split point (floor division)
left = merge_sort(arr[:mid]) ← Recursively sort left half
right = merge_sort(arr[mid:]) ← Recursively sort right half
return merge(left, right) ← Merge two sorted halves
def merge(left, right): ← Both inputs are already sorted
result = [] ← Output buffer
i = j = 0 ← Pointers into left (i) and right (j)
while i < len(left) and j < len(right): ← Both halves have remaining elements
if left[i] <= right[j]: ← <= preserves stability (left wins ties)
result.append(left[i]); i += 1 ← Pick from left, advance i
else:
result.append(right[j]); j += 1 ← Pick from right, advance j
result += left[i:] ← Drain leftovers from left (may be empty)
result += right[j:] ← Drain leftovers from right (may be empty)
return result ← Merged sorted array
nums[:] = merge_sort(nums) ← Slice assignment: mutates in-place
return nums
⚠️ Why nums[:] = ... instead of nums = ...?

nums = merge_sort(nums) rebinds the local variable to a new list — the original list object (which the LeetCode harness holds a reference to) is unchanged. nums[:] = merge_sort(nums) mutates the original list in-place via slice assignment, so the caller sees the sorted result.

Solution 2 — Randomized Quick Sort (O(n log n) average, O(1) extra space)

import random
def sortArray(nums):
def quick_sort(lo, hi): ← Sort nums[lo..hi] in-place
if lo >= hi: return ← Base case
p = random.randint(lo, hi) ← Random pivot avoids O(n²) worst case
nums[p], nums[hi] = nums[hi], nums[p] ← Swap pivot to end temporarily
pivot, w = nums[hi], lo ← pivot value, write pointer
for r in range(lo, hi): ← Scan everything except pivot
if nums[r] <= pivot:
nums[w], nums[r] = nums[r], nums[w]; w += 1 ← Elements <= pivot go to the left zone
nums[w], nums[hi] = nums[hi], nums[w] ← Place pivot in its final sorted position
quick_sort(lo, w - 1) ← Sort left partition
quick_sort(w + 1, hi) ← Sort right partition
quick_sort(0, len(nums) - 1)
return nums

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: "No built-in sort + large n" = Must Implement O(n log n)

n up to 5×10⁴ means O(n²) algorithms like Bubble or Insertion Sort would do up to 2.5×10⁹ operations — guaranteed TLE. The constraint forces you to implement Merge Sort, Quick Sort, or Heap Sort. Merge Sort is the safest: deterministic worst-case, easy to code correctly, stable.

Step 2: Explain Divide-and-Conquer Before Coding

"Merge Sort splits the array in half, recursively sorts each half, then merges them in O(n) using two pointers. Depth is O(log n) so total time is O(n log n)." Say this first — it shows you understand the algorithm, not just the code.

Step 3: Write merge() Before merge_sort()

Start with the merge(left, right) helper — it's the non-recursive core. Then merge_sort is trivially: divide → recurse → merge. Writing bottom-up demonstrates mastery and keeps you from getting lost in recursion before you've nailed the merge.

Step 4: Proactively Mention Trade-offs

"Merge Sort uses O(n) extra space. If the interviewer wants O(1), I'd switch to Heap Sort. If average-case performance matters and O(n) space is a concern, I'd use randomized Quick Sort. For simplicity + correctness guarantees, I'll go with Merge Sort."

🗣️ What to Say to the Interviewer

"I'll implement Merge Sort — it guarantees O(n log n) in all cases and is stable.
The approach: recursively split the array until single elements, then merge pairs of sorted subarrays.
The merge step uses two pointers and runs in O(n) — always pick the smaller front element.
Recursion depth is O(log n), total work per level is O(n), so overall time is O(n log n).
Space is O(n) for the temporary arrays created during each merge.
I'll write the merge helper first, then the recursive driver."
🔮 Likely Follow-up Questions
  • "Can you do it in O(1) space?" → Heap Sort (build a max-heap, extract min repeatedly); or in-place merge sort (much harder)
  • "What's Timsort?" → Python's built-in: hybrid of Merge Sort + Insertion Sort; uses Insertion Sort on small runs, then merges them
  • "Count inversions in an array?" → During merge step, count how many right-half elements are placed before a left-half element — add the remaining left count each time
  • "Sort a linked list?" → Same Merge Sort, but finding the midpoint uses fast/slow pointers (#148 Sort List)
