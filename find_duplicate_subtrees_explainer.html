LeetCode #652

Find Duplicate Subtrees

Given the root of a binary tree, return all duplicate subtrees. Each element of the result is the root node of some repeated subtree.

Medium Trees Serialize Subtrees + Hash Postorder DFS Time: O(n²) Space: O(n²)
1 Understanding the Problem ▾

We have a binary tree and need to find all subtrees that appear more than once. A subtree rooted at a node X consists of X itself plus its entire left and right subtrees — the full structure beneath it. Two subtrees are considered identical only if they have the same structure and the same values at every corresponding position.

💡 Key Insight

Two subtrees are identical if and only if: (1) their root values match, (2) their left subtrees are identical, and (3) their right subtrees are identical. This recursive definition means we can encode any subtree as a unique string fingerprint. Then duplicate detection becomes just a frequency count on strings — a problem we already know how to solve with a hash map.

Input Tree (LeetCode Example)
        1
       / \
      2   3
     /   / \
    4   2   4
       /
      4

Nodes of interest: there are three 4-leaves, and two "node 2 with left-child 4" subtrees.

Expected Output
[Node(4), Node(2)]

Leaf "4" appears 3x
  → report root once

"2 → 4" subtree appears 2x
  → report root once

Edge cases: A tree with all nodes having the same value is only a duplicate if the subtrees are structurally identical (same depth and shape, not just same value). A single-node tree has no duplicates. We report each duplicate subtree exactly once, no matter how many times it appears.

2 The Algorithm: Postorder Serialize + Frequency Hash ▾

We do a single postorder DFS (left → right → current node). At each node, we compute a serialization string that uniquely identifies the entire subtree below it. We store these strings in a hash map and watch for strings that appear exactly twice.

1 Base case — null node: Return the sentinel string "#". This is how we encode the absence of a child. Without it, a leaf node and an internal node with a child would be indistinguishable.
2 Recurse left, then right: Get the serializations of both subtrees. Because we recurse first (postorder), by the time we process the current node, both children's fingerprints are already computed and ready.
3 Build this node's key: Concatenate as f"{val},{left_serial},{right_serial}". This format is unambiguous — the comma separators combined with the recursive structure ensure uniqueness.
4 Count and detect: Increment the hash map entry for this key. If the count just reached exactly 2, add the current node to the result. Using == 2 (not >= 2) ensures we only add each duplicate root once.
5 Return the key: Pass the serialization string back up to the parent, which uses it when building its own serialization. The fingerprints compose bottom-up naturally.
🔑 Serialization Format: Why it Works

The format "val,left,right" with null sentinel "#" produces a unique string for every distinct subtree shape+value combination. A leaf node 4 → "4,#,#". The same node with a left child 4 → "4,4,#,#,#". These can never collide because the recursive nesting encodes depth.

Serialization Examples
null node             → "#"
leaf 4              → "4,#,#"
node 2 with left=4   → "2,4,#,#,#"
node 1 (root)       → "1,2,4,#,#,#,3,2,4,#,#,#,4,#,#"
3 Why It Works: The Invariant ▾
🔐 The Core Invariant

After serialize(node) returns, the returned string uniquely identifies the entire subtree rooted at node — its shape and all values. No two structurally different subtrees (or subtrees with different values at any position) produce the same string.

Proof of uniqueness: By induction. Base case: null → "#" is trivially unique. Inductive step: suppose left and right subtrees each have unique serializations. Then f"{val},{left},{right}" encodes the root value plus two already-unique strings. Since the comma-delimited format disambiguates where left ends and right begins (because null children are explicitly marked), the combined string is unique for this subtree. Any change in structure or any value change produces a different string.

Why "count == 2" is the right threshold: We want each duplicate subtree in our result exactly once. If a subtree appears k times (k ≥ 2), the first time its count hits 2 is on the second occurrence. We add it then. For occurrences 3, 4, ..., k: the condition count[key] == 2 is false (count > 2), so we skip. Exactly one representative node per duplicate subtree.

⚠️ String Concatenation is O(n) per Node

At each node, building the key string takes O(length of the serialization), which can be O(n) for a balanced tree or O(n²) for a skewed tree. Total time is O(n²) and space is O(n²) for the hash map in the worst case. This is accepted for this problem. An advanced optimization uses integer IDs (Merkle-style hashing) to get O(n) time and space, but that's usually beyond interview scope.

⚠️ Common Mistake: Using >= 2 instead of == 2

If you write if count[key] >= 2, a subtree appearing 5 times gets added to result 4 times (on occurrences 2, 3, 4, 5). The == 2 guard fires only once — on the second occurrence — regardless of how many times the subtree appears.

💡 Brute Force Comparison (for context)

A simpler but slower approach: collect all n subtree roots. For every pair, recursively check equality with isSameTree(a, b) which is O(n) per comparison. Total: O(n²) pairs × O(n) comparison = O(n³). The serialization approach beats this by encoding equality as string identity, reducing to O(n²) total.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify the Core Challenge

We need to compare subtrees for equality across different positions in the tree. The brute-force would compare every pair — that's O(n²) pairs and O(n) per comparison. Recognize this as a "reduce comparison cost" problem.

Step 2: Think About What Makes Subtrees Equal

Two subtrees are equal iff: same root value, same left subtree, same right subtree. This recursive definition naturally leads to a recursive solution. It also suggests that if we can name subtrees canonically, equality becomes string equality — O(1) with a hash map.

Step 3: Design the Serialization Format

We need a format where: (a) each unique subtree structure gets a unique name, and (b) null children are explicitly marked. Format "val,left,right" with null = "#" satisfies both. We build it bottom-up (postorder) so children's names are ready when needed.

Step 4: Apply Frequency Count

Now it's a standard frequency count: maintain a hash map of serialization → count. When count first hits 2, we have a new duplicate. Add the node once. This is exactly the "find duplicates" pattern applied to complex objects.

Step 5: State Complexity Honestly

Time: O(n²) because building string keys at each level can involve copying O(n)-length strings. Space: O(n²) for the hash map. Mention this honestly and note the integer-ID optimization if asked.

🗣️ What to Say to the Interviewer

"My first thought is brute force: compare every pair of subtrees with isSameTree. That's O(n²) pairs times O(n) comparison — O(n³) total, too slow. Key insight: if I can give every subtree a unique string ID based on its structure and values, then duplicate detection becomes just a hash map frequency count. I'll do postorder DFS — process children before the current node — so by the time I'm at node X, I already have string IDs for its left and right subtrees. I'll build this node's ID as f'{val},{left_id},{right_id}', using '#' for null children. I increment a counter for each ID. When a counter hits exactly 2, I've found a duplicate subtree — I add the current node to my result list (once, using == 2). Time: O(n²) due to string concatenation at each level. Space: O(n²) for the hash map storing potentially O(n)-length strings. This is the standard optimal approach for this problem."
