Inorder Successor in BST

Given a BST and a node p, return the in-order successor of p — the node with the smallest key greater than p.val — or null if none exists.

Medium Trees BST Property Time: O(h) Space: O(1)
1 Understanding the Problem ▾

In a Binary Search Tree, an inorder traversal (left → root → right) visits nodes in ascending sorted order. The inorder successor of a node p is simply the next node in that sorted sequence — the node with the smallest value that is strictly greater than p.val.

💡 Key Insight

We don't need to perform a full inorder traversal. The BST property (left < root < right) tells us exactly which direction to look at each node. We can navigate from root to answer in O(h) time using zero extra space.

BST {5,3,6,2,4}. Inorder: 2 → 4 → 3 → 5 → 6... Wait — let's think again. Sorted: 2,3,4,5,6. Successor of 2 is 3 (the node, not value 4). Let's use correct example below.

Two cases — draw them both before writing any code:

Case A: p has a right child

Successor = leftmost node in the right subtree. All right-subtree values are greater than p; leftmost is smallest among them.

Case B: p has NO right child

Successor = lowest ancestor for which p is in its left subtree. Must trace up — but we have no parent pointers, so we trace down from root instead.

🔑 The Unified Solution

Both cases are handled by a single elegant root-down walk: track the most recently taken "left turn" as the candidate successor. When the walk ends, that candidate is the answer.

Edge cases to consider: p has the maximum value (no successor → null), single-node tree, p is the root, p is a leaf with no right sibling above it.

2 The Algorithm: Iterative BST Walk ▾

We walk the BST from the root downward, making a binary decision at each node. We track a candidate — the smallest valid successor seen so far.

1 Initialize: successor = None (no candidate yet), node = root (start at top).
2 At each node, compare node.val with p.val:
 • If node.val > p.val: valid candidate! Record successor = node, then go left (searching for a smaller valid candidate in the left subtree).
 • If node.val <= p.val: too small (or is p itself). Go right — successors must be larger.
3 Terminate: When node becomes None (fell off the tree), successor holds the smallest valid candidate found — that's the answer.
→
node.val ≤ p.val — Too small or is p. Go RIGHT: successor must have a larger value, only found to the right.
←
node.val > p.val — Valid candidate! Record it, then go LEFT: the left subtree might contain a smaller valid candidate (between p.val and node.val).
🔑 Why Start from Root (No Parent Pointers)?

We're given only the root and p. The BST property lets us navigate from root to the right area in O(h) — we don't need parent links. Every ancestor we "left-turned from" is a valid successor candidate, and we keep the best (smallest) one as we go.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every step of the loop, successor holds the smallest node seen so far whose value is strictly greater than p.val. It is updated only when we go left — meaning the current node is a valid candidate and smaller candidates might still exist in the left subtree.

Going left after recording a candidate: If node.val > p.val, this node is valid. All nodes in node.left have values in the range (p.val, node.val) — BST property guarantees this. So we go left to search for a smaller (but still valid) successor. If we find one, we'll record it next iteration. If the left subtree is empty, we stop here and the current candidate is the answer.

Going right when node.val ≤ p.val: This node's value is too small — it cannot be a successor. The right subtree contains all values larger than node.val, so values larger than p.val live there. We must go right. We do not update successor because node.val is not a valid candidate.

⚠️ The ≤ instead of <

When node.val == p.val, we are AT node p. We still go right — p's right subtree contains the smallest values greater than p. We do NOT record p itself (we need strictly greater). This works even if the problem guarantees p exists in the tree.

Termination & correctness: Each step descends strictly down the tree (no revisiting). After O(h) steps we reach None. The final successor is the minimum over all "left-turn ancestors" of p. This is exactly the inorder successor by definition.

4 The Code (Annotated) ▾

Solution 1 — Iterative BST Walk (Interview Pick, Optimal)

def inorderSuccessor(root, p):
successor = None ← Best candidate so far; starts as None (none found yet)
node = root ← Begin traversal at root; we'll walk downward
while node: ← Loop until we fall off the tree
if node.val > p.val: ← Valid candidate: this node is strictly greater than p
successor = node ← Record as best candidate found so far
node = node.left ← Go left: left subtree may contain smaller valid candidates
else: ← node.val <= p.val: not a candidate (too small or is p)
node = node.right ← Go right: successors must be larger, they live here
return successor ← Smallest valid candidate, or None if p was largest
💡 Complexity

Time: O(h) — one root-to-leaf path. O(log n) balanced, O(n) skewed.
Space: O(1) — two pointers only. No stack, no recursion, no visited set.

⚠️ Why `node is p` not `node.val == p.val`?

BSTs can have duplicate values (problem variant). Using is checks object identity — we find the specific node p, not just any node with the same value. If you use == and there are duplicates, you might stop too early at the wrong node. Always use is for node identity checks in tree problems.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Clarify the Constraints

Ask: "Do the nodes have parent pointers?" If yes, the algorithm is different — walk right if p has right child, else walk up until you take a left turn. For this problem (standard BST), assume no parent pointers. Also ask: "Are values unique?" Affects whether to use is or ==.

Step 2: Draw the Two Cases

Sketch: Case A — p has a right child → answer is leftmost of right subtree. Case B — p has no right child → answer is the ancestor where we last went left. Then say: "I'll unify both cases with a root-down walk."

Step 3: Propose Brute Force First

"I could do a full inorder traversal and return the element right after p — O(n) time, O(h) space. Correct but ignores the BST property entirely." Then optimize.

Step 4: Derive the O(h) Optimal

"At each node: if node.val > p.val, it's a candidate — record it and go left (maybe smaller valid exists). If node.val ≤ p.val, go right. Final candidate = inorder successor. O(h) time, O(1) space."

🗣️ What to Say to the Interviewer

"The inorder successor is the smallest node strictly greater than p.val.
I'll walk from root, tracking the best candidate seen so far.
At each node: if node.val greater than p.val, record it as candidate and go left
— because left subtree might have something smaller but still valid.
Otherwise go right — must find larger values.
When I fall off the tree, the last candidate is the answer.
Time O(h), space O(1). For a balanced tree that's O(log n)."
🔮 Likely Follow-up Questions
  • "What if nodes have parent pointers?" → LeetCode #510 — go right if right child exists, else walk up until left turn
  • "What about the inorder predecessor?" → Mirror: node.val < p.val = candidate, go right; else go left
  • "What if it's a generic binary tree (not BST)?" → Must do full inorder traversal, O(n)

7 Summary & Key Takeaways ▾
Inorder successor = smallest node with value strictly greater than p.val
BST property enables O(h) root-down walk — no full traversal, no stack
node.val > p.val → record as candidate and go LEFT (search for smaller valid)
node.val ≤ p.val → go RIGHT (must find larger values; this includes visiting p itself)
O(h) time, O(1) space — optimal. Reduces to O(log n) on balanced BSTs
Use `node is p` not `node.val == p.val` in the inorder traversal variant — handles duplicates
🎯 When to Recognize the "Go Right or Up-Left" Pattern
  • "Find the smallest key greater than X in a BST" → This pattern
  • "Find the largest key less than X in a BST" → Mirror of this pattern
  • O(h) expected, no extra space → start from root and use BST property to navigate
  • No parent pointers given → use the "track candidate on left turns" technique
