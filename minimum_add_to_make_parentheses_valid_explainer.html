LeetCode #921

Minimum Add to Make Parentheses Valid

Given a string of parentheses, return the minimum number of brackets you must insert to make the string valid.

Medium Greedy Stack / Track Open-Close Time: O(n) Space: O(1)
1 Understanding the Problem ▾

A parentheses string is valid when every opening bracket ( has a matching closing bracket ) to its right, and every closing bracket ) has a matching opening bracket ( to its left. We want to know the fewest brackets to insert to repair an invalid string.

💡 Key Insight

There are exactly two failure modes in any invalid parentheses string: a ) that arrives with no unmatched ( before it (dangling close), and a ( that reaches end-of-string without ever being closed (dangling open). Each dangling bracket requires exactly one insertion to fix. The answer is simply dangling_close + dangling_open.

Invalid examples

()) → 1 dangling close → +1
(( → 2 dangling open → +2
))( → 2 dangling close + 1 open → +3
)( → 1 close + 1 open → +2

Valid examples (answer = 0)

() → perfectly matched
(()) → nested, all matched
()() → sequential, all matched
"" → empty string, trivially valid

Why "balance" alone fails: Consider )(. Its balance (sum of +1 for (, −1 for )) is 0 — looks balanced! But it needs 2 insertions. The problem with balance is that it ignores order. Our two-counter approach correctly tracks both flavors of mismatch separately.

Edge cases: Empty string → 0. All opens (((( → 4. All closes )))) → 4. Single ( → 1. Single ) → 1. Already valid → 0.

2 The Algorithm: Greedy Balance Tracking ▾

We scan the string once left to right, maintaining two integer counters. We never look ahead — we process each character greedily, resolving mismatches the moment we detect them.

1 Initialize: open = 0 (unmatched opens awaiting a close), close = 0 (dangling closes that needed a prior open). Both start at zero.
2 Encounter (: Increment open. This bracket now owes the string a matching ) later. We don't know if one will come, so we just record the debt.
3 Encounter ) with open > 0: A pending open bracket is available — pair them. Decrement open. The debt is cancelled. This is the happy path.
4 Encounter ) with open == 0: No open bracket available to match. This is a dangling close — we'd need to insert a ( before it. Increment close.
5 Return open + close: After scanning, open tells us how many unclosed ( remain — each needs a ) appended. close tells us how many ) had no match — each needed a ( prepended. Total = minimum insertions.

Decision tree at each character:

char == '('
→
open += 1   (create an IOU for a future ')')
char == ')' and open > 0
→
open -= 1   (matched! cancel one IOU)
char == ')' and open == 0
→
close += 1   (dangling — must insert a '(' before this)
🔑 Why Greedy Matching is Optimal

When a ) arrives and there is a pending (, should we always match them? Yes — always. If we delayed matching, the ( would still need to be matched eventually (or counted as dangling), and the ) could only be matched by an earlier (, which we've already passed. Greedy matching is optimal here: there's no benefit to deferring a match.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At any point during the scan, open = the number of ( characters seen so far that have not yet been matched by a ) character. Equivalently, open is the "current depth" of nesting — but only for the correctly-structured prefix. close accumulates every ) that arrived when open was zero. Both counters can only increase or hold steady — they never go negative.

Why each unmatched bracket costs exactly 1 insertion: A dangling ) requires exactly one ( inserted somewhere before it. A dangling ( requires exactly one ) inserted somewhere after it. These are independent — fixing one does not fix the other. So the minimum total insertions equals close + open.

Why we can't do better (lower bound): Every dangling close bracket cannot be matched by any existing bracket (if it could, we would have matched it greedily). Every dangling open cannot be matched by any existing bracket that comes after it (there are none left). Therefore every dangling bracket strictly requires one insertion — the algorithm's output is a tight lower bound.

⚠️ The Balance Trap: Why abs(balance) is Wrong

A common wrong approach: compute balance as +1 for ( and −1 for ), return abs(balance). Counterexample: )( → balance = 0, answer = 0 (WRONG — correct is 2). The balance approach misses dangling closes that get "cancelled out" by dangling opens. Our approach counts both types separately, avoiding this cancellation error.

4 The Code (Annotated) ▾

Solution 1 — Greedy Two-Counter (Optimal) · Time O(n) · Space O(1)

def minAddToMakeValid(s: str) -> int:
open = 0 ← count of unmatched '(' (currently owe this many ')'s)
close = 0 ← count of dangling ')' (each needed a prior '(' that wasn't there)
for c in s: ← scan every character once, O(n)
if c == '(': ← opening bracket: owe a future close
open += 1
elif open > 0: ← closing bracket, pending open available: match it
open -= 1
else: ← closing bracket, no open available: dangling close
close += 1
return open + close ← unclosed '(' + unmatched ')' = minimum insertions
⚠️ Why elif open > 0 instead of elif c == ')'?

Since the string contains only ( and ), the two branches are equivalent for valid-character input. However, writing elif open > 0 makes the intent crystal clear: "I only match this ) if there's a pending ( to pair with." It also makes the three-way logic explicit — open bracket / matched close / dangling close — which is exactly what we're tracking.

Solution 2 — Explicit Stack · Time O(n) · Space O(n)

def minAddToMakeValid(s: str) -> int:
stack = [] ← stores each unmatched '(' (stack of pending opens)
close = 0 ← count of ')' that had nothing to match
for c in s:
if c == '(': ← push a pending open to the stack
stack.append('(')
elif stack: ← ')' with a pending '(' available → pop = matched pair
stack.pop()
else: ← ')' with empty stack → dangling, must insert a '('
close += 1
return len(stack) + close ← unmatched opens + unmatched closes
💡 The Stack → Counter Optimization

Notice that the stack only ever holds '(' characters. We never inspect the stack's contents — only its size. So we can replace the entire stack with a single integer open that tracks the stack's height. This reduces space from O(n) to O(1). Same logic, same output, better memory usage.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Pattern Signal

The moment you see "parentheses" + "minimum insertions or removals", think: "Count the unmatched brackets. Each unmatched bracket = 1 operation." This is a classic greedy/stack problem. The greedy choice: process left-to-right, resolve problems immediately.

Step 2: Frame the Two Failure Modes

"A string can be invalid in exactly two ways: a ) appears with no prior unmatched ( — or the string ends with unmatched ( brackets. Let me count both." This framing immediately gives you the two-counter structure.

Step 3: Propose Stack First, Optimize to Counters

"My first approach uses a stack to track unmatched opens. When I see a ) and the stack is non-empty, I pop a match. Otherwise I count a dangling close. At the end, remaining stack size + dangling closes is my answer." Then: "Since the stack only holds identical ( characters, I can replace it with one integer. O(1) space."

Step 4: Validate with the Tricky Case

Always trace ")(" manually: index 0 is ), open=0 → close becomes 1. Index 1 is ( → open becomes 1. Return 1+1=2. Correct! This catches the "balance trap" — balance would give 0, which is wrong.

🗣️ What to Say to the Interviewer

"There are two ways this string can be invalid:
1) a ')' appears with no unmatched '(' before it — dangling close
2) a '(' reaches end-of-string unmatched — dangling open
