LeetCode #879

Profitable Schemes

Count subsets of crimes that can be committed using at most n members and generate at least minProfit total profit. Return the count modulo 10⁹+7.

Hard Dynamic Programming 3D DP — 0/1 Knapsack Time: O(k·n·minProfit) Space: O(n·minProfit)
1 Understanding the Problem ▾

You're running a gang with n members. There are k crimes (called "schemes"), each requiring a certain number of members and generating a fixed profit. You want to count how many different subsets of schemes you can pick such that:

  • The total members needed ≤ n (you don't have unlimited crew)
  • The total profit ≥ minProfit (the scheme must be worth it)

The answer can be huge, so return it modulo 10⁹ + 7.

💡 Key Insight

This is a two-constraint counting problem: you must simultaneously track how many members you've used (at most n) AND how much profit you've accumulated (at least minProfit). This forces a 3-dimensional DP state: (crime index, members used, profit so far).

Concrete example: n=5, minProfit=3, group=[2,2], profit=[2,3]

Answer = 2. The key subtlety: we count subsets of crimes, not distinct crews. The same crime committed with different members doesn't matter — only which crimes you pick.

⚠️ Tricky: Profit is a Lower Bound, Members is an Upper Bound

Member constraint is ≤ n (don't exceed gang size). Profit constraint is ≥ minProfit (must meet target). These asymmetric directions require careful handling in the recurrence. For profit, we must count all profit levels ≥ minProfit in our final answer.

Edge cases: minProfit=0 (every subset valid, just check member count), a single crime exceeding n members (skip it), crimes with 0 members (can always take them).

2 The Algorithm: 3D Knapsack DP ▾

This problem is a 0/1 Knapsack variant with two knapsack dimensions: member capacity and profit target. Each crime is an "item" you either include or exclude (0/1). The three dimensions of our DP table are:

State Dimensions

i = crime index (0 to k-1)
j = members used (0 to n)
p = profit so far (0 to minProfit)

dp[j][p] = count of ways

"After considering some crimes, how many subsets use exactly j members and generate at least (capped to) p profit?"

1 Initialize: dp[0][0] = 1 — there is exactly 1 way to use 0 members and earn 0 profit: pick no crimes at all.
2 Process each crime one by one. For crime i with g = group[i] members and p = profit[i]: try adding it to every existing subset.
3 Skip or take: For each state (j, p), we can either skip this crime (state unchanged) or take it (move to j + g members and min(p + profit_i, minProfit) profit).
4 Profit clamping (key trick!): We cap profit at minProfit because any profit ≥ minProfit is equally "good." This collapses infinite profit states into one bucket, keeping the DP tractable.
5 Collect answer: Sum dp[j][minProfit] for all j from 0 to n. These are subsets that used ≤ n members and achieved profit ≥ minProfit (stored at the clamped bucket).
📐 The Recurrence Relation
For each crime i with members g, profit p:
dp'[j][profit] = dp[j][profit] + dp[j-g][max(0, profit-p)]

where dp' is the updated table after considering crime i,
profit index is clamped: profit = min(profit + p, minProfit),
and we iterate j from n down to g (in-place update, like 0/1 knapsack).
🔑 Why Iterate j Backwards?

Just like classic 0/1 knapsack, we iterate the member dimension from n down to g when updating in-place. This ensures we don't "use the same crime twice" — if we updated dp[j+g] before dp[j], the crime might get counted again when we later process dp[j].

3 Why It Works: Optimal Substructure & Overlapping Subproblems ▾
🔐 Optimal Substructure

The count of profitable subsets from crimes [0..i] using j members with profit p can be computed from the count for crimes [0..i-1]. Each new crime either: (a) is not included — count comes directly from dp[j][p], or (b) is included — count comes from dp[j - group[i]][p - profit[i]]. These two sub-counts add up correctly because the subsets are disjoint (one without crime i, one with crime i).

🔐 Overlapping Subproblems

In a naive recursive approach, we'd compute "how many ways to achieve profit p with j members from crimes [0..i]" repeatedly from different call sites. For example, whether we're deciding to include crime 5 or crime 7, we might both need the sub-answer for "crimes [0..4] with 3 members, 2 profit." Memoization or tabulation avoids this redundancy.

The Clamping Invariant: At all times, dp[j][p] = number of subsets of crimes considered so far that use exactly j members and have profit in the "clamped bucket" p. A profit ≥ minProfit is stored at index minProfit. This invariant holds throughout the DP iteration because:

  • Base case: 1 way to use 0 members, 0 profit → dp[0][0] = 1 ✓
  • Transition: adding crime i to any valid subset gives a new valid subset → count adds correctly ✓
  • Clamping: if prev_profit + crime_profit ≥ minProfit, clamped to minProfit — the "at least minProfit" bucket accumulates ✓
💡 Why Clamping Works (Mathematical Argument)

We want to count subsets with profit ≥ minProfit. Any profit value above minProfit is "equally good" for our binary threshold. So instead of maintaining distinct profit levels 0, 1, 2, ..., ∞, we collapse everything ≥ minProfit into one level. The DP remains exact: dp[j][minProfit] = count of subsets using j members with profit ≥ minProfit.

4 The Code (Annotated) ▾

Solution 1 — Bottom-Up Tabulation (Optimal, Interview Pick)

def profitableSchemes(n, minProfit, group, profit):
MOD = 10**9 + 7 ← modulus for all arithmetic
# dp[j][p] = # subsets using exactly j members, profit clamped to p
dp = [[0] * (minProfit + 1) for _ in range(n + 1)] ← (n+1) × (minProfit+1) table
dp[0][0] = 1 ← base case: empty set, 0 members, 0 profit
for g, p in zip(group, profit): ← iterate over each crime
for j in range(n, g - 1, -1): ← members: n down to g (0/1 knapsack trick)
for k in range(minProfit, -1, -1): ← profit: minProfit down to 0
new_k = min(k + p, minProfit) ← clamp profit: don't exceed minProfit index
dp[j][new_k] = (dp[j][new_k] + dp[j-g][k]) % MOD ← add ways: skip + take
return sum(dp[j][minProfit] for j in range(n + 1)) % MOD ← sum over all valid member counts
⚠️ Why iterate k (profit) downward too?

The profit dimension is also updated in-place. If we iterated k upward, a state dp[j][k] that was just updated (with crime i included) could "feed into" dp[j][k + p], potentially counting crime i twice. Iterating downward prevents this — when we compute dp[j][k], its source dp[j-g][k-p] (conceptually) hasn't been touched yet in this crime's iteration pass.

Solution 2 — Top-Down Memoization (Recursive, same complexity)

from functools import lru_cache
def profitableSchemes(n, minProfit, group, profit):
MOD = 10**9 + 7
k = len(group)
@lru_cache(maxsize=None) ← memo cache; auto-handles repeated subproblems
def dp(i, members, prof): ← i=crime idx, members=used so far, prof=clamped profit
if i == k: ← base case: all crimes processed
return 1 if prof >= minProfit else 0 ← count this subset if profit reached target
# Option A: skip crime i
ways = dp(i + 1, members, prof) ← don't take crime i
# Option B: take crime i (if enough members remain)
if members + group[i] <= n: ← capacity check
new_p = min(prof + profit[i], minProfit) ← clamp profit for memo key
ways += dp(i + 1, members + group[i], new_p) ← take crime i
return ways % MOD
return dp(0, 0, 0) ← start: no crimes seen, 0 members, 0 profit
💡 Tabulation vs Memoization for This Problem

Tabulation (bottom-up): Fills table iteratively; space O(n · minProfit) with in-place update. Preferred in interviews — no recursion stack, easier to optimize.

Memoization (top-down): Easier to derive from the recurrence; auto-prunes branches where members > n early. Python's LRU cache handles it cleanly.

Both are O(k · n · minProfit) time and O(n · minProfit) space (after the i dimension is optimized away).

🔑 Brute Force — Enumerate All Subsets (Too Slow)

Generate all 2ᵏ subsets of crimes, check each: sum(members) ≤ n and sum(profit) ≥ minProfit. Time O(2ᵏ · k) — exponential. Fails for k=100 instantly. DP compresses this to polynomial by recognizing that only the (members, profit) state matters, not which specific previous crimes were chosen.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the "Count Subsets" + "Two Constraints" Pattern

"We're counting subsets (not finding one optimal subset). We have two simultaneous constraints: member cap (≤ n) and profit threshold (≥ minProfit). This signals a multi-dimensional DP, not greedy — greedy can't optimize for a count."

Step 2: Identify the 0/1 Knapsack Structure

"Each crime is either taken or not (0/1). The member constraint is exactly the knapsack capacity. But we have a second dimension — profit — because we need to also track that. Classic knapsack tracks one weight; this tracks two, giving us a 2D table (after the 'items' dimension is compressed away via in-place updates)."

Step 3: Define States and Transitions

"dp[j][p] = number of ways to use exactly j members and have clamped profit p. For each crime (g members, q profit), we update: dp[j][min(p+q, minProfit)] += dp[j-g][p]. Iterate j downward to enforce 0/1 property."

Step 4: Handle the Profit Lower Bound with Clamping

"Profit is a lower bound (≥ minProfit), not exact. So any profit above minProfit is 'equivalent.' We cap the profit index at minProfit throughout — this collapses the table to a finite size. At the end, sum dp[j][minProfit] for all j."

🗣️ What to Say to the Interviewer

"This looks like a 0/1 knapsack counting problem with two constraints.
State: dp[j][p] = number of crime subsets using exactly j members with clamped profit p.
For each crime: iterate j from n down to group[i], and p from minProfit down to 0.
Update: dp[j][min(p + profit[i], minProfit)] += dp[j - group[i]][p].
Base case: dp[0][0] = 1 (empty set).
Answer: sum(dp[j][minProfit] for j in 0..n) mod 10^9+7."
🔮 Likely Follow-up Questions
  • "Can you do this in O(1) space?" → Not easily; the 2D table is necessary. You CAN optimize the i-dimension (already done by in-place update), giving O(n·minProfit) space.
  • "What if crimes could be repeated?" → Unbounded knapsack: iterate j upward instead of downward.
  • "What if minProfit=0?" → Answer is the count of all subsets with ≤ n members = sum over j of dp[j][0] after the DP.
  • "Why not greedy — take most profitable first?" → Greedy fails to count; it can't enumerate all valid subsets.

7 Summary & Key Takeaways ▾
3D DP (crime × members × profit) collapses to 2D via in-place iteration (0/1 knapsack trick)
Profit is clamped at minProfit — the "at least" threshold collapses infinite profit into one bucket
Iterate member dimension downward to enforce each crime used at most once
Base case dp[0][0]=1 represents the empty subset (always valid, just may not meet profit target)
Final answer: sum all dp[j][minProfit] for j in 0..n — every valid member count with profit achieved

🎯 When to Recognize Multi-Dimensional Knapsack
  • "Count subsets satisfying TWO simultaneous constraints" → 2D DP table
  • One constraint is an upper bound (capacity), another is a lower bound (target) → clamp the lower bound
  • "Each item used at most once" → iterate capacity dimension downward
  • "Unlimited use" → iterate upward (unbounded knapsack)
