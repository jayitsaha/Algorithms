LeetCode #456

132 Pattern

Given an array of n integers nums, return true if there is a 132 pattern — a subsequence of three integers nums[i] < nums[k] < nums[j] where i < j < k.

Medium Monotonic Stack Reverse + Track Second Max Time: O(n) Space: O(n)
1 Understanding the Problem ▾

The name "132 pattern" describes a shape, not actual values. We need three indices i < j < k such that nums[i] < nums[k] < nums[j]. In plain English: find a "valley–peak–middle" shape where the dip (i) comes first, a tall peak (j) comes second, and a medium-height value (k) comes last.

The "132" name maps to the relative value order, not positional order: value at i is the smallest ("1"), value at j is the largest ("3"), and value at k is in the middle ("2"). But their positions appear as 1, 3, 2 — first smallest, then largest, then middle.

💡 Key Insight

The tricky part: the "2" (middle value) appears at position k which is after the "3" (position j). So the values read small → large → medium in position order. Scanning right-to-left lets us process j and k first, then efficiently check if any future element to the left can serve as the "1".

Edge Cases: Array length < 3 (always False). All equal values (no strict inequality). Strictly increasing/decreasing sequences. Negative numbers are valid — only relative order matters. Arrays like [-1, 3, 2, 0] — check by the definition.

2 The Algorithm: Reverse Scan with Monotonic Stack ▾

The key breakthrough: scan right-to-left. When processing index i from the right, all positions j and k (which must be to the right of i) are already processed. We can use their information to instantly check if nums[i] can serve as the "1".

We maintain two things: (1) a monotonic decreasing stack of candidate "3" values, and (2) a variable third — the largest value ever popped from the stack, which represents the best possible "2" we've seen so far.

1 Initialize: Set third = -∞ (no valid "2" established yet) and stack = []. The stack will maintain a decreasing sequence of candidate "3" values.
2 Check for "1": For each nums[i] (scanning right-to-left), first test: if nums[i] < third, we've found our "1"! We already have a valid (3, 2) pair recorded in the stack history, and nums[i] is smaller than the "2" — pattern confirmed. Return True.
3 Pop and update third: While nums[i] > stack[-1], pop from the stack. The popped value is smaller than nums[i], making nums[i] a valid "3" and the popped value a valid "2". Update third = max(third, popped) — we want the largest possible "2" to have the best chance of matching a future "1".
4 Push current element: Push nums[i] onto the stack as a candidate "3" for future (left-side) elements that might serve as "1".
5 No pattern: If we finish the full right-to-left scan without ever finding nums[i] < third, return False.
🔑 Why Scan Right-to-Left?

The 132 pattern requires i < j < k. If we fix j and look for k to the right, that's only 2 nested loops. But the real insight is: by scanning right-to-left, when we reach any index i, we've already discovered all valid (j, k) pairs to the right. The variable third captures the best "2" from those pairs. So the check at each i reduces to a single comparison: nums[i] < third?

3 Why It Works: The Invariant ▾
🔐 The Core Invariant

Stack invariant: The stack always contains elements in strictly decreasing order (bottom to top). Each element represents a candidate "3" — a local maximum from the right side that hasn't been surpassed yet.

third invariant: third always holds the largest value that has been popped from the stack. There exists some element to the right of the current scanning position that is strictly greater than third — making third a valid "2" in a confirmed (j, k) pair.

Why popping establishes a valid pair: When nums[i] > stack[-1], we pop. At this moment, nums[i] serves as "3" (the j-position value), and the popped element serves as "2" (the k-position value). The popped element was pushed earlier in our right-to-left scan, meaning it has a larger positional index than the current i — satisfying j < k. The inequality nums[j] > nums[k] is guaranteed because we only pop when nums[i] > stack[-1].

Why nums[i] < third guarantees a valid 132 pattern: When this condition triggers, third is a valid "2" with a confirmed "3" to its left in the stack history (or as the element that caused the pop). Both "3" and "2" are at positions with larger indices than i. Since nums[i] < third < (some "3" value), all three conditions — i < j < k and nums[i] < nums[k] < nums[j] — are satisfied.

⚠️ Common Confusion: Stack vs third

The stack stores candidate "3" values (peaks). third stores the best "2" value — an element that was once a candidate "3" but got popped when something larger arrived. Think of it as: stack elements are still in the running as "3"; third graduated from "3-candidate" to "confirmed-2". The largest popped value is kept because a larger "2" makes it easier to find a "1" (more elements are smaller than a larger "2").

O(n) complexity argument: Despite the nested while loop, each element is pushed onto the stack exactly once and popped at most once. So the total number of push + pop operations across the entire algorithm is at most 2n — giving O(n) overall, not O(n²).

4 The Code (Annotated) ▾

Solution 1 — Monotonic Stack, Right-to-Left (Interview Pick, O(n))

def find132pattern(nums):
third = float('-inf') ← best "2" value found so far; -inf means none yet
stack = [] ← monotonic decreasing stack: candidate "3" values
for i in range(len(nums) - 1, -1, -1): ← scan RIGHT-TO-LEFT; future j,k are already processed
if nums[i] < third: ← nums[i] is the "1": smaller than our confirmed "2"
return True ← 132 confirmed: (nums[i], third, some element > third)
while stack and nums[i] > stack[-1]: ← nums[i] is new "3"; stack top becomes valid "2"
third = max(third, stack.pop()) ← keep largest "2" seen — easier to find a smaller "1"
stack.append(nums[i]) ← push as candidate "3" for elements further left
return False ← no 132 pattern in entire array
⚠️ Why float('-inf') and not 0 or None?

Before any pops occur, third must not accidentally match a valid "1". Using -inf ensures the check nums[i] < third is always False until a real "2" is established. Any real array value satisfies x > -inf, so the first pop correctly updates third to a real value.

Solution 2 — Brute Force (O(n³), for understanding)

def find132pattern_brute(nums):
n = len(nums)
for i in range(n - 2): ← try every i
for j in range(i + 1, n - 1): ← try every j > i
for k in range(j + 1, n): ← try every k > j
if nums[i] < nums[k] < nums[j]: ← exact 132 condition
return True
return False

Solution 3 — Prefix Minimum Precomputation (O(n²), intermediate step)

def find132pattern_n2(nums):
n = len(nums)
prefix_min = [float('inf')] * n ← prefix_min[j] = min(nums[0..j-1])
for j in range(1, n):
prefix_min[j] = min(prefix_min[j-1], nums[j-1])
for j in range(1, n - 1): ← j is the "3" position
if prefix_min[j] < nums[j]: ← a "1" exists to the left of j
for k in range(j + 1, n): ← look for valid "2" to the right
if prefix_min[j] < nums[k] < nums[j]:← all 3 conditions hold
return True
return False
💡 Complexity Summary

Brute Force: Time O(n³), Space O(1) — try all triplets; correct but TLE
Prefix Min: Time O(n²), Space O(n) — precompute "1" candidates; still TLE on large input
Monotonic Stack ✓: Time O(n), Space O(n) — each element pushed and popped once

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Clarify the Pattern

Restate precisely: "I need indices i < j < k where nums[i] is smallest, nums[j] is largest, and nums[k] is in between. The '132' is the shape of values in positional order." Draw a quick sketch showing the valley-peak-middle shape.

Step 2: Brute Force to Show Understanding

"The direct approach: three nested loops checking all triplets — O(n³). Clearly too slow for n up to 2×10⁵. Can I precompute something?" Note that for each j, the "1" is just min(nums[0..j-1]) — precompute with a prefix array. Now it's O(n²). Can we do better?

Step 3: The Right-to-Left Insight

"Instead of fixing j and searching both left and right, what if I scan right-to-left? When I'm at position i, all j and k candidates (which must be to my right) are already fully explored. If I can capture the best possible (3, 2) pair from positions I've already seen, checking 'is this a valid 1?' becomes a single comparison."

Step 4: Connect to Monotonic Stack

"To track valid (3, 2) pairs efficiently, use a monotonic decreasing stack. The stack holds candidate '3' values. When we see a larger element, we pop — and the popped value becomes a confirmed '2'. We track the largest '2' ever seen in third."

🗣️ What to Say to the Interviewer

"I'll scan right-to-left with a monotonic decreasing stack and a variable 'third' for the best middle value.
At each position i: if nums[i] < third, I have my '1' — return True immediately.
Otherwise, while nums[i] is greater than the stack top, pop (establishing a valid '2') and update third.
Then push nums[i] as a new candidate '3' for future (leftward) elements.
Time O(n) since each element is pushed and popped at most once. Space O(n) for the stack."

7 Summary & Key Takeaways ▾
Scan right-to-left — by the time you process index i, all valid (j, k) pairs to the right have already been discovered
Monotonic decreasing stack holds candidate "3" values; popping reveals a valid "2"
third always stores the largest popped value — the best possible "2" seen so far
If nums[i] < third, the 132 pattern is confirmed with nums[i] as "1", third as "2", and the element that established third as "3"
O(n) time despite the while loop — each element is pushed and popped at most once
