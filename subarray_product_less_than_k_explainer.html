LeetCode #713

Subarray Product Less Than K

Given an array of positive integers nums and a positive integer k, return the number of contiguous subarrays where the product of all elements is strictly less than k.

Medium Sliding Window Product Window + Count Time: O(n) Space: O(1)
1 Understanding the Problem ▾

We need to count contiguous subarrays whose product of all elements is strictly less than k. A subarray is a contiguous slice of the original array — you cannot skip elements. "Contiguous" is the key word that makes sliding window applicable here.

💡 Key Insight

Because all numbers are positive, the product of any window only grows as we expand right and only shrinks as we remove from the left. This monotonicity is exactly what makes a sliding window viable — if adding an element makes the product too large, removing elements from the left is guaranteed to help.

Consider nums = [10, 5, 2, 6], k = 100. Walk through all subarrays:

Edge Cases: If k <= 1, no product of positive integers (each ≥ 1) can be strictly less than k. Return 0 immediately. This guard also prevents an infinite shrink loop. Single-element subarrays count only if that element is less than k.

2 The Algorithm: Sliding Window with Running Product ▾

We maintain a variable-size window [left, right] where the running product is always strictly less than k. As right advances, we include the new element; if the product becomes too large, we shrink from the left until the window is valid again. Then we count.

1 Edge-case guard: If k <= 1, return 0. Every element is at least 1, so no product can be strictly less than 1.
2 Expand right: For each right, multiply nums[right] into the running product. The window tentatively grows to include the new element.
3 Shrink left while invalid: While product >= k, divide out nums[left] and advance left. This restores window validity by removing the leftmost element.
4 Count new valid subarrays: After fixing the window, every subarray ending at right that starts anywhere from left to right is valid. Add right - left + 1 to the total.
🔑 Why right - left + 1?

With window [left, right] valid, the new subarrays introduced this step all end at right: they are [right..right], [right-1..right], …, [left..right]. That is exactly right - left + 1 subarrays. Sub-windows are also valid because removing elements from a product of positive numbers can only decrease the product.

Example: window [1..3] contributes 3 new subarrays ending at index 3

[6] → product=6
[2,6] → product=12
[5,2,6] → product=60

All three end at index 3, all have product < 100. Count = 3 - 1 + 1 = 3.

3 Why It Works: The Invariant ▾
🔐 The Invariant

After processing each right and completing the shrink loop, the window [left, right] is the longest valid window ending at right — its product is strictly less than k. Any subarray fully within this window also has product < k (since all nums are positive, removing elements shrinks the product).

Why sub-windows are guaranteed valid: Suppose window [left..right] has product P < k. For any start index left' > left, the subarray [left'..right] has product ≤ P (fewer positive factors). So every subarray ending at right with start ≥ left is automatically valid — we count them all at once with right - left + 1.

Why we never double-count: Subarrays counted at step right all end at right. Subarrays counted at step right-1 all end at right-1. No overlap — every valid subarray is counted exactly once when its rightmost index is the current right.

⚠️ Why k <= 1 Must Be Handled Up Front

All elements are positive integers (at least 1). So any single-element product is ≥ 1. If k = 1, we need product < 1 — impossible. If k = 0, same problem. Without the guard if k <= 1: return 0, the while loop would spin forever: even after shrinking the window to one element, product = nums[right] ≥ 1 ≥ k, so left would exceed right and we'd be dividing by already-removed elements — undefined behavior.

Full trace for [10, 5, 2, 6], k=100:

right=0: product=10 < 100, window=[0,0], +1, count=1
right=1: product=50 < 100, window=[0,1], +2, count=3
right=2: product=100 ≥ 100 → shrink: 100//10=10, left=1, window=[1,2], +2, count=5
right=3: product=60 < 100, window=[1,3], +3, count=8

4 The Code (Annotated) ▾

Solution 1 — Sliding Window (Optimal, Interview Pick)

def numSubarrayProductLessThanK(nums, k):
if k <= 1: ← No positive-integer product is < 1; infinite-loop guard
return 0
product = 1 ← Running product of current window (start neutral at 1)
left = 0 ← Left boundary of sliding window
count = 0 ← Accumulates answer
for right in range(len(nums)): ← Expand right pointer across all elements
product *= nums[right] ← Include nums[right] in the window
while product >= k: ← Window too large — must shrink from left
product //= nums[left] ← Remove leftmost element; // avoids float drift
left += 1 ← Advance left boundary
count += right - left + 1 ← All subarrays ending at right within valid window
return count
⚠️ Integer Division ( // ) Matters Here

When dividing a product by a factor, use // (integer floor division), not / (float). With float division, repeated multiplications and divisions can accumulate rounding errors: product that should be exactly 100 might become 99.9999… or 100.0001…, causing incorrect comparisons with k. Since all inputs are positive integers, integer division is exact.

Solution 2 — Brute Force (O(n²), for reference)

def numSubarrayProductLessThanK(nums, k):
count = 0
for i in range(len(nums)): ← Try every starting index
product = 1
for j in range(i, len(nums)): ← Extend subarray to the right
product *= nums[j]
if product >= k: ← Products only grow — safe early exit
break
count += 1 ← nums[i..j] is valid
return count
💡 Complexity Summary

Sliding Window: Time O(n) — each element enters and exits the window at most once (left and right each traverse the array once). Space O(1).
Brute Force: Time O(n²) — all (i,j) pairs considered. Space O(1). Still usable for very small n or quick prototyping.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Spot the Sliding Window Signals

"Contiguous subarray" + "constraint on a running aggregate" + "all elements positive" = textbook sliding window. The positivity guarantee means the product behaves monotonically — it grows when the window grows and shrinks when the window shrinks.

Step 2: Start with Brute Force

"The naive approach: try all O(n²) start-end pairs, accumulate product, break early when product ≥ k. O(n²) time, O(1) space. Correct but can we do O(n)?"

Step 3: Formulate the Window Invariant

"I want to maintain the largest window ending at each right whose product is < k. Because all elements are positive, once product ≥ k I can always shrink from the left to fix it — and I never need to revisit elements I've already removed."

Step 4: Derive the Counting Formula

"Once [left..right] is valid, every subarray ending at right with start ≥ left is also valid (sub-product ≤ full product). There are right-left+1 such subarrays. I add this to count each time."

Step 5: Mention the Edge Case Proactively

"I need a guard for k ≤ 1: since all nums ≥ 1, no product can be < 1. Without this guard, the while loop would run forever." This shows you think about correctness, not just the happy path.

🗣️ What to Say to the Interviewer

"Contiguous subarrays with a product constraint — classic sliding window setup.
All elements are positive, so products are monotone: bigger window = bigger product.
I'll use left and right pointers. For each right: multiply in nums[right].
If product >= k: shrink from left until valid. Then count += right - left + 1.
Edge case: k <= 1 returns 0 immediately — no positive product is strictly less than 1.
Time O(n) — each element enters and exits at most once. Space O(1)."
🔮 Likely Follow-up Questions
  • "What if elements can be negative or zero?" → Monotonicity breaks — sliding window doesn't work directly. Need DP or divide-and-conquer.
  • "Return the actual subarrays, not just count?" → At each valid (left, right) you'd collect right-left+1 new subarrays — O(n²) total space worst case.
  • "Maximum product subarray?" → Kadane variant — track both max and min product because negatives can flip (#152).
  • "What if we need product exactly equal to k?" → Use the AtMost(k) - AtMost(k-1) trick common in sliding window counting problems.
