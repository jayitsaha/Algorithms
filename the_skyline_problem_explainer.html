LeetCode #218

The Skyline Problem

Return the skyline formed by a set of buildings — a list of key points where the visible height changes.

Hard Advanced Data Structures Events + Max Heap Time: O(n log n) Space: O(n)
1 Understanding the Problem ▾

You are given a list of buildings. Each building is described by three integers: [left, right, height], where left and right are the x-coordinates of its left and right walls, and height is how tall it stands. Buildings can overlap. Return the skyline: the list of key points [x, height] where the silhouette's height changes as seen from the outside.

💡 Key Insight

Think of looking at a city from a distance. You see only the tallest building at any given x-position. The skyline is the series of step-changes in that maximum height — each change is a "key point." We need to find exactly where the max-visible height rises or falls, and output those x-coordinates with their new height.

✓ Input: buildings

[2, 9, 10]
[3, 7, 15]
[5,12, 12]

✓ Output: skyline key points

[2, 10] ← rises from 0 to 10
[3, 15] ← rises from 10 to 15
[7, 12] ← drops from 15 to 12
[9, 12] ← stays 12 (no key pt)
[12, 0] ← drops to ground

Why only left/right edges matter: Between two consecutive building edges, no building starts or ends — the set of active buildings (and therefore the max height) is constant. The skyline is flat there. We only need to check at left/right edges.

Critical edge cases: Two buildings starting at the same x (take the tallest — no intermediate key point); a building ending exactly where another starts (careful tie-breaking); all buildings at the same height (only one key point for the group); a single building (two key points: rise and drop).

2 The Algorithm: Sweep Line + Max Heap ▾

Imagine a vertical line sweeping from left to right across the x-axis. At each building edge (an event), we update a max-heap of active building heights, then check whether the maximum has changed. If it has, we emit a new key point.

1 Generate events. For each building [L, R, H], emit two events: a start at x=L with height=H, and an end at x=R with height=H. We now have 2n events total.
2 Sort events carefully. Sort by x. At the same x: starts before ends (so a building that starts exactly where another ends counts at that x). Among same-x starts, process taller first (avoid spurious low intermediate key points). Among same-x ends, process shorter first.
3 Process events with a max-heap + lazy deletion. For a start event: push the height. For an end event: mark that height as "done" in an active-count map (lazy — don't remove from heap yet). Before querying the max: pop stale (zero-count) entries from the heap top.
4 Detect changes. After processing each event, read current max from heap. If it differs from the last emitted height, append [x, cur_max] to result. Seed the heap with height 0 (the ground) so it's never empty.

🔑 Encoding Trick for Sort Order

Emit start events as (L, -H) and end events as (R, +H). Sorting these tuples lexicographically puts starts before ends at the same x (negative numbers sort before positive), and taller starts before shorter starts at the same x (more-negative first). This single sort handles all tie-breaking automatically.

3 Why It Works: The Invariant ▾
🔐 The Sweep-Line Invariant

After processing all events at position x, the max-heap's valid top equals the height of the tallest building whose x-range contains x. This is exactly the skyline height at x. Whenever this value changes relative to the previous emitted height, we have a new key point.

Why only event positions? Between two consecutive event x-values, the set of active buildings is unchanged (no building starts or ends). The max height is constant — the skyline is a flat horizontal segment. Key points can only appear at event positions.

Why lazy deletion is correct. A stale entry in the heap (a building that has ended) can only cause a wrong answer if it is at the top of the heap and we mistakenly read it as the current max. But we always clean stale tops before reading. So even if stale entries lurk deeper in the heap, they are harmless — they'll be cleaned when they eventually bubble to the top.

Why seed with height 0. After all buildings end, the heap should report height 0 (ground level). The sentinel 0 ensures we never have an empty heap and correctly emit the final [x, 0] key point.

⚠️ Common Mistake: Storing Only Heights

If you store just the height (not the building identity) in the heap, you cannot distinguish between two buildings of the same height — you won't know which one ended. Always store enough information to identify the specific building instance (e.g., store a (height, right_boundary) pair and use an active-count map indexed by height).

💡 Algorithm Deep-Dive: Sweep Line

Origin: Sweep-line algorithms were formalized by Shamos & Hoey (1976) for computational geometry. They reduce 2D problems to a sequence of 1D event-driven queries as a vertical line moves left to right.

Core pattern: Extract events at boundary x-coordinates → sort → maintain a status structure (set, heap, sorted list) describing what is active → at each event, update the structure and answer "what is the current max/min/count?"

Recognize when: "Intervals on a line," "what is active at time t," "overlap counting," "buildings/events with start and end."

💡 Algorithm Deep-Dive: Lazy-Deletion Max Heap

Why lazy? Python's heapq (and most heap libraries) support O(log n) push/pop but not O(log n) arbitrary removal. Removing a specific element mid-heap requires O(n) linear search.

Solution: Defer removal. When a building ends, decrement its count in an active dictionary. Before reading the max, pop entries from the heap top as long as their active count is 0. Each element is pushed and popped at most once → O(n log n) total.

Generalize: Lazy deletion applies to Dijkstra's algorithm (stale distance entries in the heap), K-th Largest Element in a stream, and any problem where you need a heap with conditional removal.

4 The Code (Annotated) ▾

Solution 1 — Sweep Line + Max Heap with Lazy Deletion (Optimal · Interview Pick)

import heapq
from collections import defaultdict
def getSkyline(buildings):
events = [] ← Collect all building edge events
for L, R, H in buildings:
events.append((L, -H)) ← Start: negative height so starts sort before ends at same x
events.append((R, H)) ← End: positive height; also taller starts before shorter at same x (more-negative first)
events.sort() ← Lex sort: by x then by h; starts (-H) before ends (+H) at same x
result = []
max_heap = [0] ← Min-heap with negated heights; seed with 0 (ground) that never expires
active = defaultdict(int) ← active[h] = count of buildings with height h still alive
active[0] = 1 ← The ground sentinel is always active
for x, h in events:
if h < 0: ← Start event: h is negative (encoding)
heapq.heappush(max_heap, h) ← Push -height; heap stores negatives so smallest neg = largest height
active[-h] += 1 ← Mark this height as having one more active building
else: ← End event: h is positive actual height
active[h] -= 1 ← Decrement; do NOT remove from heap (lazy deletion)
# Lazy cleanup: discard stale heap tops
while active[-max_heap[0]] == 0: ← Top of heap has no living buildings → stale
heapq.heappop(max_heap) ← Remove stale; repeat until top is valid
cur_max = -max_heap[0] ← Negate back: current tallest active building height
if not result or result[-1][1] != cur_max:← Only emit if height actually changed
result.append([x, cur_max]) ← New key point: height changed at x
return result
💡 Why the negative encoding works

Python's heapq is a min-heap. By storing -height, the smallest element (most negative) corresponds to the largest height. So -max_heap[0] gives us the maximum height without any extra bookkeeping.

⚠️ Don't forget the ground sentinel

Without active[0] = 1 and 0 in the heap, the lazy-cleanup while-loop might empty the heap entirely, causing an IndexError on max_heap[0]. The sentinel height=0 (ground) is always "active" — it represents the baseline when all buildings have ended.

💡 Complexity Summary

Sweep Line + Max Heap: Time O(n log n) — sorting 2n events + each building pushed/popped at most once. Space O(n) — heap and active map.

Brute Force: Time O(n²) — O(n) x-positions × O(n) scan. Space O(n). Fine for n≤100, fails for n=10000.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Clarify and Recognize the Pattern

Ask: "Do buildings have integer coordinates? Can heights repeat?" Then observe: you have intervals [L, R] on an x-axis with associated values (heights). You need the maximum value among all active intervals at critical x-positions. This is a classic sweep-line + ordered data structure pattern.

Step 2: State the Brute Force

"For each unique edge x-position (2n of them), scan all n buildings and find the tallest that covers x. That's O(n²). Too slow for large inputs, but correct."

Step 3: Key Observation → Events

"The skyline only changes at building edges — nowhere else. So I'll model left edges as 'start' events and right edges as 'end' events. Sort all 2n events by x. Process them left to right, maintaining the tallest active building."

Step 4: Choose Max-Heap for Dynamic Maximum

"I need the max of a dynamic set — buildings join on start events and leave on end events. A max-heap gives O(log n) push. Since Python's heapq is min-heap, I store -height. Removal isn't O(log n) directly, so I'll use lazy deletion: mark removed, clean from top before querying."

Step 5: Handle the Tie-Breaking

"Critical detail: at the same x, process starts before ends. If I encode starts as (x, -H) and ends as (x, +H) and sort lexicographically, Python's sort naturally handles this. At same x, -H (negative) sorts before +H (positive)."

🗣️ What to Say to the Interviewer

"The key observation: the skyline can only change at building edges, so I model
left edges as start events and right edges as end events — 2n events total.
After sorting events by x (with careful tie-breaking), I sweep left to right,
maintaining a max-heap of active building heights with lazy deletion.
After each event, if the heap's max changed, I emit a key point.
Time O(n log n) for sorting plus O(n log n) total heap ops. Space O(n)."
🔗 Follow-Up Questions You May Get

• "What if we need this for online queries?" — Replace heap with a sorted set (Python: sortedcontainers.SortedList) supporting O(log n) add, remove, and max.
• "What if buildings have the same height repeatedly?" — The active counter handles multiplicities correctly.
• "Can you do it without lazy deletion?" — Yes: use a SortedList and remove directly in O(log n). Trade-off: more complex data structure, same asymptotic complexity.

7 Summary & Key Takeaways ▾
The skyline changes only at building edges — model these as 2n sorted events, not a continuous sweep
A max-heap tracks the tallest active building in O(log n) per update — far better than O(n) linear scan per event
Lazy deletion avoids O(n) removal from the heap: mark ended, clean stale tops before reading max
Sort encoding (start as -H, end as +H) gives automatic tie-breaking: starts before ends at same x, taller starts first
Seed the heap with height 0 (ground sentinel) so it is never empty and correctly emits the final drop-to-zero key point
Emit a key point only when cur_max ≠ last emitted height — deduplicates consecutive identical heights
🎯 When to Use Sweep Line + Max Heap
  • Intervals on a line: buildings, meetings, tasks, sensor readings with start/end times
  • Need max/min/count of active (overlapping) elements at any position
  • O(n log n) required; O(n²) brute force is too slow
  • Keywords: "events," "ranges," "overlap," "active at x," "the highest/deepest at time t"
