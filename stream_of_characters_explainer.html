LeetCode #1032

Stream of Characters

Design a data structure that, after each character appended to a stream, reports whether any suffix of the stream matches a word from a given dictionary.

Hard Tries Reverse Trie Time: O(m) per query Space: O(W¬∑L)
1 Understanding the Problem ‚ñæ

You are given a dictionary of words at construction time, then a stream of characters arrives one character at a time. After each new character is received, you must answer: "Does any suffix of all characters seen so far exactly equal one of the dictionary words?"

A suffix of the stream is the last k characters received, for any k from 1 up to the total number of characters so far. You are not looking for arbitrary substrings ‚Äî only substrings that end at the most recently received character.

üí° Key Insight

The question "does the stream end with any dictionary word?" is a suffix matching problem. The challenge is doing this efficiently ‚Äî you cannot buffer unlimited characters and re-scan the full word list on every query. The answer must arrive in time proportional to the maximum word length, not the stream length.

‚úì Match Example

words = ["cd","f","kl"]
stream: 'a','b','c','d'
after 'd': suffix "cd" ‚àà words ‚Üí true

stream continues: 'f'
after 'f': suffix "f" ‚àà words ‚Üí true

‚úó No Match Example

words = ["cd","f","kl"]
stream: 'a','b','c'
after 'a': "a" ‚Äî no match
after 'b': "b","ab" ‚Äî no match
after 'c': "c","bc","abc" ‚Äî none in words ‚Üí false

Why Hard? The stream is online ‚Äî you can't buffer everything then query once. The dictionary can contain thousands of words of varying lengths. A na√Øve approach that checks every word against every possible suffix on every new character is O(Q √ó W √ó L). For Q=40000 queries, W=2000 words, L=2000 chars, that's 160 billion operations. We need O(m) per query where m is the max word length.

Edge cases: A single-character word (matches immediately whenever that character appears); words that are prefixes of other words; characters not appearing in any word (immediately kills all active paths); the very first character matching a one-char word.

2 The Algorithm: Reverse Trie + Active Node Set ‚ñæ
üîë The Core Idea: Reverse to Turn Suffix ‚Üí Prefix

Tries are built to match prefixes efficiently ‚Äî you walk from the root character by character. But we need suffix matching: the new character is always at the right end. Solution: reverse every dictionary word before inserting into the Trie. Now, walking the Trie from left to right with characters as they arrive naturally matches suffixes of the stream.

For words ["cd","f","kl"], we insert reversed versions: "dc", "f", "lk". When the stream reads 'd' then 'c', we walk root‚Üíd‚Üíc which is the reversed word "dc" = original "cd". Match!

1 Build (constructor): For each word, reverse it and insert into the Trie character by character. Mark the final node with is_end = True. This is O(W¬∑L) one-time work.
2 Maintain Active Set: Between queries, keep a set of currently-live Trie nodes. Each node represents "some suffix of the stream so far has matched this many characters of some reversed word." Initially the set is {root}.
3 Per query ‚Äî seed with root: At the start of every query, add root to the next active set. Root represents "start matching a brand-new word from this character." Every character is a potential start of a match.
4 Per query ‚Äî advance all active nodes: For each node in the previous active set, check if it has a child for the new character c. If yes, add that child to the next set. If no, that path dies ‚Äî it can no longer form a match.
5 Per query ‚Äî check word-end: If any node in the new active set has is_end = True, a complete word match has been found. Return True. Otherwise return False.

‚öÝÔ∏è Why active set size is bounded by max word length

Each active node sits at a specific depth in the Trie. Depth corresponds to "how many characters of a reversed word have been matched." Since the deepest Trie node is at depth = max word length m, and we can have at most one active node per depth level (because paths through the Trie don't merge), the active set has at most m nodes. So each query processes at most m nodes ‚Äî O(m) time.

3 Why It Works: The Invariant ‚ñæ
üîê The Invariant

After processing the i-th character, the active node set contains exactly the Trie nodes reachable by reading the reversed stream from position i backwards for any length k (1 ‚â§ k ‚â§ m). In other words: node N is active if and only if the k most recent stream characters (for some k) spell out the reversed-word prefix that ends at node N.

Why adding root every step is necessary and correct: Each new character is potentially the first character of a new word match (the rightmost character, since words grow rightward). The root represents "zero characters matched ‚Äî start fresh." Without re-adding root, we'd miss matches that start at the current character. By always adding root, every query automatically seeds a new potential match at depth 0.

Why the math works: Word "cd" matches stream suffix "...cd". In reversed terms: we're looking for "dc" as a prefix of the reversed stream suffix reading from the current position backwards. Walking root‚Üíd‚Üíc in the Trie while reading d then c from the stream is exactly that: each step advances one character in reverse-chronological order.

Proof of completeness: Every matching suffix is eventually caught. Any word W of length k that ends at the current position means the last k stream characters spell W. Reversed, the first k characters of the reversed stream-suffix spell W reversed. Since we seed root at every step and advance along every active path, the path root‚ÜíW[k-1]‚ÜíW[k-2]‚Üí...‚ÜíW[0] will be constructed over k consecutive queries ‚Äî and the final node W[0] is marked is_end = True.

‚öÝÔ∏è Common Mistake: Forgetting to Add Root on Every Query

If you only track the previous active set without seeding root, you'll only continue existing partial matches ‚Äî you'll never start a new match. The second character of every match beyond the first query would be missed entirely.

4 The Code (Annotated) ‚ñæ

Solution 1 ‚Äî Reverse Trie + Active Node Set (Optimal, Interview Pick)

class TrieNode: ‚Üê One node per character in the Trie
def __init__(self):
self.children = {} ‚Üê Maps char ‚Üí child TrieNode
self.is_end = False ‚Üê True if this node completes a reversed word
class StreamChecker:
def __init__(self, words: List[str]):
self.root = TrieNode() ‚Üê Root of the reverse-word Trie
for word in words: ‚Üê Insert every word reversed
node = self.root
for ch in reversed(word): ‚Üê Walk reversed word: "cd" ‚Üí walk 'd','c'
if ch not in node.children:
node.children[ch] = TrieNode()‚Üê Create new node if path doesn't exist
node = node.children[ch]
node.is_end = True ‚Üê Mark end of reversed word
self.active = {self.root} ‚Üê Active set starts with just root
def query(self, letter: str) -> bool:
nxt = {self.root} ‚Üê ALWAYS seed with root (new match can start)
for node in self.active: ‚Üê Advance every currently-live path
if letter in node.children: ‚Üê Does this path have an edge for 'letter'?
nxt.add(node.children[letter])‚Üê Yes: child becomes active next round
self.active = nxt ‚Üê Update active set (dead paths fall off)
return any(n.is_end for n in nxt) ‚Üê Any complete word found? ‚Üí True
‚öÝÔ∏è Why self.active starts as {self.root} not {}

If active were empty initially, the very first call to query() would try to advance no nodes, leaving nxt = {root} with no advancement done. That's actually fine ‚Äî but it means the first character is never checked against root's edges (only root is added to nxt, not advanced). This subtle bug is avoided by initializing active = {root} and always adding root to nxt at query start. The advancement loop does the actual work.

üí° Complexity Analysis

Build: O(W¬∑L) time ‚Äî insert W words, each of length up to L.
Space: O(W¬∑L) ‚Äî Trie stores at most W√óL nodes in worst case (no shared prefixes).
Query: O(m) per call ‚Äî active set has at most m nodes; each node does O(1) child lookup.
Total for Q queries: O(W¬∑L + Q¬∑m). Stream length doesn't matter ‚Äî this is the key win.

Solution 2 ‚Äî Brute Force with Trimmed Buffer (intuitive, but slower)

class StreamChecker:
def __init__(self, words):
self.words = set(words) ‚Üê Hash set for O(1) word lookup
self.buf = [] ‚Üê Rolling buffer of recent characters
self.maxlen = max(len(w) for w in words)‚Üê Only keep last maxlen chars needed
def query(self, letter):
self.buf.append(letter) ‚Üê Append newest char to right
if len(self.buf) > self.maxlen:
self.buf.pop(0) ‚Üê Trim oldest: only maxlen chars needed
s = ''.join(self.buf)
return any(s.endswith(w) for w in self.words)‚Üê Check all words as suffixes

Brute force: O(W¬∑m) per query ‚Äî checks every word. Fine to present first to show understanding before optimizing.

5 üé§ Interview Strategy: How to Arrive at This ‚ñæ
Step 1: Recognize ‚Äî Multi-Pattern Suffix Matching on a Stream

The signals: (a) dictionary of patterns given upfront ‚Äî preprocessing allowed; (b) characters arrive one at a time ‚Äî solution must be online; (c) "does stream end with any word" ‚Äî suffix matching. These three together point directly at a Trie with stream-processing logic.

Step 2: Propose and Dismiss Brute Force

"Na√Øve approach: buffer the stream, on each character check all words with str.endswith(). O(W¬∑m) per query. Too slow for 40000 queries with 2000 words of length 2000. I need a data structure that preprocesses the words."

Step 3: Identify the Reversal Trick

"A Trie matches prefixes efficiently. But I need suffixes ‚Äî the new character is always on the right. Key observation: if I reverse every word and insert into a Trie, then reading the stream character by character corresponds to walking the Trie through reversed-word prefixes. Suffix matching becomes prefix matching."

Step 4: Explain the Active Set

"I maintain a set of active Trie nodes between queries. Each represents a partial match in progress. On each new character: (1) always add root (new match starts); (2) advance all existing active nodes via the new character edge; (3) check if any resulting node is a word-end. Dead paths are automatically pruned when no edge exists."

Step 5: State the Complexity Bound

"The active set never exceeds m nodes (max word length), because nodes live at distinct depths and there's at most one per depth. So each query is O(m) regardless of stream length ‚Äî the key efficiency gain over brute force."

üó£Ô∏è What to Say to the Interviewer

"This is an online suffix matching problem against a dictionary.
Brute force: buffer stream, check all words with endswith ‚Äî O(W¬∑m) per query, too slow.
Better: preprocess words into a Trie. The twist: reverse every word before inserting.
Why? Suffixes grow rightward, Tries match leftward ‚Äî reversing aligns them.
Then I maintain an active node set between queries. Per character:
‚Ä¢ Add root (new match starts from this char)
‚Ä¢ Advance all active nodes via the new char's edge (dead paths drop off)
‚Ä¢ If any live node is a word-end ‚Üí return True
Active set has at most m nodes (max word length) ‚Üí O(m) per query.
Build: O(W¬∑L). Total: O(W¬∑L + Q¬∑m). Stream length doesn't affect per-query time."
üîÆ Likely Follow-up Questions
  • "What if we also need to find substring matches (not just suffixes)?" ‚Üí Need Aho-Corasick algorithm ‚Äî it uses failure links to handle all positions.
  • "What if new words can be added dynamically?" ‚Üí Inserting a reversed word into the Trie is still O(L). The active set is unaffected.
  • "Can you implement this without extra space for the active set?" ‚Üí Not really; the active set IS the state. But we could use a deque of (node, remaining_chars) pairs instead.
  • "What if some words are prefixes of others?" ‚Üí Handled naturally: multiple is_end marks can exist along one path. The set catches all of them simultaneously.
