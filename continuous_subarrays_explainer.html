LeetCode #2762

Continuous Subarrays

Count the total number of continuous subarrays where the absolute difference between any two elements is at most 2.

Medium Monotonic Queue Two Deques (Min/Max) Time: O(n) Space: O(n)
1 Understanding the Problem ▾

We're given an integer array nums. A subarray is continuous if for every pair of indices i and j within it, |nums[i] − nums[j]| ≤ 2. We must count all such subarrays.

💡 Key Insight

Instead of checking every pair, notice that if max(subarray) − min(subarray) ≤ 2, then every pairwise difference is also ≤ 2 (since max−min is the largest possible difference within the subarray). This collapses an O(n²) pairwise check to a single range comparison.

✓ Valid: [5, 4, 6]

max=6, min=4
6 − 4 = 2 ≤ 2 ✓
All pairs: |5−4|=1, |5−6|=1, |4−6|=2

✗ Invalid: [5, 4, 7]

max=7, min=4
7 − 4 = 3 > 2 ✗
|4 − 7| = 3 violates the rule

Counting trick: For a fixed right endpoint r, once we know the leftmost valid left endpoint l, all subarrays nums[l'..r] where l' ∈ [l, r] are also valid (a subarray of a valid subarray is valid). That's exactly r − l + 1 subarrays.

Edge cases: Single element (always valid: max−min=0), all identical values (every subarray valid), monotone increasing array (window grows until range exceeds 2, then shrinks), alternating far values (frequent shrinks).

2 The Algorithm: Two Monotonic Deques ▾

We maintain a sliding window [left, right] with two deques tracking the window's minimum and maximum in O(1). Each deque stores indices (not values) and maintains a monotonic ordering of values.

min_deque — increasing (front to back)
front = index of window MINIMUM

Push right: pop back entries
whose values ≥ nums[right]

Pop front: if front index < left
max_deque — decreasing (front to back)
front = index of window MAXIMUM

Push right: pop back entries
whose values ≤ nums[right]

Pop front: if front index < left
1 Initialize: min_dq = max_dq = deque(), left = 0, result = 0.
2 Expand right: For each index right, prune the back of each deque (removing elements that violate monotonicity), then push right onto both deques.
3 Shrink left if needed: While nums[max_dq[0]] − nums[min_dq[0]] > 2, advance left and evict any deque fronts whose index is now below left.
4 Accumulate count: Add right − left + 1 to result. This counts all valid subarrays ending at right.
5 Return result.
🔑 Why Deques Beat Sorted Sets

A sorted set gives O(log n) insert/delete — fine, but not optimal. Monotonic deques give amortized O(1): each index is pushed and popped at most once across the full scan. For large inputs (n ≤ 10⁵) this halves runtime constants. The key trade-off: deques only work for sliding-window queries (not arbitrary removal), which is exactly our case.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every point in the loop: min_dq[0] holds the index of the minimum element in nums[left..right], and max_dq[0] holds the index of the maximum element. The current window always satisfies max − min ≤ 2.

Why min_dq stays correct: When we push index r, we pop any back entry b where nums[b] ≥ nums[r]. Since r > b, entry r lasts longer in the window AND has a smaller or equal value — so b can never be the minimum while r is present. Popping b is safe. After pruning, the deque remains increasing from front to back.

Why max_dq stays correct: Symmetrically, we pop back entries smaller than nums[r]. Those entries can never become the window maximum while the larger r is inside the window. After pruning, the deque remains decreasing front to back.

Why the count is exact: When the window [left, right] is valid, every sub-window [left', right] for left' ∈ [left, right] is also valid (fewer elements means the range can only shrink). So exactly right − left + 1 valid subarrays end at right, all counted in one addition.

⚠️ Stale Front Entries — The Hidden Bug

When left advances past a deque's front index, that front refers to an element now outside the window. If not evicted, the deque's reported min/max is stale — it could claim the window is invalid when it's actually fine, or valid when it's not. Always check if deque[0] < left: popleft() after advancing left.

4 The Code (Annotated) ▾

Solution 1 — Two Monotonic Deques (Optimal · Interview Pick)

from collections import deque ← stdlib; no import needed on LeetCode
def continuousSubarrays(nums: list[int]) -> int:
min_dq, max_dq = deque(), deque() ← both store indices; front = min/max of window
left = 0 ← left boundary of sliding window
result = 0 ← count of valid subarrays
for right in range(len(nums)): ← expand window one step at a time
# Maintain increasing min_dq
while min_dq and nums[min_dq[-1]] >= nums[right]: ← back element ≥ new value → useless for min
min_dq.pop() ← pop it; new value is smaller AND arrives later
# Maintain decreasing max_dq
while max_dq and nums[max_dq[-1]] <= nums[right]: ← back element ≤ new value → useless for max
max_dq.pop()
min_dq.append(right) ← push current index onto both deques
max_dq.append(right)
# Shrink window until max − min ≤ 2
while nums[max_dq[0]] - nums[min_dq[0]] > 2: ← window range exceeds 2 → invalid
left += 1 ← shrink from left
if min_dq[0] < left: min_dq.popleft() ← evict stale min front if it left the window
if max_dq[0] < left: max_dq.popleft() ← evict stale max front if it left the window
result += right - left + 1 ← all (right-left+1) subarrays ending here are valid
return result
⚠️ Why >= and <= (not strict) in the pruning loops?

Using >= for min_dq means we also pop equal-valued back entries. This is correct: if two indices hold the same value, the earlier one will leave the window first — it's redundant to keep it. The deque stays as compact as possible, which is fine. Using only > would leave stale duplicates and may cause the wrong index to be reported as front after the earlier one expires.

Solution 2 — Sorted Container (Simpler, O(n log n))

from sortedcontainers import SortedList ← third-party; available on LeetCode
def continuousSubarrays(nums):
sl = SortedList() ← maintains sorted order for O(1) max/min
left = 0; result = 0
for right in range(len(nums)):
sl.add(nums[right]) ← O(log n) insert
while sl[-1] - sl[0] > 2: ← last = max, first = min
sl.remove(nums[left]) ← O(log n) remove by value
left += 1
result += right - left + 1
return result
💡 Complexity Summary

Two Deques (Optimal): Time O(n) amortized, Space O(n) — each element pushed/popped at most once per deque
Sorted Container: Time O(n log n), Space O(n) — simpler code but log factor per operation
Brute Force: Time O(n²), Space O(1) — enumerate all subarrays, compute max/min each time

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Subarray Counting Shape

"Count subarrays satisfying a constraint on their elements." When a constraint depends only on the min and max of a subarray (not on position), a sliding window works: the constraint is monotone as we grow the window. As right expands, the range can only stay or increase. Shrinking from left restores validity.

Step 2: Reduce "Every Pair" to "Max − Min"

"Checking every pair is O(n²) per window. But the extremes dominate — if max−min ≤ 2, all pairs have difference ≤ 2. This single comparison replaces the quadratic check. Now I need fast dynamic max and min as the window moves."

Step 3: Propose Two Levels of Solution

"For a clean first solution: use a SortedList — max is sl[-1], min is sl[0], O(n log n). For the optimal solution: two monotonic deques — amortized O(1) per operation, O(n) total. The deques store indices and we maintain monotonic value order by pruning the back on each push."

Step 4: Use the Right-Left Counting Trick

"Once the window [left, right] is valid, every subarray [left', right] for left' ∈ [left..right] is also valid (sub-windows of valid windows are valid). Accumulate right−left+1 at each valid step — one addition counts all subarrays ending at right."

🗣️ What to Say to the Interviewer

"The key observation: every pair having |a−b| ≤ 2 is equivalent to max−min ≤ 2 for the subarray.
So I use a sliding window. I need to query window max and min dynamically as I expand right.
I'll maintain two monotonic deques: one increasing (for min), one decreasing (for max).
When max−min exceeds 2, I advance left and evict stale deque fronts.
At each valid right, I add right−left+1 to the result — that's all subarrays ending here.
Each element enters and leaves each deque at most once → O(n) total time."
🔮 Likely Follow-up Questions
  • "What if the threshold is k instead of 2?" → Same algorithm, replace 2 with k
  • "What if we want the actual subarrays, not just count?" → Collect [left, right] pairs at each step
  • "Can you do it with one deque?" → No — you need both min and max independently; one deque can only track one extreme
  • "How does this relate to Sliding Window Maximum (#239)?" → Same max_deque technique, just without the min constraint

7 Summary & Key Takeaways ▾
Reduce "every pair" check to max−min ≤ 2 — a single O(1) comparison
Two monotonic deques give O(1) amortized window min and max (each element pushed/popped at most once)
Sliding window: grow right freely, shrink left when constraint violated
Count right−left+1 per valid step — covers every sub-window ending at right
Always evict stale deque fronts (index < left) after advancing left
