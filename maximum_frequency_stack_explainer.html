LeetCode #895

Maximum Frequency Stack

Design a stack-like data structure that always pops the element with the highest push frequency. On frequency ties, pop the most recently pushed element among the tied group.

Hard Data Structure Design Frequency Buckets Time: O(1) Space: O(n)
1 Understanding the Problem ▾

We need a custom stack supporting two operations: push(val) adds a value, and pop() removes and returns the element that has been pushed the most times overall. If multiple elements share the same maximum frequency, we return whichever was pushed most recently (i.e., it came from the "top" of its frequency group).

💡 Key Insight

Instead of one stack, maintain a dictionary of stacks indexed by frequency. group[f] is a list of all elements whose current push count equals f, ordered by the time they reached that level. Also track maxfreq — the highest currently active frequency. pop() always removes and returns group[maxfreq][-1].

⚠️ The Tie-Breaking Subtlety

When elements share the highest frequency, we pop the most recently pushed one among those tied elements. Using a Python list as a stack within each frequency group handles this automatically: list.append() preserves push order, and list.pop() removes the most recent. No sorting or extra bookkeeping needed.

2 The Algorithm: Frequency Bucket Stacks ▾

Three data structures work together in perfect O(1) harmony:

  • freq_count — dict mapping each value to its current push count
  • group — dict mapping each frequency level f to a list (stack) of values currently pushed exactly f times
  • maxfreq — an integer tracking the highest frequency level with at least one element
1 push(val): Compute new frequency f = freq_count[val] + 1. Update freq_count[val] = f. Append val to group[f]. Update maxfreq = max(maxfreq, f). All steps are O(1).
2 pop(): Pop the top of the max-frequency group: val = group[maxfreq].pop(). Decrement freq_count[val]. If group[maxfreq] is now empty, decrement maxfreq by 1. Return val.
3 Why maxfreq drops by exactly 1: Frequencies increase one push at a time. If group[k] empties, no element can have frequency k+1 or higher without passing through k first. The next non-empty group must be k−1. No scanning needed.
🔑 The "Stale Entry" Trick — Why Group Entries Aren't Cleaned Up

When 5 is pushed a second time, it's added to group[2]. Its entry in group[1] becomes "stale" but is not removed. This is intentional: when 5 is later popped from group[2] (reducing its frequency back to 1), the stale group[1] entry becomes active again and correctly represents 5 at frequency 1. No cleanup is ever needed — stale entries are harmlessly dormant until they're needed.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At any point during execution: group[f] contains, in push-arrival order, every element whose current frequency (as given by freq_count) equals f. maxfreq equals the largest f for which group[f] is non-empty.

Why pop() always returns the correct element: We want the most-frequent element, breaking ties by most-recently-pushed. The element at group[maxfreq][-1] satisfies both: it's in the highest-frequency group (most frequent), and it's at the top of that group's stack (most recently pushed to that frequency level, which is equivalent to most recently pushed overall among tied elements).

Why stale entries don't corrupt the invariant: After popping 5 from group[3], we set freq_count[5] = 2. The stale 5 in group[1] will only be reachable when maxfreq = 1. By then, 5's current frequency will be 1 (it was at 2, got popped from group[2], became 1). So the stale entry is correct again when it becomes reachable. This works because a value's journey through frequency levels is strictly sequential — it passes through every level on the way up and down.

Edge cases:

  • All pushes of the same value: maxfreq climbs to n, group[n]=[val]. Each pop unwinds one level.
  • All distinct values: all go to group[1]. Pop returns the most recently pushed — correct LIFO behavior.
  • Interleaved push/pop: maxfreq tracks current maximum correctly at every step.
4 The Code (Annotated) ▾

Solution 1 — Frequency Bucket Stacks (Optimal, O(1) per operation) — Interview Pick

class FreqStack:
def __init__(self):
self.freq_count = {} ← val → current push count
self.group = {} ← freq_level → list/stack of vals at that level
self.maxfreq = 0 ← highest frequency with ≥1 element currently
def push(self, val: int) -> None:
f = self.freq_count.get(val, 0) + 1 ← new frequency for this val
self.freq_count[val] = f ← record updated frequency
if f not in self.group: ← lazily create bucket if first time
self.group[f] = []
self.group[f].append(val) ← add to its frequency bucket (preserves order)
self.maxfreq = max(self.maxfreq, f) ← update max if this is a new high
def pop(self) -> int:
val = self.group[self.maxfreq].pop() ← top of max-frequency bucket → O(1)
self.freq_count[val] -= 1 ← this val's frequency just dropped by 1
if not self.group[self.maxfreq]: ← if that bucket is now empty...
self.maxfreq -= 1 ← ...lower maxfreq by exactly 1 (always correct)
return val
⚠️ Why defaultdict(list) is a clean alternative

Using from collections import defaultdict; self.group = defaultdict(list) removes the if f not in self.group guard — defaultdict auto-creates an empty list for any new key. Either approach is fine in interviews; the explicit check makes the logic more visible.

Solution 2 — Heap (O(log n) per operation, easier to derive)

import heapq
class FreqStack:
def __init__(self): self.freq={}; self.heap=[]; self.time=0
def push(self, val):
self.freq[val] = self.freq.get(val,0) + 1
self.time += 1
heapq.heappush(self.heap, (-self.freq[val], -self.time, val)) ← O(log n)
def pop(self):
_, _, val = heapq.heappop(self.heap) ← O(log n)
self.freq[val] -= 1
return val

The heap key is (-frequency, -push_time, val): highest frequency wins, then latest push time wins on ties. Correct but O(log n). Propose this first, then offer to optimize to O(1).

💡 Complexity Summary

Frequency Buckets (optimal): push O(1), pop O(1), Space O(n)
Heap approach: push O(log n), pop O(log n), Space O(n)
The key asymptotic win: buckets exploit frequency monotonicity (can only change by 1) to avoid sorting entirely.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize it's a Design Problem

The words "design a stack" + "most frequent" + "O(1)" are the signals. This is not a single-data-structure problem — you need to compose multiple structures. Don't rush to implement; spend 30 seconds naming what you need to track: frequency per element, and ordering for ties.

Step 2: Propose the Heap as a Starting Point

"A max-heap keyed by (−frequency, −push_time) would work correctly. push is O(log n), pop is O(log n). Can we do better?" This shows you know the standard tool and are motivated to optimize.

Step 3: Observe Frequency Monotonicity

"Frequency can only go up by 1 on push and down by 1 on pop — it's never discontinuous. So maxfreq is always a valid integer that changes by ±1. I don't need a sorted structure at all. I just need a way to group elements by frequency and track the current max."

Step 4: Design the Bucket + maxfreq Solution

"group[f] = list of elements at frequency f (stack ordering). push: freq[val]++, append to group[new freq], update maxfreq. pop: pop group[maxfreq], freq[val]--, if empty: maxfreq--. Everything O(1)."

🗣️ What to Say to the Interviewer

"I need to always return the most frequent element, with recency as a tiebreaker.
My first instinct: a max-heap on (−frequency, −push_time) gives O(log n) — that works.
But I notice frequency can only change by 1 per operation, so it's monotone. That means
I can track maxfreq as a single integer rather than sorting. I'll use a dict of stacks:
group[f] holds all elements currently at frequency f, ordered by push time.
push: freq[val]++; group[new_f].append(val); maxfreq = max(maxfreq, new_f).
pop: val = group[maxfreq].pop(); freq[val]--; if group[maxfreq] empty: maxfreq--.
Both operations are O(1). Space is O(n) total entries across all group stacks."
🔮 Likely Follow-up Questions
  • "How would you implement peek()?" → group[maxfreq][-1] — same O(1), no pop
  • "What's the max space usage?" → O(n) — every push creates exactly one entry in group and updates one entry in freq_count
  • "How is this related to LFU Cache?" → Same frequency-bucket idea, but LFU adds eviction with capacity constraints and needs to track minfreq for eviction
  • "Could you use a SortedList instead?" → Yes, but O(log n) per op — the bucket approach is strictly better here

7 Summary & Key Takeaways ▾
Bucket elements by frequency: group[f] is a stack of all elements at frequency level f
Track maxfreq as a single integer — it changes by exactly ±1 per operation (monotone frequencies)
pop() always reads group[maxfreq][-1]: highest frequency + most recently pushed at that level
Never clean up stale group entries — lower-level entries become live again after higher ones are popped
True O(1) push and pop — exploits the fact that frequency levels form a contiguous, monotone sequence

🎯 When to Recognize the Frequency Bucket Pattern
  • "Always access/remove the most (or least) frequent element" → frequency tracking structure needed
  • "O(1) per operation" → can't afford a heap; look for structural monotonicity to bypass sorting
  • "Tie-break by recency" → a list (stack) within each frequency bucket naturally handles this
  • "Frequencies change one at a time" → track max/min as a single integer, adjust by ±1
