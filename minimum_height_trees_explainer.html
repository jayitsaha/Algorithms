LeetCode #310

Minimum Height Trees

Find all nodes that, when chosen as root, produce the minimum possible tree height. There are at most 2 such nodes.

Medium Graph Remove Leaves Iteratively Time: O(n) Space: O(n)
1 Understanding the Problem ▾

We have an undirected tree (connected graph with no cycles, n nodes and n−1 edges). We can pick any node as the root. The height of the resulting rooted tree is the number of edges on the longest downward path. Different root choices produce different heights. We want to find all nodes that give the minimum height.

💡 Key Insight

The optimal root(s) are the center(s) of the tree — the node(s) at the middle of the longest path (the diameter). A tree always has exactly 1 or 2 such centers. We can find them by peeling leaves from the outside in, like peeling an onion, until 1 or 2 nodes remain.

Larger example — n=6, edges: [[3,0],[3,1],[3,2],[3,4],[5,4]]: Node 3 connects to {0,1,2,4}. Node 4 connects to {3,5}. Rooting at 3: max depth = 2 (via 4→5). Rooting at 4: max depth = 2 (via 3→0 or 3→1 or 3→2). Both give height 2. Answer: [3, 4].

Edge cases: n=1 → return [0]. n=2 → return [0,1]. Star graph (one hub, all others are leaves) → answer is the single hub. Even-diameter path → exactly 2 centers. Never 3 centers in any tree.

2 The Algorithm: Iterative Leaf Pruning (Topological Peel) ▾

Leaves — nodes with only one neighbor (degree = 1) — can never be optimal roots for n > 2. Rooting there forces the entire tree's length to hang below them. So we peel them away. The newly exposed nodes become leaves and are peeled in the next round. We continue until 1 or 2 nodes remain.

1 Handle n ≤ 2: Return list(range(n)) immediately — single or two-node trees are trivially optimal no matter which node is root.
2 Build adjacency sets + degree tracking: For each undirected edge [u, v], add v to adj[u] and u to adj[v]. The degree of each node = size of its adjacency set.
3 Find initial leaves: Any node with degree == 1 is a leaf. Collect all such nodes into a list. These form the outermost "ring".
4 Peel iteratively while remaining > 2: Subtract this round's leaf count from remaining. For each leaf, get its sole neighbor, remove the edge (discard leaf from neighbor's adj set). If neighbor's degree drops to 1, add it to the next round's leaf list.
5 Return survivors: When remaining ≤ 2, the current leaf list holds 1 or 2 center nodes. These are the MHT roots.
🔑 Connection to Topological Sort (Kahn's Algorithm)

This is structurally identical to Kahn's algorithm for topological ordering: process degree-0 nodes (source nodes), decrement in-degree of successors, enqueue new zeros. Here "degree-1 = leaf" replaces "degree-0 = source", and we stop at ≤ 2 remaining instead of processing all. Recognizing this connection in an interview demonstrates deep pattern knowledge.

⚠️ Why At Most 2 Centers?

A tree's center lies on its diameter (longest path). A path of even length has 2 middle nodes; odd length has 1. No tree can have 3 or more centers. This is a geometric fact about tree structure — it's why our algorithm terminates correctly when ≤ 2 remain.

3 Why It Works: The Invariant ▾
🔐 The Invariant

After each peeling round, the remaining nodes form a subtree that is strictly more central than the removed leaves. Specifically: for any leaf node u with neighbor v, the height when rooting at u equals 1 + height(v). Since height(v) is already optimal for the subtree not including u, removing u can only improve (or maintain) the minimum. The surviving core after all peeling is exactly the set of globally optimal roots.

Formal proof sketch: Let L be a leaf with sole neighbor v. Root the tree at L: height = 1 + (longest path starting from v not going through L) = 1 + h(v). Root at v: height = max(h(v), 1) = h(v) [since h(v) ≥ 1 when n > 2]. So height(L) > height(v) — L is always strictly worse than its neighbor. Induction across rounds shows each peeling step preserves the optimal solution in the remaining nodes.

Complexity analysis: Each of the n nodes is processed (and removed from its neighbors' adjacency sets) at most once. Each of the n−1 edges is removed once. Total: O(n) time. Adjacency sets and degree array: O(n) space.

💡 Intuition by Example: Star Graph

Consider a star: node 0 is the hub, nodes 1..n are all leaves. Round 1: peel all n leaves. Remaining = 1 (just node 0). Done! Node 0 is the center — rooting there gives height 1, while rooting at any leaf gives height 2. The algorithm peels all spokes in one round and correctly identifies the hub.

⚠️ Why We Need Adjacency Sets (Not Just a Degree Array)

When we peel a leaf, we need to know which neighbor to update. With only a degree array, we'd have to scan all edges to find the neighbor — O(n) per leaf. With adjacency sets, next(iter(adj[leaf])) gives the sole neighbor in O(1), and adj[neighbor].discard(leaf) removes the edge in O(1). Total: O(n).

4 The Code (Annotated) ▾

Solution 1 — Iterative Leaf Pruning (Optimal · Interview Pick)

def findMinHeightTrees(n, edges):
if n <= 2: ← n=1 or n=2: every node is an MHT root
return list(range(n))
adj = [set() for _ in range(n)] ← adjacency sets for O(1) discard
for u, v in edges: ← undirected: add both directions
adj[u].add(v); adj[v].add(u)
leaves = [i for i in range(n) if len(adj[i]) == 1] ← nodes with exactly 1 neighbor
remaining = n ← total unpeeled node count
while remaining > 2: ← keep peeling: stop when ≤ 2 left
remaining -= len(leaves) ← subtract this round's leaves
new_leaves = []
for leaf in leaves:
neighbor = next(iter(adj[leaf])) ← leaf has exactly 1 neighbor
adj[neighbor].discard(leaf) ← remove this edge from graph
if len(adj[neighbor]) == 1: ← neighbor now has degree 1 → new leaf
new_leaves.append(neighbor)
leaves = new_leaves ← advance to next ring
return leaves ← 1 or 2 center node(s)
⚠️ Critical: Why "remaining > 2" and not "> 1"?

Trees can have either 1 or 2 centers. Stopping at remaining ≤ 2 handles both cases. If we stopped at remaining ≤ 1, we'd peel one of the two valid centers when the diameter is even — losing a correct answer. The condition remaining > 2 is the exact right threshold.

Solution 2 — Brute Force BFS (O(n²) · Too slow for large n)

def findMinHeightTrees_brute(n, edges):
if n == 1: return [0]
adj = [[] for _ in range(n)]
for u, v in edges: adj[u].append(v); adj[v].append(u)
def bfs_height(root): ← BFS to find tree height from root
seen, q, h = {root}, [root], 0
while q:
nxt = [nb for node in q for nb in adj[node] if nb not in seen]
seen.update(nxt); q = nxt; h += bool(nxt)
return h
heights = [bfs_height(i) for i in range(n)] ← O(n) BFS × n roots = O(n²) total
min_h = min(heights)
return [i for i, h in enumerate(heights) if h == min_h]
💡 Complexity Summary

Leaf Pruning (Optimal): Time O(n) — every node and edge removed exactly once. Space O(n) for adj sets and degree tracking.
Brute Force BFS: Time O(n²) — n roots × O(n) BFS. TLEs at n = 50,000.
LeetCode constraint: n up to 2×10⁴, so O(n) is needed.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify the Structure

"n nodes, n−1 undirected edges, connected → this is a tree. We need to find optimal root(s). The key property: tree has no cycles, so BFS from any node visits all nodes."

Step 2: Brute Force Baseline

"For each of n nodes, root the tree there, BFS to compute height. Track minimum. O(n²) total. Works, but too slow for n up to 20,000."

Step 3: Geometric Observation

"Leaves can't be optimal — they're at the boundary. If I move from a leaf toward its neighbor, height can only decrease or stay the same. So the optimal root is somewhere in the 'middle' — the center of the diameter."

Step 4: Peeling Insight

"I can find the center iteratively: peel all leaves, expose new leaves, repeat. This is Kahn's topological sort on an undirected tree. I stop when ≤ 2 nodes remain. O(n) time."

Step 5: Boundary Condition

"Why stop at 2? A tree's diameter center is 1 or 2 nodes (for odd or even diameter). Can never be 3. So stopping at remaining > 2 covers both cases."

🗣️ What to Say to the Interviewer

"I notice this is a tree — n nodes, n-1 undirected edges, connected.
The brute force is: for each node, BFS to find tree height as root. O(n²).
But I can observe: leaves are never optimal roots for n > 2 — rooting there is always worst.
So I iteratively peel leaves (degree-1 nodes), exposing new leaves each round.
This is essentially Kahn's algorithm adapted for trees.
I stop when ≤ 2 nodes remain — those are the center node(s). At most 2 can exist.
Time O(n), Space O(n)."
🔮 Likely Follow-up Questions
  • "Can there be 3 or more centers?" → No. Proof: centers lie on the diameter (a path), which has at most 2 midpoints.
  • "How would you find the tree diameter?" → Two-BFS: BFS from any node to find farthest node u; BFS from u to find farthest node v; length = dist(u,v).
  • "What if the graph has cycles (not a tree)?" → This algorithm fails. It specifically exploits the tree property that each connected component eventually reduces to a center.
  • "Why adjacency sets instead of lists?" → discard() is O(1) for sets vs O(degree) for lists; both are fine here since total edge removals = O(n).

7 Summary & Key Takeaways ▾
A tree has exactly 1 or 2 centers — the midpoint(s) of its longest path (diameter)
Leaves (degree-1 nodes) are never optimal roots — peel them iteratively from outside in
Stop peeling when ≤ 2 nodes remain — they are the answer
This is Kahn's topological sort applied to undirected trees (degree-1 replaces in-degree-0)
Time O(n), Space O(n) — each node and edge processed exactly once
