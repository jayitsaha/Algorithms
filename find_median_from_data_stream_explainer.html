LeetCode #295

Find Median from Data Stream

Design a data structure that supports adding integers one by one and answering "what is the current median?" after each addition — in O(log n) per add and O(1) per query.

Hard Heaps Two Heaps addNum: O(log n) findMedian: O(1) Space: O(n)
1 Understanding the Problem ▾

The median of a sorted dataset is the middle element (when count is odd) or the average of the two middle elements (when count is even). The twist here is that numbers arrive one-by-one in a stream — you don't have them all upfront, and you must be able to answer a median query at any point.

💡 Key Insight

The median is the boundary between the lower half and upper half of all numbers seen. If you could efficiently access the largest element of the lower half and the smallest element of the upper half, you could answer median in O(1). A max-heap gives you the largest of a set in O(1). A min-heap gives you the smallest. So: one max-heap holds the lower half, one min-heap holds the upper half, and the median is at their tops.

Odd count: stream [1, 3, 5]

Left max-heap: [3, 1] top=3
Right min-heap: [5] top=5
Left has 1 more → median = 3.0

Even count: stream [1, 2, 3, 4]

Left max-heap: [2, 1] top=2
Right min-heap: [3, 4] top=3
Equal sizes → median = (2+3)/2 = 2.5

Edge cases to consider: single element (return it directly), two elements (average both), all identical elements, negative numbers, INT_MIN/MAX values, queries interleaved with adds.

⚠️ Why not sort on every query?

Sorting all n elements per query costs O(n log n) per call. If you maintain a sorted array and insert in place, each insert is O(n) to shift elements. Neither approach scales to millions of insertions with frequent median queries. The heap approach gives O(log n) insert and O(1) query — the best possible for a general data stream.

2 The Algorithm: Two Heaps — Max Left, Min Right ▾

We maintain two heaps that partition all seen elements into a lower half and an upper half. Two invariants must always hold:

I Order invariant: Every element in the left (max-heap) ≤ every element in the right (min-heap). The left's top is the "largest small number" and the right's top is the "smallest large number." Together they bracket the median.
II Balance invariant: The left heap may have at most 1 more element than the right: |len(left) − len(right)| ≤ 1, and we allow left to be the bigger one. This makes median extraction simple: if equal sizes, average both tops; if left is bigger, its top is the median.

addNum protocol — 3 steps:

1 Always push to the left heap first. In Python, simulate the max-heap by pushing the negative: heappush(left, -num).
2 Fix the order invariant. Pop left's max and push it to right: heappush(right, -heappop(left)). This ensures left's maximum is never larger than right's minimum — even if the new element was very large and "shouldn't" be in the left half.
3 Fix the balance invariant. If right now has more elements than left (because of step 2), move right's minimum back to left: heappush(left, -heappop(right)). After this, left has at most 1 more element than right.
🔑 Python Max-Heap via Negation

Python's heapq module only provides a min-heap. To use it as a max-heap, store the negatives of all values. heappush(left, -5) inserts 5. -left[0] gives you the maximum (negating the stored minimum). Always remember to negate when pushing to, and when reading from, the left heap.

3 Why It Works: The Invariant ▾
🔐 The Two-Heap Invariant

After each addNum call, the following is guaranteed:
(1) All elements in left ≤ all elements in right.
(2) len(left) == len(right) or len(left) == len(right) + 1.

These two invariants together mean: if you were to merge both heaps and sort them, the median would be exactly at position len(left)-1 (0-indexed) of the combined sorted array, which is always left[0] (when odd) or the average of left[0] and right[0] (when even).

Why pushing to left first, then cross-pushing, maintains the order invariant: No matter what value num is, after pushing to left and then unconditionally moving left's maximum to right, the new top of right is the largest value among all elements seen, and left contains everything smaller. The cross-push is the key: it doesn't just move the new element — it moves whatever ended up as left's maximum after the push, which might be an older element if num was small.

Why the balance step after cross-push: The cross-push always moves one element from left to right. Before the operation, left might have had 1 extra (odd total). After: right now has 1 extra (even total but swapped). The balance step moves right's min back to left, restoring the invariant that left can be the bigger heap.

⚠️ The "Unconditional Cross-Push" Is the Trick

Some implementations try to conditionally push to left or right based on the new number's value relative to heap tops. This creates bugs: you still need to check and fix both invariants. The unconditional approach (always push to left, always cross-push to right, then rebalance) is simpler, provably correct, and does at most 3 heap operations per addNum.

4 The Code (Annotated) ▾

Solution 1 — Two Heaps (Optimal — Interview Pick)

import heapq
class MedianFinder:
def __init__(self): ← Initialize both heaps as empty lists
self.left = [] # max-heap ← Stores negatives; left[0] = -max
self.right = [] # min-heap ← Stores positives; right[0] = min
def addNum(self, num: int) -> None:
heapq.heappush(self.left, -num) ← Step 1: always push to left (as negative)
heapq.heappush(self.right, ← Step 2: always cross-push left's max to right
-heapq.heappop(self.left)) ← Pop left's top, negate back to positive
if len(self.right) > len(self.left): ← Step 3: if right grew too big...
heapq.heappush(self.left, ← ...move right's min back to left
-heapq.heappop(self.right)) ← Negate to store as negative in left
def findMedian(self) -> float:
if len(self.left) > len(self.right): ← Odd total: left has 1 extra → it's the median
return float(-self.left[0]) ← Negate stored negative to get true max
return (-self.left[0] + self.right[0]) / 2.0 ← Even total: average both tops
⚠️ The Negation Bug — #1 Mistake

Every interaction with self.left requires negation: push -num, read -self.left[0], pop with -heappop(self.left). Forgetting any single negation corrupts the max-heap ordering. When in doubt: "store negative, read negative" — the heap holds negatives, so negate on the way in AND on the way out.

Solution 2 — Python SortedList (cleaner but library-dependent)

from sortedcontainers import SortedList
class MedianFinder:
def __init__(self): self.data = SortedList() ← Maintains sorted order automatically
def addNum(self, num): self.data.add(num) ← O(log n) insertion into sorted structure
def findMedian(self): ← O(1) index access in sorted list
n = len(self.data); mid = n // 2
if n % 2: return float(self.data[mid]) ← Odd: middle element directly
return (self.data[mid-1] + self.data[mid]) / 2.0 ← Even: average two midpoints
💡 Complexity Summary

Two Heaps (optimal): addNum O(log n) — 3 heap ops each O(log n). findMedian O(1) — peek both tops. Space O(n).
SortedList: addNum O(log n) with better constants (C implementation). findMedian O(1). Not available in interviews without imports.
Naive approach: Re-sort on each add O(n log n) or shift on insert O(n) — neither acceptable for a stream.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Pattern

Three signals trigger Two Heaps: (1) "data stream" — you're processing numbers online, not all at once. (2) "median" — you need the middle of a sorted collection. (3) "efficient per operation" — sorting the whole thing after each add is too slow. When you hear these three together, the answer is almost always Two Heaps.

Step 2: Build the Intuition Aloud

"The median is at the boundary between smaller and larger halves. If I split all elements into a lower half and an upper half, the median is either the top of the lower half (odd count) or the average of both tops (even count). A max-heap gives me the lower half's top in O(1). A min-heap gives me the upper half's top in O(1)."

Step 3: State the Invariants Explicitly

"I need two invariants: every element in the left heap must be ≤ every element in the right heap (order), and their sizes differ by at most 1 with left allowed to be bigger (balance). My addNum protocol restores both invariants in O(log n)."

Step 4: Address the Python Trick Proactively

"Since Python's heapq is a min-heap, I'll simulate the max-heap for the left side by storing negatives. Every push to left negates the value; every read from left negates again to get the real number."

🗣️ What to Say to the Interviewer

"I'll use two heaps: a max-heap 'left' for the smaller half, a min-heap 'right' for the larger half.
Invariants: left's max ≤ right's min, and left can have at most 1 more element than right.
addNum: (1) push to left, (2) always move left's max to right, (3) if right grew bigger, move right's min back to left.
findMedian: if sizes equal, average both tops. If left bigger, return left's top.
Python: heapq is min-heap only, so I store negatives in left — push -num, read -left[0].
Time: O(log n) per add (3 heap ops), O(1) per findMedian. Space: O(n)."
🔮 Likely Follow-up Questions
  • "Sliding Window Median?" → Same two-heap idea but with lazy deletion (#480) — much harder
  • "What if numbers are in range [0, 100]?" → Counting sort / bucket approach: O(1) insert, O(100) = O(1) median
  • "Can you do O(1) insert?" → Not possible for a fully general data stream — O(log n) is optimal
  • "What about the kth largest element?" → Just a single min-heap of size k (#703, #215)

7 Summary & Key Takeaways ▾
The median is at the boundary between the sorted lower half and upper half
Two heaps (max-left, min-right) maintain that boundary efficiently
Always push to left first, then unconditionally cross-push to right — this is simpler and bug-free
Rebalance if right grows bigger than left — at most one extra heap op
Python: store negatives in left heap to simulate max-heap; negate when reading back
O(log n) per add (≤ 3 heap operations), O(1) per findMedian (just peek two tops)

  • Scheduling problems where you need both smallest and largest active tasks → Two heaps
