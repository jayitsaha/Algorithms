LeetCode #81

Search in Rotated Sorted Array II

Given a rotated sorted array that may contain duplicates and a target value, return true if target exists in the array.

Medium Binary Search Handle Duplicates Time: O(n) worst Space: O(1)
1 Understanding the Problem ▾

Imagine a sorted array — say [0, 1, 1, 2, 3, 5, 6] — that was picked up at some secret pivot point and wrapped around. The result might be [3, 5, 6, 0, 1, 1, 2]. This is a rotated sorted array. Your task: determine if a target value exists. The twist vs. the classic variant (#33): the array can contain duplicate values, which introduces an ambiguity case that prevents us from always guaranteeing O(log n).

💡 Key Insight

In any binary search window on a rotated sorted array, at least one half is always fully sorted — unless the left endpoint, midpoint, and right endpoint are all equal (the duplicate ambiguity). When we can identify the sorted half, we ask: "does the target belong in that sorted range?" If yes, search there. If no, search the other half. When ambiguous, safely shrink both ends by 1.

✓ Problem #33 — No Duplicates (always O(log n))

nums = [4, 5, 6, 7, 0, 1, 2], target = 0
mid = 3, nums[mid] = 7
nums[lo]=4 <= nums[mid]=7 → left sorted
target=0 NOT in [4,7] → go right
Confidently halved search space

✗ Problem #81 — Duplicates (can hit O(n))

nums = [1, 1, 1, 1, 3, 1], target = 3
mid = 2, nums[mid] = 1
nums[lo]=1 == nums[mid]=1 == nums[hi]=1
Can't tell which half has the 3!
Must shrink: lo++, hi-- (O(n) worst)

Key Edge Cases: empty array → False; all identical values with target absent → O(n) degenerate; target at the rotation boundary; single element equal to target; all elements are duplicates of a value that isn't the target.

2 The Algorithm: Modified Binary Search with Duplicate Handling ▾

This is problem #33 (rotated sorted array search) with one additional case. The full approach: maintain a lo and hi pointer. At each iteration, compute mid. Then branch on three cases, in priority order.

1 Found check: If nums[mid] == target, return True immediately — no further work needed.
2 Duplicate ambiguity check (NEW): If nums[lo] == nums[mid] == nums[hi], both halves could contain the target. Since we know nums[mid] ≠ target, the specific values at lo and hi are also not the target (they equal nums[mid]). Shrink safely: lo += 1; hi -= 1.
3 Left half sorted: If nums[lo] ≤ nums[mid], the left half [lo, mid] is a clean sorted segment. If target lies in [nums[lo], nums[mid]), move hi = mid - 1. Otherwise, the target must be in the right half: lo = mid + 1.
4 Right half sorted (else): The right half [mid, hi] is a clean sorted segment. If target lies in (nums[mid], nums[hi]], move lo = mid + 1. Otherwise, hi = mid - 1.
5 Exhausted window: If lo > hi and we never returned True, return False.

🔑 Why Exactly One Half Is Always Sorted (when not ambiguous)

A sorted array rotated once has exactly one "break point" where values jump down. In any window, mid either lands in the first sorted segment or the second. If nums[lo] ≤ nums[mid], no break occurred in the left half — it's cleanly sorted. If that condition fails, the break is in the left half, meaning the right half is clean. There is no third possibility.

3 Why It Works: The Invariant ▾
🔐 The Loop Invariant

At the start of every iteration: if target exists in the original array, it must be located at some index within [lo, hi]. We never discard an index that might hold the target. Every step either finds the target or provably eliminates indices that cannot contain it.

Why the "left sorted" branch is correct: When nums[lo] ≤ nums[mid] and the range [nums[lo], nums[mid]) does not contain the target, every value in nums[lo..mid] is strictly in that range (the left half is sorted, so all values are between its endpoints). Since target is not in that range, target cannot be in nums[lo..mid]. Eliminating it is safe.

Why the duplicate-shrink is safe: We enter this branch only when nums[lo] == nums[mid] == nums[hi]. We already confirmed nums[mid] ≠ target. Therefore nums[lo] ≠ target and nums[hi] ≠ target (they are both equal to nums[mid]). Eliminating exactly those two indices loses nothing.

⚠️ The Unavoidable O(n) Worst Case

Consider nums = [1, 1, 1, 1, 1, 1], target = 2. At every step, nums[lo] == nums[mid] == nums[hi] == 1. Each iteration shrinks the window by exactly 2 (one from each end). After n/2 iterations the window is empty and we return False. This is genuinely O(n). You cannot avoid this without additional information about the input — there is no O(log n) algorithm for this problem in the worst case.

Termination: Each iteration either returns True or reduces the size of [lo, hi] by at least 1 (the shrink removes 2, the binary-search branches remove roughly half). The window size is always finite and strictly decreasing, so the loop always terminates.

4 The Code (Annotated) ▾

Solution 1 — Modified Binary Search (Interview Pick)

def search(nums: list[int], target: int) -> bool:
lo, hi = 0, len(nums) - 1 ← Standard binary search window
while lo <= hi: ← Valid window: at least one element remains
mid = (lo + hi) // 2 ← Floor-division avoids overflow (same as (lo+hi)/2 in Python)
if nums[mid] == target: ← Direct hit — return immediately
return True
# ── THE KEY NEW CASE: duplicate ambiguity ──
if nums[lo] == nums[mid] == nums[hi]:← All three endpoints equal — can't determine sorted half
lo += 1 ← Safe: nums[lo] == nums[mid] != target
hi -= 1 ← Safe: nums[hi] == nums[mid] != target
elif nums[lo] <= nums[mid]: ← Left half [lo..mid] is a sorted segment
if nums[lo] <= target < nums[mid]: ← Target in left sorted range?
hi = mid - 1 ← Search left
else: ← Target not in left range, must be right
lo = mid + 1 ← Search right
else: ← Right half [mid..hi] is a sorted segment
if nums[mid] < target <= nums[hi]: ← Target in right sorted range?
lo = mid + 1 ← Search right
else: ← Target not in right range, must be left
hi = mid - 1 ← Search left
return False ← Window exhausted without finding target
⚠️ Critical: Use <= not < for the sorted-half check

The condition elif nums[lo] <= nums[mid] must use <= (not strict <). When the window has exactly 2 elements (lo = mid), nums[lo] == nums[mid], and we must still treat the left half as "sorted." Using strict < would send a 2-element window into the else branch incorrectly when the left element is smaller.

Solution 2 — Linear Scan (Brute Force, O(n) always)

def search(nums: list[int], target: int) -> bool:
return target in nums ← Python's 'in' operator is a linear scan: O(n) always
💡 Complexity Summary

Modified Binary Search: Time O(log n) average / O(n) worst; Space O(1)
Linear Scan: Time O(n) always; Space O(1)
The binary search is always preferred in interviews — it shows you understand the structure of the problem.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Connect to the easier variant (#33)

"This is #33 — Search in Rotated Sorted Array — but with duplicates. I'll start from that solution and reason about what changes. In #33, we could always determine which half is sorted. The only new issue is when duplicates make all three endpoints equal."

Step 2: Walk through the duplicate-ambiguity case explicitly

"When nums[lo] == nums[mid] == nums[hi], I cannot tell which side the rotation happened on. For example, in [1,1,2,1,1], mid is 1, lo is 1, hi is 1 — the '2' could be on either side. Since nums[mid] ≠ target, nums[lo] and nums[hi] are also not target. I can safely shrink both ends."

Step 3: Proactively mention the worst-case complexity

"The worst case is O(n) — for example, [1,1,1,1,1] target=2. Every iteration we can only remove 2 elements. I can't avoid this — it's provably impossible to guarantee O(log n) when duplicates are allowed. Average case is still O(log n)."

Step 4: Dry-run a duplicate-heavy example

"I'd trace through [1,3,1,1,1], target=3. Iteration 1: mid=2, nums[mid]=1≠3; nums[lo]=1==nums[mid]==nums[hi]=1 → shrink. Iteration 2: lo=1, hi=3, mid=2, nums[mid]=1. nums[lo]=3 > nums[mid]=1 → right half [1,1,1] is sorted; 3 not in that range → hi=mid-1=1. Iteration 3: lo=hi=mid=1, nums[1]=3=target → True."

🗣️ What to Say to the Interviewer

"I'll use modified binary search. This is #33 plus one extra case for duplicates.
Each iteration: if nums[mid]==target, done. Then check: if all three — nums[lo],
nums[mid], nums[hi] — are equal, I can't determine which half is sorted, so I
shrink both endpoints by 1 (safe, since they equal nums[mid] which isn't target).
Otherwise, I identify the sorted half (nums[lo]<=nums[mid] → left sorted) and
check if target lies in that half's range. Average O(log n), worst O(n) with
heavy duplicates — that's unavoidable for this problem."
🔮 Likely Follow-up Questions
  • "What if we need the index?" → Same logic, return mid instead of True
  • "Can we always guarantee O(log n) with duplicates?" → No, proved impossible
  • "How do you find the rotation point?" → Binary search for min element (#153 / #154)
  • "What if the array was sorted and rotated multiple times?" → Same algorithm still works

7 Summary & Key Takeaways ▾
A rotated sorted array always has at least one sorted half in any window — unless all three endpoints are equal (the duplicate ambiguity)
The duplicate ambiguity is resolved with a safe 1-element shrink from both ends — only possible because those endpoints already equal nums[mid] which ≠ target
O(n) worst case is provably unavoidable with duplicates; average case remains O(log n)
The sorted-half check must use <= not < to correctly handle 2-element windows where lo == mid
