LeetCode #1248

Count Number of Nice Subarrays

Given an array of integers and an integer k, count subarrays that contain exactly k odd numbers.

Medium Sliding Window AtMost(K) − AtMost(K−1) Time: O(n) Space: O(1)
1 Understanding the Problem ▾

Given an integer array nums and an integer k, return the number of "nice" subarrays — contiguous subarrays that contain exactly k odd numbers.

A number is odd if num % 2 == 1. Even numbers are "transparent" — they extend a window without changing its odd count.

💡 Key Insight

Counting subarrays with exactly k odds is hard to do directly with one sliding window because there is no monotone "shrink" condition. The trick: exactly(k) = atMost(k) − atMost(k−1). Each "at most" query is an easy variable-window problem.

Example: nums=[1,1,2,1,1], k=2

[1,1] → 2 odds ✓
[1,1,2] → 2 odds ✓
[1,2,1] → 2 odds ✓
[2,1,1] → 2 odds ✓
Answer: 4

Why "exactly k" is tricky

Standard window: shrink when invalid.
"At most k" → shrink when odds>k ✓
"Exactly k" → shrink when odds>k?
But also skip when odds<k!
No single monotone condition.

Edge cases: k=0 (only all-even subarrays), all elements odd, all elements even, k greater than total odd count in array (answer = 0), single element array.

2 The Algorithm: AtMost(K) − AtMost(K−1) ▾

We decompose the "exactly k" problem into two "at most k" problems. Each "at most" sub-problem is solved with a standard variable sliding window in O(n) time.

atMost(k)
subarrays with ≤k odds
−
atMost(k−1)
subarrays with ≤k−1 odds

The subtraction cancels subarrays with 0 to k−1 odds, leaving only those with exactly k.

How atMost(goal) works internally: Maintain a window [left, right] with at most goal odd numbers. Expand right one step at a time. If odd count exceeds goal, advance left until the window is valid again. At each right, the valid subarrays ending there count as right − left + 1.

1 Define atMost(goal): Returns the count of subarrays with at most goal odd numbers. If goal < 0, return 0 immediately (needed for k=0 edge case).
2 Expand right: Add nums[right] % 2 to odds. This is 1 for odd numbers and 0 for even — no if-statement needed.
3 Shrink from left while invalid: While odds > goal, subtract nums[left] % 2 and advance left. After the loop, [left, right] is valid.
4 Count valid subarrays ending at right: Add right − left + 1 to result. This counts subarrays with left end anywhere in [left, right].
5 Return atMost(k) − atMost(k−1): The difference isolates exactly k odds by set subtraction.
🔑 Why right − left + 1 counts all valid subarrays at once

After shrinking, every left endpoint from index left to right gives a valid subarray ending at right. That's right − left + 1 subarrays. Each unique subarray gets counted exactly once — at its rightmost element. So summing across all right positions gives the exact total.

3 Why It Works: The Invariant ▾
🔐 The Invariant

After each inner while-loop iteration: nums[left..right] is the shortest valid window ending at right with at most goal odd numbers. Any window starting at left or later and ending at right is guaranteed to have ≤ goal odds. There are exactly right − left + 1 such windows.

Set-theoretic proof of the subtraction: Let A = all subarrays with at most k odds, B = all subarrays with at most k−1 odds. Clearly B ⊆ A. The subarrays with exactly k odds = A \ B. So |exactly k| = |A| − |B| = atMost(k) − atMost(k−1). This works because each subarray is counted once in each atMost call.

⚠️ Why you cannot skip the atMost(k−1) call

You might think: "just count subarrays where odds == k during the single window pass." But the sliding window's left pointer may have jumped past multiple starting points. You cannot retroactively distinguish between exactly-k and less-than-k subarrays within a single window pass. The two-call approach avoids this entirely.

k = 0 edge case: atMost(0) counts subarrays with zero odds (all-even runs). atMost(−1) must return 0. The guard if goal < 0: return 0 is required — without it, the while loop would never exit because negative values cannot satisfy odds > goal sensibly.

4 The Code (Annotated) ▾

Solution 1 — AtMost Sliding Window (Optimal, Interview Pick)

def numberOfSubarrays(nums, k):
def atMost(goal): ← helper: subarrays with ≤ goal odds
if goal < 0: return 0 ← guard: needed when k=0, atMost(-1)→0
res = left = odds = 0 ← result, left pointer, window's odd count
for right, num in enumerate(nums): ← right expands window one step at a time
odds += num % 2 ← 1 if odd, 0 if even — elegant tally
while odds > goal: ← window is invalid (too many odds)
odds -= nums[left] % 2 ← remove left element's contribution
left += 1 ← shrink window from the left
res += right - left + 1 ← all sub-windows ending at right are valid
return res
return atMost(k) - atMost(k - 1) ← exactly k = difference
⚠️ Using num % 2 vs checking if num is odd

Both num % 2 and num % 2 != 0 work, but odds += num % 2 is cleaner — it adds 1 for odd and 0 for even without branching. Same trick for the remove step: odds -= nums[left] % 2. Python's % always returns non-negative, so this is safe for positive integers.

Solution 2 — Prefix Sum (Alternative O(n) approach)

def numberOfSubarrays(nums, k):
count = {0: 1} ← prefix sum 0 occurs once (empty prefix)
prefix = res = 0
for num in nums: ← running count of odds from index 0
prefix += num % 2 ← prefix = # odds in nums[0..i]
res += count.get(prefix - k, 0) ← how many starts give exactly k odds here
count[prefix] = count.get(prefix, 0) + 1 ← record this prefix for future lookups
return res

The prefix sum approach is the sibling of "Subarray Sum Equals K" (#560) — replacing value sums with odd counts. It uses O(n) space for the hash map versus O(1) for the sliding window approach.

💡 Complexity Summary

Brute Force: Time O(n²), Space O(1) — enumerate all subarrays
Prefix Sum: Time O(n), Space O(n) — hash map of prefix counts
AtMost Sliding Window: Time O(n), Space O(1) — interview pick, cleanest for generalisation

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the "Exact Count in Subarray" Signal

"Contiguous subarray with exactly k [property]" is a pattern. The naive O(n²) approach enumerates all pairs. I need O(n). The property here — being odd — is simple to test. This suggests sliding window or prefix sum.

Step 2: Try Sliding Window — Spot the Obstacle

"Standard sliding window: expand right when valid, shrink left when invalid. But 'exactly k odds' violates monotonicity — I need to shrink for too many AND skip for too few. The window can't encode both constraints simultaneously."

Step 3: Reframe with AtMost

"'At most k odds' IS monotone — more elements, more opportunities to have odds. I can shrink when odds > k. So I define atMost(k), then use the identity: exactly(k) = atMost(k) − atMost(k−1). Two O(n) passes = O(n) total."

Step 4: Implement and Handle Edge Cases

"The template is clean. One subtlety: for k=0, I call atMost(−1). Guard it with 'if goal < 0: return 0'. Also, nums[right] % 2 cleanly adds 0 for even and 1 for odd — no if needed."

🗣️ What to Say to the Interviewer

"I'll use the AtMost(k) − AtMost(k−1) technique.
'Exactly k odds' has no monotone shrink condition for a sliding window.
But 'at most k odds' does — I shrink left while odds > k.
At each right, right−left+1 subarrays end there with ≤k odds.
The difference gives me exactly k. Time O(n), Space O(1).
Edge case: atMost(−1) must return 0, which I guard with an early check."
🔮 Likely Follow-up Questions
  • "Same problem but with at least k odds?" → n*(n+1)/2 total subarrays − atMost(k−1).
  • "Subarrays with exactly k distinct integers?" → Identical template! Problem #992.
  • "Can you solve it with prefix sums?" → Yes, analogous to Subarray Sum Equals K (#560).
  • "Generalise: any monotone property X instead of 'odd'?" → atMost works whenever adding elements only increases X.

7 Summary & Key Takeaways ▾
exactly(k) = atMost(k) − atMost(k−1) is the universal identity for "exact count in subarray" problems
atMost is monotone: window grows → odd count non-decreasing, so left-shrink is always well-defined
At each right, right−left+1 subarrays end there and have ≤ goal odds — counted in O(1) per position
Use num % 2 as a zero-branch indicator: 1 for odd, 0 for even — clean and fast
Guard atMost(goal < 0) → return 0 to handle the k=0 edge case without special logic
