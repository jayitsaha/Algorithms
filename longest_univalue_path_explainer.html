LeetCode #687

Longest Univalue Path

Given the root of a binary tree, return the length of the longest path where every node has the same value. The path does not need to pass through the root. Length is measured in edges.

Medium Trees DFS Postorder Return Depth · Track Max Time: O(n) Space: O(h)
1 Understanding the Problem ▾

We walk a binary tree looking for the longest chain of edges where every node on the chain carries the same value. A chain through 3 same-valued nodes contains 2 edges — length is counted in edges, not nodes. The path can run through any node in the tree, including deep interior nodes; it does not have to start at the root.

💡 Key Insight

The longest univalue path through a node N equals the sum of how far it can extend left plus how far it can extend right — but only counting directions where the immediate child matches N's value. We compute this sum locally at N and track the best globally, while returning only the single best arm upward for the parent to use.

The two-value distinction: At every node we compute two different numbers. The arm (returned upward) is the best single-direction extension the parent could use. The width (tracked globally) is left_arm + right_arm, the full span across this node. These must not be confused.

Edge cases: Single node → 0 (no edges). All different values → 0. Entire tree is same value → h-1 (longest root-to-leaf path is longest, but a "zig-zag" through two subtrees might be longer). Null root → 0.

2 The Algorithm: DFS Postorder — Return Arm, Track Max ▾

We use postorder DFS: process both children before computing the current node's contribution. A helper function dfs(node) returns the arm length — how far the univalue chain extends downward from this node in the best single direction. A nonlocal variable captures the global best.

1 Base case — null node: Return 0. A null child contributes nothing; any chain reaching here terminates.
2 Recurse left and right: Get l = dfs(node.left) and r = dfs(node.right). These give the arm lengths coming up from each subtree.
3 Extend if values match: left_arm = (l+1) if node.left and node.left.val == node.val else 0. We add 1 edge only if the child's value matches. A mismatch resets to 0 — the chain breaks.
4 Update global max: ans = max(ans, left_arm + right_arm). The path through this node is the combined span. This might be the answer even if the path never reaches the root.
5 Return the best arm: return max(left_arm, right_arm). The parent can only continue the chain in one direction from this node — it gets the best single arm, not the sum.
🔑 Why This Pattern Is Called "Return Arm, Track Max"

This exact pattern appears in Binary Tree Diameter (#543), Binary Tree Maximum Path Sum (#124), and Longest Zigzag Path (#1372). The shape is always: helper returns the best arm going down; outer variable accumulates the best diameter/span/sum seen at any node. Recognizing this template lets you solve the whole family of "tree path" problems.

3 Why It Works: The Invariant ▾
🔐 The Invariant

dfs(node) always returns the length of the longest univalue chain starting at node and going strictly downward in a single direction. This is the maximum number of edges a parent could gain by extending its own chain through this node.

Why postorder guarantees correctness: We need both children's arm values to compute left_arm and right_arm. Processing children first (postorder) ensures those values are ready when we need them. Any other traversal order would require us to compute values we don't yet have.

Why the global max trick is needed: The optimal path might be entirely inside a subtree, never touching the root. By running ans = max(ans, left_arm + right_arm) at every single node, we consider every possible node as the "turning point" of the longest path. One of those will be the global maximum.

Why we return max, not sum: A tree path is a sequence of nodes with no branching. When the parent asks "how much can I extend in your direction?", it can only use one arm (it has a single edge connecting it to this node). Returning the sum would imply the parent can branch both left and right through this node — creating an invalid fork.

🚨 The Classic Bug

return left_arm + right_arm — wrong! This tells the parent it can fork, which is not a valid path. The parent would then compute its arm as this sum + 1, and subsequent parents keep accumulating, producing completely wrong answers. Always return max(left_arm, right_arm).

4 The Code (Annotated) ▾

Solution 1 — DFS with nonlocal ans (Optimal · O(n) · Interview Pick)

def longestUnivaluePath(root): ← outer function; entry point
ans = 0 ← global best path length (in edges)
def dfs(node): ← helper; returns arm length from node downward
nonlocal ans
if not node: return 0 ← null child → arm of 0; chain ends
l = dfs(node.left) ← recurse left (postorder: children first)
r = dfs(node.right) ← recurse right
left_arm = (l+1) if node.left and node.left.val == node.val else 0 ← +1 edge only if child value matches
right_arm = (r+1) if node.right and node.right.val == node.val else 0 ← same for right
ans = max(ans, left_arm + right_arm) ← full width through this node; update global
return max(left_arm, right_arm) ← best single arm going up to parent
dfs(root) ← kick off DFS; return value is discarded
return ans ← maximum across all nodes
⚠️ Why guard with node.left and ...?

If node.left is None, accessing node.left.val raises AttributeError. The and short-circuits: if node.left is falsy, Python never evaluates node.left.val. Result: null child always gives arm = 0, which is correct.

Solution 2 — Brute Force (O(n²) — understand why it's slower)

def longestUnivaluePath(root): ← two separate passes per node → O(n²)
def arm(node, val): ← how far can we go from node matching val?
if not node or node.val != val: return 0
return 1 + max(arm(node.left, val), arm(node.right, val))
def solve(node): ← check every node as a potential path center
if not node: return 0
here = arm(node.left, node.val) + arm(node.right, node.val)
return max(here, solve(node.left), solve(node.right))
return solve(root) ← arm() re-runs for every node → quadratic
💡 Why O(n²) — and how the optimal fixes it

In the brute force, arm() is called fresh for every node in solve(). The subtree under node3 gets traversed when node3 is the center, again when node2 is the center, again when the root is the center — redundant work. The optimal DFS shares results: each node's arm is computed exactly once and reused by its parent. Classic memoization via the call stack.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify the Pattern

"This is a tree path problem where the answer can turn at an interior node. That signals the diameter-style DFS: helper returns arm lengths, outer variable tracks global max. I've seen the same shape in Binary Tree Diameter (#543)."

Step 2: Define What the Helper Returns

"My helper dfs(node) returns: the length of the longest univalue chain that starts at node and goes strictly downward in one direction. This is what the parent needs to decide whether to extend."

Step 3: Handle the Value Check

"After I get the left and right arms, I only extend them if the child's value matches the current node's value. Mismatch = arm resets to 0. This is the univalue constraint — the chain breaks on a value difference."

Step 4: Update Global, Return Local

"Update ans with left_arm + right_arm — the full width through this node. But return max(left_arm, right_arm) to the parent — it can only extend in one direction. These two values serve different purposes."

🗣️ What to Say to the Interviewer

"This resembles Binary Tree Diameter, but with a value equality constraint.
I'll use DFS postorder — process children before parent.

At each node: get l and r from children, then extend only if values match.
Update global ans with left_arm + right_arm (the full span).

Time: O(n), Space: O(h) for the call stack."
🔮 Likely Follow-up Questions
  • "Count nodes instead of edges?" → Return max(la, ra) + 1; update ans with la + ra + 1. Adjust base case to return 1 for leaf.
  • "General k-ary tree instead of binary?" → Iterate all children; sum all matching arms; return single best arm.
  • "What if values need to be strictly increasing along the path?" → Pass current value and direction downward; extend only if child is val+1.
  • "Can you avoid nonlocal?" → Use a class attribute self.ans, or a mutable list ans = [0].

7 Summary & Key Takeaways ▾
The DFS helper returns the arm length (one direction only) — NOT the combined path width through a node.
Update ans = max(ans, left_arm + right_arm) at every node — this captures paths that bend through interior nodes.
Value equality is checked after recursing into children — we always recurse, but only extend the arm when values match.
Postorder DFS is mandatory — children's return values must be known before the parent can compute its arms.
Length is measured in edges: a same-value chain through 3 nodes = 2 edges, not 3.
The pattern "return arm, track max" is reusable across Binary Tree Diameter, Maximum Path Sum, Longest Zigzag Path, and more.
🎯 Recognize "Return Arm, Track Max" When You See
  • The problem asks for the longest/heaviest path in a tree (not root-to-leaf only)
  • The path might "turn" — pass through both left and right subtrees of some node
  • The answer could be inside any subtree, not necessarily including the root
  • You need to report something different to the parent vs what you track globally
