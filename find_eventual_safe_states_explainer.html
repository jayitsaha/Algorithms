LeetCode #802

Find Eventual Safe States

Given a directed graph, return all safe nodes — nodes from which every path eventually leads to a terminal node with no outgoing edges.

Medium Graph Reverse Graph Topo Sort Time: O(V+E) Space: O(V+E)
1 Understanding the Problem ▾

A directed graph has n nodes (0 to n−1). graph[i] lists all nodes that node i has a directed edge to. A node is a terminal node if it has no outgoing edges (graph[i] = []). A node is a safe node if every possible path starting from that node eventually reaches a terminal node.

💡 Key Insight

A node is unsafe if and only if some path from it enters a cycle — because a cycle means you can loop forever and never reach a terminal. Conversely, a node is safe if all its outgoing paths lead to safe nodes (or are already terminal). This lets us propagate safety backwards from terminal nodes.

✗ Unsafe: in or leading to a cycle

0 → [1, 2]
1 → [2, 3]
3 → [0] ← cycle: 0→1→3→0
Nodes 0, 1, 3 are unsafe

✓ Safe: all paths reach a terminal

2 → [5] 5 → [] (terminal)
4 → [5] 6 → [] (terminal)
Nodes 2, 4, 5, 6 are safe
Return: [2, 4, 5, 6] sorted

Edge Cases: All nodes form one big cycle → return []. No edges at all → all nodes are terminal → return all nodes sorted. A node pointing only to safe nodes is itself safe — safety propagates backward one hop at a time.

🔑 The Reformulation

Instead of "does every path from X terminate?" — ask "is X on or connected to a cycle?" Safe = not part of, and not leading into, any cycle. This reformulation enables an elegant reverse-graph approach: start from confirmed safe nodes (terminals), and propagate safety backward through the graph.

2 The Algorithm: Reverse Graph + Kahn's BFS ▾

We reverse all edges in the graph, then run a topological-sort-style BFS (Kahn's algorithm) starting from terminal nodes as seeds. In the reversed graph, terminal nodes have no incoming edges — they are the natural starting points for propagating safety outward.

1 Build reverse graph + track out-degree: For each edge u → v in the original, add v → u in rev. Compute out_deg[u] = number of original outgoing edges from u.
2 Seed the BFS queue: Push all nodes where out_deg == 0. These terminal nodes have no exits — they are trivially safe. Nothing can "go wrong" from a dead end.
3 BFS propagation via reverse edges: Pop a safe node v. In the reverse graph, look at its neighbors u (original edges u → v). Decrement out_deg[u]. If out_deg[u] == 0, every path from u now leads to safe nodes → enqueue u.
4 Collect and sort results: All nodes ever added to the queue are safe. Nodes never added (still have out_deg > 0) are in or lead to cycles. Return the safe set sorted.

🔑 Why Reverse the Graph?

In the original graph, safety flows forward — a node is safe if all its successors are safe. To check this efficiently, we reverse direction: "is my predecessor confirmed safe?" becomes a question we can answer by decrementing a counter. When the counter hits zero (all original exits confirmed safe), the node earns its safe status. This is exactly Kahn's algorithm, repurposed for safe-node discovery.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At any point during the BFS, out_deg[u] equals the number of u's original outgoing neighbors that have not yet been confirmed safe. When out_deg[u] hits zero, every exit from u in the original graph leads to a confirmed-safe node — making u safe by definition.

Why cycle nodes never become safe: Consider a cycle A → B → C → A. Initially out_deg[A]=1, out_deg[B]=1, out_deg[C]=1. For A to reach out_deg=0, B must first become safe. For B, C must first be safe. For C, A must first be safe. This mutual dependency is never resolved — none of them ever enter the queue. They are correctly classified as unsafe.

Why the length check on the sorted output doesn't need extra logic: We simply collect all enqueued nodes into a set and sort at the end. Nodes that were never enqueued have out_deg > 0 — they are in cycles. The two sets partition the node space cleanly.

⚠️ Common Mistake: Confusing in-degree vs out-degree

In standard Kahn's topological sort, you track in-degree of the original graph. Here you track out-degree of the original graph (equivalently, in-degree of the reversed graph). Many bugs arise from mixing these up. Always anchor your thinking: out_deg[u] = number of edges u sends in the ORIGINAL graph.

DFS alternative intuition: A 3-color DFS (white=unvisited, gray=in-current-path, black=confirmed-safe) achieves the same result: if DFS from a node ever reaches a gray node, it found a back edge → cycle → unsafe. Both approaches are O(V+E), but BFS avoids recursion depth issues and the subtle 3-state color logic.

4 The Code (Annotated) ▾

Solution 1 — Reverse Graph + Kahn's BFS (Interview Pick)

from collections import deque ← BFS queue for O(1) popleft
def eventualSafeNodes(graph):
n = len(graph)
rev = [[] for _ in range(n)] ← reversed adjacency list
out_deg = [0] * n ← original out-degree per node
for u in range(n):
for v in graph[u]:
rev[v].append(u) ← reverse edge: v → u in rev
out_deg[u] += 1 ← count u's original exits
queue = deque(
i for i in range(n) if out_deg[i]==0) ← terminals: trivially safe
safe = set() ← accumulate safe nodes
while queue:
v = queue.popleft() ← v is confirmed safe
safe.add(v)
for u in rev[v]: ← who originally pointed to v?
out_deg[u] -= 1 ← one less unresolved exit for u
if out_deg[u] == 0: ← all of u's exits are now safe
queue.append(u) ← u is safe — enqueue it
return sorted(safe) ← problem requires sorted output
⚠️ Why not just check in-degree of the reverse graph?

out_deg[u] in the original graph IS the in-degree of u in the reversed graph. Both formulations are equivalent. Tracking out_deg in the original is more intuitive: it directly answers "how many of u's exits haven't been proven safe yet?"

Solution 2 — DFS with 3-Color Marking

def eventualSafeNodes(graph):
n = len(graph)
color = [0] * n # 0=white 1=gray 2=black ← 0:unseen 1:in-stack 2:safe
def dfs(node):
if color[node] == 1: return False ← back edge detected → cycle
if color[node] == 2: return True ← already confirmed safe
color[node] = 1 ← mark "currently on path"
for nei in graph[node]:
if not dfs(nei): return False ← any unsafe successor → unsafe
color[node] = 2 ← all paths safe → mark safe
return True
return [i for i in range(n) if dfs(i)] ← sorted because i=0..n-1
💡 Complexity Summary

Reverse Graph BFS: Time O(V+E) — single pass build + BFS. Space O(V+E) — reverse graph storage.
DFS 3-Color: Time O(V+E). Space O(V) — color array + recursion stack. Risk: stack overflow for deep graphs.
Interview pick: Reverse Graph BFS — iterative, cleaner to debug, no recursion hazard.

5 🔬 Algorithm Deep-Dive: Kahn's Algorithm on a Reversed Graph ▾

Kahn's Algorithm (1962) was originally designed to produce a topological ordering of a Directed Acyclic Graph (DAG). It repeatedly removes source nodes (in-degree 0) and updates neighbors' in-degrees. Any node that never reaches in-degree 0 is in a cycle — this is the key cycle-detection property we exploit.

🔐 Core Invariant (as used here)

We run Kahn's on the reversed graph, but track original out-degree. A node becomes a "source" in our BFS when its original out-degree drops to zero — meaning all original exits are confirmed safe. The algorithm implicitly finds all nodes that can be "peeled off" safely, leaving cycle participants unreachable.

The cycle-detection guarantee: In a cycle A→B→C→A, the circular dependency prevents any of them from reaching out_deg=0. No amount of safe-node propagation can break the cycle's self-referential lock. This mathematical guarantee is what makes Kahn's so powerful for cycle detection.

1 Recognize the pattern: "Nodes from which every path reaches [goal]" → reverse edges, run Kahn's from goal nodes. "Which nodes can reach a sink?" → same structure.
2 Generalize: This pattern appears in "Find all nodes that can only lead to nodes with property P." Reverse + BFS from P-nodes. Works for any monotone property that propagates backward.
3 Cycle detection for free: Nodes Kahn's never processes = nodes whose in-degree (in the BFS graph) stayed > 0 = nodes locked in mutual cycles. Zero extra code needed.
6 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Reframe "safe" as "cycle-free reachability"

Read carefully: "every path eventually reaches a terminal." The only way a path never terminates is if it enters a cycle. So safe = no cycle is reachable from this node. Unsafe = on a cycle or leading into one.

Step 2: Propose DFS brute-force with memoization

"I can DFS from each node, tracking visited nodes in the current path. If I hit a node already in the current path, I found a cycle — unsafe. With 3-color marking, this is O(V+E). But it uses recursion (stack overflow risk) and the 3-state logic is subtle."

Step 3: Insight — flip the direction of propagation

"Terminal nodes are trivially safe. Any node whose every exit leads to safe nodes is also safe. Safety propagates backward. If I reverse the graph, I can BFS outward from terminals, confirming safety one layer at a time."

Step 4: Name the technique — Kahn's on the reversed graph

"This is Kahn's topological sort algorithm, applied to the reversed graph seeded from terminal nodes. I track out-degree in the original graph. When it hits 0 for a node, all its exits are safe — enqueue it. Nodes never enqueued are stuck in cycles."

🗣️ What to Say to the Interviewer

"Safe means every path avoids cycles. So I need to find nodes not in or leading to any cycle.
My approach: build the reverse adjacency list and track original out-degree for each node.
Seed the queue with all terminal nodes — out_deg == 0 in the original graph. These are safe.
BFS: pop a confirmed safe node v. For each u in rev[v] (i.e., original had u→v): decrement out_deg[u].
If out_deg[u] drops to 0, all of u's exits are now safe — enqueue u.
Cycle nodes never get enqueued (mutual dependency keeps out_deg > 0). Collect + sort.
Time O(V+E), Space O(V+E) for the reverse adjacency list."
🔮 Likely Follow-up Questions
  • "Can you do it iteratively AND avoid the reverse graph?" → Yes, iterative DFS with an explicit stack + color array. But reverse-graph BFS is cleaner.
  • "What if there are no terminal nodes?" → Queue starts empty. BFS processes nothing. Return [] — all nodes are in cycles or lead to them.
  • "What if a node has a self-loop?" → out_deg[u] starts ≥ 1 (the self-loop). It can never reach 0 (the self-loop back to u will never be "resolved" by u itself). Correctly unsafe.
  • "Can safe and unsafe nodes be adjacent?" → Yes! Node 2 (safe, 2→5) and node 1 (unsafe, 1→2 AND 1→3) are adjacent. Safety is per-node, not per-edge.

8 Summary & Key Takeaways ▾
Safe = not in or leading into a cycle. Reformulate the problem to make this tractable.
Reverse the graph and run Kahn's BFS from terminal nodes (out_deg=0 seeds).
out_deg[u] tracks how many of u's original exits are unresolved. When it hits 0, u is safe.
Cycle participants never reach out_deg=0 — they self-lock. Zero extra code needed for cycle detection.
DFS 3-color works too but reverse-graph BFS is cleaner and avoids recursion stack overflow.
