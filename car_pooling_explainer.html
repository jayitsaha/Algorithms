LeetCode #1094

Car Pooling

Given trip requests and a vehicle capacity, determine if all trips can be completed without exceeding capacity at any location.

Medium Intervals Difference Array Time: O(n + K) Space: O(K)
1 Understanding the Problem ▾

You drive a single car on a straight road. Each trip is described as [numPassengers, from, to] — passengers board at location from and exit at location to. You have a fixed capacity seat limit. The question: does the on-board passenger count ever exceed capacity at any point?

💡 Key Insight

This is a range update + point query problem. Each trip adds passengers over the interval [from, to). We want to know if any location on that number line has a cumulative passenger count exceeding capacity. Instead of iterating every trip over every location, we can encode just the boundaries and recover actual values with a single prefix-sum sweep.

Think about the road as a number line from 0 to 1000. Each trip is a "load" event that increases occupancy from from and decreases it at to. Multiple trips may overlap at the same locations, stacking their passengers. We need to find the maximum stack height and compare it to capacity.

✓ Feasible: trips=[[2,1,5],[3,3,7]], cap=6

Loc 1-2: 2 passengers (Trip A)
Loc 3-4: 2+3=5 passengers
Loc 5-6: 3 passengers (Trip A exits)
Max=5 ≤ 6 → True

✗ Not Feasible: same trips, cap=4

Loc 1-2: 2 passengers ✓
Loc 3-4: 2+3=5 passengers
5 > 4 = capacity at loc 3
→ False

⚠️ Half-Open Interval: [from, to)

Passengers board at from and exit at to — they are NOT in the car at location to. In the difference array: place +numPass at index from and -numPass at index to (not to+1). Getting this wrong inflates occupancy by 1 extra location per trip.

Edge cases: All trips start at the same location (maximum simultaneous load), capacity of 1 with a 1-passenger trip, non-overlapping serial trips (each is fine independently), and to at the maximum boundary (1000).

2 The Algorithm: Difference Array + Prefix Sum Sweep ▾

The Difference Array technique makes range updates O(1) by recording only where values change (the boundaries), then recovering actual values with one prefix-sum pass. It is the optimal tool when you need many range additions followed by point lookups over a fixed-size coordinate space.

1 Allocate diff[0..1000]: Array of size 1001, all zeros. Locations are guaranteed in [0, 1000] so this covers the full range.
2 Encode each trip as two boundary events: diff[from] += numPass (passengers board here) and diff[to] -= numPass (passengers exit here). Two O(1) operations per trip regardless of how large the interval is.
3 Prefix-sum sweep: Maintain a running total curr = 0. Walk through diff left to right. At each location, curr += diff[loc] updates the occupancy. This correctly accounts for all trips that include this location.
4 Capacity check: If curr > capacity at any step, return False immediately. After the full sweep with no violation, return True.
📐 Core Formulas
For each trip [p, from, to]:
diff[from] += p ← passengers board
diff[to] -= p ← passengers exit

occupancy[loc] = sum(diff[0..loc]) ← prefix sum
🔑 Why Size 1001?

The problem constrains 0 ≤ from < to ≤ 1000. We need indices 0 through 1000 inclusive — that's 1001 positions. The sweep costs O(1001) = O(K) regardless of the number of trips n. When K (coordinate range) is small relative to n, difference array outperforms event-based sorting (O(n log n)).

3 Why It Works: The Invariant ▾
🔐 The Invariant

After processing all trips into the difference array: the prefix sum of diff[0..i] equals the exact number of passengers in the car at location i. This holds because every trip with from ≤ i < to has added its +p (at its start) but not yet subtracted its −p (at its end). Trips that haven't started or have already ended contribute net zero to the prefix sum at i.

Formal proof for one trip T = [p, a, b]:

  • If i < a: neither diff[a] = +p nor diff[b] = -p has been included in prefix sum up to i. Net = 0. (Passenger hasn't boarded.) ✓
  • If a ≤ i < b: diff[a] = +p included, diff[b] = -p not yet. Net = +p. (Passenger is on board.) ✓
  • If i ≥ b: both included. Net = +p + (-p) = 0. (Passenger has exited.) ✓

Summing across all trips: the prefix sum at location i is exactly the count of passengers currently on board. This is the precise quantity we compare against capacity.

⚠️ Alternative: Event Sorting (O(n log n))

You can also create two events per trip — a "board" at from and an "exit" at to — sort all events by location, and sweep through them. At ties (board and exit at same location), process exits first to avoid momentary overcounting. This generalizes to unbounded coordinates but is slower: O(n log n) vs O(n + K).

4 The Code (Annotated) ▾

Solution 1 — Difference Array (Optimal, Interview Pick)

def carPooling(trips, capacity): ← trips=[[numPass,from,to],...]; capacity=int
diff = [0] * 1001 ← range [0,1000]; 1001 slots cover every location
for numPass, start, end in trips: ← unpack each trip — note variable names match problem
diff[start] += numPass ← board event: passengers add to occupancy here
diff[end] -= numPass ← exit event: at 'to', passengers are gone (half-open)
curr = 0 ← running prefix sum = current passengers in car
for passengers in diff: ← sweep all 1001 locations left to right
curr += passengers ← update occupancy at this location
if curr > capacity: ← check AFTER update (occupancy is now accurate here)
return False ← capacity exceeded — infeasible
return True ← swept all locations without overflow — feasible
⚠️ diff[end] not diff[end+1]

Passengers exit at their destination — they are NOT present at location to. Using diff[end] -= numPass correctly removes them at the exit point. If you mistakenly wrote diff[end+1] -= numPass, passengers would still show as on-board at location to, inflating the count and potentially causing false "over capacity" results.

Solution 2 — Event Sorting (O(n log n), works for unbounded locations)

def carPooling(trips, capacity):
events = [] ← collect (location, delta) pairs
for numPass, start, end in trips:
events.append((start, +numPass)) ← board event (positive)
events.append((end, -numPass)) ← exit event (negative, sorts before boards at same loc)
events.sort() ← sort by (location, delta): exits (-) before boards (+)
curr = 0
for loc, delta in events:
curr += delta
if curr > capacity: return False
return True
💡 Complexity Summary

Difference Array: Time O(n + K), Space O(K) — K=1001 constant; best for bounded locations
Event Sorting: Time O(n log n), Space O(n) — general; works for any coordinate range
Brute Force per location: Time O(n × K), Space O(K) — check every trip at every location; too slow

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Interval-Overlap Structure

Trips are intervals on a number line. You're asking: does any point lie under more than capacity total weight? This is classic interval overlap detection. The signal: intervals + cumulative resource check → sweep line or difference array.

Step 2: Propose Brute Force

"For each location 0 to 1000, I can sum passengers from all active trips at that location. If any sum exceeds capacity, return False. This is O(n × K) — correct but slow. Let me optimize."

Step 3: Introduce Difference Array

"Instead of checking every trip at every location, I record only where changes happen. Each trip causes exactly two boundary changes: +numPass at from and -numPass at to. One prefix-sum sweep reconstructs exact occupancy at each location in O(K)."

Step 4: Confirm the Boundary

"Important: passengers exit AT to, so we place the decrement at diff[to], not diff[to+1]. When I apply the prefix sum, I add diff[loc] before checking capacity — so the decrement takes effect right at the destination."

🗣️ What to Say to the Interviewer

"This is an interval overlap problem on a bounded number line [0, 1000].
I'll use a difference array of size 1001: diff[from] += numPass, diff[to] -= numPass for each trip.
Then one prefix-sum sweep: curr += diff[loc], check if curr > capacity at each step.
O(n + K) time, O(K) space where K = 1001 (constant).
Key boundary: passengers exit AT 'to', so the decrement goes at diff[to], not diff[to+1]."
🔮 Likely Follow-up Questions
  • "Locations up to 10^9?" → Coordinate compression + event sorting (O(n log n))
  • "Minimum capacity needed?" → Same diff array; return max(prefix_sums) instead of boolean
  • "Multiple cars?" → Binary search on number of cars + greedy assignment check
  • "Which trips to drop to fit capacity?" → Becomes a different optimization problem

7 Summary & Key Takeaways ▾
Difference array converts any number of range updates into O(1) boundary writes; one prefix-sum sweep recovers exact point values
The half-open interval rule: diff[to] -= numPass (not to+1) — passengers exit at their destination, not after
When coordinates are bounded by constant K, difference array beats event sorting: O(n + K) vs O(n log n)
The same pattern extends to 2D (rectangle range updates → 2D difference array with double prefix sum)
Pattern recognition: intervals + sum at each point + bounded integer coordinates → difference array is your first instinct
