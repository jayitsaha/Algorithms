LeetCode #1110

Delete Nodes And Return Forest

Given a binary tree and a set of node values to delete, remove those nodes and return the roots of all remaining trees in the resulting forest.

Medium Trees Post-order DFS Time: O(n) Space: O(n)
1 Understanding the Problem ▾

You have a binary tree and a list of node values to delete. When you delete a node, its children become orphaned — they may become new roots of their own independent trees. Your task: return all root nodes of the surviving trees after all deletions.

💡 Key Insight

A node becomes a new forest root if and only if it is NOT deleted AND its parent WAS deleted (or it was already the original root with no parent). This parent-deleted signal must flow downward through the recursion — the natural fit is passing an is_root flag alongside each recursive call.

Edge cases to consider:

  • The original root is in the delete list — its surviving children become new roots.
  • A leaf node is deleted — no new roots are created (it simply disappears).
  • All nodes deleted — return empty list.
  • No nodes deleted — return a forest with just the original root.
  • A deleted node's parent is also deleted — the child is still a new root (the link to its grandparent is already severed).
2 The Algorithm: Post-order DFS with is_root Flag ▾

We perform a single post-order DFS — visiting left child, right child, then the node itself. This bottom-up order ensures we process children before parents, so we can sever links cleanly as we return up the call stack.

1 Convert delete list to a set — Enables O(1) membership checks. We'll do this check at every node, so it must be fast.
2 DFS with is_root parameter — Each call carries a boolean: "Is this node currently a potential forest root?" The original tree root starts with is_root=True.
3 Add to forest if (is_root AND not deleted) — A surviving orphan becomes a new forest root. We add it at the top of the call before recursing.
4 Recurse on children with correct is_root — Pass is_root=deleted to each child. If we're deleting the current node (deleted=True), the children become orphans (is_root=True). If we keep the node, children remain normal children (is_root=False).
5 Return None if deleted, else return node — Returning None severs the link at the parent level. The parent captures this: node.left = dfs(node.left, deleted) sets node.left = None when the left child is deleted.
🔑 Why Post-order Is Essential

Pre-order visits the parent before children — but at that point, we don't yet know if a child needs to be severed. Post-order visits children first, so by the time we handle a parent's deletion, its children have already been visited, captured as roots if needed, and their return values (None or themselves) are ready to assign.

3 Why It Works: The Invariant ▾
🔐 The Invariant

When dfs(node, is_root) is called, the value of is_root is always exactly correct: it is True if and only if the node's parent has been deleted (or the node is the original root). This correctness is maintained because the parent's deletion status is determined before the recursive call is made.

Why returning None severs the link: Python assignment node.left = dfs(node.left, ...) sets node.left to whatever dfs returns. If the left child is deleted, dfs returns None, so node.left = None — the link is severed automatically. No need for parent pointers or separate deletion passes.

Proof of new-root correctness: Suppose node C is a child of node P which is being deleted. When we process P (the parent's frame), we detect deleted=True and call dfs(C, is_root=True). Inside that call, if C is not in the delete set, the first conditional fires: forest.append(C). This happens before we recurse into C's subtree — so C is captured as a root before any of its descendants are processed. This ordering ensures no root is missed and no root is double-counted.

⚠️ Common Mistake: Forgetting the Original Root

The initial call must pass is_root=True for the tree's original root. If the root is NOT deleted, it must appear in the forest. Many incorrect solutions start with is_root=False and forget to handle the root as a special case — then the root node is never added to the forest, producing a wrong answer.

When the root itself is deleted: The call dfs(root, is_root=True) detects deleted=True. Even though is_root=True, the not deleted condition prevents adding root to the forest. We then call dfs(root.left, is_root=True) and dfs(root.right, is_root=True) — both surviving children become new roots. The function returns None for the deleted root.

4 The Code (Annotated) ▾

Solution 1 — Post-order DFS with is_root flag (Optimal / Interview Pick)

def delNodes(root, to_delete):
to_delete_set = set(to_delete) ← O(1) lookup; avoid O(k) list scan per node
forest = [] ← Accumulate surviving tree roots here
def dfs(node, is_root): ← is_root: True if this node may be a new forest root
if not node: return None ← Base case: null child, nothing to do
deleted = node.val in to_delete_set ← Is this node scheduled for deletion? O(1)
if is_root and not deleted: ← Surviving orphan = new forest root!
forest.append(node) ← Capture root BEFORE modifying children
node.left = dfs(node.left, deleted) ← deleted becomes child's is_root signal
node.right = dfs(node.right, deleted) ← Same for right; None return severs the link
return None if deleted else node ← None tells parent to sever this edge
dfs(root, is_root=True) ← Root is always a candidate root initially
return forest ← All surviving tree roots collected
⚠️ Why must we assign node.left = dfs(node.left, ...) instead of just calling dfs?

The return value of dfs is either the node itself (if kept) or None (if deleted). We must write this return value back to node.left. Without the assignment, calling dfs(node.left, ...) only traverses the subtree — it never updates node.left, so deleted children would still be reachable from their parent.

Solution 2 — BFS with parent tracking (Iterative Alternative)

from collections import deque
def delNodes(root, to_delete):
to_delete_set = set(to_delete)
forest = []
# Queue holds (node, parent, side) tuples ← Track where each node hangs
queue = deque([(root, None, None)]) ← Root has no parent
while queue:
node, parent, side = queue.popleft()
if node.val in to_delete_set: ← Delete this node
if parent: ← Sever parent's pointer to this node
setattr(parent, side, None)
for child, s in [(node.left,'left'),(node.right,'right')]:
if child: ← Orphaned children become new roots
forest.append(child)
queue.append((child,None,None))← No parent now (orphan)
else:
if parent is None: forest.append(node)← Root of original tree (if undeleted)
if node.left: queue.append((node.left, node,'left'))
if node.right: queue.append((node.right,node,'right'))
return forest
💡 Complexity Summary

DFS (Optimal): Time O(n) — every node visited exactly once. Space O(h) recursion stack (h = tree height; O(n) worst case skewed tree) + O(k) delete set + O(n) result.
BFS: Time O(n), Space O(n) queue. Avoids recursion stack but is more complex to implement correctly.
Prefer DFS in interviews — it's shorter, cleaner, and demonstrates elegant use of return values.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Reframe the Problem

"What makes a node become a new forest root?" — This is the central question. Answer: A node becomes a new root when its parent is deleted. So we need to detect, for each node, whether its parent just got deleted. That's a signal that travels downward through the tree.

Step 2: Identify the Need for Post-order

We need to modify parent-child links. When we delete a node, we want its parent to immediately lose its reference to it. The cleanest way: have the deleted node's recursive call return None, and have the parent assign that return value back. This requires processing children before parents — post-order.

Step 3: Design the API

dfs(node, is_root) — the function passes down "are you an orphan?" and passes up "should your parent sever its reference to you?" The is_root flag flows top-down; the return value (None vs. node) flows bottom-up. This two-way information flow through a single DFS is the elegant core of the solution.

Step 4: Handle the Original Root

The initial call is dfs(root, is_root=True). If the root isn't deleted, it gets added to forest immediately. This avoids any special-casing — the same logic handles all nodes uniformly.

🗣️ What to Say to the Interviewer

"I'll use post-order DFS — children before parents — so I can sever links as I return up the stack.
Each call carries an is_root flag. A node becomes a new forest root if is_root=True and it's not deleted.
is_root for children equals whether the current node is being deleted.
Deleted nodes return None, which severs the parent's reference automatically via assignment.
Original root starts with is_root=True. Complexity: O(n) time, O(h) space for the call stack."
🔮 Likely Follow-up Questions
  • "Can you do it without recursion?" → BFS with parent+side tracking (Solution 2 above)
  • "What if node values aren't unique?" → Use a set of node references (by id), not values
  • "What if it's an N-ary tree?" → Same approach; filter out deleted children from node.children list
  • "What's the space complexity for a balanced vs. skewed tree?" → O(log n) balanced, O(n) skewed (recursion stack)

7 Summary & Key Takeaways ▾
Post-order DFS (children before parents) is the natural traversal for bottom-up tree modifications.
Passing is_root downward elegantly propagates "your parent was just deleted" without tracking parent pointers.
Returning None from a deleted node automatically severs the parent-child link via the assignment node.left = dfs(...).
Convert to_delete to a set upfront — never do O(k) list membership tests inside a loop.
The original root must also start with is_root=True — it's a potential forest root too!
🎯 When to Recognize This Pattern
  • "Delete nodes from a tree" — almost always signals post-order DFS with return-value severance.
  • Problems where a node's fate depends on its parent's fate — pass signals downward as parameters.
  • Collecting subtree roots after structural modifications — capture before recursing.
  • Any problem needing bidirectional information flow in a tree — pass flags down; return results up.

Complexity: Time O(n) — each node visited exactly once. Space O(n) — O(h) recursion stack where h is tree height; O(n) worst case for a skewed tree.
