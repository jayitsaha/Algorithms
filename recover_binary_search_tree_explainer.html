LeetCode #99

Recover Binary Search Tree

Two nodes of a BST are swapped by mistake. Recover the tree without changing its structure.

Medium Trees In-order Traversal Time: O(n) Space: O(h)
1 Understanding the Problem ▾

A Binary Search Tree (BST) has a fundamental property: for every node, all values in its left subtree are smaller, and all values in its right subtree are larger. The most useful consequence: if you visit nodes using in-order traversal (Left → Node → Right), you get values in perfectly sorted ascending order.

This problem tells us exactly two nodes have had their values swapped. We need to find them and restore the BST — without changing any pointers or tree structure, only the two misplaced values.

💡 Key Insight

In a correct BST, in-order traversal gives a sorted sequence. Two swapped nodes create exactly 1 or 2 "inversions" — positions where a value is larger than the next one in the in-order sequence. Finding these inversions identifies the two wrong nodes.

✗ Corrupted: nodes 3 and 2 swapped

Tree: root=3, left=1, right=4
       right.left=2
In-order: 1, 3, 2, 4
Inversion: (3>2) → nodes 3 and 2 are wrong

✓ Recovered: swap values 3 ↔ 2

Tree: root=2, left=1, right=4
       right.left=3
In-order: 1, 2, 3, 4 ✓
BST property fully restored

Adjacent vs non-adjacent swaps: If the two swapped nodes are adjacent in the in-order sequence (e.g., swap nodes with values 2 and 3 in [1,2,3,4,5]), only one inversion appears. If non-adjacent (e.g., swap 1 and 4 in [1,2,3,4,5] → [4,2,3,1,5]), two inversions appear: (4>2) and (3>1). Our algorithm handles both cases with the same logic.

Edge cases: Tree with only 2 nodes (trivial swap of root and child); root itself is one of the swapped nodes; swapped nodes are far apart in the tree.

2 The Algorithm: In-order Traversal with Inversion Detection ▾

We perform a single DFS in-order traversal, carrying a prev pointer that tracks the most recently visited node. At each step, if prev.val > node.val, we've found an inversion and record the involved nodes. At the end, we swap the two identified nodes' values.

1 Initialize: first = None (the misplaced "too-large" node), second = None (the misplaced "too-small" node), prev = a sentinel with val = -infinity so the first comparison never falsely triggers.
2 Recurse left: In-order means we go to the left subtree first. This ensures we visit nodes from smallest to largest in a correct BST.
3 Process current node: Check if prev.val > node.val. If yes, an inversion is here. At the first inversion: set first = prev and second = node. At any subsequent inversion: update second = node only (first is already correct).
4 Advance prev: Set prev = node before recursing right, so the next node visited can compare against the current one.
5 Swap values: After traversal, swap first.val and second.val. No pointers change — only the integer values stored in those two nodes are exchanged.
🔑 Why set first = prev (not node) at first inversion?

At an inversion, prev.val > node.val — the previous value is too large for this point in the sorted order. So prev is the "early, too-large" wrong node (becomes first). The current node is potentially the "late, too-small" wrong node (becomes second). If there's a second inversion, second gets updated because the actual too-small node is even further right in the sequence.

Two-inversion example: swap 1 and 4 in [1,2,3,4,5] → [4,2,3,1,5]
In-order after swap:  4  →  2  →  3  →  1  →  5
Step prev=4, cur=2:  4>2 → INVERSION #1 → first=node(4), second=node(2)
Step prev=2, cur=3:  2<3 → OK
Step prev=3, cur=1:  3>1 → INVERSION #2 → second=node(1) (updated)
Swap first(4) ↔ second(1) → Result: [1,2,3,4,5] ✓
3 Why It Works: The Invariant ▾
🔐 The Invariant

Throughout the in-order traversal, prev always holds the node most recently processed. In a correct BST, prev.val < current.val always holds (the sequence is strictly increasing). Any violation of this inequality — prev.val > current.val — reveals one of the two swapped nodes.

Why a single swap creates exactly 1 or 2 inversions: Consider a sorted array [a, …, X, …, Y, …, z] where X < Y. Swapping X and Y gives [a, …, Y, …, X, …, z]. The position where Y now sits is "too large" — the element before Y (or Y itself compared to the next element smaller than Y) creates one inversion. The position where X now sits is "too small" — X compared to the element before it creates a second inversion. If X and Y were adjacent, these two violations merge into one inversion at the single boundary point.

Why updating second at every inversion is correct: The first inversion pins first correctly — it's the leftmost "out of place large" node. But second may not be final yet: a second inversion could appear later, meaning the actual too-small node is further right. Always writing second = node ensures we get the rightmost candidate, which is the true partner to first.

⚠️ Common Mistake: Early Return After First Inversion

If you return immediately when you find the first inversion, you correctly handle adjacent swaps but miss the true second node in non-adjacent swaps. Always complete the full traversal. The adjacent case still works because there is no second inversion to update second with — it stays correctly set from the first.

Space: Recursion uses O(h) call stack — O(log n) for balanced trees, O(n) worst case (degenerate/linear tree). Morris traversal can reduce this to O(1) by threading the tree, but adds significant implementation complexity.

4 The Code (Annotated) ▾

Solution 1 — In-order with prev pointer (Optimal — Interview Pick)

def recoverTree(root):
first = second = None ← two wrong nodes we need to find
class Sentinel: val = float('-inf') ← dummy node; -inf ensures first real comparison never triggers inversion
prev = Sentinel()
def inorder(node):
nonlocal first, second, prev ← needed so inner function can mutate outer variables
if not node: return ← base case: null node → nothing to do
inorder(node.left) ← L: go left first (in-order: Left → Node → Right)
if prev.val > node.val: ← inversion! prev should always be < node in a correct BST
if first is None: ← first inversion: lock in 'first' (never update again)
first = prev ← prev is too large for its position → it's the misplaced big node
second = node ← always update second (handles both 1-inversion and 2-inversion cases)
prev = node ← advance prev BEFORE going right (not after — order matters)
inorder(node.right) ← R: visit right subtree last
inorder(root)
first.val, second.val = second.val, first.val ← swap VALUES only; tree structure (pointers) untouched
⚠️ Why prev = node BEFORE inorder(node.right)?

After we process the current node (the "Node" in Left→Node→Right), we set prev = node and then recurse right. This is correct: the right subtree should see this node as its predecessor. If we set prev after the right recursion, it would be updated too late — the right subtree would compare against the wrong predecessor.

Solution 2 — Collect in-order array, sort, reassign (Brute Force)

def recoverTree(root):
nodes = [] ← collect all tree nodes in in-order order
def inorder(n):
if n: inorder(n.left); nodes.append(n); inorder(n.right)
inorder(root)
vals = sorted(n.val for n in nodes) ← what the values SHOULD be (sorted)
for node, v in zip(nodes, vals): ← pair each node with its correct value
node.val = v ← assign; restores BST. But this is O(n log n) + O(n) space
💡 Complexity Summary

In-order + prev (Solution 1 — optimal): Time O(n), Space O(h)
Collect + sort (Solution 2): Time O(n log n), Space O(n)
Morris traversal (advanced): Time O(n), Space O(1) — threads the tree; no stack

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the BST In-order Property

Whenever you see "BST" + "something is wrong" + "find/fix nodes", immediately think: in-order traversal gives sorted order. Two values swapped = one or two violations of sorted order. Reduces a complex tree problem to a simple sequence-inversion problem.

Step 2: Propose the Brute Force First

"I can collect all nodes in in-order, sort the values, and reassign. This is O(n log n) and O(n) space. It's correct but suboptimal — since only two values are wrong, I don't need to sort everything."

Step 3: Optimize with a Single Pass

"Since in-order should be sorted, I just need to find the inversions. I track a 'prev' pointer — if prev.val > curr.val, that's an inversion. First inversion sets both first and second; second inversion (if it exists) updates second. Then swap their values."

Step 4: Mention the O(1) Space Follow-up

"If O(1) space is required, Morris Traversal achieves this by temporarily threading the tree — creating back-pointers to simulate the call stack, then cleaning them up. This is complex to implement but worth mentioning as a bonus."

🗣️ What to Say to the Interviewer

"The key property: in-order traversal of a BST gives sorted values.
Two swapped nodes break this, creating exactly 1 or 2 inversions (places
where prev.val > curr.val during in-order traversal).
I'll do one DFS pass with a 'prev' pointer.
At the first inversion: first = prev (too large), second = node (too small).
At any second inversion: update second = node (it's further right).
After traversal: swap first.val and second.val.
Time O(n), Space O(h) for the recursion stack."
🔮 Likely Follow-up Questions
  • "Can you do O(1) space?" → Morris Traversal; threads left-child's rightmost node back to current; complex but doable
  • "What if THREE nodes were swapped?" → Track up to 3 inversion boundaries; same idea, more state
  • "What if we can change the tree structure?" → Collect in-order, rebuild the tree entirely (simpler but not required here)
  • "How do you verify your fix is correct?" → Run in-order on the recovered tree and confirm it's sorted

7 Summary & Key Takeaways ▾
In-order traversal of a correct BST gives strictly increasing values — this is the fundamental property to exploit
Two swapped nodes create exactly 1 inversion (adjacent swap) or 2 inversions (non-adjacent swap) in the in-order sequence
Track prev during in-order DFS; when prev.val > node.val, you've found an inversion
Set first = prev only on the first inversion; update second = node at every inversion — this handles both 1 and 2 inversion cases
Swap only the values in the two nodes — never touch tree pointers or structure
O(1) space via Morris traversal is possible but complex — always mention it as a follow-up, not the primary solution
