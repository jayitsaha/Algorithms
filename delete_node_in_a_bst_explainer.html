LeetCode #450

Delete Node in a BST

Given a root node reference of a BST and a key, delete the node with the given key. Return the root node reference of the updated BST.

Medium Trees BST Time: O(h) Space: O(h)
1 Understanding the Problem ▾

A Binary Search Tree (BST) has one golden invariant: for every node, all values in its left subtree are smaller, and all values in its right subtree are larger. Deleting a node sounds simple — until you realize you must also reattach its children without breaking this invariant.

💡 Key Insight

BST deletion has exactly three cases based on how many children the target node has. Cases 1 and 2 are trivial bypasses. Case 3 (two children) requires a non-obvious trick: replace the target's value with its in-order successor (the smallest value in its right subtree), then delete the successor — which is always easy because the successor has at most one child.

The three cases visualized:

Case 1None (leaf node)Return None to parent — pointer simply disappears
Case 2One child (left or right)Return that child — parent bypasses deleted node directly
Case 3Two childrenFind in-order successor, copy its value here, delete successor from right subtree
✗ Naive: "just remove node 5"

Node 5 has children 3 and 7.

✓ Correct: replace value, delete successor

Copy successor's value (6) into node 5.

BST property holds: {3,2,4} < 6 < {7}.

Edge cases: key not present in tree (return root unchanged), deleting the root (handled by returning the updated node), single-node tree, node with one child on either side.

2 The Algorithm: Recursive In-Order Successor Replacement ▾

The recursive approach is elegant: at every node, we return the updated subtree root to the parent. This lets parent pointers update themselves in one assignment per level — no manual pointer tracking needed.

1 Base case: If root is None, key wasn't in the tree — return None. Parent gets back None gracefully.
2 Navigate by BST property: If key < root.val, target is in the left subtree — recurse left. If key > root.val, recurse right. Set the child pointer to the return value (this is the magic — it updates the tree automatically).
3 Found the target (key == root.val): Now apply the three-case analysis.
4 Case 1/2 — Zero or one child: If no left child, return root.right (which may be None). If no right child, return root.left. Parent receives the correct replacement.
5 Case 3 — Two children: Walk to root.right, then keep going left until there's no left child. This leftmost node is the in-order successor. Copy its value into root.val. Then call deleteNode(root.right, succ.val) to cleanly remove the successor.
6 Return root: After handling any case, return root to the caller. If value was copied (Case 3) or children updated (Case 2), this is the same node. If Case 1, we already returned None or a child above.
🔑 Why the In-Order Successor?

The in-order successor is the smallest value that is larger than the deleted node. It perfectly slots between the left subtree (all smaller) and the remaining right subtree (all larger). Most importantly: the successor has no left child — it's the leftmost node in a subtree — so deleting it always resolves to Case 1 or Case 2. This prevents infinite recursion depth on Case 3.

3 Why It Works: The Invariant ▾
🔐 The Invariant

After every call to deleteNode(subtree_root, key), the returned tree is a valid BST with the key removed (if it was present). The parent can simply assign parent.left = deleteNode(parent.left, key) and the entire tree remains valid.

Why the successor replacement preserves BST order:

  • Let S = successor (leftmost in right subtree of target T)
  • S.val > all values in T's left subtree (BST property of original tree)
  • S.val < all other values in T's right subtree (S was the leftmost = minimum)
  • So replacing T.val with S.val satisfies: left subtree < S.val < right subtree ✓
  • Deleting S from right subtree is Case 1 or 2 — always terminates cleanly ✓
⚠️ Common Mistake: Moving Node Objects

Beginners try to physically move tree nodes — detaching them and re-attaching at different positions. This requires tracking grandparents, uncle nodes, and many pointer variables. The clean insight: you never need to move nodes. Just copy the successor's value into the target node, then delete the successor. The node object at the target position remains in place.

Why the recursive return pattern is correct: Python's root.left = deleteNode(root.left, key) pattern means: "my left child should become whatever deleteNode returns." If the left child gets deleted, it returns its own child (or None). If it navigates deeper, it returns itself with an updated subtree. Either way, the assignment handles all cases automatically.

4 The Code (Annotated) ▾

Solution 1 — Recursive Successor Replacement (Interview Pick, O(h) time)

def deleteNode(root, key): ← returns updated subtree root to parent
if not root: ← base case: key not in tree (fell off the edge)
return None
if key < root.val: ← key is smaller: must be in left subtree
root.left = deleteNode(root.left, key) ← update left pointer with result
elif key > root.val: ← key is larger: must be in right subtree
root.right = deleteNode(root.right, key) ← update right pointer with result
else: ← key == root.val → THIS node is the target
if not root.left: ← Case 1 (leaf) or Case 2 (right child only)
return root.right ← bypass: parent gets right child (or None)
if not root.right: ← Case 2: left child only
return root.left ← bypass: parent gets left child
# Case 3: two children → find in-order successor
succ = root.right ← start at right child
while succ.left: ← go as far left as possible
succ = succ.left ← successor = leftmost node in right subtree
root.val = succ.val ← COPY successor's value into target node
root.right = deleteNode(root.right, succ.val) ← delete successor from right subtree
return root ← return (possibly modified) current node
⚠️ Why copy the value, not splice the node?

We copy succ.val into root rather than removing root and inserting succ in its place. This avoids touching any pointers above root in the tree — the parent of root never needs to change. We only modify the right subtree, which we already have a recursive mechanism for.

Solution 2 — Iterative (O(h) time, O(1) extra space)

def deleteNode(root, key):
parent, node, go_left = None, root, False ← track parent + direction to child
while node and node.val != key: ← search phase
parent = node
if key < node.val: node = node.left; go_left = True
else: node = node.right; go_left = False
if not node: return root ← key not in tree
if not node.left: repl = node.right ← Case 1 or 2
elif not node.right: repl = node.left ← Case 2
else: ← Case 3: find successor and relink
repl = node.right; prev = node
while repl.left: prev, repl = repl, repl.left
if prev != node: prev.left = repl.right; repl.right = node.right
repl.left = node.left
if not parent: return repl ← deleted root; new root is repl
if go_left: parent.left = repl
else: parent.right = repl
return root
💡 Complexity Summary

Both solutions: Time O(h) — h is tree height. O(log n) for a balanced BST, O(n) for a degenerate (sorted-input) tree.
Recursive space: O(h) call stack. Iterative space: O(1) extra — no call stack overhead.
Interview pick: Recursive. Cleaner code, easier to explain the three-case analysis.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify the Pattern

"This is a BST modification problem. For BST problems, I always use the BST property to navigate (go left if key is smaller, right if larger). For modifications, I'll use the recursive return pattern — each call returns the updated subtree root."

Step 2: Enumerate the Three Cases Out Loud

"Once I find the node, there are three cases. No children: return None. One child: return that child. Two children: this is the interesting one — I need the in-order successor." Naming the cases upfront shows systematic thinking.

Step 3: Justify the Successor Choice

"The in-order successor is the smallest value larger than the target. It slots perfectly between the left and right subtrees. And since it's the leftmost node in the right subtree, it has no left child — so deleting it is always Case 1 or 2."

Step 4: Highlight the Value-Copy Trick

"I copy the successor's value into the current node rather than moving nodes around. This keeps the upper tree untouched — only the right subtree changes, which my recursive call handles."

🗣️ What to Say to the Interviewer

"I'll use a recursive approach where each call returns the updated subtree root.
Base case: null root means key not found, return null.
Navigate by BST property: smaller → go left, larger → go right.
When found, three cases:
Case 1/2: no left child → return right child. No right → return left child.
Case 3 (two children): find in-order successor — go right, then left as far as possible.
Copy successor's value here. Recursively delete successor from right subtree.
Successor always has no left child, so that recursive call hits Case 1 or 2.
Time O(h), Space O(h) for call stack."
🔮 Likely Follow-up Questions
  • "Can you do it iteratively?" → Yes. Track parent + direction. More code, O(1) space.
  • "Why not use the predecessor?" → Either works. Predecessor = rightmost in left subtree. Both are symmetric.
  • "What if there are duplicate keys?" → Delete only one occurrence; don't stop search early.
  • "Insert into BST instead?" → Simpler: same navigation, no deletion logic (#701).
  • "What if the tree needs rebalancing after deletion?" → AVL/Red-Black trees add rotation logic after each change.

7 Summary & Key Takeaways ▾
BST deletion has exactly 3 cases: leaf, one child, two children
Cases 1 and 2 are trivial bypasses — just return None or the existing child
Case 3: copy in-order successor's value, then delete successor from right subtree
Successor = leftmost node in right subtree; always has at most one (right) child
Recursive return pattern — root.left = deleteNode(root.left, key) — handles pointer updates automatically
Time O(h): O(log n) balanced BST, O(n) for degenerate (linked-list-like) tree
