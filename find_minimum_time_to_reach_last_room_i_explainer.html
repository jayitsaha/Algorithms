LeetCode #3341

Find Minimum Time to Reach Last Room I

Navigate an n×m dungeon grid where each room has a minimum move time requirement. Find the earliest second you can reach the bottom-right room from (0,0).

Medium Graph Modified Dijkstra Time: O(n·m·log(n·m)) Space: O(n·m)
1 Understanding the Problem ▾

You're given an n × m grid (a dungeon). Each cell moveTime[i][j] represents the earliest second at which you are allowed to enter room (i, j). You start at (0, 0) at second 0, and each move takes exactly 1 second. You can move up, down, left, or right between adjacent rooms. The goal: reach (n-1, m-1) in the minimum possible time.

💡 Key Insight

This is not plain BFS because edge "costs" vary — you might have to wait outside a room if you arrive before its moveTime threshold. This waiting turns it into a shortest-path problem with variable costs, which is exactly what Dijkstra's algorithm handles optimally.

Why BFS Fails

Worked example (3×3 grid):

moveTime = [[0,4],[4,4]] # 2×2 grid
Start: (0,0) at t=0
Move to (0,1): moveTime[0][1]=4, arrive=max(0,4)+1=5 ← must wait until t=4, then 1 sec to move
Move to (1,1): moveTime[1][1]=4, arrive=max(5,4)+1=6 ← arrive t=5 > 4, no wait needed
Answer: 6

Edge cases: moveTime[0][0] is always 0 (start room). Single-cell grid returns 0. Large moveTime values require waiting, potentially doubling the path length. The constraint says you can always reach the last room.

2 The Algorithm: Modified Dijkstra on a 2D Grid ▾

Treat the grid as a weighted directed graph where each edge weight is dynamic: it depends on when you arrive. The key modification to standard Dijkstra is in how edge weights are computed — instead of a fixed weight, the cost to enter a neighbor is max(current_time, moveTime[neighbor]) + 1.

1 Initialize: Create a dist table with infinity for all cells except dist[0][0] = 0. Push (0, 0, 0) (time, row, col) into a min-heap.
2 Extract minimum: Pop (t, r, c) — the unvisited cell reachable in the least time. If t > dist[r][c], it's a stale entry — skip it (standard Dijkstra lazy deletion).
3 Early termination: If we just popped (n-1, m-1), return its time immediately — Dijkstra guarantees this is optimal.
4 Relax neighbors: For each of the 4 adjacent cells (nr, nc), compute new_t = max(t, moveTime[nr][nc]) + 1. If new_t < dist[nr][nc], update dist and push to heap.
5 Repeat until the heap is empty or we reach the goal. Since the problem guarantees a path exists, we always terminate at step 3.
🔬 The Modified Edge Weight Formula

Standard Dijkstra: new_dist = cur_dist + weight

This problem: new_t = max(cur_t, moveTime[nr][nc]) + 1

The max represents waiting: if you arrive at time 3 but the room opens at time 10, you wait until time 10, then spend 1 second entering. If you arrive at time 12 and the room opened at 10, no waiting needed.

3 Why It Works: Dijkstra's Invariant + Algorithm Deep-Dive ▾
🔐 Dijkstra's Core Invariant

When a node is popped from the min-heap, its recorded distance is guaranteed to be optimal. No future path through unvisited nodes can do better, because all edge weights are non-negative (we can never "un-wait" time).

Why non-negative weights are required: The waiting formula max(cur_t, moveTime) + 1 always produces a result ≥ cur_t + 1 — we always move forward in time. This guarantees non-negative edge costs, making Dijkstra applicable.

📖 Dijkstra's Algorithm — Origin & Generalization

Origin: Edsger W. Dijkstra, 1956. Originally conceived to find the shortest path between two cities in a road network. Published in 1959 as a 3-page paper — one of the most influential algorithms ever.

Class of problems: Single-source shortest path on graphs with non-negative edge weights.

Core invariant: The min-heap always holds candidates ordered by their tentative shortest distance. When we pop a node, we've "settled" it — its shortest path is known.

Complexity: O((V + E) log V) with a binary heap. For an n×m grid: V = n·m, E = 2·n·m (4 edges per cell, but undirected) → O(n·m·log(n·m)).

When to recognize it: "Find minimum cost path", "weighted graph", "can wait but not go back in time", constraints are non-negative.

⚠️ Why Not BFS or Dynamic Programming?

BFS: Works for unweighted graphs (all edges = 1 second). Here, effective edge cost = max(cur_t, moveTime[nr][nc]) + 1, which varies by when you arrive. Two paths to the same cell might have different costs depending on timing.

Simple DP: Grid DP assumes you process cells in a fixed order (e.g., top-left → bottom-right). Here, you might profitably revisit cells via a detour, and the optimal sub-structure doesn't align with row/col order.

Proof sketch for this specific problem: Since max(cur_t, moveTime[nr][nc]) + 1 ≥ cur_t + 1 > cur_t, time strictly increases along any path. Therefore, once we pop cell (r, c) with time t, any other path reaching (r, c) that hasn't been finalized yet must have cost ≥ t (since min-heap), and any path through later-settled cells can only add more time. The optimality invariant holds.

4 The Code (Annotated) ▾

Solution 1 — Modified Dijkstra (Optimal, Interview Pick)

import heapq← Python's min-heap (heapq is a min-heap by default)
def minTimeToReach(moveTime):
n, m = len(moveTime), len(moveTime[0]) ← grid dimensions
dist = [[float('inf')] * m for _ in range(n)] ← best known arrival time per cell
dist[0][0] = 0 ← start at (0,0) at time 0
heap = [(0, 0, 0)] ← (time, row, col) — min-heap on time
while heap: ← process until all reachable cells settled
t, r, c = heapq.heappop(heap) ← extract cell with lowest arrival time
if t > dist[r][c]: ← stale entry? (better path found later)
continue ← skip — lazy deletion avoids expensive heap updates
if r == n-1 and c == m-1: ← reached the goal!
return t ← Dijkstra guarantees this is optimal
for dr, dc in [(-1,0),(1,0),(0,-1),(0,1)]: ← 4-directional neighbors
nr, nc = r + dr, c + dc ← candidate neighbor
if 0 <= nr < n and 0 <= nc < m: ← bounds check
new_t = max(t, moveTime[nr][nc]) + 1 ← wait if needed, then 1 sec to move
if new_t < dist[nr][nc]: ← found a shorter path to (nr, nc)?
dist[nr][nc] = new_t ← update best known time
heapq.heappush(heap, (new_t, nr, nc)) ← push to explore later
return dist[n-1][m-1] ← fallback (problem guarantees path exists)
⚠️ Why Check t > dist[r][c]?

We use lazy deletion: rather than updating existing heap entries (which is expensive), we push new entries and mark old ones stale. When we pop a stale entry (one where a shorter time was found), we skip it. This is the standard Dijkstra optimization in Python since heapq doesn't support decrease-key.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Graph Pattern

The grid is a graph: rooms are nodes, adjacency gives edges. The goal is "minimum time from source to destination." This screams shortest path. Ask: are edge costs variable? Yes — they depend on when you arrive. So standard BFS (uniform cost) won't work.

Step 2: Identify the Edge Weight Formula

What's the cost to move from (r,c) to (nr,nc)? You must wait if you arrive before moveTime[nr][nc]. So the arrival time at the neighbor is max(current_time, moveTime[nr][nc]) + 1. This is a dynamic/context-dependent edge weight.

Step 3: Confirm Dijkstra Applies

Check: are all edge costs non-negative? Yes — max(t, moveTime) + 1 ≥ t + 1 > t. Time never decreases. Dijkstra's invariant holds. Use a min-heap prioritized by arrival time, not by hops.

Step 4: Handle the "Stale Entry" pattern

Python's heapq doesn't support decrease-key. Use lazy deletion: push a new (new_t, nr, nc) whenever you improve a distance. When you pop an entry, check if t > dist[r][c] — if so, skip it. This is a standard Dijkstra interview pattern.

🗣️ What to Say to the Interviewer

"The grid is a weighted graph where edge costs depend on arrival time.
If I arrive at (r,c) at time t and want to enter (nr,nc),
I must wait until max(t, moveTime[nr][nc]), then spend 1 second moving.
Since time only moves forward, all costs are positive — Dijkstra applies.
I'll use a min-heap sorted by arrival time. Each pop gives me the
earliest-reachable unvisited cell. I stop when I pop (n-1, m-1).
Time complexity: O(n·m·log(n·m)), Space: O(n·m)."

7 Summary & Key Takeaways ▾
Grid-to-graph reduction: rooms = nodes, adjacency = edges, time = cost
Modified edge weight: max(cur_t, moveTime[nr][nc]) + 1 captures both waiting and moving
Dijkstra applies because time never decreases (non-negative edge costs)
Lazy deletion (if t > dist[r][c]: continue) avoids expensive heap updates
Early termination: return as soon as the goal is popped from the min-heap
Pattern: "minimum cost to reach destination" on weighted graph → Dijkstra
