LeetCode #236

Lowest Common Ancestor of a Binary Tree

Given a binary tree and two nodes p and q, find their lowest common ancestor — the deepest node that has both p and q as descendants (a node can be a descendant of itself).

Medium Trees DFS Postorder Find in Left/Right Subtrees Time: O(n) Space: O(h)
1 Understanding the Problem ▾

The Lowest Common Ancestor (LCA) of two nodes p and q in a binary tree is the deepest node in the tree that is an ancestor of both p and q. Crucially, the problem allows a node to be an ancestor of itself — so if p is an ancestor of q, then p is the LCA.

💡 Key Insight

A node X is the LCA of p and q if and only if one of three conditions holds at X: (1) X is p or q itself, (2) p is in X's left subtree and q is in X's right subtree, or (3) q is in X's left subtree and p is in X's right subtree. We can detect all three from a single bottom-up DFS pass.

Consider this example tree:

        3
       / \
      5   1
     / \ / \
    6  2 0  8

Edge cases: One of p or q is the direct ancestor of the other. p or q is the root. Both p and q are leaves on opposite sides. Tree is a single node.

Constraints given: Both p and q are guaranteed to exist in the tree. All node values are unique.

2 The Algorithm: Recursive Postorder DFS ▾

The elegant solution uses postorder DFS (Left → Right → Root). Each call asks its subtrees: "Did you find p or q?" Then it makes a decision based on both answers. This bottom-up approach is why postorder is natural here — we need children's results before the parent can decide.

1 Base case — null node: If root is None, return None. We've gone off the tree; nothing was found in this direction.
2 Base case — target match: If root == p or root == q, return root immediately. We've found a target. No need to recurse further — return it as a signal upward.
3 Recurse into both subtrees: Call lca(root.left, p, q) → store as left. Call lca(root.right, p, q) → store as right. Each returns None or a found node.
4 Both sides found something: If left and right are both non-None, p and q are in different subtrees — the current node is the LCA. Return root.
5 Only one side found something: Return left or right. This propagates the found result upward. The LCA will be determined at a higher node, or was already found via the "self is target" base case.
🔑 The Three-Signal Return Convention

Every call returns exactly one of three things: None (neither p nor q in this subtree), p or q (exactly one target found here — acting as a signal), or the LCA node (once determined, it bubbles up unchanged to the root). The cleverness: we reuse the same TreeNode return type for all three cases.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every node, the return value is: None if neither p nor q exists in its subtree; the LCA node if the LCA has been identified within this subtree; or whichever of p or q exists in this subtree (as an upward signal that one target was found). This invariant is maintained throughout the recursion.

Why detecting both non-None results is correct: Because we use postorder DFS, when we evaluate if left and right, the entire left subtree and entire right subtree have already been fully explored. A non-None left means one of p/q is somewhere in the left branch. A non-None right means one of p/q is somewhere in the right branch. Since p ≠ q and they can't both be in both subtrees simultaneously, the current node is definitionally the deepest node where the paths to p and q diverge — it's the LCA.

Why the ancestor case works (e.g., p=5, q=4 where 4 is under 5): When DFS reaches node 5 (which equals p), it returns 5 immediately without exploring 5's subtree. This seems like it would miss q=4. But the correctness comes from the right-subtree search: the right subtree of node 3 (root) has none of p=5 or q=4, so it returns None. Back at the root: left=Node(5), right=None → return left = Node(5). Node 5 is correctly identified as the LCA.

⚠️ Common Confusion: "Don't We Miss q When We Return Early at p?"

When root == p and we return root without recursing, we've stopped one branch of the recursion. But the parent's other branch will still search for q. The parent collects results from BOTH its left and right subtrees. If q is under p, then q never gets "found" as a separate signal — but the parent will see only left=Node(p) and right=None, and return Node(p). This is exactly correct, because p is the LCA when q is in p's subtree.

4 The Code (Annotated) ▾

Solution 1 — Recursive Postorder DFS (Optimal, Interview Pick)

def lowestCommonAncestor(root, p, q):
if root is None: ← Fell off tree: neither p nor q here
return None
if root == p or root == q: ← Hit a target: return it as an upward signal
return root
left = lowestCommonAncestor(root.left, p, q) ← Search entire left subtree
right = lowestCommonAncestor(root.right, p, q) ← Search entire right subtree
if left and right: ← Both sides found something → split here → current node IS the LCA
return root
return left or right ← Propagate whichever side found something (or None if neither did)
⚠️ What "return left or right" Actually Does

Python's or returns the first truthy value. Since TreeNode objects are truthy and None is falsy: if left=Node(5) and right=None, returns Node(5). If left=None and right=Node(1), returns Node(1). If left=None and right=None, returns None. Three outcomes from one line.

Solution 2 — Iterative with Parent Pointer Map

def lowestCommonAncestor(root, p, q):
parent = {root: None} ← Map node → its parent (root's parent is None)
stack = [root]
while p not in parent or q not in parent: ← Keep DFS until both targets are recorded
node = stack.pop()
if node.left:
parent[node.left] = node
stack.append(node.left)
if node.right:
parent[node.right] = node
stack.append(node.right)
ancestors = set() ← Collect all ancestors of p (path up to root)
while p:
ancestors.add(p)
p = parent[p]
while q not in ancestors: ← Walk q's ancestors until we hit p's lineage
q = parent[q]
return q ← First shared ancestor = LCA
💡 Complexity Summary

Recursive DFS: Time O(n) — visits each node at most once. Space O(h) call stack (O(log n) balanced, O(n) skewed/worst case).
Iterative (Parent Map): Time O(n), Space O(n) — stores parent map for all nodes. Avoids stack overflow for extremely deep trees.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Clarify the Problem Type

Ask: "Is this a BST or a general binary tree?" (This is general — no ordering property.) "Can a node be its own ancestor?" (Yes.) "Are p and q guaranteed to be in the tree?" (Yes.) Knowing it's a general tree rules out the BST value-comparison shortcut.

Step 2: Recognize the Pattern — "Need Both Children's Info First"

The LCA can only be determined at a node after knowing what both subtrees contain. This "decide after children" structure signals postorder DFS. The analogy: you can't know which employee is the shared manager until both employee records have been looked up.

Step 3: Design the Return Value Contract

What should each recursive call return? It must communicate: "I found nothing", "I found one target", or "I found the LCA." Reuse TreeNode|None — None for nothing, the target node as a signal for one-found, and the LCA node once determined. Three states in one return type.

Step 4: Code the Decision Logic

Both non-None → current node is LCA (return root). One non-None → propagate it up (return left or right). Both None → propagate None. The first time both sides are non-None is exactly the LCA — because this is postorder, the tree below has been fully searched.

🗣️ What to Say to the Interviewer

"I'll use a recursive postorder DFS. Each call returns: null if neither target is in this subtree, the target node if exactly one is found, or the LCA if both are found in different subtrees.
Base case: null node returns null. If current node equals p or q, return it immediately as a signal.
Recurse left and right. If both return non-null, p and q are on opposite sides — current node is the LCA. Otherwise return whichever is non-null.
Time O(n) — every node visited once. Space O(h) for the call stack. For a BST, I'd compare values for O(log n). Want me to code the iterative version too?"
🔮 Likely Follow-up Questions
  • "What if it's a BST?" → Compare: if both p,q < root go left; if both > root go right; else root is LCA. O(log n) for balanced.
  • "What if p or q might not exist?" → Track found_p and found_q booleans; only return LCA if both confirmed found.
  • "LCA of multiple nodes?" → Pass a set of targets; check membership instead of equality.
  • "What if tree has parent pointers?" → Walk p and q up to root, find first intersection — like finding the meeting point of two linked lists.
  • "Can you do it iteratively?" → Yes — parent pointer map (Solution 2 above). Build the map, collect p's ancestors into a set, walk q upward until you hit the set.

7 Summary & Key Takeaways ▾
Postorder DFS (children before parent) is natural when the parent's decision depends on both subtrees' results
Return type encodes three signals in one: None / found-one-target / LCA — no extra variables needed
Early return at "root == p or root == q" is correct even when one is an ancestor of the other — the parent handles the None-vs-Node logic
Time O(n), Space O(h) — each node visited exactly once; call stack depth equals tree height
For BST variant: compare values instead of searching; achieve O(log n) for balanced trees (#235)

🎯 When to Recognize the "Find in Left/Right Subtrees" Pattern
  • The answer to a tree question requires knowing what both subtrees contain simultaneously
  • Phrase "lowest/deepest/closest common..." — signals a need to detect where paths diverge
  • You can encode the answer in the return value of the recursive call itself (no global state needed)
  • Postorder traversal: same pattern applies to tree diameter, max path sum, and subtree counting problems
