Maximum Nesting Depth of the Parentheses

Given a valid parenthesization string s, return the maximum nesting depth of its parentheses.

Easy Stacks Track Max Open Count Time: O(n) Space: O(1)
1 Understanding the Problem ▾

The nesting depth of a parenthesization string is the maximum number of open parentheses that are "simultaneously active" at any point as you read left to right. Think of it as how deep you are inside nested parentheses at the deepest moment.

💡 Key Insight

Nesting depth = the maximum number of unmatched open parentheses seen at any single moment while scanning left-to-right. Every ( increases the depth by 1; every ) decreases it. The answer is the peak value of this running counter.

What to ignore: The string may contain digits, operators, letters, and other characters. Only ( and ) affect nesting depth. Everything else is depth-neutral.

Edge Cases: empty string (depth 0), no parentheses at all (depth 0), single pair "()" (depth 1), deeply nested "(((())))" (depth 4), adjacent groups "()()" (depth 1 — they don't add).

2 The Algorithm: Single-Pass Open Counter ▾

We don't need an actual stack. We only need to count how many open brackets are currently live. At each (, the count goes up; at each ), it comes down. The maximum value this count ever reaches after an increment is our answer.

1 Initialize two variables: open = 0 — the running count of currently unmatched open brackets. max_open = 0 — the peak value seen so far (our answer).
2 Scan each character c left-to-right: Only ( and ) need action. Digits, letters, operators — skip entirely.
3 On '(': First open += 1, then max_open = max(max_open, open). We increment FIRST because the depth increases the moment this bracket opens. The new depth is the candidate maximum.
4 On ')': Do open -= 1. Depth just decreased — this can never be a new maximum, so no max update needed.
5 Return max_open. The problem guarantees a valid VPS (valid parenthesization string), so open will always return to 0 at the end.
🔑 Why Not Use a Real Stack?

A traditional stack-based matching stores each open bracket and pops on close. The stack's length at any moment equals the current depth. But since every element on the stack is identical (just '('), we never need to look at individual elements — only the count matters. A single integer replaces the entire stack, reducing space from O(n) to O(1).

3 Why It Works: The Invariant ▾
🔐 The Invariant

After processing character at position i, open equals the number of open brackets that have been opened but not yet closed. This is exactly the current nesting depth — how many layers of parentheses we are currently inside.

Why increment before updating max: When we encounter (, the depth increases the moment that bracket is opened. The new nesting level starts at that character, not the next one. So we must first raise open to reflect the new depth, then check if it's a new record.

Why we never update max on ): A closing bracket decreases depth. The resulting value of open is strictly less than the value it was before the decrement. If the pre-decrement value was already ≤ max_open, the post-decrement value certainly is too. No update needed — proven by the monotonically non-increasing nature of closing events.

⚠️ Common Mistake: Max Before Increment

Writing max_open = max(max_open, open); open += 1 captures the depth before the bracket opens — which is the depth of the enclosing layer, not this layer. You'd always be one step behind, missing the actual maximum. Always increment first.

Non-parenthesis characters: Any character that isn't ( or ) leaves open unchanged, so the invariant is trivially preserved across them. The algorithm's correctness doesn't depend on what those characters are.

💡 Analogy: Elevator Floor Counter

Imagine a building where ( means "go up one floor" and ) means "go down one floor." The nesting depth question asks: what is the highest floor you ever reached? You don't need to remember every floor change — just track your current floor and the highest floor you ever touched.

4 The Code (Annotated) ▾

Solution 1 — Open Counter (Optimal, O(n) time, O(1) space) — Interview Pick

def maxDepth(s: str) -> int:
open = 0 ← count of currently unmatched '(' brackets
max_open = 0 ← peak depth ever seen — this is our answer
for c in s: ← single left-to-right pass; O(n) where n = len(s)
if c == '(': ← new open bracket: nesting level increases
open += 1 ← increment FIRST — depth is the new (higher) value
max_open = max(max_open, open) ← then check if it's a new peak
elif c == ')': ← matching close bracket: nesting level decreases
open -= 1 ← strictly less than before — can't be a new max
return max_open ← peak depth across the entire string
⚠️ Why elif Instead of else for ')'?

The string may contain non-parenthesis characters (digits, operators, etc.). Using elif c == ')' explicitly means we only decrement on actual close brackets. If we used else, a stray character would also decrement open — breaking the invariant. Always be explicit about what triggers state changes.

Solution 2 — Boolean Arithmetic (Pythonic, same complexity)

def maxDepth(s: str) -> int:
depth, ans = 0, 0
for c in s:
depth += (c == '(') - (c == ')') ← +1 for '(', -1 for ')', 0 for others
ans = max(ans, depth) ← safe: after ')' depth is lower, no new max
return ans

Solution 3 — Explicit Stack (Brute Force, O(n) time, O(n) space)

def maxDepth(s: str) -> int:
stack, max_depth = [], 0 ← actual stack of open brackets
for c in s:
if c == '(':
stack.append('(') ← push onto stack
max_depth = max(max_depth, len(stack)) ← stack length = current depth
elif c == ')':
stack.pop() ← match and remove
return max_depth
💡 Complexity Summary

Counter (Optimal): Time O(n), Space O(1) — best possible
Boolean arithmetic: Time O(n), Space O(1) — same, slightly different style
Explicit Stack: Time O(n), Space O(n) — use as stepping stone to explain the O(1) optimization

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Parentheses Pattern

"This involves parentheses — the classic tool is a stack. I'll start there: push on (, pop on ). The stack depth at any point is the nesting depth. Track the maximum."

Step 2: Propose the Stack Solution and Its Cost

"My initial approach uses an explicit stack: O(n) time, O(n) space. Every ( pushes an element, every ) pops one. The length of the stack is the current depth. I track max(len(stack))."

Step 3: Optimize — Notice What the Stack Actually Stores

"Every element I push is identical — just '('. I never inspect individual elements; I only care about the stack's size. So I can replace the entire stack with a single integer counter. O(1) space."

Step 4: Handle Non-Parenthesis Characters

"The VPS may contain other characters. I use elif c == ')' (not else) to ensure I only act on actual brackets. Non-bracket characters are depth-neutral — skip them."

🗣️ What to Say to the Interviewer

"I'll use a counter 'open' for currently active open brackets and 'max_open' for the peak.
Scan left to right. On '(': open += 1, then max_open = max(max_open, open).
On ')': open -= 1. Closing can't create a new max — depth strictly decreased.
Other characters: skip. Return max_open.
This is O(n) time, O(1) space. The key insight: I replaced the stack with its own size."
🔮 Likely Follow-up Questions
  • "What if the input may be invalid?" → If open ever goes below 0, the string is malformed. Return -1 or raise an exception.
  • "What if there are also brackets [ ] and { }?" → Three separate counters, or a dict mapping each open bracket type to its count.
  • "Remove brackets to limit depth to k? (LC 1614 variant)" → Greedy: scan left-to-right; when opening would exceed k, skip it and its matching close.
  • "Maximum depth of a valid parentheses substring?" → Same counter; reset to 0 when you hit an invalid sequence (extends to LC 32 territory).
