LeetCode #225

Implement Stack using Queues

Implement a last-in-first-out (LIFO) stack using only queue operations, supporting push, pop, top, and empty.

Easy Data Structure Design One Queue, Rotate Time: O(n) push Space: O(n)
1 Understanding the Problem ▾

A Stack is LIFO (Last-In-First-Out): the most recently pushed item comes out first. A Queue is FIFO (First-In-First-Out): the oldest item in comes out first. They order elements in opposite directions.

The constraint: implement push(x), pop(), top(), and empty() using only queue operations — enqueue (append), dequeue (popleft), peek front, size, and isEmpty. No arrays with random access allowed.

💡 Key Insight

A queue hands you elements from the front; a stack needs you to access the most recent element. The trick: after pushing a new element to the back, rotate all older elements behind it so the new element ends up at the front. Then every dequeue is a valid LIFO pop. Only one queue needed.

Queue Alone (FIFO — wrong for stack)

Enqueue: 1, 2, 3
Queue front → back: [1, 2, 3]
Dequeue → 1 (oldest pushed)
✗ Stack needs 3 (newest)

After Rotate-on-Push (LIFO — correct)

push(1): queue = [1]
push(2): rotate → [2, 1]
push(3): rotate → [3, 2, 1]
Dequeue → 3 ✓ LIFO!

Constraints & edge cases: The problem guarantees pop/top are only called on a non-empty stack. Edge cases include a stack with a single element (pop returns it immediately), and rapid push/pop alternations that must maintain the invariant throughout.

2 The Algorithm: One Queue, Rotate on Push ▾

We maintain one queue with the invariant: the front always holds the stack's top element. We pay the O(n) cost on every push to maintain this invariant, making pop and top trivially O(1).

Stack Behaviour
LIFO via rotation
1 push(x) — O(n): Enqueue x at the back. Then rotate: for each of the n−1 elements that were already in the queue, dequeue from front and re-enqueue at back. After n−1 rotations, x has moved from back to front — it is now the stack top.
2 pop() — O(1): The front of the queue is the stack top. Call popleft() and return it. No rotation needed — the next-most-recent element is already at the new front.
3 top() — O(1): Return queue[0] without removing it. The front is the stack top by our invariant.
4 empty() — O(1): Return whether the queue has zero elements. An empty deque is falsy in Python, so return not self.q works cleanly.
🔑 Two-Queue Alternative (symmetric approach)

Use two queues: push goes into the idle queue, then drain the active queue into it, then swap. Also O(n) push, O(1) pop. The one-queue rotate approach is preferred in interviews for simplicity and lower constant factors.

3 Why It Works: The Invariant ▾
🔐 The Invariant

After every push completes, the queue front holds the most recently pushed element. The queue read front-to-back mirrors the stack read top-to-bottom: most-recent first, oldest last.

Why rotation works: When we push x, it enters at position n (back). There are n−1 elements in front of it (positions 0 through n−2). We cycle each of those n−1 elements through the rotation: dequeue from front, enqueue at back. After n−1 such moves, x (which didn't move) is now at position 0 — the front. The relative order of the older elements is preserved among themselves; they just moved as a group to the right of x.

Why pop() needs no rotation: After a pop, the second-most-recently-pushed element is now at position 0 — it was at position 1 before the pop, and the dequeue of position 0 shifted it there. The invariant is automatically maintained without any extra work.

⚠️ Off-by-One in the Rotation Loop

The loop is range(len(self.q) - 1) — not range(len(self.q)). After appending x, the queue has n elements. We rotate n−1 of them (the old elements). If we rotated all n, we would cycle x all the way back to the back, undoing our work. Stop one short.

⚠️ Why not deque.appendleft() instead of rotating?

appendleft() is not a valid queue operation — it is a deque-specific method. The problem asks you to use only queue primitives (enqueue at back, dequeue from front). Using appendleft() would violate the problem's constraint.

4 The Code (Annotated) ▾
from collections import deque← O(1) popleft, unlike list.pop(0) which is O(n)
class MyStack:
def __init__(self):
self.q = deque()← single queue; invariant: front = stack top
def push(self, x: int) -> None:
self.q.append(x)← enqueue at back (x is not the top yet)
for _ in range(len(self.q) - 1):← rotate n-1 old elements behind x
self.q.append(self.q.popleft())← dequeue front → re-enqueue at back
def pop(self) -> int:
return self.q.popleft()← front = stack top → O(1) LIFO pop
def top(self) -> int:
return self.q[0]← peek front without removing → O(1)
def empty(self) -> bool:
return not self.q← empty deque is falsy → O(1)
⚠️ Use deque, NOT list, for the queue

Python's list.pop(0) is O(n) because it shifts every element left. deque.popleft() is O(1) using a doubly-linked structure internally. In an interview, always mention this distinction — it shows you understand underlying complexity, not just API names.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify the core conflict

Queue gives you the oldest element; stack needs the newest. The orderings are reversed. I need to reverse the order using only queue operations.

Step 2: Decide where to pay the O(n) cost

Reversing n elements always costs O(n). I can pay it at push time (makes pop/top O(1)) or at pop time (makes push O(1) but needs two queues). For this problem, paying at push is cleaner with one queue.

Step 3: Derive the rotation mechanics

After pushing x to the back, I need it at the front. The only queue move available is: dequeue from front, enqueue at back. If I do this for every element that was there before x, those elements cycle around, leaving x at front. That's n−1 rotations.

Step 4: Verify the invariant holds after pop

After pop removes the front (stack top), the second element is now at the front. That second element was the second-most-recently-pushed item — the correct new stack top. No extra work needed. The invariant is self-maintaining through pops.

🗣️ What to Say to the Interviewer

"Stack is LIFO; queue is FIFO — opposite orderings. I'll maintain
the invariant that the queue front always equals the stack top.
To do that, on every push I enqueue x at the back, then rotate
all n-1 existing elements to the back — this brings x to the front.
Pop and top then become trivial O(1) dequeue and peek operations.
Push is O(n), space is O(n). I only need one queue."
💡 Follow-up Questions to Expect
  • Can you do it with two queues? Yes — symmetric approach, same asymptotic complexity.
  • What if we want O(1) push and O(n) pop instead? Pay the rotation cost at pop using a second queue.
  • The reverse problem — Queue using Stacks (LC 232)? Two stacks, amortized O(1) dequeue — a classic companion.
  • Why not just use appendleft()? That's not a valid queue operation — it's a deque-specific method violating the constraint.

7 Summary & Key Takeaways ▾
Stack (LIFO) and Queue (FIFO) hand out elements in opposite order. The bridge is rotation: cycling older elements behind the new one.
Invariant: queue front = stack top. Maintaining this on push (O(n)) makes pop and top trivially O(1).
Use Python's collections.deque — popleft() is O(1). Using a list would make the "queue dequeue" O(n) and break the spirit of the constraint.
Loop is range(len(q) - 1) not range(len(q)). Rotating all n elements brings x back to the back; rotate only n-1 old elements.
The symmetric twin (LC 232 — Queue using Stacks) uses two stacks with amortized O(1) dequeue. Both are classic interview problems.
🎯 When to Recognise This Design Pattern
  • Asked to implement ADT X using only operations of ADT Y
  • The two ADTs have opposite element-access order (FIFO vs LIFO)
  • You must not use any data structure not in the permitted set
  • Interview problems saying "only push to back / pop from front" as valid ops
