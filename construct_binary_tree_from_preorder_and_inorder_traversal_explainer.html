LeetCode #105

Construct Binary Tree from Preorder and Inorder Traversal

Given preorder and inorder traversal arrays of a binary tree, reconstruct and return the original tree.

Medium Trees Find Root in Inorder Time: O(n) Space: O(n)
1 Understanding the Problem ▾

A binary tree can be fully reconstructed from its preorder and inorder traversal arrays — provided all node values are unique. You're given both arrays and must build the original tree.

💡 Key Insight: What Each Traversal Tells You

Preorder [Root, Left, Right]: The very first element is always the root of the current (sub)tree. As we recurse, the next unconsumed preorder element is always the next root.

Inorder [Left, Root, Right]: Once we know the root, we find it in the inorder array. Everything to its left belongs to the left subtree; everything to its right belongs to the right subtree. This tells us the sizes and membership of each subtree.

Example: preorder = [3, 9, 20, 15, 7], inorder = [9, 3, 15, 20, 7]

Edge cases: Single-node tree (root only), skewed trees (all nodes to one side), empty input (return null). Uniqueness of values is guaranteed — without it, reconstruction is ambiguous.

2 The Algorithm: Recursive Divide and Conquer ▾

This problem is a textbook case for divide-and-conquer recursion. Each call builds one node, then delegates left and right subtrees to recursive calls. We track the current preorder position with a shared pointer and use inorder boundary indices to know the extent of each subtree.

1 Pre-build an index map: Create a hash map idx_map[value] = inorder_index for every element. This enables O(1) root lookup instead of O(n) linear search per recursive call — reducing total time from O(n²) to O(n).
2 Take the next preorder element as root: Maintain pre_idx pointing to the next unconsumed preorder element. At each call, preorder[pre_idx] is the root of the current subtree. Advance pre_idx before recursing.
3 Locate root in inorder: Use idx_map to find mid — the root's index in the inorder array. Elements inorder[in_left..mid-1] form the left subtree; elements inorder[mid+1..in_right] form the right subtree.
4 Recurse left, then right: Build the left subtree first (this matches preorder structure — all left-subtree nodes precede right-subtree nodes in preorder). Then build the right subtree. Attach both as children of the current root.
5 Base case: If in_left > in_right, the range is empty — return None. This happens when a node has no left or right child.
🔑 Why Recurse Left Before Right?

In preorder, after the current root come ALL nodes of the left subtree, then ALL nodes of the right subtree. Our pre_idx pointer advances monotonically. If we built the right subtree first, pre_idx would be pointing at left-subtree elements — wrong assignment. Building left first consumes the correct preorder elements in the correct order.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every recursive call helper(in_left, in_right): preorder[pre_idx] is the root of the subtree whose inorder elements span exactly inorder[in_left..in_right]. After the call returns, pre_idx has advanced by exactly (in_right - in_left + 1) positions — one per node in that subtree.

Mathematical justification: A preorder and inorder sequence together uniquely determine a binary tree when all values are distinct. This is because:

  • The first preorder element uniquely identifies the root (one possible root)
  • Finding the root in inorder splits the remaining elements unambiguously into left and right groups
  • The size of the left group (mid - in_left) tells us exactly how many preorder elements belong to the left subtree
  • This reasoning applies recursively — every subtree is uniquely determined
⚠️ Why Uniqueness of Values Is Required

If duplicate values existed, finding the root in the inorder array would be ambiguous — multiple positions could match. We wouldn't know which occurrence is the actual root split point, leading to multiple valid (conflicting) trees. The problem guarantees unique values precisely to avoid this.

Edge case: When in_left > in_right, the range is empty. pre_idx does NOT advance — no preorder element is consumed for an empty subtree. Returning None correctly represents a missing child without corrupting the pointer state.

4 The Code (Annotated) ▾

Solution 1 — Optimal: Recursion with Hash Map (Interview Pick)

def buildTree(preorder, inorder):
idx_map = {v: i for i, v in enumerate(inorder)} ← Build once: value → inorder position (O(n))
pre_idx = [0] ← Mutable wrapper so inner function can advance it
def helper(in_left, in_right): ← Builds subtree for inorder[in_left..in_right]
if in_left > in_right: ← Empty range → no node; return None
return None
root_val = preorder[pre_idx[0]] ← Next preorder element IS the root here
pre_idx[0] += 1 ← Consume this root; advance pointer
root = TreeNode(root_val) ← Create the node
mid = idx_map[root_val] ← O(1) lookup: where is root in inorder?
root.left = helper(in_left, mid - 1) ← Left subtree: inorder slice BEFORE mid
root.right = helper(mid + 1, in_right) ← Right subtree: inorder slice AFTER mid
return root ← Return fully built subtree
return helper(0, len(inorder) - 1) ← Start with full range
⚠️ Why pre_idx = [0] (a list) instead of pre_idx = 0?

Python closures can read outer variables but cannot rebind them without the nonlocal keyword. Using a list (mutable container) lets us mutate pre_idx[0] in place and have the change visible across all recursive calls. Alternatively, declare nonlocal pre_idx at the top of helper and use a plain integer — both are valid and equally correct.

Solution 2 — Brute Force: Array Slicing (O(n²) — mention first, then optimize)

def buildTree(preorder, inorder):
if not preorder or not inorder: ← Base case: no nodes left
return None
root = TreeNode(preorder[0]) ← First preorder element = root
mid = inorder.index(preorder[0]) ← O(n) linear scan each call → O(n²) total
root.left = buildTree(preorder[1:mid+1], ← Slice creates new list → O(n) extra space/call
inorder[:mid])
root.right = buildTree(preorder[mid+1:],
inorder[mid+1:])
return root
💡 Complexity Summary

Optimal (Hash Map): Time O(n) — each of n nodes processed once; Space O(n) for hash map + recursion stack (O(h) where h = height; O(log n) avg, O(n) worst for skewed tree)
Brute Force (Slicing): Time O(n²) — O(n) linear search at every one of n nodes; Space O(n²) — new array copies at every recursion level
Takeaway: Pre-build lookups before recursion. Never scan inside recursion when you can precompute.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Understand What the Traversals Give You

Say: "In preorder the root is always first. In inorder the root divides left and right. So preorder gives me roots one by one, and inorder tells me who belongs left vs. right of each root." This framing naturally suggests recursion.

Step 2: Trace a Small Example by Hand

preorder=[3,9,20,15,7], inorder=[9,3,15,20,7]. "Root = 3 (preorder[0]). Find 3 in inorder: index 1. Left side = [9] (1 element). Right side = [15,20,7] (3 elements). Recurse." This hand-trace builds the algorithm live.

Step 3: Propose Brute Force, Then Optimize

"I can use inorder.index() to find the root — O(n) per call, O(n²) total. To optimize, I'll precompute a hash map from value to inorder index so each lookup is O(1). Total becomes O(n) time."

Step 4: Explain the pre_idx Trick

"Instead of slicing the preorder array, I maintain a single pointer that advances as we place each node. Because we recurse left before right, the pointer naturally walks through preorder in the correct order. No extra array copies needed."

🗣️ What to Say to the Interviewer

"Preorder always puts the root first — so preorder[0] is my root.
I find that root in the inorder array at position mid.
Everything left of mid in inorder is the left subtree; right of mid is right subtree.
I recurse on both halves. A shared pre_idx pointer advances through preorder naturally.
To avoid O(n) scans: precompute a hash map value→inorder_index for O(1) lookup.
Time: O(n). Space: O(n) for map plus recursion stack."
🔮 Likely Follow-up Questions
  • "What about inorder + postorder?" → Mirror: postorder's LAST element is root; read it right-to-left (#106)
  • "What about preorder + postorder?" → Unique only for full binary trees (#889)
  • "Can you do it iteratively?" → Yes, with a stack; same O(n) complexity but trickier to code
  • "What if values aren't unique?" → Ambiguous; problem becomes undetermined without extra constraints
  • "What's the recursion depth?" → O(h) where h is tree height; O(log n) average, O(n) worst (skewed tree)

7 Summary & Key Takeaways ▾
Preorder's first element is always the root of the current subtree
Finding root in inorder splits nodes into left vs. right subtrees
Pre-build an index map for O(1) lookups — converts O(n²) → O(n)
Recurse LEFT before RIGHT to stay in sync with preorder's structure
Use boundary indices (not slicing) to avoid O(n²) space overhead
Two traversals + unique values = unique reconstruction (fundamental theorem)

🎯 When to Recognize "Find Root in Inorder"
  • Problem gives two traversals of the same binary tree
  • One traversal is preorder or postorder (provides roots)
  • Other traversal is inorder (partitions left/right)
  • Asked to reconstruct the tree structure
  • Each recursive call needs to know: "which nodes belong to my subtree?"
