LeetCode #1762

Buildings With an Ocean View

Find every building that has an unobstructed view of the ocean to its right.

Medium Monotonic Stack Decreasing from Right Time: O(n) Space: O(1) extra
1 Understanding the Problem ▾

Imagine a row of buildings standing side-by-side, all facing the same direction. The ocean is to the right. A building has an ocean view if and only if every building to its right is strictly shorter than it. If even one taller-or-equal building stands between it and the ocean, the view is blocked.

💡 Key Insight

Building i has an ocean view if and only if heights[i] > max(heights[i+1 .. n-1]). Rephrased: it must be strictly taller than the tallest building anywhere to its right. This single condition replaces checking every individual right-side building.

✓ heights = [4, 2, 3, 1]

idx 3 (h=1): rightmost → view ✓
idx 2 (h=3): max_right=1, 3>1 → view ✓
idx 1 (h=2): max_right=3, 2<3 → blocked ✗
idx 0 (h=4): max_right=3, 4>3 → view ✓
Answer: [0, 2, 3]

✗ heights = [1, 2, 3, 4]

idx 3 (h=4): rightmost → view ✓
idx 2 (h=3): max_right=4, 3<4 → blocked ✗
idx 1 (h=2): max_right=4, 2<4 → blocked ✗
idx 0 (h=1): max_right=4, 1<4 → blocked ✗
Answer: [3] — only rightmost!

Edge cases: single building (always has view); all same height (only rightmost); strictly decreasing sequence (every building); equal-height pair (left one is blocked — same height does block).

2 The Algorithm: Right-to-Left Scan with max_right ▾

The core idea: scan from the ocean inward (right to left). As we walk left, we carry the running maximum height of everything we have already seen to the right. This gives us an O(1) answer for each building: "Is this building taller than everything to its right?"

1 Initialize: max_right = 0 and result = []. max_right tracks the tallest building seen so far (to the right of our current position). Starts at 0 because the rightmost building has nothing to its right.
2 Scan right-to-left (i = n-1 down to 0). At each building, ask: is heights[i] > max_right? If yes, this building is strictly taller than everything to its right — ocean view!
3 Collect index: If heights[i] > max_right, append i to result. Then regardless, update max_right = max(max_right, heights[i]) before moving further left.
4 Reverse and return: We collected indices right-to-left (descending). Reverse to get ascending order: return result[::-1].
🔑 Why Right-to-Left Is the Natural Direction

If we scan left-to-right, to answer "is this building the tallest to its right?" we need the suffix maximum — which requires a pre-pass or looking ahead. Scanning right-to-left lets us build that suffix maximum incrementally as we go. We already know everything to the right when we arrive at index i.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At the start of iteration for index i, max_right equals exactly max(heights[i+1], heights[i+2], ..., heights[n-1]) — the suffix maximum of everything to i's right. This invariant holds throughout by induction: at i = n-1, no elements are to the right, so max_right = 0 is correct. After processing i, we update max_right to include heights[i], making it correct for i-1.

Strict inequality matters: We use > (strictly greater than), not >=. A building of equal height does block the view — the problem says "strictly taller." So if two adjacent buildings have height 5, the left one is blocked.

Completeness — we never miss a view building: Every building with a view satisfies heights[i] > max_right at the moment we visit it. Since max_right is exactly the suffix maximum, the condition fires precisely for ocean-view buildings and nowhere else. No false positives, no false negatives.

⚠️ Common Mistake: Updating max_right Before Checking

If you write max_right = max(max_right, heights[i]) before the if heights[i] > max_right check, you include the current building in the suffix maximum, which is wrong — a building should be compared against everything strictly to its right, not including itself. Always check first, then update.

4 The Code (Annotated) ▾

Solution 1 — max_right scan (Interview Pick, O(n) time, O(1) extra space)

def findBuildings(heights):
n = len(heights) ← total number of buildings
result = [] ← will hold ocean-view indices (collected right-to-left)
max_right = 0 ← suffix max; starts 0 since rightmost has nothing to its right
for i in range(n - 1, -1, -1): ← scan from rightmost building leftward
if heights[i] > max_right: ← strictly taller than all buildings to the right → ocean view
result.append(i) ← record this index
max_right = max(max_right, heights[i]) ← update suffix max AFTER the check
return result[::-1] ← reverse: collected descending, need ascending
⚠️ Why result[::-1] instead of inserting at the front?

We could use result.insert(0, i) to keep it sorted during collection, but that is O(k) per insertion — O(k²) total in the worst case. Appending and reversing at the end is O(k) total. For large inputs, this difference is significant.

Solution 2 — Monotonic Stack (explicit pattern, O(n) time, O(n) extra space)

def findBuildings(heights):
stack = [] # heights of right-side buildings ← monotonic decreasing stack
result = []
for i in range(len(heights) - 1, -1, -1): ← right to left
while stack and stack[-1] <= heights[i]: ← pop dominated (shorter-or-equal) buildings
stack.pop()
if not stack: ← no taller building to the right → ocean view
result.append(i)
stack.append(heights[i]) ← push current to maintain decreasing order
return result[::-1] ← reverse to ascending

Solution 3 — Brute Force O(n²) (understand, don't submit)

def findBuildings(heights):
n, result = len(heights), []
for i in range(n): ← try each building
has_view = all(heights[j] < heights[i] ← check every building to the right
for j in range(i+1, n))
if has_view: result.append(i)
return result
🔬 Monotonic Stack — Algorithm Deep-Dive

What it is: A stack that enforces a strict ordering (monotonically increasing or decreasing) by popping elements that violate the invariant before each push. Used classically for "next greater/smaller element" problems.

Core invariant: After processing index i (right-to-left), the stack contains — bottom to top — a strictly decreasing sequence of heights from buildings to the right of i that could potentially block a future building. Any building that gets popped was "dominated" by the current building and can never block anything the current building doesn't already block.

Time complexity: Each element is pushed once and popped at most once → O(n) amortized total, despite the while loop inside the for loop.

Recognize when: "For each element, find the next/previous element that is strictly greater/smaller" — or — "filter elements where only comparisons from one direction matter." This generalizes to "first taller building to the right" (where stack holds indices, not heights).

💡 Complexity Summary

max_right (interview pick): Time O(n), Space O(1) extra (output list is O(k))
Monotonic Stack: Time O(n) amortized, Space O(n) for stack
Brute Force: Time O(n²), Space O(1) — fails for large n

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Restate the Problem Precisely

"Building i has an ocean view iff heights[i] is strictly greater than all heights[j] for j > i. I need all such indices in sorted order."

Step 2: Propose Brute Force

"For each building, scan everything to its right to find the maximum height. O(n²). I'd precompute suffix maximums in a second pass to bring it to O(n) time and O(n) space."

Step 3: Discover the O(1)-Space Insight

"If I scan right-to-left, I can build the suffix maximum incrementally — no pre-pass needed. By the time I reach index i, max_right already holds the suffix max of everything to its right. That eliminates the O(n) space for the suffix array."

Step 4: Mention the Monotonic Stack Extension

"If the interviewer asks 'which specific building is the first blocker for each building?' — that's the monotonic stack's domain. The max_right trick only tells you if there is a blocker, not which one."

🗣️ What to Say to the Interviewer

"A building has an ocean view iff it is strictly taller than everything to its right.
I'll scan right-to-left, tracking max_right = tallest building seen so far.
For each building: if heights[i] > max_right → ocean view, append i to result.
Then update max_right. After the loop, reverse result for ascending order.
Time O(n), Space O(1) extra. Want me to code it up?"
🔮 Likely Follow-up Questions
  • "What if the ocean is to the left?" → Scan left-to-right, track max_left.
  • "Return view distances instead of indices?" → Use monotonic stack with indices to find the first taller building (#739 style).
  • "What if buildings can move?" → Dynamic version needs a segment tree or sorted set.
  • "Heights have ties — does that matter?" → Yes! Equal height blocks view. Strict > handles it correctly.

7 Summary & Key Takeaways ▾
Scan from the ocean direction (right to left) — this is always the natural approach for "view from the right" problems
max_right = suffix maximum built on-the-fly; O(1) extra space, no pre-pass needed
Check heights[i] > max_right (strict) — equal height still blocks
Update max_right AFTER checking, not before (or you include the current building)
Collect indices right-to-left and reverse at end — O(k) vs O(k²) for front-inserts
Monotonic stack generalizes to "first blocker" queries but adds O(n) space
