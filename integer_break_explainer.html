LeetCode #343

Integer Break

Given an integer n ≥ 2, break it into k ≥ 2 positive integers that sum to n, and return the maximum product of those integers.

Medium Dynamic Programming Max Product Partition Time: O(n²) Space: O(n)
1 Understanding the Problem ▾

You are given a positive integer n. Break it into at least two positive integers that sum to n and return the maximum product of those parts. You choose both how many pieces and the size of each — the only rules are all pieces are positive integers and they sum to n.

💡 Key Insight

The product is maximised when you use 3s as much as possible. Mathematically, the optimal base is e ≈ 2.718, and the closest integer is 3. You never want 1s (they don't help the product) and you never want factors ≥ 5 (they can always be split into smaller pieces that multiply higher). The DP approach rediscovers this truth automatically, bottom-up, without needing to know the math in advance.

✓ n = 10, best split

10 = 3 + 3 + 4
Product = 3 × 3 × 4 = 36
(or 3 + 3 + 2 + 2 = 36)
Not: 5 + 5 = 25 ✗

✗ Common mistake: using 1s

7 = 1 + 1 + 1 + 4
Product = 1×1×1×4 = 4 ✗
7 = 3 + 4
Product = 3×4 = 12 ✓

Edge cases: n = 2 → only split is 1+1, answer is 1. n = 3 → best is 1+2 = 2 (not 1+1+1 = 1). For any n ≥ 4, we never include 1s in the optimal split. Numbers 4+ always benefit from further splitting.

🔑 Why This Is DP (Two Pillars)

Optimal Substructure: The best split of n uses j + remainder where the remainder is itself split optimally. "Optimally split remainder" = dp[remainder]. So the global optimum is built from local optima.
Overlapping Subproblems: dp[6] is needed when computing dp[8], dp[9], dp[10], etc. Without a cache, these would be recomputed exponentially many times.

2 The Recurrence & State Machine ▾

State definition: dp[i] = the maximum product achievable by splitting integer i into at least two positive parts.

📐 The Recurrence Relations
dp[i] = max over j from 1 to i−1 of:
    max( j × (i − j),   j × dp[i − j] )

Option A: j × (i−j) — take j, keep remainder intact (one piece)
Option B: j × dp[i−j] — take j, split remainder optimally

Why do we need both options? dp[i-j] represents splitting the remainder further — but dp[2] = 1 (forced 1+1 split), while keeping 2 intact as a factor gives just 2. When computing dp[4] with j=2: option A gives 2×2=4, option B gives 2×dp[2]=2×1=2. Option A wins! This is why you always take max(j×(i-j), j×dp[i-j]).

3 Why It Works: The Invariant ▾
🔐 The Invariant

When we compute dp[i], every cell dp[1] through dp[i-1] already holds the correct maximum product for that sub-integer. This means the inner loop over j correctly computes all possible first cuts of i, and looking up dp[i-j] gives the exact optimal sub-answer.

Proof sketch: We proceed by strong induction. Base: dp[2] = 1 is correct (only 1+1=1). Inductive step: assume dp[k] is correct for all k < i. For dp[i], the optimal split uses some first piece j*. Either the remainder (i−j*) stays as one piece → product = j* × (i−j*), or it's split further → product = j* × dp[i−j*]. Since dp[i−j*] is correct by hypothesis, taking max over all j gives the globally optimal answer for i.

Why 3 is special (the math behind the pattern): Consider splitting n into k equal parts of size n/k. The product is (n/k)^k. Taking the derivative with respect to k and setting to 0 gives the optimal piece size = e ≈ 2.718. Since pieces must be integers, 3 beats 2 for the same budget (3+3=6 gives 3×3=9, while 2+2+2=6 gives 2×2×2=8). Factor 4 ties with 2+2 (4=2×2), and any factor ≥ 5 should be broken further.

⚠️ Common Trap: Forgetting Both Options

Writing only j × dp[i-j] fails on small cases. Example: computing dp[4] with j=2: 2×dp[2]=2×1=2, but the correct answer is 4 (split 2+2, keeping both as intact pieces → 2×2=4). Always include max(j×(i-j), j×dp[i-j]).

Edge case handling: n=2: only j=1 is valid, keep=1×1=1, dp[2]=1. n=3: j=1 gives 1×2=2 (win), j=2 gives 2×1=2 (same). dp[3]=2. These small cases feed correctly into larger ones.

4 The Code (Annotated) ▾

Solution 1 — Bottom-Up Tabulation (Interview Pick, O(n²) time, O(n) space)

def integerBreak(n: int) -> int:
dp = [0] * (n + 1) ← dp[i] = max product when breaking integer i into ≥2 parts
for i in range(2, n + 1): ← start from 2 (dp[0]=dp[1]=0 already correct as base)
for j in range(1, i): ← try every possible first piece j (from 1 to i-1)
keep = j * (i - j) ← option A: keep remainder (i-j) as one unsplit number
split = j * dp[i - j] ← option B: split remainder further using the DP table
dp[i] = max(dp[i], keep, split) ← update dp[i] with the better of current, keep, split
return dp[n] ← answer is the maximum product for the full integer n

Solution 2 — Top-Down Memoization (same complexity, recursive style)

Solution 3 — Greedy / Math (O(n) time, O(1) space — mention as bonus)

def integerBreak(n: int) -> int:
if n == 2: return 1 ← 1+1 is the only valid split
if n == 3: return 2 ← 1+2 = product 2 (not 1+1+1 = 1)
product = 1
while n > 4: ← keep extracting 3s while more than 4 remains
product *= 3; n -= 3 ← 3 beats any other factor over the same budget
return product * n ← remainder is 2, 3, or 4 — all fine as-is
⚠️ Why the loop stops at n > 4

Remaining n=2: 2 is optimal (don't split). n=3: 3 is optimal. n=4: 4 = 2+2, and 2×2=4=4, so keeping 4 is equivalent — don't split. n=5 or more: 5=2+3, 2×3=6>5, so always split n≥5. That's why we stop when n≤4.

💡 Complexity Summary

Tabulation: Time O(n²) — two nested loops to n. Space O(n) for dp array.
Memoization: Same O(n²) time, O(n) space for memo dict + call stack.
Greedy/Math: Time O(n), Space O(1) — but requires the 3-insight as prior knowledge.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognise the DP signal

The problem asks to maximise a product over all ways to partition a number. "Maximise/minimise over combinatorial choices where sub-answers overlap" = DP fingerprint. Say this explicitly: "I see overlapping subproblems, so I'll use DP."

Step 2: Define the state clearly

State: dp[i] = maximum product when we split integer i into at least two positive pieces. This is the hardest step of any DP — nail it out loud before writing code.

Step 3: Write the recurrence, explain both options

For each first piece j from 1 to i−1: take max of j×(i-j) (keep remainder) and j×dp[i-j] (split remainder further). Explain why both are needed with a concrete example like dp[4].

Step 4: Mention greedy as an O(1) space bonus

After the DP solution, observe that the dp table reveals a pattern: always prefer 3s. Offer the greedy O(n)/O(1) approach as a follow-up optimisation — shows mathematical depth.

🗣️ What to Say to the Interviewer

"I see this is a maximisation over all integer partitions — classic DP candidate.
State: dp[i] = max product when breaking i into ≥2 positive pieces.
Transition: for each split point j from 1 to i-1:
dp[i] = max(dp[i], j×(i-j), j×dp[i-j])
Both options needed — dp[i-j] forces a further split, but the remainder as-is can be larger.
Base: dp[1]=0, dp[2]=1. Fill left to right. Time O(n²), Space O(n).
Bonus: the dp table reveals that 3s dominate — O(n)/O(1) greedy is also possible."
🔮 Likely Follow-up Questions
  • "Can you do O(1) space?" → Yes, greedy: use 3s, keep remainder of 2, 3, or 4.
  • "What if pieces must be at most k?" → Filter inner loop to j ≤ k.
  • "What if exactly k pieces required?" → 2D DP: dp[i][k] = max product splitting i into exactly k parts.
  • "Why is 3 optimal?" → Calculus: maximise (n/k)^k → optimal piece size = e ≈ 2.718, nearest integer is 3.
