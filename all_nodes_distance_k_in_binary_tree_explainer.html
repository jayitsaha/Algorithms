LeetCode #863

All Nodes Distance K in Binary Tree

Given a binary tree, a target node, and an integer K, return all node values that are exactly K edges away from the target.

Medium Graph Build Graph + BFS Time: O(n) Space: O(n)
1 Understanding the Problem ▾

Imagine you're standing at a node in a binary tree — call it the target. You want to find every node that is exactly K hops away, where one hop means moving along any single edge. The challenge: tree edges are normally one-directional (parent → child). To reach nodes K steps away, you might need to travel upward to a parent or ancestor, then laterally into a sibling subtree. Standard tree DFS can't do that without extra bookkeeping.

💡 Key Insight

A binary tree is just a special case of an undirected graph. If you add parent pointers (treating every tree edge as bidirectional in an adjacency list), BFS from the target naturally finds all nodes at exactly distance K — because BFS explores one layer of edges per round.

Example: target=5, K=2

Tree: 3→(5,1), 5→(6,2),
1→(0,8), 2→(7,4)

From node 5, distance 2:
7 → via 5→2→7
4 → via 5→2→4
1 → via 5→3→1 (UP!)
Answer: [7, 4, 1]

Edge Cases: K=0 returns only the target itself. If target is the root, all answers are in subtrees (no upward travel possible). If K exceeds the farthest node distance, return []. All node values are guaranteed unique (per LeetCode constraints).

2 The Algorithm: Build Undirected Graph + BFS ▾

The strategy is a two-phase approach: Phase 1 converts the tree into an undirected graph by DFS-adding parent edges to an adjacency list. Phase 2 runs standard BFS from the target node, stopping after exactly K levels.

1 Build adjacency list via DFS: Traverse the entire tree once. For every parent-child pair, add both directions: adj[child].append(parent.val) and adj[parent].append(child.val). After this pass, moving "upward" in the tree is now a regular neighbor lookup.
2 BFS from target node: Initialize a deque with target.val and a visited set containing target.val. Process the queue level by level — each level represents one additional edge of distance from the target.
3 Stop at distance K: After exactly K rounds of BFS expansion, exit the loop. All remaining nodes in the queue are at distance exactly K from the target node.
4 Return the queue contents: return list(queue). If K=0, this is just the target. If no nodes exist at distance K (too far), the queue empties before K rounds and we return [].
🔑 Why BFS and not DFS for K-distance?

BFS explores nodes in order of increasing distance — level 0, then all level-1 nodes, then all level-2, etc. This makes it natural to stop at exactly level K. DFS would require accumulating distance along each path and checking at each node, with more bookkeeping and potential revisiting overhead.

3 Why It Works: The Invariant ▾
🔐 The BFS Distance Invariant

At the start of BFS level d, every node currently in the queue is exactly d edges away from the target. This invariant holds because BFS expands all distance-d nodes before enqueueing any distance-(d+1) nodes. When we've completed exactly K rounds, the queue holds precisely the distance-K nodes.

Why the visited set is mandatory: We converted the tree to an undirected graph — every edge now appears in both directions. Without a visited set, BFS would oscillate: from node 5 we'd visit parent 3; from 3 we'd try to visit 5 again, and so on forever. The visited set ensures each node is processed at its first-reached (shortest) distance.

Correctness of the graph conversion: The original tree has exactly n−1 edges. Adding both directions creates a symmetric adjacency list with 2(n−1) entries total — still a tree structure (acyclic, connected). BFS on this symmetric structure finds correct shortest paths because trees have unique paths between any two nodes.

⚠️ Common Mistake: Forgetting the Visited Set

It feels like "I only added parent edges, so there can't be infinite loops." Wrong! From BFS's perspective, going child→parent→child→parent IS a cycle if you allow re-visiting. The visited set prevents this and is not optional in any undirected graph BFS.

4 The Code (Annotated) ▾

Solution 1 — Build Undirected Graph + BFS (Optimal, Interview Pick)

from collections import defaultdict, deque
def distanceK(root, target, k):
adj = defaultdict(list) ← adjacency list: val → [neighbor vals]
def build(node, parent): ← DFS to populate undirected adj list
if not node: return
if parent: ← skip for root (no parent)
adj[node.val].append(parent.val) ← upward edge: child → parent
adj[parent.val].append(node.val) ← downward edge: parent → child
build(node.left, node) ← recurse left, node is now parent
build(node.right, node) ← recurse right, node is now parent
build(root, None) ← start DFS from root with no parent
queue = deque([target.val]) ← BFS from target node's value
visited = {target.val} ← mark target visited immediately
dist = 0 ← current distance level from target
while queue and dist < k: ← keep expanding until we hit level k
for _ in range(len(queue)): ← snapshot: process ALL nodes at current level
node = queue.popleft() ← dequeue one node from current level
for nb in adj[node]: ← visit all neighbors (children + parent)
if nb not in visited: ← only visit each node once
visited.add(nb)
queue.append(nb) ← enqueue for NEXT level
dist += 1 ← finished current level; one step farther
return list(queue) ← after k rounds, queue = all distance-k nodes
⚠️ Why use node.val as keys, not node objects?

LeetCode guarantees all values are unique. Using integer values as keys keeps the adjacency list simple and human-readable. If values could repeat, you'd need to use the TreeNode object itself (or id(node)) as the dictionary key.

Solution 2 — Pure DFS with Distance Tracking (Alternative, O(h) space)

def distanceK_dfs(root, target, k):
res = []
def collect(node, dist): ← gather nodes exactly (k-dist) steps below node
if not node or dist > k: return
if dist == k: res.append(node.val); return
collect(node.left, dist + 1)
collect(node.right, dist + 1)
def dfs(node): ← returns dist from node to target, -1 if not found
if not node: return -1
if node == target: ← found target; collect k steps downward
collect(node, 0); return 0
left = dfs(node.left) ← check left subtree
if left != -1: ← target found in left subtree
collect(node.right, left+2) ← explore right branch at adjusted distance
if left+1 == k: res.append(node.val)
return left + 1
right = dfs(node.right) ← check right subtree
if right != -1:
collect(node.left, right+2)
if right+1 == k: res.append(node.val)
return right + 1
return -1
dfs(root); return res
💡 Complexity Summary

Build Graph + BFS (Solution 1): Time O(n) — one full DFS to build + one BFS pass. Space O(n) for adj list, visited set, queue.
Pure DFS (Solution 2): Time O(n) — each node visited at most twice. Space O(h) for recursion stack where h = tree height. Avoids the adjacency list but the logic is significantly harder to reason about and explain.
Interview choice: Solution 1. Cleaner structure, more transferable to other graph problems, and easier to explain the BFS-level intuition.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Spot the "distance in all directions" signal

The phrase "distance K from a target node" without the restriction "only downward" is the key tell. Distance problems that allow upward traversal in a tree always benefit from treating the tree as an undirected graph. Name it explicitly to the interviewer.

Step 2: Identify the two obstacles

"Two things make this hard: (1) trees have no parent pointers — you can't go upward, and (2) some K-distance nodes might require crossing the root. Both problems vanish if we build an undirected adjacency list first, adding parent edges during a DFS."

Step 3: Propose the two-phase plan

"Phase 1: One DFS to build adj[]. For each node, add bidirectional edges. Phase 2: BFS from target.val for exactly K rounds. After K rounds, whatever's in the queue is the answer. Both phases are O(n)."

Step 4: Write clean level-by-level BFS

Level-by-level BFS is the pattern: snapshot queue length at round start with for _ in range(len(queue)), then process exactly that many nodes. After finishing the loop body, increment dist. This cleanly separates levels without extra bookkeeping.

🗣️ What to Say to the Interviewer

"The challenge here is that distance K from a node can go upward through the tree — which standard tree traversal doesn't support. My approach: first, convert the tree to an undirected graph by running one DFS and adding both directions (parent↔child) to an adjacency list. Then, BFS from the target node for exactly K rounds. After K rounds, everything still in the queue is at distance K. Time O(n), Space O(n). The visited set prevents oscillation through the symmetric edges we just added."
🔮 Likely Follow-up Questions
  • "What if node values aren't unique?" → Use node references (objects) as adjacency keys instead of values
  • "Can you avoid the extra O(n) space for the adjacency list?" → Yes, the pure DFS solution (Solution 2) uses only O(h) recursion stack
  • "What if K is larger than any distance in the tree?" → BFS exhausts all nodes and queue empties before K rounds; return []
  • "Find nodes at distance ≤ K instead of exactly K?" → Return all nodes visited during BFS up to and including round K

7 Summary & Key Takeaways ▾
Trees can be treated as undirected graphs by adding parent pointers via DFS — this unlocks upward traversal
BFS naturally measures distance: after K rounds, the queue holds exactly all distance-K nodes
Level-by-level BFS uses for _ in range(len(queue)) to snapshot the current level before expanding
Always use a visited set in undirected graph BFS — symmetric edges create apparent cycles for the BFS traversal
Use node.val as keys only when values are unique; otherwise use node object references
Both DFS build phase and BFS phase are O(n) — overall O(n) time and O(n) space

🎯 When to Recognize the Build Graph + BFS Pattern
  • "Distance from a node in all directions" on a tree → convert to undirected graph
  • Need to traverse upward in a tree (toward root or ancestors) → add parent edges to adj list
  • "Find all nodes exactly K hops away" → BFS for exactly K levels
  • Unweighted shortest path on a graph → BFS (not DFS)
