LeetCode #120

Triangle

Given a triangular array of integers, find the minimum path sum from top to bottom. At each step you may move to an adjacent number in the row below.

Medium Dynamic Programming Bottom-up DP Time: O(n²) Space: O(n)
1 Understanding the Problem ▼

You are given a triangle of numbers — a list of rows where row i contains exactly i+1 elements. Starting from the apex (the single element at row 0), descend to the bottom row choosing one number per row. The movement rule: from position j in row i, you can only step to position j or j+1 in row i+1 — the two cells directly below. Your goal is to find the path with the minimum total sum.

💡 Key Insight

Every interior cell in the triangle is reachable from exactly two parents above. This means there are only two choices at each step, and the optimal cost to reach any cell depends on previously computed optimal costs of adjacent cells below it. This is the hallmark of dynamic programming: the problem has optimal substructure and overlapping subproblems.

Worked example:

triangle = [
     [2],
    [3, 4],
   [6, 5, 7],
  [4, 1, 8, 3]
]

Path  2 → 3 → 5 → 1  =  11  ✓ minimum
Path  2 → 4 → 7 → 3  =  16  suboptimal
Path  2 → 3 → 6 → 4  =  15  suboptimal
🔐 Why Greedy Fails Here

A greedy approach picks the smaller neighbor at each step. From 2, it picks 3 (smaller than 4). From 3, it picks 5 (smaller than 6). This is correct so far — but greedy doesn't see that 5 is adjacent to 1 (best option) while a different path might eventually get stuck. In larger triangles greedy consistently fails by optimizing one step at a time rather than the full path. DP computes costs for all future paths simultaneously.

2 The Algorithm: Bottom-Up DP (Tabulation) ▼

Instead of recursing top-down, we build the answer bottom-up. We start from the last row (costs are trivially known) and propagate optimal costs upward one row at a time. After processing every row, the apex cell holds the global minimum.

1 Initialize: copy the last row of the triangle into a 1D array dp. Each dp[j] = the cost of being at position j in the last row — the base case.
2 Iterate upward: for row i from len(triangle)-2 down to 0, and for each position j in that row, compute dp[j] = triangle[i][j] + min(dp[j], dp[j+1]).
3 Return dp[0]: after processing row 0 (the apex), dp[0] contains the minimum path sum for the entire triangle.
📐 Recurrence Relation
dp[j] = triangle[i][j] + min(dp[j], dp[j+1])
   for i in range(len(triangle)-2, -1, -1)
   for j in range(len(triangle[i]))

Base case: dp = triangle[-1][:]  (copy of last row)
Naive Recursion — O(2ⁿ)

Try every path top-down. From each cell branch left and right. The recursion tree doubles at each level. For a 20-row triangle, that is over 1 million paths. Many subproblems recalculated.

Bottom-Up DP — O(n²) / O(n)

Process each cell exactly once. Reuse results from below. Single 1D array overwritten in-place. No cache, no stack overflow, no exponential blowup.

💡 In-Place Magic

Because row i only reads from dp[j] and dp[j+1] — both at indices ≥ j — updating from left to right is safe. We overwrite dp[j] before we need it again, and dp[j+1] (still the row below value) is read first. This is why a single 1D array suffices instead of a 2D table.

3 Why It Works: Optimal Substructure & The Invariant ▼
🔐 The Invariant

After processing row i, for each valid column j: dp[j] = minimum sum of any path starting at position (i, j) and reaching the bottom of the triangle.

Optimal Substructure (why combining sub-answers gives the global answer): The best path from cell (i, j) to the bottom is: triangle[i][j] plus the best of the two sub-paths from (i+1, j) or (i+1, j+1) to the bottom. Because those sub-paths are independent of each other and their costs are already optimal (by our inductive assumption), taking the min gives the globally optimal cost from (i, j).

Overlapping Subproblems (why naive recursion is wasteful): Interior cell (i, j) is reachable from both (i-1, j-1) and (i-1, j). In a naive top-down recursion, dp(i, j) would be recomputed every time it appears in the call tree. In a triangle of n rows, the number of such overlapping calls grows as O(2ⁿ). DP stores each result once — O(n²) total.

💡 Edge Cases Handled Automatically
  • Single row: dp = [val]; no loop executes; return dp[0] = val. Correct.
  • Negative numbers: min() works regardless of sign. DP doesn't assume non-negativity.
  • Left/right edges: Row i has i+1 cells; j=0 and j=i are always valid boundary cells with exactly one child path accessible.

Interview Pick. O(n²) time, O(n) extra space. Iterative, no recursion stack, space-optimal.

⚠️ The [:] Slice Matters

Without [:], dp = triangle[-1] is a reference — mutating dp would corrupt the input. Always copy when you plan to overwrite in-place, or when the input is not yours to modify.

Without caching, dp(2,1) is called twice (from dp(1,0) and dp(1,1)). With @lru_cache the second call is O(1). There are O(n²) unique (i,j) pairs, each computed once.

5 🎤 Interview Strategy: How to Arrive at This ▼
Step 1 — Recognize the DP Signal

The words "minimum sum", "all paths", and "adjacent cells only" together in a fixed-structure grid scream DP. Greedy is tempting but fails (demonstrated above). Exhaustive search is too slow. DP is the sweet spot.

Step 2 — Define the State

Ask: "At position (i, j), what do I need to track?" Only the position itself — not the path taken. So dp[i][j] = minimum cost from (i,j) to the bottom. State space: O(n²).

Step 3 — Write the Recurrence

dp[i][j] = triangle[i][j] + min(dp[i+1][j], dp[i+1][j+1]). Base case: dp[n-1][j] = triangle[n-1][j]. Process rows from bottom to top.

Step 4 — Optimize to O(n) Space

Row i only depends on row i+1. Use a single 1D array. Process positions left-to-right within each row (safe because dp[j+1] from the previous row hasn't been overwritten yet when computing dp[j]).

🗣️ What to Say to the Interviewer

"I see a minimum-path problem over a structured grid — classic DP signal.

Let me define state: dp[j] = min cost from row i, position j, to the bottom.
Base case: dp = copy of last row (trivially correct).

Recurrence bottom-up: dp[j] = triangle[i][j] + min(dp[j], dp[j+1])
for i from n-2 down to 0, j from 0 to i.

Time O(n^2), Space O(n) — single rolling array, no 2D table needed.
Final answer: dp[0] after processing all rows."
💡 Common Follow-Up Questions
  • "O(1) extra space?" — Modify triangle[-1] directly (no copy). Modifies input.
  • "Top-down instead?" — Flip: dp[i][j] = triangle[i][j] + min(dp[i-1][j-1], dp[i-1][j]) from row 1 downward. Answer is min(dp[-1]).
  • "What if we want the actual path, not just the sum?" — Store parent pointers: for each cell, record which child was chosen. Traceback from apex to bottom.

7 Summary & Key Takeaways ▼
Bottom-up DP fills results from base cases upward, eliminating all redundant recomputation
The recurrence dp[j] = triangle[i][j] + min(dp[j], dp[j+1]) processes each cell exactly once in O(1)
A single 1D rolling array achieves O(n) space — no 2D table needed
Greedy fails on this problem; only DP correctly evaluates all future path costs simultaneously
Memoization and tabulation both achieve O(n²) time; tabulation beats memoization on space (O(n) vs O(n²))
🎯 When to Use Bottom-Up Path DP
  • Problem asks for min/max/count over all paths in a DAG-shaped structure
  • Cell (i,j) depends only on cells in the row/layer directly below
  • You want to avoid recursion overhead and cache allocation
  • Space optimization is important (rolling array pattern applies)

🧠 The Pattern Family

Triangle belongs to the "Grid/Shape Path DP" family. Signature: structured 2D shape, constrained movement, optimize a cumulative metric. The key diagnostic questions are: Does each cell's answer depend on a fixed small neighborhood of adjacent cells? and Is the dependency directed (no cycles)? If yes to both, DP is the right tool.
