LeetCode #142

Linked List Cycle II

Given the head of a linked list, return the node where the cycle begins. If there is no cycle, return null. You must solve it using O(1) memory.

Medium Linked List Floyd's Cycle Detection Time: O(n) Space: O(1)
1 Understanding the Problem ▾

You are given a linked list. The tail node's next pointer may point back to some earlier node — forming a loop. Your job is not just to detect whether a cycle exists (that's LeetCode #141), but to pinpoint the exact node where the cycle begins.

Think of it like a racetrack: the road from the starting gate to the track entrance is the "prefix." Once you enter the oval track, you loop forever. We want to find the entrance to the oval — not just confirm that there's a loop.

💡 Key Insight

This is a two-phase problem. Phase 1 uses Floyd's fast/slow pointer trick to confirm a cycle exists and find an arbitrary meeting point inside the cycle. Phase 2 exploits a beautiful mathematical property: reset one pointer to head and walk both pointers one step at a time — they will meet at precisely the cycle entry node. No extra memory needed.

node[4].next points to node[1]

✗ No Cycle — return null

Edge Cases: empty list → null; single node pointing to itself → return that node; cycle at head (no prefix) → return head; no cycle at all → return null; very long list with short cycle.

2 The Algorithm: Floyd's Two-Phase Detection ▾

Floyd's algorithm for cycle entry detection has two distinct phases. Each is simple to implement; the magic is in the mathematical guarantee between them.

1 Initialize: slow = fast = head. Both pointers begin at the head of the list.
2 Phase 1 — Race to detect: Each iteration: slow = slow.next (1 step), fast = fast.next.next (2 steps). Check fast and fast.next before advancing to avoid null errors. If fast reaches null → no cycle, return null. Loop until slow == fast — a cycle is confirmed, and they're at the meeting point.
3 Phase 2 — Reset one pointer: slow = head. Leave fast where it is (the meeting point). Both pointers now advance exactly one step per iteration.
4 Phase 2 — March to entry: Advance both slow and fast one step at a time. When slow == fast again, that node is the cycle entry. Return it.
🔑 Why Does Resetting to Head Work?

This is not obvious from inspection alone — it requires a mathematical proof. The key equation is F = n·C − a, where F is the prefix length, C is the cycle length, a is the distance from cycle entry to meeting point, and n is a positive integer. Section 3 walks through this derivation step by step.

3 Why It Works: Floyd's Mathematical Proof ▾
Define distances in the list: F = length from head to cycle entry (the "prefix") a = length from cycle entry to Phase-1 meeting point (inside cycle) C = total cycle length n = number of complete loops fast made extra vs. slow When Phase 1 meets (slow == fast): Distance slow traveled = F + a Distance fast traveled = F + a + n·C (fast looped n extra times) Since fast = 2 × slow: F + a + n·C = 2(F + a) n·C = F + a F = n·C − a ← the golden equation Phase 2 interpretation: Slow resets to head. Both advance 1 step/iteration. After F steps: slow → head + F steps = cycle entry (by definition of F) fast → meeting_point + F steps = (a steps past entry) + F steps = a + (n·C − a) steps past entry = n·C steps past entry = cycle entry (n full loops = back at entry) ∴ slow == fast at cycle entry ✓
🔐 The Invariant

After Phase 1, the distance from the meeting point forward to the cycle entry equals the distance from head to the cycle entry. That's the algebraic identity F = n·C − a. Walking both pointers simultaneously from these two positions guarantees convergence at the cycle entry.

Edge case: cycle starts at head (F = 0). Then F = n·C − a means a = n·C, so the meeting point is exactly at the cycle entry. After resetting slow to head, both slow and fast are already at the cycle entry → they meet immediately at head. Correct.

⚠️ Common Mistake: The Loop Condition in Phase 1

Both pointers start at head before entering the loop. If you check slow == fast before any moves, they'll always be "equal" at the start and you'll incorrectly detect a cycle. Always advance first, then compare. The standard guard while fast and fast.next with the move inside the loop body handles this correctly.

4 The Code (Annotated) ▾

Solution 1 — Floyd's Algorithm (Interview Pick, O(n) time, O(1) space)

def detectCycle(head):
slow = fast = head ← both start at head
while fast and fast.next: ← need 2 valid nodes for fast.next.next
slow = slow.next ← slow advances 1 step
fast = fast.next.next ← fast advances 2 steps
if slow == fast: ← meeting point — cycle confirmed!
slow = head ← RESET slow to head; fast stays at meeting point
while slow != fast: ← march both 1 step at a time
slow = slow.next ← slow walks from head toward cycle entry
fast = fast.next ← fast walks from meeting point toward cycle entry
return slow ← they meet exactly at cycle entry
return None ← fast reached null → no cycle
⚠️ Why check fast and fast.next (not just fast)?

Inside the loop, we do fast.next.next — so fast.next must also be non-null. If the list has an even number of nodes with no cycle, fast lands on null directly. If odd, fast lands on the last node whose next is null. Both cases need the fast.next guard to avoid a NullPointerException.

Solution 2 — Hash Set (Simpler, O(n) space)

def detectCycle(head):
visited = set() ← remember every node object we've seen
node = head
while node: ← walk until we hit null or revisit
if node in visited: ← we've been here before → cycle entry!
return node ← this is the first repeated node
visited.add(node) ← mark this node as seen
node = node.next ← advance
return None ← reached null → no cycle
💡 Complexity Summary

Floyd's Algorithm: Time O(n), Space O(1). The interview answer — demonstrates algorithmic thinking.
Hash Set: Time O(n), Space O(n). Propose this first as the "obvious" solution, then optimize.
Node modification (mark visited): O(n) time, O(1) space — but destructive; never do this in practice.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Clarify Constraints

Ask: "Should I return the node object or its value?" (Node object.) "Is O(n) extra space acceptable, or do you want O(1)?" This shows you're aware of the trade-off and aren't just diving into the hard solution.

Step 2: Propose Hash Set First

"The direct approach: walk the list storing each node in a set. The first node we've already seen is the cycle entry. O(n) time, O(n) space." Code this up if asked — it's correct and interviewers appreciate seeing it before the optimized solution.

Step 3: Introduce Floyd's for O(1) Space

"For constant space, I'll use Floyd's two-pointer algorithm. Phase 1 detects the cycle (fast/slow meet inside). Phase 2 resets one pointer to head and marches both one step at a time — they converge at the cycle entry. I can prove this is always correct."

Step 4: Be Ready to Derive the Proof

Define F (prefix), a (meeting offset from entry), C (cycle length). Show fast = 2×slow implies F = n·C − a. Then: after reset, slow walks F from head; fast walks F from the meeting point which equals n full cycles back to entry. Both arrive together.

🗣️ What to Say to the Interviewer

"I'll use Floyd's algorithm. Phase 1: slow at 1 step, fast at 2 steps.
If fast hits null → no cycle, return null.
When they meet → cycle detected; meeting node is inside the cycle.
Phase 2: reset slow to head, keep fast at meeting point.
Walk both one step at a time. They meet at the cycle entry.
The math: F = n·C − a proves this. Both pointers walk exactly F steps
to reach the entry — from different directions. O(n) time, O(1) space."
🔮 Likely Follow-up Questions
  • "What if cycle starts at head?" → F=0, Phase 2 terminates immediately — slow and fast both start at the cycle entry.
  • "Find the cycle length?" → After Phase 1 meeting, keep one fixed, advance other counting steps until they re-meet.
  • "What's the worst-case step count?" → O(n): fast enters cycle within n steps, meeting happens within one more cycle length.
  • "Can you find duplicate in array using this?" → Yes — LeetCode #287 treats array as a linked list where index is the "next" pointer.
