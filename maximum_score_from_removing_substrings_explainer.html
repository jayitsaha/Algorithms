LeetCode #1717

Maximum Score From Removing Substrings

Greedily remove the higher-value pair first — use a stack to simulate collapsing adjacencies in O(n).

Medium Greedy Stack Time: O(n) Space: O(n)
1 Understanding the Problem ▾

You are given a string s made of lowercase letters, and two integers x and y. You can repeatedly remove either the substring "ab" (earning x points) or "ba" (earning y points) from anywhere in the string. After each removal, the surrounding characters collapse together, potentially forming new pairs. Find the maximum total score achievable.

💡 Key Insight

Each 'a' and 'b' in the string can participate in at most one removal. When it can be part of either pair, always assign it to the higher-value pair first. Doing so greedily — exhaustively removing the higher-value pair before touching the lower-value pair — is provably optimal. A stack handles the "characters collapse after removal" naturally.

✗ Wrong order: remove "ba" first (x=5, y=4)

s = "aabba"
Remove "ba" (idx 2-3): +4 → "aaa" .. wait
Remove "ab" (idx 1-2): +5 → then "ba"→+4
s="aabba": remove "ba"→"aaa" (no more ab/ba)
Total = 4 ✗ (missed both "ab"s!)

✓ Correct: remove "ab" first (x=5, y=4)

s = "aabba"
Remove "ab" (idx 1-2): +5 → "aba"
Remove "ab" (idx 0-1): +5 → "a"
No "ba" left in "a"
Total = 10 ✓

Edge Cases: No 'a' or 'b' in string (score = 0); string of all one letter (no pairs); x == y (either order works equally); deeply nested "ababab" — the stack handles all of these correctly.

2 The Algorithm: Two-Pass Greedy Stack ▾

The core structure is two sequential passes, each using a stack to simulate pair removal. We pick which pair to prioritise based on which scores more.

Pass 1 — Higher Value
"ab" if x ≥ y
Exhaust high-value pairs
Pass 2 — Lower Value
"ba" then
Clean up remainder
1 Determine order: If x >= y, first = "ab", second = "ba". Otherwise swap both the pair strings and scores. This normalisation means the remove() helper is written once.
2 Pass 1 — Stack removal of first pair: Iterate over s character by character. If the stack's top equals pair[0] and the current character equals pair[1], a match is found: pop the top, add the score. Otherwise push the current character. O(n) — each character enters and exits the stack at most once.
3 Reconstruct remainder: ''.join(stack) gives the string with all first-pair occurrences removed. Pass 2 runs the same stack algorithm on this remainder for the second pair.
4 Return total score: pts1 + pts2 is the maximum achievable score.
🔑 Why a Stack — Not String.replace()?

s.replace("ab", "") in a loop is O(n) per replacement and up to O(n) replacements, giving O(n²) total. The stack processes each character exactly once — when we pop, the new top automatically becomes adjacent to the next character, perfectly simulating collapse. Two passes = O(2n) = O(n).

3 Why It Works: The Invariant ▾
🔐 The Stack Invariant

After processing index i, the stack contains all characters from s[0..i] that have not yet been matched into a target pair. Furthermore, no two adjacent characters at the top of the stack form the target pair — if they did, we would have matched them immediately. This guarantees all possible target pairs in the current window are removed greedily.

Why greedy ordering is provably optimal: Consider any 'a' that could form either "ab" (at cost x) or "ba" (at cost y) with adjacent 'b' characters. WLOG x > y. If we use the 'a' in a "ba" pair instead of "ab", we earn y instead of x — a loss of (x - y) > 0. No rearrangement of pair assignments can compensate this loss. Thus: always assign shared characters to the higher-value pair first.

⚠️ Why Two Passes Are Needed

After Pass 1, some 'a' and 'b' characters remain that were unable to form the first-pair type — perhaps they're in "ba" order when we sought "ab". These characters can still form the second-pair type, so Pass 2 captures that remaining value. A single pass cannot distinguish "I should save this 'a' for a 'b' I haven't seen yet" from "this 'b' is better matched to the 'a' behind me."

When x == y: Both passes produce equal value per removal. The greedy order doesn't matter — our code handles this by picking "ab" first (since we check x >= y), which is perfectly correct.

4 The Code (Annotated) ▾

Solution 1 — Two-Pass Greedy Stack (Optimal, O(n)) ← Interview Pick

def maximumGain(s: str, x: int, y: int) -> int:
if x < y: ← Normalise: ensure first pair always has higher (or equal) score
x, y = y, x ← Swap scores
first, second = "ba", "ab" ← Swap pair labels to match swapped scores
else:
first, second = "ab", "ba" ← "ab" scores x ≥ y, so it's first
def remove(s, pair, score): ← Reusable helper: one greedy pass via stack
stack, pts = [], 0 ← Fresh stack; local score accumulator
for c in s: ← One linear scan: O(n)
if stack and stack[-1] == pair[0] and c == pair[1]:← Top + current = target pair?
stack.pop() ← Remove top — pair is consumed
pts += score ← Claim the points for this removal
else:
stack.append(c) ← No match yet — push for future matching
return ''.join(stack), pts ← Remaining chars + points earned this pass
s, pts1 = remove(s, first, x) ← Pass 1: exhaust all high-value pairs
_, pts2 = remove(s, second, y) ← Pass 2: exhaust remaining low-value pairs
return pts1 + pts2 ← Maximum total score
⚠️ Common Mistake: Single Pass for Both Pairs

Some candidates try to handle both "ab" and "ba" in a single pass by checking two conditions. This fails because when the stack top is 'a' and the current char is 'b', you know it's "ab" — but what if the next char after removing would form "ba" with what's below? The single-pass approach creates ordering ambiguity. Two focused passes avoid this cleanly.

Solution 2 — Brute Force Loop (O(n²), for intuition)

def maximumGain_brute(s, x, y): ← Not suitable for n=10^5
result = 0
hi_pair = "ab" if x >= y else "ba" ← Higher-value pair
lo_pair = "ba" if x >= y else "ab"
hi_pts = max(x, y); lo_pts = min(x, y)
for pair, pts in [(hi_pair, hi_pts), (lo_pair, lo_pts)]:
while pair in s: ← O(n) search per iteration
s = s.replace(pair, '', 1) ← O(n) replace — two O(n) ops per removal = O(n²) total
result += pts
return result
💡 Complexity Summary

Two-Pass Stack (Optimal): Time O(n) — two single scans. Space O(n) — stack holds at most n chars.
Brute Force: Time O(n²) — O(n) per removal × O(n) removals. Will TLE for n = 10⁵.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Spot "Pair Removal With Collapse"

Any time a problem says "remove adjacent pairs and the surrounding characters become adjacent" — think stack. The stack top is always "waiting for its partner." This pattern appears in parentheses matching, duplicate removal, and here.

Step 2: Identify the Greedy Property

Two pair types compete for the same 'a' and 'b' characters. One pair scores more. Key question: "Is there ever a reason to delay using the high-value pair?" No — using a low-value pair first can only reduce or maintain the score. This is the exchange argument for greedy correctness.

Step 3: Propose Brute Force → Optimize

Start with "I could use str.replace() in a loop — O(n²), too slow for n=10⁵." Then: "If I use a stack, each character is pushed and popped at most once per pass — O(n). Two passes = O(2n) = O(n)."

Step 4: Write the DRY Helper

The stack pass logic is identical regardless of which pair we target. Write it as a parameterised helper remove(s, pair, score). This shows coding discipline and avoids duplicated code — interviewers notice this.

🗣️ What to Say to the Interviewer

"Pair removal with collapse is a stack pattern — stack top waits for its partner.
Greedy insight: since x and y differ, always remove the higher-value pair first.
Normalise at the start: if y > x, swap so 'first' always has the higher score.
Two passes: Pass 1 exhausts 'first' pair via stack — O(n). Pass 2 does same for 'second' pair on the remainder.
Each char pushed and popped at most once per pass → O(n) total. Space O(n) for the stack."
🔮 Likely Follow-up Questions
  • "What if there are k different pairs with k different scores?" → Sort pairs by descending score; do k passes
  • "Can you solve it in one pass?" → Not cleanly; two focused passes are already O(n)
  • "What if the pairs overlap (e.g., "abc" and "bca")?" → Generalise to tracking state per pair with multiple stacks
  • "Minimum removals to make string valid?" → Simpler stack counting (#1003, #1541)
