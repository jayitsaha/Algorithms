LeetCode #403

Frog Jump

A frog on stone 0 must reach the last stone. It can jump k−1, k, or k+1 units if its last jump was k. Determine if it can reach the last stone.

Hard Dynamic Programming DP with Jump Size State Time: O(n²) Space: O(n²)
1 Understanding the Problem ▾

A frog starts at position 0 (the first stone) and must reach the last stone. The key constraint is momentum: if the frog's last jump was k units, its next jump must be exactly k−1, k, or k+1 units (and must be positive — can't stay put or go backward). The very first jump must be exactly 1 unit.

💡 Key Insight

The frog's position alone is not enough to describe its situation. Two frogs at stone 5 — one arriving with jump 2, another with jump 5 — have completely different next options. The state must be (current stone position, last jump size). This two-dimensional state space is the heart of the DP.

Important constraints: If stones[1] ≠ 1, return False immediately (first jump forced to be 1). Jump size of 0 is always skipped. The stones array is sorted in ascending order with no duplicates.

⚠️ Common Confusion

The problem asks about positions (stone values), not indices. Stone at index 3 might have position/value 5. Always work with the actual position values stored in the array, not their array indices.

2 The Algorithm: Forward DP with Jump-Set per Stone ▾

We use a hash map where dp[pos] is the set of all jump sizes that can land the frog on stone at position pos. We iterate stones left to right, and for each reachable (pos, k) pair, we push forward to the next possible stones.

1 Build a stone-position set for O(1) membership queries. Also build the DP table: a dict dp = {pos: set()} for every stone position. Initialize dp[0] = {0} — the frog is at pos 0 with a sentinel "last jump" of 0.
2 Iterate stones in order. For each stone position pos and each arrival jump size k in dp[pos], try the three candidate next jumps: k−1, k, k+1.
3 Filter and propagate. For each candidate next_k: skip if next_k ≤ 0 (no backward or zero-length jumps). Skip if pos + next_k is not in the stone set. Otherwise, add next_k to dp[pos + next_k].
4 Answer: After processing all stones, check dp[stones[-1]]. If it is non-empty, the frog can reach the last stone — return True. Otherwise False.

3 Why Is This DP? Invariant, Substructure & Proof ▾

Why greedy fails: No local rule reliably guides the frog. "Jump as far as possible" may overshoot all remaining stones. "Jump as close as possible" may leave too little momentum for a distant stone. The optimal first jump depends on the entire future stone layout — a global property.

Why brute-force DFS is exponential: At each stone, the frog has up to 3 choices. With n=2000 stones this explodes to 3²⁰⁰⁰ paths. Many paths arrive at the same (stone, jump) pair. Without memoization, we recompute the same futures repeatedly.

🔐 The DP Invariant

After processing all stones up to position pos, dp[pos] contains the complete set of jump sizes k such that a valid path from stone 0 to stone pos exists with the final jump being exactly k. Every valid arrival speed is captured; no false positives exist.

Optimal Substructure: Whether the frog can complete the crossing from (pos, k) depends only on what stones exist ahead and the current momentum — not on how the frog got to pos. The subproblem "can the frog cross from (pos=5, k=2)?" is completely determined by the remaining stone positions and k=2. History before reaching stone 5 is irrelevant.

Overlapping Subproblems: Multiple paths can reach the same (stone, k) pair. Example: paths 0→1→3 (both jump 2) and 0→1→2→3 (invalid — no stone 2 here, but in general) can both arrive at the same state. Without memoization we'd explore the same sub-tree from (stone=3, k=2) multiple times. DP merges all arrivals before expanding forward.

💡 Why Sets, Not Booleans?

A stone might be reachable with different jump sizes — and those different speeds enable completely different futures. dp[5] = {2} means only jump-2 arrivals are valid; dp[5] = {2, 4} means both jump-2 and jump-4 are valid, allowing next jumps of {1,2,3} or {3,4,5} respectively. A single boolean per stone would lose this crucial information.

Complexity Proof: There are n stones. At each stone, the set of possible jump sizes is bounded by the maximum achievable jump, which itself is bounded by n (you can't jump farther than the total range of stones). So each dp[pos] set has at most O(n) elements. Total states = O(n²). Each state is processed once, with 3 constant-time candidates. Total time: O(n²). Space: O(n²) for the DP table.

4 The Code (Annotated) ▾

Solution 1 — Bottom-Up Tabulation (Interview Pick)

def canCross(stones: list[int]) -> bool:
if stones[1] != 1: ← Fast fail: first jump must be exactly 1
return False
stone_set = set(stones) ← O(1) membership test: is X a stone position?
dp = {s: set() for s in stones} ← dp[pos] = {jump sizes that can land here}
dp[0].add(0) ← Sentinel: "0 units jumped to reach start"
for pos in stones: ← Process stones left to right (sorted order)
for k in dp[pos]: ← Each valid arrival jump size at this stone
for next_k in (k-1, k, k+1): ← Shrink, keep, or grow the jump
if next_k > 0 and (pos + next_k) in stone_set:← Must advance, must hit a stone
dp[pos + next_k].add(next_k)← Record: next stone reached with jump next_k
return bool(dp[stones[-1]]) ← Non-empty = frog can reach the last stone
⚠️ Why dp[0] = {0} and not {1}?

We seed with 0 as a sentinel, not 1. When we process stone at pos=0 with k=0, we try next_k ∈ {−1, 0, 1}. Only next_k=1 is valid (positive AND stone at 0+1=1 exists). This bootstraps the mandatory first jump of 1 without any special-casing. Seeding with {1} would skip the validation that stones[1] == 1 check naturally provides.

⚠️ What if k=1 and next_k=0?

The guard next_k > 0 prevents the frog from jumping 0 units (staying in place forever). Without it, processing stone 1 with k=1 would try next_k=0 and do dp[1+0].add(0) — adding a spurious "jump 0" to stone 1's own set, which would then propagate incorrectly in the next iteration.

Solution 2 — Top-Down Memoization (alternative perspective)

def canCross(stones: list[int]) -> bool:
stone_set = set(stones)
last = stones[-1]
memo = {} ← Cache: (pos, k) → can frog reach last stone?
def dfs(pos, k):
if pos == last: return True ← Base case: already at destination
if (pos, k) in memo: return memo[(pos, k)]← Cache hit: avoid recomputation
for next_k in (k-1, k, k+1):
nxt = pos + next_k
if next_k > 0 and nxt in stone_set:
if dfs(nxt, next_k): ← Recurse: can we finish from this new state?
memo[(pos, k)] = True; return True
memo[(pos, k)] = False; return False
return dfs(0, 0)
💡 Complexity Summary

Brute-Force DFS (no memo): Time O(3ⁿ) — exponential, TLE on n > 30
Top-Down Memoization: Time O(n²), Space O(n²) — cache hits eliminate re-work
Bottom-Up Tabulation (pick this!): Time O(n²), Space O(n²) — no recursion stack, iterates naturally

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify the State Space

Say aloud: "The frog's future options depend on both WHERE it is and HOW FAST it's going. A frog at stone 5 with k=2 has very different options from one at stone 5 with k=5. So my DP state must be (position, last_jump_size) — that's a 2D state space."

Step 2: Establish Bounds

"The jump size can at most grow by 1 per step, starting from 1. After n steps, max speed is n. So at each stone, the possible jump sizes are bounded by O(n). Total states: n stones × n speeds = O(n²). This is tractable for n=2000."

Step 3: Choose dp[pos] = Set of Jump Sizes

"I'll store, for each stone, every jump size that can reach it. This is more expressive than a boolean because the jump size governs future options. Propagate forward: from (pos, k), push k−1/k/k+1 to valid next stones."

Step 4: Explain the Sentinel

"Initialize dp[0] = {0}. Sentinel 0 naturally produces the mandatory first jump of 1: from k=0, only next_k=1 is positive and lands on a stone. No special-casing needed."

🗣️ What to Say to the Interviewer

"This is DP because the frog's next valid jumps depend on its current position
AND the previous jump size — two dimensions of state.
I'll use dp[pos] = set of arrival jump sizes. For each (pos, k), I try k-1,k,k+1.
Filter: skip k≤0 and positions not in the stone set.
Answer: dp[last stone] non-empty → True, else False.
Time O(n²): n stones × up to n jump sizes each.
Space O(n²): same reason — the full DP table."
🔮 Likely Follow-up Questions
  • "Reconstruct the actual path?" → Store (prev_pos, jump) parent in each dp set entry; trace back from last stone
  • "What if jump can vary by 2 (k−2 to k+2)?" → Same DP, try 5 candidates, same O(n²) complexity
  • "Can we do better than O(n²)?" → Current lower bound for this problem appears to be O(n²); no known sub-quadratic solution
  • "Why does greedy fail?" → Give the counter-example [0,1,2,3,4,8,9,11]: always jumping max leads to getting stuck

7 Summary & Key Takeaways ▾
State = (stone position, last jump size) — never just position alone
dp[pos] = set of valid arrival jump sizes — not a single boolean
Propagate forward: from (pos, k), push k−1/k/k+1 to valid next stones
Filter: next_k > 0 (no zero/backward jumps), next stone must exist in stone_set
Sentinel dp[0]={0} elegantly bootstraps the mandatory first jump of 1
Answer: dp[last stone] non-empty → True; all sets empty at last → False

  • "Reach position X from 0, each move constrained by previous move size?" → DP with momentum/speed state
  • Current position + some "speed/history" determines future options → 2D DP state
  • Bounded branching factor (3 choices per step) → tractable even if state space is large
  • Greedy or BFS (without state) gets stuck → confirms need for DP with extra state
