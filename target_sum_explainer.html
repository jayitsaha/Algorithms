LeetCode #494

Target Sum

Assign '+' or '−' to each number. Count the number of distinct sign assignments that produce the target sum.

Medium Dynamic Programming Count Subsets to Target Time: O(n·S) Space: O(S)
1 Understanding the Problem ▾

You have an integer array nums and an integer target. For each element you choose a '+' or '−' sign. Evaluate the resulting expression and count how many distinct sign assignments equal the target.

Example Input

nums = [1, 1, 1, 1, 1]
target = 3

Answer = 5 distinct ways

−1+1+1+1+1 = 3
+1−1+1+1+1 = 3
+1+1−1+1+1 = 3
+1+1+1−1+1 = 3
+1+1+1+1−1 = 3

💡 Key Insight — The Subset Partition Trick

Let P = indices assigned '+', N = indices assigned '−'. Then:

sum(P) − sum(N) = target     (definition)
sum(P) + sum(N) = total      (all elements)
─────────────────────────────
2·sum(P) = total + target
sum(P) = (total + target) / 2 = S

The problem reduces to: count subsets of nums that sum to S. This is the classic 0/1 Knapsack count variant — a standard DP pattern.

🔐 Edge Cases
  • (total + target) is odd → impossible (S would be fractional), return 0
  • target > total or target < −total → unreachable, return 0
  • Zeros in nums → each zero multiplies count by 2 (both ± give same result), DP handles naturally
  • S < 0 → impossible (subset sums are non-negative for non-negative nums)
2 The Algorithm: DP — Count Subsets Summing to S ▾

After the partition math, we count subsets summing to S = (total + target) / 2. We maintain a 1D array dp[j] = number of ways to form sum j from items processed so far.

📐 The Recurrence Relation
dp[j] += dp[j − num]    for each num, sweeping j from S down to num
Base: dp[0] = 1    (one way to form sum 0: take nothing)
Answer: dp[S] after all items

Why sweep j from S down to num (not 0 to S)?

The reverse sweep is the 0/1 Knapsack signature. Each item can be used at most once. If we swept forward, we'd read an already-updated dp[j−num] from the same pass, effectively "using" the item multiple times (unbounded knapsack). Sweeping backward guarantees we always read dp[j−num] from the previous item's state — so each item contributes at most once per pass.

At each item: include in subset P or skip

adds ways to reach j via this item

ways without this item preserved
1 Compute S = (sum(nums) + target) // 2. If (total+target) is odd or S < 0, return 0.
2 Init dp = [0] * (S+1), then dp[0] = 1 (base case).
3 For each num in nums: sweep j from S down to num. Execute dp[j] += dp[j−num].
4 Return dp[S] — the count of '+' subsets summing to S = count of valid sign assignments.
3 Why It Works: Invariant, DP Proof, and Why Not Greedy ▾
🔐 The DP Invariant

After processing the first k items: dp[j] = the number of distinct subsets of {nums[0], ..., nums[k−1]} that sum to exactly j.

Why DP and not Greedy? Greedy works when one locally optimal choice leads to a global optimum. Here, we're counting all valid arrangements — every branch of the decision tree matters. No single "best choice" per item can capture all paths. We need to accumulate counts across all 2n possibilities — and DP does this efficiently by building on shared subproblems.

Optimal substructure: The number of ways to form sum j using the first k+1 items equals (ways using first k items that reach j) + (ways using first k items that reach j−num[k], extended by taking num[k]). The dp recurrence captures this exactly.

Overlapping subproblems: The state (item index, remaining sum) can be reached by many different sign sequences. Naively, the recursive tree has 2n leaves but only O(n × sum) distinct states. Caching (memoization) or the tabulation dp array prevents recomputation.

⚠️ The Most Common Mistake: Forward vs Backward Sweep

for j in range(num, S+1) → Wrong! This is unbounded knapsack — the same num can be used multiple times.
for j in range(S, num-1, -1) → Correct! Each num contributes at most once per outer loop iteration.

Why the partition math is exact: Every valid sign assignment defines a unique partition (P, N). Conversely, every subset summing to S defines one valid sign assignment (the subset = P, complement = N). The bijection is perfect, so dp[S] counts exactly the number of valid sign assignments.

4 The Code (Annotated) — Tabulation & Memoization ▾

Solution 1 — DP Tabulation (Bottom-Up) · O(n·S) time · O(S) space  Interview Pick

def findTargetSumWays(nums, target):
total = sum(nums)← sum of all array elements
if (total + target) % 2 != 0:← S must be a whole number; odd → impossible
return 0
S = (total + target) // 2← size of the '+' subset we need to count
if S < 0:← target too large or too small; unreachable
return 0
dp = [0] * (S + 1)← dp[j] = ways to form sum j from items so far
dp[0] = 1← base: one way to form sum 0 (take nothing)
for num in nums:← process each number exactly once (outer loop)
for j in range(S, num - 1, -1):← reverse sweep: 0/1 knapsack constraint
dp[j] += dp[j - num]← if include num: inherit ways to reach j−num
return dp[S]← total valid sign assignments
⚠️ Why return dp[S] instead of dp[target]?

dp is indexed by non-negative sums (0 to S). The original target can be negative. The transformation S=(total+target)/2 always gives a non-negative integer when valid. dp[target] would be a wrong or illegal index — always use dp[S].

Solution 2 — Memoization (Top-Down) · O(n·sum) time · O(n·sum) space

from functools import lru_cache
def findTargetSumWays(nums, target):
n = len(nums)
@lru_cache(maxsize=None)← auto-memoize all (i, remaining) pairs
def dp(i, remaining):← i = current index; remaining = sum still needed
if i == n:← base: used all numbers
return 1 if remaining == 0 else 0← did we exactly hit the target?
add = dp(i + 1, remaining - nums[i])← try '+' sign on nums[i]
sub = dp(i + 1, remaining + nums[i])← try '−' sign on nums[i]
return add + sub← total ways from this state
return dp(0, target)← start at index 0, need exactly 'target' remaining
💡 Which Approach to Prefer?

Tabulation is the interview gold standard — O(S) space, no recursion overhead, simpler call stack. Memoization is easier to derive on the fly (just write the recursion + cache). Start with memoization if you're stuck, then optimize to tabulation. Both give the same time complexity asymptotically.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the "Count Ways" Signal

"How many ways" → DP (not greedy). Brute force explores every sign combination: 2n leaves. For n=20 that's ~1M — borderline. We need to find overlapping states to collapse the tree.

Step 2: State the Recursion

Define dp(i, remaining) = ways to assign signs to nums[i..n-1] such that their signed sum = remaining. Base: dp(n, 0)=1, dp(n, x≠0)=0. Transition: dp(i, r) = dp(i+1, r−nums[i]) + dp(i+1, r+nums[i]).

Step 3: Notice the Partition Shortcut

The state space is O(n × 2·total). But if you notice the partition math — P = (total+target)/2 — you can reduce to counting subsets with sum S, giving O(S) space with the 1D DP trick.

Step 4: Apply 0/1 Knapsack Template

dp[0]=1; for each num, reverse sweep j from S to num: dp[j] += dp[j-num]. The reverse ensures each item is used at most once.

Step 5: Handle Edge Cases Before Coding

Check: (total+target) even, S≥0. Handle these at the top before the DP loop — saves debugging time.

🗣️ What to Say to the Interviewer

"'Count the ways' tells me DP. The brute force is 2^n assignments. I see overlapping subproblems: many sign sequences share the same (index, remaining) state. Let me define dp(i, remaining) = ways to complete indices i..n-1 with sum 'remaining'. Base: dp(n, 0) = 1. Transition: try +nums[i] and -nums[i]. Now for the optimization: I notice every assignment splits nums into '+' subset P and '-' subset N. Then P - N = target and P + N = total, giving P = (total + target) / 2 = S. So I only need to count subsets summing to S — a classic 0/1 knapsack count! dp = [0]*(S+1), dp[0] = 1. For each num, sweep j from S down to num: dp[j] += dp[j-num]. Time: O(n*S), Space: O(S). Edge case: return 0 if (total+target) is odd or S < 0."
🎯 Pattern Recognition Signals
  • "Assign +/− to each element" → subset partition trick
  • "Count ways" (not find min/max) → DP count variant
  • Each element used exactly once → 0/1 Knapsack (reverse sweep)
  • Non-negative integers, counting subsets → 1D DP with dp[j] += dp[j-num]

7 Summary & Key Takeaways ▾
ApproachTimeSpaceNotes
Brute Force DFS O(2n) O(n) stack Exponential; ok only for n ≤ 20
Memoization (top-down) O(n × sum) O(n × sum) Easy to derive; natural from recursion
DP Tabulation ✓ O(n × S) O(S) Optimal; partition trick cuts space
"Count ways" + each item used once → 0/1 Knapsack count. The dp recurrence is dp[j] += dp[j-num] with a reversed inner loop.
The partition math P=(total+target)/2 transforms the sign problem into subset counting — this is the key insight that makes O(S) space possible.
Reverse inner loop (j: S → num) is mandatory for 0/1 Knapsack. Forward loop gives unbounded knapsack (items reused).
Always check: (total+target) must be even and S must be ≥ 0. Return 0 immediately on failure.
Zeros in the array double the count — handled automatically since dp[j] += dp[j-0] = dp[j] adds itself each pass.
🎯 When to Apply the Count-Subsets-to-Target Pattern
  • "Assign +/− signs" → partition P and N → S = (total+target)/2 → count subsets
  • "Count subsets summing to K" → dp[j] += dp[j-num], reverse sweep, dp[K] is answer
  • Non-negative values, each used once, counting (not optimizing) → this exact template
  • If items can repeat → unbounded knapsack (forward sweep instead)
