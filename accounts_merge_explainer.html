LeetCode #721

Accounts Merge

Given a list of accounts where each account is [name, email1, email2, ...], merge accounts that share at least one common email into a single account.

Medium Graph Union-Find Time: O(E·log E) Space: O(E)
1 Understanding the Problem ▾

You're given a list of accounts. Each account is a list where the first element is the person's name and the rest are their email addresses. The twist: the same person may appear across multiple accounts, connected by a shared email.

Your task: merge all accounts that belong to the same person (identified by at least one shared email). Output each merged person as a single account: [name, sorted_email1, sorted_email2, ...].

💡 Key Insight

This is a connected components problem. Think of every email as a node. If two emails appear in the same account, they share an edge — they belong to the same person. Accounts from different rows that share even one email form one connected component. Find all components, collect their emails, and sort.

Input — fragmented accounts

["John", "j@g.com", "j@m.com"]
["John", "j@m.com", "j@y.com"]
["Mary", "m@g.com"]

Output — merged accounts

["John", "j@g.com", "j@m.com", "j@y.com"]
["Mary", "m@g.com"]

Edge Cases: A single-email account with no overlaps remains alone. Two accounts sharing a name but no email remain separate (same name ≠ same person). Emails are case-sensitive strings. The order of output accounts in the result list doesn't matter.

2 The Algorithm: Union-Find by Email ▾

Union-Find (Disjoint Set Union) is a data structure designed exactly for this: grouping items into components and merging components efficiently. We use emails as nodes, not accounts.

1 Initialize Union-Find: For every email we encounter across all accounts, set parent[email] = email (each email starts as its own root). Also build email_to_name[email] = name so we can look up the person's name later.
2 Union within each account: For every account, pick the first email as the "anchor." Union every other email in that account with the anchor. All emails in the same account must share a root after this step.
3 Collect by root: Iterate through every known email. Call find(email) to get its root. Group all emails with the same root together — they're the same person's emails.
4 Format output: For each component, look up email_to_name[root] to get the name. Sort the list of emails. Prepend the name. Return the list of all merged accounts.
🔑 Algorithm Deep-Dive: Union-Find with Path Compression

Origin: Union-Find (Disjoint Set Union) by Tarjan, 1975. Tracks a partition of a set into disjoint subsets (components) with near-O(1) merge and lookup.

Core invariant: find(x) always returns the canonical root of x's component. After path compression, x and all ancestors point directly to the root — future finds are O(1).

When to recognize: "Merge groups sharing a common X", "find connected components", "are A and B in the same group?" — Union-Find.

How Union bridges accounts across rows:

Account 0: j@g.com (anchor) ← j@m.com
Account 1: j@m.com finds root → j@g.com ← j@y.com ← BRIDGE merges both accounts!
Account 2: m@g.com (standalone)
3 Why It Works: The Invariant ▾
🔐 The Invariant

After processing all accounts: two emails share the same root if and only if they appeared together in at least one account — directly or transitively. The union operation maintains this: if A and B are unioned (same account), and B and C are unioned (another account), then A, B, C all share one root. This is exactly the transitive-closure of the "same account" relation.

Why transitivity is automatic: Suppose email A and B are in account 1, and B and C are in account 2. When we process account 1: union(A, B) → same root. When we process account 2: union(B, C). find(B) walks up to A's root (with path compression). So C is merged into A's component. A, B, C all share one root.

Path compression guarantee: find(x) recursively walks to the root, then sets parent[x] = root directly. After one find, x is one hop from the root. All subsequent finds for x are O(1). Combined with union by rank, total complexity is O(α(n)) per operation — effectively O(1).

⚠️ Common Mistake: Union by Account Index

Beginners sometimes try to union account indices instead of emails. This fails: two accounts with the same person but different emails in different rows would never be connected — there's no "edge" between their indices. The email is the edge. Always treat emails as the graph nodes.

⚠️ Why Not Union by Name?

Two people can share a name! "John" at two different companies might be two separate people with zero shared emails. Merging by name would incorrectly combine them. The problem explicitly says to merge by shared email only.

4 The Code (Annotated) ▾

Solution 1 — Union-Find (Optimal, Interview Pick)

import collections← needed for defaultdict
def accountsMerge(accounts): ← accounts: list of [name, e1, e2, ...]
parent = {} ← Union-Find parent map: email → root
email_to_name = {} ← remember account name for each email
def find(x): ← find root with path compression
if parent[x] != x: ← if x is not its own parent...
parent[x] = find(parent[x]) ← ...compress: point x directly to root
return parent[x] ← return root of component
def union(a, b): ← merge a's component with b's component
ra, rb = find(a), find(b) ← find both roots first
if ra != rb: ← only act if they are in different groups
parent[ra] = rb ← attach root of a under root of b
for account in accounts: ← pass 1: register emails and union them
name = account[0] ← first element is always the name
for email in account[1:]: ← loop over all emails in this account
if email not in parent: ← first time seeing this email?
parent[email] = email ← init: email is its own root
email_to_name[email] = name ← store the account's owner name
for email in account[2:]: ← union all emails to the first email
union(account[1], email) ← account[1] is the anchor email
components = collections.defaultdict(list) ← pass 2: group emails by their root
for email in email_to_name: ← iterate every registered email
root = find(email) ← find canonical root (path-compressed)
components[root].append(email) ← group under root
result = [] ← assemble final answer
for root, emails in components.items(): ← each component = one merged person
result.append(
[email_to_name[root]] ← name from root's stored value
+ sorted(emails)) ← sorted emails appended
return result
⚠️ Why two separate for-loops over the account's emails?

The first loop (account[1:]) initializes every email into parent before any unions happen. The second loop (account[2:]) performs unions using account[1] as anchor. If you merge these into one loop and call union(account[1], email) before account[1] is in parent, you get a KeyError. The two-loop structure ensures initialization always precedes union.

Solution 2 — DFS on Adjacency List (Alternative)

def accountsMerge(accounts):
from collections import defaultdict
email_to_name = {} ← email → account name
graph = defaultdict(set) ← adjacency list: email → {neighbors}
for account in accounts:
name, *emails = account
for email in emails:
email_to_name[email] = name
graph[emails[0]].add(email) ← connect all to first email
graph[email].add(emails[0]) ← bidirectional
visited, result = set(), []
for email in email_to_name:
if email not in visited: ← start DFS from each unvisited email
component, stack = [], [email]
while stack:
node = stack.pop()
if node in visited: continue
visited.add(node); component.append(node)
stack.extend(graph[node])
result.append([email_to_name[email]]
+ sorted(component)) ← name + sorted component
return result
💡 Complexity Summary

Let E = total emails across all accounts (sum of all account lengths minus names).
Union-Find: Time O(E·α(E)) for unions + O(E log E) for final sort = O(E log E), Space O(E)
DFS: Time O(E log E) (graph build O(E) + DFS O(E) + sort O(E log E)), Space O(E)
Both are O(E log E) overall due to the sort step.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify the Pattern Signal

"Merge groups that share a common element" is the classic Union-Find signal. The shared element is the edge connecting groups. Here, a shared email is the edge between two accounts — it proves they belong to the same person.

Step 2: Decide the Node Universe

Key question: what are the nodes? Emails, not accounts. If we use accounts as nodes, we'd have to build edges between every pair of accounts that share an email — more work. With emails as nodes, we simply union all emails within each account to one anchor, and the bridges form automatically.

Step 3: Propose DFS First, Then Upgrade

In an interview, mention the DFS approach first ("I can build an adjacency list and find connected components via DFS"). Then say: "But Union-Find is more elegant — no explicit edge list needed, and find/union are nearly O(1)."

Step 4: Handle the Name Lookup

One subtle point: how do we know the name for each component? Store email_to_name for every email. When collecting results, use email_to_name[root] — the root is always a real email in the input, so it has a stored name.

🗣️ What to Say to the Interviewer

"This is a connected components problem — emails are nodes, shared accounts create edges.
I'll use Union-Find: initialize each email as its own component.
For each account, union all emails to the first email as the anchor.
When accounts share an email, the union automatically bridges their entire email sets.
Finally, collect emails by root, sort each group, prepend the stored name.
Time O(E log E) dominated by sorting. Space O(E) for the parent map."
🔮 Likely Follow-up Questions
  • "Can you do it without Union-Find?" → Yes, DFS/BFS on adjacency list (shown above)
  • "What if there are millions of accounts?" → Path compression scales; no full graph materialized
  • "Can the same email appear in accounts with different names?" → Problem assumes consistent naming for same person; if not, store any name for the root
  • "How do you handle one account with duplicate emails?" → union(x,x) is a no-op since find(x)==find(x)

7 Summary & Key Takeaways ▾
Emails are the Union-Find nodes — NOT accounts; this is the crucial reframe
For each account, union every email to the first email (the anchor)
A shared email automatically bridges two accounts into one component
After all unions, collect emails by root, sort, and prepend the name
Path compression makes repeated find() calls essentially O(1) amortized
Name lookup uses email_to_name[root] — the root is always a real email in the input

  • Transitive grouping: A~B, B~C → A~C automatically → Union-Find
  • "Are A and B in the same group?" in O(α(n)) → Union-Find
