LeetCode #632

Smallest Range Covering Elements from K Lists

Given k sorted lists of integers, find the smallest range [a, b] that includes at least one element from each list.

Hard Heaps Min Heap + Track Max Time: O(n·k·log k) Space: O(k)
1 Understanding the Problem ▾

You have k sorted lists of integers. Find the smallest range [a, b] such that:

  • The range contains at least one element from every list
  • The range size b − a is minimized
  • On ties (equal size), pick the range with the smaller left bound
💡 Key Insight

Any valid range is anchored by real elements: the left bound is always some actual element from one list, and the right bound is always some actual element from another. We never need to consider ranges with "floating" boundaries. So the problem reduces to: pick exactly one element from each list such that (max − min) is smallest.

Example walkthrough: nums = [[4,10,15,24,26], [0,9,12,20], [5,18,22,30]]

Edge cases: k = 1 (any single element is valid, pick smallest range [v, v]); all lists length 1 (only one possible selection); duplicate values across lists; very large values (Python handles natively); empty lists (problem guarantees each list is non-empty).

2 The Algorithm: Min Heap + Track Max ▾

Think of each list as a sorted stream. We maintain a sliding window across all k streams simultaneously: a min-heap always holds exactly one "live" element per list (the current frontier), and a variable cur_max tracks the maximum across those k elements. The current range is [heap_min, cur_max].

1 Seed the heap: Push (value, list_index, element_index) for the first element of every list. Track cur_max = max of all those first elements. Record initial range = [heap[0][0], cur_max].
2 Pop the minimum: heappop removes the smallest element in the current frontier — call it (val, list_i, idx_j).
3 Termination check: If list i has no more elements (idx_j + 1 ≥ len(nums[i])), we can no longer cover all k lists — stop.
4 Advance and push: Push nums[i][idx_j + 1] into the heap. Update cur_max = max(cur_max, new_val). Check if [heap[0][0], cur_max] is the new best.
5 Repeat until termination. Return best range.
🔑 The Greedy Insight — Why Advance the Minimum?

The range is [min_val, max_val] across our k chosen frontier elements. Advancing the maximum only makes the range bigger. Advancing the minimum replaces it with the next element from the same list, which is guaranteed to be larger (the list is sorted). This raises the floor, potentially shrinking the gap to the ceiling. It's the only productive move, so it's optimal to always advance the minimum.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every iteration, the heap contains exactly one element from each of the k lists — always the frontier element (smallest unprocessed) for that list. Therefore [heap_min, cur_max] is always a valid range (covers all lists), and it is the tightest range for the current set of frontier elements.

Completeness argument (no valid range is skipped): Suppose the optimal range is [a, b] where a comes from list p. At some point in our algorithm, all elements before a in list p will have been popped. When we pop a, it is the heap minimum. At that moment, all other lists have frontier elements ≥ their respective elements in the optimal range. Therefore cur_max ≥ b. We record a range of size ≤ b − a, so we cannot miss the optimal.

Why stop when a list is exhausted: Once any list is empty, we have no element to contribute from it for any future range. The current and all future states cannot form valid ranges. We've already explored every configuration where all lists were represented.

⚠️ cur_max Never Decreases — Here's Why

When we pop the minimum and push the next element from the same list, that next element is larger (sorted list). So the heap's minimum can only rise (or stay). But the maximum? It was set by some element already in the heap — we never remove the maximum until it becomes the minimum. So cur_max = max(cur_max, new_val) is correct; we never need to recompute it from scratch.

4 The Code (Annotated) ▾

Solution 1 — Min Heap + Track Max (Optimal · O(nk log k) · Interview Pick)

import heapq
def smallestRange(nums): ← nums: list of k sorted lists
heap = [] ← min-heap of (value, list_idx, elem_idx)
cur_max = float('-inf') ← track max across all heap elements
for i, lst in enumerate(nums): ← seed heap with first elem from each list
heapq.heappush(heap, (lst[0], i, 0)) ← tuple: value, which list, index in list
cur_max = max(cur_max, lst[0]) ← initial max across all first elements
best = [heap[0][0], cur_max] ← first valid range: [heap root, max]
while True:
val, i, j = heapq.heappop(heap) ← pop smallest current frontier element
if j + 1 >= len(nums[i]): ← list i exhausted → can't cover all lists
break
nxt = nums[i][j + 1] ← next element from the same list
heapq.heappush(heap, (nxt, i, j + 1)) ← replace popped element in heap
cur_max = max(cur_max, nxt) ← max can only grow (sorted list)
new_min = heap[0][0] ← new heap root is new global minimum
if cur_max - new_min < best[1] - best[0]:← tighter than current best?
best = [new_min, cur_max] ← update best (smaller left bound wins ties)
return best
🔬 Algorithm Deep-Dive: Min-Heap for Merging K Sorted Streams

Origin & Class: Generalization of "Merge K Sorted Lists." A min-heap of size k maintains a sorted frontier across all lists simultaneously, giving O(log k) per advance instead of O(k) linear scan of all frontiers.

Core Invariant: Exactly one "live" element per list in the heap at all times — always the smallest unprocessed element from that list. The heap root is the global minimum across all live elements.

Complexity: O(nk) total elements processed × O(log k) per heap operation = O(nk log k). Space O(k) for the heap.

Recognize this pattern when: "find something optimal by choosing one element from each of k sorted sequences" — also applies to LC #23 (Merge K Lists), #378 (Kth Smallest Matrix), #373 (K Pairs Smallest Sums).

💡 Which to Use in an Interview?

Min Heap solution is preferred. It uses O(k) space vs. O(nk) for the flattened approach, and the heap insight (maintain exactly one representative per list) demonstrates a deeper understanding of the problem structure. The sliding window is valid but costs more memory and is trickier to explain.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Reframe the problem

"I need one element from each of k lists, and I want (max_chosen − min_chosen) to be as small as possible. This is fundamentally about controlling the spread of a k-element selection."

Step 2: Recognize the k-sorted-list pattern

"The lists are sorted, which means if I hold one element per list, advancing any pointer gives me a larger element from that list. I can use a min-heap to always know which element is the global minimum among my k chosen elements."

Step 3: Figure out the greedy move

"The range is [min, max]. To shrink it, I should try to raise the minimum. I cannot decrease the maximum — it's fixed by elements already in the heap. So: always pop the minimum and advance that list's pointer. This is the only productive move."

Step 4: Handle termination and max tracking

"Stop when any list is exhausted (can't maintain coverage). For the maximum: it only grows, so track it with a running variable — no need for a max-heap."

🗣️ What to Say to the Interviewer

"I need one element from each of k sorted lists such that max − min is minimized.

I'll use a min-heap seeded with the first element from each list, plus a cur_max variable.
At each step: the heap root is the range minimum, cur_max is the range maximum.
I pop the minimum — this is the only element I can replace to potentially shrink the range.
I push the next element from the same list and update cur_max.
I stop when any list is exhausted. Time O(nk · log k), Space O(k)."
🔮 Likely Follow-up Questions
  • "What if the lists are unsorted?" → Sort each first; add O(nk log n) preprocessing
  • "What if k = 1?" → Any single element works; minimum range is [lst[0], lst[0]]
  • "Can you use a different sliding window approach?" → Yes: flatten + sort + two-pointer with coverage counter (higher space cost)
  • "How does this relate to Merge K Sorted Lists?" → Same heap structure; here we stop at the optimal range instead of outputting the full merged stream

7 Summary & Key Takeaways ▾
Any valid range is bounded by actual elements: left bound = some list's element, right bound = some other list's element
Min-heap of size k maintains the frontier minimum in O(log k) per operation; track cur_max separately with a variable
Greedy rule: always advance the minimum pointer — the only move that can shrink the range
cur_max never decreases (sorted lists: the next element from any list is always ≥ the current one)
Stop when any list is exhausted — all valid range configurations have been explored
Space is O(k), regardless of how long the lists are — only k entries in the heap at once
