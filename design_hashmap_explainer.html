Design HashMap

LeetCode 706 — Implement a hash map from scratch with put, get, and remove

Easy Hash Table Separate Chaining O(n/k) per op
1 Understanding the Problem ▾

Build the engine behind Python's dict — a data structure that stores key-value pairs and supports insert, lookup, and delete in near-constant time, all without using any built-in hash table library.

We need three operations: put(key, value) to insert or update, get(key) to retrieve a value (or -1 if absent), and remove(key) to delete an entry.

💡 Key Insight

Use a hash function to convert any key into a fixed-range array index. When two different keys map to the same index (collision), store them in a chain at that bucket. This gives O(1) average-case access.

Walkthrough of the example:

put(1,1) → store (1,1) in bucket 1. put(2,2) → store (2,2) in bucket 2. get(1) → hash to bucket 1, find key 1, return 1. get(3) → bucket 3 is empty → return -1. put(2,1) → bucket 2 already has key 2, update value to 1. remove(2) → bucket 2, find and remove key 2.

2 The Algorithm: Separate Chaining ▾

A hash table has two ingredients: a hash function that maps keys to bucket indices, and a collision resolution strategy. We use separate chaining — each bucket holds a list of all entries that hash to it.

1 Allocate an array of 769 empty lists (buckets). Using a prime number minimises collision clustering.
2 Hash function: h(key) = key % 769. Maps any integer key to a bucket index 0–768.
3 put: Compute bucket index → scan chain for existing key → update value if found, else append new pair.
4 get: Compute bucket index → scan chain → return value if found, else -1.
5 remove: Compute bucket index → scan chain → delete pair if found.

3 Why It Works: The Guarantees ▾
🔐 The Invariant

Every key k lives in exactly one bucket: buckets[k % 769]. Within that bucket, keys are unique — put either updates an existing entry or appends a new one.

Correctness: The hash function is deterministic — the same key always maps to the same bucket. A linear scan of that bucket's chain guarantees we find, update, or confirm absence of any key.

Performance: With 769 buckets and at most 10⁴ operations, the average chain length is 10000/769 ≈ 13. Each operation scans at most one chain, giving O(n/k) average time where n is entries and k is bucket count.

⚠️ Why 769? Why Prime?

A prime bucket count reduces clustering. If keys share a common factor with the bucket count (e.g., all even keys with 1000 buckets), many buckets stay empty while others overflow. A prime like 769 distributes keys more uniformly across buckets.

Edge cases: get/remove on non-existent key → scan finds nothing, return -1 or no-op. put with key 0 → valid, hashes to bucket 0. Value 0 is valid (we use -1 as "not found" sentinel, not stored value).

4 The Code (Annotated) ▾
# Separate Chaining — Interview Pick
class MyHashMap:
def __init__(self):
self.size = 769 # prime bucket count
self.buckets = [[] for _ in range(self.size)] # 769 empty chains
def _hash(self, key): # maps key → bucket index
return key % self.size
def put(self, key, value): # insert or update
h = self._hash(key) # find the bucket
for i, (k, v) in enumerate(self.buckets[h]): # scan chain
if k == key: # key exists?
self.buckets[h][i] = (key, value) # update in place
return
self.buckets[h].append((key, value)) # new → append to chain
def get(self, key): # lookup
h = self._hash(key)
for k, v in self.buckets[h]: # scan chain
if k == key: # found!
return v
return -1 # not in this bucket
def remove(self, key): # delete
h = self._hash(key)
for i, (k, v) in enumerate(self.buckets[h]): # scan chain
if k == key: # found!
self.buckets[h].pop(i) # remove from list
return
⚠️ Common Mistake

Forgetting to return after update: In put, if the key already exists, you must return after updating. Otherwise the code falls through to append and creates a duplicate entry — get would return the first one, but remove would only delete one copy.

📊 Complexity

Time: O(n/k) per operation on average, where n = number of stored keys and k = 769 buckets. With at most 10⁴ calls, average chain length ≈ 13.
Space: O(k + n) = O(769 + n) for the bucket array plus all stored entries.

5 🎤 Interview Strategy ▾
Step 1: Recognize the Pattern

"Design X without built-in Y" → implement from primitives. This is a system design / data structure design problem. We need to choose a hash function and a collision strategy.

Step 2: Discuss Trade-offs

Mention the Direct Address option (O(1) but O(10⁶) space). Then propose Separate Chaining — better space, still O(1) amortised. Mention Open Addressing exists but chaining is simpler to implement correctly.

Step 3: Choose Bucket Count

Use a prime number near √(max_key) or a round number. 769 is a common choice. Explain briefly: "Prime reduces collision clustering from modular arithmetic."

Step 4: Implement Cleanly

Separate the hash function into its own method. Keep put/get/remove symmetric — each starts with hash computation, then scans the chain. This makes the code easy to explain.

🗣️ What to Say to the Interviewer

"I'll use separate chaining with a prime-sized bucket array. Each bucket stores a list of key-value pairs. The hash function is key mod 769. For put, I hash the key, scan the chain to check for duplicates, and either update or append. Get and remove follow the same hash-then-scan pattern. Average time per operation is O(n/k) where k is the bucket count."
🔄 Follow-up Questions
  • "What about resizing?" — When load factor exceeds threshold (e.g., 0.75), double bucket count and rehash all entries. Amortised O(1).
  • "Open addressing vs chaining?" — Open addressing uses less memory (no pointers) but is harder to delete from; chaining is simpler and handles high load better.
  • "Better hash function?" — Multiplicative hashing, universal hashing, or Python-style with perturbation.
