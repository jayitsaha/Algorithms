LeetCode #210

Course Schedule II

Given numCourses and a list of prerequisite pairs, return a valid ordering in which you can finish all courses, or [] if it's impossible.

Medium Graph Topological Sort Time: O(V+E) Space: O(V+E)
1 Understanding the Problem ▾

You have numCourses courses labeled 0 to numCourses - 1. Some have prerequisites: the pair [a, b] means "you must take course b before course a." Your job: return a valid order to take all courses, or return [] if it's impossible (meaning there's a circular dependency).

💡 Key Insight

Model the courses as nodes and prerequisites as directed edges in a graph. A valid ordering exists if and only if the graph contains no cycle. Finding that ordering is called a topological sort. This is exactly what Kahn's algorithm does — peel off courses with no remaining prerequisites, one BFS level at a time.

✓ Valid — numCourses=4, prereqs=[[1,0],[2,0],[3,1],[3,2]]

Graph: 0→1, 0→2, 1→3, 2→3
No cycle exists
Return: [0, 1, 2, 3] ✓

✗ Impossible — prereqs=[[0,1],[1,0]]

Graph: 0→1 AND 1→0
Cycle: 0 ↔ 1
Return: [] (impossible)

Edge cases to consider: no prerequisites at all (any order works, return any permutation), a single course (return [0]), a complete chain (only one valid order), a self-loop (course is its own prerequisite — impossible), and a graph where a subset is cyclic (entire answer is []).

2 The Algorithm: Kahn's BFS Topological Sort ▾

Think of it like a course registration system. In-degree is the number of prerequisites still required. Start with every course that has zero prerequisites — those can be taken immediately. As you "complete" them, you remove their contribution to others' counts. When a course's in-degree reaches 0, it becomes available.

1 Build the graph + compute in-degrees: Create an adjacency list. For each pair [a, b], add edge b → a (b must come before a, so completing b unlocks a). Simultaneously, increment indegree[a] — course a has one more prerequisite.
2 Seed the queue: Add all courses with indegree == 0 to a deque. These courses have no prerequisites and can be taken immediately. If there are none, a cycle exists from the start.
3 BFS loop — peel one course at a time: Dequeue a course, append it to order. For each course this unlocks (its neighbors in the graph), decrement their in-degree by 1. If a neighbor's in-degree hits 0, enqueue it — it's now available.
4 Cycle check at the end: If len(order) == numCourses, we scheduled everything — return order. If not, at least one course is stuck in a cycle (its in-degree never reached 0) — return [].
🔑 Algorithm Deep-Dive: Kahn's Algorithm (1962)

Origin: Arthur Kahn, 1962. Produces a topological ordering of any DAG in O(V+E) time using a simple BFS. Also detects cycles as a free side-effect.

Core invariant: When a course is dequeued, every one of its prerequisites has already appeared in the result. This is guaranteed because a course only enters the queue when all edges pointing to it have been "consumed" by prior BFS steps.

Why cycle detection is embedded: Nodes in a cycle always have at least one incoming edge from within the cycle. These edges are never consumed (the source never becomes processable), so those nodes never reach in-degree 0, never enter the queue, and never join the result. The final count check reveals the gap.

Recognize when: "Must do X before Y," "is a schedule possible?", "dependency ordering," "minimum steps to process all tasks."

3 Why It Works: The Invariant ▾
🔐 The Invariant

At the moment course X is appended to order, all courses that X requires have already been appended. This holds because: X enters the queue only when indegree[X] == 0. That means every course with an edge pointing to X has already been processed (dequeued and appended), which is the only way those edges get "consumed" (decremented). So the prerequisite chain is satisfied in the correct sequence.

Proof of correctness (by contradiction): Suppose course X is appended but some prerequisite P of X was not yet appended. Then edge P→X was never decremented. That means X's in-degree was never fully zeroed. But we only enqueue X when in-degree == 0 — contradiction. Therefore every appended course has all prerequisites already in the result.

Proof of cycle detection: Let C₁→C₂→…→Cₖ→C₁ be a cycle. Every node Cᵢ has at least one incoming edge from Cᵢ₋₁ (within the cycle). This edge is never consumed because Cᵢ₋₁'s in-degree never reaches 0 (for the same reason — Cᵢ₋₂ can't be processed...). By induction, no node in the cycle ever reaches in-degree 0. They never enter the queue. They never reach order. Final length check fails → return [].

⚠️ Common Mistake: Reversing the Edge Direction

For pair [a, b] meaning "b before a," the edge must go b → a in the graph. This represents "completing b unlocks a." Many beginners write graph[a].append(b) instead of graph[b].append(a). The in-degrees then get computed for the wrong nodes and the whole algorithm produces garbage results — often subtly wrong, passing easy cases but failing complex prerequisites chains.

No prerequisites case: Every course starts with in-degree 0. All enter the queue immediately. BFS processes them all. result length = numCourses. Valid — any permutation works, and we return one.

4 The Code (Annotated) ▾

Solution 1 — Kahn's BFS (Optimal — Interview Pick)

from collections import deque
def findOrder(numCourses, prerequisites):
graph = [[] for _ in range(numCourses)] ← graph[b] = list of courses that b unlocks
indegree = [0] * numCourses ← count of prerequisites each course still needs
for a, b in prerequisites: ← [a,b]: must take b before a
graph[b].append(a) ← b→a: completing b makes a available
indegree[a] += 1 ← course a requires one more course
queue = deque(i for i in range(numCourses)
if indegree[i] == 0) ← courses with no prerequisites can start now
order = [] ← accumulate valid schedule here
while queue: ← process until no more available courses
course = queue.popleft() ← take next available course (FIFO)
order.append(course) ← this course is "completed" — add to schedule
for neighbor in graph[course]: ← unlock courses that depended on this one
indegree[neighbor] -= 1 ← one fewer prerequisite remaining
if indegree[neighbor] == 0: ← all prereqs done → now available!
queue.append(neighbor) ← add to queue for processing
return order if len(order)==numCourses else []← if missing courses, cycle exists
⚠️ Why check len(order) == numCourses?

If a cycle exists, the stuck courses never reach in-degree 0 — they never enter the queue — they never get appended to order. So the result is shorter than numCourses. One length check catches all cycle scenarios, no matter how many cycles or how complex they are. Elegant.

Solution 2 — DFS with 3-Color Marking (Recursive Alternative)

def findOrder(numCourses, prerequisites):
graph = [[] for _ in range(numCourses)]
for a, b in prerequisites:
graph[b].append(a) ← same direction: b→a
WHITE, GRAY, BLACK = 0, 1, 2 ← unvisited, in DFS stack, fully done
color = [WHITE] * numCourses
result = []
def dfs(node): ← returns False if cycle found
if color[node] == GRAY: return False ← back-edge to ancestor → cycle!
if color[node] == BLACK: return True ← already fully explored → skip
color[node] = GRAY ← mark "in current DFS path"
for neighbor in graph[node]:
if not dfs(neighbor): return False
color[node] = BLACK ← all descendants explored, mark done
result.append(node) ← post-order: add AFTER all dependencies
return True
for i in range(numCourses):
if not dfs(i): return []
return result[::-1] ← reverse post-order = topological order
💡 Complexity Summary

Kahn's BFS: Time O(V+E), Space O(V+E). Every node enqueued/dequeued once; every edge processed once.
DFS 3-Color: Time O(V+E), Space O(V+E) + O(V) recursion stack. Risk of stack overflow for very deep graphs.
Brute Force: Try all V! permutations, validate each against prerequisites — O(V! × E). Completely impractical.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize "Dependency Ordering" = Topological Sort

Whenever you see "must do X before Y," "prerequisites," "dependency chain," or "is this schedule possible?" — think directed graph + topological sort. This pattern is extremely common in system design problems too (build systems, task schedulers, module loaders).

Step 2: Model the Problem as a Graph Explicitly

"I'll model courses as nodes. Pair [a,b] means edge b→a — completing b unlocks a. The question becomes: find a topological order of this DAG, or report that it's cyclic." Say this out loud — it shows clarity.

Step 3: Propose Kahn's BFS (preferred over DFS)

"I'll use Kahn's algorithm. It's iterative — no recursion stack overflow risk. Cycle detection is built-in: if result length is less than numCourses, a cycle prevented some nodes from being processed."

Step 4: Derive the 4-step Structure from First Principles

"Step 1: Build graph + compute in-degrees in one pass. Step 2: Seed queue with all in-degree-0 courses. Step 3: BFS — dequeue, append to result, decrement neighbors' in-degrees, enqueue those that hit 0. Step 4: Length check for cycle."

🗣️ What to Say to the Interviewer

"This is a topological sort problem. The 'must take b before a' relationships form a directed graph.
I'll use Kahn's algorithm: compute in-degrees, seed a queue with in-degree-0 courses,
then BFS — dequeue a course, append to result, decrement its neighbors' in-degrees.
When a neighbor hits 0, enqueue it. At the end, if result contains all courses, return it.
If not, a cycle exists — return []. Time O(V+E), Space O(V+E) where V=numCourses."
