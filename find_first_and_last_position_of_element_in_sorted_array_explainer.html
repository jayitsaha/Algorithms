LeetCode #34

Find First and Last Position of Element in Sorted Array

Given a sorted array of integers and a target value, return [first_index, last_index] of the target. Return [-1, -1] if not found. Must run in O(log n) time.

Medium Binary Search Two Binary Searches Time: O(log n) Space: O(1)
1 Understanding the Problem ▾

You have a sorted array where numbers appear in non-decreasing order. Your target may appear multiple consecutive times, forming a contiguous block — or not at all. You need two answers: the leftmost index where the target appears, and the rightmost index. If the target is absent, return [-1, -1].

💡 Key Insight

Because the array is sorted, all copies of the target form a contiguous block. Finding the range reduces to finding the left boundary and right boundary of that block. Modified binary search — where we keep going even after finding the target — finds these boundaries in O(log n).

✓ target=8 in [5,7,7,8,8,10]

Index: 0 1 2 3 4 5
Value: 5 7 7 8 8 10
8 appears at indices 3 and 4
Answer: [3, 4]

✗ target=6 in [5,7,7,8,8,10]

Index: 0 1 2 3 4 5
Value: 5 7 7 8 8 10
6 is not in the array at all
Answer: [-1, -1]

The O(log n) constraint is the decisive signal. A left-to-right scan finds both boundaries in O(n) but that violates the requirement. Sorted array + O(log n) = binary search, always.

Edge cases: empty array → [-1, -1]; target not present → [-1, -1]; entire array equals target (e.g., [8,8,8]) → [0, n-1]; single-element array; target at position 0 or n-1.

2 The Algorithm: Two Independent Binary Searches ▾

Standard binary search finds some index where the target lives — not necessarily the first or last. The modification: when nums[mid] == target, instead of returning, record the index and keep narrowing toward the boundary you care about.

1 Binary Search for First Position (biased left): When nums[mid] == target, save mid as a candidate and set right = mid - 1. This shrinks the window leftward — we might find an even earlier occurrence.
2 Binary Search for Last Position (biased right): Same logic but when nums[mid] == target, save mid and set left = mid + 1. This shrinks the window rightward — we might find a later occurrence.
3 Combine: Return [first, last]. Short-circuit: if the first search returns -1, the target is absent — skip the second search and return [-1, -1] immediately.
How the two searches differ when they hit a match:
Find First:
mid < target → go right
mid == target → save, go LEFT ←
mid > target → go left
Find Last:
mid < target → go right
mid == target → save, go RIGHT →
mid > target → go left
🔑 Why Two Full Searches, Not "Find One Then Expand"?

"Find one occurrence, then scan left/right" degrades to O(n) in the worst case — imagine [8,8,8,...,8] with n elements. Running two binary searches on the full original array keeps each at O(log n), giving a total of O(2 log n) = O(log n).

3 Why It Works: The Invariant ▾
🔐 The Invariant (Find First Search)

At every iteration, the first occurrence of target (if it exists) lies within [left, right]. When we find the target at mid, we record it but set right = mid - 1: "I have a valid answer, but there might be a better one in [left, mid-1]." When the window empties, the last recorded index is provably the leftmost occurrence.

Why "keep going left" finds the first: The array is sorted so all target copies are contiguous. If target is at mid, any target to its left has a smaller index. By shrinking right, we never miss a leftward target. The final recorded mid is the smallest index with the target value.

Symmetric reasoning for Find Last: Setting left = mid + 1 after a match explores right-of-mid — any target there has a larger index. The final recorded mid is the rightmost occurrence.

⚠️ Common Mistake: Returning Inside the Loop

Writing if nums[mid] == target: return mid gives you a position but not the first or last. Example: [1,2,2,2,3], target=2. The first mid lands at index 2 (the middle 2), but the correct first position is index 1. You must save the index and keep going.

Target absent: If nums[mid] never equals target, the result variable stays -1 and is returned as-is — no extra logic needed.

Integer overflow note (Java/C++): Use mid = left + (right - left) // 2 not (left + right) // 2. In Python integers don't overflow, but this habit matters in typed languages.

4 The Code (Annotated) ▾

Solution 1 — Two Binary Searches (Optimal, Interview Pick)

def searchRange(nums, target): ← Returns [first_idx, last_idx]
def find_boundary(go_left): ← Single helper handles both directions
left, right = 0, len(nums) - 1 ← Full array search space
result = -1 ← Pessimistic start: not found
while left <= right: ← Standard binary search loop
mid = left + (right - left) // 2 ← Safe midpoint (avoids overflow)
if nums[mid] == target: ← Match found — save it
result = mid ← This index is a valid boundary candidate
if go_left: ← Finding FIRST occurrence?
right = mid - 1 ← Go left: earlier match possible
else: ← Finding LAST occurrence
left = mid + 1 ← Go right: later match possible
elif nums[mid] < target: ← Target is to the right
left = mid + 1
else: ← nums[mid] > target; go left
right = mid - 1
return result ← -1 if never found; else boundary index
first = find_boundary(go_left=True) ← Leftward-biased binary search
if first == -1: ← Target not in array at all
return [-1, -1] ← Short-circuit: skip second search
last = find_boundary(go_left=False) ← Rightward-biased binary search
return [first, last] ← [leftmost_index, rightmost_index]
⚠️ Why not return immediately inside the loop when target is found?

An immediate return gives you some position, but binary search doesn't guarantee it's the extreme one. The loop must run to completion (search space exhausted) to guarantee the saved result is the true boundary. Counterexample: [2,2,2,2,2], target=2 — early return gives index 2; the correct first is 0, the correct last is 4.

Solution 2 — Brute Force Linear Scan (O(n))

def searchRange(nums, target):
first = last = -1 ← Track first/last hit as we scan
for i, v in enumerate(nums):
if v == target:
if first == -1: first = i ← Set first only on the very first hit
last = i ← Always update last; final value is rightmost
return [first, last]
💡 Complexity Summary

Two Binary Searches (Optimal): Time O(log n) — 2× log n comparisons; Space O(1) — only a few integer variables
Linear Scan (Brute Force): Time O(n) — scans every element; Space O(1) — but violates the problem's O(log n) constraint
Python built-ins: bisect.bisect_left() and bisect.bisect_right() implement exactly this — but implement from scratch in interviews to demonstrate understanding.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Binary Search Signal

Three conditions appear together: (1) sorted array, (2) search for a value, (3) explicit O(log n) requirement. This triple combination mandates binary search — no other approach achieves it for this problem.

Step 2: State the Brute Force First

"Naively I could scan left-to-right, recording the first and last hit — O(n) time. But the problem requires O(log n), so I need binary search." This shows you understand when and why to reach for the advanced tool.

Step 3: Articulate the Key Modification

"Standard binary search returns immediately when it finds the target. I can't do that here — I need the boundary. So when I find the target, I'll save the index and keep narrowing toward the side I care about. Two separate searches, one biased left and one biased right."

Step 4: Mention the Short-Circuit Optimization

"If the first search returns -1, the target isn't in the array — I can skip the second search entirely and return [-1, -1] immediately. This avoids a redundant O(log n) pass."

🗣️ What to Say to the Interviewer

"Sorted array + O(log n) → binary search. I'll run it twice with a small modification.
For the first occurrence: normal binary search, but when nums[mid] == target,
I save mid and set right = mid - 1 to keep exploring left.
For the last occurrence: same, but when nums[mid] == target,
I save mid and set left = mid + 1 to keep exploring right.
Each search runs independently on the full array in O(log n).
Total: O(2 log n) = O(log n) time, O(1) space."
🔮 Likely Follow-up Questions
  • "Count occurrences of target?" → last - first + 1 when first != -1, else 0.
  • "What if array is not sorted?" → Binary search impossible; O(n) scan is the best you can do.
  • "Can you use Python's bisect module?" → Yes: bisect_left gives first position, bisect_right - 1 gives last. But demonstrate from scratch first.
  • "What if there are multiple targets and you only need the count?" → bisect_right(target) - bisect_left(target) in O(log n).

7 Summary & Key Takeaways ▾
Sorted array + O(log n) constraint = binary search, always
Run binary search twice: once biased left (find first), once biased right (find last)
When target found, save index and keep narrowing — do NOT return immediately
Short-circuit: if first search returns -1, skip second search and return [-1, -1]
Use one helper with a boolean flag — avoids duplicating nearly-identical code
Count occurrences: last - first + 1 (when first != -1)
