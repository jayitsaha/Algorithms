LeetCode #1135

Connecting Cities With Minimum Cost

Given n cities and a list of connections with costs, return the minimum cost to connect all cities — or -1 if impossible.

Medium Graph Kruskal's Algorithm Union-Find Time: O(E log E) Space: O(N)
1 Understanding the Problem ▾

You have n cities labeled 1 through n, and a list of possible roads — each described as [city1, city2, cost]. You want to build the cheapest set of roads that connects every city to every other city, directly or through intermediary cities. Return the total construction cost, or -1 if it's impossible with the given roads.

💡 Key Insight

This is the classic Minimum Spanning Tree (MST) problem. A spanning tree connecting n nodes needs exactly n−1 edges (no more, no less) with no cycles. The MST is the spanning tree with the minimum total edge weight — the cheapest network of roads that keeps all cities reachable from each other.

n−1 = 3 edges needed
Only 2 connected → -1 ✗

Edge cases to consider: n=1 (already connected, return 0), disconnected graph (not enough roads to connect all cities, return -1), multiple roads between same pair (we'll always pick cheapest), all cities already connected with just one road.

2 The Algorithm: Kruskal's MST + Union-Find ▾

Kruskal's algorithm is a greedy approach: always add the cheapest road that doesn't create a cycle. The magic is in detecting cycles efficiently — that's where Union-Find (Disjoint Set Union, DSU) comes in. It tracks which cities are already connected in near-constant time.

1 Sort all roads by cost (ascending). This is the greedy foundation — we evaluate cheaper connections before expensive ones, ensuring the total cost is minimized.
2 Initialize Union-Find: each of the n cities starts as its own independent component. We have n components initially and need to reduce to exactly 1.
3 Process each road in sorted order: call find(u) and find(v) to get the root representative of each city's component. If the roots differ, these cities are in separate components.
4 Accept or reject: If different roots → union(u, v), add cost, decrement component count. Same root → skip (adding would create a cycle, violating the "spanning tree" requirement).
5 Termination: Stop as soon as components reaches 1 (all connected). If we exhaust all roads with components > 1, the graph is disconnected — return -1.
🔑 Why Greedy "Cheapest First" is Provably Optimal

The Cut Property of MSTs guarantees this: for any way to split the cities into two groups S and V−S, the minimum-weight road crossing between the groups must appear in some MST. Kruskal's exploits this property repeatedly — each accepted edge is the cheapest road crossing some cut at that moment, so it's always optimal to include it.

3 Why It Works: Union-Find Invariant ▾
🔐 The Core Invariant

At any point, find(x) returns the canonical root representative of x's connected component. Two cities x and y are in the same component if and only if find(x) == find(y). Merging two components always reduces the total count by exactly 1.

Path Compression: When find(x) traverses up to the root, it flattens the path — every node encountered points directly to the root. This makes future lookups nearly O(1). The amortized cost becomes O(α(n)) where α is the inverse Ackermann function, essentially a constant for all practical n.

Union by Rank: When merging components, attach the shallower tree under the root of the deeper tree. This keeps the maximum tree height at O(log n), complementing path compression to achieve the near-constant amortized cost.

Why exactly n−1 edges? A spanning tree on n vertices is a connected acyclic graph, which always has exactly n−1 edges by definition. Each accepted edge reduces the component count by 1 (from n down to 1), so we always accept exactly n−1 edges to span the entire graph.

⚠️ Critical: Always check for disconnected graphs

After the loop, if components > 1, some cities are unreachable from others — return -1. This check is easy to forget when implementing Kruskal's. The component count tracker handles this elegantly: we decrement on each accepted edge and check at the end.

4 The Code (Annotated) ▾
🔬 Algorithm Deep-Dive: Kruskal's MST (1956)

Origin: Joseph Bernard Kruskal Jr., 1956. Solves the Minimum Spanning Tree problem on weighted undirected graphs in O(E log E) time.

Core invariant: After processing k edges, the selected edges form a spanning forest — a collection of trees — with the minimum weight among all forests that can be extended to a valid MST. Each accepted edge is provably part of some MST (by the Cut Property).

Exchange argument (why it's optimal): Suppose Kruskal's accepted edge e but some optimal MST T* does not contain e. Adding e to T* creates a cycle. That cycle must contain some other edge e' crossing the same cut that e crosses. Since Kruskal's sorted by weight and accepted e before e', we know cost(e) ≤ cost(e'). Swapping e' for e gives a spanning tree with cost ≤ T*, contradicting T* being optimal.

Recognize when: "Connect all nodes with minimum total edge weight" → MST → Kruskal's or Prim's. Choose Kruskal's when edges are sparse or given as an edge list (easier to sort). Choose Prim's for dense graphs (adjacency matrix).

Solution 1 — Kruskal's with Union-Find (Optimal, Interview Pick)

def minimumCost(n: int, connections: list) -> int:
parent = list(range(n + 1)) ← 1-indexed: parent[i] = i means i is its own root
rank = [0] * (n + 1) ← Rank (proxy for tree height) for union by rank
def find(x): ← Returns root of x's component (with path compression)
if parent[x] != x:
parent[x] = find(parent[x]) ← Flatten: x points directly to root
return parent[x]
def union(x, y) -> bool: ← Merge x and y's components; return False if already same
rx, ry = find(x), find(y)
if rx == ry: return False ← Same root = same component = adding would cycle
if rank[rx] < rank[ry]: rx, ry = ry, rx ← Attach smaller-rank tree under larger
parent[ry] = rx ← ry's entire tree now hangs under rx
if rank[rx] == rank[ry]: rank[rx] += 1 ← Only increase rank on equal-rank merge
return True ← Successfully merged two distinct components
connections.sort(key=lambda e: e[2]) ← Sort edges cheapest first — O(E log E)
total_cost = 0 ← Accumulate the MST edge costs
components = n ← Start with n isolated components
for u, v, cost in connections:
if union(u, v): ← Edge bridges two different components
total_cost += cost
components -= 1
if components == 1: ← All cities connected — early exit!
return total_cost
return -1 ← Exhausted all edges, still disconnected
⚠️ Off-by-one: cities are 1-indexed!

The problem uses cities numbered 1 to n (not 0 to n−1). Make parent and rank arrays of size n+1 so index n is valid. A classic bug: using list(range(n)) only covers 0..n−1, causing city n to cause an IndexError or silently corrupt results.

Solution 2 — Prim's Algorithm (alternative, better for dense graphs)

import heapq
from collections import defaultdict
def minimumCost_prims(n, connections):
graph = defaultdict(list) ← Build adjacency list
for u, v, w in connections:
graph[u].append((w, v))
graph[v].append((w, u)) ← Undirected: add both directions
heap = [(0, 1)] ← Start from city 1 with cost 0
visited = set(); total = 0; count = 0
while heap and count < n:
cost, city = heapq.heappop(heap)
if city in visited: continue ← Skip already-processed cities
visited.add(city); total += cost; count += 1
for w, nxt in graph[city]: ← Explore neighbors
if nxt not in visited:
heapq.heappush(heap, (w, nxt))
return total if count == n else -1
💡 Complexity Summary

Kruskal's + Union-Find: Time O(E log E) for sort, O(E · α(n)) for union-find ops ≈ O(E log E) total. Space O(n) for parent/rank arrays.
Prim's + Min-Heap: Time O(E log V). Space O(V + E) for adjacency list + heap. Better for dense graphs where E ≫ V.
Both are optimal for MST. Kruskal's is simpler to code in interviews (sort edges, simple union-find template).

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the MST Pattern

Key signals: "connect all nodes/cities," "minimum cost," weighted undirected edges, "return -1 if impossible." These together scream Minimum Spanning Tree. The "all connected, minimum total weight" constraint is the MST definition.

Step 2: State your algorithm choices and pick one

"Two standard MST algorithms: Kruskal's (sort edges + Union-Find) and Prim's (BFS with min-heap). Since we're given an edge list, I'll use Kruskal's — sort by cost, greedily accept edges that don't create cycles." This shows you know the trade-offs.

Step 3: Code Union-Find cleanly from scratch

Interviewers love this template: parent array (1-indexed), find with path compression, union with rank. That's ~12 lines to memorize. Always mention path compression and union by rank — these are the two optimizations that make it near-O(1) amortized.

Step 4: Handle impossibility explicitly

Track component count (start at n, decrement each accepted edge). If we exhaust all edges with components > 1, return -1. Mentioning this shows you think about edge cases before coding.

🗣️ What to Say to the Interviewer

"This is a Minimum Spanning Tree problem: connect all n cities with exactly n−1 edges at minimum total cost.
I'll use Kruskal's algorithm. First sort edges by cost — O(E log E). Then initialize Union-Find with each city as its own component.
Greedily process edges: if union(u,v) succeeds (different components), add the cost and decrement components.
Stop as soon as components reaches 1. If we exhaust all edges first, the graph is disconnected — return -1.
Total: O(E log E) time, O(n) space. Union-Find uses path compression + rank for near-O(1) amortized operations."
🔮 Likely Follow-up Questions
  • "What if you can only use at most k roads?" → Accept first k valid edges; check if all connected.
  • "Can you solve this with Prim's instead?" → Yes; discuss dense vs sparse graph trade-offs.
  • "What if roads are directed (one-way)?" → Becomes Minimum Arborescence — much harder, Edmonds' algorithm.
  • "How do you find ALL possible MSTs?" → Matroid intersection — mention as advanced topic.
  • "What if some edges are already selected (fixed)?" → Modified Kruskal's: accept fixed edges first, then run standard algorithm.
