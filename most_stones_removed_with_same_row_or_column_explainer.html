LeetCode #947

Most Stones Removed with Same Row or Column

On a 2D plane, n stones are placed at integer coordinates. A stone can be removed if it shares a row or column with another stone. Return the maximum number of stones that can be removed.

Medium Graph Union-Find Time: O(n · α(n)) Space: O(n)
1 Understanding the Problem ▾

You have stones placed on a 2D grid. A stone is removable if at least one other stone (that hasn't been removed yet) shares its row or column. You want to remove as many stones as possible. What's the maximum count?

💡 Key Insight

Stones that share a row or column are connected. Connectivity is transitive: if stone A shares a row with B, and B shares a column with C, then all three are in the same group. Within any connected group of size k, you can always remove exactly k − 1 stones, leaving one survivor. Therefore: answer = total stones − number of connected components.

B at (0,2) ← same row as A
C at (1,2) ← same col as B
All connected → remove B then A

✗ Isolated stone → cannot remove

remove it. It's a survivor.

Why can we always remove k−1 from a connected group? Model the component as a tree (any spanning tree of the connectivity graph). Remove leaves first — each leaf shares a row or col with its parent, satisfying the rule. After removing the leaf, the parent becomes the new leaf. Repeat until 1 stone remains. This always gives exactly k−1 removals — order of removal doesn't matter as long as we pick non-isolated stones.

Edge cases: All n stones share one row → 1 component, answer = n−1. Every stone has a unique row AND unique column → n components, answer = 0. n = 1 → answer = 0.

2 The Algorithm: Union-Find by Row and Column ▾

We need to count connected components where stones are connected if they share a row or column. Union-Find (Disjoint Set Union) is the ideal tool. The clever twist: instead of comparing every pair of stones (O(n²)), we treat rows and columns themselves as nodes, then union a stone's row-node with its column-node.

1 Initialize Union-Find: Use a dict parent. Each node is its own root lazily (first time find(x) is called, set parent[x] = x). Encode column c as c + 10001 to avoid integer collision with row r (coordinates are 0–10000).
2 Union row and column for each stone: For every stone at (r, c), call union(r, c + 10001). This links row-node r to column-node c+10001. Two stones sharing row r both union with the same row-r node → automatically in the same component.
3 Count components: Second pass over stones. For each stone at (r, c), call find(r) and collect into a set. Unique values in the set = number of connected components.
4 Return answer: len(stones) − len(seen). Each component of size k contributes k−1 removals, and summing k−1 over all components gives total − num_components.
🔑 Why Row+Column Unioning Works

Two stones at (r, c₁) and (r, c₂) both call union(r, c₁+offset) and union(r, c₂+offset) respectively. Both operations link to the same row-node r. Therefore find(r) for both stones returns the same root → they are in the same component. No explicit pairwise comparison needed. O(n) passes replaces O(n²) work.

🔑 DFS Alternative (mention first in interviews)

Build adjacency list: add edge (i, j) if stones i and j share a row or column. Run DFS/BFS to count components. Same formula applies. Time O(n²) to build graph, Space O(n²). Mention this as your "intuitive" approach, then optimize to Union-Find.

3 Why It Works: The Invariant ▾
🔐 The Core Invariant

Within a connected component of k stones, at any point during removal, as long as at least 2 stones remain in the component, there exists at least one removable stone (one that shares a row/col with another in the component). We can always make progress until exactly 1 stone remains. Therefore, max removals = k − 1 per component.

Formal argument: A connected graph on k vertices has a spanning tree with k−1 edges. In the spanning tree, every non-root node has a parent. Remove leaves (stones whose only connection in the current tree is upward to their parent). Since the leaf shares a row or column with its parent, the removal is valid. After removal, the next leaf becomes removable. By induction, we make k−1 removals.

Why the row+column encoding is lossless: When we call union(r, c+10001), we link row-node r and col-node c+10001. Two stones with the same row r both merge into the same component as r. Two stones with the same column c both merge into the same component as c+10001. Union-Find's path compression then connects all transitively reachable nodes. The encoding never creates false positives because rows ∈ [0,10000] and cols+offset ∈ [10001,20001] are disjoint.

⚠️ Counting Roots: Row-nodes, Not All Nodes

After building the Union-Find, the parent dict contains both row-nodes and col-nodes. When counting components, call find(r) for each stone's row only — and collect into a set. Don't also add find(c+10001), as that double-counts. One root per stone's row-node is sufficient because the row-node and col-node for a stone share the same root after unioning.

4 The Code (Annotated) ▾

Solution 1 — Union-Find by Row/Column (Optimal · O(n·α(n)) · Interview Pick)

def removeStones(stones): ← n = len(stones)
parent = {} ← lazy UF: node → parent; starts empty
def find(x): ← returns root of x's component
if x not in parent: parent[x] = x ← lazy init: x is its own root
if parent[x] != x: ← not at root yet
parent[x] = find(parent[x]) ← path compression: flatten tree
return parent[x]
def union(a, b): ← merge two components
ra, rb = find(a), find(b) ← find roots of both
if ra != rb: parent[ra] = rb ← link only if different roots
for r, c in stones: ← O(n) union phase
union(r, c + 10001) ← link row r to col c (offset by 10001)
seen = set() ← will hold distinct component roots
for r, c in stones: ← O(n) count phase
seen.add(find(r)) ← root of this stone's row-node
return len(stones) - len(seen) ← n − components = max removable
⚠️ Why offset 10001?

Coordinates range 0–10000. Without offset, row index 5 and column index 5 are both the integer 5 — they'd incorrectly be treated as the same node, merging unrelated components. Adding 10001 to column indices gives columns the range [10001, 20001], fully disjoint from rows [0, 10000]. Any constant larger than 10000 works; 10001 is the minimal safe choice.

Solution 2 — DFS on Adjacency List (O(n²) — intuitive but slow)

def removeStones(stones):
n = len(stones)
adj = [[] for _ in range(n)] ← adjacency list for the stone graph
for i in range(n): ← O(n²) to build all edges
for j in range(i+1, n):
if stones[i][0]==stones[j][0] \
or stones[i][1]==stones[j][1]: ← same row or same col
adj[i].append(j); adj[j].append(i)
visited=[False]*n; comps=0
def dfs(v):
visited[v]=True
for nb in adj[v]:
if not visited[nb]: dfs(nb)
for i in range(n):
if not visited[i]: dfs(i); comps+=1 ← count connected components
return n - comps ← same formula

Union-Find (optimal): Time O(n · α(n)) ≈ O(n), Space O(n)
DFS (intuitive): Time O(n²), Space O(n²) adjacency list
Interview recommendation: Present DFS first as the natural approach, then optimize to Union-Find to demonstrate both clarity and efficiency.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Reframe — What Can't Be Removed?

Instead of asking "what can be removed?", ask "what must stay?" A stone that is the ONLY stone in its row AND the only stone in its column can never be removed. Those are the survivors — one per connected component. This reframe leads directly to: answer = n − survivors = n − components.

Step 2: Identify Connected Components

Two stones sharing a row or column are connected. This relationship is transitive: if A↔B (same row) and B↔C (same col), then A, B, C are in one group. This is exactly connected components on a graph where edges represent shared row/col.

Step 3: Choose Union-Find Over DFS

DFS needs O(n²) to build the adjacency list. Union-Find avoids pairwise comparison: union each stone's row with its column (as separate abstract nodes). Two stones sharing row r both union with node r → automatically in the same component. O(n) work total.

Step 4: Mention the Offset Trick Proactively

"I add 10001 to all column keys so row 0 and column 0 don't collide as the same integer node. This is a common gotcha in 2D Union-Find problems."

🗣️ What to Say to the Interviewer

"This is a connected-components problem. Two stones are in the same group if they
share a row or column — transitively. Within each group of k stones, we can remove k-1.
So: answer = n − number_of_components.

For the simple approach: build an adjacency list and run DFS — O(n²).

🔮 Likely Follow-up Questions
  • "What if coordinates are negative?" → Normalize to 0-based or use string keys "r_5"/"c_5" instead of integers with offsets
  • "Can you solve it with BFS?" → Yes, build adjacency list, BFS each unvisited node — same O(n²) as DFS
  • "What's the minimum stones left?" → Exactly the number of connected components — one survivor per group
  • "How does α(n) compare to log n?" → α(n) ≤ 4 for all practical n; it's the inverse Ackermann function — extremely slow-growing

7 Summary & Key Takeaways ▾
Stones sharing a row or column are "connected" — connectivity is transitive, forming components
Within a connected component of k stones, exactly k−1 can be removed; 1 survivor remains
Answer = total stones − number of connected components
Union-Find key trick: union each stone's row-node with its column-node (not stone-to-stone pairs)
Offset columns by 10001 to avoid integer collision between row 5 and column 5
