LeetCode #109

Convert Sorted List to Binary Search Tree

Given the head of a singly linked list where elements are sorted in ascending order, convert it to a height-balanced binary search tree.

Medium Divide and Conquer Linked List BST Time: O(n log n) Space: O(log n)
1 Understanding the Problem ▾

We receive a sorted linked list and must return the root of a height-balanced BST built from the same values. "Height-balanced" means no node's left and right subtree depths differ by more than one — the tree should be as shallow as possible.

💡 Key Insight

Because the list is already sorted, picking the middle element as the root automatically satisfies the BST property — every value to its left is smaller, every value to its right is larger. And picking the middle specifically ensures the left and right subtrees are as equally sized as possible, giving us the height-balanced guarantee.

Input (sorted linked list)

-10 → -3 → 0 → 5 → 9
(5 nodes, ascending order)

Output (height-balanced BST)

    0
  /   \
-3   9
/   /
-10  5

Why height-balanced matters: An unbalanced BST can degrade to O(n) for search (essentially a linked list again). A height-balanced BST guarantees O(log n) search. LeetCode's test cases accept any valid height-balanced BST — you don't need to produce the exact one shown.

Edge cases: Empty list → return None. Single node → return that node as a leaf. Two nodes → first becomes left child of second (median of two is the second when using lower-middle convention). All handled correctly by the recursive algorithm.

2 The Algorithm: Find Mid → Sever → Recurse ▾

This is a classic Divide and Conquer problem on a linked list. We cannot use array-style midpoint indexing, so we find the middle using the fast/slow pointer technique, then split the list and recurse on each half.

1 Base cases: If head is None, return None. If there is only one node (head.next is None), wrap it in a TreeNode and return. No pointer work needed.
2 Find the middle: Use slow (advances 1 step) and fast (advances 2 steps), plus a prev pointer trailing behind slow. When fast or fast.next becomes None, slow is at the middle.
3 Sever the list: Set prev.next = None. This cuts the linked list just before slow. The left half is head → ... → prev; the right half starts at slow.next.
4 Create root: Make TreeNode(slow.val). This is the root of the current subtree.
5 Recurse: root.left = sortedListToBST(head) (left half), root.right = sortedListToBST(slow.next) (right half). Return root.
🔑 Recursion Shape

sortedListToBST([-10,-3, 0, 5, 9]) mid=0 → root=TreeNode(0) left: sortedListToBST([-10,-3]) mid=-3 → root=TreeNode(-3) left: sortedListToBST([-10]) → leaf(-10) right: sortedListToBST([]) → None right: sortedListToBST([5, 9]) mid=9 → root=TreeNode(9) left: sortedListToBST([5]) → leaf(5) right: sortedListToBST([]) → None

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every recursive call, the sublist passed in is sorted and represents exactly the set of values that belong in this subtree. Choosing the median as the root ensures: (1) BST property holds by construction, and (2) left and right subtrees are as equal in size as possible, so the height is O(log n).

BST property: Since the list is sorted, all values before slow are less than slow.val, and all values after are greater. Placing slow.val as the root satisfies the BST invariant without any comparisons.

Balance guarantee: At each call of size n, we pick the median — giving left subtree size ⌊(n-1)/2⌋ and right subtree size ⌈(n-1)/2⌉. The difference is at most 1. By induction, the entire tree is height-balanced. The recursion tree has O(log n) depth.

⚠️ The Most Common Bug: Forgetting to Sever

If you forget prev.next = None, the left recursive call receives the full list from head all the way through slow and beyond — it has no way to know where to stop. You'll get infinite recursion or a malformed tree. The sever step is not optional; it is the boundary that tells each recursive call what its subproblem is.

Why slow.next gives the right half: After severing, slow is the root, and slow.next points to the first element of the right sublist. The right recursive call correctly gets elements larger than the root.

4 The Code (Annotated) ▾

Solution 1 — Find Mid + Divide & Conquer (Interview Pick, O(n log n) time, O(log n) space)

def sortedListToBST(head): ← head is the first node of the current sublist
if not head: return None ← empty sublist → no tree node
if not head.next: return TreeNode(head.val) ← single node → leaf; no pointers needed
slow, fast, prev = head, head, None ← initialize fast/slow at head, prev trails
while fast and fast.next: ← run until fast reaches the end
prev = slow ← save position one step before slow
slow = slow.next ← slow advances 1
fast = fast.next.next ← fast advances 2 (safe: checked fast.next above)
prev.next = None ← SEVER: left half ends at prev; right starts at slow.next
root = TreeNode(slow.val) ← middle element becomes the BST root
root.left = sortedListToBST(head) ← recurse on left half (head to prev)
root.right = sortedListToBST(slow.next) ← recurse on right half (slow.next onward)
return root ← return assembled subtree
⚠️ Why "fast and fast.next" — not "fast.next and fast.next.next"?

Checking fast and fast.next places slow at the lower-middle for even-length lists. If you used fast.next and fast.next.next, you'd advance one extra step, placing slow at the upper-middle. Both produce valid balanced BSTs, but getting the convention wrong leads to off-by-one errors that cause crashes on single-element sublists. The standard form is while fast and fast.next.

Solution 2 — Convert to Array then Recurse (Simpler, O(n) time, O(n) space)

def sortedListToBST_v2(head): ← trades O(n) space for simpler code + faster mid
nums = []
while head: ← collect all values from the linked list
nums.append(head.val); head = head.next
def build(lo, hi): ← recursive builder over index range
if lo > hi: return None
mid = (lo + hi) // 2 ← O(1) midpoint — array allows this!
node = TreeNode(nums[mid])
node.left = build(lo, mid - 1)
node.right = build(mid + 1, hi)
return node
return build(0, len(nums) - 1) ← kick off over full index range
💡 Complexity Summary

Find Mid (interview pick): Time O(n log n) — O(n) to find mid at each of O(log n) levels. Space O(log n) — recursion stack only, no array.
Array + Recurse: Time O(n) — O(1) midpoint access. Space O(n) for the array plus O(log n) stack.
If the interviewer asks "can you avoid the extra O(n) space?", that's your cue to pivot to Solution 1.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Read the Two Key Constraints

Sorted list → BST property is free. Height-balanced → must split as evenly as possible at every level. The combination immediately points to "pick the median, recurse left and right."

Step 2: Propose the Simpler Approach First

"The easy version: convert the list to an array in one pass, then recursively take the middle element as root. O(n) time, O(n) space. Let me code this up and then see if we can improve space."

Step 3: Optimize with Fast/Slow Pointer

"To avoid the O(n) array, I can find the middle in-place using fast/slow pointers. I also keep a prev pointer so I can sever the list before the middle node. Space becomes O(log n) — just the call stack."

Step 4: Call Out the Sever Step Explicitly

"The critical step is prev.next = None. Without it, the left recursive call doesn't know where to stop — it would eat the entire list." Interviewers love when you proactively flag the tricky part.

🗣️ What to Say to the Interviewer

"The list is already sorted, so the BST property is free — I just need to pick the right root at each level.
For height balance I must split evenly, so I'll always pick the median. Finding the median of a linked list requires the fast/slow pointer trick — O(n) per level, O(log n) levels, total O(n log n).
After finding the mid, I sever the list with prev.next = None, make slow.val the root, then recurse left on head and right on slow.next.
Space is O(log n) — just the recursion depth, no extra array needed."
🔮 Likely Follow-up Questions
  • "What if the list is unsorted?" → Sort first (O(n log n)), then apply the same algorithm. Net O(n log n).
  • "Can you do it in O(n) total time?" → Yes — flatten to array (O(n) extra space), then use O(1) midpoint index access.
  • "Verify your BST is correct?" → Inorder traversal of the output must reproduce the original sorted sequence exactly.
  • "What is the height of the resulting BST?" → O(log n) since we halve at each level and both halves differ by at most 1.

7 Summary & Key Takeaways ▾
Sorted input + BST = always use the median as root — no comparison needed
Fast/slow pointer finds the middle of a linked list in O(n) with O(1) extra space
Sever (prev.next = None) before recursing left — it's the boundary of the subproblem
Recursion depth is O(log n); each level takes O(n) total work → O(n log n) overall
Array approach is O(n) time but costs O(n) space — worth mentioning as the simpler first step

Array + Recurse O(n) O(n) Simpler; use O(1) midpoint indexing

🎯 When to Recognize This Pattern
  • Sorted sequence → balanced BST → always pick median at each level
  • Linked list → find middle → fast/slow pointer
  • "Height-balanced" in the problem → equal halving at every recursion
  • Need to find mid of linked list in any problem → this same fast/slow trick
