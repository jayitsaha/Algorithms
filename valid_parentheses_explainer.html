LeetCode #20

Valid Parentheses

Given a string s containing just the characters '(', ')', '{', '}', '[' and ']', determine if the input string is valid.

Easy Stacks Parentheses Matching Time: O(n) Space: O(n)
1 Understanding the Problem ▾

A string of brackets is valid when three rules all hold simultaneously:

1 Every open bracket must be closed by the same type of bracket — '(' by ')', '[' by ']', '{' by '}'.
2 Open brackets must be closed in the correct order. The most recently opened bracket must be the first to close (Last-In First-Out nesting).
3 There are no unmatched close brackets that appear without a prior matching open.
💡 Key Insight

Brackets must nest properly. The string "([)]" looks balanced — equal counts of each type — but it's invalid because the ')' tries to close '(' while '[' is still open inside it. Order matters, not just count. This "most recent must close first" rule is exactly what a LIFO stack enforces.

✓ Valid examples

"()" → one simple pair
"()[]{}" → three separate pairs
"{[]}" → nested, correct order
"([{}])" → triple nested correctly

✗ Invalid examples

"(]" → wrong close type
"([)]" → wrong close order
"{" → unclosed open bracket
"]" → lone close with no prior open

Edge Cases to consider: empty string (valid — vacuously all rules hold), odd-length string (always invalid — can't form pairs), single character (always invalid), all open brackets "(((" with no closes, and a close bracket before any open like ")(".

2 The Algorithm: Push Open, Match Close ▾

Think of the stack as a "waiting room for unmatched open brackets." Every open bracket is pushed in. Every close bracket asks: "Am I the right partner for whoever is waiting at the front of the line?" That front-of-the-line position is always the stack's top — and it holds the most recently seen open bracket.

1 Build the matching map: {')': '(', ']': '[', '}': '{'}. Mapping close brackets to their expected open partners allows O(1) lookups. The three keys are the close brackets; checking if c in matching tells us c is a close bracket.
2 Scan left to right: For each character, decide — is it an open bracket (push it) or a close bracket (verify the top)?
3 Open bracket → Push: stack.append(c). It will be matched later by whatever close bracket corresponds to it.
4 Close bracket → Pop and verify: Pop the top. If the stack was empty, use a sentinel '#' that can never match. Check if the popped value equals matching[c]. Mismatch → return False immediately.
5 After full scan → Check stack: Return not stack. Any remaining items in the stack are open brackets that were never closed → invalid. Empty stack → every open was correctly closed.

Matching pairs:

(↔)
[↔]
{↔}
🔑 Why a Stack Fits Perfectly

Brackets nest. The innermost open bracket must close before any outer bracket can. That's Last-In First-Out (LIFO) — the exact behavior of a stack. Whenever a problem has the rule "the most recently started thing must finish first," a stack is the right tool. This same pattern appears in call stacks, HTML tag matching, and expression parsing.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every point during the scan, the stack contains exactly all open brackets seen so far that have not yet been matched, in the order they were encountered. The top of the stack is always the most recently seen unmatched open — which is precisely what the next valid close bracket must match.

Why checking only the top is sufficient: For a string to be valid, inner brackets must close before outer ones. If the current close bracket doesn't match the top of the stack, there is no way to rearrange processing to make it valid — the mismatch is fatal. We never need to search deeper.

Why empty stack at end means valid: Every push must be balanced by a pop. If any open brackets remain, they were never closed. Since we also fail immediately on a bad close bracket (popped wrong type or found empty stack), an empty stack at the end guarantees every bracket was matched in the correct nesting order.

⚠️ Common Trap: Counting Instead of Stacking

Naive approach: count opens and closes of each type, verify equal. This fails for "([)]" — counts balance (1 each), but the ORDER is wrong. You need the stack to track order. Counting loses all positional information.

🚨 Empty Stack on Close Bracket

If you encounter a close bracket but the stack is empty, there's no open bracket waiting — fail immediately. The sentinel '#' trick handles this elegantly: top = stack.pop() if stack else '#'. Since '#' never equals any valid open bracket, the mismatch check catches it in the same branch.

4 The Code (Annotated) ▾

Solution 1 — Stack + Hash Map with Sentinel (Optimal, Interview Pick)

def isValid(s: str) -> bool:
matching = {')': '(', ']': '[', '}': '{'} ← Map each close bracket to its expected open partner
stack = [] ← Waiting room for unmatched open brackets
for c in s: ← Scan left to right, one character at a time
if c in matching: ← True iff c is a close bracket (key of dict)
top = stack.pop() if stack else '#' ← Pop top open; '#' sentinel if stack is empty
if top != matching[c]: ← Top must equal the correct matching open
return False ← Wrong type or empty stack → invalid
else: ← c is an open bracket
stack.append(c) ← Push it; will be matched by a future close
return not stack ← Valid iff no unmatched open brackets remain
⚠️ The '#' Sentinel Pattern

stack.pop() if stack else '#' collapses the empty-stack failure into the same branch as the wrong-type failure. Instead of two separate if statements, one check handles both. '#' can never equal '(', '[', or '{', so it always triggers top != matching[c]. Clean and idiomatic.

Solution 2 — Explicit Guard (clearer for beginners)

def isValid(s: str) -> bool:
stack = []
pairs = {')': '(', ']': '[', '}': '{'}
for c in s:
if c in '([{':
stack.append(c) ← Open → push
else:
if not stack or stack[-1] != pairs[c]: ← Empty or wrong top → fail
return False
stack.pop() ← Matched → pop the open
return len(stack) == 0 ← True iff all opens were matched
💡 Complexity

Both solutions: Time O(n) — one pass; each character is pushed at most once and popped at most once.
Space O(n) — stack holds at most n/2 items (worst case: all opens before any closes).
Can we do O(1) space? Not in general with three bracket types and arbitrary nesting. For a single bracket type, counting would suffice. But with three types and nesting, the stack is necessary to record which types were opened and in what order.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Pattern Signal

"Correctly nested" and "properly matched" are the key phrases. Anytime you need to remember something and verify it against the most recently seen item first, think stack. This shows up as bracket matching, HTML tag validation, function call stacks, and expression parsing.

Step 2: Propose and Refute the Naive Approach

"A first instinct: count each type of open and close bracket, verify equal counts. But that fails for '([)]' — counts balance, order doesn't. We need to track order, not just quantity." This shows critical thinking.

Step 3: Arrive at the Stack Naturally

"To enforce order: each close must match the most recently seen unmatched open. That's LIFO — a stack. Push opens onto the stack, pop and verify on closes. If the top doesn't match, or the stack was empty, it's invalid."

Step 4: Walk Through Edge Cases Out Loud

Cover: empty string (True — nothing to fail), odd length (False — can't pair), close bracket with empty stack (False), and unmatched opens at end (False). Mention the sentinel trick for the empty-stack case.

🗣️ What to Say to the Interviewer

"I'll use a stack — the classic tool for bracket matching.
Build a dict mapping each close bracket to its expected open partner.
Scan left to right: open brackets → push onto stack.
Close brackets → pop the top (or use '#' sentinel if empty).
If the popped value doesn't equal the expected open → return False.
After scanning: return True only if the stack is empty.
Time O(n), Space O(n) — each character pushed and popped at most once."
