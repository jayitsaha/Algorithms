LeetCode #716

Max Stack

Design a max stack that supports push, pop, top, peekMax, and popMax — all efficiently.

Hard Design Two Stacks DLL + TreeMap Time: O(log n) Space: O(n)
1 Understanding the Problem ▾

A Max Stack is a regular stack — last-in-first-out — with one extra superpower: you can query or remove the maximum element at any time, not just the top. Five operations must be supported:

Classic Stack Ops

push(x) — add x to top
pop() — remove & return top
top() — peek top (no remove)

Max-Specific Ops

peekMax() — view max value
popMax() — remove & return max
popMax is where it gets hard!

💡 Key Insight

You need two simultaneous orderings of the same data: insertion order (for normal stack ops) and value order (for max ops). These are usually different orderings. The core design challenge is maintaining both in sync as elements come and go.

Why is popMax() the hard operation? Imagine the stack is [2, 8, 3, 1] (top = 1). The max is 8, but it's buried under 3 and 1. To remove it from a regular stack, you'd have to pop 1, pop 3, pop 8, then push 3 and 1 back. That's O(n). Can we do better?

⚠️ The Core Tension

A stack gives O(1) access to the most recently inserted element. A max-heap gives O(1) access to the largest element. These are different elements. Marrying both access patterns without each slowing down the other is the entire design problem.

Edge Cases: Multiple elements with the same max value (popMax removes only the topmost one), single element, pushing negatives, alternating push/popMax, empty after popMax.

2 The Algorithm: Two Stacks (Interview) + DLL/SortedList (Optimal) ▾

Approach A — Two Stacks (O(1) amortized, simpler)

Use a main stack and a parallel max-tracking stack. The max-stack stores the running maximum at each level: max_stk[i] = max of stack[0..i]. This makes peekMax O(1). The cost is popMax: O(n) to find and remove a buried max, then push elements back.

1 push(x): Append x to stack. Compute running max: max(x, max_stk[-1] if max_stk else x). Append to max_stk. O(1).
2 pop() / top(): Pop (or peek) both stacks simultaneously. max_stk stays in sync — each index in max_stk corresponds to the same index in stack. O(1).
3 peekMax(): Return max_stk[-1]. The running-max invariant guarantees this is always the global maximum. O(1).
4 popMax(): buffer = []. Pop into buffer until stack top == max_val. Remove that element. Push buffer back (using self.push() to rebuild max_stk). O(n) worst, O(1) amortized.
🔑 Approach B — DLL + SortedList (O(log n) all ops)

Each push creates a node with a unique ID, added to: (1) a list simulating a doubly-linked list (tail = top of stack) and (2) a SortedList keyed by (value, uid). Lazy deletion: instead of immediately removing a node from the list, mark its uid as deleted. Both structures skip marked IDs on access. popMax finds max in O(log n) via SortedList, marks it deleted. pop() cleans up from the top, skipping any deleted IDs. All ops become O(log n) amortized.

3 Why It Works: The Invariant ▾
🔐 Two-Stacks Invariant

At every moment, max_stk[i] = maximum value among stack[0], stack[1], …, stack[i]. This means max_stk[-1] always equals the global maximum in O(1), no scanning needed.

Push maintains invariant: New max at level i+1 is max(new_val, max_stk[i]). We compute this and store it, so the invariant holds at the new level.

Pop maintains invariant: We always pop the same level from both stacks. The invariant at level i-1 was already established when that element was pushed — it's still correct.

popMax maintains invariant: We use self.pop() and self.push() internally, which always maintain the invariant. After popping the max and pushing the buffer back, each push re-establishes the running max at each new level.

⚠️ Why popMax is O(n) worst-case but O(1) amortized

Amortized analysis: each element can be moved to the buffer at most once per popMax call that targets a max below it. Across all operations, every element is pushed and popped a constant number of extra times. The total extra work across all operations is O(n) for n operations — O(1) amortized per operation.

DLL + SortedList invariant: Every live element has exactly one entry in the list AND one entry in the SortedList. Deleted uids form a "ghost" set. Both structures skip ghosts on access. This ensures neither structure grows stale — they're always consistent for the live elements.

4 The Code (Annotated) ▾

Solution 1 — Two Stacks (Interview Pick)

class MaxStack:
def __init__(self):
self.stack = [] ← main stack, stores the actual values
self.max_stk = [] ← max_stk[i] = max of stack[0..i] at all times
def push(self, x: int) -> None: ← O(1)
self.stack.append(x) ← add to main stack
cur = max(x, self.max_stk[-1] if self.max_stk else x) ← running max at this level
self.max_stk.append(cur) ← record: invariant maintained
def pop(self) -> int: ← O(1), pop both stacks together to stay in sync
self.max_stk.pop() ← remove this level's max entry
return self.stack.pop() ← remove and return the actual value
def top(self) -> int: ← O(1), peek without removing
return self.stack[-1]
def peekMax(self) -> int: ← O(1) — invariant guarantees global max is here
return self.max_stk[-1]
def popMax(self) -> int: ← O(n) worst, O(1) amortized
max_val = self.peekMax() ← the value to find and remove
buffer = [] ← temp storage for elements above the max
while self.stack[-1] != max_val: ← pop until we reach the topmost max
buffer.append(self.pop()) ← use self.pop() — keeps both stacks in sync!
self.pop() ← remove the max itself (from both stacks)
for val in reversed(buffer): ← push back in original order (buffer is reversed!)
self.push(val) ← self.push() rebuilds max_stk at each level
return max_val
⚠️ Why reversed(buffer)?

We pop top-to-bottom: first element into buffer was closest to max, last element was the stack's original top. To restore stack order, we must push from the bottom of the buffer first — that's reversed(buffer). Forgetting this reverses the stack order!

⚠️ Never raw-append inside popMax — always use self.pop() / self.push()

These methods keep stack and max_stk perfectly synchronized. If you directly manipulate the raw lists (e.g. self.stack.pop() without self.max_stk.pop()), you corrupt the running-max invariant and peekMax gives wrong answers.

Solution 2 — Lazy Deletion with SortedList (O(log n) all ops)

from sortedcontainers import SortedList
class MaxStack:
def __init__(self):
self.stk = [] ← list of (val, uid), acts as DLL (top = tail)
self.sl = SortedList() ← sorted by (val, uid); max is at sl[-1]
self.dead = set() ← uids of lazily-deleted elements
self.uid = 0 ← monotonically increasing; higher uid = more recent
def push(self, x): ← O(log n)
self.stk.append((x, self.uid)) ← tag with uid for ordering ties
self.sl.add((x, self.uid)) ← O(log n) insert into sorted structure
self.uid += 1
def _clean(self): ← skip dead entries at top of stack
while self.stk and self.stk[-1][1] in self.dead:
self.dead.discard(self.stk.pop()[1])
def pop(self): ← O(log n) amortized
self._clean() ← skip any lazily-deleted tops
val, uid = self.stk.pop()
self.sl.discard((val, uid)) ← O(log n) remove from sorted view
return val
def popMax(self): ← O(log n) — no scanning needed!
val, uid = self.sl.pop() ← O(log n): max is sl[-1]
self.dead.add(uid) ← lazy-delete from stack view
return val
💡 Why Lazy Deletion Works

popMax removes a node from the SortedList immediately (O(log n)). But finding that node in the list array would take O(n) scanning. Instead, we just record the uid as "dead" — O(1). Whenever pop() or top() accesses the list tail, it skips any dead uids. The total cleanup work is proportional to the number of operations, giving O(1) amortized cleanup per operation.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Spot the Dual-Ordering Pattern

Say out loud: "This problem needs two different orderings of the same elements — insertion order for stack ops, and value order for max ops. Whenever a design problem needs two simultaneous orderings, I combine two data structures."

Step 2: Code the Two-Stacks Solution First

"My first solution: a parallel max-tracking stack. At every level, I record the running max. push is O(1), pop is O(1), peekMax is O(1). The only trade-off is popMax — O(n) because I have to dig out a buried element. For many workloads, this is fine."

Step 3: Explain Why O(n) popMax is Hard to Avoid with Pure Stacks

"A stack only allows O(1) access at the top. If the max is buried, we must move elements to reach it — that's inherently O(n). To get O(log n) for popMax without changing the stack's access pattern, we need a second structure that tracks value-order independently."

Step 4: Upgrade to Lazy Deletion

"Optimal approach: SortedList gives O(log n) max lookup. But we can't efficiently remove a node from the middle of a list array. Solution: lazy deletion — mark the uid as dead, skip it when we encounter it from either end. This decouples the two structures while keeping them consistent."

🗣️ What to Say to the Interviewer

"I see two simultaneous access patterns: stack order and max order. I'll use two data structures.
First: a main stack + a parallel max-tracking stack. max_stk[i] = max of stack[0..i].
push: O(1), pop: O(1), top: O(1), peekMax: O(1). popMax: O(n) worst, O(1) amortized.

To get O(log n) for all ops: pair a list with a SortedList, keyed by (val, uid).
Lazy deletion: popMax removes from SortedList in O(log n), marks uid dead.
pop() skips dead uids at the tail, cleaning up lazily. All ops → O(log n) amortized."
🔮 Likely Follow-up Questions
  • "Min Stack instead?" → Same two-stacks trick, track running minimum; sorted list uses index [0]
  • "What if elements can repeat?" → Already handled — unique uid ensures the most recently pushed duplicate is the max candidate
  • "Thread safety?" → Discuss locking push/pop/popMax as atomic transactions to avoid race conditions
  • "Can you use just a heap?" → A heap can't efficiently support stack-order pop() in O(log n) — it's the wrong data structure for insertion-order access

7 Summary & Key Takeaways ▾
Max Stack needs two simultaneous orderings — combine two data structures
Two-stacks: parallel max-tracking stack stores running max per level → O(1) peekMax
popMax pops to buffer, removes max, pushes back (reversed!) — O(n) worst, O(1) amortized
Optimal: SortedList for O(log n) max lookup + lazy deletion for O(log n) stack access
Always use self.pop()/self.push() inside popMax — never raw list operations
Unique UIDs handle duplicate values: most-recently-pushed duplicate is chosen as max
