LeetCode #1552

Magnetic Force Between Two Balls

Place m balls into n baskets at given positions to maximise the minimum magnetic force (minimum pairwise distance) between any two balls.

Medium Binary Search BS on Distance Time: O(n log n + n log D) Space: O(1)
1 Understanding the Problem ▾

You have n baskets at fixed integer positions along a number line, and m balls to place — one ball per basket, but you choose which m of the n baskets to use. The "magnetic force" between two balls is just the distance between them. You want to choose positions so the weakest force (the smallest gap between any two balls) is as large as possible.

💡 Key Insight

This problem asks you to maximise a minimum value. The answer is a numeric distance, not an index. Whenever a problem asks "maximise the minimum" or "minimise the maximum" of some value you control, Binary Search on the Answer is the canonical technique: search over the possible answer values and ask "is this feasible?"

Edge cases to consider: m=1 (only one ball, no pair exists — trivially any placement works), m=2 (answer = position[last] − position[0] after sorting), m=n (forced placement, answer = min adjacent gap in sorted order), positions not sorted (must sort first), duplicate positions in input.

2 The Algorithm: Binary Search on Distance ▾

Instead of searching for where to place each ball, we search for the minimum gap value d — from 1 to (max_pos − min_pos). For each candidate d, a greedy O(n) checker tells us whether a valid placement exists. We binary search to find the largest feasible d.

1 Sort positions. This lets us always greedily scan left-to-right and place the next ball at the earliest valid basket.
2 Set search bounds. lo = 1 (minimum possible gap). hi = position[-1] − position[0] (maximum possible gap — place two balls at the extremes).
3 Binary search loop. Compute mid = (lo + hi) // 2. Run can_place(mid).
4 Greedy checker — can_place(d). Place first ball at position[0]. Scan left-to-right: place the next ball at the first basket that is ≥ last_placed + d. Count total placed. Return count ≥ m.
5 Update bounds. If can_place(mid) is True: record ans = mid, set lo = mid + 1 (try larger). If False: set hi = mid − 1 (try smaller).
6 Return ans — the largest d for which can_place returned True.

Why feasibility is monotone (the key prerequisite for BS on Answer):

If we can place m balls with minimum gap ≥ d, we can always place them with any smaller gap d' < d (just relax the constraint — the same placement still works). So feasibility(d) is a decreasing step function:

🔑 Binary Search Template: Finding the Last True

We want the rightmost True in [T,T,T,T,F,F,F]. Template: if feasible(mid): ans=mid; lo=mid+1 else hi=mid-1. At the end, ans holds the last True.

3 Why It Works: The Invariant ▾
🔐 The Greedy Invariant

In can_place(d), at every moment last holds the position of the most recently placed ball, and every ball placed so far is at least d apart from its predecessor. The greedy choice — place at the first valid basket ≥ last+d — never wastes capacity: taking an earlier valid position leaves more room for future balls.

Exchange argument for greedy correctness: Suppose an optimal placement uses position p but skips an earlier valid position q (where last+d ≤ q ≤ p). Swap p for q in the placement. The gap before q is q−last ≥ d (valid). The gap after q is p_next−q ≥ p_next−p ≥ d (since q ≤ p, p_next−q is at least as large as p_next−p). So the swap is still valid and we've shifted everything leftward — leaving even more room for subsequent balls. Therefore, greedy is always at least as good as optimal.

Why binary search finds the exact boundary: Because feasibility is monotone, the search space is a sorted sequence [T,T,…,T,F,F,…,F]. Standard binary search locates the T/F boundary in O(log(hi−lo)) iterations. We record ans each time we see T and push lo up, so ans ends at the rightmost T.

⚠️ Most Common Mistake: Forgetting to Sort

The greedy checker assumes positions are visited in increasing order. Without sorting, you might place a ball far right and then encounter a closer left position that violates the gap — the greedy won't catch this. Always position.sort() before running the algorithm.

Complexity breakdown: Sorting takes O(n log n). The binary search runs O(log D) iterations where D = max_pos − min_pos (at most 10^9, so ≈30 iterations). Each iteration runs the greedy checker in O(n). Total: O(n log n + n log D). Space is O(1) extra (the sort can be in-place).

4 The Code (Annotated) ▾

Solution 1 — Binary Search on Distance (Optimal · Interview Pick)

def maxDistance(position: list[int], m: int) -> int:
position.sort() ← Sort so greedy scan works left-to-right
def can_place(d: int) -> bool: ← Greedy feasibility: can we place m balls with gap ≥ d?
count = 1 ← Always place first ball at position[0]
last = position[0] ← Track the last placed position
for pos in position[1:]: ← Scan remaining baskets
if pos - last >= d: ← Far enough from the last placed ball
count += 1 ← Place a ball here
last = pos ← Update last placed position
if count == m: return True ← All balls placed — early exit
return count >= m ← Did we place all m balls?
lo, hi = 1, position[-1] - position[0] ← lo=1 (min gap), hi=max possible gap
ans = 0 ← Best feasible gap found so far
while lo <= hi: ← Standard binary search
mid = (lo + hi) // 2 ← Candidate minimum gap
if can_place(mid): ← Feasible?
ans = mid ← Record this as the best answer so far
lo = mid + 1 ← Try a LARGER gap (push lower bound up)
else:
hi = mid - 1 ← Too large; try a SMALLER gap
return ans ← Largest feasible minimum distance
⚠️ Why "lo = mid + 1" when feasible (not "hi = mid")?

We want the largest feasible gap — the rightmost True. When mid is feasible we record it and push lo upward to search for something better. If we wrote hi=mid instead, we'd converge on a smaller value (the lower boundary of the True region). Always use the "record and push lo up" pattern for "find last True".

💡 Complexity Summary

Binary Search (optimal): Time O(n log n + n log D) where D = max_pos − min_pos ≤ 109 → ≈30 BS iterations. Space O(1) extra.
Brute Force: Time O(C(n,m) × m) — exponential; only feasible for tiny n.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Spot the "Maximise Minimum" Signal

The phrase "maximum of the minimum force" is a canonical trigger for Binary Search on Answer. Any "max of min" or "min of max" problem where the answer is a number (not an index) and feasibility is monotone should make you think BS on Answer immediately.

Step 2: Define the Search Space

Ask: "What's the minimum possible answer?" → 1 (any adjacent gap of 1). "Maximum possible?" → max_pos − min_pos (place two balls at the extremes). This gives lo=1, hi=max−min. We will not search for positions — we search for a distance d.

Step 3: Design the Feasibility Checker

For a candidate gap d: sort positions (one-time), then greedily place balls — always pick the earliest basket that is ≥ last+d. This runs in O(n). If count ≥ m at the end, d is feasible.

Step 4: Apply the "Last True" BS Template

Feasibility(d) = True for d in [1..ans], False for d in [ans+1..hi]. Binary search for the last True: if can_place(mid) → ans=mid; lo=mid+1. Else → hi=mid-1. Return ans.

🗣️ What to Say to the Interviewer

"'Maximise the minimum' tells me to use Binary Search on the Answer.
I'll binary search over the minimum gap d, from 1 to max_pos − min_pos.
For each candidate d, I run a greedy O(n) check: sort positions once,
then place balls greedily — always grab the nearest basket ≥ last+d.
If I can fit all m balls, d is feasible — record and try larger.
Otherwise try smaller. Total: O(n log n + n log D). Space O(1)."
🔮 Likely Follow-up Questions
  • "What if positions have duplicates?" → Still works — sorting places them adjacent, greedy skips them (gap=0 < d).
  • "What if m = 2?" → Answer is always max_pos − min_pos (place at the two ends).
  • "Can you do it without sorting?" → The greedy checker needs left-to-right order. You could use a sorted set for dynamic inputs, but sorting once is optimal for this static problem.
  • "Related: Koko Eating Bananas (#875)?" → Same template: BS on speed, greedy hours checker — "minimise maximum" variant.

7 Summary & Key Takeaways ▾
"Maximise the minimum" always suggests Binary Search on Answer — search for d, not for positions
Sort positions first; the greedy checker requires left-to-right order to be correct
Greedy "take first valid basket" is provably optimal by an exchange argument

🎯 When to Recognise Binary Search on Answer
  • Problem says "maximise the minimum" or "minimise the maximum"
  • The answer is a numeric value lying in a bounded range
  • You can verify feasibility for a given answer in O(n) or O(n log n)
  • The feasibility function is monotone (if d works, all d' < d also work)
