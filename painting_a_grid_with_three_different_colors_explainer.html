LeetCode #1931

Painting a Grid With Three Different Colors

Count the number of ways to paint an m×n grid with 3 colors so that no two adjacent cells share the same color.

Hard Dynamic Programming State Compression Time: O(3^m · n) Space: O(3^m)
1 Understanding the Problem ▾

You have an m × n grid and three colors (0=Red, 1=Green, 2=Blue). Paint every cell so that no two adjacent cells (up, down, left, right — not diagonal) share the same color. Return the number of valid colorings modulo 109+7.

💡 Key Insight

The grid has only m rows (1 ≤ m ≤ 5). Each column is a vertical strip of m cells. There are only 3m ≤ 243 possible column colorings. We can encode each column as a base-3 integer (a "state"), precompute which states are internally valid and which pairs of adjacent columns are compatible, then run a column-by-column DP.

Why m ≤ 5 is the key constraint: With m = 5, there are only 35 = 243 possible column colorings. We can enumerate ALL of them. If m were large (like n), enumeration would be impossible. The small m makes state compression viable.

🔑 State Encoding

Represent each column's coloring as a base-3 integer. Row 0's color occupies the least significant trit, row 1 the next, etc.

Column [R, G, B] with m=3 → 0·3⁰ + 1·3¹ + 2·3² = 0 + 3 + 18 = 21
Column [R, R, B] (invalid) → 0·3⁰ + 0·3¹ + 2·3² = 0 + 0 + 18 = 18

Edge cases: m=1 (each column is a single cell, all 3 colorings valid); n=1 (single column, count valid states directly); m=5 n=1000 (maximum, 243 states, 1000 columns — still fast).

2 The Algorithm: Column-DP with State Compression ▾

Process the grid column by column. Each column's valid coloring is a "state" (base-3 integer). The DP accumulates the number of ways to reach each state at each column.

1 Enumerate valid column states: Iterate all 3m base-3 integers from 0 to 3m−1. Keep only those where no two adjacent rows have the same digit (trit). Call this set valid_states.
2 Precompute compatible pairs: For each pair of valid states (s, t), check that for every row r, digit(s, r) ≠ digit(t, r). Store these as compat[s] — a list of states that can follow s.
3 Initialize DP: dp[s] = 1 for all valid states s (column 0 can be any valid coloring in 1 way).
4 Iterate columns 1 to n−1: For each column, build new_dp[t] = sum of dp[s] for all s in valid_states where t is in compat[s]. Swap dp ← new_dp after each column.
5 Return answer: Sum all values in dp (total ways to complete the entire grid) modulo 109+7.

3 Why It Works: The DP Deep-Dive ▾

Why is this Dynamic Programming?

Optimal Substructure

The number of valid full-grid colorings through column j ending in state t depends only on how many valid partial colorings of columns 0..j−1 end in states compatible with t. We don't need to know which specific cells were colored — only the final column state matters.

Overlapping Subproblems

Many different left-column sequences can produce the same column state s. Without DP, we'd recount all of them every time we try to extend from s. DP memoizes: "how many ways produce state s at column j?" — computed once, used for all compatible next states.

📐 The Recurrence Relation
dp[j][t] = Σ dp[j-1][s]  for all s ∈ valid_states where compatible(s, t)

Base case: dp[0][s] = 1 for all s ∈ valid_states
Answer: Σ dp[n-1][s] for all s ∈ valid_states

What makes this "State Compression DP"?

The DP state is a compressed representation of the column coloring as a single integer. Instead of storing a list of m colors for a column, we encode it as a base-3 number. This compression lets us:

  • Store the entire DP table as a simple array/dict indexed by integers
  • Compute compatibility quickly using arithmetic (digit extraction)
  • Keep the state space tiny: 35 = 243 total states, even fewer valid ones
🔐 The Invariant

After processing column j, dp[s] holds the number of valid ways to color columns 0 through j such that column j has coloring s. The adjacency constraint is enforced both vertically (within column validation) and horizontally (compatibility check between adjacent columns). These two checks together guarantee global validity.

Why the column approach works better than row approach

We could transpose the problem and DP row-by-row. The constraint m ≤ 5 makes column compression feasible. If we instead processed row-by-row, each state would need to encode an entire row of n cells (n up to 1000), which is exponential in n. Processing by columns keeps states small because the column dimension (m ≤ 5) is fixed and small.

⚠️ Common Misconception

Students sometimes try row-by-row DP where the state encodes the entire current row. This fails because n can be 1000 — the state space 31000 is astronomically large. The key is recognizing that m is the small dimension, so states compress the m-cell column, not the n-cell row.

4 The Code (Annotated) ▾

Solution 1 — Tabulation (Bottom-Up DP) — Interview Pick

def colorTheGrid(m: int, n: int) -> int:
MOD = 10**9 + 7 ← always use MOD for counting problems
total = 3 ** m ← total possible column colorings (including invalid)
 
def get_colors(state): ← decode base-3 state to list of m colors
cols = []
for _ in range(m):
cols.append(state % 3) ← extract lowest trit
state //= 3 ← shift right in base 3
return cols
 
def is_valid_column(state): ← no two adjacent rows same color
colors = get_colors(state)
return all(colors[i] != colors[i+1] for i in range(m-1))
 
def is_compatible(s, t): ← no row shares color between adjacent columns
cs, ct = get_colors(s), get_colors(t)
return all(cs[i] != ct[i] for i in range(m))
 
# Step 1: Enumerate valid column states
valid = [s for s in range(total) if is_valid_column(s)]
 
# Step 2: Precompute compatible pairs
compat = {s: [t for t in valid if is_compatible(s, t)] for s in valid}
 
# Step 3: Initialize DP — column 0
dp = {s: 1 for s in valid} ← each valid state can be used once for col 0
 
# Step 4: Process columns 1 to n-1
for _ in range(n - 1):
new_dp = {s: 0 for s in valid} ← fresh counts for next column
for s in valid:
for t in compat[s]:
new_dp[t] = (new_dp[t] + dp[s]) % MOD ← accumulate ways
dp = new_dp
 
return sum(dp.values()) % MOD ← total ways across all final states
⚠️ Why do we precompute compat instead of checking each step?

Checking compatibility within the DP loop would mean recomputing digit-by-digit comparisons for every (s, t) pair on every column — O(n · |valid|² · m) work. Precomputing compat once costs O(|valid|² · m) and reduces the DP inner loop to a simple list traversal. For n=1000 and m=5 this is a 1000× speedup.

Solution 2 — Top-Down (Memoization)

from functools import lru_cache
def colorTheGrid(m: int, n: int) -> int:
MOD = 10**9 + 7
total = 3 ** m
def get_colors(s): ... ← same helper as before
valid = [s for s in range(total) if is_valid_column(s)]
compat = {s: [t for t in valid if is_compatible(s, t)] for s in valid}
 
@lru_cache(maxsize=None)
def dp(col, prev_state): ← ways to color cols [col..n-1] given prev_state
if col == n: return 1 ← filled all columns — valid completion
total_ways = 0
for next_s in compat[prev_state]:
total_ways = (total_ways + dp(col + 1, next_s)) % MOD
return total_ways
 
# First column: try all valid states
return sum(dp(1, s) for s in valid) % MOD
💡 Tabulation vs Memoization — Which to Use?

Both have the same asymptotic complexity. Tabulation is preferred here because: (1) it processes columns sequentially, which is intuitive; (2) it avoids recursion overhead and stack depth; (3) memory is O(|valid|) — just one dict per column. Memoization is easier to derive from the recurrence relation and is a good first step to explain to an interviewer, but then optimize to tabulation.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Notice the Small Dimension

"m ≤ 5 is tiny. This screams exponential-in-m, polynomial-in-n. Total column states = 3^m ≤ 243 — we can enumerate them all."

Step 2: Identify the State

"The adjacency constraint between columns is purely column-to-column. Once I know the coloring of column j, I know exactly which colorings are valid for column j+1. So my DP state is: current column index + previous column's coloring."

Step 3: Encode as Integer

"A column coloring is a sequence of m values from {0,1,2}. I can encode it as a base-3 integer. This makes state representation O(1) and allows quick digit extraction for compatibility checks."

Step 4: Precompute, then DP

"Separate the one-time preprocessing (valid states, compat table) from the n-column DP. This avoids repeating expensive checks inside the loop."

🗣️ What to Say to the Interviewer

"The key observation is that m ≤ 5, meaning each column has at most 3^5 = 243 colorings. I'll encode each column as a base-3 integer and call it a 'state'. First, I'll enumerate all valid states — those where no two adjacent rows in the column share a color. Then I'll precompute which pairs of valid states are 'compatible' — can sit side by side in adjacent columns. Finally, I'll run a DP: dp[s] = number of ways to reach state s at the current column. For each column, I update: new_dp[t] += dp[s] for each (s, t) compatible pair. Time complexity: O(3^m · n) — the number of valid states times n columns. Space: O(3^m) — just the DP array and compat table."

7 Summary & Key Takeaways ▾
Encode each column as a base-3 integer — one digit per row, digit value = color index
Precompute valid states (no adjacent rows same color) and compatible pairs (no row same color in adjacent columns)
Column-DP: dp[s] = number of ways to reach column state s. Transition sums over compatible predecessors
State compression works because m ≤ 5 keeps the state space at most 243 — if m were large, this approach fails
Always apply MOD at every accumulation step to prevent integer overflow
🎯 When to Use State Compression DP
  • Grid/sequence problems where one dimension is small (≤ 20 for bitmask, ≤ 5 for base-k)
  • Adjacency/coloring constraints that only depend on the boundary between sections
  • Counting valid configurations where brute force is exponential but states compress nicely
  • Keywords: "paint", "tile", "color adjacent", "no two neighbors same"
