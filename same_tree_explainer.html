LeetCode #100

Same Tree

Given the roots of two binary trees p and q, return true if they are structurally identical with every corresponding node having the same value.

Easy Trees DFS Preorder Time: O(n) Space: O(h)
1 Understanding the Problem ▾

Two binary trees are the same tree when they are identical in both structure and values. It is not enough to have the same set of values — the values must occupy the exact same positions in the tree hierarchy.

💡 Key Insight

Two trees are the same if and only if: (1) their current root nodes have the same value, (2) their left subtrees are the same tree, AND (3) their right subtrees are the same tree. This is a recursive definition — the problem at each node reduces to the same problem at its children.

✓ Same Tree: p=[1,2,3], q=[1,2,3]

p:    1      q:    1
     / \        / \
    2   3    2   3
Same structure + values → True

✗ Different: p=[1,2], q=[1,null,2]

p:    1      q:    1
     /            \
    2             2
Different structure → False

Edge cases to consider: Both trees completely empty (True — two empty trees are identical), one empty and one not (False — structural mismatch at root), single-node trees with same or different values, and deeply skewed trees that stress the recursion stack.

2 The Algorithm: Simultaneous Preorder DFS ▾

We traverse both trees simultaneously using depth-first recursion. At each pair of corresponding nodes, we check three conditions in order. Failure at any condition short-circuits the entire check.

1 Base case — both null: if p is None and q is None: return True. Two empty subtrees are trivially identical. This is the "success leaf" — every recursion eventually bottoms out here at the leaves.
2 Base case — structural mismatch: if p is None or q is None: return False. One tree has a node where the other has a gap — the structures diverge here. Return False immediately.
3 Value check: if p.val != q.val: return False. Both nodes exist but carry different values — this pair already fails. No need to check their children.
4 Recurse left AND right: Values match here. The answer depends on both subtrees matching. Return isSameTree(p.left, q.left) and isSameTree(p.right, q.right). Python's and short-circuits — if left fails, we skip the right.
isSameTree(p=1, q=1) values match (1 == 1) ✓ ├─ isSameTree(p.left=2, q.left=2) │ values match (2 == 2) ✓ │ ├─ isSameTree(None, None) → True │ └─ isSameTree(None, None) → True │ → True AND True = True └─ isSameTree(p.right=3, q.right=3) values match (3 == 3) ✓ ├─ isSameTree(None, None) → True └─ isSameTree(None, None) → True → True AND True = True Final: True AND True = True ✓
🔑 Traversal Order: Preorder (Root → Left → Right)

We check the current node pair first, then recurse into children. This gives us the earliest possible failure — we detect value mismatches or structural gaps before wasting time on entire subtrees. Any traversal order (inorder, postorder) is also correct, but preorder is most natural here.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every recursive call isSameTree(p, q), the function correctly returns True if and only if the subtree rooted at p is structurally and value-identical to the subtree rooted at q.

Why base cases are correct:

  • both None → Two empty trees are trivially the same. Correct base case. ✓
  • exactly one None → One has a node where the other has a gap. Structurally different. ✓
  • p.val != q.val → Both nodes exist but differ in value. Not the same here. ✓

Why the recursion is correct (by induction): Assume isSameTree correctly handles all trees of height ≤ k. For a tree of height k+1: we check the root pair (correct by base case logic), then use the height-k guarantee to correctly evaluate both subtrees. The AND combines them correctly — both subtrees must match for the whole tree to match.

Why nodes are visited exactly once: Each pair of corresponding positions is visited once. If either node is null, we return immediately. If values differ, we return immediately. We only recurse when we must go deeper — ensuring O(n) visits total in the worst case (all nodes match until the leaves).

⚠️ Common Trap: Accessing .val Before Null Check

If you write if p.val != q.val before checking whether p or q is None, you get an AttributeError: 'NoneType' object has no attribute 'val'. The null guards must come first. Always: check existence before accessing attributes.

4 The Code (Annotated) ▾

Solution 1 — Recursive DFS (Interview Pick, O(n) time / O(h) space)

def isSameTree(p, q): ← compare subtree at p vs subtree at q
if p is None and q is None: ← both empty → trivially identical
return True
if p is None or q is None: ← exactly one null → structural mismatch
return False
if p.val != q.val: ← both exist but values differ
return False
return (isSameTree(p.left, q.left) ← left subtrees must match AND
and isSameTree(p.right, q.right)) ← right subtrees must also match
⚠️ Why two separate null checks (not combined into one)?

If we collapsed to if p is None or q is None: return p is q, it works but is cryptic. The explicit two-step version — first check "both null", then check "exactly one null" — is far more readable and telegraphs the intent to an interviewer. Clarity wins.

Solution 2 — Iterative BFS with a Queue

from collections import deque
def isSameTree(p, q):
queue = deque([(p, q)]) ← seed queue with root pair
while queue: ← process pairs level by level
n1, n2 = queue.popleft() ← next pair to compare
if n1 is None and n2 is None: ← both null → this pair OK, skip
continue
if n1 is None or n2 is None: ← structural mismatch
return False
if n1.val != n2.val: ← value mismatch
return False
queue.append((n1.left, n2.left)) ← enqueue left children
queue.append((n1.right, n2.right)) ← enqueue right children
return True ← all pairs verified successfully
💡 Complexity Summary

Recursive: Time O(n) — each node visited once. Space O(h) call stack. Best case O(log n) balanced tree, worst case O(n) skewed tree (chain-like).
Iterative BFS: Time O(n). Space O(w) where w = max tree width. For complete tree w = n/2, so O(n). Avoids call stack overflow for million-deep trees.
Interview pick: Recursive. It's concise, elegant, and demonstrates fluency with tree recursion. Mention iterative as the production-safe alternative.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the "Simultaneous Tree Walk" Signal

Whenever a problem asks you to compare two trees, merge two trees, or check a mirror relationship, the pattern is simultaneous recursive traversal. Both pointer arguments move in lockstep. Ask: "What does sameness mean at a single node pair?" — then trust the recursion for the rest.

Step 2: Define Base Cases Before Recursive Case

List all ways the recursion must terminate before writing the recursive step. For this problem: (1) both null → True, (2) one null → False, (3) values differ → False. Writing base cases first prevents null pointer bugs and makes your intent crystal clear to the interviewer.

Step 3: State the Recursive Case in English First

Say aloud: "If both nodes exist and have the same value, the trees are the same here — and the answer depends on whether BOTH subtrees are also the same." Then translate: return isSameTree(left, left) and isSameTree(right, right). Speak, then code.

Step 4: Proactively Mention the Iterative Variant

Without being asked: "In production, very deep trees could blow the call stack. I'd use an iterative BFS with a queue of pairs — same logic, O(n) time, but stack-safe." This shows you think beyond correctness to operational concerns.

🗣️ What to Say to the Interviewer

"This has a natural recursive structure. Two trees are the same if their roots match AND both subtrees are the same.
My base cases: if both are null, return True. If exactly one is null, return False — structural mismatch.
If both exist but values differ, return False.
Otherwise: return isSameTree(p.left, q.left) and isSameTree(p.right, q.right).
Time O(n) — we visit each node pair at most once. Space O(h) for the call stack — O(log n) balanced, O(n) worst.
For very deep trees I'd switch to iterative BFS with a deque of pairs."
🔮 Likely Follow-up Questions
  • "Handle very deep/skewed trees?" → Iterative BFS with a deque — no stack overflow.
  • "Symmetric Tree (#101)?" → Mirror check: left vs right with node order flipped.
  • "Subtree of Another Tree (#572)?" → Run isSameTree at every node of the larger tree.
  • "Merge Two Binary Trees (#617)?" → Same simultaneous DFS, but combine values instead of comparing.
  • "What if trees have duplicate values?" → No change — we compare positional values, not global membership.
