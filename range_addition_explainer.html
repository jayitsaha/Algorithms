LeetCode #370

Range Addition

Apply k range-update operations on an array of zeros and return the final array — without brute-forcing each range.

Medium Prefix Sum Difference Array Time: O(n + k) Space: O(n)
1 Understanding the Problem ▾

You are given an integer n and a list of k update operations. The array starts as all zeros. Each operation is a triple [startIndex, endIndex, inc]: add inc to every element from index startIndex through endIndex, inclusive. Return the final array after all operations.

💡 Key Insight

Rather than touching every element in a range (O(n) per operation), record only where a value starts to change and where it stops. A difference array marks +inc at the start and -inc one past the end. A single prefix-sum pass then reconstructs the true values in O(n). Total: O(n + k) instead of O(n × k).

✗ Naive: loop over every element

[0,2]+100 → touch idx 0,1,2
[1,3]+200 → touch idx 1,2,3
[2,4]+50 → touch idx 2,3,4
O(n × k) — TLE for large inputs

✓ Optimal: difference array

[0,2]+100 → diff[0]+=100, diff[3]-=100
[1,3]+200 → diff[1]+=200, diff[4]-=200
[2,4]+50 → diff[2]+=50, diff[5]-=50
Prefix sum → O(n + k) total

Edge Cases: When endIndex = n - 1 (last valid index), end + 1 = n lands in the guard slot, which is safe. Operations can overlap freely. inc may be negative (decreasing a range is valid). Multiple operations on the same index accumulate correctly via addition.

2 The Algorithm: Difference Array + Prefix Sum ▾

The Difference Array technique exploits a key observation: when you add a constant to a range, the differences between adjacent elements only change at the two endpoints of that range — nowhere in between. This lets us encode each range update in O(1), then recover the full answer with one O(n) prefix-sum scan.

1 Allocate diff of size n + 1 (all zeros). The extra slot at index n absorbs writes to diff[end + 1] when end = n - 1, eliminating any bounds check.
2 Encode each operation in O(1): for [start, end, inc], write diff[start] += inc ("value rises by inc starting here") and diff[end + 1] -= inc ("cancel that rise one past the range").
3 Prefix-sum pass: scan diff[0..n-1], maintaining a running total. At each index i, running += diff[i] and then result[i] = running. The running total automatically reflects every range that covers index i.
4 Return result[0..n-1]. The guard slot (index n) is never included in the output. The answer is the prefix sum of the difference array.

🔑 Why allocate n+1 slots?

When end = n - 1 (the last valid index), end + 1 = n. If diff has only n slots, that write is out of bounds. Allocating n + 1 slots gives a safe landing zone. That guard value is computed in the prefix sum but never added to result — we only loop for i in range(n).

3 Why It Works: The Invariant ▾
🔐 The Invariant

After encoding all operations, diff[i] holds the net change in value between position i and position i-1. The prefix sum at index i accumulates all changes that started at or before i and have not yet been cancelled — this is precisely the total increment that all operations covering index i contribute.

Proof for a single operation [s, e, v]: After encoding, diff[s] += v and diff[e+1] -= v. For the prefix sum at index i:

  • i < s: neither +v nor -v reached yet. Contribution = 0.
  • s ≤ i ≤ e: +v accumulated at index s; -v at e+1 not yet reached. Net = +v. Correct!
  • i > e: both +v and -v accumulated. Net = 0. Correct!

Multiple operations superpose linearly. Because prefix sums add independently, contributions from all k operations stack without interference. Every index i naturally accumulates exactly the sum of inc values from every operation whose range contains i.

⚠️ Critical: encode ALL ops before the prefix-sum pass

You must finish writing all +inc / -inc pairs into the diff array before computing any prefix sum. Interleaving the two phases will corrupt results because a later operation's diff[start] += will add to an already-summed value.

🚫 Off-by-one: diff[end] vs diff[end+1]

Writing diff[end] -= inc (instead of diff[end+1]) incorrectly cancels the increment at the last element of the range itself. Index end is inside the range and must receive the increment. The cancellation belongs at end + 1.

4 The Code (Annotated) ▾

Solution 1 — Difference Array (Optimal, Interview Pick)

def getModifiedArray(n: int, updates: list) -> list:
diff = [0] * (n + 1) ← size n+1: guard slot at index n absorbs end+1 writes safely
for start, end, inc in updates: ← destructure each operation; O(1) per op
diff[start] += inc ← mark: value rises by inc starting at this index
diff[end + 1] -= inc ← cancel the rise one past the end of the range
result, running = [], 0 ← output list + prefix-sum accumulator
for i in range(n): ← scan indices 0..n-1 only (skip the guard)
running += diff[i] ← absorb the net change at this position
result.append(running) ← running IS the value at index i after all ops
return result ← final array; Time O(n+k), Space O(n)
⚠️ Why diff[end + 1] -= inc and not diff[end] -= inc?

Index end is still inside the range — it must receive the full increment. The subtraction must appear at the first index outside the range: end + 1. Writing it at end would short-circuit the last element, producing a wrong answer.

Solution 2 — Brute Force (O(n × k), for comparison)

def getModifiedArray_brute(n: int, updates: list) -> list:
result = [0] * n
for start, end, inc in updates: ← for each operation...
for i in range(start, end + 1): ← ...directly update every element in range — O(n)
result[i] += inc ← correct but slow; TLE when n=k=10^5
return result
💡 Complexity Summary

Difference Array: Time O(n + k), Space O(n) — k ops at O(1) each, one O(n) pass
Brute Force: Time O(n × k), Space O(n) — each op touches up to n elements
Segment Tree (lazy): Time O((n+k) log n) — overkill here; needed when queries arrive online

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: State the Naive Approach and Its Cost

"Straightforward: for each operation, loop through every element in the range and add inc. That's O(n) per operation, O(n×k) total. For n=k=10⁵ that's 10¹⁰ operations — clearly too slow."

Step 2: Identify the Redundancy

"The bottleneck: elements in the interior of a range all get the same delta — we're repeating the same write n times. What if we only mark the two boundary points where something changes?"

Step 3: Arrive at the Difference Array Idea

"Write +inc at start, and -inc at end+1. After all operations, computing the prefix sum of that 'diff' array tells me the net effect at each position — each +inc propagates forward until cancelled by the matching -inc."

Step 4: Handle the Off-by-One at the Boundary

"I size diff as n+1 so that when end = n-1, writing diff[n] doesn't go out of bounds. The n-th slot absorbs the cancellation but is never included in the output — I only iterate i in range(n)."

🗣️ What to Say to the Interviewer

"The naive approach is O(n*k). The key observation: elements within a range all get the same delta, so I only need to mark where the delta begins and ends — two O(1) writes per operation.
I'll use a difference array of size n+1. For each [start, end, inc]: diff[start] += inc; diff[end+1] -= inc. Then a single prefix-sum pass reconstructs the final values. Time: O(n+k), Space: O(n)."
🔮 Likely Follow-up Questions
  • "What if you also need to query any index in between operations?" → Segment tree with lazy propagation — O(log n) per update and query
  • "What about a 2D version (update a rectangle)?" → 2D difference array: apply the trick on both rows and columns
  • "Can inc be negative?" → Yes, same algorithm. Subtraction at start, addition at end+1
  • "What is the minimum number of operations to make the array all-ones?" → Reverse of this problem — treat the target array as a difference array

7 Summary & Key Takeaways ▾
Difference array turns every range update into two O(1) point writes
One prefix-sum pass recovers the true values — total O(n + k)
Allocate n + 1 slots to avoid bounds checking when end = n - 1
Encode ALL operations before the prefix-sum pass — never interleave
Difference array is the inverse of prefix sum: one is for queries, the other for updates

🎯 When to Recognize the Difference Array Pattern
  • "Add a value to all elements in a range" and you need the final array → difference array
  • Multiple range-increment operations, result needed offline (after all ops) → difference array
  • "Passengers board at stop A and exit at stop B" style problems → difference array
  • Any problem where range-writes are heavy but individual reads are light → difference array
