LeetCode #328

Odd Even Linked List

Group all nodes at odd indices first, then nodes at even indices, preserving relative order within each group. Solve in O(1) extra space.

Medium Linked List Two Separate Lists Time: O(n) Space: O(1)
1 Understanding the Problem ▾

You receive a singly linked list. Your job is to reorder it in-place so that all nodes at odd positions (1st, 3rd, 5th, …) come first, followed by all nodes at even positions (2nd, 4th, 6th, …). The relative order within each group must be preserved — odd-indexed nodes appear in their original left-to-right order, and same for even-indexed nodes.

💡 Key Insight

Think of the list as two interleaved sub-lists that have been zipped together. Our goal is to unzip them — collect the odd-indexed chain and the even-indexed chain separately, then attach the even chain to the tail of the odd chain. We only touch pointers, never node values, so no extra memory is needed.

✗ Original (interleaved)

1 → 2 → 3 → 4 → 5
pos: 1 2 3 4 5
odd: 1 3 5
even: 2 4

✓ Reordered (two groups)

1 → 3 → 5 → 2 → 4
[odd chain] → [even chain]
Relative order preserved
O(1) space — pointer ops only

Edge Cases to consider: empty list (return null), single node (no reordering needed), two-node list (already correct — first is odd, second is even), lists where the last node is even-indexed vs odd-indexed (affects the loop termination condition).

⚠️ Index vs Value — Critical Distinction

The problem is about position indices (1-based), NOT the stored values. If the list is [2 → 4 → 6], node at position 1 (value=2) is odd-indexed, position 2 (value=4) is even-indexed. Don't let "even/odd" in the problem name mislead you into testing value parity.

2 The Algorithm: Two Separate Lists (Weave and Concatenate) ▾

We maintain two "chain tails" that grow simultaneously. In each loop iteration we advance both tails by two positions (skipping over the other group), building up two separate chains in place. When no more pairs exist, we stitch the even chain onto the odd chain's tail.

1 Guard clause: If the list has 0 or 1 nodes, return head immediately — the list is trivially in the correct order.
2 Initialize: odd = head (tail of odd chain, starts at position 1). even = head.next (tail of even chain, starts at position 2). even_head = even saves the start of the even chain — we'll need this after the loop to attach.
3 Weave loop: While even is not null AND even.next is not null: (a) odd.next = even.next — skip even to reach next odd. (b) odd = odd.next — advance odd tail. (c) even.next = odd.next — skip odd to reach next even. (d) even = even.next — advance even tail.
4 Concatenate: After the loop, odd is the tail of the odd chain. Set odd.next = even_head to attach the full even chain. Return head.

🔑 Why Not Just Collect Values?

A brute-force approach: scan the list, collect odd-indexed values in one array and even-indexed in another, then overwrite node values. O(n) time but O(n) space. The two-separate-lists technique is also O(n) time but only O(1) space — we operate entirely by re-wiring next pointers, never allocating new storage.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At the start of every loop iteration, odd is the current tail of a correctly-ordered odd-position chain, and even is the current tail of a correctly-ordered even-position chain. All nodes already processed maintain their original relative order within their respective groups. The two chains grow strictly left-to-right, one pair per iteration.

Why relative order is preserved: We process original nodes strictly in order of position: 1, 2, 3, 4, 5, … Each node is appended to its group's chain in that same order. Since we never skip or reorder within a group, relative ordering is automatically maintained.

Why the loop terminates correctly: The condition while even and even.next handles both list lengths. For an odd-length list (like n=5), after processing the last pair (3,4), odd is at 5 and even is at 4 — even.next is null, loop exits. For an even-length list (like n=4), after processing pair (3,4), even becomes null, loop exits. Both cases are clean.

⚠️ Order of the Four Lines Inside the Loop Matters

The update sequence must be: (1) odd.next = even.next, (2) odd = odd.next, (3) even.next = odd.next, (4) even = even.next. You MUST advance odd before setting even.next, because step 3 needs to see the newly-advanced odd's next pointer (which is the next even node). Getting steps 2 and 3 swapped produces a corrupt list.

Why even_head is saved before the loop: As we weave, we overwrite next pointers. After the loop, even sits at the last even node (or null). We have no way to walk back to the chain's start. Saving even_head = even before any modifications gives us a stable anchor — the very first even-indexed node — that we can use to splice the chains together.

4 The Code (Annotated) ▾

Solution 1 — Two Separate Lists (Optimal, O(n) / O(1)) — Interview Pick

def oddEvenList(head): ← accepts head of singly-linked list
if not head or not head.next: ← 0 or 1 node → trivially correct, return as-is
return head
odd = head ← odd chain tail starts at position 1
even = head.next ← even chain tail starts at position 2
even_head = even ← SAVE even chain head — must do before any rewiring
while even and even.next: ← loop while there's an even node AND a next odd node
odd.next = even.next ← Step 1: odd tail skips over even, points to next odd
odd = odd.next ← Step 2: advance odd tail to that next odd node
even.next = odd.next ← Step 3: even tail skips over (advanced) odd, to next even
even = even.next ← Step 4: advance even tail
odd.next = even_head ← Stitch: tail of odd chain → head of even chain
return head ← head (pos 1) is still the start of the result
⚠️ What if we set even.next BEFORE advancing odd?

If we did steps 3 and 2 in the wrong order — even.next = odd.next first, then odd = odd.next — then odd.next at step 3 would still point to the next even node (not the next odd). We'd be linking the even chain to an even node, producing a corrupt result. Steps 2 then 3 is the only valid order.

Solution 2 — Value Overwrite with Arrays (O(n) time, O(n) space — simpler but suboptimal)

def oddEvenList(head):
odds, evens, node, pos = [], [], head, 1
while node: ← collect values into two arrays
(odds if pos % 2 else evens).append(node.val)
node, pos = node.next, pos + 1
node = head
for val in odds + evens: ← overwrite node values in new order
node.val, node = val, node.next
return head
💡 Complexity Summary

Two Separate Lists (optimal): Time O(n) — single pass through the list. Space O(1) — only 3 extra pointer variables.
Value Overwrite: Time O(n) — two passes. Space O(n) — stores all n values in arrays.
Interview note: Always lead with the O(1) space pointer-rewiring solution. Mention value overwrite only as the naive approach you're improving upon.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Pattern Signal

Key signals: "reorder linked list in-place", "group by index property", "O(1) extra space". Whenever you see "partition a linked list while preserving relative order within groups", think Two Separate Lists. You build each group's chain simultaneously, then concatenate.

Step 2: Identify What State to Track

Ask yourself: what's the minimum state needed? (1) Tail of odd chain — so I can append to it. (2) Tail of even chain — so I can append to it. (3) Head of even chain — so I can attach it at the end. That's it. Three pointers total. No arrays needed.

Step 3: Simulate on a 5-Node Example

Draw [1 → 2 → 3 → 4 → 5] on paper. Walk through the four pointer assignments per iteration. This will naturally surface the correct update order (odd.next first, advance odd, even.next second, advance even) and make the loop termination condition obvious.

Step 4: Verify Both List-Length Parities

Test n=5 (odd-length) and n=4 (even-length) to confirm the loop condition while even and even.next handles both. Showing this rigor is a clear signal of interviewing maturity — most candidates forget to check edge cases.

🗣️ What to Say to the Interviewer

"I want O(1) space, so I'll rewire pointers rather than store any values.
I'll maintain two chain tails: odd (starts at head) and even (starts at head.next).
I also save even_head before the loop — I'll need that for the final concatenation.
Each iteration: odd.next skips to next odd, I advance odd; even.next skips to next even, I advance even.
Loop condition: while even and even.next — covers both odd-length and even-length lists.
After the loop: odd.next = even_head to stitch the chains. Return head.
Time O(n), Space O(1)."
🔮 Likely Follow-up Questions
  • "What if we want even-indexed nodes first?" → Swap the initialization: start even = head, odd = head.next, even_head = even. Concatenate even chain then odd chain.
  • "Partition list by value less than k?" → Same two-chain sub-pattern (#86 Partition List) — split criterion changes but structure is identical.
  • "What if the list has exactly 2 nodes?" → even = node2, even.next is null → loop doesn't execute → odd.next = even_head connects correctly. Test it!
  • "Can you do it recursively?" → Yes, but the recursive stack uses O(n) space, defeating the purpose. Stick with the iterative approach.
