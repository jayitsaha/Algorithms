LeetCode #424

Longest Repeating Character Replacement

Find the length of the longest substring containing the same letter after replacing at most k characters.

Medium Sliding Window Max Freq + Window Size Check Time: O(n) Space: O(1)
1 Understanding the Problem ▾

Given a string s and an integer k, you can pick any substring and change at most k of its characters to any letter. Your goal: find the longest substring that can be made into a single repeating character (all same letter) using at most k changes.

For example, s = "AABABBA", k = 1. Consider the substring "AABA" (indices 0–3). It has three A's and one B. We can replace that one B with an A, giving "AAAA" — length 4 using just 1 replacement. Can we do better? No — any length-5 window needs at least 2 changes.

💡 Key Insight

In any window, the optimal strategy is to keep the most-frequent character and replace all others. The number of replacements needed = (window size) − (count of the most frequent character). A window is valid if this is ≤ k.

Valid window condition: (R − L + 1) − max_freq ≤ k
where max_freq = count of the most-frequent character inside the window
✓ Valid: "AABA", k=1

A appears 3 times (max_freq = 3)
window size = 4
need = 4 − 3 = 1 ≤ k=1 ✓
Replace 'B' → "AAAA"

✗ Invalid: "AABAB", k=1

A appears 3 times (max_freq = 3)
window size = 5
need = 5 − 3 = 2 > k=1 ✗
Cannot make uniform with 1 swap

Edge Cases: k ≥ len(s) → the whole string is the answer (replace everything). k = 0 → find the longest run of a single character. All same characters → return len(s). Single character → return 1.

2 The Algorithm: Variable Sliding Window ▾

We use a variable sliding window — two pointers L and R defining the window boundaries. We always try to grow the window by advancing R. When the window becomes invalid (needs more than k replacements), we shrink it by advancing L — but only by exactly 1.

1 Initialize: L = 0, max_freq = 0, result = 0, and a frequency dictionary count tracking char counts inside the window.
2 Expand right: Move R one step right. Add s[R] to the window: count[s[R]] += 1. Update max_freq = max(max_freq, count[s[R]]).
3 Validate the window: Compute need = (R − L + 1) − max_freq. If need > k, the window is invalid — we need more replacements than allowed.
4 Shrink if invalid: Remove s[L] from the window (count[s[L]] -= 1) and advance L += 1. Crucially: do this only once (not a while-loop). This slides the window forward without shrinking it below the best-found size.
5 Track the best: result = max(result, R − L + 1). After all positions, return result.
🔑 The Clever Insight: Window Never Shrinks Below Best

When the window becomes invalid, instead of collapsing it (while-loop shrink), we slide it forward — remove one from the left, add one to the right. The window size stays the same. This means max_freq is intentionally "stale" after a shrink — we only update it upward when we find a bigger-than-current-best window. The window either grows or stays the same size.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every step, the window [L, R] has size equal to the best window we have found so far. max_freq records the highest single-character count ever seen across all windows visited. The window never shrinks below that best size — it can only grow.

Why max_freq never decreases: When we shrink from the left, the true max frequency inside the window might drop. We don't recalculate it. Is this a bug? No! Here's why: after shrinking, window size = R − L + 1 (same as before). With the stale (higher) max_freq, the validity check becomes window_size − stale_max_freq ≤ k — which evaluates to the same size. We won't grow unless a new character gives us a higher max_freq — which means we've found a genuinely better window.

Why shrink by 1 instead of a while-loop: In most sliding window problems, you shrink until valid. Here we don't, because we only care about windows at least as long as our current best. A window of the same size as our current best — even if invalid — doesn't hurt us (we just slide it forward). We expand only when the window can grow to something better.

⚠️ Common Mistake: Recalculating max_freq After Shrink

Some solutions scan all keys to recompute max_freq after removing s[L]. This makes the algorithm O(n·26) instead of O(n), but more importantly it's unnecessary — the stale max_freq is correct by the logic above. Only update max_freq upward when adding to the window.

Correctness proof sketch: Every optimal window [L*, R*] will be visited by our algorithm because: (1) R visits every index as the right endpoint, (2) L only advances when forced — so for any right endpoint where the optimal window starts at L*, our L ≤ L*, meaning our window is at least as wide. The maximum R − L + 1 seen equals the optimal answer.

4 The Code (Annotated) ▾

Solution 1 — Optimal Sliding Window (Interview Pick), O(n)

def characterReplacement(s: str, k: int) -> int:
count = {} ← tracks char frequencies inside current window
max_freq = 0 ← highest single-char count seen (only grows)
L = 0 ← left boundary of sliding window
result = 0 ← best (longest valid) window seen so far
for R in range(len(s)): ← R expands the window rightward
count[s[R]] = count.get(s[R],0) + 1← include s[R] in the window
max_freq = max(max_freq, count[s[R]])← only update upward — never recalculate
if (R - L + 1) - max_freq > k: ← need more replacements than budget? → invalid
count[s[L]] -= 1 ← remove leftmost char from freq map
L += 1 ← slide window left boundary forward by 1
result = max(result, R - L + 1) ← window size = best seen (grows or stays same)
return result
⚠️ if vs while for shrinking

Using if (not while) means we shrink by exactly 1 when invalid. This is correct here because: we added 1 to the right, so validity dropped by at most 1. Removing 1 from the left restores it. We never need to shrink more than once per step.

Solution 2 — Brute Force O(n²) — for understanding

def characterReplacement_brute(s, k): ← O(n²) — TLE for n=100,000
n, result = len(s), 0
for L in range(n): ← try every left endpoint
count = {}
for R in range(L, n): ← try every right endpoint
count[s[R]] = count.get(s[R],0) + 1
max_freq = max(count.values()) ← O(26) scan every inner step
if (R-L+1) - max_freq <= k: ← is this window valid?
result = max(result, R-L+1)
return result
💡 Complexity Summary

Optimal Sliding Window: Time O(n) — L and R each traverse once (2n total moves). Space O(26) = O(1) for uppercase A–Z.
Brute Force: Time O(n²·26) — n² windows × 26 to find max frequency. Space O(26).

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Spot the Sliding Window Signal

"Longest substring/subarray satisfying a constraint" → almost always a sliding window. The constraint here is "at most k replacements." That's a budget constraint on a window — textbook sliding window territory.

Step 2: Derive the Validity Formula

"To make all chars the same, I keep the most frequent one and replace everything else. Replacements needed = window_size − max_frequency_in_window. Valid if this ≤ k." Write this formula explicitly — it's the key to the solution.

Step 3: Propose Brute Force, Then Optimize

"Brute force: try all O(n²) windows. For each, count characters in O(26). Check validity. O(n²) total." Then: "I can optimize with a sliding window — track max_freq as I expand R, shrink L when invalid. O(n) total."

Step 4: Explain the max_freq Trick

"The non-obvious part: max_freq never decreases. After a shrink, the window stays the same size. We only allow the window to grow when a new character count exceeds the current max_freq — meaning we found a genuinely better window."

🗣️ What to Say to the Interviewer

"This is a sliding window problem — I'm looking for the longest substring with a budget constraint.
The key formula: (window size) − (count of most-frequent char) = replacements needed.
Valid window: replacements needed ≤ k.
I'll expand R right each step, update max_freq upward, and if the window is invalid,
I'll shrink from the left by exactly 1 — not a while loop, because I only care about
windows larger than my current best. This gives O(n) time, O(1) space."
🔮 Likely Follow-up Questions
  • "What if the input contains lowercase letters?" → Use a dict instead of a fixed-26 array. Functionally identical.
  • "What if k = 0?" → Find the longest run of a single character. Same algorithm, validity condition just becomes max_freq == window_size.
  • "Can you generalize to numbers?" → Yes. #1004 Max Consecutive Ones III is the binary version: replace at most k zeros.
  • "Prove max_freq never needs to decrease." → After shrink, window same size. We only grow when new char exceeds old max_freq — a strictly better situation. The stale value is harmless.
