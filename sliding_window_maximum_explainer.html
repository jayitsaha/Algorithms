LeetCode #239

Sliding Window Maximum

Given an integer array nums and an integer k, return an array of the maximum value in each contiguous window of size k as it slides from left to right.

Hard Monotonic Queue Decreasing Deque Time: O(n) Space: O(k)
1 Understanding the Problem ▾

Imagine placing a magnifying glass of width k over an array. At each position, you can see exactly k numbers. You slide the glass one position to the right, and ask: what is the biggest number I can see right now? Do this for every valid position and collect all the answers.

💡 Key Insight

Finding the max of a static array is O(n) — easy. The challenge is that our "array" changes every step: one element leaves from the left, one enters from the right. The naive approach (re-scan all k elements) costs O(k) per step, giving O(n·k) total. For large k, this is prohibitively slow. The question is: can we track the maximum in O(1) per step, amortized?

Example: nums=[1,3,-1,-3,5,3,6,7], k=3

Window [ 1, 3,-1,-3, 5, 3, 6, 7] → max = 3

Window [ 1, 3,-1,-3, 5, 3, 6, 7] → max = 5
Window [ 1, 3,-1,-3, 5, 3, 6, 7] → max = 6

2 The Algorithm: Monotonic Decreasing Deque ▾

The key observation: when element v enters the window, any element already in the window that is both older and smaller than v can be permanently discarded. Why? For every future window that still contains that smaller element, it also contains v (since v came later). And v is larger. So the smaller element will never be the maximum of any future window.

We use a deque (double-ended queue) storing array indices (not values), with the invariant that the values at those indices are strictly decreasing from front to back. The front is always the current window's maximum index.

1 Evict expired (front): If dq[0] <= i - k, the front index is no longer in the current window [i-k+1, i]. Remove it with popleft(). Repeat until the front is valid or deque is empty.
2 Remove dominated (back): While the back of the deque has value ≤ nums[i], pop it. That index is now permanently useless — it is dominated by index i which is larger and newer.
3 Append current index: Push i to the back. The deque is now still decreasing (new element appended, all dominated elements removed).
4 Record max when window is full: Once i >= k-1, the first full window exists. The front of the deque, nums[dq[0]], is the window maximum. Append it to result.

🔑 Why Indices, Not Values?

We need to check whether the front element is still inside the window. To do that, we compare its index against the current window boundary. If we stored values, we would have no way to know when an element expires.

3 Why It Works: The Invariant ▾
🔐 The Invariant (Always True)

After processing index i, the deque contains a subsequence of indices from the current window [max(0, i-k+1), i], in increasing order, such that their corresponding values are strictly decreasing front-to-back. The front index points to the window maximum.

Proof that discarding dominated elements is safe: Suppose index j < i is in the deque and nums[j] <= nums[i]. For any window W containing j, since j < i, if j is still in W, then i is in W too (because windows are contiguous and i came after j but within the same window lifespan). Since nums[i] >= nums[j], the maximum of W is at least nums[i], so nums[j] can never be the maximum of W. Discarding j is lossless.

Why is the complexity O(n)? Each of the n indices is pushed to the deque exactly once and popped at most once (either from back when dominated, or from front when expired). Total deque operations: O(n). Result array construction: O(n-k+1). Overall: O(n).

⚠️ Order of Operations Matters

Always evict expired elements from the front before removing dominated elements from the back. If you swap the order, you may incorrectly remove valid candidates based on a comparison against a stale deque state. The eviction step cleans up the past; the dominated-removal step prepares for the future.

Edge case — equal values: When nums[dq[-1]] <= v, we pop even if equal. Why? The new index i is strictly newer (larger index). If two elements have the same value, the older one will expire from the window sooner, so we keep the newer one. Using <= correctly prioritizes the index that stays valid longest.

4 The Code (Annotated) ▾

Solution 1 — Monotonic Deque (Optimal — Interview Pick)

from collections import deque ← O(1) popleft + pop, unlike list.pop(0)
def maxSlidingWindow(nums, k):
dq = deque() # stores indices ← indices, not values!
result = []
for i, v in enumerate(nums): ← single O(n) pass
# Step 1: Evict expired front
while dq and dq[0] <= i - k: ← dq[0] is out of [i-k+1, i]
dq.popleft() ← O(1) removal from front
# Step 2: Remove dominated back
while dq and nums[dq[-1]] <= v:← smaller+older → useless
dq.pop() ← O(1) removal from back
# Step 3: Append current index
dq.append(i) ← deque remains decreasing
# Step 4: Record max
if i >= k - 1: ← first full window at i = k-1
result.append(nums[dq[0]]) ← front is always the max
return result
⚠️ Common Mistake: Using a List Instead of deque

Python's list.pop(0) is O(n) because it shifts all elements. Always use collections.deque for O(1) popleft(). Using a plain list turns the solution back to O(n²) in the worst case.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize Sliding Window + Extreme Value

Keywords: "sliding window of size k", "maximum" (or minimum). This pattern signals you need an efficient way to maintain the extreme value across window slides — classic monotonic queue territory.

Step 2: State the Brute Force First

"I can use max() on each window slice — O(n·k). For n=10⁵ and k=50000, that's 5×10⁹ operations. The interviewer will nod and ask for better. This buys you time to think."

Step 3: Ask — What Can We Throw Away?

Think about what information becomes useless. Key realization: any element that is smaller than the newest element AND entered the window earlier will never be the maximum of any future window. It's safe to permanently discard it right now.

Step 4: Choose the Right Data Structure

We need: O(1) pop from front (expired), O(1) pop from back (dominated), O(1) push to back (new element), O(1) peek at front (current max). Only a deque provides all four. Explain this reasoning explicitly.

Step 5: Justify O(n) Amortized

Each of the n elements is pushed once and popped at most once (total: ≤ 2n operations). Therefore the algorithm is O(n) despite the nested while loops.

🗣️ What to Say

"The naive approach scans k elements per window for O(n·k). The insight is that when element v enters, any element in the deque that is both smaller AND older is permanently dominated — v is bigger and will outlast it in the window. We can discard those useless elements immediately. A deque lets us remove from both ends in O(1): we pop expired indices from the front, and pop dominated indices from the back before appending the new one. The front always holds the window maximum. Since each element enters and leaves the deque at most once, total time is O(n)."
💡 Follow-up Questions

Window minimum? Use a monotonic increasing deque — pop from back when new value is smaller or equal. All other logic is identical.

What if k is variable? The deque approach only works for fixed-k expiry. For variable windows you would need a segment tree or sparse table for O(1) or O(log n) range max queries.

What about using a heap? A max-heap works but costs O(n log n) due to heap insertions and the need to lazily remove expired elements — valid but suboptimal compared to the deque.

7 Summary & Key Takeaways ▾
Store indices (not values) in the deque — you need indices to check window expiry.
Pop from the back any element dominated by the new incoming value — it can never be a future max.
Pop from the front any index that has left the window — its contribution is over.
The front of the deque is always the current window maximum — O(1) lookup per step.
Each element enters the deque once and leaves at most once — total: O(n) amortized.
Space is O(k) — the deque holds at most k live indices at any moment.
🎯 Pattern Recognition Signals
  • Fixed-size sliding window + range max or min
  • Brute force O(n·k) is too slow and you need O(n)
  • A newly arrived element can permanently invalidate earlier elements
  • You need O(1) access to the current extreme value at every step
🧠 Analogy: The Ticket Queue

Imagine a concert ticket queue. Each person holds a numbered ticket (their value). When a stronger performer arrives, all weaker performers already in queue who will leave before this person get dismissed immediately — they'll never headline while this act is available. When someone's time slot expires, they leave from the front. At all times, the headliner is whoever is at the front of the queue.
