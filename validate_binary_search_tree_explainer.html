LeetCode #98

Validate Binary Search Tree

Given the root of a binary tree, determine if it is a valid binary search tree (BST).

Medium Trees DFS · In-order Time: O(n) Space: O(h)
1 Understanding the Problem ▾

A Binary Search Tree (BST) has a strict ordering property: every node's value must be greater than all values in its left subtree and less than all values in its right subtree. This must hold for every node in the entire tree.

💡 Key Insight

The BST property is global, not local. It is not enough to check that each node is greater than its left child and less than its right child. You must enforce that every node lives within a valid range inherited from all of its ancestors above it.

✓ Valid BST: [5, 3, 7, 1, 4]

✗ Invalid: [5, 1, 4, null, null, 3, 6]

  1   4  ← 4 < 5 (right child!)
     / \
    3  6 ← local check passes, global fails

Formal BST definition: For every node N with value V — all values in N's left subtree are strictly less than V, and all values in N's right subtree are strictly greater than V. Duplicates are not allowed in a standard BST.

Edge cases to consider: single node (always valid), null root (valid), nodes with integer boundary values (use float('-inf')/float('+inf') to avoid overflow), and the classic trap where a subtree appears locally valid but violates an ancestor constraint.

2 The Algorithm: DFS with Inherited Bounds ▾

We perform a DFS and pass inherited range bounds (low, high) down the tree. When we descend left, we tighten the upper bound to the current node's value. When we descend right, we tighten the lower bound. Every node must fit strictly within its inherited range.

1 Start at root, bounds (-∞, +∞): The root has no ancestor constraints — any value is initially acceptable. Call validate(root, -inf, +inf).
2 Base case — null node: An empty subtree is always a valid BST. Return True. This cleanly handles leaf children without special-casing.
3 Check current node: If node.val ≤ low or node.val ≥ high, this node is out of bounds. Return False immediately (fail fast).
4 Recurse left, tighten upper bound: The left subtree's values must be less than the current node. Call validate(node.left, low, node.val).
5 Recurse right, tighten lower bound: The right subtree's values must be greater than the current node. Call validate(node.right, node.val, high).
6 Return the AND: Both subtrees must be valid. Return left_valid AND right_valid. Python short-circuits — if left fails, right is never evaluated.
🔑 Alternative: In-Order Traversal Check

In-order traversal of any valid BST produces a strictly increasing sequence. We can traverse in-order, tracking the previously visited value, and fail immediately if any value is not strictly greater than the previous. Both approaches are O(n) time and O(h) space — the bounds approach is more general (no external state needed); the in-order approach is easier to explain verbally.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every recursive call validate(node, low, high): if this call returns True, then every node in the subtree rooted at node has a value strictly in the open interval (low, high). The range (low, high) captures all constraints inherited from every ancestor above this node.

Why bounds propagation captures the global property: When we go left from a node with value V, every node in the left subtree must (a) be less than V (BST property from V), and (b) satisfy the lower bound inherited from V's ancestors. So the new range is (low, V). Symmetrically, going right gives (V, high). Every constraint from the entire path from root to the current node is encoded in (low, high).

Why local parent-child comparison fails (the classic trap): Consider root=5, right child=4, with 4's children being 3 and 6. Locally every parent-child pair looks valid. But 4 is a right child of 5, so 4 must be greater than 5 — and it isn't. The bounds approach catches this immediately: when recursing right from root 5, the lower bound becomes 5. Node 4 fails the check 4 > 5.

⚠️ Why strict inequality — <= and >= in the failure check

The check is node.val <= low or node.val >= high. This enforces strict BST ordering — equal values are not allowed. A common mistake is using < and > (non-strict), which would incorrectly accept duplicates. Bounds are exclusive: a node's value must be strictly between low and high, not equal to either.

Why float('-inf') and float('inf') for initial bounds: Using None to mean "no bound" requires special-casing the root and every comparison. Using infinity keeps the comparison logic uniform everywhere — every node, including the root, runs the exact same val <= low or val >= high check. The root simply trivially passes since any finite integer is inside (-inf, +inf).

4 The Code (Annotated) ▾

Solution 1 — DFS with Inherited Bounds (Interview Pick, O(n) time, O(h) space)

def isValidBST(root: TreeNode) -> bool:
def validate(node, low, high): ← helper: carries inherited range constraints
if node is None: ← base case: empty subtree is always valid
return True
if node.val <= low or node.val >= high: ← node must be STRICTLY inside (low, high)
return False ← out of inherited range: BST violated
return (validate(node.left, low, node.val) ← left: upper bound tightens to current val
and validate(node.right, node.val, high))← right: lower bound tightens to current val
return validate(root, float('-inf'), float('inf'))← root starts unconstrained
⚠️ Common Mistake: Checking Only Immediate Children

Writing node.val > node.left.val and node.val < node.right.val only validates parent-child relationships. It misses global violations where a node's value is wrong relative to a distant ancestor. Always propagate bounds through the entire recursion.

Solution 2 — In-Order Traversal with Previous Value Tracking

def isValidBST(root: TreeNode) -> bool:
prev = [float('-inf')] ← list to allow mutation inside nested function
def inorder(node):
if node is None: return True ← null leaf is trivially valid
if not inorder(node.left): return False ← recurse LEFT first (in-order)
if node.val <= prev[0]: return False ← current must be STRICTLY greater than previous
prev[0] = node.val ← update: this node is now the "previous"
return inorder(node.right) ← recurse RIGHT last (in-order)
return inorder(root)
💡 Complexity Summary

Both approaches: Time O(n) — each node is visited exactly once. Space O(h) for the recursion stack, where h is the tree height. O(log n) for balanced trees, O(n) worst-case for a completely skewed tree (effectively a linked list).

Bounds DFS: No extra state — bounds are passed as function arguments.
In-order traversal: Needs a prev variable (use a list or nonlocal).

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify the Pattern

Tree problem asking to validate a property at every node → DFS. The property involves relationships across multiple levels (not just parent-child) → we need to carry context (bounds) downward through the recursion, not just local comparisons.

Step 2: Construct the Counter-Example to Motivate Bounds

Draw root=5, right child=4, with 4's children 3 and 6. Every parent-child pair looks valid locally. But 4 is in the right subtree of 5, so 4 must be > 5. Show the interviewer why local checks fail, then derive: we need a range from ancestors.

Step 3: Define the Range and Recursion

Root range: (-∞, +∞). Going left narrows the upper bound. Going right narrows the lower bound. State the base case (null → True) and the failure case (value outside bounds → False). The recursion writes itself.

Step 4: Offer the Alternative (In-Order)

"There's an alternative: traverse in-order (left-root-right). A valid BST always produces a strictly increasing sequence. I track the previous value and fail if current ≤ previous." This shows depth — you know two clean approaches.

🗣️ What to Say to the Interviewer

"The BST property is global — each node must be greater than everything in its left
subtree, not just its immediate left child. Parent-child comparison is insufficient.

My approach: DFS where each call receives inherited bounds (low, high).
At root: bounds are (-∞, +∞). Going left from node V: upper bound becomes V.
Going right from node V: lower bound becomes V.
A node fails if its value is not strictly inside (low, high).

Time O(n) — each node visited once. Space O(h) for the call stack.

BST validity is global — every node must satisfy constraints from ALL its ancestors, not just its parent
Carry inherited bounds (low, high) through DFS: tighten upper going left, tighten lower going right
Use float('-inf') and float('inf') as initial bounds — no special-casing needed at the root
