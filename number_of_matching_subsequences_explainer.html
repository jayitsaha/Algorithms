LeetCode #792

Number of Matching Subsequences

Given a string s and an array of words, return the count of words[i] that are subsequences of s.

Medium Hash Tables Bucket by First Char Time: O(|s| + Σ|w|) Space: O(Σ|w|)
1 Understanding the Problem ▾

A string w is a subsequence of string s if all characters of w appear in s in the same relative order — but not necessarily consecutively. Characters in s can be skipped. For example, "ace" is a subsequence of "abcde" because we can pick positions 0 ('a'), 2 ('c'), 4 ('e').

💡 Key Insight

We're checking many words against the same string s. The naive approach — check each word one at a time — works, but rescans s from the start for every word. The smart approach processes every character of s exactly once, advancing all waiting words simultaneously each time a character is seen.

✓ Subsequence: "ace" in "abcde"

s:  a b c d e
    ^   ^   ^
w:  a   c   e
Picked idx 0, 2, 4 → True

✗ Not Subsequence: "aec" in "abcde"

s:  a b c d e
Found 'a' at 0, 'e' at 4
'c' must come after 'e' → False

The challenge: words can have up to 5000 entries, each up to length 50. s can be up to length 50000. Checking each word by rescanning s gives O(|s| × n) — up to 250 million operations. We need something smarter.

Edge cases to consider: a word longer than s can never match; a word containing a character absent from s fails instantly; duplicate words in words each count separately — even if they're the same string.

2 The Algorithm: Bucket-by-First-Char (Waiting List) ▾

Think of each word as a pending request sitting in a queue, waiting for its next required character. We group all words by the character they're currently waiting for, then scan s once. When we encounter character c, we resolve all words waiting for c in one sweep — advancing each one to wait for its next character, or completing it if it's done.

1 Build waiting buckets: Create a dictionary with keys 'a' to 'z'. For each word w, store it as a pointer tuple (w, 1) in waiting[w[0]]. The index 1 means "next character needed is w[1] (after we match w[0])."
2 Scan s once: For each character c in s, collect all entries from waiting[c]. Immediately clear that bucket — we're about to re-enroll promoted words.
3 Advance or complete each word: For each (word, idx) retrieved from the bucket: if idx == len(word), the word is fully matched — increment the result counter. Otherwise, re-enroll as (word, idx+1) in waiting[word[idx]].
4 Return count: After exhausting s, return the total number of words that completed. Words still sitting in buckets never found all their characters — they don't count.
🔑 Why This Is Brilliant: The Paradigm Shift

Naive: "for each word, scan s" — O(n × |s|).
Bucket: "for each char in s, advance all words waiting on it" — O(|s| + Σ|wᵢ|).

We've flipped the loop order. Each character of s does O(words waiting on it) work. Each word's pointer advances at most len(word) times total. The cost of s no longer scales with n. This is analogous to event-driven simulation: each word is an agent waiting for its next event.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At any point during the scan of s[0..i-1], a word w stored as (w, j) in some bucket means: "the first j characters of w have already been matched using characters from s[0..i-1], and w is now waiting for w[j] to appear next in s."

Why the order guarantee holds: Each promotion from bucket w[j-1] to bucket w[j] happens at some position i in s where s[i] == w[j-1]. Because we process s strictly left to right and promotions only happen in response to the current s[i], the sequence of indices in s at which a word's characters are matched is strictly increasing. That's exactly the definition of a subsequence.

Why a word that completes is guaranteed to match: When idx == len(word), all len(word) characters have been matched at strictly increasing indices in s. This is a valid subsequence by construction — no cheating possible.

Why a word that doesn't complete cannot be a subsequence: If s is exhausted and a word is still in a bucket, there wasn't a character in s to trigger its next promotion. Since we process every character of s, this means that character simply doesn't appear in the remaining s — so no valid match exists.

⚠️ Critical Implementation Detail: Clear Before Re-Insert

When processing waiting[c], you must collect all entries and clear the bucket immediately before re-inserting advanced entries. If you don't, a word whose next character happens to also be c would be re-inserted into the same bucket and then picked up again in the same iteration — matching the same position in s twice. This breaks the strictly-increasing index invariant.

💡 Analogy: The Waiting Room

Imagine a waiting room with 26 doors labeled a–z. Each word enters through its first letter's door. A caller walks through all doors in order (scanning s). When the caller shouts 'c', everyone in the 'c' room steps forward — those done leave with a ticket (count++); the rest immediately re-queue at their next door. The caller shouts each letter only once per pass. Simple, parallel, and O(|s|) total announcements.

4 The Code (Annotated) ▾

Solution 1 — Bucket by First Char (Optimal, Interview Pick)

from collections import defaultdict
def numMatchingSubseq(s, words):
waiting = defaultdict(list) ← 26 lists; each keyed by the char a word needs next
for w in words: ← enroll every word at index 1 (waiting for w[0])
waiting[w[0]].append((w, 1)) ← index=1 means "w[0] will be consumed when w[0] fires"
count = 0
for c in s: ← single pass over s; O(|s|)
advance = waiting[c] ← all words currently needing char c
waiting[c] = [] ← CRITICAL: clear before re-inserting to avoid double-processing
for w, i in advance: ← process each word that was waiting for c
if i == len(w): ← i past last index? word is fully matched
count += 1 ← this word is a valid subsequence of s
else: ← more characters to match; advance pointer
waiting[w[i]].append((w, i+1)) ← re-enroll waiting for the next needed char
return count
⚠️ Why store index=1, not index=0, during enrollment?

When we put word w into waiting[w[0]], we store index 1 (not 0) because the stored index represents "the next character to match after the bucket character fires." When the bucket for w[0] triggers, we get back (w, 1): that means w[0] is already consumed, and w[1] is what we need next. If we stored 0, we'd re-check w[0] unnecessarily on every trigger.

Solution 2 — Brute Force Two-Pointer per Word (O(n × |s|))

def numMatchingSubseq_brute(s, words):
def is_subseq(w, s): ← O(|s|) per word
i = 0 ← pointer into w
for c in s: ← scan all of s every time
if i < len(w) and c == w[i]: i += 1
return i == len(w) ← True if all of w was consumed
return sum(is_subseq(w, s) for w in words)
💡 Complexity Summary

Bucket Approach: Time O(|s| + Σ|wᵢ|), Space O(Σ|wᵢ|) — one linear pass over s
Brute Force: Time O(n × |s|), Space O(1) extra — simpler but slow for large inputs
Binary Search Variant: Precompute next_char[i][c] arrays; each word is O(|w| × log|s|). Best when s is huge and reused many times.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Spot the Bottleneck

The naive approach is O(n × |s|). Ask: "Are we doing redundant work?" Yes — we rescan s from the beginning for every word. The signal: many queries against the same large structure. This is a classic preprocessing / amortization opportunity.

Step 2: Flip the Loop Order

Instead of "for each word, scan s", think "for each character in s, which words does it unlock?" When you see a 'c' in s, all words whose next needed character is 'c' can advance simultaneously. This collapses n independent passes into one.

Step 3: Design the Bucket Structure

Use a hash map of lists (one list per letter). Each list holds (word, next_idx) tuples. Scan s once; for each character, drain its bucket. Words that finish count; others re-enroll for their next character. Space is O(total word length).

Step 4: Justify the Complexity

Each character of s triggers O(words waiting on it) work. Each word's pointer can advance at most len(word) times total. So total work across all characters = O(Σ|wᵢ|) from promotions + O(|s|) from scanning = O(|s| + Σ|wᵢ|).

🗣️ What to Say to the Interviewer

"The naive approach checks each word against s independently: O(n × |s|).
I notice we can do better by processing s just once. The key insight:
group words by the character they're currently waiting for.
As I scan s left to right, each character c advances all words in bucket[c].
Words that complete count toward the answer; others re-queue for their next char.
Time: O(|s| + total word length). Space: O(total word length) for the buckets."
🔮 Likely Follow-up Questions
  • "What if s changes dynamically?" → Precompute next[i][c]: for position i, what's the next occurrence of c? O(|s|×26) build, O(|w|×log|s|) or O(|w|) per query
  • "Can you deduplicate duplicate words?" → Yes, group by (word, idx) with a count; each advance moves the whole group, reducing redundant work
  • "What if words aren't lowercase ASCII?" → Replace the fixed-26 assumption with a full hash map; same algorithm, same complexity
  • "Is a Trie-based solution possible?" → Yes: build a Trie of all words; walk each Trie path as you scan s. Equivalent complexity, slightly different memory layout

7 Summary & Key Takeaways ▾
Scan s exactly once — regardless of how many words there are (decoupled cost)
Bucket words by their "next needed character" so all matching words advance together
Always clear the bucket before re-inserting promoted entries — prevents double-processing
Each promotion corresponds to exactly one subsequence character match at a strictly increasing s-index
Time O(|s| + Σ|wᵢ|): a dramatic improvement over O(n × |s|) brute force when n is large

🎯 When to Recognize the Bucket-by-Char Pattern
  • Many queries against the same string → batch them with a bucket dispatch
  • "Subsequence" matching for multiple words simultaneously → one-pass bucket advance
  • Each query has a "current need" that changes incrementally → group by current need
  • Analogous patterns: event-driven simulation, Aho-Corasick, multi-pointer sliding windows
