LeetCode 535 — Design a URL shortening service

1 Understanding the Problem ▼

Design a URL shortening service like TinyURL. Given a long URL, generate a short URL. Given a short URL, return the original long URL.

Long URL: https://leetcode.com/problems/design-tinyurl
↓ encode()
Short URL: http://tinyurl.com/abc123
↓ decode()
Long URL: https://leetcode.com/problems/design-tinyurl
💡 Key Insight

This is fundamentally a bijection problem — each long URL maps to exactly one short code, and each short code maps back to exactly one long URL. We need two hash maps (or one bidirectional mapping).

Core Requirements:

  • encode(longUrl) → returns a unique short URL
  • decode(shortUrl) → returns the original long URL
  • Same long URL should return same short URL (idempotent)
2 The Algorithm: Counter-Based Encoding ▼

The simplest approach: use an auto-incrementing counter to generate unique short codes. Convert the counter to a short string using Base62.

3Otherwise, convert counter to Base62 string as code
4Store: long→short and short→long mappings
5Increment counter, return short URL

3 Why It Works: The Invariant ▼
🔐 The Invariant

Every counter value maps to a unique Base62 string. Since the counter only increases, no two URLs ever get the same code. The hash maps maintain perfect bidirectional mapping.

Why Bijection is Maintained

  • Unique forward: Each long URL gets exactly one short code (checked before assigning)
  • Unique reverse: Each counter value is used only once, so each short code maps to one URL
  • Idempotent: Re-encoding same URL returns same short code (lookup first)

Edge Cases

  • Empty URL: Works — gets code "0"
  • Same URL twice: Returns same short code (idempotent)
  • Very long URL: Short code length unaffected
  • Counter overflow: At 56B+ URLs with 6 chars — extend to 7 chars
⚠️ Real-World Considerations

Sequential IDs are guessable — attackers can enumerate URLs. Production systems use random codes with collision detection, or encode user ID + timestamp.

4 The Code (Annotated) ▼

Solution 1 — Counter-Based (Interview Pick)

class Codec:
BASE = "0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ"
PREFIX = "http://tinyurl.com/"
def __init__(self):
self.counter = 0
self.long_to_short = {} # Long URL → short code
self.short_to_long = {} # Short code → long URL
def _to_base62(self, num: int) -> str:
if num == 0: return "0"
result = []
while num:
result.append(self.BASE[num % 62])
num //= 62
return "".join(reversed(result))
def encode(self, longUrl: str) -> str:
if longUrl in self.long_to_short: # Already encoded
return self.PREFIX + self.long_to_short[longUrl]
code = self._to_base62(self.counter)
self.long_to_short[longUrl] = code
self.short_to_long[code] = longUrl
self.counter += 1
return self.PREFIX + code
def decode(self, shortUrl: str) -> str:
code = shortUrl[len(self.PREFIX):] # Extract code
return self.short_to_long[code]

Solution 2 — Random String (Production-Style)

import random
import string
class Codec:
CHARS = string.ascii_letters + string.digits
CODE_LEN = 6
PREFIX = "http://tinyurl.com/"
def __init__(self):
self.long_to_short = {}
self.short_to_long = {}
def encode(self, longUrl: str) -> str:
if longUrl in self.long_to_short:
return self.PREFIX + self.long_to_short[longUrl]
while True: # Generate until no collision
code = "".join(random.choices(self.CHARS, k=self.CODE_LEN))
if code not in self.short_to_long:
break
self.long_to_short[longUrl] = code
self.short_to_long[code] = longUrl
return self.PREFIX + code
def decode(self, shortUrl: str) -> str:
code = shortUrl[len(self.PREFIX):]
return self.short_to_long[code]
💡 Which to Use in Interview?

Counter-based for LeetCode (simple, no imports). Mention random approach for follow-up discussion on production systems.

5 🎤 Interview Strategy: How to Arrive at This ▼
Step 1: Recognize the Pattern

"Encode/decode" = bijection. Need forward and reverse mapping. Think hash maps immediately.

Step 2: Choose a Code Generation Strategy

Options: counter, random, hash. Counter is simplest for interview — explain trade-offs.

Step 3: Handle Idempotency

Same input should give same output. Check if URL already encoded before generating new code.

Step 4: Discuss Follow-ups

Scalability, collision handling, expiration, analytics. Shows system design thinking.

🗣️ What to Say to the Interviewer

"This is a bijection problem — I need to map long URLs to unique short codes and back. I'll use two hash maps: one for encode lookup, one for decode lookup. For code generation, I'll start with a simple counter converted to Base62 — this gives short codes and guarantees uniqueness. I'll check if a URL is already encoded to ensure idempotency. For production, I'd discuss random codes with collision detection to prevent enumeration attacks."
⚠️ Common Follow-up Questions
  • Scalability: Distributed counter? Sharding by hash?
  • Persistence: How to store mappings durably?
  • Expiration: TTL for short URLs?
  • Analytics: Track click counts?
