LeetCode #7

Reverse Integer

Extract digits one at a time with modulo, rebuild in reverse order — then bound-check for 32-bit overflow.
🏁 Medium Math Check Overflow O(log x) Time · O(1) Space
1
Understanding the Problem
▾

Given a signed 32-bit integer x, return its digits written in reverse order — but only if the result still fits inside a signed 32-bit integer range [-2³¹, 2³¹-1] = [-2147483648, 2147483647]. If the reversed value would overflow, return 0.

Concrete examples:

  • x = 123 → digits right-to-left: 3, 2, 1 → 321. Fits in 32-bit. Return 321.
  • x = -120 → digits of 120 reversed = 021 = 21 → with sign: -21. Return -21.
  • x = 1534236469 → reversed = 9646324351, but 9646324351 > 2147483647. Return 0.
  • x = 0 → 0.
💡 Key Insight

An integer is just a sequence of decimal digits. We can peel the last digit using x % 10 and discard it with x // 10. Each digit peeled from the right of x gets pushed onto the right of a growing rev accumulator — which is exactly reversed order. The only extra step is checking the 32-bit bound after.

Why overflow matters

In Python, integers have arbitrary precision — no automatic overflow. But the problem specifies 32-bit constraints: if the mathematical result of reversing would not fit in a signed 32-bit int, we must return 0. We simply compare against the constants -2**31 and 2**31 - 1 at the end.

⚠️ Common Misconception

Students often think "just reverse the string". That works, but: (1) It requires special sign handling; (2) Interviewers usually ask this specifically to test modulo/division fluency. Expect a follow-up: "Can you do it without converting to a string?" The modulo approach answers that directly.

2
The Algorithm: Digit-by-Digit Reversal
▾

Think of the number's digits like a stack of coins. We pop from the right of x (using % 10) and push onto the right of rev (via rev * 10 + digit). Because we reverse the order of insertion, the final rev is the mirror image of x.

1
Capture sign, work with abs: Record sign = +1 or -1. Set x = abs(x) to avoid Python's negative-modulo behavior.
2
Initialize rev = 0 — this is the accumulator that will hold the reversed result.
3
Extract last digit: digit = x % 10 gives the right-most decimal digit (always 0–9 since x is non-negative).
4
Append to rev: rev = rev * 10 + digit. Multiplying by 10 shifts existing digits left one place, making room on the right for the new digit.
5
Strip the digit: x = x // 10. Integer-division drops the least-significant digit. Repeat steps 3–5 while x > 0.
6
Reapply sign & bounds-check: rev *= sign. If -2**31 <= rev <= 2**31-1, return rev; else return 0.

Worked example: x = 1234

Iterationdigit = x % 10x //= 10rev = rev*10+digit
11234 % 10 = 4x → 1230 * 10 + 4 = 4
2123 % 10 = 3x → 124 * 10 + 3 = 43
312 % 10 = 2x → 143 * 10 + 2 = 432
41 % 10 = 1x → 0 (stop)432 * 10 + 1 = 4321
4321 is in [-2147483648, 2147483647] ✓ → return 4321

Why It Works: The Invariant
▾
🔐 The Loop Invariant

At the start of iteration k, rev holds the reversal of the last k digits already consumed from the original number, and x holds the remaining prefix. After all digits are consumed (x = 0), rev is the complete reversal.

Why rev * 10 + digit reverses correctly: Multiplying by 10 is a "left-shift" in decimal — it pushes all current digits one place to higher significance. Adding digit places it at position 10⁰ (ones place). So each digit that was originally at the right end of x ends up at progressively higher positions in rev. This is precisely the mirror-image ordering.

32-bit overflow check

The signed 32-bit range is exactly [-2,147,483,648, 2,147,483,647]. Note the asymmetry: the minimum is one larger in magnitude than the maximum. This means even if abs(x) is in range, its reversal might not be.

❌ Overflows → return 0

x = 1534236469
reversed = 9646324351
9646324351 > 2147483647

✓ Fits → return rev

x = 123
reversed = 321
321 <= 2147483647

Edge cases

  • Trailing zeros: x = 120 → reversed = 021 = 21. Trailing zeros in the original become leading zeros in the reversal, which vanish naturally since we never store them.
  • x = 0: Loop never executes; rev = 0; return 0.
  • Single digit (0–9): One iteration, rev = digit = x; always in range.
  • INT_MIN = -2147483648: abs reversed = 8463847412, which overflows → return 0.
📚 Complexity

Time: O(log₁₀ x) — one iteration per digit; a d-digit number requires d iterations, and d = ⌊log₁₀ x⌋ + 1.
Space: O(1) — three integer variables (sign, x, rev) regardless of input size.

The Code (Annotated)
▾
def reverse(x: int) -> int:
sign = 1 if x >= 0 else -1# capture ± sign; work with positive x
x = abs(x) # strip sign: Python -123 % 10 = 7, not -3!
rev = 0 # accumulator for the reversed number
while x != 0: # loop while there are digits remaining
digit = x % 10 # peel rightmost digit (always 0–9)
x //= 10 # discard that digit from x
rev = rev * 10 + digit # shift rev left ×10, slot digit in on right
rev *= sign # restore original sign
INT_MIN, INT_MAX = -2**31, 2**31 - 1
return rev if INT_MIN <= rev <= INT_MAX else 0 # bounds check
⚠️ Why work with abs(x)?

Python's % follows floor division, so -123 % 10 = 7 (not -3). If we tried to loop on negative x, the digit extraction would be wrong. Taking abs(x) upfront and restoring the sign at the end is the cleanest fix.

🛠️ Java / C++ Variant (pre-loop overflow check)

In statically-typed languages, rev * 10 itself can overflow before we even add the digit. The guard is:
if (rev > Integer.MAX_VALUE / 10 || (rev == Integer.MAX_VALUE / 10 && digit > 7)) return 0;
This prevents the multiplication overflow from happening at all.

🎤 Interview Strategy: How to Arrive at This
▾
Step 1 — Recognize the Pattern

Signals: "integer", "reverse digits", "without string conversion", "32-bit overflow". This is digit manipulation via modulo arithmetic — a fundamental math pattern.

Step 2 — Name the key operations

x % 10 = last digit. x // 10 = drop last digit. rev * 10 + digit = build from right. Say these aloud to the interviewer to show you understand the primitives.

Step 3 — Address the constraint proactively

Mention overflow before being asked. In Python: post-loop bound check. In Java/C++: pre-multiplication guard. The interviewer is looking for this awareness.

Step 4 — Handle sign cleanly

Explain why you work with abs(x) and restore sign at the end. This shows you know Python's floor-modulo behavior and aren't just hoping it works.

🗣️ What to Say

"I'll use modular arithmetic — no string conversion needed.

x % 10 gives the last digit. x // 10 drops it.
rev = rev * 10 + digit builds the reversal in O(1) per digit.
I'll take abs(x) upfront to avoid Python's floor-modulo on negatives,
and restore the sign after the loop.

Finally, I compare against [-2³¹, 2³¹-1] and return 0 if it overflows.
Time: O(log x) — one pass per digit. Space: O(1)."

Likely follow-up questions

  • "How would this differ in Java?" — Pre-multiplication overflow check: if (rev > INT_MAX/10 || ...) return 0 before each rev = rev * 10 + digit.
  • "What if you could use a string?" — Reverse str(abs(x)), parse to int, reapply sign, check bounds. O(log x) time but O(log x) space for the string.
  • "Related: Palindrome Number (LC 9)" — Same digit-peeling, but stop at the midpoint and compare halves — no need to reverse fully.

Summary & Key Takeaways
▾
✓ Core Takeaways
  • x % 10 peels the last digit; x // 10 discards it — these two operations iterate through decimal digits.
  • rev = rev * 10 + digit builds the reversal: every new digit is placed at the rightmost position, shifting prior digits left.
  • Handle the sign by working with abs(x) throughout and reapplying at the end. This avoids floor-division ambiguity in Python.
  • Python: post-loop bounds check. Java/C++: pre-multiplication check before each rev * 10.
🎯 When to Use This Pattern
  • Any "digit manipulation without string conversion" problem.
  • Palindrome Number — reverse the back half of digits and compare.
  • Happy Number — repeated digit-square sum uses the same % 10 loop.
  • Whenever you see "32-bit signed" + "overflow" — always check bounds.
