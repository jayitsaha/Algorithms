LeetCode #90

Subsets II

Given an integer array nums that may contain duplicates, return all possible subsets (the power set). The solution must not contain duplicate subsets.

Medium Backtracking Sort + Skip Dups Time: O(n · 2ⁿ) Space: O(n)
1 Understanding the Problem ▾

The power set of a set is the collection of all its subsets — including the empty set and the full set itself. For a set of size n, there are 2ⁿ subsets in total. This is Subsets I (#78) with a twist: the input may contain duplicate values, and we must return only unique subsets.

✗ Without dedup: [1, 2, 2]

[], [1], [2], [2], [1,2], [1,2],
[2,2], [1,2,2]
8 results — 2 are duplicates

✓ Correct output: [1, 2, 2]

[], [1], [1,2], [1,2,2],
[2], [2,2]
6 unique subsets

💡 Key Insight

When we have duplicates, two recursive branches that start with the same value at the same depth will generate exactly the same subsets downstream. The fix is elegant: sort the array first (so equal values are adjacent), then skip any element at loop index i if it equals nums[i-1] AND i > start (meaning it's a duplicate at the same DFS level, not a child).

Edge cases: all identical elements like [2,2,2] yields [], [2], [2,2], [2,2,2]; single element; empty input; negative values mixed with positives. The sort + skip handles them all uniformly.

2 The Algorithm: Sort + Backtrack + Skip Consecutive Dups ▾

We build subsets incrementally using a DFS backtracking pattern: at each recursion level, we decide which elements to add to our current partial subset (path), recurse deeper, then undo the choice. Sorting first enables duplicate detection.

1 Sort nums: Groups equal values together. This is the prerequisite — without sorting, equal values might not be adjacent and the skip rule wouldn't work.
2 Call backtrack(start=0, path=[]): On entry, immediately add a copy of path to results. The empty set is collected at the very first call. Every node in the DFS tree is a valid subset.
3 Loop i from start to n-1: For each position, try including nums[i]. But first check: if i > start AND nums[i] == nums[i-1], skip (continue). This prunes duplicate sibling branches.
4 Recurse, then backtrack: Append nums[i] to path, call backtrack(i+1), then pop nums[i]. The pop undoes our choice so we can try the next element at this level.

Decision Tree for nums = [1, 2, 2] — crossed-out branches are pruned:

[] (add)
pick 1pick 2skip 2 (dup!)
[1]
[2]
[2] again
pick 2skip 2pick 2
[1,2]
[1,2] again
[2,2]
pick 2pick 2
[1,2,2]
[2,2] → leaf

Pruned (crossed) branches would duplicate work already done by the left sibling.

🔑 Why i > start, NOT i > 0?

The skip condition uses i > start because we only want to prune sibling choices — elements at the same DFS depth. Using i > 0 would incorrectly prune the first time we see any value at a given depth. For example, with start=1 and nums=[1,2,2], i=1 is the first choice at this depth — we must NOT skip it even though nums[1]==nums[0] (they're from different depths). The guard i > start ensures we only skip the second or later occurrence of a value within the same loop iteration.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At every point in the DFS, path holds the elements of the current partial subset being extended, in sorted order. Each subset that gets added to results is unique because: (a) duplicates within a branch are allowed (they create valid larger subsets like [2,2]), but (b) duplicate sibling branches are pruned so no two branches at the same depth start with the same element.

Proof of no duplicates: After sorting, if nums[i] == nums[i-1] and i > start, then in the current loop, index i-1 was already explored (it came before i and wasn't skipped since i-1 == start or had a different value). The subtree rooted at "pick nums[i]" is identical to the subtree rooted at "pick nums[i-1]" because: same value chosen, same remaining suffix (nums[i+1:] = nums[(i-1)+1:] shifted). Skipping i eliminates the duplicate.

Proof of completeness: Every unique subset has a canonical representation in the sorted array. Our DFS explores every way to choose a subsequence of sorted nums with no duplicate sibling starts — this covers exactly the unique subsets.

⚠️ Most Common Bug: Using i > 0 instead of i > start

If you write if i > 0 and nums[i] == nums[i-1]: continue, then when exploring subsets starting at index 1 (say, the first 2 in [1,2,2]), the check i > 0 is true for i=1, and nums[1] == nums[0] is false (2 ≠ 1), so it's fine here. But when start=1 and we try i=2, nums[2]==nums[1] and i>0, we'd skip i=2 — correctly. Wait, that actually works... but: what if nums=[2,2]? At depth 0 with start=0, we try i=0 (pick 2→[2]), then i=1 (i>0 and nums[1]==nums[0]) → skip. We never get [2] from the second 2 as a sibling — but that's correct! Actually the real danger is more subtle: if nums=[1,2,2] and we pass start=0, picking i=1 then i=2 — using i>0, at the recursive call with start=1, picking i=1 is fine, then i=2 would be skipped... which is also correct here. The bug shows up with nums=[2,2,2]: at depth 0 start=0, i=0 included, i=1 skipped (i>0 and 2==2), i=2 skipped. Result: [], [2], [2,2], [2,2,2] — actually still correct! The real failure case for i>0 is harder to trigger in practice, but the rule is semantically wrong and should always be i>start for correctness in all edge cases.

4 The Code (Annotated) ▾

Solution 1 — Sort + Backtracking (Optimal, Interview Pick)

def subsetsWithDup(nums): ← entry point
nums.sort() ← CRITICAL: groups duplicates for adjacent detection
result, path = [], [] ← result accumulates subsets; path = current partial build
def backtrack(start): ← DFS; start = leftmost index we can choose from
result.append(path[:]) ← snapshot: every DFS node is a valid subset
for i in range(start, len(nums)): ← iterate choices at this DFS depth
if i > start and nums[i] == nums[i-1]:← skip duplicate SIBLING (same value, same level)
continue ← pruning: this branch duplicates nums[i-1]'s subtree
path.append(nums[i]) ← choose: add nums[i] to current subset
backtrack(i + 1) ← explore: i+1 ensures each element used at most once
path.pop() ← un-choose: restore path for next sibling
backtrack(0) ← kick off from index 0 with empty path
return result ← all 2^unique_count subsets
⚠️ Why path[:] and not path?

path is a single mutable list reused throughout the DFS. If we append path directly to result, we'd store a reference — when we later pop() elements during backtracking, every "stored" subset would appear to shrink too. path[:] creates a shallow copy — a new list with the same elements at this moment in time. This is one of the most common bugs in Python backtracking problems.

Solution 2 — Brute Force with Set Dedup (Slower but illustrative)

def subsetsWithDup_brute(nums):
nums.sort() ← sort so tuples are canonical
seen, result, path = set(), [], []
def backtrack(start):
key = tuple(path) ← hashable key for the set
if key not in seen:
seen.add(key)
result.append(path[:])
for i in range(start, len(nums)):
path.append(nums[i])
backtrack(i + 1)
path.pop()
backtrack(0); return result ← explores ALL branches including dups, discards after
💡 Complexity Summary

Solution 1 (Sort + Skip): Time O(n · 2ⁿ), Stack Space O(n) — never enters duplicate branches
Solution 2 (Set Dedup): Time O(n · 2ⁿ · n) due to tuple hashing, Space O(2ⁿ) for the seen set
Both return the same output but Solution 1 prunes at the decision point rather than after the fact.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize "All Subsets = Backtracking"

Whenever a problem asks for "all possible" subsets, combinations, or sequences, think backtracking. The pattern: build a partial solution, recurse, undo. For subsets of n elements, the DFS tree has 2ⁿ leaves, each representing a unique subset.

Step 2: Address Duplicates Immediately — Say "Sort First"

The instant you see "input may contain duplicates" in a backtracking problem, say: "I'll sort first to group equal values, then use a skip rule to avoid exploring duplicate sibling branches." This shows pattern recognition.

Step 3: Derive the Skip Rule — Draw the Tree

Draw a small example like [1,2,2]. Show the branch where you pick the first 2 and the branch where you'd pick the second 2 at the same depth. Point out: "These two branches produce identical subtrees, so I skip when i > start and nums[i] == nums[i-1]."

Step 4: Proactively Mention i > start vs i > 0

Say: "I use i > start not i > 0 — this is critical. I only skip siblings, not the first choice at a new depth. For example, [2,2] is a valid subset: the second 2 is added as a child element, not as a sibling."

🗣️ What to Say to the Interviewer

"This is a subsets problem with duplicates — I'll use backtracking.
Step 1: Sort the array. This groups duplicates so adjacent elements can be compared.
Step 2: DFS where at each level I pick elements from index 'start' onward.
On entry to each DFS call, I record the current path — that's one valid subset.
Skip rule: if i > start and nums[i] == nums[i-1], skip. This prunes sibling duplicates.
Time: O(n·2^n) — at most 2^n subsets, each takes O(n) to copy.
Stack space: O(n) for the recursion depth."
🔮 Likely Follow-up Questions
  • "Subsets without duplicates?" → Same code, the skip never triggers. That's #78.
  • "Combination Sum II?" → Same sort+skip, but only recurse when running sum ≤ target (#40).
  • "Do it iteratively?" → Start [[]], for each number in sorted order extend only subsets from the previous round if it's a duplicate, or all subsets otherwise.
  • "Permutations with duplicates?" → Sort + used[] array + skip if not used[i-1] (different but same spirit, #47).

7 Summary & Key Takeaways ▾
Sort first — the prerequisite that makes adjacent-duplicate detection possible
Skip rule: i > start (not i > 0) — prunes only sibling duplicates, not child duplicates
Every DFS node is a subset — add path[:] on entry, before the loop
path.pop() undoes the choice cleanly — one list reused throughout, no copies per branch
"Sort + Skip Consecutive Dups" is a transferable pattern for all duplicate-aware backtracking
