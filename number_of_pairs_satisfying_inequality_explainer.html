LeetCode #2426

Number of Pairs Satisfying Inequality

Given arrays nums1, nums2 and integer diff, count pairs (i,j) where i < j and nums1[i] − nums1[j] ≤ nums2[i] − nums2[j] + diff.

Hard Advanced Data Structures BIT + Merge Sort Time: O(n log n) Space: O(n)
1 Understanding the Problem ▾

We're given two integer arrays nums1 and nums2 of the same length n, plus an integer diff. We want to count all pairs (i, j) where i < j and the inequality nums1[i] − nums1[j] ≤ nums2[i] − nums2[j] + diff holds.

💡 Key Transformation

Rearrange the inequality: nums1[i] − nums1[j] ≤ nums2[i] − nums2[j] + diff
Move terms: (nums1[i] − nums2[i]) − (nums1[j] − nums2[j]) ≤ diff
Define A[k] = nums1[k] − nums2[k] for every k. The condition becomes:
A[i] − A[j] ≤ diff, equivalently A[i] − diff ≤ A[j].
Now for each j, we count how many previous i's satisfy: A[j] ≥ A[i] − diff.

✓ Valid Pair: A[i]=3, A[j]=2, diff=1

A[i] − A[j] = 3 − 2 = 1
1 ≤ diff (1) → Valid!

✗ Invalid: A[i]=5, A[j]=1, diff=1

A[i] − A[j] = 5 − 1 = 4
4 > diff (1) → Invalid

Edge cases: n=1 (zero pairs possible), all values identical (all pairs valid if diff≥0), large negative diff (no pairs), all differences equal (all n*(n-1)/2 pairs valid).

🔑 Why This Is Hard

A brute-force O(n²) approach checks every (i,j) pair. With n up to 10⁵, that's 10¹⁰ operations — too slow. We need O(n log n) using either a BIT (Binary Indexed Tree / Fenwick Tree) with coordinate compression, or an enhanced merge sort that counts inversions as it splits.

2 The Algorithm: BIT with Coordinate Compression ▾

After the transformation to array A[], we need: for each j, count how many indices i < j satisfy A[j] ≥ A[i] − diff, i.e., A[i] ≤ A[j] + diff. This is a classic "count elements seen so far with value ≤ threshold" — perfect for a BIT (Fenwick Tree).

1 Transform: Compute A[k] = nums1[k] − nums2[k] for all k. The problem reduces to counting pairs (i < j) where A[i] ≤ A[j] + diff.
2 Coordinate compress: BIT indices must be positive integers. Map all distinct values of A[] (plus A[j]+diff thresholds) to indices 1..m using sorted rank.
3 Sweep left to right (j = 0..n-1): For each j, query the BIT for "how many elements inserted so far have value ≤ A[j] + diff". Add the count to the answer. Then insert A[j] into the BIT.
4 Return answer: The accumulated count is the number of valid pairs.
BIT Structure — BIT[i] stores count of A values mapped to positions in this subtree
BIT[8]covers 1-8
BIT[4]covers 1-4
BIT[6]covers 5-6
BIT[2]covers 1-2
BIT[3]covers 3
BIT[5]covers 5
BIT[7]covers 7
BIT[1]covers 1
BIT[8]covers 8

Each BIT query (prefix sum up to index k) takes O(log m) time. Each update takes O(log m). With n elements, total time is O(n log n).

💡 Alternative: Merge Sort Inversion Counting

The same problem can be solved via modified merge sort. During the merge step, when we merge two halves, we count how many elements in the left half satisfy A[i] ≤ A[j] + diff for each A[j] in the right half. Since both halves are sorted, a two-pointer scan handles this in O(n) per level, giving O(n log n) overall.

3 🔬 Algorithm Deep-Dive: BIT & Merge Sort Inversion Counting ▾

Binary Indexed Tree (Fenwick Tree) — invented by Peter Fenwick in 1994. It answers prefix-sum queries and point-updates in O(log n) time and O(n) space, making it a compact alternative to a Segment Tree for cumulative frequency problems.

🔐 BIT Core Invariant
BIT[i] = frequency of all elements with compressed index in range [i − lowbit(i) + 1, i]
where lowbit(i) = i & (−i) — the lowest set bit of i
Query(k) = sum BIT[k] + BIT[k − lowbit(k)] + ... (remove lowest bit each time)
Update(i) = BIT[i] += 1, BIT[i + lowbit(i)] += 1, ... (propagate up)
🔑 Why BIT Works Here

After coordinate compression, each A[j] maps to an integer rank r. Query(rank(A[j]+diff)) gives the count of all previously-inserted values with rank ≤ rank(A[j]+diff), which is exactly the count of i < j with A[i] ≤ A[j]+diff. The tree structure lets us maintain this prefix sum dynamically with logarithmic overhead.

Merge Sort Inversion Counting — a classic Divide & Conquer technique. The insight: during merge sort, when we merge a sorted left half L and sorted right half R, for any element R[j], all elements in L that are "too large" are ALSO "too large" for all future R[k] (k>j). We use a pointer p that advances monotonically, giving O(n) work per merge level.

def merge_count(A, diff): ← divide and conquer entry
if len(A) <= 1: return 0, A ← base case
mid = len(A) // 2 ← split
lc, L = merge_count(A[:mid], diff)← recurse left
rc, R = merge_count(A[mid:], diff)← recurse right
count = lc + rc ← accumulate subproblem counts
p = 0 ← pointer into L
for rj in R: ← for each right element
while p < len(L) and L[p] <= rj + diff:← advance p while valid
p += 1 ← p valid elements in L satisfy condition
count += p ← all L[0..p-1] are valid for this rj
merged = sorted(L + R) ← merge for next level
return count, merged
💡 When to Recognize These Patterns
  • BIT: "Count elements seen so far with value ≤ X / ≥ X" — classic BIT + coordinate compression
  • Merge Sort Count: "Count cross-half pairs satisfying ordering condition" — modified merge sort
  • Both appear when: counting pairs/inversions, rank queries on streaming data
4 The Code (Annotated) ▾

Solution 1 — BIT + Coordinate Compression (Interview Pick, O(n log n))

def numberOfPairs(nums1, nums2, diff): ← main function
A = [a - b for a, b in zip(nums1, nums2)] ← transform: A[k] = nums1[k] - nums2[k]
# Coordinate compression
vals = sorted(set(A + [a + diff for a in A])) ← all values that appear in queries/updates
rank = {v: i+1 for i, v in enumerate(vals)} ← 1-indexed rank map (BIT needs ≥ 1)
m = len(vals) ← BIT size
bit = [0] * (m + 1) ← BIT array (1-indexed)
def update(i): ← point-update: add 1 at index i
while i <= m: ← propagate up via lowest set bit
bit[i] += 1
i += i & (-i) ← i += lowbit(i) moves to parent
def query(i): ← prefix sum: count elements with rank ≤ i
s = 0
while i > 0: ← traverse down via lowest set bit
s += bit[i]
i -= i & (-i) ← i -= lowbit(i) removes lowest bit
return s
ans = 0
for a in A: ← sweep left to right
# query: count i<j where A[i] <= a + diff
threshold_rank = rank[a + diff] ← upper bound rank for this j's threshold
ans += query(threshold_rank) ← count previously-seen values ≤ threshold
update(rank[a]) ← insert current A[j] into BIT
return ans
⚠️ Why include a+diff in coordinate compression?

When we query for "count of values ≤ a+diff", we need a+diff to have a valid rank in our mapping. If we only compress values of A[], a+diff might not be in the map. We include both A[k] and A[k]+diff for all k in the compression set to guarantee all query thresholds are mapped.

⚠️ Why query BEFORE update (not after)?

For each j, we count pairs (i, j) where i < j. If we insert A[j] first then query, we'd also count the pair (j, j), which is invalid (requires i ≠ j). Querying first ensures only the strictly-previous elements are counted.

Solution 2 — Merge Sort Inversion Counting (O(n log n) alternative)

def numberOfPairs(nums1, nums2, diff):
A = [a - b for a, b in zip(nums1, nums2)]
def merge_sort(arr):
if len(arr) <= 1: return 0, arr ← base: single element, 0 pairs
mid = len(arr) // 2
lc, left = merge_sort(arr[:mid]) ← sort left half, count its pairs
rc, right = merge_sort(arr[mid:]) ← sort right half, count its pairs
count = lc + rc ← pairs entirely in one half
p = 0 ← pointer into left half
for rj in right: ← for each right element (index j)
while p < len(left) and left[p] <= rj + diff:← advance while L[p] ≤ rj+diff
p += 1
count += p ← all left[0..p-1] satisfy condition
merged = [] ← standard merge for sorted order
i = j = 0
while i < len(left) and j < len(right):
if left[i] <= right[j]: merged.append(left[i]); i += 1
else: merged.append(right[j]); j += 1
merged += left[i:] + right[j:]
return count, merged
ans, _ = merge_sort(A)
return ans

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Algebraically Simplify

"The condition involves four arrays combined. Let me simplify: rearranging the inequality gives (nums1[i]-nums2[i]) ≤ (nums1[j]-nums2[j]) + diff. If I define A[k] = nums1[k]-nums2[k], the problem becomes: count pairs i<j where A[i] ≤ A[j]+diff."

Step 2: Recognize the Pattern

"This is 'count previous elements ≤ threshold'. That's a prefix-count query — BIT. Or I can think of it as counting modified inversions — Merge Sort. With n up to 10⁵, I need O(n log n). BIT with coordinate compression is my first choice."

Step 3: Handle Coordinate Compression

"BIT needs positive integer indices, but A[k] can be negative. I collect all unique values (A[k] and A[k]+diff for all k), sort them, and assign 1-based ranks. Now I work in rank-space. The relative ordering is preserved, so prefix queries still give correct counts."

Step 4: Propose the Sweep

"I sweep j from 0 to n-1. For each j: query BIT up to rank(A[j]+diff) to count valid i's; add to answer; then insert A[j] into BIT. The order — query first, then insert — ensures i < j."

🗣️ What to Say to the Interviewer

"First let me simplify the condition. Defining A[k] = nums1[k] - nums2[k], it reduces to counting pairs i<j where A[i] - A[j] ≤ diff. Brute force is O(n²) — too slow. I notice that for each j, I need 'how many previous values A[i] are ≤ A[j]+diff'. That's a prefix count on a dynamically-growing set — a Binary Indexed Tree handles this in O(log n) per operation. Since A values can be negative, I'll coordinate-compress them to 1-based integer ranks. I need to include both A[k] and A[k]+diff in the compression to cover all query thresholds. Sweep j=0..n-1: query BIT(rank(A[j]+diff)), add to answer, then insert rank(A[j]). Total: O(n log n) time, O(n) space."
⚠️ Common Mistakes to Avoid
  • Forgetting to include A[k]+diff in coordinate compression (causes KeyError on query)
  • Updating BIT before querying (counts (j,j) pairs, off by n)
  • Using 0-indexed BIT (lowbit(0) = 0, causing infinite loop)
  • Not handling duplicate values in coordinate compression (use a set)

7 Summary & Key Takeaways ▾
Algebraic simplification (A[k] = nums1[k]−nums2[k]) collapses a 4-array problem into a 1D counting problem
BIT (Fenwick Tree) gives O(log n) prefix-count queries and point-updates — ideal for "count previous elements ≤ threshold"
Coordinate compression maps arbitrary (possibly negative) values to 1-based ranks, satisfying BIT's index constraints
Always include both A[k] and A[k]+diff in the compression — queries need ranks for threshold values, not just element values
Query before insert to enforce the i < j constraint — never count self-pairs
Merge Sort inversion counting is an elegant alternative requiring no coordinate compression, but slightly more complex code
🎯 When to Use BIT + Coordinate Compression
  • Counting pairs/elements satisfying an ordering condition dynamically
  • Rank queries on a stream of integers (online algorithm)
  • Problems reducible to "count how many previous values fall in range [lo, hi]"
  • Values can be large/negative — compress to small positive integers first
