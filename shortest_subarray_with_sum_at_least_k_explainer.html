LeetCode #862

Shortest Subarray with Sum at Least K

Given an integer array nums and integer k, return the length of the shortest non-empty subarray whose sum is at least k. Return -1 if no such subarray exists.

Hard Monotonic Queue Prefix Sum + Deque Time: O(n) Space: O(n)
1 Understanding the Problem ▾

We need the shortest contiguous subarray whose elements sum to at least k. The critical difficulty is that nums can contain negative numbers. This single fact makes the problem jump from Medium to Hard and forces a completely different algorithmic approach.

⚡ Why Negative Numbers Make This Hard

Without negatives, a classic two-pointer sliding window works in O(n): expand right until sum ≥ k, then shrink left. With negatives, adding more elements can decrease the sum, and removing elements from the left can increase it. The sliding window invariant — "a longer window always has a larger sum" — breaks entirely.

💡 Key Insight: Prefix Sums Reframe Everything

Define prefix[j] = nums[0] + ... + nums[j−1] with prefix[0] = 0. The sum of subarray nums[i..j−1] equals prefix[j] − prefix[i]. So the problem becomes: find the minimum j−i with i < j and prefix[j] − prefix[i] ≥ k. This is a pair-search problem on the prefix array — and it admits a clean O(n) solution with a monotonic deque.

Edge Cases: all negatives (no valid subarray → return −1), single element ≥ k (answer = 1), k ≤ 0 (any single element works → answer = 1), large arrays where O(n²) brute force times out (n up to 10⁵).

2 The Algorithm: Prefix Sum + Monotonic Deque ▾

After computing the prefix sum array (length n+1), we scan through it with index j (the "right boundary"). We maintain a monotone increasing deque of indices into the prefix array — increasing in both index value and prefix value. The front of the deque is always the smallest prefix value seen so far among valid left-boundary candidates.

1 Build prefix array: prefix[0] = 0, prefix[j] = prefix[j−1] + nums[j−1]. Length is n+1.
2 For each j from 0 to n: j is the right boundary of a subarray ending at nums[j−1]. Check if any stored left boundary forms a valid pair.
3 Pop from front (valid pair found): While prefix[j] − prefix[dq.front()] ≥ k, record length j − dq.front(), update answer, and pop the front. We pop because future j' > j using the same front gives a longer (worse) subarray.
4 Pop from back (maintain monotonicity): While prefix[j] ≤ prefix[dq.back()], pop the back. The back is dominated by j: it has a larger or equal prefix value AND an earlier index — any future use of back as a left boundary is strictly beaten by using j instead.
5 Push j: Add j to the back of the deque. It's now a valid left-boundary candidate for future right boundaries.
6 Return the minimum length found, or −1 if no valid pair was ever recorded.
🔑 The Two Deque Operations and Their Purposes

Front pop = "I found the best subarray for this left boundary; discard it."
Back pop = "This old left boundary can never be optimal; discard it before it's ever used."

The deque acts as a sliding-window minimum structure, but applied to the problem of finding minimal-length valid pairs instead of a fixed-size range.

3 Why It Works: The Invariant ▾
🔐 The Deque Invariant

At all times, the indices in the deque are in strictly increasing order of both their index value AND their corresponding prefix value. That is, if deque = [i₁, i₂, ..., iₘ] then i₁ < i₂ < ... < iₘ AND prefix[i₁] < prefix[i₂] < ... < prefix[iₘ].

Why the back-pop is lossless: Suppose we have index m in the deque and we're about to push j with prefix[j] ≤ prefix[m]. For any future right boundary r > j: if prefix[r] − prefix[m] ≥ k, then prefix[r] − prefix[j] ≥ k too (since prefix[j] ≤ prefix[m]). But the length would be r−j < r−m (since j > m), so j gives a shorter subarray. Index m is strictly dominated — removing it loses no optimal answer.

Why the front-pop is lossless: When we find prefix[j] − prefix[front] ≥ k, we record j−front. For any future r > j: r−front > j−front, so front can only produce longer (worse) answers going forward. It's safe to permanently discard front.

Completeness: Every optimal pair (i, j) is examined. Index i enters the deque when j=i is processed and stays until either (a) a later j finds a valid answer using i (front pop) or (b) a later index dominates i (back pop). In case (b), i was not optimal — there exists a better pair using the dominating index.

⚠️ Why We Check the Front BEFORE the Back

Step 3 (front pop) must come before step 4 (back pop). If we pushed j first and then checked the front, we'd potentially miss valid pairs: the front might be valid with the current j, but we'd corrupt the deque before checking. Always: check front → prune back → push.

Amortised O(n) analysis: Each of the n+1 prefix indices is pushed at most once and popped at most once (total: at most once from the front, at most once from the back). Total operations ≤ 2(n+1) = O(n). The entire algorithm is O(n) time and O(n) space for the prefix array and deque.

4 The Code (Annotated) ▾

Solution 1 — Prefix Sum + Monotonic Deque (Optimal — Interview Pick)

from collections import deque
def shortestSubarray(nums, k): ← Works with negatives too
n = len(nums)
prefix = [0] * (n + 1) ← prefix[0]=0; length n+1
for i in range(n): ← Build prefix sums in O(n)
prefix[i + 1] = prefix[i] + nums[i]
dq = deque() ← Monotone increasing deque of indices
ans = float('inf') ← Best (shortest) length found so far
for j in range(n + 1): ← j is right boundary in prefix array
while dq and prefix[j] - prefix[dq[0]] >= k: ← Front gives valid pair!
ans = min(ans, j - dq.popleft()) ← Record; pop (future j's only worse)
while dq and prefix[j] <= prefix[dq[-1]]: ← j dominates back; prune
dq.pop() ← Remove dominated index
dq.append(j) ← j is candidate left boundary
return ans if ans != float('inf') else -1
⚠️ Common Mistake: Swapping the While-Loop Order

The front-check while loop (valid pair detection) must come BEFORE the back-prune while loop. Reversing them means j could get pushed before checking if it forms a valid pair with existing front — corrupting results. This is the most common implementation bug.

Solution 2 — Brute Force (O(n²), for comparison)

def shortestSubarray_brute(nums, k):
n, ans = len(nums), float('inf')
for i in range(n): ← Try each starting position
curr = 0
for j in range(i, n): ← Extend to each ending position
curr += nums[j]
if curr >= k: ← Valid subarray found
ans = min(ans, j - i + 1)
break ← Smallest window from i is j−i+1
return ans if ans != float('inf') else -1
💡 Complexity Summary

Brute Force: Time O(n²), Space O(1). Correct but TLEs for n > ~10⁴.
Prefix + Monotonic Deque: Time O(n), Space O(n). Each index pushed and popped at most once.
O(n) is optimal — we must examine every element at least once.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Negative Number Blocker

"My first instinct is sliding window — it works for minimum-length subarray sum problems. But the array has negatives. Adding elements can reduce the sum, so expanding the window doesn't monotonically increase it. Sliding window doesn't apply."

Step 2: Reframe with Prefix Sums

"Subarray sum from index i to j−1 equals prefix[j] − prefix[i]. I need minimum j−i with i < j and prefix[j] − prefix[i] ≥ k. This is a pair-search on the prefix array — now I can think about efficient data structures."

Step 3: Connect to Monotone Queue Pattern

"For each j, I want the largest i < j such that prefix[i] ≤ prefix[j] − k. If I maintain a deque with indices in increasing prefix-value order, the front is always the best candidate. I check it, pop it if valid (future j only gives worse), and prune the back if j dominates it."

Step 4: State the Complexity Argument

"Each of the n+1 prefix indices is pushed once and popped at most once from either end. Total: O(n) operations. Space is O(n) for the prefix array and deque — we can't do better than O(n) time since we must read every input."

🗣️ What to Say to the Interviewer

"Negatives break sliding window — I'll use prefix sums.
prefix[j] − prefix[i] ≥ k is the condition for nums[i..j−1] to be valid.
I want the smallest j−i. So for each j, find the largest valid i.
Maintain a monotone increasing deque of prefix indices.
Front pop when valid: record length, discard (future j only longer).
Back pop when dominated: prefix[j] ≤ prefix[back] means back is useless.
Push j. Total O(n) time, O(n) space."
🔮 Likely Follow-up Questions
  • "What if all numbers are non-negative?" → Use a two-pointer sliding window — O(n), simpler
  • "Can you track the actual subarray, not just its length?" → Store (start, length) in ans and track which front index gave the minimum
  • "What if we want strictly greater than k?" → Change ≥ k to > k in the while condition
  • "Could you use a segment tree instead?" → Yes, O(n log n), but the deque is strictly better

7 Summary & Key Takeaways ▾
Negative numbers break sliding window — the first thing to check in any subarray-sum problem
Prefix sums transform "subarray sum" into a pair-search query: prefix[j] − prefix[i] ≥ k
Monotone increasing deque gives O(1) amortised access to the best left boundary candidate
Pop front when valid — any future j is farther away from the same i, so length only grows
Pop back when j dominates — same logic flipped: j is a strictly better left boundary than back
Every index enters and exits the deque at most once → O(n) total
