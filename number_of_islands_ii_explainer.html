LeetCode #305

Number of Islands II

Given an m×n grid of water, process a list of land-adding operations and return the number of islands after each one.

Hard Graph Online Union-Find Time: O(k·α(mn)) Space: O(mn)
1 Understanding the Problem ▾

Imagine an m × n ocean. One by one, land masses are dropped into it. After each drop, report how many distinct islands exist. An island is a group of land cells connected horizontally or vertically — diagonal doesn't count.

💡 Key Insight

Each land addition either creates a new island (+1) or merges adjacent islands (−k for merging k distinct neighbour components). Land is never removed — operations are purely additive. This monotone-union property makes it a perfect fit for Online Union-Find.

After adding (0,0) and (1,1)

[L, 0, 0]

2 islands — diagonal gaps separate them

After adding (0,1) — the bridge

[L, L, 0]
[0, L, 0]
[0, 0, 0]
1 island — (0,1) bridges both

Edge Cases: Duplicate positions (second occurrence is a no-op — island count doesn't change), single-cell islands, all positions the same cell, grid filling to form one giant island.

2 The Algorithm: Online Union-Find ▾

The word "online" is critical — we must answer after each operation, not after processing all of them. Rerunning BFS/DFS from scratch costs O(mn) per query and O(k·mn) total — too slow. Union-Find achieves near-O(1) per operation with path compression + union by rank.

1 Flatten coordinates: Map each 2D cell (r, c) to a 1D index r × n + c. Store parent and rank as sparse dicts — only land cells get entries.
2 Add land: If position is already land (already in parent), skip — append current count and continue. Otherwise, set parent[id] = id, rank[id] = 0, and increment count += 1 (new isolated island).
3 Merge neighbours: Check all 4 adjacent cells. For each that is already land, call union(current, neighbour). If their roots differ (truly separate components), the union succeeds → count -= 1.
4 Record: Append current count to the result list. Repeat for every operation.

Why this count invariant works: At all times, count equals the number of distinct connected land components. New land → +1 always. Successful union of two different components → −1 always. These two rules together keep count exact.

3 Why It Works: The Invariant ▾
🔐 The Core Invariant

At all times: count = number of distinct connected land components. Adding a new land cell increases this by 1 (it starts isolated). Each successful union with a distinct neighbour decreases this by 1 (two components become one). The guard find(a) != find(b) ensures we only decrement on real mergers.

Path compression is safe: When find(x) recursively flattens the path to root, it changes which nodes store the root pointer — but the root itself doesn't change. All nodes that were in the same component before still have the same root after. The grouping is preserved.

Union by rank keeps trees shallow: Without rank, a long chain of unions could create a linear tree of depth O(n), making find O(n). With union by rank (always attach the shallower tree under the deeper), tree height stays O(log n). Combined with path compression, amortized cost drops to O(α(n)) per operation.

⚠️ The Duplicate-Add Bug

If the same position appears twice in the operations list and you forget to check if idx in parent, you'll reinitialize that cell's parent to itself, breaking any unions that already incorporated it. Always skip re-adding an existing land cell — just append the current count unchanged.

Why BFS/DFS doesn't work online efficiently: DFS from scratch after each add must re-traverse all existing land cells in the worst case. With k operations on an mn grid, worst-case is O(k·mn). Union-Find reduces this to O(k·α(mn)) — for k=10,000 and mn=1,000,000, that's roughly 1010 vs 104 operations.

4 The Code (Annotated) ▾
def numIslands2(m, n, positions):
parent = {} ← sparse dict; keys are flat indices of LAND cells only
rank = {} ← tree depth hint for keeping unions shallow
count = 0 ← running island count — the answer after each op
result = [] ← collects count after each operation
def find(x): ← returns the canonical root of x's component
if parent[x] != x:
parent[x] = find(parent[x]) ← path compression: flatten path to root
return parent[x]
def union(a, b): ← merge two components; returns True if they were different
ra, rb = find(a), find(b)
if ra == rb: return False ← same root → already connected
if rank[ra] < rank[rb]: ra, rb = rb, ra ← always attach smaller tree under larger
parent[rb] = ra ← rb's root is now a child of ra's root
if rank[ra] == rank[rb]: rank[ra] += 1 ← equal ranks: height increases by 1
return True ← merger happened — caller should decrement count
for r, c in positions:
idx = r * n + c ← 2D → 1D flat index
if idx in parent: ← duplicate position: already land, no change
result.append(count)
continue
parent[idx] = idx ← new cell is its own component (self-loop)
rank[idx] = 0 ← starts as a single-node tree of height 0
count += 1 ← always: new land = new isolated island
for dr, dc in [(-1,0),(1,0),(0,-1),(0,1)]:
nr, nc = r + dr, c + dc
nidx = nr * n + nc
if 0 <= nr < m and 0 <= nc < n and nidx in parent:
if union(idx, nidx): ← only decrement if roots were different
count -= 1
result.append(count) ← snapshot after this operation
return result
⚠️ Sparse dict vs full array

You can use parent = [-1] * (m*n) and check parent[idx] != -1 for "is land". The sparse dict approach here is cleaner for sparse inputs (few operations relative to grid size) and makes the "is this cell land?" check implicit in if idx in parent.

💡 Inverse Ackermann — What O(α(n)) Really Means

Path compression alone gives O(log n) amortized. Union by rank alone gives O(log n) worst-case. Together they achieve O(α(n)) per operation, where α is the inverse Ackermann function. For any realistic n (say, n = 1080, the number of atoms in the observable universe), α(n) ≤ 4. For all practical purposes this is O(1).

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Spot the "Online" Signal

The phrase "return the number after each operation" means you can't defer computation. Propose DFS/BFS first as O(k·mn) and note it's too slow for the given constraints. This frames the problem correctly.

Step 2: Identify the Monotone-Union Structure

Ask yourself: "Are operations only additions?" Yes — land never turns back to water. "Do I need to track connected components dynamically?" Yes. These two signals together point directly to Union-Find. Union-Find is the go-to data structure for online dynamic connectivity when only merges (not splits) happen.

Step 3: Define the Count Update Rule

The insight is simple: +1 for every new land cell, −1 for every distinct-component merge with a neighbour. Articulate this invariant explicitly — it demonstrates you understand why the algorithm works, not just how to code it.

Step 4: Code Modularly

Write three clearly separated pieces: find(x) with path compression, union(a, b) with rank, and the main loop. This structure is readable, testable, and mirrors the conceptual decomposition — exactly what interviewers want to see.

🗣️ What to Say to the Interviewer

"My first thought is BFS or DFS after each add — but that's O(mn) per query and O(k·mn) total, which is too slow. Since land is only added (never removed), each operation either creates a new component or merges existing ones. This is the classic online Union-Find problem pattern. I'll use a parent dict (sparse, only for land cells) and a rank dict. For each operation: mark the cell as land, increment count by 1, then check 4 neighbours. For each land neighbour, call union. If the union actually merges two different components (find(a) != find(b)), decrement count. With path compression and union by rank, each operation is O(α(mn)) amortized — essentially O(1). Total complexity: O(k·α(mn)) time, O(mn) space. I also need to handle duplicates: if a position appears twice in the list, skip the second occurrence and just append the current count."
🔎 Follow-up Questions to Expect
  • "What if we also had remove operations?" → Union-Find can't undo splits. Use offline processing (reverse the removes, process deletions as additions in reverse) or a Link-Cut tree for fully dynamic connectivity
  • "How would you handle this with multiple concurrent readers?" → parent and rank arrays would need atomic operations or fine-grained locking per component
  • "Can you reduce space below O(mn)?" → Only if positions are sparse; the sparse dict already does this — O(k) instead of O(mn) when k << mn
  • "Prove path compression doesn't corrupt the union structure" → Path compression only changes which node stores the root pointer; the root identity is unchanged, so all components remain correct

7 Summary & Key Takeaways ▾
Online queries after each addition → Union-Find, not BFS/DFS-from-scratch
Island count rule: +1 for new land, −1 for each distinct-component neighbour union
Always guard union with find(a) != find(b) before decrementing — same-component call must be a no-op
Duplicate positions must be skipped — reinitializing an existing land cell corrupts union structure
Path compression + union by rank achieves O(α(n)) amortized — effectively O(1) per operation
Sparse dict (parent only for land cells) handles giant sparse grids elegantly without allocating full mn array

🎯 When to Recognize the Online Union-Find Pattern
  • "Add edges/nodes online, count connected components" → Union-Find with running count
  • "Report answer after each modification" → maintain invariant incrementally, don't recompute
  • Operations are only additions (never removals) → UF is valid (no undo/split needed)
  • Grid with horizontal/vertical adjacency + growing regions → flatten 2D to 1D indices
  • "When does the graph become fully connected?" → UF, stop when count == 1
