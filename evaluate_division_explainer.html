LeetCode #399

Evaluate Division

Given equations a/b = k and a list of queries, compute each division result — or −1.0 if it cannot be determined.

Medium Graph Union-Find with Weights BFS Weighted Time: O((E+Q)·α(V)) Space: O(V+E)
1 Understanding the Problem ▾

You are given a list of equations — each one encodes a ratio between two variables. For example, a / b = 2.0 means "a is twice b." From these ratios, answer each query: "what is a / c?" If the two variables are connected (directly or indirectly through a chain of equations), multiply the ratios along the path. If they are disconnected or unknown, return −1.0.

💡 Key Insight: Division Telescopes Along a Path

Each equation A / B = k gives us two facts: A→B with weight k, and B→A with weight 1/k. To answer A / C, find a path A → B → C and multiply the weights: (A/B) × (B/C) = A/C. The intermediate variable B cancels out — this is exactly how fractions work!

Weights: ×2.0 then ×3.0
a/c = 2.0 × 3.0 = 6.0
c/a = (1/3) × (1/2) = 0.1667

Edge cases: Query a/a where a is a known variable → 1.0. Query a/a where a is unknown → −1.0. Variables in the same equation are always connected (they form one component). Two variables with no common equation chain → −1.0.

2 The Algorithm: Two Approaches ▾

Approach A — BFS on Weighted Graph (easiest to understand and implement):

1 Build adjacency list: For each equation A/B = k, add graph[A][B] = k and graph[B][A] = 1/k. This gives us the full graph with both directions.
2 For each query (src, dst): If src or dst not in graph, return −1.0. If src == dst, return 1.0. Otherwise start BFS from src.
3 BFS with accumulated product: Queue stores (node, cumulative_ratio). Each step multiplies by the edge weight. When we reach dst, return the accumulated ratio.
4 If BFS exhausts without finding dst: The nodes are in different components — return −1.0.
🔑 Approach B — Weighted Union-Find (Optimal for many queries)

Each node stores weight[x] = x / root(x). Union merges two components, computing the link weight from the given ratio. Find compresses the path while multiplying weights. Query checks if src and dst share a root; if so, returns weight[src] / weight[dst]. This is O(α(V)) per query — effectively O(1). BFS is O(V+E) per query. For large Q, Union-Find wins decisively.

3 Why It Works: The Invariant ▾
🔐 The BFS Invariant

Every entry in the BFS queue carries a cumulative ratio: the exact value of src / current_node. When we dequeue (node, ratio) and step to neighbor nbr with edge weight w, the new ratio is ratio × w = src / node × node / nbr = src / nbr. The invariant is maintained at every step.

Why multiplication gives the answer: Ratios compose multiplicatively because of fraction arithmetic:

a/b = 2, b/c = 3
a/c = (a/b) × (b/c) = 2 × 3 = 6 ← b cancels
a/d = (a/b) × (b/c) × (c/d) ← all intermediates cancel
🔐 Weighted Union-Find Invariant

At all times: weight[x] = x / root(x). Path compression maintains this by multiplying weights as we flatten the tree. When we union two components given A/B = k, we set weight[rootA] = k × weight[B] / weight[A] — this is derived from: rootA/rootB = (A/rootA)⁻¹ × (A/B) × (B/rootB) = k × ratioB / ratioA.

⚠️ Path Compression Danger in Weighted UF

Standard UF compresses by pointing each node directly to root. Here you must also accumulate the weight as you traverse. Specifically, in find(x): save orig = parent[x], recurse to compress, then weight[x] *= weight[orig]. Doing weight[x] *= weight[parent[x]] after the recursive call would read the already-updated parent — a subtle bug that corrupts all answers silently.

4 The Code (Annotated) ▾

Solution 1 — BFS on Weighted Graph (clean, interview-friendly)

from collections import defaultdict, deque
def calcEquation(equations, values, queries):
graph = defaultdict(dict) ← adj list: graph[u][v] = weight u/v
for (a, b), k in zip(equations, values):
graph[a][b] = k ← forward: a/b = k
graph[b][a] = 1.0 / k ← reverse: b/a = 1/k
def bfs(src, dst):
if src not in graph or dst not in graph:
return -1.0 ← unknown variable — no info
if src == dst: return 1.0 ← same variable: ratio is always 1
visited = {src}
queue = deque([(src, 1.0)]) ← (node, cumulative ratio src/node)
while queue:
node, ratio = queue.popleft()
for nbr, w in graph[node].items():
if nbr == dst:
return ratio * w ← found! ratio = src/node, w = node/dst → src/dst
if nbr not in visited:
visited.add(nbr)
queue.append((nbr, ratio * w))
return -1.0 ← BFS exhausted, no path found
return [bfs(s, d) for s, d in queries]

Solution 2 — Weighted Union-Find (Interview Pick, optimal)

def calcEquation(equations, values, queries):
parent, weight = {}, {} ← weight[x] = x / parent[x] always
def find(x): ← returns root; compresses path with weight chain
if parent[x] != x:
orig = parent[x] ← save BEFORE recursion changes parent[x]
parent[x] = find(parent[x]) ← path compression: now parent[x] is root
weight[x] *= weight[orig] ← x/root = (x/orig) × (orig/root)
return parent[x]
def union(a, b, k): ← encode a/b = k into the structure
for v in (a, b): ← initialize new nodes
if v not in parent: parent[v] = v; weight[v] = 1.0
ra, rb = find(a), find(b)
if ra == rb: return ← already same component
parent[ra] = rb
weight[ra] = k * weight[b] / weight[a] ← rootA/rootB = k·(B/rootB)/(A/rootA)
for (a, b), k in zip(equations, values): union(a, b, k)
res = []
for src, dst in queries:
if src not in parent or dst not in parent: res.append(-1.0)
elif find(src) != find(dst): res.append(-1.0)
else: res.append(weight[src] / weight[dst])← (src/root)/(dst/root) = src/dst ✓
return res
💡 Complexity Comparison

BFS per query: Build O(E), Per query O(V+E) → Total O(E + Q·(V+E))
Weighted Union-Find: Build O(E·α(V)), Per query O(α(V)) → Total O((E+Q)·α(V)) ≈ O(E+Q)
For typical inputs (small V, large Q), Union-Find can be 10–100× faster.

★ Algorithm Deep-Dive: Weighted Union-Find ▾

Origin: Union-Find (disjoint-set union) was introduced by Galler & Fisher (1964) and refined by Tarjan (1975) with path compression and union by rank. The weighted variant extends the standard structure to track numerical relationships between nodes and their roots.

FeatureStandard UFWeighted UF (this problem)
Per-node stateparent pointer onlyparent pointer + weight (= ratio to parent)
Find semanticssame root = same componentsame root AND weight[x] = x/root(x)
Path compressionupdate parent → rootupdate parent → root AND multiply weights
Union semanticslink two rootslink roots AND compute link weight from given ratio
Query answerconnected? yes/noif same component: weight[src]/weight[dst] = src/dst
🔑 Core Invariant

weight[x] always equals x / root(x). Every find() call and every union() call preserves this. The power: once built, any query in the same component reduces to a single division of two stored weights.

When to recognize this pattern:

  • "Given pairwise ratios, compute transitive ratios" → Weighted UF or BFS
  • "Pairs of variables linked by multiplicative relationships" → Weighted UF
  • "Connectivity + associated numeric value" → Weighted UF over standard UF
  • Many queries over same variable relationships → UF amortizes to O(1)/query
5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Reframe as a Graph

Each equation links two variables. Queries ask for transitive relationships. That's a graph — variables are nodes, equations are weighted edges. Immediately think: build an adjacency list and traverse it.

Step 2: Propose BFS First (safe, clear)

"I'll build a bidirectional weighted graph: A→B with weight k, B→A with weight 1/k. For each query, BFS from src to dst, multiplying edge weights. Return −1 if dst is unreachable." This always gets the problem right and is easy to code.

Step 3: Offer Weighted Union-Find as Optimization

"If we have many queries and few unique variables, we can do better. Weighted Union-Find precomputes all ratios in O(E·α) during build-time, then answers each query in O(α) — essentially O(1) per query." Show you know when to reach for the advanced tool.

Step 4: Derive the Union Weight Formula

If asked: "Given A/B = k, how do you set weight[rootA]?" Walk through: weight[A] = A/rootA, weight[B] = B/rootB. We need rootA/rootB. From A/B = k: rootA/rootB = k × (B/rootB) / (A/rootA) = k × weight[B] / weight[A]. This proves you understand the invariant.

🗣️ What to Say to the Interviewer

"This is a graph connectivity problem with weighted edges. Each equation A/B=k gives a directed edge A→B with weight k, and B→A with weight 1/k.
For each query, I BFS from src to dst, multiplying edge weights. The invariant is that at each BFS node, the accumulated product equals src/currentNode. When we reach dst, the product equals src/dst.
Return -1 if either variable is unknown or BFS ends without finding dst.
For the optimal solution, I'd use Weighted Union-Find: each node stores its ratio to the component root. Queries are O(α(n)) ≈ O(1)."
🔮 Likely Follow-up Questions
  • "What if equations conflict?" → Take first-seen; or detect contradiction when union finds same root with different implied weight
  • "Can you handle a/a queries?" → Yes: if a is known → 1.0; if unknown → −1.0
  • "What if we add equations dynamically?" → Union-Find handles this naturally with each union() call
  • "How do you handle floating point?" → Could track numerator/denominator as integers using GCD normalization

7 Summary & Key Takeaways ▾
Model each equation A/B=k as two directed edges: A→B (weight k) and B→A (weight 1/k)
Division telescopes: (A/B)×(B/C) = A/C — intermediate variables cancel algebraically
BFS accumulates a product along the path; the invariant is ratio = src/currentNode at each step
Weighted Union-Find is the optimal solution: O(α(V)) per query, essentially O(1)
In Weighted UF: weight[x] = x/root(x) always. Path compression MUST multiply weights through the chain
