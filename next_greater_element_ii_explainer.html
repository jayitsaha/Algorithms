LeetCode #503

Next Greater Element II

Given a circular integer array nums, return the next greater number for every element. Because the array is circular, you may wrap around from the end back to the beginning.

Medium Monotonic Stack Circular Array, 2 Passes Time: O(n) Space: O(n)
1 Understanding the Problem ▾

For every index i in the array, we want the first value to the right that is strictly greater than nums[i]. The key twist: the array is circular, so after the last element we can wrap around to the front. If no greater element exists even after a full loop, return -1.

💡 Key Insight

A circular array of length n behaves exactly like a linear array of length 2n where the second half repeats the first. We don't need to actually duplicate it — we simulate two passes using the modulo trick: nums[i % n]. One full circle is always enough to resolve every element.

✓ Example: nums = [1, 2, 1]

index 0 (val=1) → next greater: 2 at idx 1
index 1 (val=2) → wrap around: 1, then 1... no one > 2 → -1
index 2 (val=1) → wrap: idx 0 is 1 (not >), idx 1 is 2 (> 1!) → 2
Output: [2, -1, 2]

✗ Non-circular mistake: nums = [1, 2, 1]

index 2 (val=1): look right → end of array!
Naive answer: -1 (WRONG)
Must wrap around: idx 0=1 (no), idx 1=2 (>1!) → 2
Circular lookup is essential!

Edge Cases: Strictly decreasing array like [5, 4, 3] — no circular next-greater exists for index 0 (maximum); output [-1, 5, 5] since 3 and 4 find 5 by wrapping. All-equal elements like [2, 2, 2] → all -1. Single element [42] → [-1].

2 The Algorithm: Monotonic Decreasing Stack + 2-Pass Simulation ▾

We maintain a monotonic decreasing stack of indices. Each index on the stack is "waiting" to find its next greater element. When we encounter a value larger than the element at the top of the stack, we pop and record the answer for that index.

To handle circularity without duplicating the array, we run the loop from i = 0 to 2n - 1 and access nums[i % n]. We only push during the first pass (i < n). The second pass only triggers pops, resolving any indices still waiting.

1 Initialize: result = [-1] * n (default: no greater found), and stack = [] to hold waiting indices.
2 Two-pass loop: for i in range(2 * n). Use nums[i % n] as the current value, simulating circular traversal.
3 Pop and resolve: While the stack is non-empty and nums[stack[-1]] < nums[i % n], pop the top index and set result[top] = nums[i % n]. This index found its next greater element!
4 Push only in first pass: If i < n, push index i onto the stack. During the second pass (i ≥ n), we only pop — never push — to avoid duplicates.
5 Return result: Indices still on the stack after both passes have no circular next greater — their answer remains -1.
🔑 The Monotonic Stack Invariant

The stack always maintains indices whose values are in non-increasing order from bottom to top. When we see a value larger than the stack top, that larger value is exactly the "next greater element" for the top — by the ordering invariant, nothing between the top and the current position was larger (otherwise the top would have been popped earlier).

⚠️ Why We Store Indices, Not Values

We push the index i onto the stack, not the value nums[i]. When we pop, we need to know which position in result to write the answer. Storing values would lose this information.

3 Why It Works: The Invariant ▾
🔐 The Core Invariant

At any moment during the scan, every index on the stack represents an element that has not yet seen anything greater to its right. Stack values (from bottom to top) are in non-increasing order. When a larger value arrives, we have found the first such greater element for everything it "beats" on the stack — because we processed elements left to right without skipping any.

Why two passes suffice: In a circular array of length n, every element's "next greater" is at most n - 1 positions ahead (wrapping). After pass 1 (i = 0..n-1) any unresolved index is waiting for something in the wrap-around portion. Pass 2 (i = n..2n-1, equivalent to checking indices 0..n-1 again) covers exactly that wrap-around. No element needs to look more than once around the full circle.

Why "push only in pass 1" ensures correctness: Every actual array index (0 to n-1) gets pushed exactly once during pass 1. During pass 2, each index in 0..n-1 is visited again as a potential resolver (a larger value that can pop stacked entries), but the stacked entries themselves don't get re-pushed. This guarantees O(n) total push+pop operations.

When the answer is truly -1: Any element whose value is the global maximum in the array can never have a next greater — even circularly — so it stays -1. For example in [5, 4, 3], the element 5 at index 0 has no answer (nothing is greater than 5 anywhere). But 4 and 3 do find 5 by wrapping, so their answers are 5.

⚠️ Strict vs Non-Strict Greater

The condition is < (strictly less than), meaning we need strictly greater. If two equal elements appear consecutively, they don't resolve each other. For example in [2, 2, 2], nothing is strictly greater than 2 — all answers are -1.

4 The Code (Annotated) ▾

Solution 1 — Monotonic Stack, 2-Pass Circular (Optimal, Interview Pick)

def nextGreaterElements(nums): ← circular array input
n = len(nums) ← store n; used in modulo and range
result = [-1] * n ← default -1: assume no next greater
stack = [] ← indices waiting for their next greater
for i in range(2 * n): ← simulate two circular passes
while stack and nums[stack[-1]] < nums[i % n]: ← found next greater for top?
result[stack.pop()] = nums[i % n] ← record the answer and remove from stack
if i < n: ← only push indices during first pass
stack.append(i) ← this index is now waiting
return result ← unresolved indices stay -1
⚠️ Common Mistake: Forgetting the i < n Guard

If you push during both passes (remove the if i < n guard), you'll push index 0 a second time during pass 2 as index 5 (when n=5). This creates a duplicate on the stack, and when it gets popped, you'd overwrite result[0] incorrectly. Always push only in the first pass.

Solution 2 — Brute Force O(n²)

def nextGreaterElements_brute(nums): ← correct but slow for large n
n = len(nums)
result = [-1] * n
for i in range(n): ← for each element
for j in range(1, n): ← look at up to n-1 elements ahead
nxt = nums[(i + j) % n] ← circular index via modulo
if nxt > nums[i]: ← found first greater?
result[i] = nxt; break ← record and stop inner loop
return result
💡 Complexity Summary

Brute Force: Time O(n²) — for each of n elements, we scan up to n-1 others. Space O(n) for output only.
Monotonic Stack (Optimal): Time O(n) — each index is pushed once and popped at most once across the 2n iterations. Space O(n) for the stack (worst case all indices are stacked, e.g. strictly decreasing input) plus O(n) for output.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the "Next Greater" Pattern

"Next greater/smaller element to the right" is the signature monotonic stack pattern. The word "first" is key — we need the very next one, not any greater element. A decreasing stack that pops when it finds something larger gives us exactly the "first greater" in one left-to-right scan.

Step 2: Handle the Circular Twist

Say to the interviewer: "The circular part means I need to potentially look past the end. I can simulate this with two passes using i % n — no need to copy the array. One full circle is enough, so range(2*n) covers it."

Step 3: Justify Push-Only-Once Rule

"I only push during the first pass. Each index 0..n-1 needs to appear on the stack at most once. The second pass uses each value to resolve (pop) pending stack entries — if we pushed again, we'd get duplicates and corrupt answers."

Step 4: Analyze Amortized Complexity

"Despite the nested while loop, each index is pushed exactly once and popped at most once. Total stack operations across 2n iterations: at most n pushes + n pops = O(n). So the overall time is O(n), not O(n²)."

🗣️ What to Say to the Interviewer

"This is a 'next greater element' problem — classic monotonic stack territory.
The circular twist: I'll loop 2n times using i % n, so pass 2 covers the wrap-around.
I keep a decreasing stack of indices. When I see a larger value, it resolves everything smaller on the stack.
I only push in the first pass (i < n) to avoid duplicate entries.
Each index pushed exactly once, popped at most once → O(n) amortized. Stack is O(n) space."
🔮 Likely Follow-up Questions
  • "What if we want next SMALLER element?" → Monotonic increasing stack (pop when a smaller value is found)
  • "What if the array is not circular?" → Single pass from 0 to n-1, same algorithm (LeetCode #496)
  • "What if we want the number of days until a warmer temperature?" → Store distances, not values (Daily Temperatures #739)
  • "Can you solve it with O(1) extra space?" → No, the stack is needed; best case Ω(1) stack only if input is increasing
