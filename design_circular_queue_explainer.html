LeetCode #622

Design Circular Queue

Design a circular queue that supports enqueue, dequeue, peek front/rear, and isEmpty/isFull — all in O(1) time using a fixed-size array with modulo wrapping.

Medium Design Array with Head/Tail Time: O(1) all ops Space: O(k)
1 Understanding the Problem ▾

A circular queue (also called a ring buffer) is a fixed-capacity data structure that reuses array space as elements are removed. Unlike a naive queue that wastes memory at the front after dequeuing, a circular queue wraps the end of the array back to the beginning.

💡 Key Insight

Two index pointers — head (the dequeue side) and tail (the next write slot) — chase each other around the array. When either pointer reaches index k−1 and advances, it wraps back to 0 using modulo arithmetic: ptr = (ptr + 1) % k. A separate count field tracks occupancy and resolves the head==tail ambiguity (empty vs. full).

2 The Algorithm: Array with Head/Tail/Count ▾

The whole design fits in four fields: data (fixed array), head (front index), tail (next-write index), and count (current occupancy). Every operation is a small arithmetic expression — no loops, no shifting.

1 Init: Allocate data = [0] * k. Set head = tail = count = 0. Store capacity as self.k. The queue is empty and head==tail; count=0 signals the empty state.
2 enQueue(val): Guard on isFull(). Write data[tail] = val. Advance tail = (tail + 1) % k. Increment count. Return True. The modulo wraps tail from k−1 back to 0 so the next write lands in the recycled front slots.
3 deQueue(): Guard on isEmpty(). Advance head = (head + 1) % k. Decrement count. Return True. We never physically erase the old value — count ensures it won't be read again.
4 Front(): Guard on empty. Return data[head]. Head always points directly at the oldest element.
5 Rear(): Guard on empty. Return data[(tail - 1 + k) % k]. Tail points to the next write slot, so the last valid element is one position before it. Adding k before modulo prevents negative when tail==0.
🔑 Why a count Field Instead of Comparing head and tail?

When head == tail, the queue could be either empty (just drained) or completely full (just filled). Without a disambiguating signal, we can't tell them apart. A count field resolves this cleanly with zero wasted array slots. The alternative — sacrifice one slot and check (tail+1)%k == head for full — wastes a slot and is more confusing.

3 Why It Works: The Invariant ▾
🔐 The Invariant

At all times, the count live elements occupy positions data[head], data[(head+1)%k], ..., data[(head+count-1)%k] — exactly the region from head up to (but not including) tail, wrapping around the array as needed. Formally: tail = (head + count) % k always holds.

Why modulo wrapping is correct: The expression (i + 1) % k maps any valid index to the next one in the ring, cycling from k−1 back to 0. Since we advance pointers one step at a time, they always stay within [0, k). No element is ever skipped, double-read, or double-written.

Why Rear() uses (tail − 1 + k) % k: Tail is the next write position. The most recently written element sits one slot before it. Subtracting 1 and then applying modulo would give −1 when tail==0 — adding k first keeps the arithmetic positive: (0 − 1 + 4) % 4 = 3.

⚠️ Common Mistake: Rear() using data[tail]

Beginners often write return data[self.tail] for Rear, which reads the next slot to be written, not the last element. When the queue has just been enqueued into, that slot holds stale or uninitialized data. Always use data[(self.tail - 1 + self.k) % self.k].

Lifecycle of a slot: A slot starts empty, receives a value on enQueue (tail writes to it, then tail advances away), holds that value until the front dequeues past it (head advances away), and becomes logically empty again — available for the next tail write. The ring's circumference is always k; head and tail are runners around the same track.

4 The Code (Annotated) ▾

Solution 1 — Array with Head/Tail + Count (Optimal, O(1) all ops)

class MyCircularQueue:
def __init__(self, k: int):
self.data = [0] * k ← Fixed array; values are irrelevant until written
self.head = 0 ← Next element to dequeue lives here
self.tail = 0 ← Next enqueue will write here (NOT the last element)
self.count = 0 ← Disambiguates empty (0) from full (k) when head==tail
self.k = k ← Capacity stored for modulo arithmetic
def enQueue(self, value: int) -> bool:
if self.isFull(): return False ← O(1) guard; return immediately, no array changes
self.data[self.tail] = value ← Write the value at the current tail slot
self.tail = (self.tail + 1) % self.k ← Advance tail; wraps k-1 → 0
self.count += 1 ← Track occupancy for isEmpty/isFull checks
return True
def deQueue(self) -> bool:
if self.isEmpty(): return False ← O(1) guard; nothing to remove
self.head = (self.head + 1) % self.k ← Advance head past the old front element
self.count -= 1 ← Old value logically gone; slot reclaimed
return True
def Front(self) -> int:
if self.isEmpty(): return -1 ← Contract: return -1 for empty
return self.data[self.head] ← head always points at the front element
def Rear(self) -> int:
if self.isEmpty(): return -1 ← Contract: return -1 for empty
return self.data[(self.tail-1+self.k) % self.k] ← One slot BEFORE tail; +k prevents negative
def isEmpty(self) -> bool: return self.count == 0
def isFull(self) -> bool: return self.count == self.k
⚠️ Don't use Python's built-in collections!

The problem specifically asks you to implement the circular buffer mechanics yourself. Using collections.deque, list.pop(0), or queue.Queue misses the point entirely. The interviewer wants to see you manage head, tail, count, and modulo arithmetic from scratch. Show your understanding of how the ring buffer works internally.

Solution 2 — Linked List Implementation (for comparison)

class Node:
def __init__(self, val): self.val = val; self.next = None
class MyCircularQueue:
def __init__(self, k): ← Singly-linked, head and tail node pointers
self.size = 0; self.k = k
self.head = self.tail = None
def enQueue(self, val): ← Append new node to tail
if self.isFull(): return False
node = Node(val)
if self.tail: self.tail.next = node
self.tail = node
if not self.head: self.head = node
self.size += 1; return True
def deQueue(self): ← Advance head to next node
if self.isEmpty(): return False
self.head = self.head.next
if not self.head: self.tail = None ← Queue now empty; reset tail too
self.size -= 1; return True
def Front(self): return -1 if self.isEmpty() else self.head.val
def Rear(self): return -1 if self.isEmpty() else self.tail.val
def isEmpty(self): return self.size == 0
def isFull(self): return self.size == self.k
💡 Array vs Linked List Trade-offs

Array (preferred): O(1) time all ops, O(k) space (fixed), excellent cache locality, no per-node allocation overhead. Ideal when max capacity is known upfront.
Linked List: Same O(1) time, slightly more space (pointer per node), more complex empty-list handling (must sync head and tail), no wasted slots for sparse queues. Rear() is trivially self.tail.val — no modulo needed.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Name the core problem

"A naive queue that tracks a front pointer accumulates dead space on the left. A circular buffer reuses that space by treating the array as a ring — when a pointer reaches the end, it wraps to slot 0 using modulo. This gives O(1) enqueue AND O(1) dequeue with no shifting."

Step 2: Choose fields explicitly

"I need: a fixed array of size k, a head pointer (where I dequeue from), a tail pointer (where I next enqueue), and a count (to distinguish empty vs. full when head==tail). That's it — four fields, six methods, all O(1)."

Step 3: Address the head==tail ambiguity proactively

"When head==tail, the queue is either empty or full. I'll use count to distinguish. The alternative — waste one slot — gives only k−1 usable slots. My approach uses all k slots by keeping an explicit count."

Step 4: Derive Rear() carefully

"Tail always points to the NEXT slot to write. So the last valid element is data[(tail−1+k)%k]. I add k before modulo to handle tail==0 correctly — (0−1+k)%k = k−1, which is exactly where the last element lives."

🗣️ What to Say to the Interviewer

"I'll use a fixed array of size k with head, tail, and count.
enQueue: check isFull, write at data[tail], advance tail=(tail+1)%k, count++.
deQueue: check isEmpty, advance head=(head+1)%k, count--.
Front: data[head]. Rear: data[(tail-1+k)%k] — tail points past the last element.
isEmpty: count==0. isFull: count==k. All six ops are O(1) time, O(k) space total."
🔮 Likely Follow-up Questions
  • "Thread-safe version?" → Wrap enQueue/deQueue with a threading.Lock(); use condition variables for blocking enqueue/dequeue on full/empty
  • "Without a count field?" → Sacrifice one slot: full when (tail+1)%k == head. Capacity becomes k−1
  • "Design a circular deque instead?" → LeetCode #641: add enQueueFront (head moves backward) and deQueueRear (tail moves backward) using (ptr-1+k)%k
  • "What's this used for in practice?" → OS I/O buffers, producer-consumer queues, audio/video streaming buffers, CPU instruction pipelines

7 Summary & Key Takeaways ▾
Circular queue reuses array slots via modulo wrapping — no shifting, no resizing
Three fields drive everything: head (dequeue side), tail (next write), count (occupancy)
count resolves the head==tail ambiguity: 0 means empty, k means full
Rear() is data[(tail−1+k)%k] — tail is always one ahead of the last element
All 6 operations are O(1) time; total memory is O(k) (the array)

🎯 When to Recognize the Ring Buffer Pattern
  • "Design a queue/buffer with a fixed maximum capacity"

  • Memory should be reused rather than grown
  • Sliding window over time (Hit Counter, Sliding Window Maximum)
