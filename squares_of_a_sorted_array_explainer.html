LeetCode #977

Squares of a Sorted Array

Given an integer array sorted in non-decreasing order, return an array of the squares of each number, also sorted in non-decreasing order.

Easy Two Pointers Opposite Direction Time: O(n) Space: O(n)
1 Understanding the Problem ▾

You get a sorted array that may contain negative numbers. The task is simple to state — square every element and return the result in sorted order — but there is a subtle trap: squaring negatives makes them positive, and this scrambles the sorted order.

For example, [-4, -1, 0, 3, 10] is sorted, but squaring each element gives [16, 1, 0, 9, 100], which is completely out of order. A naïve sort of the squares takes O(n log n). Can we use the existing sorted order to do better?

💡 Key Insight

In a sorted array containing negatives, the largest squared values always live at the two ends — the leftmost element is the most negative (largest absolute value on the left) and the rightmost is the largest positive. The smallest squares are near zero in the middle. This gives us a two-pointer approach: compare the two ends, pick the larger square, and fill the result from the back.

Left end — most negative
-4 → |-4| = 4 → sq = 16
-1 → |-1| = 1 → sq = 1
↔ compare
Right end — most positive
3 → |3| = 3 → sq = 9
10 → |10| = 10 → sq = 100

Edge cases to think about: all positives (left pointer does nothing — right always wins), all negatives (right pointer does nothing — left always wins), all zeros, arrays of length 1, and pairs like [-3, 3] where both ends have equal absolute value (ties are fine — pick either).

2 The Algorithm: Opposite-Direction Two Pointers, Fill from Back ▾

The key trick is filling the output array from back to front. Placing the smallest square at the front is hard — we do not know where the minimum is without scanning everything. But placing the largest square at the back is easy — we always know the max is at one of the two ends of the remaining window.

1 Initialize three pointers: left = 0 (start of input), right = n-1 (end of input), write = n-1 (back of the output array). Pre-allocate result = [0] * n.
2 Compute squares at both ends: ls = nums[left]² and rs = nums[right]². These are the two candidates for the current maximum square.
3 Pick the larger square: If ls >= rs, place ls at result[write] and advance left += 1. Otherwise place rs and move right -= 1.
4 Always decrement write: After every placement, do write -= 1. This moves the output cursor one slot toward the front.
5 Repeat until left > right: When the pointers cross, every element has been consumed exactly once. Return result.
🔑 Analogy: Merging Two Sorted Lists

Think of the negatives (reversed) and the positives as two separate sorted lists of absolute values: [4, 1] (from the negatives, largest first) and [3, 10] (positives, largest first). Merging two sorted descending lists by always picking the larger head, and placing it into the result from back to front, is exactly what the two-pointer algorithm does.

3 Why It Works: The Invariant ▾
🔐 The Loop Invariant

At every iteration, result[write+1 .. n-1] contains the correctly placed squares for all already-consumed elements, and every value in that suffix is ≥ every square not yet placed. We always pick the current maximum, so the invariant is maintained on every step.

Why the maximum is always at an end: The input array is sorted. The element at left is the most negative remaining (and therefore has the largest absolute value on the left side). The element at right is the largest remaining positive. All elements strictly between them have strictly smaller absolute values. Therefore max(|nums[left]|, |nums[right]|) >= |nums[i]| for any i between left and right. The maximum square in the current window is always at one of the two ends.

Why filling from back works: We pick the maximum at each step and place it at the current back of the result (write). After n steps, write has gone from n-1 to -1 and every slot has been filled. Since we always placed the current max, the result is in descending order from back, which means ascending order from front — exactly what we want.

⚠️ Why not fill from the front?

To fill from the front we would need to find the minimum squared value first. That minimum lives somewhere in the middle of the array (near zero) — we cannot efficiently find it without scanning every element. Filling from the back is only possible because the maximum is always guaranteed to be at one of the two ends.

Correctness of tie-breaking (ls >= rs): When both ends have equal squared values (e.g., [-3, 3]), either can go first. Using >= means left wins ties, but both values end up placed correctly across two consecutive iterations regardless of which side you favor.

4 The Code (Annotated) ▾

Solution 1 — Two Pointers, Fill from Back (Optimal, O(n) time) — Interview Pick

def sortedSquares(nums):
n = len(nums) ← cache length; used multiple times
result = [0] * n ← pre-allocate output (avoids list.append overhead)
left, right, write = 0, n - 1, n - 1 ← left/right scan input; write fills result from back
while left <= right: ← loop until all elements consumed
ls = nums[left] ** 2 ← square of leftmost remaining element
rs = nums[right] ** 2 ← square of rightmost remaining element
if ls >= rs: ← left is larger (or tied): left wins
result[write] = ls ← place larger square at current back position
left += 1 ← consume left element, shrink window from left
else: ← right is strictly larger: right wins
result[write] = rs ← place right square at current back position
right -= 1 ← consume right element, shrink window from right
write -= 1 ← ALWAYS move write left after every placement
return result ← result is completely filled in ascending order
⚠️ Common Mistake: Forgetting to decrement write unconditionally

Every loop iteration places exactly one value and must advance the write pointer. A mistake is putting write -= 1 inside only one branch of the if/else. Both branches place a value, so write -= 1 belongs after the if/else, unconditionally.

Solution 2 — Naïve: Square and Sort (simpler code, O(n log n))

def sortedSquares(nums):
return sorted(x * x for x in nums) ← square every element, then sort. Ignores sorted input.
💡 Complexity Summary

Two Pointers (Optimal): Time O(n) — single pass; Space O(n) — output array (unavoidable; input is read-only)
Naïve Sort: Time O(n log n) — sort dominates; Space O(n) — output or sorted list
There is no O(1)-extra-space variant: we must produce a new sorted array. Any attempt to sort in-place would corrupt the input before reading all values.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Notice the "sorted input" hint

Sorted input is never accidental in a problem statement. It signals: "there is an O(n) or O(log n) solution that exploits this." Two pointers and binary search are the two main tools for sorted arrays.

Step 2: Propose the naïve approach first

"Simplest approach: square every element, then sort the result. O(n log n) time, O(n) space. Correct, but it throws away the sorted structure. Can we leverage it?"

Step 3: State the key observation

"After squaring, the largest values live at the two ends of the sorted input — the most negative and the most positive. The minimum is somewhere in the middle. Largest at the ends → two pointers approaching from opposite sides."

Step 4: Explain fill-from-back

"I'll fill the output from back to front. At each step, compare the squares at left and right; place the larger at result[write], advance the corresponding pointer, decrement write. When pointers cross, every slot is filled in ascending order."

🗣️ What to Say to the Interviewer

"The key insight: in a sorted array, the largest squared value is always at one of the two ends.
I'll use left and right pointers starting at the ends, and a write pointer starting at the back of the output.
Each iteration: compare nums[left]² and nums[right]². Place the larger at result[write--].
Advance whichever pointer gave the larger value, and repeat until left > right.
Time O(n), Space O(n) for the output — we can't avoid that since we must return a new array."
🔮 Likely Follow-up Questions
  • "Can you do it in-place?" → No practical O(n) in-place approach: squaring destroys the sorted order before all values are read.
  • "What if the array is unsorted?" → Sort first O(n log n) then square, or use a heap. The two-pointer trick requires sorted input.
  • "What about floating point numbers?" → Same algorithm — the comparison of squares works for any orderable numeric type.
  • "Merge Sorted Array" (#88) → Identical fill-from-back pattern, two read pointers into two separate arrays.

7 Summary & Key Takeaways ▾
Sorted input always hints at two pointers or binary search — look for it explicitly
In a sorted array, the largest absolute value is always at one of the two ends
Fill the output from back to front: placing the max is easy; placing the min is hard
Three pointers (left, right, write) is the canonical pattern for this type of merge
O(n) space for the output is unavoidable — no practical O(1)-extra-space variant exists

🎯 When to Recognize This Pattern
  • Input is sorted and the problem asks for a sorted transformation
  • Largest (or smallest) value lives at one of the two ends of the array
  • Merging two implicit sorted sequences (negatives reversed + positives)
  • Any problem involving absolute values on a sorted array
