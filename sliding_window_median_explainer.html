LeetCode #480

Sliding Window Median

Given an array nums and an integer k, return the median of each sliding window of size k as it moves from left to right.

Hard Heaps Two Heaps + Lazy Deletion Time: O(n log k) Space: O(k)
1 Understanding the Problem ▾

You have an array nums and a window of size k. The window starts at index 0 covering nums[0..k-1], then slides one position right, covering nums[1..k], and so on until the window reaches the end. For each window position, compute the median of the k elements in that window and record it. Return all medians in order.

The median of k sorted values is: the middle element if k is odd (at index k//2), or the average of the two middle elements if k is even. Crucially, the answer must be returned as a float, even when the median is a whole number.

💡 Key Insight

The median sits right at the boundary between the lower half and the upper half of the sorted window. If we maintain two heaps — a max-heap for the lower half and a min-heap for the upper half — we always have O(1) access to both boundary elements (the tops of the two heaps). The challenge is keeping both heaps up-to-date as the window slides, especially for deletions, which heaps don't support natively in O(log n).

Example: nums=[1,3,-1,-3,5,3], k=3

[1, 3,-1] → sorted[-1,1,3] → median 1.0
[3,-1,-3] → sorted[-3,-1,3] → median -1.0
[-1,-3, 5] → sorted[-3,-1,5] → median -1.0
[-3, 5, 3] → sorted[-3,3,5] → median 3.0
[5, 3, 6] → sorted[3,5,6] → median 5.0
[3, 6, 7] → sorted[3,6,7] → median 6.0

Why naive O(nk log k) is too slow

Sort each window: O(k log k) per step
n windows total → O(nk log k)
n=10^5, k=5×10^4:
~10^5 × 5×10^4 × 17 ≈ 8.5×10^10 ops
→ TLE. Need O(n log k).

Edge cases to consider: k=1 (every element is its own median), all equal elements, negative numbers, integer overflow in even-k averages (use float division, not integer), duplicates in the window, and correct handling when a deleted element is near the top vs deep in a heap.

2 The Algorithm: Two Heaps with Lazy Deletion ▾

We maintain two heaps that together represent the sorted window at all times:

lo — max-heap
lower half of window
top = max of lower half
⟷
hi — min-heap
upper half of window
top = min of upper half
Invariant: every lo element ≤ every hi element. |lo| = |hi| or |lo| = |hi| + 1.
Median = lo.top (k odd) or (lo.top + hi.top) / 2.0 (k even)

Python only has a min-heap (heapq), so we store values in lo as their negatives — pushing -x and reading -lo[0] simulates a max-heap. The critical trick for sliding: lazy deletion. When an element leaves the window, we do not remove it from the heap immediately (that would require O(k) search). Instead we add it to an invalid counter and skip it when it surfaces to the top during a future pop.

1 Seed the first window: For each of the first k elements, call push(x). Push routes x to lo (if x ≤ lo top) or hi (otherwise), then rebalances to maintain the size invariant.
2 Read the median: Clean stale tops, then compute: if lo_size == hi_size, return (lo.top + hi.top) / 2.0; else return float(lo.top). Append to result.
3 Remove the outgoing element (lazy): Mark invalid[out] += 1. Determine which heap it belongs to (compare with lo.top) and decrement that heap's effective size counter.
4 Add the incoming element: Call push(nums[i]). This also rebalances, compensating for the one element that was "removed" in step 3.
5 Lazy clean-up (inside every pop/peek): Before reading a heap's top, keep popping while the top is in the invalid map. Each clean-up costs O(log k) but is amortized — each element is cleaned at most once.
🔑 Why Lazy Deletion Works

Heaps guarantee their top is correct, but elements deeper in the heap are unordered. Lazy deletion exploits this: we don't care that stale elements are deep in the heap — we just need valid tops. When a stale element eventually floats up to the top (which it will, as real elements get popped ahead of it), we discard it then. The cost is amortized O(log k) per deletion, the same as a real deletion would cost.

3 Why It Works: The Invariant ▾
🔐 The Two Invariants We Must Maintain

Invariant 1 (ordering): max(lo) ≤ min(hi). Every element in the lower half is at most every element in the upper half. This makes the tops the two elements bracketing the median.

Invariant 2 (balance): lo_size == hi_size OR lo_size == hi_size + 1. The lower half is equal in size to the upper half, or has exactly one more element (for odd window sizes).

Why these invariants give us the median: If the sorted window has k elements, lo holds the bottom ⌈k/2⌉ of them and hi holds the top ⌊k/2⌋. The median is either the single middle element (lo.top when k is odd) or the average of the two bracketing elements (lo.top and hi.top when k is even).

How invariant 1 is maintained: When we push x, we check x vs lo.top. If x is in the wrong heap after a size-triggered rebalance move, the moved element crosses the boundary correctly — the heap tops enforce the ordering. We never place a large value in lo without moving the new lo.max to hi if it violates invariant 1.

How invariant 2 is maintained: After every push, we check: if lo_size > hi_size + 1, pop lo's max and push it to hi. If hi_size > lo_size, pop hi's min and push it to lo. One rebalance move suffices because each push changes one side's size by exactly 1.

⚠️ The Subtle Trap: Effective Size vs Physical Size

After lazy deletion, an invalid element is still physically in the heap — len(heap) is larger than the effective size. You must maintain separate lo_size and hi_size integers and use them for balance decisions. The moment you use len(lo), your balance logic will be wrong, leading to incorrect medians.

Why lazy deletion doesn't break invariant 1: Invalid elements remain in the heap but are logically absent. When we determine which heap an outgoing element "belongs to" (step 3 of the algorithm), we compare it to the clean lo.top. The size adjustment is immediate, so the balance-check after the subsequent push will rebalance correctly even if the just-deleted element hasn't been physically removed yet.

4 The Code (Annotated) ▾

Solution 1 — Two Heaps + Lazy Deletion (Optimal, Interview Pick)

import heapq
from collections import defaultdict
def medianSlidingWindow(nums, k):
lo, hi = [], [] ← lo: max-heap (stored negated); hi: min-heap
invalid = defaultdict(int) ← invalid[v] = # of copies of v lazily marked for removal
lo_size = hi_size = 0 ← effective (non-invalid) count in each heap
def add_to_heap(x): ← route x to correct heap, then rebalance
nonlocal lo_size, hi_size
if not lo or x <= -lo[0]: ← x belongs in lower half (≤ current lo max)
heapq.heappush(lo, -x); lo_size += 1
else:
heapq.heappush(hi, x); hi_size += 1 ← x > lo max: belongs in upper half
rebalance() ← fix size invariant after this push
def rebalance(): ← move one element to restore |lo|-|hi| ≤ 1
nonlocal lo_size, hi_size
if lo_size > hi_size + 1: ← lo too big: move its max to hi
heapq.heappush(hi, -heapq.heappop(lo))
lo_size -= 1; hi_size += 1
elif hi_size > lo_size: ← hi too big: move its min to lo
heapq.heappush(lo, -heapq.heappop(hi))
hi_size -= 1; lo_size += 1
def clean(heap, neg): ← pop invalid tops; neg=True for lo (values stored negated)
while heap and invalid[(-1 if neg else 1) * heap[0]] > 0:
invalid[(-1 if neg else 1) * heap[0]] -= 1
heapq.heappop(heap)
def get_median():
clean(lo, True); clean(hi, False) ← discard stale tops before reading
if lo_size == hi_size: ← even window: average the two midpoints
return (-lo[0] + hi[0]) / 2.0
return float(-lo[0]) ← odd window: lo.top is the exact middle
for x in nums[:k]: ← seed the first window
add_to_heap(x)
result = [get_median()] ← first window's median
for i in range(k, len(nums)):
out = nums[i - k] ← element leaving the window
invalid[out] += 1 ← mark for lazy removal
clean(lo, True); clean(hi, False) ← clean so lo[0] is valid for comparison below
if lo and out <= -lo[0]: ← decide which side loses a valid slot
lo_size -= 1
else:
hi_size -= 1
add_to_heap(nums[i]) ← add incoming, then rebalance
result.append(get_median())
return result
⚠️ Common Pitfall: Reading lo[0] Without Cleaning First

In the "which heap did the outgoing element belong to" check, you must compare out against a valid lo top. If you skip cleaning, lo[0] might be a stale invalid element, and you'll route the size adjustment to the wrong heap — corrupting your effective sizes and every subsequent median.

Solution 2 — SortedList (simpler, same asymptotic complexity)

from sortedcontainers import SortedList
def medianSlidingWindow(nums, k):
sl = SortedList(nums[:k]) ← O(k log k) — sorted list backed by balanced BST
result = []
def median(): ← O(1) index access on sorted structure
if k % 2: return float(sl[k // 2]) ← odd: exact middle element
return (sl[k//2 - 1] + sl[k//2]) / 2.0 ← even: average two middles
result.append(median())
for i in range(k, len(nums)):
sl.add(nums[i]) ← O(log k) insertion
sl.remove(nums[i - k]) ← O(log k) removal by value
result.append(median())
return result
💡 When to Use Each Approach

SortedList: If the interview allows sortedcontainers (common in Python-friendly interviews), this is your fastest-to-write correct solution. Very readable.

Two Heaps + Lazy Deletion: The canonical answer. Expected in system design or senior-level interviews. Shows you understand heap internals, amortized analysis, and why lazy deletion is necessary.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Pattern — "Sliding Window + Order Statistic"

Signals: "sliding window" + "median." The median requires knowing the sorted rank of elements. Brute-force sort per window is O(k log k) per step — mention this immediately to show you understand the bottleneck. You need a structure that maintains sorted order dynamically.

Step 2: Connect to the Prerequisite — Find Median from Data Stream (#295)

The two-heap structure for dynamic median (LeetCode #295) gives O(log n) insert and O(1) median. The new challenge: the window slides, so elements must also be removed. Mention this connection — it signals pattern recognition and shows you're building on prior knowledge.

Step 3: Address the Deletion Problem — Lazy Deletion

Heaps don't support efficient arbitrary deletion. Your options: (a) use a sorted balanced BST (SortedList in Python); (b) use lazy deletion — mark elements as invalid, clean them up when they float to the top. Explain the amortized O(log k) cost of lazy deletion.

Step 4: Mention the Size-Tracking Subtlety

Because invalid elements still occupy physical space in the heaps, you can't use len(heap) for balance decisions. Explain that you maintain separate lo_size and hi_size counters representing effective (non-invalid) counts.

🔮 Likely Follow-up Questions
  • "What's the space complexity?" → O(k) for the heaps. The invalid map is at most O(n) across the whole run but at most O(k) at any single point (each element in the window can be invalidated at most once before being cleaned).
  • "What if elements are very large, risking overflow?" → Use float division (a + b) / 2.0 — Python integers don't overflow, but in Java/C++ cast to long or use double.
  • "Can you do better than O(n log k)?" → O(n log k) is likely optimal for this problem; obtaining any order statistic over k items fundamentally requires O(log k) work per update.
  • "How does this differ from #295?" → #295 has no sliding window — no deletions needed. This problem adds the lazy deletion trick on top of the two-heap structure.

7 Summary & Key Takeaways ▾
Two heaps split the window: lo (max-heap) holds the lower half, hi (min-heap) holds the upper half — median lives at their tops
Lazy deletion avoids O(k) search: mark outgoing element invalid, discard it when it floats to the top — O(log k) amortized
Track lo_size and hi_size (effective counts) separately from len(lo) and len(hi) — physical sizes differ from logical sizes due to stale entries
Rebalance after every push: if |lo_size - hi_size| > 1, move one element across — at most one move per step
Always clean heap tops before comparing or reading — an invalid top gives a wrong lo.max value, corrupting size-accounting
Even-k median: average lo.top and hi.top as floats — never use integer division, negative values make this non-obvious

🎯 When to Recognize the Two-Heap + Lazy Deletion Pattern
  • You need the median of a dynamically changing window or stream
  • Elements must be both inserted and removed from a priority-queue structure
  • You need O(1) access to the maximum of the lower half AND minimum of the upper half simultaneously
  • k is large enough that sorting is too slow but you can't afford a segment tree
