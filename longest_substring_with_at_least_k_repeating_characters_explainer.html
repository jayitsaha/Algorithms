LeetCode #395

Longest Substring with At Least K Repeating Characters

Find the length of the longest substring where every character appears at least k times.

Medium Divide & Conquer Sliding Window Time: O(n·26) Space: O(n)
1 Understanding the Problem ▾

Given a string s and an integer k, find the length of the longest substring such that every character in it appears at least k times. The substring can contain multiple distinct characters — the only requirement is that none of them appear fewer than k times within that substring.

✓ Valid: s="aaabbb", k=3 → 6

Full string "aaabbb":
a appears 3 times ≥ 3 ✓
b appears 3 times ≥ 3 ✓
Answer = 6

✗ Tricky: s="aaabbbcc", k=3

Full string: c=2 < 3 ✗
Can't include 'c' in valid window
Best = "aaabbb" → 6

💡 Key Insight

Any character that appears fewer than k times in the current window is a mandatory split point — no valid substring can contain it. If character c appears m < k times in the whole string, then any substring containing c has count(c) ≤ m < k, which violates the condition. This observation is the foundation of the Divide & Conquer approach.

Edge Cases: k=1 (all characters valid, return len(s)); k > len(s) (impossible, return 0); all same character like "aaaa" with k=2 (answer is 4); all unique characters with k=2 (answer is 0); empty string (return 0).

2 The Algorithm: Divide & Conquer on Bad Characters ▾

The elegant insight: a character with count < k in the current window is provably impossible to include in any valid answer within that window. It acts as a guaranteed wall — split there, and recurse on each fragment independently.

1 Base case: If the string is empty, return 0. If every character in the current window appears ≥ k times, return the full length (no split needed).
2 Count frequencies: Build a frequency map for all characters in the current window.
3 Find a bad character: Find any character whose count is less than k. This character cannot appear in any valid answer — it is a mandatory divider.
4 Split and recurse: Split the string on every occurrence of the bad character, then recursively solve each fragment. Return the max over all fragments.
D&C Tree — s="aaabbbcc", k=3
"aaabbbcc" [a=3,b=3,c=2]
↓ c=2 < 3 → bad char is 'c', split: ["aaabbb", ""]
"aaabbb" [a=3,b=3]
""
↓ all ≥ 3 ✓ → len=6            ↓ len=0 → return 0
return 6 ✓
return 0
↑ max(6, 0) = 6 ← Answer
🔑 Alternative: Sliding Window with Unique-Count Enumeration

Because naive sliding window has no clear monotonic shrink condition (adding a character can fix a low count), we add an outer loop: iterate target_unique from 1 to 26. For each fixed unique count, use a standard two-pointer: shrink left when unique > target_unique, update answer when all unique chars also have count ≥ k. Both approaches run in O(n·26) time.

3 Why It Works: The Invariant ▾
🔐 The Core Invariant

At every recursive call on substring s[l..r], if character c has freq(c) < k within that substring, then no valid answer can contain c. Therefore every valid substring within s[l..r] lies entirely within one of the fragments created by splitting on c. The recursion covers all such fragments, so no valid answer is ever missed.

Why we can pick any bad character: It doesn't matter which bad character we pick to split on — any one of them is sufficient. The fragments produced by splitting on character c will be further processed by deeper recursive calls, which may discover and split on additional bad characters within each fragment.

Recursion depth is bounded by |Σ| = 26: Each call either returns immediately (all valid) or splits on a character that cannot appear in any sub-call's answer. After at most 26 such splits (one per distinct letter), we run out of bad characters and hit base cases everywhere. Therefore the recursion depth is at most 26, not proportional to n.

⚠️ Why Naive Sliding Window Doesn't Work Here

Standard sliding window requires a monotonic shrink condition: "when the window becomes invalid, shrinking from left makes it more valid." But here, removing a character can drop its count below k, making the window worse. And adding a character can bring a count up to k, making it better. There is no consistent direction — the window value is not monotone in window size. The unique-count enumeration trick adds back the missing monotonicity.

4 The Code (Annotated) ▾

Solution 1 — Divide & Conquer (Interview Pick)

from collections import Counter
def longestSubstring(s: str, k: int) -> int:
if len(s) == 0: ← base case: empty string
return 0
freq = Counter(s) ← count every char in current window
if all(v >= k for v in freq.values()): ← all chars meet threshold?
return len(s) ← entire string is valid!
# Find the first bad character (count < k)
bad = next(c for c in freq if freq[c] < k) ← this char cannot be in any valid answer
# Split on every occurrence of bad char
parts = s.split(bad) ← guaranteed: bad cannot cross fragment boundary
return max(longestSubstring(p, k) for p in parts) ← recurse on each fragment, take best
⚠️ Why max(), not sum() over fragments?

We want the longest single contiguous substring, not the combined length of all valid fragments. Each fragment is independent — the answer lives entirely within one of them. We take max, not sum.

Solution 2 — Sliding Window with Unique-Count Enumeration

def longestSubstring(s: str, k: int) -> int:
unique_count = len(set(s)) ← max distinct chars; outer loop bound
result = 0
for target in range(1, unique_count + 1): ← fix: window must have exactly 'target' unique chars
freq = [0] * 26 ← frequency of each letter in current window
left = cur_unique = cur_valid = 0 ← left ptr, distinct chars, chars with count ≥ k
for right in range(len(s)):
idx = ord(s[right]) - ord('a')
if freq[idx] == 0: cur_unique += 1 ← new char enters window
freq[idx] += 1
if freq[idx] == k: cur_valid += 1 ← char just hit the k threshold
while cur_unique > target: ← too many unique chars — shrink left
li = ord(s[left]) - ord('a')
if freq[li] == k: cur_valid -= 1 ← losing a char at exactly k threshold
freq[li] -= 1
if freq[li] == 0: cur_unique -= 1 ← char leaves window entirely
left += 1
if cur_unique == target == cur_valid: ← all chars in window appear ≥ k times!
result = max(result, right - left + 1)
return result
💡 Complexity Comparison

Divide & Conquer: O(n·|Σ|) time — each level does O(n) work, depth ≤ 26. O(n) space for recursion stack (worst case: each call halves the remaining string).
Sliding Window: O(n·|Σ|) time — 26 outer iterations × O(n) two-pointer pass each. O(|Σ|) = O(1) extra space for the frequency array.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Identify Why Greedy Sliding Window Fails

Say: "My first instinct is a sliding window, but I notice there's no monotonic shrink condition. Adding a character can raise its count from k-1 to k (making the window better), not worse. So I can't simply shrink when a count drops below k."

Step 2: Discover the D&C Insight via Elimination

Ask yourself: "Is there anything I can immediately rule out?" Any character with count < k in the current range is provably forbidden — it cannot appear in any valid answer. That makes it a guaranteed split point. This leads directly to the recursive approach: count → find bad → split → recurse.

Step 3: Bound the Complexity

"Each recursive call either returns immediately (all chars valid) or splits on a bad character. The bad char can no longer appear in sub-calls' windows. Since there are at most 26 distinct characters, we can split at most 26 times total — so recursion depth is bounded by |Σ|, not n."

Step 4: Offer the Iterative Bonus

"There's also an iterative O(n·26) sliding window version: enumerate the target number of unique characters (1 to 26) in the outer loop. This fixes the monotonicity — we shrink when unique > target. This avoids recursion and uses O(1) space."

🗣️ What to Say to the Interviewer

"Naive sliding window fails here — no monotonic shrink condition exists.
Key observation: a char with count < k in the current window is forbidden — it's a guaranteed split point.
Algorithm: count frequencies, find first bad char, split on it, recurse on fragments, return max.
Depth ≤ 26 (one per distinct character), each level O(n) → total O(26n) = O(n).
Space: O(n) for the recursion stack.
Iterative variant: outer loop over target unique count 1..26, inner two-pointer — same time, O(1) space."
🔮 Likely Follow-up Questions
  • "What if k=1?" → Every char is valid, return len(s) immediately (no split needed)
  • "What if the alphabet is larger (e.g., Unicode)?" → D&C still works; use a dict for freq. Sliding window outer loop becomes impractical for large alphabets.
  • "Can you do it without recursion?" → Yes — the sliding window enumeration approach is fully iterative
  • "What is the worst case for D&C space?" → O(n) for the call stack if we always split at the middle

7 Summary & Key Takeaways ▾
A character with count < k in the current window cannot appear in any valid answer — it is a mandatory split point
Divide & Conquer: split on bad chars, recurse on fragments, take max — depth ≤ 26 (bounded by alphabet size)
Naive sliding window fails because adding a char can raise a count (no monotonic shrink condition)
Sliding window works if we enumerate target unique-char count (1–26) in the outer loop to restore monotonicity
Both approaches are O(n·26) ≈ O(n) for lowercase English; D&C uses O(n) stack, sliding window uses O(1)
When sliding window breaks monotonicity, ask: "what fixed property in an outer loop restores it?"

🎯 When to Recognize This Pattern
  • "Longest substring where ALL characters meet a threshold" → Divide & Conquer on bad chars
  • "Sliding window feels right but shrink condition isn't monotonic" → Enumerate a fixed property in outer loop
  • "Small fixed alphabet (≤ 26 chars)" → O(n·26) is effectively O(n); safe to iterate over all possible values
  • "Some element provably can't be in any valid answer" → It's a split point → D&C
