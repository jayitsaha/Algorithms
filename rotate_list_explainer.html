LeetCode #61

Rotate List

Given the head of a linked list, rotate the list to the right by k places.

Medium Linked List Find New Head Position Time: O(n) Space: O(1)
1 Understanding the Problem ▾

"Rotate right by k" means the last k nodes detach from the tail and reattach at the front of the list. Each single rotation takes the very last node and moves it to the head. Doing this k times in a row gives us the fully rotated list.

💡 Key Insight

For a list of length n, rotating right by k is the same as finding the node at position n − (k mod n) (0-indexed) and making it the new head. Everything before it becomes the tail, stitched to the original tail. This is a cut-and-reattach at one precise position — no element-by-element movement needed.

✓ Example: [1→2→3→4→5], k=2

n=5, effective k = 2 mod 5 = 2
New head at index: 5 − 2 = 3 (value 4)
New tail: index 2 (value 3)
Result: 4→5→1→2→3

⚠ Edge: k=5 (full rotation = no-op)

n=5, effective k = 5 mod 5 = 0
k becomes 0 → return list unchanged
(every node is back where it started)
Result: 1→2→3→4→5

Edge Cases to watch: empty list (return null), single node (always unchanged), k=0 (no rotation), k that is a multiple of n (no-op after mod), and very large k (mod handles it).

2 The Algorithm: Make Circular, Find New Head ▾

The elegant trick: instead of moving k nodes one by one (slow), form a ring by connecting the tail back to the head, then find exactly where to cut the ring. One cut in the right place produces the rotated list in O(n) total time.

1 Count length & find tail: Traverse the list once. Count nodes (starting at n=1 for head) and stop when tail.next is None. This single pass gives us n and a reference to the tail node.
2 Normalize k: Set k = k % n. Rotating by n is the identity — every node returns to its start position. If k becomes 0 after this, return head immediately.
3 Form a circle: Set tail.next = head. The list is now a ring. Any node can reach any other node by following .next. This sets us up for the elegant cut.
4 Walk to the new tail: Starting from head, advance n − k − 1 times. You land on the node at 0-indexed position n − k − 1 — this is the new tail (it will point to null when we cut).
5 Cut and return: Save new_head = new_tail.next. Set new_tail.next = None to break the circle. The ring is now a list again, starting at new_head. Return new_head.
🔑 Why exactly n − k − 1 steps?

After normalizing k (1 ≤ k ≤ n−1), the new head is at 0-indexed position n − k. The new tail is one step before it: position n − k − 1. Starting from position 0 (head), we need to advance exactly n − k − 1 positions. This formula is always safe: with 1 ≤ k ≤ n−1, the value n−k−1 is between 0 and n−2 inclusive.

3 Why It Works: The Invariant ▾
🔐 The Core Invariant

When a list of n nodes is rotated right by k positions, the node originally at 0-indexed position n − k becomes the new head, and the node at position n − k − 1 becomes the new tail. This is always true, regardless of k (after normalizing with mod n).

Proof by example: [1, 2, 3, 4, 5], k=2, n=5. New head index = 5 − 2 = 3 → value 4. New tail index = 5 − 2 − 1 = 2 → value 3. Result should be [4, 5, 1, 2, 3]. ✓

Why the circle works: After connecting tail → head, the structure is a ring. The rotated list is this ring read starting from position n−k. We don't actually loop the ring; we cut it at precisely the right spot. The ring formation is purely a conceptual/implementation tool — it lets us avoid tracking two separate pointers simultaneously.

Why k mod n is essential: Rotating n times is the identity — every node ends up where it started. So rotating by k and by k+n give the same result. For any k, only k mod n rotations are "real work." Without this normalization, k=5 on a 5-node list would cause us to walk 0 positions (still correct) but k=7 on a 5-node list would incorrectly try to walk 2 positions past the tail.

⚠️ Off-By-One: Why − 1 at the end?

We want the new tail (to set its .next = None), not the new head. The new tail is one node before the new head. So we walk to index n−k−1, not n−k. If you accidentally walk to n−k, you land on the new head and would need a separate "prev" reference — more error-prone. Using n−k−1 keeps the logic clean.

4 The Code (Annotated) ▾

Solution 1 — Make Circular & Cut (Optimal, Interview Pick)

def rotateRight(head, k):
if not head or not head.next or k == 0: ← quick exits: empty, 1-node, or zero-rotation
return head
tail, n = head, 1 ← tail starts at head, n=1 (head already counted)
while tail.next: ← walk until true tail (tail.next is None)
tail = tail.next ← advance to next node
n += 1 ← count it
k %= n ← normalize: k=5 on n=5 → k=0 (no-op)
if k == 0: return head ← full rotations cancel — return unchanged
tail.next = head ← form the circle: last node → first node
new_tail = head ← start from head to walk n-k-1 steps
for _ in range(n - k - 1): ← n-k-1 advances → lands on new tail node
new_tail = new_tail.next ← advance pointer
new_head = new_tail.next ← node after new_tail is the new head
new_tail.next = None ← break the circle here — new_tail is now the tail
return new_head ← rotated list starts here ✓
⚠️ Why does n start at 1, not 0?

We assign tail = head before the loop, so head is already the first node we've "visited." The loop counts subsequent nodes as tail moves forward. Starting n at 1 means when the loop ends (tail.next = None), n correctly equals the total number of nodes. Starting at 0 would give n = length − 1.

Solution 2 — Brute Force (Rotate One at a Time, O(n × k))

def rotateRight_brute(head, k):
if not head or not head.next: return head ← base cases
n, cur = 0, head
while cur: n += 1; cur = cur.next ← count length
k %= n ← normalize
for _ in range(k): ← k individual one-step rotations
cur = head
while cur.next.next: cur = cur.next ← find second-to-last node
cur.next.next = head ← last node now points to old head
head = cur.next ← new head is the old last node
cur.next = None ← detach old last node from second-to-last
return head ← correct but O(n*k) — TLE for large k
💡 Complexity Summary

Brute Force: Time O(n × k) — k passes of O(n) each. Space O(1). Correct but too slow.
Make Circular & Cut: Time O(n) — exactly 2 linear traversals. Space O(1) — only 3 pointer variables (tail, new_tail, new_head). This is always the right answer in an interview.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Spot the Pattern Signal

"The problem asks me to move the last k nodes to the front. The critical observation: we're finding a specific position from the end of the list. 'k-th from end' is a classic linked list pattern — signal that we're dealing with a Find New Head Position problem."

Step 2: Handle k > n Immediately

"First thing I check: what if k is larger than the list length? Rotating by exactly n is a no-op. So k = k % n. If k becomes 0, I can return head right away. This is an important edge case that interviewers love to probe."

Step 3: The Circle Insight

"Instead of moving k nodes one by one — which is O(n × k) — I can think of the list as a ring. Connect tail to head, then find where to cut. The cut is at position n−k from the start. Total work: two O(n) passes, O(1) space."

Step 4: Walk to n − k − 1, Not n − k

"I need to land on the new tail so I can set new_tail.next = None. That's index n−k−1, not n−k. Then new_head = new_tail.next gives me the new head. Clean, no extra bookkeeping."

🗣️ What to Say to the Interviewer

"First edge cases: empty list, single node, or k=0 are immediate returns.
Then I'll traverse once to count n and land on the tail.
k = k % n to normalize — rotating by n is identity. If k=0 after that, return head.
Connect tail.next = head to form a ring. Now I need to find where to cut.
Walk n−k−1 steps from head to land on the new tail.
new_head = new_tail.next. Set new_tail.next = None. Return new_head.
Time O(n) — two passes. Space O(1) — three pointer variables."
🔮 Likely Follow-up Questions
  • "Can you do it in one pass?" → Two pointers k+1 apart: advance fast k+1 steps, then move both until fast hits null — slow is the new tail.
  • "What about rotate left by k?" → Equivalent to rotate right by n−k. Same code with k = n − (k % n).
  • "Doubly-linked list?" → Same logic, plus update .prev pointers at the cut and the join points.
  • "What if k is negative?" → Python % handles negatives naturally; in C++/Java, use ((k % n) + n) % n.

7 Summary & Key Takeaways ▾
Always normalize k first: k = k % n. Rotating by n is always a no-op
Form a circle (tail.next = head), then find where to cut — two passes, O(n) total
New tail is at index n−k−1; new head is the next node (index n−k)
Start n at 1 (head is already counted before the counting loop)
O(1) space — only three extra pointer variables: tail, new_tail, new_head

🎯 When to Use the "Find New Head Position" Pattern
  • Problem asks to move last k elements/nodes to the front
  • Need to find the k-th node from the end of a linked list
  • Rearranging list segments (not individual nodes) in-place
  • k could exceed list length → always reduce with mod n
