LeetCode #173

Binary Search Tree Iterator

Implement a lazy in-order iterator over a BST that returns the next smallest integer in O(1) amortized time using only O(h) extra space.

Medium Trees Controlled In-order Stack Time: O(1) avg Space: O(h)
1 Understanding the Problem ▾

You are given the root of a Binary Search Tree (BST). Implement an iterator class with two methods: next() which returns the next smallest integer, and hasNext() which tells you if more integers remain. Both must use O(h) extra space (h = tree height) and O(1) average time.

The fundamental property that unlocks this problem: in-order traversal of a BST (Left → Root → Right) visits nodes in ascending sorted order. If you walked the full tree recursively, you'd see every value from smallest to largest. The challenge is doing this lazily — producing one value at a time on demand, without pre-computing the full list.

💡 Key Insight

Think of in-order traversal as following a "path of leftmost nodes." At any point in the traversal, the next value to visit is always the leftmost reachable node from the current frontier. An explicit stack can remember these frontier nodes between next() calls — the stack always holds the "left spine" from the current position downward.

✓ What We Want (lazy, O(h) space)

it = BSTIterator(root)
it.next() → 1
it.next() → 3
it.next() → 7
it.hasNext() → True
it.next() → 9
it.hasNext() → False

✗ What We Avoid (eager, O(n) space)

# collect all values upfront
vals = []
inorder(root, vals)
# O(n) memory used even if
# caller only calls next() once

Edge Cases: Single-node tree (one push, one pop, done). Right-skewed tree (stack depth always 1 — each push_left only pushes one node). Left-skewed tree (stack depth starts at n, decreases monotonically — still O(h) at any moment). The problem guarantees next() is only called when hasNext() is true.

2 The Algorithm: Controlled In-order Stack ▾

The core idea is to simulate recursive in-order traversal using an explicit stack. The stack at any point holds exactly the left spine — the path from some "current" node down to its deepest left descendant. The top is always the next value to emit.

1 Init: Create an empty stack. Call _push_left(root) — walk leftward from root, pushing every node until we hit None. After this, stack top = the globally smallest node.
2 hasNext(): Simply return len(self.stack) > 0. If the stack is non-empty, more values remain. O(1) time, no traversal needed.
3 next(): Pop the top node (this is the minimum remaining). Then call _push_left(node.right) to push the left spine of the popped node's right subtree. Return node.val.
4 _push_left(node): Loop while node is not None: push node, then go left (node = node.left). This anchors the iterator at the leftmost reachable node from any starting point.
🔑 Why the Left Spine Works

In a BST, the in-order successor of any node is the leftmost node of its right subtree (if it has one), or an ancestor node (which is already on the stack waiting). By maintaining the left spine, the stack encodes exactly these "waiting ancestors" plus the path to the current frontier. No node is ever missed or double-counted.

3 Why It Works: The Invariant ▾
🔐 The Stack Invariant

At all times, the stack contains exactly the nodes that are still pending in in-order traversal, stored from bottom (largest pending) to top (smallest pending = next output). The top of the stack is always the globally next smallest unvisited node. Stack elements are always in descending order top-to-bottom.

Why popping gives the next smallest: In-order traversal visits Left → Node → Right. The stack simulates this: when we push a left spine from some root, everything in the left subtree has been pushed or was already visited. The top is the leftmost node in whatever subtree we most recently entered — which is definitionally the minimum of that subtree and thus the global minimum of what remains.

Why O(1) amortized per call: Each of the n nodes is pushed onto the stack exactly once (when first encountered going leftward) and popped exactly once (when it becomes the next answer). Total push work across all n calls = O(n). Total pop work = O(n). Averaged over n calls, that is O(1) per call. Some individual calls are O(k) where k is the left-spine length of a right child, but this is bounded by total O(n).

⚠️ O(h) vs O(n) Space — The Distinction

The stack holds at most O(h) nodes at any snapshot. This is because the stack always holds a single path from some node down to a leaf — and any path in a tree has at most h nodes. The total work is O(n), but the space at any instant is O(h). For a balanced BST, h = O(log n). The brute-force approach stores all n values — O(n) even at init.

Right-skewed tree (degenerate, all right children): _push_left always pushes only the current node (no left children). Stack depth stays at 1 throughout. Very fast.

Left-skewed tree (all left children): Init pushes all n nodes (O(n) = O(h) here since h=n). Then each next() pops one and pushes nothing (no right children). Stack decreases by 1 each time. This is the worst case for init but still O(h) space.

4 The Code (Annotated) ▾

Solution 1 — Controlled In-order Stack (Optimal, Interview Pick)

class BSTIterator:
def __init__(self, root): ← O(h) time and space; called once
self.stack = [] ← Explicit stack replaces recursion's call stack
self._push_left(root) ← Populate the initial left spine
def _push_left(self, node): ← Helper: push node + every left descendant
while node: ← None means we've reached the leaf's left child
self.stack.append(node) ← Push: this node is "pending" for in-order visit
node = node.left ← Always go left to find the minimum
def hasNext(self) -> bool: ← O(1): non-empty stack means more values exist
return len(self.stack) > 0
def next(self) -> int: ← O(1) amortized; each node is pushed/popped once
node = self.stack.pop() ← Top = next smallest; remove it
self._push_left(node.right) ← Explore left spine of right subtree (may push 0-h nodes)
return node.val ← Emit this value; in-order guarantees ascending
⚠️ Order Matters: Pop First, Then Push Right

We pop first to get the node's value, then push its right subtree's left spine. This matches in-order: visit the node (pop), then recurse right (push_left of right child). If we pushed the right subtree before popping, we'd process right before reporting the current node — that would be wrong order.

Solution 2 — Flatten BST to List at Init (Simple, O(n) space)

class BSTIterator:
def __init__(self, root): ← O(n) time AND O(n) space — bad for large trees
self.vals, self.idx = [], 0
def inorder(node): ← Standard recursive DFS in-order
if not node: return
inorder(node.left)
self.vals.append(node.val)
inorder(node.right)
inorder(root) ← Collect ALL values upfront
def hasNext(self): return self.idx < len(self.vals)
def next(self): ← O(1), but O(n) already spent at init
v = self.vals[self.idx]; self.idx += 1; return v
💡 Complexity Summary

Controlled Stack: Init O(h), hasNext O(1), next O(1) amort, Space O(h) ← use this
Flatten List: Init O(n), hasNext O(1), next O(1), Space O(n) ← mention first, then optimize
Real Python generator: yield statements handle this automatically — the runtime does exactly what our explicit stack does.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Traversal

"Next smallest in a BST" is in-order traversal. Signal: BST + ascending order + iterator interface = controlled in-order with explicit stack.

Step 2: Propose Brute Force First

"Simple approach: traverse the whole BST at init, store sorted values in a list, use an index. O(n) space. I'll mention this as baseline, then optimize."

Step 3: Identify the Space Constraint

"The problem requires O(h) space — h is tree height. The brute force uses O(n). I need lazy evaluation: compute the next value only when next() is called."

Step 4: Derive the Stack Invariant

"I'll simulate recursion with a stack. In in-order, I'd go left until leaf, then visit, then recurse right. My stack holds the 'go left until leaf' path. Pop = visit. Then push_left(right child) = recurse right."

🗣️ What to Say to the Interviewer

"Since BST in-order gives sorted order, I need to simulate in-order traversal lazily.
I'll maintain a stack holding the left spine from any current node.
At init: push_left(root) — this loads the initial left spine. Stack top = global minimum.
hasNext(): stack non-empty → True. O(1).
next(): pop the top node (next smallest). Push left spine of its right child. Return val.
Each node pushed once, popped once → O(1) amortized, O(h) space at any point."
