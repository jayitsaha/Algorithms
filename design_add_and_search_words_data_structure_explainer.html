LeetCode #211

Design Add and Search Words Data Structure

Design a data structure that supports adding new words and searching if a string matches any previously added string, where '.' matches any single letter.

Medium Tries Trie + DFS for Wildcards addWord: O(m) search: O(26^d)
1 Understanding the Problem ▾

We need a data structure with two operations: addWord(word) which stores a word, and search(word) which returns true if the query matches any stored word. The twist: a query can contain '.', which acts as a wildcard matching any single letter.

💡 Key Insight

A Trie handles addWord efficiently by sharing common prefixes. The wildcard '.' turns search into a DFS — at each '.' position we branch into all existing children and recurse. This is the Trie + DFS for Wildcards pattern.

✓ Words Stored

addWord("bad")
addWord("dad")
addWord("mad")

? Query Results

search("pad") → False
search("bad") → True
search(".ad") → True ('.' = b, d, or m)
search("b..") → True ('..' = any 2 chars)

Edge Cases: Query is all wildcards like "..." (tries every path), query matches prefix but not full word (must check is_end), query character not in trie at all (dead end), empty string, single-character words.

🔑 Why Not a Hash Set?

A set handles exact matches in O(1), but wildcards break it: to match "b.." against a set you'd have to scan all stored words of length 3 starting with 'b'. A Trie organizes words by shared prefixes — the wildcard DFS only follows edges that were actually inserted, pruning entire subtrees automatically. This is the structural advantage that makes Trie the right choice here.

2 The Algorithm: Trie + Recursive DFS ▾

Build a Trie (prefix tree). Each node stores: a dictionary children mapping characters to child nodes, and a boolean is_end marking where complete words end. addWord traverses the trie creating nodes along the way. search uses a recursive DFS that handles '.' by branching into all existing children.

1 Define TrieNode: children = {} (char to TrieNode), is_end = False. The root node represents the empty prefix. No character is stored in the node itself — characters live on the edges.
2 addWord(word): Start at root. For each character: if the edge doesn't exist, create a new child node. Move to the child. After the last character, set is_end = True on the current node. Time O(m), m = word length.
3 search(word) — base case: If we've consumed all characters (index == len(word)), return node.is_end. This distinguishes "reached a node" from "reached the end of a stored word." Critical: a prefix like "ba" must not match if only "bad" was added.
4 search(word) — literal character: If word[i] is a letter, follow that specific child edge. If the edge doesn't exist, return False. Recurse with index + 1. This is O(m) — one path from root to leaf.
5 search(word) — wildcard '.': Iterate over all existing children of the current node. For each, recursively call DFS with index + 1. If any recursive call returns True, immediately return True. If none match, return False. This is the branching step.
⚠️ Wildcard Worst-Case Complexity

A query of all wildcards like "..." against a trie full of 3-letter words forces exploring up to 26 × 26 × 26 paths in the absolute worst case. In practice, only nodes that were actually inserted exist, so the true branching factor is bounded by the actual vocabulary. The key insight for interviews: O(26^d × m) where d = number of '.' and m = word length.

3 Why It Works: The Invariant ▾
🔐 The Trie Invariant

Every path from root to a node marked is_end=True spells exactly one added word. Conversely, every added word corresponds to exactly one such path. The DFS invariant: at call dfs(node, i), node is the Trie node reached after matching the first i characters of the query — where literal chars follow their exact edge and wildcards may have followed any existing edge.

Why '.' must try all children: When a word was added, we don't store which letter was used at each position in any queryable form — we only stored the letters as edges. At a '.' position, any letter could have been stored, so we must try every edge that exists at that node. We don't try non-existent edges because those paths were never inserted.

Why checking is_end at the base case is essential: The Trie stores words, not just characters. The word "ba" is NOT stored even if "bad" is — because when "bad" was added, only the node for 'd' got is_end=True. Checking is_end at the exact moment we've consumed all query characters ensures we only match complete words.

💡 Why DFS Is Correct for Wildcards

The DFS explores the subset of the trie reachable from the current node. For a literal character, that subset is a single child (or nothing). For '.', it's the set of all existing children. By definition, the set of all paths in the trie from the current position that lead to an is_end node is exactly the set of words that could match. DFS is guaranteed to find all such paths (completeness) and will only find nodes in the trie (soundness — no spurious matches).

4 The Code (Annotated) ▾

Solution 1 — Trie + Recursive DFS (Optimal / Interview Pick)

class TrieNode:
def __init__(self):
self.children = {} ← dict: char → TrieNode. Only letters stored are keys
self.is_end = False ← True when a complete word ends at this node
class WordDictionary:
def __init__(self):
self.root = TrieNode() ← Root = empty prefix. Every word path starts here
def addWord(self, word: str) -> None:
node = self.root ← Reset to root for each new word
for ch in word: ← Walk character by character
if ch not in node.children: ← Edge missing? Create it
node.children[ch] = TrieNode()
node = node.children[ch] ← Move to next node along edge
node.is_end = True ← This node now marks end of a complete word
def search(self, word: str) -> bool:
def dfs(node, i):
if i == len(word): ← Base case: consumed all characters
return node.is_end ← Match only if a word ends exactly here
ch = word[i]
if ch == '.': ← Wildcard: try every existing child
for child in node.children.values():← .values() gives nodes, not chars
if dfs(child, i + 1):
return True ← Short-circuit: first success is enough
return False ← No child path worked → dead end
else: ← Literal char: follow exactly that edge
if ch not in node.children: return False
return dfs(node.children[ch], i + 1)
return dfs(self.root, 0) ← Start DFS from root at index 0
⚠️ Common Mistake: children.values() vs children.keys()

For the wildcard branch, iterate node.children.values() to get the child TrieNode objects — those are what you pass to the next DFS call. A common error is iterating .keys() (which gives characters) or .items() without unpacking. You're passing nodes to dfs(child, i+1), not characters.

Solution 2 — Hash Map (Brute Force, simpler to state)

class WordDictionary:
def __init__(self):
self.words = [] ← Store all words in a flat list
def addWord(self, word): self.words.append(word)
def search(self, word):
import re
pattern = re.compile('^' + word.replace('.', '[a-z]') + '$')
return any(pattern.match(w) for w in self.words) ← O(n*m) — slow

The regex approach is easy to state in an interview as a brute-force baseline. It's O(n×m) per search where n = total words. The Trie solution is strictly better because it avoids re-scanning all words on every search.

💡 Complexity Summary

addWord: Time O(m) — one pass through word. Space O(m) amortized per new word (shared prefix nodes reused)
search (all literals): Time O(m) — single root-to-leaf path
search (with d wildcards): Time O(26^d × m) worst case; O(actual_paths × m) in practice
Total space: O(N×m) where N = number of words, m = average length

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the Signals

"The problem asks me to store strings and support pattern matching with wildcards. 'Store strings' + 'prefix structure' = Trie. 'Wildcard matching' = DFS/backtracking. These two signals together point directly to Trie + DFS."

Step 2: State the Brute Force

"Brute force: store all words in a list. For search, check each word with regex or character-by-character matching. O(n×m) per search — too slow if many words. We can do better by organizing the structure."

Step 3: Propose Trie + Explain addWord

"A Trie groups words by shared prefixes. addWord: traverse letter by letter, creating nodes as needed, mark is_end at the final character. This is the standard Trie insert — O(m)."

Step 4: Explain the Wildcard DFS Insight

"For search with '.': at a literal character I follow exactly that edge. At '.' I don't know which letter was used, so I try ALL children recursively — this is a DFS. I short-circuit and return True as soon as any branch succeeds. If all children fail, return False."

🗣️ What to Say to the Interviewer

"I'll use a Trie. Each TrieNode has children dict and is_end bool.
addWord: walk the trie creating nodes, set is_end=True at the end. O(m).
search: I'll write a recursive DFS helper dfs(node, i).
Base case: i == len(word) → return node.is_end.
If word[i] is a letter: follow that child if exists, else False.
If word[i] is '.': try all children recursively, return True if any succeeds.
Worst case: O(26^d) for d wildcards, but in practice bounded by actual words stored."
🔮 Likely Follow-up Questions
  • "How would you handle deleteWord?" → Set is_end=False; optionally prune leaf nodes upward with a post-order check
  • "What if '.' appears at every position?" → Already handled — branching at every level. O(26^m) worst case, which you'd mention upfront
  • "How would you get all matching words, not just a boolean?" → Change DFS to collect results rather than short-circuit on first match
  • "Can you use '*' for zero-or-more characters?" → Different problem — that's regex matching (#10, #44), needs DP

7 Summary & Key Takeaways ▾
A Trie stores words character by character, sharing common prefixes — addWord is O(m) and space-efficient for large vocabularies with shared prefixes
Literal search follows a single root-to-leaf path: O(m) — same as hash lookup for exact matching
Wildcard '.' at a node triggers DFS into all existing children — branching factor equals the actual children count, not 26
The base case checks is_end, not just "reached a node" — this correctly distinguishes words from their prefixes
This Trie + DFS pattern extends naturally: multiple wildcards, deleteWord, getAllMatches, autocomplete suggestions
