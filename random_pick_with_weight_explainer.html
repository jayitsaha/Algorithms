LeetCode #528

Random Pick with Weight

Implement pickIndex() to return a random index proportional to its weight — heavier weights must be chosen more often.

Medium Binary Search Prefix Sum Time: O(log n) per pick Space: O(n)
1 Understanding the Problem ▾

You are given an array w where w[i] is the weight of index i. Implement a class with two methods: __init__(w) and pickIndex(). Each call to pickIndex() must return a random index such that the probability of returning index i equals exactly w[i] / sum(w).

Example: w = [1, 3, 2] — total is 6. Call pickIndex() many times and you should see index 0 about 1/6 of the time, index 1 about 3/6 = 50% of the time, and index 2 about 2/6 of the time. Heavier = more likely.

💡 Key Insight

Think of each weight as a segment on a number line from 0 to total_weight. Pick a random point anywhere on that line. Whichever segment the point lands in — return that index. The probability of landing in segment i equals exactly w[i] / total by uniform distribution. Prefix sums give us the boundaries; binary search finds the segment in O(log n).

Edge cases: Single element (always return 0 — prefix=[w[0]], target always in only segment), all weights equal (uniform random), very large weights (prefix sum fine in Python), one weight much larger than others.

2 The Algorithm: Prefix Sum + Binary Search ▾

A two-phase approach: build the "weighted ruler" once in __init__, then use binary search every time pickIndex is called. Construction is O(n); each pick is O(log n).

1 Build prefix sums (constructor): Convert w = [1, 3, 2] to running totals prefix = [1, 4, 6]. Each value is the right boundary of that index's segment. After this, prefix[i] = w[0] + w[1] + ... + w[i].
2 Pick a random target: target = random.uniform(0, prefix[-1]). This generates a uniformly random real number in [0, total_weight) — a random point on the weighted number line.
3 Binary search for the segment: Find the leftmost index where prefix[i] > target. Since prefix sums are strictly increasing (all weights are ≥ 1), this is a standard lower-bound binary search. The answer is the first index whose cumulative sum exceeds the target.
4 Return the index: The leftmost index where prefix[i] > target is the segment the random point fell into. Return that index. This is equivalent to bisect.bisect_right(prefix, target) in Python.
🔑 bisect_right vs bisect_left

We use bisect_right(prefix, target) which returns the insertion point after any equal elements. This means if target == prefix[i], we return i+1 — the next segment's start. This is correct: segment i covers the half-open interval (prefix[i-1], prefix[i]], so an exact hit on the upper boundary belongs to segment i. With random.uniform, exact integer hits have probability zero, so both variants work in practice. bisect_right is the safe, idiomatic choice.

3 Why It Works: The Invariant ▾
🔐 The Binary Search Invariant

At every iteration, the correct answer (the index of the segment containing target) lies within [lo, hi]. We halve the search space each step by asking: "Does prefix[mid] exceed the target?" If yes, the answer is at mid or to its left. If no, the answer is strictly to the right. When lo == hi, we have found it.

Probability correctness — proof sketch: Index i is returned when the random target falls in the interval (prefix[i-1], prefix[i]] (define prefix[-1] = 0). The length of this interval is exactly w[i]. Since target is uniform on [0, prefix[-1]), the probability of returning index i is w[i] / sum(w). All intervals are disjoint and tile the full range — the probabilities sum to 1.

Why the prefix array is always sorted: All weights are positive integers (w[i] ≥ 1 by constraint). So each prefix sum is strictly greater than the previous one. A sorted array is required for binary search to work correctly — verified by the constraints.

⚠️ Common Mistake: Using random.randint vs random.uniform

Both work, but with different boundary handling. If using random.randint(1, total) (integer target), use bisect_left(prefix, target) — because segment boundaries are integers and we want the leftmost prefix that is ≥ target. If using random.uniform(0, total) (float target), use bisect_right(prefix, target) — to find the first prefix strictly greater than the float target. Mixing these up causes off-by-one errors on exact boundaries.

4 The Code (Annotated) ▾

Solution 1 — Prefix Sum + bisect_right (Optimal, Interview Pick)

import random, bisect
class Solution:
def __init__(self, w: list[int]): ← Called once; amortized cost spreads over all picks
self.prefix = [] ← Cumulative sum array; will be strictly increasing
total = 0 ← Running sum tracks right boundary of current segment
for wt in w: ← One pass through weights — O(n)
total += wt ← Advance the right boundary by this weight
self.prefix.append(total) ← prefix[i] = sum of w[0..i], the right edge of segment i
def pickIndex(self) -> int: ← Called many times; must be O(log n) per call
target = random.uniform(0, self.prefix[-1])← Uniformly random real in [0, total_weight)
return bisect.bisect_right(self.prefix, target)← First prefix > target = segment we landed in
⚠️ Why bisect_right not bisect_left here?

bisect_right(arr, target) finds the position after any value equal to target. Since our target is a float from random.uniform and prefix values are integers, equality almost never happens — both work in practice. But bisect_right is the correct choice: if somehow target == prefix[i], it returns i+1, placing the boundary point in the next segment. The interval for index i is (prefix[i-1], prefix[i]] — closed on the right — so the boundary belongs to i, not i+1. However, with random.uniform(0, total) this scenario has probability zero. Stick with bisect_right.

Solution 2 — Expand to flat list (Brute Force, for contrast)

class Solution:
def __init__(self, w):
self.pool = [] ← Expand each index by its weight count
for i, wt in enumerate(w):
self.pool.extend([i] * wt) ← e.g., w=[1,3,2] → pool=[0,1,1,1,2,2]
def pickIndex(self) -> int:
return random.choice(self.pool) ← O(1) pick, but O(sum(w)) space — BAD for large weights
💡 Complexity Summary

Prefix Sum + bisect_right: Init O(n), pickIndex O(log n), Space O(n)
Expand to flat list: Init O(n + sum(w)), pickIndex O(1), Space O(sum(w)) — breaks when weights reach 10⁵
Alias Method (advanced): Init O(n), pickIndex O(1), Space O(n) — used in production ML systems; harder to implement
Interview answer: Prefix Sum + Binary Search. Handles any weight values, clean to implement.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognise the weighted random sampling pattern

Key signals: "pick randomly" + "probability proportional to weight". This is a classic pattern. The mental model is: assign each index a segment of a number line proportional to its weight, then throw a dart at the line uniformly at random.

Step 2: Propose the brute force — expand to flat array

"Simplest approach: expand index i into the pool w[i] times. Use random.choice. O(1) picks, but space is O(sum(w)). If weights are up to 10⁵, the pool could have 10⁸ entries. Too slow to build, too much memory."

Step 3: Derive the optimisation via number-line insight

"Instead of materialising the expanded pool, I represent the segments using prefix sums — compact O(n) structure. To find which segment a random point falls in, binary search the sorted prefix array. O(n) build, O(log n) per pick."

Step 4: Code and verify on the example

"w=[1,3,2] → prefix=[1,4,6]. target=2.8. bisect_right([1,4,6], 2.8)=1. Return index 1. Check: 2.8 is in (1,4], which is index 1's segment of width 3/6 — correct probability 50%."

🗣️ What to Say to the Interviewer

"I'll model this as a weighted number line. Each index gets a segment of width equal to its weight.
In __init__, I build a prefix sum array — O(n) time and space.
In pickIndex, I generate target = random.uniform(0, total_weight), then binary search the prefix
array for the leftmost entry exceeding target. That's my answer. O(log n) per pick.
Python's bisect.bisect_right handles this exactly. Let me code it up..."
🔮 Likely Follow-up Questions
  • "What if weights are updated after construction?" → Use a Fenwick tree (BIT) for O(log n) updates + O(log n) prefix queries
  • "Can you achieve O(1) picks?" → Yes: Vose's Alias Method — O(n) init, O(1) pick, same O(n) space
  • "How would you unit-test this?" → Run N picks, compare empirical frequency to expected w[i]/total with a chi-squared test
  • "What does Python's random.choices do internally?" → Exactly prefix sums + bisect — same algorithm!

7 Summary & Key Takeaways ▾
Convert weights to prefix sums once in __init__ — this builds the "weighted number line" in O(n)
A random float in [0, total) falls into segment i with probability w[i]/total — correct by uniform distribution
Binary search on the sorted prefix array locates the correct segment in O(log n) per call
Python's bisect.bisect_right is the one-liner that handles all boundary cases cleanly
For dynamic weights (updates between picks), upgrade to a Fenwick tree for O(log n) updates
