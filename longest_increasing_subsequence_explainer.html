LeetCode #300

Longest Increasing Subsequence

Given an integer array nums, return the length of the longest strictly increasing subsequence.

Medium Dynamic Programming Binary Search DP: O(n²) · Patience: O(n log n) Space: O(n)
1 Understanding the Problem ▾

A subsequence is formed by deleting some (or no) elements from the array without changing the relative order of the remaining elements. We want the longest one that is strictly increasing — each element must be strictly greater than the one before it.

💡 Key Insight

We are not looking for a contiguous subarray — we can skip elements. But we cannot reorder them. The problem reduces to: for each element, what is the longest increasing run that can end at exactly that position?

✓ Valid LIS from [10,9,2,5,3,7,101,18]

[2, 5, 7, 18] — length 4
[2, 5, 7, 101] — length 4
[2, 3, 7, 18] — length 4
Answer: 4

✗ Invalid — these are NOT valid

[3, 2, 7] — not increasing
[5, 5, 7] — must be STRICTLY increasing
[101, 7] — reversed (must preserve order)

Edge cases: single element → 1. All same values → 1. All decreasing → 1. Already sorted → n.

2 The Algorithm: Two Approaches ▾

Approach A — O(n²) DP (Classic LIS)

Define dp[i] = length of the longest increasing subsequence ending exactly at index i. Base case: every element alone is LIS of length 1. Then for each i, look back at all j < i: if nums[j] < nums[i], we can extend the subsequence ending at j.

1 Initialise: dp = [1] * n. Base case: each element is its own LIS of length 1.
2 Fill i = 1 to n−1: For each i, scan all j < i. If nums[j] < nums[i], update dp[i] = max(dp[i], dp[j] + 1).
3 Answer: max(dp) — the LIS can end anywhere, so take the global max.

Approach B — O(n log n) Patience Sort / Binary Search

Maintain a tails array where tails[k] stores the smallest possible tail of any increasing subsequence of length k+1 seen so far. For each element, binary search in tails to find where it fits — extend or replace.

1 Initialise: tails = []. This array remains sorted at all times.
2 For each num: binary search (bisect_left) to find the leftmost position where tails[pos] >= num.
3 Replace or Append: If pos == len(tails), append num (found longer subsequence). Otherwise replace tails[pos] = num (record smaller tail for same length — better future).
4 Answer: len(tails). The length is always correct even though tails may not itself be a valid subsequence.
🔑 Which to Present in Interviews

Lead with O(n²) DP — it directly encodes the recurrence and demonstrates DP understanding. Then offer the O(n log n) patience sort as an optimization. Interviewers often specifically ask for both.

3 Why It Works: DP Invariant & Patience Sort Proof ▾

Why is this Dynamic Programming?

  • Optimal Substructure: The LIS ending at position i is built from the best LIS ending at some earlier j (where nums[j] < nums[i]) plus one element. Sub-problem solutions feed directly into larger solutions.
  • Overlapping Subproblems: Without memoization, computing dp(i) recursively would recompute dp(j) for the same j many times across different outer i values.
📐 The Recurrence
dp[i] = 1 + max(dp[j]  for all j < i where nums[j] < nums[i])
Base case: dp[i] = 1   (the element itself is a LIS of length 1)
Answer: max(dp[0], dp[1], ..., dp[n−1])
🔐 DP Invariant

After processing index i, dp[i] is the definitive length of the longest increasing subsequence ending exactly at nums[i]. All dp values for indices less than i are already final because the inner loop only looks backward.

Why does Patience Sort give the correct length?

tails is always sorted, and tails[k] is the minimum possible tail element of any increasing subsequence of length k+1. Key points:

  • Replacing tails[pos] with a smaller num records that "same-length LIS exists with smaller tail," making future extensions easier.
  • Appending means we found a strictly longer subsequence.
  • The length of tails always equals the LIS length — the elements may not be a valid subsequence themselves.
⚠️ Common Trap: Confusing tails with the actual LIS

After patience sort on [10,9,2,5,3,7,101,18], tails ends as [2,3,7,18]. This happens to be a valid LIS, but in general, replacements can break this. For example, after [3,1,2]: tails=[1,2], but the actual LIS is [1,2] or [3] (length 2). If you need to reconstruct the subsequence, you must track parent pointers in the O(n²) DP.

4 The Code (Annotated) ▾

Solution 1 — O(n²) DP (Classic — start here in interviews)

def lengthOfLIS(nums): ← takes integer list
n = len(nums) ← store once for clarity
dp = [1] * n ← base: every element is a LIS of length 1
for i in range(1, n): ← outer: try to compute LIS ending at i
for j in range(i): ← inner: check all previous indices
if nums[j] < nums[i]: ← strictly increasing: j can precede i
dp[i] = max(dp[i], dp[j] + 1) ← extend best subsequence ending at j
return max(dp) ← LIS can end anywhere, not just last element
⚠️ Why max(dp), not dp[-1]?

The LIS does not have to end at the last element. For [5,1,2,3], dp = [1,1,2,3] and the answer is 3 (ending at index 3), which is dp[-1]. But for [5,4,3,2,1], dp = [1,1,1,1,1] and all are equally valid. And for [3,5,6,2,7], the answer is dp at index 4 (value 4), not dp[-1]=4 which happens to agree here but not always. Always use max(dp).

Solution 2 — O(n log n) Patience Sort with Binary Search (Optimal)

import bisect ← Python standard library binary search
def lengthOfLIS(nums):
tails = [] ← tails[k] = smallest tail of LIS of length k+1
for num in nums: ← process each number once: O(n) iterations
pos = bisect.bisect_left(tails, num) ← O(log n): leftmost pos where tails[pos] >= num
if pos == len(tails): ← num larger than all tails → new longer LIS
tails.append(num) ← extend tails array
else:
tails[pos] = num ← replace: smaller tail = easier future extension
return len(tails) ← length of tails = LIS length
💡 Complexity Summary

O(n²) DP: Time O(n²) two nested loops, Space O(n) for dp array
O(n log n) Patience: Time O(n log n) binary search per element, Space O(n) for tails array
Both use O(n) extra space. For n ≤ 2500, O(n²) is fine. For n up to 10⁵, use patience sort.

5 🎤 Interview Strategy: How to Arrive at This ▾
Step 1: Recognize the DP Signal

"This is a subsequence optimization problem. Subsequences + maximization = think DP. The LIS sub-pattern specifically: dp[i] = best answer ending at index i. This anchoring trick is how we break the problem into independent subproblems."

Step 2: Define the State Precisely

"Let dp[i] = length of the longest increasing subsequence ending at exactly index i. The key word is 'ending at' — I fix the right endpoint and optimize over all choices for the left portion."

Step 3: State the Recurrence

"dp[i] = 1 + max(dp[j] for j < i where nums[j] < nums[i]). Every element starts at 1 (itself). We extend the best prior subsequence whose tail is smaller than nums[i]."

Step 4: Propose the O(n log n) Optimization

"If n is large: maintain a sorted tails array using patience sorting. For each new number, bisect_left finds insertion point in O(log n). Replace or append. Length of tails = answer. This is O(n log n) total."

🗣️ What to Say to the Interviewer

"This is a subsequence optimization problem — DP is the right tool.
I'll define dp[i] = length of LIS ending at index i. Base case: dp[i] = 1 for all i.
For each i, I scan all j < i. If nums[j] < nums[i], update dp[i] = max(dp[i], dp[j]+1).
Return max(dp). Time O(n²), Space O(n).

For O(n log n): I maintain a sorted tails array. For each number,
bisect_left finds where it belongs. If past the end, append (longer LIS found).
Otherwise replace that position (smaller tail = better). Return len(tails)."

  • "Longest / count something in a subsequence keeping relative order" → LIS-style DP
  • Elements can be skipped, order must be preserved, optimize a monotone condition
  • If "2D ordering" or "envelopes" → sort one dimension, run LIS on the other
